# Innovation Prior

Reconstructed first-person **reasoning traces** for famous and recent papers.

Finished papers are written theorem-first: the polished answer arrives with the struggle
edited out. **Innovation Prior** runs that backwards. For each paper, the method is rewritten
as a first-person *discovery* trace â€” re-derived from the pain points and prior art of its
time, hitting walls and self-correcting until it lands on the idea and the code.

This repository is also a static website (GitHub Pages, served from the repo **root**) that
showcases those traces.

## The three artifacts

Every method ships three Markdown files under `methods/<slug>/results/`:

| File | What it is |
| --- | --- |
| `context.md` | The pre-method ground: the research question, prior art, and exactly what hurt at the time. Uses `##` section headers. |
| `reasoning.md` | The long-form first-person derivation trace â€” walls and self-corrections included. **The main event.** Long continuous prose with unicode math and some LaTeX. |
| `answer.md` | The distilled summary plus reference code (code blocks) â€” where the trace lands. |

These artifacts are generated by the **paper-to-reasoning** skill, which transforms a finished
paper into the chain of thought of how the method was actually derived.

## SFT training data

Beyond the website, the traces are assembled into **supervised fine-tuning data** in the
LLaMA-Factory ShareGPT format. Build scripts and full documentation live in
[`sft/`](sft/README.md). The processed data ships **gzipped** in `sft/`
(`innovation_sft.jsonl.gz`, `maintain_sft.jsonl.gz` â€” `gunzip -k` to use); the raw `.jsonl` are
git-ignored and regenerable with the scripts. Two datasets, trained **together in one mixed run**:

- **`innovation_sft.jsonl`** (`sft/build_sft.py`) â€” our annotated data: each method as a Q&A, each
  trajectory / agentic ladder as a multi-turn conversation (with tool use), the answer being the
  scientist write-up (`train_answer`), reasoning in `<think>`, and the discovery **year** injected
  into every system prompt as meta-conditioning. Multi-turn data is emitted in two framings â€”
  *full* (history keeps its reasoning) and *folded* (prior rounds' reasoning emptied, only the
  current round trained) â€” so the model learns to derive against both kinds of history.
- **`maintain_sft.jsonl`** (`sft/build_maintain.py`) â€” a capability-**maintenance** set: public
  **Qwen**-distilled traces (khazarai, WithinUsAI, armand `pi`/Claude-Code, nvidia Open-SWE) mixed
  in as on-policy replay to preserve the base Qwen model's original abilities (against catastrophic
  forgetting). Includes both reasoning and Qwen non-reasoning traces, plus reconstructed tool
  declarations for the agentic ones.

### Patched LLaMA-Factory

Mixing all of this in **one** training run needs two per-example controls that upstream
LLaMA-Factory lacks, so we maintain a fork:

> **https://github.com/Imbernoulli/LLaMA-Factory** â€” branch **`feat/per-turn-loss-mask`**
> (`INNOVATION_PRIOR.md` there documents exactly what changed and why)

1. **Per-turn `loss`** â€” a `"loss": false` turn stays as context but is excluded from the loss.
   This folds prior rounds while still training **every** action of the current round â€” which
   `mask_history` (all-turns or last-turn-only) cannot express.
2. **Per-example `enable_thinking`** â€” an `"enable_thinking": false` example renders its empty
   `<think></think>` into the *prompt*, not the loss, so **non-reasoning** data (e.g. Qwen's
   non-thinking traces) trains in the same thinking run **without** teaching the model to emit an
   empty think block. *(Verified: such data trains zero `<think>` tokens.)*

To train: fill the two stub paths in the fork's `data/dataset_info.json`, then run a single config
with `dataset: innovation_sft,innovation_maintain`, `template: qwen3` (or `qwen3_5`),
`mask_history: false`. The per-example metadata is baked into the data, so one global config trains
every case correctly. Full details: [`sft/README.md`](sft/README.md).

## The website

A pure-static single-page app â€” no build step, all relative paths, so it works as-is on
GitHub Pages.

- `index.html` â€” the app shell (hero/intro, sidebar, tabbed reading pane).
- `assets/style.css` â€” clean, responsive, light/dark via `prefers-color-scheme`; comfortable
  long-form reading column (~760px); horizontally scrollable code blocks; sidebar collapses
  on mobile.
- `assets/app.js` â€” data-driven logic: loads `methods.json`, builds the domain-grouped
  sidebar with search, and fetches `methods/${slug}/results/${tab}.md` on demand.
- `methods.json` â€” the single source of truth: an array of `{slug, title, domain, arxiv}`.

Rendering uses CDN libraries (jsDelivr): **marked.js** for Markdown, **KaTeX** + auto-render
for math (`$...$`, `$$...$$`, `\(...\)`, `\[...\]`, rendered *after* Markdown), and
**highlight.js** for code. The default tab is **Reasoning**. Routes are deep-linkable via the
URL hash (`#<slug>/<tab>`) with working browser back/forward.

### Running locally

The app fetches Markdown over `fetch()`, which browsers block on the `file://` protocol, so
serve it over HTTP:

```bash
python3 -m http.server 8000
# then open http://localhost:8000
```

On GitHub Pages, enable Pages for the repository root branch â€” nothing else is required.

## Adding a method

1. Create the folder `methods/myslug/results/` and drop in `context.md`, `reasoning.md`, and
   `answer.md` (e.g. generated by the paper-to-reasoning skill).
2. Add one entry to `methods.json`:

   ```json
   { "slug": "myslug", "title": "My Paper Title", "domain": "Some Domain", "arxiv": "2401.00000" }
   ```

   Entries are ordered by domain, then title. New domains automatically get their own
   collapsible group in the sidebar.

That's it â€” the site is fully data-driven from `methods.json` and references only the methods
listed there. Other top-level folders (each method's `src/`, `code/`, `notes/`, `refs/` raw
materials, which are gitignored) are never scanned or exposed.
