# Slitting Coils When Knives Crawl, Not Jump

## Problem
A slitter cuts a coil of width `W` lengthwise into narrower strips using `K` knives.
The knife positions `0 < p_1 < p_2 < ... < p_K < W` split the coil into `K+1` segments
of widths `p_1-0, p_2-p_1, ..., W-p_K` (a **setup**). A segment whose width exactly
equals one of the `n` ordered widths counts as product of that width; any other
segment is **deckle trim** (waste). Running a setup for `r` linear units produces `r`
copies of each segment and consumes `r*W` of coil material, regardless of how much of
that width was actually useful.

You run a **sequence** of setups (a campaign) so that, summed over all setups, the
copies produced of each demanded width meet its quantity. Between *consecutive*
setups, every one of the `K` knives may move by **at most `m` millimeters** (knives
are physical blades on a rail: they crawl, they cannot jump). Your first setup is
likewise bounded by `m` from the given starting positions `S`.

Choosing the widths of one setup freely (a **multiset** summing to `W`) is easy in
isolation; the difficulty is that a setup with excellent widths may sit far from the
setups you need before and after it, forcing costly detours just to get the knives
there. A high-quality campaign trades pattern quality against pattern reachability.

## Input (stdin)
```
W K m
S_1 S_2 ... S_K
n
width_1 qty_1
...
width_n qty_n
maxSetups setupCost penalty
```
`S` is strictly increasing, each in `[1, W-1]`. Each `width_i` is a distinct integer in
`[1, W-K-1]` (room for the other `K-1` real slots plus a positive remainder);
`qty_i >= 1`. `maxSetups` bounds how many setups you may use.

## Output (stdout)
```
T
p_1 p_2 ... p_K r
   (T lines total, one per setup, in campaign order)
```
`1 <= T <= maxSetups`. Each line gives the `K` knife positions of that setup (strictly
increasing integers in `[1, W-1]`) and its run length `r >= 1`.

## Feasibility (violating ANY of these scores `Ratio: 0.0`)
- `1 <= T <= maxSetups`; each `p_j` is an integer in `[1, W-1]`; `p_1 < p_2 < ... < p_K`.
- Each `r >= 1`.
- **Knife-motion bound**: for setup 1, `|p_j - S_j| <= m` for every knife `j`; for every
  later setup `t`, `|p_j(t) - p_j(t-1)| <= m` for every knife `j`.
- No trailing tokens after the campaign.

Demand is **not** a hard constraint -- you may finish with unmet quantity, but it is
penalized heavily in the objective below (see Scoring).

## Objective
Minimize
```
F = TotalMaterial + setupCost * T + penalty * Shortfall
TotalMaterial = sum over setups of (r * W)
Shortfall     = sum over widths i of max(0, qty_i - produced_i) * width_i
```
where `produced_i` is the total copies of `width_i` across all setups (any segment
counts toward `width_i` whenever its width exactly equals `width_i`, regardless of
slot).

## Scoring
The checker builds its own baseline `B`: visit each demanded width once, **in the
input order given**, using a pattern that repeats that width as many times as fits
(up to `K` copies) -- but produce only **one** copy per run (no batching) and bridge
to the next width with no reordering. This always meets demand. Then:
```
sc = min(1000.0, 100.0 * B / max(1, F))
Ratio = sc / 1000.0
```
Matching the baseline scores `Ratio = 0.1`; a campaign roughly `10x` cheaper caps at
`Ratio = 1.0`.

## Constraints
- `30 <= W <= 3000`, `2 <= K <= 8`, `1 <= n <= 16`, `1 <= m <= W`.
- Time limit 5s, memory 512MB.

## Example
`W=60 K=2 m=25`, `S=(20,40)`, demand: `width=15 qty=3`, `width=25 qty=2`,
`maxSetups=10 setupCost=50 penalty=6`.

A valid campaign:
```
2
15 30 2
25 50 1
```
Setup 1 (`15,30`, moves `5,10 <= 25` from `S` OK) has segments `15,15,30`, producing
`2*2=4` copies of width 15 (demand 3, met) for `2*60=120` material. Setup 2 (`25,50`,
moves `10,20 <= 25` OK) has segments `25,25,10`, producing `1*2=2` copies of width 25
(demand 2, met) for `60` material. `TotalMaterial=180`, `+setupCost*T=100`,
`Shortfall=0`, so `F = 280`.
The baseline visits width 15 then width 25 in input order: reposition to `(15,30)`
(1 setup) then 3 single-unit visits there, then reposition to `(25,50)` (1 setup)
then 2 single-unit visits. `setups=1+3+1+2=7`, `material=420`, `+setupCost*7=350`,
so `B = 770`. `sc = min(1000, 100*770/280) = 275.0`, so `Ratio = 0.275`.
