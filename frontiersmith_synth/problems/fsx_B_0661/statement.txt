# Quarry Ignition: Routing a Chain Detonation Through Shielded Charges

## Problem

A quarry contains `N` charges, numbered `1..N`, placed at integer points on a plane.
Charge `i` has a value `v_i`, a blast radius `r_i`, a detonation threshold `t_i`, and
an **ignitable** flag `c_i`. If `c_i = 1` you may set the charge off directly; if
`c_i = 0` it is **shielded** and can only be set off by the chain reaction itself.

Charge `i`'s blast reaches charge `j` (`i != j`) exactly when the Euclidean distance
between them is at most `r_i`. This reach relation is **directed**: `i` reaching `j`
does not imply `j` reaches `i`, since radii differ.

You choose exactly `M` **distinct** ignitable charges to detonate first (your
*ignition set* `S`). The chain reaction then runs to completion in deterministic,
simultaneous rounds:

* Round 0: every charge in `S` detonates (direct ignition bypasses the threshold).
* Round `k+1`: any not-yet-detonated charge `j` detonates once the number of
  **distinct** charges that were already detonated by round `k` and whose blast
  reaches `j` is `>= t_j`.
* The process stops when a round detonates nothing new. The final detonated set is
  the **closure** of `S` under the reach relation and the thresholds -- it does not
  depend on round order, only on `S`.

Your score is the total value of every charge in the closure. A charge with `r_i = 0`
can still be detonated by others; it just never triggers anything itself.

**The trap:** igniting a tight cluster of charges that each reach many neighbours
maximizes what you can see in one glance (lots of immediate blasts), but a shielded
high-value charge may need `t_j >= 2` *distinct*, well-separated chains to arrive
simultaneously -- something a dense local cluster never routes to. Likewise, the
single highest-value charges are often isolated (small `r_i`) and detonate nothing
beyond themselves. Reasoning about the full multi-hop closure -- and which sparse set
of entry points routes through the shielded gaps -- is what wins.

## Input

```
N M
x_1 y_1 v_1 r_1 t_1 c_1
...
x_N y_N v_N r_N t_N c_N
```

## Output

Print `M` distinct integers in `1..N`: the indices of your ignition set. Every
printed index must have `c_i = 1`.

## Feasibility

Feasible iff exactly `M` integers are printed, all in `[1,N]`, all pairwise
distinct, and every one has `c_i = 1`. Any violation (including malformed tokens)
scores `0`.

## Scoring

Let `F` be the total value of the closure of your ignition set. Let `B` be the total
value of the closure of the **baseline** ignition set: the `M` ignitable charges with
the largest `v_i` (ties broken by smaller index) -- picked by value alone, with no
regard to reach. The generator guarantees enough ignitable charges exist and `B > 0`.

```
score = min(1000, 100 * F / B) / 1000
```

so the value-only baseline always scores exactly `0.1`; routing `10x` more value than
it caps the score at `1.0`.

## Constraints

* `1 <= N <= 250`, `1 <= M <= 15`, and `M` never exceeds the number of ignitable
  charges.
* `0 <= x_i, y_i <= 20000`, `1 <= v_i <= 5000`, `0 <= r_i <= 400`, `1 <= t_i <= 4`,
  `c_i in {0,1}`.
* Time limit: 4 s. Memory limit: 512 MB.

## Example

```
5 2
0 0 3 6 1 1
12 0 3 6 1 1
6 0 100 0 2 0
500 500 50 0 1 1
500 600 40 0 1 1
```

Charges 1 and 2 are cheap, short-range charges `6` apart from shielded charge 3
(value `100`, threshold `2`). Charges 4 and 5 are expensive, isolated (`r=0`)
decoys.

Baseline picks the two highest-value ignitable charges, `{4, 5}`: neither reaches
anything, so the closure is just `{4,5}`, `B = 90`, giving the calibration score
`0.1`.

Igniting `{1, 2}` instead: both reach charge 3 (distance `6 <= 6`), and both are
distinct sources, so `2 >= t_3 = 2` and charge 3 detonates too. Closure
`= {1,2,3}`, `F = 3+3+100 = 106`. Score `= min(1000, 100*106/90)/1000 ≈ 0.1178` --
better, by routing two cheap charges through the shielded gap instead of grabbing
the two priciest isolated ones.
