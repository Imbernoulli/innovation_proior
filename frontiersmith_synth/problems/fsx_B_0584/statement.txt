# Compiling a Task Graph on a Tiny Scratchpad

## Problem
You compile `N` tasks on a machine whose only fast memory is a scratchpad that holds `C`
data pages, managed as an **exact LRU cache** (least-recently-used eviction). Each task
reads a small set of data pages. The tasks form a DAG: some tasks must be compiled before
others. You choose the compilation order — any linear order that respects the DAG — to
**minimize the number of cache misses**.

When a task is compiled it accesses its pages **in increasing page-id order**. Accessing a
page already resident is a hit; otherwise it is a miss: the page is loaded, and if the
scratchpad is full the least-recently-used page is evicted first. The scratchpad starts
empty. A page access moves that page to most-recently-used.

The catch is that a page can be shared by tasks scattered all over the DAG. Whether such a
page stays resident between two of its uses depends entirely on how many *distinct* pages
you touch in between — i.e. on the **reuse-distance profile your ordering authors**. Simply
clustering tasks that share a page tightens local reuse but can push a globally-recurring
page's reuse distance past `C`, so it is reloaded again and again. Doing well means shaping
the whole reuse-distance histogram under the DAG constraints, not just chasing the nearest
shared page.

## Input
```
N M C E
(N lines) k  p_1 p_2 ... p_k      # task i touches k pages (1<=k<=4), 0<=p_j<M, strictly increasing
(E lines) u v                      # precedence edge: task u must be compiled before task v
```
Tasks are numbered `1..N`; pages `0..M-1`. Every edge has `u<v`, so the identity order
`1,2,...,N` is always a valid compilation order (the reference order).

## Output
Print a permutation of `1..N`: your compilation order (space- or newline-separated).

## Feasibility
The output must be a permutation of `1..N`, and for every precedence edge `(u,v)` task `u`
must appear before task `v`. Any other output scores 0.

## Objective
Let `F` be the total number of LRU cache misses produced by simulating your order under the
model above (cold misses plus capacity misses). **Minimize `F`.**

## Scoring
Let `B` be the number of misses of the reference order `1,2,...,N`. Your score for a test is
```
score = min(1.0, 0.1 * B / F)
```
Lower misses give a higher score. Reproducing the reference order scores `0.1`; halving its
misses scores `0.2`; the score is capped at `1.0`. The final score is the mean over all tests.

## Constraints
- `1 <= N <= 4000`, `2 <= C <= M <= ~1200`, `1 <= k <= 4`.
- Time limit: 4 s. Memory limit: 512 MB. Each input file is at most 5 MB.
- Scoring is fully deterministic.

## Example
```
9 5 3 6
2 0 2
2 1 3
2 0 4
1 2
1 3
1 4
1 2
1 3
1 4
1 4
2 5
3 6
4 7
5 8
6 9
```
Here page `0` is shared by tasks `1` and `3` (two different cluster roots), the cache holds
`C=3` pages, and tasks form three chains `1->4->7`, `2->5->8`, `3->6->9`. The reference order
`1..9` interleaves the chains and suffers `B=8` misses. Merely clustering each chain together
still reloads page `0` and scores only `7` misses. The order `1 4 7 3 6 9 2 5 8` compiles the
two chains that both touch page `0` back-to-back, so page `0` stays resident and only `F=5`
misses occur:
```
score = 0.1 * 8 / 5 = 0.160
```
