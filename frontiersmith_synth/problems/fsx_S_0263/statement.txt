# Neon Midway: Simultaneous Ride Circuit

## Problem

The **Neon Midway** carnival runs a sprawling circuit of thrill rides. During the
grand *Night Surge* show, the operator wants to light up as many rides as
possible **at the same instant** to maximize ticket revenue — but not every pair
of rides can run simultaneously.

Ride `i`, if switched on for the surge, earns an estimated revenue `w[i]`
(headliner coasters and haunted mansions pull far more than the kiddie carousels).

Certain **pairs of rides cannot run together**: two rides may draw from the same
power substation, share a crowd-flow lane on the midway that would jam, or trip
a common safety interlock. The engineers hand you a list of such **conflicting
pairs**. If two rides conflict, at most one of them may be lit during the surge.

Choose a set of rides to light — a set containing **no conflicting pair** — that
maximizes the total revenue.

Formally this is a **maximum-weight independent set** on the general conflict
graph whose vertices are rides and whose edges are conflicting pairs. At this
scale it is NP-hard, the optimum is unknown, and many heuristics (weight-greedy,
weight-per-degree greedy, randomized restarts, local-search swaps) are all
viable and produce genuinely different circuits.

## Input

The first line contains two integers:

```
n m
```

- `n` — number of rides, numbered `1..n`.
- `m` — number of conflicting pairs.

The second line contains `n` integers `w[1] w[2] ... w[n]` — the surge revenue of
each ride (`1 <= w[i] <= 1000`).

Each of the next `m` lines contains two integers:

```
u v
```

meaning rides `u` and `v` conflict (`1 <= u, v <= n`, `u != v`). Each unordered
pair is listed at most once; conflicts are symmetric.

## Output

On the first line print `c` — the number of rides you light (`0 <= c <= n`).
On the following lines (or the same line, whitespace-separated) print the `c`
chosen ride indices, each in `1..n`, all distinct.

Printing `1` followed by any single ride index is always valid (one lit ride
never conflicts with itself).

## Feasibility

An output is feasible iff:

- `0 <= c <= n`;
- every printed index is in `1..n` and all `c` indices are distinct;
- no two chosen rides form a conflicting pair.

Any infeasible output scores `0`.

## Objective

Let `F` be the total revenue of your lit rides (sum of `w[i]` over chosen rides).
You are **maximizing** `F`.

## Scoring

The checker builds a reference circuit `B` by the **index-order greedy** rule:
scan rides `1, 2, ..., n` and light each ride whenever it conflicts with no
already-lit ride. Let `B` be that reference circuit's total revenue. The score
for a test is

```
ratio = min(1.0, 0.1 * F / B)
```

Reproducing the index-order reference circuit scores exactly `0.1`; reaching a
ten-fold-of-reference revenue caps the score at `1.0`. The overall score is the
mean ratio across all tests.

## Constraints

- `1 <= n <= 5000`.
- `0 <= m <= n*(n-1)/2`.
- `1 <= w[i] <= 1000`.
- Time limit: 5 s. Memory limit: 512 MB.

## Example

Input:

```
5 4
10 8 7 9 6
1 2
2 3
3 4
4 5
```

The conflict graph is a path `1-2-3-4-5`.

Index-order greedy reference: light ride `1` (revenue 10), skip `2` (conflicts
with 1), light `3` (revenue 7), skip `4` (conflicts with 3), light `5`
(revenue 6). So the reference circuit is `{1,3,5}` with `B = 10 + 7 + 6 = 23`.

Consider lighting `{2, 4}`: revenue `8 + 9 = 17`, which is feasible (2 and 4 do
not conflict). A valid worked output:

```
2
2 4
```

scores `ratio = min(1.0, 0.1 * 17 / 23) = 0.0739`. Reproducing `{1,3,5}` scores
`ratio = min(1.0, 0.1 * 23 / 23) = 0.1`. The optimal circuit here is `{1,3,5}`
with revenue `23`.
