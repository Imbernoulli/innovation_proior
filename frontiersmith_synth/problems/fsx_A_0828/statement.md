# The One-Big-Term Sequence

An integer sequence `a(0), a(1), a(2), ...` is generated by a hidden,
constant-coefficient **linear recurrence** of some order `D` (unknown to
you). Equivalently, its ordinary generating function is a **rational
function** `P(x)/Q(x)` whose denominator factors into `D` distinct linear
factors, so the sequence is an exact sum of pure geometric terms:

```
a(n) = c_1 * r_1^n + c_2 * r_2^n + ... + c_D * r_D^n
```

for fixed integers `c_i` (coefficients) and `r_i` (roots, all `>= 2`). You
are shown only the first `T` terms — a short prefix. Your job is to predict
`a(n)` **far** beyond `n = T`, at indices many multiples of `T` away.

## Input (stdin)

```
T t
0  a(0)
1  a(1)
...
T-1  a(T-1)
```

`t` is the test id (informational only). `T` training rows follow, each an
index and the (possibly large, possibly negative) integer value at that
index.

## Output (stdout): a closed-form expression

Print **one line**: a single arithmetic expression in the variable `n`,
using only `+ - * /`, parentheses, numeric constants, and `n` itself (no
function calls, no other names). The grader evaluates your expression at a
set of held-out indices, all strictly greater than `T`, that are **never
shown to you**.

**Illustrative FORM only — NOT the hidden law:**

```
2*n + 5
```

This just shows the required syntax (a plain expression in `n`); the actual
hidden sequences in this problem are sums of geometric terms, not
polynomials, and every test id uses different roots and coefficients that
you must discover from the data.

## Feasibility

The expression must parse under the grammar above (only `n`, numeric
constants, and `+ - * / **`), stay within a generous size/complexity bound,
and evaluate to a finite number at every held-out index. Any violation
scores `0`.

## Objective (minimise)

Let `err_i` be the relative error of your prediction at held-out index `i`
(clipped to `[0,1]`), `MRE` their mean, and `nodes` the number of syntax
nodes in your expression. The grader forms

```
F = MRE + LAMBDA * nodes
B = MRE_of(single-geometric-term-from-last-two-points) + LAMBDA * 5
Ratio = min(1000, 100 * B / F) / 1000
```

with a small fixed `LAMBDA`. Reproducing the checker's own naive baseline
(extrapolate a single ratio from the last two visible points) gives Ratio
close to `0.1`. Lower held-out error raises the score; needlessly many terms
tax it back down. Report the highest Ratio you can.

## Why the visible window is a trap

Every hidden sequence here is dominated, throughout the ENTIRE visible
window, by one huge term. The other terms have tiny coefficients (as small
as `+-1`) — utterly invisible against the dominant term while `n <= T` — but
their roots are slightly larger, so each one is on an exponential collision
course to eventually overtake the dominant term at some index past `T`. A
numerical fit of a small, fixed recurrence order to the visible window will
match it almost perfectly (there is essentially one behaviour to see) and
will extrapolate confidently — and wrongly, by orders of magnitude, once a
held-out index passes a crossover it never knew existed. Only an **exact**
algebraic recovery of the true minimal order (not a numerically "good
enough" fit) can see the small terms that a short window structurally
cannot rule out, and correctly predicts what happens after they take over.

## Constraints

Time limit 5 s, memory 512 MB. `T` is at most a few dozen rows; every
instance file is a few hundred bytes. Scoring is fully deterministic.
