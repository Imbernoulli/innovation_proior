# Halley Station: Flexible Field-Work Schedule

## Problem

A remote polar research base (Halley Station) must complete a batch of **field-work jobs**
before the winter darkness sets in. Each **job** — drilling an ice core, deploying a weather
mast, servicing a seismometer — is a fixed **chain of tasks** that must be carried out **in
order** (you cannot log the ice-core samples before the core has actually been drilled).

Unlike a classic assembly line, the base is **cross-equipped**: most tasks can be handled by
**any one of several shared assets** — the two snow tractors, the drilling rig, the generator
shed, the comms lab — but the *time a task takes depends on which asset you pick* (the older
tractor is slower; the drilling rig does a bore fast but is a bottleneck everyone wants).

Each shared asset can perform only **one task at a time**. Your dispatcher must therefore make
**two coupled decisions** for every task: *which* eligible asset to use, and *when* to start it.

Your goal: choose an asset and a start time for every task so that all precedence and
one-task-per-asset constraints hold, and the **makespan** — the moment the last task finishes —
is as **small** as possible.

This is a *flexible* job-shop scheduling problem. The optimum makespan is NP-hard to compute;
many machine-selection and dispatch strategies are viable and produce different schedules.

## Input

The first line contains two integers:

```
J M
```

- `J` — number of jobs, numbered `1..J`.
- `M` — number of shared assets (machines), numbered `1..M`.

Then the `J` jobs are given in order. Each job starts with a line containing one integer
`k_j`, the number of tasks in job `j`. Each of the next `k_j` lines describes one task, in the
order the tasks must be performed:

```
e  a_1 d_1  a_2 d_2  ...  a_e d_e
```

- `e` — the number of **eligible assets** for this task (`1 <= e <= M`).
- Each pair `a_r d_r` means: if you run this task on asset `a_r` (`1 <= a_r <= M`), it takes
  `d_r` time units (`1 <= d_r <= 99`). The eligible assets `a_r` of a single task are distinct.

## Output

For every task, in the exact order the tasks appear in the input (all tasks of job `1` in
order, then all tasks of job `2`, and so on), print **two integers**:

```
a  s
```

- `a` — the asset you assign to that task; it **must** be one of the task's eligible assets.
- `s` — the start time, a nonnegative integer.

Print one task's pair per line (or any whitespace layout). The output is a flat list of exactly
`2 * (sum_j k_j)` integers.

## Feasibility

If a task is placed on asset `a` at start `s` and its duration on `a` is `d`, it occupies the
half-open interval `[s, s + d)`. An output is feasible iff:

- every chosen asset `a` is one of that task's eligible assets;
- every start time `s` is an integer in `[0, 2*10^9]`;
- **precedence**: within each job, for consecutive tasks `x` then `y`,
  `s(y) >= s(x) + d(x)` (the finish time of `x` on its chosen asset);
- **no overlap**: any two tasks assigned to the *same* asset occupy disjoint intervals.

Any infeasible output scores `0`.

## Objective

Let `F` be the **makespan**: the maximum over all tasks of `s(task) + d(task)`. You are
**minimizing** `F`.

## Scoring

Let `B` be the makespan of the *fully sequential* fallback: for every task pick its **first
listed** eligible asset, and run every task back-to-back on one global timeline. This is always
feasible, and equals the sum of every task's first-listed duration. The score for a test is

```
ratio = min(1.0, 0.1 * B / F)
```

The sequential fallback (`F = B`) scores `0.1`; a schedule ten times shorter (`F = B / 10`)
caps the score at `1.0`. The overall score is the mean ratio across all tests.

## Constraints

- `1 <= J <= 14`, `1 <= M <= 8`.
- Total number of tasks `sum_j k_j <= 200`.
- `1 <= e <= M`; eligible assets of a task are distinct; `1 <= d_r <= 99`.
- Time limit: 5 s. Memory limit: 512 MB.

## Example

Input:

```
2 2
2
2 1 3 2 5
1 2 2
2
1 2 4
2 1 1 2 6
```

Job 1 has task A `(asset 1: 3, asset 2: 5)` then task B `(asset 2: 2)`. Job 2 has task C
`(asset 2: 4)` then task D `(asset 1: 1, asset 2: 6)`. The first-listed durations are
`3, 2, 4, 1`, so the sequential fallback makespan is `B = 3 + 2 + 4 + 1 = 10`.

A valid interleaved schedule:

- task A: asset `1`, start `0`, occupies `[0,3)`
- task B: asset `2`, start `4`, occupies `[4,6)`
- task C: asset `2`, start `0`, occupies `[0,4)`
- task D: asset `1`, start `3`, occupies `[3,4)`

Asset 1 hosts `[0,3)` and `[3,4)` (disjoint); asset 2 hosts `[0,4)` and `[4,6)` (disjoint);
both jobs obey precedence (`4 >= 0+3` and `3 >= 0+4`? no — check: task D starts at 3, task C
finishes at 4, so this particular D placement would violate precedence). Use instead task D:
asset `1`, start `4`, occupies `[4,5)`. Then asset 1 hosts `[0,3)` and `[4,5)` (disjoint) and
precedence holds (`4 >= 0+4`). The makespan is `F = 6`.

Output (asset then start, in task order):

```
1 0
2 4
2 0
1 4
```

Score for this test: `min(1.0, 0.1 * 10 / 6) = 0.16667`.
