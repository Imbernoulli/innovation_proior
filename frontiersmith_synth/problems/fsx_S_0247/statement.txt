# Luminet: Degree-Capped Smart-City Lighting Backhaul

## Problem

A city is rolling out a **smart streetlight network**. The plan lists `N` fixed
locations across the districts. Each location is one of two kinds:

- a **light pole** (a *terminal*): it carries a lamp and a controller and **must** be
  wired into the backhaul so it can be monitored and dimmed centrally;
- a **junction cabinet** (a *Steiner point*): it carries no lamp, but the crews can drop
  a passive splice/relay box there and use it purely to route fibre-power cable more
  cheaply. A junction may be wired in or left unused entirely.

You lay **cable segments**, each joining two locations, costing trench length proportional
to the straight-line distance between them. The finished network must connect **all light
poles** into a single component (poles must be able to reach one another along cable,
possibly hopping through junction cabinets). Junction cabinets that you do not need may be
left out.

Every location has a limited number of cable ports: location `i` can be an endpoint of
**at most `cap_i`** cable segments. You want to wire all the light poles together for the
**smallest total trench length**, never exceeding any port cap.

With unlimited ports and no junctions this would be a spanning tree; the per-location port
caps *and* the freedom to route through optional junction cabinets make this a
degree-constrained Steiner network, for which no efficient optimal algorithm is known.
Heuristics are expected.

## Input

```
N
x_1 y_1 t_1 cap_1
x_2 y_2 t_2 cap_2
...
x_N y_N t_N cap_N
```

- Line 1: integer `N`, the number of locations.
- Each of the next `N` lines describes location `i` (numbered `1..N`): integer coordinates
  `x_i`, `y_i`, a flag `t_i` (`1` = light pole / terminal, `0` = junction cabinet), and the
  port cap `cap_i`.
- All location coordinates are distinct. There are at least `2` light poles, and every
  location has `cap_i >= 2`, so a connected wiring always exists (e.g. a single chain
  through the light poles).

## Output

Print your cable layout as a list of segments.

```
E
a_1 b_1
a_2 b_2
...
a_E b_E
```

- Line 1: integer `E`, the number of cable segments you lay.
- Each of the next `E` lines gives one segment between locations `a_k` and `b_k`
  (`1 <= a_k, b_k <= N`, `a_k != b_k`). Segments are undirected; do not list the same
  unordered pair twice.

## Feasibility

An output is feasible if and only if all of the following hold:

- each segment joins two distinct valid locations, and no unordered pair appears twice;
- for every location `i`, the number of segments incident to `i` is at most `cap_i`;
- **every light pole** (`t_i = 1`) lies in a single connected component of the layout
  (all poles can reach one another). Junction cabinets need not be connected or used.

Any output that violates feasibility scores 0.

## Objective

The **length** of a cable segment between locations `p` and `q` is the Euclidean distance
rounded to the nearest integer:

```
len(p, q) = round( sqrt( (x_p - x_q)^2 + (y_p - y_q)^2 ) )
```

Minimize the total trench length

```
F = sum over all laid segments (a, b) of len(a, b).
```

## Scoring

Let `B` be the length of the **pole-chain baseline**: wire the light poles in input index
order into a single path (consecutive poles by increasing index), ignoring all junction
cabinets. This is always feasible because every pole has `cap_i >= 2`. Your score on a test
is

```
ratio = min(1.0, (B / max(1, F)) / 10)
```

so a layout that merely matches the pole-chain baseline scores `0.1`, one half its length
scores `0.2`, and reaching one tenth of the baseline length (or shorter) reaches the capped
score `1.0`. Larger is better. Per-test scores are averaged.

## Constraints

- `2 <= N <= 600`
- at least `2` light poles (`t_i = 1`)
- `0 <= x_i, y_i <= 10000`
- `2 <= cap_i <= 4`
- Time limit: 5s
- Memory limit: 512 MiB

## Example

Input:
```
5
0 0 1 2
0 10 1 3
10 10 1 2
10 0 1 3
5 5 0 4
```

Locations 1..4 are light poles on the corners of a 10x10 square; location 5 is a junction
cabinet in the centre. The pole-chain baseline wires `1-2, 2-3, 3-4` (three sides), so
`B = 10 + 10 + 10 = 30`.

Consider the star through the central cabinet: `5-1, 5-2, 5-3, 5-4`. Cabinet 5 has degree 4
(cap 4, ok); each pole has degree 1. All poles reach one another through the cabinet, so it
is feasible. Each spoke is `round(sqrt(50)) = 7`, giving `F = 7 + 7 + 7 + 7 = 28` and
`ratio = min(1.0, (30 / 28) / 10) = 0.107...` — using the optional junction cabinet as a
hub beats the pole-only chain.

The chain itself (`1-2, 2-3, 3-4`) has `F = 30 = B` and scores exactly `0.1`.
