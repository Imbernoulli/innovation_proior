Let me start from what actually hurts. I'm fine-tuning a language model on math with RL, and the thing I keep tripping over is the advantage. In PPO the advantage comes out of a learned critic V, and that critic is killing me on two fronts at once: it's a second network the size of the policy, so it roughly doubles my memory and compute on a single GPU, and — worse — in this setting the reward only lands on the very last token of the response, so the critic has almost no signal to learn an accurate per-token value from. It's expensive *and* it's the unreliable part. So my first instinct is the obvious one: get rid of V entirely. But the critic was never decoration; its whole job was to be the baseline that I subtract from the return to reduce the variance of the policy gradient. If I delete it, I have a hole exactly where the baseline used to be, and the entire question becomes: what do I put there?

Let me get the structure of the gradient straight before I choose a baseline, because the answer to "what baseline" lives in that structure. The reward in this task is only at EOS — there is no meaningful intermediate reward, no real per-token credit to assign from the environment. So modeling each token as its own timestep with a bootstrapped value at every step is solving a problem I don't have. The honest model is the bandit one: the whole generation o is a single action, the prompt q is the state, and I'm maximizing E_{o~π_θ}[ R(o,q) ] with the score-function gradient E_{o~π_θ}[ R(o,q) ∇_θ log π_θ(o|q) ]. That's plain REINFORCE on full sequences. And with reward only at the end and a single action, the natural discount is γ = 1 — there's nothing to discount, the return of the action *is* the terminal reward.

Before I commit to that, let me check what the full GAE machinery actually does at γ = λ = 1 with V ≡ 0, since I'd like the critic-free skeleton to be GAE's own limit rather than a separate ad hoc thing. GAE is A_t = Σ_l (γλ)^l δ_{t+l} with δ_t = r_t + γ V(o_{t+1}) − V(o_t). Take V ≡ 0, γ = λ = 1, and put the reward only on the last step: take a length-6 response with r = (0,0,0,0,0,1). Then δ = (0,0,0,0,0,1), and A_t = Σ_{l≥0} δ_{t+l} sums every δ from t onward, so A_t = 1 for every t — I just traced it and all six tokens get advantage 1, the full terminal return, with no bootstrapping. So GAE at this limit collapses to "every token of the response carries the sequence's whole return," which is precisely the bandit picture. Good — the skeleton is full-sequence REINFORCE, γ = 1, and a baseline I have to manufacture from the rewards themselves instead of from a value network.

The variance-reduction fact I'm leaning on: for any baseline b that doesn't depend on the sampled action, E[(R − b) ∇log π] = E[R ∇log π] (because E[b ∇log π] = b E[∇log π] = b·∇∫π = 0), so subtracting b leaves the gradient unbiased while shrinking its variance if b correlates with R. So I'm free to pick any action-independent b, and I want one that's close to R on average. With the critic gone, the cheapest source of "what R typically is for this prompt" is other samples for the same prompt.

So who's already done this and where do they stop? RLOO's move is clean: sample k responses per prompt, and baseline each one with the mean of the *other* k−1, A_i = r(o_i) − (1/(k−1)) Σ_{j≠i} r(o_j). The leave-one-out is the elegant part — the baseline for o_i genuinely doesn't depend on o_i, so the estimator stays exactly unbiased, no fudge. But stare at the variance of that baseline. With k = 4 or 8, I'm estimating "the typical reward for this prompt" from three or seven samples. That's a noisy baseline, and — this is the part that nags me — the advantages it produces for *different* prompts are never put on any common scale. Prompt A might have rewards clustered near 1, prompt B near 0; the leave-one-out advantages from A and from B land in incomparable ranges and get summed into one gradient as if they were on the same footing.

GRPO tries to fix the scale problem head-on: don't just subtract the group mean, also divide by the group standard deviation, A_i = (r_i − mean(r)) / (std(r) + ε), broadcast to every token. Now within each prompt the advantages are standardized to roughly unit spread, so prompts of different reward scales contribute comparably. That feels right at first. But two things start to bother me, and one of them I think I can actually pin down rather than just suspect.

The cheap-to-see problem first: the group is tiny, k = 4 or 8. Picture a prompt where all k sampled responses happen to get the same reward — easy prompt everyone solves, or a hard one everyone fails. I have to be precise here: under the exact formula A_i = (r_i − mean)/(std + ε), the centered numerator vanishes with the std, so the standardized residual is not literally unbounded; it is bounded. The problem is still real, just subtler. A four-sample std is a brittle scale estimate: exact ties give no ranking signal, one discrete reward flip can create an order-one standardized residual from a tiny group, and every token in that prompt inherits a scale chosen by only those few samples. The statistic that is supposed to stabilize scale is itself the noisiest object in the estimator.

Now the deeper problem, the one I want to nail down rather than wave at: is (r_i − mean)/std even an unbiased estimate of the per-sample advantage? My gut says the numerator and denominator are computed from the *same* handful of rewards, so they can't be independent, and if they're correlated then dividing one by the other biases the result. Let me actually check, because if it's true it's the real reason to abandon local normalization, not just the small-sample brittleness.

Set up the cleanest version. Suppose the rewards in a group are r_i = θ + ε_i, with ε_i ~ N(0, σ²) i.i.d. for i = 1..N — a true baseline θ plus zero-mean noise. The thing I'd *want* the advantage to recover is ε_i, how much better than baseline this sample is. GRPO's estimator is A_i = (ε_i − ε̄)/D with ε̄ = (1/N)Σ_j ε_j and D = sqrt((1/N) Σ_j (ε_j − ε̄)²). I'll compute E[A_i | ε_i] and see if it equals ε_i.

Numerator first. ε_i − ε̄ = ε_i − (1/N)(ε_i + Σ_{j≠i} ε_j) = (1 − 1/N) ε_i − (1/N) Σ_{j≠i} ε_j. Condition on ε_i; the other ε_j are still zero-mean and independent of ε_i, so E[ε_i − ε̄ | ε_i] = (1 − 1/N) ε_i. So if the denominator were a constant, I'd have a clean (1 − 1/N) shrinkage — a known, harmless scaling. The whole question is whether D behaves like a constant given ε_i, or whether it secretly carries ε_i.

So compute E[D² | ε_i]. Write D² = (1/N) Σ_j ε_j² − ε̄². Conditioning on ε_i leaves the j≠i terms i.i.d. N(0,σ²), so E[Σ_j ε_j² | ε_i] = ε_i² + (N−1)σ² (the ε_i² is fixed, each other ε_j² contributes σ²). For the ε̄² term, ε̄ = (1/N)(ε_i + Σ_{j≠i} ε_j), so
E[ε̄² | ε_i] = (1/N²) E[(ε_i + Σ_{j≠i} ε_j)² | ε_i] = (1/N²)( ε_i² + 2ε_i·0 + E[(Σ_{j≠i} ε_j)²] ) = (ε_i² + (N−1)σ²)/N²,
using E[Σ_{j≠i} ε_j] = 0 and Var(Σ_{j≠i} ε_j) = (N−1)σ². Subtract:
E[D² | ε_i] = (1/N)(ε_i² + (N−1)σ²) − (ε_i² + (N−1)σ²)/N² = ((N−1)²/N²) σ² + ((N−1)/N²) ε_i².
Call it α + β ε_i² with α = ((N−1)²/N²)σ² and β = (N−1)/N² > 0. The conditional second moment of the denominator *grows with* ε_i². If that algebra is right, the denominator is not independent of the sample — a sample with a large |ε_i| systematically inflates the group spread that it's then divided by.

Let me not trust the hand algebra blind — it's the load-bearing step. With σ = 1 I'll Monte-Carlo E[D²|ε_i] (fix ε_1, draw the other N−1, average D² over four million draws) against α + βε_1². At N = 4 the formula gives α = 9/16 = 0.5625, β = 3/16 = 0.1875; for ε_1 ∈ {0, 1, 2} that predicts 0.5625, 0.7500, 1.3125, and the simulation returns 0.5624, 0.7499, 1.3123. At N = 8 it predicts 0.7656, 0.8750, 1.2031 and returns 0.7658, 0.8754, 1.2034. Four-decimal agreement, so the formula holds and the ε_i² coupling is real, not an artifact of my algebra.

I need E[1/D | ε_i], not E[D² | ε_i], so let me push through the nonlinearity, but carefully. Let g(ε_i) = E[1/D | ε_i] = E[f(D²) | ε_i] with f(x) = x^{−1/2}, and Taylor-expand f around μ = E[D² | ε_i]:
f(x) = μ^{−1/2} − (1/2) μ^{−3/2} (x − μ) + (3/8) μ^{−5/2} (x − μ)² + O((x−μ)³).
Take the conditional expectation: the linear term dies because E[x − μ | ε_i] = 0 by construction, and the second-order approximation is g(ε_i) ≈ μ^{−1/2} + (3/8) Var(D² | ε_i) μ^{−5/2}. Even the leading scale μ^{−1/2} = (α + β ε_i²)^{−1/2} already changes with ε_i², so the denominator is not behaving like a constant independent of this sample. But I should not make the tempting symmetry shortcut here. If I decompose A_i = (1 − 1/N)(ε_i/D) − (1/N)(Σ_{j≠i} ε_j)(1/D), the second term is not killed just because Σ_{j≠i} ε_j is symmetric: D² contains (ε_i + Σ_{j≠i} ε_j)², so with ε_i fixed it has a linear cross term in that sum. The odd-times-even argument would be false.

There is a cleaner way to prove the bias without that bad shortcut. For any realized group, set z_j = ε_j − ε̄, so Σ_j z_j = 0, D² = (1/N)Σ_j z_j², and A_i = z_i/D. Given z_i, the other centered residuals sum to −z_i, so their squared sum is at least z_i²/(N−1), with equality when they are all equal. Therefore Σ_j z_j² ≥ z_i² + z_i²/(N−1) = N z_i²/(N−1), hence D² ≥ z_i²/(N−1), and |A_i| ≤ sqrt(N−1). The normalized local-std advantage is bounded for every finite N, while ε_i under the Gaussian model is unbounded. A bounded conditional expectation cannot equal the unbounded function ε_i for all ε_i. So the finite-group local-std estimator is biased.

Let me see this bias directly rather than only inferring it, because the boundedness argument tells me the estimator *can't* track ε_i but not how badly it fails at moderate ε_i. So I'll Monte-Carlo E[A_1 | ε_1] at N = 4 and compare it to what an unbiased-up-to-shrinkage estimator would give, (1 − 1/N) ε_1. For ε_1 = 0.5, 1, 2, 3 the harmless-shrinkage prediction is 0.375, 0.75, 1.50, 2.25, but the simulation returns 0.54, 0.95, 1.37, 1.53. At small ε_1 it actually overshoots the (1−1/N) line, and at large ε_1 it falls far below it and saturates: ε_1 = 3 gives 1.53, nowhere near 2.25, pinned under the bound sqrt(3) ≈ 1.732 (the empirical max |A_1| over the draws is 1.7321, hitting the bound exactly). So the conditional expectation is a bent, saturating function of ε_1, not the line ε_1 nor even a clean multiple of it. Large advantages get compressed by construction, precisely because the sample that creates a large centered numerator also helps create the denominator that divides it. The Taylor calculation, the boundedness argument, and this simulation all point to the same failure.

So the bias and the brittle scaling trace to the same root: the standard deviation is computed from the same tiny sample as the numerator. The fix can't be "estimate the std more carefully within the group" — the group is small by construction and I can't enlarge it without paying k× the rollout cost per prompt. Let me look at where the bias goes as the sample pool grows. The conditional denominator scale is controlled by μ(ε_i) = α + β ε_i², and as N → ∞, β = (N−1)/N² → 0 while α = ((N−1)²/N²)σ² → σ², so the dependence on ε_i should disappear and the denominator should converge to the true σ. Let me put numbers on that decay so I know whether "large" means hundreds or millions: β runs 0.1875, 0.0586, 0.0154, 0.0039, 0.00098 at N = 4, 16, 64, 256, 1024, and the numerator factor (1 − 1/N) runs 0.75, 0.9375, 0.984, 0.996, 0.999. So by N on the order of 1000 the sample-dependent term β is down by ~190× from N = 4 and the shrinkage factor is within 0.1% of 1 — the normalization statistics have essentially stopped depending on the individual sample. The disease is *small N*, and the cure is more samples in whatever pool I standardize over.

But I can't grow the per-prompt group. What other pool do I have? The whole batch. In one optimization step I have a global batch of hundreds or a thousand-plus responses across many prompts. If I standardize over *that* pool instead of over each four-sample group, then the mean and std I divide by are computed from N_global on the order of 1024, not N_group = 4. By the decay I just tabulated, those batch statistics sit at the N ≈ 1000 end where β ≈ 0.001 and the bias from numerator/denominator coupling is ~190× smaller than in the four-sample group. The std is also no longer at the mercy of one uninformative prompt — it's averaged over the whole batch, so a single tied group cannot set the scale for itself. Both failures weaken together, for the same reason: a large pool makes the normalization statistics behave much more like constants.

And standardizing advantages over a pool isn't exotic — subtracting the mean and dividing by the std of the advantages over a minibatch (whitening) before they hit the policy loss is a standard PPO implementation detail; the large-scale study by Andrychowicz et al. (2020) lists per-minibatch advantage normalization among its design knobs as common practice. What I'm doing is taking that established whitening and choosing the pool to be the *global* batch rather than the prompt group, precisely because the size of the pool is what controls the bias and the stability.

So a first cut: drop the per-group std entirely, and instead whiten the advantages across the full batch. But what do I feed into the whitening — the raw rewards, or something centered first? Here I don't want to throw away the one good thing the group gives me. The per-prompt structure carries real information: prompts differ in difficulty, and "this response beat the typical response *to this prompt*" is a more meaningful signal than "this response's raw reward is high," which mostly tells me the prompt was easy. That's exactly the RLOO/baseline intuition, and it has a second benefit I care about in this RLVR setting: subtracting the group mean *reshapes* the reward. Let me check the reshaping claim with both verifier conventions, since I want this to be robust to which one I happen to use. If the verifier emits 0/1 and a group's outcomes are (1,1,0,0), the mean is 0.5 and centering gives (+0.5, +0.5, −0.5, −0.5). If it emits −1/1 and the same group is (1,1,−1,−1), the mean is 0 and centering gives (+1,+1,−1,−1). Both land symmetric about zero — the location is normalized either way, so I don't have to redesign rewards for 0/1 vs −1/1, the centering handles the offset automatically. The group mean still includes the sample itself, so it is not the leave-one-out unbiased baseline; it introduces the finite (1 − 1/N) shrinkage I already accounted for. But it is a stable centering step. It's the *denominator*, the local std, that was biased and brittle. So keep the group mean as local reward reshaping; throw away the local std.

That gives me a two-step estimator. Step one, reshape with a local baseline: A'_i = r_i − mean_{group(i)}(r). Step two, stabilize with global statistics: standardize A' over the whole batch, A_i^norm = (A'_i − mean_batch(A')) / (std_batch(A') + ε). The group mean handles per-prompt difficulty and reward-scale reshaping; the batch std handles cross-prompt scale and is computed from a pool large enough that any one sample has little leverage. The dangerous small-sample std is simply gone.

Edge case I have to handle before I trust it: what if a prompt has only one sample in this batch (group size 1)? Then there's no "other responses" to form a mean, and "subtract the group mean" would subtract the sample from itself and give exactly zero advantage — wiping the signal. The honest fallback is to set the group mean to 0 for singletons: keep the raw score, let the *global* whitening center it relative to the rest of the batch. So singletons get no local baseline but still get the batch-level standardization. If I stopped after group-mean centering, I would have removed the brittle local std but left the gradient magnitudes unscaled across prompts; the batch whitening is the part that puts all valid tokens back on one common scale.

Now, tokens. The actor loss is per-token: it wants an advantage at every response position, not one scalar per sequence. The reward is a scalar at EOS, and my A'_i is one scalar per response. The standard move is to broadcast that scalar to every valid token of the response — every token of a correct response shares the same outcome advantage. But here's a choice in *when* I whiten: do I whiten the per-sequence scalars (one number per response) and then broadcast, or do I broadcast first and whiten over the pool of *tokens*? Let me think about what each pool means. If I whiten per-sequence, every response counts once regardless of length. If I broadcast to tokens first and then whiten over all valid tokens in the batch, a 600-token response contributes 600 entries to the mean and std while a 50-token response contributes 50 — the statistics become *length-weighted*. For a token-level policy loss that's the consistent choice: I'm standardizing the actual per-token quantities that enter the gradient, over the actual set of tokens that enter the gradient. Each token's advantage is put on a common scale with every other token the loss will touch. So: broadcast the centered per-sequence advantage to all valid tokens, mask out the padding, then whiten over all valid response tokens in the batch, then mask again to be safe. The return tensor this critic-free API hands downstream can be the same tensor as the advantages; there is no separate bootstrapped value target to compute.

Let me make the whitening concrete because the small constants matter. masked_mean is the sum over masked entries divided by the count (with a tiny floor so an empty mask can't divide by zero). masked_var subtracts that mean, squares, masked-means again, and applies Bessel's correction n/(n−1), matching the usual unbiased sample-variance estimate before the inverse-square-root scale. masked_whiten is then (v − mean) · rsqrt(var + 1e−8), with the 1e−8 as a numerical floor on the variance. Here that floor sits under a batch statistic computed from all valid response tokens; it is a guard, not the source of the scale.

Now the KL regularizer, because for the reasoning-task variant I add it as a separate loss term rather than baking it into the reward, and I have to pick the estimator carefully. I'm sampling responses from π_θ and want to keep it near the frozen reference π_ref. Let ℓ(y) = log π_θ(y) − log π_ref(y). The true reverse KL is J_RKL = E_{y~π_θ}[ℓ(y)]. If I differentiate the expectation as a distributional objective, I get E[(ℓ+1)∇log π_θ], and the +1 term vanishes because E[∇log π_θ] = 0, so the reverse-KL gradient is E[ℓ ∇log π_θ]. But the code path for a separate KL loss has a fixed rollout batch and backprops through the current log-probabilities of those sampled tokens. So the per-sample loss I want is the one whose fixed-sample derivative is ℓ ∇log π_θ. That distinction matters; scalar KL estimation and loss-gradient estimation are not the same question.

Three sample forms are on the table — k1 = ℓ, k2 = (1/2)ℓ², k3 = exp(−ℓ) − 1 + ℓ — and I keep wanting to argue about their gradients by hand, where I am most likely to drop a sign or a frozen-detach. So let me just build a one-token two-action softmax policy with a frozen reference, autograd each surrogate, and read off the actual coefficient multiplying ∇log π_θ(y). With logits (0.3, −0.7), reference logits (0.1, 0.2), sampled token y the first action, autograd gives: ∇k1 = (0.2689, −0.2689), exactly ∇log π_θ(y), coefficient 1. ∇k2 = (0.1160, −0.1160), which is ℓ · ∇log π_θ with ℓ = 0.4311 (and 0.4311 × 0.2689 = 0.1160, matches). ∇k3 = (0.0942, −0.0942) = (1 − exp(−ℓ)) · ∇log π_θ with 1 − exp(−ℓ) = 0.3502 (0.3502 × 0.2689 = 0.0942, matches). So the three gradients are coefficient·∇log π_θ with coefficients 1, ℓ, and 1 − exp(−ℓ) respectively — confirmed numerically, not just claimed.

Now read those coefficients against what I need. The reverse-KL loss gradient is ℓ ∇log π_θ, so k2 is exactly right — its coefficient *is* ℓ. k1's coefficient is the constant 1: as a fixed-sample loss its derivative is just ∇log π_θ, missing the log-ratio multiplier; averaged exactly on-policy that score has expectation zero, so on a finite batch it's mostly an unweighted push on sampled tokens rather than a force proportional to how far they drifted. k1 is fine as a reward-shaping cost where the policy-gradient machinery supplies the score multiplier, but it's the wrong standalone loss gradient. k3's coefficient is 1 − exp(−ℓ) = 1 − π_ref/π_θ; in expectation the constant-1 part drops like any score term and leaves −E_{y~π_θ}[(π_ref/π_θ)∇log π_θ] = −E_{y~π_ref}[∇log π_θ], the *forward*-KL gradient — wrong direction for a reverse-KL loss.

And the k3 coefficient is not just direction-wrong, it's dangerous, which I can see by sweeping the ratio π_θ/π_ref. At ratio 1 both k2 and k3 coefficients are 0. As π_θ(y) drops relative to π_ref, k2's coefficient ℓ = log(ratio) goes −2.30, −4.61, −6.91, −11.51 at ratio 0.1, 0.01, 1e−3, 1e−5 — it grows like log, mildly. k3's coefficient 1 − 1/ratio goes −9, −99, −999, −99999 over the same ratios — it grows like 1/ratio, i.e. it explodes exactly when the policy has driven a token's probability far below the reference, which is the regime a KL penalty is supposed to handle gracefully. Numerically it's computed through exp(log π_ref − log π_θ), so it overflows there too. So k3 may be a low-variance scalar KL estimator near the reference, but as a standalone loss gradient it is both direction-mismatched and prone to blow up. k2 = (1/2)(log(π_θ/π_ref))² it is: correct reverse-KL loss gradient, no importance ratio, no exp, coefficient that grows only logarithmically. The full objective for the reasoning variant is then L = L_PPO(A^norm) − λ J_{k2 as loss}(θ): the clipped surrogate using my whitened advantages, minus a k2 KL-loss term.

At this point the construction is just PPO with three changes: the critic V is deleted; GAE is taken at λ = 1, γ = 1 so the advantage is just the full return (no bootstrapping, consistent with reward-only-at-EOS, as I traced above); and the baseline that V used to provide is replaced by a two-step normalization — subtract the per-prompt group mean, then whiten across the global batch. The clipped surrogate, the old-policy ratio, the reference KL — all unchanged. I've only swapped the source of the advantage.

Let me write the estimator as the code that drops into the registry, filling the single slot the harness left me. It takes per-token rewards, the response mask, and the per-sample group ids; it returns per-token advantages and returns. Everything runs under no_grad — this is advantage computation, not part of the differentiable loss.

```python
from collections import defaultdict
from typing import Optional

import torch
import verl.utils.torch_functional as verl_F
from verl.trainer.config import AlgoConfig


@register_adv_est(AdvantageEstimator.REINFORCE_PLUS_PLUS_BASELINE)
def compute_reinforce_plus_plus_baseline_outcome_advantage(
    token_level_rewards: torch.Tensor,   # (bs, response_length), outcome reward already placed on tokens
    response_mask: torch.Tensor,         # (bs, response_length) 1 on valid response tokens
    index: torch.Tensor,                 # (bs,) group id; equal id == responses to the same prompt
    epsilon: float = 1e-6,
    config: Optional[AlgoConfig] = None,
    **kwargs,
) -> tuple[torch.Tensor, torch.Tensor]:
    response_length = token_level_rewards.shape[-1]
    # outcome scalar per response: reward only lives at EOS, so summing the row recovers it (bandit return)
    scores = token_level_rewards.sum(dim=-1)            # (bs,)

    id2score = defaultdict(list)                        # gather scores by prompt group
    id2mean = {}

    with torch.no_grad():
        bsz = scores.shape[0]
        for i in range(bsz):
            id2score[index[i]].append(scores[i])
        for idx in id2score:
            if len(id2score[idx]) == 1:
                # singleton group: no local baseline available -> 0, let global whitening center it
                id2mean[idx] = torch.tensor(0.0)
            elif len(id2score[idx]) > 1:
                # local centering = group mean (the stable part of group normalization;
                # reshapes rewards so 0/1 and -1/1 schemes both center near zero)
                id2mean[idx] = torch.mean(torch.stack(id2score[idx]))
            else:
                raise ValueError(f"no score in prompt index: {idx}")
        for i in range(bsz):
            # step 1: subtract the per-prompt group mean (centering / reshaping)
            scores[i] = scores[i] - id2mean[index[i]]

        # broadcast the centered per-sequence advantage to every valid token, mask padding
        scores = scores.unsqueeze(-1).tile([1, response_length]) * response_mask
        # step 2: whiten over ALL valid response tokens in the batch; one sample has little leverage,
        # and the token-level pool makes the stats length-weighted, matching the per-token policy loss
        scores = verl_F.masked_whiten(scores, response_mask) * response_mask

    # no critic target is computed, so this API returns the same tensor for advantages and returns
    return scores, scores
```

So the causal chain, start to finish. The critic was an expensive, unreliable baseline in the sparse-reward LLM regime, so I deleted it and modeled the generation as a single bandit action with γ = 1 — which I confirmed is exactly GAE's own λ = γ = 1, V ≡ 0 limit, every token taking the full terminal return — leaving a hole where the baseline used to be. RLOO's leave-one-out and GRPO's group-standardization both fill it from the same prompt's few samples, but the small group is the problem: GRPO's local std is a noisy prompt-sized scale, and the local-std estimator is biased for every finite N because the centered numerator and denominator are computed from the same sample. I checked this three ways that agree: E[D²|ε_i] = α + βε_i² matched the simulation to four decimals; the centered-residual bound forces |A_i| ≤ sqrt(N−1); and the simulated E[A_1|ε_1] at N = 4 saturates under that bound (1.53 at ε_1 = 3 vs the harmless-shrinkage 2.25). The same numerics showed β decaying ~190× from N = 4 to N ≈ 1000, which pointed straight at the fix: keep the stable, reward-reshaping group mean (verified to center both 0/1 and −1/1 rewards about zero), but compute the *scale* normalization over the global batch, whose large pool makes the statistics behave like constants; this is standard PPO advantage whitening, just pooled globally. Broadcasting the centered scalar to tokens and whitening over all valid tokens makes the standardization length-weighted and consistent with the per-token loss, with the variance floor now just a numerical guard under a batch variance. For the separate KL loss, autograd confirmed the three surrogates have gradients 1·, ℓ·, and (1−exp(−ℓ))·∇log π_θ; only k2's coefficient ℓ is the reverse-KL gradient, and a ratio sweep showed k3's coefficient blowing up like 1/ratio (−99999 at ratio 1e−5) while k2's grows only like log. Net: PPO with no critic, γ = λ = 1, and a group-mean-then-global-whiten baseline.
