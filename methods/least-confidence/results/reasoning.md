Let me start from what actually costs me. I have a flood of unlabeled text and a classifier I want to train, but every label has to come from a human annotator, and that is the expensive part — the text is free, the labels are not. So the real question is not "how do I fit a classifier" but "given that the oracle will only label a fixed number of examples, which examples do I ask about?" Every label I spend is a label I can't spend elsewhere, and right now I'm spending most of them badly.

Here is exactly how badly. Suppose the category I care about is rare — say one example in a thousand is positive. If I draw the examples to label uniformly at random from the pool, then five hundred labels buy me, in expectation, about five hundred negatives and essentially zero positives. I cannot train a classifier to recognize a class from a sample that contains none of it. And it's not just the rare-class case: as my training set grows, the marginal value of one more random label drops, because most examples I draw at random are ones the current classifier already gets right, so labeling them confirms what I already know and teaches nothing. The labels are being wasted on the easy, settled part of the input space while the examples that would actually move the classifier — the ambiguous ones near the boundary, the rare positives — almost never get drawn. So I want to *choose* which examples to label, not draw them at random. The whole game is picking the informative ones.

What does "informative" even mean here, and who has a handle on it? The cleanest theoretical statement I know comes from thinking of learning as search through a version space — the set of all hypotheses in my model class still consistent with the labels I've seen. Early on that set is huge; each label I add must be consistent with the true concept, so it carves the set down. The question "which example is worth labeling" becomes "which example, once labeled, carves the most off the version space." And there's a beautiful object that pins this down: the region of uncertainty. Consider the set of input points on which two still-consistent hypotheses disagree,

  R = { x : there exist c1, c2 in my class, both consistent with every label so far, with c1(x) != c2(x) }.

Think about what a label does depending on where the example sits. If x is *outside* R, then every consistent hypothesis already agrees on x — labeling it can't contradict any of them, so the version space doesn't shrink at all, the label is wasted. If x is *inside* R, then some consistent hypotheses say one thing and others say the opposite; whatever the true label turns out to be, it kills off all the consistent hypotheses that guessed wrong. So an example only ever informs me if it lies in R. And the mass of R, alpha = Pr[x in R], is doing double duty: as long as the true concept is still in the version space, it bounds the error of whatever consistent hypothesis I pick (any disagreement between the target and my hypothesis lives inside R, so my hypothesis can only be wrong on R, hence error <= alpha), and it's the probability that a random draw lands somewhere useful. As I label more, R shrinks, alpha shrinks, and a random draw lands in the useful region with probability alpha, which is heading toward zero. That's the decaying efficiency I was complaining about, now named: drawing over the whole domain, most points fall outside R and do nothing. So the fix, in principle, is obvious — only ask about points inside R. Then every query shrinks R and efficiency doesn't decay.

So can I just do that? Let me try the most direct thing the theory suggests and see where it breaks. The cleanest version is: let the learner ask the oracle about *any* point it wants — a membership query. The learner picks the point that most resolves its confusion, synthesizes it, and hands it over. In concept-learning theory this is enormously powerful; you can identify a target from far fewer queries than passive learning because you probe exactly where you're confused. Let me imagine doing it on my text problem. The learner constructs the input it's most uncertain about and asks the human, "is this in the category?" But the input it constructs is a point in feature space, and there's no reason a synthesized point in word-feature space corresponds to any real document. People who tried this on images got back artificial hybrid characters with no recognizable symbol in them — gibberish the annotator can't classify. For text it's worse: a synthesized "document" assembled to maximize confusion is a meaningless bag of words, not a sentence anyone can read and label. So synthesis is dead on arrival here. The oracle is a human and the inputs have to be natural language; I can't fabricate the query. Wall.

Fine — don't synthesize, filter. I have a giant pool of *real* unlabeled examples sitting right there. Instead of constructing the ideal query point, draw real examples and only ask the oracle about the ones that fall inside the region of uncertainty. Every query is then a genuine document the human can read, and the in-R guarantee still holds: each one shrinks R. This is exactly the selective-sampling move and it feels right — it keeps the efficiency argument while killing the gibberish problem. So the rule would be: for each unlabeled example, test whether it's in R; if it is, it's a candidate to label.

But now I have to actually compute "is x in R," and R is defined through the *entire* version space. To test membership I'd need to find two consistent hypotheses disagreeing on x, which means maintaining the whole set of consistent hypotheses and re-deriving it after every single label. For the trivial concept classes in the theory you can do this; for the classifiers I actually train on text — logistic-regression-style models over thousands of word features — enumerating or even representing the version space is hopeless, and I'd have to redo it every round. Worse, "in R or not" is a hard binary test, and what I have is a budget: the oracle will label n examples this round, so I don't want a yes/no membership test, I want to *rank* the pool and take the top n. The region of uncertainty is the right idea and an unusable object. Wall.

So how do people approximate R without the version space? The committee idea: instead of representing all consistent hypotheses, draw two of them at random from the version space, run both on an incoming example, and treat their *disagreement* as the signal "this point is inside R" — because if two consistent hypotheses disagree on x, then x is by definition in R. That's clever; it samples R instead of computing it. And there's real theory behind it: after examining m random examples, a disagreement-querying learner can need only logarithmically many labels while the generalization error still falls like O(1/m) in the examined stream, so measured against the number of labels the error falls exponentially fast. That's the kind of speedup I'm after. So let me try to adopt it. To run it I need to (a) draw hypotheses uniformly at random from the version space, (b) assume the data are clean enough that a consistent perfect classifier actually exists, and (c) keep a committee of models rather than the single classifier I'm already training. On my problem all three crack. My text labels are noisy and my category definitions are messy, so there is no perfect deterministic classifier consistent with all the data — the version space can be empty, which makes "draw from it" meaningless. Sampling the version space uniformly isn't something I can do for these models anyway. And maintaining a committee is extra machinery on top of training. The committee gives a sound proxy for R under assumptions I don't have. Wall again — but this one is informative, because it tells me what I actually need: a proxy for "x is in the region of uncertainty" that (i) uses only the single classifier I'm already training, (ii) doesn't assume noise-free realizability, and (iii) produces a *ranking*, a scalar I can sort, so I can spend a budget.

Let me look hard at what a committee's disagreement is really measuring, because the answer is hiding there. Two hypotheses disagree on x precisely when x is the kind of point where the evidence so far doesn't determine the label — where, if I had to commit, I'd be guessing. The committee detects that by sampling two classifiers and seeing them split. But my single trained classifier already carries that information directly, if it's a probabilistic classifier: it outputs a posterior, its own degree of belief about the label of x. Where two consistent hypotheses would split, a well-trained probabilistic classifier sits near the fence — its posterior is close to ambivalent. Where all consistent hypotheses agree, the posterior is confident. So the classifier's own *uncertainty about its prediction* is a single-model, scalar surrogate for "x is in the region of uncertainty." I don't need a committee to tell me the model is on the fence; I can read it off the posterior. This dodges all three cracks at once: one classifier, no realizability assumption (a posterior is defined whether or not a perfect classifier exists), and it's a number I can sort.

Make it concrete for the simplest case, two classes C and its complement. The classifier estimates the posterior P(C | x), the probability x belongs to C. The decision boundary, under equal misclassification costs, is at P(C | x) = 0.5 — above it I'd call x positive, below it negative. So where is the classifier least certain of its decision? Exactly at 0.5: there the two outcomes are equiprobable, the model is maximally on the fence, and a label there resolves the most. An example with P(C | x) = 0.99 the model is almost sure is positive; one with 0.5 it cannot call at all. So the rule is to query the examples whose posterior is *nearest 0.5*. That's the scalar I wanted: score each unlabeled example by how close its posterior sits to the boundary, sort, take the n closest to 0.5. It's a soft, rankable, single-model stand-in for "inside R," and it falls straight out of "the boundary is where the model is least certain, and least-certain points are the ones in the region of uncertainty."

Let me sanity-check this against the rare-class disaster that started everything. With random sampling I drowned in negatives. With nearest-0.5 selection, once the classifier has even a weak notion of the positive class, the examples it finds genuinely borderline — posterior near 0.5 — are concentrated around the boundary between the rare positives and the negatives, which is exactly where the few positives and the confusable negatives live. So I'm spending labels on the contested region rather than re-confirming the bulk of obvious negatives. That's the behavior I wanted: chase the boundary, not the mass.

Now I have to generalize past two classes, because in general there are several categories and the posterior is a whole distribution over classes, not a single number I can compare to 0.5. What's the analogue of "nearest the boundary"? Picture the classifier looking at x: it computes posteriors over all classes and, if forced to decide, picks the most probable one, yhat = argmax_y P(y | x), and acts with confidence P(yhat | x) — the probability mass on its chosen class. When is it most certain of that decision? When P(yhat | x) is near 1 — the top class dominates, the rest are near zero. When is it least certain? When P(yhat | x) is small — even the *best* class only barely wins, the mass is spread out, the model is close to guessing. So the natural multi-class uncertainty score is one minus the confidence in the top class,

  uncertainty(x) = 1 - P(yhat | x) = 1 - max_y P(y | x),

and I query the examples that maximize it — equivalently, the examples with the *smallest* maximum posterior. Read it back into the binary case to make sure it's the same rule: with two classes the larger posterior is max(p, 1-p) = 0.5 + |p - 0.5|, so minimizing the top-class probability is exactly minimizing the distance from 0.5. So "smallest top-class probability" reduces to "posterior nearest 0.5" when there are two classes. Same rule, generalized. Good — the multi-class form is genuinely the same idea, not a different heuristic bolted on.

And 1 - max_y P(y | x) has a clean reading I like even better than "distance to the boundary": it's the model's own estimate of the probability that it will *mislabel* x. If the model commits to yhat, it's wrong whenever the true class is something other than yhat, and under its own posterior the probability of that is exactly 1 - P(yhat | x). So I'm querying the examples the model believes it is most likely to get wrong — its expected 0/1 loss, point by point. That's a very satisfying thing to spend labels on: the examples it expects to misclassify are precisely the ones whose true labels would correct it. It costs nothing beyond the forward pass I already do to classify, since the posterior is right there.

Before I lock it in, let me poke at what this score throws away, because I want to know its weakness and whether a richer score is worth the cost. 1 - max only looks at the single most probable class; it ignores how the remaining probability is distributed. Two examples can have the same top-class probability of, say, 0.5 — but in one the other 0.5 sits entirely on a single runner-up class, and in the other it's smeared over eight classes. Intuitively the first is the more useful query if my goal is to settle a specific classification decision: the model is torn between two specific classes, and the label would tell it which. So I could instead score by the margin between the top two classes, P(yhat1 | x) - P(yhat2 | x), and query the smallest margins — that uses the runner-up the top-only score discards. Push it further and I could use the whole distribution's entropy, -sum_y P(y | x) log P(y | x), which is maximal when the posterior is flat and uses every class. These are real refinements; in the binary case all three give the same query ordering — top-only, margin, and entropy all choose the posterior nearest 0.5, since with two classes the top probability, the gap to the runner-up, and the entropy are all monotone functions of |p - 0.5|. They only differ once there are three or more classes, where they weight the non-top mass differently. Entropy gives lower priority to edge cases where one class is merely highly unlikely; the top-only and margin scores still value them if the remaining classes are confusable. Entropy is the natural uncertainty measure if the loss I care about is log-loss, while the top-only and margin scores aim more directly at decision uncertainty under 0/1 classification error, since they prefer examples that would help the model decide between the classes it is about to confuse. For the cleanest rule I want the minimal, most direct form — the one that scores nothing but the quantity the classifier already exposes, its top-class probability, and that equals the model's own misclassification probability. That's 1 - max_y P(y | x). Margin and entropy are the refinements I'd consider if the top-only score left value on the table; the simplest expression of "label what the model is least sure of" is least confidence itself.

Now wrap it in the loop, because the selection is only half of it. The classifier I use to score the pool is the one trained on the labels I have *so far*, and after I label this round's batch I retrain on the enlarged set, and *that* classifier scores the next round. So it's a sequential, iterative process: an initial classifier seeds it, each round it scores the unlabeled pool by uncertainty, sends the n least-confident examples to the oracle, folds the new labels in, and retrains. The selection is exactly as good as the current classifier — a deficient classifier this round picks examples that, once labeled, tend to compensate for its deficiency next round, which is the exploratory feedback that makes it work. One caveat I can already see: the initial classifier matters. If I start from nothing, there can be a long stretch where the model has no idea where the positives are and the uncertainty score is meaningless, so it flails until it stumbles onto a positive; seeding it with a handful of known positives plus a few random examples gives the uncertainty score something real to point at from the first round.

So the acquisition rule is just: take the currently-unlabeled examples, get the trained classifier's posterior over classes for each, score by the top-class probability, and return the n with the smallest top-class probability — the n the model is least confident about, equivalently the n that maximize 1 - max_y P(y | x).

```python
import numpy as np
from .strategy import Strategy


class LeastConfidence(Strategy):

    def __init__(self, X, Y, idxs_lb, net, handler, args):
        super(LeastConfidence, self).__init__(X, Y, idxs_lb, net, handler, args)

    def query(self, n):
        # the pool examples not yet labeled
        idxs_unlabeled = np.arange(self.n_pool)[~self.idxs_lb]
        # current classifier's posterior P(class | x) for each unlabeled example: [m, n_classes]
        probs = self.predict_prob(self.X[idxs_unlabeled],
                                  np.asarray(self.Y)[idxs_unlabeled])
        # confidence in the predicted (top) class: max_c P(c | x)
        U = probs.max(1)[0]
        # least confident = smallest max-prob; sort ascending and take the first n
        # (argmin of max_c P  <=>  argmax of 1 - max_c P; in the binary case <=> P nearest 0.5)
        return idxs_unlabeled[U.sort()[1][:n]]
```

Random labeling wastes the budget on examples the classifier already handles and starves the rare, boundary cases, and the version-space picture explains why — only examples in the region of uncertainty R inform the learner, and random draws hit R with a probability that decays to zero. Querying inside R is the fix, but synthesizing the ideal query point produces unlabelable gibberish on text, and computing R exactly needs the whole version space, which is intractable and gives a binary test rather than a ranking. A committee approximates R by sampling two hypotheses and querying on disagreement, but that needs noise-free realizability, uniform version-space sampling, and a committee — none of which I have. The disagreement a committee detects is just "the model is on the fence," and a single probabilistic classifier already reports that as its posterior: nearest 0.5 in two classes, and in general the smallest top-class probability, 1 - max_y P(y | x), which is exactly the model's own estimated probability of mislabeling x. Querying the n examples that maximize it gives a soft, single-model, rankable surrogate for "inside the region of uncertainty," reduces to nearest-0.5 in the binary case, costs nothing beyond the forward pass, and slots straight into the retrain-each-round loop as the acquisition rule.
