Let me start from what actually costs me. I have a flood of unlabeled text and a classifier I want to train, but every label has to come from a human annotator, and that is the expensive part — the text is free, the labels are not. So the real question is not "how do I fit a classifier" but "given that the oracle will only label a fixed number of examples, which examples do I ask about?" Every label I spend is a label I can't spend elsewhere, and right now I'm spending most of them badly.

Here is exactly how badly, and I want to put a number on it rather than wave at it. Suppose the category I care about is rare — one example in a thousand is positive, p = 0.001. If I draw the examples to label uniformly at random from the pool, the count of positives among n labels is Binomial(n, 0.001), with mean 0.001·n. For n = 500 that mean is 0.5, and the chance of getting *zero* positives is (1 - 0.001)^500 = 0.999^500 ≈ 0.606. So more than half the time, five hundred labels buy me not a single positive, and the expected haul is half of one. I cannot train a classifier to recognize a class from a sample that, more likely than not, contains none of it. And it's not just the rare-class case: as my training set grows, the marginal value of one more random label drops, because most examples I draw at random are ones the current classifier already gets right, so labeling them confirms what I already know and teaches nothing. The labels are being wasted on the easy, settled part of the input space while the examples that would actually move the classifier — the ambiguous ones near the boundary, the rare positives — almost never get drawn. So I want to *choose* which examples to label, not draw them at random. The whole game is picking the informative ones.

What does "informative" even mean here, and who has a handle on it? The cleanest theoretical statement I know comes from thinking of learning as search through a version space — the set of all hypotheses in my model class still consistent with the labels I've seen. Early on that set is huge; each label I add must be consistent with the true concept, so it carves the set down. The question "which example is worth labeling" becomes "which example, once labeled, carves the most off the version space." And there's a sharp object that pins this down: the region of uncertainty. Consider the set of input points on which two still-consistent hypotheses disagree,

  R = { x : there exist c1, c2 in my class, both consistent with every label so far, with c1(x) != c2(x) }.

Think about what a label does depending on where the example sits. If x is *outside* R, then every consistent hypothesis already agrees on x — labeling it can't contradict any of them, so the version space doesn't shrink at all, the label is wasted. If x is *inside* R, then some consistent hypotheses say one thing and others say the opposite; whatever the true label turns out to be, it kills off all the consistent hypotheses that guessed wrong. So an example only ever informs me if it lies in R. And the mass of R, alpha = Pr[x in R], is doing double duty: as long as the true concept is still in the version space, it bounds the error of whatever consistent hypothesis I pick (any disagreement between the target and my hypothesis lives inside R, so my hypothesis can only be wrong on R, hence error <= alpha), and it's the probability that a random draw lands somewhere useful. As I label more, R shrinks, alpha shrinks, and a random draw lands in the useful region with probability alpha, which is heading toward zero. That's the decaying efficiency I was complaining about, now named: drawing over the whole domain, most points fall outside R and do nothing. So the fix, in principle, looks obvious — only ask about points inside R. Then every query shrinks R and efficiency doesn't decay. Let me see whether that "in principle" survives contact with my actual problem, because there are at least two ways it could fail to be implementable.

The cleanest version of "ask about points inside R" lets the learner ask the oracle about *any* point it wants — a membership query. The learner picks the point that most resolves its confusion, synthesizes it, and hands it over. In concept-learning theory this is enormously powerful; you can identify a target from far fewer queries than passive learning because you probe exactly where you're confused. Let me imagine doing it on my text problem. The learner constructs the input it's most uncertain about and asks the human, "is this in the category?" But the input it constructs is a point in feature space, and there's no reason a synthesized point in word-feature space corresponds to any real document. People who tried this on images got back artificial hybrid characters with no recognizable symbol in them — gibberish the annotator can't classify. For text it's worse: a synthesized "document" assembled to maximize confusion is a meaningless bag of words, not a sentence anyone can read and label. So synthesis is dead on arrival here. The oracle is a human and the inputs have to be natural language; I can't fabricate the query. Wall.

Fine — don't synthesize, filter. I have a giant pool of *real* unlabeled examples sitting right there. Instead of constructing the ideal query point, draw real examples and only ask the oracle about the ones that fall inside the region of uncertainty. Every query is then a genuine document the human can read, and the in-R guarantee still holds: each one shrinks R. This keeps the efficiency argument while killing the gibberish problem. So the rule would be: for each unlabeled example, test whether it's in R; if it is, it's a candidate to label.

But now I have to actually compute "is x in R," and R is defined through the *entire* version space. To test membership I'd need to find two consistent hypotheses disagreeing on x, which means maintaining the whole set of consistent hypotheses and re-deriving it after every single label. For the trivial concept classes in the theory you can do this; for the classifiers I actually train on text — logistic-regression-style models over thousands of word features — enumerating or even representing the version space is hopeless, and I'd have to redo it every round. And there's a second problem I keep tripping over: "in R or not" is a hard binary test, but what I have is a budget. The oracle will label n examples this round, so I don't want a yes/no membership test — I want to *rank* the pool and take the top n. The region of uncertainty is the right idea and an unusable object. Wall.

So how do people approximate R without the version space? The committee idea: instead of representing all consistent hypotheses, draw two of them at random from the version space, run both on an incoming example, and treat their *disagreement* as the signal "this point is inside R" — because if two consistent hypotheses disagree on x, then x is by definition in R. That's clever; it samples R instead of computing it. And there's real theory behind it: after examining m random examples, a disagreement-querying learner can need only logarithmically many labels while the generalization error still falls like O(1/m) in the examined stream, so measured against the number of labels the error falls exponentially fast. That's the kind of speedup I'm after. So let me try to actually run it and see if the assumptions hold on my data. To run it I need to (a) draw hypotheses uniformly at random from the version space, (b) assume the data are clean enough that a consistent perfect classifier actually exists, and (c) keep a committee of models rather than the single classifier I'm already training. On my problem all three crack. My text labels are noisy and my category definitions are messy, so there is no perfect deterministic classifier consistent with all the data — the version space can be empty, which makes "draw from it" meaningless. Sampling the version space uniformly isn't something I can do for these models anyway. And maintaining a committee is extra machinery on top of training. The committee gives a sound proxy for R under assumptions I don't have. Wall again — but this one is informative, because it tells me what I actually need: a proxy for "x is in the region of uncertainty" that (i) uses only the single classifier I'm already training, (ii) doesn't assume noise-free realizability, and (iii) produces a *ranking*, a scalar I can sort, so I can spend a budget.

Let me look hard at what a committee's disagreement is really measuring, because the answer may be hiding there. Two hypotheses disagree on x precisely when x is the kind of point where the evidence so far doesn't determine the label — where, if I had to commit, I'd be guessing. The committee detects that by sampling two classifiers and seeing them split. But my single trained classifier already carries that information directly, if it's a probabilistic classifier: it outputs a posterior, its own degree of belief about the label of x. Where two consistent hypotheses would split, a well-trained probabilistic classifier should sit near the fence — its posterior close to ambivalent. Where all consistent hypotheses agree, the posterior should be confident. So the classifier's own *uncertainty about its prediction* could be a single-model, scalar surrogate for "x is in the region of uncertainty." That would dodge all three cracks at once: one classifier, no realizability assumption (a posterior is defined whether or not a perfect classifier exists), and a number I can sort. Let me make it concrete and check whether it really behaves the way I'm hoping.

Take the simplest case, two classes C and its complement. The classifier estimates the posterior p = P(C | x), the probability x belongs to C. The decision boundary, under equal misclassification costs, is at p = 0.5 — above it I'd call x positive, below it negative. So where is the classifier least certain of its decision? Exactly at 0.5: there the two outcomes are equiprobable, the model is maximally on the fence, and a label there resolves the most. An example with p = 0.99 the model is almost sure is positive; one with 0.5 it cannot call at all. So the candidate rule is to query the examples whose posterior is *nearest 0.5*. That's a scalar I can sort: score each unlabeled example by how close its posterior sits to the boundary, take the n closest to 0.5.

Before I trust this I want to check it against the rare-class disaster that started everything, because "near the boundary" is only useful if the boundary is where the rare positives actually are. Suppose the classifier has learned even a weak notion of the positive class. The obvious negatives — text with none of the words that signal the category — get a small p, far below 0.5, so the nearest-0.5 rule ignores them. The obvious positives, if any, get a large p, also ignored. The examples it scores near 0.5 are the ones where the positive-signaling words and the negative context are in tension — exactly the confusable region where the handful of true positives sit alongside the negatives that look like them. So the rule concentrates the budget on the contested region instead of re-confirming the bulk of obvious negatives. Concretely: of those 500 random labels, ~499 were obvious negatives sitting near p = 0, which this rule would never have spent a label on. That's the failure mode of random sampling stated as exactly what the new rule avoids. Good — it does the thing I needed, at least in this picture.

Now I have to generalize past two classes, because in general there are several categories and the posterior is a whole distribution over classes, not a single number I can compare to 0.5. What's the analogue of "nearest the boundary"? Picture the classifier looking at x: it computes posteriors over all classes and, if forced to decide, picks the most probable one, yhat = argmax_y P(y | x), and acts with confidence P(yhat | x) — the probability mass on its chosen class. When is it most certain of that decision? When P(yhat | x) is near 1 — the top class dominates, the rest are near zero. When is it least certain? When P(yhat | x) is small — even the *best* class only barely wins, the mass is spread out, the model is close to guessing. So a natural multi-class uncertainty score is one minus the confidence in the top class,

  uncertainty(x) = 1 - P(yhat | x) = 1 - max_y P(y | x),

and I query the examples that maximize it — equivalently, the examples with the *smallest* maximum posterior.

I should check that this multi-class score actually reduces to the binary rule I just convinced myself of, not merely that it sounds like it. In the two-class case the top-class probability is max(p, 1-p). I claimed that minimizing it is the same as minimizing the distance from 0.5, which rests on the identity max(p, 1-p) = 0.5 + |p - 0.5|. Let me verify that holds across the range rather than assume it. Sweeping p from 0 to 1 in steps of 0.1: at p = 0.1, max(0.1, 0.9) = 0.9 and 0.5 + |0.1 - 0.5| = 0.5 + 0.4 = 0.9; at p = 0.5, max = 0.5 and 0.5 + 0 = 0.5; at p = 0.7, max = 0.7 and 0.5 + 0.2 = 0.7; at p = 0.9, max = 0.9 and 0.5 + 0.4 = 0.9. They match at every point I check, and the reason is plain once written: |p - 0.5| is exactly how far the larger of {p, 1-p} sits above 0.5. So minimizing the top-class probability *is* minimizing |p - 0.5|, i.e. choosing the posterior nearest 0.5. The multi-class form is genuinely the same idea specialized, not a different heuristic bolted on.

And 1 - max_y P(y | x) has a second reading that I find more convincing than "distance to the boundary": it's the model's own estimate of the probability that it will *mislabel* x. If the model commits to yhat, it's wrong whenever the true class is something other than yhat, and under its own posterior the probability of that is exactly 1 - P(yhat | x). So I'd be querying the examples the model believes it is most likely to get wrong — its expected 0/1 loss, point by point. That's a sensible thing to spend labels on: the examples it expects to misclassify are precisely the ones whose true labels would correct it. And it costs nothing beyond the forward pass I already do to classify, since the posterior is right there.

Before I lock it in, let me poke at what this score throws away, because I want to know its weakness and whether a richer score is worth the cost. 1 - max only looks at the single most probable class; it ignores how the remaining probability is distributed. So I can construct two examples the score must treat as identical and ask whether they really deserve the same priority. Take a three-class problem and two posteriors with the same top-class probability 0.5: A = (0.50, 0.48, 0.02) and B = (0.50, 0.25, 0.25). The least-confidence score is 1 - 0.50 = 0.50 for both, so it ranks them equal. But they are not equally useful queries. In A the model is torn between two specific classes — it nearly can't tell class 1 from class 2 — and a single label would settle which. In B the leftover mass is split evenly; the model is diffusely unsure. If my goal is to resolve a specific classification decision, A is the better query, and least confidence is blind to the difference. Two refinements see it. The margin score, P(yhat1 | x) - P(yhat2 | x) with smallest margin queried first, gives margin(A) = 0.50 - 0.48 = 0.02 and margin(B) = 0.50 - 0.25 = 0.25, so it ranks A ahead of B — it prefers the two-way-torn example, using exactly the runner-up that the top-only score discarded. The entropy score, -sum_y P(y | x) log P(y | x) with largest queried first, gives entropy(A) ≈ 0.777 and entropy(B) ≈ 1.040, so it ranks B ahead of A — it prefers the flatter, more diffuse posterior, because it weighs every class. So on this one example the three scores genuinely disagree about which is the more informative query, and they disagree in directions that match their definitions: margin chases two-way confusions, entropy chases diffuse uncertainty, least confidence ties them.

That disagreement only appears once there are three or more classes. Let me confirm the three collapse to one rule in the binary case, since that's what makes least confidence a safe minimal choice there. Sweeping p with posterior (p, 1-p): 1 - max is 0.5, 0.4, 0.3, 0.2, 0.1 as p goes 0.5, 0.6, 0.7, 0.8, 0.9; margin is 0.0, 0.2, 0.4, 0.6, 0.8 over the same p; entropy is 0.693, 0.673, 0.611, 0.500, 0.325. All three are extremal at p = 0.5 and monotone in |p - 0.5| — one decreasing, margin increasing, entropy decreasing — so sorting by any of them gives the *same* ordering of binary examples. They can only differ with three or more classes, where they weight the non-top mass differently, which is exactly what the A/B example showed.

So which do I want as the canonical rule? Margin and entropy are real refinements, and if I cared about log-loss specifically, entropy is the natural matching uncertainty measure, while margin and the top-only score aim more directly at decision uncertainty under 0/1 classification error, since they prefer examples that would help the model decide between the classes it is about to confuse. But for the cleanest, minimal rule I want the form that scores nothing but the quantity the classifier already exposes — its top-class probability — and that equals the model's own misclassification probability. That's 1 - max_y P(y | x): least confidence itself. Margin and entropy are the refinements I'd reach for if the top-only score left value on the table; the simplest expression of "label what the model is least sure of" is least confidence.

Now wrap it in the loop, because the selection is only half of it. The classifier I use to score the pool is the one trained on the labels I have *so far*, and after I label this round's batch I retrain on the enlarged set, and *that* classifier scores the next round. So it's a sequential, iterative process: an initial classifier seeds it, each round it scores the unlabeled pool by uncertainty, sends the n least-confident examples to the oracle, folds the new labels in, and retrains. The selection is exactly as good as the current classifier — a deficient classifier this round picks examples that, once labeled, tend to compensate for its deficiency next round, which is the exploratory feedback that makes it work. One caveat I can already see, and it's the same rare-class trap from the start: if I begin from nothing, there can be a long stretch where the model has no idea where the positives are and the uncertainty score is meaningless, so it flails until it stumbles onto a positive. Recall that with p = 0.001 a cold random start has a ~60% chance of finding zero positives in 500 draws, so "stumble onto a positive" can take a very long time. Seeding the loop with a handful of known positives plus a few random examples gives the uncertainty score something real to point at from the first round.

So the acquisition rule is: take the currently-unlabeled examples, get the trained classifier's posterior over classes for each, score by the top-class probability, and return the n with the smallest top-class probability — the n the model is least confident about, equivalently the n that maximize 1 - max_y P(y | x).

```python
import numpy as np
from .strategy import Strategy


class LeastConfidence(Strategy):

    def __init__(self, X, Y, idxs_lb, net, handler, args):
        super(LeastConfidence, self).__init__(X, Y, idxs_lb, net, handler, args)

    def query(self, n):
        # the pool examples not yet labeled
        idxs_unlabeled = np.arange(self.n_pool)[~self.idxs_lb]
        # current classifier's posterior P(class | x) for each unlabeled example: [m, n_classes]
        probs = self.predict_prob(self.X[idxs_unlabeled],
                                  np.asarray(self.Y)[idxs_unlabeled])
        # confidence in the predicted (top) class: max_c P(c | x)
        U = probs.max(1)[0]
        # least confident = smallest max-prob; sort ascending and take the first n
        # (argmin of max_c P  <=>  argmax of 1 - max_c P; in the binary case <=> P nearest 0.5)
        return idxs_unlabeled[U.sort()[1][:n]]
```

Let me trace this once on a tiny pool to be sure the indexing returns what I mean. Say five unlabeled examples with binary posteriors as rows of `probs`: (0.9, 0.1), (0.55, 0.45), (0.5, 0.5), (0.2, 0.8), (0.99, 0.01). The top-class probabilities `U = probs.max(1)[0]` are 0.9, 0.55, 0.5, 0.8, 0.99. Sorting `U` ascending orders the rows by confidence, smallest first, giving index order [2, 1, 3, 0, 4]; with n = 2 the rule returns rows 2 and 1, whose top-class probabilities are 0.5 and 0.55 — the two the model is least sure of, and in this binary case the two whose posteriors sit nearest 0.5 (|0.5 - 0.5| = 0 and |0.55 - 0.5| = 0.05). Rows 0, 3, 4 with confidences 0.9, 0.8, 0.99 are correctly left for later. That matches the rule, and confirms `.max(1)[0]` takes the top-class probability per example and `.sort()[1][:n]` returns the indices of the n smallest. Mapping those positions back through `idxs_unlabeled` yields pool indices, which is what `query` must return.

Random labeling wastes the budget on examples the classifier already handles and starves the rare, boundary cases — for a 0.001-frequency class, ~60% of the time 500 random labels return no positive at all — and the version-space picture explains why: only examples in the region of uncertainty R inform the learner, and random draws hit R with a probability that decays to zero. Querying inside R is the fix, but synthesizing the ideal query point produces unlabelable gibberish on text, and computing R exactly needs the whole version space, which is intractable and gives a binary test rather than a ranking. A committee approximates R by sampling two hypotheses and querying on disagreement, but that needs noise-free realizability, uniform version-space sampling, and a committee — none of which I have. The disagreement a committee detects is just "the model is on the fence," and a single probabilistic classifier already reports that as its posterior: nearest 0.5 in two classes (verified via max(p, 1-p) = 0.5 + |p - 0.5|), and in general the smallest top-class probability, 1 - max_y P(y | x), which is exactly the model's own estimated probability of mislabeling x. Querying the n examples that maximize it gives a soft, single-model, rankable surrogate for "inside the region of uncertainty," reduces to nearest-0.5 in the binary case, costs nothing beyond the forward pass, and slots straight into the retrain-each-round loop as the acquisition rule.
