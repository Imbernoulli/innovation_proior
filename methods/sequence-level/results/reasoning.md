Let me start from what is actually breaking. I am training a large language model with RL — roll out responses from a frozen old policy, score them, take a few gradient steps on the current policy with a PPO-style clipped surrogate — and at scale it falls apart. Big models, sparsely activated mixture-of-experts, long responses: the reward curve climbs for a while and then the whole thing collapses, and the collapse is irreversible. I revert to an earlier checkpoint, I retune the clip ranges, I lengthen the generation budget, I swap the query set — none of it brings the model back. That irreversibility is the tell. A merely noisy optimizer would wobble and recover; a collapse that repeats even after rewinding and changing the obvious knobs points to a structural problem in the gradient being computed, step after step, until the model can't function. So I shouldn't be hunting for a better learning rate or a gentler schedule. I should be asking whether the gradient I'm computing is even the gradient I think it is.

What am I computing, exactly? The off-policy setup forces an importance-sampling correction. I have a batch of responses generated by the old policy π_old, I partition it into mini-batches and take several steps on π_θ, so the data is genuinely off-policy and I reweight each contribution by how much more or less likely it is under the current parameters than under the old ones. PPO's form is the clipped surrogate: with the per-token ratio r_t(θ) = π_θ(a_t|s_t)/π_old(a_t|s_t), which equals 1 at θ_old, maximize E_t[ min( r_t Â_t, clip(r_t, 1−ε, 1+ε) Â_t ) ]. The clip kills the incentive to push the ratio outside the band around 1; the min makes it a pessimistic lower bound. Fine. PPO needs a critic to produce Â_t, a value model about the size of the policy — expensive, and fragile to get right for long responses — so in practice I'm using GRPO, which throws the critic away and reads the advantage off a group: sample G responses to the same query, and set Â_i = (r(x,y_i) − mean_j r(x,y_j)) / std_j r(x,y_j), the same scalar shared across every token of y_i. Then the objective is the PPO surrogate with the per-token ratio w_{i,t}(θ) = π_θ(y_{i,t}|x,y_{i,<t})/π_old(y_{i,t}|x,y_{i,<t}), clipped per token, with a KL term I'll keep fixed and set aside. Cheap, no critic, and it works at moderate scale. So why does it die at large scale?

Let me look hard at that per-token ratio, because that's the one piece that's doing the off-policy correction and the one piece I'm least sure about. Importance sampling is a precise statistical statement: E_{z∼π_tar}[f(z)] = E_{z∼π_beh}[ (π_tar(z)/π_beh(z)) f(z) ]. The reweighting π_tar/π_beh corrects the distribution mismatch, but — and this is the part I keep glossing over — it only does so *in aggregate*. The right-hand side is an expectation over π_beh; to actually realize the correction you average the weighted function over many samples from π_beh. With one sample, (π_tar(z)/π_beh(z)) f(z) is an unbiased estimate of the target expectation, sure, but its variance can be enormous, set by how heavy-tailed the ratio is. IS is a many-sample device. Averaging is not optional decoration; it's the mechanism.

Now stare at w_{i,t}. At each token position t I have exactly one realized token, y_{i,t}, drawn from the old next-token distribution π_old(·|x,y_{i,<t}). The "importance weight" w_{i,t} = π_θ(y_{i,t}|·)/π_old(y_{i,t}|·) is built from that single draw from that position's distribution. A single weighted draw is still a formal IS estimator, but there is no averaging over the next-token distribution — there is one sample per position — so the correction is all variance and no practical averaging mechanism. Inside a clipped policy-gradient loss, that means the token ratio is no longer behaving like a reliable distribution-correction device; it is behaving like a noisy per-token multiplier. Each w_{i,t} multiplies that token's contribution to the gradient, and there are |y_i| of them mixed into one response's gradient estimate.

This is the crack. Let me follow it. The per-token weights don't cancel — they're not centered at 1 with symmetric spread that averages out; as an active gradient weight, a positive-advantage token can keep w_{i,t} anywhere in (0, 1+ε], while a negative-advantage token can keep it anywhere in [1−ε, +∞). The asymmetry is not negligible. And these weights accumulate: a long response is a long run of noisy factors feeding the per-token gradients, so the variance the scheme injects grows with response length. Worse, the per-token clipping doesn't damp this — it amplifies it. Clipping decides, token by token, whether to keep or zero each noisy contribution based on whether that noisy ratio crossed a threshold, so the clip itself is being driven by the noise. The longer the response and the larger the model, the more these accumulate, and at some point the gradient is dominated by this manufactured noise rather than by the learning signal, the update walks the model somewhere it can't recover from, and it collapses. The irreversibility now makes sense: it's not a transient, it's a steady corruption of the gradient that eventually destroys the policy.

So the diagnosis is sharper than "GRPO is unstable." The diagnosis is: GRPO applies an importance-sampling weight at a granularity where there is no averaging over the behavior distribution — one sample per next-token distribution — and the consequence is high-variance gradient noise that scales with length and is made worse by clipping. The fix can't be a better clip range. It has to be to apply the off-policy correction at a granularity where the weight is a coherent sample-level quantity.

Where does it mean something? Step back to what's being rewarded. The verifier scores the whole response y_i and hands back one number r(x,y_i); the group-relative advantage Â_i is one scalar for the whole response. The reward lives at the sequence level. But the optimization — the IS weight, the clip — lives at the token level. There's a mismatch of units. The thing I'm rewarding is a sequence; the thing I'm correcting and clipping is a token. If I want the off-policy correction to be coherent, the unit of the objective should match the unit of the reward. Reward is per-sequence, so do importance sampling per-sequence.

What's the per-sequence ratio? The natural object is π_θ(y_i|x)/π_old(y_i|x) — the likelihood of the whole response under the new policy over its likelihood under the old one. And unlike the per-token w, this has a clean meaning: it's how far the response y_i, drawn from π_old, sits relative to π_θ, a single coherent measure of the off-policy-ness of that entire sample. That's the quantity a sequence-level clip should be reading, and it aligns with the sequence-level reward. Treating a whole response's likelihood as the thing to optimize against isn't exotic — operating directly on sequence likelihood π_θ(y|x) is a known move in controllable generation, where a loss acting on log π_θ(y|x) shifts mass over entire sequences. So the building block is reasonable.

But I can't just drop π_θ(y_i|x)/π_old(y_i|x) into the surrogate as-is, and let me see why before I commit. The response likelihood is a product over |y_i| tokens, π_θ(y_i|x) = Π_t π_θ(y_{i,t}|x,y_{i,<t}), so the ratio is a product of |y_i| per-token ratios. Take its log: log[π_θ(y_i|x)/π_old(y_i|x)] = Σ_t log(π_θ(y_{i,t}|·)/π_old(y_{i,t}|·)), a *sum* over tokens of per-token log-ratios. Each per-token log-ratio is a small number that fluctuates as θ moves; sum a few hundred or a few thousand of them and tiny per-token shifts compound. Let me put a number on it before I trust the worry. Say early in an update the per-token log-ratio has standard deviation around 0.02 — small, the policies are close. If the tokens are roughly independent, the sum over L tokens has standard deviation 0.02·√L. So for L=50 the summed log-ratio has std ≈ 0.14, for L=500 it's ≈ 0.45, for L=2000 it's ≈ 0.89. Exponentiate: I simulated this and the raw sequence ratio exp(Σ) lands in roughly [0.72, 1.39] for L=50 but spreads to about [0.12, 7.9] for L=2000, with the worst single draw over 30×. Two things fall out of those numbers at once. First, the spread genuinely blows up with length — a 2000-token response routinely produces a sequence ratio 8× off 1, so I've traded per-token noise for a quantity that's even wilder in absolute magnitude. Second, and this is the killer for clipping: the scale *depends on length*. The L=50 distribution lives in [0.72, 1.39] and the L=2000 one in [0.12, 7.9]; no single clip band around 1 can be correct for both — a band that barely touches the short responses would discard most of the long ones. I'd need a different ε per length, which is untenable.

Both problems point the same way. The sum-over-tokens grows with length and its scale depends on length. So divide by the length. Define the per-token *average* log-ratio,

  (1/|y_i|) Σ_t log( π_θ(y_{i,t}|·) / π_old(y_{i,t}|·) ),

and exponentiate that:

  s_i(θ) = ( π_θ(y_i|x) / π_old(y_i|x) )^{1/|y_i|} = exp( (1/|y_i|) Σ_t log( π_θ(y_{i,t}|·) / π_old(y_{i,t}|·) ) ).

This is the geometric mean of the per-token ratios. Let me re-run the same simulation with the average in place of the sum and see whether it actually fixes the two failures. The average of L per-token log-ratios with per-token std 0.02 has std 0.02/√L — the √L now divides instead of multiplies. Numerically: for L=50 the simulated s_i has std ≈ 0.0028 and its 1st–99th percentile band is about ±0.007; for L=500, std ≈ 0.0009, band ±0.002; for L=2000, std ≈ 0.00045, band ±0.001. Both failures are gone. The numerical range is now per-token regardless of length — all three lengths put s_i within a hair of 1, in bands that overlap heavily, so one clip band ε can serve all of them. And it's a genuine variance reducer: where the raw sum's spread *grew* as 0.02·√L, the average's spread *shrinks* as 0.02/√L, so a longer response gives a *more* stable s_i, not a less stable one — a few tokens swinging hard can drag a sum dramatically but barely move a mean over hundreds of tokens. So length normalization does exactly the two jobs the raw product failed at, and the numbers show it cleanly: it controls the variance and it unifies the numerical range across lengths. The 1/|y_i| isn't cosmetic; it's the thing that makes a sequence-level ratio usable at all.

And it answers the question I parked earlier about the clip width. Those bands tell me the scale directly: s_i sits within about ±10⁻³ of 1 for realistic lengths, with the tightest case (L=2000) inside ±10⁻³ and even the short L=50 case under ±10⁻². So a meaningful clip on s_i wants a band a few times 10⁻⁴ wide — orders of magnitude tighter than GRPO's ε≈0.2, which lives where it does because the per-token w it clips has the *un-averaged* spread (in the same simulation a single-token ratio has std ≈ 0.02 and a ±0.05 band, so ε≈0.2 sits a comfortable few times above its typical spread). Same clipping idea, a natural scale set entirely by how the ratio is defined, and now I have it from numbers rather than a guess.

Now build the objective around it. Same clipped-surrogate shape as PPO, but with s_i in place of the per-token ratio, the group-relative advantage Â_i, and the average and clip both at the sequence level:

  J_GSPO(θ) = E_{x∼D, {y_i}∼π_old(·|x)}[ (1/G) Σ_i min( s_i(θ) Â_i, clip(s_i(θ), 1−ε, 1+ε) Â_i ) ].

Clipping a whole response, not individual tokens, to throw out responses that are too off-policy — which is what s_i measures — and that matches the sequence-level reward and the sequence-level advantage. One thing I should flag to myself: because s_i is defined so differently from the per-token w — it's a length-normalized geometric mean sitting very close to 1, since each per-token log-ratio is small and the average of small numbers is small — the clip width ε that makes sense here is presumably narrower than GRPO's, but I don't actually know by how much, and a clip band is useless if it's the wrong scale. I'll fix the scale with numbers later, once I've pinned down how tightly s_i hugs 1; right now the structure is what matters.

Let me check that this actually does what I claimed to the gradient, because the whole motivation was "the per-token weighting injects noise; weight things coherently instead." So I need the gradient of J_GSPO and I need to compare it term-for-term with GRPO's. Drop the clipping for the derivation (it's a region selector, not the heart of the gradient). The unclipped objective per response is s_i(θ) Â_i, and Â_i doesn't depend on θ. I want ∇_θ E[(1/G) Σ_i s_i Â_i]. The clean way is the log-derivative trick: ∇_θ s_i = s_i ∇_θ log s_i, so

  ∇_θ J_GSPO = E[ (1/G) Σ_i s_i(θ) Â_i ∇_θ log s_i(θ) ].

Now ∇_θ log s_i. From s_i = exp((1/|y_i|) Σ_t log π_θ(y_{i,t}|·) − (same with π_old)), the log is log s_i = (1/|y_i|) Σ_t [ log π_θ(y_{i,t}|·) − log π_old(y_{i,t}|·) ], and only the π_θ terms depend on θ, so

  ∇_θ log s_i(θ) = (1/|y_i|) Σ_t ∇_θ log π_θ(y_{i,t}|x, y_{i,<t}).

Substitute back:

  ∇_θ J_GSPO = E[ (1/G) Σ_i ( π_θ(y_i|x)/π_old(y_i|x) )^{1/|y_i|} Â_i · (1/|y_i|) Σ_t ∇_θ log π_θ(y_{i,t}|·) ].

Now do the same for GRPO so I can line them up. Its unclipped per-token objective is w_{i,t} Â_{i,t} with Â_{i,t} = Â_i, and ∇_θ(w_{i,t} Â_i): since w_{i,t} = π_θ(y_{i,t}|·)/π_old(y_{i,t}|·), I have ∇_θ w_{i,t} = w_{i,t} ∇_θ log π_θ(y_{i,t}|·), so

  ∇_θ J_GRPO = E[ (1/G) Σ_i Â_i · (1/|y_i|) Σ_t w_{i,t}(θ) ∇_θ log π_θ(y_{i,t}|·) ].

Put them side by side. Both are, per response, Â_i times an average over tokens of (something) · ∇_θ log π_θ(y_{i,t}|·). The only difference — the *entire* difference — is the "something," the weight multiplying each token's score-function gradient ∇_θ log π_θ(y_{i,t}|·). In GRPO that weight is the per-token w_{i,t}, a different number for every token. In GSPO it is the single s_i, the same number for every token in the response. So the two objectives produce the same family of gradients up to one substitution: GRPO weights the per-token log-likelihood gradients *unequally* by their individual noisy ratios; GSPO weights them all *equally* by one coherent sequence-level ratio. The unequal weights are exactly the object the diagnosis pointed at — they can range over (0,1+ε] for positive advantage and [1−ε,∞) for negative advantage, they're not negligible, and their idiosyncratic position-by-position variation is the noise that accumulates over the response. Collapsing them to a single s_i per response removes that variation from the gradient by construction. So the original worry — "the per-token weighting injects noise" — and the fix — "weight things coherently instead" — turn out to be the same statement read in two directions: the per-token disparity *is* the instability, and equal weighting *is* its removal. That's exactly the change I wanted from the gradient, not a side effect of it.

There's a corollary I should pick up on the MoE side, because that was the most violent failure. In a sparsely activated model the experts that fire for a given response can shift after a single gradient update — measurably, around a tenth of the activated experts change for the same sample on a deep MoE, and more in deeper models. Each token's likelihood depends on which experts fired, so the per-token ratios w_{i,t} swing drastically across an update — the numerator and denominator are effectively evaluated through different sub-networks. That's catastrophic for a scheme whose gradient weight *is* the per-token ratio. But s_i uses the length-normalized aggregate sequence likelihood π_θ(y_i|x), not an exposed weight for any individual token's likelihood. If the model keeps its overall language-modeling ability, the averaged sequence log-ratio should be far less likely to lurch when individual experts reshuffle, because the per-token wiggles are pooled in the length-normalized sum. The workaround I would otherwise need is routing replay: cache the experts activated under π_old and force π_θ to reuse them when computing token ratios, so the numerator and denominator share an activated sub-network. That workaround stabilizes the token-level ratio but costs memory, communication, and usable MoE capacity. If the objective no longer asks any single token's ratio to be stable across the update, the reason for that workaround largely disappears; the quantity I need stable is the sequence average.

The same unit change matters for the training system, not just the math. In a disaggregated RL stack, the rollout engine and the training engine may not produce bit-identical likelihoods because they use different kernels, precisions, or batching. If the loss depends on every token-level ratio, those small per-token discrepancies become active gradient weights, so the conservative thing is to recompute old-policy log-probabilities in the training engine. If the loss depends on a length-normalized sequence likelihood instead, the small discrepancies are averaged before they become the ratio. That makes the objective much more tolerant of using the rollout-side likelihoods directly, especially in partial-rollout or multi-turn settings where recomputation is expensive.

Now I want to be honest about the clip-fraction consequence, because it is easy to misread. Clipping at the sequence level means when s_i lands outside the band, the *whole response* is excluded from the gradient — all its tokens at once. Per-token clipping only drops the offending tokens. So sequence-level clipping can mark many more token positions as clipped simply because the response is the unit being judged. The thing to hold onto is *why that is acceptable*: the dropped tokens belong to a response that is too off-policy as a whole, which is the coherent unit to judge; whereas tokens retained under per-token clipping can still carry noisy per-token weights that, by the gradient analysis above, are an unreliable learning signal. The point is not to maximize the raw count of token gradients. The point is to keep gradients whose off-policy weight is meaningful.

The clip magnitude I already pinned down from the spread simulation: s_i sits within a few times 10⁻⁴ of 1 for realistic lengths, so the band wants to be a few times 10⁻⁴ wide, with the left and right edges allowed to differ (a tighter floor than ceiling, carrying over the decoupled-clip idea) — versus GRPO's ε≈0.2, which lives a couple of orders of magnitude higher because the un-averaged per-token ratio it clips spreads that much more. Same clipping idea, different natural scale, purely because of how the ratio is defined. I'll read ε from config rather than bake it in, since the right number is small and easy to get wrong.

Now the implementation question, and this is where I have to be careful, because the clean math object s_i is one scalar per sequence but the autodiff graph is built over per-token log-probs. I want two things at once: in the forward pass, every token of response i should see the *value* s_i (broadcast to the whole sequence), so the clip and the surrogate use the sequence-level ratio; and in the backward pass, the gradient should flow through the per-token log π_θ(y_{i,t}|·) with the equal weight s_i, exactly as my gradient derivation says. If I literally compute s_i as a per-sequence scalar and broadcast it, the autodiff will still differentiate it correctly — ∇ log s_i = (1/|y_i|) Σ_t ∇ log π_θ — so that's fine in principle. But there's a more flexible construction that also lets me attach a *per-token* advantage later if I ever want finer-grained control (multi-turn settings, say), while staying numerically identical to GSPO when the advantage is shared. Define a token-level surrogate ratio

  s_{i,t}(θ) = sg[ s_i(θ) ] · π_θ(y_{i,t}|·) / sg[ π_θ(y_{i,t}|·) ],

where sg[·] is stop-gradient — detach in PyTorch. Numerically the second factor π_θ/sg[π_θ] is 1, so s_{i,t} should equal s_i token-for-token; the forward value should be exactly the sequence ratio broadcast to every token. And the *gradient* should be carried only by the un-detached π_θ(y_{i,t}|·) in the numerator: ∇_θ s_{i,t} = sg[s_i] · ∇_θ π_θ(y_{i,t}|·)/sg[π_θ(y_{i,t}|·)] = s_i · ∇_θ log π_θ(y_{i,t}|·) at the evaluation point. That's a strong claim about what autograd will actually do, and stop-gradient constructions are exactly where I tend to be wrong, so I won't take it on faith — I'll build the tensor on a tiny input and read off both the forward value and the gradient. Two responses, lengths 5 and 3 (the second padded), small random per-token log-probs, the log-space form log s_{i,t} = sg[log s_i] + log π_θ − sg[log π_θ]. The forward check: exp of that, compared against exp(neg_kl_seq) broadcast to every masked token — the maximum absolute difference comes out 0.0, so the forward value really is s_i on every token. The backward check: differentiate Σ(s_{i,t}·mask) with respect to the per-token log-prob (whose own score is 1 here) and compare against s_i broadcast — maximum difference again 0.0. And the gradient rows are the thing I actually care about: for the length-5 response every one of the five entries is 1.00498, and for the length-3 response every one of the three real entries is 1.00047 with zeros on the two padded positions. Every token of a response gets the *same* weight s_i — the equal-weighting from the gradient derivation, now visible in the numbers rather than argued. So summing the token-level surrogate with possibly-per-token advantages reproduces ∇J = E[(1/G)Σ_i s_i · (1/|y_i|) Σ_t Â_{i,t} ∇log π_θ(y_{i,t}|·)], which collapses to the GSPO gradient when Â_{i,t} = Â_i. Forward value is the sequence ratio, backward path is the per-token log-prob, advantage can be per-token — flexibility for free, and the tiny-input trace says it's numerically GSPO. I'll implement this form.

In log space the same construction is cleaner, which is what I'll actually code, because I already have the per-token log-ratios. log s_{i,t} = sg[log s_i] + log π_θ(y_{i,t}|·) − sg[log π_θ(y_{i,t}|·)]. I have log_prob = log π_θ per token and old_log_prob = log π_old per token. The per-token log-ratio is negative_approx_kl = log_prob − old_log_prob (it's the negative of the usual approximate KL, hence the name). The sequence average is the masked mean of that over the response, neg_kl_seq = Σ_t (negative_approx_kl · mask) / Σ_t mask, one scalar per row. Then

  log_seq_ratio = log_prob − sg[log_prob] + sg[neg_kl_seq],

broadcast neg_kl_seq across the token axis; this is exactly sg[log s_i] (the detached sequence average) plus the live-minus-detached per-token log_prob that carries the gradient. Exponentiate to get the per-token ratio whose value is s_i and whose gradient is the per-token score function. One numerical guard: before exp, clamp log_seq_ratio at a safe ceiling so a pathological row can't overflow. Where to put the ceiling? The legitimate value of log_seq_ratio is neg_kl_seq, the average per-token log-ratio, which from the simulation above sits around ±10⁻³ — so a legitimate exp(log_seq_ratio) is exp(±10⁻³) ≈ 1.001. For a row to even approach a clamp at 10 its *average* per-token log-ratio would have to be 10, i.e. every token roughly e¹⁰ ≈ 22000× more likely under π_θ than π_old — utterly impossible for a model that still functions. So a ceiling of 10 sits about four orders of magnitude above any legitimate value and only ever fires on a genuinely diverged row; exp(10) ≈ 22000 is finite and harmless in fp32. A max of 10 it is.

Then it's the ordinary clipped-surrogate machinery. With seq_ratio = exp(log_seq_ratio):
the unclipped per-position loss is pg_losses1 = −Â · seq_ratio, the clipped one is
pg_losses2 = −Â · clip(seq_ratio, 1−ε_low, 1+ε_high), and I take the elementwise max of the two as the loss (max of the negatives = min of the surrogates, the pessimistic bound). The clip fraction is the masked mean of [pg_losses2 > pg_losses1], the fraction of positions where the clipped branch bit. The reported KL is the masked mean of −negative_approx_kl.

The one piece that must follow the sequence-level structure is the *aggregation*. The objective is (1/G) Σ_i [ per-response quantity ], and the per-response quantity is itself a token-mean over that response (because s_i was a length-normalized, per-token object and I broadcast the loss across tokens). So I aggregate as seq-mean-token-mean: for each sequence take the token-mean of its per-position losses over the masked tokens, then take the mean over sequences. If I instead used a flat token-mean over the whole batch, long responses would dominate the loss in proportion to their length, undoing the length normalization I built s_i to have. The aggregation has to mirror the objective's two-level (1/G)(1/|y_i|) structure, so seq-mean-token-mean it is, not the default token-mean.

Let me write it as the policy-loss function that fills the slot, grounded in the verl trainer's primitives — the per-token log-probs and advantages come in, the masked-mean and aggregation helpers exist, the clip band is read from config:

```python
from typing import Any, Optional

import torch
import verl.utils.torch_functional as verl_F
from verl.workers.config import ActorConfig
from verl.trainer.ppo.core_algos import agg_loss, register_policy_loss


@register_policy_loss("gspo")
def compute_policy_loss_gspo(
    old_log_prob: torch.Tensor,      # (bs, response_length): log π_old per token
    log_prob: torch.Tensor,          # (bs, response_length): log π_θ  per token
    advantages: torch.Tensor,        # (bs, response_length): group-relative advantage
    response_mask: torch.Tensor,     # (bs, response_length): 1 on real response tokens
    loss_agg_mode: str = "seq-mean-token-mean",
    config: Optional[ActorConfig] = None,
    rollout_is_weights: torch.Tensor | None = None,
) -> tuple[torch.Tensor, dict[str, Any]]:
    """Compute the clipped policy objective and related metrics for the sequence-level loss."""
    assert config is not None
    assert isinstance(config, ActorConfig)
    # s_i hugs 1 (it is exp of an *average* per-token log-ratio), so this band is far
    # narrower than a token-level ratio's; read it from config, don't hardcode ε.
    clip_ratio_low = config.clip_ratio_low if config.clip_ratio_low is not None else config.clip_ratio
    clip_ratio_high = config.clip_ratio_high if config.clip_ratio_high is not None else config.clip_ratio

    # per-token log-ratio  log π_θ − log π_old  (the negative approximate KL)
    negative_approx_kl = log_prob - old_log_prob

    # length-normalized sequence log-ratio: (1/|y_i|) Σ_t (log π_θ − log π_old), masked mean
    seq_lengths = torch.sum(response_mask, dim=-1).clamp(min=1)
    negative_approx_kl_seq = torch.sum(negative_approx_kl * response_mask, dim=-1) / seq_lengths

    # straight-through: forward value is the broadcast sequence ratio s_i;
    # backward path is the per-token log π_θ, so each token's score gradient is weighted by s_i.
    #   log s_{i,t} = sg[log s_i] + log π_θ − sg[log π_θ]
    log_seq_importance_ratio = (
        log_prob - log_prob.detach() + negative_approx_kl_seq.detach().unsqueeze(-1)
    )
    log_seq_importance_ratio = torch.clamp(log_seq_importance_ratio, max=10.0)
    seq_importance_ratio = torch.exp(log_seq_importance_ratio)

    # PPO-style clipped surrogate, but with the sequence ratio and sequence-level clip
    pg_losses1 = -advantages * seq_importance_ratio
    pg_losses2 = -advantages * torch.clamp(seq_importance_ratio, 1 - clip_ratio_low, 1 + clip_ratio_high)
    pg_losses = torch.maximum(pg_losses1, pg_losses2)       # max of negatives = min of surrogates

    if rollout_is_weights is not None:
        pg_losses = pg_losses * rollout_is_weights

    # objective is (1/G) Σ_i (1/|y_i|) Σ_t [...]  → per-seq token-mean, then mean over sequences
    pg_loss = agg_loss(
        loss_mat=pg_losses, loss_mask=response_mask,
        loss_agg_mode="seq-mean-token-mean", **config.global_batch_info,
    )

    pg_clipfrac = verl_F.masked_mean(torch.gt(pg_losses2, pg_losses1).float(), response_mask)
    pg_clipfrac_lower = torch.tensor(0.0, device=pg_loss.device)
    ppo_kl = verl_F.masked_mean(-negative_approx_kl, response_mask)
    pg_metrics = {
        "actor/pg_clipfrac": pg_clipfrac.detach().item(),
        "actor/ppo_kl": ppo_kl.detach().item(),
        "actor/pg_clipfrac_lower": pg_clipfrac_lower.detach().item(),
    }
    return pg_loss, pg_metrics
```
