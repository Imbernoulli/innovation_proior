When I train a GRU-based translation model, adding layer normalization transforms the loss-versus-steps curve: where the unnormalized baseline still sits around $7.0$, the normalized model is already down near $5.4$ after the same number of steps. That is a large, real win, and it is why the layer became standard for sequence models and Transformers. But plot the two losses against wall-clock *time* instead of steps and the gap collapses to about $5.9$ at equal time. The layer is doing genuine work, but I pay for it on every forward pass — and in a recurrent network that means on every timestep of every layer, where the per-step arithmetic is multiplied by the sequence length and the depth. So a good fraction of the per-step gain is handed straight back as slower steps. The question is whether I am paying for something I do not need. To answer it I have to be precise about what the standard layer actually computes, because if part of the mechanism is dead weight I want to delete it. The neighboring options do not help here: BatchNorm pools statistics across the mini-batch, so it cannot cleanly handle variable-length sequences and needs running averages at test time; WeightNorm reparameterizes the weights as $w = g\,v/\lVert v\rVert$ but never looks at the activation distribution, so it is not invariant to input or dataset re-scaling and has not matched BatchNorm's accuracy; and plain Euclidean normalization of activations has never been shown to work as a wholesale replacement. The standard layer is the one to beat, and the goal is to keep its stabilization while cutting its cost — which first requires knowing *which part* of it is load-bearing.

For summed inputs $a \in \mathbb{R}^n$ with $a_i = \sum_j w_{ij} x_j$, the standard layer computes $\mu = \frac{1}{n}\sum_i a_i$ and $\sigma = \sqrt{\frac{1}{n}\sum_i (a_i - \mu)^2}$, then standardizes $\bar a_i = \frac{a_i - \mu}{\sigma}\, g_i$ with a learned per-neuron gain $g$, and outputs $y_i = f(\bar a_i + b_i)$. Mechanically this bundles two distinct operations: subtracting the mean (recentering the vector to zero mean) and dividing by the standard deviation (rescaling it to unit spread). They feel like one move, "standardize," but they are two, they cost differently, and they buy different things. The cleanest way to see what each buys is through invariances. Shift every summed input by a constant, $a \to a + c\mathbf{1}$ — which a recentered weight matrix can produce — and $\mu$ shifts by the same $c$, so $a - \mu$ is unchanged: the mean-subtraction is exactly what gives *re-centering invariance*. Scale everything, $a \to \alpha a$ — from scaling the weights or inputs by $\alpha$ — and $\mu \to \alpha\mu$, $\sigma \to \alpha\sigma$, so $\frac{\alpha a - \alpha\mu}{\alpha\sigma} = \frac{a-\mu}{\sigma}$ is unchanged: the division by $\sigma$ is what gives *re-scaling invariance*. Two operations, two invariances, cleanly separated.

The prevailing story credits both invariances, but that is a story, not a measurement. Stabilizing training is fundamentally about controlling *spread* — keeping hidden states and gradients from blowing up or vanishing as they propagate. Subtracting the mean recenters the cloud of activations but does nothing to its variance: $\mathrm{var}(a - \mu) = \mathrm{var}(a)$. The operation that actually pulls the variance to a fixed scale regardless of how large the weights have grown is the *division by $\sigma$*. So I bet that re-scaling is the load-bearing invariance and re-centering is the weaker suspect, and I act on it: drop the mean-subtraction entirely and keep only the scale normalization. I propose **RMSNorm** (Root Mean Square Layer Normalization). Once $\mu$ is gone I can no longer divide by $\sigma$, because $\sigma$ is *defined* through $\mu$ as the spread around the mean. The natural mean-free measure of scale is the spread around the origin — the root mean square:

$$\mathrm{RMS}(a) = \sqrt{\frac{1}{n}\sum_{i=1}^n a_i^2},\qquad \bar a_i = \frac{a_i}{\mathrm{RMS}(a)}\, g_i .$$

No $\mu$, no subtraction, one statistic. A sanity check confirms this is a generalization rather than a different animal: if the summed inputs already have zero mean, then $\sigma = \sqrt{\frac{1}{n}\sum_i a_i^2} = \mathrm{RMS}(a)$ and the two rules coincide exactly. I have not changed the layer wildly; I have switched off the recentering and left the mean wherever it lands.

Dropping the mean costs an invariance, and I want to know precisely which one moves. RMS is linear, $\mathrm{RMS}(\alpha a) = |\alpha|\,\mathrm{RMS}(a) = \alpha\,\mathrm{RMS}(a)$ for $\alpha > 0$, so $\frac{\alpha a}{\alpha\,\mathrm{RMS}(a)} = \frac{a}{\mathrm{RMS}(a)}$: re-scaling invariance survives intact — exactly the one I argued matters. But a shift $a \to a + c\mathbf{1}$ has nothing to subtract it, so re-centering invariance is gone. Writing the layer in general form $y = f\big(\frac{Wx}{\mathrm{RMS}(a)}\odot g + b\big)$ with $a = Wx$, scaling the whole weight matrix $W' = \delta W$ gives $a' = \delta a$, $\mathrm{RMS}(a') = \delta\,\mathrm{RMS}(a)$, and $\frac{\delta Wx}{\delta\,\mathrm{RMS}(a)} = \frac{Wx}{\mathrm{RMS}(a)}$ — invariant to weight-matrix re-scaling; the identical cancellation holds for scaling the input, a batch, or the whole dataset, since each example cancels its own $\delta$. But scaling individual weight *rows* by different $\delta_i$ sends $a_i \to \delta_i a_i$ while $\mathrm{RMS}(a)$ pools all neurons, so no single $\delta_i$ factors out of $a_i/\mathrm{RMS}(a)$: RMSNorm is not invariant to per-weight-vector re-scaling — and neither was the standard layer, for the same pooling reason. That is WeightNorm's invariance, bought by normalizing each weight vector alone, at the price of never seeing the activation distribution. So RMSNorm matches the standard layer's invariance table everywhere except it loses weight-matrix re-centering — one checkbox, and it is precisely the one I am gambling does not matter.

There is a geometric reading worth holding onto: $\frac{a}{\mathrm{RMS}(a)}$ satisfies $\sum_i\big(\frac{a_i}{\mathrm{RMS}}\big)^2 = n$, so RMS pins the summed-input vector to the sphere of radius $\sqrt{n}$ regardless of how inputs and weights were scaled. That immediately raises the obvious cheaper-looking alternative: divide by the plain Euclidean norm $\lVert a\rVert = \sqrt{\sum_i a_i^2}$, which differs from RMS only by the constant $\sqrt{n}$ since $\lVert a\rVert = \sqrt{n}\,\mathrm{RMS}(a)$, and which WeightNorm already uses successfully on weights. But $\lVert a\rVert$ forces $a$ onto the *unit* sphere, radius $1$ independent of $n$. RMS forces it onto the radius-$\sqrt{n}$ sphere, which grows with layer width. That matters because a fixed unit radius squashes a $1000$-dimensional and a $10$-dimensional vector to the same length, so each coordinate of the wide vector becomes tiny while each coordinate of the narrow one does not, and the per-coordinate scale diverges across layers. The $\frac{1}{n}$ inside the root normalizes each *coordinate* to roughly unit scale regardless of width, making the scheme behave consistently across layers of different size. The factor I would have been tempted to drop is the one carrying the robustness, which is why plain L2 normalization of activations does not serve as a drop-in.

The real test of a normalization layer is not its forward invariances but whether it keeps the *gradients* well-behaved, since helping the optimizer is the whole point. Write $v = \frac{Wx}{\mathrm{RMS}(a)}\odot g + b$, so $y = f(v)$. The easy parameters: $b$ enters additively, so $\partial L/\partial b = \partial L/\partial v$; $g$ multiplies the normalized inputs elementwise, so $\partial L/\partial g = \partial L/\partial v \odot \frac{Wx}{\mathrm{RMS}(a)}$. Both are invariant to scaling $x$ or $W$ — the normalized inputs do not move, and $b$'s gradient never saw the scale — and the $g$-gradient is proportional to the *normalized* inputs, so $g$ always sees controlled-magnitude quantities and its own scale does not drift. The weight matrix is where it gets interesting, because RMS is quadratic in $a = Wx$ and so $W$ enters through both numerator and denominator. With $S = \mathrm{RMS}(a)$, $\partial S/\partial a_k = \frac{a_k}{nS}$, and the quotient rule gives $\frac{\partial(a_j/S)}{\partial a_k} = \frac{1}{S}\big(\delta_{jk} - \frac{a_j a_k}{nS^2}\big)$, so the Jacobian of the normalized vector with respect to $a$ is

$$R = \frac{1}{\mathrm{RMS}(a)}\Big(I - \frac{(Wx)(Wx)^\top}{n\,\mathrm{RMS}(a)^2}\Big).$$

This is $1/S$ times identity minus a rank-one outer-product correction, and the correction projects out the radial direction — perturbing $a$ along itself leaves $a/\mathrm{RMS}(a)$ fixed, so $R$ must annihilate that direction. Indeed $R\cdot a = \frac{1}{S}\big(a - a\frac{a^\top a}{nS^2}\big) = 0$ because $a^\top a = nS^2$, confirming the derivation. Naming the upstream gradient $u = \partial L/\partial v$ and $z = a/S$, the gradient entering $z$ is $q = g\odot u$, and since $R$ is symmetric and $a = Wx$, one training case contributes

$$\frac{\partial L}{\partial W} = \big(R(g\odot u)\big)\,x^\top$$

(summed over a batch). The structural fact lives in $R$. Scale $x$ or $W$ by $\delta$: $a \to \delta a$, $\mathrm{RMS}(a)\to\delta\,\mathrm{RMS}(a)$, and the $\delta^2$ in the outer-product cancels, leaving $R' = \frac{1}{\delta}R$. Feed that back. If I scaled the *input*, the $x^\top$ factor is multiplied by $\delta$ while $R$ is multiplied by $1/\delta$, so $\partial L/\partial W$ is invariant — rescaling the data does not jolt the weight gradients. If I scaled the *weights*, $x$ is unchanged so only $R$ moves and $\partial L/\partial W$ is multiplied by $1/\delta$: a layer whose weights have grown large automatically receives *smaller* gradients, which damps further growth. That is an implicit, per-layer learning-rate adaptation falling straight out of the quadratic form of RMS — big weights, small steps — with no schedule and no extra parameters. None of this came from the recentering I dropped; all of it comes from the re-scaling structure I kept. And the bias is off by default precisely because the shift only existed to restore a location after recentering, and there is no recentering to undo.

The original motivation was cost, so the accounting: the standard layer needs one reduction for $\mu$, a second over the centered squares for $\sigma$, plus an elementwise subtraction of $\mu$ across all $n$ entries; RMSNorm needs a single reduction — sum of squares — a reciprocal-square-root, and an elementwise multiply. One fewer reduction and the entire subtraction pass gone. In a recurrent net that constant is multiplied by sequence length and depth, turning into the wall-clock speedup in exactly the regime where the time-versus-steps gap was widest. The cost can go further. The $n$ neurons are roughly identically distributed, so I do not strictly need all of them to estimate the root mean square: take the first $k = \lceil n\,p\rceil$ entries for a fraction $p$, form $\overline{\mathrm{RMS}}(a) = \sqrt{\frac{1}{k}\sum_{i=1}^k a_i^2}$, and normalize *all* $n$ entries by it. Call it partial, or **pRMSNorm**. The only property the invariance proofs used was linearity, and $\overline{\mathrm{RMS}}(\alpha a) = \alpha\,\overline{\mathrm{RMS}}(a)$ holds too, so pRMSNorm inherits the identical invariance table while the reduction shrinks roughly in proportion to $k/n$. The honest caveat is that $\overline{\mathrm{RMS}}$ is a noisy, generally biased estimate of the full-vector RMS, and since $1/\mathrm{RMS}$ sits in the denominator of the gradient, a denominator estimated from too few samples can occasionally come out small and make the gradient spike; so $p$ is a cost-versus-stability knob, not a free reduction, and gradients tend to explode if it is pushed too small. The forward is almost nothing, which is the point.

```python
import torch
import torch.nn as nn


class RMSNorm(nn.Module):
    def __init__(self, d, p=-1., eps=1e-8, bias=False):
        """
        d:    layer width (number of summed inputs / neurons).
        p:    partial ratio; this implementation treats 0 <= p <= 1 as partial,
              so practical partial use should keep p > 0. p < 0 or p > 1 disables partial mode.
        eps:  numerical floor on the denominator.
        bias: optional shift; off by default since RMSNorm does no mean-subtraction
              and therefore enforces no re-centering invariance to restore.
        """
        super().__init__()
        self.eps, self.d, self.p, self.bias = eps, d, p, bias
        self.scale = nn.Parameter(torch.ones(d))          # gain g, init 1
        self.register_parameter("scale", self.scale)
        if bias:
            self.offset = nn.Parameter(torch.zeros(d))
            self.register_parameter("offset", self.offset)

    def forward(self, x):
        if self.p < 0. or self.p > 1.:
            norm_x = x.norm(2, dim=-1, keepdim=True)      # ||a|| = sqrt(sum a_i^2): one reduction
            d_x = self.d
        else:
            partial_size = int(self.d * self.p)           # use first int(d*p) entries
            partial_x, _ = torch.split(x, [partial_size, self.d - partial_size], dim=-1)
            norm_x = partial_x.norm(2, dim=-1, keepdim=True)
            d_x = partial_size

        rms_x = norm_x * d_x ** (-1. / 2)                 # full RMS or partial estimate = selected norm / sqrt(d_x)
        x_normed = x / (rms_x + self.eps)                 # a_i / RMS(a), no mean subtracted

        if self.bias:
            return self.scale * x_normed + self.offset
        return self.scale * x_normed                      # a_i / RMS(a) * g_i
```
