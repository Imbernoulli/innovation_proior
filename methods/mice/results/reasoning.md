Let me start from what actually goes wrong when I try to fill a table with holes in more than one column. I have `n` rows and `p` variables, and `NaN`s scattered all over — not confined to a single column, but in several at once, and in different rows for different columns. I want a completed table I can hand to whatever analysis comes next. The cheapest thing is to replace each column's missing entries by that column's mean. I know exactly why that is bad, and I should say it precisely rather than wave at it. Mean-filling does two separate damaging things. It ignores every other variable, so if cholesterol and BMI move together, a row with a high BMI and a missing cholesterol gets the *global* average cholesterol, throwing away the one piece of information I had. And it inserts the identical constant into every hole in a column, which adds zero scatter: the column's variance shrinks (I replaced a spread-out set of unknowns by a single number repeated), and every correlation that column has with the others is pulled toward zero, because constants do not co-vary with anything. So mean imputation is wrong on relationships and wrong on uncertainty, in two independent ways. I need to fix both.

Fixing the first one — using the other variables — has an obvious shape. To fill `Y_j`, regress `Y_j` on the other variables `Y_{-j}` over the rows where `Y_j` was observed, and predict the missing rows. That borrows strength across the correlated columns, exactly what mean-filling refused to do. And it has a licence: under the missing-at-random assumption (and the table here is missing-completely-at-random, which is the strict special case), Rubin showed the missingness mechanism is ignorable, meaning the conditional distribution of the missing given the observed is the right thing to draw from. So `P(Y_j_mis | Y_j_obs, Y_{-j})` — which is exactly what a regression of `Y_j` on `Y_{-j}` estimates — is a legitimate object to impute from. Good. But the moment I write "regress `Y_j` on the other variables," I hit the tangle that makes this hard: the other variables are *also* full of holes. I cannot regress cholesterol on BMI if BMI itself has missing entries in the very rows I care about. And it is circular: BMI would help me fill cholesterol, but cholesterol would equally help me fill BMI. There is no variable I can safely treat as a complete predictor, because none of them is complete. So the single-column regression idea, taken literally, does not even get off the ground on multivariate missingness.

The textbook way around "I need a joint, and everything depends on everything" is to write down one big joint distribution over all `p` variables and draw the missing entries from it. Schafer's approach: assume `Y ~ N(mu, Sigma)`, a `p`-variate normal with unstructured covariance, and run data-augmentation MCMC — alternately draw the parameters given the (currently completed) data, then draw the missing data given the parameters. When the data really are jointly normal this is beautiful: every imputation comes from a single coherent probability model, and both the parameter uncertainty and the residual uncertainty fall straight out of the MCMC. But look at the data I actually have. The columns are mixed: some continuous and skewed and bounded below by zero, some binary, some ordered categories. A multivariate normal is flatly wrong for a binary column — it will hand me continuous fills like 0.37 for a yes/no variable, and negative fills for a strictly positive one. I could try to build a fancier joint — a general-location model, latent normals under the categoricals — but each variable type I want to honor forces another layer onto the joint, and the relations can be nonlinear or constrained in ways no convenient multivariate family captures. The joint is the bottleneck: the whole method is hostage to my ability to write down, and sample from, one global distribution for heterogeneous data. For the data in front of me, I cannot.

So let me stare at what I actually *can* write down. I cannot specify a sensible joint `P(Y | theta)`. But for any *single* column, given the others, I can almost always pick a sensible model: a linear regression for a continuous variable, a logistic regression for a binary one, an ordered-logit for an ordinal one. The per-variable conditional is something I understand and can build; the joint is not. So what if I never form the joint at all, and instead specify the imputation model one column at a time — `P(Y_j | Y_{-j}, theta_j)` for each `j`, each with its own little parameter `theta_j` and its own appropriate functional form? That sidesteps the entire bottleneck. The cost, which I should be honest about now, is that a collection of conditionals chosen independently need not correspond to *any* joint distribution — they can be mutually incompatible, in the sense that no `P(Y)` has all of them as its conditionals. With a joint model, coherence is automatic; here I am giving it up. I will have to come back to whether that bites.

Now, granting per-column conditionals, how do I beat the circularity? The circularity was: to fill `Y_1` I need `Y_2` filled, to fill `Y_2` I need `Y_1` filled. The way out is to break the deadlock with a rough start and then refine. Fill *every* column's holes first, with anything cheap — a random draw from that column's own observed values, say, just to get a complete table to stand on. Now every variable is (crudely) complete, so I *can* run the single-column regression for each one. So I sweep: take `Y_1`, throw away its current fill, regress `Y_1` on the *current* values of all the other columns over the rows where `Y_1` was really observed, and re-impute `Y_1`'s holes from that regression. Then move to `Y_2`, regress it on the now-updated others (including the freshly re-imputed `Y_1`), re-impute. Go through all `p` columns. That is one pass. Then do it again, and again. Each variable keeps getting re-imputed in light of the latest fills of the rest, so the cross-variable dependence propagates around the cycle instead of being frozen at whatever the crude initialization happened to be. The chained structure — fill column by column, looping — is what dissolves the circular dependence: nobody is ever treated as a fixed complete predictor; everybody is conditionally re-estimated given everybody else's current state, repeatedly.

I want to recognize what this iteration actually *is*, because if I just call it "alternating regressions" I will mis-tune it. Look at the abstract move stripped of the regression details: I am replacing each block `Y_j` by a sample from its full conditional given everything else, `P(Y_j | Y_{-j}, theta_j)`, cycling through the blocks `j = 1, ..., p`, over and over. That is the exact shape of a Gibbs sampler — sweep the coordinates, each time redraw one from its conditional given the current rest. So *if* the per-column conditionals I picked were the genuine conditionals of some single joint `P(Y | theta)`, the sweep would be Gibbs sampling from that joint, and its standard theory would apply: after the chain forgets its start, the completed tables are draws from the correct posterior predictive distribution of the missing data. I should hold the "if" lightly — I have not checked that my hand-picked conditionals correspond to any joint, and that is a real gap I'll have to return to. But the recognition is still useful even before I settle that, because it tells me what each step must look like to be a *proper* conditional draw rather than a point estimate: at column `j` I should draw the *parameters* `theta_j` from their posterior given the current data, and then draw `Y_j`'s missing entries given those drawn parameters — two draws, not a single plug-in prediction. A plug-in would be conditioning on a point estimate of `theta_j` as if it were known, which is not sampling from the conditional. So the chained-regression procedure I arrived at by sheer necessity (it was the only escape from the circularity) lines up with a principled MCMC reading, and that reading hands me the per-step recipe. I will need that recipe next.

But here is the part that decides whether this is usable in practice: how fast does the chain settle? A general Gibbs sampler over a high-dimensional posterior can take a very long time to forget its starting point, and if this one did, an iterate-the-table scheme would be hopeless. Let me think about what the chain actually carries from one iteration to the next. When I re-impute `Y_j` at iteration `t`, I condition on the *current values* of the other columns — but I do not condition on `Y_j`'s own previous imputation at all; I threw it away and regenerated it from scratch using the rows where `Y_j` was observed plus the others. So the only way iteration `t-1`'s imputation of `Y_j` influences iteration `t`'s imputation of `Y_j` is *indirectly*, through how it changed the other columns, which then feed back as predictors. There is no direct autocorrelation in `Y_j`'s own imputed values across iterations — they are coupled only through the other variables' relationships. That is a weak coupling, which suggests the dependence on the initial fill washes out in a handful of sweeps rather than hundreds.

I should not just assert that, though — it is the load-bearing claim, so let me actually watch it on a small case I can run in my head and on paper. Take two correlated columns, `Y_1` and `Y_2 = 0.8·Y_1 + noise`, so their correlation is about 0.84, with `n = 200` rows and 20% of each column knocked out (and never both holes in the same row, so each missing entry always has its partner observed). Start from the mean fill: every hole in a column gets that column's observed average. Since the columns are standardized to roughly unit scale, the root-mean-squared error of the mean fill against the held-out truth comes out around 0.93 — about what I'd expect, since I'm replacing each unknown by a constant and eating its full spread. Now run the point sweep: regress `Y_1` on the current `Y_2` over `Y_1`'s observed rows and overwrite `Y_1`'s holes with the fitted values, then do `Y_2` on the updated `Y_1`. After that first sweep the RMSE drops to about 0.56 — the cross-variable information alone nearly halves the error, exactly the gain mean-filling threw away. Then I keep sweeping, and the interesting thing is how little happens next: sweep two moves it to ~0.580, sweep three to ~0.580, and from there it is pinned to the fourth decimal — 0.5799, 0.5799, 0.5799. (The tiny up-tick from 0.56 to 0.58 after the first sweep is the chain relaxing off the lucky-but-inconsistent first pass onto its fixpoint, not divergence.) So the fills stop moving after two or three sweeps, not hundreds. That is the fast-settling behavior the weak-coupling argument predicted, now seen rather than hoped for: a small number of iterations — a handful, a couple dozen at most — is enough, and there is no long burn-in to chase. That is what makes the whole scheme cheap enough to be practical.

Now I owe myself the proper per-column step, because "regress and predict" is the plug-in version, and the Gibbs reading just told me a plug-in is not a proper draw. Take a continuous column `Y_j`, call the response `y` (its observed part, `n1` values) and let `X` be the design matrix of the other variables on those same observed rows, with `q` columns. The proper Bayesian linear regression draw has to produce, for the missing rows, values that reflect *both* the uncertainty in the regression itself and the irreducible residual scatter. The raw least-squares algebra wants `(X'X)^{-1}`, but real tables give me collinearity, small samples, and sometimes empty cells, so I stabilize the inverse immediately. Form `S = X'X`, make `diag(S)` the diagonal matrix with the same diagonal as `S`, and set `V = (S + kappa * diag(S))^{-1}` with `kappa = 0.0001`. Then the fitted center of the posterior draw is `beta_hat = V X'y`, and the residual sum of squares is `SSE = (y - X beta_hat)'(y - X beta_hat)`. Under the standard noninformative prior on the normal linear model, the residual variance draw comes from the scaled inverse-chi-squared form: draw `g ~ chi^2_{n1 - q}` and set `sigma_dot^2 = SSE / g`. The degrees of freedom are `n1 - q`, observed rows minus fitted coefficients; the numerator is the same residual sum of squares, but the chi-squared denominator lets the noise scale fluctuate instead of freezing it. Given that drawn `sigma_dot^2`, the coefficient draw is normal with covariance `sigma_dot^2 V`, so I draw `z1 ~ N(0, I_q)` and set `beta_dot = beta_hat + sigma_dot * chol(V) * z1`. Finally I form the imputations on the missing rows: predict `X_mis beta_dot` and add the residual scatter, `y_imp = X_mis beta_dot + sigma_dot * z2` with `z2 ~ N(0, I_{n0})` over the `n0` missing rows. Every sign and factor is doing work: the ridge enters inside `V`, the residual variance is `SSE/g`, the coefficient jitter is scaled by `sigma_dot`, and the last `sigma_dot * z2` is added to the missing-row prediction rather than to the fitted observed rows.

Let me make sure I see why each of the three random pieces is there, because dropping any of them silently breaks validity. The chi-squared draw of `sigma_dot` says "I do not know the noise level exactly." The normal draw of `beta_dot` says "I do not know the regression coefficients exactly" — that is parameter uncertainty. And the additive `sigma_dot * z2` says "even with the true model, the value scatters around its conditional mean." Now recall what multiple imputation needs from these draws. I will make `m` completed tables, estimate `Q_hat_h` with complete-data variance `U_h` on each completed table, then pool by `Qbar = (1/m) sum_h Q_hat_h`, `Ubar = (1/m) sum_h U_h`, `B = (1/(m-1)) sum_h (Q_hat_h - Qbar)^2`, and `T = Ubar + (1 + 1/m) B`. `B` is the spread of my estimate across the `m` completions, the extra variance that exists because there were missing values.

Let me actually push the `B = 0` claim through the formula rather than wave at it, since it is the whole reason a single best-guess fill is dangerous. Suppose the fill is deterministic — I plug in `beta_hat` and drop both noise draws. Then every one of the `m` completed tables is byte-for-byte identical, so every per-imputation estimate is the same number; call it `q`. With `m = 5` and `Q_hat_h = q` for all `h`, the mean is `Qbar = q`, and `B = (1/(m-1)) sum_h (q - q)^2 = (1/4)(0+0+0+0+0) = 0` exactly. Then `T = Ubar + (1 + 1/m)·0 = Ubar`: the total variance collapses to the within-imputation part, the confidence interval is computed as if there were no missing data at all, and it comes out too narrow — falsely confident tests follow. Contrast a stochastic fill where the five estimates genuinely scatter, say `3.10, 3.20, 3.05, 3.25, 3.12`: now `Qbar = 3.144` and `B = (1/4)·sum (Q_hat_h - 3.144)^2`, which works out to about `0.0064` — a strictly positive between-imputation term that pushes `T` above `Ubar` and widens the interval to acknowledge the missing data. So the formula confirms it: `B` is nonzero only if the `m` imputations genuinely differ, and they differ only if the per-column draw injected randomness. That settles why a deterministic regression fill is invalid for inference. The middle case — adding the residual noise `z2` but keeping `beta_hat` fixed (a stochastic regression with frozen coefficients) — would make the completions differ a bit, so `B > 0`, but it omits the parameter-uncertainty draw; the literature reports that variant's interval coverage at roughly 0.908 instead of the nominal 0.95, under-covering because it pretends to know the regression exactly. I can't reproduce that 0.908 figure on paper here — it needs a coverage simulation — so I'll take it as the reported value and note it's consistent with the direction of my `B` argument: leaving out a source of randomness shrinks `B`, which shrinks `T`, which under-covers. So all three random draws are load-bearing: the chi-squared for the noise level, the normal for the coefficients, the additive residual for the scatter. They are the machinery that makes `B` come out the right size.

The small ridge inside `V` is not a change in the logic of the draw; it is the numerical guard that lets the draw exist on real tables. If a column has many predictors, few observed rows, two nearly duplicate predictors, or a categorical level with an empty cell, the unregularized inverse can explode. Adding `kappa * diag(S)` shrinks only enough to keep the solve stable, and because the same `V` appears in `beta_hat` and in `sigma_dot^2 V`, the point estimate and the coefficient uncertainty stay matched to the same stabilized regression.

So now I have an honest per-column proper draw, and the chained Gibbs sweep over the columns, and the fast-mixing argument that says a dozen sweeps suffice. Let me also notice an alternative for the per-column draw that I will want when the linear-normal model is shaky, because it falls out of the same regression and is more robust. The danger with the parametric draw `X_mis beta_dot + sigma_dot z2` is that it can produce values the variable could never actually take — a negative count, a continuous value for something discrete, a fill outside the observed range if the linear model is misspecified or the variable is skewed. What if, instead of *generating* a value from the normal, I use the regression only to *match*, and donate a real observed value? Compute a predicted mean for every observed case using `beta_hat`, and a predicted mean for each missing case using the drawn `beta_dot`; for a given missing case, find the handful of observed cases (say five) whose predicted means are closest to its predicted mean — the donor pool — pick one at random, and impute *that donor's actually-observed* `y`. This is predictive mean matching. Because every imputed value is a value that genuinely occurred in the data, it automatically respects the variable's bounds, its skew, its discreteness, its heteroscedasticity — none of which I had to model. It is robust to the linear model being wrong, because the linear model is used only to define "similar," not to manufacture the number. The randomness needed for valid `B` is still there: `beta_dot` is a posterior draw (so the matching shifts across the `m` completions), and the random pick among the donors adds the residual-like scatter. The trade is that it cannot extrapolate beyond the observed range and needs enough donors to choose from. So I have two solid per-column engines — the parametric normal draw and the matching draw — both built on the same regression, and both compatible with the chained sweep.

Let me circle back to the incompatibility worry I flagged, because I gave up joint coherence to get the per-column freedom. My conditionals — a linear regression here, a logistic regression there — need not be the conditionals of any single joint distribution. Two linear regressions are jointly compatible (they correspond to a bivariate normal under mild conditions), but the moment I mix a linear and, say, an ordinal model, there may be no joint with both as conditionals, so the Gibbs interpretation strictly speaking no longer points at a well-defined stationary distribution. How much does this actually hurt? The little theory and simulation evidence available suggests the damage is mild in practice — the method "works" across a wide range of cases even when strict compatibility fails. And I can hedge it: the order in which I visit the columns matters when the conditionals are incompatible (a compatible Gibbs sampler is order-invariant; an incompatible one is not), so I should impute in a sensible order rather than arbitrarily. A natural choice is to go from the columns with the *least* missingness to the columns with the most: the least-missing columns are the ones I can estimate most reliably from real observed data, so doing them first means that by the time I reach the heavily-missing columns, their predictors are already as good as they are going to get. That ordering also tends to make the early sweeps informative rather than wasted. So incompatibility is a real theoretical blemish, contained in practice by keeping the per-column models reasonable and the visit order sensible.

Now I want to step back to what this benchmark is actually asking, because it may change the per-column engine I should ship. The task is not to do valid statistical inference with pooled standard errors across `m` imputations — it is to produce *one* completed table and be scored on two things: the root-mean-squared error of the filled values against the held-out truth, and the performance of a downstream model trained on the completed table. RMSE penalizes squared error, so I should ask which single fill `c` minimizes the expected squared error against the (random, from my model's view) true value `Y`. Let me just expand it. Write `mu = E[Y | observed]` for the conditional mean. Then `E[(Y - c)^2] = E[(Y - mu + mu - c)^2] = E[(Y - mu)^2] + 2(mu - c)·E[Y - mu] + (mu - c)^2`. The cross term vanishes because `E[Y - mu] = 0` by definition of `mu`, leaving `E[(Y - c)^2] = Var(Y | observed) + (mu - c)^2`. The first piece does not depend on `c`; the second is a square minimized at `c = mu` where it is zero. So the squared-error-optimal point fill is the conditional mean, and any other choice pays a penalty equal to `(mu - c)^2`.

Let me sanity-check that algebra numerically before I lean on it. Take a "conditional distribution" for one missing value with mean `mu ≈ 0.50` and variance about `1.70`, and sweep `c` around `mu`: at `c = mu - 0.5` the empirical `E[(Y-c)^2]` is `1.947`, at `c = mu - 0.2` it is `1.737`, at `c = mu` it is `1.697`, at `c = mu + 0.2` it is `1.737`, at `c = mu + 0.5` it is `1.947`. It bottoms out exactly at `c = mu`, the minimum equals the variance `1.697`, and each off-center value sits above it by precisely `(mu - c)^2` (e.g. `0.2^2 = 0.04`, and `1.737 - 1.697 = 0.040`). The decomposition holds. The consequence is sharp: a *proper* multiple-imputation draw deliberately adds scatter around `mu` (the `sigma_dot z2` residual and the `beta_dot` jitter) — exactly what makes `B` come out right for inference — but against a fixed held-out truth that scatter is a nonzero `(mu - c)^2` penalty in expectation, so it *raises* RMSE. The very randomness that is mandatory for valid inference is counterproductive for reconstruction. So for this objective I want the *point* version of the chained scheme: at each column, regress and impute the predicted mean, with no residual noise and no parameter draw. The iteration over columns stays — that is what handles the multivariate circularity and borrows cross-variable strength — but each step becomes a deterministic conditional-mean regression rather than a stochastic posterior draw. It is the same algorithm with the randomness switched off, and the squared-error decomposition is what tells me switching it off is correct for reconstruction accuracy and downstream prediction rather than calibrated uncertainty.

That decision also fixes my per-column regressor, and I want it to be the self-tuning kind so I never have to hand-pick a ridge per column across heterogeneous tables. A plain least-squares fit will be unstable on the collinear / small-`n` columns; a fixed ridge needs a penalty I would have to guess. The clean answer is a Bayesian linear regression that sets its own regularization by maximizing the evidence — MacKay's framework as implemented in Bayesian ridge regression. Center `X` and `y` when fitting the intercept, keep that intercept outside the probabilistic coefficient prior, put a Gaussian prior on the centered coefficients with precision `lambda`, and give the noise precision `alpha`. The posterior mean coefficients solve `coef = (lambda/alpha * I + X'X)^{-1} X' y`, so the ridge strength is the ratio `lambda/alpha`. The precisions are estimated from the data by iterating the evidence updates: over the eigenvalues `d_i` of `X'X`, compute `gamma = sum_i (alpha * d_i) / (lambda + alpha * d_i)`, then with sklearn's tiny Gamma hyperpriors use `lambda = (gamma + 2 * lambda_1) / (sum coef^2 + 2 * lambda_2)` and `alpha = (n - gamma + 2 * alpha_1) / (SSE + 2 * alpha_2)`. With the default `1e-6` hyperprior constants this is essentially the familiar `gamma / sum coef^2` and `(n - gamma) / SSE`, but the code path is the hyperprior version. So the penalty `lambda/alpha` is chosen automatically from the data: a collinear or small-`n` column gets shrunk more, a well-determined one less, with no manual tuning. For the prediction I want, I just take this regressor's posterior-mean prediction — the conditional mean — which is exactly the point fill that minimizes squared error. The same fitted model can report a predictive standard deviation; if I want sklearn's stochastic multiple-imputation branch, it samples a truncated normal from that predictive mean and standard deviation. For this objective I set `sample_posterior=False` and take the mean.

Let me assemble the loop concretely, in the form I will actually run. Initialize the table: fill every column's `NaN`s with that column's observed mean — any complete starting table will do, and the mean is the cheapest sensible one; the first sweep overwrites it anyway. Decide a visit order: ascending in fraction-missing, so the least-incomplete columns are imputed first. Then for a fixed number of rounds, sweep the columns in that order; at each column with missing entries, fit the self-tuning Bayesian linear regression on the rows where that column was observed (predictors = all the other columns at their current fill), predict the rows where it is missing, and write those predictions back, so the next column in the sweep sees the update immediately. After each full sweep, check how much the imputed entries changed from the previous sweep; if the largest change (relative to the scale of the observed data) drops below a small tolerance, stop early — the fixpoint has effectively been reached and further sweeps will not move the fills meaningfully, and chasing exact convergence rarely improves the downstream score anyway. Otherwise stop at the round cap. At the end, restore the originally-observed entries (they should never have been altered) and return the completed table.

A couple of the constants deserve their reasons stated. The round cap: I argued the chain mixes fast because a column's imputation re-enters only through the other variables, so a few-dozen sweeps is comfortably enough; I will cap at a few tens and lean on the early-stop to finish sooner when the fills settle. The convergence tolerance is measured as the infinity-norm of the change in the table between successive sweeps, divided by the magnitude of the observed data, compared against a small number like `1e-3` — a scale-free "the fills have stopped moving" test. The mean initialization is just a starting complete table. The ascending visit order is the reliability-ordering argument above. None of these is doing the real work; the real work is the chained conditional-mean regression with a self-regularizing per-column model.

So let me write it, filling the single empty slot in the transformer — the rule that turns a `NaN`-laden table into a finite, relationship-respecting completed one — with the chained-regression scheme, using the well-tested iterative-imputer machinery so I am not re-implementing the round-robin and convergence bookkeeping by hand:

```python
import numpy as np
from sklearn.base import BaseEstimator, TransformerMixin


class CustomImputer(BaseEstimator, TransformerMixin):
    """Chained-equations imputation: model each incomplete column as a
    regression on all the others, in a round-robin, iterated to a fixpoint.

    Per-column engine is a self-regularizing Bayesian linear regression whose
    ridge penalty is set by evidence maximization, so collinear / small-n
    columns don't blow up. We impute the conditional mean (the point version),
    which is what minimizes squared-error reconstruction.
    """

    def __init__(self, random_state=42, max_iter=30):
        self.random_state = random_state
        self.max_iter = max_iter

    def _build(self):
        # enable the (formerly experimental) iterative imputer, then build it
        from sklearn.experimental import enable_iterative_imputer  # noqa: F401
        from sklearn.impute import IterativeImputer
        from sklearn.linear_model import BayesianRidge
        return IterativeImputer(
            estimator=BayesianRidge(),       # self-tuning ridge per column (MacKay evidence)
            sample_posterior=False,           # point fills: posterior-mean prediction, then clipping
            max_iter=self.max_iter,          # cap on round-robin sweeps; early-stops sooner
            random_state=self.random_state,
            imputation_order="ascending",    # least-missing columns first (most reliable predictors)
            initial_strategy="mean",         # start from a complete table (column means)
            tol=1e-3,                        # stop when the fills stop moving (scaled inf-norm)
        )

    def fit(self, X, y=None):
        # X: (n_samples, n_features) with NaN for missing. No labels used.
        self._imputer = self._build()
        self._imputer.fit(X)
        return self

    def transform(self, X):
        # Round-robin regression fills every NaN; observed entries are restored.
        return self._imputer.transform(X)

    def fit_transform(self, X, y=None):
        self._imputer = self._build()
        return self._imputer.fit_transform(X)
```
