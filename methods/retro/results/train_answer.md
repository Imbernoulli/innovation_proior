Autoregressive language modeling factorizes text as $p(x_1,\dots,x_n)=\prod_i p(x_i\mid x_{<i})$, and the dominant recipe for making a neural LM better has been brute scale: more parameters, more data, more compute. That works, and scaling laws make it predictable, but the single parameter count is doing two unrelated jobs. Part of the gain from scale is more *computation* — more layers, more attention, more nonlinear mixing per token, genuine reasoning capacity. The other part is more *memorization* — the larger network simply stores more facts and phrasings from the training set in its weights, which is why large LMs memorize hard enough to reproduce training chunks verbatim. These two are fused in the parameter count, and the giant model pays the full compute cost on every forward pass regardless of which job a given prediction actually needs. The question I want to settle is whether they can be decoupled: can the memorization benefit come from somewhere cheap — an external text database I look things up in at prediction time — so that parameters are spent only on computation? If so, a semi-parametric LM could match a far larger purely-parametric one at much less compute, and its knowledge would be inspectable (I can read what was retrieved) and updatable (swap the database) instead of frozen in weights. Two constraints make this hard at the scale that matters: the database has to be allowed to grow to *trillions* of tokens — the regime where retrieval could substitute for tens of billions of parameters, far beyond the few-billion-token stores prior work used — and incorporating the retrieved text must cost time *linear* in the amount retrieved, since a quadratic cost would just move the expense rather than remove it.

The existing retrieval-augmented LMs each fail one of those constraints, and they fail in instructive, opposite ways. The frozen-retriever interpolation methods — continuous cache and kNN-LM — retrieve, at inference, *tokens* whose stored activation resembles the current one, build a distribution from those tokens, and *interpolate* it with the LM's own next-token distribution. Because the retriever is frozen, these scale, and they bolt onto any model with no retraining; kNN-LM even pushed the store to all of Wikipedia. But the network *never reasons over the retrieved text* — it only mixes two output probability vectors at the very end, so a retrieved passage can nudge the final distribution but cannot enter the model's internal computation, be combined with the context, or be reasoned about. And it retrieves at the granularity of individual tokens, which is storage-heavy and throws away passage structure. The other family — the end-to-end-trained dense retrievers, DPR, REALM, RAG, FiD — *do* let the model reason over retrieved text, and dominate QA. But REALM-style training has to search the database during training and *periodically re-embed and re-index the whole corpus* as the retriever's parameters drift, which is exactly what caps the database at a few billion tokens; you cannot re-index a trillion-token store every few hundred steps. And that whole line is built for QA: one question, one retrieval on the prompt, a short answer — none of it is designed to model *arbitrary long text*, where the beginning of a 2000-token sequence might need one set of documents and its end a completely different set. So frozen retrieval scales but cannot reason; trainable retrieval reasons but cannot scale, and neither retrieves repeatedly for different parts of the same sequence.

I propose RETRO, a Retrieval-Enhanced Transformer that gets both at once by splitting the system at exactly the right seam: keep the *retriever frozen* and make the *fusion trainable*. The retriever is a pre-trained BERT used only to embed a span of text into a fixed key — mean-pooled over time — and never trained, so the keys never drift, the database can be embedded once and grown to trillions of tokens, and (crucially) the nearest neighbours of every chunk are fixed and can be precomputed once and saved alongside the data, which is what makes on-the-fly retrieval during training unnecessary. The reasoning then happens *downstream* of retrieval, in a trainable encoder plus cross-attention that lets the parametric network actually combine the retrieved text with its context. That single split is the whole resolution: scale comes from the frozen, precomputable retriever; reasoning comes from the trainable fusion.

Retrieval granularity is the next decision. Per-token retrieval, as in kNN-LM, would mean a trillion keys and a trillion lookups and would discard the passage structure I want to reason over. So I retrieve at the level of contiguous *chunks*: split each $n=2048$-token sequence into $l$ chunks of size $m=64$ (so $l=32$), and retrieve the $k$ nearest neighbours per chunk. This cuts the number of lookups per sequence from $n$ to $l$, a factor of $m$, and each retrieved unit is a coherent span. The choice $m=64$ balances two pressures — small enough that a chunk is topically coherent so its single BERT embedding is a meaningful query, large enough that $l$ stays modest. The database is a key→value store. For each chunk $N$ in the corpus the key is its frozen embedding $\mathrm{BERT}(N)$, and the value is *two* contiguous spans, $[N,F]$: the matched chunk $N$ itself and $F$, its *continuation* in the source document. Storing the continuation is deliberate — the matched chunk $N$ is what is *similar* to my current chunk, giving alignment and lexical evidence, but what I actually want to predict is what comes *next*, and $F$ is a concrete example of "text that followed something like my current chunk." Keeping both only fixes the value length at two chunks; with $N$ and $F$ each of length $64$, a retrieval $\mathrm{Ret}(C)$ has shape $k\times r$ with $r=128$. To retrieve for a query chunk $C$ I embed it the same way and take approximate nearest neighbours by L2 distance $d(C,N)=\lVert \mathrm{BERT}(C)-\mathrm{BERT}(N)\rVert^2$ through an approximate-nearest-neighbour index (SCaNN), which returns top-$k$ in $O(\log T)$ for a $T$-element store — about $10$ms even over a two-trillion-token database, amortized over a $64$-token chunk and so negligible. One hazard of a frozen retriever from the training corpus is that a neighbour could come from the *same document* as the training sequence, so $\mathrm{Ret}(C_u)$ might literally contain $C_{u+1}$; I filter out same-document neighbours to block that leak.

The objective has to keep the whole thing causal, and the precise conditioning is where the design lives. The retrieval-enhanced log-likelihood is
$$L(X\mid\theta,D)=\sum_{u=1}^{l}\sum_{i=1}^{m}\ell_\theta\!\left(x_{(u-1)m+i}\,\middle|\,(x_j)_{j<(u-1)m+i},\,(\mathrm{Ret}(C_{u'}))_{u'<u}\right).$$
The condition on retrieval is $u'<u$ — strictly *earlier* chunks. A token in chunk $u$ may depend on all earlier tokens (ordinary autoregression) and on the neighbours retrieved for chunks strictly before it; for the first chunk that previous-neighbour set is empty. The reason for this exact condition is the trap I must avoid: $\mathrm{Ret}(C_u)$ is retrieved using $C_u$ itself and includes continuations $F$ that are "what tends to follow text like $C_u$," so if tokens *inside* $C_u$ were allowed to attend to $\mathrm{Ret}(C_u)$ the model would leak information about $C_u$'s own content and likely continuation into the prediction of $C_u$'s own tokens — it could cheat, and sampling would be inconsistent because I would need $C_u$'s neighbours before having generated $C_u$. So $\mathrm{Ret}(C_u)$ is computed after $C_u$ is known and is first useful for predicting $C_{u+1}$. That keeps the model autoregressive and samplable: to generate $C_{u+1}$, retrieve on the already-generated $C_u$ and condition; the added retrieval cost during sampling is linear in the number of chunks, dwarfed by the usual quadratic token-sampling cost.

To fold the neighbours into the decoder I use encoder-decoder cross-attention, the standard differentiable way to let a decoder attend over encoded auxiliary content. A small bidirectional Transformer encoder turns the retrieved neighbours into representations $E$, and the main LM decoder attends over $E$. I do not put cross-attention in every layer — that would pay the retrieval path at every depth — but interleave a retrieval block among standard LM blocks, one every three layers starting from layer $6$. A retrieval block is $\mathrm{Retro}(H,E)=\mathrm{FFW}(\mathrm{CCA}(\mathrm{Attn}(H),E))$ and a plain block is $\mathrm{LM}(H)=\mathrm{FFW}(\mathrm{Attn}(H))$, where $\mathrm{CCA}$ is the chunked cross-attention operator. As long as $\mathrm{Attn}$ is causal, $\mathrm{FFW}$ positionwise, and $\mathrm{CCA}$ respects causality, any stack of these plus a readout head realizes exactly the likelihood above. One subtlety in the encoder is easy to miss: I want the neighbour encoding to be *aware of the chunk that retrieved it*, so the encoder emphasizes what is relevant to the current context rather than blindly summarizing a passage. So the neighbour encoder is *conditioned on $H_u$*, the decoder's activations for chunk $C_u$, via cross-attention inside the encoder; the retrieval representation is thereby differentiably modulated by the retrieving chunk, and the LM-loss gradient flows back through the encoder's use of $H_u$ even though the retriever that *selected* the neighbours is frozen. All $l\cdot k$ neighbours are encoded in parallel into $E\in\mathbb{R}^{l\times k\times r\times d'}$, with $E_u$ the encoded neighbours for chunk $u$.

Chunked cross-attention is where causality could break, and getting the alignment right is the load-bearing step. Chunk $C_u$ occupies positions $(u-1)m+1,\dots,um$, and its neighbours encode information about the *whole* of $C_u$, including its last token at position $um$. So predicting the token at position $um$ — the last token of $C_u$ — must not use $\mathrm{Ret}(C_u)$, or that would be leakage; the first prediction that may legitimately use $\mathrm{Ret}(C_u)$ is the *next* token, at position $um+1$, the first token of $C_{u+1}$, and it stays legitimate for all of $C_{u+1}$. I implement this by defining *attending chunks* shifted by one token relative to the retrieval chunks: the attending chunk for $E_u$ is $H_u^+=(h_{um+i-1})_{i\in[1,m]}$, the activations at positions $um,\,um+1,\dots,um+m-1$. The resolution of the apparent boundary conflict is the standard one for autoregressive cross-attention: the activation at position $um$, after attending to $E_u$, *feeds the prediction of position $um+1$*, which is allowed since $u<u+1$ — generally the cross-attention output at position $um+i-1$ is used to predict position $um+i$, the $i$-th token of $C_{u+1}$. The one-token shift is exactly what makes "the last token of $C_u$ is the first position to access $E_u$" consistent with "no token uses its own chunk's neighbours." There are $l-1$ attending chunks, $u$ from $1$ to $l-1$, and
$$\mathrm{CCA}(H,E)_{um+i-1}=\mathrm{CrossAttention}(h_{um+i-1},E_u),\qquad i\in[1,m].$$
The first $m-1$ positions (inside $C_1$) have no previous chunk's neighbours to attend to, so there $\mathrm{CCA}$ is the identity, $\mathrm{CCA}(H,E)_j=h_j$ for $j\in[1,m-1]$; in a residual implementation those positions are simply left unchanged. Inside each chunk's cross-attention I merge the neighbour and time dimensions of $E_u$ so the $k$ neighbours of $r$ tokens become one attended sequence of length $k\cdot r$, and the decoder token attends across time and across neighbours at once. The single-head cross-attention operator, with parameters $K,Q\in\mathbb{R}^{d\times c}$ and $V\in\mathbb{R}^{d\times d}$, for a query $h\in\mathbb{R}^d$ over $Y\in\mathbb{R}^{T\times d}$, is
$$\mathrm{CrossAttention}(h,Y)=\mathrm{softmax}(Y K Q^\top h)\,Y V,$$
softmax over the $T$ attended positions, multi-head in practice. Because a data chunk and its neighbours are roughly positionally aligned — both "starting at the same place" — I use *relative* rather than absolute positional encodings in this cross-attention, with the distance between data token $i$ of $C_u^+$ and retrieval token $i'$ given by $d(i,i')=i-i'+l-1$, turned into positional logits added to the content logits, so the model can learn to attend to the neighbour position aligned with its own.

This meets the two constraints I started from. The cost is *linear*: each decoder token attends only to its single preceding chunk's $k\cdot r$ retrieved tokens — a fixed amount independent of sequence length — so the total cross-attention cost is $O(n\cdot k\cdot r)$, which means I can crank up $k$ and the database size at evaluation without compute exploding. And causality does not cost long-range retrieval: each $\mathrm{CCA}$ touches only $E_{u-1}$, but the *self-attention* layers propagate information forward across chunks, so the activations of a token in chunk $u$ depend on *all* prior retrievals $\mathrm{Ret}(C_{u'})_{u'<u}$ even though no single cross-attention reaches more than one chunk's neighbours. The dependence on everything retrieved so far comes for free through self-attention while cross-attention pays for only one chunk's worth — the trick that reconciles linear cost with conditioning on all prior retrievals. The padding follows the alignment exactly: take the decoder activations of shape $(n,d)$, drop the first $m-1$ positions, pad the tail and reshape to $(l,m,d)$ to form the shifted attending chunks; cross-attend each against its $k$ neighbours of length $r$, merging into $k\cdot r$ positions, adding relative-position logits, softmax, weighted sum of values; reshape back to $(n,d)$ and pad the front with $m-1$ zeros to restore alignment, truncating to length $n$, so the residual update leaves the first $m-1$ positions unchanged as required.

The rest is a standard decoder-only Transformer with two minimal modernizations, RMSNorm instead of LayerNorm and relative (Transformer-XL style) positions instead of absolute. The neighbour encoder is small and shared across all neighbours and chunks — $d'=896$, a couple of layers, about $+19$M parameters — and its relative cost shrinks as the base model grows (roughly $+30\%$ params at $132$M but only $+8\%$ at $7$B). Because the whole setup retrieves directly from the training data, evaluation must be leakage-aware: a retrieval LM can cheat by copy-pasting a leaked training chunk to "predict" a leaked evaluation chunk. So in addition to deduplicating the training set (dropping documents with $\geq0.8$ $13$-gram MinHash-Jaccard overlap to any validation or test document), I report *filtered* bits-per-byte — for each evaluation chunk I find its nearest training chunks, measure the maximal contiguous token overlap $r(C)\in[0,1]$, and report metrics restricted to chunks with overlap below a threshold $\alpha$, separating genuine generalization from memorized copying. Finally the retrieval pathway is general: it can inject any source, not just the pretraining corpus. For open-domain QA I fine-tune the pretrained retrieval model on DPR's retrieved Wikipedia passages as neighbours, formatting the input as "question: {q} \n answer: {a}", left-padding so "answer:" ends the first $64$-token chunk to align the answer with the first retrieving position, and feeding the top-$20$ DPR passages through chunked cross-attention — the same architecture doing QA by swapping what is in the database.

```python
import jax
import jax.numpy as jnp

Q = jnp.zeros((d, d))
K = jnp.zeros((d, d))
V = jnp.zeros((d, d))

def cross_attention(chunk, neighbour):
    return (chunk @ Q) @ (neighbour @ K).T, neighbour @ V

def multi_neighbour_cross_attention(chunk, neighbours):
    logits, values = jnp.vectorize(cross_attention,
        signature="(m,d),(r,d)->(m,r),(r,d)")(chunk, neighbours)
    logits += relative_positional_encodings(m, r)[None, :, :]
    logits = jnp.moveaxis(logits, 0, -1).reshape((m, r * k))
    values = jnp.moveaxis(values, 0, 1).reshape((r * k, d))
    return jax.nn.softmax(logits) @ values

def chunked_cross_attention_update(observation, neighbours):
    attending = jnp.pad(observation[m - 1:],
                        ((0, m - 1), (0, 0)),
                        mode="constant").reshape(l, m, d)
    out = jnp.vectorize(multi_neighbour_cross_attention,
                        signature="(m,d),(k,r,d)->(m,d)")(attending, neighbours)
    return jnp.pad(out.reshape(n, d),
                   ((m - 1, 0), (0, 0)),
                   mode="constant")[:n]

def encode_neighbours(retrieved_tokens, H):
    H_chunks = H.reshape(l, m, H.shape[-1])
    E = embed_enc(retrieved_tokens)
    for p_enc in range(1, L_enc + 1):
        E = bidirectional_attn(E)
        if p_enc in P_enc:
            E = encoder_cross_attn(E, H_chunks)
        E = encoder_ffw(E)
    return E

def forward(X, retrieved_tokens):
    H = embed(X)
    E = None
    for p in range(1, L + 1):
        H = causal_attn(H)
        if p == min(P):
            E = encode_neighbours(retrieved_tokens, H)
        if p in P:
            H = H + chunked_cross_attention_update(H, E)
        H = ffw(H)
    return readout(H)
```
