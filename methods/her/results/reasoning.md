OK, let me think this through from scratch. I have an agent and a task specified by a goal — get the box to this spot, get the bits into this pattern — and I want to specify the task as honestly as I can: a reward that is `-1` on every step until the goal is reached and `0` once it is. `r(s,a) = -[s' is not at the goal]`. That's the thing I actually care about, did the task get done, and it costs me nothing to write down. No distance metric, no weighting of terms, no domain knowledge. Beautiful in principle.

In practice it's a disaster, and I want to be precise about *why*, because the precise reason is going to tell me what to fix. The agent starts exploring, more or less at random because it hasn't learned anything yet. In a state space of any real size, the probability that a random walk lands exactly on the goal is essentially zero. So across an entire episode, every single reward it sees is `-1`. Across a thousand episodes, every reward is `-1`. Now think about what the value learner does with that. The Bellman target is `y_t = r_t + gamma * max_{a'} Q(s_{t+1}, a')`. If `r_t` is `-1` everywhere and `Q` is initialized to roughly a constant, then `y_t` is the same constant-ish number for every transition. The regression target carries no information that distinguishes a good action from a bad one. There is no gradient that points toward the goal, because nothing in the data has ever *been* near the goal in the sense of receiving a different reward. The agent cannot learn the task because it has never once observed success. It's not that it learned the wrong thing — it's that there is nothing to bootstrap from.

So my first instinct: this is an exploration problem, right? The agent isn't reaching the goal, so push it to explore harder. Add an intrinsic curiosity bonus, count-based novelty `alpha/sqrt(N(s))`, or an information-gain exploration term, or train an ensemble of value functions and act optimistically. Make it go to new places. Let me actually think about whether that helps. Those methods make the agent *visit more distinct states*. But stare at the bit-flipping toy for a second: `S = {0,1}^n`, each action flips one bit, reward `-1` unless the bit-string equals the target. For `n = 50` there are `2^50` states. Even if my exploration bonus drove me to visit a fabulous diversity of states, the chance I arrange all 50 bits into the *exact* target pattern is still one in `2^50`. The problem was never that I'm revisiting the same boring states. The problem is that the *reward* is silent — even after I visit a brand new state, the environment tells me nothing about whether that state was progress. State coverage and reward signal are different things, and exploration bonuses fix the first one. So this is a wall. Better exploration doesn't touch the actual difficulty.

Fine, the standard answer. Shape the reward. Instead of binary, give a dense `-||s_object - g||^2`, a smooth thing that gets bigger as I get closer. Now every step has an informative gradient and the toy is trivially solvable. Why don't I just do this and go home? Three reasons, and each one bothers me more than the last. First, I had to *invent* that distance function. For pushing a box on a table it's a Euclidean distance; for stacking bricks, existing shaped rewards use five separate terms that someone hand-tuned the weights on. That's domain knowledge and RL expertise re-spent on every new task, which is exactly the thing I was trying to escape by writing a binary reward in the first place. Second — and this one is subtle and I've actually watched it happen — a shaped reward *punishes* moving the wrong way. If the agent can't yet manipulate the object precisely, the dense reward makes "touch the object and accidentally push it slightly away" look worse than "don't touch the object at all," so the agent learns a degenerate policy where it never engages with the object. The shaping actively suppresses the exploration I need. Third, and most fundamental: the shaped reward is a *proxy*. What I care about is binary success. Optimizing `-||·||^2` is optimizing a surrogate, and the optimum of the surrogate need not be the policy that maximizes success. Shaping is a compromise on the metric I actually want. So shaping "works" but it's a retreat from the whole premise. I want to keep the sparse binary reward and *still* learn.

So now I'm stuck between two walls. Sparse reward gives me no signal; shaped reward betrays the premise and misbehaves. Let me put the algorithm aside entirely and think about what's actually happening in one of these failed episodes, concretely, like a human watching it.

I'm learning to shoot a puck into a net on the left. I take a shot. It misses the net and ends up over on the right. A standard learner concludes: that sequence of actions did not score, reward `-1`, learn almost nothing. And that conclusion is correct, *given the goal was the left net*. But watch what a person does. A person says: huh, well, that would have been a perfect shot if the net had been on the right. The trajectory failed at the task I set, but it *succeeded* at a different task — the task of putting the puck where it actually ended up. The episode that "failed" is a flawless demonstration of reaching its own endpoint.

Let me stay with that, because it feels like the crack in the wall. The trajectory is `s_0, s_1, ..., s_T` and the goal `g` was not satisfied at any of them, so I got `-1` the whole way. But the agent *did* reach `s_T`. It definitely, reproducibly, by these exact actions, got the system into state `s_T`. What if I just... pretend that was the goal all along? Relabel the episode as if `g' = s_T` had been the target. Then I replay the same transitions, but I recompute the reward against `g'`. And now the last transition lands on `g'` by construction — the reward there is `0`, not `-1`. Suddenly this episode contains a transition where the reward is *different*, where success was observed. The constant-`-1` curse is broken, not by changing the environment, not by shaping, but by changing the *question I ask of the data I already have*. Every episode, no matter how badly it failed at the real goal, is a successful episode for *some* goal — the goal it happened to achieve. So every episode can yield a positive, informative learning signal.

Now I have to check: is this even a legal thing to do? Can I take a transition that was collected while pursuing goal `g` and feed it to my value function as if the goal had been `g'`? This only makes sense if my value function takes the goal as an input in the first place. And there is exactly the right tool for that: a goal-conditioned value function, the universal value function idea. Instead of `Q(s,a)` I learn `Q(s, a, g)` — one network, state and goal both as inputs. It's trained by the same Bellman bootstrap, `Q*(s,a,g) = E_{s'}[ r_g(s,a) + gamma * max_{a'} Q*(s', a', g) ]`, and because a single network shares parameters across all goals, the value estimate *generalizes* across goals — it can give a sensible value for a goal it was never explicitly trained on. That's exactly what I need: a transition is well-defined input under *any* goal I choose to pair it with, not only the goal that generated it. The actor and critic become `pi(s, g)` and `Q(s, a, g)` — concatenate the goal onto the state and feed it in. Good. The relabeling has somewhere to live.

Second legality check, and this one is the real constraint. When I relabel a transition's goal from `g` to `g'`, I am changing the reward, and in effect I am changing which policy the data is "about" — the data was generated by a behavioral policy conditioned on `g`, but I'm now using it to learn the value of reaching `g'`. That is precisely an *off-policy* update: learning the value of one thing from data generated while doing another. Is that allowed? Yes — but only for off-policy algorithms, the ones whose Bellman target uses `max_{a'} Q(s', a', g)` or a separate target policy `pi'(s', g)` rather than the action that was actually taken under the old goal. The target doesn't care which policy generated the transition; it only needs the tuple `(s, a, r_{g'}, s')` to be a valid one-step sample of the environment dynamics under goal `g'`. And it is — the dynamics `p(s'|s,a)` don't depend on the goal at all, only the reward does. So I can relabel freely as long as I recompute the reward consistently and as long as my learner is off-policy: a value-based method with a max, or a deterministic actor-critic with a target policy. On-policy methods, which need the data to come from the current policy, are out. That's a clean, sharp condition.

Let me write the multi-goal setup down properly so I'm not hand-waving. There's a goal space `G`. Each goal `g` comes with a predicate `f_g: S -> {0,1}` — does this state satisfy goal `g`? The reward is `r_g(s, a) = -[ f_g(s') = 0 ]`, minus-one until the predicate is satisfied. In the simplest case the goal *is* a desired state, `G = S` and `f_g(s) = [s = g]`. More usefully, a goal can pin down only part of the state — for an object-position task, `G = R^3` and `f_g(s) = [ ||g - s_object|| <= eps ]`, the goal only constrains where the object is, within a tolerance. The one assumption I need for relabeling is a map `m: S -> G` such that `f_{m(s)}(s) = 1` for every state — given any state I reached, I can name a goal that this state satisfies. That's barely an assumption: for `G = S` it's the identity; for the object-position case `m(s) = s_object`, just read off the object's position. Every state I land in is, via `m`, a goal that I demonstrably achieved. So the relabel goal for an episode ending in `s_T` is `m(s_T)`.

Now, *which* states do I relabel with? The endpoint `m(s_T)` is the obvious one — call that the `final` strategy, replay each trajectory with the goal it ended at. Let me make sure I see why even this minimal version is powerful before I get fancy. Take an episode `s_0, ..., s_T`. Store every transition twice. Once with the real goal `g` and its real reward `r(s_t, a_t, g)` — that's just ordinary experience replay, and I keep it because at test time the goal really is `g` and I mustn't forget that distribution. And once relabeled: for each transition store `(s_t, g', a_t, r'=r(s_t, a_t, g'), s_{t+1}, g')` with `g' = m(s_T)`. The relabeled trajectory copy is guaranteed to contain at least one rewarding transition — the last one — so half of the replayed trajectory copies now include a non-constant reward signal. The value function finally has something to chew on: it learns "from `s_{T-1}`, this action reaches `g' = m(s_T)`," and because `Q(s,a,g)` generalizes across goals, that knowledge bleeds into estimates for nearby goals, including, eventually, the real ones. Learning becomes possible.

Can I do better than `final`? The endpoint is just one achieved goal; the trajectory passed through `T` of them. The most natural generalization: when I'm replaying transition `t`, relabel it with some state the agent reached *later in the same episode* — pick a random index after `t` and use `m(s_future)` as the goal. Call that `future`. Why would `future` beat `final`? Think about what a relabel goal teaches. If I relabel transition `t` with a goal that the agent reached only a few steps after `t`, I'm giving the value function the signal "from here, this action sequence gets you to that nearby place" — local, near-horizon credit, which is exactly the granular information the bootstrap needs to chain together. The endpoint `s_T` might be many steps and a lot of irrelevant motion away from transition `t`; a near-future achieved state is a tighter, more directly attributable target. `future` gives the learner a whole spectrum of horizons — short, medium, up to the end — instead of only the longest one. And there's a second, lovelier reason it helps, which I notice when I ask where the relabel goals come from: they come from states the agent *actually reaches*. Early in training, with a clumsy policy, the achieved states cluster near the start — so the relabel goals are easy, reach-from-here goals, and the agent learns those. As the policy improves, the achieved states spread out toward the real targets, so the relabel goals automatically get harder. The relabeling generates its own curriculum, and I never had to design it, never had to control the initial-state distribution, never had to hand-order tasks easy-to-hard. The curriculum falls out of "relabel with what you actually achieved" for free.

There are two other sampling choices I should keep in the search space. `episode`: relabel with `k` random achieved states from the same episode, not necessarily after `t`. `random`: relabel with `k` random achieved states from anywhere in training so far. The common knob is `k`, how many relabeled goals I generate per real transition — equivalently the ratio of relabeled data to ordinary data in the buffer. I expect `future` to win because of the near-horizon-credit argument: `episode` and `random` will hand the learner relabel goals that have nothing to do with the transition being replayed (a goal from before `t`, or from a totally different episode), which is much weaker supervision; `future` is the only one that says "goal reachable from *here*, soon."

But `k` can't just be cranked up. If I make `k` huge, almost every sampled update is relabeled data, and the fraction of updates still carrying the *real* goal `g` shrinks toward nothing. That matters because at test time I'm evaluated on the real goal distribution, and if the learner barely sees that distribution, the value function's estimates there get starved and the policy drifts away from the objective I actually care about. So there's a tension: more relabeling means more reward signal but a more distorted goal distribution. I want a modest `k`, enough hindsight to break the sparse-reward curse, not so much that the real objective is drowned out; `k = 4` is a natural starting point because it makes most updates informative while still leaving one original-goal update for every four relabeled ones.

Let me convert `k` into the actual sampling knob I'll implement. I want a fraction `future_p` of sampled transitions to be relabeled with a future goal and the rest left with the original goal, such that the relabeled-to-original ratio is `k : 1`. If `future_p` is the relabeled fraction, then relabeled-to-original is `future_p : (1 - future_p) = k : 1`, so `future_p = k / (1 + k) = 1 - 1/(1+k)`. For `k = 4` that's `future_p = 0.8`. Clean.

One subtlety about *when* to do the relabeling. I don't want to bloat the buffer by physically storing `k+1` copies of every transition. Better: store each episode once, as full trajectories `{o, ag, g, u}` where `ag` is the achieved-goal sequence `m(s_t)` computed per step. Then do the relabeling *lazily, at sampling time* — when I draw a minibatch, for each sampled transition I decide with probability `future_p` whether to swap its goal for a future achieved goal from its own episode, and I recompute its reward on the spot. Same statistical effect, far less memory, and it lets me change strategies without re-collecting data.

Now the off-policy learner underneath. For continuous control I'll use a deterministic actor-critic. Critic `Q(s, a, g)`, actor `pi(s, g)`, target copies of both. The critic's Bellman target for a (possibly relabeled) transition `(s, a, r, s', g)` is `y = r + gamma * Q'(s', pi'(s', g), g)` — note `g` is held fixed across the one-step backup; I relabel the goal *and* recompute `r` against that same `g`, consistently, and then the target uses that `g` throughout. The critic regresses `Q(s,a,g)` onto `y` with `y` treated as constant; the actor maximizes the critic, `L_actor = -E[ Q(s, pi(s,g), g) ]`, gradient flowing through the critic into the actor. Targets tracked by polyak averaging. Standard, except every input is goal-conditioned and the rewards come from the relabeling.

A couple of details I shouldn't gloss over because they bite in practice. The reward lives in `{-1, 0}`, so the return is bounded: with discount `gamma`, the most negative possible value is `-1 - gamma - gamma^2 - ... = -1/(1-gamma)` and the best is `0`. The critic's regression target can transiently overshoot these bounds and destabilize, so I clip the critic target to `[-1/(1-gamma), 0]` — the actual range of achievable values. With `gamma = 0.98` that's `[-50, 0]`. Cheap and it keeps the critic honest. And the actor: I squash its output through a `tanh` and rescale to the control range, but `tanh` saturates and kills gradients if the preactivations grow large, so I add the square of the preactivations to the actor's loss as a mild penalty keeping it in the linear regime. Inputs to the networks come at wildly different magnitudes (joint angles vs. velocities vs. positions), so I normalize observations to zero mean and unit variance using running statistics and clip to `[-5, 5]`. For exploration the behavioral policy is the actor with, say, 20% chance of a uniform random action and otherwise additive Gaussian noise at a few percent of the action range — I need real exploration on top of the relabeling, because the relabeling only helps me learn from what I *did* reach; it can't make me reach somewhere new.

Let me now write the thing that makes all of this go — the sampler that turns stored episodes into a training minibatch, with the relabeling baked in. This is where `future_p`, `m`, and the reward recomputation all come together.

```python
import numpy as np

def make_sample_her_transitions(replay_strategy, replay_k, reward_fun):
    # k controls hindsight-to-real ratio; future_p is the relabeled fraction.
    # relabeled:real = future_p:(1-future_p) = k:1  =>  future_p = 1 - 1/(1+k)
    if replay_strategy == 'future':
        future_p = 1 - (1. / (1 + replay_k))
    else:                       # 'none' = ordinary replay, no relabeling
        future_p = 0

    def _sample_her_transitions(episode_batch, batch_size_in_transitions):
        # u/g have T action steps; o/ag have T+1 states, so future_t may hit T.
        # 'ag' is the achieved-goal sequence m(s_t).
        T = episode_batch['u'].shape[1]
        rollout_batch_size = episode_batch['u'].shape[0]
        batch_size = batch_size_in_transitions

        # pick a random (episode, timestep) per sampled transition
        episode_idxs = np.random.randint(0, rollout_batch_size, batch_size)
        t_samples = np.random.randint(T, size=batch_size)
        transitions = {key: episode_batch[key][episode_idxs, t_samples].copy()
                       for key in episode_batch.keys()}

        # choose which transitions get a relabeled goal
        her_indexes = np.where(np.random.uniform(size=batch_size) < future_p)
        # for those, pick a timestep strictly AFTER t in the same episode ...
        future_offset = (np.random.uniform(size=batch_size) * (T - t_samples)).astype(int)
        future_t = (t_samples + 1 + future_offset)[her_indexes]

        # ... and swap the goal for the achieved goal at that future step:
        # "pretend the thing I actually reached later was the goal all along"
        future_ag = episode_batch['ag'][episode_idxs[her_indexes], future_t]
        transitions['g'][her_indexes] = future_ag

        # recompute reward against the (possibly swapped) goal -- this is what
        # turns a failed episode into a rewarding one: r against m(s_future) is
        # not stuck at -1.
        info = {k.replace('info_', ''): v for k, v in transitions.items()
                if k.startswith('info_')}
        transitions['r'] = reward_fun(ag_2=transitions['ag_2'],
                                      g=transitions['g'], info=info)

        transitions = {k: transitions[k].reshape(batch_size, *transitions[k].shape[1:])
                       for k in transitions.keys()}
        return transitions

    return _sample_her_transitions
```

I end up with this chain: a binary success reward is the honest task spec but leaves the value learner with a constant `-1` and nothing to bootstrap from; better exploration doesn't help because the missing thing is reward signal, not state coverage; reward shaping helps but resurrects the domain knowledge I was trying to avoid, distorts exploration, and optimizes a proxy. The way out is that a trajectory which fails the real goal still *succeeds* at reaching wherever it ended up — so I relabel the transition with an achieved goal `m(s_future)` and recompute the reward against it, which is legal precisely because a goal-conditioned `Q(s,a,g)` accepts any goal as input and an off-policy update doesn't care which policy generated the data. Relabel with future achieved states rather than just the endpoint, because near-horizon achieved goals give tight, attributable credit and automatically form an easy-to-hard curriculum as the policy improves; keep the hindsight-to-real ratio `k` modest so the real goal distribution isn't drowned out; do the relabeling lazily at sample time; and drop all of it onto a goal-conditioned off-policy actor-critic with value-clipping and observation normalization. Every episode, however badly it failed, now teaches the agent something true.
