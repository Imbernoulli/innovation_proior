Let me start from what actually goes wrong, because the symptom is concrete: I'm pretraining a big decoder-only Transformer in bfloat16, and the run keeps blowing up. The loss sits on its nice downward curve and then, with no warning, jumps — a spike — sometimes recoverable, sometimes not, and the bigger I make the model the more often it happens. Gradient clipping is on, the global norm is capped at 1.0, and the spikes come anyway, which already tells me something: whatever is going wrong has produced a state from which even a norm-limited step is bad, so clipping is treating a symptom downstream of the real problem. And there's a slower companion to the spikes — the gradient norm tends to drift upward over the run, growing the longer I train, and the spike frequency rises with it. So I have two coupled things: a slow growth and sudden discontinuities. I don't want to keep restarting from checkpoints and skipping data batches to dodge spikes; that's a band-aid that costs real compute. I want to know what's drifting, and stop it at the source.

Where would a drift like that even live? The training loop has a lot of moving parts, but most of them — the attention, the MLPs, the layer norms — have normalization stapled to them that keeps their scales in check. The one place in the whole forward pass where a wide vector of unbounded real numbers gets pushed through an exponential with nothing normalizing it first is the final softmax over the vocabulary. That's the suspect. So let me stare hard at exactly that layer and write down what the objective there really is, not the way it's usually quoted but in a form I can manipulate.

For one position, I have a logit vector `l = (l_1, ..., l_V)` over the vocabulary, the predicted distribution is `p_j = exp(l_j)/Z` with the normalizer `Z = sum_k exp(l_k)`, and the cross-entropy against the true next token `y` is the negative log-probability of that token. Write it out:

  `CE = -log p_y = -( l_y - log Z ) = log Z - l_y`,

where `log Z = log sum_k exp(l_k)` is the log of the partition function. So the loss I'm minimizing decomposes, exactly, into two pieces pulling in opposite directions: raise the true-token logit `l_y`, and lower the log-partition `log Z`. That's the whole game cross-entropy plays.

Now I notice something about that form: `CE = log Z - l_y` reads like it depends on the logits only through their differences, since both `l_y` and `log Z` shift together when I move the logits around. Let me check whether that's actually true, because if it is, it has consequences. Add a constant `c` to every single logit, `l -> l + c·1`. The softmax: `exp(l_j + c)/sum_k exp(l_k + c) = e^c exp(l_j) / (e^c sum_k exp(l_k)) = p_j`, the `e^c` cancels, so the prediction doesn't move at all. And the two pieces of the loss individually: `l_y -> l_y + c`, while `Z -> e^c Z` so `log Z -> log Z + c`. Inside `CE = log Z - l_y` the two `c`'s cancel. So the algebra says cross-entropy is invariant to a uniform shift of all the logits, while `log Z` moves one-for-one with the shift.

A numeric instance confirms it: for `l = (2, -1, 0.5, 3)`, `y = 3`, `log Z = 3.3841` and `CE = 0.3841`. Shifting every logit by `c = 5.7` leaves the softmax unchanged to floating-point noise (max coordinate difference `2.2e-16`) and `CE` unchanged at `0.3841`, while `log Z` moves to exactly `9.0841 = 3.3841 + 5.7` — `log Z` is the coordinate along the direction the loss is blind to.

This is the crux. There is a whole one-parameter family of logit vectors — `l + c·1` for every real `c` — that produce identical predictions and identical cross-entropy. Along that family, the predicted distribution is frozen, but `log Z` ranges over the entire real line: as `c -> +infinity`, `log Z -> +infinity`, and the individual logits grow without bound, all while the loss the optimizer sees is constant. Cross-entropy pins down the logit gaps — the differences that set the probabilities — and says nothing whatsoever about the overall level. The level is a free gauge. The optimizer has no instruction, from the loss, about where to sit along that gauge orbit.

So which way does the gauge actually drift in practice? It's a free direction, so a priori it could wander either way, but I should not just guess — let me actually run gradient descent and watch. I'll set up a small position: 100 logits, a fixed random "shape" for the gaps, and start the level already drifted up (add a constant of 40 to everything). Then run plain GD on cross-entropy alone and track `log Z` over 5000 steps. Starting `log Z` is 45.1, and the trajectory is `45.1 -> 48.6 (step 100) -> 51.1 (step 1000) -> 52.7 (step 5000)` — monotonically up, and still climbing at the end. So the gauge doesn't wander randomly and it doesn't go down: pure cross-entropy actively inflates the level. That's not an accident of my init; it's because raising the true-token logit `l_y` is one of the two things CE rewards, and with no counter-pressure on the level the optimizer takes the cheap route of pushing logits up. This matches the documented behavior — activations feeding the final softmax grow over the run — and it's exactly the slow growth I'm seeing in the gradient norm. The cross-entropy is indifferent to the one quantity, the absolute logit level (equivalently `log Z`), that I now have direct evidence is drifting up.

And why would a drifting `log Z` be dangerous here specifically, in bfloat16? Because of where these large numbers go: straight into an exponential. bfloat16 keeps float32's eight exponent bits, so it has the same enormous range, but it has only seven mantissa bits against float32's twenty-three. A float stores a fixed number of mantissa bits within each octave `[2^k, 2^{k+1})`, so the gap between representable numbers — the unit in the last place — scales like `2^k · 2^{-mantissa}`, which means the absolute roundoff error grows with the magnitude of the number. Two facts fall out: for the same number, bfloat16 rounds about `2^{23-7} = 65536` times more coarsely than float32; and the larger a number gets, the coarser its rounding. Now push a large, coarsely-rounded logit through `exp`. The differential is `d(exp x) = exp(x)·dx`, so a small absolute error `dx` in the argument becomes a relative error of size `dx` in the output of the exponential. Make the logits large and `dx` (the roundoff) grows with them, so the softmax outputs get corrupted by an amount that grows as the logits drift up.

Let me build a concrete case and actually run the arithmetic, because "grows with magnitude" is exactly the kind of claim that's either marginal or alarming and I can't tell which until I compute it. Take a position with ten logits at 128 and one at 128.5. First, is 128.5 even representable in bfloat16? The value 128 sits in the octave `[2^7, 2^8) = [128, 256)`, where the ULP is `2^7 · 2^{-7} = 1.0`. So the representable bfloat16 values near 128 are spaced a full 1.0 apart, and 128.5 is half a ULP up — right at the rounding boundary, and round-to-nearest-even sends it down to 128.0. The 0.5 gap simply does not survive storage. Now the consequence: with the true logits, the softmax weight on the distinguished token is `exp(0.5) / (10·exp(0) + exp(0.5))` (after subtracting the max), which I compute to be `1.6487 / 11.6487 = 0.1415`. After the 128.5 rounds to 128, all eleven logits are equal, so the distinguished token gets `1/11 = 0.0909`. That's a drop from `0.1415` to `0.0909`, a `35.8%` relative collapse, manufactured entirely by one rounding event. So at magnitude 128 the effect is not marginal at all; it's catastrophic for that token's probability. And the same two logits at magnitude ~3 instead of 128 would have a ULP of `2^1·2^{-7} = 0.0156`, so the 0.5 gap is `0.5/0.0156 = 32` ULPs wide and survives cleanly — the corruption really is a function of how large the logits have drifted, exactly as the `d(exp x) = exp(x)dx` argument said.

So here is the chain I now believe: cross-entropy lets the level drift, and gradient descent on it actively drives the level up (I watched it); large logits are where bfloat16's exponential becomes unfaithful, as the 128 example just showed; an unfaithful exponential corrupts the softmax and, when it tips over, produces a spike. The slow growth and the sudden discontinuity would then be the same disease seen at two timescales. That's a hypothesis with a mechanism, and the GD trajectory is evidence for the first link of it.

So I need a restoring force on the gauge. Cross-entropy won't supply it — by construction it can't, it's gauge-invariant — so I have to add something that does care about the level. What do I want that something to prefer? I want the logits not to drift large; equivalently I want `log Z` not to drift large. A natural target value to pin the gauge to is `log Z = 0`, i.e. `Z = 1`, i.e. `sum_k exp(l_k) = 1`: that's the small, numerically-safe end of the orbit. If I had freedom to choose the gauge `c` for a given prediction, the `c*` that sets `log Z = 0` is `c* = -log Z(l)`, and it leaves the predictions untouched (it's a gauge move) while killing the magnitude. So the hard version of what I want is "project onto the slice `log Z = 0` of every gauge orbit." But I don't want to hard-project — that's a constraint I'd have to enforce, and it would fight the optimizer. I want a soft nudge: add a penalty to the loss that is minimized when `log Z = 0` and grows as `log Z` moves away from it, then let the optimizer trade it off against the cross-entropy.

What's the right shape for that penalty as a function of `log Z`? Call `s = log Z`. I want a function `phi(s)` minimized at `s = 0`. The candidates I can think of: `|s|`, `s^2`, or just `s`. Take the signed `s` first and see if it survives: penalizing `s` itself has gradient `+1` everywhere, no minimum, so it would shove `log Z -> -infinity`, sending the whole gauge level downward without ever settling. That's not a restoring force, it's a one-way push — ruled out. So the penalty must be symmetric in `s`, penalizing large excursions in either direction: the representative with `Z` near 0 is not more meaningful than the representative with `Z` huge; both are far from the slice I want. `|s|` is symmetric and minimized at 0, but it has a kink at `s = 0`: its gradient is `sign(s)`, a constant pull of fixed strength right up to the minimum, which keeps yanking even when `log Z` is already tiny, and it's nonsmooth exactly where I want the system to settle. `s^2` is symmetric, minimized at 0, smooth everywhere including at the minimum, convex, and — this is the property I want — its gradient is `2s`, proportional to the violation: gentle when `log Z` is already near 0, stronger the further it has drifted. That matches the behavior of a restoring force; it leaves a well-behaved gauge alone and pulls harder on a runaway. So I'll take the penalty to be `(log Z)^2` — square the log-partition. Whether it actually behaves as a restoring force is a claim about the gradient and the dynamics it produces, so that's what I work out next.

Let me write the augmented per-position objective and work out its gradient, because the gradient is what tells me whether this actually does what I think:

  `L = CE + lambda·(log Z)^2 = (log Z - l_y) + lambda·(log Z)^2`,

with `lambda` a small coefficient I'll pin down later. I need the gradient with respect to each logit `l_j`. The key derivative is `d(log Z)/d l_j`. Since `log Z = log sum_k exp(l_k)`,

  `d(log Z)/d l_j = exp(l_j) / sum_k exp(l_k) = exp(l_j)/Z = p_j`.

So the derivative of the log-partition with respect to a logit is just that token's softmax probability. Now the cross-entropy part: `d(log Z - l_y)/d l_j = p_j - 1{j = y}`, the familiar "predicted minus one-hot." And the penalty part, by the chain rule:

  `d/d l_j [ lambda·(log Z)^2 ] = lambda · 2 log Z · d(log Z)/d l_j = 2 lambda·log Z · p_j`.

Stack them:

  `dL/d l_j = (p_j - 1{j=y}) + 2 lambda·log Z · p_j`.

A check against autograd on a random 6-vector in float64 agrees with this formula coordinate by coordinate to `1.1e-16` — machine epsilon.

Now read what the new term does. It's `2 lambda log Z · p_j` — a probability-weighted pull on every logit. When `log Z > 0` (the logits have drifted up, the dangerous case), this term is positive on every coordinate, weighted by `p_j`, so the gradient-descent step subtracts it: it pushes down hardest on the logits that currently contribute most to `Z`, shrinking `log Z` back toward 0. When `log Z < 0` it reverses sign and pushes those same high-probability coordinates back up. It is not a perfectly uniform shift, so it can still alter gaps at finite `lambda`; the point is that its scalar driver is the otherwise-free `log Z`, while cross-entropy remains the term that primarily sculpts the probability gaps.

But "shrinks `log Z` back toward 0" is the dynamical claim, and the gradient sign alone doesn't tell me how strong the effect is against the upward drift I measured. So let me rerun the same 100-logit experiment, this time with the penalty on, and watch `log Z` for a few values of `lambda`. With `lambda = 0` (the baseline) the trajectory was `45.1 -> 52.7` and rising. With `lambda = 1e-4`: `45.1 -> 48.5 (step 100) -> 49.4 (step 1000) -> 49.2 (step 5000)` — the climb is arrested. It doesn't drive `log Z` to 0; it stops it from running away and parks it on a plateau around 49 instead of letting it pass 52 and keep going. That actually sharpens my understanding of what `lambda = 1e-4` does: the restoring drive at the plateau is `2·lambda·log Z = 2·1e-4·49 ≈ 0.0098` per step, just enough to cancel the per-step upward pull from cross-entropy, so the two reach a standoff. With `lambda = 1e-2` instead: `45.1 -> 45.1 (step 100) -> 41.1 (step 1000) -> 27.8 (step 5000)` — now the penalty dominates and actively crushes the level down, restoring drive `2·1e-2·49 ≈ 0.98` per step. So the picture is honest now: a small `lambda` does not pin `log Z` to zero, it neutralizes the drift, and a larger `lambda` overpowers it. The proportional-strength property I wanted from `s^2` is visible — the drive scales with `log Z` and with `lambda` — and the canonical small coefficient's job is to stop the runaway, not to zero out the partition — a real restoring force, gentle by design.

Now does arresting the drift in `log Z` actually bound the thing I care about — the logit magnitudes that feed the exponential? I'd like a bound, not a hope. The log-sum-exp is sandwiched: on one side, `Z = sum_k exp(l_k) >= exp(max_k l_k)`, so `log Z >= max_k l_k`. On the other, `Z <= V · exp(max_k l_k)`, so `log Z <= max_k l_k + log V`. Putting them together,

  `max_k l_k <= log Z <= max_k l_k + log V`.

Let me sanity-check that sandwich numerically before I lean on it, because it's the link from "control `log Z`" to "control the worst logit." I generate 2000 random logit vectors of size `V = 50000` at a wide range of scales and, for each, check `logZ - max_l >= 0` and `logZ - max_l - log V <= 0`. The minimum of `logZ - max_l` over all of them is `0.0` (attained when one logit dominates), and the maximum of `logZ - max_l - log V` is `-0.416 <= 0` (the upper bound is loose unless all logits are equal). `log V = 10.82`. So the bound holds, and it's tight on the lower side. Reading it the way I need: `log Z - log V <= max_k l_k <= log Z`. If the penalty holds `log Z` near 0 — or even just keeps it from running away — then the largest logit cannot become large and positive; for a 50k vocabulary it lives no more than about 11 nats below `log Z` and never above it. This does not pin every logit to a point — low-probability logits can be very negative — but it caps the top logits, the ones that dominate the exponential and the softmax probabilities. That is the difference between "the exponent sees numbers near the normal log-probability scale" and "the exponent is free to see 128," which is precisely the regime where bfloat16's exponential stays faithful, where `dx` roundoff is small because the relevant numbers themselves are small.

That same target has another consequence worth noting. `log Z = 0` means `sum_k exp(l_k) = 1`. Combine that with what cross-entropy is already doing — driving the softmax to match the targets — and the raw logits themselves start to behave like normalized log-probabilities: `exp(l_j)` summing to one means `l_j` is approximately `log p_j` directly, not up to an arbitrary additive constant. So the same penalty that keeps the numbers small also nudges the logits toward being honest log-probabilities rather than log-probabilities-plus-junk. (At `lambda = 1e-4` the partition isn't driven all the way to `Z = 1`, as the dynamics showed, so this is a tendency rather than an exact identity — but it pulls in the right direction.) The gauge-fixing and the "make the logits meaningful" goal are the same move; I didn't have to choose between them.

Let me make sure I'm not reinventing something already on the table, and that I've picked the right lever. The standard over-confidence fix at this layer is label smoothing — soften the one-hot target into `(1-eps)` on the true class plus `eps/V` spread over the rest, so the loss becomes a mixture of the true cross-entropy and a uniform-target cross-entropy. Does that solve my problem? Look at what it touches. Label smoothing changes the target distribution — it deliberately fits the model to a softened distribution instead of the data, trading a bit of true log-likelihood for calibration. That's the wrong contract here: I want to keep modeling the same next-token distribution, not lower a number by changing what is being modeled. And mechanically, label smoothing acts on the logit gap between the true class and the rest — it caps how far apart the classes get pushed — which is a statement about differences, the gauge-invariant part. By the shift-invariance I checked at the very start, anything that depends only on the predicted distribution is blind to the level; label smoothing is exactly such a thing, so it does nothing to the free level. It's pulling on the wrong handle. My penalty is the complement — it touches the level through `log Z`, leaves the target distribution alone, and so it can sit on top of plain cross-entropy without becoming label smoothing in disguise. Different lever, and the right one for a numerical drift.

What about clipping the logits directly — just cap them at some value before the softmax? That's the brute-force version and it's worse, for a reason that's specific to the roundoff story. The whole point is that large numbers are already corrupted by roundoff by the time they exist; clipping happens after that corruption, so it clamps an already-wrong value, and a hard clip is itself a discontinuity — in effect another roundoff, a sharp kink the optimizer has to route gradients around. The penalty, by contrast, is smooth and acts during optimization to discourage the model from ever producing the large logits in the first place, so the numbers are small and faithful before they hit the exp, not clamped after they've already gone bad. Prevention at the source beats a hard cap downstream. And the reactive option I already have, gradient-norm clipping, is even further downstream — it acts after the bad step has been computed, which is consistent with my opening observation that clipping at norm 1.0 didn't stop the spikes. The penalty is the only one of these that addresses the cause.

Now `lambda`. It has two jobs to balance, and the dynamics experiment already gave me the scale empirically — let me reconstruct the order-of-magnitude argument that agrees with it. It must be small enough that the penalty is a gentle regularizer and cross-entropy stays essentially the maximum-likelihood loss, with the penalty only nudging the gauge, or I'd be back to changing what's modeled. And it must be large enough that the restoring force actually keeps `log Z` from running away. Scale check: cross-entropy per token is order one-to-ten nats early and settles to a few nats; a healthy `log Z` (when the gauge is well-behaved) is order one, so `(log Z)^2` is order one too. If I take `lambda = 1e-4`, the penalty contributes on the order of `1e-4` against a cross-entropy of order one — about a ten-thousandth of the CE scale — yet with gradient `2 lambda log Z` it still applies a steady downward pull on the level that, as I watched, accumulates over many steps to hold `log Z` down and arrest the climb. That's the sweet spot the simulation showed: at `1e-4` the modeling objective is untouched but the runaway is stopped, whereas `1e-2` started visibly dragging the level toward zero and would begin competing with cross-entropy for control of the logits. Much smaller than `1e-4` and the restoring force would be too weak to win against the drift. `1e-4` is the canonical large-vocabulary coefficient that keeps the penalty a regularizer and not a second objective; I'll use it, and the simulation gives me confidence it's the right order of magnitude.

A couple of implementation details I have to get right or the whole thing is subtly wrong. First, reduction: cross-entropy here ignores padding / packed-boundary positions, the ones whose target is the sentinel `-1`, and averages over the valid positions. My penalty has to be reduced the same way — averaged over exactly the valid positions — for two reasons. The logits at ignored positions are garbage I never train, so penalizing their `log Z` is meaningless and would inject noise. And if I averaged the penalty over a different count than cross-entropy uses, the effective `lambda` would no longer be the clean ratio I just reasoned about; the two terms have to be on the same per-token footing for `1e-4` to mean what I think it means. So: mask to `target != -1`, take the mean of `(log Z)^2` over those positions.

Second, numerics of computing `log Z` itself — I'd be embarrassed to introduce an instability while fixing one. `log Z = log sum_k exp(l_k)` computed naively overflows the moment any logit is largish, which is the whole situation I'm in. The standard stabilized log-sum-exp subtracts the max first: `log Z = max_k l_k + log sum_k exp(l_k - max_k l_k)`, which is exact and never exponentiates a positive argument. I'll use the library `logsumexp` primitive, which does exactly this. Note the loop back to the 128.5 example: the same max-subtraction trick that makes `logsumexp` safe is the operation that, in that corrupted example, was the source of roundoff trouble when the logits were huge — which is one more reason to keep the logits small in the first place, and exactly what my penalty does.

Third, the code should keep the loss exactly in the simple softmax-loss shape I just derived. The primitive computation is: compute `log_z = logsumexp(logits)`, form `log_softmax = logits - log_z`, take the negative true-token log-softmax as cross-entropy, and then, if the auxiliary coefficient is nonzero, add `lambda * log_z^2` at that same per-position shape. In the hard-target `compute_loss(logits, targets)` slot I can delegate the first part to the library cross-entropy, but the auxiliary term must still be literally the same per-position `log_z` squared. That keeps the implementation faithful to the loss: no temperature, no target smoothing, no clipped logits, just the ordinary cross-entropy plus the small log-partition penalty.

Let me put it together against the actual interface I have to fill — a function that takes the logit tensor of shape `(B, T, V)` and the targets of shape `(B, T)`, with `-1` ignored, and returns the scalar to backpropagate. Flatten the batch and time into one axis of positions, get cross-entropy the usual way (it handles the ignore index and the averaging), compute `log Z` per position with the stabilized primitive, mask to the valid positions, square, average, and add with the small coefficient:

```python
import torch
import torch.nn.functional as F


def compute_loss(logits, targets):
    # logits: (B, T, V); targets: (B, T) with -1 marking ignored positions.
    flat_logits = logits.view(-1, logits.size(-1))     # (B*T, V)
    flat_targets = targets.view(-1)                     # (B*T,)

    # Primary objective: plain next-token cross-entropy, CE = log Z - l_y,
    # averaged over valid positions (ignore_index=-1).
    ce_loss = F.cross_entropy(flat_logits, flat_targets, ignore_index=-1)

    # Auxiliary softmax z-loss:
    # add lambda * square(log_z), where log_z = logsumexp(logits) is the
    # max-subtraction-stabilized softmax log-partition.
    log_z = torch.logsumexp(flat_logits, dim=-1)        # (B*T,)
    mask = flat_targets != -1                            # same valid positions as CE
    z_loss = (log_z[mask] ** 2).mean()                  # mean over valid positions

    # lambda = 1e-4, the canonical large-vocabulary coefficient.
    # Its logit gradient contribution is 2*lambda*log_z*p_j.
    return ce_loss + 1e-4 * z_loss
```
