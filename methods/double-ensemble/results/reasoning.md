I start with the part that is tempting but wrong. I have a strong tabular learner and I can make it into an ensemble, so the easy move is to train several copies and average them. That helps with variance, but it does not answer the two finance-specific questions in front of me. Which rows should the next member pay attention to? Which columns should it see? If I leave both choices uniform, I get diversity, but it is blind diversity.

The row question is dangerous because the usual boosting instinct points in the right general direction and the wrong detailed direction. A row with high current loss might be a useful hard case, close to a real boundary or to a rare but repeatable pattern. It might also be an unforecastable shock. If I simply say high loss means high weight, I give the next model more incentive to memorize noise. But if I only down-weight all high-loss rows, I also lose the hard rows that actually teach the model something. So I need a second signal that separates hard-but-learnable from hard-because-noisy.

A boosted tree already gives me more than a final loss. While it is being trained, every row has a loss after tree 1, after tree 2, and so on. That path says something that the final value hides. If a row is useful to the decision surface, its relative loss should tend to improve as the learner accumulates structure. If a row is mostly noise, its relative loss may stay flat or rise because the model is not finding a stable pattern for it. This is the extra axis I need: not just how large the current loss is, but how the row's loss moves during the previous member's training.

Raw losses across iterations are not directly comparable, because the whole loss scale changes as trees are added. I should rank-normalize across samples at each iteration. Let C be the N by T loss-curve matrix for the previous member, and let norm(C) replace each column by percentile ranks. Then I summarize the beginning and ending of each row's path with C_start and C_end, the row means over the first and last ten percent of iterations. Ten percent is a stabilizer: it avoids making the decision from a single early or late tree while still comparing the two ends of the trajectory.

Now I need the signs to be exactly right. For the current ensemble loss vector L, I use h1 = norm(-L), not norm(L). That means an easy low-loss row has a large h1, while a high-loss row has a small h1. Because the final weight will be inverse in h, a small h1 increases the row's weight. So h1 is the boosting-like part: it raises hard and noisy rows and suppresses easy rows. By itself it has the same noise problem as boosting.

For the trajectory term I use h2 = norm(C_end / C_start). A descending relative loss curve gives a small ratio, hence a small h2, hence a larger inverse weight. A flat or rising relative curve gives a larger ratio, hence a larger h2, hence a smaller weight. This term is what removes the sign error: high current loss alone would chase noise, but a noisy row should not also have the favorable descending trajectory. Combining the terms gives h = alpha1 h1 + alpha2 h2. Hard learnable rows have small h1 because current loss remains meaningful and small h2 because the loss path is descending, so their h is small and their weight is large. Easy rows have large h1 and moderate h2, so their weight is small to moderate. Noisy rows have small h1 but large h2, so the two terms counteract each other instead of letting noise dominate.

I still do not want one outlier row to set an extreme weight. I bin the h values into B bins and use the mean h inside each bin. If row i falls in bin b_i, the weight is w_i = 1 / (gamma^k * mean_h[b_i] + 0.1). The +0.1 prevents the denominator from approaching zero. The gamma^k factor, with gamma between 0 and 1, makes later members less aggressive: as k grows, gamma^k shrinks, the h-dependent part of the denominator matters less, and the weights move toward a common value. So the early members use the reweighting signal sharply, and later members become closer to uniform rather than escalating the chase.

That settles the row side. The feature side has a different failure mode. If I rank features once and always keep only the strongest ones, every member learns the same regime-specific vocabulary. If the current market regime makes a factor look weak, a hard cutoff may discard a factor that matters later. I want feature subsets that prefer useful columns but still preserve diversity.

The cheapest model-agnostic importance test is to perturb one column and see how much the current ensemble's loss changes. For feature f, I shuffle that column across training samples to form X_f, predict with the current ensemble, and compute L_f. Then g_f = mean(L_f - L) / (std(L_f - L) + epsilon). A large positive g_f means the shuffled feature increases loss consistently, so the current ensemble relies on it. Dividing by the standard deviation keeps a feature with a noisy, inconsistent effect from looking important just because its mean shift is large in a few rows.

The selection rule should not be a top-k cutoff. I bin features by g-value and sample a fixed fraction from each bin, with the largest fraction for the highest-g bin and smaller but nonzero fractions for weaker bins. The ceiling in ceil(r_d N_d) matters because a small bin should not vanish just because its expected sample count is fractional. The result is a member that mostly sees strong features, but also sees a tail of weaker or dormant features. That gives each member a different column view without pretending weak-current-regime features are useless forever.

The training loop is now straightforward. Start with uniform row weights and all features. Train member k on the current weights and feature subset. If it is not the last member, retrieve the previous member's per-tree loss curve, compute the current ensemble's training loss, update the next row weights with the binned inverse h formula, and choose the next feature subset with shuffled-column g-values. Prediction discards all of this training-time machinery: each stored member predicts on the feature subset it was trained on, and the wrapper averages the member predictions, optionally with configured member weights.

The simple combination rule is an average over members. If I later expose member weights, the safest default remains all ones; the feature-selection score should still be read as the current ensemble's average reliance unless I deliberately redesign it for weighted members. The sample-reweighting routine is the place where a sign mistake would be fatal, so I pin it down before moving on: rank(-loss_values), rank(C_end / C_start), and the inverse denominator 1 / (decay**k * h_avg + 0.1).

So the final construction is a sequential ensemble with two between-member updates. The sample update uses final ensemble loss only to say where error remains, then uses the previous member's loss trajectory to avoid confusing noisy rows with useful hard rows. The feature update uses shuffled-column loss changes to estimate reliance, then samples across reliance bins instead of collapsing to one fixed subset. The constants are deliberately conservative: rank normalization for robustness, first and last ten percent windows for stable trajectory endpoints, binning for regularized weights/importances, +0.1 for a denominator floor, gamma in [0, 1] for later-member annealing, and nonzero feature sampling ratios so diversity remains present all the way down the bins.
