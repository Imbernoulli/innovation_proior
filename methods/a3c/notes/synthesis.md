# A3C — Synthesis notes (Phase 1.5)

## The pain point at the time (2015–2016)

Deep RL had one spectacular success: DQN (Mnih et al. 2013, 2015). It learned Atari from pixels with a CNN Q-network. But the recipe that made it work was widely believed to be *necessary*: combining a deep net with naive online RL was thought to be "fundamentally unstable." The instability story:

- An online RL agent visits states along a trajectory. Consecutive states/observations are highly correlated (s_t and s_{t+1} look almost the same). SGD assumes roughly i.i.d. samples; feeding it a stream of correlated gradients makes updates point in similar directions for long stretches, so the net overfits to the current region of state space and the optimization is high-variance / non-stationary.
- Worse, with a bootstrapped TD target the target depends on the *same* network being updated, and the policy that generates data changes as θ changes — a moving data distribution chasing a moving target. This is the "deadly triad" (function approximation + bootstrapping + off-policy) that can diverge.

DQN's fix = **experience replay**: store transitions (s,a,r,s') in a big buffer (~1M), sample minibatches uniformly at random. This (a) breaks temporal correlation by mixing transitions from many different times, and (b) re-uses each transition many times (data efficiency), and (c) averages the data distribution over many past policies → more stationary. Plus a **target network** θ⁻ frozen for I_target steps to stabilize the bootstrap target.

### The cost of replay — the crack to exploit
- Replay only works for **off-policy** algorithms. The buffer contains data generated by *old* policies; to learn from it correctly the algorithm must be able to update from data not generated by the current policy. Q-learning is off-policy (it learns Q* regardless of behavior policy), so it's fine. But this **rules out on-policy methods** — Sarsa, actor-critic, plain policy gradient — whose updates are only valid for data drawn from the current policy. So the most natural policy-gradient / actor-critic family is locked out of the "deep + stable" club.
- Replay costs **memory** (store a million frames) and **compute per real interaction** (each step you also replay a minibatch).
- To scale, the only known route was **more hardware**: DQN needs a GPU; Gorila (Nair et al. 2015) distributes across 100 actors + 30 parameter servers = 130 machines, each with its own replay memory, gradients shipped to a central server.

So: the field believed stability *requires* replay, and replay *requires* off-policy and *requires* lots of memory/hardware. The research question: **can we decorrelate updates and stabilize deep RL WITHOUT a replay buffer — so that on-policy methods become usable and so it runs on one commodity machine?**

## The key insight (the thing to land on)

Replay's *real* job is to make the stream of gradients fed to SGD look decorrelated and roughly stationary. There's a second way to get decorrelation that has nothing to do with storing the past: **run many actors in parallel, each in its own environment instance, each with its own (possibly different) exploration policy, all updating one shared set of parameters asynchronously.** At any instant the workers are in different states, exploring different regions, so the batch of updates hitting the shared θ at any moment is drawn from a *diverse* mixture of states — decorrelated *in time across workers* instead of *across a stored buffer*. Parallelism replaces replay as the decorrelating mechanism.

Consequences that fall out:
- Because decorrelation no longer comes from a buffer of *old-policy* data, you are **not forced off-policy** anymore. On-policy methods (Sarsa, n-step Q, actor-critic) become directly usable — each worker's data is fresh, generated by (nearly) the current shared policy. This *unlocks* the actor-critic family that was previously excluded.
- No buffer → **no replay memory** (memory + per-step replay compute saved).
- Many CPU cores on one machine instead of GPU/cluster: training time ≈ linear speedup in #workers, and the decorrelation effect means it also *stabilizes* (not just speeds up).

The single-machine multi-thread choice (vs Gorila's machines + parameter server) enables **Hogwild!**-style lock-free updates (Recht et al. 2011): on shared memory, multiple threads do gradient updates to shared θ without locking; for sparse-ish/SGD updates the races are provably benign and you avoid all the communication/synchronization cost. That's why it's on one box, not a cluster.

## The four algorithms (all share the same skeleton)

Skeleton per thread: copy shared θ → θ' (thread-local); roll out up to t_max steps in own env with own exploration; compute accumulated gradient d θ over those steps; asynchronously apply dθ to shared θ (Hogwild, no lock); repeat. Global shared counter T.

1. **Async one-step Q-learning** (Alg 1): each step, target y = r + γ max_{a'} Q(s',a';θ⁻) (0 if terminal); accumulate ∂(y−Q(s,a;θ))²/∂θ; apply every I_AsyncUpdate steps or at episode end. Shared **target network** θ⁻ updated every I_target. Accumulating grads over several steps ≈ minibatch, and reduces threads overwriting each other.
   - Per-thread **different ε** for ε-greedy: each thread periodically samples ε from a distribution (3-point: anneal to 0.1, 0.01, 0.5 with prob 0.4/0.3/0.3). Diverse exploration = more decorrelation + better exploration.
2. **Async one-step Sarsa**: identical to (1) but target y = r + γ Q(s',a';θ⁻) where a' is the *action actually taken* in s' (on-policy TD). Still uses target net + accumulation.
3. **Async n-step Q-learning** (Supp Alg 2): **forward view**. Act up to t_max steps; from the last state bootstrap R = max_a Q(s_t,a;θ⁻) (0 if terminal); walk backward R ← r_i + γ R accumulating ∂(R−Q(s_i,a_i;θ'))²/∂θ' for each i. So the last state gets a 1-step target, second-last a 2-step target, …, up to t_max-step. One reward propagates to up to n preceding (s,a) in a single update → faster credit assignment than one-step.
   - Forward view (explicit n-step returns) chosen over backward view (eligibility traces) because it's easier with momentum optimizers and BPTT for recurrent nets.
4. **Async advantage actor-critic (A3C)** (Supp Alg 3): maintain policy π(a|s;θ) and value V(s;θ_v). Same forward-view n-step rollout. Bootstrap R = V(s_t;θ'_v) (0 if terminal). Walk backward R ← r_i + γR; advantage estimate A = R − V(s_i;θ'_v); accumulate
     - policy grad: dθ ← dθ + ∇_{θ'} log π(a_i|s_i;θ') (R − V(s_i;θ'_v))
     - value grad: dθ_v ← dθ_v + ∂(R − V(s_i;θ'_v))²/∂θ'_v
   - Share most params: one CNN, softmax head for π, single linear head for V (all non-output layers shared).
   - **Entropy regularization** added to policy objective: ∇_{θ'} log π(a_i|s_i;θ')(R−V) + β ∇_{θ'} H(π(s_i;θ')). H is policy entropy; β controls strength (0.01). Reason: discourage premature convergence to a near-deterministic suboptimal policy; keeps exploration alive (Williams & Peng 1991).

## Derivations to live out in reasoning.md

### Policy gradient → REINFORCE → baseline → advantage actor-critic
- Objective J(θ) = E[R_t]. Policy gradient theorem / score-function: ∇_θ E[R_t] = E[ ∇_θ log π(a_t|s_t;θ) R_t ] (REINFORCE, Williams 1992). Unbiased but very high variance because R_t is a noisy Monte-Carlo return.
- Subtract a state-dependent baseline b(s_t): E[∇_θ log π(a|s) b(s)] = b(s) Σ_a ∇_θ π(a|s) = b(s) ∇_θ Σ_a π(a|s) = b(s) ∇_θ 1 = 0. So baseline doesn't bias the gradient but reduces variance. Best baseline ≈ V^π(s).
- With b(s)=V(s): the multiplier R_t − V(s_t). Since E[R_t | s,a] = Q^π(s,a), this is ≈ Q^π(s,a) − V^π(s) = A^π(s,a), the **advantage**. → actor-critic: actor = π, critic = V.
- A3C estimates A with the n-step bootstrapped return: Â = Σ_{i=0}^{k-1} γ^i r_{t+i} + γ^k V(s_{t+k}) − V(s_t), k ≤ t_max. The first part (n-step return) is the Q-estimate; minus V(s_t) is the advantage. This is exactly R − V(s_t) computed in the backward loop.

### Why baseline reduces variance but a constant doesn't help much
- Variance of g = ∇logπ (R−b). Var minimized over b roughly when b ≈ E[(∇logπ)² R]/E[(∇logπ)²] ≈ weighted V. A learned V(s) tracks the return's mean per state → cancels the large state-to-state baseline variation, leaving only the *action* advantage. Constant baseline only removes overall mean, not per-state variation.

### n-step return bias/variance tradeoff
- 1-step target r+γV(s'): low variance (one random reward), high bias (depends on imperfect V). Monte-Carlo R_t: unbiased, high variance. n-step interpolates: bootstraps after n real rewards → less bias than 1-step, less variance than full MC. t_max controls it.

### Entropy term
- H(π) = −Σ_a π(a|s) log π(a|s). Adding β H to the maximization objective; ∇_θ H pushes π toward uniform (higher entropy) → prevents the softmax from saturating to a single action early. Without it, an early lucky action gets reinforced, π collapses, exploration dies, stuck in local optimum.

### Why parallelism decorrelates (make this quantitative-ish)
- Single online agent: gradients g_t, g_{t+1} along a trajectory are strongly positively correlated (similar states) → effective batch is "1 direction repeated," high-variance drift. With N workers in different states, the aggregate update over a wall-clock window is (1/N)Σ over N nearly-independent states → like an i.i.d. minibatch of size N. Variance of the mean drops ~1/N and the temporal correlation that replay was fighting is gone. So parallelism gives both the minibatch-averaging *and* the decorrelation that replay provided, without storage and without going off-policy.

### Hogwild! justification
- Lock-free async SGD on shared θ: when updates are sparse or step sizes small, the probability two threads write the same coordinate simultaneously is low, and the stale/raced updates act like bounded noise; Recht et al. proved near-optimal convergence rates. So skipping locks (which would serialize and kill throughput) is safe → enables the cheap single-machine design.

### Shared RMSProp
- Non-centered RMSProp: g = α g + (1−α)Δθ²; θ ← θ − η Δθ/√(g+ε), elementwise. Per-parameter adaptive LR. In async setting, choice: per-thread g vs shared g. Shared g (updated lock-free) is more robust to LR/init than per-thread or momentum SGD, and saves one parameter-vector-sized copy per thread (memory). (Canonical PyTorch reimplementations often use shared Adam instead — same idea: a shared adaptive optimizer state.)

## Design-decision → why table

| Decision | Why / alternative rejected |
|---|---|
| No replay buffer | Decorrelation comes from parallel diverse actors instead; frees on-policy methods, saves memory/compute. |
| Parallel actors on shared model | Replaces replay as decorrelator; gives ~linear speedup + stability. |
| Single machine + threads (vs Gorila cluster + param server) | Removes gradient/param communication cost; enables Hogwild lock-free updates. |
| Hogwild (no locks) | Locks serialize and destroy throughput; races provably benign for small/sparse SGD steps. |
| Different exploration per thread (ε from a distribution) | Maximizes data diversity → more decorrelation + better exploration robustness. |
| Accumulate grads over t_max steps before applying | Acts like a minibatch (lower-variance grad); fewer threads overwriting each other; trade compute vs data efficiency. |
| Target network θ⁻ (value methods) | Stabilize bootstrap target (moving-target problem), as in DQN. |
| n-step returns, forward view | One reward updates n predecessors → faster credit assignment; forward view easier with momentum + BPTT than eligibility traces. |
| Advantage = n-step return − V (actor-critic) | Baseline V cuts policy-gradient variance without bias; advantage gives per-action signal. |
| Share CNN body between π and V heads | One representation, less compute/params; value learning regularizes features. |
| Entropy bonus βH, β=0.01 | Prevent premature collapse to deterministic suboptimal policy; keep exploration. |
| Shared RMSProp | More robust to LR/init than momentum SGD or per-thread RMSProp; saves a per-thread copy. |
| t_max=5 (Atari), γ=0.99, α=0.99, 16 threads | Standard short rollout; γ horizon; RMSProp decay; cores on one CPU. |

## Canonical implementation (grounding for code)
Kostrikov pytorch-a3c (files in code/): `main.py` spawns shared_model.share_memory() + N train processes + 1 test; `train.py` is the per-worker loop: sync local model ← shared, roll out num_steps, bootstrap R from V (or 0 if done), backward loop computing R, advantage = R−V, value_loss += 0.5 advantage², GAE (λ; with λ=1 reduces to n-step advantage) for policy_loss, policy_loss −= logπ·advantage + entropy_coef·entropy; backward total loss = policy_loss + 0.5·value_loss; clip grad norm; ensure_shared_grads copies local .grad into shared model; optimizer.step() on shared params (SharedAdam). `model.py` = CNN + LSTM + critic_linear + actor_linear. `my_optim.py` = SharedAdam with share_memory'd state (the modern stand-in for shared RMSProp). The answer code will mirror this; I'll keep entropy + advantage policy loss + n-step bootstrap front and center and note GAE λ=1 ≡ the paper's n-step advantage.
