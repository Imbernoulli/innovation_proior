Let me start from the thing that actually nags at me. I have good evidence that a handful of LLM agents talking to each other beat a single agent: one drafts, another critiques, the first revises, and the revision repairs first-pass mistakes a lone model just commits and ships. Specialization plus a critique-then-fix loop recovers errors. But every demonstration of this I can point to uses a tiny, fixed cast — fewer than ten agents, a few dozen at the outside. Nobody treats the *number* of collaborators as something to turn. And that's strange, because in ordinary deep learning the single most reliable lever is scale: add neurons and quality climbs along a smooth curve, sometimes with a new ability appearing abruptly past a threshold. If a group of agents is even loosely like a network of neurons — each agent a unit that processes and passes on information — then the question writes itself: does adding more collaborating agents keep paying off, and what is the shape of the curve relating agent count to quality? I cannot answer that by running four agents. I need a way to instantiate collaboration among arbitrarily many agents, and then sweep the number and watch.

So before any cleverness, what is the bare minimum I need? I need, for a chosen population of agents, a definite specification of who passes work to whom; a definite order in which they act; and — this is the one I suspect will bite — a way to keep each agent's reading burden from blowing up as the population grows. Let me look at what's already on the table and see exactly where each falls short of "instantiate collaboration among N agents and scale N."

The chain is the simplest thing that works: agent 0 drafts, hands to 1 who refines, hands to 2, and so on, a waterfall — design, code, test in the software case. It genuinely refines: each step catches what the last missed. But as a scaling substrate it's hopeless. Its only structural quantity is its length, so "more agents" just means a longer line; the depth grows linearly while the breadth stays one. There's no parallel exploration — every agent sees exactly one upstream draft, the immediately preceding one, so the whole collaboration is a single thread of consciousness with no diversity of starting points. If I want to study scale, the chain gives me one boring axis.

The star fixes breadth and loses depth: a hub broadcasts to many workers who each produce a variant in parallel. Now I have diversity — many independent attempts at once — but no iterative refinement; the parallel branches never get improved by a second pass, and worse, if everything has to come back together, *someone* has to read all of it. That's the first hint of the cost problem. Trees, like AgentVerse arranges, sit in between — branch out hierarchically, reflect, refine — but they're hand-assembled, they tend to collapse back toward a star in practice, and they're observed to fall over once you push past a few dozen agents. Why do they fall over? Hold that thought; the answer is the same answer that will eventually constrain everything.

Tree of Thoughts and Graph of Thoughts are operating one level down — over the *thoughts* of a single model, not over a society of role-playing agents — but they're worth staring at because they fought a structural battle I'm about to fight. Tree of Thoughts models reasoning as a tree: explore branches, look ahead, backtrack, prune. Graph of Thoughts says explicitly that the tree is too rigid, and the specific rigidity it names is the one I care about — in a tree, two separate branches can never be recombined. It generalizes to an arbitrary graph of thoughts precisely so that distinct lines of work can be *aggregated*, merged into a combined thought that takes the strengths of each. That's the lever the star was missing: a structure where many parallel drafts can flow back together and be fused, not just produced side by side. Good — so whatever I build wants to support *convergence*, the merging of multiple incoming pieces of work, and not only the divergence the star gives. But Graph of Thoughts also does a thing I should be wary of: it permits feedback loops, cycles in the graph. The moment I allow a cycle, "who acts after whom" stops being well-defined — information can flow backward and I'd need some task-specific rule to break the loop. For a structure I want to scale and reuse across heterogeneous tasks, an arbitrary cyclic graph is a liability.

And then GPTSwarm, which is the closest to where my instinct is pointing: it represents agent systems as computational graphs — node is an operation, an LLM or tool call; edge is information flow; an agent is a sub-graph; it focuses on directed acyclic graphs — and it *optimizes* the node prompts and the edge connectivity with reinforcement learning or evolution. So the graph-of-agents abstraction is already validated. But its whole thrust is to *learn* the right connectivity for a given task, with every node and edge hand-customized; that's laborious, it doesn't generalize cleanly across tasks, and — critically for me — it is the wrong question. I don't want to learn one optimal wiring for one task. I want to *fix* a structure, scale its size, and measure how collaboration behaves. The contribution I'm circling isn't an optimizer over graphs; it's the graph as a controlled, general, scalable organizing principle, and a way to keep it sane at a thousand agents.

Pull the common thread out of all of that. Every one of these is a special case of "nodes that produce work, connected by directed edges along which work flows." The chain is a path; the star is a single source fanning out; the tree is a branching hierarchy; what Graph of Thoughts wants is a graph where branches reconverge. So I shouldn't pick one — I should adopt the general object they're all instances of and then study the instances within it. A directed graph over agent nodes, with edges carrying the flow of work. And it must be acyclic: acyclicity is exactly what gives me a clean, task-independent order to act in and forbids work flowing backward, so I never need a bespoke cycle-breaking rule. A directed acyclic graph, then — G = (V, E), V the agent nodes, E the directed edges, no cycles. The chain, star, tree, mesh, and the reconverging graphs all live inside this one object as particular edge sets, which means I can hold the *machinery* fixed and vary only the *shape* — and that is precisely the apparatus a scaling study needs.

Now, what sits on a node and what sits on an edge? The naive reading is: agents are nodes, edges are just arrows. But the collaboration that works is a *two*-part thing — somebody produces an artifact, and somebody else looks at it and says "here's what's wrong, fix this." Those are different jobs. If I make every agent a node and leave edges as inert arrows, I have to bolt the critique behaviour onto the nodes somehow, and it gets tangled — is a node critiquing itself, or its predecessor, and in what role? Let me try the cleaner split. Put an *actor* on each node whose job is to produce the artifact, and put a *critic* on each edge whose job is to take the upstream actor's artifact and issue an instruction for improving it. An edge connects exactly two artifact-producers, an upstream and a downstream one — and "the thing that lives between two producers and tells the downstream one how to improve on the upstream one" is *exactly* a critic. The bipartition isn't decoration; it's the natural division of labor the working collaboration already has, made structural. Write the agentization as an operator: a_i = rho(v_i) puts an actor on node v_i, a_ij = rho(<v_i, v_j>) puts a critic on edge <v_i, v_j>, where rho wraps a foundation model with a role profile and memory. A graph with |V| nodes and |E| edges thus deploys |V| + |E| agents — far more agents than nodes once the graph is dense, which is a feature: the critics are where the refinement pressure comes from.

With agents pinned to nodes and edges, I need an order to run them in. A node's actor can't sensibly act until it has the artifacts of everything pointing into it — its inputs. That's a dependency constraint, and the dependency constraint over a DAG has a name: a topological order, where every node is visited only after all its predecessors. So I traverse in topological order, and within that order each edge runs as a small dialogue: the upstream actor a_i hands its artifact to the edge critic a_ij; a_ij reflects and instructs; then a_ij passes the instruction to the downstream actor a_j, which refines. Formally the order has to interleave them — for every edge <v_i, v_j>, the actor of i comes before the critic of ij, which comes before the actor of j: I(a_i) < I(a_ij) < I(a_j). The per-edge interaction is a dual-agent multi-turn loop — request, reply, request, reply, capped at a few rounds — so each edge implements an iterative critique-and-refine, which is exactly multi-agent self-refinement: one actor's draft, run through a critic, becomes the next actor's improved draft. Each edge needs at least two such turns, so a graph runs on the order of 2|E| interaction rounds.

I deliberately set up the critic on the edge and the actor on the node before deciding anything about the *shape* of the graph, because now I get to ask the real question — which shapes are good? — with all the machinery shape-independent. But first I have to confront the cost problem I keep deferring, because it decides which shapes are even *reachable*.

Here is the trap. Suppose every agent, when it acts, gets to see the entire transcript of every interaction it depends on — all the back-and-forth, all the reasoning, not just the final drafts. Picture the densest interesting graph, a mesh where every earlier agent connects to every later one, and look at the agent at the very end, the sink, the one that depends on everyone. How much does it have to read? It inherits the dialogue of every interaction upstream of it, and in a fully connected graph of n agents the number of pairwise interactions is on the order of n choose 2, which is n(n-1)/2. So the sink's context grows like n^2. Let me make that concrete with the lengths involved: call t the task length, p the profile length, i the average instruction length, s the average artifact length, m the cap on rounds per adjacent pair. Without any pruning, the sink's total token load is roughly

  O(n)_{w/o} = t + p + s + (2m - 1)(i + s) ( n(n-1)/2 + 2(n-2) ),

where the (2m-1)(i+s) is the token cost of one fully-unrolled adjacent interaction (instructions plus artifacts across its rounds), the n(n-1)/2 counts the mesh's internal edges, and the 2(n-2) accounts for the sentinel hookups at the ends. For large n the dominant term is the product of (2m-1)(i+s)/2 with n^2 — call the constant C ≡ (2m-1)(i+s)/2 — so the load is approximately C·n^2, growing with the square of the population. That is fatal. Quadratic context means token cost and latency that blow up super-linearly, and an absolute context length that smashes through the model's window. This is *why* the hierarchical systems fell over past a few dozen agents — the thing I told myself to hold onto earlier. If I do nothing about this, I can never reach hundreds of agents, let alone a thousand, and the scaling study I want is dead on arrival. Wall.

So what is actually worth propagating? When an actor finishes an interaction, the valuable output is the *refined artifact* — the new draft. The detailed back-and-forth that produced it, the critic's full reasoning, the dead ends — those mattered for *making* the artifact but they don't need to travel onward; whatever was learned is already baked into the draft. So split memory: short-term memory holds the working context *within* a single interaction, and long-term memory keeps only the *final artifact* of each completed interaction, discarding the transcript. Then the only thing that flows along an edge to the next agent is one artifact, not a growing pile of dialogues. Now redo the sink's accounting. With only artifacts propagating, the sink no longer inherits n^2 worth of dialogue; it sees, per its dependencies, a bounded amount of artifact-and-instruction. Tracking the same terms,

  O(n)_{w/}  = t + p + s + m(i + s) ( (n - 1) + 2(n - 2) ),

where the edge-count factor is now linear in n rather than quadratic — the (n-1) and 2(n-2) replace the n(n-1)/2 — because each agent passes forward a single distilled artifact instead of an unrolled transcript per pairwise interaction. For large n this is approximately Cbar·n with Cbar ≡ 3m(i + s): *linear*. Propagating only the artifact decouples context length from quadratic to linear, and linear is the difference between a few dozen agents and a thousand. The memory mechanism isn't a tidiness nicety; it's the precondition for any of the dense, interesting shapes to be runnable at scale.

But propagating only the artifact creates a new little problem, and it's the one Graph of Thoughts already told me to expect. If a node has *several* incoming edges — if work converges on it — then it receives several artifacts, several distinct upstream drafts. With the transcript thrown away, it can't reconstruct a merged conversation; it has just the drafts. So a convergent node must *aggregate*: fuse the incoming artifacts into one before it refines. Not pick the best, not concatenate — fuse, taking the strengths of each into a single combined artifact, the "non-linear" merge that Graph of Thoughts argued a graph buys you over a tree. So the dynamics are: at a divergent node, the artifact branches outward to several successors, each of which refines it from a different angle; at a convergent node, the several incoming artifacts are aggregated into one, then refined. Divergence spreads diversity; convergence synthesizes it. And convergence is the operation that actually turns parallel breadth into improved quality — without it, breadth is just unused parallelism, the star's wasted variants.

Now I finally get to the question I built all this to ask: of the shapes living inside the DAG, which one should I reach for when I want *balance*? Let me lay out the extremes I already understand. The chain is pure depth: n agents, depth n, breadth one, every node has exactly one predecessor so there's never any aggregation, never any diversity — a single refining thread. The star is pure breadth: depth essentially two, breadth n, one round of parallel exploration with at most a single convergence at the end. The mesh is pure density: every earlier node feeds every later node, which is maximal aggregation but also the most expensive, and — I should be honest — overwhelming, because a node that must fuse *everyone's* artifact faces information overload that can impede rather than help refinement. None of these is balanced. The chain has no width; the star has no depth; the mesh has too much of everything.

What I want is a shape that interleaves the two virtues: rounds of parallel diverse refinement (width), stacked into successive stages of iterative improvement (depth), with convergence at every stage so diversity is repeatedly synthesized rather than merely produced once. Stand back and that description is a familiar object — it is exactly the layout of a multilayer perceptron. Partition the agents into ordered *layers*; within a layer the agents act in parallel (they don't feed each other, so they're independent diverse refinements — width); between one layer and the next, connect them so artifacts flow forward stage by stage (depth); and make the inter-layer connection *full* — every agent in a layer feeds every agent in the next — so each node in a downstream layer is convergent over the *entire* previous layer and aggregates all of its diversity before refining. That single design choice, full bipartite connection between adjacent layers, is what makes every non-source agent a heavy aggregator: it sees the whole spread of the prior stage and fuses it. A sparser inter-layer connection would let some downstream agents see only part of the previous layer's diversity, wasting exactly the synthesis I'm building this shape to get. Full connection between layers, then.

How many layers, and how wide? This is where the memory mechanism reaches back and constrains me, and I should take it seriously rather than picking a number. Recall that propagating only the artifact, while it saves me from quadratic context, has a cost: it can make *deep* topologies lose track of distant agents — by the time an artifact has been refined through many stages, the early context is long gone, and a deep structure can suffer artifact "rollbacks," a later agent effectively undoing good work it can no longer see the justification for. So for a fixed number of agents I should prefer *shallow and wide* over *deep and narrow*. I want few layers and many agents per layer. A chain is the pathological deep case — n layers of one. I want to crush the depth without collapsing into the star's single burst of breadth. The natural shallow-but-not-trivial choice is to let the number of layers grow only *logarithmically* in the population: set the layer count to about log2 of the number of agents. Then depth grows like log N while width grows like N / log N — overwhelmingly wide, gently deep, which is exactly the regime where memory control is least likely to lose distant agents and where the per-stage aggregation has lots of diversity to fuse. Logarithmic depth is the sweet spot between the chain's fatal depth and the star's depthlessness.

Let me also sanity-check the cost of this shape so I know it's reachable. The edges are the sum over adjacent layer pairs of (size of layer)·(size of next layer). With about log N layers each of width about N/log N, that's roughly (log N)·(N/log N)^2 = N^2 / log N edges — comfortably below the mesh's N^2/2, and above the chain's N. So the layered shape is denser than a chain, meaning more interaction and more refinement opportunity, but cheaper than a mesh, and it avoids the mesh's information overload by only ever asking a node to aggregate one layer's worth, not everyone's. It sits in the moderate-density middle, which is where I'd bet balance lives.

Now make the layer partition concrete, because the construction has to handle any N. Set the number of layers L = floor(log2 N). Divide the N agents as evenly as possible: each layer gets floor(N / L) agents, and the remainder N mod L — the few agents that don't divide evenly — I dump into the first layer. Why the first? The first layer is the source layer; its agents have no peer predecessors, so they're the pure parallel-exploration stage, the broadest sampling of independent starting drafts. Making the source layer the widest is exactly right: more independent initial attempts means more diversity entering the network, and everything downstream is aggregation and refinement of that initial spread. Then I lay the agents out in index order — agents 0..(size of layer 0 - 1) are layer 0, the next block is layer 1, and so on — compute each layer's start and end index by accumulating the sizes, and wire every node in layer i to every node in layer i+1. Because agents are numbered layer by layer in order, every edge automatically runs from a lower index to a higher index, so the index order is already a valid topological order — no separate sort needed, and the DAG-ness is free.

Let me actually walk the small case I'll run by default, N = 4, to make sure the construction does something sensible and not degenerate. L = floor(log2 4) = 2 layers. Each layer gets floor(4/2) = 2 agents, remainder 4 mod 2 = 0, so layers are [2, 2]. Layer 0 is agents {0, 1}, layer 1 is agents {2, 3}. Full bipartite between them: 0->2, 0->3, 1->2, 1->3 — four edges. That's a clean 2x2 block: two agents draft independently in parallel (diversity), then each of two downstream agents aggregates *both* drafts and refines (synthesis), and the runtime collects the two refined artifacts at the output sentinel. Depth two, width two, every downstream node a full aggregator — width and depth both present at the smallest interesting size. Good.

Now I have to be careful at the very bottom, because the logarithm bites. The canonical body is meaningful once there are at least two agents. If N = 2, then L = floor(log2 2) = 1 — a single layer. With one layer there are no *adjacent* layer pairs, so the inter-layer wiring produces *zero* edges: the two agents just sit in one layer with no peer edges between them. N = 3 is the same, floor(log2 3) = floor(1.58) = 1, one layer, no edges. For those two cases the runtime can still connect every source and sink through its sentinels, but the layered topology itself is vacuous. N = 1 is different: floor(log2 1) = 0, so the raw multi-layer body has no layer count to divide by; that is the separate single-agent variant, and a standalone scaffold should return an empty edge list before entering the canonical body. So the first non-vacuous layered case is N = 4. L = floor(log2 4) = 2 gives [2, 2] and four edges. N = 8 gives L = 3, sizes floor(8/3)=2 each plus remainder 8 mod 3 = 2 into the first, so [4, 2, 2] — a wide source layer of four feeding two stages of two, twelve edges. N = 16 gives L = 4, [4,4,4,4], forty-eight edges. The pattern holds: a fat front layer, logarithmically few stages, every downstream node aggregating its whole predecessor layer.

One more reflection on *why* I expect this balanced shape to be worth having, derived rather than asserted. Every node beyond the first layer is convergent — it has an entire layer of predecessors. Convergent nodes are the aggregators, and aggregation is where an artifact gets revised by being forced to reconcile several incoming drafts; the more convergent a node, the more often and more substantially its artifact gets revised, and revised artifacts tend to grow longer and more complete as they absorb the strengths of what they fuse. So a topology thick with convergent nodes should produce markedly more revision and longer, richer artifacts than a chain, where no node is ever convergent and each artifact is touched by exactly one refinement per step. That's the upside, and it's structural: I built convergence into every layer transition on purpose. The matching downside I should keep in view is that aggregation is *harder* than divergence — fusing several artifacts into one coherent improved artifact is a more demanding act than branching one artifact into several refinements, so heavy convergence is where the quality risk concentrates. The layered shape leans into convergence deliberately to get the revision-and-length benefit; the alternative shapes that lean toward divergence trade some of that synthesis for easier, safer propagation. There's no free lunch among the shapes — which is the entire reason I want the general DAG machinery, so I can pick the shape to fit the task's openness and my compute budget rather than committing to one.

So let me write the edge generator — the one empty slot — for the layered shape, grounded in the construction I just walked: log-based layer count, even split with the remainder in the front layer, cumulative indices, full bipartite wiring between adjacent layers, returning the directed edges.

```python
import math


def generate_topology(node_num: int) -> list[tuple[int, int]]:
    """Layered (MLP-shaped) collaboration DAG over agents 0 .. node_num-1.

    Agents are partitioned into ~log2(node_num) ordered layers of roughly equal
    size; every agent in a layer feeds every agent in the next layer. Within a
    layer the agents act in parallel (diverse, independent refinements -> width);
    successive layers stack iterative refinement (depth); each non-source agent is
    convergent over its entire previous layer, so it aggregates that layer's
    diversity before refining (synthesis). Wide-and-shallow by design so the
    artifact-only memory mechanism does not lose distant agents.
    """
    if node_num <= 1:
        return []

    # depth grows only logarithmically: few stages, wide layers
    layer_num = int(math.log(node_num, 2))

    # split agents as evenly as possible across layers...
    layers = [node_num // layer_num for _ in range(layer_num)]
    # ...and dump the remainder into the source layer, making the broadest
    # parallel-exploration stage at the front (most independent initial drafts).
    layers[0] += node_num % layer_num

    # cumulative [start, end) index range for each layer; agents are numbered
    # layer by layer, so every edge runs low-index -> high-index (a valid
    # topological order for free, and the DAG property is automatic).
    end_ids, start_ids = [layers[0]], [0]
    for i in range(1, len(layers)):
        start_ids.append(end_ids[-1])
        end_ids.append(end_ids[-1] + layers[i])

    # full bipartite connection between adjacent layers: every node in layer i
    # feeds every node in layer i+1, so each downstream node aggregates the whole
    # previous layer's spread of artifacts.
    edges = []
    for i in range(len(layers) - 1):
        for u in range(start_ids[i], end_ids[i]):
            for v in range(start_ids[i + 1], end_ids[i + 1]):
                edges.append((u, v))
    return edges
```

The causal chain is now complete. I need to scale collaboration the way deep learning scales neurons, but the existing systems fix a tiny agent count and give me no scalable, swappable structure: the chain is pure depth with no breadth or scaling knob; the star is pure breadth with no refinement; trees are brittle under large contexts; Tree of Thoughts is a rigid tree at the wrong granularity; Graph of Thoughts shows that a *graph* lets parallel branches reconverge and be aggregated but allows cycles and works over a single model's thoughts; GPTSwarm validates the agents-as-DAG abstraction but spends its effort *learning* one task-specific wiring rather than holding structure fixed and scaling it. Pulling out the common object, they are all edge sets in a directed acyclic graph, so I use the DAG itself — acyclic for a clean task-independent order and no backflow — as the general organizing principle, put a producing actor on every node and a critiquing critic on every edge to make the working draft-then-fix division of labor structural, and run the agents in topological order with each edge a dual-agent refine loop. The killer constraint is context: letting agents read full transcripts makes the sink's load grow like n^2, so I propagate only the final artifact of each interaction — dropping the leading context term from C·n^2 to Cbar·n, the linear regime that makes large agent populations feasible — and, because pruning transcripts means a convergent node receives several bare drafts, I aggregate incoming artifacts into one before refining, which is what turns parallel breadth into synthesized quality. Then, wanting balance rather than pure depth, breadth, or density, I use the multilayer-perceptron layout: ordered layers, parallel within a layer for width, stacked for depth, fully connected between adjacent layers so every downstream node aggregates its whole predecessor layer. Memory control's penalty on deep structures keeps the depth shallow, so I set the layer count to about log2 of the population — depth ~ log N, width ~ N/log N — split the agents evenly with the remainder fattening the source layer, index them layer by layer so the numbering is already a valid topological order, and wire full bipartite blocks between adjacent layers, yielding roughly N^2/log N edges, denser than a chain and cheaper than a mesh. The edge generator drops straight into the runtime's one open slot.
