I start with the thing that makes hidden units hard. If a network has only input units connected straight to output units, the error at an output can tell me directly how to change the connecting weights. But as soon as I insert hidden units, the training data no longer tells me what those hidden units should do. The only quantity I can compare with the desired answer is the final output, and yet I need a direction for every weight below it. I need the scalar error at the top to assign credit to all the internal computations that helped create it.

The most literal route is to perturb one weight, run the whole network again, and see whether the error goes up or down. For weight w_i I can use (E(w + h e_i) - E(w)) / h. That is tempting because it treats the network as a black box, but the cost is wrong in exactly the case I care about. If there are P weights, one gradient step costs P extra forward evaluations, and the step size h adds its own numerical compromise. Making h small reduces truncation error but increases cancellation. This is not a learning rule for a large network; it is a diagnostic.

I can also try to write the derivative formulas out symbolically. For one or two layers, that is tolerable. But a network is a composition with shared intermediate values. If I differentiate the full expression as an expression, products duplicate their subexpressions, nested products duplicate them again, and the derivative expression can become much larger than the computation I started with. Worse, every change in the architecture asks me to do the algebra again. I do not want a giant formula. I want a procedure that follows the computation itself.

Forward sensitivity is closer. While I evaluate every unit, I can carry along the derivative of that unit with respect to one selected weight. Addition, multiplication, and the nonlinearity all have local derivative rules, so the chain rule pushes this sensitivity forward cleanly. But this still has the wrong asymmetry. One forward sensitivity sweep answers one question: how does the output change if this one input direction changes? I have many weights and one scalar error. Carrying one direction at a time makes the cost scale with the number of weights.

So I should stop looking at the computation from the weights outward. The output is one scalar. Every intermediate value matters only through its effect on that scalar. Let the computation create intermediate values v_1, v_2, ..., v_N, with the loss L as the final scalar. For any intermediate v_i, the quantity I want to track is not "how does v_i change when one particular weight changes?" but "how does L change when v_i changes?" Write that as vbar_i = dL/dv_i. If I know vbar for a child value v_j that directly used v_i, then the contribution passing through that child is vbar_j * dv_j/dv_i. If v_i feeds several later values, the chain rule says the effects add:

  vbar_i = sum over children j of vbar_j * dv_j/dv_i.

This looks like the chain rule with the order of traversal reversed, but I do not yet trust that the "effects add" step and the reversed order actually reproduce the true gradient — that is the part that could be wrong, and it is exactly the part forward sensitivity gets for free. The case that worries me is a value used more than once, because that is where the accumulation has to happen and where a naive single-pass could double-count or miss a path. So before I build anything around this, I want to run the recurrence on a concrete computation small enough to differentiate three independent ways and compare.

Take a = 2, b = -3, and a computation that deliberately reuses an intermediate:

  u = a * b
  v = tanh(u)
  L = v + u.

Here u is used twice — once directly in L and once through v — so it is the shared-subexpression case I am unsure about. Numerically u = -6, v = tanh(-6) = -0.99998771, and L = v + u = -6.99998771.

Now the reverse sweep. Initialize Lbar = dL/dL = 1. The last operation is L = v + u, an addition, which sends its adjoint unchanged to both inputs, so vbar = 1 and ubar starts at 1 (the direct path). Next is v = tanh(u), which sends vbar * (1 - v^2) into u, accumulated on top of what is already there:

  ubar = 1 + 1 * (1 - v^2) = 1 + (1 - 0.99997542) = 1.00002458.

Then u = a * b sends ubar * b to a and ubar * a to b:

  abar = ubar * b = 1.00002458 * (-3) = -3.00007373,
  bbar = ubar * a = 1.00002458 *  2  =  2.00004915.

I want to know whether those two numbers are actually dL/da and dL/db. Forward sensitivity gives me an independent route. Pushing the direction da=1, db=0 forward: du = b = -3, dv = (1 - v^2) du = -0.00007373, dL = dv + du = -3.00007373. Pushing da=0, db=1: du = a = 2, dv = (1 - v^2) du = 0.00004915, dL = dv + du = 2.00004915. These match the reverse-sweep abar and bbar to every digit. And central finite differences with h = 1e-6 give dL/da = -3.0000737 and dL/db = 2.0000492, agreeing to the precision finite differences can offer. Three independent computations land on the same gradient, and the place where they could have diverged — the doubly-used u — is exactly where the reverse sweep had to add 1 (direct path) and (1 - v^2) (path through v). If I had overwritten instead of accumulated, ubar would have been (1 - v^2) ≈ 0.0000246, and abar would have come out near zero instead of -3. So the accumulation is doing real work, and it produces the right answer. That settles the worry: visiting the operation list in reverse and accumulating each child's contribution reconstructs the true gradient.

The reason the reversed order is safe is topological: when I sweep the operation list backward, every child of v_i is processed before v_i, so by the time I reach v_i it has already received the total effect on L from every path through every child, and I can read it off as final. I initialize the final scalar with Lbar = 1, and then every operation sends its adjoint to its parents. An addition z = a + b sends zbar to both a and b. A multiplication z = a b sends zbar * b to a and zbar * a to b. A nonlinearity z = phi(a) sends zbar * phi'(a) to a. Shared subexpressions are not duplicated; their incoming contributions are summed in one number, as I just watched happen at u. The derivative computation is dynamic programming over the graph.

Now I can ask whether the textbook neural-network delta formulas are this same sweep written in layer notation, or something I have to derive separately. For an output unit j with loss contribution 1/2 (y_j - d_j)^2, the adjoint on y_j is dE/dy_j = y_j - d_j. If y_j = phi(x_j), then pushing that adjoint back through the activation gives the sensitivity to the unit's total input,

  delta_j = dE/dx_j = (y_j - d_j) phi'(x_j).

The total input is x_j = sum_i w_ij y_i + b_j. The weight w_ij appears locally as the product w_ij y_i, whose partial with respect to w_ij is y_i, so the adjoint sweep deposits

  dE/dw_ij = delta_j y_i.

The lower unit output y_i feeds the loss through every outgoing connection, so its adjoint accumulates over those connections, dE/dy_i = sum_j delta_j w_ij, and if y_i = phi(x_i),

  delta_i = dE/dx_i = phi'(x_i) sum_j w_ij delta_j.

I do not want to take it on faith that these layer-level formulas are correct just because they have the shape of the graph sweep, so I check one link numerically. Take a hidden unit i feeding one output unit j, with x_i = 0.5, w_ij = -1.5, b_j = 0.3, desired output d = 1, and phi = tanh. Forward: y_i = tanh(0.5) = 0.46212, x_j = w_ij y_i + b_j = -0.39318, y_j = tanh(x_j) = -0.37409, E = 1/2 (y_j - d)^2. Then delta_j = (y_j - d) phi'(x_j) = (-1.37409)(1 - y_j^2) = -1.18179, and the formula gives dE/dw_ij = delta_j y_i = -0.54613 and delta_i = phi'(x_i) w_ij delta_j = (1 - y_i^2)(-1.5)(-1.18203) = 1.39413. Differencing E directly: perturbing w_ij gives dE/dw_ij = -0.54613, and perturbing x_i gives dE/dx_i = 1.39413. Both match the formulas. So the layer recurrence is the graph sweep specialized to affine-plus-nonlinearity, not a separate fact — start with the output errors, convert them to deltas by the activation derivative at the already-computed forward activations, move one layer down, sum the weighted deltas from the layer above, and read weight gradients as "receiving unit delta times sending unit activity." Every edge is used once forward and once backward. I never perturb a weight, and I never build a symbolic derivative expression; I reuse the values the forward pass already produced and send one scalar sensitivity backward through each local operation.

The storage requirement follows from what the local derivatives need. The local derivative at a node usually depends on the forward value there: phi'(x_i) needs x_i or y_i, and d(w_ij y_i)/dw_ij needs y_i. So the forward pass must leave behind the activations or an operation tape. If I unroll an iterative network through time, the backward pass needs the history of unit states at each time step; if the same weight is reused at multiple unrolled positions, the gradient for the actual tied weight is the sum of the gradients for all its occurrences — the same accumulation I verified at u, now across time. That is the price of the backward sweep: memory for the path the computation took. The payoff is that one scalar loss gives me all parameter derivatives in one reverse traversal.

This also tells me the method is not really about neural networks. A layered net is only one kind of directed acyclic computation once it is evaluated. The general rule is: run the program forward, record the intermediate values and dependency edges, set the output adjoint to one, traverse operations in reverse topological order, and for each operation add output_adjoint times the local partial derivative into each input adjoint. If the original program costs C elementary operations, the backward pass touches the same local operations in reverse and costs only a small constant multiple of C. For a function from many parameters to one loss, that turns the gradient cost from "one forward-like sweep per parameter" into "one forward pass plus one backward pass" — which is the asymmetry that made forward sensitivity the wrong tool here.

The learning rule is then plain gradient descent in weight space. Accumulate dE/dw over one case or over a batch of cases, and change each weight by -eta dE/dw. A momentum term can update a velocity rather than the position directly, and small random initial weights break symmetry so hidden units do not all receive identical signals. Those are optimization details. The core computation is the adjoint sweep: one scalar error at the end, local chain-rule messages moving backward, all partial derivatives collected at the leaves. I can now write the method as a small graph algorithm: topologically order the values from the forward evaluation, zero every sensitivity, set the loss sensitivity to one, visit the values in reverse order, and add each child's sensitivity times each local partial into the parent; the adjoints sitting on the parameter leaves are exactly the gradient components I checked above.
