Let me start from the thing that actually hurts, because everything else in cryptography seems solved and this one piece refuses to be. Two people want to talk privately over a wire someone is recording. The whole apparatus we have — a cipher family `{S_K}`, the transmitter sending `C = S_K(P)`, the receiver computing `S_K^{-1}(C) = P` — works beautifully the instant both of them hold the same `K`. And that is exactly the catch. They have to *already* share `K`, and `K` has to get to both of them over some channel the eavesdropper can't touch: a courier, registered mail, a trusted operator carrying an attaché case chained to his wrist. The secure channel exists only to carry the key; it's too slow and too costly to carry the message. So the secret message rides the cheap public wire and the secret key rides the expensive private one, and the private one is the bottleneck for the whole edifice.

For a pair of old correspondents that's fine — they met once, they planned ahead. But picture what cheap digital hardware is about to make ordinary: a terminal calling a bank, a cash dispenser, electronic mail between firms that have never dealt with each other, two strangers who want to do business *this afternoon*. They cannot wait for a courier. And it's worse than one awkward courier trip, because the cost is combinatorial. In a network of `n` subscribers, any of the `n(n-1)/2` pairs might someday want to speak in private. A hundred users is already about five thousand keys; a thousand users, half a million; the whole telephone system, something like twenty million billion. You cannot ship that many keys in advance, and you cannot make every first contact wait. The key-distribution channel is the single barrier standing between cryptography and the networked world.

So I want to ask the question that the field has quietly treated as closed: do the two parties *really* have to share a secret in advance? Stated baldly it sounds impossible — if they could somehow send a secret key over the open wire without ever having met, why couldn't they just send the secret *message* the same way and skip the key entirely? The question seems to answer itself in the negative. But "seems to answer itself" is exactly the kind of assumption I should distrust, because it's an assumption about the *structure* of the problem, not a theorem.

Let me look hard at what we're actually allowed to assume, because Shannon already drew the only firm line here. He showed that *perfect* secrecy — the cryptogram telling the enemy nothing, the a-posteriori message probabilities exactly equal to the a-priori ones — is achievable, but only if the key carries at least as much entropy as the message, `H(K) ≥ H(M)`. If you want to send a long message in perfect secrecy you need an equally long key, the one-time pad, provably unbreakable and hopelessly impractical because the key is as long as everything you'll ever say. So nobody actually uses unconditional security. Everybody uses a few hundred bits of key and accepts something weaker: the system is secure not because the cryptogram is information-theoretically empty, but because *recovering the message costs too much to bother*. The security has quietly migrated from information to computation. That migration is the crack I want to push on. Once "secure" means "computationally infeasible to break," the whole game is about the *gap between what's cheap to do and what's ruinously expensive to undo* — and that is a statement about algorithms, not about entropy.

And there's a hard ceiling I have to be honest about up front, because it tells me what kind of solution I can possibly be looking for. Whatever I build, if the public information determines the secret information uniquely among a finite set — and it must, if the receiver is going to recover a definite message — then an opponent with *unlimited* computation can always just search that finite set. So nothing of this kind can ever be unconditionally secure. I am not going to beat Shannon. I am going to *trade* his unbreakability for a computational wall: a thing that is feasible for the legitimate parties and infeasible, by some enormous factor, for the eavesdropper. Good. That's the right register, and it tells me where to look — at the new theory of the cost of computation.

Because that theory now exists. Complexity theory has started sorting problems by difficulty: `P` for what a deterministic machine does in polynomial time, "easy"; `NP` for what you can *verify* in polynomial time, or equivalently find in polynomial time if you had unlimited parallelism. And Karp gave us twenty-one `NP`-complete problems — travelling salesman, satisfiability, graph colouring, the knapsack — all interlocked so that a fast algorithm for any one cracks them all, and everyone believes there is none. That is *precisely* the shape I want: a thing easy to do forward, believed infeasible to undo. I'll keep one worry in my pocket — `NP`-completeness is worst-case, and a cipher needs a problem hard on the *typical* random instance, not just on some adversarial one — but the vocabulary is finally here.

Now, where in current practice does this "easy forward, hard backward" asymmetry already quietly do useful work? It's sitting in the login system. The clever thing Needham's password scheme does is store not the password `PW` but its image `f(PW)` under a function easy to compute and infeasible to invert. At login you compute `f` of what's typed and compare. Steal the whole password table and you've got nothing you can use — the entries aren't passwords, and the login routine itself rejects them. Evans and Kantrowitz and Weiss, and Purdy, have been publishing constructions for such functions; Purdy's are sparse polynomials of enormous degree over a finite field, whose roots cost far more to find than the polynomial costs to evaluate. So a *one-way function* — easy one way, infeasible the other — is a real object, not a fantasy.

Let me also stare at the *other* problem on my desk, the one that looks unrelated: signatures. A paper contract is signed; the signature is hard to forge and the holder can wave it in front of a judge. A digital message can be copied perfectly, bit for bit, so on its face a "digital signature" is a contradiction — anything I can recognize, I can copy, so how can it be unforgeable? And our current authentication doesn't really solve it. We append an authenticator that's a function of the message and a *shared* secret key, plus the date and time. That stops a third party from forging, fine. But the receiver holds the same key — so *he* could have manufactured the authenticator himself. Against a dispute between sender and receiver it's useless, and disputes are the whole reason signatures exist: the broker forging a client's order, the client disavowing the order that lost money. Lamport showed a partial fix with a one-way function — publish images of pairs of secret vectors, reveal one per message bit — but it blows the message up a hundred-fold and it's clearly a workaround, not the idea.

Here's what I keep circling. Look at the two real defenses we have and notice they protect *opposite* flanks. The IFF challenge-and-response — radar throws a random challenge, the aircraft must return the correctly encrypted reply, and because the challenge never repeats a recorded reply is worthless next time — that beats an *eavesdropper* who'd replay, but it falls apart the moment someone *captures the radar* and reads its key, because then he can fool any radar keyed the same. The login one-way function is the mirror image: capture the password table (the analogue of capturing the radar) and you still can't log in, because the table holds `f(PW)`, not `PW` — but the login message itself never changes, so anyone who *intercepts* it walks right in. Challenge/response survives interception and dies to capture; the one-way function survives capture and dies to interception. Each one guards exactly the flank the other exposes.

So what would it take to guard *both at once*? In the IFF picture, the challenger throws a question and judges the answer. What if the challenger could throw a question *it cannot itself answer*, yet *can* still judge? Then capturing the challenger gives you nothing — it doesn't know the answers either — and the changing challenge already kills replay. Hold that thought against the login one-way function `f`: easy forward, hard back. The challenger issues a value `y` in the *range* of `f` and demands a pre-image `x` with `f(x) = y`. Nobody can answer that — `f` is one-way — *unless* there's some private back-door knowledge. And the challenger, holding the forward algorithm for `f`, can check any claimed answer instantly by computing `f(x)` and seeing if it equals `y`.

A plain one-way function is not enough; I want a one-way function with a *secret way back* — a one-way function that someone in possession of a particular piece of secret information can run *backwards*. Call it a trap-door one-way function. The holder of the trap-door can produce pre-images on demand; everyone else, even holding the entire forward algorithm in the open, cannot. And the instant I say that, both of my hard problems dissolve into one object, and they were never really two.

If the "challenge" is not random noise but the *message itself*, the signature falls out. To sign, I — the trap-door holder — run the function *backwards* on the message, producing the unique pre-image only I could produce. Anyone, holding the public forward algorithm, runs it forward and checks that it lands on my message. They can *verify* the signature and they cannot *forge* it, because forging means inverting the one-way function without the trap-door. A signature that's easy to recognize and infeasible to counterfeit, on a perfectly copyable digital message — the thing that looked contradictory. It even settles disputes: the verifier can check but can't generate, so he can't have faked it, and a judge can run the same public check.

Now turn the same object the other way and key distribution falls out too. To send *me* a private message, you run the *forward* function — which is public, anyone can — on your message. Only I, with the trap-door, can run it backwards and read it. The forward direction can be published to the entire world without leaking the backward direction, because that's exactly what one-wayness *means*. So I can print my forward function in a directory, next to my name and address, like a telephone listing. Anyone can encrypt to me; only I can decrypt. The key to encrypt and the key to decrypt have come apart — encryption is *public*, decryption is *private*, and computing the private one from the public one is infeasible. That separation is the whole thing. It's why I'd call this *public-key* cryptography: the enciphering capability is a thing you publish.

I want to be careful about what the discovery actually *is* here, because it isn't a construction. I don't yet have a single concrete trap-door function I trust. What I have is the recognition that the two problems — key distribution and signatures — each of which seemed unsolvable by its very definition, can be solved *at all*, and that the solutions come in *one package*: invert a publishable one-way function with a private trap-door. Two strangers can now hold a private conversation with no prior contact, each encrypting under the other's public function. The directory replaces the courier. The combinatorial key-distribution nightmare evaporates: every user generates *one* pair locally and publishes half of it; there is nothing to pre-share with anyone.

Let me pin the definitions down precisely, because the precision is load-bearing and I already see one place it's easy to get wrong. A public-key cryptosystem is a pair of families `{E_K}` and `{D_K}` of invertible transformations on a finite message space `{M}`, with four properties. One: for every `K`, `E_K` is the inverse of `D_K`, so `D_K(E_K(M)) = M`. Two: for every `K`, both `E_K` and `D_K` are easy to compute. Three: for almost every `K`, it is computationally infeasible to derive `D_K` — or any algorithm equivalent to it — from `E_K`. Four: for every `K`, it is feasible to generate the inverse pair `(E_K, D_K)` from `K`. Property three is what lets me publish `E_K` and keep `D_K` secret. Property four is what lets each user roll his own pair from a single random `K` drawn off a noisy diode, so no one ever has to hand anyone a key. The public directory of `E_K`'s doesn't even need read-protection — quite the opposite, I *want* it read — only protection against someone *rewriting* an entry, which is cheap since the file changes rarely.

And now the subtle definition, the one-way function itself, where the obvious phrasing is wrong. `f` is one-way if for any `x` in the domain `f(x)` is easy to compute, yet for almost all `y` in the range it is computationally infeasible to solve `y = f(x)` for *any* argument `x'` with `f(x') = y`. The trap I must avoid: this is *not* the ordinary mathematical sense of "non-invertible," where the inverse fails to be unique. Non-uniqueness would *help* the attacker, not hurt him — if many `x` map to the same `y`, he only needs to find *one* of them, and that's easier. The degenerate extreme makes it vivid: if `f(x) = c` for every `x`, then any `x` is a pre-image and inversion is trivial. So one-wayness can't mean "the inverse isn't a function." It must mean: given `y` and the public `f`, it is overwhelmingly hard to produce *any* `x` at all that `f` maps to `y`. A small amount of many-to-one is tolerable, even expected, but the function mustn't be too degenerate. Getting this definition backwards would have me building the easiest functions to break and calling them hard.

A trap-door one-way function, then, is a function that genuinely *does* have an easily computed inverse — so it's not really one-way in the strict sense — but for which, given only an algorithm for the forward direction, finding that easy inverse is infeasible; only through certain trap-door information (say, the random bit string that generated the `(E, D)` pair) can the easy inverse be recovered. A public-key cryptosystem *is* exactly a set of trap-door one-way functions. The picture I keep is a combination lock: anyone who knows the combination opens it in seconds, while a skilled locksmith needs hours — and yet if the owner *forgets* the combination he's no better off than the locksmith. The trap-door is the combination; lose it and you become an ordinary outsider to your own function.

The pieces order themselves in a useful way, and the ordering tells me which problem is genuinely the hard one. A cryptosystem secure against known-plaintext attack already gives me a one-way function for free: fix a plaintext `P_0` and define `f(K) = S_K(P_0)`; inverting `f` means finding the key from a single known plaintext-cryptogram pair, which is exactly what "secure against known plaintext" forbids. A public-key cryptosystem gives a one-way authentication system (sign with `D_A`, verify with `E_A`), but the converse doesn't appear to hold — so building a full public-key *cryptosystem* is strictly harder than building one-way authentication. And a public-key cryptosystem gives a public-key *distribution* system but not conversely. So the cleanest, most reachable target is the *weakest* of these: just let two parties agree on a shared key over the public wire. I don't need the full encrypt-to-anyone machinery for that; I need *less*.

So the question narrows to the most concrete possible thing: what is an actual function I can build this on? Three candidates are on my desk and I'll try each.

The first, just to fix ideas, is linear algebra. Take an invertible binary `n×n` matrix `E`, encipher the message vector `m` by `Em`, decipher with `D = E^{-1}`. Enciphering and deciphering each cost about `n^2` operations. And I can manufacture an `(E, D)` pair without inverting anything: start from the identity, do random elementary row and column operations to build an arbitrary invertible `E`, and run the same operations in reverse order on the identity to get `E^{-1}` — the sequence of operations is itself drawn from a random bit string. So far so good. But the eavesdropper's job — recover `D` from `E` — is just *matrix inversion*, and matrix inversion costs only about `n^3`. The ratio of cryptanalytic to legitimate effort is therefore at most `n`. To get a ratio of even `10^6` I'd need absurd block sizes, and there's no exponential gap anywhere in sight. It's useless as a cipher. But it's clarifying: it shows me *exactly* what's missing. The forward and backward problems here are both polynomial and only a factor of `n` apart. I need a pair where forward is polynomial and backward is *believed exponential*.

The second candidate is the knapsack, straight from Karp's `NP`-complete list. Define `f(x) = a · x` where `a = (a_1, ..., a_n)` is a known integer vector and `x` is a binary `n`-vector. Computing the sum is trivial, at most `n` additions. Inverting — given the sum `s`, find which subset of the `a_i` adds to it — is the knapsack problem, `NP`-complete, believed infeasible for `n` past a hundred or so. This is genuinely a one-way function, and it's tempting. But two things hold me back from it as the *key-agreement* answer. Its hardness is worst-case, and I'd have to choose the `a_i` carefully so a *random* instance is hard — if, say, the `a_i` are tiny the sum has few bits and the thing is wildly degenerate, easily inverted, and if `a_i = 2^{i-1}` inversion is just reading off binary digits. And more fundamentally, a one-way function alone gives me login and signatures, but it doesn't, by itself, let *two* parties *combine* their secrets into a *common* one. There's no composition law in `a · x` that two people can each contribute to. Knapsack is a fine one-way function; it is not, on its face, a meeting-in-the-middle.

The third candidate is exponentiation, and it's the one Gill keeps pointing me at — work in a finite field, raise a fixed base to a power, and lean on the fact that the inverse problem, the discrete logarithm, is considered very hard. Let me set it up properly. Take a prime `q` and a fixed *primitive* element `α` of the field `GF(q)` — primitive meaning its powers run through *all* the nonzero elements `1, 2, ..., q-1`. Define the forward map

    Y = α^X mod q,    for 1 ≤ X ≤ q-1.

Forward is cheap by repeated squaring: to get `α^X` I square and multiply down the binary expansion of `X`, at most `2·log_2 q` multiplications. Concretely `α^37 = α^(32+4+1) = ((((α)^2)^2...)^2)·(α^4)·α` — a handful of squarings, not thirty-seven multiplications. Backward is the discrete logarithm,

    X = log_α Y mod q,

and the best algorithm anyone knows costs on the order of `q^(1/2)` operations for a well-chosen `q`. So if `q` is a `b`-bit prime, forward is about `2b` multiplications — polynomial in `b` — while backward is about `2^(b/2)` — exponential in `b`. *There* is the gap the matrix never gave me. With `b = 200`: at most four hundred multiplications to go forward, against `2^100`, roughly `10^30`, operations to come back. The cryptanalytic effort grows *exponentially* relative to the legitimate effort. That's a different universe from the matrix's factor of `n`, and a different universe from Merkle's puzzles, which buy the honest parties only a *quadratic* edge — work `N` for them, `N^2` for the eavesdropper — an edge so modest that practical bandwidth limits cap the ratio around `10^4` and a determined opponent is only a square-root behind. Exponential beats quadratic decisively.

But I still have the knapsack's objection ringing: a one-way function gives signatures, not a *shared key*. So why is exponentiation different? Let me actually try to *use* it for agreement and see what happens, because this is where I keep getting stuck. Tables of exponentials never gave me a clean signature mechanism — that scheme failed — but Marty and I keep twisting exponentials around, turning them over, trying to make them fit something, because they feel too structured to discard. Let me push two secrets through the same base and stare at the algebra.

Suppose I pick a secret `X_A` and publish `Y_A = α^{X_A} mod q`. You pick a secret `X_B` and publish `Y_B = α^{X_B} mod q`. Now I take *your* public value and raise it to *my* secret:

    Y_B^{X_A} = (α^{X_B})^{X_A} = α^{X_B X_A} mod q.

And you take *my* public value and raise it to *your* secret:

    Y_A^{X_B} = (α^{X_A})^{X_B} = α^{X_A X_B} mod q.

Stare at the exponents. `X_B X_A` and `X_A X_B`. They're *equal* — multiplication of the exponents commutes. We land on the *same field element*, `α^{X_A X_B} mod q`, without ever having exchanged a secret. *That's* the structure the knapsack didn't have and the matrix didn't have: exponentiation carries a composition law, `(α^a)^b = α^{ab} = (α^b)^a`, so two people can each stamp their own secret exponent onto the public value, in *either order*, and meet in the middle. The one-wayness keeps the eavesdropper out; the *commutativity of the exponents* lets the two of us in. This isn't just a hard-to-invert function — it's a hard-to-invert function with exactly the algebraic handle two parties need to agree.

So here is the protocol, fully. We agree, in public, on a prime `q` and a primitive element `α` — these are not secret, anyone may know them. Each user `i` draws a random `X_i` uniformly from `{1, ..., q-1}`, keeps it secret, and places

    Y_i = α^{X_i} mod q

in a public file beside his name and address. When users `i` and `j` want a private key, each computes

    K_ij = α^{X_i X_j} mod q,

user `i` by raising the `Y_j` he reads from the file to his own secret `X_i`,

    K_ij = Y_j^{X_i} mod q = (α^{X_j})^{X_i} = α^{X_i X_j} mod q,

and user `j` symmetrically by `K_ij = Y_i^{X_j} mod q`. They now share `K_ij`, and they use it as the key in an ordinary fast symmetric cipher for the actual conversation.

What must the eavesdropper do? He sees `q`, `α`, `Y_i`, `Y_j`. He wants `K_ij = α^{X_i X_j}`. If he could take a discrete logarithm he'd recover `X_i` from `Y_i`, then compute `Y_j^{X_i}` and he's in — so if logs mod `q` are easy the system is broken, no question. The honest question is the converse, and I have to be scrupulous here: I do *not* have a proof that recovering `K_ij` from `Y_i` and `Y_j` *requires* taking a logarithm. Maybe there's a way to get `α^{X_i X_j}` from `α^{X_i}` and `α^{X_j}` without ever isolating an exponent. I don't see one — every route I can find goes through extracting `X_i` or `X_j` first — but "I don't see one" is not a theorem, and I'll say so plainly. The security *rests* on the difficulty of discrete logarithm mod `q`; if someone finds an algorithm whose cost grows like `log_2 q` instead of `q^(1/2)`, we're sunk. The flip side of that exposure is genuinely appealing, though: the problem statement is so simple and clean that it might equally well admit a *proof* of its difficulty someday — a hardness theorem of the kind cryptography has never had.

Let me make the security accounting concrete and also catch the sizing constraint, because the exponential gap forces `q` to be large. The legitimate work is one exponentiation, at most `2·log_2 q` multiplications. The attack is one discrete log, about `q^(1/2) = 2^(b/2)` operations for a `b`-bit `q`. If `b = 200`, that's at most four hundred multiplications against `2^100 ≈ 10^30`. The work ratio is enormous — but notice it's `2^(b/2)`, so to keep the attack out of reach I need `b` solidly large; a small `b` would put `2^(b/2)` within a computer's reach even though the legitimate side stays cheap. So `q` wants to be hundreds, even a thousand, bits — and then the public file stores just one `b`-bit number per user, and each terminal needs only a few `b`-bit words and a random-number source. One more structural point I shouldn't gloss: this has to be *two-way*. The exchange needs fresh randomness from *both* ends — `X_i` from one, `X_j` from the other — precisely so that the resulting key depends on a secret only each party holds. If only one side spoke, there'd be nothing distinguishing the legitimate receiver from the eavesdropper who hears the same single transmission; it's the *independent* random draw at each end that splits them apart.

Let me run a tiny instance by hand to make sure I haven't fooled myself, with a `q` small enough to check but a real primitive root. Take `q = 23` and `α = 5`; the powers of `5` mod `23` cycle through all of `1..22`, so `5` is primitive. Let `A` choose `X_A = 6`: `Y_A = 5^6 mod 23`. Now `5^2 = 25 ≡ 2`, `5^4 ≡ 2^2 = 4`, `5^6 = 5^4 · 5^2 ≡ 4 · 2 = 8`, so `Y_A = 8`. Let `B` choose `X_B = 15`: `Y_B = 5^15 mod 23`; `5^8 ≡ 4^2 = 16`, `5^15 = 5^8 · 5^4 · 5^2 · 5^1 ≡ 16 · 4 · 2 · 5 = 640 ≡ 640 - 27·23 = 640 - 621 = 19`, so `Y_B = 19`. Now `A` computes `Y_B^{X_A} = 19^6 mod 23`: `19 ≡ -4`, `(-4)^2 = 16`, `(-4)^4 ≡ 16^2 = 256 ≡ 256 - 11·23 = 3`, `19^6 ≡ 3 · 16 = 48 ≡ 2`. And `B` computes `Y_A^{X_B} = 8^15 mod 23`: `8 = 2^3` so `8^15 = 2^45`, and the order of `2` mod `23` is `11` (since `2^11 ≡ 1`), so `2^45 = 2^(44+1) = (2^11)^4 · 2 ≡ 2`. Both get `2`. And `α^{X_A X_B} = 5^90 mod 23` should agree: `90 = 4·22 + 2`, and `5^22 ≡ 1` by Fermat, so `5^90 ≡ 5^2 ≡ 2`. All three routes give `2`. The exponents commuted, the two parties met, and the only public data was `23, 5, 8, 19`.

That is the key-distribution problem solved over a purely public channel: the conceptual leap is now concrete in a few lines of modular arithmetic. And I'll note where it sits relative to the fuller dream: this gives me *agreement on a shared key*, the weakest and most reachable of the three reductions, the one I argued was easiest. It does *not* by itself give me the full encrypt-to-anyone public-key cryptosystem or the signature — those want an actual trap-door one-way function I can invert with private information, and the cleanest candidates for that are still only candidates: perhaps a knapsack one-way function can somehow be given a private way back, or perhaps the difficulty of factoring can be exploited, since multiplying is easy and undoing the product appears brutal. But the *framework* is now firm: one-way functions, trap-door one-way functions, public enciphering keys, the splitting of encryption from decryption — and within it, exponential key exchange stands as a complete, working solution to the problem that was blocking everything.

Let me close the loop on the whole chain, because the order matters and I don't want to lose it. The pain was that secrecy demanded a pre-shared key carried over a private channel, and in a network that's combinatorially hopeless; signatures looked separately impossible. Shannon forced me off unconditional security and onto computational security, which turned the problem into one about the *gap* between easy and hard computation — and complexity theory just handed us a language for exactly that gap. The login one-way function and the IFF challenge/response each guarded one flank; fusing them — a one-way function the challenger can't invert but can judge, then a one-way function with a private *trap-door* — collapsed signatures and key distribution into a single object and split encryption from decryption, which is public-key cryptography. Hunting for a concrete function with a *big*, exponential easy-vs-hard gap ruled out the matrix (gap only `n`) and, for *agreement* specifically, the knapsack (one-way but no composition law), and landed on modular exponentiation — where the discrete logarithm supplies the one-way hardness and, crucially, the commutativity `(α^a)^b = (α^b)^a` supplies the algebraic handle that lets two strangers, broadcasting only `α^{X_i}`, compute the same secret `α^{X_i X_j}` that no eavesdropper holding only the two public powers can reach. The courier is gone; the directory took its place.

```python
# Exponential key exchange over a prime field.
# Public, agreed in advance, NOT secret: a prime q and a primitive element alpha of GF(q).
# Security rests on the difficulty of the discrete logarithm mod q:
#   forward  Y = alpha^X mod q     costs ~2*log2(q) multiplications (repeated squaring)
#   backward X = log_alpha(Y) mod q costs ~q^(1/2) by the best known algorithm  -> exponential gap.

import secrets

def random_int(low, high):
    # draw uniformly from the inclusive range [low, high]
    if low > high:
        raise ValueError("empty range")
    return low + secrets.randbelow(high - low + 1)

def modexp(base, exp, mod):
    # repeated-squaring exponentiation: O(log exp) multiplications mod `mod`
    result = 1
    base %= mod
    while exp > 0:
        if exp & 1:
            result = (result * base) % mod
        base = (base * base) % mod
        exp >>= 1
    return result

class PublicParameters:
    def __init__(self, q, alpha):
        self.q = q          # large prime (hundreds to ~1000 bits, since attack ~ 2^(b/2))
        self.alpha = alpha  # primitive element: its powers run through all of 1..q-1

def generate_keypair(params):
    # each user draws a secret exponent and publishes only alpha^X
    X = random_int(1, params.q - 1)          # secret, never leaves the terminal
    Y = modexp(params.alpha, X, params.q)    # public: goes in the directory beside name/address
    return X, Y

def derive_shared_secret(my_secret_X, their_public_Y, params):
    # raise the OTHER party's public value to MY secret exponent:
    #   (alpha^X_other)^X_mine = alpha^(X_mine * X_other) mod q
    # the other party, doing the symmetric computation, lands on the SAME element,
    # because the exponents commute -- this is the structure a bare one-way function lacks.
    return modexp(their_public_Y, my_secret_X, params.q)

# An eavesdropper holds q, alpha, Y_i, Y_j but not X_i or X_j; to reach
# alpha^(X_i X_j) he must (as far as anyone knows) first take a discrete log.
```
