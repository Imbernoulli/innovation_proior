OK, let me think this through from scratch. The thing everyone complains about with Transformers is that attention is quadratic in the sequence length, and the instinctive reading of "quadratic" is "too many FLOPs." So the whole field has been chasing FLOP reduction: make the score matrix sparse so you only compute a fraction of its entries, or approximate the softmax with a low-rank factorization so the cost becomes linear. And yet вҖ” this is the thing that nags at me вҖ” a lot of these clever linear-attention methods, when you actually time them on a GPU, are not faster than plain dense attention. Some are slower. They reduce the FLOPs and the wall clock doesn't move. That's the surprise: if "quadratic FLOPs" were really the problem, cutting the arithmetic should cut the time, and it doesn't. So the time was never being spent on arithmetic in the first place.

So where is it being spent? Let me look at what attention actually does, step by step, as a sequence of GPU kernels. Inputs Q, K, V are each NГ—d, living in HBM, the big slow off-chip memory. Step one, S = QK^T: read Q and K from HBM, multiply, write S вҖ” and S is NГ—N вҖ” back to HBM. Step two, P = softmax(S): read the whole NГ—N S back from HBM, exponentiate and normalize row-wise, write the NГ—N P back to HBM. Step three, O = PV: read P and V from HBM, multiply, write O. Stare at this for a second. We wrote an NГ—N matrix to HBM, read it back, wrote another NГ—N matrix, read it back. For N=1K that's a million entries per matrix, round-tripped. The matmuls QK^T and PV have decent arithmetic intensity, but the softmax in the middle is a pure elementwise-plus-reduction over NВІ elements: almost no arithmetic per byte, it just reads and writes the giant matrix. That step is memory-bound. And masking and dropout, when present, are more elementwise passes over the same NГ—N matrix вҖ” more memory-bound traffic.

Now the hardware fact that turns this from an annoyance into the whole story: on a modern GPU, HBM is huge but slow вҖ” order 1.5вҖ“2 TB/s вҖ” and there's a tiny on-chip SRAM that's about ten times faster in bandwidth but only ~100 KB per multiprocessor. And crucially, over GPU generations compute has been getting faster relative to memory bandwidth. So a memory-bound op like the attention softmax is bottlenecked by the rate at which it can drag the NГ—N matrix across the HBM bus. That's the time. The FLOP-reduction methods don't touch this вҖ” they still materialize matrices in HBM, sometimes with messier access patterns, so the dominant cost is untouched. The principle that's missing is being *IO-aware*: count and minimize the reads and writes between HBM and SRAM, not the FLOPs. There's an old framework for exactly this, the external-memory / IO-complexity model вҖ” you count transfers between a small fast memory and a large slow one. Nobody's been applying it to attention. Let me adopt it as the objective. The thing to minimize is HBM accesses.

Let me get a number to anchor on. Standard attention's HBM traffic: QK^T reads Q,K (that's Оҳ(Nd)) and writes S (Оҳ(NВІ)); softmax reads S and writes P (Оҳ(NВІ)); PV reads P,V and writes O (Оҳ(NВІ) + Оҳ(Nd)). Total Оҳ(Nd + NВІ). The NВІ term is the killer and it's all from the intermediate matrices. So the goal crystallizes: *never write the NГ—N attention matrix to HBM, and never read it back.* Compute O = softmax(QK^T)V while keeping the big matrix on-chip, or better, never forming the whole thing at all.

The obvious move is kernel fusion вҖ” load Q, K, V once, do the matmul, the softmax, the second matmul all on-chip, write only O. That would collapse all those round-trips. But there are two walls. First wall: the softmax in the middle. Softmax over a row needs the whole row at once вҖ” it needs the max over the row for numerical stability and the sum over the row for the denominator. The row of S has length N. If N is large the row of S вҖ” let alone the whole matrix вҖ” doesn't fit in the little SRAM. So I can't just "load it all and fuse"; the reduction seems to demand the entire row be present. Second wall: this is for training, and the backward pass needs S and P to compute gradients. The standard fused-kernel trick still has to spill the intermediates to HBM precisely so the backward can read them вҖ” which puts the NВІ traffic right back. So naive fusion doesn't save training. Two challenges, then: (i) do the softmax reduction without ever holding a whole row; (ii) get through the backward pass without storing the NГ—N matrix.

Take the first wall. The softmax couples all the columns of a row through the max and the sum вҖ” that coupling is what seems to force materializing the whole row. Is there a way to compute a softmax incrementally, seeing the entries a chunk at a time, and still get the exact answer? Let me think about what a numerically-stable softmax even is. For a vector x, you take m = max_i x_i, form f_i = exp(x_i вҲ’ m), let в„“ = ОЈ_i f_i, and softmax = f/в„“.   в„“ = e^{m^{(1)} вҲ’ m} в„“^{(1)} + e^{m^{(2)} вҲ’ m} в„“^{(2)}.

The coupling is broken. I never need the whole row; I need two running scalars.

But wait вҖ” I don't actually want the softmax weights for their own sake. I want O = PV, the weighted sum of value vectors. When the max and the denominator both change, I have to re-scale the existing o_i the same way I re-scaled в„“_i. That last line is the crux of the whole thing. No NГ—N matrix anywhere. This is genuinely just the online-softmax recurrence promoted from a scalar denominator to the vector output.

Now make it concrete as a tiled algorithm. The SRAM has size M. I'll split K and V into column blocks of B_c rows each, and Q (and the output O and the stats) into row blocks of B_r rows each. Let me make sure this actually computes the exact softmax and not some approximation, because that's the whole selling point вҖ” exact attention. I'll prove it by induction on the outer-loop index j. And the cost: each inner step does two matmuls of size B_rГ—B_cГ—d, i.e. O(B_r B_c d) FLOPs, executed T_c T_r вүҲ NВІ/(B_c B_r) times, totalling O(NВІd) вҖ” same arithmetic order as standard attention. The only extra memory beyond inputs and output is the O(N) of statistics (m, в„“). Good: linear extra memory, exact result, all on-chip.

Now the second wall: the backward pass. Naively, gradients need S and P, the NГ—N matrices. If I store them I'm back to O(NВІ) memory and HBM traffic; if I don't store them I seem stuck. There's a known escape вҖ” gradient checkpointing: don't store the big intermediate, recompute it in the backward. Someone's already suggested doing this for attention to cap the memory footprint, by checkpointing the memory-efficient forward and letting autodiff recompute. But generic checkpointing has a reputation: it trades speed for memory вҖ” you pay extra FLOPs to recompute and you eat that as slower wall-clock. Here's the twist that's special to our setting, though: in an IO-bound regime, recomputation might actually be *free or better* on the clock, because the recompute happens on-chip in fast SRAM and saves the HBM round-trips of the NГ—N matrix that storing-and-reloading would cost. The extra FLOPs are cheap; the avoided HBM accesses are expensive. So recomputation could *buy* speed instead of costing it. But to make it tight I don't want to lean on a generic autodiff-checkpoint of the forward вҖ” that would recompute too much and re-store per-block temporaries. Let me derive the backward by hand and see exactly the minimal thing I need to recompute and the minimal thing I need to save.

Set up: scalar loss ПҶ, output gradient dO = вҲӮПҶ/вҲӮO, want dQ, dK, dV. Let the raw dot-product score be R_{ij} = q_iВ·k_j and the score fed to softmax be S_{ij} = П„R_{ij}.   dS_{i:} = (diag(P_{i:}) вҲ’ P_{i:} P_{i:}^T) dP_{i:} = P_{i:} вҲҳ dP_{i:} вҲ’ (P_{i:}^T dP_{i:}) P_{i:},

  dq_i = П„ОЈ_j dS_{ij} k_j = П„ОЈ_j P_{ij}(dP_{ij} вҲ’ D_i) k_j = П„ОЈ_j (e^{S_{ij}}/L_i)(do_iВ·v_j вҲ’ D_i) k_j,

Every one of dV, dQ, dK is now a sum I can accumulate block by block, recomputing the needed P_{ij} on-chip from the saved (m, в„“) and the inputs, and computing D_i as do_iВ·o_i. So the backward, like the forward, is O(NВІ) FLOPs and O(N) extra memory, and вҖ” the payoff вҖ” it never reads an NГ—N matrix from HBM. I save from the forward only O, the stats (m, в„“), and, if dropout is on, the random-number-generator state (so I can regenerate the same dropout mask in the backward instead of storing the NГ—N mask вҖ” storing it would be O(NВІ), regenerating it is free in memory). This is strictly better than checkpointing the forward generically: I recompute only the attention block S, P that I need, I don't recompute or re-store per-block temporary outputs, and I keep a single running copy of things.

This also sharpens a contrast with the prior linear-memory approach. That one also used online softmax and also recomputed in the backward, but it was aiming at the *footprint* вҖ” the peak bytes вҖ” and it summarized each block by a separate temporary output, then combined all the block outputs at the end, which means holding one output per block; and its backward leaned on generic gradient checkpointing. By instead incrementally updating one running O_i (the rescale-and-add recurrence above) I keep a single output copy, and by deriving the backward analytically I recompute only S and P. The deeper difference is the objective: minimizing HBM accesses, not bytes. And note minimizing accesses also bounds the footprint вҖ” if an algorithm makes A accesses its memory is at most A вҖ” so I get the footprint win for free, but the access count is what controls the wall clock.

So let me actually quantify the HBM accesses and confirm the win is real, because that's the claim the whole method rests on. Trace the forward kernel's traffic. Each element of K and V is loaded exactly once вҖ” that's the outer loop, Оҳ(Nd). The inner loop makes one pass over Q and O for each outer block, so T_c passes over Q and O, each pass moving Оҳ(Nd). Total HBM accesses вүҲ Оҳ(Nd + NdВ·T_c) = Оҳ(NdВ·T_c). Now T_c = N/B_c, and B_c is set by how much fits in SRAM of size M. The constraints: K_j and V_j blocks are B_cГ—d, need B_c d = O(M), so B_c = O(M/d); Q_i, O_i are B_rГ—d, need B_r = O(M/d); and the score block S_ij is B_rГ—B_c, need B_r B_c = O(M).   Оҳ(Nd В· Nd/M) = Оҳ(NВІ dВІ / M).

Compare to standard attention's Оҳ(Nd + NВІ). The ratio of the leading terms is (NВІ dВІ/M) / NВІ = dВІ/M. For real values вҖ” d is 64вҖ“128, M is around 100 KB вҖ” dВІ is many times smaller than M, so NВІdВІ/M is many times smaller than NВІ. That's the multiple-fold reduction in HBM traffic, and since attention is memory-bound that's the wall-clock speedup. It even survives the extra recomputation FLOPs in the backward: the backward's traffic is the same Оҳ(NВІdВІ/M) by the identical argument (each K,V element loaded once; T_c passes over Q, O, dO and over dQ), so more FLOPs but far fewer accesses, and the clock follows the accesses. The entire detour through IO-awareness was the point.

Is this access count optimal, or could a cleverer schedule do asymptotically better? Let me see how low it could possibly go. Suppose some exact-attention algorithm achieved o(NВІdВІ/M) accesses *for all* M in the meaningful range [d, Nd]. Look at the top of that range, M = Оҳ(Nd) вҖ” SRAM large enough to hold a constant number of the input matrices. There o(NВІdВІ/M) = o(NВІdВІ/(Nd)) = o(Nd). So o(Nd) is impossible вҖ” contradiction. Hence no exact attention algorithm can beat Оҳ(NВІdВІ/M) across all M; the tiling is asymptotically optimal in HBM accesses over the SRAM range, at least via this subrange argument. (A full M-parameterized lower bound would be nicer, but the subrange contradiction is the standard external-memory style and it pins optimality at M = Оҳ(Nd).)

A couple of constants I glossed need nailing down, since they decide whether the kernel actually fits. The block sizes I'll set to B_c = вҢҲM/(4d)вҢү and B_r = min(вҢҲM/(4d)вҢү, d). The 1/4 is a budget split: the kernel has to hold several d-wide blocks resident at once вҖ” a Q block, a K block, a V block, and the running O block вҖ” so I divide the M-element SRAM budget by about four to leave room for all of them (plus the small score tile and stats). Capping B_r at d is what enforces the third constraint B_r B_c = O(M): with B_c вүҲ M/d, the product B_r B_c вүҲ MВ·(B_r/d), so I need B_r вүӨ d to keep that O(M). Asymptotically this is exactly B_c = Оҳ(M/d), B_r = Оҳ(min(M/d, d)), the same values the IO-complexity bound assumed.

One more pass to fold in the real-kernel niceties. The softmax scaling П„ (typically 1/вҲҡd) just multiplies S_ij before everything; masking sets masked entries of S_ij to вҲ’вҲһ before the row-max (so they contribute zero weight); dropout multiplies PМғ_ij elementwise by a mask whose entries are 1/(1вҲ’p) or 0, applied to the probabilities before the V multiply, and reproduced in the backward from the saved RNG state. And there's a tidy bookkeeping simplification real implementations use: during the forward kernel I still keep the temporary running max m_i because the accumulator is based at that max, but the statistic I need to save for the backward is only lse_i = m_i + log в„“_i. Each block uses m_new = max(m_i, m_blk), folds the denominator as lse_i вҶҗ m_new + log(e^{lse_i вҲ’ m_new} + в„“_blk), and at the very end normalizes the accumulator once by o_scale = exp(m_i вҲ’ lse_i). The backward reconstructs P_{ij} directly as exp(S_ijВ·П„ вҲ’ lse_i). Mathematically identical to the (m, в„“) recurrence I derived; it just saves the one scalar that is convenient to recompute from.

And the extension that falls out almost for free: if I only care about a block-sparse attention pattern вҖ” a 0/1 mask over blocks of the score matrix вҖ” then in the loops I simply skip the blocks the mask zeroes out. Same kernel, an `if` around the inner body. The HBM accesses scale by s, the fraction of nonzero blocks, giving Оҳ(Nd + NВІdВІs/M) вҖ” the Nd term survives because I still have to write the whole output O вҖ” so for s = N^{вҲ’1/2} this is Оҳ(NвҲҡN) and for s = N^{вҲ’1}log N it's Оҳ(N log N) accesses. A natural pattern to use is a butterfly sparsity, since butterfly factorizations can express arbitrary structured patterns with near-optimal cost and are hardware-friendly. But the core is the dense exact algorithm; block-sparsity is just skipping iterations.

Let me write the forward as the kernel I'd actually run, mirroring the structure of a real fused implementation, with the running statistics and the rescale-and-add accumulation as the body вҖ” and folding (m, в„“) into the log-sum-exp form since that's what the kernel stores.

```python
# FlashAttention forward, one query-block program. Streams over K/V blocks,
# keeps per-row running max m_i, running log-sum-exp lse_i, and output
# accumulator acc_o; rescales the accumulator each block; normalizes once at end.
def flash_attn_fwd_block(q, K, V, scale, BLOCK_N, is_causal=False):
    # q: (B_r, d) for this query block, already in SRAM.
    m_i   = full((q.shape[0],), -inf)     # running row max
    lse_i = full((q.shape[0],), -inf)     # running log-sum-exp
    acc_o = zeros((q.shape[0], q.shape[1]))  # running output accumulator

    for k_blk, v_blk in iter_blocks(K, V, BLOCK_N):     # load each K,V block once
        s = (q @ k_blk.T) * scale                       # S_ij block, on chip, B_r x B_c
        s = apply_mask(s, is_causal)                    # set masked entries to -inf

        m_ij  = maximum(m_i, rowmax(s))                 # tentative new max
        p     = exp(s - m_ij[:, None])                  # P~_ij = exp(S_ij - m), on chip
        l_ij  = rowsum(p)                               # this block's denominator piece

        acc_o = acc_o * exp(m_i - m_ij)[:, None]        # rescale old accumulator to new max
        acc_o = acc_o + p @ v_blk                       # add this block's P~ V contribution

        m_i   = m_ij                                    # advance running max
        lse_i = m_ij + log(exp(lse_i - m_ij) + l_ij)    # fold block into running log-sum-exp

    o = acc_o * exp(m_i - lse_i)[:, None]               # single final normalization by 1/L_i
    return o, lse_i                                     # save lse_i (O(N) stat) for backward
```

And the backward, as the hand-derived block recurrence вҖ” recompute P from the saved lse, get D_i as the doВ·o dot product, accumulate dV, dQ, dK:

```python
# FlashAttention backward. Recomputes P on-chip from lse; D_i = rowsum(O .* dO).
def flash_attn_bwd_block(q, K, V, o, do, lse, scale, BLOCK_N):
    D_i = rowsum(o * do)                                # D_i = do_i . o_i  (size-d dot, per row)
    dq  = zeros_like(q)
    for k_blk, v_blk, dk_blk, dv_blk in iter_blocks_bwd(K, V, BLOCK_N):
        s  = (q @ k_blk.T) * scale
        p  = exp(s - lse[:, None])                      # recomputed P_ij, no N x N stored
        dv_blk += p.T @ do                              # dV += P^T dO
        dp = do @ v_blk.T                               # dP_ij = dO V^T
        ds = p * (dp - D_i[:, None]) * scale            # dS_ij = P_ij (dP_ij - D_i), with tau
        dq += ds @ k_blk                                # dQ_i += dS_ij K_j
        dk_blk += ds.T @ q                              # dK_j += dS_ij^T Q_i
    return dq  # dk, dv accumulated in place
```

The whole chain, traced backward: attention is memory-bound, so its runtime is the HBM traffic of round-tripping the NГ—N score and probability matrices, not the FLOPs вҖ” which is why FLOP-cutting approximations didn't speed it up. So the objective becomes minimizing HBM accesses. To never write the NГ—N matrix, fuse everything into one kernel; the obstacle is that softmax needs a whole row, which online softmax dissolves by carrying a running max and denominator and rescaling partial results вҖ” promoted here to also rescale the running output accumulator, so the exact output is built block by block on-chip (proved exact by induction). The other obstacle, needing the NГ—N matrix for the backward, is dissolved by recomputing the attention block on-chip from a saved O(N) statistic and the inputs вҖ” which is a win, not a cost, because the avoided HBM accesses dwarf the extra FLOPs вҖ” together with the identity D_i = do_iВ·o_i that turns a length-N row reduction into a size-d dot product. Counting accesses gives Оҳ(NВІdВІ/M), a many-fold reduction from Оҳ(NВІ) since dВІ вүӘ M, and a subrange argument shows that's optimal for exact attention. Skipping zero blocks extends it to block-sparse with accesses scaled by the sparsity.
