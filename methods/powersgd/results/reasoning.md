I start from the communication primitive, because that is where most tempting compressors quietly fail. Dense distributed SGD is expensive, but it has one property I cannot lose: the object being communicated is a fixed-shape tensor, and aggregation is addition. Ring or tree all-reduce can exploit that associativity, so the message does not become more structurally complicated as I add workers. If I send a compressed object whose structure depends on each worker's local gradient, then the sum of those objects may no longer be a compressed object of the same type. Once that happens I am pushed into gather, decode, merge, or re-encode logic, and the nominal compression ratio stops predicting wall-clock speed. So the property I am really hunting for is a compressor whose encoding commutes with averaging across workers.

That requirement immediately strains the familiar coordinate schemes. Top-k with memory is attractive as an optimizer, because a residual can carry all dropped coordinates forward, but worker A's top coordinates and worker B's top coordinates are not the same set. Their sum has the union support, and that union can grow with the number of workers. Sign methods are tiny, but signs are nonlinear and majority vote is not averaging. Stochastic quantization is easier to analyze because it is unbiased, but quantized codes are still not generally closed under addition in the way a dense tensor is. The common failure is the same: the encoding is a nonlinear function of the local data, so averaging the encodings is not the encoding of the average.

The gradient tensors themselves suggest a different structure to exploit. A fully connected layer gives me a matrix gradient. A convolutional filter gradient can be flattened into a matrix by keeping output channels as rows and folding input and kernel dimensions into columns. Matrices can be low-rank even when their entries are not sparse. If an `m x n` gradient is well approximated by rank `r`, then a factorization `P Q^T` costs `(m+n)r` numbers instead of `mn`, which is the right scale for aggressive compression. The obvious way to get the best rank-`r` factorization is a truncated SVD, so let me test whether that route survives the averaging requirement. The best rank-`r` approximation is not a linear function of `M`: the singular subspaces of `M_1` and of `M_2` need not align, so the SVD of the average is not built from the SVDs of the parts. If every worker truncates its own SVD and we average the factors, we get the same structural failure as top-k, now in spectral coordinates. So I should drop the demand for the *optimal* low-rank approximation and instead look for one whose expensive steps are *linear* in `M`.

Subspace iteration has exactly that shape. Start from a query matrix `Q`, compute `P = M Q`, orthogonalize `P`, and project back with `Q = M^T P`. Both matrix multiplications are linear in `M` as long as the *other* factor is held fixed and shared across workers. That linearity is the whole game, so I want to check that it really makes the distributed computation collapse to a single-machine one. The order of operations has to be: every worker uses the *same* query `Q` and computes `P_w = M_w Q` locally; we all-reduce the mean to get `P = (1/W) sum_w M_w Q`; only *then* does everyone orthogonalize that common `P`; each worker computes `Q_w = M_w^T P` against the shared left factor; and a second all-reduce gives `Q = (1/W) sum_w M_w^T P`. The reconstruction is `P Q^T`.

I want to be sure this is identical to running plain subspace iteration on the averaged gradient `Mbar = (1/W) sum_w M_w`, and not merely close. Two workers, `m,n,r = 3,4,1`, random `M_1, M_2`, shared random `Q`. Running the two-reduction procedure above and, separately, one step of subspace iteration directly on `Mbar`, the two reconstructed matrices agree to `6.7e-16` — machine zero. That is the linearity paying off: `(1/W) sum_w M_w Q = Mbar Q` because `Q` is shared, and after orthogonalizing the *aggregated* `P`, `(1/W) sum_w M_w^T P = Mbar^T P` because `P` is now shared. The compressed all-reduce computes exactly what a single machine holding the average gradient would have computed. That is the closure property I needed.

The ordering is load-bearing, and I want to catch the natural shortcut that breaks it before I trust the design. The tempting one-shot hook computes *both* local factors before any communication: each worker orthogonalizes its own `P_w = M_w Q`, forms `Q_w = M_w^T P_w` against its *own* left factor, and we all-reduce both factors in a single round. This is appealing because it is one collective instead of two. I tried it on the same two-worker example. The reconstruction differs from the correct one by up to `0.41` in absolute entries, and as an approximation of `Mbar` it is strictly worse: relative-to-truth the one-shot error is larger (Frobenius `1.93` versus `1.86` for the correct version). The reason is exactly the linearity argument run in reverse — orthogonalizing `P_w` *per worker* makes `Q_w = M_w^T orth(M_w Q)` a nonlinear function of `M_w`, so averaging the `Q_w` no longer equals `Mbar^T P`. The two reductions are not an inefficiency to be merged away; collapsing them silently reintroduces the nonlinearity I spent the whole design avoiding. It only coincides with the correct algorithm when `W = 1`, where there is nothing to average.

Now I have a compressor that is implementable and all-reduceable, but I have to be honest that it is biased. A single step of subspace iteration does not even reach the optimal rank-`r` approximation, let alone preserve the full gradient. I checked the size of that gap on a `4 x 5` matrix with `r = 2`: the one-step reconstruction has Frobenius error `1.98` from `M`, while the optimal rank-2 SVD has error `1.19`, which equals the trailing singular mass `1.19` exactly as it must. So I am leaving roughly `0.8` of Frobenius mass on the floor relative to even the best rank-2 object, and far more relative to the dense gradient. I cannot pretend the reconstructed matrix is an unbiased stochastic gradient.

This is where error feedback has to enter, and I want to pin down precisely what it stores. Before compression I add the worker's residual to the current gradient, in the same units the optimizer will update with: `M = g + e`. After reconstruction I store the part this step failed to send. In the single-worker case that is unambiguous: `e_next = M - P Q^T`. I verified the bookkeeping numerically on the `4 x 5`, `r = 2` example — after compressing, `M` equals `approx + e` to machine precision, and `approx` indeed has rank 2. So the local buffer holds exactly the discarded component, and it is reintroduced next step; nothing is silently dropped, only delayed. (In the distributed version the natural reference invariant is the local corrected matrix minus the *averaged* low-rank update; I will keep the W=1 identity as the one I can check exactly here, and treat the distributed version as the consistent generalization.)

The residual analysis is what lets me say anything about convergence. Under a contraction condition `||C(x) - x||^2 <= (1 - delta) ||x||^2`, the residual stays bounded, the corrected iterate `x_tilde = x_t - e_t` follows an SGD-like recurrence, and smoothness recovers SGD's asymptotic rate up to higher-order terms. I want to be careful not to overclaim here: this is a statement about the corrected iterate, not a claim that each actual compressed step equals the dense step, and the contraction constant for the one-step warm-started low-rank map on a *changing* matrix is not something I can certify from a single example. I expect it holds in the regime where the gradient is genuinely low-rank and the subspace drifts slowly; I would want to verify it across a real training run before stating it as a theorem.

That dependence on slow drift points at the next design choice. One step of subspace iteration is a weak rank estimate, so starting from a fresh random `Q` every step throws away everything learned. Under small SGD steps the gradient distribution changes gradually, so the dominant subspace this step should be close to the previous one — which means reusing the previous `Q` turns the per-step single iteration into a continued power iteration across steps. I tested this on a `20 x 20` sequence with a fixed dominant rank-2 structure plus small per-step drift, `r = 2`. Cold start (fresh random `Q` each step) gives a mean relative reconstruction error of `0.53` over the run; warm start (carry the previous `Q` forward and re-orthogonalize) gives `0.28`. Both begin identically at the first step (`0.227`, same initial `Q`), and warm start pulls ahead immediately afterward as it accumulates the subspace across iterations. So the warm start is not a minor optimization; it roughly halves the error, and it is what makes a single matrix multiply per side good enough. The underlying picture is Oja-like stochastic subspace tracking; I am treating it as an empirical and algorithmic justification, not as a closed-form theorem for the one-step changing-matrix case.

The orthogonalization placement matters because the factors otherwise absorb arbitrary scale. The algorithm orthogonalizes the aggregated `P` before computing `Q = M^T P`. It also pays to normalize the warm-started `Q` before using it, since after carrying it forward and adding a new `M^T P` it is no longer orthonormal and its scale would leak into `P`. So the per-step sequence is: normalize the warm-started `Q`, compute and reduce `P`, orthogonalize the reduced `P`, compute and reduce `Q`, store that reduced `Q` for the next step, and normalize it again at the start of the next step. This keeps the factorization scale-stable across the warm-start chain.

There is a rank-interface question. The natural knob for this method is the rank itself: rank one, two, four, and so on, because rank directly sets both the compression ratio `(m+n)r / mn` and the approximation quality. The scaffold instead hands me a `compress_ratio`, so if I must obey that interface I can invert it: solve `(m+n)r <= ratio * mn` for `r` and cap at `min(m,n)`. But that ratio-derived rank is a translation layer, not the essence of the method, so I will expose a fixed `rank` parameter as the primary control and fall back to `compress_ratio` only when no rank is supplied. I should also refuse to compress when it does not actually save communication: if `(m+n)r` is not meaningfully smaller than `mn` — for instance a near-square matrix at moderate rank — the encode/decode work buys nothing, so a minimum-compression-rate threshold sends those tensors dense. Vector-shaped tensors have no useful matrix rank and are small, so they pass through dense as well.

Putting the pieces together, the artifact should say exactly what it does and no more. It trades the dense gradient all-reduce for two fixed-shape low-rank all-reduces — and I have checked that this exactly equals subspace iteration on the averaged gradient, that merging the two reductions into one genuinely breaks that equality, that the stored residual is exactly the discarded component, and that warm-starting the query roughly halves the reconstruction error. It uses error feedback so the omitted part stays in local memory instead of disappearing, with a convergence guarantee that I can only honestly state in the corrected-iterate sense under a contraction assumption. The systems win is likewise conditional: when bandwidth is the bottleneck and the encoded matrices are much smaller, wall time drops; when bandwidth is high, communication overlaps compute well, or encode/decode overhead dominates, an optimized synchronous SGD can still win.
