I am training a very deep transformer — dozens of layers, billions of parameters — and to fit it in memory and run it fast I do the arithmetic in 16-bit, and it dies. Within the first thousand iterations the loss goes to NaN: not a gradual divergence I could chase with a smaller learning rate, but a hard overflow, because FP16 has a ceiling around $6.5\times10^4$, and once a value crosses it the activation becomes infinity and any statistic computed from it — a sum, a variance inside a normalization — becomes NaN and poisons everything downstream. So something in the forward pass is manufacturing enormous numbers, and the job is to find where and stop it without disturbing the data, tokenizer, optimizer, schedule, or the attention and MLP internals.

The block in use is the standard pre-normalized one, $x \leftarrow x + \mathrm{Sublayer}(\mathrm{LN}(x))$, and pre-normalization is the default for a reason I do not want to give up. The original transformer put the norm *after* the residual addition, $x \leftarrow \mathrm{LN}(x + \mathrm{Sublayer}(x))$ (Post-LN), and the mean-field analysis of Xiong et al. (2020) shows that when the norm sits on the residual *sum*, the expected gradient norm of the parameters near the output is $O(d\sqrt{\ln d})$, *independent of depth* $L$ — large from step one, which is why Post-LN needs a learning-rate warmup and is delicate. Moving the norm inside the residual branch makes the residual path a clean identity highway and gives a last-layer gradient of $O(d\sqrt{\ln d / L})$, which actually *shrinks* with depth, is well-behaved at initialization, and trains without warmup. That gradient health is the whole appeal of pre-normalization. But the same analysis records the *forward*-side fact that turns out to be load-bearing: in Post-LN the residual sum has constant expected squared norm and is renormalized back to scale $\sqrt{d}$, while in Pre-LN the residual stream *grows* with depth, $(1 + l/2)\,d \le \mathbb{E}\,\|x_l\|^2 \le (1 + 3l/2)\,d$. And of course it does — the residual path is never normalized (that is the point), so every sublayer's raw output is added onto the highway and stays. The norm only ever touches the *copy* fed into a sublayer, never the highway itself. So pre-normalization gives good gradients *because* the residual path is un-normalized, and gives forward growth *because* the residual path is un-normalized: one property, two consequences. That is why the naive fixes are both wrong. Reverting to Post-LN flattens the forward scale but brings back the $O(d\sqrt{\ln d})$ output gradients and the warmup. And merely lowering the learning rate or shrinking the init fights an exponential with a constant — it delays the overflow by a few hundred iterations but never removes the mechanism.

The mechanism deserves a closer look, because linear growth in the *expected* norm is gentle — a few hundred thousand in squared norm at sixty layers is only a per-coordinate scale of order tens, nowhere near the FP16 ceiling. What kills me is the *worst* coordinate, and the fact that the growth compounds. Write the recursion for the stream, $x_{l+1} = x_l + b_l$ with $b_l = \mathrm{Sublayer}(\mathrm{LN}(x_l))$, and take the coordinate-wise variance loosely: $\mathrm{Var}(x_{l+1}) = \mathrm{Var}(x_l) + \mathrm{Var}(b_l) + 2\,\mathrm{Cov}(x_l, b_l)$. If $b_l$ were a fixed-scale, zero-mean perturbation independent of the stream, this would just be $\mathrm{Var}(x_l) + \text{const}$, the gentle linear growth. But $b_l$ is *computed from* $x_l$, and once training has moved the weights even slightly, $b_l$ is positively correlated with $x_l$ along the directions the layer has learned to amplify, so the covariance term is positive and the recursion behaves like $\mathrm{Var}(x_l)\cdot(1 + \text{something})$ — geometric. A multiplicative factor a hair above one applied sixty times is the difference between "a few hundred thousand" and "$10^4$–$10^5$." Why would a branch output be large precisely where the stream is already large? Because $\mathrm{LN}(x_l)$ emits a vector of unit per-coordinate variance, hence $L_2$ norm of order $\sqrt{d}$ — normalization does not produce small numbers, it produces numbers of scale $\sqrt{d}$ — and transformers carry a few persistent outlier dimensions that the sublayer's projections reproduce at scale $10^1$–$10^2$. Added back onto the highway, those directions make the next layer's input larger in exactly those coordinates, so the next branch output is larger there too. The large directions feed themselves; the un-normalized residual add is the wire that closes the loop, and after enough layers the worst coordinate crosses the FP16 ceiling.

I propose Sandwich-LN, with RMSNorm. The intervention follows directly from where the loop can be cut. I cannot touch the stream (the identity path I am protecting), cannot remove the pre-sublayer norm (it gives the normalized input and the gradient health), and cannot touch the sublayer internals (off-limits, and anyway the problem is that their *output* is large, not what they compute). The one remaining wire is the branch output, in the gap between the sublayer and the residual add. So normalize *there*: $$x_{l+1} = x_l + \mathrm{Norm}\big(\mathrm{Sublayer}(\mathrm{Norm}(x_l))\big).$$ A normalized branch term, times its learned gain, has a controlled mean-square scale — call it $\beta_0$ — regardless of how large the raw sublayer output was, so the recursion becomes $\mathrm{Var}(x_{l+1}) \approx \mathrm{Var}(x_l) + \beta_0$. The compounding factor is gone; the geometric growth collapses back to the benign additive growth, now *enforced* on the branch contribution rather than hoped for. The branch can no longer hand a raw $10^2$ output to the highway.

The critical thing is that this is *not* Post-LN, and the distinction is the entire point. Post-LN normalizes the residual *sum*, $x \leftarrow \mathrm{LN}(x + b)$; Sandwich-LN normalizes the *branch output* and adds it to an un-normalized stream, $x \leftarrow x + \mathrm{Norm}(b)$. The identity highway from $x_l$ to $x_{l+1}$ stays a clean un-normalized skip; the norm sits inside the branch, off to the side. That is exactly the distinction Xiong's gradient analysis cares about: Post-LN's bad $O(d\sqrt{\ln d})$ output gradient comes from the gradient flowing back along the identity connection having to pass through a normalization at every layer, whereas here the gradient along the highway passes through *nothing*. So I keep the gradient health of Pre-LN and gain the forward-scale control of Post-LN, because I normalized the branch rather than the sum. The branch-end norm even helps on the backward pass: a normalization layer's Jacobian carries a factor of roughly $1/\|a\|$, so when a branch produces a huge activation $a$, the gradient flowing back into that branch's parameters is *damped* by $1/\|a\|$ — the opposite of plain Pre-LN, where a huge forward activation means a huge gradient through the output projection. The fix is stabilizing on both passes. At initialization nothing is perturbed in a bad direction: the gain is one and there is no bias, the branch output is still a well-behaved moderate-scale vector (the explosion is a training-time phenomenon needing learned weights), so the post-norm merely reparameterizes its magnitude without introducing a depth-dependent factor or touching the identity path. Each sublayer is therefore sandwiched between a pre-norm, which earns the well-behaved-at-init gradient, and a post-norm on the branch output, which earns the controlled branch scale.

Separately, since the block now runs four norm calls instead of two, the per-call cost matters, and that is reason to ask whether I am paying for something I do not need. LayerNorm does two things: it subtracts the mean (re-centering) and divides by the standard deviation (re-scaling), two reductions over the channel dimension plus a subtraction. But the whole job I am asking these norms to do is *scale control*, and subtracting the mean shifts the vector without changing its variance or the variance of the gradients — re-centering moves the cloud, re-scaling shrinks it. If the re-scaling is what stabilizes, the re-centering is dispensable, so I normalize by the root mean square alone: $$\bar a_i = \frac{a_i}{\mathrm{RMS}(a)}\, g_i, \qquad \mathrm{RMS}(a) = \sqrt{\frac{1}{n}\sum_{i=1}^{n} a_i^2},$$ with a learned per-channel gain $g$ (init 1) and no bias, because no mean is removed so there is no shift to restore. When the input already has zero mean, $\mathrm{RMS}(a)$ is exactly the standard deviation, so RMSNorm coincides with LayerNorm; I drop only the part that handles a nonzero mean. The property I care about survives intact: for a positive scale $\alpha$, $\mathrm{RMS}(\alpha a) = \alpha\,\mathrm{RMS}(a)$, so dividing by it makes the output invariant to positive rescaling of the input or the whole weight matrix — that is the re-scaling invariance I argued is load-bearing. (For a negative scalar the denominator picks up $|\alpha|$ and the sign stays in the numerator, so the usual invariance claim is the positive magnitude-scaling case.) What I give up — invariance to shifts and to per-weight-vector rescaling — is exactly what I argued does not control scale. There is even a bonus: the weight gradient is invariant to positive input scaling and inversely correlated with positive weight scale, so bigger weights get smaller gradients — an implicit per-layer learning-rate adaptor, fitting for a method whose purpose is keeping scales in line. The cost is one reduction (the sum of squares) plus a reciprocal-square-root, against LayerNorm's two reductions and a subtraction, a real and recurring saving across four norms per block.

Two numerical details matter, because the entire motivation was FP16 overflow and I must not reintroduce it inside the norm. The mean-of-squares is computed in fp32 even when the activations are fp16 — squaring fp16 values and summing thousands of them over a wide channel dimension is precisely what loses significance or overflows — and only after taking the reciprocal square root is the normalized activation cast back before the gain is applied. And an $\epsilon = 10^{-5}$ goes inside the root, $\mathrm{rsqrt}(\mathrm{mean}(x^2) + \epsilon)$, so a degenerate all-zero row does not divide by zero; it is sized well below any healthy activation RMS, flooring the pathological case without perturbing the normal one. The whole block becomes $x \leftarrow x + \mathrm{Norm}_{\text{post}1}(\mathrm{Attn}(\mathrm{Norm}_{\text{pre}1}(x)))$ then $x \leftarrow x + \mathrm{Norm}_{\text{post}2}(\mathrm{MLP}(\mathrm{Norm}_{\text{pre}2}(x)))$, and it drops into the existing decoder by replacing only the normalization op and the block wiring. In the edit surface I keep the scaffold class name `LayerNorm` and the `(ndim, bias)` signature, ignoring `bias`, so every construction site including the final norm picks up the RMSNorm internals unchanged, and the embeddings, stack, head, optimizer, and schedule are untouched.

```python
import torch
import torch.nn as nn


class LayerNorm(nn.Module):
    """RMSNorm internals kept under the scaffold's LayerNorm class name."""

    def __init__(self, ndim, bias):
        super().__init__()
        self.weight = nn.Parameter(torch.ones(ndim))   # gain g (init 1); no bias
        self.eps = 1e-5

    def forward(self, input):
        # fp32 reduction for fp16-safe sum of squares; 1 / sqrt(mean(x^2) + eps)
        rms = input.float().pow(2).mean(-1, keepdim=True).add(self.eps).rsqrt()
        return (input * rms).type_as(input) * self.weight


class Block(nn.Module):
    """Sandwich-LN block: each sublayer wrapped by a pre-norm and a post-norm on the
    branch output, added to an un-normalized residual stream."""

    def __init__(self, config):
        super().__init__()
        self.ln_pre1  = LayerNorm(config.n_embd, bias=config.bias)   # before attention
        self.attn     = CausalSelfAttention(config)
        self.ln_post1 = LayerNorm(config.n_embd, bias=config.bias)   # on attention's branch output
        self.ln_pre2  = LayerNorm(config.n_embd, bias=config.bias)   # before MLP
        self.mlp      = MLP(config)
        self.ln_post2 = LayerNorm(config.n_embd, bias=config.bias)   # on the MLP's branch output

    def forward(self, x):
        x = x + self.ln_post1(self.attn(self.ln_pre1(x)))   # x + Norm(Attn(Norm(x)))
        x = x + self.ln_post2(self.mlp(self.ln_pre2(x)))    # x + Norm(MLP (Norm(x)))
        return x
```
