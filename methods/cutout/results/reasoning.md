I start from the fact that the usual regularizer is failing in a very specific place. Dropout is excellent when a dense hidden unit can be made unreliable and every other unit has to stand on its own, but a convolutional layer is not like that. Its parameters are shared, its local neighborhoods are correlated, and neighboring pixels or activations often carry almost the same evidence. If I remove one point, the nearby points still carry the missing signal. So in a convolutional stack, pointwise dropout is not really removing visual evidence; it is mostly adding noise and asking filters to be robust to that noise.

The feature-map variants point in a more promising direction but do not quite address the thing I care about. Dropping a whole feature map is stronger than dropping scattered positions, and targeting the most active responses is also plausible, but these operations happen after the image has already been transformed into multiple channels. Evidence removed in one map may still exist in another. The model sees an inconsistent representation rather than an image where a cue is actually absent. That would explain why these methods can help in isolation but lose their edge when batch normalization and ordinary augmentation are already in the system — though I should treat that as a hypothesis I cannot settle from first principles, only by comparing error rates on the strong baseline.

If the diagnosis is right, the removal has to happen before the feature maps are made. If I remove visual evidence at the input, then every downstream map has to live with the same missing evidence. There is no parallel channel that quietly preserves it. The network can only infer the missing part from the remaining context. Whether that actually helps is an empirical question, but at least the mechanism matches the failure mode I am worried about, in a way the feature-map variants do not.

The next question is how the input should be damaged. Scattered pixel corruption is too easy because neighboring pixels repair it statistically — the same redundancy argument that sank pointwise dropout applies again, now at the input. Denoising autoencoders already show that input corruption can teach useful features, and context encoders show that a larger missing region forces more global understanding than isolated pixel noise. I do not need the reconstruction task, but I can keep the lesson: the corruption has to be spatially structured enough that local redundancy cannot simply undo it.

So the operation I will try is: during training, choose a location in the image and set one connected region to zero. For the shape I default to a square, on the reasoning that the amount removed should matter more than the exact outline, and a square leaves only one real knob, its side length. The label stays unchanged, because the training example is meant to be an occluded view of the same object, not a mixture with a different class.

Now the placement of the square. My first instinct is to force every removed square to sit fully inside the image, which sounds cleaner. But I should think through what that does to the distribution of training images. If the patch is always fully contained, every training image loses the same full-sized chunk of evidence — the occlusion is uniform in strength. An alternative is to sample the center uniformly over all image coordinates and clip the patch at the boundary, so a center near an edge removes only part of the nominal square.

Take a 32x32 image with side length L=16 and the clipped scheme: sample center (y, x), set y1=max(y-L//2,0), y2=min(y+L//2,h), x1=max(x-L//2,0), x2=min(x+L//2,w), and zero mask[y1:y2, x1:x2]. With the center at (16,16) the box is [8,24)x[8,24), which is exactly 16x16 = 256 pixels removed — the full square, as expected when nothing is clipped. With the center at (1,1) the box is [0,9)x[0,9), so only 9x9 = 81 pixels are removed and 92% of the image survives. With the center exactly at (0,0) the box is [0,8)x[0,8): just 64 pixels, the single inner quadrant of the square. So clipping does not merely shave a few border pixels; near an edge it removes a small fraction of the nominal patch, and the training set ends up spanning a real range from strongly occluded to barely occluded. That spread is exactly what I would want a regularizer to provide — and it also means some images are nearly untouched, which is the same gentleness I would otherwise have to add by applying the patch with probability less than one. The fully-contained variant throws that spread away. I will go with sample-then-clip.

There is an edge case in this slice arithmetic that matters for the code, not just the idealized picture. Because Python slices exclude the upper endpoint, the removed side is y2-y1 = 2*(L//2), not L. For even L this is exactly L, so L=16 gives a true 16x16 patch, matching the 256 pixels computed above. But if someone passes an odd L it bites: with L=15, y1=16-7=9 and y2=16+7=23, so a centered box is [9,23)x[9,23), a side of 14 and an area of 14x14 = 196, i.e. 2*(15//2)=14 per side, not 15. The lengths I expect to tune are even, so this never affects the reported settings, but the operation is genuinely "remove 2*(L//2) per side, clipped," and I should keep that exact, not the idealized "L by L."

Next, the fill value, and here the order of operations in the pipeline matters more than it looks. The patch zeros pixels, but zero means very different things depending on whether it is applied before or after normalization. If I zero in raw pixel space I get a hard black patch; if I zero after per-channel normalization I get whatever raw value maps to normalized 0. With the CIFAR channel mean 125.3 and std 63.0 (both on the 0..255 raw scale), normalization sends raw r to (r-mean)/std. A hole that is zero in normalized space therefore decodes to raw = 0*std + mean = mean exactly — numerically, 0 -> 125.3 on the 0..255 scale, the dataset mean for that channel. The other order is much worse: a raw-black pixel (r=0) normalizes to (0-0.491)/0.247 ≈ -1.99, a strong negative outlier in every masked position. Feeding hundreds of -1.99 values into a batch-normalized network would shift the batch statistics far more than necessary and inject a spurious "this region is extremely dark" signal. So I want the mask applied after ToTensor and Normalize: the hole then sits at the per-channel mean, which carries no class information and barely perturbs batch statistics. The pipeline order is not cosmetic; it is what makes the patch a soft erasure instead of a black box.

There is no test-time scaling, and it is worth being explicit about why, since dropout trains the reflex to add one. Dropout scales at test time because it is approximating the expectation of a multiplicative random network, and the train/test distributions of activations must be matched. This operation is not a stochastic sub-network; it is data augmentation. Training sees modified images, evaluation sees ordinary images, exactly like random crops and flips, neither of which is undone at test time. So the network, loss, optimizer, labels, and learning-rate schedule all stay as they are.

Before committing, I want to revisit a more elaborate version I was tempted by, because if it is clearly better I should pay its cost. The idea: instead of a random location, store each image's maximally activated feature map, upsample it, threshold it, and hide that salient region in the next epoch — actively pushing the model off whatever cue it currently leans on. That is a more pointed attack on over-reliance, and it sounds like it should beat a blind random square. The cost is a saliency bookkeeping loop: a per-image stored map, an extra forward pass or cache, and a thresholding step every epoch. I cannot resolve this analytically, so the honest test is whether the targeted version actually beats a random fixed-size square on validation error. In the comparisons that decide it, the random version matches the targeted one — and once a one-line random transform equals the saliency-driven loop, the loop is pure overhead. So I drop it. This is a genuine subtraction: the targeted version was the first thing I built and it earned removal by not winning, not by being argued away.

The method I am left with is a tiny input transform: build an all-ones h by w mask, sample one or more centers uniformly, zero the clipped square around each center, expand the mask across channels, and multiply the normalized image by it. It is applied only during training, usually with one hole, with the side length selected on validation data. I expect the best length to depend on how fine-grained the task is, not just the image size: CIFAR-100 asks for a hundred-way distinction where the discriminating cue can be a small, specific detail rather than the whole object's coarse shape, while CIFAR-10's ten broader categories can lean more on holistic context. A patch large enough to help the ten-way task by forcing it off big, dominant cues could just as easily erase the one nuanced detail the hundred-way task actually needs — so I should expect a smaller optimal length as the classes get finer-grained, not a single global value. That argues for tuning the length per benchmark on the held-out split rather than fixing one number; the values that come out of that search are 16 for CIFAR-10, 8 for CIFAR-100, and 20 for SVHN, with the usual CIFAR schedule multiplying the learning rate by 0.2 at epochs 60, 120, and 160. What makes this different from the variants I rejected is not a new loss or architecture but where and how the evidence goes missing: a contiguous region erased at the input, at the dataset mean, before any channel can preserve it. Whether that translates into lower test error is the thing the experiments have to settle, but every design choice above is aimed squarely at that one mechanism.
