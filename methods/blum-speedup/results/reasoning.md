The starting intuition is the one algorithm designers use every day: a problem has many implementations, and analysis tries to find the one with the best asymptotic behavior. Even when constants and machine details differ, we usually expect there to be a meaningful optimum, or at least a stable complexity class that captures the best known and possibly best possible way to solve the problem. I want to know whether that expectation is forced by computability, or whether it is something we import from the well-behaved problems we usually study.

Let me first fix the setting carefully, because the whole question lives or dies on the axioms. Programs are enumerated as indices, and a complexity measure assigns to index `i` a cost `Phi_i(n)` on input `n`. The two axioms I will lean on are: `Phi_i(n)` is defined exactly when program `i` halts on `n` (same domain as the function), and the predicate "`Phi_i(n) = m`" is decidable. These are deliberately weak. They cover running time and space — for time, `Phi_i(n)` is the step count, defined iff the machine halts, and "does it halt in exactly `m` steps" is decidable by simulating `m` steps. So whatever I conclude is not an artifact of a perverse cost rule; it has to hold for the resources I actually care about.

Now the real question. Take a single total computable predicate `f`. It has infinitely many indices computing it. Is one of those indices asymptotically best? The naive picture says yes: start from any program, find an avoidable inefficiency, remove it, repeat until nothing avoidable remains. The implicit assumption is that "remove an inefficiency" is a process that terminates, i.e. that the space of programs for `f`, ordered by eventual cost, has a minimum (or a small class of minima). I should not assume that. The descending-chain intuition only forbids infinite descent in a well-founded order; eventual cost on almost all inputs is not obviously well-founded.

So let me ask the sharper question first, before trying to build anything: could there be a computable `f` where, given any program for it, I can always exhibit another program for the same `f` that is eventually faster — not by a constant, but by a margin I get to prescribe with a total computable function `r`? Concretely, I want: for every index `i` with `phi_i = f`, some index `j` with `phi_j = f` and `Phi_i(n) > r(n, Phi_j(n))` for almost all `n`.

Before building anything I want to rule out an apparent contradiction. Suppose such an `f` exists. Take any program `P_0`. Get a faster `P_1`. Get a faster `P_2`. Iterate. With `r(n,m) = 2m`, say, I get `Phi_{P_0}(n) > 2 Phi_{P_1}(n) > 4 Phi_{P_2}(n) > ...` for large `n`. So `Phi_{P_k}(n) < 2^{-k} Phi_{P_0}(n)` eventually. If I could take `k` as large as I like and have the inequality hold at a *fixed* `n`, repeatedly halving would eventually drive the cost at `n` negative, which is absurd for any natural-number-valued cost.

It doesn't collapse, and tracing through why is informative. Fix `n`. The chain gives, for each `k`, an inequality that holds for all but finitely many inputs — but the finite exception set depends on `k`, and nothing says it stays bounded. Concretely: step `k` guarantees `Phi_{P_{k-1}}(n) > 2 Phi_{P_k}(n)` only for `n >= N_k`. If the thresholds `N_k` were bounded by some fixed `N*`, then every `n >= N*` would satisfy the whole chain at once and the negative-cost absurdity would follow — so consistency forces the `N_k` to be unbounded. And once they are unbounded, for any fixed `n` I can find a `k` with `N_k > n`, which breaks the telescoped product at that link and the chain says nothing further about `n`. The negative-cost argument needs one fixed `n` to satisfy infinitely many steps of the product at once, and unbounded `N_k` denies me exactly that. So the statement is not self-contradictory, and the exception sets *must* grow without bound along the chain — that is a feature of the construction, not an embarrassment.

The same trace exposes a second thing I would have gotten wrong if I hadn't done it. The chain `P_0, P_1, P_2, ...` is real, but it cannot be effectively generated. If I had a single algorithm that, given `P_k`, produced `P_{k+1}` together with the exception bound `N_{k+1}`, I could in principle diagonalize to build a program that is faster than itself, or compute `f` with unboundedly decreasing cost in a uniform way — and that *would* contradict `f` being a fixed total function with a well-defined cost at each `n`. So the speedup, if it exists, has to be non-uniform: the faster program exists but is not handed to me by an effective procedure applied to the slower one. This is the price of consistency, and it tells me what kind of construction to look for. Not "an optimizer that improves any given program," but "a `f` so arranged that improvability is forced for every program while the improvement is not effectively locatable."

That reframing is what makes a construction plausible. The way to force improvability for *every* program at once is diagonalization against the cost functions. I want to define `f` by a process that, as it decides `f(n)` for more and more `n`, reserves the right to make any particular program `i` look bad — pay a large cost on infinitely many inputs — unless that program has already been "used up" by being beaten. Roughly, `f` is built to cancel each index: index `i` is forced to be eventually dominated by some later index, so no `i` can be the bottom. The construction has to interleave infinitely many such requirements and keep `f` total computable. I will not reproduce the full bookkeeping here, but the shape is a priority/cancellation argument over indices, and the consistency check above is exactly what guarantees the interleaving does not demand the impossible (a fixed input beaten infinitely often).

The strength of the claim depends entirely on how aggressive `r` is allowed to be. Two limiting cases make that concrete.

Case `r(n, m) = c·m` for a constant `c`. Then the statement reads: every program can be replaced by one that is eventually a factor `c` faster. On its own this looks like ordinary linear speedup and could be dismissed as a coding/batching artifact in models that admit it. So the constant-factor case is necessary but not impressive.

Case `r(n, m) = 2^m`, say, or any fast-growing total computable `r`. Now the replacement program `P_{k+1}` satisfies `Phi_{P_k}(n) > 2^{Phi_{P_{k+1}}(n)}` eventually, i.e. `Phi_{P_{k+1}}(n) < log_2 Phi_{P_k}(n)`. Chaining a finite number of such steps — valid once `n` exceeds the largest of that finite set of admissible thresholds — the cost drops like `Phi, log Phi, log log Phi, ...` over that finite depth: an enormously fast collapse, far beyond any constant factor. And crucially the construction is supposed to deliver this for *whatever* total computable `r` I name in advance, because `r` is a parameter of the build, not a consequence of it. The quantifier order is what makes this honest: `r` is chosen first, then `f` is constructed depending on `r`. It is not one `f` that beats every `r` simultaneously — that would be too strong and almost certainly false; the true, weaker statement is that for each `r` there is an `f`. Re-reading my target inequality, `Phi_i(n) > r(n, Phi_j(n))`, that is exactly the right shape — `r` is fixed outside the "for every `i` there is `j`" quantifier. So the strong-looking improvement is legitimate precisely because `f` is allowed to depend on `r`.

The "infinite chain" I used earlier as a hypothetical is in fact forced. Pick any index `i` for `f`. The theorem gives `j` computing the same `f` with `Phi_i > r(n, Phi_j)` a.e. But `j` is *also* just an index for `f`, so the theorem applies to `j` and yields `j'`, and so on — nothing in the statement privileges `i`; the hypothesis is only "computes `f`." So the chain regenerates indefinitely, and by the consistency trace above it does so without contradiction. That is the structural payload: not "this one program is bad," but "the relation of-eventual-cost on programs for `f` has no minimal element."

I should also state plainly what this does *not* say, because it is easy to overclaim. It does not say every computable problem behaves this way — it is existential, "there is an `f`," for each fixed `r`. Many concrete problems do have optimal algorithms, tight lower bounds, and robust completeness theories, and nothing here touches them. The "almost all inputs" qualifier is also doing real work: a replacement program is allowed to lose on finitely many small inputs, so the speedup is asymptotic, not pointwise — which is exactly why the consistency argument went through.

So: the descending-chain intuition that says "keep optimizing until you bottom out" presumes the order of programs-by-eventual-cost is well-founded, and nothing about computability grants that. For each Blum measure and each total computable `r`, diagonalizing against the cost of every index produces a total computable `f` such that every program computing `f` is eventually beaten, by the `r`-margin and on almost all inputs, by another program computing the same `f` — a genuine counterexample to the best-algorithm picture, not a coding trick. Complexity, in full generality, can be a layered structure with no bottom rather than a single asymptotic destination. The part of this I have verified directly is the consistency argument and the quantifier order; the existence of `f` itself rests on the diagonal construction, whose bookkeeping I have only sketched here.
