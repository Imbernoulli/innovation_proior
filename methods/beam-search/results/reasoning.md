Let me start from the thing that actually hurts. I have an utterance — a few seconds of speech, sliced into ten-millisecond samples — and a network that encodes every legal sentence and every pronunciation of every word as paths of states. Decoding means finding the path through that network whose states best match the acoustics. That is a search over a layered graph: a decision point advances me from one network state to the next, layer `t` holds the network states I could be sitting in after `t` advances, and each node fans out to a handful of successor states. The score of a path is the product of per-step match probabilities. I want the highest-scoring path.

The naive thing is to keep every live path. But every state branches, so the number of live paths roughly multiplies at each step; after a few hundred steps the frontier is astronomically large. I cannot store it, let alone score it. So the whole game is: which partial paths do I carry forward, and which do I throw away, step by step?

Let me lay out what I already know how to do, because the answer is going to be a compromise between these.

One option: search all paths in parallel and merge. If two different paths arrive at the *same* network state at the same step, the future is identical from there on, so I only need to keep whichever arrived with the better score — the worse one can never overtake. That collapse is what makes a forward sweep over the network exact and tractable: I keep one best score per network state and update them all, step to step, never backtracking. It finds the globally optimal path in time proportional to (number of states) × (number of steps). Beautiful — and that's essentially what the Dragon system does, treating recognition as a clean probabilistic computation over a Markov network with no backtracking: it updates the state probabilities of the network on every ten-millisecond sample. The catch is right there in the cost: *every* network state gets its probability updated on *every* sample. When the network is large and each update is heavy, that full-breadth sweep runs forty-five to two hundred times real time. The exactness is paid for by touching the entire breadth at every step, and most of that breadth is hopeless — vast regions of the network that no good path will ever go near, updated anyway. One lever to cut that cost is segmentation — grouping consecutive samples that clearly belong together into a single larger unit, so the number of forward steps drops below the number of raw samples; that shrinks the depth but not the breadth, and the breadth is the real problem.

The opposite option: keep a single path. At each step, extend the one path I'm holding by its single best-scoring successor. Memory is nothing, time is linear in the number of steps — exactly the budget I want. This is the spirit of a best-first commitment, like Hearsay-I chasing the most promising syntactic path. But a single greedy commitment is brittle: one frame where the locally best successor is globally wrong, and the whole decode is ruined, with no second candidate in hand to recover with. Hearsay-I patches this with backtracking — when the committed path goes bad, unwind and try another. And backtracking, in a graph this deep and bushy, is exactly the cost I'm trying to escape: every dead end throws away the work spent reaching it, and the unwinding dominates the runtime. I want forward-only.

So I'm pinned between two extremes. Full breadth: exact, backtrack-free, but updates everything — too slow. Single path: cheap and forward-only, but so brittle it needs backtracking to be usable, which kills the "forward-only" win. Let me look harder at *why* each extreme is wrong, because the fix should fall out of that.

Full breadth is wrong because it spends effort uniformly. At any frame, the live network states are wildly unequal — a few are matching the acoustics well, and an enormous tail is matching badly and getting worse. The exact sweep updates the awful tail with the same diligence as the promising few. That's the waste. The single path is wrong for the mirror reason: it spends *all* its effort on exactly one state, so a single mistake has nowhere to recover.

Both failures point at the same missing idea. I don't want to keep *all* states, and I don't want to keep *one* state — I want to keep the *good* states and discard the obviously-doomed tail. How many is "good"? That's a knob. Call it the breadth I allow myself: at each frame, of all the successor states my current live set generates, keep only the most promising handful and throw the rest away. Not all of them (that's the exact sweep), not one of them (that's greedy), but the top few.

Let me make this precise as a forward sweep. I hold a set of live partial paths — the *frontier*. To go from frame `t` to frame `t+1`: take every path in the frontier, generate all its successor extensions, score each extension, then prune the pool back down to the best few, and that pruned set is my new frontier. Repeat to the last frame; the best complete path in the final frontier is my answer. Memory per frame is bounded by that "few," and total work is (few) × (branching) × (frames) — linear in depth, no backtracking, no full-breadth update. The doomed tail is never touched because it never survives a single pruning step.

What is "the best few" measured by? Here's a subtlety I have to get right, because it's the whole difference between this working and not. If I rank surviving paths purely by score-so-far, I bias toward paths that happen to have had easy frames early; a path that's slightly behind now but sitting in a region about to match beautifully gets cut. What I actually want to rank by is an estimate of the *final* score of the best completion of each path — score accumulated *plus* an estimate of what's still to come. That's the same instinct as evaluating a node by cost-so-far plus a heuristic estimate of cost-to-go, the way an informed best-first search orders its frontier with `f = g + h`. Here `g` is the score along the partial path and `h` is an optimistic estimate of the rest. The pruning keeps the top few by `g + h`, not by `g` alone. Good — the evaluation function is doing real work: it lets a currently-behind-but-promising path survive.

Now, how wide should "the few" be? Watch the two limits and the knob reveals itself. If I let the surviving set be unbounded — keep *every* successor every step — I prune nothing: the surviving set is the whole layer, so I'm back to the exact full-breadth sweep, which is exhaustive (exponentially many distinct paths in a tree, or, once I merge paths that reach the same network state, the full-label forward sweep over the network). If I shrink the surviving set to exactly one, I keep only the single best successor each step, and I'm back to greedy hill-climbing down one path. So this one number — the width of the surviving set — is a dial that slides continuously from "exact and unaffordable" at one end to "greedy and brittle" at the other. Every intermediate setting trades a little optimality for a lot of speed. That dial is the whole idea. Call its value `W`, the width of the kept frontier.

Let me think about which way quality moves as I turn the dial. Bigger `W`: I keep more near-miss alternatives around the best path, so a path that the evaluation function temporarily under-ranks has more room to survive and later prove itself — fewer good completions get prematurely pruned, so on the whole the answer tends to get better, while cost goes up linearly in `W`. Smaller `W`: cheaper and faster, but the pruning is more aggressive and more likely to cut the eventual winner. I should be careful not to oversell this as a guarantee — because the evaluation function is only an estimate of the final score, a wider beam can occasionally crowd out the state that a narrower beam would have ridden to the win, so quality is a *trend* in `W`, not a monotone law; what *is* certain is that once the beam is wide enough to never prune the eventual winner, I recover the exact optimum, and at `W = ∞` I'm back to the exact sweep. The cost stays *linear* in `W` and in depth (taking the per-step evaluation and the top-`W` selection as the unit of work), never exponential. That's the property the exact methods couldn't give me: a tunable, predictable, linear-time search. In speech terms, keeping a beam of near-miss alternatives around the best path and pruning the rest at each decision point contains the exponential growth without any backtracking, and gets a ten-to-twenty-five-fold speedup over updating the whole network — and I can spend more `W` exactly where paths get confusable and the search naturally narrows where the right path is obvious.

Now let me worry about *how* I prune, because there's a design choice hiding in "keep the top few" that I glossed. Two ways to define the surviving set:

One: keep a fixed count — sort the successor pool by `g + h` and take the top `W`. Clean, the memory bound is exactly `W`, trivially predictable.

Two: keep everything within a threshold of the layer's best — survive if your score is within some factor of the best score at this frame. This is adaptive: where many paths are nearly tied (the acoustics are ambiguous, lots of confusable candidates) the surviving set swells; where one path dominates (the right word is obvious) it collapses to almost nothing. That's appealing because it spends breadth exactly where the problem is hard and almost none where it's easy — the variable-width behavior I wanted. The cost is that I can't bound the memory a priori; a genuinely ambiguous stretch could blow the surviving set up.

Both are versions of the same move — prune to a *beam* of the most promising survivors, the rest discarded. The threshold version is the more natural fit for the speech decoder, where "best is whatever is within a heuristic threshold on the state probabilities" lets the number of live paths float: many where they're confusable, few where the answer is obvious. The fixed-count version is the more portable one when I want a hard memory guarantee. I'll think of them as the same algorithm with two pruning rules; the fixed `W` is the one I can reason about cleanly and the one I'll build, with the understanding that a threshold can replace the hard cutoff when adaptivity matters more than a memory bound.

Let me also question the cost of the evaluation itself, because I waved at "score each extension" as if it were free. The accurate evaluation — the `g + h` that orders paths well — can be expensive (a real estimate of the best completion's total cost is not cheap). If I run it on *every* successor of every frontier node, I might spend more on ranking than the pruning saves. There's a cheaper, myopic score available too — just the immediate cost of the last decision. So a refinement: use the cheap local score first to knock the successor pool down to a larger intermediate set — a *filter width* — and only then pay for the expensive accurate evaluation on those survivors, finally keeping the best `W`. Two-stage pruning: a cheap filter to a filter width, then the costly global evaluation to the beam width. It buys most of the quality of full accurate ranking at a fraction of the evaluation cost. I'll keep this in mind as the knob to reach for when the evaluation function is the bottleneck; for a problem where evaluation is cheap I can skip it and rank the whole pool directly.

Let me pressure-test the core idea on something concrete and away from speech, to convince myself it isn't a speech-only trick — a 0/1 knapsack. `n` items, each with a weight and a value, a capacity, maximize total value. Build the solution one item at a time: layer `t` decides item `t` — take it or skip it. A state records how many items I've decided, the value accumulated, and the capacity left. The exact frontier doubles each layer — `2^n` subsets — so it's the same explosion. Beam search says: at each item, generate both children (take / skip) of every live state, rank the pool, keep the top `W`. What's `g + h` here? `g` is the value packed so far; for `h` I want an *optimistic* upper bound on the best completion. The clean optimistic bound is the fractional (LP) relaxation: fill the remaining capacity with the remaining items in *descending value-density order*, allowing the last item to be taken fractionally. The order matters — it has to be descending density, because that fractional fill is the optimum of the LP relaxation of the integer remainder, and the LP optimum is an upper bound on the integer optimum; fill them in any other order (say their original index order) and the number I get is just *some* feasible fractional value, which can fall *below* the true best completion and would no longer be an over-estimate. So: descending density, fractional last item, and it never *under*estimates the true best completion — an over-estimate of remaining value, exactly the right thing to *rank* by because it ranks a state by the best it could possibly still become. Keep the top `W` by `value-so-far + fractional-bound`. The best feasible (fully-decided, or partially-decided-and-the-rest-skipped) state I ever see is the answer.

And I can predict the behavior before running it, from the structure alone. At `W = 1` I keep only the single most optimistic state each layer — essentially a greedy fill — and it'll land somewhere short of optimal because one early commitment can be wrong. As I raise `W`, I carry more near-miss subsets forward, so the subset that the bound temporarily under-ranks has room to survive and ultimately win; the achieved value should climb toward the true optimum, and at some finite `W` (when the beam is wide enough to never prune the eventual winner) it *hits* the optimum exactly — and any wider `W` just spends more for the same answer. I shouldn't promise the climb is strictly monotone — since the bound only estimates the completion, there's no law forbidding a wider beam from briefly doing worse on a given instance — but the trend is up, and the flatten-at-optimum endpoint is guaranteed once the beam covers the winner. That's the signature I expect: a value that climbs with `W` and then flattens at the optimum, with cost growing linearly in `W` the whole way. If I had the exact optimum from brute force on a small instance, the gap would shrink to zero as `W` grows. That climb-then-flatten is the whole value proposition made visible.

There's a failure mode I should name now, because wide `W` doesn't automatically fix it. Suppose I crank `W` up, expecting proportionally more exploration — and instead the achieved value barely improves. Why? Look at *what* the top-`W` states at some deep layer actually are. The beam is a greedy filter at every layer, so a single strong early decision tends to dominate: the `W` survivors are very often `W` tiny variations of the *same* strong prefix — take-item-3-and-5, take-3-and-5-and-skip-7, take-3-and-5-and-9 — all descended from one or two lucky early choices. They look like `W` different states but they're really one lineage wearing `W` coats. The beam has *breadth* in count and almost none in *diversity*. A genuinely different, initially-unpromising prefix that needed a few more layers to reveal its quality got pruned at layer two and never came back. So doubling `W` mostly buys more near-duplicates of the incumbent lineage, not new lineages — diminishing returns that look like the method plateauing.

How do I inject diversity without just making `W` enormous? A few moves are available. I could dedup — if two states are effectively identical (same key cells, by a hash), keep one; that stops the beam wasting slots on literal twins. I could refuse to let too many same-score states crowd the beam. Or I could pick survivors *stochastically* — instead of the deterministic top `W`, sample `W` of them with probability rising in their score, so an occasional underdog lineage survives the cut and gets a chance to develop. Each of these widens the *kinds* of states in the beam rather than their count.

But there's a cleaner structural fix, and it comes from staring at *what the beam discards and never reconsiders*. The plain beam, at each layer, commits to one frozen set of `W` survivors and the rest are gone forever. The states that were ranked 11th-through-50th at layer two — some of which were one good decision away from becoming the best lineage — are simply lost. What if I never permanently discard them? Keep, for *each layer*, a priority queue of every state that ever reached that layer, ordered by the evaluation. Now do this: one *pass* walks the layers in order, and at each layer it pops just the *single* best un-expanded state from that layer's queue, expands it, and pushes its children into the next layer's queue. One pass is therefore exactly a width-one beam search run to the end — but it leaves all the runners-up sitting in the queues. Then I start a *second* pass from layer zero: pop the *next*-best state at each layer, expand it, push its children. This second pass tends to develop a different lineage — built from states the first width-one pass left sitting in the queues — all the way to the end. A third pass does it again. Keep launching passes until my time budget is spent.

Why is this better? Each pass is a fresh width-one descent that, at every layer, takes the *next*-best unexpanded state rather than the same best one the earlier passes already took, so it tends to follow a different lineage than its predecessors. After `k` passes I've grown on the order of `k` decent solutions with more variety between them than `W` near-duplicates of a single lineage. The diversity I was missing comes from *time* (more passes) rather than from *width* (a fatter frozen beam). And there's a bonus when I'm working against a fixed time budget: I never have to choose `W`. I just loop passes until time is up — it's an anytime search that keeps improving the best-found solution the longer I let it run, and time management collapses to "run until the budget is spent." The price is memory: I'm now retaining all the states across all layers in those queues, not just the previous layer's `W`. That's the trade — keep everything, get diversity and anytime behavior; or keep only `W`, get a hard memory bound but risk the one-lineage trap.

So the family is clear in my head now. The core is bounded-breadth forward search: layer by layer, expand the live set, rank by an evaluation that estimates the final score, keep the best `W` (or everything within a threshold), never backtrack — a dial from exact at `W = ∞` to greedy at `W = 1`. When accurate evaluation is the bottleneck, filter cheaply to a wider intermediate set first, then evaluate expensively down to `W`. When the frozen beam collapses into one lineage, restore diversity — dedup, sample survivors, or, best of all, keep a per-layer queue and run repeated width-one passes until time runs out for an anytime, self-tuning search.

Let me write the core — knapsack, deterministic fixed-`W`, ranked by the optimistic fractional bound — and watch the climb-then-flatten I predicted.

```python
from dataclasses import dataclass, field

@dataclass(order=True)
class State:
    key: float                              # rank by value-so-far + optimistic bound
    idx: int = field(compare=False)         # how many items decided (the layer)
    value: int = field(compare=False)       # value packed so far  (this is g)
    cap: int = field(compare=False)         # remaining capacity
    chosen: tuple = field(compare=False)    # item indices taken

def bound(value, cap, idx, items):
    # h: LP-relaxation optimistic completion -- fill the rest in DESCENDING
    # value/weight order, last item fractional. That descending order is the LP
    # optimum of the integer remainder, so it never underestimates the best
    # completion; it is exactly what we want to rank by.
    b = value
    for w, v in sorted(items[idx:], key=lambda iv: iv[1] / iv[0], reverse=True):
        if w <= cap:
            cap -= w; b += v
        else:
            b += v * (cap / w); break
    return b

def beam_search(items, capacity, W):
    n = len(items)
    layer = [State(0.0, 0, 0, capacity, ())]      # frontier at layer 0
    best = layer[0]
    for _ in range(n):                            # one layer per item, forward only
        pool = []
        for s in layer:                           # expand every live state
            w, v = items[s.idx]
            # skip item s.idx
            pool.append(State(bound(s.value, s.cap, s.idx+1, items),
                              s.idx+1, s.value, s.cap, s.chosen))
            # take item s.idx, if it fits
            if w <= s.cap:
                nv, nc = s.value + v, s.cap - w
                pool.append(State(bound(nv, nc, s.idx+1, items),
                                  s.idx+1, nv, nc, s.chosen + (s.idx,)))
        pool.sort(key=lambda s: s.key, reverse=True)   # rank by g + h
        layer = pool[:W]                               # PRUNE to beam width W
        for s in layer:                                # track best feasible value
            if s.value > best.value:
                best = s
    return best.value, best.chosen
```

The pruning is the single line `layer = pool[:W]`: generate the whole layer's offspring, rank them by `value-so-far + optimistic-bound`, keep `W`, discard the rest, never look back. Drop in a brute-force `2^n` enumerator on a small instance for the exact optimum, sweep `W` from 1 upward, and the achieved value should climb and then flatten exactly at that optimum — the dial doing its job, exactness recovered at finite width, cost linear in `W` throughout.

So the chain is: the exact full-breadth forward sweep is correct but touches all the doomed breadth and is too slow; a single greedy path is cheap and forward-only but one bad decision ruins it and recovering needs the backtracking I'm trying to avoid; the fix is to keep neither all states nor one but the best `W`, ranked by an estimate of the final score, pruned forward each layer with no backtracking — a width dial from exact to greedy, with cost linear in depth and in `W` once the per-step evaluation and the top-`W` selection are taken as the unit of work; filter cheaply before evaluating expensively when ranking is the bottleneck; and when the frozen beam collapses to a single lineage, recover diversity by deduping, sampling, or keeping per-layer queues and replaying width-one passes until the time budget is spent.
