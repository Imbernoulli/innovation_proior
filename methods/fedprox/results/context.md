## Research question

We want to train a single shared model across a large network of devices — phones,
sensors, wearables — without ever moving their data off-device. Formally we minimize a
finite-sum objective spread across the network,

```
min_w  f(w) = sum_{k=1}^N p_k F_k(w),   p_k = n_k / n,   sum_k p_k = 1,
```

where `F_k(w) = E_{x~D_k}[ f_k(w; x) ]` is device `k`'s local empirical risk over its own
data distribution `D_k`, `n_k` is its number of samples and `n = sum_k n_k`. The `F_k` may be
non-convex (neural nets). Two features distinguish this setting from ordinary data-center
distributed optimization:

- **Statistical heterogeneity.** Each device's data is generated by its own usage, so the
  `D_k` are *non-identically distributed*. The local risks `F_k` therefore differ from one
  another and from the global `f`; the minimizer of any one `F_k` need not be near the
  minimizer of `f`. This is the departure from the IID assumption that data-center
  distributed optimization is built on.
- **Systems heterogeneity.** Devices differ in compute, memory, network (3G/4G/wifi),
  and battery, and many are intermittently offline. Different devices can do very different
  amounts of work in a fixed wall-clock window.

Communication is the binding constraint: rounds are slow and expensive, on-device datasets
are small relative to total compute, so the regime is "do a lot of local computation, talk to
the server rarely, and have only a small fraction of devices participate per round." The
question is how to train the shared model under non-IID data and under devices that do
different amounts of work per round.

## Background

The prevailing recipe is **local-update / low-participation** federated optimization: rather
than sending a gradient per minibatch to the server (which would need far too many rounds), a
small subset of devices each run *several* local SGD steps on their own data and send back the
resulting model, and the server merges these models. This trades extra on-device computation
for far fewer communication rounds — the right trade when communication dominates.

The load-bearing concepts:

- **Local empirical risk as a surrogate.** Each device replaces the global objective with its
  own `F_k`, which it can evaluate and differentiate locally. The "amount of local work"
  (number of local epochs / iterations) is a tunable knob that moves the operating point
  along the computation-vs-communication curve: more local work per round means fewer rounds,
  and a more-exactly-solved *local* problem.
- **Model averaging from a shared start.** A diagnostic observation about plain
  weight-averaging: average two networks trained from the *same* random initialization on
  different data and the averaged model lands in a good, low-loss region of the loss surface;
  average two networks trained from *different* initializations and the average lands in a bad
  region, worse than either parent. So a *shared* starting point each round is what makes
  averaging the returned models sensible: the local solutions sit in a common basin and their
  average is meaningful.
- **The local-epoch trade-off.** With heterogeneous `F_k`, running *more* local epochs drives
  each device toward the optimum of *its own* `F_k`, which under non-IID data is displaced
  from the global optimum. The number of local epochs that works depends on the data and the
  available resources, and would in principle differ by device and by round.
- **Regularized local subproblems as a known optimization tool.** Data-center distributed
  methods stabilize local subproblems with regularization or constraints around the current
  iterate. The generic mathematical fact: if a differentiable function has Hessian bounded
  below, `∇^2 F >= -L_- I`, adding a quadratic regularizer centered at a point shifts the
  Hessian upward by the regularization coefficient, so with enough curvature added the local
  subproblem becomes strongly convex even if `F` itself is non-convex. This is standard
  proximal-point / trust-region machinery.
- **Measuring how non-IID the network is.** To quantify convergence under heterogeneity, one
  bounds how much the local gradients disagree with the global gradient. A family of
  "growth"/"diversity" conditions of the form
  `E_k||∇F_k(w)||^2 ≤ B^2 ||∇f(w)||^2` (Schmidt & Le Roux 2013's strong growth condition;
  Vaswani et al. 2018; Yin et al. 2018) does this in the IID / over-parameterized
  setting: `B = 1` when all `F_k` coincide, and `B` grows with disagreement. A bounded-variance
  condition `E_k||∇F_k(w) - ∇f(w)||^2 ≤ σ^2` is a simpler, more restrictive sibling.

## Baselines

These are the methods a new federated optimizer would be measured against.

**FedSGD (McMahan et al., AISTATS 2017).** The minimal federated method. A `C`-fraction of
clients each compute the gradient of their local risk at the current global model,
`g_k = ∇F_k(w_t)` over all their local data, and send it up; the server takes one
sample-weighted gradient step,

```
w_{t+1} = w_t - eta * sum_k (n_k / n) g_k = w_t - eta ∇f(w_t),
```

since `sum_k (n_k/n) g_k = ∇f(w_t)`. This is exact full-batch (over participating data)
gradient descent on `f`: one global gradient step per communication round.

**FedAvg (McMahan et al., AISTATS 2017).** Push more computation into each round. From a
shared `w_t`, each selected device `k` runs `E` epochs of local SGD on `F_k` with step `eta`,
producing `w_k^{t+1}`; the server averages the returned models weighted by sample count,

```
each k in S_t:  w_k^{t+1} <- E epochs of SGD on F_k starting from w_t
server:         w^{t+1}   = sum_{k in S_t} (n_k / sum_{j in S_t} n_j) * w_k^{t+1}.
```

`E = 1` (one full-batch local step) recovers FedSGD; larger `E` does more local work and cuts
the round count by `10x–100x` in practice. It is the de-facto method. Systems heterogeneity is
handled by dropping any device that cannot complete `E` epochs within the round's time window.

**Elastic Averaging SGD (Zhang, Choromanska & LeCun, NeurIPS 2015).** A data-center
distributed method that couples each worker's local parameters `x_i` to a shared center
variable `hat_x` through an elastic (quadratic) force `(rho/2)||x_i - hat_x||^2`, letting
workers explore locally while a moving average pulls them toward consensus. It maintains a
separate moving-average center with its own update rule, uses SGD as the local solver, and has
been analyzed for quadratic objectives.

**DANE / AIDE (Shamir, Srebro & Zhang, ICML 2014; Reddi et al. 2016).** Communication-efficient
distributed Newton-type methods for the data-center setting. Each machine solves a local
subproblem that augments its local function with both a quadratic anchor *and* a linear
gradient-correction term,

```
w_i^{t} = argmin_w  F_i(w) - (∇F_i(w_{t-1}) - eta ∇f(w_{t-1}))^T w + (mu/2)||w - w_{t-1}||^2,
```

so the local gradient direction is corrected by the gap between the local and the global
gradient. The correction uses the full global gradient `∇f(w_{t-1})` each round, computed with
all machines participating.

**Parallel / local SGD analyses (Stich 2018; Yu et al. 2018; Wang & Joshi 2018; Zhou & Cong
2017; and the IID parallel-SGD line).** A body of work that proves convergence for
local-update schemes resembling FedAvg, assuming IID data or that all devices participate every
round, with the local solver fixed as SGD/GD.

## Evaluation settings

The natural yardsticks already in use for federated optimization:

- **Synthetic non-IID classification.** Generate per-device data from a softmax model
  `y = argmax(softmax(W_k x + b_k))`, `x in R^60`, with `W_k, b_k ~ N(u_k, 1)`,
  `u_k ~ N(0, α)` and inputs `x_k ~ N(v_k, Σ)`, `Σ` diagonal with `Σ_{jj} = j^{-1.2}`,
  per-device mean `v_k` drawn from a `N(B_k, 1)`, `B_k ~ N(0, β)`. The two knobs `α`
  (how much local models differ) and `β` (how much local data differs) let you dial
  statistical heterogeneity precisely; an IID variant fixes shared `W, b` and a common input
  distribution. 30 devices, samples per device following a power law.
- **MNIST**, multinomial logistic regression (convex), distributed over 1,000 devices with
  each device holding only 2 digit classes and a power-law sample count — a heterogeneous,
  unbalanced split.
- **FEMNIST**, the 62-class federated EMNIST, multinomial logistic regression, with a small
  subset of classes per device (heterogeneous partition), ~200 devices.
- **Shakespeare** next-character prediction, a stacked character-level LSTM (8-D embedding,
  two LSTM layers, dense + softmax over the character vocabulary, 80-character unroll), with
  each speaking role in the plays as a separate device — naturally non-IID and unbalanced.
- **Sent140** tweet sentiment, a two-layer LSTM with pretrained GloVe embeddings, one Twitter
  account per device.
- Protocol: a small fixed number of devices selected per round (e.g. 10), local SGD with a
  grid-searched, non-decayed learning rate and a small batch size, an 80/20 per-device
  train/test split. Systems heterogeneity is simulated by assigning different devices
  different amounts of local work and by forcing a fraction of them to be stragglers.
  Reported quantities are training loss and test accuracy versus communication rounds.
  Standard federated tooling (LEAF benchmark, TensorFlow Federated) provides the datasets and
  harness.

## Code framework

The substrate is the local-update federated harness shared with the baselines: a server loop
that selects a subset of devices, broadcasts the current global model, collects the models the
devices return, and merges them; and a client routine that loads the broadcast model and runs
a local solver on the device's data. The baseline merge rule is sample-weighted averaging, and
plain client-side SGD already exists. The unresolved slots are the client-side update rule and
the policy for devices that complete different amounts of work.
