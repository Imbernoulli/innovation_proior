Let me start from what actually breaks when I run the obvious recipe. I have a base model, a verifiable reward — the answer is right or it isn't, +1 or −1 — and I'm training it to reason for a long time before answering, tens of thousands of tokens of chain of thought. No critic: training a value network the size of the policy is too expensive, and with the reward arriving only at the last token a per-token value would be junk anyway, so I do the group-relative thing. For each prompt q I sample a group of G responses from the old policy, score each one, and use the group's own mean as the baseline: the advantage of every token in response i is the centered, std-normalized scalar reward Â_{i,t} = (R_i − mean(R))/std(R), broadcast across all of o_i's tokens, and I optimize the clipped surrogate in the importance ratio r_{i,t} = π_θ/π_old. This is just the GRPO recipe, and in principle it should work. In practice the run is well below the known strong runs and it's sick in three specific ways, and I want to diagnose each before I touch anything, because if I just start adding tricks I'll never know which wound I'm treating.

First wound: the entropy collapses. Early in training the policy's output entropy drops fast and the G responses for a given prompt become nearly carbon copies of each other. That's exploration dying — the policy commits to a deterministic mode before it's found a good one, and once the group is identical there's nothing to learn relative to. Let me figure out *why* the clipped surrogate would do this, because the clip is supposed to be a neutral trust region. Take a positive advantage, Â > 0, meaning I want to push these tokens up. The clip bounds the ratio to 1+ε with ε = 0.2, so a token can grow at most to π_old·(1+ε). Now look at two tokens: one already likely, π_old = 0.9, and one unlikely, π_old = 0.01. The likely one is allowed to climb to 0.9·1.2 = 1.08 — effectively uncapped, it can go all the way to near 1. The unlikely one is capped at 0.01·1.2 = 0.012 — it can barely move. So the *same* clip threshold is generous to a token that's already winning and miserly to a token that's still exploring. Step after step, the high-probability "exploitation" tokens get reinforced freely while the low-probability "exploration" tokens are throttled, and the distribution sharpens onto a few tokens. That's the entropy collapse, and it's baked into using one symmetric clip for tokens at wildly different probabilities. If I just raise ε uniformly I'd loosen the lower clip too, and loosening the lower bound lets the surrogate drive a token's probability all the way to 0 — that *suppresses* exploration tokens to extinction and collapses the sampling space from the other side. So the move has to be asymmetric: decouple the lower and upper clip into ε_low and ε_high, and raise only ε_high. Keep ε_low where it is so I don't annihilate low-probability tokens; lift ε_high so an exploration token at 0.01 has real room to grow. Something like ε_low = 0.2, ε_high = 0.28. The asymmetry isn't cosmetic — it's the whole point: the upper clip was the thing strangling exploration, so that's the only one I want to loosen.

Second wound: a growing fraction of my groups produce zero gradient. Watch what happens when all G responses to a prompt are correct. Every R_i = +1, so mean(R) = +1, so R_i − mean(R) = 0 for every member — the advantage is exactly zero for the whole group, and a zero advantage means a zero policy gradient. Same thing if all G are wrong. So any prompt the model has fully mastered (or has no chance on) contributes nothing, and as training proceeds the model masters more prompts, so the count of all-correct groups climbs and climbs. The batch is nominally full but the *effective* number of gradient-bearing prompts keeps shrinking, which both shrinks the gradient magnitude and inflates its variance — fewer real samples behind each update. The fix has to keep the batch full of prompts whose group is mixed. So: over-sample, and filter out any prompt whose group is all-correct or all-wrong, keep resampling until the batch is filled with prompts where 0 < (number correct) < G. Dynamic sampling — the per-batch generation cost floats, but generation is dominated by the long-tail slow samples anyway, so filtering the trivially-resolved prompts barely costs wall-clock and gives every prompt in the batch a live gradient. I can write that straight into the objective as a constraint: s.t. 0 < |{o_i : is_equivalent(a, o_i)}| < G.

Now the third wound, and this is the one I want to sit with, because it's the subtle one and it's about length. The average response length drifts up over training. Everyone reads rising length as the model learning to reason more, the "emergence" of long chain-of-thought. But when I actually look, it's the *incorrect* responses that grow fastest and longest, and a lot of the growth is garbage — repetition, padding, gibberish — while accuracy stalls. That doesn't look like reasoning emerging. That looks like the optimizer being *paid* to be verbose. So let me ask: is there something in the objective that rewards length per se, independent of quality? Stare at the loss aggregation. GRPO computes the loss sample-level: it averages the per-token losses *within* each response first, then averages across responses. Written out, the weight in front of response i's token sum is (1/G)·(1/|o_i|). So a single token's weight in the total loss is (1/G)·(1/|o_i|) — and that *shrinks as |o_i| grows*. A token sitting inside a long response counts for less than a token inside a short one.

Let me trace what that does, sign by sign, because the direction is everything. Suppose response i is correct, Â_{i,t} > 0, so the gradient on each of its tokens pushes their probability up. The magnitude of that per-token push is proportional to (1/|o_i|). So a *short* correct response gets a *bigger* per-token push than a *long* correct response. The optimizer is being told: when you're right, prefer to be brief — short correct answers get reinforced harder. Now suppose response i is incorrect, Â_{i,t} < 0, so the per-token gradient pushes probabilities *down*, a penalty. That penalty is also scaled by (1/|o_i|). So a *long* wrong response gets a *smaller* per-token penalty than a short wrong one. The optimizer is being told: when you're wrong, it's cheaper to be long — pad the wrong answer and you'll be punished less per token. Put the two together and the dynamics are obvious: correct answers are pressured shorter, incorrect answers are pressured longer, and since at any moment a real fraction of responses are incorrect, the *average* length climbs, dragged up by the wrong ones bloating. The "emergent long reasoning" is partly just this aggregation artifact — the 1/|o_i| is an unintended length lever, and it's pointing the wrong way on the responses that matter. And it's not even unique to this recipe; the same per-sequence length normalization is sitting in a lot of open-source PPO loss code, which averages each response by its own valid-token count — a habit probably inherited from pretraining, where dividing by a fixed context length is just a numerical-stability convenience. Here, where the divisor |o_i| is *not* constant, it silently injects a length bias.

So the length bias is real and I can read its direction straight off the 1/|o_i| factor. There's a fourth, related thing while I'm in the advantage: I also divide the centered reward by std(R), the per-group standard deviation. Groups with small spread — prompts that are nearly all-correct or all-wrong — get a large weight, because dividing by a small std blows the advantage up. Advantage normalization is a standard variance-reduction trick, but it's normally done over a whole *batch*; doing it per-prompt makes the objective weight prompts unevenly by how easy or hard they happened to be, a difficulty bias on top of the length bias. I'll keep that in the back of my mind — both the 1/|o_i| and the /std are normalizers that look innocent but reshape the objective.

How do I kill the length bias at its root? The disease is that a token's weight depends on the length of the response it happens to live in. The cure is to make every token count equally, full stop. So instead of averaging within a response and then across responses, aggregate at the *token* level across the whole batch: weight every token by 1/(Σ_i |o_i|), the reciprocal of the total token count in the batch, with no per-response 1/|o_i| in sight.

  J(θ) = E[ (1/ Σ_i |o_i|) Σ_i Σ_t min( r_{i,t} Â_{i,t}, clip(r_{i,t}, 1−ε_low, 1+ε_high) Â_{i,t} ) ].

Now every token has the same weight regardless of its host sequence's length. Walk the consequences back through the same sign analysis: a long correct response, having more tokens, now contributes *more* total influence than a short correct one — which is what I want, more reasoning that's right should matter more — and from the perspective of any single token, a pattern that raises or lowers reward is promoted or suppressed by the same amount no matter how long the response it appeared in. The pad-the-wrong-answer loophole closes: each extra junk token in a long wrong response now carries its own full-weight penalty instead of a discounted one. Longer sequences influence the gradient in proportion to their length, not in inverse proportion to it. That's the token-level policy gradient loss, and it falls out of "make the per-token weight constant" rather than being a knob I tuned.

There's still the truncation problem, which is its own length pathology. I cap generation at some maximum length and truncate anything that runs over. The default move is to give a truncated sample the punitive reward, −1, treat it as a failure. But think about what that does: a response can be *reasoning soundly* and simply not have finished within the budget, and I'm slapping it with the full failure penalty purely for being long. That's reward noise — I'm telling the model its correct-so-far reasoning was bad, which confuses the signal about what good reasoning even is. As a diagnostic, the cleanest version of "truncation reward is noise" is to mask out the loss of every truncated sample entirely, so they contribute no gradient at all. Overlong filtering would localize whether the cliff is the source of noise. But filtering has its own failure mode if I stop there: if truncated samples contribute nothing, the model gets no signal at all about length, so over time long responses can just accumulate — it never learns *not* to run over.

So I want something gentler than a hard −1 cliff at the boundary and gentler than blanket masking: a length-aware penalty that says "going over is bad, and the further over you go the worse it is," ramping in smoothly rather than detonating at the cap. Set up a buffer of width L_cache just below the hard maximum L_max. Inside that buffer, the penalty grows linearly from 0 (at the start of the buffer) to −1 (at the hard cap), and past the cap it's a flat −1. Let me write it as a piecewise function of the length |y|, added to the rule reward:

  R_length(y) = 0                                     if |y| ≤ L_max − L_cache,
              = ((L_max − L_cache) − |y|) / L_cache   if L_max − L_cache < |y| ≤ L_max,
              = −1                                     if |y| > L_max.

Check the endpoints: at |y| = L_max − L_cache the middle expression is 0, continuous with the first piece; at |y| = L_max it's (−L_cache)/L_cache = −1, continuous with the third. So it's a continuous linear ramp from 0 down to −1 across the buffer, then floored. The longer the response within the danger zone, the larger the penalty, but a response that's only slightly over pays only a little — no sharp cliff that would make the truncation boundary a source of high-variance reward. With the budget I'm using, that's L_max − L_cache = 16,384 expected tokens and a 4,096-token soft buffer, so the hard cap is 20,480. This signals "avoid excessive length" smoothly instead of punishing length as if it were wrongness.

One more thing I should clean up while I'm here: the β·D_KL(π_θ || π_ref) term. It was put into alignment RL to keep the policy from wandering too far from the initial model — there the whole point is to *stay close*. But here the entire goal is for the reasoning policy to move a long way from its initialization; the distribution is *supposed* to shift dramatically as it learns to think. So a term whose only job is to pull me back toward the starting point is fighting the objective. Drop it. And the reward itself stays a rule, not a learned model — +1 if the answer is equivalent to the ground truth, −1 otherwise — because a learned reward model invites reward hacking, and for verifiable math the rule is exactly the signal I want (which also means I should curate the data so answers are single integers a parser can check without ambiguity).

So the full recipe is: decouple the clip and raise ε_high to keep exploration alive; oversample-and-filter to keep every group gradient-bearing; aggregate the loss at the token level to kill the length bias; soft-punish overlong responses to denoise truncation; drop the reference-KL; rule reward. That's a coherent objective:

  J(θ) = E_{(q,a)~D, {o_i}~π_old}[ (1/Σ_i|o_i|) Σ_i Σ_t min( r_{i,t}(θ) Â_{i,t},
          clip(r_{i,t}(θ), 1−ε_low, 1+ε_high) Â_{i,t} ) ],   s.t. 0 < |{o_i : is_equivalent(a,o_i)}| < G,
  r_{i,t}(θ) = π_θ(o_{i,t}|q,o_{i,<t}) / π_old(o_{i,t}|q,o_{i,<t}),
  Â_{i,t} = (R_i − mean(R)) / std(R).

Now I want to step back to the length bias specifically, because I'd like a treatment I can apply at the *reward* stage — one knob, upstream of the advantage estimator — rather than only deep inside the loss aggregation. The reason is practical: the reward manager hands me one scalar per response, sitting at the last valid token, as a (batch, response_length) tensor, and then the group-relative estimator runs on it. If I can pre-shape that scalar so the length bias is counteracted before the group baseline is even formed, I get the length correction without rewriting the loss reduction. The question is: what single per-response multiplier f(T) of the scalar reward, as a function of the response length T, undoes the length effect?

Let me think about how a response's gradient contribution scales with T. The group-relative advantage is a single scalar per response, and it's broadcast uniformly across all T of that response's tokens. So if I just sum the per-token contributions of a response, the total scales like (advantage)·T — a longer response, carrying the same scalar advantage replicated over more tokens, contributes proportionally more in aggregate. That's the verbosity pressure showing up again. If I wanted to flatten that completely — make the per-response contribution independent of T — I'd divide by T, i.e. f(T) = 1/T. But hold on: dividing by T is *exactly* the 1/|o_i| sample-level normalization I just diagnosed as the disease. Full 1/T normalization doesn't merely damp length, it flips the sign-side behavior: correct responses get reinforced harder when they are shorter, and wrong responses get penalized less when they are longer. So 1/T overcorrects into the brevity/under-penalty pattern I am trying to escape. And the uncorrected end, f(T) = 1, leaves the full T-scale verbosity pressure. I want something *between* those endpoints — a partial correction that dampens verbosity without recreating the sample-level normalization.

Write the multiplier as f(T) = T^{-α}. If α = 0, I do nothing and a response's total contribution scales like T^1. If α = 1, I divide by length and the total contribution scales like T^0, the full sequence-level averaging that causes the sign problem above. The midpoint exponent is α = 1/2: divide the scalar by √T. Then the total contribution scales like r·T/√T = r·√T. That is not an unbiased-policy-gradient proof and it is not length-invariant; it is the halfway exponent between the two bad extremes. It leaves some legitimate extra influence for a longer response because it contains more tokens, but it cuts the raw linear length incentive down to a square-root one. That is the gentle correction I need: lean against verbosity without paying for it with the full brevity bias of 1/T.

I have to design this knowing what happens *downstream*, though, because this runs before the group-relative estimator, which will still subtract the group mean and divide by the group std on top of whatever I return. So I'm not the only normalizer in the pipeline — I'm pre-shaping the reward magnitudes that then go into (R − mean(R))/std(R). Dividing each response's scalar by √T changes the *relative* magnitudes of long versus short responses *within a group* before the baseline is formed: a long response's reward is shrunk more than a short one's, so the centering and std are computed from already length-shaped rewards. That's the interaction I want. If I'd divided by T, the long responses would be shrunk so hard that after group normalization the policy would chase brevity; √T is light enough to dampen without that.

So now the implementation, filling the one empty slot — the reward transformation the estimator consumes. The reward manager gives me token_level_scores, shape (bs, response_length), with each response's outcome scalar sitting at its last valid token, plus response_mask marking the valid tokens. I recover the per-response scalar by summing over the token axis — sum, not mean, because the scalar lives at exactly one position and summing the (mostly-zero) row recovers it, whereas a mean would divide by the sequence length and tangle itself up with the very length correction I'm trying to apply deliberately. I read each response's true length T from the mask, response_mask.sum(dim=-1) — the count of valid tokens, not the padded seq_len. I divide each scalar by √T, clamping T at a floor of 1 so an empty response doesn't divide by zero, and adding a tiny epsilon as a further numerical floor. Then I have to put the rescaled scalar *back* at the last valid token, because the downstream estimator expects the outcome reward to sit there — I find the last valid index as response_mask.sum(dim=-1) − 1 (clamped at 0), scatter the rescaled scalar there in a zeroed tensor, and multiply by the mask to zero anything outside the valid region. All of it under no_grad, because this is a reward transformation, not part of the differentiable graph.

```python
import torch
import numpy as np
from typing import Optional


def normalize_rewards(
    token_level_scores: torch.Tensor,   # (bs, response_length): outcome scalar at last valid token
    response_mask: torch.Tensor,        # (bs, response_length): 1 on valid response tokens
    index: np.ndarray = None,           # (bs,): group / prompt id (unused by this rule)
    epsilon: float = 1e-6,
    config: Optional[object] = None,
    **kwargs,
) -> torch.Tensor:                      # (bs, response_length): length-corrected reward tensor
    """Divide each response's scalar reward by sqrt(response_length) before re-placing it
    at the last valid token. Counter-balances the verbosity pressure of the group-mean
    broadcast without flipping into a brevity bias (which dividing by length, i.e. the
    sample-level 1/|o| factor, would cause)."""
    with torch.no_grad():
        bsz, seq_len = token_level_scores.shape
        scores = token_level_scores.sum(dim=-1)              # (bs,): recover per-response scalar

        lengths = response_mask.sum(dim=-1).to(scores.dtype) # (bs,): T = # valid tokens (not seq_len)
        denom = torch.sqrt(lengths.clamp(min=1.0)) + epsilon # sqrt(T), floored against sqrt(0)/0-div
        scores = scores / denom                              # pre-divide scalar by sqrt(T)

        out = torch.zeros_like(token_level_scores)
        last_idx = response_mask.long().sum(dim=-1) - 1      # (bs,): index of last valid token
        last_idx = last_idx.clamp(min=0)
        out[torch.arange(bsz, device=out.device), last_idx] = scores   # re-place rescaled scalar
        return out * response_mask                           # keep "outcome reward at last token" only
```

Let me close the loop on why this is the right shape and not three unrelated patches. The single thread is length: the recipe's loss arithmetic and its truncation handling and this reward transform are all fighting the same pathology — a length bias whose direction I derived from the 1/|o_i| factor (brief-when-right, long-when-wrong, average drifts up). The token-level loss kills it at the root by making every token's weight constant; the soft overlong punishment denoises the truncation boundary so length is discouraged smoothly rather than punished as wrongness; and this √T reward-stage transform is the upstream, one-knob proxy — it pre-divides each response's scalar by √T, the exponent halfway between "do nothing" (which leaves the linear verbosity drift) and "divide by length" (which overcorrects into the sample-level brevity/under-penalty bias), so it damps length pressure without pretending to be the exact policy-gradient objective. Around that length core sit the other two repairs to the same naive run: decouple-and-raise the upper clip so exploration tokens aren't strangled and entropy doesn't collapse, and oversample-and-filter so every group carries a live gradient — with the reference-KL dropped because the reasoning policy is meant to move far from its start, and the reward kept a rule because verifiable math doesn't need a hackable learned model.
