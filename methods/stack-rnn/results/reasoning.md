I start with the simplest recurrent predictor and try to make the failure precise. I read a token `x_t`, update `h_t = sigmoid(U x_t + R h_{t-1})`, and predict `y_t = softmax(V h_t)`. This is a perfectly good way to let the past influence the present, but every fact about the prefix has to live inside one fixed-width vector. On a short `a^n b^n` stream the trajectory can look as if it is counting, because the training lengths carve out a small region of state space where "after this many `a`s, expect this many `b`s" works. But the test deliberately leaves that region. If the rule asks for arbitrary depth, the memory resource itself has to grow with depth; otherwise I am asking a fixed vector to behave like an unbounded data structure.

The LSTM makes me more careful. It is not fair to say recurrent nets cannot remember. With gates and a linear cell, a unit can keep nearly constant error flow and can learn to increment or decrement a quantity. That really can behave like a counter on one-turn languages. But now the shape of the task matters. A counter is enough for "how many closers remain?" when the alphabet and order are simple. It is not enough for "( [ ( ) ] )", where the next legal closer is determined by the last unmatched opener, not just by the number of open brackets. Reversal makes the same point in a cleaner way: I need the last symbol I stored to be the first symbol I read back. The missing memory is last-in-first-out.

So I want the finite recurrent controller from an SRN, but I want to attach a stack to it. That immediately gives the right abstract machine for context-free-style structure: a finite state plus a stack is a pushdown automaton. Prior neural pushdown automata show that this bias is not arbitrary. They put an external stack beside a recurrent controller and learn small grammars. But I need the whole thing to train from next-token loss by backpropagation, and a hard stack action is the obstruction. If the controller chooses exactly PUSH or exactly POP with an argmax, a small change in a weight usually leaves the chosen action unchanged. The forward behavior is identical, the loss is identical, and the useful gradient through the action is zero. Reinforcement-style credit assignment is possible in principle, but a wrong early stack action corrupts a long rollout, which is exactly the high-variance setting I am trying to avoid.

The escape is to stop choosing one action during training. Let the controller produce a softmax distribution over actions and let the memory execute the weighted mixture of the pure action outcomes. I first write the two-action version because the indices are easiest to audit. The controller emits `a_t = softmax(A h_t)`, with entries for PUSH and POP. The stack is a vector `s_t` with top at index 0. If I push, the new top is a learned value and every old cell moves one slot deeper. If I pop, every old cell moves one slot toward the top. Therefore the top cell must be `s_t[0] = a_t[PUSH] sigmoid(D h_t) + a_t[POP] s_{t-1}[1]`. For a deeper cell `i > 0`, the push source is the previous cell above it and the pop source is the previous cell below it: `s_t[i] = a_t[PUSH] s_{t-1}[i-1] + a_t[POP] s_{t-1}[i+1]`. The signs and indices are forced by the geometry of the stack: push shifts down, so destination `i` reads old `i-1`; pop shifts up, so destination `i` reads old `i+1`.

The top value `sigmoid(D h_t)` is also doing work. It is not an arbitrary unbounded write. It is a scalar summary of the current hidden state, clipped through a sigmoid into `(0,1)`. That gives the stack a stable numeric range. Now I need to decide what a pop reads past the bottom. If I fill with zero, an empty slot can be confused with a real pushed value near zero. If I fill with a value outside the pushed range, the controller can detect emptiness. I choose `-1`. Then an all-empty stack is a stack whose slots are `-1`, and a pop that frees the bottom contributes `-1` into that freed bottom slot. This bottom case is easy to omit, but it is essential: after a pop, the formula for interior cells handles `i < p-1`, while the new last cell has no old `i+1`, so it receives the empty sentinel.

Now I feed the stack back into the controller. The hidden update becomes `h_t = sigmoid(U x_t + R h_{t-1} + P s_{t-1}^k)`, where `s_{t-1}^k` is the top `k` stack entries. The experiments use `k = 2`, which gives the controller a tiny window into the stack rather than only the single top. For the synthetic algorithmic tasks I can set `R = 0` and force long-range state to pass through the structured memory; for language modeling I can restore the ordinary hidden recurrence. The output remains `y_t = softmax(V h_t)`. I do not need to concatenate the stack top into the output head, because the stack has already influenced `h_t`.

I also need a no-op action. A single stack that solves `a^n b^n` can push during the `a` phase and pop during the `b` phase without ever waiting. But as soon as I run multiple stacks in parallel, one stack may need to hold steady while another drains. Adding NO-OP is the exact unchanged-cell case, not a new kind of write. The top becomes `s_t[0] = a_t[PUSH] sigmoid(D h_t) + a_t[POP] s_{t-1}[1] + a_t[NO-OP] s_{t-1}[0]`; each deeper cell becomes `s_t[i] = a_t[PUSH] s_{t-1}[i-1] + a_t[POP] s_{t-1}[i+1] + a_t[NO-OP] s_{t-1}[i]`. The three action weights sum to one, so this is a convex superposition of the pushed, popped, and unchanged stacks.

Several details now line up. Multiple stacks are not decorative; one stack performs one action per step, so harder patterns need parallel bookkeeping. The stacks interact only through the shared hidden layer, which is the finite controller. The model is continuous during training because the action probabilities, pushed value, hidden update, and output are all differentiable. Backpropagation can send error into the pushed value, into the action softmax, into the previous stack entries, and backward through time to the earlier decision that created a bad entry. That is the gradient path the hard-action stack lacked.

Training is still not magically easy. The local statistics of these streams create shallow basins: a bigram-like predictor can get much of the entropy right while ignoring the rare forced boundary where the real algorithm is tested. So I keep the ordinary SGD/BPTT setup, truncate BPTT to about 50 steps, clip gradients hard, start around learning rate `0.1`, halve it when validation entropy stops improving, and increase the length parameter during training. For harder patterns I accept random restarts as a search wrapper over SGD. The continuous relaxation makes the model trainable, but the learned behavior is still program-like enough that initialization can decide whether the controller finds the right discipline.

One last issue appears only when I care about very long strings. The soft action distribution never has to be exactly one-hot. A push with probability `0.999` still leaves a small pop/no-op trace, and hundreds of tiny traces smear the stack. At test time I can take the argmax action and execute the corresponding hard stack move. That is not changing the learned program; it is collapsing the relaxed controller to the discrete stack behavior it has learned. The training-time object is differentiable, and the test-time object is crisp enough to run the same finite controller at depths beyond training. That is the complete method: a recurrent controller whose action distribution defines a differentiable superposition of stack operations, with sentinel emptiness, top-of-stack feedback, optional no-op, and multiple parallel stacks when the algorithm needs more than one piece of last-in-first-out bookkeeping.
