Let me start from what actually hurts. Every time I train one of these transformer language models, the single knob that sets my cost is `L`, the length of the input subsequence I train on. Attention compares every token to every other token, so compute and memory both scale like `L^2`. And `L` pays off — a token predicted with 1000 tokens of left context has a lower loss than one predicted with 100 — so I'm always tempted to crank it up, and always punished for it on the clock. What I really want is to train on short sequences, which is cheap, and then at inference feed sequences *longer* than `L` and have the model keep working, ideally do better because it sees more context. The recurrent models people trained before this were routinely trained short and run long, and everyone assumed that generalization was free. The hope was the transformer would inherit it. So the question is sharp: I train at `L`, I evaluate at `L_valid > L`, does my perplexity hold? And if it doesn't, why not, and can I fix it by changing only how position enters the model — without spending one extra millisecond, one extra megabyte, or one extra parameter over the cheapest thing I already have?

First I need to know whether the problem is even in the position mechanism, or whether the transformer just structurally can't accept more tokens. Let me check the second possibility, because if it's true I'm wasting my time. A transformer layer: embedding lookup acts per token, the feedforward sublayer acts per token, the softmax classifier acts per token, and attention's parameters don't depend on length at all — it already has to eat variable-length inputs because of causal masking. So every function in the stack maps `n` input vectors to `n` output vectors for *any* `n`. Nothing in the weights cares how many tokens show up. The architecture can physically accept a longer input. So if it breaks at `L_valid > L`, it's not the layers; it has to be the one thing I inject that *does* depend on absolute position — the position signal I add at the bottom.

Let me actually look at how that signal behaves out of range. The standard move is to add a vector to each token embedding that encodes its position. If those vectors are learned, one per slot `0..L-1`, then position `L` and beyond simply has no vector — there's nothing to add, the behavior is *undefined* past `L`. Dead on arrival for extrapolation; I can rule learned-absolute out by inspection. The cleverer option is sinusoidal: each coordinate of the position vector is a sine or cosine of the position at a geometric range of frequencies, `sin(p / 10000^{2k/d})` and the cosine counterparts. This is a *function* of position, defined for every integer, so it produces *some* vector at position `L+500`. The original transformer paper speculated exactly this — that because the encoding extends to any position, the model might extrapolate. So let me trust that and just try it: train at `L = 512`, then score the validation set at `L + k` for `k` from a handful up to thousands, and watch the perplexity.

What I see is deflating. Perplexity improves for the first ~20 tokens past `L`, holds flat to maybe `k = 50`, and then climbs — degrades — and keeps climbing as `k` grows. Train at `L = 1024` and it's the same shape, improving to about `k = 50` and then falling apart. So "the encoding is defined everywhere" is not the same as "the model can use it everywhere." And I can say *why*: yes, `sin(p/...)` is defined at `p = L+500`, but during training the model only ever saw the *particular phase combinations* that occur over `[0, L)`. Past `L`, the joint pattern of all those sinusoids across the dimensions is a configuration it never encountered — it's out of distribution. The model learned responses tuned to in-range signals; feed it an out-of-range signal and it has no calibrated response. The cheap, parameter-free method that "should" extrapolate doesn't, and now I understand the mechanism: it's not that the function dies, it's that the model's *learned interpretation* of the function only covers `[0, L)`.

To be sure this is the position method and not some confound, I hold everything else fixed — architecture, data, seed, training budget, epochs — and change *only* the position scheme. If extrapolation behavior moves when nothing but the position method moves, the position method is the cause. So let me survey what else is on the table and run the same train-short-test-long probe on each, because if a different position method extrapolates under identical conditions, that simultaneously proves the diagnosis and shows me what property buys the win.

Rotary first. Instead of adding a vector at the input, it rotates the query and key vectors in *every* attention layer by an angle proportional to their absolute position. The neat thing falls out of trigonometry: a query rotated by angle `θ·i` dotted with a key rotated by `θ·j` depends only on `i - j`, the *relative* offset — the absolute rotations cancel in the inner product. So although it's built from absolute positions, the model effectively sees relative distances. Two structural differences from the additive scheme jump out. One, it injects position at *every* layer, not just the bottom. Two — and I want to hold onto this — it touches only the queries and keys, never the *values*. The output of an attention sublayer is a weighted sum of value vectors; if I never put position into the values, then the layer's output carries no explicit absolute-position component, only whatever the attention *weights* did with relative distance. I run the probe: rotary extrapolates noticeably better than sinusoidal — it keeps improving for the first ~200 extra tokens at `L=512`, ~100 at `L=1024` — but the per-layer rotations cost me; training and inference are slower and heavier than the plain sinusoidal baseline. Better extrapolation, but it broke bar (2): efficiency.

Now the relative-bias family — Shaw et al.'s scheme as used in T5. This one adds nothing to the embeddings at all. It computes the query–key dot products as usual, then adds a *learned scalar* to each score that depends only on the distance `i - j` between query and key, shared across the network and tuned per head. So the score-vs-distance relationship is a learned lookup. The distances are *bucketed*: each small distance gets its own learned bias, and distances past some cutoff are pooled into shared buckets. The intuition behind bucketing is appealing for my problem — a never-before-seen large distance just falls into an already-learned far bucket, so there's *something* defined out there. Same two structural properties as rotary: position enters at every layer, only through the query–key comparison, never into the values. I run the probe, and this is the best extrapolator yet — perplexity keeps improving for several hundred extra tokens, `k≈600` at `L=512`, `k≈800` at `L=1024`. This is the result that nails the diagnosis: under identical everything-else, swapping in this position method buys real extrapolation. So extrapolation *is* reachable by changing the position representation. But the cost is brutal: computing and gathering the bucketed per-distance bias makes it at least about twice as slow as sinusoidal on my GPU/PyTorch setup, and it adds learned parameters. The whole point was train-short-and-cheap; if I train this on `L=512` to extrapolate to 1024 it runs at half the words-per-second of just training sinusoidal at 1024, so I've saved nothing. It cleared bar (1) and blew bars (2) and (3).

So here's where I'm standing, and I want to read the ladder carefully because the gaps are pointing somewhere. Sinusoidal and learned-absolute are cheap and parameter-free but go out of distribution past `L` — they break. Rotary and the relative bias extrapolate, the relative bias best of all — but they're slow and the bias adds parameters. And the two that extrapolate share a property the two that don't lack: they put position *only* into the query–key scores, never into the values, so the residual stream carries no explicit absolute position. That property keeps coming up, so let me take it as a working hypothesis: position should live in the attention scores, not in the values, and not as a vector added at the bottom.

But the relative bias being *learned and bucketed* is precisely what makes it expensive — the gather over distance buckets and the parameters per bucket. And the learned lookup is *also* a subtle extrapolation hazard: the far buckets are trained on whatever far distances appeared in `[0, L)`, so even bucketing only partly defines behavior out of range. What if I keep the "add a bias to the score that depends on distance" idea but make the bias-vs-distance function so simple it needs no parameters and no gather, and is defined — smoothly — for *every* distance, including distances I never trained on? The disease with sinusoidal was that the signal past `L` is a novel, out-of-distribution pattern. The cure is a position signal whose value at distance `d` is the *obvious continuation* of its values at small `d`, with no learned dial that could be miscalibrated out of range. The simplest such function of distance is a straight line. So: after the query–key dot product, before the softmax, add a bias that is *linear* in the relative distance `i - j`.

Which direction should the line go? I want a recency bias — distant tokens should matter less than nearby ones, all else equal — and crucially I want that preference to be *the same shape* at every distance so that distance 5000 is just "distance 4 but more so," never a regime the model hasn't conceptually seen. So make the bias *more negative* the farther the key is from the query. For query `i` attending over keys `j = 1..i`, the relative distance is `i - j ≥ 0`, and I subtract a penalty proportional to it. Write the row of biases the query `i` adds to its scores over keys `1..i` as

  `m · [-(i-1), -(i-2), ..., -2, -1, 0]`,

so the key right next to the query (`j = i`, distance 0) gets penalty 0, the key one step back gets `-m`, two steps back `-2m`, and so on, linearly. The attention computation becomes `softmax(q_i K^T + m·[-(i-1),...,-1,0])`. The scalar `m > 0` is a per-head slope. That's it — no embedding added anywhere, no learned bias, no bucket gather. And it's defined for *any* distance: at distance 5000 the penalty is just `5000·m`, the same line rather than a new absolute-position vector. This directly attacks the failure I diagnosed: the thing the model interprets is a scalar that grows linearly and predictably, not a high-dimensional phase pattern that goes novel past `L`.

Let me sanity-check this against the property I extracted from the survey. Does it keep position out of the values? Yes — the bias is added to the *scores*, the values are untouched, so the attention output is a position-free weighted average; absolute position never enters the residual stream. Does it inject at every layer? Yes — every attention sublayer adds the same bias, so position reasserts itself at every layer, exactly like rotary and the relative bias and unlike the bottom-only additive scheme. Good, it has the two structural properties I associated with the extrapolators, but it's parameter-free and a single subtraction.

Now I have to face the slope `m`. One slope for all heads would force every head to the same recency preference, and that's clearly wrong — some heads should look locally, some should keep a long view. So `m` is per-head. With `H` heads I need `H` slopes, and I want them to span a range of recency scales: a head with large `m` is penalized hard for looking far, so it becomes a local head; a head with tiny `m` barely penalizes distance, so it keeps a long-range view. The question is how to lay out `H` slopes across `(0, 1)`. Let me reason about what "tiny `m`" buys: the long-range heads are the precious ones — they're the ones that can use far context at all — so I want *more resolution near zero*, i.e. the slopes should bunch up toward 0 and spread out toward 1. A geometric sequence does exactly that: equal ratios mean equal *log*-spacing, which packs points densely near the small end. So make the `H` slopes a geometric sequence.

Which geometric sequence? Let me pin it for the common case of 8 heads and then generalize. I want the slopes to live in `(0, 1)` and span a wide range of scales. Take the sequence `1/2, 1/4, 1/8, ..., 1/256` — that's `2^{-1}, 2^{-2}, ..., 2^{-8}` for 8 heads. Start `1/2`, ratio `1/2`. Cleanly inside `(0,1)`, geometrically spaced, densest near zero, eight distinct recency scales from "halve attention every step or two" down to "barely notice distance." That's the 8-head set. Now generalize to `n` heads while keeping the *same character* — start and ratio equal, geometric, spanning to roughly `2^{-8}` at the small end so the longest-range head is about as long-range as in the 8-head case. If I want `n` slopes `r, r^2, ..., r^n` with the smallest being about `2^{-8}`, then `r^n = 2^{-8}`, so `r = 2^{-8/n}`, and I set start = ratio = `2^{-8/n}`. Check `n=8`: `r = 2^{-1} = 1/2`, giving exactly `1/2,...,1/256`. Good, the general rule reduces to the case I designed it from. The implementation writes the same number as `2^{-2^{-(log2(n) - 3)}}`, which equals `2^{-8/n}` because `8/n = 2^{3-log2(n)}`; that form is convenient when the head count is a power of two.

What if `n` isn't a power of 2, like 16 — wait, 16 *is* a power of 2; the awkward cases are things like 12. For a power of 2 like 16, I apply the formula directly: `r = 2^{-8/16} = 2^{-1/2}`, giving `2^{-0.5}, 2^{-1}, 2^{-1.5}, ..., 2^{-8}` — start `1/sqrt(2)`, ratio `1/sqrt(2)`, sixteen slopes. This is a half-step refinement of the 8-head grid: it keeps the old integer powers and adds the half-integer powers, including the leading `2^{-0.5}`. For a genuinely non-power-of-2 `n`, the clean power-of-two formula doesn't directly apply, so the workaround is: take the slopes for the closest power of 2 below `n`, then borrow the extra slopes I need from the *next* power of 2's set by taking every other one, grabbing only as many as I'm short. It's a hack to preserve the good spacing when `n` isn't a power of two, and it matters little because in practice the models I train use power-of-two head counts.

Should the slopes be learned? My instinct says let gradient descent find them — but I should think about whether that helps *extrapolation* specifically, which is the whole point. Learned slopes would be tuned to the distances seen in `[0, L)`; there's no signal during training that tells the optimizer how the slope should behave for the far distances that only appear at inference, so a learned slope could overfit the in-range recency statistics and generalize worse out of range. A fixed geometric set, by contrast, is the same line everywhere and can't be miscalibrated to the training length. And learning the slopes means they're parameters that flow through the backward pass, which costs speed. So I'll fix them before training. The character I'm relying on — slopes in `(0,1)`, densest near zero — is what I'd want to land on; I don't need a learned dial to find it.

One subtlety I almost missed: the scaling. In standard attention the scores are divided by `sqrt(d_k)` to keep the dot-product variance from blowing up the softmax — that scaling exists to tame a *random* quantity, the dot product of two learned vectors. My bias is not a random dot product; it's a deterministic geometric penalty I'm choosing directly in score-space. If I also divided it by `sqrt(d_k)`, I'd just be rescaling slopes I already get to choose freely, so it would be redundant at best and would couple my slope choice to the head dimension for no reason. So the linear bias is added *after* (and outside of) the `sqrt(d_k)` scaling — it is not scaled by `sqrt(d_k)`. The slopes are defined directly against the post-scaling scores.

Now the implementation, and here's where it gets pleasingly cheap. I'm already adding a causal mask to the scores before the softmax — an upper-triangular matrix of `-inf` that zeroes out attention to future tokens. My linear bias is *also* an additive term on the scores before the softmax. So I don't add a new operation at all: I fold the linear bias into the same mask matrix. For each head `h` with slope `m_h`, the bias matrix has entry `-m_h·(i - j)` at `(i, j)` for `j ≤ i`, and `-inf` for `j > i`. Add it once, reuse it. There is genuinely no runtime penalty, because the addition was already happening.

Let me get the matrix exactly right, because there's a trick that makes it even cheaper and I want to be sure it's valid. Writing positions one-based, the honest bias for query `i` is the row `m_h·[-(i-1), ..., -1, 0]` — a staircase where each row's "0" sits on the diagonal. In zero-based implementation indices, the finite entry for query position `i` and key position `j ≤ i` is `-m_h(i - j) = m_hj - m_hi`. Building that staircase per row is a bit of work. But watch: I'm about to feed each row through a softmax, and softmax is invariant to adding a constant to all entries of a row — `softmax(x + c) = softmax(x)`. The honest finite row and the simpler row `m_h·[0, 1, 2, ..., L-1]` differ, within the unmasked part `j ≤ i`, by the row constant `-m_hi`. Since softmax kills per-row constants, the simple matrix `m_h·[0,1,...,L-1]` (every row identical, then masked by the causal `-inf`) produces the *exact same attention weights* as the honest staircase. This works precisely because the bias is *linear* — a nonlinear penalty wouldn't decompose into a per-row constant — which is yet another reason linearity was the right choice. So I build `slope[h] * arange(L)` broadcast across rows, one cheap outer construction per head, and let the causal `-inf` mask handle `j > i`.

What's the memory cost of this? The plain causal mask is one `L×L` matrix shared across heads. Mine needs a different slope per head, so the mask becomes `H × L × L` — `H` times bigger. In absolute terms that's small (tens of megabytes at the sizes I care about), and the headline point survives: because I can now *train at a smaller `L`*, the quadratic `L^2` savings on the activations dwarf this `H×` overhead on the mask. Net, against a sinusoidal model trained at the *same* length, I'm within a fraction of a percent of memory and identical in runtime and parameters; against a sinusoidal model I'd otherwise have had to train *long*, I save real gigabytes and wall-clock.

Now I should be honest with myself about *what* this is buying and not over-claim it, because I want to understand the mechanism, not just celebrate the perplexity curve. When I score with nonoverlapping segments and grow `L_valid`, perplexity drops — good. But there are two completely different reasons that could happen. One: the model is genuinely exploiting context longer than `L_train`, spotting patterns across more than `L` tokens. Two: longer segments simply have a smaller *fraction* of context-starved predictions. That second effect is the early token curse — predictions near the start of each nonoverlapping segment have almost no left context (the relevant prior tokens were in the discarded previous segment), so they carry high loss and inflate perplexity. If `L_valid = 1000`, maybe 10% of predictions have ≤100 tokens of context; double to `L_valid = 2000` and only ~5% do. So merely *accepting* a longer input, even learning nothing new about long range, lowers perplexity by diluting the cursed early predictions.

I can disentangle these with a thought experiment turned into a measurement protocol I already have: sliding-window inference with stride 1. Move the window one token per forward pass, so *every* prediction gets the maximal `L`-token context — that removes the early token curse entirely, because no prediction is context-starved anymore. If the linear-bias model's nonoverlapping gains came from genuinely using `>L` context (reason one), then under stride-1 windows its perplexity should keep *improving* as `L_valid` grows. If the gains came purely from curse dilution (reason two), then under stride-1 windows — where the curse is already gone — growing `L_valid` should leave perplexity *flat*. Run it: the sinusoidal model's perplexity still explodes as `L_valid` grows even with sliding windows (it's broken by out-of-distribution position signals regardless). The linear-bias model holds *flat* — it doesn't get worse, but it doesn't improve either. So the honest reading is reason two: the extrapolation gains come from the method's ability to keep functioning on longer inputs, which then reduces the early token curse, not (at least not yet) from attending past `L_train`. The relative-bias model behaves the same way under this test. That's a real limitation to name and a direction for later, but it doesn't lessen the value: at `L_valid = L_train` the linear bias matches or beats the alternatives while being simpler and parameter-free, and at `L_valid > L_train` it gives better perplexity than the only other cheap option (plain nonoverlapping) without the crushing cost of the slow sliding window.

Let me also make sure I haven't quietly broken anything by removing position embeddings entirely. There is *no* additive position vector anywhere now — the token embeddings go in raw, and the *only* thing that tells the model about order is the per-head linear penalty on the scores. Is that enough order information? It is, because the penalty depends on the signed distance `i - j` and, combined with the causal mask (which already enforces "only look left"), every key's contribution is monotonically discounted by how far back it sits, differently per head. The model gets a rich, multi-scale notion of "how far away" from the spread of slopes, applied at every layer. And it's a recency *inductive bias* in the literal sense: distant query–key pairs are penalized, the penalty growing with distance, at a head-specific rate. That's a deliberately strong prior toward locality, which is exactly the prior that text rewards.

So let me write the whole thing into the harness, filling the one empty slot — the position mechanism and where it acts. The slopes computed once from the head count; the per-head bias folded into the additive pre-softmax mask using the linear/translation-invariance trick; the values and embeddings left completely position-free:

```python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


def _slopes(n_heads):
    # Per-head geometric slopes in (0,1), densest near 0 (long-range heads).
    # Power-of-2 case: start = ratio = 2^(-8/n), i.e. r, r^2, ..., r^n with r^n ~ 2^-8.
    def pow2(n):
        start = 2.0 ** (-(2.0 ** -(math.log2(n) - 3.0)))   # == 2^(-8/n)
        return [start ** (i + 1) for i in range(n)]
    if math.log2(n_heads).is_integer():
        return torch.tensor(pow2(n_heads))
    # non-power-of-2: closest pow2 set, then interleave extras from the next set
    closest = 2 ** math.floor(math.log2(n_heads))
    extra = pow2(2 * closest)[0::2][: n_heads - closest]
    return torch.tensor(pow2(closest) + extra)


class ALiBiAttention(nn.Module):
    """Multi-head causal attention with a linear per-head distance bias added
    to the scores before softmax. No position embeddings anywhere."""

    def __init__(self, d_model, n_heads):
        super().__init__()
        self.n_heads, self.d_k = n_heads, d_model // n_heads
        self.qkv = nn.Linear(d_model, 3 * d_model)
        self.out = nn.Linear(d_model, d_model)
        self.register_buffer("slopes", _slopes(n_heads), persistent=False)  # [H]

    def _bias(self, T, device, dtype):
        # softmax is shift-invariant per row and the bias is linear, so the
        # honest staircase m*[-(i-1),...,0] can be replaced by m*[0,1,...,T-1]
        # broadcast over rows; the causal -inf mask handles j > i.
        pos = torch.arange(T, device=device, dtype=dtype)           # [T]
        bias = self.slopes.to(device, dtype)[:, None, None] * pos[None, None, :]  # [H,1,T]
        causal = torch.triu(torch.full((T, T), float("-inf"), device=device, dtype=dtype), 1)
        return (bias + causal)[None, ...]                           # [1,H,T,T] broadcast over batch

    def forward(self, x):                       # x: [B, T, d_model] (no position added)
        B, T, _ = x.shape
        q, k, v = self.qkv(x).chunk(3, dim=-1)
        q = q.view(B, T, self.n_heads, self.d_k).transpose(1, 2)    # [B,H,T,d_k]
        k = k.view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        v = v.view(B, T, self.n_heads, self.d_k).transpose(1, 2)
        scores = (q @ k.transpose(-2, -1)) / (self.d_k ** 0.5)      # sqrt(d_k) scaling
        scores = scores + self._bias(T, x.device, scores.dtype)     # bias NOT scaled by sqrt(d_k)
        attn = F.softmax(scores, dim=-1)
        return self.out((attn @ v).transpose(1, 2).reshape(B, T, -1))
```

Walking the chain once more so I'm sure it holds. I wanted to train short and run long, but the transformer trained at `L` broke past `L`, and since the layers are length-agnostic the fault had to be the position signal — confirmed by holding everything fixed and swapping only the position method. Sinusoidal "should" extrapolate because it's a function of every position, but the model only learned the in-range phase patterns, so out-of-range signals are out of distribution and it degrades; learned-absolute has no behavior past `L` at all. Rotary and the bucketed relative bias *do* extrapolate, the relative bias best, proving the position method is the lever — but both are slow, the relative bias adds parameters, and both share the property that position lives only in the query–key scores, never in the values. I kept that property and discarded their cost by replacing the learned, bucketed bias with the simplest distance rule that continues past the training range: a straight line, `-m·(i-j)`, a fixed per-head slope, more negative the farther back the key, added to the scores before softmax. Per-head geometric slopes `2^{-8/n}, ..., 2^{-8}` give a spread of recency scales packed densely near zero where the long-range heads live; fixed not learned, because learned slopes can only overfit the training-length statistics; not scaled by `sqrt(d_k)`, because the bias is a chosen deterministic penalty, not a variance-prone dot product. Folding the linear bias into the existing causal mask — and using softmax's per-row shift-invariance plus the bias's linearity to replace the staircase with a single broadcast row — makes it cost no extra operations and almost no extra memory. And the stride-1 sliding-window probe tells me, honestly, that the longer-input gains come from defeating the early token curse rather than from attending past `L_train` yet — a flat-perplexity robustness to length, achieved with no embeddings, no parameters, no added runtime.
