Let me start from what actually hurts. I am training a transformer language model with AdamW, one enormous stochastic run, and the only knob I get to shape over the whole run is the learning rate as a function of step. Everyone uses cosine: warmup linearly to a peak `eta`, then a half-cosine down to about a tenth of the peak, with the cosine cycle stretched to cover the whole run. It works. But the thing that keeps biting me is that the cosine has a length parameter `T` — the step where it first hits its minimum — and I have to pick it before I start. And the empirical record is brutally specific about what `T` has to be. Kaplan's group looked at the summed learning rate over a run and found the loss keeps dropping as that sum grows, which says: don't decay early, because decaying early means the rate spends less total mass up high; so `T < S` (cycle shorter than the run) is leaving performance on the table. And Chinchilla looked at the other side and found `T > S` — the cosine not finished decaying by the end — actually *drops* performance, while `T = S` is both the best loss and the most efficient. Put those two together and the optimum is pinned exactly at `T = S`. The cosine has to be precisely as long as the run.

That is a much worse constraint than it first looks. It means the schedule is welded to a token budget I commit to up front. Say I train and then want to go further. I can't just keep going — the rate is already sitting at its floor, `0.1 eta`, and it makes almost no progress down there. If I try to re-warm the rate to keep exploring, the loss spikes and the run claws its way back only slowly; people in the continual-learning literature report re-warming hurting and even inducing forgetting. And worse than the continuation problem is the science problem: every scaling-law study, every data-mixture comparison, every architecture ablation is really the question "how would this model have done at a *different* number of tokens?" — and because the cosine has to match the length, I cannot read that off one run. I have to retrain from scratch at each target length. For `m` model sizes crossed with `m` token counts that's `O(m^2)` full pretraining runs. That's the cost that makes scaling research a luxury.

So what do I actually want? A schedule that hits the same final loss as a well-tuned cosine, but that does *not* need the total number of steps fixed in advance — so I can stop or continue on demand — and whose intermediate checkpoints I can reuse, so probing the model at many token counts is roughly linear effort instead of quadratic. And it has to be purely a schedule-layer change: same model, same data, same AdamW, same total update budget. Nothing else moves.

Let me stare at *why* the cosine works at all, because if I understand that I can try to keep the part that matters and throw away the part that hurts. The folklore, going back to warm-restart SGD and the large-batch work, is a trade-off: a *high* rate early helps the optimizer explore the loss landscape, and a *slow anneal* to a low rate at the end is what lets a deep net settle into a good minimum. The cosine bundles those two jobs — explore high, then settle low — into one smooth curve of a fixed length. But notice: there is no law that says they have to be the *same* curve. The reason `T` is welded to `S` is precisely that they're fused. If I could *separate* the high-rate exploration phase from the low-rate settling phase, then the exploration phase could run for an undetermined length — as long as I like — and the settling could be a short tail I tack on whenever I decide to stop. The length commitment would just dissolve.

So let me split it deliberately into stages. Warmup linearly to the peak — that part's settled, everyone does it and as long as it's long enough it barely affects the final loss. Then, instead of immediately starting to decay, hold the rate *constant* at the peak `eta` for the bulk of training. Call that the stable stage. Then, at the end, a decay stage that brings the rate down. Three pieces: warmup, stable, decay. Write it as

```
WSD(s) =  (s/W) * eta            if s < W        # warmup
          eta                    if W < s < T    # stable, constant peak
          f(s - T) * eta         if T < s < S    # decay
```

where `f` is some decreasing function from 1 down toward 0, `T` is where the decay starts, `S` the end. The lovely thing about this shape is that the stable stage is a single reusable trunk: I run it as long as I want, and whenever I want a finished model at the current step, I branch off a decay. I never had to know `S` when I started. And for scaling studies I run *one* long stable trunk per model size and branch a cheap decay from checkpoints at `10N, 20N, ... 60N` tokens — that's the difference between `O(m^2)` and `O(m)`.

But this only buys me anything if the decay actually recovers the cosine's final loss. Does holding constant and then decaying late really match annealing the whole way down a cosine? I genuinely don't know yet, and it's the load-bearing assumption — if a constant trunk plus a short tail lands at a worse loss than the cosine, the whole budget-free story is moot. So let me think hard about what a constant-then-decay run does. First, is the shape even new? This constant-plus-short-cooldown shape isn't. Zhai and co. used an "infinite learning rate" schedule for scaling vision transformers: warmup, then a constant (or reciprocal-sqrt) main phase, then a short sharp linear decay, exactly so the final decay could be branched at different lengths. And DeepSeek's multi-step LLM schedule is basically a staged constant-then-decay too. So the *shape* has precedent; what's missing is anyone treating it as the tool for budget-free LLM training and cheap scaling, and anyone asking what the decay tail should actually look like. That's the gap I'm in.

Now, what happens at the moment I start decaying from a constant plateau? Here's the observation that I have to take as my anchor, because it is reported and it is startling: when I anneal from the constant high rate, the loss — both training and validation — drops sharply and fast, and it reaches or beats the cosine-at-`T=S` value. Not a gentle convergence; a sudden cliff that tracks the rate coming down. In the original WSD experiments, a decay of about 10% of the total tokens is enough while 2.5% falls short; in the later cooldown study, longer sweeps show benefits plateauing around 20%, and long runs can need less than 20% if the absolute number of cooldown steps is large enough. That's almost too good: the constant trunk looked mediocre because its loss sat high, and then a short decay collapses the gap. I need to understand the mechanism, because if I don't, I won't know how to shape `f` or how long to make the decay.

Let me look at what's actually going on in the weights and gradients during that decay, because the numbers are peculiar. The per-step weight update size tracks the learning rate — fine, that's mechanical, AdamW's step scales with the rate. But here's the strange part: during the *stable* stage the updates are large and yet the loss barely moves; during the *decay* stage the updates are small and yet the loss plummets. Big steps, no progress; small steps, huge progress. That's backwards from how I'd naively think about optimization. And when I look at the gradients in the decay phase: the gradient norm shrinks, consecutive-step gradients start *agreeing* — their cosine turns positive, meaning the optimizer is moving in a consistent direction step after step instead of thrashing — the first-order directional derivative along the trajectory falls off roughly in lockstep with the rate, and the curvature of the loss along the path climbs by about an order of magnitude. Rising curvature, consistent direction, shrinking gradient: that's the signature of dropping into the bottom of a sharp local optimum. And one more clincher: if I take a checkpoint from *before* the decay and a checkpoint from *after* the decay and just walk in a straight line between them in weight space — linear interpolation of the two weight vectors — the loss falls smoothly the whole way. The two checkpoints aren't separated by a barrier; they're in one connected basin.

Put those facts together and a picture forms that explains the strange "big steps, no progress / small steps, huge progress" mismatch. Picture the loss surface as a long valley with a river running along its floor: there's essentially one nearly-flat direction — down the river — along which steady progress is possible, and it's flanked by steep walls, the sharp directions across the valley. A high constant learning rate does two things at once. Along the river (flat direction) it carries the iterate fast and far — real progress, lots of it. But across the valley (sharp directions) a high rate makes the iterate *overshoot and oscillate* between the walls; it can't sit at the river's bottom because each big step throws it back up the opposite bank. So during the stable stage the weights are genuinely moving a lot (big updates) and genuinely making progress down the river — but the measured loss stays *elevated*, because the iterate is perpetually parked partway up the steep walls, not at the channel floor. The loss you read off is dominated by that cross-valley displacement. Now lower the rate. The steps across the sharp directions shrink below the overshoot threshold, the oscillation dies, and the iterate simply drops the short distance from the wall to the river bottom. Small steps — but each one removes cross-valley loss that the high rate was holding it away from, so the loss falls fast. That's the cliff. And it's why straight-line interpolation works: pre-decay and post-decay differ mostly by that one short cross-valley descent into the same channel, so the line between them stays in the basin.

This reframes the whole schedule. The stable stage's job is to travel as far down the river as possible; that's why I want it high and long. The decay stage's job is purely to kill the cross-valley oscillation and let the iterate fall into the channel; that's a *short* operation — you only have to descend the width of the valley, not the length of the river — which is exactly why ~10% of tokens is enough and why the loss drop is so abrupt. The exploration and the settling really are separable, and the decay is the cheap part. Good. So my three-stage split isn't just an engineering convenience for budget-free training; it's aligned with the actual geometry.

Now the real open question, the one the earlier constant-plus-cooldown schedules do not pin down: what should `f`, the shape of the decay, be? The Zhai-style ancestor just used linear, `f(p) = 1 - p` over decay-progress `p` going 0 to 1. Linear is the obvious default. But is it the right shape? Let me reason from what the decay is for, mechanically. The decay's job is to get the cross-valley step size down below the overshoot threshold so the iterate can settle, while still leaving enough nontrivial learning rate to keep moving. With a fixed number of decay steps, where do I want to spend them? Early in the decay the rate is still high-ish; those steps are still more transition than settling. The productive settling happens once the rate is low enough that the cross-valley oscillation has actually died. So I want to get through the high part of the decay quickly and then allocate more of the cooldown to lower rates. Linear spends equal step-density at every rate level, which by the oscillation story is wasteful: it lingers at rates that are still throwing the iterate up the walls.

So I want a tail that drops fast at the start and then becomes gentle. The simplest such function is `1 - sqrt(p)`: it equals 1 at `p = 0` and 0 at `p = 1` like linear, but it bends the other way. Let me actually quantify how it differs from linear instead of asserting it bends the right way. Its derivative is `-1/(2 sqrt(p))`. I evaluate that: at `p = 0.25` the slope is `-1`, at `p = 0.5` it is `-0.707`, at `p = 1` it is `-0.5`, and as `p -> 0` it blows up (`-5` already at `p = 0.01`) — so the drop is steepest right at the start and relaxes to `-1/2` by the end, where linear is a flat `-1` throughout. Now compare the actual multipliers at matched cooldown progress. Since `sqrt(p) > p` on `(0,1)`, `1 - sqrt(p) < 1 - p` everywhere inside the interval, so the sqrt rate is always below linear there. Putting numbers to it: at `p = 0.25`, linear sits at `0.75 eta` and sqrt at `0.50 eta`; at `p = 0.5`, linear is `0.50 eta` and sqrt is `0.293 eta`; at `p = 0.75`, linear is `0.25 eta` and sqrt is `0.134 eta`. So at every interior point the sqrt schedule is sitting at a meaningfully lower rate.

The cleaner way to see the "front-loading" is to ask how much of the cooldown each schedule spends *below* a given rate, since that low-rate tail is where the settling happens. Solve `1 - sqrt(p) = c` for the crossing point: `p = (1-c)^2`, versus linear `p = 1-c`. For `c = 0.5 eta`, sqrt crosses at `p = 0.25` and linear at `p = 0.5` — so sqrt spends the last `0.75` of the cooldown below half the peak while linear spends only `0.5`. For `c = 0.25 eta`: sqrt `0.438` of the cooldown below it, linear `0.25`. For `c = 0.1 eta`: sqrt `0.19`, linear `0.10`. So sqrt allocates roughly 1.5-2x more of its cooldown to the low-rate settling regime at every threshold I check. That is concretely the "quench the high part fast, then settle long and low" profile the river-valley picture asks for, and now I've verified it numerically rather than taken it on faith. Whether that geometric advantage actually translates into lower final loss is an empirical question I can't settle from the schedule shape alone; the reported cooldown ablations are what I'd lean on, and they do put `1 - sqrt(p)` ahead of linear in the tested 10-20% settings with the gap widening for longer runs — consistent with this reasoning, though I'd want to confirm it on my own runs before trusting it as universal.

Let me write the decay with a floor so it interpolates cleanly between the peak and the minimum rate `min_lr`, rather than crashing to exactly zero. With decay-progress `p = (s - T)/(S - T)` running 0 to 1,

```
lr(s) = min_lr + (eta - min_lr) * (1 - sqrt(p)).
```

At `p = 0`, `lr = eta`, continuous with the stable stage. At `p = 1`, `lr = min_lr`. With `min_lr = 0` this is the pure multiplicative `(1 - sqrt(p))` cooldown; with a nonzero `min_lr` it is the same multiplier interpolated down to a floor instead of to zero.

Before I commit to the square root I should check whether "fast early drop" can be overdone, because nothing in the argument so far says the exponent `1/2` is special — it just says "below linear." Generalize the tail to `1 - p^a`, with `a = 1` the linear case and smaller `a` more front-loaded. Let me see how far the front-loading goes as I shrink `a`. The fraction of the cooldown spent below a rate `c` is `1 - (1-c)^{1/a}` (inverting `1 - p^a = c`). Take `c = 0.1 eta`: at `a = 1` that fraction is `0.10`, at `a = 0.5` it is `0.19`, and at `a = 0.2` it is `1 - 0.9^5 = 0.41`, at `a = 0.1` it is `1 - 0.9^{10} = 0.65`. So an exponent of `0.1` parks two-thirds of the entire cooldown below a tenth of the peak rate. Under the oscillation story that sounds aggressive, but it is plausibly *too* aggressive: it strands the run at a rate so low that it can barely make progress for most of the cooldown — the iterate has settled into the channel and is then just sitting there at a near-dead rate. That's a real tension I can't resolve from the geometry alone, since the geometry says "lower is better for quenching" but says nothing about the cost of going too low too soon. So the choice of exponent has to come down to the empirical sweep, and I should not pretend the square root is forced by algebra. What's reported lines up with the caution above: `a = 0.1` and `a = 0.2` are noticeably worse, the remaining exponents below `0.5` differ only marginally, and `a = 0.5` comes out on top — a sweet spot, front-loaded but not extreme. Gentler-than-linear tails (a within-decay cosine, or `1 - p^2`) sit on the other side, keeping the rate high too long, and they fall behind for the practical 10-20% cooldown lengths. I'll take `a = 1/2` as the best tested compromise in this family rather than a derived optimum.

How long should the decay be? The geometry already says it should be short — I only need to descend the valley's width, not travel the river's length. Let me sweep it in my head and against what's reported. Far too few decay steps — say 2.5% of the run — and the rate is still falling when the run ends: the iterate hasn't finished dropping into the channel, so I read off a half-quenched loss. That's the regime that falls short. Around 10% of total tokens, the cooldown is reliably enough to reach the bottom. Push the fraction higher and the benefit plateaus — once the oscillation is fully quenched and the iterate is at the river bottom, extra low-rate steps buy little — so somewhere around 10-20% is the practical band. And what really matters is the *absolute* number of cooldown steps, not the fraction: on a very long run even a small fraction can be plenty, because that small fraction is still many thousands of steps of settling. So I want the decay as short as I can get away with — to keep checkpoint-probing cheap — but not so short it leaves the cooldown unfinished. For a fixed-length run I'll take the last ~20% as the decay window; for very long runs a smaller fraction is fine as long as the absolute step count is large.

One more thing to settle: the floor `min_lr`. The canonical cooldown often goes to zero, and the reference schedule exposes this as `final_lr_factor=0.0`. Lower final rates help pretraining loss; in fact, making cosine decay below the common 10% floor can match the best `(1-sqrt)` cooldown loss. But downstream metrics can be hurt by exact-zero annealing, plausibly because the model saturates too early. So the maximum rate and the final rate should be set independently. I will keep `min_lr` as a parameter: `min_lr=0` reproduces the canonical cooldown-to-zero, and a small positive floor is available when downstream evaluation prefers it.

Let me also be honest about the warmup. I'm keeping it linear and I'm not over-thinking its length: as long as it's long enough, the final performance is insensitive to it. A slightly longer warmup is a mild, safe choice — it just means the rate ramps to peak a bit more gently before the long stable stage. The interface gives me `warmup_iters`, and I can be conservative and use a generous warmup fraction; it costs me nothing in final loss.

So now I assemble the whole schedule against the actual interface I have to fill: `get_lr(it, warmup_iters, lr_decay_iters, learning_rate, min_lr)`, called every iteration to set AdamW's rate, where `learning_rate` is the peak `eta`, `min_lr` the floor, and `lr_decay_iters` is the horizon by which the schedule has fully decayed. Three branches. If `it` is in warmup, ramp linearly from near zero to the peak: `learning_rate * (it+1)/(warmup_iters+1)`. Otherwise, figure out where the decay window starts — the last fraction of the horizon, say the final 20%: `decay_start = int(lr_decay_iters * 0.8)`. If `it` is past `decay_start`, I'm in the cooldown: compute `p = (it - decay_start)/(lr_decay_iters - decay_start)`, clamp it into `[0,1]`, and return `min_lr + (learning_rate - min_lr) * (1 - sqrt(p))`. The clamp matters because for `it >= lr_decay_iters`, the canonical schedule stays at the final learning-rate factor. Otherwise — between warmup and `decay_start` — I'm in the stable stage and just return the constant peak `learning_rate`. That's the entire method: three short branches, one square root.

Let me write it as the code I'd actually drop in:

```python
import math


def get_lr(it, warmup_iters, lr_decay_iters, learning_rate, min_lr):
    """WSD schedule with (1-sqrt) cooldown.

    warmup -> constant stable plateau at the peak -> short final cooldown that
    front-loads the rate drop (1 - sqrt(progress)) so the cross-valley
    oscillation is quenched fast, then settles into the basin at low rate.
    """
    # Warmup: linear ramp from ~0 up to the peak rate.
    if it < warmup_iters:
        return learning_rate * (it + 1) / (warmup_iters + 1)

    # Decay window: the final fraction of the horizon (~20% of steps).
    decay_start = int(lr_decay_iters * 0.8)

    # Cooldown: (1 - sqrt(progress)) front-loads the drop.
    if it >= decay_start:
        decay_len = max(1, lr_decay_iters - decay_start)
        p = (it - decay_start) / decay_len
        p = min(max(p, 0.0), 1.0)               # clamp to [0, 1]
        coeff = 1.0 - math.sqrt(p)              # 1 at p=0, 0 at p=1
        return min_lr + (learning_rate - min_lr) * coeff

    # Stable stage: hold the peak rate. This is the reusable trunk — it can run
    # for an undetermined length, with a cooldown branched off on demand.
    return learning_rate
```
