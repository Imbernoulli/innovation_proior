I start from the part that feels wasteful rather than from the part that already works. Residual shortcuts let me train networks that would otherwise be impossible, but the empirical exchange rate is poor: once I am near the best thin residual models, a small accuracy gain asks for a very large increase in layers. That is a warning that depth is not being converted into useful representation efficiently.

The identity shortcut explains why. A residual block adds `F(x_l, W_l)` to `x_l`, so the signal has an exact path around the residual weights. That is the optimization gift, but it is also an escape hatch. In backpropagation, the gradient can travel through the shortcut even if the residual branch is weak. So I should not assume that every extra block is learning a new useful transformation. Stochastic depth makes this less speculative: if whole residual blocks can be randomly skipped during training and the network still improves, then many blocks in a very deep residual chain are not individually indispensable.

My first instinct is to change the block, but I need to separate several possibilities. A block can become stronger by adding more convolutions, by changing the mix of `3x3` and `1x1` convolutions, or by increasing the channel count. Larger spatial filters are not the right first experiment because the surrounding evidence from VGG and Inception already favors small filters, and larger filters would blur the question with an older design choice. I keep the filters at `3x3` or `1x1` and ask what the residual block is really buying.

The bottleneck block is also the wrong object for the small-image question I am trying to answer. It exists to make a block cheaper so I can stack more layers. If the thing under suspicion is the habit of spending nearly everything on depth, then a thinning device would bake in the old answer. I keep the same number of feature planes across the residual branch and use pre-activation, because the identity-mapping work has already shown that `BN -> ReLU -> conv` is the better residual-unit order.

Now I test block structure at comparable parameter count. I compare the basic `B(3,3)` block against variants with `1x1` layers such as `B(3,1)`, `B(1,3)`, `B(1,3,1)`, `B(3,1,3)`, and `B(3,1,1)`. This is not the place for a dramatic conclusion: the results are close. `B(3,3)` is the best by a small margin, while `B(3,1)` and `B(3,1,3)` are very near it and cheaper. That tells me the exact internal mix is not the main source of leverage. The simple two-`3x3` block is still the cleanest choice, but not because every alternative collapses.

Then I test whether each block should contain more convolutions. The tempting thought is that if a block is the useful unit, making each block deeper should make each unit stronger. I want to test that fairly, so I hold the total number of convolution layers and the parameter count fixed and only vary the deepening factor `l` — the number of convolutions inside a block. If the thought were right, larger `l` should win. The experiment goes the other way. `l=2` wins; `l=1` is too weak, and `l=3` or `l=4` gets worse. I have to take the prediction as refuted and ask why. With the total convolution budget fixed, deeper blocks mean fewer residual blocks and therefore fewer shortcuts. The shortcut is not just a passive wire; it is part of the optimization machinery. Removing too many of those addition points makes the network harder to optimize, and that cost outweighs whatever a deeper internal transform buys. So the deepening idea is a dead end at fixed budget, and the basic two-convolution unit stands.

The block settles down to the basic pre-activation two-convolution unit. The remaining axis is channel count. This is the axis the thin residual designs deliberately minimized in order to buy depth, and now I have reasons to distrust that trade. Width costs quadratically in parameters and FLOPs, while more blocks cost linearly, so on paper width looks expensive. But wall-clock training is not paper arithmetic. A thousand small sequential convolutions underuse a GPU; fewer larger convolutions give the hardware more parallel work. I should measure accuracy per training time, not only parameter count.

Before I commit to width, I want a concrete sense of how the channel multiplier `k` actually moves the parameter budget, because "quadratic" should be visible in the numbers and not just asserted. Take a fixed depth and let the three groups carry stage widths `16, 16k, 32k, 64k` (the first group already widened, the stem left at 16). At the smallest setting `k=1` the network is essentially the thin residual baseline; a 16-layer version comes out around `0.18M` parameters. Pushing the same 16-layer skeleton to `k=8` gives widths `16, 128, 256, 512` and lands near `11M` parameters — roughly a sixty-fold increase. Eight times the width for sixty-some times the parameters is the quadratic showing up exactly where it should (channel count enters both the input and output dimension of each conv). So width is genuinely expensive per unit; the bet is only worth making if accuracy-per-time pays it back.

When I sweep the feature-plane multiplier and depth, the pattern starts to point one way. At fixed modest depths, increasing the channel multiplier keeps improving the CIFAR results over the tested range. At fixed large multipliers, increasing depth helps at first but then stops helping or hurts. The claim I most want to nail down is the head-to-head against the extreme thin baseline, because that is the comparison the depth-first habit rests on. The pre-activation residual reference at 1001 layers on CIFAR sits around `10.2M` parameters. So I check what a deliberately wide-but-shallow model of comparable size looks like: a 40-layer network with `k=4` has widths `16, 64, 128, 256` and `(40-4)/6 = 6` blocks per group, and when I count its parameters it comes out at about `8.95M` — the same order of magnitude as the 1001-layer model, slightly under it. That is the comparison I wanted to be honest about: it is genuinely matched on parameter scale, not just hand-waved as "comparable." A model with one twenty-fifth the depth and the same parameter budget that trains far faster and is competitive on accuracy is direct evidence that the old depth-to-width ratio was one expensive point in the design space, not a law. Pushing further, a 28-layer model at `k=10` is much larger — widths `16, 160, 320, 640`, about `36.5M` parameters — but it still trains successfully and beats the thin 1001-layer reference, which tells me the width axis has real headroom left where the depth axis had run out.

This also changes how I think about regularization. Increasing channels creates many more parameters — going from the `0.18M` thin model to tens of millions is most of a two-order-of-magnitude jump — and batch normalization alone is not a complete answer, especially when the data do not get much augmentation. SVHN is the clean test: no augmentation, just scaling to `[0,1]`, and the training curves can show the model driving training loss very low while test error stalls. That is the regime where I expect to need extra stochastic regularization, and it is where I would watch the train/test gap most closely to confirm the model is overfitting rather than underfitting.

Where the dropout goes is not a free choice. Putting it on the shortcut would randomly damage the one path that lets information and gradients move cleanly through the net, which fights the reason residual nets train in the first place; I would expect that to hurt optimization, so I rule it out rather than test it to destruction. The dropout has to live inside the residual branch, between the two convolutions, after a ReLU. The clean placement is first BN/ReLU and convolution, then second BN/ReLU, then dropout, then the second convolution. That placement perturbs the residual transformation without corrupting the identity path, and it gives the following block's batch normalization a less stale activation distribution.

The final shape is therefore narrow in concept even though it is wide in channels: keep the simple pre-activation `B(3,3)` residual block, use far fewer blocks than the thousand-layer baseline, multiply the feature planes in the three residual groups by `k`, and place dropout only inside the residual branch when regularization is needed. The depth parameter should satisfy `(depth - 4) % 6 == 0`, giving `N = (depth - 4) / 6` blocks per group. Sanity-checking that convention against the names I want to use: `depth = 28` gives `(28-4) % 6 = 0` and `N = 4`, `depth = 16` gives `N = 2`, `depth = 40` gives `N = 6` — all integers, and the `% 6 != 0` depths are correctly excluded, so the assertion will catch a malformed depth instead of silently truncating. The stage widths are `16`, `16k`, `32k`, and `64k`; downsampling occurs at the first blocks of the second and third residual groups; the classifier ends with BN, ReLU, `8x8` average pooling, and a linear layer.

The implementation details need to make the shortcut semantics exact. If the input and output planes match, the addition uses the raw identity input. If the planes change, the shortcut is a `1x1` projection with the block stride applied to the first preactivated tensor, so the shortcut and residual tensors align. Dropout comes after the second BN/ReLU and before the second convolution. Convolutions have no bias. The weight scale should be He/MSR fan-in, with standard deviation `sqrt(2/(kW*kH*nInputPlane))`, so a PyTorch version should use `mode="fan_in"`, not fan-out.

Before trusting any of this I push a small batch through the assembled `WRN-28-10` to confirm the shapes line up the way the design assumes. A `32x32` input enters the stem at width 16, group 1 keeps `32x32` while widening to `160`, group 2 strides down to `16x16` at `320`, group 3 strides down to `8x8` at `640`; the `8x8` average pool then collapses the spatial dimensions to a single `640`-vector that the linear layer maps to the class logits. The output of a two-image batch comes back as `(2, 10)`, exactly the `(batch, num_classes)` I need, and the intermediate maps are `16x32x32 -> 160x32x32 -> 320x16x16 -> 640x8x8` — the two stride-2 transitions land where the design puts the downsampling, and the final `8x8` matches the pooling window with nothing left over. The projections fire exactly at the three group boundaries where the channel count jumps, which is what keeps the identity additions well-defined everywhere else. That end-to-end check is what convinces me the channel-width design is actually realizable as written, not just on paper.
