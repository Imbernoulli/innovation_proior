Let me start from what actually costs money when I serve one of these models. A Transformer is, to first approximation, a stack of linear projections — Q, K, V, the attention output, the feed-forward matrices — and at inference almost all of the arithmetic and almost all of the memory traffic is those matmuls. In FP16 every one of them is a floating-point multiply-and-accumulate, and floating-point multiplies are the expensive part: on a 7nm process a multiply costs several times what an add costs, and the chip's hard ceiling is power, so the FP multiplies are literally the thing limiting throughput. And before I even compute, I have to haul the weight matrices out of DRAM into the on-chip SRAM, and that bandwidth is the other bottleneck, growing with model size. So if I want a model that is cheap to run, the lever is obvious: make the linear layers stop doing floating-point multiplies, and make the weights small enough that loading them is cheap.

The reflex move is quantization — fewer bits per weight. But the way everyone does it is post-training: train a normal FP model, then snap the weights onto a coarse grid afterward. I keep coming back to why that has to break at low bit-width. Post-training quantization can only place the quantized weights *near* the float values it was handed; it has no way to go look for a different solution. And there is no reason the good low-bit network lives anywhere near the float optimum — the float optimum was found in a continuous landscape that the grid doesn't respect. Empirically this is exactly what happens: 8-bit is basically free, 4-bit holds with effort, but 2-bit is already lossy and 1-bit is a catastrophe — perplexities blowing up by twenty-plus orders of magnitude. So post-training is a dead end for the regime I want, which is *very* few bits.

The alternative is to make the weights low-bit *during training* — quantization-aware training. Then the optimizer is searching directly in (a relaxation of) the low-bit space, and it can find a network that is actually good when quantized, not a float network mutilated afterward. This is the right frame. But it has its own difficulties, and I should be honest about them before I commit. The forward operator that maps a real weight onto a discrete value — sign, or round-then-clip — is piecewise constant, so its derivative is zero almost everywhere. Backprop through it transmits nothing; the gradient dies at the quantizer. And second, even if I get *some* gradient, the latent real-valued weight gets nudged by tiny SGD steps, and a tiny nudge almost never moves a real weight across a quantization boundary, so the discrete weight just sits there while the latent weight drifts. Both of these are real walls, and I'll need an answer to each.

Take the gradient wall first, because without it nothing else matters. The function is non-differentiable; I can't use its true derivative because it's zero. So I lie about it in the backward pass. This is the straight-through estimator: in the forward pass I use the discrete value, but in the backward pass I pretend the discretizer was the identity function and let the gradient flow straight through to the latent weight. Is that justified? It's biased — I'm propagating the gradient of a function I didn't actually compute. But for a single layer it has the right sign: increasing the latent weight tends to increase (or keep) the discrete weight, so a gradient that says "increase the latent weight to decrease the loss" is pointing the right way, even if its magnitude is wrong. That's the deal I take. In code I want a one-liner that's exact in the forward and an identity in the backward: `x_q = x + (quant(x) - x).detach()`. Let me check it really does both halves. Forward: `x + (quant(x) - x) = quant(x)`, since `.detach()` only blocks gradient, not value — so the forward value is exactly the quantized one. Backward: the detached bracket is a constant w.r.t. `x`, so `d x_q / d x = d x / d x = 1`. I don't fully trust my own algebra on autograd, so let me actually run it — quantize `[0.37, −1.23, 0.8]` through `(x*4).round()/4` wrapped this way, sum, and read the gradient: the forward comes back equal to `quant(x)`, and `x.grad` is `[1, 1, 1]` for every element. Good — forward discrete, backward identity, confirmed by the autograd engine and not just by my reading of it.

The second wall — tiny updates never flipping a discrete weight — has a known answer too, and it's why I can't just store the discrete weights. I keep a full-precision *latent* weight that accumulates the updates, and I quantize it on the fly in every forward pass. The latent weight can drift by a hair on each step, and those hairs accumulate until eventually it crosses a boundary and the discrete weight flips. At inference I throw the latent weight away and keep only the discrete one. So the picture is: latent FP weight is the thing the optimizer touches, discrete weight is the thing the matmul touches, STE bridges the two. Good. Now the real design question: *what* discretizer, and with *what* scale?

The crudest discretizer is the sign — push every weight to `+1` or `-1`. But raw `sign(W)` throws away all magnitude information, and the network's outputs would be wildly off-scale. I need a scalar that rescales the discrete tensor back into the neighborhood of the real one. So I want `W ≈ α·B` with `B = sign(W)` and some positive `α`, and I'd better pick `α` to minimize the damage. The natural measure of damage is the squared error `J(B, α) = ||W − α·B||²`, and I can actually solve this. Expand it: `J(B, α) = α²·BᵀB − 2α·WᵀB + WᵀW`. Now `B ∈ {±1}ⁿ`, so `BᵀB = n` is just a constant, and `WᵀW = c` is fixed because `W` is given. So `J = α²·n − 2α·WᵀB + c`. Minimizing over `B` (with `α > 0`) is the same as *maximizing* `WᵀB = Σᵢ WᵢBᵢ`, and term by term the best I can do is `Bᵢ = +1` when `Wᵢ ≥ 0` and `Bᵢ = −1` when `Wᵢ < 0` — that is, `B* = sign(W)`, confirming sign is the right discretizer for this objective. Then for `α`, take `∂J/∂α = 2α·n − 2·WᵀB = 0`, so `α* = WᵀB*/n = Wᵀsign(W)/n = (1/n)Σᵢ|Wᵢ| = mean(|W|)`. So the L2-optimal scale isn't something I tune — it falls out as the *mean absolute value of the weights*. That's a satisfying result: sign for direction, absmean for scale, and a matmul against a `{-1,+1}` tensor needs no multiplies at all — only additions and subtractions, plus one scalar multiply by `α` at the very end to put the output back on scale.

There's a small refinement I've seen help in the binary-Transformer work: centralize the weights before taking the sign. Subtract the mean of the real weights first, so the binarized set is zero-centered, `W̃ = sign(W − W̄)` with the absmean scale on top. The intuition is information capacity — if the real weights are biased toward one side, plain `sign` sends almost all of them to the same value and the binary tensor carries almost no information; centering first balances the `+1`/`−1` populations and uses the one bit fully. I may not need this exact centering operation later, but the lesson is clear: the discrete levels have to be used symmetrically around zero.

Now activations. The weights are discrete, but the activations flowing in are still real, and an integer-friendly matmul wants them quantized too. Here the established tool is absmax quantization: to put a tensor into the signed `b`-bit range `[−Qb, Qb]` with `Qb = 2^{b-1}`, scale by `Qb/||x||_∞` — divide by the absolute maximum, multiply by `Qb`, round, clip — and dequantize by the inverse. For 8 bits `Qb = 128`, so the integer range is essentially `[−128, 127]`. Why absmax and not something fancier? Because it's parameter-free and symmetric, and symmetry matters: a symmetric `[−Qb, Qb]` scale keeps zero mapping to zero, so the quantized matmul stays a clean signed integer operation with no zero-point offset to carry around. But there's a known trap. Past a certain scale, activations grow large-magnitude *outlier* features — a tiny fraction of dimensions, maybe a tenth of a percent, that are systematically huge and show up in every layer. If I use one absmax over the whole tensor, that one outlier sets `||x||_∞`, and every other value gets squashed into a sliver of the 8-bit range and loses all its precision. The fix is granularity: don't take one scale per tensor, take one **per token** (per row). Then an outlier only blows up the scale of its own row; the other rows keep their precision. So: per-token absmax, 8-bit, symmetric.

Let me write down what I have as a binary layer and check it actually trains stably, because there's a variance issue lurking. Forward: `y = W̃ · x̃` where `W̃ = β·sign(W − W̄)` and `x̃` is the absmax-quantized activation. Suppose, as a back-of-envelope, the entries of `W` and `x` are independent, identically distributed, and independent of each other. Then for an output coordinate `y = Σⱼ w̃ⱼ x̃ⱼ` over `n` terms, `Var(y) = n·Var(w̃ x̃) = n·E[w̃²]·E[x̃²]`. The binarized weight has magnitude `β = mean(|W|)`, so `E[w̃²] = β²`, giving `Var(y) = n·β²·E[x̃²]`. The question is what `n·β²` actually equals — I'd like it near `1`, since a full-precision layer with standard init keeps `Var(y)` at the scale of `1`, and that order-1 output is exactly the condition that keeps deep Transformers from exploding or vanishing through the residual stream. So let me actually compute `n·β²` rather than wave at it. If the latent weights are `W ~ N(0, σ²)`, then `β = E|W| = σ·√(2/π)`, so `n·β² = n·σ²·(2/π)`. Now plug in the variance-preserving fan-in init, `σ² = 1/n` — the standard choice that keeps a real linear layer's output at unit scale. The `n` cancels: `n·β² = (2/π) ≈ 0.637`. So it is *not* `1` — it's an order-1 constant a bit below `1`, and the cancellation of `n` is the substantive part: the result is scale-free, independent of the layer width, which is what I actually care about. Let me push the check one step further, because the ternary version (which I'm heading toward) gates some weights to zero, and those zeros lower `E[w̃²]`. I simulate `n=10⁵` Gaussian weights, run them through the round-clip-then-dequantize I'll define below, and measure: `n·E[w̃²]` comes out near `0.44`, matching `(2/π)·p` where `p ≈ 0.69` is the surviving (nonzero) fraction. So the honest statement is `Var(y) ≈ 0.44·E[x̃²]` for the ternary layer — order-1, not `1`. That gap is the whole point: the matmul does *not* preserve activation energy on its own, it sits at a fixed sub-unity multiple of `E[x̃²]`. What keeps the residual stream from slowly decaying layer after layer is whatever pins `E[x̃²]` itself, and the clean way to pin it is to normalize the activations *before* I quantize them — a LayerNorm-family op right before the activation quantizer forces the normalized activation energy to a fixed scale, so after quantization the output variance stays at a fixed order through depth. Where to put it? Inside the sublayer, before the projection — sub-layer normalization, the stabilizer that large-scale Transformers already rely on. So the binary BitLinear is `y = W̃ · Quant(LN(x)) · (β·γ/Qb)`, with `β = mean(|W|)` the weight scale and `γ = ||LN(x)||_∞` the activation scale, the two dequant factors folded into the output.

One more stability lever I keep noticing. The latent-update-doesn't-flip-the-weight problem is worst at the very start of training, when I most want the weights to move fast. A small latent step rarely flips a discrete weight, so the effective update is sluggish and biased early on. The simplest cure is a *large learning rate* — make the latent steps big enough that they actually cross boundaries. And here's a pleasant asymmetry: a full-precision Transformer would diverge at that learning rate, but the discrete model tolerates it, because the discretization itself is a kind of regularizer that damps the runaway. So the 1-bit model can be trained at a learning rate that would blow up its float twin, and it *benefits* from it. I'll lean on that.

So I have a working binary recipe. But binary is the thing post-training failed at, and I have a nagging worry that two values is just too few — every weight is forced to participate at full magnitude `±β`; there's no way for the layer to say "this connection should be off." Watch what `sign` does to a weight that's essentially zero, a connection the network would rather not have: it still gets shoved to `+β` or `−β` at full strength, picking up whichever side of zero noise happened to land it on. The two-valued set has no way to express *absence*. Stare at that for a second — what I want is a third option, a way for the quantizer to set a weight to actually-zero and gate that connection off entirely.

What does adding a zero buy me concretely? Two things. First, capacity in the obvious counting sense: a set of three values carries `log₂(3) ≈ 1.58` bits per weight instead of one, so I'm spending barely more than a bit and getting a qualitatively richer representation. Second, and this is the real prize, a weight set to exactly zero contributes *nothing* to the matmul — its term drops out — so the zero acts as explicit feature filtering: the layer can learn which connections to switch off, and those switched-off connections cost zero arithmetic. That's not just expressiveness, it's free sparsity that a hardware kernel can exploit. So I want the discrete set to be `{−1, 0, +1}` — ternary, scaled.

Now I have to redo the discretizer-and-scale question for three values. I can't reuse the binary derivation directly — `sign` only ever emits `±1`. I need a rule that rounds a scaled weight to the *nearest* of `{−1, 0, +1}`. The construction I reach for: normalize the weight by some scale `γ`, round to the nearest integer, and clip into `[−1, 1]`. Define `RoundClip(x, a, b) = max(a, min(b, round(x)))`, and quantize as `W̃ = RoundClip(W / (γ + ε), −1, 1)`. Let me check this really is "snap to the nearest of `{−1, 0, +1}`" and find where the zero-band falls. A normalized value `W/γ` below `0.5` in magnitude rounds to `0`; between `0.5` and `1.5` rounds to `±1`; beyond `1.5` rounds to `±2` but the clip pulls it back to `±1`. So the rule gates a weight to zero exactly when `|W/γ| < 0.5`, i.e. `|W| < γ/2`. The `ε` is just a guard against dividing by zero when a whole tensor is degenerate. I don't want to trust the boundary arithmetic on faith, so I generate `10⁵` weights, run the round-clip, and audit it directly: every weight with `|W| < γ/2` does map to `0` (zero violations), and no weight in `[γ/2, 1.5γ)` is wrongly zeroed — the band is exactly where I derived it. So the small-magnitude weights, the weak connections, are precisely the ones switched off; that is the feature-filtering I was reaching for, now verified on actual numbers rather than read off the formula.

What should `γ` be? I want the same flavor of answer the binary case gave me — a parameter-free scale that comes from the data, not a knob. The natural carry-over from the binary derivation is the absmean again: `γ = (1/nm)·Σᵢⱼ|Wᵢⱼ|`, the mean absolute value over the whole weight matrix. Let me sanity-check that it sets the zero-threshold sensibly, and actually count the resulting populations rather than guess at "balanced." The threshold is `γ/2 = mean(|W|)/2`, so a weight gets gated off when its magnitude is below half the average magnitude — the genuinely small ones. For Gaussian latent weights I can even predict the zero fraction in closed form: `γ = σ·√(2/π)`, so `γ/2 = 0.399·σ`, and the zero fraction is `P(|W| < γ/2) = P(|Z| < 0.399) = 0.310`. The simulation above lands at `0.310` zeros with the remaining `0.690` split almost evenly between `−1` and `+1` (`0.344` vs `0.345`) — so absmean gives roughly a third zeros and a balanced two-thirds `±1`, which is a genuinely well-used ternary alphabet, not a degenerate one. That settles "nearly balanced" as a measured fact, not a hope. Now the dial behavior: if I'd made `γ` larger, the threshold `γ/2` rises, more weights fall below it, the tensor gets sparser — more zeros. So `γ` is also the sparsity dial: bigger `γ`, more zeros. The absmean choice is the neutral parameter-free setting that produces this balanced split; pushing `γ` up just to force more zeros would trade away useful connections. (There's also the option of *learning* the scale and threshold per layer — an elastic quantizer — but a learnable scale adds parameters and is touchier to train, and I specifically need to crank the learning rate, so I want the most stable, parameter-free quantizer I can get. Absmean it is.)

Now, why ternary and not some other small set? I should pin this down rather than wave at it. Two-valued `{−1, +1}` I've already argued is too rigid: no off switch, and a near-zero latent weight still becomes a full-strength connection. What about a *non-symmetric* set like `{0, 1}`? That breaks the thing I liked about symmetric quantization: with no negative value the matmul can't subtract, the representation is biased, and the optimizer has to fight a one-sided weight alphabet. The symmetry of `{−1, 0, +1}` around zero is doing real work: it keeps the matmul a balanced signed add/subtract and keeps the discrete representation centered. Could I go richer — `{−2, −1, 0, 1}` or `{−2, −1, 0, 1, 2}`? The `±2` is tempting because a multiply-by-2 is just a bit-shift, so I'd keep the multiplication-free property. But each added value costs bits, packing complexity, and a larger kernel contract. The first set that gives me negative, zero, and positive is `{−1, 0, +1}`. Stop there unless the math forces another value, and it doesn't.

I also want to revisit the activation handling now that I've committed to ternary, because there's a simplification I left on the table. In the binary recipe I followed the older convention of treating activations differently before nonlinearities — shifting them by their minimum into `[0, Qb]` so everything is non-negative, which needs a zero-point offset `η = min(x)`. But carrying a per-tensor zero-point around is annoying for both the implementation and any downstream kernel — it's an extra additive term that has to be tracked through the matmul. Do I actually need it? The reason for the `[0, Qb]` shift was to match the non-negative output of a ReLU, but I'm scaling activations per token with absmax anyway, and a symmetric `[−Qb, Qb]` range maps zero to zero with no offset. So I'll drop the asymmetric pre-nonlinearity scaling entirely and just use symmetric per-token absmax everywhere. It is simpler and friendlier to a low-bit kernel. So: activations are scaled per token to `[−Qb, Qb]` in the formula, with `Qb = 2^{b-1} = 128` for `b = 8`, and the actual int8 code clamps to `[-128, 127]` with scale `127 / max|x|`, no zero-point.

Let me also settle the backbone, because the layer has to live somewhere. I'll build on the LLaMA-style decoder — RMSNorm instead of full LayerNorm, the SwiGLU gated feed-forward, rotary position embeddings, no bias terms. This isn't a quantization decision; it's so the resulting model drops straight into the existing open-source serving stack with minimal friction. The one quantization-relevant interaction: the sub-layer normalization I needed for variance preservation is an RMSNorm, and I can *fuse* it into the BitLinear itself — put the RMSNorm at the front of the layer's forward. Then the surrounding decoder block can drop its own pre-attention and pre-FFN normalization, because BitLinear already normalizes its input. One normalization, in the right place, doing double duty.

Now which projections do I quantize? Only the parametric linear projections — attention Q/K/V/output and the feed-forward matrices — the ones that dominate compute and memory. I deliberately leave the rest high precision: the residual connections and normalizations are negligible compute; and I keep the input/output embeddings in full precision because the model has to produce high-precision probabilities to sample from, and a coarse embedding there would corrupt the distribution. So it's the projection matmuls that go low-bit, nothing else.

Let me assemble the BitLinear forward and make sure every piece is the thing I derived. The latent weight `w` is a normal high-precision parameter. In the forward I: normalize the input with the fused RMSNorm; quantize that normalized activation to 8-bit per-token absmax, wrapped in STE so the backward is identity; quantize the weight to ternary absmean RoundClip, also STE-wrapped; do the matmul with `F.linear`; and the dequant factors are folded in by the way I write the quantizers. Concretely I'll write each quantizer to *return the dequantized value* — quantize then immediately divide back by the scale — so that the value flowing through `F.linear` is already on the right scale and I don't have to track separate scalars through the matmul. For the weight, the scale is `1/mean(|W|)`: multiply by it, round, clip to `[−1, 1]`, divide back. For the activation, the scale is `127/max(|x|)` per token: multiply, round, clip to `[−128, 127]`, divide back. The STE wrapper `x + (q(x) − x).detach()` makes the forward use `q(x)` and the backward use the identity, for both the weight and the activation. That's the whole layer:

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


def activation_quant(x):
    """Per-token 8-bit absmax quantization (symmetric, no zero-point).
    Scale each row (token) by 127 / max|x|, round to the int8 range, divide back."""
    scale = 127.0 / x.abs().max(dim=-1, keepdim=True).values.clamp_(min=1e-5)
    y = (x * scale).round().clamp_(-128, 127) / scale       # quantize then dequantize
    return y


def weight_quant(w):
    """Ternary {-1, 0, +1} absmean quantization.
    scale = 1 / mean|W| (so |w|<mean|W|/2 -> 0: feature-filtering threshold);
    round to nearest of {-1,0,1}, clip, divide back by the scale."""
    scale = 1.0 / w.abs().mean().clamp_(min=1e-5)
    u = (w * scale).round().clamp_(-1, 1) / scale           # RoundClip then dequantize
    return u


def rmsnorm(x, eps=1e-5):
    return x * torch.rsqrt(x.pow(2).mean(dim=-1, keepdim=True) + eps)


class BitLinear(nn.Linear):
    """Native low-bit linear: ternary weights, 8-bit activations, every forward pass.
    Built on nn.Linear so self.weight is the high-precision latent parameter; the
    fused RMSNorm lets the surrounding LLaMA block drop its own pre-projection norm."""

    def forward(self, x):
        w = self.weight                                     # latent FP weight (accumulates updates)
        x_norm = rmsnorm(x)                                 # variance-preserving normalization, fused in
        # STE: forward uses the quantized value, backward is the identity
        x_quant = x_norm + (activation_quant(x_norm) - x_norm).detach()
        w_quant = w + (weight_quant(w) - w).detach()
        y = F.linear(x_quant, w_quant)                      # the (mostly) add/subtract matmul
        return y
```

This gives the concrete object I need: `self.weight` is the latent FP weight that fixes the tiny-update-doesn't-flip problem; `rmsnorm` is the sub-layer normalization that holds the output variance near `1`; `activation_quant` is symmetric per-token 8-bit absmax that survives outlier features; `weight_quant` is the ternary absmean RoundClip whose threshold gates the weak connections to zero; the two `.detach()` wrappers are the straight-through estimator; `F.linear` against a `{−1, 0, +1}` tensor is the matmul that needs almost no floating-point multiplies. For inference I'd offline-quantize the weights once to ternary, drop the STE entirely, fuse the RMSNorm into the activation quantizer, and replace `F.linear` with a low-bit kernel that does the integer add/subtract and applies the weight and activation dequant scales afterward — but the trained object is exactly the layer above.

There are a couple of training-time choices the layer doesn't show that I should reason out, because they follow from the same dynamics. The optimizer is Adam, betas around `(0.9, 0.95)`, short warmup. The learning rate I argued should be large; the refinement is a two-stage schedule — a high peak rate in the first half, then decayed midway to a lower rate for the second half. Why two stages and not a smooth decay? Because the loss curve for low-bit training isn't the gentle exponential of full-precision training; it has an S-shape, with a large drop happening *late*, near the end. A high rate early drives the fast flipping I need while the weights are sorting themselves out; then dropping the rate midway lets the model settle into that late descent cleanly, rather than rattling around. The same two-stage logic governs weight decay. Early on I use the standard `0.1` from the LLaMA recipe; in the second half I turn weight decay *off*. The reason is specific to latent weights: the magnitude of a latent weight acts as a confidence score for its discrete value — a large-magnitude latent weight is far from any boundary and won't flip, a near-zero one is on the fence. Weight decay shrinks latent magnitudes, which lowers confidence and makes the discrete weights flip more often. That churn is useful early, when the model is still finding its assignment of `{−1, 0, +1}`; but in the second half I want the discrete weights to *commit* and the model to converge, so I remove the decay and let the latent magnitudes grow into confident, stable assignments.

Let me trace the whole causal chain once. Serving cost is dominated by the linear projections' FP multiplies and weight memory, so I want weights natively in a tiny discrete set on every forward pass — but post-training quantization can't get there below ~4 bits because it can only perturb a finished float model, so I train the discreteness in from the start. Training through a non-differentiable quantizer needs the straight-through estimator (forward discrete, backward identity), and tiny SGD steps that never flip a discrete weight need a high-precision latent weight that accumulates updates and is quantized on the fly. The L2-optimal way to approximate a real tensor by a signed-binary one is sign-for-direction and mean-absolute-value-for-scale — that's the absmean. But two values can't express an off connection, so I add a zero: round-then-clip a weight normalized by its absmean onto `{−1, 0, +1}`, which gates the below-half-average weights to zero — explicit feature filtering at `log₂(3) ≈ 1.58` bits, symmetric so the matmul stays a balanced add/subtract; `{0,1}` would be unstable and more values would be needless. Activations go to 8-bit symmetric per-token absmax, per-token so an outlier feature can't crush the rest, symmetric so there's no zero-point to track. The matmul doesn't preserve activation energy — I computed it to sit at a fixed order-1 multiple of `E[x̃²]` (about `0.44` for the ternary layer, scale-free in the width `n`), so what holds the residual stream steady through depth is pinning `E[x̃²]` itself: an RMSNorm before the quantizer, fused into the layer. And because small latent updates flip discrete weights sluggishly — worst at the start — I train at a large learning rate the float twin couldn't survive, on a two-stage schedule matched to the S-shaped low-bit loss curve, with weight decay used as a confidence/flip knob that I switch off in the second half so the discrete weights commit.
