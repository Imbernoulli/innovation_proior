Let me start from what actually goes wrong. I have a long-tailed image dataset — a few head classes with thousands of examples each, a long tail of classes with a dozen or fewer — and an ordinary CNN trained with cross-entropy on it is useless on the tail. That's not mysterious: the network minimizes average loss, the average is dominated by the head, and the cheapest way to drive it down is to nail the head and let the rare classes collapse. So I need to fight the imbalance somewhere. Resampling is the obvious lever — over-sample the tail, under-sample the head — but in deep feature learning both hurt: over-sampling just repeats the same handful of tail images over and over, which slows training and overfits to those few real pixels, and under-sampling throws away head images the representation actually needs. I'd rather leave the data exactly as it is and fix the *loss*: multiply each example's loss by a per-class factor that compensates for how rare its class is. So the entire question reduces to one vector — a weight `w_c` per class — and I need to decide what it should be.

There is a principled reason to reweight at all, and it's worth getting straight because it tells me what a weight *means*. Cost-sensitive decision theory: if misclassifications carry different costs, the Bayes-optimal action is to predict the class of minimum expected cost. Elkan worked this out cleanly for two classes — predict positive iff `P(y=1|x) ≥ p*` with the threshold `p* = (c10 - c00)/(c10 - c00 + c01 - c11)`, where `c10` is the cost of a false positive and `c01` of a false negative. A standard classifier is trained to operate at the `0.5` threshold, and Elkan's theorem says that to shift it to a desired `p*` you can rescale the number of negative examples by `(p*/(1-p*))·((1-p0)/p0)` — or, if the learner takes per-example weights, *weight* each negative by that same factor instead of physically resampling. So a class weight is literally standing in for the cost of getting that class wrong. For the imbalance problem I want the cost to grow with rarity, so that a mistake on a rare class hurts as much as a mistake on a common one. Fine — but Elkan hands me the *mechanism* (reweight by class) and leaves the *magnitude* to me. How much should the cost grow with rarity?

The textbook answer is: declare every class equally important, which means cancel the empirical prior — weight inversely to frequency, `w_c ∝ 1/n_c`. Let me check it does what it claims. A sample from class `c` contributes `(1/n_c)·L` to the loss; there are `n_c` of them, so class `c`'s total contribution is `n_c·(1/n_c)·L = L`, the same for every class regardless of size. Good, that's exactly "rebalance to uniform." And this is widely used. But on genuinely long-tailed real data it performs *badly* — people training on real large-scale long-tail distributions, word-frequency data, weakly-supervised image collections, report that pure inverse frequency degrades performance and they had to back off to a softer rule. I want to see the failure quantitatively before I trust my hunch about its cause. Take a concrete imbalance: three classes with counts `1000, 100, 5`, the kind of 200:1 spread you get at imbalance factor 200. Normalize the inverse-frequency weights to sum to `C = 3`. The raw factors are `1/1000, 1/100, 1/5 = 0.001, 0.01, 0.2`, summing to `0.211`; scaling to sum to `3` gives `(0.0142, 0.142, 2.844)`. So the five-image tail class gets weight `2.84`, the thousand-image head class `0.0142` — a ratio of exactly `200`, the count ratio, pushed straight through. Now ask what that ratio does as the head grows. If the head class had `10000` images instead of `1000`, inverse frequency would hand the tail `2000×` the head weight; at `100000`, `20000×`. The ratio is the raw count ratio and it grows without bound. But those five tail images are not five independent looks at the concept — in real data they're near-duplicates of each other plus whatever the augmentation pipeline manufactures from them. So inverse frequency is amplifying five noisy, highly redundant gradients by an unbounded factor until they dominate training. It implicitly assumes `n_c` samples are worth `n_c` times as much as one sample — that the value of data is *linear* in the count — and for a redundant tail class that assumption is false, and the failure mode is exactly this runaway ratio.

So what did the practitioners switch to? A smoothed version, `w_c ∝ 1/sqrt(n_c)`. And empirically it's better — the square root compresses the dynamic range, so a 1000:1 count gap becomes only about 32:1 in weight, the tail gets boosted without the violent over-amplification. But stare at it and it's deeply unsatisfying. Why `1/2`? Why not `1/3`, or `2/3`? There's no derivation behind the exponent, nothing connects it to any property of the dataset, and it's a single frozen point — it sits *somewhere* between "no reweighting" (`w_c = 1` for all) and "full inverse frequency" (`w_c = 1/n_c`), but it gives me no handle to slide along that axis depending on how bad my imbalance is or how the dataset is structured. I have two extremes I understand and one mysterious heuristic wedged between them with a hardcoded exponent. What I actually want is a *family* — one interpretable parameter that, turned to one end, gives no reweighting, and turned to the other, gives inverse frequency, and in between gives a principled interpolation that I can match to the data. The sqrt is a symptom that the real quantity I should be reweighting by is *not* the raw count `n_c`.

Let me chase that. Inverse frequency's hidden assumption is that data value is linear in count. The sqrt's improvement is a hint that the true value grows *sublinearly* — that the marginal benefit of one more example shrinks as the class gets bigger. Why would it shrink? Because real images of a class overlap in information. The first image of a bird species teaches the network a lot; the tenth is largely redundant with the first nine; the thousandth is, for most purposes, a near-duplicate of something already seen, especially once you fold in the augmentation. Random crop, horizontal flip, rescale — the network is trained on a cloud of transformations around each image, so every image plus its augmentations covers a little *region* of feature space, not a point, and a new image whose region is already covered by previous ones adds essentially nothing. So the right question isn't "how many samples does this class have" but "how much non-redundant feature space does this class actually *cover*". That's a saturating quantity: it climbs fast at first and then flattens as the class's region fills up.

Let me try to make that precise, because if I can write down the expected covered volume I'll have a count-substitute that builds in diminishing returns by construction. Think of a single class. Let its feature region have total volume `N` — `N ≥ 1`, the amount of genuinely distinct stuff there is to learn about this class. Each sample is a unit-volume subset of that region (the sample plus all its near-duplicates and augmentations), and subsets can overlap. Sampling the class is then a covering process: I drop down `n` unit-volume subsets at random and ask how much of the region of volume `N` gets covered. The covered volume is what the model can actually learn from; call it `E_n`, the *effective number* of samples — the expected covered volume after `n` draws. This is exactly the random-covering problem: cover a big set with i.i.d. random small sets, expected covered volume. The trouble is that the honest version of this is brutal — the expected covered volume depends on the shape of the subsets and the dimension of the feature space, and there's no clean closed form. I need to simplify hard enough to get a formula but keep the saturation.

Here's the simplification that buys me everything: forbid *partial* overlap. When I drop the `n`-th sample, say it lands either *entirely inside* the already-covered region — adding nothing — or *entirely outside* it — adding its full unit volume. No half-in, half-out. Crude, yes, but it keeps the one feature I care about, that a new sample helps only to the extent the region isn't already covered, and it'll turn out to give exactly the saturating behavior with a one-line formula. What's the probability the `n`-th sample lands inside? If placement is uniform over the region, that probability is just the fraction already covered: after `n-1` samples the expected covered volume is `E_{n-1}`, the total is `N`, so `p = E_{n-1}/N`. With probability `p` the new sample is swallowed and the volume stays `E_{n-1}`; with probability `1-p` it lands outside and the volume becomes `E_{n-1} + 1`. Take the expectation:

```
E_n = p·E_{n-1} + (1 - p)·(E_{n-1} + 1)
    = E_{n-1} + (1 - p)
    = E_{n-1} + 1 - E_{n-1}/N
    = 1 + ((N-1)/N)·E_{n-1}.
```

And the base case is `E_1 = 1`: the first sample overlaps nothing, so it covers its full unit volume. So I have a clean linear recurrence, `E_n = 1 + ((N-1)/N)·E_{n-1}`, `E_1 = 1`. The coefficient `(N-1)/N` keeps showing up, so let me name it `β = (N-1)/N`. Since `N ≥ 1`, `β ∈ [0, 1)`. Now solve the recurrence. It's a first-order linear recurrence with constant coefficient `β` and constant forcing `1`; I'll guess the closed form and prove it by induction. Conjecture `E_n = (1 - β^n)/(1 - β)`. Base: `E_1 = (1 - β)/(1 - β) = 1` ✓. Inductive step: assume `E_{n-1} = (1 - β^{n-1})/(1 - β)`. Then

```
E_n = 1 + β·(1 - β^{n-1})/(1 - β)
    = [(1 - β) + β·(1 - β^{n-1})] / (1 - β)
    = [1 - β + β - β^n] / (1 - β)
    = (1 - β^n)/(1 - β).
```

The `-β` and `+β` cancel and the `β·β^{n-1}` collapses to `β^n`. So `E_n = (1 - β^n)/(1 - β)` for all `n`. That's a single closed form in one parameter `β`. Induction can hide an algebra slip, so let me run the recurrence forward numerically and compare. Take `β = 0.99`. Iterating `E_n = 1 + 0.99·E_{n-1}` from `E_1 = 1`: `E_2 = 1.99`, `E_3 = 2.9701`, `E_4 = 3.9404`, `E_5 = 4.9010`. The closed form `(1 - 0.99^n)/0.01` gives `1.99000, 2.97010, 3.94040, 4.90100` for `n = 2,3,4,5` — same numbers to the digits. The recurrence and the closed form agree, so the algebra held.

I want to read this formula two more ways before I trust it. First, it's a geometric series: `(1 - β^n)/(1 - β) = 1 + β + β^2 + ... + β^{n-1} = Σ_{j=1}^{n} β^{j-1}`. So the `j`-th sample contributes `β^{j-1}` to the effective count — the first contributes `1`, the second `β`, the third `β^2`, geometrically decaying. That is diminishing returns, made explicit: each successive sample is worth a factor `β` less than the last, because it's that much more likely to fall in already-covered territory. The saturation isn't bolted on; it's the geometric structure. Second, push `n → ∞`: the total effective number a class can ever reach is `Σ_{j=1}^{∞} β^{j-1} = 1/(1 - β)`. That limit should be the total volume `N` — you can't cover more than the whole region. And indeed `1/(1 - β) = 1/(1 - (N-1)/N) = 1/(1/N) = N`, exactly consistent with my definition `β = (N-1)/N`. So `β` is just a reparametrization of `N`, the number of genuinely distinct prototypes in a class. Let me watch the saturation actually happen, again at `β = 0.99` so `N = 100`. The closed form gives `E_10 = 9.56`, `E_50 = 39.50`, `E_100 = 63.40`, `E_500 = 99.34`, `E_1000 = 99.996`, `E_5000 = 100.00`. Early on, with `n` well below `N`, `E_n` tracks `n` fairly closely (`9.56` against `10`). But by `n = 500` it has stalled near `99`, and after that thousands more samples barely move it off `N = 100`. The effective count climbs and then flattens against its ceiling — which is the behavior I was after, and I didn't impose it, the recurrence produced it.

Now the two limits of `β`, because that's where this had better reduce to things I already understand if it's going to be trustworthy. Take `β = 0`, i.e. `N = 1`. Then `E_n = (1 - 0^n)/(1 - 0) = 1` for every `n` (with `0^n = 0` for `n ≥ 1`). One prototype: every sample is a duplicate of the one thing there is to learn, so the effective number is `1` no matter how many you collect. The other end, `β → 1`, i.e. `N → ∞`: I claim `E_n → n`. I can't just plug in `β = 1` — that's `0/0`. So L'Hôpital on the ratio in `β`. Let `f(β) = 1 - β^n` and `g(β) = 1 - β`. As `β → 1`, `f(β) → 1 - 1 = 0` and `g(β) → 0`, so it's the indeterminate `0/0`; `g'(β) = -1 ≠ 0` near `1`, and the derivative ratio `f'(β)/g'(β) = (-n β^{n-1})/(-1) = n β^{n-1} → n·1 = n` as `β → 1`, a finite limit, so L'Hôpital applies and `lim_{β→1} E_n = lim_{β→1} f'(β)/g'(β) = n`. So when `N` is huge — infinitely many distinct prototypes, no overlap at all — the effective number equals the actual count, every sample is unique. Exactly the two ends I needed.

Watch what those two limits do to the *weight*. I want the raw weight inversely proportional to the effective number, `r_c = 1/E_{n_c} = (1 - β)/(1 - β^{n_c})`, and then I can normalize all class weights together. At `β = 0`: every raw weight is `(1 - 0)/(1 - 0) = 1`, so after any sum-preserving normalization every class still has weight `1` — no reweighting at all. At `β → 1`: `E_{n_c} → n_c`, so the raw weight is proportional to `1/n_c`; after a sum-to-`C` normalization that should be inverse class frequency on the usual loss scale. I don't want to take "should be" on faith — these are exactly the two prior rules, so if this construction is the right generalization, sweeping `β` on my `1000, 100, 5` example had better reproduce them numerically at the two ends. Let me compute the normalized CB weights across `β`:

```
beta = 0.0       weights = (1.000, 1.000, 1.000)
beta = 0.9       weights = (0.675, 0.675, 1.649)
beta = 0.99      weights = (0.131, 0.206, 2.664)
beta = 0.999     weights = (0.022, 0.148, 2.829)
beta = 0.9999    weights = (0.0149, 0.143, 2.842)
beta = 0.99999   weights = (0.0143, 0.142, 2.843)
```

The `β = 0` row is `(1, 1, 1)` exactly — no reweighting, as the algebra promised. And the bottom rows march toward `(0.0142, 0.142, 2.844)`, which is precisely the normalized inverse-frequency vector I computed earlier from `1/1000, 1/100, 1/5`. So the two endpoints are not approximately the old rules, they *are* the old rules, recovered numerically. The single knob `β` slides continuously from *no reweighting* (`β = 0`) to *inverse frequency* (`β → 1`). The intermediate rows are new territory: at `β = 0.9` the tail weight is only `1.65` and the two larger classes sit near `0.68` — a much gentler boost than inverse frequency's `2.84` vs `0.014`. That gentler interior is the regime the smoothed-sqrt heuristic was groping for; for comparison the normalized `1/sqrt(n_c)` weights on this example are `(0.164, 0.518, 2.318)`, which sits between the `β = 0.99` and `β = 0.999` rows. So the sqrt rule is, numerically, just one frozen point inside this family — except here it's not a hardcoded exponent but a point on a curve I can move along, and the curve has a meaning: `β = (N-1)/N` encodes how much information overlap I believe the data has. The two rules I already trusted fall out as the endpoints and the mysterious middle becomes a tunable, interpretable interior.

A couple of design points I should pin down before I write the loss. Why assume one shared `N` — one `β` — for the whole dataset, rather than a per-class `N_c`? In principle each class has its own number of prototypes. But I can't identify `N_c` per class without extra information about each class's internal diversity, which I don't have; the per-class counts `n_c` alone don't tell me how redundant each class's images are. So the only thing I can actually fit is a single dataset-wide `β`, and it already captures the dataset-level diminishing-returns behavior — which is the regime where inverse frequency was failing. (If I later had a way to estimate per-class diversity, per-class `β_c` would be the natural extension.) And why does the *grain* of the dataset suggest different `β`? Because `β = (N-1)/N` is monotone in `N`, and `N` is the number of unique prototypes per class. A fine-grained class — one specific bird species — has fewer genuinely distinct prototypes than a coarse class — "bird" in general; its images saturate the region faster, so its effective `N` is smaller, so a *smaller* `β` is appropriate. A coarse, varied class wants a larger `N` and hence a `β` closer to `1`, nearer pure inverse frequency. So I shouldn't expect one `β` to be right across datasets of different granularity; `β` is the dial that lets me match the reweighting to how much redundancy the data carries, which is precisely the adaptivity the bare sqrt couldn't offer.

Now turn the effective number into the actual loss. For class `i`, the raw count-based factor is `r_i = 1/E_{n_i} = (1 - β)/(1 - β^{n_i})`. One nuisance: if I just multiply the loss by these raw factors, I quietly rescale the *total* loss magnitude — at `β = 0` every factor is `1` and the total is the ordinary loss, but at large `β` the summed scale changes, which silently changes the effective step size of an optimizer I'm supposed to hold fixed. So I normalize after computing the raw factors:

```
α_i = r_i / Σ_{j=1}^{C} r_j · C.
```

That makes `Σ_{i=1}^{C} α_i = C`, keeping the total loss on roughly the same scale as the unweighted loss for any `β`, so turning the reweighting on or sweeping `β` doesn't smuggle in a learning-rate change. (At `β = 0` this normalization is automatically the identity, since all raw factors are already `1` and sum to `C`.) With that, for a sample with true label `y` in a class of `n_y` training samples, the class-balanced loss is

```
CB(p, y) = α_y·L(p, y),    where the raw factor before normalization is (1 - β)/(1 - β^{n_y}),
```

where `L` is whatever base loss I'm using. And `L` is genuinely arbitrary here — the weight depends only on the class count, not on the predicted probabilities or the form of `L`, so this is loss-agnostic and model-agnostic. Drop in softmax cross-entropy and I get the weighted softmax form, `CB = -α_y·log(softmax(z)_y)`. Drop in sigmoid cross-entropy (treat each class as an independent one-vs-all predictor, which has the nice property of not assuming mutual exclusivity and unifying single-label with multi-label) and I get the same normalized class factor multiplying the one-vs-all sigmoid loss.

Now drop in focal loss and look at what the structure comes out to. Focal loss carries an α-balanced variant, `FL = -α_t (1 - p_t)^γ log(p_t)`, where `α_t` was always left as a free per-class hyperparameter — usually set by inverse frequency, or just tuned by hand. My class-balanced focal loss is `CB_focal = -α_y Σ_i (1 - p_i^t)^γ log(p_i^t)`, with `α_y` obtained by normalizing the raw factor `(1 - β)/(1 - β^{n_y})` across classes. Term by term that has the same shape as the α-balanced focal loss — same `(1-p_t)^γ`, same per-class `α` multiplier — but the `α` is no longer a dangling knob: the effective number supplies it from the class count. And the two multipliers don't interfere, because they read different things: `(1-p_t)^γ` depends only on the predicted probability `p_t` (sample difficulty) and `α_y` depends only on the class count `n_y`, so they're independent factors on the same loss. Focal handles difficulty, the effective-number weight handles count, and the product is well-defined whatever `γ` and `β` are.

Let me make sure the asymptotics-in-`n` behave sensibly for the loss too, because the runaway ratio is exactly where inverse frequency went wrong. For a tail class with small `n_y`, `1 - β^{n_y}` is small, so the raw factor is large — the tail is boosted. But the boost is capped by the geometry. With the series reading, `E_{n_y} = 1 + β + ... + β^{n_y - 1}`, so a one-sample class has `E_1 = 1` and raw factor `1` before the shared normalizer. The real difference is at moderate and large counts: inverse frequency keeps dividing by the full `n_c`, but the effective number saturates toward `N = 1/(1-β)`. Let me put the same growing-head test I used to indict inverse frequency back onto this rule and see whether it actually behaves better. Tail class `n = 5`, `β = 0.99` so `N = 100`: `E_5 = 4.90`, raw factor `1/E_5 = 0.204`. Head class `n = 1000`: `E_1000 = 99.996`, raw factor `0.0100`. Raw ratio tail/head `= 0.204/0.0100 = 20.4`. Now grow the head the way I did before. At `n = 10000`, `E ≈ 100.0` still, so the raw ratio is `0.204/0.0100 = 20.4` again; at `n = 100000`, `E` is still pinned at `100`, ratio still `20.4`. Compare the inverse-frequency ratios on the identical setup: `200`, `2000`, `20000`. So where inverse frequency's tail/head ratio grew linearly with the head count without bound, the effective-number ratio is frozen near `20`, i.e. near `N`, no matter how large the head gets. Once a class has many more than about `N` samples its raw factor stops shrinking — it is already "covered." That bounded ratio is the difference between the two rules.

So I can write the rule. The computation is pure — it sees only the per-class counts — and it returns the weight vector the cross-entropy criterion will use. Three lines do the math, then a normalization:

```python
import torch


def compute_class_weights(class_counts, num_classes, config):
    """Per-class weights from the effective number of samples.

    Effective number  E_n = (1 - beta^n) / (1 - beta),  beta = (N-1)/N in [0,1).
    Weight            w_c proportional to 1 / E_{n_c} = (1 - beta) / (1 - beta^{n_c}),
    then normalized so the weights sum to num_classes (keeps the loss scale fixed).
    beta = 0    -> all weights 1            (no reweighting)
    beta -> 1   -> w_c proportional to 1/n_c (inverse class frequency)
    """
    # beta = (N-1)/N encodes assumed data overlap; one shared value per dataset.
    beta = config.get("beta", 0.9999)

    counts = class_counts.float()
    # E_{n_c} = (1 - beta^{n_c}) / (1 - beta); the (1 - beta) cancels in 1/E_n, so:
    effective_num = 1.0 - torch.pow(beta, counts)        # 1 - beta^{n_c}
    weights = (1.0 - beta) / effective_num               # (1 - beta) / (1 - beta^{n_c})  ∝ 1/E_{n_c}

    # normalize so sum_c weights[c] = num_classes -> total loss stays on the unweighted scale
    weights = weights / weights.sum() * num_classes
    return weights
```

The chain is complete. I started stuck: cross-entropy on long-tailed data collapses the tail, resampling damages deep feature learning, so I reweight the loss per class — and cost-sensitive theory licenses that but won't tell me the magnitude. Inverse frequency, the obvious magnitude, assumes data value is linear in count and over-amplifies the redundant, noisy tail until it dominates; the field's patch, inverse-sqrt frequency, works better only by accident of a hardcoded exponent with no principle and no knob. The hint in the sqrt — value grows sublinearly because real images overlap in information — led me to recast "how many samples" as "how much non-redundant feature volume a class covers," a random-covering problem; simplifying overlap to all-in-or-all-out gave the recurrence `E_n = 1 + ((N-1)/N)·E_{n-1} = 1 + β·E_{n-1}`, which solves to `E_n = (1-β^n)/(1-β)`, a geometric series whose `j`-th term `β^{j-1}` is diminishing returns made explicit and whose limit `1/(1-β) = N` recovers the number of prototypes. Raw weighting by `1/E_{n_c}` and then normalizing the weights to sum to `C` gives a one-parameter family that slides from no reweighting (`β=0`) to normalized inverse frequency (`β→1`), with the smoothed heuristic now a principled, tunable, dataset-matched interior point; the normalization holds the loss scale fixed; and because the weight depends only on the count, it drops into softmax, sigmoid, or focal loss unchanged — in the focal case supplying exactly the per-class `α_t` that focal loss had left as a free knob.
