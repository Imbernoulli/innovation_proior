I start from the uneasiness around self-reference. A program that says "use my own code" sounds circular if I picture it as needing to inspect a mysterious current self. But computability theory does not need that primitive. It first turns programs into data by numbering them, then supplies ordinary computable operations over those numbers. Once descriptions are data, the question becomes whether a number can be arranged to stand in for the program that is eventually produced.

The effective numbering gives me `phi_e`, the behavior of the program with index `e`. This is only a naming convention until I add two effective capabilities: a universal evaluator that can run `e` on input `x`, and a specialization function from the `s-m-n` theorem. Specialization is the decisive operational move. If `q` is a two-input program, `s(q, a)` is a one-input program with `a` already baked into the first slot. So code can mention other code, and a compiler-like operation can manufacture new code from old code and data.

Now suppose I want a program that behaves like `Q(p, x)`, where `p` is the program's own index. I do not try to guess `p` directly. I build a helper program `B(q, x)` that computes `Q(s(q, q), x)`. This is computable because `s` is computable and `Q` is computable. Let `b` be an index for `B`. Then define `p = s(b, b)`. This is a finite, effective construction: take the code of `B` and specialize it with its own index as the static argument.

The fixed point appears when I unfold the definitions. Running `p` on `x` is the same as running `B(b, x)`, because `p = s(b, b)`. But `B(b, x)` was defined to run `Q(s(b, b), x)`. Since `s(b, b)` is exactly `p`, the behavior is `Q(p, x)`. Therefore `phi_p(x) ~= Q(p, x)` for every input `x`. No semantic oracle was used, and no step required knowing in advance what `phi_p` computes.

This is why the theorem is not a paradox. The construction is circular only when described informally after the fact; operationally it is staged. First there is a computable transformer from program texts to program texts. Then an index for that transformer is chosen. Then specialization produces a concrete index. Finally, when the program runs, the identity `p = s(b, b)` makes its own description available as a normal parameter. A paradox would require inconsistent truth conditions such as "this sentence is false." Here there is only an equality of partial computations produced by a fixed-point construction.

The Rogers version expresses the same mechanism from the angle of program transformation. Given any total computable `f` mapping indices to indices, construct a specification `Q(e, x) = phi_{f(e)}(x)`. Kleene's theorem gives `p` with `phi_p(x) ~= Q(p, x)`, hence `phi_p ~= phi_{f(p)}`. The fixed point is behavioral, not necessarily textual. The transformer may rewrite the program, add dead code, or compile it through another representation; the theorem says some index survives the transformation at the level of the computed partial function.

The deeper lesson is that self-reference is a capability created by representability. Once the system can enumerate programs and effectively produce specialized descriptions, it can close the loop on a program's own index. The theorem does not say every program can introspect its source by itself. It says for every computable way of using a would-be self-description, there is a program whose construction supplies exactly the description that use requires.

That also explains why quines are only the visible tip. A quine comes from choosing `Q(e, x)` to print `e` or a text represented by `e`. Recursive definitions come from choosing `Q` to call the computation named by `e` on smaller inputs. Self-recognition comes from choosing `Q(e, x)` to compare `x` with `e`. In all cases the same fixed-point skeleton turns an external parameter slot into the program's own description.
