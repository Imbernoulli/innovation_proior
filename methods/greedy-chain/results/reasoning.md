Let me start from what actually breaks. I have a frozen large language model and I want it to solve two very different-feeling kinds of tasks with nothing but a prompt: knowledge-heavy questions where it has to find and combine facts, and long interactive tasks where it has to drive an environment over many steps to a goal. And right now I have two separate tricks for these, each of which is good at exactly the thing the other is bad at, and I keep running into the gap between them.

Trick one is to make the model reason. If I show it a few examples whose answers don't jump straight to the result but instead write out the intermediate steps — for a word problem, "Roger started with 5 balls, 2 cans of 3 is 6, 5 + 6 = 11, the answer is 11" — then on a new problem it imitates that form and produces its own chain of intermediate steps before answering. Wei et al. showed that sufficiently large models can produce this kind of multi-step procedure for arithmetic, commonsense, and symbolic tasks. So the model clearly *can* string together a plan and execute it in language. But watch what happens the moment the chain needs a fact the model doesn't actually hold. It doesn't stop and say "I don't know this." It writes down *something* — a plausible date, a plausible founder, a plausible nationality — and then, because every later token is conditioned on the text already written, that invented fact becomes a premise the rest of the chain reasons *from*. One hallucination near the top and the whole chain is quietly poisoned downstream. This isn't a rare glitch; it has to be the characteristic failure mode of reasoning-by-prompting, because the chain is generated and consumed entirely inside the model. There is no point in the process where the model looks at anything outside itself. It's a closed box reasoning over its own pretraining residue.

Could I just sample harder? Wang et al.'s self-consistency says: don't greedily decode one chain, sample a bunch of them at some temperature, let each finish with its own answer, and take the majority vote. And it does help: a correct answer is usually reachable by several different routes, so the votes concentrate on it. But think about *why* it helps and you immediately see where it can't. It helps when the model knows the facts and just needs to be steered away from one unlucky reasoning path; the diversity averages out path-level noise. It does nothing about *missing* knowledge, because none of the sampled chains ever consults anything new; they're all drawing from the same internal well. If the model simply doesn't have the fact, or has it wrong, many sampled chains can march confidently to the same wrong answer, and a majority vote over wrong answers is still wrong. Sampling more of an ungrounded process gives me more ungrounded process. The missing piece is grounding, and self-consistency treats the symptom.

So the obvious thing the reasoning side is missing is contact with the world. And I do have machinery for that — that's trick two. I can use the same frozen model as a *policy*: turn the environment's observation into text, ask the model for the next action, execute it, feed back the new observation, repeat. People have built real systems this way — a model that drives a web browser to answer questions, a model that picks robot skills, a model that plays a text game. So acting-by-prompting is a thing that works. Let me see if I can just bolt the world onto the reasoning problem with it. Give the model a tiny Wikipedia interface — it can search for an entity and read the first few sentences, it can look up a string on the current page like Ctrl-F, it can declare it's finished with an answer — and let it generate a sequence of these actions to answer a multi-hop question.

Let me actually run this in my head on a real multi-hop question, the kind that needs two hops: "What is the elevation range for the area that the eastern sector of the Colorado orogeny extends into?" Pure-action policy: Search[Colorado orogeny]. Observation: it was an episode of mountain building in Colorado. Now the model has to emit the next action straight from the history with nothing in between. It might Lookup[eastern sector], get "the eastern sector extends into the High Plains," then Search[High Plains], get a disambiguation stub, then Search[High Plains (United States)], get "the High Plains rise in elevation from around 1,800 to 7,000 ft." So far so good — the action sequence is even mechanically fine. But now comes the step that I think kills it: the model has, sitting in its observation history, the question, the chain of searches, and the sentence with the elevation range, and it has to map all of that *at once* onto the single correct final action, Finish[1,800 to 7,000 ft]. That mapping — read the whole trajectory, realize the elevation sentence is the answer to the original multi-hop question, and emit it — is exactly the implicit, computation-heavy step the action-only setup gives the model no room to perform. There's no place in the sequence where it can write down "the eastern sector goes into the High Plains, so I need the High Plains' elevation, and that observation gives it, so the answer is..." It has to do all of that silently, in one forward pass, and commit to an action token. I'd predict two characteristic failures from this: an act-only policy gathers useful observations and still fails to synthesize the answer, or it wanders and repeats a search because nothing in the context records what it is trying to do. I can't measure the rates from my desk, but the structural reason for both is the same missing scratchpad.

Make it worse — a genuinely long task, like a household text game where I have to examine an object under the desklamp across a large simulated home. The action-only policy has to decompose this implicitly into "find the object, find the desklamp, bring them together, turn on the lamp," track which part it's done, and use commonsense about where a desklamp probably is (on a desk, a shelf, a dresser) — all with no scratchpad. So I'd expect the predictable things: it loses the goal halfway through, it forgets it already checked a drawer and checks it again, it can't tell when a subgoal is finished. The model has plenty of commonsense from pretraining about where desklamps live, but the act-only format gives it no moment to *say* that commonsense and condition on it.

So I'm stuck between two failures that are almost mirror images. Reasoning-only: a rich internal monologue, decomposition, exception-handling, synthesis — all the cognitive moves — but a closed box, no way to fetch or correct a fact, so it hallucinates. Acting-only: real contact with the world, real observations flowing in — but no place to think between actions, so it can't decompose, can't track, can't synthesize. One has thought without grounding; the other has grounding without thought.

Now, the prior work on the acting side had *started* to add something language-shaped between actions. Web browser agents show that a language model can navigate pages and answer questions, but they learn that behavior from imitation and human-feedback reinforcement rather than from an explicit thinking step. Robot skill systems go another way: the model scores which skill would help the instruction, and a learned value or affordance estimate says whether that skill is feasible in the current state; the product grounds the choice, but the model is still choosing among skills rather than writing a working memory. The closest closed-loop setup pipes the environment's feedback back into the prompt — a success/failure signal, a description of the current scene, a human's reply — so the model can replan against fresh state. And that's genuinely a step: it's a closed loop, observations re-enter the model. But look hard at *what kind* of language it injects, because that's the crux. The load-bearing text is tied to the environment: here's what just happened, here's the state, here's what's left to do. It's the world talking back to the model. It still leaves no general-purpose scratchpad where the model can decompose the goal, recall that desklamps tend to be on desks, notice an exception and adjust the plan, or work out what an observation *means* for the next move. The "monologue" there is state feedback more than free-form internal thought. So even the closest prior system is grounding-plus-feedback, not a general mechanism for reasoning and acting to share the same running state.

Let me sit with the mirror-image structure, because the symmetry is suspicious. Reasoning generates internal text that helps it plan and synthesize but touches nothing; acting touches the world but generates no internal text. What if the reason they help opposite failure modes is that they're literally the same kind of operation — emitting a token sequence — pointed at two different targets, and I've been keeping them in two different boxes for no good reason? When I reason, I produce language that updates my own working context. When I act, I produce a token sequence that the environment consumes and answers. The first changes only what I'm conditioning on; the second changes the world and then changes what I'm conditioning on. Both are just "the model writes the next chunk of the sequence." Why should the model have to leave the trajectory to think?

Let me make that precise with the agent-environment frame. At step `t` the agent has context `c_t = (o_1, a_1, ..., o_{t-1}, a_{t-1}, o_t)` and picks an action `a_t` from the action space `A` under `pi(a_t | c_t)`. The thing I keep needing — and keep being unable to express — is an action whose *only* effect is to add some reasoning into the context for the next step to use, without poking the environment. The environment-changing actions live in `A`. Internal reasoning lives in the space of language, call it `L`. So enlarge the action space: let the agent choose from `A_hat = A ∪ L`. An ordinary action `a_t ∈ A` does what it always did — it hits the environment and the environment returns an observation. But a language action — call it a *thought* — `a_hat_t ∈ L` is defined to have no effect on the environment, and therefore produces no observation. What it does instead is compose information by reasoning over the current context and update it: `c_{t+1} = (c_t, a_hat_t)`. The thought is a free move that writes onto the scratchpad. The next step — whether it's another thought or a real action — conditions on it.

This is cheap to write down, which makes me suspicious that I've just relabeled the problem rather than solved it. So before I commit, let me actually run the augmented policy step by step on the same Colorado question and watch whether the specific step that killed the act-only policy now has somewhere to happen — and watch for the place it could still break. Thought 1: "I need to search Colorado orogeny, find the area the eastern sector extends into, then find that area's elevation range." That single sentence *is* the decomposition the act-only policy could never externalize — and now it's in the context, so every later step is conditioned on the plan. Action 1: Search[Colorado orogeny]. Observation: mountain-building episode, no mention of the eastern sector. Here is the first place I worried it might break: the observation didn't contain what the plan asked for. Thought 2: "It doesn't mention the eastern sector, so I should look up eastern sector." So the thought *can* carry exception-handling — the observation contradicted the plan and the next thought repaired it, in language, grounded on what the world just said. Lookup[eastern sector] -> "extends into the High Plains." Thought 3: "So the eastern sector goes into the High Plains; I need the High Plains' elevation range." Search[High Plains] -> a disambiguation stub. Thought 4: "I need High Plains (United States) instead." Search[High Plains (United States)] -> "rise in elevation from around 1,800 to 7,000 ft." Thought 5: "So the answer is 1,800 to 7,000 ft." Finish[1,800 to 7,000 ft]. Now go back to the exact step that choked the act-only policy: the silent map from the whole trajectory onto the final action. In this trace it isn't silent — it's Thought 5, an explicit sentence that conditions directly on the Observation that grounds it, and the Finish only has to copy what the thought already said. The step that had no room to happen now has a place to happen. That's the half I was unsure about (reason-to-act). The other half I already get for free: when Thought 3 needed the High Plains' elevation, the policy could Search for it instead of inventing a number, so the reasoning never has to fabricate the fact it lacks (act-to-reason). The two failure modes were complementary because the two abilities are complementary, and this trace shows the interleaving lets each cover the other on at least this instance. I'd still want to confirm the win on a batch of HotpotQA and FEVER, not one hand-run question — a single favorable trace is suggestive, not proof — but the mechanism does what I needed it to do here.

Now I have to confront the cost of making the action space `A_hat = A ∪ L`. `L` is the space of *all* language — unbounded. A policy that has to learn, from scratch, which of infinitely many possible sentences is a useful thought at this moment is hopeless; there's no signal dense enough to pin that down. So I can't be learning this from nothing. But I don't have to — that's exactly what a big pretrained model already gives me. The reason chain-of-thought worked at all is that the model's language prior already knows what a sensible next reasoning step looks like; the few-shot exemplars just point it at the form. So I lean on the same thing: a frozen large model, prompted with a handful of human-written trajectories that interleave thoughts, actions, and observations, and asked to continue the pattern on a new instance. The unbounded `L` is tamed by the strong language prior, not by search. No gradient updates, no reward signal — just demonstrations. And the binding cost becomes annotation, not training, so the prompt format has to be easy for a human to write.

How do I write those exemplars? Here's the part I like: I don't have to design a thought format or a special syntax. A demonstration is just a human solving the task and *typing their thoughts in plain language on top of the actions they take* — "I need to search X, then find Y"; "the paragraph doesn't say Z, maybe look up W"; "X was started in 1844"; "1844 < 1989, so X came first"; "so the answer is..." Let me sanity-check that comparison-style exemplar arithmetic, since I'm about to rely on the model imitating it: a HotpotQA question like "Which started first, X (1844) or Y (1989)?" needs the thought to compare two years and pick the earlier. 1844 < 1989, so X started first — the right answer is X, and the comparison is the kind of one-line check the thought slot is exactly meant to hold. The thought space is whatever language can express, so I never have to enumerate thought *types* or choose a rigid template; the same flexible mechanism handles decomposing a question, injecting a commonsense fact, doing a little arithmetic like that comparison, reformulating a failed search, and stating the final answer. That flexibility is the whole reason for putting reasoning in `L` rather than in some fixed slot; the model is recognizing a form it already understands.

Now, *how often* should a thought appear? My QA walk-through had a thought before essentially every action — dense interleaving — and that felt right there, because every step of a multi-hop question benefits from a thought: decompose, respond to the observation, reformulate, synthesize. But push that pattern onto the long household task and I expect it to backfire. That task can run through many primitive actions; if I force a full reasoning sentence before every "go to drawer" move, I balloon the trajectory, and since this is in-context learning the whole running prompt has to fit in the context window. Let me put rough numbers on it so I'm not hand-waving: a household episode can easily be a few dozen primitive moves, and a useful thought sentence is on the order of a dozen-plus tokens, so a thought-before-every-action policy adds hundreds of tokens of mostly-routine reasoning to a prompt that already carries the instruction and several full exemplars. A thought before every micro-action would crowd the budget and bury the few genuinely useful thoughts in noise. What I actually want there is thoughts only at the junctions that matter: decompose the goal up front, note when a subgoal is done, decide the next subgoal, recall where an object is likely to be. Those are sparse and their positions are task-dependent — I can't hand-place them, and I shouldn't, because the model is better positioned than I am to judge when a thought earns its tokens. So: for reasoning-heavy tasks I interleave thought and action densely in the exemplars; for decision tasks with long horizons I write exemplars where thoughts appear sparsely, and let the model decide for itself when to emit a thought versus an action. The same augmented action space `A_hat` supports both regimes; only the density of the demonstrations differs.

The policy is now "predict the next token sequence and route it," and the routing is what makes it work. The running prompt is: a short instruction describing the task and the available actions, then the few-shot interleaved exemplars, then this instance's initial observation. At each step I want the model to produce the next thought *and* the next action, and I want to draw a hard line between what the model writes and what the environment writes, because the model must never get to hallucinate an observation. So I prompt for the thought-and-action and I make the model *stop generating* right before the observation slot. Concretely: I append "Thought i:" and call the model with a stop sequence at "\nObservation i:". The model fills in the thought and, on the same continuation, the action ("...\nAction i: Search[High Plains]"), and halts before it can invent an observation. I split that continuation into the thought and the action. Then *I* execute the action against the environment — `env.step(action)` — and *I* paste the real observation back into the prompt. So one model call yields a thought and an action together; the observation always comes from the world.

Before I trust that "split the continuation into the thought and the action" line, I should actually run the split on the strings the model will hand me, because this is exactly where a clean-looking design quietly breaks. Take a well-formed continuation, the body after "Thought i:": `"I need to search Colorado orogeny, find the area the eastern sector extends into, then find the elevation range.\nAction 1: Search[Colorado orogeny]"`. Splitting on `"\nAction 1: "` gives two pieces: the thought `"I need to search Colorado orogeny, ... then find the elevation range."` and the action `"Search[Colorado orogeny]"`. Good — that's the clean case, and it does unpack into exactly two parts. But the model won't always cooperate. Suppose it just writes a thought and forgets the action, e.g. `"It doesn't mention the eastern sector. Maybe I should look that up."` — there's no `"\nAction 1: "` substring at all, so the split returns a single-element list and the two-target unpack raises `ValueError: not enough values to unpack (expected 2, got 1)`. So I can't assume the split succeeds; I have to catch that. And there's a second way it goes wrong that I'd have missed if I hadn't traced it: if the continuation contains the action marker *twice* (the model rambles and writes a second "Action 1:"), the split yields three pieces and the same two-target unpack raises `ValueError: too many values to unpack (expected 2)`. The convenient fact is that *both* malformations raise the same `ValueError`, so one `except ValueError` handles them together. In the handler I don't want to crash and I don't want to throw away the step, so I keep the first line of what the model gave as the thought and make a second, narrower call that prompts explicitly for "Action i:" and stops at the newline — that always yields a single parseable action. I count that as a bad call because it is a real extra model call, and tracking the rate tells me how often the format is misfiring. So the robust version of the routing is: try the clean split; on `ValueError` (either flavor), fall back to thought-plus-a-second-action-call.

One more concrete detail I only catch by looking at what `env.step` expects. The environment's command verbs are lowercase — `search`, `lookup`, `finish` — but the model, imitating the instruction's capitalized "Search/Lookup/Finish", emits `Search[High Plains]`. If I pass that straight in, the parser won't recognize the verb. The fix is to lowercase just the first character: `action[0].lower() + action[1:]` turns `"Search[High Plains]"` into `"search[High Plains]"` — it touches only the leading verb letter and leaves the bracketed argument's capitalization (`High Plains`) untouched, which is what I want, since the entity name's case can matter for the search. The thought, as defined, touched nothing in the environment; only this action did. After appending "Thought i: ... / Action i: ... / Observation i: ..." the prompt is ready for step i+1, and the model conditions on everything: its own prior reasoning and the real observations.

A couple more rough edges around termination. The loop has to stop somewhere. The model signals completion with a Finish action; when the environment returns done, I stop. But it might not finish on its own within a bounded interaction budget, so I cap the steps. If I hit the cap without a Finish, I send the environment `finish[]` to close the episode with an empty answer instead of pretending the model found one — an empty answer scores as wrong, which is the honest outcome, rather than leaving the episode open. Greedy decoding throughout: one trajectory, deterministic, no tree search, no backtracking. With greedy decoding the model can fall into repeating a previous thought-and-action verbatim, unable to break out of the loop — that's a real cost of committing to a single deterministic chain, and the step cap is what bounds the damage when it happens. But the single greedy chain is the simple, reproducible core, and everything fancier (sampling, voting, search) can be layered on top of it later.

One more thing I notice while staring at the action interface, because the design of the actions themselves matters. The Wikipedia interface I gave the model is deliberately *weak* — search returns only the first few sentences of an exactly-named page (or, if the page doesn't exist, just a list of similar entity names), and lookup only steps to the next sentence containing a string. I could have wired in a strong neural retriever that fetches the most relevant passage. But a strong retriever would do the model's thinking for it: it would surface the answer-bearing sentence without the model ever having to reason about *what* to look for or *how* to reformulate when a page is missing. By keeping the interface as crude as a human Ctrl-F'ing through Wikipedia, I force the retrieval to be *driven* by explicit reasoning — when Search[Adam Clayton Powell] returns the wrong "Adam Clayton Powell" and a list of similar names, the model has to think "to find the documentary I should search Adam Clayton Powell (film)" and reformulate. The weakness of the actions is what gives the thoughts something load-bearing to do.

The core policy is the single greedy chain that fills the one empty slot in the agent harness. The structure is exactly the routing I just traced: one model call per step yields thought-plus-action with a stop before the observation; I try the clean split and fall back on `ValueError`; I lowercase the verb and execute the action; I paste back the real observation; I append and continue; I stop on Finish or close at the cap with `finish[]`.

```python
def llm(prompt, stop=None):

    temperature 0, stopping when any string in `stop` is produced."""

INSTRUCTION = (
    "Solve a question answering task with interleaving Thought, Action, Observation "
    "steps. Thought can reason about the current situation, and Action can be three "
    "types:\n"
    "(1) Search[entity] -- returns the first sentences of the entity's wiki page, or "
    "similar entity names if it does not exist.\n"
    "(2) Lookup[keyword] -- returns the next sentence on the current page containing "
    "keyword.\n"
    "(3) Finish[answer] -- returns the answer and finishes the task.\n"
    "Here are some examples.\n"
)


def greedy_chain(env, max_steps=7, idx=None):
    """Greedy single chain: alternate model-generated Thought+Action with real
    environment Observations, in one trajectory, no backtracking."""
    question = env.reset(idx=idx)
    prompt = INSTRUCTION + FEWSHOT_EXEMPLARS + question + "\n"
    n_calls, n_badcalls = 0, 0
    done = False
    reward, info = 0, {}
    for i in range(1, max_steps + 1):
        n_calls += 1
        # One call yields this step's thought AND action; stop BEFORE the observation
        # slot so the model can never fabricate an observation -- the world fills it in.
        thought_action = llm(prompt + f"Thought {i}:", stop=[f"\nObservation {i}:"])
        try:
            thought, action = thought_action.strip().split(f"\nAction {i}: ")
        except ValueError:
            # Model didn't emit a clean Action; take the thought, then ask narrowly
            # for the action alone and stop at the newline.
            n_badcalls += 1
            n_calls += 1
            thought = thought_action.strip().split("\n")[0]
            action = llm(
                prompt + f"Thought {i}: {thought}\nAction {i}:", stop=["\n"]
            ).strip()

        # The action -- and only the action -- touches the environment; the thought
        # was an inert language action that just updated the context.
        obs, reward, done, info = env.step(action[0].lower() + action[1:])
        obs = obs.replace("\\n", "")

        # Fold thought, action, and the REAL observation back into the running context
        # so the next step conditions on prior reasoning and grounded facts alike.
        prompt += (
            f"Thought {i}: {thought}\n"
            f"Action {i}: {action}\n"
            f"Observation {i}: {obs}\n"
        )
        if done:                       # the model emitted Finish[...]; task closed
            break

    if not done:                       # hit the step cap without finishing
        obs, reward, done, info = env.step("finish[]")   # close with an empty finish
    info.update({"n_calls": n_calls, "n_badcalls": n_badcalls, "traj": prompt})
    return reward, info
```
