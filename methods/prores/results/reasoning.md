Let me start from what actually goes wrong when I push a Transformer language model deeper. I stack identical blocks, each one writing the output of an attention or feed-forward sublayer back onto a shared residual stream, `x_{l+1} = x_l + F_l(x_l)`. The skip `x_l` is the highway: it's why deep nets train at all, because it gives gradients a clean path backward and lets each block begin as a small nudge to the identity. And yet two things break as the stack grows. Training gets jittery — loss spikes, gradient spikes, the deeper I go the more fragile the run, to the point where I'm dropping the learning rate or leaning on a warmup just to not diverge. And worse, when I finally do train a deep model and probe it, the deepest layers turn out to be doing almost nothing — I can lop them off with barely a change in loss. So I'm paying for depth twice over: it destabilizes the run, and the depth I bought is half-dead. I want both fixed at once, and I want it cheap: no new parameters to speak of, no surgery on attention or the FFN, and ideally it should work whether I'm in pre-norm, post-norm, or one of the hybrids.

Why are the deep layers dead, concretely? Let me trace the pre-norm recursion, since pre-norm is what I actually run. The block is `x_{l+1} = x_l + F(Norm(x_l))`. The normalization fixes the scale of the *branch input*, the thing fed into attention or the FFN, but it does nothing to the *stream* `x_l` that I keep adding onto. So the stream variance keeps accumulating with depth. Make that quantitative under the usual zero-mean normal-weight assumptions. The variance scale satisfies `σ²_{x_l} = σ²_{x_1}·Θ(∏_{k=1}^{l-1}(1 + 1/σ_{x_k}))` — every layer contributes a factor `(1 + 1/σ_{x_k})` above one to the running stream scale. Bound it at depth `L`: the slowest growth is linear, `Θ(L) ≤ σ²_{x_L}`, and the dangerous upper branch is exponential, `σ²_{x_L} ≤ Θ(exp L)`.

That bracket is wide enough that I should figure out which end I'm actually near, so let me just iterate the recursion. Start `σ²_1 = 1` and run `σ²_{l+1} = σ²_l·(1 + 1/σ_l)`. I get `σ²` at layers `(1, 8, 16, 32, 64) = (1.0, 17.2, 63.7, 250.0, 1001)`. That is not linear-in-`l`: linear would put layer 64 near 64, and I'm at ~1000. Look at the ratios instead — `σ²_32/σ²_16 = 3.92` and `σ²_64/σ²_32 = 4.00`. The variance roughly *quadruples every time I double the depth*, i.e. `σ²_l ∝ l²` (and `σ_l ∝ l`), a power law that sits well above the linear floor. So in practice the stream scale climbs fast with depth — much closer to the dangerous end of the bracket than the benign one — and I should expect this to be what makes the top layers useless, though I still have to show *how* a growing `σ` kills a layer.

Now here's why climbing variance kills the deep layers, and it's not obvious until I write the Jacobian. The block map is `Pre-LN(x) = x + f(Norm(x))`, so `∂Pre-LN(x)/∂x = I + (∂f/∂Norm)·(∂Norm/∂x)`. Look at that second term. `Norm` divides by the stream's own scale, so `∂Norm/∂x` carries a `1/σ_{x_l}` — when the stream variance is large, the normalization is dividing by a big number and its Jacobian is tiny. So the whole second term shrinks as `σ_{x_l}` grows, and the block Jacobian should collapse toward `I`. Chain that across the whole stack and the end-to-end derivative `‖∂y_L/∂x_1‖` is bounded by a product `∏_l(1 + A/σ_{x_l} + B/σ²_{x_l})` for some constants `A, B`.

I want to know whether that product actually saturates or whether it just grows slowly, so let me feed the `σ_l` I just computed into it with `A=B=1`. Over 64 layers the product is `5034`. Now stretch to 256 layers — four times as deep — and it's `83383`. That extra `192` layers multiplied the end-to-end Jacobian by only `16.5x`, even though they are more than half the network. The per-layer factor tells the same story directly: at `l=1` the block contributes `1 + A/σ + B/σ² = 3.0` (log `1.10`), at `l=32` it's `1.07` (log `0.065`), at `l=128` it's `1.016`, at `l=256` it's `1.008` (log `0.008`). The log-contributions are falling like `1/σ ∝ 1/l`, so the sum `Σ log(1 + A/σ_l + ...)` grows like `Σ 1/l` — it diverges, but only *logarithmically*, which is why I can quadruple the depth and barely move the Jacobian. A deep block contributing a factor of `1.008` is doing almost nothing: its Jacobian is `≈ I`, so changing its input barely changes its output. So the deepest layers in a deep pre-norm net really are pushed toward identity behavior, which is exactly the redundancy the pruning probes keep finding: chop the top third and very little changes. The variance growth isn't a cosmetic numerical annoyance; it's the mechanism that turns deep layers into dead weight.

So one school of thought says: kill the variance growth and the deep layers come back to life. That's the LayerNorm-Scaling move — divide the branch input by `√l` at layer `l`, `x_{l+1} = x_l + F(Norm(x_l)/√l)`, which deliberately down-weights deeper residuals so the variance curve flattens toward linear, keeping the deep Jacobians off `I`. It works, and the math above tells me *why* it works: shrink the per-layer increment with depth and the product `∏(1 + 1/σ)` — which in my iterated recursion is what drove `σ²_l` up to `~l²` — can't run away. But sit with the shape of that fix. The factor is `1/√l` — it depends only on depth, and it is *static*, baked in for the entire run. As I make the model very deep, `1/√l` clamps the deepest branches hard, permanently. Early on that's helpful, it's controlling the explosion. But late in training, when everything has stabilized and I *want* those deep layers learning at full strength, the `1/√l` is still throttling them. I'm trading "dead because the variance exploded" for "dead because I clamped them." The cure is right but it's frozen in time, and that's the seam I want to pull on.

Let me look at the other school, the one that thinks the disease is instability of the *update* rather than variance of the *activation*. Fixup, T-Fixup, DeepNorm: they pin down that what blows up a deep Transformer is the *model update* — one optimizer step moves the function by an unbounded amount when the net is deep — and they bound that update by a constant. DeepNorm makes it concrete: `x_{l+1} = Norm(α·x_l + F_β(x_l))` with a *constant* skip scale `α` — `(2L)^{1/4}` for a decoder-only `L`-layer stack — plus a matching branch-weight scale `β = (8L)^{-1/4}`, and they prove the update is bounded at init; it trains past a thousand layers. Beautiful, but notice *where* the bound lives: it's derived from the network's behavior *at initialization*, and `α` and `β` are constants applied uniformly for all of training. The early chaotic phase is exactly when the update is worst, so a bound sized for that moment makes sense then. But training isn't one regime. There's a warmup where updates are large and chaotic, then a long stable phase of small gradual changes, then a decay — that's literally why people run Warmup-Stable-Decay learning-rate schedules. A constraint calibrated for the worst instant of training and held fixed through the stable phase is over-conservative once I'm in the stable phase; it caps the learning capacity of the layers it's protecting. Same frozen-in-time problem as `1/√l`, just from the update side instead of the activation side.

And then there's the scalar-on-the-branch family — ReZero, SkipInit — which is the closest in form to what I keep circling. ReZero puts one scalar on each branch and *learns* it: `x_{i+1} = x_i + α_i·F[W_i](x_i)`, every `α_i` initialized to 0. That zero init is lovely. At `t=0` the whole network is *exactly* the identity, `x_{l+1} = x_l` for every block, which is trivially well-conditioned. To see how sharp the knife is, take the toy stack `x_L = (1+αw)^L x_0` with input-output Jacobian `(1+αw)^L`, fix a modest branch gain `w=0.5` and `L=64`, and evaluate: `α=0` gives Jacobian `1.0` exactly — the input passes through untouched — while `α=1` gives `(1.5)^64 ≈ 1.9e11`, an eleven-orders-of-magnitude blow-up. And it isn't a knife-edge at `α=0`: even `α=0.1` already gives `(1.05)^64 ≈ 22.7`. So the input signal is preserved only in a narrow neighborhood of zero, and starting *at* zero is what buys the clean conditioning. SkipInit is the same idea, a single learnable per-branch scalar init at 0, enough to train thousand-layer nets without any norm at all; and the reason being-near-identity-at-init is so valuable was spelled out by De and Smith — in a normalized residual net the variance into block `ℓ` grows like `Var(x_ℓ) ≈ Var(x_{ℓ-1}) + 1 ≈ ℓ`, so after the next residual add the new branch accounts for only a `1/(ℓ+1)` slice of the output variance at init. A healthy deep net *starts dominated by its skips*, hugging identity, and that's precisely what keeps its gradients conditioned. So "be identity at init" is a target I should engineer for on purpose.

But here's where ReZero and SkipInit leave me hanging, and it's the crux. Their `α_i` is *learned*, by gradient descent, *independently per layer*. Nothing says the shallow branches turn on before the deep ones. The optimizer is free to grow a deep `α` first if that locally lowers the loss — and given how entangled the stack is, that's not a hypothetical. Stare at the dependency structure for a second. A deep layer's input is the running output of every shallow layer beneath it; a shallow layer's gradient is back-propagated *through* every deep layer above it. So early in training, when a deep branch with freshly-random weights starts contributing, it does two bad things at once: it injects noise into the intermediate representation that all the layers above it now have to consume, and — because gradients flow back through it — it corrupts the gradient signal that the shallow layers below are using to figure out what *they* should compute. The shallow layers are trying to lay down a stable foundation, and the deep layers, firing early with garbage, are simultaneously eating that half-built foundation and poisoning the feedback it's built from. A *learned* scalar gives me no control over this; ReZero's only guarantee is "start at zero," and after step one it's the optimizer's call. The mechanism, like all the others, is fixed at initialization.

So let me name the common failure precisely, because all three families share it. Every one of these levers — depth-aware init, the constant DeepNorm `α`, the static `1/√l`, the zero-init learnable scalar — sets up the *first step* and then either freezes or hands the wheel to the optimizer. None of them is *training-phase-aware*. But I have two facts staring at me that are explicitly *about* the trajectory, not the init. One: training is staged — chaotic warmup, then stable, then decay. Two: the layers converge unevenly — shallow layers settle into their final representation earlier than deep ones; that's measured. Put those next to the dependency analysis and a different kind of fix suggests itself. The problem isn't only *how much* each branch contributes (a magnitude question, which the static factors answer) — it's *when*, and in *what order across depth*, each branch is allowed to start contributing. That's a question about the whole trajectory, and nobody's tool answers it because they all act at one instant.

What would I actually want? I want each layer to start at the identity — `α=0`, keep ReZero's good init — and then I want the contributions to *come on in order*: shallow first, deep last, with the deep ones held back until the shallow ones beneath them have had time to stabilize. And I want this to be something I *impose*, not something I hope the optimizer discovers, because the whole point is that the optimizer left to itself does *not* respect the order — that's the ReZero gap. So the scalar should not be a parameter at all. It should be a *predefined* function I set, that ramps from 0 up to 1 over the course of training, and whose ramp is *slower for deeper layers*. Let me write that as `x_{l+1} = x_l + α(l,t)·F(Norm(x_l))`, where `α(l,t)` depends on both the layer index `l` and the training step `t`, and is fixed by me, not learned. At `t=0`, `α(l,0)=0` for all `l` — exact identity, the property I wanted from ReZero. As `t→∞`, `α(l,t)→1` — the block becomes plain vanilla Pre-LN, so I lose nothing in the limit; the deep layers end up at full strength, which is exactly what the static `1/√l` fails to allow. In between, the *schedule* of how `α` climbs is where I encode "shallow first."

Now I have to pin down the shape of `α(l,t)`. Take the simplest monotone ramp from 0 to 1: linear in `t`, clipped at 1. `α(l,t) = min(t/τ_l, 1)`, where `τ_l` is the number of steps layer `l` takes to finish warming up — the step at which its scalar first reaches 1. The whole game is now in choosing the `τ_l`. I want deeper layers to take longer, so `τ_l` should increase with `l`. The cleanest choice that does that and has a single knob is to make `τ_l` linear in `l`: `τ_l = T·l`, where `T` is the warmup length of the *first* layer. Then layer 1 finishes at `T`, layer 2 at `2T`, layer `l` at `T·l`, and the deepest layer `L` finishes at `T·L`. That gives me

  α(l,t) = min( t / (T·l), 1 ),  l = 1,…,L.

Read what this does. At any fixed `t` during the ramp, the shallow layers (small `l`, small `T·l`) have already reached `α=1` while the deep layers are still partway up — the residual branches "switch on" in a wave that sweeps from shallow to deep. The total warmup, the step at which *every* layer has reached full strength, is `T·L`. After that, every `α(l,t)=1` and I'm running exactly vanilla Pre-LN — so whatever the schedule did, it did during the first `T·L` steps and then got out of the way. That's the property the static methods can't have: the constraint is real early and *gone* late.

Now read the three properties I wanted off the schedule directly, with `T=1000, L=32`. Identity at init: `α(l,0)=min(0,1)=0` for every `l`, so `x_{l+1}=x_l` exactly — ReZero's clean start for free, and the variance can't explode at init because nothing is being added to the stream yet. For the shallow-before-deep wave, count how many of the 32 layers have reached full strength (`α=1`) at a sweep of steps: at `t=500`, `0/32` (even layer 1 is only at `α=0.5`); at `t=1000`, `1/32` (layer 1 just hit 1, layer 5 at `0.20`, layer 9 at `0.11`); at `t=4000`, `4/32`; at `t=16000`, `16/32` — exactly half the stack full, the rest mid-ramp; at `t=32000`, `32/32`. The branches switch on as a front that sweeps shallow to deep, and it's a *gradual* front, not a step: at any mid-training step there's a band of partially-on layers between the saturated shallow ones and the still-near-zero deep ones. Bounded update across time *and* depth falls out of the same counts: early on only the shallow layers have nonzero `α`, so only a few layers can move the function at all, and they move it by a fraction `α<1` — the update is throttled as a function of *both* depth and training progress, not a single init-time constant, and the `0/32 → 16/32 → 32/32` progression is the constraint relaxing itself layer by layer, tight when updates are chaotic and fully released when they're stable. Read the table column-wise and it gives the ordering: a deep layer's `α` is still ~`0.02` at `t=500` while layer 1 is already at `0.5` and climbing, so the deep layer stays quiet through the shallow layers' large early updates and only comes on once they've largely settled — it builds on a stabilized input instead of amplifying early-init noise, and it isn't injecting garbage back through the gradient path while the shallow layers are still finding their footing.

And the activation explosion? I computed earlier that with all branches at full strength the stream variance climbs like `σ²_l ∝ l²` — the iteration gave `σ_l ≈ l/2` (`σ_8≈4.1`, `σ_32≈15.8`, `σ_64≈31.6`), so layer 64 sits near `1000`. LayerNorm-Scaling cures that by shrinking deep increments statically. My schedule attacks the same accumulation in time, and the recurrence I already wrote down makes that quantitative. Its per-layer factor, `(1 + 1/σ_{x_k})`, is one plus the branch-to-stream scale ratio, so scaling the branch by `α(k,t)` turns it into `(1 + α(k,t)/σ_{x_k})` — the increment the recurrence adds each layer is scaled down by `α(k,t)`. Take a step `t ≥ T` first, so `m = t/T ≥ 1` layers are fully warmed (for `t < T` every `α` is below `t/T` and the whole stack is uniformly throttled — strictly quieter than what I'm about to derive). Through the prefix `k ≤ m` the recursion is exactly vanilla, so `σ_m ≈ m/2` and the variance at depth `m` is `~m²/4`. Beyond the front the damped recurrence, in continuum form, reads `dσ²/dl = (m/l)·σ`, i.e. `dσ/dl = m/(2l)`, which integrates to `σ(l) ≈ (m/2)·(1 + ln(l/m))`: past the warmed front the stream scale grows only *logarithmically* in depth, not linearly. So the variance profile at time `t` is the vanilla quadratic out to depth `m = t/T` and essentially flat beyond it — at `t=1000` with `T=1000` (`m=1`) the top of a 64-layer stack sits near `σ² ≈ (1/4)(1 + ln 64)² ≈ 7`, against the vanilla `~1000` — and the profile steepens smoothly as the front advances, reaching the full quadratic only at `t = T·L`, exactly when the last layer finishes warming up, by which point the shallow stream the deep layers sit on has already organized. So I get the early variance control of `1/√l` without its permanent late-phase clamp, because my factor goes to 1 and the static one never does.

The "predefined, not learned" choice deserves a hard look, since ReZero's instinct was to learn it and learning is usually the safer bet. Could I just learn `α(l,t)`? No — that's precisely ReZero, and it has no mechanism to enforce shallow-before-deep; the optimizer sets each `α` to whatever minimizes the *current* loss, and there's no term in the loss that rewards "let the shallow layers settle first." The ordering is a *prior* I hold about how training should proceed, drawn from the convergence-order and dependency facts, and a prior is something I impose, not something I ask the data to rediscover. Making `α` a fixed schedule with zero learnable parameters is the point: it costs nothing, it's deterministic, and it *guarantees* the order. The remaining worry — that a hand-set shape might be wrong — is answerable only by checking whether the qualitative shape is what carries the benefit, which means putting alternative schedules next to mine.

The quickest way to see whether the specific shape is load-bearing or arbitrary is to put three schedules side by side at one step. Take `t=1000`, `L=32`, and read off `α` for layer 1 and layer 32. Shallow-first gives `(layer 1, layer 32) = (1.0, 0.031)`. The "equal" schedule `α(l,t)=min(t/T,1)` gives `(1.0, 1.0)`. The reverse schedule `τ_l = T·(L−l+1)` gives `(0.031, 1.0)` — an exact mirror image, deep on, shallow off. Now reason from those numbers. The equal schedule still delays the chaotic init updates (good — both layers are at 0 at `t=0`), but at `t=1000` it has switched on *all 32* branches at once, deep and shallow together, so I'm back to deep layers firing while shallow representations are unstable, just postponed by `T`. I'd expect this to help less than the staggered version and to be brittle: a long shared warmup means I'm scaling every residual up simultaneously after the delay, which can dump a coordinated burst of representation and gradient noise into the stack and even diverge. (As an aside, this "equal" schedule on a Post-LN block is essentially what BranchNorm does — a single global ramp — a known but weaker special case.) The reverse schedule's `(0.031, 1.0)` is the worst case made explicit: at `t=1000` the deepest, freshly-random branch is already at full strength while the bottom layer is essentially off. That actively does the harmful thing — it lets the deep, randomly-initialized branches dominate optimization early while the shallow layers are starved — which is exactly the divergence pattern Post-LN is notorious for, the last layers hogging the update while the bottom layers get vanishing gradients and never escape. A longer warmup makes *that* worse, cementing the deep-dominant regime before the shallow layers get a say. So it's the *order* — shallow before deep — doing the work, not merely "delay everything": the same total delay with the order flipped gives me the pathological configuration. The linear-in-`l` stagger is the simplest schedule that encodes the right order.

The static-vs-dynamic distinction is the whole thesis, so it's worth stating as a direct comparison. Suppose I freeze the constraint instead of ramping it — e.g. hold every layer at a fixed fraction forever, `α(l)=1/√l` (which is just LayerNorm-Scaling) or `α(l)=1/L`. Compare that to the matching schedule that *starts* at that fraction and then relaxes it up to 1 — a "stagewise" ramp `clip((t−T(l−1))/T, 0, 1)` rescaled to begin at `1/√l` and climb to 1. Both impose the same gentle constraint at init; but the static one keeps strangling the deep layers through the entire stable phase, while the relaxing one hands them back their full capacity once they're past the dangerous early regime. If my reading is right — bounded update is needed *at init*, not forever — the relaxing version should win and the static one should look needlessly conservative, especially for the architectures (like Post-LN) that most need the early protection. That's the clean falsifiable consequence of the whole training-phase-aware idea.

There's a small family of refinements on the basic linear ramp worth keeping in my pocket. The raw `α=min(t/(T l),1)` climbs at constant slope; I could make the early part gentler with `α = (min(t/(T l),1))²` — quadratic, so the branch comes on *very* softly at first, which is what I'd want for a fundamentally less-stable block like Post-LN, where easing in gently — the identity-at-init idea taken seriously — matters most — or faster with the square-root `(min(t/(T l),1))^{1/2}`. The total warmup `T·L` is the same for all three; only the curvature differs. The linear one is the natural default — it's the plainest interpolation from identity to vanilla — and I'd reach for the square only on the touchier normalizations. None of these change the parameter count: they're all just different arithmetic on `t`, `T`, `l`.

Now the one free number, `T`. It sets the timescale of the whole thing: layer 1 warms over `T` steps, the whole model over `T·L`. Two failure modes bracket it. If `T` is too small — say the warmup finishes in a few hundred steps — I've barely delayed anything past the chaotic init and I've thrown away most of the benefit; the branches snap on almost immediately. If `T` is too large — so `T·L` eats a big fraction of my total training budget — then I'm spending most of training with my deep layers artificially weakened, and they never get enough full-strength steps to actually learn; the run ends before the deep capacity is used. So `T` wants to be large enough to meaningfully outlast the warmup phase but small enough that `T·L` is a modest slice of training. With a 100k-step budget, let me just tabulate `T·L` against the depths I actually sweep (12 to 120) for `T=1000`: that's `12k, 32k, 64k, 96k, 120k` at `L = 12, 32, 64, 96, 120`. The first four all sit inside 100k — at `L=32` the `32k` total warmup is a comfortable third of the run, leaving two-thirds of training at full strength — so `T=1000` is a single value I can set once and apply across that whole range without tuning. But the last one fails the check: `1000·120 = 120{,}000 > 100{,}000`, so the deepest layers would *never* reach `α=1` before the run ends — they'd be permanently throttled, which is exactly the failure I'm trying to avoid. So the budget check redirects me at extreme depth: I have to keep `T·L ≤` total steps, and at `L=120` that means dropping `T` to `500` (`500·120 = 60{,}000`, back inside 100k). That's the one principled adjustment — not a free choice but one the budget forces.

A couple of choices I made without comment, now justified. I'm multiplying only the *branch* `F(Norm(x_l))` by `α`, and leaving the skip `x_l` at weight 1. That's deliberate: the skip is the identity highway and the gradient path; if I scaled *it* by `α` I'd be attenuating the signal and the gradient flow, and at `α=0` the network would collapse to zero output instead of to the identity. Keeping the skip at 1 and dialing the branch from 0 to 1 means `α=0` gives *exactly* identity and `α=1` gives *exactly* vanilla — the two endpoints I want — with a clean interpolation between. And this generalizes to every normalization variant the same way: wherever a block has a residual branch, I multiply that branch by `α(l,t)`. Pre-LN: `x_l + α·F(Norm(x_l))`. Post-LN: `Norm(x_l + α·F(x_l))`. Sandwich-LN: `x_l + α·Norm(F(Norm(x_l)))`. DeepNorm: `Norm(α_const·x_l + α(l,t)·F_β(x_l))` — its constant skip-scaling stays, and my time-scalar rides on the branch. LayerNorm-Scaling: `x_l + α(l,t)·F(Norm(x_l)/√l)` — its `1/√l` and my ramp compose. The principle — schedule the branch contribution — is architecture-agnostic, which is why I expect it to help across all of them, with the post-norm-flavored ones (Post-LN especially) benefiting most because they're the ones whose deep layers normally dominate and destabilize, exactly what the shallow-first ordering corrects.

Let me now turn this into code I'd actually run, because the implementation has one wrinkle worth getting right. The scalar `α(l,t)` depends on the global training step `t`, so I need to know the step before the residual write. The cleanest single-stream version carries the step as a non-trainable buffer on the model and computes the per-layer `α` right in the loop that runs the blocks. There's a second, smaller wrinkle: I want `x ← x + α·(branch contribution)`, but in the nanoGPT-style harness the block already folds the skip in and returns `x + branch`, i.e. `block_out = x + delta` where `delta` is the sublayer contribution. So rather than reach inside the block to multiply only the branch, I recover the contribution as `delta = block_out − x` and write `x ← x + α·(block_out − x)`. Expanding it to make sure the skip isn't accidentally scaled: substituting `block_out = x + delta` gives `x_new = x + α·((x + delta) − x) = x + α·delta` — the coefficient on the skip `x` is exactly `1`, the coefficient on the branch `delta` is exactly `α`. The endpoints confirm it: `α=0` gives `x_new = x` (pure identity), `α=1` gives `x_new = x + delta` (exact vanilla Pre-LN), so the one-liner is a faithful interpolation with no attenuation of the highway. I read the current step before the block loop, so the first training forward is truly `t=0` and every branch is off; then I advance the buffer after this forward has used that step. For each block `i` (1-indexed layer `l = i+1`) I set `α = min(step/(T·l), 1)`; once a layer's `α` has reached 1 I just take `block_out` directly, which is identical to vanilla. The whole change, as it sits in the forward loop:

```python
step = self._warmup_step.item()      # non-trainable buffer, read before the loop: first training forward is t=0
for i, block in enumerate(self.transformer.h):
    block_out = block(x)             # = x + delta (Pre-LN residual)
    if self.training and step < T * (i + 1):          # layer l = i+1 still warming up
        x = x + min(step / (T * (i + 1)), 1.0) * (block_out - x)   # scale only the branch
    else:
        x = block_out                # alpha == 1: exact vanilla Pre-LN
# ... final norm, head, loss all unchanged ...
if self.training:
    self._warmup_step += 1           # advance after this forward has used the step
```

Two extra lines of arithmetic, one integer buffer for the step, zero learnable parameters, and the optimizer is completely untouched.
