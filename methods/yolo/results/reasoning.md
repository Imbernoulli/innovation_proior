I start with the shape of the detection problem, because the awkward part is not the definition. Given an image, I want every object, each with a class and a tight box. The awkward part is that the best systems around me do not look like that simple mapping. They look like pipelines.

DPM is a sliding-window model over a HOG pyramid: a root filter, part filters, deformation costs, then box prediction and non-max suppression. It has a real spatial model, but the features, classifier, geometry correction, and post-processing are separate pieces. R-CNN replaces exhaustive sliding windows with Selective Search proposals, then warps each proposal, runs a CNN, scores with per-class SVMs, adjusts boxes, and suppresses duplicates. Fast R-CNN shares the convolutional pass and trains the proposal classifier and box regressor together, but the proposals still come from a separate algorithm. Faster R-CNN moves proposal generation into a neural network, but the design is still propose first, classify and refine second.

Two pressures keep showing up. One is speed. If I have to generate thousands of regions or classify each crop separately, real-time video is out of reach. Sharing convolution helps, but a proposal stage and a second per-region decision stage still impose latency. The other pressure is trainability. A pipeline optimizes proposal recall, classification accuracy, box regression, and duplicate suppression as separate objectives. I cannot simply backpropagate through the whole detector as one function of the image and the final detection targets.

There is also a structural context problem. A region classifier sees a cropped patch. If the patch locally resembles an object, the classifier has little access to the surrounding scene that might reveal it as background. This is not a claim that any particular model must fail on every such crop; it is a consequence of deciding locally after proposals have already been cut out. If every box prediction is instead made from features of the full image, the score can depend on both local appearance and global context.

So what I want is one network, one image-sized forward pass, and one detection objective. The moment I write that down, I hit a wall: a detector must output a variable-size set, one entry per object, while a neural network head naturally emits a fixed-size tensor. The obvious patch is to allocate a fixed list of, say, 98 output boxes and let the network fill them in. But then I owe an answer to "which output slot owns which ground-truth object?", and that answer is not free. If the slots are interchangeable, two of them can chase the same central object while a second object goes unclaimed, and the optimizer sees no reason to break the tie one way rather than the other. I want to convince myself this is a real problem and not a worry I can wave away.

Suppose I assign each ground-truth object to whichever slot currently predicts the closest box, recomputed every step (the natural "let the network sort it out" rule). Consider two nearly identical training images, each with one object near the image center. In the first image slot 5 happens to be marginally closer and gets the gradient; in the second, slot 6 is marginally closer. Now both slot 5 and slot 6 are being pulled toward center-object configurations, on different images, with nothing telling them apart. Over a batch the assignment can flip slot-by-slot, and the coordinate gradient any single slot receives is an average over whichever objects happened to select it. That is exactly the symmetry I feared: the slots never differentiate, because their only distinguishing feature is a tie-break that flips. A fixed output size alone does not solve the assignment problem; it just hides it. I need the slot-to-object map to be determined by something stable about the image, not by the network's own current guesses.

The most stable thing I have is location. If I divide the image into an `S x S` grid and declare that the cell containing an object's center is responsible for that object, then the slot-to-object map is fixed by the data, not by the network. A cell owns only the objects whose centers fall inside it. The same two-image thought experiment now resolves: a center-object lands in the same center cell both times, so the same output location is trained on it consistently. As a bonus, because each object has a single intended owner cell, two distant cells are not encouraged to both report it, which suppresses duplicates before any post-processing even runs.

That still leaves what happens inside one cell. If a cell predicts a single box, then across the training set that one position is asked to represent a tall standing person on one image, a wide car on another, a squat object on a third. A single regressor minimizing squared error will settle near the average of those shapes and fit none of them. So I let a cell predict `B` boxes, hoping the predictors can specialize by shape. How many? I want the output small for speed, so I want the smallest `B` that still lets predictors separate. `B=1` cannot separate anything. `B=2` is the smallest value that can, and for VOC, where same-cell objects are usually one dominant object plus the occasional second, two hypotheses is a plausible match. I will take `B=2` provisionally and let the loss tell me whether two predictors actually differentiate.

For two predictors to specialize rather than both drift to the average, only one of them can be trained on a given object. So within a cell that contains an object, I pick the box predictor whose current prediction best matches the ground truth, measured by IOU, and give only that predictor the coordinate gradient for that object. This is self-reinforcing in the direction I want: if predictor A is already slightly better on tall boxes, it gets selected on tall objects, improves on them, and stays the tall specialist; predictor B is then free to claim the wide ones. I should check the degenerate start, though. Early in training every predicted box may have zero overlap with the truth, so IOU gives me no signal to pick a responsible box and the rule is undefined. I need a deterministic fallback so that exactly one predictor is responsible even then; the smallest coordinate RMSE to the truth is a tie-break that is always defined and still prefers the closer box.

Now the confidence channel. I want it to mean more than "something is in this cell." A score I will threshold at test time should be high only when a box both contains an object and localizes it well. So I define the confidence target as `Pr(Object) * IOU(pred, truth)`. Read off the consequences: in an empty cell `Pr(Object)=0`, so the target is `0` for every box there; in an object cell the responsible box should predict its actual current IOU with the truth, not a flat `1`. A loosely placed box around a real object then correctly reports low confidence, because it is not yet a good detection. The target is not a constant label but a moving quantity the predictor is chasing as its own boxes improve.

Should each of the `B` boxes carry its own class vector? The two boxes in a cell are competing geometric hypotheses for the same cell-owned object, not two independent objects, so duplicating the class distribution per box would be redundant and would also dilute the class gradient across boxes. I predict one shared set of conditional class probabilities per cell, `Pr(Class_i | Object)`. The reason this is worth doing is what it buys at test time. The cell gives me `Pr(Class_i | Object)`; each box gives me a confidence which is `Pr(Object) * IOU`. Multiply them and I want to land on something interpretable. Let me actually do the algebra on a concrete cell: say `Pr(Object)=0.8`, `IOU=0.6`, and `Pr(Class_i | Object)=0.7`. Then confidence is `0.8 * 0.6 = 0.48`, and the class-specific score is `0.7 * 0.48 = 0.336`. Now compare with what I would call the "honest" target, `Pr(Class_i) * IOU`: since presence of the class implies an object is present, `Pr(Class_i) = Pr(Class_i | Object) * Pr(Object) = 0.7 * 0.8 = 0.56`, and `0.56 * 0.6 = 0.336`. The two agree exactly. So the per-cell-class, per-box-confidence factorization is not just a storage saving; the test-time product `Pr(Class_i | Object) * confidence` literally equals `Pr(Class_i) * IOU`, a class-specific score that already folds in both label belief and localization quality. Good — that is the number I want to threshold.

With `S=7`, `B=2`, `C=20` for VOC, the fixed output is `S x S x (B*5 + C) = 7 x 7 x 30 = 1470` values, and the number of candidate boxes the network ever proposes is `S x S x B = 7 x 7 x 2 = 98`. I want to register how small that is against the baselines: Selective Search hands R-CNN roughly 2000 regions per image, each pushed through the CNN; here the entire image yields 98 boxes from one forward pass. That two-orders-of-magnitude compression in the number of evaluated boxes is the structural reason this design has a chance at video frame rates, where the proposal-plus-per-region pipelines do not.

For the backbone I want a strong classifier feature extractor, but detection data is too scarce to learn good general features from scratch. I pretrain the convolutional layers on ImageNet at `224 x 224`, then switch to `448 x 448` for detection, because localizing boxes needs finer spatial resolution than naming a whole image does. GoogLeNet suggests the useful compute-saving pattern: put `1 x 1` channel reductions before the expensive `3 x 3` convolutions. I do not need the full inception branching; a straight stack of `1 x 1` and `3 x 3` convolutional layers, then two fully connected layers, can emit the flat `1470`-vector. Hidden layers use leaky ReLU with slope `0.1`; the final layer is linear, because the outputs include unbounded coordinates and I do not want to squash them.

Now the loss. The path of least resistance is plain sum-squared error over the whole `1470`-vector, which is trivially differentiable. But I should look at what it actually does before adopting it, because it is badly balanced in three ways. First, it weights a coordinate error and a class-probability error identically, even though a detector lives or dies by its boxes. Second, and worse, it is dominated by empty cells. Let me count on a small `2 x 2` grid with `B=2` and a single object in one cell: there are `4 x 2 = 8` boxes in total, of which exactly one is the responsible box (target confidence = its IOU) and the other `7` are no-object boxes (target confidence = 0). So seven of eight confidence terms pull toward zero. On the real `7 x 7` grid the imbalance is far steeper. If every no-object confidence term pushes at full strength, the network's easiest minimizer is to predict near-zero confidence everywhere, which swamps the rare positive signal and collapses recall.

I counter both with two scalar weights. `lambda_coord = 5` multiplies the coordinate terms so localization carries a larger gradient than classification. `lambda_noobj = 0.5` multiplies the confidence terms of boxes that are not responsible for an object, so the flood of empty-cell confidence terms cannot dominate the gradient. The exact case split matters, and the `2 x 2` count above tells me precisely what it must be. In an empty cell, both boxes are no-object terms; no coordinate or class loss applies. In the object cell, the responsible box gets coordinate loss and object-confidence loss (target = its IOU), and class loss is applied once for the cell, since the class distribution is conditional on an object being present there. The subtle case is the *other*, unused box in the object cell: in my `2 x 2` trace it is one of the seven no-object terms, not zero of them. So the implementation cannot simply key the no-object loss on "empty cell" — it must key it on "not the responsible box," which is `1 - responsible` over all boxes. If I keyed it on emptiness instead, the unused box in every object cell would receive no confidence gradient at all and would be free to emit spurious high confidence at test time. The indicator I need is therefore: `1_i^obj` for cells with an object, `1_ij^obj` for the single responsible predictor, and `1_ij^noobj = 1 - 1_ij^obj` for every other predictor, which by construction partitions all boxes into responsible-or-not with nothing double-counted or dropped.

There is a third squared-error mismatch, on the box sizes specifically. The same absolute width error should be more damaging on a small object than on a large one — a 5%-of-image width error can wreck a bird's IOU while barely moving a bus's — but raw squared error does not see that. I can check it: with target width `w*` and a fixed absolute error `d`, the raw penalty is `(w* - (w*+d))^2 = d^2`, which is constant in `w*` and so penalizes the bird and the bus identically. I want a parameterization where the same `d` costs more for small `w*`. Predicting `sqrt(w)` instead of `w` does this, because near a small `w*` the square-root function is steeper. Let me put numbers on it with `d = 0.05`: at `w* = 0.05` the penalty `(sqrt(0.05) - sqrt(0.10))^2 ≈ 0.00858`; at `w* = 0.9` it is `(sqrt(0.9) - sqrt(0.95))^2 ≈ 0.00068`. So the same 5% error costs about `12.7x` more on the small box than on the large one under the square-root parameterization, versus a flat `1x` ratio under raw widths. That is the asymmetry I wanted. So the network predicts `sqrt(w)` and `sqrt(h)`; at decode time I square those channels to recover `w, h`, and in the coordinate loss the targets are `sqrt(w*)` and `sqrt(h*)`. It is a cheap fix and it does not require changing the loss form, only the units of two channels.

Assembling the objective from these indicators is now mechanical. Under `1_ij^obj`: a `lambda_coord`-weighted squared error on `(x, y)` and on `(sqrt(w), sqrt(h))`, plus a squared error on confidence with target equal to the responsible box's IOU. Under `1_ij^noobj`: a `lambda_noobj`-weighted squared error driving confidence to `0`. Under `1_i^obj`: a squared error on the cell's `C` class probabilities. Nothing else contributes. The implementation has to honor exactly the three subtleties I derived: no-object loss keyed on `1 - responsible` (not on emptiness), responsible-box confidence target equal to its IOU (not `1`), and box width/height channels carried in square-root units (not square-rooted after the fact inside the loss).

Inference stays deliberately small, which is the whole point. One forward pass produces the `7 x 7 x 30` tensor. For each cell and box I decode the center by adding the cell's grid offset and dividing by `S`, square the predicted root-width and root-height to get `w, h`, and multiply the box confidence by the cell's class probabilities to get the `Pr(Class_i) * IOU` scores I verified above. I threshold those class-specific scores and run a light NMS pass over what survives. Because each object has a unique owner cell, the grid has already done most of the de-duplication; NMS is cleanup for the cases the grid does not cover — large objects or ones straddling a cell boundary, which two adjacent cells may both fire on.

The same choices that buy the speed impose the limits, and I should be honest about them rather than discover them later. Each cell has only `B=2` boxes and one shared class distribution, so if several small objects fall with their centers in one cell, the cell physically cannot represent them all — a flock of birds is a structural failure mode, not a tuning issue. The final feature grid is coarse, so small objects and unusual aspect ratios are hard. And the square-root size trick, which I checked does rebalance the loss toward small boxes, still leaves the objective as squared error, which is only a proxy for IOU and for average precision — minimizing it is not the same as maximizing the metric I am actually scored on. These are the costs of compressing detection into a single fixed tensor from one full-image pass; I am accepting them in exchange for the 98-box, one-forward-pass speed.
