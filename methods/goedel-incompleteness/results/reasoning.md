I'm trying to settle Hilbert's question in its sharpest form: is there a formal theory T of arithmetic — fixed alphabet, a mechanically recognizable set of axioms, finitely many rules of inference — that derives every arithmetical truth of the intended domain, and whose own consistency can be certified by finitary means from inside mathematics itself? The natural first attempt at making "certified from inside" precise is to reduce the consistency of a stronger theory, analysis, down to the consistency of arithmetic, and the natural way to run that reduction is to let arithmetic talk about which of its own sentences are true.

So suppose there is an arithmetical formula True(x), meant to hold of exactly the codes of true sentences, carrying the property that has to hold if the name is earned: for every sentence phi, True(code(phi)) holds exactly when phi does. Suppose too that I have some means of manufacturing self-referential sentences — I still owe myself that machinery, but grant it for a moment. Then nothing stops me from building a sentence L with L holding exactly when True(code(L)) fails: a formalized liar sentence. Then True(code(L)) holds exactly when L does, which holds exactly when True(code(L)) fails — True(code(L)) is equivalent to its own negation, an outright logical contradiction, not merely an unresolved sentence. So no arithmetical formula can carry that disquotational property; truth-in-the-intended-model cannot be captured by a predicate of arithmetic at all. That rules out the truth-based reduction entirely — not because it is technically inconvenient, but because assuming the needed predicate exists collides head-on with the very self-reference I clearly do have available.

This pushes me toward a different predicate, one that is syntactic rather than semantic: instead of asking whether a sentence is true, ask whether it has a proof. Provability, Prov_T(x), reports only that some finite object — a derivation — exists; it makes no claim about the sentence's truth value. That difference matters structurally, not just stylistically. "True" and "provable" are not the same concept, so a sentence asserting its own unprovability is not thereby asserting its own falsity, the way the liar sentence asserted its own falsity. If the same fixed-point trick that just wrecked True(x) produces a sentence G_T with G_T holding exactly when Prov_T(code(G_T)) fails, that biconditional is not of the collapsing shape "G_T iff not G_T" — it is "G_T iff no proof of G_T exists," which a consistent theory and its intended model can both survive without contradiction. Provability, unlike truth, is safe to self-apply. It is also the more honest choice on independent grounds: whether a finite string is a valid derivation is a mechanically checkable, contentless fact, whereas "true" imports semantic commitments I would rather not need. Both reasons point the same way, so I build the whole argument on Prov_T and never introduce True(x) at all.

Making Prov_T concrete is bookkeeping, but load-bearing bookkeeping. From outside the system, a T-proof is a finite sequence of formulas in which every line is an axiom or follows from earlier lines by one of finitely many rules; because the axioms are effectively recognizable, checking whether a candidate sequence is a valid proof is itself a finite mechanical procedure. Assign numbers to symbols, and by an effective sequence coding (prime-exponent coding will do) assign numbers to finite strings and to finite sequences of strings, so that formulas and proofs become numbers. The syntactic predicates "n codes a formula" and "p codes a valid T-proof of the formula with code n" become primitive recursive relations on numbers, and because a theory that can express elementary arithmetic can represent every primitive recursive relation by a formula, there is an arithmetical Proof_T(p, n) that mirrors the proof relation exactly. Provability is then Prov_T(n) := there exists p such that Proof_T(p, n). Nothing here talks about truth; it is a syntactic proxy, checkable in principle line by line, which is exactly why it was safe to self-apply above.

Now the real construction problem: I want a sentence G_T that holds exactly when Prov_T(code(G_T)) fails — a sentence about its own unprovability. Writing this down by fiat is circular: to state "the formula whose code equals my own code" I would need to already know my own code before I have finished specifying myself. I need an indirect route, and the shape I want is the one Cantor already showed me: index a family of objects by the very set it ranges over, and land on a member defined in terms of its own index — except where Cantor's diagonal entry is built to differ from every indexed member, mine has to be built to coincide with itself.

The lever is substitution. Given the code e of a formula phi(x) with one free variable and a number m, there is a completely mechanical operation — plug the numeral for m into the one free slot of phi — that produces the code of the resulting sentence phi(m). Call this operation diag; it is an effective numerical function on codes, and since the theory represents every effectively computable function, there is an arithmetical Sub(e, m, z) that holds exactly when z is the code of the result of substituting the numeral for m into the formula coded by e.

Build an auxiliary formula that is not yet self-referential: B(x) := there exists y such that Sub(x, x, y) and not Prov_T(y). Read informally, B(x) says "the formula you get by substituting x's own code into the formula x codes for is unprovable" — a claim about an arbitrary code x, nothing circular about it yet.

Now apply B to its own code. Let b = code(B), and set G_T := B(b), substituting the numeral for b into B's free slot. Unwind what this says: there exists y such that Sub(b, b, y) and not Prov_T(y). But Sub(b, b, y) holds exactly when y = diag(b), and diag(b) — substitute b's own code into the formula B that b codes for — is, by construction, exactly the code of B(b), which is code(G_T). So the only y satisfying Sub(b, b, y) is code(G_T), and the existential collapses: G_T is equivalent to not Prov_T(code(G_T)). I did not write "the sentence asserting its own unprovability" directly; I built a formula about substituting a code into itself, then fed that formula its own code, and the self-reference fell out because diag(b) landed on G_T's own code. This is the diagonal pattern doing real work, re-engineered rather than merely reused: the indexed family is one-free-variable formulas indexed by their own Godel numbers, and the "diagonal entry" is not the one that differs from every indexed member but the one produced by substituting an index into its own formula.

Suppose T proves G_T. Then because Proof_T mirrors actual T-proofs, T also proves Prov_T(code(G_T)) — T can certify that a proof of G_T exists. But T proving G_T means T proves not Prov_T(code(G_T)) as well, since that is what G_T says. So T proves a sentence and its negation: T is inconsistent. Contrapositive: if T is consistent, T does not prove G_T. And since G_T literally asserts that no number codes a T-proof of G_T, and indeed no such proof exists, G_T is true of the actual natural numbers. So there is a true arithmetical sentence T cannot prove: T is incomplete.

I would like consistency alone to also rule out T proving not-G_T, so that G_T is genuinely undecidable rather than merely unprovable, but the direct argument I have needs more. If T proves not-G_T, i.e. T proves there exists p with Proof_T(p, code(G_T)), I want to say that no actual number witnesses this, since G_T has no real proof, and that a theory asserting an existential for which every specific instance is refutable is defective. That is exactly what omega-consistency buys me: if T proves not Proof_T(n, code(G_T)) for every specific numeral n, an omega-consistent T cannot also prove the existential there exists p Proof_T(p, code(G_T)). Nothing weaker in my hands gets me there. It is an unsatisfying reliance — omega-consistency is a strange extra hypothesis to need — but I do not yet see how to avoid it, so I state the full undecidability result under that extra assumption, and the bare non-provability of G_T under consistency alone.

The same machinery formulates the theory's own consistency: Con(T) := not Prov_T(code of a fixed contradiction), read as "no number codes a T-proof of falsehood." I would like T to prove Con(T) implies G_T, which would let me read off a second theorem directly from the first. Getting that implication requires reproducing the first theorem's argument inside T rather than just reasoning about T from outside, and that in turn requires Prov_T to behave, provably in T, the way a provability predicate should: whenever T actually proves a sentence phi, T proves Prov_T(code(phi)); T proves that Prov_T distributes over modus ponens; and, least obvious but true of the Prov_T I built, T proves Prov_T(code(phi)) implies Prov_T(code(Prov_T(code(phi)))) — the theory can certify its own certifications, because Proof_T is itself a primitive recursive predicate and T proves enough about its own primitive recursive functions to track that. Granting these, T proves Con(T) implies G_T. If T also proved Con(T), T would prove G_T, contradicting the first theorem for a consistent T. So a consistent, sufficiently arithmetical T cannot prove its own standard consistency statement. That is the sharper blow to Hilbert's program: not merely incompleteness somewhere in arithmetic, but the specific finitary-looking statement Hilbert most wanted proved, Con(T), turns out to be exactly the kind T cannot reach.

Looking back at where this leaves the three things I started with: the liar paradox gave me self-reference, but at the price of an outright contradiction, useless for a theorem about a consistent theory — that is precisely why I had to swap truth for provability, the one predicate whose self-application yields an independent sentence instead of a collapse. The outside, Hilbert-style consistency proof gave me a way to reason about T's syntax, but only from a vantage point external to T; arithmetization is what moves that reasoning inside T's own language, so a statement about proofs becomes a statement about numbers T itself can quantify over. And Cantor's diagonal gave me the pattern — index a family by its own index set, hit a self-referential member — but the pattern had to be re-engineered around substitution, because I need a formula that ends up equal to the entry generated from its own index, not one that differs from every entry. None of the three, on its own, gets me a theorem; stitched together — syntactic provability standing in for truth, arithmetization moving proof theory inside arithmetic, substitution replaying the diagonal pattern in self-referential form — they produce a concrete undecidable arithmetical sentence and, from it, the impossibility of a sufficiently strong consistent theory proving its own consistency.
