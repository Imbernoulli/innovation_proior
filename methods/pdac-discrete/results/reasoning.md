I start from the constraint that the editable method is only a local fill inside an already token-based VLA trunk, and the trunk itself is not a blank backbone. It is the released OpenVLA-7B checkpoint, and that checkpoint was already fine-tuned at large scale on Open-X precisely by predicting these same 256 action-bin tokens, so its embedding and output rows for that tail block of the vocabulary are not random weights waiting to be trained — they already encode a calibrated notion of what each bin means, learned from far more data than this LIBERO-Goal run will ever see. If I add a separate regression or diffusion decoder on top of the action-slot hidden states, I can certainly emit continuous actions, and the trunk's visual and language grounding still helps me get there, but the decoder head itself starts from nothing: fresh parameters, either a custom loss or an inference-time denoising loop, and no inherited notion of action calibration, all of which have to be learned inside a single seed and a limited step budget. The token route pays none of that cost. It reuses whatever the released checkpoint already learned for its own action vocabulary, so only the LoRA adapters have to move, not a freshly initialized head. So the cleanest path is to make each action coordinate something the trunk can already predict: a token.

That means I need to quantize the normalized action, but I need to quantize it in exactly the same way as the existing OpenVLA tokenizer. The physical action statistics live outside this local method. Each dataset dimension is normalized through the stored quantile statistics into `[-1, 1]`; the local tokenization then clips to that interval. With `N = 256`, the code constructs `edges = np.linspace(-1, 1, N)` and `centers = (edges[:-1] + edges[1:]) / 2`. This is a slightly treacherous convention: there are 256 edge values but only 255 center values. The symbol count is still 256 because `np.digitize` can return an endpoint index as well as the interval indices.

Let me check the encode cases rather than hand-wave them. After clipping, the minimum value `-1` digitizes to `d = 1`, and the maximum value `+1` digitizes to `d = N`. Interior values produce indices between those endpoints. The tokenizer maps the digitized index to a token by `token_id = V - d`, where `V` is the effective vocabulary size used by the tokenizer or action model. So the minimum endpoint maps to `V - 1`, the maximum endpoint maps to `V - N`, and every emitted action token lies in the half-open block `[V - N, V)`. The `action_token_begin_idx = V - (N + 1)` value in the tokenizer is only a threshold for masks: training code uses `token_id > action_token_begin_idx`, which includes `[V - N, V)` and does not include the extra token `V - (N + 1)`.

Now I invert it. If a predicted token id is `t`, the digitized value is `d = V - t`; that is the same subtraction run backwards. To index centers I need zero-based interval indices, so I compute `d - 1`. The lower endpoint case is fine: `t = V - 1` gives `d = 1` and center index `0`. The upper endpoint case would be out of range: `t = V - N` gives `d = N` and center index `N - 1`, but the last valid center index is `N - 2`. The canonical decode therefore clips `d - 1` into `[0, centers.shape[0] - 1]`, which is `[0, N - 2]`. This means the exact-max token and the last ordinary interval token both reconstruct to the last center. That is not a new design choice in the scaffold; it is the released tokenizer's edge-case handling.

The next issue is where the argmax is allowed to look. The action slots produce logits over the whole language vocabulary, but most of those tokens are not valid action symbols. If I argmax over all logits, a normal language token can win and then the de-tokenizer will treat an arbitrary id as an action bin. The RT-2/OpenVLA action-token convention says the valid action ids are the least-used tail tokens, so at inference I slice the logits to `[V - N, V)`, take `argmax` inside that slice, and add the start offset back to recover the real token id. Greedy decode is enough here because the runtime wants one deterministic normalized action chunk, not a sampled language continuation.

Training then becomes the native discrete objective. I pass zero action-slot features into the PD&AC runtime because this method does not condition the decoder on noisy actions, previous predicted continuous actions, or any learned action-side inputs. The trunk sees the prompt, images, optional state, and empty action slots; the runtime's discrete helper computes next-token cross-entropy on the action labels and ignores prompt/non-action positions. The method itself owns no parameters, so all learning happens through the LoRA-adapted trunk and the existing token embedding/output weights.

I also need to keep the chunking semantics straight. The original OpenVLA action string is seven tokens for one timestep. The PD&AC scaffold lays out `K * 7` action slots for `K = NUM_ACTIONS_CHUNK`, and the same per-slot token arithmetic applies independently at every slot. I take the action-logit slice from `num_prompt_tokens` through `num_prompt_tokens + K * 7`, restrict each slot to the valid token block, invert token ids to bin indices, look up centers, and reshape to `(batch, K, 7)`. The method stops at normalized actions; the runtime owns the quantile-based un-normalization into physical robot units.

This gives me the actual fill I want: no new learned head, no custom loss, no off-by-one drift from the tokenizer, no full-vocabulary action decode, and no leakage of un-normalization or benchmark bookkeeping into the editable method. The tradeoff is real: cross-entropy treats neighboring bins as unrelated classes and the code's edge convention gives only 255 distinct centers for 256 token ids. But those are the tradeoffs of the canonical discrete action-token path, and preserving them exactly is the point of this scaffold fill.
