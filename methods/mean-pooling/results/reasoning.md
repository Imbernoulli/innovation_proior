Let me start from the thing that's actually in my hands. The encoder hands me, for each item, not one vector but a whole set of them: a sentence of n tokens comes out as `{h_1, ..., h_n}`, each `h_i` a D-dimensional contextual vector; a spatial location with V meteorological variables comes out as V vectors, one per variable. And the next thing in the pipeline — a cosine-similarity comparison, a linear classifier, the next backbone block — wants exactly one D-vector per item. So I have to manufacture a function `g({h_1, ..., h_n}) -> R^D` that collapses the set to a single vector. That's the whole job. The question is what `g` should be, and I want to derive it rather than guess, because every reduction I've seen people use is justified by a sentence of intuition and then it turns out to matter a lot which one you pick.

What constraints does the set itself force on `g`? The members have no canonical order at the point where I pool them. The tokens of a sentence do come in a reading order, but by the time the transformer has run, each `h_i` already carries context from the whole sequence through self-attention; the *pooling* step is just looking at a bag of vectors, and there's no reason the bag's arrangement should change the summary. For the variables at a location it's even starker — there's no privileged ordering of temperature, pressure, humidity that the aggregation should exploit. So I need `g` to be invariant to permuting its inputs: shuffle the members, get the same vector. The second constraint is that n varies — sentence to sentence, and across train/test — so `g` has to eat any set size, including ones it never saw. The third is subtler and I'll come back to it hard: the raw output shouldn't be dominated by how many members there are, because a fixed linear head, a dot product, or a Euclidean comparison will see vector scale, and if a 50-member set and a 5-member set land at wildly different magnitudes, length will masquerade as content. And the fourth: I want the cleanest possible default, ideally with no learnable parameters, so it's the transparent baseline that anything fancier has to beat.

Permutation invariance is the sharp one, so let me lean on it. If `g({h_i})` must be unchanged under every reordering of the `h_i`, then `g` cannot reference any member by its position — it can only see the members as an unordered multiset. The functions that do that are the *symmetric* ones: reductions that combine the members through an operation that doesn't care about order. The order-independent combiners I actually have are the elementwise associative-commutative ones — sum, and elementwise max (and min) — and anything built from them, like the average. Concatenating the members would be order-dependent *and* size-dependent, so that's out immediately: concat of n vectors has dimension nD, which isn't even a fixed size, and it bakes in an order. If I feed an orderless variable set through a recurrent reader and take the last state, I have simply chosen an artificial order and made the answer depend on that choice. So I'm down to symmetric elementwise reductions: sum, max, mean, or some weighting of them.

Start with the most obvious candidate, the one a single special token gives you. The pretrained transformer comes with a designated classification slot whose final hidden vector is *called* the aggregate sequence representation — the architecture is built so self-attention can funnel a summary of the whole sequence into that one position. Tempting: it's already there, it's already one vector, zero work. But two things stop me. First, it's a bottleneck — one vector is being asked to carry the entire set, and there's no guarantee self-attention actually packed everything useful into it rather than whatever the training objective rewarded. And that's the second thing, the decisive one: that slot was trained on the pretraining objective — next-sentence prediction — not on "be a good summary for cosine similarity." The people who built it say it outright, that this vector is not a meaningful sentence representation without fine-tuning, since it was trained for that other task. So off the shelf, reading off the special token gives me a vector shaped by the wrong objective. And worse for my setting: a per-variable tokenizer at a spatial location doesn't even *have* a designated summary token — there are just V variable vectors, none of them privileged. So pick-one-token fails both on quality (trained for the wrong thing) and on availability (no such token in the general set-pooling case). I want a reduction that uses *all* the members and presupposes no special slot.

So back to the symmetric reductions over all members. Take sum first, `g = Σ_i h_i`, the simplest order-free combination. It's permutation-invariant, eats any size, no parameters. But now the third constraint bites. Let each coordinate of a member look like a draw with mean μ and variance σ², and use the roughly-independent case as the scale calculation. The sum has expectation `nμ`, so whenever μ is not zero its location moves linearly with the number of members. Even in the centered part, the variance of the sum is `nσ²`, so the standard deviation grows like `√n · σ`. A 5-member set and a 50-member set therefore sit at systematically different raw scales just because their counts differ. A pure cosine comparison after L2 normalization can hide a global scalar, because `Σ_i h_i` and `(1/n)Σ_i h_i` point in the same direction for a fixed item, but a fixed linear head, a dot product, a Euclidean distance, or any consumer of the unnormalized norm will read length as content. Sum lets set size leak straight into the representation. Wall.

To remove that count leak without adding parameters, I can ask what exponent on n would make the pooled location stop moving with set size. If I scale the sum as `Σ_i h_i / n^α`, the expectation is `n^(1-α) μ`; for any nonzero μ, the only way to make that independent of n is `α = 1`. That gives `g = (1/n) Σ_i h_i`. Now the expected location is `E[g] = (1/n) · nμ = μ`, independent of n, and the per-coordinate variance is `Var[g] = (1/n²) · nσ² = σ²/n` when the members are roughly independent. So the pooled vector estimates the same per-member center no matter how many members the set has, and larger sets simply estimate that center more tightly. This is the mean. There's a second, cleaner way to see why it is the right parameter-free combiner and not just a convenient one. Any weighted average `Σ_i w_i h_i` with `Σ_i w_i = 1` is size-stable; among all such weightings, which do I pick when I have no reason to prefer any member over another? The weighting that treats every member identically — `w_i = 1/n` for all i — is the only fixed permutation-symmetric choice, and it's the maximum-entropy weighting (flattest possible) given no information distinguishing the members. So the uniform mean isn't a lazy default; it's the forced default the moment I refuse to use parameters to score the members. Anything else is asserting a preference I don't have.

That observation also tells me exactly how the mean relates to the fancier thing I could build. The expressive reduction is a content-weighted average: let a learned query score each member and softmax the scores into weights, `u = Σ_i α_i h_i`, `α_i = softmax(score(h_i))` — attention pooling, the self-attentive sentence encoder. It's strictly more expressive, it can learn to downweight junk members and emphasize informative ones. But it introduces learnable parameters (the query, the scoring map), so it's no longer the parameter-free default; it can overfit when data is scarce; and it gives up the transparency of a fixed rule. Crucially, the mean is its *zero-information limit*: set the query to give equal scores to everything (a zero query, a uniform softmax) and `α_i = 1/n`, and attention pooling *is* the mean. So the mean is the honest floor — the value attention collapses to when it has learned nothing — which is exactly what makes it the baseline that any attentive aggregator must clear to justify its parameters.

Now the other symmetric reduction that people actually use and that beats the mean in some settings: elementwise max, `g({h_i})_d = max_i h_{i,d}`, take the strongest activation per dimension across the set. I shouldn't dismiss it, because for a BiLSTM sentence encoder max-pooling over the hidden states reportedly worked *better* than mean. Why would it? Max keeps, per feature, the single most extreme member and ignores the rest — so if a feature's signal is concentrated in one salient member (a keyword, an outlier point in a cloud), max grabs it cleanly while the mean would dilute it by averaging in n−1 uninformative members. So max is the right reduction when the answer is driven by an *extreme* member. But that's also its failure mode: when the set's information is spread across many comparably-informative members — which is the regime I expect for aggregating, say, 48 meteorological variables that all carry real signal, or for a sentence where meaning is distributed — max throws away everything but the per-dimension winner, and it's non-smooth: per dimension the gradient flows to exactly one member, so most of the set gets no learning signal on that coordinate. The mean uses every member equally and is smooth in all of them. So the choice between mean and max is a bias-about-the-set decision: concentrated-in-an-extreme → max; spread-across-the-members → mean. For comparably-informative members the mean is the safer default, and that's the regime of the variable-aggregation problem, so mean it is — with max kept on the shelf as the alternative for extreme-driven sets.

Let me push on the size-stability one more level. The mean's expectation is size-independent, but its spread is not: `Var[(1/n)Σ h_i] = σ²/n` per coordinate, so its standard deviation shrinks like `σ/√n`. That's usually a feature (more members → a more stable estimate of μ), but it has a flip side: for very large sets the deviations that distinguish one large set from another can become small, while small sets keep a much noisier spread. So if the object I want is not the per-member center but an aggregate signal with comparable fluctuation scale across set sizes, dividing by n over-shrinks the large ones. The same exponent check tells me where the intermediate normalizer lives. The sum has standard deviation `√n · σ`; after scaling by `n^α`, the standard deviation is `n^(1/2-α) σ`. To make that spread constant in n, I need `α = 1/2`, so the pooled aggregate is `Σ_i h_i / √n`, with standard deviation `σ` in the roughly-independent case. That's the sqrt-length variant, and now I can see exactly what each normalizer stabilizes: dividing by n stabilizes the mean estimate (`E = μ`, variance `σ²/n`), and dividing by √n stabilizes the aggregate fluctuation scale (standard deviation about σ for all n). That second motivation matches the transformer sentence-encoder choice to divide the summed representation by `√length` to reduce sentence-length effects. Both are defensible; they answer slightly different questions. The cleanest uniform-average reading is the plain mean ÷n, so that's my primary; the ÷√n form is the variance-stabilizing sibling I'll keep alongside it.

I've been writing `(1/n) Σ_i h_i` as if I always know n and as if every slot in my tensor is a real member. In a real batch that's not true. Variable-size sets get padded to a common length N so they fit in a `[B, N, D]` tensor, and the pad slots are filler — not members. If I just call `.mean` over the member axis of that tensor, I commit two sins at once: I average in the pad vectors (which aren't part of the set), and I divide by N (the padded length) instead of n (the true count). Both corrupt the result, and the corruption is size-dependent in exactly the way I worked to remove. So the real reduction has to *mask*. I carry a mask `m_i ∈ {0,1}`, 1 for a real member and 0 for padding, and compute the mean over real members only:

  g = ( Σ_i m_i · h_i ) / ( Σ_i m_i ).

The numerator zeroes out the pad vectors before summing, and the denominator is the true count n = Σ_i m_i, not N. That's the masked mean, and it reduces to the plain mean exactly when every slot is real (all m_i = 1, so the denominator is N = n and the numerator is the full sum) — which is the case in the variable-aggregation setting, where every location has all V variables present and there's no padding at all. So `x.mean` over the variable axis and the masked mean agree there; the mask machinery only earns its keep when sets are genuinely ragged.

One more guard. The denominator Σ_i m_i is a count, and a count can be zero — an all-pad row, an empty set after some filtering. Then I'd divide by zero and get NaNs that poison the batch. So I floor the denominator at a tiny positive constant before dividing: `clamp(Σ_i m_i, min=1e-9)`. For any nonempty set the count is ≥ 1, vastly larger than 1e-9, so the clamp is invisible; it only fires on the degenerate empty set, where it turns a 0/0 into 0/1e-9 = 0 (since the masked numerator is also zero there) — a defined output instead of a NaN. It's the same role ε plays in any normalization: never change the healthy case, just keep the denominator off zero.

So the reduction I've landed on, in words: zero the non-members, sum over the set axis, divide by the true member count floored away from zero. Permutation-invariant (sum and count are both symmetric), eats any set size, expectation independent of size, zero parameters, and the uniform/maximum-entropy/zero-query-attention default. Let me write it as the function the pipeline actually calls, filling the slot. The member vectors come in as `[B, N, D]` and the mask as `[B, N]`; I expand the mask to `[B, N, D]` so it multiplies every channel, sum over the member axis N, and divide by the clamped masked count:

```python
import torch
import torch.nn as nn


class SetReducer(nn.Module):
    """Mean pooling: collapse a variable-size set of D-dim member vectors to ONE
    D-dim vector per item by averaging over the real (non-pad) members.
    No learnable parameters."""

    def __init__(self, embed_dim, sqrt_len=False):
        super().__init__()
        self.embed_dim = embed_dim
        self.sqrt_len = sqrt_len   # divide by n (mean) or by sqrt(n) (scale-stable variant)

    def forward(self, tokens, mask):
        # tokens: [B, N, D]   mask: [B, N]  (1.0 real member, 0.0 padding)
        input_mask_expanded = mask.unsqueeze(-1).expand(tokens.size()).float()
        summed = torch.sum(tokens * input_mask_expanded, dim=1)      # zero pads, sum over the set
        count = torch.clamp(input_mask_expanded.sum(dim=1), min=1e-9) # true count, floored
        if self.sqrt_len:
            return summed / torch.sqrt(count)            # ÷sqrt(n): aggregate scale ~const in n
        return summed / count                            # ÷n: the uniform mean over the set
```

And the special case the variable-aggregation task actually needs is even barer, because there every set is full — all V variables present at every spatial location, no padding, so the mask is all ones and the masked mean is just the plain average over the variable axis. The aggregator receives `[B, V, L, D]` and must return `[B, L, D]`; averaging over the variable axis V is the whole operation:

```python
import torch.nn as nn


class VariableAggregator(nn.Module):
    """Mean pooling over the V variable tokens at each spatial location.
    No learnable parameters; the parameter-free lower-bound aggregator."""

    def __init__(self, embed_dim, num_heads, num_vars):
        super().__init__()
        self.embed_dim = embed_dim
        self.num_heads = num_heads      # unused: mean pooling has no attention
        self.num_vars = num_vars        # unused: mean is size-agnostic

    def forward(self, x):
        # x: [B, V, L, D] -> uniform average over the variable axis V
        return x.mean(dim=1)            # [B, L, D]  (masked mean with an all-ones mask)
```

Walking the chain back: I needed to collapse an unordered, variable-size set of contextual member vectors to one fixed-size vector. Order-independence forced a symmetric reduction, ruling out concatenation and any recurrent reader that imposes an order. Picking a single designated token failed — it's a bottleneck trained for the wrong objective, and a per-variable set has no such token. Among symmetric combiners, the bare sum leaked set size into the output magnitude, so I divided by the count to get the mean, whose expectation is provably size-independent and which is the unique uniform / maximum-entropy / zero-query-attention weighting when no member is privileged — making it the honest parameter-free floor that any learned attentive pooling must beat. Max pooling stays as the alternative for sets whose signal lives in an extreme member, but the mean is the safe default for sets of comparably-informative members. The ÷√n sibling stabilizes aggregate scale rather than the mean estimate, for when set sizes vary widely. Real batches pad ragged sets, so the implementable form is the masked mean — sum the real members, divide by the true count clamped off zero — which collapses to a plain average when every set is full, as in the variable-aggregation case where the aggregator is simply `x.mean(dim=1)`.
