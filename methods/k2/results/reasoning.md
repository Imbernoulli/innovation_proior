Let me start from what actually goes wrong. I am fine-tuning a policy `ֿ€_־¸` with a reward, and I have to tether it to a frozen reference `ֿ€_ref` so it does not wander off into degenerate high-reward gibberish. The tether everyone uses is a KL penalty: add something proportional to how far `ֿ€_־¸` has drifted from `ֿ€_ref`, with a coefficient I get to tune. The honest object I want is `KL(ֿ€_־¸ג€–ֿ€_ref) = ־£_v ֿ€_־¸(v) log(ֿ€_־¸(v)/ֿ€_ref(v))`, a sum over the whole vocabulary at every token position. On a language model that vocabulary is tens of thousands of tokens, and I would have to form and differentiate through that whole sum at every position of every sequence in every minibatch. That is not happening. What I actually have lying around, cheaply, is the log-probability of the token that was *sampled*, under both models: two scalars per token, `logprob = log ֿ€_־¸(x)` and `ref_logprob = log ֿ€_ref(x)`. The cleanest starting point is Monte-Carlo. Done?

No. Stare at `k1 = ־”` for a second as a *random variable*. KL is always nonnegative, but `־”` for an individual sampled token is `log(ֿ€_־¸/ֿ€_ref)` and that is negative whenever the policy assigns the sampled token a *lower* probability than the reference does ג€” which is roughly half the tokens. So my estimator is positive on half its samples and negative on the other half, and the true value it is estimating is a small positive number sitting near zero. The signal is a small mean riding on top of a big two-sided spread. On a toy where I can actually compute the true KL ג€” say `q = N(0,1)`, `p = N(0.1,1)`, true KL `ג‰ˆ 0.005` ג€” the per-sample standard deviation of `k1` is something like twenty times the true value. Twenty. To average that down to a usable penalty I would need an absurd number of samples. As a value estimator `k1` is unbiased but uselessly noisy.

But the variance is not even the part that should worry me most. Let me differentiate `k1` as a loss, because that is what I am going to do to it. `k1 = log ֿ€_־¸(x) גˆ’ log ֿ€_ref(x)`. The reference is frozen, `גˆ‡_־¸ log ֿ€_ref = 0`, so `גˆ‡_־¸ k1 = גˆ‡_־¸ log ֿ€_־¸(x)`. Now average that over the on-policy samples: `E_{xגˆ¼ֿ€_־¸}[גˆ‡_־¸ log ֿ€_־¸(x)] = גˆ« ֿ€_־¸ גˆ‡_־¸ log ֿ€_־¸ = גˆ« גˆ‡_־¸ ֿ€_־¸ = גˆ‡_־¸ גˆ« ֿ€_־¸ = גˆ‡_־¸ 1 = 0`. The expected gradient is *zero*. This is the same score-function identity that makes policy gradients work ג€” the mean of the score is zero ג€” and here it bites me: my KL penalty, which reports a correct KL value, has a gradient that in expectation pushes the policy *nowhere*. At any reasonable batch size the regularizing signal averages out to nothing. So `k1` fails on two counts ג€” high-variance value *and* a gradient that vanishes ג€” and the second failure is the fatal one for the KL-loss path, because the whole point of putting the penalty in the loss is that its gradient should pull `ֿ€_־¸` back toward `ֿ€_ref`. Wall.

So I want two things at once: a value that is one-signed and low-variance like the true KL, and a gradient that does not collapse to zero. Let me think about each.

The variance and one-signedness first, because that is the more obvious fix. The disease is that `־”` takes both signs. The crude cure is to take its absolute value, `|־”|` ג€” now it is nonnegative and its spread is smaller. But look at the gradient of `|־”|`: it is `sign(־”)ֲ·גˆ‡_־¸ log ֿ€_־¸`. The magnitude of the log-ratio has been thrown away; the gradient only knows *which side* of the reference the policy is on, not *how far*. That means the strength of the correction does not scale with the size of the drift ג€” a token that has drifted enormously gets pushed back exactly as hard as one that has barely moved. That is a coarse, sign-only tether, not a faithful KL gradient. And there is a kink at `־”=0`. So `|־”|` patches the value but mangles the gradient in a different way.

Let me try to be smarter about the one-signedness. What single function of `־”` is nonnegative, smooth, small when `־”` is small, and grows with `|־”|` symmetrically? The square, `־”ֲ²`. It is `ג‰¥0`, it is smooth, it vanishes only when the policy matches the reference on that token, and ג€” unlike `|־”|` ג€” its size *does* scale with the drift. As a value, `ֲ½־”ֲ²` (I will explain the half shortly) is biased, because it is not literally the KL, but it is one-signed and it turns a two-sided small-mean random variable into a local distance. That is promising only if I can explain why its mean should track KL near the reference rather than merely be some unrelated positive number.

But why on earth should `ֲ½־”ֲ²` have *low bias* as an estimate of the KL value? That is not obvious ג€” `־”ֲ²` and `־”` are different functions, their means need not be close. Let me chase this, because if I understand it I will trust the estimator. The mean I am computing is `E_{xגˆ¼ֿ€_־¸}[ֲ½(log(ֿ€_־¸/ֿ€_ref))ֲ²]`. Write it in the same generator form used for f-divergence local expansions, `E_{xגˆ¼q}[f(p(x)/q(x))]`. With `q = ֿ€_־¸` (the sampling distribution) and ratio `ֿ€_ref/ֿ€_־¸`, my squared estimator uses `f(x) = ֲ½(log x)ֲ²` evaluated at `x = ֿ€_ref/ֿ€_־¸`: indeed `ֲ½(log(ֿ€_ref/ֿ€_־¸))ֲ² = ֲ½(log(ֿ€_־¸/ֿ€_ref))ֲ² = ֲ½־”ֲ²`, the square killing the sign flip. The KL value I actually want, `E_{xגˆ¼ֿ€_־¸}[log(ֿ€_־¸/ֿ€_ref)] = E[גˆ’log(ֿ€_ref/ֿ€_־¸)]`, uses the generator `f(x)=גˆ’log x`. The squared generator is not globally convex for every positive `x`, so I should not lean on global divergence guarantees; the question I care about for a penalty near the reference is the local Taylor geometry around ratio `1`.

Now the fact that makes this click: these generator-based distances have the same second-order shape when the two distributions are close. Concretely, for a parametric family near coincidence, the expansion is `D_f(p_0, p_־¸) = (f''(1)/2)ֲ·־¸^T F ־¸ + O(־¸ֲ³)`, where `F` is the Fisher information ג€” the leading behavior is fixed entirely by the single number `f''(1)`, times the same Fisher quadratic form. So if my estimator's `f` and KL's `f` have the *same* `f''(1)`, their expectations agree to second order ג€” they are the *same* local quadratic distance, and the bias is third-order-small whenever `ֿ€_־¸` is near `ֿ€_ref`, which is exactly the regime a KL *penalty* is trying to keep me in. Let me check the two second derivatives. For KL, `f(x) = גˆ’log x`, `f'(x) = גˆ’1/x`, `f''(x) = 1/xֲ²`, so `f''(1) = 1`. For my estimator, `f(x) = ֲ½(log x)ֲ²`, `f'(x) = (log x)/x`, `f''(x) = (1 גˆ’ log x)/xֲ²`, so `f''(1) = (1 גˆ’ 0)/1 = 1`. They match. And that match is *why* I want the factor of one-half: without it, `f(x) = (log x)ֲ²` gives `f''(1) = 2`, and the estimator would locally look like *twice* the KL ג€” a constant factor I would then have to absorb into the coefficient. The `ֲ½` is what makes the estimator agree with KL to second order on the nose, so the coefficient `kl_loss_coef` keeps its local meaning. The bias grows once `ֿ€_־¸` and `ֿ€_ref` separate enough that the higher-order terms wake up. Fine: I accept a value that is biased far from the reference, because near the reference it is calibrated and that is where the penalty is supposed to operate.

Now the gradient, which is the thing that actually killed `k1`. Differentiate `ֲ½־”ֲ²` as a loss. `גˆ‡_־¸ (ֲ½־”ֲ²) = ־”ֲ·גˆ‡_־¸ ־” = ־”ֲ·גˆ‡_־¸ log ֿ€_־¸` (again the reference term drops). So the per-token gradient is `(log ֿ€_־¸ גˆ’ log ֿ€_ref)ֲ·גˆ‡_־¸ log ֿ€_־¸`. Take the on-policy expectation: `E_{xגˆ¼ֿ€_־¸}[־”ֲ·גˆ‡_־¸ log ֿ€_־¸]`. Is this the true KL gradient? Let me compute the true gradient from scratch and compare. `גˆ‡_־¸ KL(ֿ€_־¸ג€–ֿ€_ref) = גˆ‡_־¸ גˆ« ֿ€_־¸ log(ֿ€_־¸/ֿ€_ref) = גˆ« (גˆ‡_־¸ ֿ€_־¸) log(ֿ€_־¸/ֿ€_ref) + גˆ« ֿ€_־¸ גˆ‡_־¸ log(ֿ€_־¸/ֿ€_ref)`. The first integral is `גˆ« ֿ€_־¸ גˆ‡_־¸ log ֿ€_־¸ ֲ· log(ֿ€_־¸/ֿ€_ref) = E[־”ֲ·גˆ‡_־¸ log ֿ€_־¸]`. The second is `גˆ« ֿ€_־¸ גˆ‡_־¸ log ֿ€_־¸ = E[גˆ‡_־¸ log ֿ€_־¸] = 0` by the score identity. So `גˆ‡_־¸ KL = E_{xגˆ¼ֿ€_־¸}[־”ֲ·גˆ‡_־¸ log ֿ€_־¸]` ג€” exactly the expectation of my squared estimator's gradient. The squared estimator's gradient is an *unbiased estimator of the true KL gradient*.

Sit with how strange and good that is. `k1` had the right *value* but a *zero* gradient. The squared estimator has the *wrong* value (biased) but the *right* gradient. They are mirror images, and which one I want depends on what I do with the number. In the KL-loss path I backpropagate it ג€” the gradient is the entire point ג€” so I want the one with the correct gradient even though its value is biased. The thing that flipped a useless-gradient estimator into a correct-gradient one was precisely the squaring: `גˆ‡(ֲ½־”ֲ²) = ־”ֲ·גˆ‡־”` reattaches the log-ratio magnitude `־”` as the coefficient on the score `גˆ‡log ֿ€_־¸`, and that coefficient is exactly the weight the true KL gradient demands. The `k1` gradient was `1ֲ·גˆ‡log ֿ€_־¸` ג€” coefficient one, which averages to zero against the score. The squared estimator's gradient is `־”ֲ·גˆ‡log ֿ€_־¸` ג€” coefficient `־”`, which does *not* average to zero against the score, because `־”` is correlated with the score in just the way the KL gradient needs. This is the whole reason the squared form exists.

Let me make sure I am not fooling myself by checking whether there is a value-correct, low-variance estimator that also has the right gradient ג€” because if there is, I would prefer it. The standard variance-reduction move is a control variate: take `k1` and add a term with zero expectation that is negatively correlated with it, to cancel the noise without moving the mean. What zero-mean quantity do I have? With `r = ֿ€_ref/ֿ€_־¸`, `E_{xגˆ¼ֿ€_־¸}[r] = גˆ« ֿ€_־¸ (ֿ€_ref/ֿ€_־¸) = גˆ« ֿ€_ref = 1`, so `r גˆ’ 1` has mean zero. So `גˆ’log r + ־»(r גˆ’ 1)` is, for any `־»`, an unbiased estimator of `KL(ֿ€_־¸ג€–ֿ€_ref)` (recall `גˆ’log r = גˆ’log(ֿ€_ref/ֿ€_־¸) = ־”`). I could minimize its variance over `־»`, but that optimal `־»` depends on `ֿ€_־¸, ֿ€_ref` analytically ג€” not available. There is a cheap principled choice though: `log r ג‰₪ r גˆ’ 1` always (concavity of `log`, the tangent at `1`), so taking `־» = 1` makes `גˆ’log r + (r גˆ’ 1) = (r גˆ’ 1) גˆ’ log r ג‰¥ 0` ג€” the always-nonnegative vertical gap between `log` and its tangent line, a Bregman divergence. Call it `k3 = (r גˆ’ 1) גˆ’ log r = e^{גˆ’־”} גˆ’ 1 + ־”`. As a *value* this is gorgeous: unbiased, nonnegative, and built from the natural control variate whose only job is to cut variance without moving the mean. If value were what I cared about, `k3` would be the natural winner.

But I am backpropagating, so let me differentiate `k3` too. `k3 = e^{גˆ’־”} גˆ’ 1 + ־”`, with `־” = log ֿ€_־¸ גˆ’ log ֿ€_ref` and `r = e^{גˆ’־”}`. `גˆ‡_־¸ k3 = (גˆ’e^{גˆ’־”} + 1)ֲ·גˆ‡_־¸ ־” = (1 גˆ’ r)ֲ·גˆ‡_־¸ log ֿ€_־¸`. Take the expectation: `E[(1 גˆ’ r)ֲ·גˆ‡_־¸ log ֿ€_־¸] = E[גˆ‡_־¸ log ֿ€_־¸] גˆ’ E[rֲ·גˆ‡_־¸ log ֿ€_־¸] = 0 גˆ’ גˆ« ֿ€_־¸ (ֿ€_ref/ֿ€_־¸) גˆ‡_־¸ log ֿ€_־¸ = גˆ’גˆ« ֿ€_ref גˆ‡_־¸ log ֿ€_־¸`. That is `גˆ’E_{xגˆ¼ֿ€_ref}[גˆ‡_־¸ log ֿ€_־¸]`, an expectation under the *reference*, not under `ֿ€_־¸`, and it is not equal to `גˆ‡_־¸ KL = E_{xגˆ¼ֿ€_־¸}[־”ֲ·גˆ‡_־¸ log ֿ€_־¸]` in general. So `k3`, the value-best estimator, has a *biased* gradient ג€” and in practice the bias is not tiny: once `ֿ€_־¸` drifts from `ֿ€_ref`, the `r = ֿ€_ref/ֿ€_־¸` factor swings wildly (a token rare under the policy but common under the reference makes `r` blow up), and the gradient it produces drifts substantially from the KL gradient. So `k3` and `k1` sit on one side ג€” correct value, wrong gradient ג€” and the squared estimator sits alone on the other ג€” wrong value, correct gradient. For a penalty I differentiate, the squared estimator is the only one of the three whose gradient actually optimizes the KL.

There is one more attraction to `k3` I should weigh against and then set down: there is a clean way to get *both* a good value and the right gradient if I am willing to split forward from backward. I can take the *value* from `k3` (nonnegative, low-variance, a faithful KL number for logging and for the magnitude of the penalty) and the *gradient* from the squared estimator (correct), via a straight-through construction: compute `s = ֲ½־”ֲ²`, and return `s גˆ’ stop_grad(s) + stop_grad(k3_value)`. The forward value is `k3`'s; the backward gradient is `s`'s, i.e. `־”ֲ·גˆ‡log ֿ€_־¸`. That is the best of both when I explicitly ask for a `+` variant: base estimator in the forward pass, squared estimator in the backward pass. But it costs the `k3` exponential and its clamping, and it is a strictly more elaborate object. For the plain penalty, the squared estimator alone already carries the one property I cannot do without ג€” the correct gradient ג€” and it needs no exponential at all, so it cannot overflow on a rare token the way `e^{גˆ’־”}` can. I will reach for the straight-through hybrid only when I specifically want the cleaner forward value; the base estimator is the square.

So the estimator is `ֲ½(log ֿ€_־¸(x) גˆ’ log ֿ€_ref(x))ֲ²` per token, and I want to nail down the few remaining choices. The factor `ֲ½`: I already have two independent reasons for it, and they agree, which is reassuring. The local generator reason ג€” `f''(1) = 1` so the estimator's expectation matches KL to second order ג€” and the gradient reason: `גˆ‡(ֲ½־”ֲ²) = ־”ֲ·גˆ‡log ֿ€_־¸` with coefficient exactly `1` on `־”`, so the gradient *is* the KL gradient with no stray factor of two to launder into the coefficient. Drop the `ֲ½` and both the value calibration and the gradient pick up a factor of two; keep it and everything is unit-clean. Numerical safety: the square involves no exponential, no logarithm beyond the log-probs I already hold, so there is nothing to overflow ג€” `־”ֲ²` is just a subtraction and a multiply. This is the squared estimator's quiet advantage over `k3`, which has to clamp `ref_logprob גˆ’ logprob` before `exp` and clamp its output afterward to keep a far-out token from producing an enormous or non-finite penalty. The plain square branch does not need those clamps. Gradient connectivity: the whole construction lives or dies on the gradient flowing back into `logprob`, so I must *not* wrap this in any no-grad context ג€” `logprob` carries `requires_grad`, `־” = logprob גˆ’ ref_logprob` keeps it, and `ֲ½־”ֲ²` keeps it. The reference log-prob is frozen and contributes no gradient, which is exactly what I want. Shape: I map two `(batch, response_length)` tensors elementwise to one `(batch, response_length)` tensor of per-token penalties; the surrounding loop masks it by the response mask, aggregates to a scalar, scales by `kl_loss_coef`, and adds it to the policy-gradient loss.

Let me write it as the body that fills the one open slot in the training step ג€” the elementwise map from the two log-prob tensors to the per-token penalty:

```python
import torch


def compute_kl_penalty(
    logprob: torch.Tensor,      # (batch, response_length) ג€” log ֿ€_־¸(x), current policy (requires grad)
    ref_logprob: torch.Tensor,  # (batch, response_length) ג€” log ֿ€_ref(x), frozen reference
) -> torch.Tensor:              # (batch, response_length) ג€” per-token KL penalty
    # ־” = log ֿ€_־¸ גˆ’ log ֿ€_ref ; the reference term carries no gradient (frozen model).
    # NOTE: no torch.no_grad() ג€” the gradient must flow back into logprob.
    log_ratio = logprob - ref_logprob

    # k2 / MSE estimator: ֲ½ ־”ֲ².
    #   value: biased, but its local generator has f''(1)=1, so it matches KL to 2nd
    #          order near ֿ€_־¸ג‰ˆֿ€_ref; always ג‰¥ 0; low variance.
    #   gradient: גˆ‡(ֲ½־”ֲ²) = ־”ֲ·גˆ‡log ֿ€_־¸, whose on-policy expectation IS גˆ‡KL(ֿ€_־¸ג€–ֿ€_ref)
    #             ג€” the correct KL gradient, which is what gets backpropagated.
    #   numerics: no exp/log here ג†’ nothing to overflow; the ֲ½ keeps both the value
    #             calibration (f''(1)=1) and the gradient coefficient on ־” exactly 1.
    return 0.5 * log_ratio.square()
```

So the causal chain. I needed a per-token KL penalty built from just the two sampled log-probs, cheap and safe, that I would *backpropagate*. The naive log-ratio `k1` had the correct KL value but was high-variance and, worse, its gradient `גˆ‡log ֿ€_־¸` averages to zero by the score-function identity, so it tethers nothing. Forcing nonnegativity with `|־”|` fixed the value's sign but gave a sign-only gradient that ignores the magnitude of the drift. Squaring the log-ratio gave a nonnegative, low-variance value that ג€” because the local generator `ֲ½(logֲ·)ֲ²` has the same `f''(1)=1` as KL's `גˆ’log` generator ג€” matches the true KL to second order near the reference, so its value bias is small exactly where a penalty operates; the `ֲ½` is what calibrates that match and also keeps the gradient coefficient clean. And the decisive property: `גˆ‡(ֲ½־”ֲ²) = ־”ֲ·גˆ‡log ֿ€_־¸`, whose expectation is exactly the true KL gradient, so the squared estimator ג€” alone among the cheap options ג€” backpropagates the right thing. The control-variate `k3` is the value-best estimator (unbiased, low variance, nonnegative) but its gradient `(1גˆ’r)גˆ‡log ֿ€_־¸` is biased and needs an overflow-prone exponential, so it is the better *logged value* and the worse *loss*; when I want both I can take `k3`'s value with the square's gradient via a straight-through, but the base penalty I differentiate is `ֲ½(logprob גˆ’ ref_logprob)ֲ²`.
