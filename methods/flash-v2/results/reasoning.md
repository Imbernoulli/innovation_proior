I have a fused exact-attention kernel that already solved the memory problem — it tiles Q, K, V into blocks, streams them through SRAM, uses the online-softmax statistics to combine blocks, and never writes the N×N score matrix to HBM, so its memory is linear in N and it does many-fold fewer HBM accesses than the materialized implementation, `Θ(N²d²/M)` against `Θ(Nd + N²)`. That bought 2-4x over standard attention. And yet when I profile it on an A100 the forward pass sits at only about 25-40% of the device's theoretical FLOPs/s, while an optimized GEMM on the same card runs at 80-90% of peak. So the kernel is no longer waiting on memory; the Tensor Cores are idle for some other reason. The whole game now is: keep the exact output and the linear memory, but find where the *work* is being scheduled badly and fix it so I get GEMM-like utilization.

Before I touch anything I need to internalize the cost model, because it tells me what "scheduled badly" even means here. The A100 does 312 TFLOPs/s of FP16/BF16 matmul on its Tensor Cores but only 19.5 TFLOPs/s of non-matmul FP32 on the regular ALUs. Divide those: a non-matmul FLOP is about 16x more expensive than a matmul FLOP. That number reframes the whole problem. In attention the matmuls are QKᵀ and PV; everything else — the exponentials, the row-maxes, the rescalings, the divisions, the mask comparisons — is non-matmul. By raw FLOP count the non-matmul part is small, a couple of N×N elementwise passes against two N×N×d GEMMs. But by *time* it can dominate, because each of those cheap-looking FLOPs costs 16 matmul-FLOPs of wall clock. So my first instinct — "the kernel is exact and IO-optimal, what's left to do?" — is wrong. There's a whole axis I wasn't counting: every elementwise operation in the inner loop is disproportionately expensive, and if I can move non-matmul work out of the hot loop without changing the answer, I directly buy back throughput. Let me hold that thought and go look at the inner loop I actually have.

The forward loop I inherited keeps a running output block O_i together with the online-softmax statistics m_i (running max) and ℓ_i (running normalizer), and on every K/V block j it does the combine-and-rescale. Written out, the update is `O_i ← diag(ℓ_i^new)^{-1}(diag(ℓ_i) e^{m_i - m_i^new} O_i + e^{m̃_{ij} - m_i^new} P̃_{ij} V_j)`. Stare at that. Inside the loop, every single block, I divide the running output by the new normalizer `diag(ℓ_i^new)^{-1}`. That's a per-element division over the whole B_r×d output tile, on the regular ALUs, once per column block — and a division is exactly the 16x-expensive kind of op I just decided to hunt. Why am I dividing every block? Only because I insist the running O_i be the *correctly normalized* partial output at every step. But I don't actually need that at every step — I only need the *final* O_i, at the end of the loop, to be normalized. So the question is whether I can carry the output *unnormalized* through the loop and divide once at the very end. The danger is the running max: the normalizer and the partial output are both taken relative to whatever the max was at the time, and the max can jump on any block, so deferring the division has to interact correctly with the max-rescaling. I don't trust that from the algebra alone — let me work it out, then check it on actual numbers.

Take the simplest case, two blocks. Block 1: m⁽¹⁾ = rowmax(S⁽¹⁾), ℓ⁽¹⁾ = rowsum(e^{S⁽¹⁾ - m⁽¹⁾}), and instead of forming the normalized P̃⁽¹⁾V₁ I keep the unnormalized accumulator Õ⁽¹⁾ = e^{S⁽¹⁾ - m⁽¹⁾} V₁. Block 2: the new max m⁽²⁾ = max(m⁽¹⁾, rowmax(S⁽²⁾)) = m. The normalizer combines with the online-softmax correction, ℓ⁽²⁾ = e^{m⁽¹⁾ - m⁽²⁾} ℓ⁽¹⁾ + rowsum(e^{S⁽²⁾ - m⁽²⁾}), which equals rowsum(e^{S⁽¹⁾ - m}) + rowsum(e^{S⁽²⁾ - m}) = ℓ, the right full-row normalizer. Now the accumulator: I rescale the old unnormalized accumulator by just the max-correction factor and add the new unnormalized contribution, Õ⁽²⁾ = diag(e^{m⁽¹⁾ - m⁽²⁾}) Õ⁽¹⁾ + e^{S⁽²⁾ - m⁽²⁾} V₂. Substitute Õ⁽¹⁾: that's e^{m⁽¹⁾ - m} e^{S⁽¹⁾ - m⁽¹⁾} V₁ + e^{S⁽²⁾ - m} V₂ = e^{S⁽¹⁾ - m} V₁ + e^{S⁽²⁾ - m} V₂ — the two exponents both telescope down to the global max m. Then at the end O⁽²⁾ = diag(ℓ⁽²⁾)^{-1} Õ⁽²⁾ = (e^{S⁽¹⁾-m}V₁ + e^{S⁽²⁾-m}V₂)/ℓ, which is what plain softmax-then-weighted-sum gives. The algebra says exact; I want a number on it, picking scores where the max deliberately jumps on block 2 (the failure mode I was worried about). One query row, scores [0.5, 1.0] on block 1 and [3.0, -1.0] on block 2 (so the global max 3.0 lives in block 2), V rows [1,0],[0,1] then [1,1],[2,0]. Running the deferred-normalization update against a plain softmax over all four scores: the kernel path gives O = [0.90530368, 0.91875235] and the reference gives [0.90530368, 0.91875235], max abs difference 2.2e-16 — machine epsilon, so the deferred division and the max-correction compose correctly even when the max jumps. The check also pins down one implementation detail I'd have gotten wrong otherwise: I seed ℓ_i = 1.0 (not 0), because on the first block m_i = -∞ makes alpha = e^{-∞ - m_new} = 0, which zeroes out that seed — so the seed value is irrelevant as long as the first alpha is 0, and 1.0 is the convenient choice. What deferring saves: the accumulator still needs the per-block max-correction rescale `diag(e^{m⁽¹⁾ - m⁽²⁾})` (unavoidable — the past sum was taken relative to the old max), but the *division* by the normalizer is gone from the loop entirely and happens once after the last block. `T_c` divisions per output row collapse to one. First chunk of non-matmul work out of the hot path, answer unchanged.

While I'm in the statistics, there's a second little economy. For the backward pass I need to be able to recompute P = softmax(S), and naively I'd save both m and ℓ for that. But P = e^{S - m}/ℓ = e^{S - m - log ℓ} = e^{S - (m + log ℓ)}. So in the mathematical, natural-exponential view I only need the single logsumexp L = m + log(ℓ); one vector replaces two. In the base-2 kernel, the same object is stored in base-2 units as M = m₂ + log2(ℓ), and the backward recomputes P = exp2(S₂ - M). Half the statistics memory, and it's the natural object anyway — the row's log-partition, just expressed in the base the kernel is using. Store that per row block; drop the separate m and ℓ.

Now the exponentials themselves. The inner loop calls exp on a B_r×B_c tile every block — that's a lot of the non-matmul time, and it's transcendental, the most expensive ALU work there is. I can't remove the exponentials, the softmax needs them. But the GPU's special-function units compute base-2 exponent, exp2, natively and faster than natural exp. And e^x = 2^{x·log₂e} with log₂e = 1.44269504, so I can convert: instead of e^{S - m} I compute exp2((S - m)·log₂e). At first that looks like I just *added* a multiply by 1.44269504 to every element — trading the cost of natural exp for exp2 plus a per-element scale, which is no win and maybe a loss. But watch where the scale can go. The scores are S = (QKᵀ)·sm_scale, and I'm about to multiply S by log₂e anyway. Both sm_scale and log₂e are constants for the whole kernel. So fold them together and push them onto Q *at load time*: load q′ = q·(sm_scale·log₂e) once, before the loop even starts. Then QKᵀ already comes out as S·log₂e, and the inner loop is pure exp2 with no extra per-element multiply. Check it against the natural path directly, on numbers. Take q = [0.3, -0.7, 1.2, 0.05], k = [1.1, 0.2, -0.4, 0.9], sm_scale = 1/√64. Natural path: exp(sm_scale·⟨q,k⟩) = 0.96983919459. Folded path: pre-scale q by sm_scale·log₂e, dot with k, then 2^(·) = 0.96983919461. Relative difference 1.9e-11 — the identity holds (the tiny gap is just float rounding in the constant), and the running-max comparison and the subtraction are then all consistently in the log₂ scale because everything entering the loop is. The scale multiply, which used to be one multiply per element of the N×N score matrix, becomes one multiply per element of the B_r×d query tile, done once. exp becomes exp2 (cheaper SFU op) and the scaling leaves the hot loop. Another slice of non-matmul work, gone, answer unchanged.

Causal masking is the next non-matmul sink, and it's worse than it looks because it's a comparison plus a conditional write `tl.where(row ≥ col, qk, -inf)` over the whole tile, plus then exponentiating a tile full of -inf to zero. But I'm already working in blocks, so I can reason about masking at the block level instead of the element level. Fix a query row block i (rows in `[start_m·B_r, (start_m+1)·B_r)`). A K/V column block j covers columns `[j·B_c, (j+1)·B_c)`. There are three kinds of blocks. If the block is entirely above the diagonal — every column index exceeds every row index in the tile, i.e. j·B_c ≥ (start_m+1)·B_r — then the whole tile is masked to zero and contributes nothing: I can skip the block outright, no QKᵀ, no exp, no PV. If the block is entirely below the diagonal — every column index is ≤ every row index, i.e. (j+1)·B_c ≤ start_m·B_r — then nothing is masked and I can skip the `tl.where` entirely: just compute the block normally. Only the blocks straddling the diagonal actually need the elementwise mask. So instead of masking every block, I split the column loop into two passes: a first pass over the strictly-below-diagonal blocks with no mask at all (cutting `non_causal_end` to a B_c multiple, `floor(start_m·B_r / B_c)·B_c`), and a second pass over just the boundary block(s) where I apply `tl.where`.

The per-row-block boundary count depends on the block shape, so I want the exact number, not just "about half." Take a concrete config, B_r=128, B_c=64 at seqlen=16384 — 128 row blocks, 256 column blocks. Mask-every-block would iterate 128×256 = 32768 block-iterations. Summing the two-pass counts per row block (pass1 = floor(start_m·128/64) blocks, pass2 = the remaining blocks up to (start_m+1)·128) gives 16512 block-iterations. So the two-pass scheme skips 32768−16512 = 16256, i.e. 49.6% of the blocks never get computed — close to the naive intuition of half, and now I have the actual number (block-count ratio 1.98x). One thing the count corrects: the boundary needs 2 masked blocks per row block on average here, not one — "one boundary block per row block" only holds when B_r = B_c (square tiles); with B_r=128, B_c=64 the diagonal cuts across two column blocks. That's still a vanishing fraction (2 of up to 256), so the `tl.where` cost is tiny, but it's 2 not 1, worth knowing for the loop bounds. The other thing the count makes honest: the 1.98x is purely *block count*. Wall-clock won't be 1.98x faster on the causal forward, because the blocks I *keep* (the whole lower triangle) still cost their full time — the speedup over a mask-everything kernel is real but smaller than 2x, since I'm only saving the upper-triangle compute and the masking overhead, not changing the lower triangle. So: the whole upper triangle never computed, the mask applied to ~2 blocks per row block, and the elementwise masking overhead nearly eliminated — a meaningful causal speedup, just not literally 2x in wall-clock.

So the inner-loop cleanup is a family of the same move: find a non-matmul operation in the inner loop and either defer it to the end, convert it to a cheaper unit, or skip it by blocks — and each one I either proved or checked numerically leaves the output exact. But reducing non-matmul FLOPs only helps if the device is busy in the first place, and I have a nagging second observation from the profile: low occupancy. So let me look at how the work is spread across the SMs, because that's a different bottleneck entirely.

The kernel I inherited launches one thread block per attention head, and parallelizes over batch × heads. On an A100 there are 108 SMs, and a thread block runs on one SM, so I want at least ~108 thread blocks (more, to hide latency) to fill the machine. Batch × heads is fine when the batch is big. But the whole reason I care about this kernel is *long* sequences, and long sequences usually come with *small* batches — you can't fit many 16k-token sequences in memory at once. So exactly in the regime I'm optimizing for, batch × heads can drop below 108, and most of the GPU sits idle. I need another dimension to parallelize over. What's available? The sequence length. Here's the structural fact that makes it possible: softmax couples a row across the *key* dimension — to normalize row i I need all of row i's scores — but different *query rows are completely independent*. Row i's output depends on Q_i, all of K, all of V, but not on any other query row. So I can hand each query row block to its own thread block and they never need to talk to each other.

But the loop I have is structured the wrong way for that. Its outer loop runs over K/V *column* blocks j and the inner loop over Q *row* blocks i, which is why it keeps reloading the running O_i and its statistics from HBM — different column-block iterations of the outer loop touch the same row block, so the partial output has to live in HBM between them. If instead I make the outer loop run over Q *row* blocks i, and the inner loop over K/V column blocks j, then each outer iteration is a self-contained computation of one output row block: load Q_i once, loop over all of K and V accumulating the running (Õ_i, m_i, ℓ_i) entirely in SRAM/registers, normalize, write O_i once. Two things follow. First, no row block's partial output ever round-trips to HBM mid-computation, because everything one output block needs is consumed within its own outer iteration. Second — the payoff I'm after — each outer iteration is *independent*, so I can give each one its own thread block. That's the whole motivation, so put a number on it. The launch grid becomes (number of row blocks) × (batch) × (heads). At seqlen 16k with B_r = 128 that's 16384/128 = 128 row blocks per head. So even at the worst case I actually care about — batch 1, and say 8 heads — that's 128 × 1 × 8 = 1024 thread blocks, against 108 SMs. The old grid of just batch × heads would have been 1 × 8 = 8 thread blocks, filling 8 of 108 SMs (~7% occupancy) and leaving the machine almost entirely idle. So the loop swap takes occupancy from ~7% to comfortably oversubscribed in exactly the long-sequence/small-batch regime that motivated the whole kernel. The seqlen dimension is what restores occupancy when batch × heads alone couldn't, and it comes for free from reordering the loops — no extra arithmetic, just a different mapping of the same work onto the grid.

Now I have the SMs full and the inner loop stripped of cheap-but-slow ops. One layer left: inside a single thread block, how do its warps divide the work? A thread block has 4 or 8 warps, and warps communicate only through shared memory with a sync in between, which is itself overhead I'd like to avoid. The scheme I inherited is "split-K": within the block, K and V are split across the warps and Q is shared by all of them. So each warp computes a partial QKᵀV using its slice of K and V — but its slice covers only some of the columns of the row, so no warp has the *full* row's output; the warps' partial results have to be written out to shared memory, the block synchronizes, and then the partials are summed to get the block's output. Every block, every warp writes its partial to shared memory and reads back others' — exactly the shared-memory round-trips I keep finding are pure overhead. Why is it laid out this way? Because in the old outer-loop-over-K structure, K was the dimension being streamed, so it felt natural to split it. But I just swapped the loops. Now the row is the independent unit, so the natural thing is to split the *rows* across warps instead — "split-Q": split Q across the warps, keep K and V shared by all of them. Warp w owns a contiguous slice of the row block's rows. It computes its slice of QKᵀ against the shared K — that's a full set of scores for *its* rows. It runs the online softmax on its own rows; the running max and normalizer are per-row, and these are its rows, so no other warp's data enters. It multiplies its P slice by the shared V to get its slice of the output. At no step did warp w read a quantity that warp w′ produced — the only shared inputs are K and V, which are read-only, not partial results. So the block's output is the plain concatenation of the warps' row slices: no shared-memory reduction, no extra sync, no round-trip. Contrast that trace with split-K, where each warp held a slice of the *columns* (the contraction axis): a warp's QKᵀV over its column slice is only a *partial* sum for every row — to get any row's actual output you must add the other warps' partials, which forces the write-to-shared-memory / sync / sum reduction. The difference comes down to which axis each scheme cuts along: split-Q cuts along rows, which don't couple, so the slices are final; split-K cut along the contraction, which is summed over, so the slices are partial. The loop swap is what made rows the available axis to cut, so split-Q only becomes possible *after* the swap.

So three levers, all preserving the exact output: fewer non-matmul FLOPs in the inner loop (defer normalization, store only logsumexp, exp2 with the scale folded into Q, two-pass causal that skips half the blocks); parallelize the outer loop over row blocks across the sequence dimension to fill the SMs; and split Q across warps to kill the inter-warp shared-memory reduction. There's one knob left to set, the block sizes B_r×B_c. Bigger blocks mean fewer iterations of the loop and so fewer shared-memory loads/stores per output element — good — but they need more registers and more shared memory to hold the tiles, and past a point the register file spills (catastrophic slowdown) or the shared memory simply doesn't fit and the kernel won't launch. The sweet spot depends on the head dimension d, because d sets how wide each tile is, and on how much shared memory the device has. In practice there are only about four sensible choices, {64,128}×{64,128}, and I pick per head dimension — smaller d can afford a larger B_r (e.g. 128×64), large d (256) needs the smaller 64×64 to stay within registers and shared memory. I'll tune these by hand per head dim for now; there are few enough cases that it's not worth autotuning yet, though autotuning the block sizes is the obvious next step.

Each change carries a different kind of justification: the deferred normalization is telescoped algebraically above and checked numerically against a reference softmax on a case where the max jumps, agreeing to 2.2e-16. The logsumexp substitution is a pure algebraic identity, P = e^{S - (m + log ℓ)}, nothing to check beyond reading it. The exp2 conversion is the identity e^x = 2^{x log₂e} with the scale folded into Q, verified numerically to 1.9e-11. The two-pass causal only *skips* blocks that are entirely masked and only *omits the mask* on blocks where no element would be masked, so block-by-block it computes the same thing as masking every block, and the block count confirmed the kept set is exactly the lower triangle plus the diagonal boundary. The loop swap and the warp split-Q rearrange *where* arithmetic happens, not *what* arithmetic — each output row is still softmax(Q_i Kᵀ) V over all of K, V, and no warp ever consumes another warp's partial. What I have *not* run is the assembled kernel on a GPU against a reference at FP16 tolerance — the individual transformations check out, but FP16 accumulation order can surprise you even when the math is right, so that end-to-end max-abs-difference test is still owed. Modulo that, the kernel returns O = softmax(QKᵀ)V in O(N²d) FLOPs with O(N) extra memory for the stored logsumexp — same correctness and memory guarantees as the IO-aware predecessor, just scheduled to keep the Tensor Cores busy.

Now assemble the pieces into the kernel body and the grid/block choice the harness left empty: one program owns one query row block of one head, loops over K/V column blocks with the unnormalized accumulator and the online statistics in registers, causal split into the two passes derived above, exp2 with the scale pre-folded into q. The per-block update is exactly the recurrence traced above:

```python
q = tl.load(q_ptrs, mask=offs_m[:, None] < seqlen, other=0.0)
q = (q * (sm_scale * 1.44269504)).to(tl.float16)           # scale + log2(e) folded in once

m_i = tl.zeros([BLOCK_M], dtype=tl.float32) - float("inf")
l_i = tl.zeros([BLOCK_M], dtype=tl.float32) + 1.0           # seed is irrelevant: first alpha = 0
acc = tl.zeros([BLOCK_M, BLOCK_DMODEL], dtype=tl.float32)   # UNNORMALIZED; divide once at the end

for start_n in range(0, non_causal_end, BLOCK_N):           # pass 1: no mask needed
    k = tl.load(k_ptrs, mask=(start_n + offs_n[:, None]) < seqlen, other=0.0)
    qk = tl.dot(q, tl.trans(k))                             # already in log2 units
    m_new = tl.maximum(m_i, tl.max(qk, axis=1))
    alpha = tl.math.exp2(m_i - m_new)                        # max-correction factor
    p = tl.math.exp2(qk - m_new[:, None])
    l_i = l_i * alpha + tl.sum(p, axis=1)
    v = tl.load(v_ptrs, mask=(start_n + offs_n[:, None]) < seqlen, other=0.0)
    acc = acc * alpha[:, None] + tl.dot(p.to(v.dtype), v)    # rescale, no divide
    m_i = m_new
# pass 2 repeats this body over just the diagonal boundary block(s), with
# qk = tl.where(row >= col, qk, -inf) inserted before the max/exp2 above

acc = acc / l_i[:, None]              # the single division, at the very end
m_i = m_i + tl.math.log2(l_i)         # base-2 logsumexp, stored for backward
```

Wrap that with a grid over (row blocks, batch × heads) and the per-head-dim block sizes settled above, and the forward pass is done.

Three levers, all output-preserving: strip non-matmul work out of the inner loop, swap the loops so the sequence dimension fills the SMs, split Q instead of K across warps so no partial ever needs combining. What I haven't done is run the assembled kernel on an A100 against a reference and profile it — the individual pieces check out on paper and on toy numbers, but only that run says whether the achieved TFLOPs/s actually moves from the ~25-40% I started at toward GEMM's 80-90%.
