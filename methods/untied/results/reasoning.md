Let me start from where the vocabulary actually touches the model, because that is the only place this question can live. A language model meets its words twice. On the way in, the current token is a one-hot `c` and I look it up in a matrix `U` of shape `C × H` — one row per vocabulary word, `H` hidden units — to get a dense vector `Uᵀc` that the body consumes. On the way out, after the body has chewed on the history and produced an activation `h ∈ R^H`, I need one real number per vocabulary word, a score, and the cheapest thing that turns `h` into `C` scores is another `C × H` matrix `V`: the logits are `V h`, then softmax, then the next-word probabilities. So I have two matrices of identical shape, both indexed by the same vocabulary, both `C × H`. And the tempting observation is right there: they have the same shape and index the same words, so why carry two? Use one matrix `S` at both ends — look the word up with row `S_k` on the way in, score it with the same row `S_k` on the way out. That instantly removes a `C·H` block of parameters, which for any realistic vocabulary is the single largest chunk of the whole model. Free win, apparently.

But "same shape, same index" is not "same job," and before I commit to sharing I want to know what each end is actually being asked to do. Think about what I want the *input* matrix to capture. When the model reads a word, I want the body to *react* to synonyms in nearly the same way — if "couch" and "sofa" produce almost the same perturbation of the hidden state, then everything downstream generalizes across them for free. So the input embedding wants similar rows for words that play the same role *as inputs to the dynamics*. Now the *output* matrix. `V` is literally the weight matrix of a `C`-way softmax classifier over the next token: the score of word `k` is `V_kᵀ h`, an inner product of word `k`'s row with the current activation. For that classifier to be good, I want words that are *interchangeable as continuations* — words that should be predicted in the same contexts — to have nearby rows, so they get similar scores against any `h`. These two notions of "similar words" rhyme but they are not the same. One is "this word pushes the state the same way when read." The other is "this word should be scored the same way when predicted." A representation tuned for the first is not automatically tuned for the second. So already I am suspicious of forcing one matrix to be both.

I can make the suspicion concrete instead of hand-wavy by looking at how these matrices are actually *trained*, because gradients don't lie about what a parameter is being pushed to become. Take the per-step negative log-likelihood, `L_t = −log p_t(o_t | i_{1:t})`, where `o_t` is the true next word and `p_t(o_t|·) = exp(V_{o_t}ᵀ h) / Σ_x exp(V_xᵀ h)`. Let me differentiate with respect to the two matrices separately, as if they're independent — which is the configuration where they *are* independent. Start with the output matrix `V`, because the loss depends on it directly through the logits. The logit for word `k` is `z_k = V_kᵀ h`, and standard softmax-cross-entropy gives `∂L_t/∂z_k = p_t(k|·) − 1{k = o_t}`. Then `∂L_t/∂V_k = (∂L_t/∂z_k) · (∂z_k/∂V_k) = (p_t(k|·) − 1{k=o_t}) · h`. So row by row: for the true word, `∂L_t/∂V_{o_t} = (p_t(o_t|·) − 1) h`, a pull (the coefficient is negative since `p ≤ 1`) that increases `V_{o_t}ᵀ h`; for every other word `k`, `∂L_t/∂V_k = p_t(k|·) h`, a push that *decreases* its score, with magnitude proportional to how much probability the model currently wastes on it. Every single row of `V` gets a nonzero gradient at every step, because `p_t(k|·) > 0` for all `k`. The whole output matrix is densely trained, all the time.

Now the input matrix `U`, in the same independent setting. The loss depends on `U` only through `h`, and `h` depends on `U` only through the rows that were actually looked up — and at this step only the current input word `i_t` was looked up, so only row `U_{i_t}` is in the computational path. For every other row, `∂L_t/∂U_k = 0` when `k ≠ i_t`. For the row that was used, by the chain rule through `h`, `∂L_t/∂U_{i_t} = (∂L_t/∂h) · (∂h/∂U_{i_t})`, and `∂L_t/∂h = Σ_k (∂L_t/∂z_k)(∂z_k/∂h) = Σ_k (p_t(k|·) − 1{k=o_t}) V_k = Σ_x p_t(x|·) V_xᵀ − V_{o_t}ᵀ` (transposing into row convention), so `∂L_t/∂U_{i_t} = (Σ_x p_t(x|·) V_xᵀ − V_{o_t}ᵀ) · ∂h/∂U_{i_t}`. The shape of this is completely different from the output side. Only *one* row of `U` moves per step — the current input word — and all the others sit frozen. A word that is rare as an *input* gets its input row updated only a handful of times in the entire run, because its row only learns on the steps where that word is the one being read.

So here is the picture I now have, and it's sharper than the intuition I started with. Even if `U` and `V` had the same shape and the same vocabulary index — which they do — they are driven by *structurally different gradient streams*: `V` is dense (every row, every step), `U` is sparse (one row, the current input, per step). These are two different training dynamics. They are not one object wearing two hats; they are two objects being shaped by two different forces, for two different purposes. That's a strong reason to let them be two matrices.

But wait — I should not just assert "two forces so two matrices." I should ask the honest question: what actually happens if I *do* tie them, `S = U = V`? Maybe the two forces combine into something fine. When `S` plays both roles, the gradient on a row of `S` is the sum of its input-role gradient and its output-role gradient. So let me add the cases up carefully. Take a row `k`. Its output-role gradient is always present (dense): if `k` is the true word it's the `(p−1)h` pull, otherwise the `p_t(k|·) h` push. Its input-role gradient is present only if `k = i_t`, the current input word. So for almost every row — every `k ≠ i_t` — the input role contributes *nothing* this step, and the update on `S_k` is exactly the output-role update. Only the single row `k = i_t` gets both contributions added together. Stare at that one special row. It collects an input-role term (the `k = i_t` case of `∂/∂U`) plus an output-role term — and which output-role case applies? Almost always `i_t ≠ o_t`: the current input word is almost never the very next word, because words seldom repeat back to back. So on row `i_t` the output-role term is the `k ≠ o_t` case, `p_t(i_t|·) h` — and `p_t(i_t|·)` is the probability the model assigns to emitting *the word it just read as the next word*, which is tiny, essentially zero, because immediate repetition is rare. So the special row is locally dominated by its input-role term, not by its output-role term.

Add it up across the whole matrix, not just that one row, and something uncomfortable falls out. At a given step, `C−1` rows are output-only; one row gets input plus a nearly negligible output term. For a fixed word, most timesteps are timesteps when that word is *not* the current input, so most of its updates are also output-only; only on the relatively sparse occasions when the word is read does it get the input-role term. So over training, the tied matrix `S` evolves to look much more like what the output embedding `V` would have been. The reason is not that the output term dominates on the current-input row — it does not — but that the output-role update is dense across rows and the input-role update is sparse across time. The input role isn't getting a fair say; tying doesn't *blend* the two representations, it makes the shared matrix track the output role more closely and leaves the input role underrepresented. If I cared about the input representation being good in its own right, tying quietly throws away some of that freedom.

So now I'm leaning hard toward keeping them separate. But I have to be careful, because there's a known result floating around that says sharing the input and output word matrices is actively *bad*, and if that's a law I'd better understand it before I conclude anything. The setting where it's bad is word2vec skip-gram. There the two matrices are a "center" and a "context" matrix, and the argument (Goldberg & Levy) is that a word is rarely its own context. If I force the center and context vectors to be the same vector, the self-score for word `i` is `U_iᵀU_i = ||U_i||²`. The model wants the probability of `i` appearing as its own context to be tiny, so it tries to make that self-score small; because the self-score is a squared norm, the only way to do that directly is to shrink the vector toward zero. So tying is harmful there. Does that condemn tying — or is it specific to skip-gram? The hinge is the body. In skip-gram there is essentially no model body: the "activation" the output matrix scores against *is the input vector itself*, `h ≈ U_{i_t}` (an identity body). That's why the self-prediction pathology bites — the same vector appears on both sides of the inner product. But in a real language model there is a deep body — an LSTM, a transformer stack — between the input lookup and the output score. The activation `h` is a nonlinear function of the *entire history*, not of the current input row directly. `h` is no longer `U_{i_t}`; it's `f(U_{i_1}, …, U_{i_t})` run through layers. So the self-prediction term doesn't transmit the same way: there's no `U_{i_t}ᵀ U_{i_t}` to collapse, because the thing `V` scores against isn't the input row anymore. The body *decouples* the two ends. The word2vec result isn't a law against tying; it's a statement about the identity-body case. With a body in between, that particular norm-collapse argument no longer applies.

That actually reframes the whole question, and it's a relief, because it means tying is not forbidden and untying is not forced — both are admissible, and I get to choose on the merits. So what is the choice, precisely? Tying is a *constraint*: it forces `U = V`, collapsing two matrices into one. Untying is the *unconstrained* configuration: two independent matrices, each free to follow its own gradient stream to its own optimum. The unconstrained version is the more expressive one by definition — it contains the tied solution as the special case `U = V` and everything around it. Tying buys two things: it removes a `C·H` parameter block (the largest in the model), and it imposes a constraint that, like any parameter-sharing constraint, regularizes — it forces every output-classifier row to be the same row that embeds that word on input. Those are benefits when parameter reduction and regularization are the binding needs. Untying gives the opposite trade: it spends the `C·H` parameters so the output classifier can have its own rows — the logit map is no longer the same table as the input lookup, and the output head can carve out predictive directions the input lookup has no reason to represent — at the cost of more parameters to regularize. So the derivation is not "sharing is illegal"; it is "sharing is an extra constraint, and I only impose it when that constraint is worth more than the output-specific capacity it removes."

I can sharpen *why* untying gives real extra capacity rather than redundant parameters, because "more parameters" is not automatically "more useful." Think about what tying actually pins down. When `S = V = U`, the logit for word `k` is `S_kᵀ h`, and `S_k` is *also* the vector that, when word `k` is read, perturbs the state. So a single vector has to simultaneously be a good "how I move the state when read" direction and a good "how I'm scored when predicted" direction. There's no reason the optimal directions for those two coincide — recall the two gradient streams pull `U` and `V` toward different things. Inan et al. give the strongest argument in the other direction: start from an augmented KL loss whose soft target is built from the input embedding, take the high-temperature limit, and zero gradient asks the output logits `Wh_t` to match `Lᵀu_t`. Under the rank assumptions in that calculation, this means the column space of `W` matches the column space of `Lᵀ`, so there exists a square map `A` with `W = LᵀA` and `Wh_t = LᵀAh_t`. That is a real reason to align the spaces; reusing the embedding in the output projection makes the alignment explicit as `W = Lᵀ` and leaves the body to absorb the map `A` into `h`. But "same useful subspace under that augmented objective" is still softer than "the two row tables must be literally identical under ordinary cross-entropy." Untying gives the reconciliation room to be word-specific: the output rows can specialize to the predictive task while the input rows specialize to the reactive task, and the body no longer has to spend capacity undoing a constraint I imposed for free-parameter reasons. If I choose capacity over that regularizer, this is the capacity I am buying.

So my decision is to *untie*: keep the input token embedding and the output projection as two separate matrices. Now let me turn that into the actual code, in the harness I have. The input side is a module that owns the token lookup and hands the output projection a weight; the body and the loss are fixed. The tied default would have the embedding module hand back its own `wte.weight`, so input and output share. To untie, the embedding module needs to own a *separate* parameter of shape `(vocab_size, n_embd)` and return *that* from `get_lm_head_weight()`, leaving `wte` to be the input lookup alone.

One detail I have to get right is initialization of that new output matrix, because the harness wires `lm_head.weight = embedding.get_lm_head_weight()` and then trains it. Picture step zero. Every other module is initialized at the usual small scale (std `0.02` for the GPT-2 family). If I initialize the separate output matrix the same way, the very first logits `V h` are small random numbers — a slightly random initial next-word distribution that the first updates have to spend effort flattening out. There's a cleaner starting point. If I initialize the output matrix to *zero*, the initial logits are exactly `0` for every word, so the softmax is exactly uniform, and the initial loss is exactly `ln(C)` — the maximum-entropy, "I know nothing yet" baseline, with no spurious large logits to fight. From there the output rows grow from zero under their dense per-step gradient (`(p − 1)h` on the true word, `p_t(k|·) h` on the rest), which is well-behaved. So I zero-init the separate output projection, and keep the input embedding `wte` at the standard `normal(0, 0.02)` so the input pathway carries signal from the first step — the input side I *want* to be non-degenerate immediately, because its rows learn slowly (sparse, one-per-step), while the output side I'm happy to grow from a clean uniform start because it learns fast (dense, every step). The asymmetry in the inits mirrors the asymmetry in the gradient streams I derived.

The position embedding and the body I leave exactly as they are: untying is an intervention on the *relationship between the two word matrices*, nothing else. The learned absolute position embedding stays added to the token embedding in `forward`, and it stays excluded from the reported parameter count (so this can't be a back door to buying capacity through positions). There's no per-layer value-embedding residual here, so `get_value_embed` returns `None`. The output projection has no bias, matching the fixed harness — the bias was never part of the tying question.

So the slot fills in like this:

```python
import torch
import torch.nn as nn


class TokenEmbedding(nn.Module):
    """Token + position embedding with a separate output projection: the matrix used
    to score tokens on the way out is not the input lookup."""

    def __init__(self, config):
        super().__init__()
        self.wte = nn.Embedding(config.vocab_size, config.n_embd)   # input lookup (sparse grad)
        self.wpe = nn.Embedding(config.block_size, config.n_embd)   # learned positions
        self.drop = nn.Dropout(config.dropout)
        self.block_size = config.block_size
        self.n_embd = config.n_embd
        self.vocab_size = config.vocab_size
        # Separate output-projection weight — NOT tied to wte.
        self._lm_head_weight = nn.Parameter(torch.empty(config.vocab_size, config.n_embd))
        # Zero-init: initial logits = 0 -> uniform softmax -> loss starts at ln(vocab_size),
        # the max-entropy start; rows grow under the dense (p - 1{k=o_t})*h gradient.
        nn.init.zeros_(self._lm_head_weight)

    def forward(self, idx):
        b, t = idx.size()
        tok_emb = self.wte(idx)                                     # only U_{i_t} learns per step
        pos = torch.arange(0, t, dtype=torch.long, device=idx.device)
        pos_emb = self.wpe(pos)
        return self.drop(tok_emb + pos_emb)                         # (B, T, n_embd)

    def get_lm_head_weight(self):
        return self._lm_head_weight                                 # the OUTPUT matrix, free of wte

    def get_num_pos_params(self):
        return self.wpe.weight.numel()                             # positions excluded from the count

    def get_value_embed(self, layer_idx):
        return None                                                # no per-layer residual here
```

Tracing the chain back: a language model meets its vocabulary at two ends, and the cheap move is to use one matrix for both. But the two ends ask for different things — the input wants synonyms to drive the body alike, the output wants interchangeable continuations to score alike — and the gradients confirm it, because the input matrix is trained by a sparse one-row-per-step signal while the output matrix is trained by a dense every-row signal. Tie them and almost every row-time update is output-only; the single current-input row gets an input-role term, but its output-role term is scaled by the near-zero probability of immediate repetition. So the shared matrix tracks the output role more closely and leaves the input role underrepresented. The word2vec result that sharing is harmful turns out to be about the identity-body case; a real model body decouples the two ends, so tying is admissible but not mandatory — it is a regularizing constraint that removes the model's largest parameter block. Untying is the unconstrained configuration: it spends those parameters to give the output classifier its own degrees of freedom, so the classifier rows no longer have to be the same vectors used by the input lookup. The code is the minimal expression of that: a separate, zero-initialized output-projection parameter returned by `get_lm_head_weight()`, with the input embedding, the positions, and the body left untouched.
