Let me start from what actually hurts. I'm handed five classes I've never seen, with five — sometimes one — labeled images each, and I have to label fresh query images of those classes right now, no retraining. The obvious move, fine-tune a deep net on twenty-five images, is hopeless: millions of weights, a couple dozen examples, it memorizes the support set and tells me nothing about the queries. Yet a person glances at one example of a new character and recognizes it instantly. So the lesson the data scarcity is screaming at me is that the answer cannot be "fit more parameters per task." It has to be a *prior* — an inductive bias so strong and so simple that twenty-five points are enough to pin it down. Everything I design from here should be biased toward the simplest hypothesis that still separates the classes, because every extra knob I add is another thing five images per class cannot constrain.

What's already working that I should not throw away? The episodic idea from the matching-networks line is clearly right, and I want to be precise about why. The failure mode of ordinary training is a mismatch: you train on big balanced minibatches and then test on tiny N-way K-shot tasks, so the objective the network optimized is not the objective it's judged on. The fix is to make training look exactly like testing — sample an episode of N classes, K support examples each, and a query set, and directly optimize the network's accuracy on that query set conditioned on that support set. Then training is a Monte Carlo estimate of the actual few-shot test distribution, and in a regime this data-starved that faithfulness is itself a powerful regularizer: the network is never allowed to rely on anything it won't have at test time. I'm keeping episodic training wholesale. It's the *classifier inside the episode* I need to rethink.

So look hard at what matching networks does inside an episode. It embeds every support and query image, and to classify a query x̂ it takes a weighted vote over support labels, ŷ = Σ_i a(x̂, x_i) y_i, with weights a(x̂, x_i) = exp(c(f(x̂), g(x_i))) / Σ_j exp(c(f(x̂), g(x_j))) — a softmax over the cosine similarity between the embedded query and each embedded support point. Read structurally, that's a differentiable weighted nearest-neighbor classifier living in the embedding space, and it's lovely because it's nonparametric: no per-class weights to fit, so nothing to overfit. But stare at it from the standpoint of "what does the model carry around to make a prediction." It keeps *every individual support point* and assigns *every one of them* its own attention weight. The prediction for a query is a sum over all support examples; the state the classifier holds grows with the number of support points. With five-shot that's a vote over twenty-five separate kernels, and there is no notion of a class as a single thing — a class is just whichever scattered support points happen to wear its label. That feels like more representational machinery than the problem deserves. And then on top of that the strong variant, full context embeddings, runs a bidirectional LSTM over the support set to contextualize the embeddings — which both adds parameters I can't afford to fit on twenty-five images *and* imposes an arbitrary sequential ordering on a set that has no order. The bias is going the wrong way: toward more complexity, when the data is begging for less.

So let me push on the thing that nags me: do I really need one weight per support point? What is the *minimal* summary of a class I could compare a query against? If I had to describe a class with a single object — independent of whether it has one example or fifty — the most natural single object for a cloud of vectors is its center. Take the mean of the embedded support points for the class. One vector per class, no matter how many examples, nothing to fit. Call it the prototype, c_k = (1/|S_k|) Σ_{(x_i,y_i)∈S_k} f_φ(x_i). Then classify a query by which prototype it's closest to. That immediately fixes the two things that bothered me: the per-class state is one vector regardless of shot, and a "class" is now a concrete entity, not a bag of points. The nearest-class-mean idea isn't new — there's prior work that represents each class by the mean of its examples and classifies by nearest mean, exactly so you can drop in a new class at near-zero cost by just averaging its examples. But that work used a *linear* embedding (a learned Mahalanobis metric on fixed features) and was built for the regime where each class brings many examples; and when it wanted nonlinearity it had to bolt on a separate k-means partitioning step in input space, decoupled from the metric learning. I want the embedding itself to be a deep nonlinear network, learned end-to-end by the episodic objective, with the mean computed *in the learned space*. So: a deep f_φ, prototypes = means of embedded support, classify by distance — and train the whole thing episodically so f_φ learns to make the means separable.

Now I have to turn "nearest prototype" into something differentiable I can train. A hard argmin over distances has no gradient. The standard cure is a softmax: define a distribution over classes for a query x as p_φ(y = k | x) = exp(−d(f_φ(x), c_k)) / Σ_{k'} exp(−d(f_φ(x), c_{k'})), and minimize the negative log-probability of the true class, J(φ) = −log p_φ(y = k | x), averaged over the query points of an episode. The closer the query embedding is to the right prototype relative to the others, the smaller the loss; gradients flow through the distances, through the prototypes, into f_φ. That's clean and it's just a cross-entropy on logits that happen to be negative distances. Notice I want the negative log of p over the *correct* class k, and the normalizer runs over all classes k′ — let me be careful not to fuse those indices, the term in the numerator's exponent uses c_k and the denominator sums exp(−d(·, c_{k'})) over every k′ including k.

But which d? Cosine is the default in the matching-networks line, and it would be the lazy choice. Let me not pick by habit. I have two free design objects now — the distance and the prototype — and I just committed to making the prototype the *mean*. So the right question is: for which distances is "the mean" actually the right representative of a class, in the sense of being the point that best stands in for all the support points? That's a precise question and it has a precise answer. Consider any class as a little clustering problem: I have its support embeddings, I want one representative point, and "best representative" should mean the point minimizing the total distance to the class's points. So I'm asking: for which d is argmin_c Σ_{x∈S_k} d(f(x), c) equal to the mean of the f(x)? There's a known characterization that points at the Bregman divergences — d_φ(z, z′) = φ(z) − φ(z′) − (z − z′)ᵀ∇φ(z′) for a strictly convex φ — but I don't want to take it on faith, so let me differentiate and see for myself. Fix the set and the divergence and differentiate g(c) = Σ_x d_φ(x, c) with respect to c. Inside, d_φ(x, c) = φ(x) − φ(c) − (x − c)ᵀ∇φ(c). The φ(x) term doesn't depend on c. For h(c) = (x − c)ᵀ∇φ(c), the product rule gives ∇h(c) = −∇φ(c) + ∇²φ(c)(x − c): the first term comes from differentiating x − c, and the Hessian term comes from differentiating ∇φ(c). Therefore ∇_c d_φ(x, c) = −∇φ(c) − ∇h(c) = −∇²φ(c)(x − c) = ∇²φ(c)(c − x). Summing over x gives ∇g(c) = ∇²φ(c)(|S_k|c − Σ_x x). Under the regular strict-convexity assumptions, the Hessian is positive definite, so the gradient vanishes exactly when |S_k|c = Σ_x x, i.e. c = (1/|S_k|) Σ_x x. The mean. Let me put numbers on the φ = ‖·‖² case to be sure I didn't slip a sign: with the three-point cluster {(2,0),(0,2),(3,1)} the mean is (5/3, 1), and the gradient of Σ_x‖x − c‖² is 2(3c − Σx) = 2(3·(5/3,1) − (5,3)) = 2((5,3) − (5,3)) = (0,0) — it does vanish at the mean. So for a Bregman divergence the mean is the stationary (and, by convexity, minimizing) representative, which is the prototype I already chose. There's a converse worth keeping in mind: among smooth distortions, this mean-as-minimizer property singles out the Bregman divergences; nothing outside that family makes the mean the right summary. So if I want the mean to be a defensible class summary, I'd better pick a Bregman divergence — otherwise I'm averaging points that the distance doesn't agree the average represents.

Is cosine a Bregman divergence? Cosine distance, 1 − ⟨z,z′⟩/(‖z‖‖z′‖), doesn't look like one — it cares only about angle and throws away magnitude — but rather than wave my hands I should check whether its best-representative is the mean, since that's the property I actually need. Take the same cluster {(2,0),(0,2),(3,1)}, whose mean (5/3, 1) points at angle 31.0°. Cosine ignores the length of the candidate center, so I only have to search over its direction: minimize Σ_x [1 − cos(angle between x and the direction θ)] over θ in the first quadrant. Sweeping θ, the total bottoms out at θ ≈ 34.0°, with direction ≈ (0.829, 0.560) — not the mean's (0.857, 0.514) and not the mean's 31.0° angle. So cosine's optimal class representative is a *different* point from the mean (it tilts toward the longer vectors, which dominate the unnormalized inner products). That settles it: if I average embeddings to make a prototype and then score by cosine, I'm being inconsistent — I summarized the class with the mean, but cosine doesn't agree the mean is the right summary. Squared Euclidean, d(z, z′) = ‖z − z′‖², on the other hand *is* a Bregman divergence — take φ(z) = ‖z‖², then φ(z) − φ(z′) − (z − z′)ᵀ∇φ(z′) = ‖z‖² − ‖z′‖² − (z − z′)ᵀ(2z′) = ‖z‖² − ‖z′‖² − 2zᵀz′ + 2‖z′‖² = ‖z‖² − 2zᵀz′ + ‖z′‖² = ‖z − z′‖², and I checked above its representative *is* the mean. This gives me a concrete empirical prediction to test on the benchmarks: for a mean-prototype classifier, squared Euclidean should improve over cosine, and the gap should be *larger* here than for the point-wise matching classifier, because the averaging step is exactly where cosine is inconsistent. I'll use squared Euclidean.

There's a deeper reading of "squared Euclidean is a Bregman divergence" that tells me what modeling assumption I'm silently making, and I want to surface it because it's the kind of thing that bites you later. There's a bijection between regular exponential-family densities and regular Bregman divergences: any p_ψ(z | θ) = exp{zᵀθ − ψ(θ) − g_ψ(z)} can be written as exp{−d_φ(z, μ(θ)) − g_φ(z)} for the Bregman divergence d_φ paired with that family. So consider modeling each class in embedding space as one component of an equally-weighted mixture, p(z | Γ) = Σ_k π_k p_ψ(z | θ_k) with π_k = 1/K and the component mean μ(θ_k) = c_k. The posterior over which component generated z is p(y = k | z) = π_k exp(−d_φ(z, c_k)) / Σ_{k'} π_{k'} exp(−d_φ(z, c_{k'})), and with equal π_k those cancel, leaving exactly the softmax-over-negative-divergence-to-prototypes that I wrote down for classification. So my classifier *is* inference in an exponential-family mixture with one component per class, and the choice of d picks the class-conditional density. Squared Euclidean corresponds to a spherical Gaussian family with a common fixed scale, up to the convention-dependent constant multiplying the distance. That's a strong, simple assumption — every class is a round blob of the same size around its mean — and in the few-shot regime "strong and simple" is exactly the bias I argued I want. It also flags the natural generalization (pick a different Bregman divergence for a different class-conditional family, e.g. a diagonal-Gaussian with per-dimension variances) so I know where to push if spherical turns out too rigid. I'll come back to that.

Before I lock in squared Euclidean I should worry about one thing: it's a *quadratic* in the embedding — am I accidentally building a weak classifier? Let me expand the exponent and look. −‖f_φ(x) − c_k‖² = −f_φ(x)ᵀf_φ(x) + 2 c_kᵀ f_φ(x) − c_kᵀ c_k. The first term, −f_φ(x)ᵀf_φ(x), is the same for every class k — it doesn't depend on k — so when I put this through the softmax over k it should cancel top and bottom, leaving 2 c_kᵀ f_φ(x) − c_kᵀ c_k, which is *linear* in f_φ(x): write it as w_kᵀ f_φ(x) + b_k with w_k = 2c_k and b_k = −c_kᵀc_k. "Cancels in the softmax" is the kind of claim I've been burned by before, so let me actually run both sides on random numbers instead of trusting the algebra. Pick a 4-dimensional embedding, three random prototypes, a random query f. Softmax of the full logits −‖f − c_k‖² gives (0.0385, 0.3090, 0.6525). Softmax of the reduced linear logits w_kᵀf + b_k with w_k = 2c_k, b_k = −‖c_k‖² gives (0.0385, 0.3090, 0.6525) — identical to all the digits I printed. So the quadratic and the induced-linear classifier are the *same* classifier; dropping the shared −f_φ(x)ᵀf_φ(x) term genuinely leaves a plain linear head on top of the embedding, with weights and biases read directly off the prototypes — no extra parameters, the linear layer is *induced* by the means. At first that worried me — "only linear?" — but it's actually the right amount of expressiveness for this regime. A linear head has almost nothing to overfit (its weights are determined by five-image means, not free parameters), and all the nonlinearity I need lives in f_φ, the deep embedding, which is trained across thousands of episodes where it *can* be constrained. That's the same division of labor every modern deep classifier uses — a powerful nonlinear feature extractor and a linear readout — so I'm not giving anything up; I'm putting the capacity where the data can pay for it and keeping the per-task head parameter-free.

Now a sanity check that I haven't strayed from the method I'm reacting to. What happens in the one-shot case, |S_k| = 1? Then the prototype c_k is just the single support embedding f_φ(x_k) — averaging one point returns the point, which I can see immediately: the mean of {z} is z. So my classifier becomes "softmax over negative distance from the query to each single support embedding," which is a (soft) nearest-neighbor over support points — the same shape the matching-networks classifier takes in one-shot (the only difference being its cosine vs. my squared-Euclidean distance, and that whole question I've already settled). So with a fixed embedding the two methods make the same decision when there's one example per class, modulo the distance. That's reassuring: I haven't invented something disconnected from the baseline, I've found a multi-shot *generalization* where, instead of voting over individual points, I first collapse each class to its mean and vote over classes. The methods only diverge at K > 1, where I summarize and they don't — and the summary is the whole point.

Should I allow more than one prototype per class — say cluster each class into a few sub-prototypes to capture multimodal classes? It's tempting, but think about what it costs. With a fixed number of prototypes greater than one per class I'd need a *partitioning* of each class's support points into sub-clusters, and the only ways I know to do that (k-means, or the multi-modal extensions in the nearest-class-mean line) are a *separate* step, decoupled from the gradient updates of f_φ. That breaks the thing I value most: end-to-end training by ordinary SGD on the episodic loss. One prototype per class keeps the entire method differentiable and learnable in one loop, and the deep embedding can already warp space so a class becomes unimodal around its mean. So: one prototype per class, and let f_φ do the work of making that adequate. Same logic kills FCE — a per-episode contextual re-embedding would add LSTM parameters and an arbitrary set ordering, and in a regime where twenty-five images can't even fit a linear head's worth of free parameters, learning a custom embedding per episode is exactly the over-parameterization I'm trying to avoid. The simple inductive bias is the feature, not a compromise.

Let me also pin down the training objective concretely as it runs per episode, because the way episodes are built is itself a design choice. I sample N_C classes for the episode, and within each chosen class I split its examples into a support set S_k of N_S points and a query set Q_k of N_Q points (disjoint). I compute each prototype from the support points only, c_k = (1/N_S) Σ_{(x_i,y_i)∈S_k} f_φ(x_i) — and I should be careful here, the normalizer is the number of *support* points N_S = |S_k|, not the number of classes; it's the per-class average. Then for every query point (x, y) with true class k I accumulate the loss (1/(N_C N_Q)) [ d(f_φ(x), c_k) + log Σ_{k'} exp(−d(f_φ(x), c_{k'})) ], which is exactly −log p_φ(y = k | x): the d(f_φ(x), c_k) is the negative of the numerator's exponent and the log-sum-exp is the log normalizer over all classes k′. Average over the query points, backprop, step. That's the whole episode.

Two episode-composition choices remain, and I'll reason them out rather than guess. First, how many classes per training episode — should training "way" match test "way"? The naive answer is yes, match the test condition exactly. But think about what the embedding has to learn. If I train on 5-way episodes, f_φ only ever has to make 5 prototypes mutually separable at a time; if I train on 20-way or 30-way episodes, it has to keep many more prototypes simultaneously separable, which is a strictly harder constraint on the geometry of the embedding. A harder training task forces finer-grained distinctions into f_φ, and an embedding that can separate 30 classes at once will comfortably separate 5 at test time. So I expect training with *higher* way than test to help, and I'll treat training way as something to tune upward on a validation set rather than pin to the test way. Second, the shot: here the asymmetry argument doesn't apply the same way, because the prototype's statistical character changes with N_S — a one-shot prototype is a raw embedding, a five-shot prototype is an average of five, and these live at different noise levels. So I want the network to see prototypes at test-time noise during training, which means matching the training shot to the test shot. Way: train harder (higher). Shot: train matched.

One loose end on the modeling side: the spherical-Gaussian assumption baked into squared Euclidean says every class is a round, equal-size blob. The Bregman/exponential-family bijection tells me exactly how to relax it — switch to a Bregman divergence corresponding to, say, a diagonal Gaussian, which amounts to learning a per-dimension (and possibly per-class) variance and scaling the squared differences by its inverse. I'd want to try fitting a variance per dimension per class. But I should predict the likely outcome from my own setup: the embedding f_φ is already free to rescale and rotate space however it likes, so it can *absorb* per-dimension variance into the embedding itself — there's a redundancy between "let f_φ stretch the axes" and "fit a separate per-axis variance." When two parts of a model can express the same thing, adding the second usually buys nothing and costs parameters I can't afford here. So my expectation is that the extra per-class variance parameters won't help and the spherical assumption is enough; I'd keep them out unless validation says otherwise. That keeps the method at its minimal, parameter-free-head form.

There's a zero-shot variant that falls out almost for free, and it's a good test of whether the prototype abstraction is really the right level. In zero-shot I'm not given support *images* of the new class — I'm given class meta-data, an attribute vector v_k describing the class. But my classifier only ever needed a *prototype* per class; it never cared where the prototype came from. So I just produce the prototype by embedding the meta-data with a separate learned map, c_k = g_ϑ(v_k), and everything downstream — softmax over squared distance to prototypes, episodic NLL training — is unchanged. The only wrinkle is that the meta-data and the images live in different domains, so the two embeddings f (query, image) and g (prototype, attributes) aren't naturally on the same scale; empirically it helps to constrain the prototype embedding g to unit length while leaving the query embedding f free. That the same machinery handles both few-shot and zero-shot by swapping how the prototype is built is exactly the sign that "summarize each class by one prototype, classify by distance" was the right primitive.

Let me now write the method as it runs and then as code, filling the three empty slots in the harness. Per episode the loss computation is:

```
Input: episode classes V (|V| = N_C); for each k in V, support S_k (N_S each) and query Q_k (N_Q each)
for k in V:
    c_k = (1 / N_S) * sum_{(x_i, y_i) in S_k} f_phi(x_i)          # prototype = mean of embedded support
J = 0
for k in V:
    for (x, y) in Q_k:                                            # y == k
        # NLL of the true class under softmax over -squared-distance to prototypes
        J += (1 / (N_C * N_Q)) * [ ||f_phi(x) - c_k||^2 + log sum_{k'} exp(-||f_phi(x) - c_{k'}||^2) ]
return J
```

And dropped into the episodic classifier harness, the three slots become: summarize the support set into prototypes (the per-class mean of embedded support), score queries by negative squared distance to those prototypes, and train with cross-entropy on those scores (which is exactly the NLL above, since cross-entropy applies log-softmax to the logits).

```python
import torch
import torch.nn.functional as F
from torch import Tensor


class CustomFewShotMethod(FewShotClassifier):
    """Prototypical Networks: each class is summarized by the mean of its embedded
    support examples (its prototype); a query is classified by a softmax over its
    (negative squared) Euclidean distance to the prototypes."""

    def __init__(self):
        backbone = make_backbone(use_pooling=True)        # deep nonlinear embedding f_phi
        # use_softmax=False: return raw negative squared distances as logits; cross-entropy log-softmaxes them
        super().__init__(backbone=backbone, use_softmax=False)

    def process_support_set(self, support_images: Tensor, support_labels: Tensor):
        support_features = self.compute_features(support_images)
        self._raise_error_if_features_are_multi_dimensional(support_features)
        # Episode labels are local 0..n_way-1; prototype c_k is the mean support feature for class k.
        n_way = int(torch.unique(support_labels).numel())
        self.prototypes = torch.stack([
            support_features[support_labels == label].mean(dim=0)
            for label in range(n_way)
        ])

    def forward(self, query_images: Tensor) -> Tensor:
        query_features = self.compute_features(query_images)        # f_phi(query)
        self._raise_error_if_features_are_multi_dimensional(query_features)
        # logits = -squared distance to each prototype; argmax = nearest prototype.
        # The quadratic -||.||^2 reduces to a linear head after the query-only term cancels in softmax.
        sq_dist = ((query_features[:, None, :] - self.prototypes[None, :, :]) ** 2).sum(dim=2)
        scores = -sq_dist
        return self.softmax_if_specified(scores)

    @staticmethod
    def is_transductive() -> bool:
        # queries are classified independently; no query-query information is shared
        return False

    def compute_loss(self, scores: Tensor, labels: Tensor) -> Tensor:
        # -log p_phi(y=k|x): cross-entropy on negative squared-distance logits == the episodic NLL
        return F.cross_entropy(scores, labels)
```
