## Research question

Given a set of objects in a multidimensional feature space and a distance function over them,
assign to each object a meaningful measure of how *exceptional* it is — how much it looks like
it was generated by a different mechanism than the bulk of the data — without labels and without
assuming a parametric distribution.

Real datasets are not homogeneous: they contain several groups of points with very different
internal densities — a tight, dense cluster next to a loose, sparse one, perhaps with a few stray
points in between. How should a detector quantify exceptionality on such data?

## Background

By the late 1990s, knowledge discovery in databases (KDD) had matured around finding *frequent*
structure — association rules, classification, clustering. Outliers were the converse problem
and had received far less attention; in much of that work they appeared only as "noise" to be
tolerated or discarded so the main patterns came out cleanly. Yet for many applications — credit
card and telecom fraud, criminal activity in e-commerce, surprising performance statistics of
athletes — the rare, exceptional case is the whole point. "One person's noise is another
person's signal."

The intuitive definition everyone reached for is Hawkins's (1980): an outlier is an observation
that deviates so much from the others as to arouse suspicion that it was generated by a
different mechanism. Evocative, but not operational, and not obviously multidimensional. Three
families of methods tried to make it concrete:

- **Statistical, distribution-based (discordancy tests).** Fit a standard distribution (normal,
  Poisson, …) to the data and flag points in its low-probability tail; over a hundred such
  discordancy tests exist for different distributions and outlier types (Barnett & Lewis).
  Most of these tests are *univariate* (one attribute at a time) and *distribution-based*: in a
  real multidimensional dataset you usually do not know the generating distribution. The verdict
  is binary — a point is, or is not, discordant.

- **Depth-based.** Organize the data into layers by some notion of depth (e.g. convex-hull
  peeling, Tukey depth), expecting outliers in the shallow layers. This sidesteps distribution
  fitting and in principle handles many dimensions. It relies on computing
  `k`-dimensional convex hulls, whose lower-bound complexity is `Ω(N^{⌈k/2⌉})`; the published
  algorithms are only acceptable for `k ≤ 2`, maybe 3. Again binary.

- **Distance-based** (detailed under Baselines). A point is an outlier if "most" other points
  are "far" from it. This generalizes the statistical tests, scales with dimension, and needs
  only a metric.

There is also a body of work on density-based *clustering* (DBSCAN and relatives such as CLARANS,
BIRCH, STING, WaveCluster, DenClue, and CLIQUE). These can tolerate
exceptions, but only as a by-product: their objective is to find clusters, the points left over
are lumped together as "noise," and that noise/not-noise verdict is binary and tied to a single
global density threshold (DBSCAN's `MinPts` and `ε`). They quantify nothing about *how* outlying
a point is. What is reusable from them is the idea of using a count of nearby objects — `MinPts` —
to define a neighborhood without committing to a particular volume in advance.

## Baselines

**Distance-based outliers — DB(p, D) (Knorr & Ng, VLDB 1998).** The notion that made Hawkins
operational and multidimensional. An object `O` in a dataset `T` is a `DB(p, D)`-outlier if at
least a fraction `p` of the objects in `T` lie at distance greater than `D` from `O`:

```
O is a DB(p, D)-outlier  ⇔  |{ q ∈ T : d(O, q) ≤ D }|  ≤  (1 − p)·|T|.
```

This is appealing and powerful. It needs only a metric, so unlike the discordancy tests it is
not restricted to one attribute or a known distribution, and unlike the depth methods it does
not build convex hulls, so it is computationally feasible for `k ≥ 5` and beyond. It even
*unifies* the statistical tests: for a normal distribution, being a `3σ` outlier coincides with
being a `DB(0.9988, 0.13σ)`-outlier; for a Poisson(`μ=3`), with a `DB(0.9892, 1)`-outlier — so
the right `p`, `D` recover the classical tail tests, while the same definition keeps working when
no distribution fits. The distance-based algorithms include an index-based and a nested-loop
method at `O(k·N²)`,
and a cell-based algorithm linear in `N` (exponential in `k`, best for `k ≤ 4`). The verdict is
binary — an object either is or is not a `DB(p, D)`-outlier, with no degree of exceptionality
to rank by. The user must choose both `p` and `D`, and `D` is a *single distance applied
to the whole dataset*.

**`Dᵏₙ` outliers — rank by distance to the k-th neighbor (Ramaswamy, Rastogi, Shim).** A
refinement of the distance-based view that removes one parameter and adds a ranking.
For a point `p`, let `Dᵏ(p)` be the distance from `p` to its `k`-th nearest neighbor; declare
the `n` points with the largest `Dᵏ(p)` to be the outliers:

```
p is a Dᵏₙ-outlier  ⇔  at most n − 1 points p′ have  Dᵏ(p′) > Dᵏ(p).
```

`Dᵏ(p)` is a clean local-sparsity proxy — a point in a sparse neighborhood has to reach far for
its `k`-th neighbor — and ranking by it both supplies the "strength" ordering that `DB(p, D)`
lacked and replaces the awkward distance `D` with a more intuitive "how many outliers do I
want" count `n`. The associated pruning algorithm scales well in `N` and `k`.

## Evaluation settings

The yardsticks already in use for outlier work of this kind:

- **Low-dimensional synthetic datasets** with deliberately mixed structure — e.g. a Gaussian
  cluster plus uniform clusters of different densities, with a few planted strays — used to
  visualize and sanity-check what a detector flags, since ground truth is known by construction.
- **Real performance-statistics databases** where a domain expert can confirm whether a flagged
  case is genuinely exceptional: National Hockey League player statistics (used by the
  distance-based work, e.g. 3-D subspaces of points/plus-minus/penalty-minutes, and
  goals/shooting-percentage), and German Bundesliga soccer-player records (games played, goals
  per game, position), where the "outliers" should correspond to recognizably special players.
- **Higher-dimensional real data** to test that the definitions stay meaningful as dimension
  grows — e.g. ~64-dimensional color-histogram feature vectors extracted from images.
- **Performance protocol:** wall-clock runtime of the neighborhood computation as a function of
  dataset size `N` and dimension, typically with a spatial index (e.g. an X-tree / R-tree
  variant) materializing the `k`-nearest neighbors, since the neighbor search dominates cost.

Neighborhood-size choices are naturally swept when comparing nearest-neighbor detectors, and the
qualitative behavior of the resulting rankings is inspected rather than reduced to a single tuned
setting.

## Code framework

A detector here receives an unlabeled feature matrix and a distance metric, builds a neighbor
index, and assigns each point a real score (higher = more anomalous). Everything the substrate
needs already exists: a `k`-nearest-neighbor engine (`sklearn.neighbors.NearestNeighbors`, with
KD-tree / ball-tree / brute-force backends) that returns, for every point, its neighbors and their
distances; `numpy` for the elementwise arithmetic; and whatever feature scaling the upstream
pipeline chooses before distances are computed.

```python
import numpy as np
from sklearn.neighbors import NearestNeighbors


class AnomalyDetector:
    """Fits on unlabeled features and assigns raw / thresholded / anomaly scores.
    The neighbor engine and arithmetic already exist; the scoring rule does not."""

    def __init__(
        self,
        n_neighbors=20,
        *,
        algorithm="auto",
        leaf_size=30,
        metric="minkowski",
        p=2,
        metric_params=None,
        n_jobs=None,
    ):
        self.n_neighbors = n_neighbors          # neighborhood size (count of nearby objects)
        self.algorithm = algorithm
        self.leaf_size = leaf_size
        self.metric = metric
        self.p = p
        self.metric_params = metric_params
        self.n_jobs = n_jobs
        self._nn = None
        self._k = None
        self._train_state = None

    def fit(self, X):
        # Build the neighbor index on the training features (no labels used).
        X = np.asarray(X, dtype=float)
        k = max(1, min(self.n_neighbors, len(X) - 1))
        self._k = k
        self._nn = NearestNeighbors(
            n_neighbors=k + 1,
            algorithm=self.algorithm,
            leaf_size=self.leaf_size,
            metric=self.metric,
            p=self.p,
            metric_params=self.metric_params,
            n_jobs=self.n_jobs,
        ).fit(X)
        # distances[i], indices[i]: the k nearest training points of training point i,
        # excluding the point itself.
        distances, indices = self._nn.kneighbors(X)
        distances, indices = distances[:, 1:], indices[:, 1:]
        # TODO: from the neighbor distances/indices, build whatever per-point training
        #       statistics the scoring rule we will design needs, and store them.
        self._train_state = None
        return self

    def score_samples(self, X):
        # k nearest TRAINING points of each query point
        distances, indices = self._nn.kneighbors(X, n_neighbors=self._k)
        # TODO: the scoring rule we will design.
        #       Turn the query points' neighbor distances/indices and the stored training
        #       statistics into raw per-point scores.
        raw_scores = np.zeros(len(X))
        return raw_scores

    def decision_function(self, X):
        # TODO: shift or threshold the raw scores if the API needs a decision score.
        decision_scores = self.score_samples(X)
        return decision_scores

    def anomaly_score(self, X):
        # TODO: map raw scores to the calling convention higher = more anomalous.
        scores = np.zeros(len(X))
        return scores
```

The neighbor engine hands over each point's `k` nearest distances and the indices of those
neighbors; `fit` may precompute and store per-point training statistics, and the scoring methods
turn a query point's neighborhood, together with those stored statistics, into raw, thresholded,
or higher-is-more-anomalous scores.
