I start with the part that is actually hard: detection is not classification with a different label set. Classification lets me collapse the whole image to one decision, but detection asks for every object and for boxes that pass a geometric overlap test. The CNN result on ImageNet is too strong to ignore, yet it arrives from a setting with abundant image-level labels and no need to localize. If I simply train the same high-capacity network from scratch on boxed detection data, I expect overfitting. If I simply run it as a dense sliding-window classifier, I inherit a localization problem from the network's own geometry. So I need to separate the two difficulties: get good candidate locations somehow, and use the CNN where it is strongest, as a recognizer of visual content.

The first tempting path is to regress object coordinates directly from an image. That sounds clean because the output is already a box. But the signal is too global and too ambiguous: an image may contain several instances, the network has to decide how many boxes to emit, and concurrent deep regression results on VOC are not competitive enough to make this the main route. Direct regression can be useful later as a correction, but I do not want it to be the whole localization mechanism.

The second familiar path is sliding windows. I could scan many positions and scales, crop each window, and classify it. Shallow CNN detectors have worked like this for constrained categories because their feature maps retain enough spatial resolution. A deeper ImageNet-style CNN is different. High layers have large receptive fields and a coarse stride in the input. A unit near the top is responding to a large image patch, and nearby top-layer positions are not separated by a pixel or two. That coarseness is useful for recognition invariance but bad for precise window placement. I could make the network shallower to recover resolution, but then I give up the learned hierarchy that motivated the whole attempt.

The third path is more promising: do not ask the CNN to search all positions. Let a category-independent process propose regions that may contain objects, then classify those regions. This "recognition using regions" view changes the division of labor. A proposal algorithm handles "where might an object be?" without knowing the class. The CNN handles "what does this region look like?" That is the right split because region proposals can be high-recall and cheap, while the CNN can spend its capacity on visual recognition.

Selective search is a good proposal source for this role. It begins with an oversegmentation, then greedily merges neighboring regions that look compatible. The similarity is deliberately plural: color, texture, size, and fill terms catch different reasons that pieces of an object should be grouped. Running variants over several color spaces and similarity combinations gives diversity. The output is a few thousand class-independent boxes in fast mode, not hundreds of thousands of sliding windows. It is also useful scientifically because prior region-based systems used the same proposal family, so a comparison can isolate the representation change.

Now I have candidate rectangles, but the CNN accepts a fixed-size image. I need a transform from an arbitrary rectangle to a 227 x 227 input. I can preserve aspect ratio by enclosing the proposal in a tight square, but that may include a lot of background. I can mask out the outside context, but that creates an artificial input unlike natural ImageNet crops. Or I can anisotropically warp the rectangle to the fixed size, accepting shape distortion in exchange for filling the CNN input with the candidate. I also need a small amount of context around the proposal because objects are not recognized in isolation. The practical choice is the simple one: warp the proposal after expanding it so that the original box has a border of p = 16 pixels in the transformed coordinate frame. If the expanded source rectangle crosses the image boundary, the missing pixels are image-mean pixels before mean subtraction, which become zeros after mean subtraction. I have to be precise here: the padding is defined at the target scale, so the source crop is expanded by the factor 227 / (227 - 2p), with rounding and clipping, rather than by a naive p/227 fraction of the original box.

With proposals and warping in place, the detection architecture is almost forced. For each image, generate roughly two thousand boxes. For each box, crop, pad, warp, subtract the mean, and forward it through the CNN. Take a fixed-length feature vector, normally fc7 for detection, and train one linear classifier per object class. This shares the expensive CNN work across all classes: one feature matrix for the image, one matrix product with the SVM weight matrix, then per-class suppression of duplicate boxes.

The next problem is training the CNN. Detection data alone is too small for a large network, but ImageNet classification gives me a supervised auxiliary task with enough labels to learn general visual features. So I first use a CNN trained on ILSVRC classification. Then I adapt it to the detection domain by replacing the ImageNet 1000-way classification layer with an N+1-way layer for N object classes plus background and continuing SGD on warped proposal windows. I lower the learning rate to 0.001 so fine-tuning adjusts the initialization without destroying it.

Fine-tuning needs labels for proposal windows. I want many positives because I am updating the entire network, not just a small classifier. So for this stage I use a loose rule: assign each proposal to the ground-truth box with maximum IoU, and if that overlap is at least 0.5, label the proposal as that box's class; otherwise label it as background. This creates many jittered positive examples around each object. Since positives are rare among proposals, each minibatch is biased toward them: 32 positive windows and 96 background windows in a batch of 128. The loose label rule is not ideal for precise localization, but it gives the network enough examples to adapt its features to warped object regions.

After fine-tuning, I still do not want to rely only on the final softmax as the detector. The softmax has been trained with loose positives and randomly sampled background windows. A detection classifier has a different job. For a class like car, the clean positives are the actual ground-truth boxes, and the important negatives include background and near misses that the current classifier confuses with the class. So I train separate linear SVMs on the fixed CNN features. For each class, positives are only ground-truth boxes of that class. Negatives are proposals whose overlap with every instance of that class is below 0.3. Proposals in the grey zone are ignored for that class. The 0.3 threshold is not arbitrary decoration; validation shows that moving it to 0.5 or to 0 hurts AP by several points, because the classifier either accepts too many imprecise near positives or treats too many ambiguous regions as hard negatives.

Hard-negative mining is necessary because most proposal windows are background and many are trivially easy. I do not need to keep every easy negative in memory. I start with low-overlap negatives, train the linear SVM, scan images, add only negatives that violate the margin or score too high, retrain as needed, and evict negatives that become safely easy. This is inherited from the detection tradition, but it fits the CNN feature setting well: the feature cache is large, so training should spend memory on confusing examples.

At test time the classifier side is straightforward. I run selective search, warp each proposal, compute CNN features, scale features the same way as in training, and score every proposal for every class with the SVMs. Many proposals overlap the same object, so I apply greedy non-maximum suppression independently per class. The threshold can be learned or fixed in a demo, but the operation is the same: sort by score, keep the best remaining box, and discard lower-scored boxes with too much IoU overlap.

The remaining errors tell me what to fix next. If the CNN features are doing their job, many false positives should not be pure background confusions; they should be boxes that are on or near the right object but not tight enough. That is exactly the regime where regression becomes sensible. I rejected direct box regression as the whole detector, but I can use regression as a small correction from an already plausible proposal to a nearby ground-truth box.

The box transform has to be scale-invariant. Let P = (P_x, P_y, P_w, P_h) be a proposal in center, width, height coordinates, and let G = (G_x, G_y, G_w, G_h) be its assigned ground-truth box. A raw pixel shift is the wrong target because the same 10-pixel error means different things for a tiny box and a large box. So I define center targets as fractions of the proposal size: t_x = (G_x - P_x) / P_w and t_y = (G_y - P_y) / P_h. Width and height should be multiplicative, not additive, because doubling and halving should be symmetric and predicted sizes must stay positive. So I use t_w = log(G_w / P_w) and t_h = log(G_h / P_h).

I learn four functions d_x(P), d_y(P), d_w(P), and d_h(P). Each is linear in the proposal's pool5 feature vector. Once those functions predict a correction, I invert the parameterization: Ghat_x = P_w d_x(P) + P_x, Ghat_y = P_h d_y(P) + P_y, Ghat_w = P_w exp(d_w(P)), and Ghat_h = P_h exp(d_h(P)). The signs check out. If d_x equals (G_x - P_x) / P_w, then P_w d_x + P_x equals G_x. If d_w equals log(G_w / P_w), then P_w exp(d_w) equals G_w. There is no missing negative sign, no missing scale constant, and the exponential is on the width and height corrections only.

The regression fit itself is ridge regression with lambda set by validation, and the objective is the standard sum of squared target errors plus lambda times the squared weight norm. The implementation adds practical conditioning: it scales the CNN features, appends a bias feature, centers and decorrelates the four regression targets before fitting, then unwhitens predictions at test time. Those steps do not change the target definitions or the inverse transform; they make the linear solves better behaved.

I also need to be careful about which training pairs the regressor sees. If a proposal is nowhere near any object, asking a linear model to move it across the image to a ground-truth box is meaningless. So I assign a proposal to its max-overlap ground-truth box only if the overlap is above 0.6, discard unassigned proposals, and train class-specific regressors. At test time I predict once for each scored detection, clip the adjusted coordinates to the image, keep the original SVM score, and run the usual evaluation path. Iterating the correction would require redoing features or trusting the same features for a new box; validation says one pass is enough.

Now the whole design is coherent. Region proposals handle the coarse search over object-like locations. Warped proposal windows let a fixed-input CNN score arbitrary regions. Supervised ImageNet pretraining supplies a rich hierarchy when detection labels are scarce, and fine-tuning adapts that hierarchy to the warped-window domain. Strict per-class SVMs with hard-negative mining supply sharper detection boundaries than the loose fine-tuning softmax. Greedy NMS removes duplicate hits from the redundant proposal set. Finally, class-specific ridge regression tightens the boxes using scale-invariant center shifts and log-space size corrections. The method is not a single clever formula; it is a careful division of labor that lets each older tool do the part it is good at while the CNN supplies the missing learned representation.
