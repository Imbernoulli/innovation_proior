I start with the part that is actually hard: detection is not classification with a different label set. Classification lets me collapse the whole image to one decision, but detection asks for every object and for boxes that pass a geometric overlap test. The CNN result on ImageNet is too strong to ignore, yet it arrives from a setting with abundant image-level labels and no need to localize. If I simply train the same high-capacity network from scratch on boxed detection data, I expect overfitting. If I simply run it as a dense sliding-window classifier, I inherit a localization problem from the network's own geometry. So I need to separate the two difficulties: get good candidate locations somehow, and use the CNN where it is strongest, as a recognizer of visual content.

The first tempting path is to regress object coordinates directly from an image. That sounds clean because the output is already a box. But the signal is too global and too ambiguous: an image may contain several instances, the network has to decide how many boxes to emit, and concurrent deep regression results on VOC are not competitive enough to make this the main route. Direct regression can be useful later as a correction, but I do not want it to be the whole localization mechanism.

The second familiar path is sliding windows. I could scan many positions and scales, crop each window, and classify it. Shallow CNN detectors have worked like this for constrained categories because their feature maps retain enough spatial resolution. A deeper ImageNet-style CNN is different. High layers have large receptive fields and a coarse stride in the input. A unit near the top is responding to a large image patch, and nearby top-layer positions are not separated by a pixel or two. That coarseness is useful for recognition invariance but bad for precise window placement. I could make the network shallower to recover resolution, but then I give up the learned hierarchy that motivated the whole attempt.

A third option avoids the geometry problem entirely: do not ask the CNN to search all positions. Let a category-independent process propose a modest set of regions that may contain objects, then classify only those regions. This splits the work along the grain of what each part is good at. A proposal algorithm answers "where might an object be?" without committing to a class, which it can do with high recall on a few thousand windows; the CNN answers "what does this region look like?", which is the question its ImageNet training already trained it to answer. Neither piece is forced into a job it does badly, where direct regression overloaded the CNN with the counting-and-placing problem and sliding windows overloaded it with a stride its high layers cannot honor. This is the path I want to develop.

Selective search is a good proposal source for this role. It begins with an oversegmentation, then greedily merges neighboring regions that look compatible. The similarity is deliberately plural: color, texture, size, and fill terms catch different reasons that pieces of an object should be grouped. Running variants over several color spaces and similarity combinations gives diversity. The output is a few thousand class-independent boxes in fast mode, not hundreds of thousands of sliding windows. It is also useful scientifically because prior region-based systems used the same proposal family, so a comparison can isolate the representation change.

Now I have candidate rectangles, but the CNN accepts a fixed-size image. I need a transform from an arbitrary rectangle to a 227 x 227 input. I can preserve aspect ratio by enclosing the proposal in a tight square, but that may include a lot of background. I can mask out the outside context, but that creates an artificial input unlike natural ImageNet crops. Or I can anisotropically warp the rectangle to the fixed size, accepting shape distortion in exchange for filling the CNN input with the candidate. I also need a small amount of context around the proposal because objects are not recognized in isolation. The simplest thing that gives me both is to warp the proposal after expanding it so that the original box keeps a border of some p pixels in the transformed coordinate frame.

The detail I keep getting wrong if I am careless is where p lives. It is tempting to expand the source rectangle by p pixels, but p is supposed to be a margin in the 227 frame, not in the original image, and the warp scale differs per proposal. The clean way is to define the expansion so that the original box, after warping, occupies the central 227 - 2p pixels. That means scaling the source rectangle about its center by 227 / (227 - 2p). Let me check that this scale actually buys p = 16 pixels of target-frame border and not something else. With p = 16 the factor is 227 / (227 - 32) = 227 / 195 = 1.1641. Take a source box of width 80: expanding it gives a source width of 80 * 1.1641 = 93.13, and the warp maps that 93.13-wide source onto the full 227, so one source pixel becomes 227 / 93.13 = 2.4375 target pixels. The original 80-wide box therefore lands as 80 * 2.4375 = 195.0 pixels in the target frame, leaving (227 - 195) / 2 = 16.0 pixels on each side. So the border is exactly p, independent of the source width, which is what I wanted; a naive p/227 fraction of the original box would have given a margin that shrank for large proposals and grew for small ones. If the expanded source rectangle crosses the image boundary, the missing pixels are image-mean pixels before mean subtraction, which become zeros after mean subtraction.

With proposals and warping in place, the detection architecture is almost forced. For each image, generate roughly two thousand boxes. For each box, crop, pad, warp, subtract the mean, and forward it through the CNN. Take a fixed-length feature vector, normally fc7 for detection, and train one linear classifier per object class. This shares the expensive CNN work across all classes: one feature matrix for the image, one matrix product with the SVM weight matrix, then per-class suppression of duplicate boxes.

The next problem is training the CNN. Detection data alone is too small for a large network, but ImageNet classification gives me a supervised auxiliary task with enough labels to learn general visual features. So I first use a CNN trained on ILSVRC classification. Then I adapt it to the detection domain by replacing the ImageNet 1000-way classification layer with an N+1-way layer for N object classes plus background and continuing SGD on warped proposal windows. I lower the learning rate to 0.001 so fine-tuning adjusts the initialization without destroying it.

Fine-tuning needs labels for proposal windows. I want many positives because I am updating the entire network, not just a small classifier. So for this stage I use a loose rule: assign each proposal to the ground-truth box with maximum IoU, and if that overlap is at least 0.5, label the proposal as that box's class; otherwise label it as background. This creates many jittered positive examples around each object. Since positives are rare among proposals, each minibatch is biased toward them: 32 positive windows and 96 background windows in a batch of 128. The loose label rule is not ideal for precise localization, but it gives the network enough examples to adapt its features to warped object regions.

After fine-tuning, I am uneasy about relying only on the final softmax as the detector. The softmax has been trained with loose 0.5-IoU positives and randomly sampled background windows. A detection classifier has a different job. For a class like car, the clean positives are the actual ground-truth boxes, and the important negatives include background and near misses that the current classifier confuses with the class. So I train separate linear SVMs on the fixed CNN features. For each class, positives are only ground-truth boxes of that class. Negatives are proposals whose overlap with every instance of that class is below some threshold; proposals in the grey zone above it but below a positive cutoff are ignored for that class, so the classifier never sees a half-object labelled either way.

The negative threshold is the one knob here I cannot reason my way to a single number on. I can bound the two failure modes, though. If I set it at 0.5, then a proposal overlapping a car at 0.45 counts as a negative even though it is mostly car, and I am teaching the SVM to push down boxes that are arguably correct detections, which should hurt recall on tight boxes. If I set it at 0, then every proposal that is not literally a ground-truth box is a negative, including ones overlapping the car at 0.4, and now near-duplicates of the positive get used as hard negatives, which should make the decision boundary fight itself. Both extremes are plausibly bad for opposite reasons, and the right value is some intermediate cutoff where a proposal is "clearly not this object." I would pick it by grid search on a held-out set rather than argue it; my prior is something low like 0.3, but I would only trust it after seeing the AP curve over thresholds, and I would not be surprised if the difference between 0.3 and a neighbour is a couple of AP points.

Hard-negative mining is necessary because most proposal windows are background and many are trivially easy. I do not need to keep every easy negative in memory. I start with low-overlap negatives, train the linear SVM, scan images, add only negatives that violate the margin or score too high, retrain as needed, and evict negatives that become safely easy. This is inherited from the detection tradition, but it fits the CNN feature setting well: the feature cache is large, so training should spend memory on confusing examples.

At test time the classifier side is straightforward. I run selective search, warp each proposal, compute CNN features, scale features the same way as in training, and score every proposal for every class with the SVMs. Many proposals overlap the same object, so I apply greedy non-maximum suppression independently per class. The threshold can be learned or fixed in a demo, but the operation is the same: sort by score, keep the best remaining box, and discard lower-scored boxes with too much IoU overlap.

The remaining errors tell me what to fix next. If the CNN features are doing their job, many false positives should not be pure background confusions; they should be boxes that are on or near the right object but not tight enough. That is exactly the regime where regression becomes sensible. I rejected direct box regression as the whole detector, but I can use regression as a small correction from an already plausible proposal to a nearby ground-truth box.

The box transform has to be scale-invariant. Let P = (P_x, P_y, P_w, P_h) be a proposal in center, width, height coordinates, and let G = (G_x, G_y, G_w, G_h) be its assigned ground-truth box. A raw pixel shift is the wrong target because the same 10-pixel error means different things for a tiny box and a large box. So I define center targets as fractions of the proposal size: t_x = (G_x - P_x) / P_w and t_y = (G_y - P_y) / P_h.

For width and height I have a real choice between an additive target G_w - P_w and a multiplicative one. Both can be inverted, so invertibility does not decide it. What decides it is symmetry and positivity. Suppose a proposal should double in width versus a proposal that should halve: with additive targets and P_w = 80 those are +80 and -40, two different magnitudes for what are mirror-image corrections, and an unbounded negative prediction could even drive a predicted width negative. With the log target, doubling is log 2 = +0.693 and halving is log(1/2) = -0.693, equal magnitude and opposite sign, and exp() of anything stays positive. The log scaling is the one that treats "twice as big" and "half as big" as the same size of mistake, so I use t_w = log(G_w / P_w) and t_h = log(G_h / P_h).

I learn four functions d_x(P), d_y(P), d_w(P), and d_h(P). Each is linear in the proposal's pool5 feature vector. Once those functions predict a correction, I invert the parameterization: Ghat_x = P_w d_x(P) + P_x, Ghat_y = P_h d_y(P) + P_y, Ghat_w = P_w exp(d_w(P)), and Ghat_h = P_h exp(d_h(P)). Before I trust this I want to run the round trip on a concrete pair, because an off-by-one in which width divides which target is exactly the kind of bug that still trains and just regresses to slightly wrong boxes. Take P = (100, 80, 40, 60) and a target G = (110, 70, 50, 48). The targets are t_x = (110 - 100)/40 = 0.25, t_y = (70 - 80)/60 = -0.1667, t_w = log(50/40) = 0.2231, t_h = log(48/60) = -0.2231. Now feed those in as the predictions and invert: Ghat_x = 40 * 0.25 + 100 = 110, Ghat_y = 60 * (-0.1667) + 80 = 70, Ghat_w = 40 * exp(0.2231) = 40 * 1.25 = 50, Ghat_h = 60 * exp(-0.2231) = 60 * 0.8 = 48. That returns G = (110, 70, 50, 48) exactly. So when the regressor predicts the targets perfectly the transform is the identity on G, the exponential sits only on width and height, and there is no stray sign or missing scale constant; a perfect predictor cannot be sabotaged by the parameterization.

The regression fit itself is ridge regression with lambda set by validation, and the objective is the standard sum of squared target errors plus lambda times the squared weight norm. The implementation adds practical conditioning: it scales the CNN features, appends a bias feature, centers and decorrelates the four regression targets before fitting, then unwhitens predictions at test time. Those steps do not change the target definitions or the inverse transform; they make the linear solves better behaved.

I also need to be careful about which training pairs the regressor sees. If a proposal is nowhere near any object, asking a linear model to move it across the image to a ground-truth box is meaningless: the targets t_w = log(G_w / P_w) blow up when P barely overlaps G, and the pool5 feature of a mostly-background window carries little information about where the object actually is. So I only train on proposals whose max overlap with a ground-truth box is high, above 0.6, assign each to its max-overlap box, discard the rest, and train class-specific regressors. At test time I predict once for each scored detection, clip the adjusted coordinates to the image, keep the original SVM score, and run the usual evaluation path. I could iterate the correction, but a second pass would feed the regressor pool5 features that belong to the original proposal window, not to the moved box, so the inputs and the box would be inconsistent; recomputing features for every moved box is expensive, and I would expect most of the gain to come from the first nudge anyway. I would do a single pass and check on a validation split whether a second pass actually helps before paying for it.

Stepping back, the pieces now fit together without anything left dangling. Region proposals handle the coarse search over object-like locations, so the CNN never has to scan all positions at a stride its high layers cannot resolve. Warped proposal windows let a fixed-input CNN score arbitrary regions, with the padding pinned to the target frame so context is the same fraction for every box. Supervised ImageNet pretraining supplies a rich hierarchy when detection labels are scarce, and fine-tuning with the loose 0.5 rule adapts that hierarchy to the warped-window domain on many jittered positives. Strict per-class SVMs with hard-negative mining then re-draw sharper decision boundaries than the loose fine-tuning softmax. Greedy NMS removes duplicate hits from the redundant proposal set. And the round-trip-verified box regression tightens what is left using scale-invariant center shifts and log-space size corrections. What carried the work was not one formula but the division of labor: each older tool does the part it is good at, and the CNN supplies the learned representation that the hand-designed features could not.
