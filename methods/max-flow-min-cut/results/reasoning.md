I start with the only inequality I can trust. If I take any feasible flow and any cut separating the source from the sink, the flow value cannot exceed the cut capacity. I do not want to leave that as a picture in my head, because the exact slack in this inequality is going to matter. I sum the net outflow over all vertices on the source side `S`. Conservation erases every internal vertex, and every arc with both ends in `S` appears once positively and once negatively. What survives is `value(f) = sum_{S->T} f - sum_{T->S} f`. Then I drop the backward term, because it is nonnegative, and I replace forward flow by forward capacity. So `value(f) <= cap(S,T)`.

I want to know where the slack in that chain comes from, because the answer tells me what it would take to make the inequality an equality. Two steps lost ground. The first was dropping `sum_{T->S} f`; that loss is zero exactly when every arc crossing backward from `T` into `S` carries no flow. The second was replacing forward flow by forward capacity; that loss is zero exactly when every arc crossing forward from `S` into `T` is saturated. So a cut that is full in the forward direction and idle in the backward direction would force `value(f) = cap(S,T)`. If that ever happened, neither object could move: no larger flow could squeeze through a forward-saturated cut, and no cheaper cut could drop below the flow it already matches. I do not have such a configuration yet, but I now have a precise target to aim a construction at.

Let me sanity-check that the two conditions are not vacuous by hand on the smallest interesting network. Take vertices `s, a, b, t` with arcs `s->a`, `s->b`, `a->b`, `a->t`, `b->t`, every capacity `1`. Suppose I route one unit `s->a->t` and one unit `s->b->t`, leaving `a->b` empty. Take `S = {s}`, `T = {a,b,t}`. The forward arcs across this cut are `s->a` and `s->b`, both at flow `1` of capacity `1`, both saturated. There are no backward arcs from `T` into `s`. So both conditions hold, and the prediction says `value(f) = cap(S,T)`. Checking directly: `value = 2`, and `cap(S,T) = c(s,a) + c(s,b) = 2`. They agree. And to confirm the slack story, take the different cut `S' = {s,a}`: its forward arcs are `s->b`, `a->b`, `a->t`, total capacity `3`, strictly above `2`. The upper bound still holds but is loose there, exactly because `a->b` crosses forward carrying no flow — an unsaturated forward arc, which is one of the two things my analysis said creates slack. So the equality conditions are doing real work, not describing an empty set.

The obvious way to build a large flow is to find a source-to-sink route with unused capacity, push as much as the tightest arc allows, and repeat. Before I trust that, I want to know whether it can paint itself into a corner. Stay with the same one-unit network. Suppose the search first finds the route `s->a->b->t` and pushes one unit. Now `s->a`, `a->b`, `b->t` all carry one unit and are full. I look for another forward route with spare capacity. From `s` I can still reach `b` via `s->b`. But from `b` the only outgoing arc is `b->t`, which is now full. There is no other forward path, so a purely additive procedure stops here at `value = 1`.

Is `1` actually the best? I can route `s->a->t` and `s->b->t` independently, two units, with `a->b` untouched — that is the flow I just checked by hand, value `2`. So the additive procedure stopped a full unit short of the optimum. The trouble is not capacity; it is irreversibility. The early choice to spend `a->b` was wrong, and the method has no operation that says "take that unit back and send it the other way." If the procedure cannot undo its own routing, then stopping tells me nothing about optimality.

So I need to make undoing a normal move rather than an exception. For a current flow `f`, a step from `u` to `v` should be available in two ways. If there is a real arc `u->v` that is not full, I can push more through it, up to `c(u,v)-f(u,v)`. If there is a real arc `v->u` that currently carries flow, I can cancel some of that flow, up to `f(v,u)`. Both changes move flow from `u` toward `v` in the vertex balances; they are interchangeable as far as conservation cares. So I define the available room from `u` to `v` as `c_f(u,v) = c(u,v) - f(u,v) + f(v,u)`, treating missing arcs as zero. The graph of positive `c_f` values is the graph of changes I can still make.

Now I go back to the stuck flow and look at it through this residual lens, to see whether the missing undo reappears as a path. The stuck flow had `f(s,a)=1`, `f(a,b)=1`, `f(b,t)=1`, everything else zero. Computing `c_f` on each ordered pair: `c_f(s,b) = 1-0+0 = 1`; `c_f(b,a) = 0-0+f(a,b) = 1`, the cancellation room on the wrongly used diagonal; `c_f(a,t) = 1-0+0 = 1`. So the residual graph contains the path `s -> b -> a -> t`. That is not a forward route in the original network — its middle step `b -> a` is a cancellation of the `a->b` unit. Pushing one unit along it removes the unit from `a->b`, adds a unit on `s->b` and a unit on `a->t`, and the result is exactly the two-unit flow `s->a->t`, `s->b->t`. The undo I could not express before is just an ordinary residual path. So a residual source-to-sink path is not a heuristic hint; it is a concrete recipe for a strictly larger flow.

Let me make sure that recipe is always safe, not just on this example. If the residual graph has a source-to-sink path, I take the smallest residual room on it, call it `F`, and apply `F` to every step. Feasibility survives because `F` is no larger than the room at any step: where I add forward flow, `F <= c-f` keeps it under capacity; where I cancel reverse flow, `F <= f(v,u)` keeps that flow nonnegative. Conservation survives because each interior vertex receives `F` units of net change from the step before and sends `F` units to the step after. At the source, net outflow rises by `F`; at the sink, net inflow rises by `F`. So along any residual `s`-`t` path the value strictly increases. The improving case is fully handled.

The case that decides everything is the other one: no residual source-to-sink path exists. I should not read that as a computational dead end; I should ask what structure the failure forces. Let `S` be the vertices still reachable from the source in the residual graph, and let `T` be the rest. The sink is unreachable, so `s in S` and `t in T`, and this is a genuine cut. Because `S` is exactly the residually reachable set, no residual edge leaves `S`.

Now I push that single fact through the definition of `c_f`. Take an original arc `u->v` with `u in S`, `v in T`. If it had any unused forward capacity, then `c_f(u,v) >= c(u,v)-f(u,v) > 0`, so the residual edge `u->v` would leave `S` and make `v` reachable — impossible. Hence every forward arc across the cut is saturated. Take an original arc `v->u` going backward, `v in T`, `u in S`. If it carried positive flow, then `c_f(u,v) >= f(v,u) > 0`, so again `v` would be reachable from `u` — impossible. Hence every backward arc across the cut carries zero flow.

These are precisely the two equality conditions I extracted from the weak-duality slack at the very start. I did not assume them; the absence of a residual path produced them. So for this `S`, `value(f) = cap(S,T)`. The same failed search that says "the flow cannot be augmented" also hands me the cut that says "and it does not need to be." I can check the shape against my small example: the recovered two-unit flow has residual graph reachable set `S = {s}` (from `s`, the only positive-room out-arcs are to `a` and `b`, but `c_f(s,a) = 1-1+0 = 0` and `c_f(s,b)=1-1+0=0`, both saturated, so nothing is reachable past `s`). That cut has capacity `c(s,a)+c(s,b) = 2`, equal to the flow value, with both forward arcs saturated and no backward arcs at all — exactly the predicted shape.

So the procedure is constructive because it lives in only two states. If the residual graph has a source-to-sink path, I augment and strictly increase the flow. If it has no such path, the source-reachable set is a minimum cut and the current flow is maximum. There is no leftover state where I am stuck without a certificate, because the certificate is not a separate argument — it is the exact residue left behind when augmentation becomes impossible.

I still have to be sure the procedure stops. With integer capacities, I start from the zero flow, and every residual capacity is then an integer. Each augmentation uses the minimum of positive residual capacities, so it raises the value by at least one whole unit. The value is bounded above by any source cut, for instance the capacity of all arcs leaving the source. A strictly increasing integer sequence with an upper bound is finite, so the process terminates, and because every update adds or subtracts an integer amount, every arc flow stays integral throughout. Rational capacities reduce to this by clearing denominators.

Termination is not the same as efficiency, though, and arbitrary path choices can be genuinely bad. A small middle arc can be used forward, then cancelled, then used again, so the flow crawls up one unit at a time even when a large flow is available; with irrational capacities, careless choices need not terminate at all. To control this I choose shortest augmenting paths in the residual graph, found by breadth-first search. With that rule, residual distances from the source never decrease: any residual edge created by an augmentation is the reverse of an edge on the just-used shortest path, and using it to bring a vertex closer would contradict the layering that breadth-first search just imposed. An edge that becomes a bottleneck and disappears can only return after a reverse use has pushed its endpoint's distance up by at least two. Distances are bounded by the number of vertices, so each edge is a bottleneck only `O(V)` times. That gives `O(VE)` augmentations and `O(VE^2)` total time.

What the residual graph really is, then, is a live ledger of every local change the current flow still admits, including undoing earlier choices. While a residual path reaches the sink, the flow is improvable, and the path is the improvement. When no residual path reaches the sink, the vertices still reachable from the source are exactly the cut whose forward side has no slack and whose backward side carries no flow. The maximum flow and the minimum cut are not two theorems that happen to share a number; they are the two faces of the single moment when augmentation runs out.
