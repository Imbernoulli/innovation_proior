I start from the uncomfortable part of the problem: the learner receives reinforcement, not a target. If a unit outputs an action and the world returns one scalar `r`, there is no direct error signal saying that this unit should have output a different value. In the older adaptive search element picture, this is handled by trying actions stochastically and by keeping eligibility traces, but that still leaves me with a question I cannot ignore: why should multiplying reinforcement by a local trace be a gradient step rather than just a plausible reinforcement rule?

The first correction is to stop trying to differentiate the realized reward. The environment may be unknown, discontinuous, delayed, or effectively a black box. The differentiable object I can actually write down is the probability law of the stochastic unit. For unit `i`, I have a conditional distribution `g_i(y_i | w_i, x_i)`. The objective is not the reward from this one sample, but the expected reinforcement under the current weights, `J(W) = E[r | W]`. If I can differentiate that expectation with respect to a weight, I have a real hill-climbing target.

I condition on the sampled output of the unit. Holding `W` and the unit input `x_i` fixed, the expected reward decomposes over the unit's possible outputs. For a discrete unit, `E[r | W, x_i] = sum_y g_i(y | w_i, x_i) E[r | W, x_i, Y_i = y]`. There is one thing I have to check before differentiating this sum: once the unit's output `Y_i = y` is fixed, does the downstream reward distribution still depend on the weight `w_ij`? In a feedforward network `w_ij` only enters through how the unit produces its output; conditioned on the output value, the rest of the computation and the environment see only `y`, not `w_ij`. So `E[r | W, x_i, Y_i = y]` is constant in `w_ij`, and the only `w_ij`-dependence left in the sum is through the probabilities `g_i(y)`. Differentiating term by term gives

```text
d E[r | W, x_i] / d w_ij = sum_y E[r | Y_i = y] d g_i(y) / d w_ij.
```

This is close, but it is not yet a sample rule. A sample gives me one realized `Y_i`, not the whole sum over possible outputs. The natural first guess is to drop the sum and use the single realized term `r * d g_i(Y_i) / d w_ij`. I want to know whether the expectation of that quantity, over `Y_i ~ g_i`, equals the derivative above. It does not, and I can see the mismatch without abstraction by writing the expectation out: `E[ r * d g_i(Y_i)/d w_ij ] = sum_y g_i(y) E[r|y] d g_i(y)/d w_ij`. There is an extra factor of `g_i(y)` inside the sum that the target derivative does not have. So this naive rule is biased by exactly the sampling probability.

Let me make that failure concrete so I am not fooling myself with symbols. Take a single Bernoulli unit with two outputs `y in {0,1}`, `p = Pr(Y=1) = 0.3`, and a deterministic reward `r(0)=1`, `r(1)=4`. Then `J(p) = (1-p)*1 + p*4 = 1 + 3p`, so the true derivative is exactly `dJ/dp = 3`. The naive rule samples `y`, computes `r * dg(y)/dp` with `dg(1)/dp = 1`, `dg(0)/dp = -1`, and averages: `0.3*(4*1) + 0.7*(1*(-1)) = 1.2 - 0.7 = 0.5`. That is `0.5`, not `3`. The naive estimator is off by a factor that comes from the probabilities, exactly as predicted. So I genuinely cannot just drop the sum.

The fix is to put back what the sampling took out. Since `d g_i(y) = g_i(y) d log g_i(y)`, I can write the target derivative as `sum_y g_i(y) E[r|y] d log g_i(y) / d w_ij`, and now the leading `g_i(y)` is exactly the sampling weight. So the single-sample quantity `r * d log g_i(Y_i) / d w_ij` has expectation `sum_y g_i(y) E[r|y] d log g_i(y)/d w_ij`, which is the derivative I wanted. Checking on the same Bernoulli example: `d log g(1)/dp = 1/p` and `d log g(0)/dp = -1/(1-p)`, so the expectation is `0.3*(4 * 1/0.3) + 0.7*(1 * (-1/0.7)) = 4 - 1 = 3`. That matches `dJ/dp = 3`. The log-probability derivative is not a cosmetic rewrite; it is the correction that turns a sampled output into an unbiased estimate of a derivative of an expectation. In modern terms this is the score-function or likelihood-ratio estimator; the local log-derivative term is the characteristic eligibility, and the resulting family of algorithms is what Williams names REINFORCE.

Now I have to check that this local result really survives the network setting. A downstream weight can affect reward through later units and the environment, but the argument above isolates one stochastic unit at a time. For `w_ij`, the input `x_i` comes from upstream parts of the feedforward network, so its distribution is independent of this downstream weight, and I can average the conditional result over `x_i` without disturbing the derivative. That yields `E[Delta w_ij | W] = alpha_ij d E[r | W] / d w_ij` when the learning-rate factor is constant for that weight. Summing over weights gives the network statement: the expected update has a nonnegative inner product with the true gradient, and with one common positive learning rate it is exactly proportional to the gradient.

The baseline is the next place where the derivation could break. Subtracting a reference level from reward is useful because raw reinforcement may be noisy or shifted, but if the baseline changes the mean of the estimator then I am no longer running the same gradient estimator. The estimator with baseline is `(r - b) d log g_i(Y_i)/d w_ij`, whose expectation is the unbiased term minus `b * E[d log g_i(Y_i)/d w_ij]`. So the baseline is harmless precisely when that expected score is zero. For a discrete distribution, `E[d log g_i(Y_i) / d w_ij] = sum_y g_i(y) d log g_i(y) / d w_ij = sum_y d g_i(y) / d w_ij = d/dw_ij ( sum_y g_i(y) ) = d 1 / d w_ij = 0`. I can confirm this is not just a symbolic coincidence on the Bernoulli case: `0.3*(1/0.3) + 0.7*(-1/0.7) = 1 - 1 = 0`, so a baseline of, say, `b=2.5` shifts the estimator by `2.5 * 0 = 0` and the expectation stays at `3`. The condition this exposes is sharper than "subtract whatever seems convenient": the baseline must be conditionally independent of the very sample whose score it multiplies, or it will not cancel.

With that condition, the update becomes `Delta w_ij = alpha_ij (r - b_ij) e_ij`, where `e_ij = d log g_i(Y_i | w_i, x_i) / d w_ij`. This is the formula that makes the old reward-modulated trace picture precise. It says which trace is needed: not any activity trace, but the score of the sampled output under the unit's own distribution. It also says why centering the reward can reduce variance without biasing the direction.

The Bernoulli logistic case is worth working out because it tells me whether the abstract score reduces to something local and familiar. If a unit emits `y in {0,1}` with `Pr(Y=1) = p`, then `d log g(y)/dp = (y - p)/(p(1-p))`. Now let `p = sigma(s)` with `s = sum_j w_ij x_j`, so `dp/ds = p(1-p)` and `ds/dw_ij = x_j`. The chain rule gives `e_ij = [ (y-p)/(p(1-p)) ] * [ p(1-p) ] * x_j`, and the `p(1-p)` factors are supposed to cancel to leave `(y_i - p_i) x_j`. I do not want to trust a cancellation I only wrote symbolically, so I check it by finite differences. With `(y, w, x) = (1, 0.4, 2.0)`, `p = sigma(0.8) = 0.690`, the analytic `(y-p)x = (1-0.690)*2 = 0.620`; the centered finite difference of `log g` in `w` gives `0.6200510`. Two more points, `(0, -0.7, 1.5)` and `(1, 1.1, -0.5)`, give `-0.3888377` vs `-0.3888377` and `-0.3170678` vs `-0.3170678`. The cancellation is real. That `(y_i - p_i) x_j` local form is what makes associative reward-inaction and related actor rules appear as special cases rather than separate inventions.

The delayed-reward case initially looks harder because the reinforcement can arrive after many stochastic choices. If I treat the whole episode as one sampled object, however, its probability is a product over the stochastic choices made through time. The log of that product is a sum, so the score for the trajectory is the sum of the per-time scores, and the same reward can multiply the accumulated eligibility: `Delta w_ij = alpha_ij (r - b_ij) sum_t e_ij(t)`. This is what unfolding the recurrent or time-dependent network amounts to. The eligibility trace is then not merely a memory trick; it is the online way to accumulate the score of the sampled trajectory before terminal reinforcement is known.

This also explains the policy-gradient form. A stochastic policy samples actions from `pi_theta(a_t | s_t)`. The probability of a trajectory contains the product of the action probabilities controlled by `theta`; environment transitions may be unknown, but if they do not depend on `theta` directly, their derivatives drop out of the log-derivative just as `E[r | Y_i = y]` did in the single-unit argument. The score of the trajectory is `sum_t grad_theta log pi_theta(a_t | s_t)`. Multiplying that by return gives an unbiased stochastic gradient estimator. Replacing the full return with reward-to-go or with an advantage is a variance and credit-assignment improvement that preserves the same log-probability score structure, as long as the subtracted part has zero expectation under the same baseline argument I checked above.

The Gaussian case keeps me honest because the method is not only for binary action units. For a Gaussian unit with mean `mu` and standard deviation `sigma`, the score with respect to `mu` is `(y - mu) / sigma^2`, and the score with respect to `sigma` is `((y - mu)^2 - sigma^2) / sigma^3`. The zero-mean and unbiasedness claims should hold here too, but the continuous case is exactly where I most want a check rather than a symbol, so I sample it. With `mu = 1.5`, `sigma = 0.8`, reward `r(y) = y` (so `J = mu`, `dJ/dmu = 1`), four million draws give `E[(y-mu)/sigma^2] ≈ -0.0002` (the score averages to zero), `E[r * score] ≈ 0.9994` (matching `dJ/dmu = 1`), and with a baseline `b = 5` still `≈ 1.0004`. So the same principle carries over to a continuous density: use the derivative of the log density of the sampled output, and the baseline does not bias it.

There is also a boundary with ordinary backpropagation. If the network has deterministic hidden layers feeding stochastic output units, the score at the stochastic outputs can be propagated backward through deterministic functions: the stochastic node contributes the log-probability derivative, and ordinary chain rule carries it to earlier deterministic weights. For reparameterizable continuous noise one can sometimes differentiate through the sampled value itself, such as `y = mu + sigma z` for Gaussian noise. But that is a different pathwise route and is not available for generic discrete units, where there is nothing to differentiate through. The score-function route stays valuable because it does not require differentiating the reward through the sample at all.

The modern implementation artifact is almost terse after all this. In the Spinning Up examples, the policy produces a distribution, the taken action's log probability is read, and the loss is the negative mean of `logp * weights`. When `weights` is the episode return, reward-to-go, or advantage, automatic differentiation of that loss produces the negative of the same estimator: `grad ( -(logp * w) ) = -(w * grad logp)`, and `grad logp` is exactly the score `e_ij` summed over time. Minimizing the loss is therefore ascending the estimated expected return, and the line `loss_pi = -(logp * adv).mean()` is a compact descendant of the single-unit result.

I also have to keep the limitations in frame. The whole derivation above is about the estimator's mean. Every check I ran was an expectation: `E[r * score] = dJ/dtheta`. Unbiasedness says nothing about variance or about how fast learning converges, and a single scalar reward can be a very noisy multiplier. The same Bernoulli expectation that averaged to exactly `3` came from per-sample values of `4 * (1/0.3) = 13.3` and `1 * (-1/0.7) = -1.43`, a spread of order ten around a mean of three; a poorly chosen baseline would only widen that. These algorithms can be slow, can settle at poor optima, and do not come with a general convergence proof. The score-function estimator is broadly applicable, including to black-box or discrete stochastic systems, but its practicality depends on variance reduction, which is why baselines, critics, reward-to-go, and advantage estimates are wrapped around the unbiased core rather than being optional decoration.

So the reconstructed reasoning is this: stochasticity creates a probability distribution I can differentiate even when the reward mechanism is opaque. Dropping the sum naively is biased by the sampling probability, as the Bernoulli example showed (`0.5` instead of `3`); weighting the sampled output by the derivative of its log probability exactly cancels that probability and recovers the derivative of the expectation (back to `3`). Multiplying that score by reward gives an unbiased stochastic gradient estimate; subtracting an output-independent baseline leaves the mean unchanged because the expected score is zero; summing scores through time handles delayed reinforcement; and the same structure becomes the policy-gradient update used in modern code.
