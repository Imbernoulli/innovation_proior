I start with the wrong kind of reduction. A Karp reduction tells me that exact solving is hard, but it has no reason to preserve a gap. If a satisfiable formula maps to a target instance with optimum `m`, an unsatisfiable formula may still map to an instance with optimum `m - 1`, or `0.99m`, or something else too close to separate by approximation. The reduction proves exact hardness precisely because it only cares about the boundary between perfect and imperfect. Approximation needs a numerical margin, and I do not yet know where such a margin should come from.

At the same time, proof verification is becoming stranger than the NP definition suggests. In NP I read the whole witness. In interactive proofs the verifier flips coins and checks algebraic claims through random challenges. LFKN show me that a global sum of a low-degree polynomial over a Boolean cube can be checked by stripping variables one at a time and ending at one random point. BFL show that with noncommunicating provers this reaches NEXP. So the verification picture already contains the ingredients I want: randomness, algebra, and consistency checks that turn global truth into local tests. But the scale is wrong. I do not want a large interactive protocol for NEXP. I want a static proof for NP that can be sampled.

A verifier-to-clique construction gives the first clue that the approximation problem and the proof-checking problem are not separate. If a verifier uses `r` random bits and reads `q` proof bits, I can make a graph whose vertices are accepting local views: one random string together with one assignment to the queried proof locations that makes the verifier accept. I join two vertices when the two local views are consistent. Now a clique is exactly a collection of accepting views that can be explained by one global proof string. If the input is true, the honest proof supplies one accepting view for every random string. If the input is false, soundness says no proof can cover more than the accepting fraction allowed by the verifier. The clique gap is the soundness gap.

That changes the target. I no longer search for a clever approximation gadget for each problem. I search for a proof format whose local soundness gap can be exported. The constraint-system version makes this exact. Given a verifier with logarithmic randomness and constant queries, I create one constraint for each random string, with variables equal to the proof locations the verifier can read. The violated-constraint fraction is the verifier's rejection probability. Conversely, if I can reduce NP instances to constraint systems that are satisfiable in the yes case and have a constant unsatisfied fraction in the no case, then a verifier only has to pick one random constraint and check its variables. So a constant-query proof system and gap-CSP hardness are the same statement written in two languages.

This tells me what the verifier must accomplish, but it also explains why the ordinary witness cannot work. In a usual satisfying assignment, an error can be localized. If one bit is wrong, random sampling probably misses it. For local checking to be possible, a false proof has to be wrong in many places. I need a proof representation with distance. That points to error-correcting codes, but distance alone is not enough. The verifier must also check that the encoded object satisfies a computation. The code has to carry algebra.

Low-degree polynomials have exactly the combination I need. Two distinct degree-`d` polynomials over a field disagree on at least a `1 - d/|F|` fraction of points, and when `d/|F| <= 1/2`, any function within distance less than `1/4` has a unique nearest degree-`d` polynomial. This means a purported proof is not just a list of witness bits. It is a redundant algebraic table, and if it is lying about a global computation, the lie should show up across many random locations.

Now arithmetization becomes useful at the NP scale. I can encode a Boolean computation as polynomial identities over a finite field. A satisfying assignment becomes a low-degree object whose values obey those identities. Sum-check reasoning explains how a global algebraic condition can be reduced to random evaluations. Low-degree tests, following the self-testing and robust-polynomial line, explain how to test that the alleged table is close to some low-degree polynomial rather than an arbitrary forged function. The verifier is no longer asking "is this witness correct if I read it all?" It is asking "do these few local shadows look like they come from one algebraic object that could encode a correct computation?"

But I immediately hit the parameter wall. The algebraic verifier may use only logarithmic randomness, because a random point in the chosen field domain can be named with `O(log n)` bits. Yet its answers are field elements, line restrictions, or curve restrictions; measured as bits, they are polylogarithmic. I have made the proof locally meaningful but not constant-query in the bit sense. The very redundancy that spreads errors also makes each local answer too large.

The way around this is to stop reading the long answers directly. For a fixed input and random string, the outer verifier's action is small: it chooses a few locations and then applies a small decision circuit to the answers. Instead of reading those answers in plain form, I can ask another proof checker to verify the small claim that these encoded answers satisfy that decision circuit. This is composition. The outer verifier supplies low randomness and algebraic robustness. The inner verifier checks the tiny decision problem created by the outer verifier's random choice.

This only works if I discipline the verifier. The queries have to be nonadaptive, because the inner instance must be a fixed circuit determined by the input and random string before any proof contents are read. The outer verifier also has to be robust or in normal form, because the inner verifier proves that the encoded answer tuple satisfies the decision circuit, and that must really imply that the outer verifier would accept. Otherwise composition would certify the wrong local statement.

This is the composition step. Randomness becomes the outer randomness plus the inner randomness on a much smaller instance. Query complexity becomes the inner query complexity. Soundness composes as `e + e1 - e e1`, so it stays bounded away from 1 when the pieces are chosen with constant error. The important fact is that the inner verifier is run on the outer verifier's decision circuit, whose size depends on the outer answer size, not on the original input size. The recursion attacks the bottleneck directly.

One composition is not enough, so I iterate. Start with the algebraic verifier: good randomness, long answers. Compose it with an inner verifier for the small circuit. The answer size drops from polylogarithmic in `n` to polylogarithmic in `log n`. Compose again using the smaller circuit size. The answer size falls to constants while the total randomness remains `O(log n)`, because all added randomness is for tiny instances. I amplify the inner verifiers to small constant error before composing; the composed errors are bounded by `2/16` and then `3/16`, which is safely below the standard `1/2` soundness threshold. Curve aggregation handles the related problem of multiple algebraic point queries: ask for the restriction to a low-degree curve through the points, and check consistency at one random point, with a bad curve answer accepted with probability at most `d(l - 1)/|F|`.

Now the original approximation gap is no mystery. A false statement cannot be supported by a proof that is locally correct almost everywhere, because the proof has been forced into a robust algebraic form and its local checks compose. When I translate that verifier into constraints, or into MAX-3SAT clauses, the rejection probability becomes a fraction of violated constraints or clauses. The proof string is doing double duty: it is a locally testable encoding of global correctness and a generator of optimization gaps.

This is the distinctive proof mindset. I am not merely combining NP witnesses, interactive proofs, and codes. I am changing what a proof is so that global correctness leaves many local traces. NP verification gives the relation but not robustness. Interactive proofs give algebraic random checks but not a static constant-access proof for NP. Error-correcting codes give distance but not a computation-aware local test. Approximation theory asks for gaps but does not know how to manufacture them. The construction succeeds by making the proof itself gap-bearing: if the claim is false, constantly many random local views must fail, and that local failure can be exported as hardness of approximation.
