OK, the concrete thing in front of me is a bibliographer's search that won't finish. The corpus is the cumulated citation tapes — a few years of them, something like a hundred and fifty thousand citations, call it ten million characters. The query is a Boolean function over a list of keywords: find every title that contains both "ion" and "bombardment", say, and there might be a couple dozen keywords in one prescription. The program we have does the obvious thing: for each keyword, walk it down the whole text looking for that keyword, then do the next keyword, and so on. And it is slow enough that someone burned through a six-hundred-dollar machine budget before the search even came back. So let me figure out why it's slow, exactly, and what the cost *should* be.

The cost of the obvious method is plainly (number of keywords) × (length of text), more or less. Each keyword does its own full sweep of the ten million characters. If I have twenty-four keywords I read the text twenty-four times. And that's the whole problem in one line: the keyword count *multiplies* the text length. There's no reason it should. The text is the same text. If I'm standing at one position in the text reading a character, I'd like that single read to make progress on *all* the keywords at once, not on one of them. The number I want the cost to depend on is the length of the text, plus the work to set up whatever machinery, plus however much output we actually have to print — and *not* the number of keywords. That's the target: cost essentially independent of how many keywords are in the query.

So what do I have to build with. There's the single-keyword story, which has just been solved cleanly — Knuth, Morris and Pratt. For one pattern of length m in text of length n you can do O(m+n), and the trick is that the text pointer never backs up. You precompute a little table off the pattern: when you've matched the first j−1 characters and then the j-th fails, you don't slide the pattern over by one and re-read; you slide it by exactly the amount that keeps the longest already-matched prefix still aligned. Concretely their table entry — call it f[j] — is the length of the longest proper prefix of pattern[1..j−1] that is also a suffix of it. The longest *border*. You failed at position j, but maybe the last few characters you matched are themselves the start of the pattern, so you don't throw all that away; you fall back to the end of that shorter already-matched prefix and keep going from the same text character. And they compute that table in O(m) by, beautifully, running the matching procedure of the pattern against itself. The thing I want to hold onto from KMP is not the single-pattern bookkeeping; it's the *shape* of the idea: on a failure, the most you can keep is "the longest matched prefix that's still viable," and that quantity depends only on where you were, so you can precompute it.

Now I have a *set* of keywords, not one. What's the right structure for a set of strings? A trie. Insert each keyword as a path from a root; shared prefixes share nodes. So "he" and "hers" and "his" all leave the root on "h" and walk down the same first edge; "she" leaves on "s". Reading one text symbol moves me one edge down. And here's the nice part: a path from the root spelling out u means I have currently matched u as a substring ending right here, and u is a prefix of at least one keyword. If a node is the end of a complete keyword, I tag it with that keyword. So walking the text through the trie matches, at the current position, every keyword that *starts* where my current walk started — all of them simultaneously, in one descent. That's already the "all keywords at once" property I wanted, at least for keywords anchored at the same start.

But it only handles keywords that start at my walk's start. The moment I'm at a node and the next text character has no edge out of that node, I've fallen off the trie. With one pattern KMP told me how far to back off; here I have no such rule for the tree, so the naive fix is to give up, jump back to the root, restart the walk at the *next* text position, and try again. And that's a per-position restart — I'm re-reading the text once per starting position, which is exactly the k·n disease in a different costume. So the trie alone isn't enough. What's missing is precisely a KMP-style failure rule, but for the tree instead of the line.

Let me think about what "fall back" should mean on the trie. Suppose I'm sitting at the node whose path spells "she" and the next character fails — there's no edge for it. I've just read the text characters …s,h,e ending here. I can't keep "she", fine. But what *can* I keep? The longest suffix of "she" that is still the prefix of some keyword. The suffix "he" is a prefix of the keyword "he" (and of "hers"). So I shouldn't go all the way back to the root and lose everything; I should jump to the node for "he" and try the failing character from *there*. That jump — from the node for "she" to the node for "he" — is exactly KMP's failure idea, lifted from one pattern to the whole set. So I want, for every node s with path-string u, a link to the node whose string is the longest proper suffix of u that is also a prefix of some keyword. Call it the failure link f(s). On the line, KMP's f was "longest proper prefix that's also a suffix of the same pattern." On the tree it becomes "longest proper suffix of my string that is also a prefix of *some* keyword" — and "is a prefix of some keyword" is the same as "is the string of some node in the trie." So f(s) is just the deepest *other* node whose string is a suffix of s's string. That's the generalization. It's the same content as KMP — keep the longest still-viable matched piece — but the "still-viable matched piece" now ranges over all keywords, which is exactly what makes one pass serve all of them.

Let me get the search loop straight first, assuming I have these links. Current state s, current symbol a. If there's a trie edge on a out of s — a goto — take it: that's a longer match, advance. If there's no edge, follow the failure link to f(s) and try a again from there; keep following failure links until either some node has an edge on a, or I get back to the root. From the root, if there's still no edge on a, I just stay at the root and move on to the next character. I want to make that last case clean, so let me give the root a self-loop on every symbol that doesn't start any keyword: define the goto of the root on such a symbol to be the root itself, never "fail". Then the root never fails, the failure-following always terminates (worst case at the root, which has an edge on everything), and every step consumes exactly one input symbol with no special case. That's worth doing — without the self-loop I'd need an extra guard for "fell off at the root," and the termination argument for the inner loop would be murkier.

There's one more thing the search has to do, and staring at "she" shows it. When I reach the node for "she", I've also, sitting inside it, just matched "he" — "he" is a suffix of "she" and "he" is itself a keyword. So at that single position I must report *both* "she" and "he". A node has to announce not only the keyword that ends exactly there, but every keyword that is a suffix of its string. And "keyword that is a suffix of my string" — those are reached by following failure links: f("she") is "he", and "he" is a keyword. I could chase those links at every match, but that re-walks the suffix chain over and over. Better: precompute it. When I set f(s), I also fold the report-set of f(s) into the report-set of s. Then each node's report-set already contains every keyword that is a suffix of its string, and at search time reporting is just "emit this node's set," constant per keyword reported. These are the output sets, collapsed at build time.

Now the real question: how do I actually *compute* f for every node? I can't compute f(s) before I know f for shorter strings, because the definition of f(s) leans on suffixes of s, which are shorter. So process nodes in order of increasing depth — breadth-first from the root. Depth-1 nodes (single characters) have no proper non-empty suffix that's a prefix of a keyword, so their failure link is the root: f = 0 for all depth-1 states. Good, base case. Now suppose I've computed f for everything shallower than depth d, and I want f for a depth-d node. Every depth-d node s is reached by exactly one edge: there's a parent r at depth d−1 and a symbol a with goto(r,a)=s. The path-string of s is (string of r) followed by a. I want the longest proper suffix of that which is a prefix of some keyword. A proper suffix of "(r's string)·a" has to be a suffix of r's string, followed by a; it cannot start from r itself, because that would be the whole string again. So I should begin at the best proper suffix of r, then the next best, and so on: f(r), f(f(r)), … . Set state = f(r); while there's no goto on a out of state, set state = f(state); when goto(state,a) succeeds, that target is f(s). And because the root has a goto on every symbol (the self-loop), this while-loop always terminates. After setting f(s), fold output(f(s)) into output(s). Process the states in breadth-first (depth) order so that whenever I compute f(s), the f of every shallower state — in particular every state on the chain from f(r) down — is already known, and its output set has already been completed.

Let me run the whole thing on a small set to make sure I believe it: keywords {he, she, his, hers}. Build the trie. Root is 0. "he": 0→1 on h, 1→2 on e, tag state 2 with "he". "she": 0→3 on s, 3→4 on h, 4→5 on e, tag 5 with "she". "his": h already goes 0→1, then 1→6 on i, 6→7 on s, tag 7 with "his". "hers": h is 0→1, e is 1→2 (reuse), then 2→8 on r, 8→9 on s, tag 9 with "hers". Then add the root self-loop on every symbol other than h and s. Now the failure links, by depth. Depth 1: states 1 (h) and 3 (s), both f=0. Depth 2: states 2 (he), 6 (hi via state… wait, 6 is reached from 1 on i, string "hi"), and 4 (sh). For f(2): parent is 1, symbol e; set state=f(1)=0; goto(0,e)=0 (self-loop, no keyword starts with e), so f(2)=0. For f(6): parent 1, symbol i; state=f(1)=0; goto(0,i)=0; f(6)=0. For f(4): parent 3, symbol h; state=f(3)=0; goto(0,h)=1; so f(4)=1. Depth 3: states 5 (she), 7 (his), 8 (her). For f(5): parent 4, symbol e; state=f(4)=1; goto(1,e)=2; so f(5)=2 — and now fold output(2)={he} into output(5), giving output(5)={she, he}. There it is: the node for "she" will report both. For f(7): parent 6, symbol s; state=f(6)=0; goto(0,s)=3; f(7)=3. For f(8): parent 2, symbol r; state=f(2)=0; goto(0,r)=0; f(8)=0. Depth 4: state 9 (hers), parent 8 symbol s; state=f(8)=0; goto(0,s)=3; f(9)=3, fold output(3)=∅, output(9)={hers}. So f over states 1..9 is 0,0,0,1,2,0,3,0,3 — and that is exactly the failure table I'd want.

Now drive "ushers" through it. State 0. 'u': no edge, root self-loop, stay 0. 's': goto(0,s)=3, state 3. 'h': goto(3,h)=4, state 4. 'e': goto(4,e)=5, state 5 — output(5)={she, he}, so at this position (the 'e' is the 4th character) report both. 'r': goto(5,r) fails; follow f(5)=2; goto(2,r)=8; state 8, no output. 's': goto(8,s)=9; state 9 — output(9)={hers}, report it. So in "ushers" I get "she" and "he" ending at the 'e', and "hers" ending at the final 's', overlapping the earlier matches. That's right — those are exactly the substrings of "ushers" that are keywords. The single pass found all of them, overlaps included, and the work didn't care that there were four keywords rather than one.

I should make sure this is actually *correct*, not just right on one example, because the whole edifice rests on f meaning what I claimed. Let me prove the failure-link characterization: f(s)=t if and only if the string v of t is the longest proper suffix of the string u of s that is also a prefix of some keyword. Induct on the depth of s, equivalently on |u|. Base: depth 1, u is a single symbol, its only proper suffix is the empty string, represented by the root; and I set f=0 there. Holds. Inductive step: u = a₁…a_j, j>1; let r be the parent, so r's string is a₁…a_{j−1} and a_j is the last symbol. Consider the chain r₁=f(r), r₂=f(r₁), … . By the inductive hypothesis applied to r (which is shallower), r₁'s string is the longest proper suffix of a₁…a_{j−1} that's a prefix of some keyword; r₂'s string is the longest proper suffix of *that* which is a prefix of some keyword; and so on — so as I walk the chain I'm enumerating the viable suffixes of a₁…a_{j−1} in strictly decreasing length. My loop walks this chain skipping any r_i that has no goto on a_j, and stops at the first r_n with goto(r_n, a_j)=t≠fail. The string of t is (string of r_n)·a_j. Since r_n's string is the longest viable suffix of a₁…a_{j−1} that *can* be extended by a_j, appending a_j gives the longest proper suffix of u=a₁…a_j that is a prefix of some keyword. And I set f(s)=t=goto(r_n,a_j). So f(s) is exactly that longest viable suffix-state. Done — and note the loop reaches the root in the worst case, where goto on a_j always succeeds because of the self-loop, so there's always such an r_n.

Then the output sets. Claim: output(s) is exactly the set of keywords that are suffixes of s's string u. Induct on depth. At creation I put a keyword y into output of the node whose string is y, so y∈output(node(y)). For the inductive step at state s of depth d: any keyword added at creation is u itself (if u is a keyword); any keyword added by the merge came from output(f(s)), and by induction those are keywords that are suffixes of f(s)'s string, which by the failure-link characterization is a suffix of u — so all of them are suffixes of u, as claimed. Conversely, any keyword y that's a suffix of u is the string of some node, hence is in that node's output; if y=u it's there directly, and if y is a proper suffix of u then by the failure characterization and induction y is in output(f(s)), and since I process states in increasing depth the merge from f(s) (shallower) into s has already happened, so y∈output(s). The two directions match, so output is exactly right — every keyword ending at this position, and only those.

And the search invariant: after processing the j-th text character, the machine is in the state whose string is the longest suffix of a₁…a_j that is a prefix of some keyword. The same suffix argument works against the text instead of against a single trie edge: the goto-then-failure loop, starting from the previous state (the longest viable suffix of a₁…a_{j−1}), follows failure links until it can extend by a_j, which lands on the longest viable suffix of a₁…a_j. Combined with the output claim, at every position the machine emits precisely the keywords that end there. So the machine is valid: it reports keyword y ending at position i exactly when y is a substring of the text ending at i. Correct, all of them, single pass.

Now is it actually fast — is the search really independent of the keyword count? The construction first: building the trie and tagging outputs is one walk per keyword, so O(sum of keyword lengths). The failure computation: the only thing to worry about is the inner while-loop, state←f(state), which could in principle run many times. But the same depth potential still controls it. Along any one keyword path, every goto down the trie raises depth by one and every failure step lowers depth by at least one, so the failure steps charged while that path is being made consistent are fewer than the number of symbols on the path. The construction processes shared trie edges only once; if I upper-bound it by charging each processed edge to any keyword path that contains it, summing over all keywords can only overcount. So the total number of state←f(state) executions is O(sum of keyword lengths). With the output merge represented by linked lists, the merge is constant time at each edge because output(s) and output(f(s)) are disjoint at that moment; the small Python version below materializes sets for clarity, but the abstract machine gets the linear construction bound. Both build steps are linear in the total keyword length — and nothing in there depends on the text.

The search itself: in each cycle the machine makes zero or more failure transitions and then exactly one goto. Same potential argument with depth. A trie goto can raise depth by at most one — and a root self-loop raises it by zero — while every failure transition lowers depth by at least one. The first failure cannot occur until an earlier goto has raised the depth above zero, and depth never goes negative. So over n input symbols, the total number of failure transitions is strictly less than the n goto transitions, and the total number of state transitions is under 2n. Fewer than 2n state transitions to scan a text of length n, *regardless of how many keywords there are or how long they are.* That is exactly the property I was after at the very start — the keyword count has dropped out of the search cost entirely. (For a single keyword this collapses to the KMP bound; their finer analysis even shows at most O(log d) failure transitions in any one cycle for one pattern, via the Fibonacci-string worst case, but the simple amortized 2n is all I need for the multi-keyword machine.) The only remaining cost that *can* depend on the keywords is printing the answer: if a flood of keywords genuinely ends at many positions — the degenerate {a, a², …, a^k} against aⁿ — then there's that much output to emit, but any correct algorithm must emit the same answer, so that's intrinsic, not overhead.

One refinement I can see, the same one KMP noticed. My failure link from a node sometimes points to a node that will *also* immediately fail on the same symbol, costing a wasted hop. If state 4 ("sh") fails on a symbol that isn't 'e', I go to f(4)=1 ("h"); but if the only continuation from 1 were 'e' as well — if "his" weren't in the set — then 1 would fail too on that symbol and I'd be re-testing something I already know fails. KMP's `next` table fixes this: define an optimized link f′ where f′(s)=f′(f(s)) whenever the failure target offers no goto symbol that s doesn't already offer, and f′(s)=f(s) otherwise — collapsing chains of links that can't possibly help. It saves the occasional redundant hop. But there's a cleaner way to kill *all* failure transitions if I want to: precompute, for every state and every symbol, the state I'd end up in after doing all the failure hops and then the goto — a genuine deterministic next-move function δ. Define δ(0,a)=goto(0,a) for the root, and for any other state r, δ(r,a)=goto(r,a) if that edge exists, else δ(r,a)=δ(f(r),a) — and since I build δ in breadth-first order, δ(f(r),·) is already filled in because f(r) is shallower. Then search is just state←δ(state, a), exactly one transition per character, no inner loop at all. That's the full DFA. It's at most twice as fast in transition count (the 2n becomes n), but it's bigger to store, and in practice the machine sits in the root state most of the time — where there are no failure transitions to save anyway — so the speedup "would virtually never be achieved." That's why the failure-link version is the one to keep by default, with the DFA available when the transitions genuinely dominate. For storing goto itself, the same practical note: a full two-dimensional state×alphabet array gives O(1) lookups but eats memory when the alphabet and keyword set are large; storing only the non-fail edges as a small per-state list costs memory proportional to the real edges, with hot states like the root kept as direct-index tables. A sensible compromise, and it doesn't touch the asymptotics.

So the whole thing falls out as: build a trie of all the keywords with a self-looping root (the goto graph), breadth-first-compute a failure link on every node generalizing KMP's longest-viable-suffix rule from one pattern to the set, fold suffix-keyword reports along those links so every node announces all keywords ending there, then run one pass over the text taking a goto when I can and a failure link when I can't, emitting matches as I go — fewer than 2n non-output transitions, build linear in total keyword length, search cost independent of the number of keywords, and optionally collapse the failure transitions into a one-move-per-symbol deterministic automaton. Here it is.

```python
from collections import deque


class AhoCorasick:
    def __init__(self):
        self.goto = [{}]        # goto[state][symbol] -> state; missing == fail
        self.output = [set()]   # keywords that END at this state
        self.fail = [0]         # failure link: deepest proper-suffix state

    def add_keyword(self, word):
        state = 0
        for ch in word:
            nxt = self.goto[state].get(ch)
            if nxt is None:
                nxt = len(self.goto)
                self.goto.append({})
                self.output.append(set())
                self.fail.append(0)
                self.goto[state][ch] = nxt
            state = nxt
        self.output[state].add(word)

    def build(self):
        queue = deque()
        for ch, s in self.goto[0].items():
            self.fail[s] = 0
            queue.append(s)
        while queue:
            r = queue.popleft()
            for ch, s in self.goto[r].items():
                queue.append(s)
                state = self.fail[r]
                while ch not in self.goto[state] and state != 0:
                    state = self.fail[state]
                self.fail[s] = self.goto[state].get(ch, 0)
                if self.fail[s] == s:
                    self.fail[s] = 0
                self.output[s] |= self.output[self.fail[s]]
        return self

    def search(self, text):
        state = 0
        for i, ch in enumerate(text):
            while ch not in self.goto[state] and state != 0:
                state = self.fail[state]
            state = self.goto[state].get(ch, 0)
            for word in self.output[state]:
                yield (i - len(word) + 1, word)


def build_matcher(keywords):
    ac = AhoCorasick()
    for w in keywords:
        ac.add_keyword(w)
    return ac.build()


if __name__ == "__main__":
    ac = build_matcher(["he", "she", "his", "hers"])
    for start, word in sorted(ac.search("ushers")):
        print(f"{word}@{start}")   # she@1, he@2, hers@2
```
