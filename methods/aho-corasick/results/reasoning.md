OK, the concrete thing in front of me is a bibliographer's search that won't finish. The corpus is the cumulated citation tapes — a few years of them, something like a hundred and fifty thousand citations, call it ten million characters. The query is a Boolean function over a list of keywords: find every title that contains both "ion" and "bombardment", say, and there might be a couple dozen keywords in one prescription. The program we have does the obvious thing: for each keyword, walk it down the whole text looking for that keyword, then do the next keyword, and so on. And it is slow enough that someone burned through a six-hundred-dollar machine budget before the search even came back. So let me figure out why it's slow, exactly, and what the cost *should* be.

The cost of the obvious method is plainly (number of keywords) × (length of text), more or less. Each keyword does its own full sweep of the ten million characters. If I have twenty-four keywords I read the text twenty-four times. And that's the whole problem in one line: the keyword count *multiplies* the text length. There's no reason it should. The text is the same text. If I'm standing at one position in the text reading a character, I'd like that single read to make progress on *all* the keywords at once, not on one of them. The number I want the cost to depend on is the length of the text, plus the work to set up whatever machinery, plus however much output we actually have to print — and *not* the number of keywords. That's the target: cost essentially independent of how many keywords are in the query.

So what do I have to build with. There's the single-keyword story, which has just been solved cleanly — Knuth, Morris and Pratt. For one pattern of length m in text of length n you can do O(m+n), and the trick is that the text pointer never backs up. You precompute a little table off the pattern: when you've matched the first j−1 characters and then the j-th fails, you don't slide the pattern over by one and re-read; you slide it by exactly the amount that keeps the longest already-matched prefix still aligned. Concretely their table entry — call it f[j] — is the length of the longest proper prefix of pattern[1..j−1] that is also a suffix of it. The longest *border*. You failed at position j, but maybe the last few characters you matched are themselves the start of the pattern, so you don't throw all that away; you fall back to the end of that shorter already-matched prefix and keep going from the same text character. And they compute that table in O(m) by, beautifully, running the matching procedure of the pattern against itself. The thing I want to hold onto from KMP is not the single-pattern bookkeeping; it's the *shape* of the idea: on a failure, the most you can keep is "the longest matched prefix that's still viable," and that quantity depends only on where you were, so you can precompute it.

Now I have a *set* of keywords, not one. What's the right structure for a set of strings? A trie. Insert each keyword as a path from a root; shared prefixes share nodes. So "he" and "hers" and "his" all leave the root on "h" and walk down the same first edge; "she" leaves on "s". Reading one text symbol moves me one edge down. And here's the nice part: a path from the root spelling out u means I have currently matched u as a substring ending right here, and u is a prefix of at least one keyword. If a node is the end of a complete keyword, I tag it with that keyword. So walking the text through the trie matches, at the current position, every keyword that *starts* where my current walk started — all of them simultaneously, in one descent. That's already the "all keywords at once" property I wanted, at least for keywords anchored at the same start.

But it only handles keywords that start at my walk's start. The moment I'm at a node and the next text character has no edge out of that node, I've fallen off the trie. With one pattern KMP told me how far to back off; here I have no such rule for the tree, so the naive fix is to give up, jump back to the root, restart the walk at the *next* text position, and try again. And that's a per-position restart — I'm re-reading the text once per starting position, which is exactly the k·n disease in a different costume. So the trie alone isn't enough. What's missing is precisely a KMP-style failure rule, but for the tree instead of the line.

Let me think about what "fall back" should mean on the trie. Suppose I'm sitting at the node whose path spells "she" and the next character fails — there's no edge for it. I've just read the text characters …s,h,e ending here. I can't keep "she", fine. But what *can* I keep? The longest suffix of "she" that is still the prefix of some keyword. The suffix "he" is a prefix of the keyword "he" and of "hers". So I shouldn't go all the way back to the root and lose everything; I should jump to the node for "he" and try the failing character from *there*. That jump — from the node for "she" to the node for "he" — is exactly KMP's failure idea, lifted from one pattern to the whole set. So I want, for every node s with path-string u, a link to the node whose string is the longest proper suffix of u that is also a prefix of some keyword. Call it the failure link f(s). On the line, KMP's f was "longest proper prefix that's also a suffix of the same pattern." On the tree it seems to become "longest proper suffix of my string that is also a prefix of *some* keyword" — and "is a prefix of some keyword" is the same as "is the string of some node in the trie," so f(s) would be the deepest *other* node whose string is a suffix of s's string. The content I'm carrying over from KMP is "keep the longest still-viable matched piece," but the "still-viable matched piece" now has to range over all keywords rather than one pattern. I don't yet know that following these links gives a single-pass-for-all-keywords scan; that's a claim I'll have to actually check once I can compute f and run the machine.

Let me get the search loop straight first, assuming I have these links. Current state s, current symbol a. If there's a trie edge on a out of s — a goto — take it: that's a longer match, advance. If there's no edge, follow the failure link to f(s) and try a again from there; keep following failure links until either some node has an edge on a, or I get back to the root. From the root, if there's still no edge on a, I just stay at the root and move on to the next character. I want to make that last case clean, so let me give the root a self-loop on every symbol that doesn't start any keyword: define the goto of the root on such a symbol to be the root itself, never "fail". Then the root never fails, the failure-following always terminates, and every step consumes exactly one input symbol with no special case. That's worth doing — without the self-loop I'd need an extra guard for "fell off at the root," and the termination argument for the inner loop would be murkier.

There's one more thing the search has to do, and staring at "she" shows it. When I reach the node for "she", I've also, sitting inside it, just matched "he" — "he" is a suffix of "she" and "he" is itself a keyword. So at that single position I must report *both* "she" and "he". A node has to announce not only the keyword that ends exactly there, but every keyword that is a suffix of its string. And "keyword that is a suffix of my string" is exactly the information sitting along the failure chain: f("she") is "he", and "he" has a terminal tag. I could chase the whole failure chain at every text position, but that would inspect empty suffix states over and over. Better: while building the machine, give each state an output link to the nearest failure ancestor that has a real keyword ending there; if the failure ancestor is empty, inherit its output link. Then reporting from a state is local terminal tags first, followed by the terminal tags reached through output links. Conceptually that is the same output function — all keywords whose strings are suffixes of this state's string — but the code stores it without copying all suffix outputs into every state.

Now the real question: how do I actually *compute* f for every node? I can't compute f(s) before I know f for shorter strings, because the definition of f(s) leans on suffixes of s, which are shorter. So process nodes in order of increasing depth — breadth-first from the root. Depth-1 nodes, the single characters, have no proper non-empty suffix that's a prefix of a keyword, so their failure link is the root: f = 0 for all depth-1 states. Good, base case. Now suppose I've computed f for everything shallower than depth d, and I want f for a depth-d node. Every depth-d node s is reached by exactly one edge: there's a parent r at depth d−1 and a symbol a with goto(r,a)=s. The path-string of s is the string of r followed by a. I want the longest proper suffix of that which is a prefix of some keyword. A proper suffix of "r's string followed by a" has to be a suffix of r's string, followed by a; it cannot start from r itself, because that would be the whole string again. So I should begin at the best proper suffix of r, then the next best, and so on: f(r), f(f(r)), … . Set state = f(r); while there's no goto on a out of state, set state = f(state); when goto(state,a) succeeds, that target is f(s). And because the root has a goto on every symbol, this while-loop always terminates. After setting f(s), set the output link of s to f(s) if f(s) has a terminal keyword, and otherwise to the output link already computed for f(s). Process the states in breadth-first order so that whenever I compute f(s), both f(s) and its output link are already known.

Let me run the whole thing on a small set to make sure I believe it: keywords {he, she, his, hers}. Build the trie. Root is 0. "he": 0→1 on h, 1→2 on e, tag state 2 with "he". "she": 0→3 on s, 3→4 on h, 4→5 on e, tag 5 with "she". "his": h already goes 0→1, then 1→6 on i, 6→7 on s, tag 7 with "his". "hers": h is 0→1, e is 1→2, then 2→8 on r, 8→9 on s, tag 9 with "hers". Then add the root self-loop on every symbol other than h and s. Now the failure links, by depth. Depth 1: states 1 and 3 both have f=0. Depth 2: for state 2, parent 1 and symbol e give state=f(1)=0 and goto(0,e)=0, so f(2)=0. For state 6, parent 1 and symbol i give f(6)=0. For state 4, parent 3 and symbol h give state=f(3)=0 and goto(0,h)=1, so f(4)=1. Depth 3: for state 5, parent 4 and symbol e give state=f(4)=1 and goto(1,e)=2, so f(5)=2 — and because state 2 is terminal for "he", output-link(5)=2. There it is: the node for "she" reports its own local output "she" and then follows the output link to report "he". For state 7, parent 6 and symbol s give f(7)=3; state 3 is not terminal, so there is no output link. For state 8, parent 2 and symbol r give f(8)=0. Depth 4: state 9 has parent 8 and symbol s, so f(9)=3, again with no output link through state 3. So by hand f over states 1..9 is 0,0,0,1,2,0,3,0,3, with the one nontrivial output link out_link(5)=2. I don't want to trust a hand trace of nine states, so I code the build exactly as derived and dump the arrays: it prints fail = [0,0,0,0,1,2,0,3,0,3] (index 0 the root) and out_link[5]=2, every other out_link zero. That agrees with what I worked out, so at least the construction does what I think it does on this set.

Now drive "ushers" through it. State 0. 'u': no edge, root self-loop, stay 0. 's': goto(0,s)=3, state 3. 'h': goto(3,h)=4, state 4. 'e': goto(4,e)=5, state 5 — the local output is "she", and output-link(5)=2 gives "he", so at this position report both. 'r': goto(5,r) fails; follow f(5)=2; goto(2,r)=8; state 8, no output. 's': goto(8,s)=9; state 9 — local output "hers", report it. So in "ushers" I get "she" and "he" ending at the 'e', and "hers" ending at the final 's', overlapping the earlier matches — as (start, keyword) pairs, {(1,"she"), (2,"he"), (2,"hers")}. I want to be sure that's the *complete* answer and I haven't silently dropped or invented a match, so I check it the dumb exhaustive way: enumerate every substring of "ushers" by start and end, keep the ones that are in {he,she,his,hers}, and compare that set against what the machine emitted. Brute force gives exactly {(1,"she"), (2,"he"), (2,"hers")} and the two sets are equal. So on this input the single pass found all of them and only them, overlaps included, and the work didn't care that there were four keywords rather than one. (Note "his" never fires — there's no "his" in "ushers" — which is the right kind of negative to see.)

I should make sure this is actually *correct*, not just right on one example, because the whole edifice rests on f meaning what I claimed. Let me prove the failure-link characterization: f(s)=t if and only if the string v of t is the longest proper suffix of the string u of s that is also a prefix of some keyword. Induct on the depth of s, equivalently on |u|. Base: depth 1, u is a single symbol, its only proper suffix is the empty string, represented by the root; and I set f=0 there. Holds. Inductive step: u = a₁…a_j, j>1; let r be the parent, so r's string is a₁…a_{j−1} and a_j is the last symbol. Consider the chain r₁=f(r), r₂=f(r₁), … . By the inductive hypothesis applied to r, r₁'s string is the longest proper suffix of a₁…a_{j−1} that's a prefix of some keyword; r₂'s string is the longest proper suffix of *that* which is a prefix of some keyword; and so on — so as I walk the chain I'm enumerating the viable suffixes of a₁…a_{j−1} in strictly decreasing length. My loop walks this chain skipping any r_i that has no goto on a_j, and stops at the first r_n with goto(r_n, a_j)=t. The string of t is the string of r_n followed by a_j. Since r_n's string is the longest viable suffix of a₁…a_{j−1} that can be extended by a_j, appending a_j gives the longest proper suffix of u=a₁…a_j that is a prefix of some keyword. And I set f(s)=t=goto(r_n,a_j). So f(s) is exactly that longest viable suffix-state. Done — and note the loop reaches the root in the worst case, where goto on a_j always succeeds because of the self-loop, so there's always such an r_n.

Then the output links. Claim: the reports made from state s are exactly the keywords that are suffixes of s's string u. A local terminal tag is obviously correct: if u itself is a keyword, that keyword ends here. For proper suffix keywords, the failure-link characterization says the failure chain from s walks the viable proper suffix states of u in decreasing length. The construction of the output link skips the nonterminal states on that same chain and lands on the nearest terminal one; repeating output links therefore visits exactly the terminal states on the failure chain, and no other states. So every keyword reported through an output link is a suffix of u. Conversely, if a keyword y is a proper suffix of u, its trie node lies somewhere on the failure chain from s; since that node is terminal, the repeated nearest-terminal jumps cannot skip it and must eventually visit it. The two directions match, so local output plus output-link output is exactly right — every keyword ending at this position, and only those.

And the search invariant: after processing the j-th text character, the machine is in the state whose string is the longest suffix of a₁…a_j that is a prefix of some keyword. The same suffix argument works against the text instead of against a single trie edge: the goto-then-failure loop, starting from the previous state, follows failure links until it can extend by a_j, which lands on the longest viable suffix of a₁…a_j. Combined with the output-link claim, at every position the machine emits precisely the keywords that end there. So the induction says the machine reports keyword y ending at position i exactly when y is a substring of the text ending at i — every match, only matches, in one pass. That's the conclusion I was trying to reach, and the small "ushers" exhaustiveness check is at least consistent with it. To get a little more evidence than a proof-plus-one-example, I run the machine against a brute-force all-substrings matcher on 200 random fifteen-character strings over the keyword alphabet; the two agree on every one. Not a proof of correctness, but it would have caught an off-by-one in the reported start position or a missed suffix output, and it doesn't.

Now is it actually fast — is the search really independent of the keyword count? The construction first: building the trie and tagging outputs is one walk per keyword, so O(sum of keyword lengths). The failure computation: the only thing to worry about is the inner while-loop, state←f(state), which could in principle run many times. But the same depth potential still controls it. Along any one keyword path, every goto down the trie raises depth by one and every failure step lowers depth by at least one, so the failure steps charged while that path is being made consistent are fewer than the number of symbols on the path. The construction processes shared trie edges only once; if I upper-bound it by charging each processed edge to any keyword path that contains it, summing over all keywords can only overcount. So the total number of state←f(state) executions is O(sum of keyword lengths). The output-link assignment is constant time at each edge: check whether f(s) has local output, otherwise copy the already-computed output link of f(s). Both build steps are linear in the total keyword length — and nothing in there depends on the text.

The search itself: in each cycle the machine makes zero or more failure transitions and then exactly one goto. Same potential argument with depth. A trie goto can raise depth by at most one — and a root self-loop raises it by zero — while every failure transition lowers depth by at least one. The first failure cannot occur until an earlier goto has raised the depth above zero, and depth never goes negative. So over n input symbols, the total number of failure transitions is strictly less than the n goto transitions, and the total number of state transitions is under 2n. Fewer than 2n state transitions to scan a text of length n, regardless of how many keywords there are or how long they are. The potential argument is the kind of thing that's easy to fool myself about, so I instrument the loop with two counters and read off the actuals. On "ushers" through the four-keyword machine: 1 failure transition and 6 gotos, total 7, comfortably under 2·6=12. For the case that's supposed to stress failure transitions — the degenerate {a, a², …, a^6} against a⁴⁰, where the trie is one long "a"-chain and matches end at almost every position — I count 34 failure transitions and 40 gotos, total 74, still under 2·40=80, and the strict "fewer failures than gotos" inequality holds (34 < 40). So the measured counts sit under the bound, the keyword count really has dropped out of the search cost, and the only thing left that scales with the keywords is printing the answer: that same a⁴⁰ run emits 225 matches, because every a^i ends at every late-enough position, but any correct algorithm must emit that same set, so it's intrinsic output, not matching overhead.

One refinement I can see, the same one KMP noticed. My failure link from a node sometimes points to a node that will also immediately fail on the same symbol, costing a wasted hop. If state 4, the node for "sh", fails on a symbol that isn't e, I go to f(4)=1, the node for "h"; but if the only continuation from 1 were e as well — if "his" weren't in the set — then 1 would fail too on that symbol and I'd be re-testing something I already know fails. KMP's next table fixes this by collapsing chains of links that cannot possibly help. But there's a cleaner way to kill all failure transitions if I want to: precompute, for every state and every symbol, the state I'd end up in after doing all the failure hops and then the goto — a genuine deterministic next-move function δ. Define δ(0,a)=goto(0,a) for the root, and for any other state r, δ(r,a)=goto(r,a) if that edge exists, else δ(r,a)=δ(f(r),a) — and since I build δ in breadth-first order, δ(f(r),·) is already filled in because f(r) is shallower. Then search is just state←δ(state, a), exactly one transition per character, no inner loop at all. That's the full DFA. Before I believe it's the *same* machine, I build δ over the alphabet of the four-keyword example and run a δ-driven search against the failure-link search on 200 random texts; they produce identical match sets every time, so the determinization didn't change the language, only how I get there. It's at most twice as fast in transition count, but it's bigger to store, and in practice the machine sits in the root state most of the time, where there are no failure transitions to save anyway — the degenerate run above is the *favorable* case for δ and even there it only saves 34 transitions out of 74. On ordinary text the saving would be far smaller, so the speedup would virtually never be worth the storage. That's why the failure-link version is the one to keep by default, with the DFA available when the transitions genuinely dominate. For storing goto itself, the same practical note: a full two-dimensional state-by-alphabet array gives O(1) lookups but eats memory when the alphabet and keyword set are large; storing only the non-fail edges as a small per-state list costs memory proportional to the real edges, with hot states like the root kept as direct-index tables. A sensible compromise, and it doesn't touch the asymptotics.

So the whole thing falls out as: build a trie of all the keywords with a self-looping root, breadth-first-compute a failure link on every node generalizing KMP's longest-viable-suffix rule from one pattern to the set, add output links so suffix keywords are reported without re-walking empty failure states, then run one pass over the text taking a goto when I can and a failure link when I can't, emitting matches as I go — fewer than 2n non-output transitions, build linear in total keyword length, search cost independent of the number of keywords, and optionally collapse the failure transitions into a one-move-per-symbol deterministic automaton. Here it is as a single-file program: it reads `k`, then `k` keywords, then the text, and prints every `start_index keyword` match (0-indexed start, overlaps included), sorted by `(start, keyword)`. The build folds the failure recovery straight into `go` so each symbol is one deterministic transition (the `to_dfa` refinement), keeping the inner search loop branch-free.

If I am not confident I can correctly BFS-propagate goto and failure links into the unified `go` DFA table within the budget, I will keep the simpler explicit failure-link search loop and ship that correct version instead.
