I start with a Transformer that is already useful, so I do not want to change the modeling story unless the resource arithmetic forces me to. The first arithmetic check is almost embarrassing: a very large reported Transformer layer has about $0.5$B parameters, which is about $2$GB in 32-bit floats. A batch of $8$ sequences of length $64\text{K}$ with width $1024$ is also about $64\text{K}\times 1\text{K}\times 8 \approx 0.5$B floats, another $2$GB. If I only had to hold one layer's parameters and one layer's activations, a long-sequence model should fit on a single accelerator. It does not, so the estimate is missing the real multipliers.

The missing terms are three different failures of the same hope. Backpropagation stores activations for every layer, so the single-layer activation cost is multiplied by $n_l$. The feed-forward sublayer is position-wise but wide, often $d_{ff}=4096$ when $d_{model}=1024$, so one intermediate is $[b,l,d_{ff}]$ rather than $[b,l,d_{model}]$. Attention then forms the $QK^\top$ scores; at $l=64\text{K}$ even one head and one sequence gives a $64\text{K}\times64\text{K}$ matrix, about $16$GB in fp32. I cannot fix one of these and declare victory. The useful target is to remove the $l^2$ attention term, the $n_l$ activation multiplier, and the $d_{ff}$ peak-memory multiplier.

The attention term is the most structural, so I attack it first. For query $q_i$, dense attention computes a softmax over all dot products $q_i\cdot k_j$. The softmax is dominated by large logits, so most far keys contribute almost nothing — and if that is true, I am paying $l^2$ to compute a matrix whose rows are mostly negligible. The temptation is to keep, for each query, only the keys with the largest dot products. But I cannot find those keys without first computing all dot products, which is exactly the quadratic work I am trying to avoid. So the real question is whether I can guess the large-dot-product keys *cheaply*, before forming the scores.

Cheaply finding near neighbors in a high-dimensional space is what locality-sensitive hashing does. If I hash vectors so that angularly close ones land in the same bucket, then "keys with large $q_i\cdot k_j$" become "keys in $q_i$'s bucket", and I never form the dense matrix. Dot product is a query norm times a cosine once keys are unit-normalized, so an angular hash is the right family. A simple angular scheme is to draw $R\in\mathbb{R}^{d_k\times b/2}$, compute $xR$, concatenate the negation, and define
$$h(x)=\arg\max([xR;\,-xR]).$$
That gives $b$ signed-axis buckets from $b/2$ random directions. Nearby directions collide with high probability, far directions usually do not, and the whole operation is a batched matrix multiply plus an argmax.

There is a mismatch before hashing can be used inside a Transformer. In ordinary multi-head self-attention, $Q$ and $K$ come from different learned projections. A query and the key it should attend to can have a large dot product without living in a common hash geometry, so "same bucket" would not track "should attend". The clean way to make those agree is to share the query-key projection: produce one $QK$ tensor and a separate $V$ tensor. Then I use the query as the unnormalized query vector, but a unit-normalized version of the same vector as the key, $k_j=q_j/\lVert q_j\rVert$. That keeps the key direction aligned with the angular hash while leaving $\lVert q_i\rVert$ as a per-query softmax temperature.

Sharing $QK$ has a consequence I should check rather than wave past: now a position's own key is $q_i/\lVert q_i\rVert$, so the self logit is $q_i\cdot(q_i/\lVert q_i\rVert)=\lVert q_i\rVert$. For any other key, $q_i\cdot k_j=\lVert q_i\rVert\cos\theta_{ij}\le\lVert q_i\rVert$. So the self logit is the *largest* logit in every row. I confirmed this numerically on random $q$: building $k=q/\lVert q\rVert$ and the score matrix $S=qk^\top$, the diagonal equals the row maximum for every row, and equals $\lVert q_i\rVert$ exactly. That is a problem — if I let a position attend to itself freely it will mostly attend to itself, and the layer drifts toward identity. I will have to suppress the self term explicitly, which I return to once the batching is settled.

Now I can write the attention target for one position as
$$o_i=\sum_{j\in\mathcal{P}_i}\exp(q_i\cdot k_j-z(i,\mathcal{P}_i))v_j,$$
where $z(i,\mathcal{P}_i)=\log\sum_{j\in\mathcal{P}_i}\exp(q_i\cdot k_j)$. In causal attention the dense allowed set would be $\mathcal{P}_i=\{j:j\le i\}$. For hashed attention I want the content set $\mathcal{P}_i=\{j:h(q_i)=h(k_j)\}$, intersected with the causal mask when the model is autoregressive. For batching I may compute over a larger chunk-local set $\widetilde{\mathcal{P}}_i$ and subtract a mask term $m(j,\mathcal{P}_i)$, with $m=\infty$ outside the true set and $0$ inside it.

The next obstacle is not mathematical but mechanical: hash buckets are ragged. A GPU wants fixed-size matrix multiplications, not a variable number of keys per query. I sort positions by the pair $(h(q_i),i)$, so items in the same bucket become contiguous and sequence order breaks ties. Then I cut the sorted sequence into chunks of length $m$ and let each chunk attend to itself and one previous chunk. In sorted coordinates the batch set is
$$\widetilde{\mathcal{P}}_i=\left\{j:\left\lfloor\frac{s_i}{m}\right\rfloor-1\le \left\lfloor\frac{s_j}{m}\right\rfloor\le \left\lfloor\frac{s_i}{m}\right\rfloor\right\}.$$
My first instinct was to claim that if every bucket is smaller than $m$, this chunk-plus-one-back rule covers all true same-bucket partners — the bucket fits inside two adjacent chunks, so surely both members are reachable. Before relying on it I checked a tiny case by brute force, and it is false. Take $m=3$ and three buckets of size two each, sorted as `[A A B B C C]` at positions $0..5$; chunks are $\{0,1,2\}$ and $\{3,4,5\}$. Bucket B straddles the boundary: position $2$ is in chunk $0$, position $3$ in chunk $1$. The query at position $2$ may look at its own chunk and the *previous* one, but its partner at position $3$ is in the *next* chunk, so it is missed. Sweeping random size distributions with every bucket strictly below $m$, this happens in essentially every sequence. The look-one-back rule is asymmetric, and a boundary-straddling bucket always loses the partner that sits on the later side from the earlier member's view. So chunking is not an exact cover of the same-bucket set; it is an approximation, and I should size and repeat it to make the misses rare rather than pretend they are gone. Since the average bucket has size $l/n_{buckets}$, I choose $m=2l/n_{buckets}$, equivalently $n_{buckets}=2l/m$, so the chunk is twice the average bucket size and most buckets sit comfortably inside a chunk pair. Sorting contributes the $\log l$ factor; the chunked dot products scale with $l$ times a fixed local window and the number of hash rounds.

The boundary misses, plus the basic fact that a single random rotation can put two genuine neighbors in different buckets, tell me one hash is not enough. I use $n_{rounds}$ independent hashes and take the union of their content sets,
$$\mathcal{P}_i=\bigcup_r \mathcal{P}_i^{(r)},\qquad \mathcal{P}_i^{(r)}=\{j:h^{(r)}(q_i)=h^{(r)}(q_j)\}.$$
Combining rounds is where I have to be careful, because each round computes its own local softmax and the same key can appear in several rounds. The obvious recipe is to rebase each round's output by its log-partition onto the union partition and add:
$$o_i=\sum_r \exp(z(i,\mathcal{P}_i^{(r)})-z(i,\mathcal{P}_i))\,o_i^{(r)}.$$
I worried this double-counts overlapping keys, so I tested it on a toy query with two rounds seeing key sets $\{0,1,2,3,4\}$ and $\{3,4,5,6,7\}$ (overlap on $3,4$). Comparing the rebased sum against the exact softmax over the union — each distinct key counted once — the rebased sum is off by $0.27$ in the output vector. So the worry is real: keys $3$ and $4$ get counted twice and the result is wrong. The fix is to give each appearance a fractional weight. Let $N_{i,j}=|\{r:j\in\mathcal{P}_i^{(r)}\}|$ and subtract $\log N_{i,j}$ inside the round mask, so a key present in $N_{i,j}$ rounds gets weight $1/N_{i,j}$ each, summing to one. Putting $-\log N_{i,j}$ into both rounds' logits before the local softmaxes and the rebasing, I re-ran the same toy: the corrected combination now matches the exact union softmax to $5.6\times10^{-17}$. So the round combiner is only exact as a set-union with the $\log N_{i,j}$ correction; the rebasing sign — the local round denominator removed by $o_i^{(r)}$, then $\exp(z^{(r)}-z)$ moving that local softmax onto the union denominator — is what carries each round onto the common scale.

The masks have two separate jobs. Causality cannot use sorted order, because sorting has scrambled positions, so I carry the original position index through the same permutation and compare original indices inside each chunk. The shared-QK self case is the one I flagged earlier: since the self logit is the row maximum, I must subtract a penalty for $i=j$. The question is whether the penalty should be infinite. I checked the degenerate row directly: the first causal token has no legal key $j<i$, so after the causal mask its only finite entry is its own self logit. With a finite penalty of $10^5$, that lone row is $\lVert q_0\rVert-10^5$; its softmax over a single finite entry is exactly $1.0$ and $\log z$ is finite — the row is well defined. With an infinite penalty the only entry becomes $-\infty$, $\log z=-\infty$, and the softmax is NaN. So the penalty must be finite. It still suppresses self-attention whenever a real key exists: $\exp(-10^5)$ underflows to $0$, so against any genuine bucket-mate the self contribution vanishes, while a token left alone can still attend to itself and stay numerically sane.

So the attention part becomes shared-QK, angular hashing, sorting by bucket and position, chunk-local attention with one previous chunk, causal and self masks from carried position indices, and multiple independent hashes combined with the duplicate correction. The dense $l^2$ score matrix is gone. With fixed chunk length and $n_{buckets}=2l/m$, the remaining attention work is linear in length apart from sorting and the number of rounds.

That still leaves the layer-storage multiplier. A usual residual layer $y=x+F(x)$ cannot be inverted without solving for $x$, but a reversible block can. I split the state into two full-width halves and compute
$$y_1=x_1+F(x_2),\qquad y_2=x_2+G(y_1).$$
Given $(y_1,y_2)$, I recover
$$x_2=y_2-G(y_1),\qquad x_1=y_1-F(x_2).$$
This is the load-bearing claim for the whole memory saving, so I verified the round-trip rather than trusting the algebra: with $F,G$ as small random $\tanh$ maps, computing $(y_1,y_2)$ forward and then inverting recovers $x_1,x_2$ to $4.4\times10^{-16}$ — machine precision, no information lost. For a Transformer block, $F$ is the attention sublayer and $G$ is the feed-forward sublayer, with normalization inside those residual functions. During the backward pass I reconstruct each layer's inputs from its outputs and immediately compute the local gradients. I do not need to cache every layer's activations; the $n_l$ activation multiplier disappears. The recomputation does assume $F$ and $G$ are deterministic — if dropout or any other randomness lives inside them, the reverse pass must replay the same random choices as the forward pass, or the reconstructed inputs will not match.

The feed-forward multiplier is easier because the FFN is independent across sequence positions. I do not need the $[b,l,d_{ff}]$ intermediate for all positions at once. I split the sequence axis into chunks, apply the same feed-forward network to one chunk at a time, and concatenate the results:
$$Y_2=\left[X_2^{(1)}+\mathrm{FF}(Y_1^{(1)});\ldots;X_2^{(c)}+\mathrm{FF}(Y_1^{(c)})\right].$$
Because the FFN acts pointwise along the length axis, concatenating per-chunk outputs is exactly the dense FFN output — this changes peak memory but not the function. The same chunking applies to the reverse computation, the backward pass, and the final large-vocabulary log-probability calculation.

Putting the pieces together, the irreducible activation scale is roughly $bld_{model}$, but the former multipliers around it have been removed. LSH attention removes the $l^2$ attention term, reversible layers remove the depth multiplier on activations, and feed-forward chunking removes the peak $d_{ff}$ intermediate. Parameters still grow with depth, so a very deep model may also stream inactive layer parameters from host memory; the long batch-times-length computation gives enough work per layer to amortize that transfer. The result is still a Transformer in its modeling ingredients, but its memory behavior is no longer dictated by dense attention scores and stored per-layer activations.
