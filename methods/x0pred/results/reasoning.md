I want to begin from the sampler rather than from the historical loss, because the sampler is the thing that will consume whatever the network emits. The forward noising equation is `x_t = a_t x_0 + b_t eps`, with `a_t^2 + b_t^2 = 1`. A DDIM step is easiest to read as: estimate the clean image, infer the corresponding noise direction from the current latent, then move to the next noise level by recombining those two pieces. In symbols, once I have `x_hat`, the implied noise is `eps_hat = (x_t - a_t x_hat) / b_t`, and the deterministic step is `x_prev = a_prev x_hat + b_prev eps_hat`. So the object that sits in the middle of every step is not the raw network output by itself; it is the clean-image estimate produced from that output.

The default answer is to make the network output `eps`. I understand why. The Gaussian score is `grad_{x_t} log q(x_t | x_0) = -(x_t - a_t x_0) / b_t^2 = -eps / b_t`, so noise prediction is score prediction up to a known positive scale. Its loss also has a useful weighting. If `x_hat = (x_t - b_t eps_hat) / a_t`, then `eps - eps_hat = (a_t / b_t)(x_hat - x_0)`, so flat noise MSE is `(a_t^2 / b_t^2) ||x_hat - x_0||^2`, an SNR-weighted clean-image loss. That pours capacity into the low-noise detail end and gives little weight to the pure-noise end, which made sense for many-step sampling.

But the same inversion is exactly where the few-step trouble lives. The clean-image error induced by a small epsilon error is multiplied by `b_t / a_t`. At high noise, `a_t` approaches zero, so the clean-image estimate can change violently even if the raw epsilon output changes only a little. At the zero-SNR endpoint there is no clean-image estimate implied by epsilon at all: `x_t` is pure noise and the formula divides by zero. With hundreds of steps, clipping and later denoising can repair a bad early estimate. With a compressed deterministic sampler, the first few low-SNR steps are too large to treat that as harmless.

So I separate two questions that are easy to blur together. The parameterization question is: what coordinate should the raw output live in, and how do I turn it into `x_hat`? The loss-weighting question is: after I choose that coordinate, how much should each noise level contribute to training? Epsilon prediction couples the two because flat epsilon MSE automatically creates an SNR-weighted clean-image loss. If I switch coordinates, I should not pretend the weighting question disappears; it just becomes explicit.

For the parameterization itself, the most direct stable choice is to put the network output in the same space as the sampler's central quantity. Train the network to emit the clean image. Then the recovery function is the identity: the raw output already is `x_hat`. There is no division by `a_t`, so the low-SNR clean-image estimate remains a bounded tensor supplied by the model rather than an amplified residual. This does not claim that a flat data-space loss is always the best weighting. Plain `||x_hat - x_0||^2` gives equal weight to all noise levels, including very noisy latents where many clean images are compatible with the same observation and the squared-error optimum is a conditional mean. If the harness has a weighting hook, I would use the stable x-output coordinate with a low-SNR-preserving weight such as truncated SNR. But the two editable functions in front of me only define the coordinate and inverse, and those functions are unambiguous.

I need to check the endpoint cases. At low noise, `a_t -> 1` and `b_t -> 0`, the target is just the almost-visible image, so the model is being asked to copy fine detail from an input that already contains it. The DDIM formula may form an implied `eps_hat` by dividing by small `b_t`, but the final step can return `x_hat` directly, and the clean-image estimate itself has no singularity. At high noise, `a_t -> 0` and `b_t -> 1`, the model is being asked for a hard denoising prediction from nearly pure noise; that is statistically hard, but it is not an algebraic blow-up. The output is still a candidate image, and the sampler can use it directly. That is the distinction I care about: hard regression is a capacity and weighting issue; division by a vanishing coefficient is a parameterization issue.

The implementation therefore has no schedule arithmetic in the two functions. That absence is not an omission; it is the point. The target is the clean tensor that came from the dataset, and the inverse conversion is the identity on the network output.

Now I check that against a full reference implementation rather than just the algebra. In the Google diffusion-distillation code, the `mean_type == "x"` branch sets `model_x = model_output`; the epsilon and velocity branches are the ones that perform conversions. The DDIM step then uses `model_x` and the implied `model_eps`. The CIFAR configuration sets `mean_type = "x"`, so this raw-output interpretation is exactly the data-coordinate parameterization used there. The same config also sets `mean_loss_weight_type = "snr_trunc"`, which means a fully faithful CIFAR run does not use unweighted x-MSE: it keeps the x-output parameterization and weights x-space error by `max(SNR, 1)` through `maximum(x_mse, eps_mse)`. That is outside the two-function scaffold, but it matters for describing the method faithfully.

Walking back through the chain, the sampler needs `x_hat`; epsilon prediction learns a score-like coordinate but recovers `x_hat` by dividing by `a_t`, which fails at low SNR; direct clean-image prediction makes the network output already be `x_hat`, so the recovery is identity and the output-to-sampler interface is stable. The price is that loss weighting must be handled deliberately. In the minimal scaffold, the correct coupled functions are `target = x_0` and `predict_x0 = identity`; in the canonical CIFAR implementation, that same x-output coordinate is paired with truncated-SNR loss weighting.
