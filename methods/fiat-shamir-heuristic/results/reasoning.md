I start with the identification protocol and I try to locate exactly where the interaction does work. The prover sends a fresh commitment, the verifier answers with a random challenge, and only then does the prover compute the response. The algebraic check at the end is not mysterious by itself; the important pressure is that the response is tied to a commitment that is already fixed. If the prover can answer too many possible challenges for the same commitment, the prover is effectively exposing the hidden square-root information that the hardness assumption says should be unavailable.

At first I am tempted to say that interaction is essential because the verifier must be present. That is too broad. In this protocol the verifier does not contribute a secret, a trapdoor, or a private computation. The verifier's challenge matrix contains no information. Its job is narrower and sharper: it arrives after the commitment and is unpredictable at that moment. The verifier is acting as a public source of fresh challenge bits, not as a knowledgeable participant.

That observation changes the problem. If the verifier's only irreplaceable contribution is unpredictability after the commitment, then I should not try to simulate the whole verifier. I should preserve the causal order. The signer can first create the commitments. Then some public rule must determine the challenge from the message and those commitments. The rule has to be deterministic so every later verifier can recompute it, but it has to behave like a random function so the signer cannot know the challenge before fixing the commitments.

Now the signature problem becomes clearer. An ordinary identification transcript is weak as evidence because the verifier can invent a dialogue after the fact by choosing questions and answers. I need a transcript whose challenge is not under the verifier's later control. If the challenge is forced to equal the output of a public random-looking function on the message and the commitments, then a verifier cannot freely choose it afterward. The transcript has to be self-consistent with a value that everyone else can recompute.

I have to be careful here because it is easy to slide into a useless hash-after-proof idea. Hashing a completed transcript as a digest would only label the transcript; it would not recreate the moment where the prover faces a challenge after committing. The function output must sit exactly in the verifier-challenge slot. The input must include the values that are already fixed at that point, especially the commitments, and in the signature setting it must include the message. Otherwise the prover may be able to choose a context that makes a convenient challenge appear, or reuse a proof where it should not apply.

In the original modular-square-root scheme this means I sample the `r_i` values and compute the commitments `z_i = r_i^2 mod n` first. The public identity values are `u_j = f(I, j)`, and the card's secrets satisfy `s_j^2 * u_j = 1 mod n`, so selected `s_j` factors can be canceled by selected `u_j` factors during verification. I then compute a public function on `m, z_1, ..., z_t` and parse its first `k*t` bits as the whole challenge matrix. Only after that do I compute the responses `y_i = r_i * product_{j:e_ij=1} s_j mod n`. A verifier reconstructs `z_i = y_i^2 * product_{j:e_ij=1} u_j mod n` from the response and the supplied challenge matrix, then checks that the same function on the message and reconstructed commitments gives that exact matrix. The verifier is no longer online; the verifier has become a consistency check against the transcript-determined challenge.

The same reasoning abstracts beyond this particular number-theoretic protocol. In any three-move public-coin proof, I can view the transcript as commitment, challenge, response. If the challenge is only public randomness, I can derive it from a public function treated as random, applied to the public statement and transcript prefix, and from the message when I want a signature. The proof or signature is accepted only when the recomputed challenge makes the original verifier predicate accept. What matters is not the word "hash" but the role assignment: the hash replaces the verifier's unpredictable public coin at the precise point where that coin protects soundness.
