I start by separating the cost of attention from the role it plays. In a Transformer encoder, the feed-forward sublayer acts independently at each token and mixes only hidden features. The self-attention sublayer is the part that moves information across token positions. Its dot products, softmax, heads, and value projections make a powerful content-dependent mixer, but the structural job I need from the sublayer is narrower: after it runs, each output position should be some function of the whole sequence, so the next per-token transformation can use information that has already crossed positions.

If the job is only "let every position see every other position," then approximating the softmax all-pairs computation is not the only route. I could replace the attention sublayer with a different operation that mixes positions and let the same residual, layer norm, and feed-forward machinery do the nonlinear modeling. The most direct thing to try is a learned dense matrix along the sequence axis, possibly paired with a learned dense matrix along the hidden axis. This is worth trying first not because I expect it to be the answer but because it isolates one variable: it removes query-key attention entirely while still giving every output position a learned linear combination of every input position. If an encoder built this way trains at all, then content-dependent dot products are not load-bearing for the cross-position mixing, and I can keep looking for something cheaper to put in the slot.

But the learned sequence matrix runs into a wall I can see without training anything. It is `N x N`, so multiplying by it is still quadratic in sequence length, exactly the cost I was trying to escape. Worse, its entries are tied to one chosen maximum length: a model trained at length 512 has a 512x512 mixing matrix and cannot be applied to a longer input without re-learning. So a learned dense mixer answers the first question — attention's specific form is not required — but it does not give me a length-friendly replacement. What I actually want is a *fixed* linear map that (i) sums over all input positions, so it is globally mixing, and (ii) has a fast multiplication algorithm so I am not paying the dense `N x N` cost. Those two requirements together point away from arbitrary learned matrices and toward structured transforms.

The discrete Fourier transform fits both requirements. With the forward convention used by NumPy and JAX, a length-`N` vector is mapped by

`X_k = sum_{n=0}^{N-1} x_n exp(-2*pi*i*n*k/N)`.

Each output coefficient is a sum over every input position, so the operation is dense and global, and it carries no learned parameters at all. The sign here is the negative-exponent forward transform, the one `jnp.fft.fftn` and `np.fft.fftn` compute. Direct multiplication by the DFT matrix is quadratic like any dense matrix, but the FFT computes the same transform in `O(N log N)` for the usual composite lengths — so requirement (ii) is met by the algorithm, not by sparsifying the matrix.

Before going further I want to check the "global" claim concretely rather than trust the formula, and I want to know how the transform should be applied to a two-axis array. The encoder input is not a single vector; it has a sequence axis and a hidden axis. The sequence axis is the one that must replace attention's position mixing. I could also transform the hidden axis — the feed-forward sublayer already mixes hidden features, so a hidden transform is not the essential move, but applying both one-dimensional transforms gives a separable two-dimensional transform. I want to confirm three things on a small array: that a 2D DFT really equals doing the two 1D axis transforms in succession, that the order does not matter, and that the resulting real operator is dense. Taking a 6x4 array `x`:

```
Xhidden = fft(x, axis=1);  Xboth = fft(Xhidden, axis=0)
Xfftn   = fftn(x)
allclose(Xboth, Xfftn)            -> True,  max|.| = 0.0
allclose(Xboth, fft(fft(x,0),1))  -> True,  max|.| ~ 1.8e-15
```

So `fftn` is exactly the composition of the per-axis transforms, and swapping the axis order changes nothing beyond floating-point noise. That is what I would expect from separability — the two 1D transforms act on different axes, so their matrices act on opposite sides of `x` and commute — but it is reassuring to see it land at machine precision rather than to assert it. The candidate sublayer is therefore a two-dimensional forward DFT over the hidden and sequence axes.

That transform is complex-valued, which is the next practical wall. I do not want to carry complex numbers into the feed-forward layers, the output heads, and the loss; that would touch the whole rest of the encoder. The cheap fix is to take the real part. The question is *where*. If I take the real part right after the first (hidden) transform and then run the second (sequence) transform, do I lose anything compared to taking the real part once at the very end? On a 5x3 array:

```
y_end = fft(fft(x, axis=1),        axis=0).real
y_mid = fft(fft(x, axis=1).real,   axis=0).real
allclose(y_end, y_mid) -> False,   max|y_end - y_mid| = 1.37
```

So the two are genuinely different — discarding the imaginary part after the first transform throws away information that the second transform would otherwise have used. Taking an absolute value would deform the operation even more. The clean choice is to keep the complex intermediate through both transforms and take the real part exactly once at the end:

`y = Re(F_seq(F_hidden(x)))`.

I also wanted to confirm that this real-at-the-end operator is still a dense global mixer and not accidentally something sparse. I build its implied real linear operator by pushing each standard basis array through `Re(fftn(.))` and reading off the columns: every one of the resulting entries is nonzero (fraction of nonzeros = 1.0). So after the real part is taken, each output coordinate still depends on every input coordinate. Requirement (i) survives the projection to the reals.

Next I have to be precise about normalization, because the model is supposed to admit two interchangeable implementations: an FFT path for speed and a cached DFT-matrix path for short sequences or for hardware without a good FFT kernel. A Vandermonde DFT matrix is sometimes written with a `1/sqrt(N)` factor for a unitary convention, so I should check which convention makes the matrix path agree with `fftn`. Building `Fseq = scipy.linalg.dft(N)` and `Fhid = scipy.linalg.dft(d)` and forming `Fseq @ x @ Fhid.T`:

```
allclose(Fseq @ x @ Fhid.T, fftn(x)) -> True,  max|.| ~ 1.9e-14
```

`scipy.linalg.dft` defaults to the *unnormalized* forward DFT, and with that convention the matrix path reproduces `fftn` to floating-point precision. To see the size of the trap, I insert the unitary factors and compare magnitudes:

```
ratio |fftn(x)| / |(Fseq/sqrt(N)) @ x @ (Fhid/sqrt(d)).T|  ->  4.899...
sqrt(N*d) with N=6, d=4                                    ->  4.899...
```

That ratio is exactly `sqrt(N*d)`, not merely close to it, so this isn't rounding noise — it's a real fork between two legitimate conventions, and the two conventions disagree by that whole factor. If I build the cached-matrix path with the textbook unitary normalization while leaving the FFT path as `jnp.fft.fftn`, then "the same" mixer would silently rescale every activation depending on which code path a given sequence length happened to route through, which defeats the entire point of offering two interchangeable implementations. So the matrix path has to use the raw, unnormalized Vandermonde matrix — `scipy.linalg.dft(N)` with no `1/sqrt(N)` — so that switching between FFT and cached-matrix execution for the same sequence length is invisible to the rest of the network. Which implementation to route to isn't fixed either: I want the FFT for long sequences where `O(N log N)` matters, and the direct matrix product when the hardware's matrix units are simply better exercised than its FFT kernels — a question I have to come back to once I can measure real hardware, not just complexity classes.

Before spending a training run on this, I want a reason to expect the mixer will do more than shuffle activations around uselessly. The property I actually need from the sublayer is that composing it with the pointwise feed-forward layer should be able to express something like a convolution — a token's post-mixing representation depending on a weighted neighborhood of the whole sequence, not just a fixed reshuffling of it. The Fourier transform gives me that through its own duality: multiplying by a set of coefficients after transforming into frequency space is, back in the original domain, the same operation as convolving with the inverse transform of those coefficients. So an encoder block that goes real space to frequency space (my mixer) to a per-token nonlinearity is not merely permuting information across positions — the shape of a global convolution is already folded into the block, with the feed-forward layer able to act as the (implicit) kernel. That's a genuinely different mechanism from attention's per-example, content-dependent weights, but it isn't a mechanism-free one, which is the reassurance I want before trusting a parameter-free sublayer with real modeling work.

The cheapest way to stress-test whether it's specifically the Fourier structure doing that work, rather than just "any fixed dense global map," is to swap in a matrix with no structure at all — freeze two random matrices, one across positions and one across features, in exactly the slot the DFT occupies, and change nothing else about the encoder. If that trains about as well as the Fourier version, I haven't found anything specific to the Fourier transform; I would have only rediscovered that almost any dense projection beats no mixing at all. It doesn't come close. The random-matrix encoder lands at 56.6 average on GLUE — barely ahead of the encoder with the mixing sublayer removed entirely (49.3, which mostly fails outright on the binary-classification tasks) and far behind the Fourier version's 76.7. So the earlier lesson generalizes past the first learned-matrix attempt: it is not enough for the operator to be fixed, dense, and global. It has to be structured in a way a random matrix isn't. That rules out the cheap escape hatch of "any frozen matrix will do," and points at the specific algebraic content of the DFT — the roots-of-unity coefficients, not just their shape — as where the real work is happening.

Given that the structure matters, I want to know how much of it is load-bearing, so I try other classical structured transforms in the same slot. The Hadamard transform is the cheapest alternative available — its matrix has only two distinct entries, `+-1`, instead of `N` roots of unity — and it is slightly faster to compute, but it costs about two points of GLUE accuracy. The discrete cosine transform is a closer relative of the DFT, and it produces real output natively, so it looks like it should recover the accuracy that discarding the imaginary part costs — but it actually costs about four points, the largest gap of any alternative I try. The Hartley transform is the one that matches: `H = Re(F) - Im(F)` is also real-valued by construction, and it lands at the identical 76.7 GLUE average as the Fourier mixer. So being real-valued from the start is not itself the source of any advantage — the Hartley transform reaches the same place by a different route than "take the real part of the complex DFT once, at the end," which is a second, independent confirmation that it is specifically the roots-of-unity kernel, present in both the Fourier and Hartley cases and absent from Hadamard, that the mixing needs. I keep the Fourier form rather than switching to Hartley because the DFT is the one with a mature `O(N log N)` FFT and decades of hardware-tuned kernels behind it; Hartley buys nothing extra for that cost.

Having settled on the transform, I still want to check I'm not leaving easy accuracy on the table by refusing to learn anything in this sublayer. I try learned weights in several places — an elementwise gain on the (sequence, hidden) array before or after the transform, learned square matrices multiplying the sequence and hidden axes the way the very first dense-mixing attempt did, and, most directly, complex learned weights added straight into the DFT matrix itself so the operator can drift away from the exact roots-of-unity kernel during training. None of it helps. Most configurations degrade accuracy or destabilize training, and the ones that don't hurt accuracy don't improve it either — they only slow the model down by adding parameters to update every step. The version that perturbs the DFT matrix directly is the most telling: it stays stable, but the learned matrix never moves anywhere better than the fixed DFT it started from. That reads like the DFT sitting at a local optimum for this role, not like a starting point I happened to undershoot from.

One more piece of the design falls out almost as a side effect of the transform's own algebra. Since the DFT coefficient for the `n`-th input and `k`-th output position already carries an explicit `n` in its phase, the transform is sensitive to absolute position before any position embedding is ever added — the same reason sinusoidal position embeddings work in the first place. So this mixer should tolerate dropping the learned position embeddings without losing much, and it does. I keep them anyway, not because the mixer needs them, but because removing them would confound any accuracy comparison against BERT, which relies on position embeddings entirely to know where a token sits.

The two-implementation design is not a hedge against a hypothetical missing FFT kernel; it is a response to something that only shows up once I look at real GPU and TPU numbers rather than trust asymptotic complexity alone. On GPUs, the FFT is the faster path at every sequence length I measure, from 512 up to 8192, matching the `O(N log N)` expectation. On TPUs the ranking flips at moderate lengths: the `O(N^2)` cached-matrix path is faster than the FFT for sequences up to roughly 4096 tokens, and only the FFT overtakes it beyond that. That's a hardware-maturity fact, not a mathematical one — TPU matrix units are so heavily optimized, and TPU FFT kernels comparatively so much less so, that a worse asymptotic algorithm wins in exactly the regime a 512-token BERT-style workload lives in. This is why the normalization check earlier wasn't a pedantic detour: a model whose matrix path and FFT path don't compute the identical function would give different answers depending on which hardware happened to run it.

Once trained with the ordinary BERT recipe — masked-language modeling and next-sentence prediction, matched model sizes, matched optimizer — the parameter-free mixer reaches 76.7 average on GLUE against BERT-Base's 83.3, about 92% of it, while training 80% faster on GPUs and 70% faster on TPUs. Measured in isolation, the mixing sublayer itself is roughly 10 to 22 times faster than self-attention depending on model size and hardware; the reason the whole-model speedup is "only" 70-80% rather than that much larger factor is that the feed-forward sublayers, unchanged and shared with the attention baseline, are what the total step time is bottlenecked on once the mixing cost stops dominating. What I did not expect going in is that the accuracy gap shrinks rather than grows at the Large size — FNet-Large reaches about 97% of BERT-Large, not 92% — and that lines up with the training-stability pattern running through every check above: every model with a fixed or absent mixing sublayer (Fourier, random, no-mixing at all) trains more stably than the ones with learned mixing weights (the dense linear attempt, and attention itself), and that stability advantage compounds as the model gets larger and harder to train. And if I want the remaining fraction of accuracy back, I don't have to abandon the idea: swapping self-attention back in for just the last two layers, leaving every lower layer as a Fourier mixer, recovers 97-99% of BERT's accuracy for far less speed cost than putting attention back everywhere. Content-dependent mixing still buys something — the discovery is that it only needs to be purchased in a couple of layers near the output, not paid for at every layer in the stack.
