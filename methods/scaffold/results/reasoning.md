Let me start from what actually goes wrong when I train a model across many clients without moving their data. The thing I'm minimizing is an average of per-client losses, f(x) = (1/N) ОЈ_i f_i(x), and each f_i is itself a stochastic objective вҖ” I only ever see g_i(x), a noisy unbiased gradient of f_i, computed on a minibatch sitting on client i. The brutal constraint is the network: talking to clients is slow and flaky, so the only resource I really care about is the number of *communication rounds*. That forces a pattern where each round a client copies the current server model, runs many local steps on its own data, and ships back only the change. And on top of that, in any round I can only reach a small subset S of the N clients, and вҖ” the part that bites вҖ” the clients' data distributions are arbitrarily different, so the f_i are arbitrarily different functions, with their individual minimizers x_i* scattered far apart and far from the true x*.

So what do I have today? The obvious recipe: each sampled client sets y_i вҶҗ x, takes K SGD steps y_i вҶҗ y_i вҲ’ О·_l g_i(y_i), and the server averages the results, x вҶҗ x + (О·_g/|S|) ОЈ_{iвҲҲS}(y_i вҲ’ x). For identical clients this is just parallel SGD and it's lovely. But picture two clients with very different data. Client 1's local steps walk y_1 toward x_1*, client 2's walk y_2 toward x_2*. Average the two endpoints and the server lands near (x_1* + x_2*)/2 = (1/N)ОЈ_i x_i*. And that point is *not* x*. There's no reason the average of the per-client optima should be the optimum of the average loss вҖ” unless the clients are identical, in which case all the x_i* coincide. So the aggregated step is systematically biased toward (1/N)ОЈ_i x_i*, and the size of that bias is exactly the dissimilarity of the clients. Let me name it: client-drift. And the nasty thing is it doesn't come from stochastic noise вҖ” set Пғ = 0, give everyone full-batch gradients, let all clients participate, and the drift is still there, because it's a property of where the *deterministic* local trajectories go. The more local steps K I take to save communication, the further each y_i drifts toward its own x_i*, so the very lever I pull to be communication-efficient makes the bias worse. To stay stable I'm forced to shrink О·_l, which directly slows me down. That's the wall: heterogeneity turns my best efficiency lever into a liability.

Let me quantify the heterogeneity so I can reason about it precisely. The clean way is gradient dissimilarity: there exist G, B with (1/N)ОЈ_i ||вҲҮf_i(x)||ВІ вүӨ GВІ + BВІ||вҲҮf(x)||ВІ for all x. G measures how much the per-client gradients disagree even where the global gradient is small вҖ” at x*, вҲҮf(x*) = 0 but the individual вҲҮf_i(x*) need not vanish, and their spread is G. I'd like to know whether that G is something I can tune away or whether it's structural, so let me build the smallest instance that isolates it and actually run the plain method on it. Take two one-dimensional clients, f_1(x) = (Ој/2)xВІ + Gx with вҲҮf_1 = Ојx + G, and f_2(x) = вҲ’Gx with вҲҮf_2 = вҲ’G. The average is f = (Ој/4)xВІ, so вҲҮf = (Ојx)/2, and the global optimum is x* = 0. The constant G has cancelled in f вҖ” it's pure heterogeneity, invisible to the global objective вҖ” yet each client's own gradient carries it. With full batch (Пғ = 0) and both clients participating every round, the round map is exactly affine in x, so I can solve for its fixed point in closed form. Client 1 runs y вҶҗ (1 вҲ’ О·_l Ој)y вҲ’ О·_l G for K steps; client 2 runs y вҶҗ y + О·_l G; the server averages the two displacements. Solving the resulting affine map for its fixed point and sweeping О·_l (Ој = 1, G = 10, K = 10, О·_g = 1) I get x_вҲһ = 0.933 at О·_l = 0.02, 0.458 at 0.01, 0.227 at 0.005, 0.0903 at 0.002, 0.0451 at 0.001 вҖ” a clean factor-of-2 each time I halve О·_l. So the fixed point is nonzero for *every* positive О·_l, with the bias scaling like О·_lВ·GВ·(something); it only reaches x* = 0 in the limit О·_l вҶ’ 0, where progress also stops. That settles it numerically: under heterogeneity the plain method has a residual bias I cannot tune to zero at any usable step size, and its size is governed by G. Anything I build has to make the dependence on G *vanish*, not merely shrink with the step вҖ” otherwise I haven't solved heterogeneity, I've only rescaled it.

What's been tried? The first instinct is to stop the clients from wandering: add a penalty (Ој/2)||w вҲ’ x||ВІ to each client's local objective, so the local gradient becomes g_i(w) + Ој(w вҲ’ x), anchoring w to the shared model. Let me think about whether that fixes the drift. It shrinks how *far* w can move in K steps, yes. But stare at the corrected direction: g_i(w) + Ој(w вҲ’ x) still has g_i(w) in it, and g_i(w) points toward x_i*. The penalty pulls back toward x, it doesn't rotate the direction toward x*. So at any fixed distance from x the client is still heading toward its own optimum, just on a shorter leash. The aggregate is still biased toward (1/N)ОЈ x_i* вҖ” the bias is throttled, not removed вҖ” and I pay for the throttling with slower progress. If I work out its rate it comes out the same as plain averaging up to the heterogeneity term; there's no actual escape from G this way. Damping the *magnitude* of the local move is the wrong knob. I need to fix the *direction*.

So let me ask the right question: what direction *should* the client take? If communication were free, the ideal local update on client i would use the global gradient, y_i вҶҗ y_i вҲ’ О·_l вҲҮf(y_i) = y_i вҲ’ О·_l (1/N) ОЈ_j вҲҮf_j(y_i). That's unbiased toward x* by construction вҖ” it's gradient descent on f itself вҖ” and it would converge beautifully even for one client at a time, with zero client-drift, because every client would be descending the *same* function f. The whole problem is that computing (1/N)ОЈ_j вҲҮf_j(y_i) needs every other client's gradient at y_i, which is exactly the communication I can't afford every step. So the real task is sharp: approximate the global direction (1/N)ОЈ_j вҲҮf_j(y_i) using only client i's own gradient g_i(y_i) plus some cheap state, and do it well enough that the approximation error doesn't depend on how heterogeneous the clients are.

Now I recognize this shape. This is the variance-reduction / control-variate situation. The classical move: to estimate E[X] when I have a correlated Y whose mean E[Y] I know, use X вҲ’ Y + E[Y]; it's unbiased, E[X вҲ’ Y + E[Y]] = E[X], and its variance is Var(X) + Var(Y) вҲ’ 2Cov(X,Y), which is small when X and Y are tightly correlated. In finite-sum optimization that's exactly how SAGA and SVRG kill the variance of the sampled gradient: the SGD direction X = g_j(x) gets corrected to g_j(x) вҲ’ g_j(ПҶ_j) + (1/n)ОЈ_i g_i(ПҶ_i), where g_j(ПҶ_j) is a stale stored gradient for the sampled component j and (1/n)ОЈ_i g_i(ПҶ_i) is the table average. The correction subtracts the part of g_j(x) that is *specific to component j* and adds back the average вҖ” it turns a one-component sample into something whose mean is the full-sum gradient. The analogy to my setting writes itself: my "components" are the clients, the "sampled component" is client i, and the thing I want is for the corrected local gradient on client i to have mean (1/N)ОЈ_j вҲҮf_j вҖ” the full-average gradient вҖ” instead of вҲҮf_i.

So let me posit two pieces of state. Each client keeps a control variate c_i that is meant to track its own gradient, c_i вүҲ g_i(y_i); the server keeps a control variate c that is meant to track the average gradient, c вүҲ (1/N)ОЈ_j g_j(y_i). Then the corrected local direction is g_i(y_i) вҲ’ c_i + c. Check what it estimates: if c_i вүҲ g_i(y_i) and c вүҲ (1/N)ОЈ_j g_j(y_i), then g_i(y_i) вҲ’ c_i + c вүҲ g_i(y_i) вҲ’ g_i(y_i) + (1/N)ОЈ_j g_j(y_i) = (1/N)ОЈ_j g_j(y_i), which is exactly the ideal global direction. The local update becomes

  y_i вҶҗ y_i вҲ’ О·_l (g_i(y_i) вҲ’ c_i + c).

The correction term (c вҲ’ c_i) is, intuitively, an estimate of the drift: c_i is where *my* gradient points, c is where the *average* gradient points, and (c вҲ’ c_i) is the rotation that bends my local step away from x_i* and back toward x*. That's the structural fix the penalty couldn't do вҖ” it operates on the direction, not the leash length. And there's a consistency condition I should impose for c to mean "the average": keep the invariant c = (1/N) ОЈ_i c_i. If c is literally the average of the client control variates, then c being a good estimate of the average gradient follows from each c_i being a good estimate of its own gradient. Initialize everything to zero, c = c_i = 0, which trivially satisfies the invariant; and notice that if I freeze every c_i вүЎ 0 forever, the correction (c вҲ’ c_i) vanishes and I recover plain averaging exactly. So this is a strict generalization of local averaging, with the extra state used only to change the local direction.

I want to know whether this actually removes the G dependence or just relocates it вҖ” that's the whole point. Let me compare the corrected update against the ideal one and bound the residual

  ОЈ_i ||(вҲҮf_i(y) вҲ’ c_i + c) вҲ’ вҲҮf(y)||ВІ

by hand. Write e_i := c_i вҲ’ вҲҮf_i(y), the per-client tracking error, and use the invariant c = (1/N)ОЈ_j c_j together with вҲҮf(y) = (1/N)ОЈ_j вҲҮf_j(y), so that c вҲ’ вҲҮf(y) = (1/N)ОЈ_j (c_j вҲ’ вҲҮf_j(y)) = Д“, the *mean* tracking error. Then a single client's residual is

  (вҲҮf_i(y) вҲ’ c_i + c) вҲ’ вҲҮf(y) = вҲ’e_i + Д“.

That's a clean cancellation: the corrected direction lands wrong by exactly (mean tracking error вҲ’ this client's tracking error). Summing the squares, ОЈ_i ||вҲ’e_i + Д“||ВІ = ОЈ_i ||e_i||ВІ вҲ’ N||Д“||ВІ вҖ” the standard variance decomposition (the cross term вҲ’2вҹЁe_i, Д“вҹ© sums to вҲ’2N||Д“||ВІ, leaving ОЈ||e_i||ВІ вҲ’ N||Д“||ВІ). Since N||Д“||ВІ вүҘ 0,

  ОЈ_i ||(вҲҮf_i(y) вҲ’ c_i + c) вҲ’ вҲҮf(y)||ВІ = ОЈ_i ||c_i вҲ’ вҲҮf_i(y)||ВІ вҲ’ N||Д“||ВІ вүӨ ОЈ_i ||c_i вҲ’ вҲҮf_i(y)||ВІ.

So it's an inequality, not an identity, and the slack is exactly N times the squared mean tracking error вҖ” which only helps, since dropping a non-negative term gives a valid upper bound. What matters is the upshot: the residual depends *only* on how well each c_i tracks вҲҮf_i(y) вҖ” and *not at all* on G, not on how dissimilar the clients are. That's the escape. Plain averaging's residual is ||вҲҮf_i(y) вҲ’ вҲҮf(y)||, the raw gradient dissimilarity bounded by G; mine is the tracking error ||c_i вҲ’ вҲҮf_i(y)||, which I get to control. And here's why I can control it: f_i is ОІ-smooth, so вҲҮf_i(y) doesn't move fast as y moves; if c_i is a recently-computed gradient of f_i it stays close to вҲҮf_i(y). So I should keep the c_i *stateful across rounds* вҖ” a client holds onto its c_i and refreshes it when it participates вҖ” because smoothness makes a slightly stale gradient still a good approximation. Statefulness isn't an accident; it's what makes the cheap approximation valid.

Now, how exactly do I refresh c_i? The honest version: after the local work, recompute the client's gradient and set c_iвҒә = g_i(x), the gradient at the server model x. That needs one extra pass over the local data beyond the K training steps вҖ” call it Option I, clean and stable but costing an extra gradient computation. Can I avoid the extra pass and reuse the gradients I already computed during the K local steps? Let me telescope the local trajectory. Over the round, y_{i,0} = x and each step is y_{i,k} = y_{i,kвҲ’1} вҲ’ О·_l (g_i(y_{i,kвҲ’1}) + c вҲ’ c_i). Sum the increments from k = 1 to K:

  y_{i,K} вҲ’ x = вҲ’О·_l ОЈ_{k=1}^K (g_i(y_{i,kвҲ’1}) + c вҲ’ c_i) = вҲ’О·_l [ ОЈ_{k=1}^K g_i(y_{i,kвҲ’1}) + K(c вҲ’ c_i) ].

Rearrange to isolate the average local gradient:

  (1/(K О·_l)) (x вҲ’ y_{i,K}) = (1/K) ОЈ_{k=1}^K g_i(y_{i,kвҲ’1}) + (c вҲ’ c_i).

The left side is just the total displacement divided by KО·_l вҖ” quantities the server already has, since it sees x and y_{i,K}. So if I define the refresh as

  c_iвҒә = c_i вҲ’ c + (1/(K О·_l)) (x вҲ’ y_{i,K}),

then substituting the identity gives c_iвҒә = c_i вҲ’ c + (1/K)ОЈ_k g_i(y_{i,kвҲ’1}) + (c вҲ’ c_i) = (1/K) ОЈ_{k=1}^K g_i(y_{i,kвҲ’1}) вҖ” the average of the K minibatch gradients I *already computed* during local training, for *free*, no extra pass. That's Option II.

Option II tracks the gradient at the *visited* iterates rather than at x, so the two refreshes are not interchangeable, and I want to know how much that matters before defaulting to it. Let me run both to the fixed point on the heterogeneous two-client instance (Ој = 1, G = 10, K = 10, О·_g = 1), where I know x* = 0. With Option I (c_iвҒә = g_i(x)) the model goes to |x вҲ’ x*| = 4.5eвҲ’5 by R = 200 and 8.9eвҲ’16 by R = 1000 вҖ” exact convergence вҖ” and the controls settle at c_1 вҶ’ 10 = вҲҮf_1(0), c_2 вҶ’ вҲ’10 = вҲҮf_2(0), the true gradients at the optimum. So Option I genuinely kills the G-floor that defeated plain averaging: same instance, same G, bias gone. I expected Option II to do the same, but running it to R = 50000 it parks at |x вҲ’ x*| = 0.229 and *stays* there, with c_1 вҶ’ 5, c_2 вҶ’ вҲ’5 вҖ” not the true gradients. That stopped me. It's not a transient; it's a biased fixed point. Hunting for the cause: at K = 1 Option II converges to 0 exactly (there's no within-round drift to corrupt the gradient average), but the residual grows with K (0.025 at K = 2, 0.101 at K = 5, 0.229 at K = 10, 0.492 at K = 20), and вҖ” the tell вҖ” it scales linearly with О·_l: halving О·_l from 0.01 to 0.005 to 0.0025 to 0.00125 takes |x вҲ’ x*| from 0.2291 to 0.1135 to 0.0565 to 0.0282, a clean factor of вүҲ0.5 each time. So Option II carries an O(О·_l) bias for K > 1 because it averages g_i over the *drifted* iterates y_{i,k} instead of at x; that error vanishes as О·_l вҶ’ 0, exactly the small-step regime the convergence analysis will need anyway, whereas plain averaging's bias also scaled like О·_l but never vanished relative to the target because it was an order-G term, not a tracking error. That reconciles it: Option II is sound but only in the small-О·_l regime; Option I is fixed-point-exact for any О·_l. Option II costs nothing and is what the telescoping identity hands me for free, so I'll keep it as the default while remembering this O(О·_l) caveat, and keep Option I as the more stable fallback.

Crucially, the local update step uses plain SGD вҖ” just g_i plus the fixed correction (c вҲ’ c_i) added to the gradient, then a vanilla step. I should resist the urge to bolt on momentum here, because the whole Option-II refresh rests on the exact telescoping identity ОЈ_k (y_{i,kвҲ’1} вҲ’ y_{i,k}) = О·_l ОЈ_k v_{i,k} with v_{i,k} = g_i + c вҲ’ c_i; a momentum buffer would break that clean relationship between the net displacement and the sum of corrected gradients, and I'd lose the free control-variate update. Plain SGD it is.

Now the server side вҖ” aggregation. The model aggregates the usual way: x вҶҗ x + (О·_g/|S|) ОЈ_{iвҲҲS}(y_i вҲ’ x), the average client displacement scaled by a global step О·_g. But the control variate c needs care because of sampling. Only the S participating clients refreshed their c_i this round; the other N вҲ’ S kept their old c_i. I want to maintain c = (1/N)ОЈ_i c_i, the average over *all* clients. The change in the global average is

  О”c = (1/N) ОЈ_i (c_i^{new} вҲ’ c_i^{old}) = (1/N) ОЈ_{iвҲҲS} (c_iвҒә вҲ’ c_i),

since only the sampled clients changed. So the server update is

  c вҶҗ c + (1/N) ОЈ_{iвҲҲS} (c_iвҒә вҲ’ c_i).

Note the divisor is N, not |S| вҖ” that's the subtle but essential point. Each participating client sends back its delta О”c_i = c_iвҒә вҲ’ c_i, the server sums those S deltas and divides by N. Equivalently, c вҶҗ c + (|S|/N) В· (1/|S|) ОЈ_{iвҲҲS} О”c_i: take the *mean* of the sampled clients' control deltas and scale it down by the participation fraction |S|/N. If I'd divided by |S| instead, c would no longer equal the true average (1/N)ОЈ_i c_i вҖ” it would over-weight the few clients I happened to sample, the invariant the whole correction relies on would break, and the drift estimate (c вҲ’ c_i) would be wrong. The |S|/N factor is exactly what keeps c honest as the all-client average despite seeing only S of them.

Why two separate step sizes, О·_l local and О·_g global? Because they trade off against the drift differently. The effective per-round step is О·Мғ = K О·_l О·_g, and that's what sets the descent speed. But the drift accumulated within a round scales with О·_l (how big each *local* step is), while the global progress scales with О·Мғ. So I can make О·_g large and О·_l correspondingly small at fixed О·Мғ: same net progress per round, but smaller local steps mean less within-round drift. Decoupling the two gives me a knob to suppress drift without sacrificing speed вҖ” which is also why a careful analysis of even the plain method improves once you allow О·_g вү  1. In practice I'll often just set О·_g = 1, but the freedom matters for the theory.

Removing the heterogeneity dependence is a strong claim, so I need the rate, not just the mechanism. I'll track three error quantities round to round. First the client-drift E_r = (1/(KN)) ОЈ_{k,i} E||y_{i,k}^r вҲ’ x^{rвҲ’1}||ВІ, how far clients wander. Second the control-lag C_r = (1/N) ОЈ_i ||E[c_i^r] вҲ’ вҲҮf_i(x*)||ВІ, how stale the control variates are relative to the gradients at the optimum. Third the distance to optimum ||x вҲ’ x*||ВІ. The plan is the standard one: bound the second moment of the server update, bound how the drift and the lag evolve, and assemble a one-round contraction.

Start with the variance of the server update. О”x = вҲ’(О·Мғ/(KS)) ОЈ_{k,iвҲҲS} (g_i(y_{i,kвҲ’1}) + c вҲ’ c_i). I want E||О”x||ВІ. Split it with the relaxed triangle inequality ||ОЈ v_j||ВІ вүӨ П„ ОЈ||v_j||ВІ into four groups: the gradient-vs-its-mean part (which contributes the within-client variance ПғВІ/(KS) after separating mean from noise), the c term, the (вҲҮf_i(x*) вҲ’ c_i) term, and the (вҲҮf_i(x) вҲ’ вҲҮf_i(x*)) term. The last one is вүӨ 2ОІ(f(x) вҲ’ f*) by smoothness-convexity. Using c = (1/N)ОЈ c_i to fold the c and (вҲҮf_i(x*) вҲ’ c_i) pieces together into the control-lag, and Lipschitzness to turn ОЈ||вҲҮf_i(y_{i,kвҲ’1}) вҲ’ вҲҮf_i(x)||ВІ into ОІВІВ·(drift), I get

  E||x^r вҲ’ x^{rвҲ’1}||ВІ вүӨ 8ОІ О·МғВІ (f(x) вҲ’ f*) + 8 О·МғВІ C_{rвҲ’1} + 4 О·МғВІ ОІВІ E_r + 12 О·МғВІ ПғВІ/(KS).

Good вҖ” the server-update size is controlled by suboptimality, lag, drift, and noise, with no bare G anywhere.

Next, how does the control-lag evolve? After round r, c_iвҒә = (1/K)ОЈ_k g_i(y_{i,kвҲ’1}) for the sampled clients (probability S/N each) and c_iвҒә = c_i otherwise. Taking expectations, E[c_i^r] = (1 вҲ’ S/N) E[c_i^{rвҲ’1}] + (S/N)В·(1/K)ОЈ_k E[вҲҮf_i(y_{i,kвҲ’1})]. Plug into the definition of C_r, use Jensen to pull the average inside the norm, then the relaxed triangle inequality plus Lipschitzness (вҲҮf_i(y_{i,kвҲ’1}) вҲ’ вҲҮf_i(x*) split through вҲҮf_i(x^{rвҲ’1})) to land

  C_r вүӨ (1 вҲ’ S/N) C_{rвҲ’1} + (S/N)( 4ОІ(f(x) вҲ’ f*) + 2ОІВІ E_r ).

So the lag *contracts* by (1 вҲ’ S/N) each round вҖ” it forgets stale information at the participation rate вҖ” and is refreshed by terms proportional to current suboptimality and drift. The more clients I sample, the faster the lag heals.

Now the drift itself. Expand E||y_{i,k} вҲ’ x||ВІ one local step at a time. The update inside is y_i вҲ’ О·_l(g_i(y_i) + c вҲ’ c_i); separate the noise (О·_lВІПғВІ), then use the relaxed triangle inequality with parameter a = 1/(KвҲ’1) to split into a contractive piece and the correction piece. The contractive piece is ||y_i вҲ’ О·_lвҲҮf_i(y_i) + О·_lвҲҮf_i(x) вҲ’ x||ВІ: this is exactly the contractive-mapping lemma вҖ” a gradient step on a ОІ-smooth Ој-convex function is a contraction for О·_l вүӨ 1/ОІ вҖ” so it's вүӨ ||y_i вҲ’ x||ВІ. The correction piece T_3 = (1/N)ОЈ_j||c вҲ’ c_j + вҲҮf_j(x)||ВІ expands, using c = (1/N)ОЈc_j and smoothness, into вүӨ (6/N)ОЈ_j||c_j вҲ’ вҲҮf_j(x*)||ВІ + 6ОІ(f(x) вҲ’ f*). Assemble into a within-round recursion

  (1/N)ОЈ_i E||y_{i,k} вҲ’ x||ВІ вүӨ (1 + 1/(KвҲ’1)) (1/N)ОЈ_i E||y_{i,kвҲ’1} вҲ’ x||ВІ + 7О·_lВІПғВІ + 6О·_lВІKОІ(f(x)вҲ’f*) + 6KО·_lВІ C_{rвҲ’1},

K = 3: (2)((1.5)Ві вҲ’ 1) = 2(3.375 вҲ’ 1) = 4.75 вүӨ 9. For K вүҘ 4 the slick bound is (1+1/(KвҲ’1))^K < exp(K/(KвҲ’1)) вүӨ exp(4/3) вүҲ 3.79, so the whole expression is < (KвҲ’1)(exp(4/3) вҲ’ 1) < (KвҲ’1)(2.79) < 3K. To be sure the constant 3 isn't optimistic I tabulated the ratio (KвҲ’1)((1+1/(KвҲ’1))^K вҲ’ 1)/(3K) up to K = 10вҒө: it rises monotonically from 0.50 (K=2) toward a limit of (eвҲ’1)/3 вүҲ 0.573, never exceeding 0.573, so 3K is a valid bound with room to spare. So the accumulated drift is bounded by 3K times the per-step terms, and after averaging and multiplying by 3ОІ,

  3ОІ О·Мғ E_r вүӨ (2О·МғВІ/3) C_{rвҲ’1} + (О·Мғ/(25О·_gВІ))(f(x) вҲ’ f*) + (О·МғВІ/(KО·_gВІ)) ПғВІ,

Assemble. I want a Lyapunov function that contracts. The model distance alone won't close because the server-variance bound feeds on C_{rвҲ’1}, and C evolves on its own. Unroll it. With weights w_r = (1 вҲ’ ОјО·Мғ/2)^{1вҲ’r} the suboptimality terms telescope, and tuning О·Мғ (small enough for the contraction, large enough to make progress) gives, in the strongly convex case, a number of rounds

Read the terms. Take Пғ = 0, K = 1, S = 1, О·_g = 1, and use Option I (c_iвҒә = g_i(x)). Let me push on one more question, because it's been nagging: when are local steps actually *worth it*? Plain averaging gets *worse* with more local steps under heterogeneity (more drift), but intuitively, if the clients are similar, taking many local steps should let each client make real progress on a shared landscape and save communication. Which notion of similarity governs this? Not gradient dissimilarity G вҖ” that can be huge even when local steps help. Let me look at the corrected update with a first-order expansion around x, taking quadratics for clarity (so the Hessians A_i are constant). The corrected direction is вҲҮf_i(y) вҲ’ вҲҮf_i(x) + вҲҮf(x). Taylor-expand each piece about x:

  вҲҮf_i(y) вҲ’ вҲҮf_i(x) вүҲ вҲҮВІf_i(x)(y вҲ’ x),   and   вҲҮf(x) вүҲ вҲҮf(y) + вҲҮВІf(x)(x вҲ’ y).

Add them:

  вҲҮf_i(y) вҲ’ вҲҮf_i(x) + вҲҮf(x) вүҲ вҲҮf(y) + (вҲҮВІf_i(x) вҲ’ вҲҮВІf(x))(y вҲ’ x).

So the corrected local direction equals the *ideal* global direction вҲҮf(y) plus an error (вҲҮВІf_i(x) вҲ’ вҲҮВІf(x))(y вҲ’ x). For quadratics the Hessians are constant, so this expansion isn't an approximation вҖ” the first-order Taylor remainder is zero identically, and the corrected direction is exactly the ideal one plus (вҲҮВІf_i(x) вҲ’ вҲҮВІf(x))(y вҲ’ x). That error is bounded by ОҙВ·||y вҲ’ x|| where Оҙ is the *Hessian* dissimilarity ||вҲҮВІf_i вҲ’ вҲҮВІf|| вүӨ Оҙ. Not G вҖ” Оҙ. The drift therefore contracts through ||I вҲ’ О·(A_i вҲ’ A)||ВІ вүӨ (1 + О·Оҙ)ВІ вүӨ 1 + 3О·Оҙ, so it stays controlled exactly when the Hessians are close. Working the quadratic analysis through, the Пғ = 0 communication rate becomes (ОІ + ОҙK)/(ОјK) = ОІ/(ОјK) + Оҙ/Ој. The ОІ/(ОјK) term falls with local work until around K = ОІ/Оҙ, where it is comparable to the Оҙ/Ој floor; beyond that, extra local steps mainly spend computation because the communication bound has saturated at the Hessian-mismatch term. If the clients share a common curvature (Оҙ = 0) the rate improves *linearly* with K вҖ” local steps are pure profit вҖ” even if their optima x_i* are arbitrarily far apart and G is unbounded. That's the surprise the gradient-dissimilarity lens completely misses: what makes local steps pay off is similarity of the *Hessians*, not closeness of the optima. And it explains the contrast with the exact proximal-point ancestor that also takes "large K": solving the subproblem exactly buys a ОҙВІ/ОјВІ dependence, while taking K stochastic gradient steps and correcting them with control variates buys Оҙ/Ој вҖ” a quadratic improvement вҖ” because the control variate fixes the direction directly instead of relying on a proximal anchor.

Let me also confirm the partial-participation story holds together, since sampling is where naive variance reduction usually falls apart. The danger is that with only S of N clients refreshing their c_i, the global c drifts away from the true average gradient. The defense is the invariant c = (1/N)ОЈ_i c_i, which the |S|/N-scaled server update c вҶҗ c + (1/N)ОЈ_{iвҲҲS}(c_iвҒә вҲ’ c_i) is supposed to preserve. With N = 6, S = 3, feeding arbitrary refresh targets to the sampled clients for 300 rounds and tracking ||c вҲ’ mean_i c_i|| at each step: with the divide-by-N update the gap stays at 1.2eвҲ’15 (machine zero) for all 300 rounds, so c tracks the true all-client mean exactly despite only ever touching half the clients. To be sure the N is load-bearing and not incidental I reran the same sampled stream with a divide-by-|S| update, and the gap grew to вүҲ1.9 вҖ” c no longer equals mean_i c_i, the invariant the whole correction depends on is broken, and the over-weighting of the sampled clients is precisely the failure I worried about. So the |S|/N factor is doing real work, not cosmetic bookkeeping. Granting the invariant, the control-lag recursion C_r вүӨ (1 вҲ’ S/N)C_{rвҲ’1} + вҖҰ shows the rest: the lag introduced by *not* updating N вҲ’ S clients decays geometrically at rate (1 вҲ’ S/N) and only adds the N/S term to the round count. The method is "client-variance-reduced": it's relatively unaffected by which clients I happen to sample, which is exactly what I need when availability is out of my control.

So the translation into the federated harness's open slots is direct, piece for piece: the state to keep is the server control variate and one control variate per client, both zero-initialized so c = mean_i c_i holds from the start; the client-side transform adds the fixed correction (c вҲ’ c_i) to the gradient before every plain-SGD step, so the local optimizer never changes; the free refresh is exactly the Option-II formula, c_iвҒә = c_i вҲ’ c + (x вҲ’ y_i)/(KО·_l), built from the pre- and post-round parameters the harness already hands back; and the aggregation step moves x by the О·_g-scaled mean client displacement while updating c by the |S|/N-scaled mean of the client control deltas вҖ” the one place a careless implementation (dividing by |S| instead of N) would silently break the invariant everything else relies on.
