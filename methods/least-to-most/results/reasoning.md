Let me look at exactly where chain-of-thought breaks, because the break is so clean it points straight at the fix. I take the last-letter task: given a list of words, output the concatenation of their last letters. "thinking, machine" → "ge". I write CoT exemplars on 2-word lists, with a nice rationale — "last letter of thinking is g, last letter of machine is e, concatenate to ge" — and the model nails 2-word test lists. Then I test on length-12 lists. It falls apart, and the longer the list, the worse it gets, the accuracy sliding down with each added word. Same story on the compositional benchmark where I map commands to action sequences: exemplars on short commands, and the model can't stretch to the long ones in the length split. Same on math: it handles problems with as many steps as the exemplars and degrades as the gold solution needs more steps.

So the failure has a name — the test problem is *harder* than the demonstrations, and CoT can't bridge that. It can imitate the kind of reasoning it was shown; it cannot extend it to a deeper or longer instance. Why not? Look at what a CoT exemplar actually is. It's a single rationale that solves the *entire* problem in one continuous shot. For a 2-word list that rationale is short and the model reproduces its shape. For a 12-word list the model would have to produce a much longer rationale of the same shape, in one pass, holding all twelve steps coherently — and that one-pass extrapolation is exactly what it can't do. The exemplar taught it "here is a complete solution of this size," and a size it hasn't seen is out of reach.

Now contrast with how a person handles the 12-word list. Nobody solves it in one mental shot. You take the first word, get its last letter, then the next word, append, then the next — you've turned one hard 12-step problem into a sequence of trivial 1-step problems, each building on the last. The reason a person generalizes from easy to hard is that they *decompose* the hard instance into easy pieces and chain them. CoT skips this; it tries to leap the whole gap at once. So let me push on the decomposition route and see where it leads.

The first move, then: don't ask the model to solve the hard problem in one rationale. First break the problem into a sequence of simpler subproblems, then solve those subproblems one at a time. The open question is what "subproblem" should mean and whether the subproblems can stay independent. Let me hold that question and pin down what decomposition itself buys me.

Decomposition isn't new, so I want to be careful about which version I'm reaching for. The usual versions split a question into *independent* subquestions, answer each on its own, and aggregate at the end — and they often need trained models to do the splitting and the aggregating. Let me check whether independent subquestions can possibly work for the last-letter task. The 12-word answer is the concatenation of twelve last letters in order. If I split into twelve independent "what is the last letter of word k" subproblems and answer each in isolation, the model never has to hold a 12-long string — each subproblem is trivially easy and demonstration-sized. So far that looks fine. The trouble is the aggregation: something still has to concatenate twelve answers in the right order into one string, and that concatenation *is* the 12-step thing the model couldn't do in one pass. Independent decomposition just relocates the hard step to the aggregator. So independence isn't enough on its own; the hardness has to be dissolved, not moved.

That tells me what the subproblems have to look like. Instead of "last letter of word k," make subproblem k be "the concatenation up to word k" — and let it *use* the answer to subproblem k−1. Then subproblem k is "take the previous answer and append one letter," which is demonstration-sized, and there is no separate aggregation step at the end, because the last subproblem *is* the full answer. The subproblems are dependent and ordered: later ones genuinely need the answers to earlier ones as building blocks. That dependency is the thing that lets me both keep each step easy *and* not smuggle the hard step into an aggregator.

And both stages — splitting and solving — I want done by the *same frozen LM* via few-shot prompting, no training, no separate decomposition model. Decompose by prompting, solve by prompting.

Stage one, decomposition. I write a prompt whose exemplars demonstrate how to break a problem into an ordered list of subproblems, then I append the actual problem and let the model produce its decomposition. For the last-letter task the decomposition of a list is its sequence of growing prefixes. The solving exemplars will cover a two-word base case, so the chain starts there and extends one word at a time: "think, machine", "think, machine, learning", "think, machine, learning, reasoning". One exemplar teaches the model to expand any list into this chain of sublists, and it should apply the same expansion to an arbitrary-length list.

Stage two, sequential solving — and this is where the carry-over has to be built into the prompt itself, so let me think about what the solving exemplars need to demonstrate. I want the model to learn: to answer the current subproblem, *use* the answer to the previous one. So the solving exemplars can't just be independent worked examples; they have to model the dependency. Here is the pair I'd use for last-letter. First exemplar: Q: "think, machine" → A: last letter of "think" is "k", last letter of "machine" is "e", concatenating gives "ke", so "think, machine" outputs "ke". Second exemplar: Q: "think, machine, learning" → A: *"think, machine" outputs "ke"*. The last letter of "learning" is "g". Concatenating "ke", "g" leads to "keg", so the answer is "keg". The second exemplar doesn't recompute the last letters of "think" and "machine"; it *starts from the previous answer* "ke" and extends it. The two exemplars together are a base case and a recursive step: given the answer to the (k−1)-prefix, compute the k-prefix by appending one letter.

So the procedure at solve time: decompose the input list into its chain of sublists, with the original list as the final subproblem. Then walk the chain. For each sublist S, build a solving prompt = [the base-and-recursive exemplars] + [the previously solved sublist/answer pairs so far] + [S], send it to the model, take the answer, and append that (sublist, answer) pair to the running context before moving to the next S. The last answer, produced for the original list, is the final solution.

Before I trust this, let me actually run the loop in my head on a list that's longer than either exemplar, to make sure the threading produces the right string and not just a plausible-looking one. Take "think, machine, learning, reasoning" — one word past the longest exemplar. Decomposition should give the chain: "think, machine"; "think, machine, learning"; "think, machine, learning, reasoning".

- Step 1, S = "think, machine". History is empty. The base-case exemplar covers exactly this, so the answer is "ke". History now holds ("think, machine" → "ke").
- Step 2, S = "think, machine, learning". The recursive exemplar plus the history shows the model: "think, machine" outputs "ke". Last letter of "learning" is "g". Concatenate "ke", "g" → "keg". History now also holds ("think, machine, learning" → "keg").
- Step 3, S = "think, machine, learning, reasoning". This is the new size, the one CoT couldn't reach. But the prompt now contains ("think, machine, learning" → "keg") sitting right above it, in exactly the shape the recursive exemplar taught. So the model should do: "think, machine, learning" outputs "keg". Last letter of "reasoning" is "g". Concatenate "keg", "g" → "kegg". Final answer "kegg".

Now check that "kegg" is actually right by the *definition* of the task, computed independently of the loop: last letters of think/machine/learning/reasoning are k, e, g, g, and their concatenation in order is "kegg". It matches. And notice what the step-3 prompt never asked the model to do: it never recomputed k, e, g from the first three words. It only had to find one last letter ("g" of "reasoning") and append it to a string it was handed. That is a 1-step problem, identical in difficulty to the base case, even though the instance is now length 4 — and the same argument runs unchanged at length 12, where step 12 is still "find one letter, append to the handed string." The hardness really did get absorbed into the *number* of steps rather than the size of any single step. That's the check I wanted: the construction doesn't just look like it generalizes, the threaded string comes out equal to the direct definition while every individual step stays demonstration-sized.

This also retroactively settles the independent-vs-dependent question from before. If I'd used independent solving exemplars — each recomputing the whole concatenation from scratch — then step 3 would have had to produce "kegg" by finding all four last letters in one rationale, which is the length-4 one-pass problem I started out unable to do. The dependency is doing real work: it is the difference between step 3 being a 1-letter append and step 3 being a full 4-letter recomputation.

Worth pinning down precisely how this differs from CoT on the same task, since both can use the *same* two lists. The CoT prompt's rationale for "think, machine, learning" recomputes everything from scratch — last letter of think, of machine, of learning, concatenate all three — independent of the "think, machine" exemplar above it. CoT's exemplars are independent; the ones I just designed are deliberately dependent, the second built on the first. That single design difference — does the later answer reuse the earlier one — is the lever I just saw move the worked example from a length-4 recomputation to a length-1 append.

Now the compositional command→action task, because it stresses the decomposition in a way prefixes don't. Here a long command like "look opposite right thrice after walk" composes shorter commands with connectives. So decomposition isn't prefixing; it's parsing the command into its constituent shorter commands in dependency order. The decomposition exemplars demonstrate exactly that: Q: "look opposite right thrice after walk" → A: "look opposite right thrice" can be solved by "look opposite right", "look opposite right thrice"; "walk" can be solved by "walk"; so the prerequisite list is "look opposite right", "look opposite right thrice", "walk". Let me sanity-check that this list is genuinely in dependency order: "look opposite right" is a prerequisite of "look opposite right thrice" (thrice repeats it), and "walk" is independent of both — so any order that puts "look opposite right" before "look opposite right thrice" is valid, which this list does. Then the solve loop appends the original command as the final mapping question, so the model first translates the three pieces and then composes the full command from those translations.

The mapping prompt — shared with the CoT baseline — teaches how to map each command to actions, and crucially how a command's output is *composed* from its parts' outputs: "the output of 'jump left' concatenates the output of 'turn left' and the output of 'jump'", and so on. One practical wrinkle I have to handle: a long command can expand to a very long action string, and spelling out "LOOK LOOK LOOK" inline will blow the model's context budget. So in the intermediate representations I use a compact Python-ish notation, "LOOK" * 3, which keeps each step short; a tiny postprocessing script expands these expressions at the end. The decomposition prompt and the mapping prompt are separate here (8 decomposition exemplars, 14 mapping exemplars chosen to cover the command semantics), whereas for last-letter the two pieces are smaller.

For math word problems the two stages don't always have to be separate decomposition and solving prompts. I can fold decomposition and subproblem solving into one generated response: an exemplar first says "Let's break this problem down: 1. <subquestion 1> 2. <subquestion 2>" and then answers them in sequence within the same response, each numbered answer using the earlier ones. Let me check that the carry-over still functions when it's folded like this, on a tiny instance: "Elsa has 5 apples; Anna has 2 more than Elsa; how many together?" Subquestion 1, how many does Anna have — Elsa's 5 plus 2 is 7. Subquestion 2, how many together — and this answer must *use* subquestion 1's: 5 + 7 = 12. The dependency is intact even inside one response: the "12" can't be produced without the "7" computed just above it. So merging the stages doesn't break the carry-over property; it just inlines it. In the revised one-shot GSM8K setup, the initial prompt ends after "Let's break down this problem:", then the model's reply is appended along with "The answer is:" for one short follow-up request that extracts the final answer. Same spirit — decompose, then solve sequentially with carry-over — without needing the full two-prompt loop used for last-letter and SCAN.

So why does this generalize to harder instances when CoT doesn't? The worked length-4 trace already showed the mechanism concretely, and the argument is the same at any length: the difficulty the model faces at any single step is bounded by the *subproblem*, not by the whole problem. Each subproblem is no harder than the exemplars — one letter to append, one short command to map, one arithmetic step — regardless of how long or deep the overall instance is. The instance's difficulty is absorbed by the *number* of steps, and the model only ever does one easy step at a time, with the prior answers handed to it. A 12-word list is just as easy per step as a 2-word list; there are simply more steps. That's the easy-to-hard bridge: hold each step at demonstration-level difficulty and let the chain be as long as the instance demands. It also stays fully interpretable and needs no training — both stages are just the frozen LM following few-shot demonstrations. And it composes with the other prompting tricks: each subproblem's solution can itself be a chain of thought, and I can self-consistency-sample the answers; the decomposition-and-thread idea is orthogonal to those.

The code is the two-stage loop made literal: decompose once, then solve the subproblems in order, threading each answer into the next prompt.

```python
def llm(prompt, stop=None):
    ...   # frozen LM completion

# Stage-1 exemplars: how to break a problem into an ordered list of subproblems.
# (last-letter: a list -> growing prefixes, starting at the two-word base case)
DECOMPOSITION_EXEMPLARS = '''Q: "think, machine, learning, reasoning"
A: "think, machine", "think, machine, learning", "think, machine, learning, reasoning"
'''

# Stage-2 exemplars: a base case + a recursive step that BUILDS ON the previous answer.
SOLUTION_EXEMPLARS = '''Q: "think, machine"
A: The last letter of "think" is "k". The last letter of "machine" is "e". \
Concatenating "k", "e" leads to "ke". So, "think, machine" outputs "ke".

Q: "think, machine, learning"
A: "think, machine" outputs "ke". The last letter of "learning" is "g". \
Concatenating "ke", "g" leads to "keg". So, "think, machine, learning" outputs "keg".
'''

def decompose(question):
    out = llm(DECOMPOSITION_EXEMPLARS + f'Q: {question}\nA:')
    return parse_subproblems(out)         # -> ordered list of sub-instances

def ensure_final_subproblem(subproblems, question):
    return subproblems if subproblems and subproblems[-1] == question else [*subproblems, question]

def solve(question):
    subproblems = ensure_final_subproblem(
        decompose(question), question
    )                                     # stage 1: ordered prerequisites + original problem
    history = ""                          # accumulated (subproblem, answer) pairs = the building blocks
    answer = None
    for S in subproblems:                 # stage 2: solve in order, each on top of prior answers
        prompt = SOLUTION_EXEMPLARS + history + f'Q: {S}\nA:'
        answer = llm(prompt).strip()
        history += f'Q: {S}\nA: {answer}\n\n'   # thread this answer into the next subproblem's prompt
    return answer                         # answer to the final subproblem = the solution
```

Tracing the loop's `history` on "think, machine, learning, reasoning" is exactly the hand-computation above: after the three iterations `history` ends with `Q: "think, machine, learning"\nA: ... keg` in place, so the final iteration's prompt hands "keg" to the model and the returned `answer` is "kegg" — equal to the directly-computed concatenation of last letters, with no single step ever larger than the base exemplar.

So the causal chain: chain-of-thought solves the whole problem in one rationale, so it can't extend to instances harder than its exemplars — accuracy slides as test difficulty exceeds demonstration difficulty. Mimic how people generalize easy-to-hard: decompose the hard instance into an ordered sequence of easy prerequisite subproblems, make the original problem the final subproblem, then solve the sequence one at a time with each answer building on the last — not independent subproblems plus an aggregator, since the aggregator would just inherit the hard step. Do both stages with the same frozen LM by few-shot prompting — decomposition exemplars that peel the problem apart, and solving exemplars that demonstrate a base case plus a recursive step reusing the previous answer. Then each step the model faces is only as hard as a demonstration, and the instance's difficulty is carried entirely by the *number* of steps, which is what lets a frozen model solve problems far harder than anything in its prompt, with no training.
