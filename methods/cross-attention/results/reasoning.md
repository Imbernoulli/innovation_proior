Let me start from what actually breaks when I try to build one network that can eat all of this weather and climate data. A single example is a stack of physical fields on a latitude-longitude grid: 2 m temperature, 10 m wind components, geopotential at several pressure levels, temperature and humidity at several more, some static surface masks. Shape it the obvious way and it's a tensor `V × H × W` — `V` variables, each an `H × W` map. The whole reason I want one network is that there's an enormous amount of this data lying around: decades of reanalysis, and far more from climate-simulation archives, dozens of model runs each. If I could pretrain on all of it the way language and vision pretrain on web-scale corpora, I'd have a foundation model for the atmosphere. But here's the rub, and it's not subtle: those data sources don't agree on what `V` is. One simulation gives me temperature and humidity, another wind and geopotential, a third some overlapping-but-different mix. And even at deployment I'll often want to feed only a subset of variables, or one a particular product never recorded. So the variable set is not a constant of the problem. It's an input.

Now look at how every standard model would ingest `V × H × W`. It treats the `V` axis as the channel axis. A ResNet's or UNet's first convolution has `in_channels = V`; a Vision Transformer cuts the image into `p × p` patches and folds the channels into each patch vector of size `p² · V`. Either way the very first layer's input dimension is welded to exactly `V`. The moment a different dataset hands me a different set of variables, that first layer doesn't fit anymore. I'd have to rebuild and retrain the input stage for every variable set. That single fact kills the dream: I can't pretrain across heterogeneous sources, can't accept a subset at finetune time, can't take in a variable that wasn't in the original channel list. And there's a quieter loss too — the instant that first convolution sums across the `V` input channels, the identity of each variable is gone; nothing downstream knows that *this* contribution was temperature and *that* one was humidity. The channel view throws away exactly the structure I need to be flexible about.

So the channel axis is the wrong place to put the variables. What's the alternative? The thing that made ViT flexible about *spatial* size was treating the image as a *set of tokens* instead of a fixed-shape array — a token sequence can be any length, so the model stops caring about the exact grid. Let me carry that idea over to the variable axis. Instead of stacking the variables into channels, give each variable its own tokens. Tokenize each variable's `H × W` map separately: cut it into `h × w` patches, linearly embed each patch (one variable is a single-channel map, so this is just a `p × p` conv with one input channel projecting to `D`). Do that for every variable present in this particular input. Now I have, for each spatial patch location, one `D`-dimensional token per variable — a tensor `V × h × w × D`. Crucially, `V` is no longer baked into any weight matrix; it's just how many tokens I happen to produce this time. To keep them distinguishable I'll keep a small learnable embedding per variable — one trainable vector for each variable in my full vocabulary of possible variables, added to that variable's tokens, the same trick as a positional embedding but along the variable axis instead of the spatial axis — so that downstream I can still tell a temperature token from a humidity token. Pick out the embeddings for the variables actually present, add them in, done. The variable set is now genuinely an argument, not an architectural commitment. I can pretrain on whatever mix a source gives me; I can finetune on a subset; I can even handle a variable unseen in one run, as long as it's in the vocabulary I allocated embeddings for. This is the flexibility I was after.

Except I've just created two real problems, and I should be honest about both before I get attached to this.

The first is cost. By turning variables into tokens I multiplied my sequence length by `V`. The backbone is a Transformer, and self-attention is quadratic in sequence length. A sequence of length `V · h · w` costs `O((V · h · w)²)` — the `V` factor comes inside the square, so the cost scales like `V²` on top of the spatial cost. With `V` reaching a few dozen on realistic inputs, that's not a constant I can wave away; it's the difference between feasible and not. The channel view at least kept the sequence length at `h · w`. I've traded its rigidity for an `O(V²)` blowup.

The second is semantics. The tokens in this long sequence are wildly heterogeneous. A token summarizing geopotential at 500 hPa and a token summarizing specific humidity at 925 hPa and a token summarizing 10 m wind are different physical objects — different units, different scales, different dynamics. Even after I normalize each variable to zero mean and unit variance so they coexist numerically, asking a stack of self-attention layers to make sense of a sequence that interleaves all these groundings is asking a lot. The backbone would spend capacity just sorting out which tokens are commensurable before it could do anything useful. The channel view, for all its faults, at least handed the network one coherent feature vector per location.

So I'm boxed in. Channels: cheap and coherent but rigid. Per-variable tokens: flexible but `O(V²)` and a heterogeneous mess. I don't want to give up the flexibility — it's the whole point — so the question becomes: can I keep per-variable tokenization on the *input* side, where the flexibility lives, but reduce the `V` tokens at each spatial location back down to *one* token before the backbone ever sees them? If I could do that, the backbone's sequence is `h · w` again — the `V` factor and its quadratic blowup vanish — and if the reduction also re-expresses the heterogeneous variable tokens in one shared form, the semantic mess is cleaned up too. Both problems are really the same problem: at each spatial location, collapse a *set* of `V` variable tokens into a *single* `D`-vector. Let me focus entirely on that reduction. It happens independently at every one of the `h · w` locations, so I can think about a single location: I have `V` vectors in `R^D`, I want one vector in `R^D`, and `V` can be anything.

What's the simplest collapse? Average them. Take the per-location mean over the `V` variable tokens. Zero parameters, and it trivially accepts any `V`. Let me think about whether it's good enough before I reach for anything heavier. Two things bother me. It weights every variable identically, at every location, in every atmospheric state — but the informativeness of a variable is *not* uniform. At a given place and condition, maybe the wind tokens carry the signal and the static surface mask carries almost nothing; the mean can't tell. And it just adds up the differently-grounded raw tokens, so the muddied-vector problem from a moment ago doesn't actually get solved — I've averaged the soup, not clarified it. The mean is a fine floor, a sanity baseline, but it's content-blind by construction.

Can I fix the content-blindness cheaply? Attach a trainable scalar weight to each variable, softmax the scalars across variables so they're a proper convex combination, take that weighted sum. Now temperature can count more than the land-sea mask on average. Better — but stare at what those weights depend on. They depend on *nothing about the current input*. Once trained, variable `v` gets weight `w_v` no matter what its token actually says at this location, no matter the state of the atmosphere. The combination is still content-independent; I've only replaced a uniform average with a fixed non-uniform one. And it still just rescales-and-adds the raw tokens — no re-projection into a common space. So this buys me a little but leaves the core defect: the weighting can't react to the data.

What I actually want is a reduction whose weights are *content-dependent* — the weight on each variable token decided by what that token (and the others) currently say — and which re-expresses the tokens in a shared representation rather than averaging raw ones. Let me write down what that means at one location. I have value vectors and I want output `= Σ_v α_v · (something I do to token v)`, where the weights `α_v` are a function of the tokens themselves, normalized to a convex combination so the output stays a proper pooling and accepts any `V`. The "normalized, data-dependent convex combination over a set" is exactly the shape of a softmax. And "the weight comes from how well each token matches some reference" is exactly a compatibility score. I keep circling the same object: this is attention. The output is a weighted average of values, the weights are a softmax of query-key compatibilities, the softmax runs over the *set* of keys so it's invariant to their order and defined for any number of them — which is precisely the any-`V`, set-valued behavior I need. Attention isn't one option among several here; it's the thing that matches every requirement I just wrote down.

But attention needs a query, keys, and values. The keys and values are obvious — they're the `V` variable tokens at this location; I let the layer learn the projections, so values become `W^V` applied to each token (there's my re-projection into a shared space, for free) and keys become `W^K` applied to each. What's the query, though? In the standard sequence setting each output position has its own query coming from somewhere. Here I want exactly *one* output token per location, summarizing the whole set. So I want exactly one query — and there's no natural source for it, because the output token isn't a transformed version of any particular input token; it's a summary of all of them. So let the query be a free parameter. A single learnable vector, the same one reused at every spatial location and every example, that the network trains to ask the right question of the variable set: "given these `V` tokens, what's the right convex combination of their values to summarize this location?" One trainable query, cross-attending into the `V` keys and values.

I think I've seen this exact shape before. ViT's `[class]` token is a learnable vector prepended to the patch sequence whose job, after attention mixes it with the patches, is to be the pooled summary of the whole image — a single trainable query that gathers a set of tokens into one. What I'm describing is the same move pointed at the *variable* set at each location instead of at the spatial set. So at least the trainable-query-summarizes-a-set pattern isn't something I'm inventing from nothing; it's a known way to turn a set into a vector. That doesn't prove it does what I want here, though — I should pin down the actual behavior before I trust it.

Let me write the reduction at one location concretely. Let `q ∈ R^D` be the learnable query and `x_1, …, x_V ∈ R^D` the variable tokens. With projections `W^Q, W^K, W^V`,

```
score_v = (W^Q q) · (W^K x_v) / √d_k
α      = softmax over v of (score_1, …, score_V)
out    = Σ_v α_v (W^V x_v),
```

and then an output projection `W^O` on `out`. The weights `α_v` now depend on the actual token contents through the dot products, so the pooling reacts to the data; `W^V` re-expresses each variable token before mixing, so the output lives in a unified representation; and the softmax over `v` is a convex combination over the set, so any `V` works and order doesn't matter. The `1/√d_k` is not decoration — without it, for query/key components that are roughly independent with unit variance the dot product `Σ_{i=1}^{d_k} q_i k_i` has variance `d_k`, so its magnitude grows like `√d_k`; large logits push the softmax toward one-hot, where its Jacobian collapses and the gradient through the attention weights becomes tiny. Dividing by `√d_k` keeps the logits unit-variance and the softmax responsive. Exactly the reason it's there in ordinary attention, and I need it here for the same reason.

One head or several? A single query-key softmax produces one weighting pattern over the variables — one way of deciding which variables matter. But "which variables matter" can have several simultaneous answers: one head might learn to surface the thermodynamic variables, another the dynamical ones, and blurring those into a single softmax would force one compromise weighting. So run several attention heads in parallel, each with its own projections to a smaller `d_k = D / num_heads` subspace, concatenate their outputs, and mix with `W^O`. Different heads keep distinct "which variables matter for *this* aspect" patterns distinct instead of averaging them together, and because each head works in a `D / num_heads` subspace the total cost is about that of a single full-width head. Concatenation alone never lets the heads talk; `W^O` is what mixes their disjoint subspaces back into one `D`-vector. So: multi-head cross-attention, query = the single learnable vector, keys = values = the variable tokens.

Now the cost, which was half my motivation. After this reduction each spatial location emits one token, so the backbone's sequence is back to `h · w` — the `V` factor is gone and with it the `O(V²)` blowup; backbone self-attention is `O((h · w)²)` again, independent of how many variables I fed. And the reduction itself is cheap: at each location it's a softmax over `V` keys and a weighted sum, with `q` a single query, so it's `O(V)` per location, `O(V · h · w)` total — linear in `V`, not quadratic. Let me put the field's actual numbers on it to see whether the factor is worth chasing or just asymptotic bookkeeping. At 5.625° the spatial token count is `h·w = 16·32 = 512`. The full variable-and-space self-attention pays `(V·512)²`; the aggregated path pays `512²` in the backbone plus the `V·512` reduction. For `V = 4` that's about `4.2M` versus `264k` — a 16× gap. For `V = 12`, `38M` versus `268k`, 141×. For `V = 48`, the ERA5 default, `604M` versus `287k`, a 2100× gap. The quadratic factor isn't a rounding error; at realistic variable counts it's three orders of magnitude, which is the difference between training this and not. I pay a linear pass to delete that. That's the trade I wanted.

How many such layers? I only need to turn a set into one vector; a single cross-attention layer with a single query already does that and already produces exactly one output token per location. Stacking more would re-attend a length-one sequence into the variable set again, which buys little for the cost. One layer, one query token. Keep it minimal.

Let me think about initialization, because a learnable query starting from a bad place could make the early training ugly. The hope is that if I set `q = 0` the layer starts out behaving like mean pooling and only specializes from there — but I should check that rather than assert it, because the dot-product score is `(W^Q q + b_q) · (W^K x_v + b_k)/√d_k`, and the biases `b_q, b_k` that `nn.MultiheadAttention` carries are not obviously irrelevant. With `q = 0` the query side is just the constant `b_q`, so `score_v = b_q · (W^K x_v + b_k)/√d_k`, which in general still *varies with `x_v`*. Uniform weights are not automatic from `q = 0` alone; they need `b_q = 0` so that every score collapses to `0` and the softmax sees all-equal logits. PyTorch happens to initialize `in_proj_bias` to zeros, so the condition should hold at step zero, but let me actually run it rather than reason about it on paper.

I build the module with `D = 8`, two heads, and feed a random `[1, V=3, L=4, 8]` input with the query zero-initialized. The attention weights come back `[0.3333, 0.3333, 0.3333]` for every location — exactly uniform `1/V`, and the same for a second random key-set, so the uniformity is a property of the init and not of the particular tokens. Permuting the three variables changes the output by `3e-8` (floating-point noise), confirming the set really is treated order-free. Then the sharper question: is the zero-init output *literally* mean pooling, or only uniformly-weighted? I compute the layer's output on the full `V`-set and separately push the per-location mean of the tokens through the same layer (a length-1 "set" of just the mean). They agree to `6e-8`. So at `q = 0` the layer computes `W^O(W^V · mean_v x_v + b)` — the mean of the variable tokens, re-projected through the value and output maps. It is mean pooling, lifted into the layer's learned representation, exactly the safe prior I wanted: "treat all variables equally" before anything is learned. And to be sure the bias condition is the real lever and not a coincidence, I force a nonzero `b_q` and rerun: the weights become `[0.17, 0.27, 0.56]`, no longer uniform. So the uniform start genuinely depends on `b_q = 0`, which the default init gives me; good to know, and a thing I'd want to preserve if I ever changed the init. Zero-initialize the query, then; it begins as mean pooling and gradient descent moves it toward content-dependent weights.

That computed equivalence also tells me how to see the whole family, and now it's not a hopeful analogy — I just watched the layer *become* mean pooling at one corner of its parameter space. Mean pooling is the case where the softmax is pinned uniform (`softmax(0)`), with the value and output projections collapsing to a passthrough of the average. The fixed-learned-scalar-weights baseline is the case where the logits are learned constants for the variables rather than functions of the token contents. The full thing is the same machine with the weights set free to depend on the data and the values re-projected. So I haven't bolted on a third unrelated option beside the two cheap baselines; they sit *inside* this layer as degenerate settings of its weights — one with uniform weights, one with content-independent weights. That the simpler baselines fall out as special cases, and I can hit one of them numerically, is the closest thing I have to a check that I picked the right general object rather than an arbitrary one.

Now let me make the shapes concrete for an implementation, because the indexing is the only fiddly part. The tokenizer hands me `x` of shape `[B, V, L, D]`: batch `B`, `V` variables, `L = h · w` spatial locations, dimension `D`. The reduction is per-location and per-example and identical at every location, so I should treat every `(example, location)` pair as an independent little aggregation problem with a `V`-element set. So permute to bring the variable axis next to `D` — `[B, L, V, D]` — and fold the `B` and `L` axes together into one big batch: `[B·L, V, D]`. Now I have `B·L` independent problems, each a set of `V` tokens of width `D`. The query is the one learnable vector `[1, 1, D]`; expand it across the `B·L` batch to `[B·L, 1, D]`. Run multi-head cross-attention with that query against the `V` tokens as keys and values: out is `[B·L, 1, D]`. Drop the length-1 query axis to `[B·L, D]`, then unfold the batch back to `[B, L, D]`. That's the one token per location the backbone wants.

In code this is a single `nn.MultiheadAttention` with `batch_first=True` and a single `nn.Parameter` for the query. The dimensions: `embed_dim = D = 1024`, `num_heads = 16` so `d_k = D / num_heads = 64` per head — the same 64-wide heads the large Transformers use, a size that's empirically a good granularity. The variable count `V` never enters the module's weights at all; it's read from the input tensor's shape at runtime, which is the entire point — the module is agnostic to how many variables I feed it. The query is zero-initialized as argued. Let me fill the slot:

```python
import torch
import torch.nn as nn


class VariableAggregator(nn.Module):
    """Reduce per-variable tokens at each spatial location to one token via
    a learnable-query cross-attention over the variable set.

    Input:  x : [B, V, L, D]   (V variable tokens at each of L spatial patches)
    Output:     [B, L, D]      (one token per spatial location)
    V is read from the input at runtime, so any number of variables is accepted.
    """

    def __init__(self, embed_dim, num_heads, num_vars):
        super().__init__()
        self.embed_dim = embed_dim
        self.num_heads = num_heads
        self.num_vars = num_vars
        # one learnable query vector; zero-init starts with uniform weights over
        # the projected variable tokens, then learns content-dependent weights
        self.var_query = nn.Parameter(torch.zeros(1, 1, embed_dim), requires_grad=True)
        # multi-head cross-attention: query attends over the V tokens (keys = values)
        self.var_agg = nn.MultiheadAttention(embed_dim, num_heads, batch_first=True)

    def forward(self, x):
        b, v, l, d = x.shape
        x = x.permute(0, 2, 1, 3)   # [B, L, V, D]: bring the variable set next to D
        x = x.reshape(b * l, v, d)  # [B*L, V, D]: each (example, location) is its own set

        query = self.var_query.expand(b * l, -1, -1)  # [B*L, 1, D]: the shared learnable query
        out, _ = self.var_agg(query, x, x)            # [B*L, 1, D]: softmax over the V tokens
        out = out.squeeze(1)                          # [B*L, D]: drop the length-1 query axis

        out = out.reshape(b, l, d)  # [B, L, D]: one token per spatial location
        return out
```

Inside that `nn.MultiheadAttention`, for one location with query `q` and the `V` tokens, the packed input projection forms `W^Q q`, `W^K x_v`, `W^V x_v` per head; the per-head scores `(W^Q q)·(W^K x_v)/√64` go through a softmax over the `V` keys; the weighted sum of `W^V x_v` is computed per head; the heads concatenate and pass through the output projection `W^O`. That matches the per-location reduction I wrote down by hand, batched over all `B·L` locations at once — and the small run above is the part of that I could actually check: it returned `[B, L, D]` for `V` in `{1, 3, 7, 20}`, so the variable count really is read from the shape and never from a weight, and the output was permutation-invariant in the variables to floating-point precision. The one thing I have not verified here is that training actually drives the query off its uniform start toward useful content-dependent weights; that's a claim about optimization, not algebra, and I'd want to confirm it by watching the learned attention weights on real ERA5 — I expect different variables to dominate at different locations, but I can't settle that on paper.

Stepping back over the chain: I wanted one network that ingests an arbitrary set of physical variables, but the channel view welds the input layer to a fixed variable count and discards variable identity, so I moved the variables off the channel axis and onto a token axis — per-variable tokenization with a learnable per-variable embedding — which makes the variable set a runtime argument. That created an `O(V²)` self-attention blowup (measured above at ~2100× for `V = 48`) and a sequence of heterogeneous, differently-grounded tokens. Both reduce to one need: collapse the `V` variable tokens at each location into a single unified token before the backbone. Mean pooling does it but is content-blind; fixed learned scalar weights are still content-independent; what I needed was a normalized, data-dependent convex combination over a set that also re-projects the tokens — which is attention, with the `V` tokens as keys and values and a *single learnable query* (the same trainable-query-summarizes-a-set move as ViT's class token) producing one output per location. The `1/√d_k` keeps the softmax from saturating; multiple heads keep several "which variables matter" patterns distinct; one layer and one query suffice; zero-initializing the query gives uniform attention at the start — which I checked numerically collapses the layer onto mean pooling, placing both cheap baselines inside it as degenerate cases. The reduction is `O(V)` per location and returns the backbone's sequence to `h · w`, deleting the quadratic factor while unifying the token semantics — the two problems I created, solved by the same module.
