I started from an uncomfortable observation about direct preference training. The setup is attractive: we fit a policy from triples $(x, y_w, y_l)$ where $y_w$ is preferred to $y_l$, with no separate reward model and no RL loop, and the loss is just a supervised binary cross-entropy on a reference-relative log-ratio gap. Writing $\rho(y) = \log \pi_\theta(y\mid x) - \log \pi_{\text{ref}}(y\mid x)$, the standard direct loss is $-\mathbb{E}\,\log\sigma\!\big(\beta\,(\rho(y_w) - \rho(y_l))\big)$, whose implicit reward is $\hat r(x,y) = \beta\,\log(\pi_\theta(y\mid x)/\pi_{\text{ref}}(y\mid x))$. The trouble is that this objective is a function of a single scalar, the gap $\rho(y_w) - \rho(y_l)$, and a gap can be improved either because $\rho(y_w)$ rises or because $\rho(y_l)$ falls. The loss does not distinguish these two routes. So I can train, watch the chosen-minus-rejected margin grow, and still see the preferred completion's own log-probability fall. That matters whenever the preferred completion is not merely "better than the other sample" but is an actual satisfactory target — a distilled reasoning trace, a response converted from a labeled dataset, a positive that is supposed to stay likely under the new policy.

The existing options do not close this hole. Preferred-only supervised fine-tuning maximizes $\log\pi_\theta(y_w\mid x)$ and so directly protects positive likelihood, but it throws away the negative completion and loses the pairwise contrast that makes preference data valuable. SLiC-HF uses a margin ranking loss plus a cross-entropy regularizer, but it abandons the reference-relative Bradley-Terry parameterization. IPO replaces the saturating logistic with a squared regression of the log-ratio gap toward a fixed target, which controls gap overfitting but never imposes a one-sided penalty on the preferred likelihood dropping below the reference. None of them supply restoring pressure on $y_w$ while keeping the clean direct-preference setup intact. The failure is sharpest on low-edit-distance pairs. When $y_w$ and $y_l$ share almost all of their tokens — the GSM8K/MetaMath case where the dispreferred trace just corrupts one intermediate calculation, with normalized edit distance around $6.5\%$ — "push the chosen sequence up and the rejected sequence down" is not separable. For positions before the first differing token $m$, the preferred and rejected terms are identical and cancel. After $m$, the update is governed by how the next-token distributions under the two prefixes differ. Using the softmax identity $\partial \log s_i/\partial\theta_j = \mathbf{1}\{i=j\} - s_j$, the descent direction on a later token's logit from the DPO log-ratio term is $s_j^{\,y_l} - s_j^{\,y_w}$. If the starting model is already decent, then for the correct token index $i$ we have $s_i^{w}\ge s_i^{l}$, so that direction is non-positive: the correct later-token logit is pushed *down*, and the wrong tokens up. The contrastive update makes the continuation after the edit worse for exactly the trace we wanted to keep.

I propose DPO-Positive, or DPOP: DPO augmented with a soft, one-sided preferred-likelihood penalty placed *inside* the logit. The fix should not replace the gap, because the gap is the useful contrast; it should add pressure only when the preferred completion has fallen below the reference, i.e. only when $\rho(y_w) < 0$. An unconditional $-\rho(y_w)$ term would keep pushing even when $\pi_\theta$ already makes $y_w$ more likely than $\pi_{\text{ref}}$, collapsing the objective toward preferred-only SFT and discarding the one-sided character I need. The right shape is a soft floor — zero when $\rho(y_w)\ge 0$, positive when $\rho(y_w) < 0$ — which is exactly the hinge $\max(0, -\rho(y_w)) = \max\!\big(0,\ \log\pi_{\text{ref}}(y_w\mid x) - \log\pi_\theta(y_w\mid x)\big)$, measured in the same sequence-log-prob units as the DPO logit. The crucial design choice is placement: the penalty goes inside the DPO logit, not as a second additive loss outside the $\log\sigma$. The pre-sigmoid score becomes
$$\rho(y_w) - \rho(y_l) - \lambda\,\max\!\big(0,\, -\rho(y_w)\big),$$
and the loss is
$$L_{\text{DPOP}} = -\mathbb{E}\,\log\sigma\!\left(\beta\left[\log\frac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)} - \log\frac{\pi_\theta(y_l\mid x)}{\pi_{\text{ref}}(y_l\mid x)} - \lambda\,\max\!\Big(0,\ \log\frac{\pi_{\text{ref}}(y_w\mid x)}{\pi_\theta(y_w\mid x)}\Big)\right]\right).$$

What makes this the right object is that it changes only the case that was broken. With $\lambda = 0$ it is exactly DPO. On any example with $\rho(y_w)\ge 0$ the hinge and its derivative are both zero, so the update is exactly ordinary DPO — the contrast is untouched wherever the preferred likelihood is already healthy. Below the floor, where $\rho(y_w) < 0$, the logit becomes $(1+\lambda)\,\rho(y_w) - \rho(y_l)$, which is the ordinary DPO preferred update plus an extra $\lambda$ times the preferred log-probability update. The sign check is the load-bearing step: the penalty is *subtracted* from the logit, the outer loss wants the logit *larger*, so minimizing the loss drives the positive penalty down by raising $\log\pi_\theta(y_w\mid x)$ relative to $\log\pi_{\text{ref}}(y_w\mid x)$ — restoring pressure, applied only when needed, with the rejected-completion contrast still present. Re-running the one-token analysis with the hinge active confirms it fixes the wrong-way continuation. For the correct token's own logit the descent direction becomes $\lambda\,(1 - s_i^{w}) + s_i^{l} - s_i^{w}$, and for any other token $j\neq i$ it becomes $-(\lambda+1)\,s_j^{w} + s_j^{l}$. Since $s_i^{w}\le 1$, a large enough $\lambda$ makes the correct-token direction positive; since $s_j^{w} > 0$ for softmax probabilities, a large enough $\lambda$ makes the off-token directions negative. That is precisely the case split I wanted: below the preferred-likelihood floor the update reverses DPO's wrong-way move, and at or above the floor it is plain DPO.

The two hyperparameters read off the formula directly. $\beta$ scales the entire logit, as in DPO; $\lambda$ scales the one-sided preferred-likelihood term before that $\beta$ multiplication. I am careful not to claim $\lambda$ is a hard guarantee that $\rho(y_w)$ can never go negative — it is a soft restoring pressure. Too small and the objective is essentially DPO; too large and it dominates the contrast and starts to look like preferred-only likelihood training. The default recipe uses $\beta = 0.3$ and $\lambda = 50$, with AdamW at learning rate $5\times10^{-5}$ for 1000 steps, sweeping $\beta\in\{0.1, 0.3, 1.0\}$ and $\lambda\in\{5, 50, 500\}$. One implementation detail matters for correctness: use summed, not length-averaged, completion log-probabilities, since the Bradley-Terry form is defined on summed sequence log-probs. Translated into the trainer's tensors, the standard DPO score before $\beta$ is $(\text{policy\_chosen} - \text{policy\_rejected}) - (\text{reference\_chosen} - \text{reference\_rejected})$, the active penalty is $\mathrm{relu}(\text{reference\_chosen} - \text{policy\_chosen}) = \mathrm{relu}(-\rho(y_w))$, and DPOP simply subtracts $\lambda\cdot\text{penalty}$ from that score before applying $\beta$ and $\log\sigma$. With label smoothing zero it reduces to the clean equation; with smoothing enabled it uses the standard TRL form with the flipped $-\beta\cdot\text{logits}$ term.

```python
import torch
import torch.nn.functional as F


def dpop_loss(policy_chosen_logps, policy_rejected_logps,
              reference_chosen_logps, reference_rejected_logps,
              beta, lambda_dpop, label_smoothing=0.0):
    """DPOP on summed completion log-probs."""
    pi_logratios = policy_chosen_logps - policy_rejected_logps
    ref_logratios = reference_chosen_logps - reference_rejected_logps
    logits = pi_logratios - ref_logratios

    penalty = torch.clamp(reference_chosen_logps - policy_chosen_logps, min=0.0)
    logits = logits - lambda_dpop * penalty

    losses = (
        -F.logsigmoid(beta * logits) * (1.0 - label_smoothing)
        - F.logsigmoid(-beta * logits) * label_smoothing
    )
    chosen_rewards = beta * (policy_chosen_logps - reference_chosen_logps).detach()
    rejected_rewards = beta * (policy_rejected_logps - reference_rejected_logps).detach()
    return losses, chosen_rewards, rejected_rewards
```
