I want a layer that mixes information across the positions of a sequence — when I compute the representation at one position, I need to pull in whatever is relevant from the other positions, no matter how far away they are. Today I do this with recurrence: thread a hidden state left to right, `h_t = f(h_{t-1}, x_t)`, and let each position see the past through that one running state. Two things about this grate on me. First, it's inherently sequential — I literally cannot compute `h_t` until `h_{t-1}` is done — so I can't parallelize across positions within a single example, and that's the binding constraint exactly when sequences get long. Second, and this is the deeper one, every long-range dependency has to survive a long chain. For position `i` to influence position `j`, the signal travels `|i-j|` steps of state overwriting, and it's well established that the longer the path a forward or backward signal has to traverse, the harder that dependency is to learn — the gradient washes out along the chain. So recurrence gives me `O(n)` sequential operations and an `O(n)` path between distant positions, and both are exactly backwards from what I want.

Convolution buys back the parallelism — stack of conv layers, all positions computed at once, `O(1)` sequential — but a kernel of width `k` only looks at a window, so to connect two far-apart positions I need a stack deep enough to bridge them, and the path length grows with distance again: linearly for plain kernels, logarithmically with dilation. Better, but I haven't actually removed the thing I care about. What would it take to make the path length *constant* — any position reaching any other in a single hop, `O(1)` sequential and `O(1)` path? Let me hold those two numbers as the target and ask what such a layer would have to look like.

If every reader position gets to look at every holder position directly, then for a sequence of length `n` I'm forming something for all `n^2` ordered pairs. So the per-layer compute is going to scale like `n^2` times the width `d`, i.e. `O(n^2 · d)`. Is that affordable? Compare to recurrence's `O(n · d^2)`. The two cross over at `n = d`: the all-pairs layer is cheaper exactly when `n < d`, when the sequence is shorter than the representation is wide. And for the sequences I care about — sentences, tokenized into subword units so the vocabulary stays bounded — `n` really is typically below `d`. So in the common regime I'm not paying extra for the all-pairs connection; it's actually the cheaper layer, and it's the one with `O(1)` sequential ops and `O(1)` path. The all-pairs read is worth chasing.

Now, what *is* the read? I keep coming back to the picture of a soft dictionary lookup. Give each position three roles. As a reader, it emits a query — a vector saying "here is what I'm looking for." As a holder, it emits a key — "here is what I am, what I advertise" — and a value — "here is what I'd contribute if you read me." The output for a reader is then a blend of the holders' values, and the blend should be weighted by how well the reader's query matches each holder's key. Stack the queries into a matrix `Q`, one row per reader; the keys into `K`, the values into `V`, one row per holder. So I need two ingredients: a *compatibility* function that turns a (query, key) pair into a score, and a *normalization* that turns a reader's row of scores into a set of blend weights.

Start with the normalization, because that part feels forced. The weights should be nonnegative — a negative weight on a value makes no sense as "how much of this do I read" — and they should sum to one, so the output is a genuine weighted average, a convex combination of the values, and stays in their convex hull rather than blowing up. And I want the whole thing smoothly differentiable so I can train through it. The standard object that takes a row of real scores and returns a nonnegative, sum-to-one, smooth weighting is the softmax: `softmax(z)_i = exp(z_i) / Σ_j exp(z_j)`. So a reader's output is `Σ_j softmax(scores)_j · v_j` — exactly the soft-alignment idea that already made translation work, where the decoder computes weights `α_ij = exp(e_ij)/Σ_k exp(e_ik)` over the source annotations and reads back the expected annotation `c_i = Σ_j α_ij h_j`. I'm reusing that machinery wholesale; the question is what goes inside as the energy `e_ij`.

The incumbent answer for that energy is an additive score: a little feed-forward net over the reader and holder vectors, `e_ij = v_a^T tanh(W_a s_{i-1} + U_a h_j)`. It works, it's differentiable, it's the original. But I want to stress-test it against my actual constraint, which is *this layer runs many times, in every layer of a deep stack*, not once as a bolt-on. Look at what that additive score costs. For each of the `n^2` (reader, holder) pairs I have to evaluate an MLP: a matrix multiply to get `W_a s` and `U_a h`, an elementwise `tanh`, and an inner product with `v_a`. That's a per-pair nonlinear function — it does not collapse into a single big matrix multiply. When attention is a one-shot read over the source per decoder step, that's fine. But as the primary mixing operation I'm calling over and over, the constant factor and the memory traffic of evaluating a small network `n^2` times per layer, in every layer of the stack, is the wrong cost profile. The thing I'm running everywhere has to be something the hardware loves, and what the hardware loves is one large dense matrix multiply.

So can the compatibility be a single matmul? The simplest compatibility between a query vector and a key vector that's just a number measuring alignment is the dot product, `q · k = qᵀk`. And here's what's appealing: the entire matrix of scores for all pairs is `Q Kᵀ` — one matrix multiplication, `(n × d_k)·(d_k × n) = (n × n)`. No per-pair MLP, no per-pair nonlinearity; one GEMM that the hardware runs at peak throughput. That's the multiplicative score, and it's why I'd reach for it over the additive MLP even though, in raw theoretical complexity, the two are similar — the difference that matters is that the dot product *is* a single optimized matrix multiply, and I'm going to call this thing constantly. So my read is shaping up to be `softmax(Q Kᵀ) V`: form all scores in one matmul, softmax each reader's row, blend the values in a second matmul. Two big GEMMs and a softmax in between.

But I should be honest that I'm trading something for that single matmul, and I half-remember why the dot-product score wasn't simply adopted everywhere already. There's a reported result that nags at me: when people study these attention mechanisms across a range of widths, the additive score and the multiplicative score are neck and neck at small attention dimension — both around the same BLEU at width 128, 256, 512 — but as the attention dimension grows toward 1024, the *multiplicative* one falls off a cliff (BLEU around 18 where the additive holds near 22), while the additive one holds steady. A several-point BLEU collapse, and only for the dot-product variant, and only at large width. That's a width-dependent failure mode of the dot product specifically. If I'm going to make the dot product the heart of a deep, wide model, I cannot wave that away — wide is exactly where I'm headed. I need to understand why a plain dot product might break as the vectors get wide, and whether it's fixable — what happens to the magnitude of `q · k` as `d_k` grows?

Model the entries as random for the back-of-envelope: suppose the components of `q` and of `k` are independent of each other, each with mean 0 and variance 1. That's a reasonable stand-in for what normalization tries to enforce on the activations anyway. Then the dot product is `q · k = Σ_{i=1}^{d_k} q_i k_i`. Its mean: `E[q·k] = Σ_i E[q_i k_i] = Σ_i E[q_i] E[k_i] = 0`, using independence to split the expectation and then mean-0 on each factor. So the score is centered at zero, good. Now its spread — the variance. Since the mean is 0, `Var(q·k) = E[(q·k)^2] = E[(Σ_i q_i k_i)(Σ_j q_j k_j)] = Σ_i Σ_j E[q_i k_i q_j k_j]`. Split that double sum into the diagonal `i = j` and the off-diagonal `i ≠ j`. On the diagonal, `E[q_i^2 k_i^2] = E[q_i^2] E[k_i^2] = 1 · 1 = 1` (independence of `q_i` and `k_i`, then variance 1 each since mean is 0). Off-diagonal, `i ≠ j`: `E[q_i k_i q_j k_j] = E[q_i] E[k_i] E[q_j] E[k_j] = 0`, because all four factors are independent and each has mean 0 — the whole cross term vanishes. So only the `d_k` diagonal terms survive, each contributing 1:

  Var(q·k) = Σ_{i=1}^{d_k} 1 = d_k.

So the logits feeding the softmax have mean 0 but variance `d_k` — standard deviation `√d_k`. As the vectors get wider, the *scores themselves get bigger*, growing like `√d_k`, purely because I'm summing more terms. Does that actually hurt the softmax? Take `d_k = 64`, so the logit std is `√64 = 8`, and write down one representative reader's row of four raw dot-product logits at that spread: `[8.0, −2.0, 1.0, −4.0]`. Softmax of that row comes out to about `[0.999, 0.000, 0.001, 0.000]` — essentially one-hot, the top logit has eaten everything. Now I want to know what that does to learning, so I look at the softmax Jacobian `∂softmax_i/∂z_j = softmax_i(δ_ij − softmax_j)`; a quick scalar proxy for "how much can this row still move" is the diagonal mass `Σ_i p_i(1 − p_i)`. For this near-one-hot row that sum is about `0.0019` — the gradient has all but vanished. So a wide model, *just by being wide*, can drive the dot-product logits up to scale `√d_k`, saturate the softmax into a near-one-hot, near-zero-gradient corner, and choke off learning. That gives me a concrete mechanism for the width-1024 collapse I was worried about: the additive score puts the pair through a learned hidden layer with a bounded `tanh`, while the bare dot product sums `d_k` raw products and inflates with `√d_k`. The cliff isn't mysterious; it tracks this variance growth.

And the fix drops right out of the same arithmetic, because I know *exactly* by how much the scores are too big. The standard deviation is `√d_k`, and I'd like the per-logit scale back around 1 regardless of width — that's the regime where the softmax has healthy gradients. So divide the scores by their standard deviation: divide `q · k` by `√d_k`. Then the scaled score has variance `d_k / (√d_k)^2 = d_k/d_k = 1`, independent of `d_k`. Re-running the same row through the fix: dividing `[8.0, −2.0, 1.0, −4.0]` by `√64 = 8` gives `[1.0, −0.25, 0.125, −0.5]`, and softmax of *that* is about `[0.519, 0.149, 0.216, 0.116]` — a real spread of weights, no longer collapsed onto one holder. Its gradient diagonal mass is about `0.648`, versus `0.0019` before scaling — the same row goes from essentially frozen to fully responsive, a couple of orders of magnitude more gradient, just from the one division. So scaling by `√d_k` is not cosmetic; on this row it's the difference between a softmax that can learn and one that can't. Note the factor has to be `√d_k`, not `d_k` — I'm normalizing the *standard deviation* of the logit to 1, which is the square root of the variance, so I divide by the square root. If I'd divided by `d_k` I'd have crushed the variance to `1/d_k`, over-shrinking the logits and washing out the very differences I want the softmax to sharpen. `√d_k` is the unique scale that lands the std at 1. So the compatibility I'll use is `(q · k)/√d_k`, and the full read is

  Attention(Q, K, V) = softmax( Q Kᵀ / √d_k ) V.

The additive MLP I started from avoids this raw width-growing sum, but it pays for that robustness with a per-pair network I can't afford to run everywhere. The scaled dot product keeps the single-GEMM cost *and* recovers the numerical robustness, by spending one cheap scalar division. That's the trade, and after watching the two softmax rows side by side it's a trade I'm willing to make.

Tally the actual cost, since the single-matmul argument only pays off if the total works out. `Q Kᵀ`: `(n × d_k) × (d_k × n)` is `O(n^2 d_k)`. Softmax over each of the `n` rows of length `n`: `O(n^2)`. Then `(scores) V`: `(n × n) × (n × d_v)` is `O(n^2 d_v)`. So `O(n^2 d)` overall, two dense matmuls and a softmax — and the sequential depth is `O(1)` (the matmuls and softmax are all parallel across positions) and every reader sees every holder directly, so the path length between any two positions is `O(1)`. That is the row of the table I set as the target back at the start: connect all positions, constant sequential ops, constant path.

Now the autoregressive case, because the layer has to work inside a decoder that generates left to right. When I'm producing position `i`, it must not be allowed to read positions `j > i` — those are the future, they don't exist yet at generation time, and letting training peek at them would be cheating. So in the score matrix `S = Q Kᵀ/√d_k`, every entry `S_ij` with `j > i` is illegal and has to contribute exactly zero weight after the softmax. How do I kill a softmax weight cleanly? The naive move is to compute the softmax over all logits and then zero out the illegal entries — check whether that actually leaves a valid distribution. Take reader `i` with three holders and scaled logits `[0.5, 1.0, 2.0]`, where holder `j=2` is the future and must be forbidden. Softmax over all three is about `[0.140, 0.231, 0.628]`. If I now zero the illegal third entry I get `[0.140, 0.231, 0]`, which sums to `0.371` — not 1. The denominator still summed the illegal `exp`, so the surviving legal weights no longer form a convex combination; I'd be silently down-weighting every legal holder by whatever mass the future grabbed. That's wrong, and the failure is quiet enough that it would just quietly hurt the model rather than crash. So zero-after-softmax is out.

The clean way is to make the illegal logits `−∞` *before* the softmax. Then `exp(−∞) = 0`, those entries drop out of both numerator and denominator, and the softmax renormalizes over exactly the legal holders. Confirm on the same row: set the third logit to `−∞`, so the row is `[0.5, 1.0, −∞]`, and softmax gives about `[0.378, 0.622, 0]`, which sums to 1. And it should equal the softmax of just the two legal logits `[0.5, 1.0]` on their own — running that, I get `[0.378, 0.622]`, the same two numbers. So masking with `−∞` before the softmax is identical to having never offered the future as a holder at all; the legal weights are a proper distribution on their own. Concretely: build a mask that is the lower triangle (including the diagonal) — reader `i` may see holder `j` iff `j ≤ i` — and add a bias matrix that is `0` on legal entries and `−∞` on the upper triangle, then softmax. So the masked read is `softmax(Q Kᵀ/√d_k + M) V` with `M_ij = 0` for `j ≤ i` and `−∞` otherwise. The future is masked out inside the same operation, no extra plumbing around it. For an encoder with no causality there's simply no mask and every position reads every position — the "dense," fully-connected read, every one of the `n^2` pairs live.

One implementation reflex before I write code. Writing this as `scores = Q @ K.transpose / sqrt(d_k); p = softmax(scores); out = p @ V` is correct and is the transparent way to see the math, but it materializes the full `n × n` score matrix in memory — and for long sequences that matrix is the dominant memory cost and the dominant memory traffic. The same function can also be exposed as one library primitive that receives the three tensors, the scale, and either a causal flag or an attention bias, then computes `softmax(QKᵀ·scale + mask)V`. On backends with a fused kernel, that primitive can avoid writing the whole score matrix out to slow memory, but the mathematical contract is unchanged: same scaling, same masking, same softmax, same two contractions. So the explicit three-step version is the reference definition, and the fused form is the faithful production route for the identical computation. The fused call takes the queries, keys, values, either an attention mask or an `is_causal` flag (which builds the lower-triangular `−∞` bias for me), and the `scale`, which I set to `1/√d_k` — the exact scaling I just derived — defaulting to `1/√(head dim)` when I don't pass it.

So let me fill the empty slot in the mixing layer — the cross-position read — with what I've landed on. The transparent reference version and the fused production form compute the same scaled-dot-product blend:

```python
import math
import torch
import torch.nn.functional as F


def mix_positions(q, k, v, mask=None, scale=None):
    """Scaled dot-product attention: softmax(Q Kᵀ / √d_k + mask) V.

    q, k, v: (B, H, N, D). mask: optional boolean (True = allowed) or additive bias.
    Reference (explicit) form — materializes the N×N score matrix; clear, not memory-lean.
    """
    D = q.shape[-1]
    scale = scale if scale is not None else 1.0 / math.sqrt(D)   # divide logits by √d_k: std→1
    scores = torch.matmul(q, k.transpose(-2, -1)) * scale        # Q Kᵀ / √d_k, one GEMM, (.,N,N)
    if mask is not None:                                         # forbid illegal (reader, holder) pairs
        if mask.dtype == torch.bool:
            scores = scores.masked_fill(mask.logical_not(), float("-inf"))  # −∞ BEFORE softmax
        else:
            scores = scores + mask                              # additive bias, e.g. 0 on legal and −∞ illegal
    p = torch.softmax(scores, dim=-1)                           # nonneg, rows sum to 1 (over legal holders)
    return torch.matmul(p, v)                                   # blend values: second GEMM, back to (.,N,D)


def mix_positions_fused(q, k, v, attn_mask=None, is_causal=False, scale=None):
    """Same function through PyTorch SDPA. is_causal builds the lower-triangular
    −∞ bias internally; scale defaults to 1/√(head dim)."""
    D = q.shape[-1]
    scale = scale if scale is not None else 1.0 / math.sqrt(D)
    return F.scaled_dot_product_attention(
        q, k, v, attn_mask=attn_mask, dropout_p=0.0, is_causal=is_causal, scale=scale,
    )
```
