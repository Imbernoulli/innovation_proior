I start with the training bottleneck, not with the architecture. I want a large predictor because the environment is a stream of images and actions, and a recurrent net can carry the temporal context that a single frame does not contain. But I do not want to train that whole recurrent net from sparse reward. The reward signal answers only whether a whole behavior worked, while the prediction problem gives me a dense supervised target at every time step. So I separate what can be learned cheaply from prediction from what must be learned from reward. The large part should learn to compress and predict experience; the reward-optimized part should be as small as I can make it.

The image stream gives me the first split. Predicting directly in pixel space is too expensive and too tied to irrelevant visual detail, so I first need a compact latent state. A plain autoencoder would compress frames, but its latent space can be jagged and unconstrained. Since the temporal model will later sample latent vectors and feed them back as if they were observations, I need those off-training-point latent samples to remain meaningful. A variational autoencoder gives me that. It maps each frame to a Gaussian posterior, samples \(z=\mu+\exp(\log\sigma^2/2)\epsilon\), reconstructs the frame, and pays a KL penalty toward \(N(0,I)\). The reconstruction term preserves the visible state; the KL term keeps the code smooth and capacity-limited enough that generated latent points are not immediately nonsensical.

Now I need the temporal model. If I let an RNN predict a single next latent vector, it will average distinct futures. That is exactly wrong for the survival task: "a projectile is launched" and "no projectile is launched" are separate futures, not two noisy samples around one mean. I need a distribution, and more specifically I need a multimodal one. A mixture density output layer gives the recurrent model that ability. Given the current latent, the action, and the recurrent state, it outputs mixture weights, means, and log standard deviations for the next latent. Training is then just negative log likelihood of the observed next latent under the predicted mixture.

I have to be precise about what this likelihood means. I do not need one joint mixture component for the whole latent vector; a simpler factorization can give each scalar coordinate its own \(K\)-component one-dimensional Gaussian mixture and average the resulting scalar losses across coordinates, time, and batch. For a scalar target \(y\), the loss is
\[
-\log \sum_{k=1}^K \pi_k \frac{1}{\sqrt{2\pi}\sigma_k}
\exp\left(-\frac12\left(\frac{y-\mu_k}{\sigma_k}\right)^2\right),
\]
with \(\log \pi\) normalized by log-sum-exp and \(\sigma_k=\exp(\mathrm{logstd}_k)\). Averaging this scalar loss over all latent coordinates is not the same as a full diagonal Gaussian mixture with one shared component index for the whole vector. It is a simpler factorized mixture, and the sampling code matches it by choosing a mixture component separately for each latent coordinate.

The controller can now stay tiny. It does not need to own the visual and temporal capacity; it can read the current latent and the recurrent model's state. The latent tells it what the present frame contains, while the recurrent state summarizes the predictive context that the density model uses for the next latent. A single linear map is enough to test the hypothesis: map those features to an action and bound the action to the environment's range. The car case uses \(z\) and the LSTM output \(h\), so the input is \(32+256=288\) and the controller has \(288 \cdot 3+3=867\) parameters. The survival case exposes \(z\), the LSTM cell state \(c\), and the LSTM output \(h\), so the input is \(64+512+512=1088\), and the one-output linear map has 1088 weights and no bias.

Because the controller is this small, CMA-ES is a reasonable optimizer. It only needs a scalar return from each rollout, and each candidate policy can be evaluated independently. That keeps reward credit assignment away from the large VAE and recurrent model. The large models get dense unsupervised losses; the controller gets black-box selection pressure through average rollout return.

The same recurrent density model can also be used as a latent environment, but only if I handle its failure mode. In the survival task, the model predicts both the next latent and a restart or done signal. A virtual environment can sample the next latent from the mixture, feed that latent back into the RNN, return a reward of one for each step survived, and terminate when the restart logit crosses zero or the maximum episode length is reached. Training a controller inside this virtual environment is cheap because there is no pixel renderer and no game engine.

But that also creates the model-exploitation problem. A controller optimized inside an approximate simulator will search the simulator's weaknesses. If the recurrent model is too deterministic, the policy may discover hidden-state trajectories where projectiles disappear or never launch. This is not a real survival skill; it is a simulator exploit. The mixture model gives me a practical control knob. Sampling divides the mixture logits by a temperature and multiplies Gaussian noise by \(\sqrt{\tau}\). Low temperature collapses toward deterministic modes and makes the dream easy to exploit. Higher temperature makes generated futures noisier, which forces the controller to survive a harder and less predictable simulation. Too much temperature destroys useful structure, so the knob trades realism against exploitability.

This gives the method in its final shape. Learn a frame VAE from random-policy observations. Encode recorded frames into latent statistics and resample \(z\) while training the recurrent density model so it does not overfit one latent draw. Train a small linear controller with CMA-ES, either in the real environment using \(z\) and recurrent state as features, or in the learned latent environment for the survival task. The important constraint is that the predictive model is trained from dense observation data, while the reward-trained search space remains small enough for black-box optimization.
