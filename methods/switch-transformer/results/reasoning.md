I start from the scaling pressure rather than from a routing trick. A dense Transformer gives me one reliable feed-forward computation per token, but it also forces every token to use every feed-forward parameter. If I widen the layer, both parameter count and FLOPs rise. If I shard a dense layer because it no longer fits on one accelerator, I add communication to every forward and backward pass. So what I want is a layer with many possible feed-forward parameters but only one small slice active for any particular token.

Mixture-of-experts is the natural first object to inspect. I can put \(N\) feed-forward networks behind a router, compute \(p_i(x)=\exp h_i(x)/\sum_j\exp h_j(x)\), select the top \(k\) experts, and return \(\sum_{i\in\mathcal T}p_i(x)E_i(x)\). That already separates total capacity from per-token expert compute. The problem is the cost of the route. With top-2 routing, every token claims two expert slots, runs two FFNs, and may need to be sent to two devices and gathered back twice. If communication and capacity buffers are the practical bottlenecks, the first question I should settle is whether the second expert is actually buying me anything, or whether I am paying it out of habit.

The old objection to \(k=1\) is that the router would not learn, because the argmax is hard. Let me see if that objection survives contact with the actual gradient. I write the top-1 output as \(y=p_e(x)E_e(x)\), where \(e=\arg\max_i p_i(x)\). The identity of \(e\) is not differentiable, but the selected probability \(p_e(x)\) is still a smooth softmax value multiplying the expert output. Holding the chosen expert fixed for the local derivative, if a downstream loss has gradient \(g=\partial\ell/\partial y\), I should get
\[
  \frac{\partial \ell}{\partial h_j}
  = \langle g,E_e(x)\rangle\,p_e(x)\,(\mathbf 1\{j=e\}-p_j(x)).
\]
I do not fully trust a softmax Jacobian I wrote down by hand, so I check it. With \(N=4\), a random logit vector \(h\), a random fixed expert output \(E_e\), and a random downstream gradient \(g\), I form \(y=p_e(x)E_e(x)\), backpropagate \(\langle g,y\rangle\), and compare \(\partial\ell/\partial h\) against the closed form above term by term. The two agree to \(7\times10^{-9}\). So a top-1 router does carry a real gradient through the selected gate; what it lacks is counterfactual feedback about the experts it did not choose, not differentiability itself. That reframes the second expert from "needed for learning" to "needed for exploration," and exploration is a problem I might solve more cheaply than by running a whole extra FFN per token. I will hold onto that and see if a lighter exploration mechanism is enough.

Now the hardware constraint bites. The router's choices are dynamic, but the expert input tensors need static sizes. For \(T\) routed tokens and \(N\) experts, the even share is \(T/N\), so I reserve \(C=(T/N)\) times a capacity factor for each expert, with integer rounding and a minimum capacity in implementation. If an expert receives fewer than \(C\) tokens, the empty slots are padding. If it receives more than \(C\), I cannot grow the buffer, so the overflowed tokens must skip this sparse FFN contribution. That is only acceptable because the Transformer block adds the FFN output to the residual stream; a dropped token contributes zero from the sparse FFN and passes through the residual unchanged.

I have to be exact about the slot cases, because an off-by-one error changes which token is dropped, and a token dropped at capacity \(C\) versus \(C+1\) is a silently different layer. The idea is an exclusive cumulative count per expert: the first token routed to an expert gets position 0, the next gets 1, and a token is kept exactly when its position is \(<C\). Let me run it on a deliberately lopsided case to make sure the boundary lands where I think. Take 6 tokens, 2 experts, capacity factor 1.0, so \(C=\lfloor 6\cdot 1.0/2\rfloor=3\), and a routing pattern \([0,0,1,0,0,1]\) that sends four tokens to expert 0 and two to expert 1. Computing `(cumsum(one_hot)-1)*one_hot` summed over experts gives positions \([0,1,0,2,3,1]\), and `position < 3` gives keep \([\text{T},\text{T},\text{T},\text{T},\text{F},\text{T}]\). So expert 0's first three tokens (indices 0,1,3) survive and its fourth (index 4, position 3) is the one dropped, while both of expert 1's tokens stay. That is exactly the behavior I want: the buffer fills in arrival order and the overflow is precisely the token that would have needed slot \(C\). The inclusive-count convention (first token has count 1, keep when count \(\le C\), store slot = count\(-1\)) is the same rule shifted by one in indexing; I will use the exclusive `cumsum(one_hot) - 1` with `position < capacity` form since it matches the switch gate directly.

Dropping is only rare if the router spreads tokens out, so I need a loss that actively discourages collapse onto a few experts. The hard load I care about is
\[
  f_i=\frac{1}{T}\sum_x \mathbf 1\{\arg\max_j p_j(x)=i\},
\]
the actual fraction of tokens sent to expert \(i\). But that is an argmax count and has no useful gradient. The smooth quantity I can move is
\[
  P_i=\frac{1}{T}\sum_x p_i(x),
\]
the average probability mass assigned to expert \(i\). My first instinct is to just penalize the spread of \(P\) — push the soft probabilities toward uniform. But I should be skeptical of that, because \(P\) and \(f\) can disagree: the router can put broad, near-uniform soft mass everywhere while its argmax still piles onto one expert, and it is the argmax that fills the capacity buffer. A penalty on \(P\) alone would not feel the experts that are actually overloaded by hard assignments. So I want a term that reacts to where the hard load \(f\) currently is. Coupling the two as \(\sum_i f_i P_i\), with gradients stopped through \(f\), uses \(f\) as a fixed weighting that points the differentiable \(P\) at exactly the overloaded experts. I scale it by \(N\) and write the per-layer term as \(\alpha N\sum_i f_i P_i\).

I want to confirm this term pushes logits in the direction I claimed, not the opposite, and that the \(N\) scaling does what I think. For one token the load term contributes \(\alpha N\sum_i f_i p_i(x)/T\), and differentiating through the softmax should give
\[
  \frac{\partial L_{aux}}{\partial h_j(x)}
  = \frac{\alpha N}{T}\,p_j(x)\left(f_j-\sum_i f_i p_i(x)\right).
\]
Rather than trust the sign by inspection, I build a small case: \(N=3\), \(T=5\) random logits, take \(f\) from the argmax counts with gradient detached, \(P\) from the mean softmax, and backpropagate \(\alpha N\sum_i f_i P_i\). The autograd logit gradient matches the closed form above to \(7\times10^{-9}\), so the formula is right. Reading the sign off it: the bracket is positive for any expert whose hard load \(f_j\) exceeds the probability-weighted average \(\sum_i f_i p_i(x)\), so under gradient descent its logit is pushed down; for underloaded experts the bracket is negative and the logit is pushed up. In the same run \(f=[0.4,0.4,0.2]\), and the mean logit gradient per expert is \([+0.0068,+0.0063,-0.0131]\) — the two overloaded experts get pushed down, the underloaded one gets pushed up. That is the load-balancing pressure I wanted, confirmed rather than assumed.

I also want the loss to behave well as the expert count changes, otherwise tuning \(\alpha\) at one \(N\) would not transfer. At the uniform target \(f_i=P_i=1/N\) the unweighted dot product \(\sum_i f_i P_i\) is \(N\cdot(1/N)^2=1/N\), so multiplying by \(N\) gives a uniform-routing loss value of exactly 1, i.e. \(L_{aux}=\alpha\). I check this numerically for \(N\in\{2,8,64,128\}\) and get \(1.000000\) in every case, so the scale really is \(N\)-independent and \(\alpha\) means the same thing across model sizes. The same run confirms the bookkeeping identity I will need when reading reference code: `mean(f * P) * N**2` returns the same number as \(N\sum_i f_i P_i\), since the mean over experts already divides the sum by \(N\).

Precision is the next failure mode I expect, and it is concentrated in a specific place. The fragile operation is not the expert matrix multiplication, which is robust in low precision; it is the router softmax and the hard decisions downstream of it. If I run the whole sparse model in float32 I make the all-to-all routing traffic more expensive. If I keep the router softmax in bfloat16, the exponentials and small mantissa can perturb the probabilities enough to flip borderline argmax decisions and destabilize which tokens fill which buffers — and unlike a dense layer, a perturbed routing decision changes the computation graph, not just a number. The compromise that isolates the sensitive part is local selective precision: cast router inputs to float32, compute logits and softmax in float32, choose the expert and form the combine weight there, then cast the combine and dispatch tensors back to the model dtype before they cross devices. The expensive float32 is then confined to a tiny router projection and never to the all-to-all payload.

There is a related lever at initialization. Large early router logits make the softmax sharp, and a sharp softmax is exactly where small numerical perturbations flip the argmax. The standard truncated-normal init uses standard deviation \(\sqrt{s/n}\) with \(n\) the fan-in; reducing the default scale \(s\) by a factor of ten shrinks early activations and router logits, which keeps the early softmax soft while routing is still random and untrained. I would frame this as a stability condition I had to add to make hard top-1 switching trainable at scale, not as part of the routing math itself, and I would want to confirm empirically that without it early training is where divergence shows up.

That leaves the exploration gap I flagged earlier: with top-1, once an expert wins a token the model never observes what the runner-up would have done. I could recover exploration by sampling the expert from the softmax instead of taking the argmax, but that injects high-variance noise into every routing decision and into the capacity buffers, which fights the stability I just bought. A lighter perturbation is to jitter the router input during training, multiplying hidden states by uniform noise near 1 before the router projection. That can flip the borderline argmax decisions — the ones where two experts are nearly tied, which is where exploration actually helps — while leaving confident assignments essentially untouched, and it costs almost nothing. This is the cheap exploration mechanism I was hoping would substitute for the second expert.

Before fixing the design I weigh two tempting extensions and decide against both. First, when a token overflows its first-choice expert I could reroute it to its next-choice expert instead of dropping it, which would reduce drops. But that sends the token to an expert the router ranked lower and complicates the dispatch path, whereas the residual skip is simpler and preserves the token's representation rather than forcing a mismatched expert computation; given that the residual already carries dropped tokens forward unchanged, the extra machinery does not earn its complexity in a first stable design. Second, I could make the attention projections sparse too, but the feed-forward sublayer is the high-parameter, token-independent part of the block, and is the safer single place to insert sparsity first.

Putting the pieces together, the layer that survives all of these checks is deliberately narrow in its moving parts. A float32 router computes softmax probabilities, picks the top-1 expert, keeps the selected gate as the differentiable scalar (whose gradient I verified flows), assigns a 0-based slot within the expert capacity (whose boundary I traced), and zeros the gate for overflow. Experts are ordinary feed-forward networks. The sparse layer returns only the expert update, so dropped tokens stay zero in that update and survive through the block residual. The training loss is the language-modeling loss plus \(\alpha N\sum_i f_i P_i\) from each sparse layer, whose gradient sign and \(N\)-independent scale I checked. The end result is not a general top-k MoE; it is the leanest expert route I could justify — one token, one expert, one balancing loss, and float32 only where the router needs it.
