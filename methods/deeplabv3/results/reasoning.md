I start with the part that is easiest to state but hardest to ignore: a classification network is good at recognizing objects partly because it throws spatial resolution away. Stride-2 convolutions and pooling layers keep compressing the image until the last feature map is about one thirty-second of the input size. That is a good trade for image-level classification, but it is the wrong endpoint for a pixel-level problem. If I attach a classifier there and upsample, I am asking the model to recover detail after the network has already discarded it.

I still want the pretrained network. The late filters know useful semantics, and throwing them away would make the segmentation problem harder. So I need a way to change where those filters are evaluated, not what they are. The stride is the culprit: if I remove a late stride-2 operation, the next feature map becomes twice as dense. But now the later filters see a smaller span of the original image, because the downsampling step they expected is gone.

The compensating move is to spread the filter taps out. For output location **i**, I compute `y[**i**] = sum_**k** x[**i** + r * **k**] * w[**k**]`. The sign and indexing matter: this is the cross-correlation convention used by convolutional libraries, with a plus offset so each output reads input positions ahead of it. The rate `r` is the input sampling stride inside the filter. If `r = 1`, nothing changes; if `r = 2`, adjacent learned weights read every other input position. So when I remove one stride-2 downsampling, I set the following convolutions to rate 2. When I remove two such downsamplings, the corresponding later stages use rates 2 and 4. The parameter count does not change, and the output grid stays dense.

That gives me a controllable `output_stride`. The original classification backbone is at 32. Removing the last downsampling and using rate 2 gives 16. Removing the last two and using rates 2 and 4 gives 8. This solves the resolution problem without learning a separate decoder first.

Now I have to handle scale. One object might be tiny, another might fill the image. A single field of view cannot be ideal for both. I could run an image pyramid, but then every scale pays for a full backbone pass. I could add more residual blocks in cascade, but if those blocks keep striding the feature map collapses again. If I keep them atrous instead, I can grow context while holding output stride fixed. The multi-grid variant is a way to vary the unit dilation rates inside the later ResNet stage: a base rate multiplied by `(1, 2, 4)` gives, for example, `(2, 4, 8)` at output stride 16.

The parallel version is more direct. I can feed the same dense feature map to several branches: one local `1x1` branch and several `3x3` atrous branches with different rates. Each branch has the same spatial size but a different effective field of view. Concatenating them gives one feature map that contains several context scales in a single forward pass.

Before I trust "just use a large rate" as a global-context mechanism, I need to check the boundary case. For a `3x3` atrous filter on an `N x N` feature map with same padding, the offsets are `{-r, 0, r}` in each dimension. At a given row, the center tap is always valid, the upper tap is valid only for rows `i >= r`, and the lower tap is valid only for rows `i + r < N`. Across one dimension, the total valid one-dimensional taps over all `N` positions is `N + 2 * max(N - r, 0)`. In two dimensions, the average normalized valid tap count is therefore `((N + 2 * max(N - r, 0)) / (3N))^2`.

That formula catches every case I care about. At `r = 1`, almost every interior location has all nine taps, with only the border losing taps. As `r` grows, fewer spatial positions have valid outer taps. Once `r >= N`, the normalized count is exactly `1/9`, because only the center tap can ever hit the feature map. So a very large-rate `3x3` branch is not a true global branch. It degenerates into a pointwise `1x1` operation over most or all positions.

The fix is to add global context explicitly. I pool the final feature map over its spatial extent, pass that image-level vector through a `1x1` convolution with 256 filters and batch normalization, then bilinearly upsample it back to the feature-map size. Now the pyramid has a branch that really sees the whole image, not just a sparse local stencil whose outer weights are mostly padding.

Putting the head together, I use five branches at output stride 16: an image-level pooling branch, a `1x1` branch, and three `3x3` atrous branches at rates `(6, 12, 18)`. Each branch produces 256 channels with batch normalization and ReLU. I concatenate the branches, project the concatenation with another `1x1` convolution to 256 channels, apply batch normalization, ReLU, and a light dropout, and then use a final `1x1` convolution to produce class logits. If I evaluate at output stride 8, I double the atrous rates to `(12, 24, 36)` so the rates match the denser feature grid.

I do not add the DenseCRF from the earlier systems, because the goal is now a single trainable segmentation network. I also do not add a decoder or an extra post-ASPP `3x3` refinement layer in the core head, because those belong to later variants or other library adaptations, not to this method's ASPP formulation. The final shape is: dilated pretrained backbone, multi-rate ASPP with image-level context and trained batch normalization, final `1x1` logits, and bilinear upsampling to the input grid.
