I start with the part that is easiest to state but hardest to ignore: a classification network is good at recognizing objects partly because it throws spatial resolution away. Stride-2 convolutions and pooling layers keep compressing the image until the last feature map is about one thirty-second of the input size. That is a good trade for image-level classification, but it is the wrong endpoint for a pixel-level problem. If I attach a classifier there and upsample, I am asking the model to recover detail after the network has already discarded it.

I still want the pretrained network. The late filters know useful semantics, and throwing them away would make the segmentation problem harder. So I need a way to change where those filters are evaluated, not what they are. The stride is the culprit: if I remove a late stride-2 operation, the next feature map becomes twice as dense. But now the later filters see a smaller span of the original image, because the downsampling step they expected is gone.

The compensating move is to spread the filter taps out. For output location **i**, I compute `y[**i**] = sum_**k** x[**i** + r * **k**] * w[**k**]`. The sign and indexing matter: this is the cross-correlation convention used by convolutional libraries, with a plus offset so each output reads input positions ahead of it. The rate `r` is the input sampling stride inside the filter. If `r = 1`, nothing changes; if `r = 2`, adjacent learned weights read every other input position. So when I remove one stride-2 downsampling, I set the following convolutions to rate 2. When I remove two such downsamplings, the corresponding later stages use rates 2 and 4. The parameter count does not change, and the output grid stays dense.

Let me make sure the bookkeeping actually comes out where I want. Define `output_stride` as input resolution divided by feature-map resolution. The backbone starts at 32. Remove the last downsampling and the feature map doubles in each dimension, so `output_stride` halves to 16; the convolutions that used to follow a stride-2 step now run at rate 2 to keep their original input footprint. Remove the last two downsamplings and it halves again to 8, with the two affected stages at rates 2 and 4. So I have a knob — one removed stride for 16, two for 8 — and I have not had to learn a separate decoder to get there. Good; that is the resolution side handled.

Now I have to handle scale. One object might be tiny, another might fill the image. A single field of view cannot be ideal for both. I could run an image pyramid, but then every scale pays for a full backbone pass. I could add more residual blocks in cascade, but if those blocks keep striding the feature map collapses again. If I keep them atrous instead, I can grow context while holding output stride fixed. There is a multi-grid idea here: vary the unit dilation rates inside the later ResNet stage by a small pattern. If the base rate for the stage is 2 and the per-block multipliers are `(1, 2, 4)`, the three blocks run at `2*(1,2,4) = (2, 4, 8)`. So the stage spans several fields of view internally without changing its output stride.

The parallel arrangement is more direct than stacking, and it does not depend on getting a cascade's strides right. I can feed the same dense feature map to several branches at once: one local `1x1` branch and several `3x3` atrous branches with different rates. Each branch keeps the same spatial size but reads a different effective field of view. Concatenating them gives one feature map carrying several context scales from a single forward pass.

The obvious temptation now is to grab global context for free by just cranking one branch's rate way up — make `r` large enough that a `3x3` filter reaches across the whole feature map. Before I trust that, I want to know what a `3x3` atrous filter actually reads as `r` grows, because with same padding the outer taps start landing outside the map. Take a `3x3` filter on an `N x N` feature map; the offsets are `{-r, 0, r}` in each dimension. Along one dimension, the center tap (offset 0) is valid at every one of the N rows. The upper tap (offset `-r`) is valid only when `i - r >= 0`, i.e. for `i` from `r` to `N-1`, which is `max(N - r, 0)` rows. The lower tap (offset `+r`) is valid only when `i + r <= N - 1`, also `max(N - r, 0)` rows. So the total valid one-dimensional taps summed over all N positions is `N + 2 * max(N - r, 0)`, and the per-position average normalized by 3 is `(N + 2 * max(N - r, 0)) / (3N)`. Taps in the two dimensions are independent, so the 2D normalized valid-tap fraction is the square,

```text
((N + 2 * max(N - r, 0)) / (3N))^2.
```

I do not want to trust an algebra step like that on faith, so I check it directly against a brute-force count of in-bounds taps. On a small `N = 5` grid: at `r = 1` the formula gives `((5 + 2*4)/15)^2 = (13/15)^2 = 0.751`, which times 9 is `6.76` taps on average — matching a direct count that loses taps only at the border. At `r = 2` it gives `((5 + 2*3)/15)^2 = (11/15)^2 = 0.538`, i.e. `4.84` taps, again matching the explicit count. And at `r = 5`, `max(N - r, 0) = 0`, so the fraction is `(5/15)^2 = 1/9` and the average is exactly `1` tap — only the center can ever land inside. The same equality holds for any `r >= N`. So the tempting "just use an enormous rate" branch does not see the whole image; it degenerates into a pointwise `1x1` operation, weighting one location and discarding the other eight taps to padding. That kills the free-global-context idea: a single overgrown rate gives me a `1x1`, not a global view.

That same computation also tells me how large a rate I can usefully pick before the branch starts hollowing out, which I need to set the actual ASPP rates. At `output_stride = 16` with a typical PASCAL crop of 513, the feature map is `N = ceil(513/16) = 33`, and the branch collapses to `1/9` only once `r >= 33`. So rates comfortably below that stay informative: the formula gives `0.772` of taps valid at `r = 6`, `0.574` at `r = 12`, and `0.405` at `r = 18` — about `6.95`, `5.17`, and `3.64` of the 9 taps respectively. Rates `(6, 12, 18)` therefore give me three genuinely different, still-substantial fields of view rather than three branches sliding toward a `1x1`. If I move to `output_stride = 8`, the map doubles to `N = 65`, so I double the rates to `(12, 24, 36)`; those land at the same valid-tap fractions (`0.769`, `0.568`, `0.398`) on the denser grid, which is why the rates have to scale with the output stride rather than stay fixed.

The thing the atrous branches cannot supply — confirmed by that `1/9` limit — is true image-level context, so I add it explicitly as its own branch. I pool the final feature map over its whole spatial extent, pass that single image-level vector through a `1x1` convolution with 256 filters and batch normalization, then bilinearly upsample it back to the feature-map size. That branch genuinely sees everything, by averaging rather than by a sparse stencil whose outer weights would be mostly padding.

Putting the head together, I use five branches at output stride 16: the image-level pooling branch, a `1x1` branch, and three `3x3` atrous branches at rates `(6, 12, 18)`. Each branch produces 256 channels with batch normalization and ReLU. I concatenate the branches, project the concatenation with another `1x1` convolution to 256 channels, apply batch normalization, ReLU, and a light dropout, and then use a final `1x1` convolution to produce class logits. At output stride 8 I use the doubled rates `(12, 24, 36)` for the reason just computed.

I do not add the DenseCRF from the earlier systems, because the goal is now a single trainable segmentation network. I also do not add a decoder or an extra post-ASPP `3x3` refinement layer in the core head, because those belong to later variants or other library adaptations, not to this method's ASPP formulation. The final shape is: dilated pretrained backbone, multi-rate ASPP with image-level context and trained batch normalization, final `1x1` logits, and bilinear upsampling to the input grid.
