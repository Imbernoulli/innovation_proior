Let me start from what actually hurts. I have a team of `n` agents in a partially observable cooperative task — a DEC-POMDP with one shared reward. Each agent acts on its own local observation `o_i` and they all get the same `R(s, A)`. I want to train them with policy gradients, because on-policy PG is conceptually clean and I already trust PPO in single-agent settings. But everyone around me believes on-policy methods are sample-inefficient here, and the methods that win the cooperative benchmarks are off-policy: value-decomposition Q-learning and centralized-critic actor-critics. So if I'm going to make a plain PG method competitive, the win has to come from somewhere specific, not from a clever new objective. Let me figure out where.

The variance wall is the obvious enemy. The policy gradient is `E[Σ_t ∇log π(a^t|o^t) G_t]`, and `G_t` is a Monte-Carlo return with brutal variance. The standard fix is a baseline: subtract something that doesn't depend on the action, replace the return with an advantage. The identity I lean on is `E[∇log π(a|s) · b(s)] = 0` for any `b` that depends only on the state, so subtracting a baseline leaves the gradient unbiased and only changes its variance. In single-agent RL the baseline of choice is the value function `V(s)`, and I fold it into GAE: with `δ_t = r_t + γ V(s_{t+1}) − V(s_t)`, the advantage is `Â_t = Σ_{l≥0} (γλ)^l δ_{t+l}`, `λ` trading bias for variance. Every term of that sum is an error of `V`. So the variance of my gradient is, term by term, controlled by how good `V` is. That's the lever. If I want a PG method to be competitive here, the value function — the critic — is what I have to get right, and "get right" means get its *variance contribution* down.

Now the subtlety that has no single-agent analogue. My actor must only look at `o_i`, because at execution time that's all it has. But the critic is *only used during training*. I'm training in a simulator. The baseline never has to be deployed — it's discarded after training. So nothing forces the critic to restrict itself to `o_i`. The baseline-correctness identity said the baseline can depend on anything independent of the action given the state; it never said it had to depend only on what the actor sees. This is the crack I want to pry open: at training time I have the global state `s` and every agent's observation, and the critic is free to use all of it. The actor stays decentralized; the critic goes centralized. That's the whole CTDE idea — the partial-observability constraint binds at execution, not at training — and it's exactly what lets a *training-only* baseline see more than the policy.

So the question sharpens to: what does the centralized critic condition on, and how is it built? Let me look hard at how the prior centralized methods answer that, because I suspect they over-answer it.

MADDPG gives each agent a centralized *action-value* critic `Q_i(x, a_1,…,a_n)` — global info plus the joint action of all agents. The reason it conditions on the joint action is real: from agent `i`'s viewpoint the environment is non-stationary because the others are learning, but if you *give* the critic everyone's actions, then conditioned on the joint action the world looks stationary again, so the critic target stops chasing a moving distribution. That's a genuine insight. But look at the cost. There's a separate critic per agent. And every critic's input carries the joint action, so the input dimension grows with `n` and the critic is tied to every agent's *current policy* — change one policy and every `Q_i` is stale. COMA cleans up the "one critic per agent" part — a single `Q(s, A)` for the team plus a counterfactual baseline that marginalizes agent `i`'s own action with the others held fixed, `A_i = Q(s,A) − Σ_{a'_i} π_i(a'_i|o_i) Q(s, (A_{−i}, a'_i))`, which is a lovely answer to credit assignment. But it's still an *action-value* critic taking the joint action, and now I have to evaluate it once per candidate action of agent `i` to form the baseline. Value decomposition — VDN's `Q_tot = Σ_i Q_i`, QMIX's monotonic mixing of the `Q_i` through a state-conditioned hypernetwork — also conditions on actions (they're `Q`-functions) and pays for decentralizability with a restriction on the representable class: VDN can't represent interactions where one agent's best action depends on another's simultaneous choice, and QMIX's monotonicity, while richer, is still a constraint on the function class. And all of these are off-policy, dragging the whole replay-buffer, target-network apparatus.

Here's the thing that's bugging me. All of them condition the critic on *actions*. Why? In MADDPG the reason was the stationarity argument, and that's about deterministic-policy off-policy learning. In COMA the joint action is there because the *baseline itself* is counterfactual over actions. But for *my* purpose — I just want a baseline to subtract inside GAE, a variance-reduction term — what do I actually need? Stare at the advantage identity again: the baseline only has to depend on the state, not the action. GAE needs `V(s_t)`, a *state* value, to form the residuals `δ_t`. It never needs `Q`. If I use a centralized *state-value* `V(s)` rather than a centralized action-value, I get the variance reduction I came for, and I shed everything that made `Q` painful: no joint-action input, so no scaling with `n` and no coupling of the critic to every policy; no action-space enumeration for a counterfactual; and no monotonicity/additivity constraint, because I'm not factorizing a joint `Q` into per-agent pieces at all — I'm just learning one scalar function of the state. The methods that "win" took on all that machinery, and for a PG baseline most of it is unnecessary. Let me commit to a centralized `V(s)` and see how far the plain single-agent PPO recipe carries if I just feed it global information.

So the design collapses to: keep single-agent PPO essentially unchanged — the clipped surrogate `min(r_t Â_t, clip(r_t, 1−ε, 1+ε) Â_t)` with `r_t = π_θ(a_t|o_t)/π_{θ_old}(a_t|o_t)`, GAE for the advantage, the standard implementation lore — and the *only* thing that distinguishes this from independent local PPO is what the value function `V` looks at. The actor reads `o_i`; the value function reads global information. That is a very small change, which means the input and training details have to be exactly right.

Now, what global information? Naively "the global state" — but there isn't one canonical global state, and the choice turns out to matter enormously. Let me reason through the options, because I have a diagnostic clue I can't ignore: independent local-only PPO is reported to be *surprisingly strong* on hard SMAC maps, sometimes beating centralized PPO that uses the environment's global state. That's backwards. Centralization is supposed to *help* — the critic sees strictly more, so its baseline should be at least as good. If a centralized critic does worse than a purely local one, the centralization must be feeding the critic the wrong thing. The input is the suspect. Let me trace why.

Option one, the environment-provided global state — call it EP. It's a single agent-agnostic vector the simulator hands me: in SMAC, positions, health, shield, weapon-cooldown for all units. Compact, truly global. But it's the *same vector for every agent*, and being agent-agnostic it drops agent-specific local features: the agent's own id, its available movement and attack actions, its relative distances to allies and enemies within its sight. Now picture the value of a situation for agent `i`. It often depends on those very local features — whether `i` can currently attack, where it is relative to the enemy. If the critic never sees them, its `V` is systematically wrong in a way that varies by agent, and the GAE residuals inherit that error as bias and variance. So a critic on EP can genuinely be *worse* than a local critic that at least sees `o_i`. That's the reversal explained: EP throws away the local information that the value depends on. The "centralization hurt" wasn't centralization failing — it was the wrong centralized input.

Option two, concatenate all local observations — call it CL. Stack `(o_1,…,o_n)` into one big vector and feed that to every agent's critic. Now no local feature is missing. But two problems. The dimension grows linearly with `n` and with the size of each `o_i`, so on a 27-agent map the critic input is enormous and value learning gets hard and sample-hungry. And — less obvious but real — concatenating *observations* still isn't the same as the true global state: if there's information no agent observes at all, CL doesn't contain it either, so it can miss genuinely global structure while paying the full dimensionality price. CL fixes EP's local-omission but at a cost, and incompletely.

So I'm caught between EP (compact, global, but misses local) and CL (has local, but high-dimensional and can miss global). Each leaves something on the floor. What I actually want is *both* the global picture and the agent-specific local detail, fed to the critic. The natural move is to stop choosing and just concatenate: for agent `i`, form the critic input as the environment global state `s` together with that agent's own observation `o_i`. Call it the agent-specific global state — agent-specific because each agent gets a *different* critic input (`s` is shared, `o_i` is not). This gives the critic the comprehensive global picture from EP *and* the local features EP was dropping, without blowing up to the full CL concatenation of everybody's observations.

Wait — "agent-specific input" has a second payoff I should pin down, because it interacts with a decision I haven't made yet: do I share critic parameters across agents? The agents are homogeneous in most of these tasks — same observation and action spaces — and parameter sharing (one network for all agents) is known to make learning much more sample-efficient, since every agent's data trains the same weights. But if I share one critic network and feed it the same global state for every agent, it can only output one value, identical for all agents. That's fine if the true team value is genuinely agent-agnostic, but it can't express a value that differs by agent's role or situation. The agent-specific input fixes exactly this: because each agent feeds in *its own* `o_i` (which carries its id and local features), one shared network produces *different* values for different agents. The agent-specific component is what lets a single shared critic emit per-agent values. So the "agent-specific" choice and the "share parameters" choice are the same insight from two sides: I want one network (cheap, sample-efficient) that still distinguishes agents (expressive), and routing each agent's local identity into the input buys me both.

I can push the minimal version of this and see the skeleton clearly. Suppose I don't even have access to a rich `o_i` to append, or I want the leanest agent-specific signal — what is the *smallest* thing that makes a shared network agent-specific? An identity. Append a one-hot agent index to the global state: input for agent `i` is `concat(s, e_i)` where `e_i` is the `i`-th row of the `n×n` identity. The shared network now sees the same `s` but a different one-hot per agent, so it can fit a different value per agent — at minimum capturing each agent's role. This is the agent-specific construction stripped to its identity component, and it's the cleanest thing to actually code first. The full SMAC version replaces (or augments) the one-hot with the real `o_i`, which carries the id implicitly plus all the local features.

There's a wrinkle in the agent-specific-by-concatenation idea I should chase down before I'm happy with it. If I concatenate `s` and `o_i`, and `s` and `o_i` *overlap* — both contain, say, the enemy's health — then the critic input double-counts those features. That doesn't add information; it just inflates the input dimension, which I just argued (against CL) makes value learning harder. In SMAC this overlap is real: the environment state and the local observation share a lot. So the honest version is: take the agent-specific state, then *remove the repeated features* — keep `s`, append only the parts of `o_i` not already in `s`. Same information content as the full concatenation, smaller input. Call it the feature-pruned agent-specific state. It needs me to know which features overlap (a bit of per-environment bookkeeping), but when I have that knowledge it strictly dominates the un-pruned version on input dimensionality at no information cost. So I have a small family — agent-specific (AS) and its feature-pruned form (FP) — and the design principle behind both is the one sentence I'd actually keep: *give the value function both global and agent-specific local features, and don't let it carry redundant dimensions.*

Let me now write the critic concretely, because I want to see the shapes line up with how the learner will call it. The learner hands me a batch with `state` of shape `(B, T, state_dim)` and `obs` of shape `(B, T, n_agents, obs_dim)`, and it expects a per-agent value `(B, T, n_agents, 1)` back — it'll squeeze the trailing 1 later. So inside the critic I take the global state, broadcast it across the agent axis so every agent sees the same `s`, attach the agent-specific component (the one-hot `e_i`, or the pruned `o_i`), concatenate along the last dim, and run a small MLP that ends in a scalar. The network itself is nothing exotic — two hidden layers, width 64, ReLU, a linear head to 1 — the standard PPO MLP. I'm deliberately not inventing an architecture; the whole point is that the architecture is generic and the *input* is the design.

```python
state = batch["state"][:, ts]                                      # (B, T, state_dim)
state = state.unsqueeze(2).expand(-1, -1, n_agents, -1)            # (B, T, n, state_dim)
agent_id = th.eye(n_agents, device=batch.device)
agent_id = agent_id.unsqueeze(0).unsqueeze(0).expand(B, T, -1, -1) # (B, T, n, n)
inputs = th.cat([state, agent_id], dim=-1)                         # (B, T, n, state_dim + n)
x = F.relu(fc1(inputs)); x = F.relu(fc2(x)); q = fc3(x)       # (B, T, n, 1)
```

That's the agent-specific central-V critic in its minimal identity form: shared MLP over `(s, e_i)`. The AS/FP variants replace this lean identity signal with value inputs that include the agent's own local features, pruning features that already appear in `s` when that bookkeeping is available. Good — the critic is built. But I'm not done, because a baseline is only as useful as it is *well-trained*, and a couple of things about training `V` in this setting will wreck it if I'm naive. Let me hit those walls deliberately.

First wall: the value targets drift wildly. The critic regresses to returns, and in some of these tasks the returns span orders of magnitude — in the particle `Spread` task episode returns run from below −200 up toward 0, and over the course of training the realized returns shift as the policy improves. A neural net regressing to targets whose scale moves by orders of magnitude is unstable: the gradients are dominated by the large-magnitude targets and the learning rate that's right early is wrong later. The cure I know from single-agent RL is to normalize the regression target by a running mean and standard deviation — regress `V` to the *normalized* target, and when I need the actual value (for GAE) I *denormalize* the network's output back to the real scale. So the critic learns in a standardized space while the advantage is computed in the true reward space. But there's a trap in naive normalization: every time I update the running mean/std, the normalization map changes, which retroactively changes what the network's *existing* outputs mean — for inputs it had already fit perfectly, the target just moved under it. That's a self-inflicted non-stationarity on top of the one I'm trying to fix. The clean resolution (PopArt, van Hasselt et al.) is to absorb the statistics into the *last linear layer*: whenever the running mean `μ` and std `σ` change from `(μ_old, σ_old)` to `(μ_new, σ_new)`, rescale the output layer's weights and bias by `W ← W σ_old/σ_new`, `b ← (σ_old b + μ_old − μ_new)/σ_new`, so the network's *de-normalized output is preserved exactly* across the statistics update while the *targets* get adaptively rescaled. The running statistics themselves I keep as an EMA with a debiasing term (start the accumulators at zero, divide by the running sum of weights, so the early estimate isn't biased toward zero). So value normalization addresses exactly the scale drift that would otherwise make value regression unstable. I'll fold it into the critic's output head.

Second wall: PPO reuses each batch over several epochs, and in a multi-agent setting that reuse is more dangerous than in single-agent. The reason is the non-stationarity that's intrinsic to MARL: every agent is part of every other agent's environment, so when I update one policy I've changed the world the others were optimizing against. If I take many gradient epochs on the same batch, or split it into many small minibatches, the policies move a lot per data collection, and the value function (and the other agents' implicit assumptions) go stale fast. So the same instinct that motivates PPO's clip — limit how far the policy moves per update — has to be applied *harder* here. Concretely: use few epochs (on the order of 5–15, fewer for harder tasks), avoid splitting into many minibatches (at most two, often one), and keep the clip `ε` small (under 0.2 — smaller `ε` means each update changes the policy less, trading a little speed for a lot of stability). The clip should apply to the *value* as well as the policy: regress `V` but clamp its change from the previous value, `value_pred_clipped = V_old + clamp(V − V_old, −ε, +ε)`, and take the max of the clipped and unclipped value errors, so the critic also can't jump too far in one update. And use more data per gradient rather than more passes over the same data — a larger batch gives a lower-variance gradient, which matters more when the underlying problem is non-stationary. These are the single-agent PPO knobs, retuned because the multi-agent non-stationarity makes "limit the per-update change" the dominant concern. The unifying principle is one sentence: in MARL, anything that lets the policies move too much per data collection destabilizes the moving-target value learning, so dial all of those knobs toward smaller, more conservative updates.

Third wall, and this one is peculiar to environments where agents can die: SMAC. An agent can be killed partway through an episode while its teammates fight on. What does the critic see for a dead agent, and what target should it learn? If I just keep feeding the agent-specific global state, here's the failure: when the agent dies, its agent-specific local features (available actions, relative distances) go to zero, but the agent-agnostic global features stay nonzero, so the critic input lurches into a region it rarely visits — a sharp distribution shift. Dead states are a small fraction of the data (roughly a fifth of timesteps), so these rare, out-of-distribution inputs get fit poorly, the value error there is large, and that error propagates through GAE into the policy gradient. Worse, training on these odd inputs may corrupt the value network's feature representation for the states that matter. The fix that follows from the diagnosis: when an agent is dead, replace its critic input with a *single fixed vector* — a zero vector with the agent's id appended (`0_a`). Now all of an agent's post-death timesteps map to the *same* input, so instead of a smear of badly-fit out-of-distribution states the critic only has to fit one thing: the average post-death value for that agent, which it can do well. Keep the agent id in that vector, because (the same agent-specific logic as before) the post-death value differs by the agent's role, and zeroing the id too would collapse different agents' deaths into one indistinguishable input — which hurts exactly on maps where agents play different roles. This is "death masking." There's a tempting alternative I should knock down: just *skip* value learning for dead agents — treat death as a terminal state, replace the reward at the death step `d` with the accumulated post-death return `R_d = Σ_{t≥d} γ^{t−d} r_t`, and only compute GAE on alive steps. It's theoretically clean (death = terminal, post-death return = terminal reward). But it quietly damages GAE's whole purpose at that step: GAE is an exponentially weighted blend of `k`-step returns precisely to trade bias for variance, and if I cram the entire `(T−d)`-step return into the single timestep `d`, the 1-step return estimate at `d` *becomes* a `(T−d)`-step Monte-Carlo estimate — much higher variance, no value-function truncation, defeating the bias-variance control I adopted GAE for. So the death mask (fix the input to `0_a`) beats both leaving the global state on (distribution shift) and skipping value learning (variance blow-up). I'll mask.

Let me also be honest about what carries the actor side, since the critic only matters through it. The actor is unchanged single-agent PPO on local observations: the clipped surrogate with importance ratio `r = π_θ(a_i|o_i)/π_{θ_old}(a_i|o_i)`, an entropy bonus `σ S[π_θ(o_i)]` to keep exploration alive, GAE advantages `Â` computed with the critic's denormalized `V`, advantage normalization across the batch before the surrogate, orthogonal initialization, Adam. Homogeneous agents share the actor network too. Everything that's load-bearing for single-agent PPO stays load-bearing; the moving part is upstream, in what `V` sees.

So I can write the actor and critic objectives down cleanly. Over valid batch-time-agent positions `(b,t,i)`, the actor maximizes
`L(θ) = (1/|M|) Σ_{(b,t,i)∈M} min(r_{θ,bti} Â_{bti}, clip(r_{θ,bti}, 1−ε, 1+ε) Â_{bti}) + σ (1/|M|) Σ_{(b,t,i)∈M} S[π_θ(o_{bti})]`,
with `r_{θ,bti} = π_θ(a_{bti}|o_{bti})/π_{θ_old}(a_{bti}|o_{bti})`, and the critic minimizes the clipped value loss
`L(φ) = (1/|M|) Σ_{(b,t,i)∈M} max((V_φ(x_{bti}) − R̂_{bti})^2, (V_old(x_{bti}) + clamp(V_φ(x_{bti}) − V_old(x_{bti}), −ε, ε) − R̂_{bti})^2)`,
with `R̂` the discounted reward-to-go. In the PopArt implementation the target is normalized before the value error is formed, while `V` is denormalized when I compute GAE: `δ_t = r_t + γ V(s_{t+1}) − V(s_t)`, `A_t = Σ_{l≥0}(γλ)^l δ_{t+l}`. If the networks are recurrent (GRU), the same sums run over the rollout time axis and I train through time with BPTT on fixed-length chunks of the trajectory.

Now let me pin the constants, because they shouldn't be arbitrary. `γ = 0.99` is the standard discount. `λ = 0.95` for GAE is the usual bias-variance sweet spot — high enough to be near-unbiased, low enough to cut variance. Two hidden layers of width 64 with ReLU is the plain PPO MLP in the on-policy recipe; orthogonal initialization keeps activations well-scaled at init and improves stability. Adam with `ε = 1e-5`, gradient-norm clipping at 10, value loss as a Huber loss with `δ = 10` (robust to the occasional large value error rather than letting it dominate the squared loss). Clip `ε < 0.2`, tuned down on harder tasks for stability. Epochs 5–15, minibatches ≤ 2. These constants are inherited from PPO practice and the multi-agent-specific calls (small clip, few epochs, big batch, value normalization, death mask, agent-specific input) are the ones I derived above from the non-stationarity and partial-observability of the setting.

Let me drop the whole thing into the critic slot the learner expects — a module that takes `(scheme, args)`, declares itself a value head, and on `forward(batch, t)` returns `(B, T, n_agents, 1)`. The lean shared-critic version broadcasts the global state to every agent and appends an agent one-hot; if I ask the harness for individual observations or previous joint actions, the same input builder appends those blocks too. In the on-policy trainer the environment wrapper can preassemble richer AS or FP value inputs as `cent_obs`; the critic then maps that centralized observation to a scalar value.

```python
import torch as th
import torch.nn as nn
import torch.nn.functional as F


class CentralVCritic(nn.Module):
    """Shared central-V critic that returns (B, T, n_agents, 1)."""

    def __init__(self, scheme, args):
        super().__init__()
        self.args = args
        self.n_agents = args.n_agents
        self.n_actions = args.n_actions
        self.output_type = "v"

        input_shape = scheme["state"]["vshape"]
        if args.obs_individual_obs:
            input_shape += scheme["obs"]["vshape"] * self.n_agents
        if args.obs_last_action:
            input_shape += scheme["actions_onehot"]["vshape"][0] * self.n_agents
        input_shape += self.n_agents                 # agent-specific one-hot
        self.fc1 = nn.Linear(input_shape, args.hidden_dim)
        self.fc2 = nn.Linear(args.hidden_dim, args.hidden_dim)
        self.fc3 = nn.Linear(args.hidden_dim, 1)

    def forward(self, batch, t=None):
        bs = batch.batch_size
        max_t = batch.max_seq_length if t is None else 1
        ts = slice(None) if t is None else slice(t, t + 1)

        inputs = []
        state = batch["state"][:, ts].unsqueeze(2).repeat(1, 1, self.n_agents, 1)
        inputs.append(state)                                          # (B, T, n, state_dim)

        if self.args.obs_individual_obs:
            obs = batch["obs"][:, ts].view(bs, max_t, -1)
            inputs.append(obs.unsqueeze(2).repeat(1, 1, self.n_agents, 1))

        if self.args.obs_last_action:
            if t == 0:
                last_actions = th.zeros_like(batch["actions_onehot"][:, 0:1])
            elif isinstance(t, int):
                last_actions = batch["actions_onehot"][:, slice(t - 1, t)]
            else:
                last_actions = th.cat(
                    [th.zeros_like(batch["actions_onehot"][:, 0:1]), batch["actions_onehot"][:, :-1]],
                    dim=1,
                )
            inputs.append(last_actions.view(bs, max_t, 1, -1).repeat(1, 1, self.n_agents, 1))

        agent_id = th.eye(self.n_agents, device=batch.device)
        inputs.append(agent_id.unsqueeze(0).unsqueeze(0).expand(bs, max_t, -1, -1))
        inputs = th.cat(inputs, dim=-1)

        x = F.relu(self.fc1(inputs))
        x = F.relu(self.fc2(x))
        return self.fc3(x)
```

Let me retrace the causal chain so I'm sure it holds together. I wanted a plain on-policy PG method to be competitive in cooperative MARL, and the variance identity told me the value function is the lever, since GAE's variance is term-by-term the error of `V`. The CTDE observation — partial observability binds only at execution — meant the *training-only* critic is free to use the global state the actor can't. Surveying the centralized prior art, they all conditioned the critic on *actions* (centralized `Q`, counterfactual baselines, factorized joint `Q`), paying for it with input that scales with `n`, action enumeration, or a restricted representable class — but a GAE baseline only needs a *state* value, so a centralized `V(s)` buys the variance reduction while shedding all of that. Then the diagnostic that local PPO can beat centralized PPO pointed straight at the critic's *input*: the environment global state omits agent-specific local features, so its value is biased where local detail matters; concatenating all observations restores local info but is high-dimensional and can still miss truly-global structure. Feeding the critic *both* — the global state plus the agent's own observation (agent-specific), with redundant features pruned — resolves the trade-off, and the agent-specific component doubles as the thing that lets one shared (parameter-sharing) network emit per-agent values. Training that critic well then forced three more moves out of the setting itself: value-target normalization that preserves outputs while rescaling targets, because returns span orders of magnitude; conservative PPO knobs (small clip, few epochs, few minibatches, large batch, clipped value loss) because MARL non-stationarity punishes large per-update changes; and death masking — collapse a dead agent's input to a single fixed id-tagged vector — because otherwise rare out-of-distribution dead states wreck the value estimate, while the theoretically clean alternative of skipping them raises GAE's variance. The result is single-agent PPO with one thing changed: a centralized state-value critic over an agent-specific global state, dropped into the learner as a module returning a per-agent value.
