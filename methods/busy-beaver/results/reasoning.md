I want to pin down, as a number, how far a Turing machine of bounded size can go on a blank tape before it must either halt or run forever. The machines of a fixed size are a finite world: with `n` internal states and two tape symbols, a transition table assigns to each of the `2n` pairs (state, scanned symbol) a triple — symbol to write, direction to move, next state — and each entry is drawn from a finite set. So there are finitely many `n`-state machines, and in principle I can list every one of them. Each, started on a blank tape, runs a definite course: it halts after a definite number of steps, or it doesn't.

That makes an obvious quantity. List all `n`-state machines, throw away the ones that never stop, and over the survivors take the maximum number of steps before halting; call it `S(n)`. Or take the maximum number of `1`s left on the tape at halt; call it `Sigma(n)`. Both are well-defined finite numbers, because the survivor set is finite and each survivor contributes a finite count. The whole question is whether this innocent-looking maximum is something I can actually compute.

Let me get my hands on the smallest cases, partly to fix the convention and partly because I don't trust a definition I haven't run. Take the standard setup: `n` working states plus one halting state `H`, the head starts on a blank (all-`0`) tape, and the transition into `H` still writes its symbol and counts as a step.

For `n = 1` there is one working state, state `0`, and two table entries — what to do on a scanned `0` and on a scanned `1`. Each entry chooses a write bit, a direction, and a next state in `{0, H}`. That is a tiny finite catalog, and I can sweep it. Doing so, the longest a halting one-state machine runs is one step, and the most `1`s it can leave is one: `S(1) = 1`, `Sigma(1) = 1`. That is forced — to halt at all on the first step, the machine reads the blank, writes a `1`, and goes to `H`; anything that returns to state `0` either loops or immediately halts having done nothing more.

For `n = 2` the catalog is bigger but still finite: four entries, each with `2 × 2 × 3 = 12` choices, so `12^4 = 20736` machines. I enumerate all of them, simulate each for a generous cap, and over those that halt I take the maxima. The result is `S(2) = 6`, `Sigma(2) = 4`. Let me actually look at a machine that achieves it, because if the maximum is real there should be a concrete witness:

```
state 0, read 0  ->  write 1, move L, goto 1
state 0, read 1  ->  write 1, move R, goto 1
state 1, read 0  ->  write 1, move R, goto 0
state 1, read 1  ->  write 1, move L, goto H
```

Hand-running it from a blank tape: state `0` reads `0`, writes `1`, steps left into state `1`; state `1` reads `0`, writes `1`, steps right into state `0`; and so on. After six transitions it enters `H`, and the tape holds four `1`s. So `S(2) = 6` and `Sigma(2) = 4` are not just numbers I trust — I have the witness in hand. Good: the definition produces sane, checkable values at the bottom, and these match the accepted small entries. The quantity is real.

Now the part that worried me from the start: the word "survivors." To compute `S(n)` I have to identify, among the `n`-state machines, which ones halt. For a machine that does halt, simulation is a certificate — run it and watch it stop. For a machine that never halts, plain simulation gives me nothing that terminates. At `n = 1` and `n = 2` I sidestepped this by simulating with a cap and trusting that nothing real ran past it, but that trust is exactly what I cannot bottle. If I want `S(n)` in general I need to know, for the non-halting machines, that they truly never stop — and there is no uniform bound I'm allowed to assume in advance.

Let me see whether `S` could be computable anyway, by supposing it is and seeing what that buys me. Suppose some algorithm outputs `S(n)` on input `n`. Then here is a decision procedure for halting, restricted to `n`-state machines on a blank tape: given such a machine, compute `S(n)`, simulate the machine for `S(n)` steps, and report "halts" if it has halted by then, "never halts" otherwise. This is correct: every halting `n`-state machine halts in at most `S(n)` steps by the definition of `S(n)` as the maximum over halting machines, so if it hasn't stopped by step `S(n)` it never will. The "never halts" verdict, which simulation alone could never justify, is now justified by the bound.

I should check this doesn't secretly need more than `n` states. Any machine with a fixed-size description can be re-encoded with a known number of states, and "does this fixed machine halt on blank tape" is already the hard core of the halting problem — the blank-tape halting problem is itself undecidable. So a computable `S` would decide it: pick the machine's state count `n`, get `S(n)`, simulate that long, decide. That contradicts undecidability. Therefore no algorithm computes `S`. I notice the same `S(n)`-step simulation also recovers the full halting tape and hence `Sigma(n)`, so an algorithm for `Sigma` would let me reconstruct halting behavior too; `Sigma` is uncomputable by the same route. The asymmetry between halting and non-halting machines, which I felt at `n = 1`, is the whole reason — `S` would erase it, and it can't be erased.

So `S` is finite at every `n` yet computed by no algorithm. That is already strange enough that I want to understand *how fast* it must grow, because an uncomputable function had better outpace the computable ones — otherwise some computable function would dominate it and I could ask why the domination doesn't help.

Here is the pressure. Take any computable function `f`. I'd like to argue `S` eventually exceeds `f`, and the way to argue it is to build a small machine that out-runs `f` and then stops. Concretely: there is a Turing machine that, on a blank tape, writes the input it wants, simulates `f`, and runs for at least `f(k)` steps before halting — say, by computing `f(k)` and then counting down. If that machine uses `c` states for some fixed bookkeeping plus a number of states that grows slowly in `k`, then for the size `m` it occupies, `S(m)` is at least the running time of this machine, which is at least `f(k)`. As `k` increases, `m` increases too, but only mildly, so for infinitely many sizes `S(m)` overtakes `f` evaluated at the corresponding argument. I want to be careful not to overstate the strength of this — the exact "for all but finitely many `n`" form needs the state count of the `f`-running machine bounded carefully against `n`, and I'd want to write that encoding out in full to be sure of the precise quantifier. What I'm confident of from the construction is that no single computable `f` can stay above `S` forever, because I can always engineer a halting machine whose size is modest and whose runtime is `f` of something larger. That is consistent with what the uncomputability proof already told me: if some computable `f` dominated `S` everywhere, I could simulate any `n`-state machine for `f(n) >= S(n)` steps and decide halting with a computable bound, which I've just shown is impossible. The domination failure and the uncomputability are two views of the same wall.

Stepping back, what I've actually built is not "a function that is large." Many computable functions are large. What is particular here is that the largeness is indexed by description size and is forced by undecidability: at each size `n` there is a longest finite run, that number would settle every halting question among machines of that size, and that is precisely why no algorithm can name it. The small values are knowable and I checked two of them by hand. Larger exact values are finite classification problems that get monstrous quickly; partial searches only ever give lower bounds, never the value, because closing a value means certifying that every remaining machine of that size that hasn't halted never will. The honest summary is that this quantity turns the negative halting theorem into a concrete growth curve: a boundary that can be named, ranked, and approached from below, but never drawn in full by any algorithm.
