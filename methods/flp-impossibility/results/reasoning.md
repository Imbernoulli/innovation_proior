There's an algorithm on the table that's supposed to solve asynchronous commit, and my job is to prove it correct — someone proposed it, a way to get a set of data-manager processes to agree, all-commit or all-abort, even when things go wrong. It looks reasonable. I just have to nail down why it always terminates. That should be the easy part.

Let me set up what "agree" even means here, because the commit problem is the thing I actually care about and it's cleaner to think about a stripped-down version. Each process starts with a private bit — call it its vote, 0 or 1, abort or commit. They run around exchanging messages and eventually each process writes down a final answer. I want three properties. They must all write the *same* answer — that's the whole point, a split commit corrupts the database. The answer mustn't be hard-wired — if it always wrote 0 regardless of inputs, that "agreement" would be a cheat, so I'll insist both 0 and 1 are achievable outcomes for some inputs. And every healthy process must eventually write *something* — it has to terminate. Agreement, non-triviality, termination. If I can get all three even when one process dies, I'm done.

The environment is the harsh part, and I have to be honest about it. Fully asynchronous: I make no assumption about how fast any process runs relative to any other, and no assumption about how long a message takes to arrive. Messages are delivered correctly, exactly once — the network itself is reliable — but the delay is unbounded and they can arrive out of order. No shared clock. And one process might just crash: stop dead at some moment and send nothing more. Not lie, not go haywire — just halt. The mildest possible fault, on top of a reliable network. If anything should be survivable, it's that.

So I try to verify the algorithm terminates. And I keep hitting the same wall. There's a point in the run where the algorithm is waiting to hear from some process before it commits. If that process is slow, everybody waits. Fine, slow is temporary, eventually the message comes. But what if that process *crashed*? Then the message never comes — and here's the thing that won't go away: **the waiting processes cannot tell the difference.** A crashed process and a slow process look *identical* from the outside. Silence is silence. The only way to break the wait would be to time out — "I've waited long enough, I'll assume it's dead" — but I have no clock and no bound on delay, so any timeout I pick can be exceeded by a process that's merely slow, and then I'd be treating a *live* process as dead and risk disagreeing with it. I can't time out. So I wait. Forever, if the process really did die at that moment.

I patch it. I add a round so the protocol doesn't depend on that one process at that one instant. And the wall just moves: now there's a *different* point where the protocol waits on a single process before it can safely commit, and the same crash at *that* point stalls it. Every patch relocates the window of vulnerability; none closes it. People say in the folklore that this window is just sloppy engineering. But I've now tried to engineer it away several times and it always reappears, in the same shape, around the same indistinguishability of dead from slow.

Let me stop trying to make the algorithm work and entertain the opposite: maybe *no* deterministic protocol works. Maybe the window is not an accident — maybe it's a law. That feels strong. The synchronous people already solve agreement: Pease, Shostak and Lamport get interactive consistency for n ≥ 3m+1 even against *lying* processes, and the Byzantine Generals result tolerates malice with more than two-thirds loyal. Those work. But stare at *why* they work — they run in rounds, with negligible/bounded delay, and a process *knows* when a round is over. In a round model, silence carries information: if I expected a message this round and got none, the sender is faulty, and I act on that. That's exactly the move I'm forbidden from making. Their agreement is *bought with synchrony.* And the Two Generals can't agree at all, but that's because their channel *loses* messages — an infinite regress of acknowledgements. My channel is reliable; nothing is lost. So the hardness I'm fighting isn't message loss, and it isn't malice. It's the bare asynchrony plus one crash. If that alone defeats determinism, it's a clean and surprising statement: one crash, reliable network, no liars, and still impossible.

To prove an impossibility I need to make the model painfully precise, because the whole thing will turn on tiny structural facts. A process is an automaton: a one-bit input register, an output register that starts blank and can be written *once* with 0 or 1 (write-once — once you've decided you can't un-decide), unbounded internal storage, and a deterministic transition function. Communication is a message buffer holding a *multiset* of (destination, value) pairs that have been sent but not delivered. Two operations: send drops a message into the buffer; receive(p) either deletes some message addressed to p and hands p its value, or — and this is the part that encodes "arbitrary delay" — returns a null marker and leaves the buffer untouched, *even if* a message for p is sitting there. The only fairness I'll demand of the message system: if a process performs receive infinitely often, every message addressed to it is eventually delivered. So the system can stall a message any *finite* number of times, but not forever, against a process that keeps listening.

Now the whole state of the system at an instant: a **configuration** — every process's internal state, plus the buffer contents. The system moves one **step** at a time. A step is one process p doing receive(p), getting either a particular real message or null, and then — deterministically, since the transition function is deterministic — updating its state and sending out a finite set of messages. Because everything's deterministic, the step is completely pinned down by the pair (p, m): *who* moved and *what message event* they received. I'll call that pair an **event** e = (p, m), and write e(C) for the configuration you reach by applying it to C. A **schedule** is a sequence of events applied one after another; the resulting configuration after a finite schedule σ I'll write σ(C). Configurations reachable from an initial one are *accessible*; those are the only ones I'll ever talk about.

Two structural facts jump out, and I should bank them now because I suspect they're load-bearing. First: a process can *always* take a step, because (p, null) is always applicable — receiving nothing is legal. So the system never deadlocks for structural reasons; if it fails to decide, that's not because nobody can move, it's because moving never *commits* to a value. Second, and this one I can feel will be the engine of the whole proof: **steps by different processes commute.** Suppose from C I run schedule σ₁ to reach C₁ and σ₂ to reach C₂, and the *sets of processes* that take steps in σ₁ and in σ₂ are disjoint. Then σ₁ and σ₂ don't interact in the state that matters to their own events. A process in σ₁ never reads a process's local state from σ₂; the message events consumed by σ₁ are the ones that made σ₁ applicable from C, so σ₂ does not remove them, and symmetrically for σ₂. New messages sent by the first schedule may sit in the buffer, but they are not the messages the second schedule was already scheduled to receive. So I can apply σ₂ after σ₁, and σ₁ after σ₂, and both orders perform the same local updates, consume the same old messages, add the same new messages, and land in the *same* configuration C₃. A little commutative diamond. It's immediate from the locality of the model, but it's going to do enormous work, because it says the adversary scheduling the messages can shuffle the order of non-interfering steps freely without the system being able to notice.

Now I need the right invariant for "still undecided," and this is where I have to think, not compute. What does it mean for the system to have *not yet committed*? Not that no process has written its output — that's too coarse. It means: from here, the future could still go either way. So look at a configuration C and collect the set of decision values of all configurations reachable from C — call it V(C). If both 0 and 1 are in V(C), the decision genuinely isn't settled yet: there's a way to reach a 0-decision and a way to reach a 1-decision. Call such a C **bivalent**. If V(C) is a single value, C is **univalent** — 0-valent or 1-valent — the outcome is already sealed even if nobody's written it down yet. (V is never empty: there's always an admissible run from any accessible configuration, and by termination it decides something.) Reachability can only shrink this set: once C is i-valent, every successor is i-valent too, because an opposite-valent or bivalent successor would give C a path to the other value. Bivalence is exactly "the decision is not yet predetermined." That's the invariant. If I can keep the system bivalent forever, it can never have decided — because a decided configuration has a value in it, and once a value is written, agreement forbids the other value ever appearing, so a decided configuration is univalent. Forever-bivalent means forever-undecided. That's my target: an admissible run that stays bivalent at every step.

Two things to prove, then. The system has to *start* undecided — there must be a bivalent initial configuration, or the adversary has nothing to work with. And from any bivalent configuration the adversary must be able to *reach another* bivalent configuration while still making progress. Get both, and I can chain them into an infinite undeciding run. Let me get the starting point first.

Suppose, for contradiction, that *every* initial configuration is univalent. By non-triviality, some initial configuration decides 0 and some decides 1 — both values are reachable outcomes, so there's a 0-valent initial configuration and a 1-valent one. Now, the initial configurations are just the input vectors — each process's starting bit. Say two initial configurations are **adjacent** if they differ in exactly one process's input bit. The input vectors form a hypercube: you can walk from any vector to any other by flipping one bit at a time, and the whole thing is connected. So somewhere along the walk from a 0-valent initial configuration to a 1-valent one, two *adjacent* configurations must straddle the boundary — a 0-valent C₀ and a 1-valent C₁ differing only in the input bit of a single process, call it p.

p is allowed to be the one faulty process, and that turns the indistinguishability-of-dead into a proof tool. Consider an admissible deciding run starting from C₀ in which **p takes no steps at all** — p has crashed before doing anything. It exists because the protocol must tolerate one crash, so even with p dead the others decide; let σ be its schedule, and it decides 0 (since C₀ is 0-valent). Now apply that very same σ to C₁. Is it applicable? Yes — p never moves in σ, so σ never touches p, and C₀ and C₁ are *identical except for p's input bit*, which nothing in σ ever reads because the only process that could read p's input is p, and p is silent. So σ runs on C₁ exactly as it ran on C₀, and the two runs reach configurations that differ *only* in p's untouched internal state. The decision is made by the other processes, who can't tell C₀ from C₁ apart — so they decide the *same* value, 0. But C₁ is supposed to be 1-valent: every reachable configuration from it decides 1. We just reached a 0-decision from C₁. Contradiction. So at least one of C₀, C₁ was bivalent after all, and a bivalent initial configuration exists.

The crash of the single process that *distinguishes* two adjacent inputs makes the two inputs indistinguishable to everyone else, so they can't decide differently — which forces undecidedness at the boundary. From a bivalent configuration, I still need the adversary to step to another bivalent one.

Let me think about what could go wrong. I'm in a bivalent configuration C. I want to apply some particular event e = (p, m) — because to build a *fair* run I'll eventually need to let each process move and each message be received, so I can't just apply whatever I like; I have to be able to apply a *designated* e and still stay bivalent. So fix an event e applicable to C. Claim: among the configurations I can reach by running some schedule out of C whose *last* event is e, at least one is bivalent.

Suppose not. Let me name things carefully. Let 𝒞 be the set of all configurations reachable from C by schedules that *don't* apply e. And let 𝒟 be the set of configurations e(E) for E in 𝒞 — i.e. take any reachable configuration that hasn't done e yet, then do e. (One quick check: e stays applicable everywhere in 𝒞. e = (p, m) is applicable at C, meaning if m is a real message then that message event is in the buffer; along any schedule that never applies e, that event is never consumed — the system can keep delaying it — and if m is null it's always applicable. So e applies to every E in 𝒞.) My assumption-for-contradiction is that **every configuration in 𝒟 is univalent.**

First, 𝒟 contains both flavors. Take E₀, a 0-valent configuration reachable from C — exists because C is bivalent — and E₁, a 1-valent one. For each i: if Eᵢ is in 𝒞 (e not yet applied on the way to it), then Fᵢ := e(Eᵢ) is in 𝒟, and Fᵢ is i-valent because every successor of an i-valent configuration is still i-valent. If instead e *was* already applied on the way to Eᵢ, then some Fᵢ in 𝒟 lies on the path and Eᵢ is reachable from it. That Fᵢ is univalent by assumption. It cannot be (1−i)-valent, because then every configuration reachable from Fᵢ, including Eᵢ, would be forced to decide 1−i, while Eᵢ is i-valent. So Fᵢ is i-valent in this case too. Thus 𝒟 has a 0-valent configuration and a 1-valent one.

Now, the e-free reachable set 𝒞 is connected by the schedules that generate it, and the image 𝒟 contains both valences, so somewhere the valence flips between *neighbors*. Precisely: call two configurations neighbors if one is a single step from the other. By walking inside 𝒞 from a preimage of a 0-valent member of 𝒟 to a preimage of a 1-valent member, I can find neighboring C₀ and C₁ in 𝒞 such that D₀ = e(C₀) is 0-valent and D₁ = e(C₁) is 1-valent. Since they're neighbors, one comes from the other by a single event; without loss of generality C₁ = e′(C₀) for some event e′ = (p′, m′). So I have a little picture: from C₀, doing e gives the 0-valent D₀; doing e′ gives C₁, and then doing e gives the 1-valent D₁. The single step e′ is what flips the eventual decision. Everything now rides on who p′ is relative to p.

Case 1: p′ ≠ p. Then e and e′ are steps by *different* processes, so by the commutativity diamond they don't interfere — applying e then e′ equals applying e′ then e. In particular D₁ = e(C₁) = e(e′(C₀)) = e′(e(C₀)) = e′(D₀). So D₁ is a single step e′ *out of* D₀. But D₀ is 0-valent, and every successor of a 0-valent configuration is 0-valent. So D₁ is 0-valent. But D₁ is 1-valent. That's the contradiction — disjoint processes commute, so the order of e and e′ can't possibly be what decides the value.

So the flip must be caused by e′ being a step of the *same* process p. Case 2: p′ = p. Now both e and e′ are receives by the one process p — the decision is hinging on which message p receives, in which order. This is exactly the picture I couldn't shake: the whole commitment is localized to two messages arriving at *one node*. So let me silence that node and see what the rest of the system can do. Take any finite deciding continuation from C₀ in which **p takes no steps** — it exists, because from this accessible configuration the protocol must still tolerate p crashing, so with p frozen the others eventually reach a decision. Let its schedule be σ, and let A = σ(C₀) be the configuration it decides in. Crucially, σ touches only processes *other than* p, while e and e′ are both steps of p — so σ commutes with e and with e′, by the diamond.

Apply σ to C₀ and then do e: by commutativity, e(σ(C₀)) = σ(e(C₀)) = σ(D₀). So e(A) is reachable from D₀, which is 0-valent — hence e(A) is 0-valent. Now apply σ to C₀, then do e′ (getting σ applied to C₁ = e′(C₀), i.e. reaching σ(C₁)), then do e: by commutativity e(e′(σ(C₀))) = σ(e(e′(C₀))) = σ(e(C₁)) = σ(D₁). So e(e′(A)) is reachable from D₁, which is 1-valent — hence e(e′(A)) is 1-valent. Step back and look at A: from A, the single step e leads to a 0-valent configuration, and the two steps e′ then e lead to a 1-valent one. So from A I can still reach a 0-decision and a 1-decision — **A is bivalent.** But A is the configuration at the *end of a deciding run* — by assumption that run decides, so A is univalent. Contradiction.

Both cases collapse. So my assumption — that every configuration in 𝒟 is univalent — is false. There is a bivalent configuration reachable from C by a schedule ending in the designated event e. The system can always be coaxed to apply a chosen e and still come out undecided.

The two cases together say something I should hold onto. The moment the protocol commits to a value, that commitment is the product of one step. If that decisive step is a receive by some process *other than* p, commutativity says its effect doesn't depend on its order relative to p's pending step — so it can't be what tips a balanced configuration, contradiction. So the decisive step has to be p's own receive — the commitment is localized to the *order in which one process, p, takes in two specific messages.* And that's precisely the order that becomes invisible the instant p crashes: freeze p, and the rest of the system, by the diamond, reaches a configuration from which *both* values are still live. The thing that decides is the thing that disappears under a single crash.

Now I chain the two halves into a run that never decides — and I have to be careful that the run I build is *admissible*, not just non-deciding, or the impossibility is vacuous (a run where I starve some process or never deliver a message proves nothing; the protocol is allowed to not decide on unfair runs). Start from a bivalent initial configuration C₀ — exists by the first half. Keep a queue of the processes in some order, and order the message buffer by send-time, earliest first. I'll run in stages, and maintain the invariant that every stage begins in a bivalent configuration.

A stage, starting from bivalent C with process p at the head of the queue: let m be the earliest message addressed to p in C's buffer, or null if there is none, and set e = (p, m). By the second half, there's a bivalent configuration C′ reachable from C by a schedule whose *last* event is e — run that schedule; it's the stage. Then move p to the back of the queue and repeat from C′.

Why is the resulting infinite run admissible? Because each process reaches the head of the queue infinitely often, and the last event of that stage is a step by that process, so each process takes infinitely many steps — nobody is starved, so in fact *no* process is faulty in this run (better than one crash — zero). And because every stage where p is at the head ends by receiving p's *earliest* pending message, every message eventually becomes p's earliest pending message at one of p's turns and gets delivered — no message is delayed forever. So the run satisfies every fairness condition: at most one faulty process (here none), every message to a live process eventually received. It's admissible. And every stage starts and ends bivalent; since valence only shrinks along steps, no intermediate configuration inside the stage can have become univalent and then returned to bivalent. So the configuration is *always* bivalent, and no process is *ever* in a decision state. An admissible run that never decides.

That's the contradiction with the assumption I started this whole derivation from — that some protocol P is partially correct *and* decides on every admissible run. No such P exists.

So the window of vulnerability was never an engineering defect. It's a theorem: in a fully asynchronous system where even one process may crash, no deterministic protocol can guarantee consensus. The adversary scheduling messages can keep the system bivalent forever, and bivalent means undecided. The reason, in one breath: the proof may freeze the one process whose receive would expose the decision value, and once that indistinguishability is available, a fair scheduler can keep postponing commitment without actually crashing anyone.

Let me sharpen the boundary by asking what I'd need to give up to make agreement *possible*, because the proof tells me exactly where it leaned. It leaned twice on the same move: silence the one process p that holds the decisive distinction, and let the others run. That move is available because (a) the system is asynchronous, so a slow p is indistinguishable from a dead p, and (b) a process may die at a bad moment *during* the run. Suppose I forbid (b) — no process dies during execution — but still allow processes to be *dead from the start*, unknown which, as long as a strict majority are alive initially. Can I agree then? I think yes, and it's worth constructing, because it shows the impossibility is tight: it's death *during* the run, not mere uncertainty about who's alive, that kills consensus.

Two stages. Let L = ⌈(N+1)/2⌉, a bare majority. Stage one: every process broadcasts its id and then listens until it has heard from L−1 *others*. Form a directed graph G with an edge i→j whenever j heard directly from i; every process that finishes records L−1 incoming edges, and because no one dies mid-run and a majority are alive, every initially alive process eventually collects them. Stage two: build the transitive closure G⁺. Each process broadcasts its id, its initial value, and the names of the L−1 it heard from; then it waits for stage-two messages from every ancestor it currently knows of, learning new ancestors from those messages, until all currently-known ancestors have been heard from. When it settles, it knows all of its ancestors in G and the incident edges that let it compute the relevant part of G⁺. Now it computes the *initial clique* of G⁺ — a clique with no incoming edges — using the fact that a node k is in it exactly when k is an ancestor of every node that is an ancestor of k. For any node in such a source clique, its L−1 predecessors cannot come from outside the clique, so the clique has at least L members including the node itself. Two distinct initial cliques would be disjoint source components, and two disjoint sets of size at least L cannot both fit inside N processes. So there is *exactly one* initial clique, it has size at least L, and every process that finishes the second stage learns precisely which processes comprise it *and* their initial values. All processes decide by one fixed rule applied to the initial values of that one common clique — and since they all see the same clique and the same values, they all decide the same thing. Agreement, validity, termination — under the restricted failure pattern. The impossibility and this protocol meet exactly at the line: asynchronous agreement survives initial-only failures but not a single mid-run crash.

The causal chain, start to end: I wanted a deterministic protocol that always agrees despite one crash; every attempt stalled on a single process whose silence I couldn't read as death-or-slowness; so I asked whether the stall is fundamental and chose bivalence — "both outcomes still reachable" — as the formal stand-in for "undecided." A bivalent *initial* configuration must exist, because crashing the one process that distinguishes two adjacent inputs makes the rest decide identically, forcing undecidedness at the boundary. And from any bivalent configuration a bivalent continuation is reachable after any designated event, because the decisive step that would seal a value is necessarily one process's own receive (disjoint steps commute, so a third party's step can't do the sealing), and that receive can be hidden behind that process's crash, leaving the system with both values still live. Scheduling those moves round-robin gives an admissible, perfectly fair run whose configurations remain bivalent and therefore decide nothing — so total correctness in an asynchronous system with one possible crash is unattainable.
