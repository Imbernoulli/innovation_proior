Let me start from where the cost actually sits. In a decoder block I have two sublayers, attention and the position-wise feed-forward network, and the FFN is the heavier of the two by parameter count — it projects up to a hidden width of 4d, applies a pointwise nonlinearity, and projects back down, so its two matrices alone are 8d^2 parameters against attention's 4d^2. Everyone pours effort into attention. The FFN, meanwhile, is `act(x W1) W2` and the only thing in it that isn't a plain linear map is that one scalar function `act`, which I inherited without thinking — ReLU because the original Transformer used it, or GELU because BERT and GPT switched to it and the codebase followed. So if I want a cheap win in language-model pretraining, the most under-examined knob in the whole architecture is that single pointwise function. And the constraint I want to respect is strict: change only `act`, keep the `(B,T,d)->(B,T,d)` shape, don't touch attention or normalization or the optimizer or the data, and ideally add no parameters and no new hyperparameter, so the change is something anyone can drop into an existing codebase and just run. The question is whether the activation that got inherited is actually the best one for this slot, and I have no reason to believe it is.

So what are my options today, and what does each really do. ReLU is `max(0, z)`: kill the negatives, pass the positives through unchanged. The killing-negatives part is doing real work — it makes the layer sparse, only some units fire — but stare at the positive branch and it's literally the identity, `z`. So the FFN's entire nonlinearity is the hard gate at the origin; above zero it's a linear map, and a unit that's strongly on is treated on exactly the same linear scale as a unit that's barely on. The function is piecewise linear with a sharp kink. GELU smooths that kink — `z·Phi(z)` where `Phi` is the Gaussian CDF, derived as the expected value of `z` times a Bernoulli(`Phi(z)`) keep-mask, a deterministic stand-in for input-dependent dropout — and it adds a gentle dip just below zero. It's nicer to optimize. But what is GELU doing for a *strongly* activated unit? `Phi(z) -> 1` as `z` grows, so `GELU(z) -> z`: asymptotically it's the identity too. Swish, `z·sigmoid(beta z)`, came out of an actual reinforcement-learning search over scalar activation functions, which already tells me searching for `act` is worth it — but its winner is self-gated, smooth, and again `sigmoid(beta z) -> 1`, so `Swish(z) -> z` for large positive `z`.

Let me not just assert that "they're all asymptotically the identity" — let me put numbers on it and see how fast the convergence actually is, because if it only kicks in at `z = 100` it doesn't matter for real activations. Tabulating the four functions at a few positive inputs:

```
z      ReLU    GELU    Swish    ReLU^2
0.5    0.500   0.346   0.311     0.250
1      1.000   0.841   0.731     1.000
2      2.000   1.954   1.762     4.000
4      4.000   4.000   3.928    16.000
8      8.000   8.000   7.997    64.000
```

This is more decisive than I expected. By `z = 4`, GELU has already reached `4.000` and Swish `3.928` — both have effectively collapsed onto the line `y = z` while the input is still a perfectly ordinary activation magnitude, not some extreme tail value. So the "asymptotically linear" property isn't a curiosity at infinity; over the entire range a hidden unit actually occupies, GELU and Swish *are* the identity to three decimals. They differ from ReLU only in the narrow band near the origin (at `z = 0.5` they pull down to 0.35/0.31, below ReLU's 0.50) and agree with it completely the moment the unit fires with any conviction. The unexamined assumption baked into all of them is now visible in the numbers: the positive branch should be linear, and the only freedom anyone has exercised is the exact shape of the soft knee around zero.

That assumption is exactly the thing I should poke. Why should the positive branch be linear? When a unit is strongly activated, that's the network expressing high confidence in some feature; a linear positive branch says "twice the evidence, twice the output," which is fine, but it gives me no way to make a few strongly-firing units *dominate* the layer's output more than their linear share. If instead the positive branch grew faster than linearly, the loud units would pull ahead of the quiet ones, sharpening the layer into a more selective, more contrastive map — and if I keep killing the negatives, I keep the sparsity. Is there any precedent for faster-than-linear rectified functions actually helping, or am I just speculating? There is, and it's an old one I half-remember from associative memory. Krotov and Hopfield studied dense associative memories with a rectified polynomial energy, `F_n(x) = x^n` for `x >= 0` and `0` for `x < 0`. The `n = 2` energy case is the standard quadratic Hopfield network; as `n` increases, each term in the energy gets *sharper*, and a sharper energy can pack more memories into the same number of neurons — they work out a capacity `K^max = alpha_n N^{n-1}` that grows with `n`. In the feed-forward neural-network view, the activation degree is shifted down by one from the energy power, so I should not blindly copy their notation into a Transformer. What I can take from it is the open question they pose directly: above the threshold, should the output grow linearly, sub-linearly, or faster than linearly, and is there something better than ReLU? So the idea that a rectified, faster-than-linear nonlinearity has a real upside is sitting in the energy-based-memory literature, where "spikier above threshold" buys capacity. The catch is that it lives in shallow associative-memory models and on toy classification, never in a deep Transformer FFN.

So I want a rectified function that grows faster than linearly. The natural family is the rectified powers `f_p(z) = max(0, z)^p`. If "faster than linear" is the lever, I want the *smallest* step beyond linear, because I'm about to stack this thing through dozens of layers and I want to perturb the known-good architecture as little as possible — a wildly sharper activation is more likely to blow up training than to help. The `p = 1` case is just ReLU, the linear-positive-branch incumbent. The next integer up is `p = 2`: `act(z) = max(0, z)^2`. That is the minimal step out of the linear family, so it's where I'll start — but I should pressure-test it from a few angles before I trust it, because "minimal" doesn't mean "safe."

First, do I really need the rectification, or could I just square? Tempting to say `act(z) = z^2` and skip the `max`. But `z^2` is even — it forgets the sign of `z` entirely, so a strongly negative pre-activation and a strongly positive one produce the identical large output. That throws away the gating that made ReLU useful in the first place: the whole point was that negative pre-activations mean "this feature is absent, output nothing." A raw square turns "absent" into "very present," which is backwards, and it also makes the function symmetric and non-monotonic in a way that destroys the sparsity. So I can't square the raw pre-activation. I need to rectify first — zero out the negatives so off units stay off — and *then* square the surviving positive part. That's exactly `max(0, z)^2`, the `p = 2` rectified polynomial, not the bare parabola. Good: the rectification and the squaring are both load-bearing, and in that order.

Second, what does this function look like, precisely, so I'm not fooling myself about the asymptotics or the smoothness. For `z <= 0` it's flat at `0`. For `z > 0` it's `z^2`. The value is continuous at the origin — both branches give `0` there. The derivative is the interesting case, because ReLU's hard kink lives exactly there. Differentiating: `0` for `z < 0`, and `d/dz (z^2) = 2z` for `z > 0`. Let me check the two one-sided limits at the origin instead of eyeballing it. From the left, the slope is flat, so `lim_{z->0-} act'(z) = 0`. From the right, `lim_{z->0+} 2z = 0`. Both are `0`, so the derivative is *continuous* at the origin — `C^1`. Contrast ReLU, whose left slope is `0` and right slope is `1`: those don't match, which is precisely the hard kink. So squared ReLU has removed it; the function leaves zero smoothly with zero initial slope and then accelerates. (A quick finite-difference sanity check confirms it: with `h = 1e-6`, `[act(h) - act(0)]/h = 1e-6 ≈ 0` and the left difference is `0`, whereas for ReLU the same right difference is `1.0` — the kink shows up immediately.)

Now compare the growth. ReLU and GELU and Swish all satisfy `act(z)/z -> 1` as `z -> +inf` — and from the table above they reach that limit by `z = 4`. Here `act(z)/z = z`, which doesn't converge to anything; reading it off the same table, `ReLU^2/z` is `0.5, 1, 2, 4, 8` at those inputs — it just equals `z`. The function grows *quadratically*, without bound relative to the input. That is a genuinely different growth regime, not a cosmetic reshaping near zero. To make the "loud units dominate" claim concrete: take one unit at `z = 1` and another at `z = 2`, twice as activated. Under GELU the outputs are `0.841` and `1.954`, a ratio of `2.32` — close to the linear `2.0`, the strong unit gets roughly its proportional share. Under squared ReLU they are `1` and `4`, a ratio of `4.0`: the unit that's twice as activated produces *four* times the output. That's the quadratic dominance I was reaching for, the deep-network analogue of the "spikier activation packs more capacity" story from the memory model — and now I've seen it as a number rather than assumed it.

Third, is `max(0, z)^2` secretly something the field already found valuable? If it is, I get its benefits without paying its costs. The other thing people put in this FFN slot is the GLU family. A Gated Linear Unit replaces one projection with a product of two: Dauphin's `(xW) ⊗ sigmoid(xV)`, with a bilinear variant `(xW) ⊗ (xV)` that drops the sigmoid, and Shazeer's Transformer version puts these in the FFN as `ReGLU(x) = max(0, xW) ⊗ xV`, plus `GEGLU` and `SwiGLU` for GELU/Swish gates. The reason GLU was supposed to help, per Dauphin, is gradient flow: the product `a ⊗ sigma(b)` has a linear path `a ⊗ sigma(b)` whose gradient `∇a ⊗ sigma(b)` isn't squashed by a saturating derivative the way a `tanh` gate's would be, so signal survives through depth. And separately there's the representational argument — a product of two learned quantities, a multiplicative interaction, strictly enlarges the function class a layer can express versus a purely additive one. ReGLU is the ReLU-gated member of that family. So let me write ReGLU out componentwise and compare it to my candidate. ReGLU's `i`-th hidden unit, before the down-projection, is `max(0, (xW)_i) · (xV)_i` — the ReLU-gated value of one projection times the raw value of a second, independent projection.

What happens to ReGLU if the two projections are forced to be the *same* affine map, `V = W` and, if biases are present, the value bias is tied to the gate bias? Then unit `i` is `max(0, z_i) · z_i` for the shared pre-activation `z_i = (xW + b)_i`. Check the two cases. If `z_i > 0`: `max(0, z_i) = z_i`, so the product is `z_i · z_i = z_i^2`. If `z_i <= 0`: `max(0, z_i) = 0`, so the product is `0 · z_i = 0`. Put them together: `max(0, z_i) · z_i` is exactly `max(0, z_i)^2` for every `z_i`. So squared ReLU applied after an affine projection is precisely the tied-affine special case of ReGLU. The gate and the value are the same hidden pre-activation. That does not give me the full independent two-projection GLU family, but it does give me the same kind of multiplicative interaction — a product of two copies of the pre-activation, the cheapest possible degree-2 term — and it keeps a nonsaturating positive branch rather than a sigmoid gate whose derivative can saturate.

That tying is the whole point. ReGLU, GEGLU, and SwiGLU each use *three* weight matrices — the gate projection `W`, the value projection `V`, and the down-projection `W2` — whereas the plain FFN uses two. To keep parameters and FLOPs matched against the two-matrix baseline, the GLU papers shrink the hidden width `d_ff` by a factor of two-thirds, from 3072 to 2048 in their T5-base comparison. Tying the gate and value maps collapses the extra projection, so I'm back to *two* matrices — `c_fc` and `c_proj` — and I can keep the full 4d hidden width untouched. No third matrix, no width reduction, no new hyperparameter. The algebra does not prove empirical dominance over independent GLU variants; it says I can test the core multiplicative, faster-than-linear idea in the exact parameter and shape budget of the activation I'm replacing. And it's simpler than GELU or Swish per element: no `erf`, no `exp`, no `sigmoid`, just a rectify and a square.

Let me also be honest about why I stop at `p = 2` and don't chase higher rectified powers. Higher `p` grows faster and would dominate even harder, so why not? My first instinct was to say larger powers overflow, but let me check that before I lean on it. In bf16 the max representable value is about `3.4e38`, so `max(0, z)^p` only overflows when `z > (3.4e38)^{1/p}` — that's `z > 7e12` for `p = 3` and still `z > 2.6e6` for `p = 6`. No realistic post-LayerNorm pre-activation gets within many orders of magnitude of that, so literal overflow is *not* the real objection; I'd have been asserting a problem that doesn't occur. The honest objection is dynamic range and gradient conditioning, and I can quantify it. Take a "small but on" unit at `z = 0.1` and a "moderately loud" one at `z = 5`. The ratio of their outputs is `(5/0.1)^p = 50^p`: that's `50` for ReLU, `2500` for `p = 2`, and `1.25e5` for `p = 3`. The gradients scale as `p·z^{p-1}`, so the slope ratio between the same two units is `50` for ReLU, `50` for `p = 2`... actually let me compute it — `(2·5)/(2·0.1) = 50` for `p = 2`, and `(3·25)/(3·0.01) = 2500` for `p = 3`. So every increment in `p` multiplies the spread between a quiet and a loud unit by another large factor, in both the forward values and the gradients, and that spread compounds through a deep stack. `p = 2` already widens the value spread from 50x to 2500x versus ReLU; `p = 3` pushes it to 1.25e5x, which is the regime where I'd genuinely worry about a few units swamping everything and destabilizing optimization. So the principled stopping point isn't "overflow" — it's that `p = 2` is the smallest step that buys the faster-than-linear effect while keeping the dynamic range within a couple orders of magnitude of ReLU's. It's also the one that coincides exactly with tied-affine ReGLU (cubic and up don't have that clean single-product interpretation — they're products of three or more copies, a different object). I can't claim from this desk that `p = 2` empirically beats `p = 3`; what I can say is `p = 2` is the lowest-risk member to validate first, and I'd want to actually run `p = 3` head-to-head before believing it's worse. If the cheap version doesn't help, chasing higher powers would be premature anyway.

One more design point to nail down so I don't leave a hole: where does the squaring sit relative to the rest of the FFN. It goes exactly where the old activation went — between the up-projection and the down-projection, applied pointwise to the 4d hidden state. The up-projection `c_fc` plays the role of `W` (it produces the pre-activation `z = xW` that gets both rectified and squared), and `c_proj` is the down-projection `W2`. Dropout, if the codebase has it, sits after the activation as before; the change is orthogonal to it. Bias terms are inherited from the surrounding FFN — the activation swap doesn't care. So this really is a one-line edit to the forward pass: replace the inherited `act` with rectify-then-square.

Let me write it into the FFN, filling the single empty slot in the sublayer:

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


class MLP(nn.Module):
    """Position-wise FFN with a squared-ReLU activation.
    Up-project to 4x width, apply max(0, z)^2 pointwise, project back down.
    Two matrices, full 4x width, no extra parameters vs a ReLU/GELU FFN."""

    def __init__(self, config):
        super().__init__()
        self.c_fc = nn.Linear(config.n_embd, 4 * config.n_embd, bias=config.bias)
        self.c_proj = nn.Linear(4 * config.n_embd, config.n_embd, bias=config.bias)
        self.dropout = nn.Dropout(config.dropout)

    def forward(self, x):
        x = self.c_fc(x)            # z = x W1 : up-projection to the 4x hidden state
        x = F.relu(x).square()      # max(0, z)^2 : rectify (keep sparsity), then square
                                    #   (= tied-map ReGLU max(0,z)*z; quadratic positive branch)
        x = self.c_proj(x)          # down-projection back to n_embd
        x = self.dropout(x)
        return x
```

So the causal chain. I started from the observation that the FFN is the heavy sublayer and its one pointwise nonlinearity is the part of the Transformer nobody re-examines — inherited, not chosen. Looking hard at the incumbents, ReLU's positive branch is the bare identity and GELU and Swish, for all their smoothness near zero, are *also* asymptotically linear on the positive side; they all silently assume the positive branch should be roughly linear. Questioning that assumption pointed at faster-than-linear activations, for which the dense-associative-memory line already gives a reason to care about sharper rectified growth — but only the smallest step beyond linear is safe to stack deep, so the `p = 2` rectified polynomial `max(0, z)^2`. Checking it can't be done by squaring the raw input (that's even and kills the gating); the rectify-then-square order is forced. Working out the function shows a genuinely different, quadratic growth regime, with a derivative `2z` that leaves the origin smoothly and has no saturation plateau on the positive side. Then the unification: `max(0, z)^2 = max(0, z) · z` is exactly ReGLU when the gate and value use the same affine pre-activation, so this is the tied-map special case of the gated-linear-unit family — a multiplicative degree-2 term with two matrices instead of three, the full 4x width instead of a 2/3-shrunk one, no extra parameters, no new hyperparameter, and no transcendental to evaluate. It drops into the existing FFN as a one-line replacement of the inherited activation: `x = F.relu(x).square()`.
