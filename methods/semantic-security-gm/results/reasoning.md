Let me start from the thing that actually bothers me, which is a card game over the telephone. Two people want to deal cards without a deck — they exchange encrypted cards over the wire. I have public-key encryption to do this: Diffie and Hellman gave the concept, Rivest, Shamir and Adleman gave a concrete trapdoor, Rabin gave another. So I can encrypt a card and send it. And yet the game can still be broken. Suppose my opponent, looking only at the ciphertext of a card he is not supposed to see, can compute its *color*. Red or black, one bit. He never recovers the card. He never inverts the encryption. He just learns one bit, and that bit is enough to ruin the game. Lipton actually showed this for mental poker built on RSA — a bit that must stay hidden falls right out.

So the notion of security I inherited is failing me, and I should look hard at what that notion even says. Diffie and Hellman say a system is secure if it is *computationally infeasible to recover the plaintext or the key*. Their whole analysis is a work-factor analysis: how many operations to invert. RSA inherits this — its security rests on it being hard to take s-th roots mod n without the factorization. Rabin inherits it and even sharpens it: he proves that extracting square roots mod n, for a noticeable fraction of residues, would let you factor n, because if you ever get two square roots x and y of the same residue with x ≠ ±y, then gcd(n, x±y) hands you a factor. That is a beautiful "break implies factor" theorem.

But look at what it protects. It protects the *whole message*. It says nothing whatsoever about the color of the card. The adversary in the card game does not want the whole card; he wants one bit, and the theorem about inverting the function is silent about that bit. The gap between "can't invert" and "can't learn anything" is exactly where my game dies.

Let me make the gap explicit, because I want to know if it is a fixable accident of RSA or something structural. Take a deterministic encryption E, RSA say. If some visible output bit varies with m — parity for an integer-valued example, or any ciphertext bit on which E(m₀) and E(m₁) differ — then that bit defines a nonconstant predicate of the plaintext, and the ciphertext gives it away for free. Even if I do not know in advance which output bit varies, the deeper problem is already visible: the ciphertext is a fixed public function of the message. A deterministic encoding turns features of E(m) into features of m, computable by anyone, no trapdoor needed.

And it gets worse the moment I remember the key is public. If E is deterministic, then equal plaintexts give equal ciphertexts. So the ciphertext leaks message *equality*: if two cards encrypt to the same string, they are the same card. Worse still, the adversary holds E. He can take any guess m₀, compute E(m₀) himself, and compare it to the intercepted ciphertext. "Is this card the ace of spades?" — he encrypts the ace of spades and checks for a match. A deterministic public-key scheme can never hide whether the message equals a guess. The publicness of the key is a gift to the attacker as long as encryption is a function.

That last observation is doing a lot of work, so let me sit on it. The reason the adversary can re-encrypt-and-compare is that for him E(m) is a fixed target: one plaintext, one ciphertext. If instead each plaintext had *many* possible ciphertexts, and encryption tossed coins to choose among them, then the adversary's own encryption of his guess would almost surely not equal the intercepted one even when his guess is right — because they are independent samples from a large set of encodings. The re-encrypt-and-compare attack dissolves. So whatever a secure scheme is, it cannot be a function. It has to be randomized. I will keep that as a hard conclusion: **secure encryption must be probabilistic.**

Before I build anything, I have to settle the harder question, the one the card game forced on me: what should "secure" *mean*? Let me try to write down what I actually want. I want that the ciphertext gives the adversary no help in computing anything about the plaintext. "Anything" — the color, the suit, a hash, the identity, any predicate, recursive or not, because partial information in a real application can be any function whatsoever. And "no help" relative to what? Relative to what he could already do without the ciphertext. He always knows the message distribution; if 90% of cards in play are red, he can guess "red" and be right 90% of the time with no ciphertext at all. I cannot stop that. What I can demand is that the ciphertext does not improve on it.

So here is the definition I want to reach for. For any function f on the message space, let p_f be the best a-priori probability of guessing f(m): p_f = max over values v of the probability that f(m) = v. With no ciphertext, the adversary guesses the most likely value and wins with probability p_f, and nothing beats that. If f is allowed to depend on the public encryption algorithm E, I just write p_E for the same maximum after E is fixed. I will say the scheme is **semantically secure** if for every such f_E, an adversary *given the ciphertext* still cannot compute f_E(m) with probability better than p_E by a non-negligible amount, against any polynomially bounded adversary. Whatever is efficiently computable about the cleartext given the ciphertext is efficiently computable without it. That is the sentence I have been circling.

Let me pin it down as a game so it is testable, and let me be generous to the adversary so the definition is strong. The encryption algorithm E is public. Game one: pick m from the distribution, ask the adversary for f(m) with no ciphertext — his best is p_f. Game two: pick m, give him one ciphertext drawn from E(m), ask for f(m). Game three, the strong one: let the adversary even *choose* the function f after seeing E, then give him a ciphertext and ask for f(m). The scheme is semantically secure if he cannot win game three with probability non-negligibly above p_f. Note I let f be utterly arbitrary — not necessarily polynomial-time, not even recursive — because the *adversary's* job of computing it must be hard, and I do not want to assume f is easy. This is the polynomially bounded shadow of Shannon's perfect secrecy: Shannon asked that the a-posteriori message probabilities equal the a-priori ones for an unbounded adversary, which costs you a key as long as the message; I am asking the same thing but only against poly-bounded adversaries, which is what lets the key be short.

Now I have a definition I believe in. The trouble is it quantifies over *all functions f*. That is a nightmare to prove anything against — to certify a scheme secure I would have to rule out every conceivable f, recursive or not. I need a handle. Let me look for an equivalent condition that is a single clean test.

Here is the test I want instead: the adversary cannot tell two messages apart. Concretely — let a poly-bounded "message-finder" produce two messages m₀, m₁; encrypt one of them at random; hand the ciphertext to a poly-bounded "line-tapper" and ask which message it was. Call the scheme **indistinguishable** (I had been calling it polynomially secure) if no such finder/tapper pair can win non-negligibly above one-half: formally, if for all m₀, m₁ the finder can produce, the probability the tapper outputs 1 on an encryption of m₀ differs from the probability it outputs 1 on an encryption of m₁ by at most 1/poly(k). This is one game, no quantifier over functions. And notice it instantly explains the determinism failure: a deterministic scheme is never indistinguishable, because the finder outputs any m₀ ≠ m₁, their ciphertexts differ as fixed strings, and the tapper distinguishes them trivially. Good — the definition has the right scheme on the wrong side of the line.

Are the two definitions the same thing? I should prove it, both ways, or I am just hoping.

One direction is short. Suppose a scheme is *not* indistinguishable: some finder produces m₀, m₁ and a tapper T tells their ciphertexts apart with gap δ greater than 1/poly. I want to convert that into a break of semantic security. Put the message distribution uniformly on m₀ and m₁, and let f(m)=1 exactly on m₁. With no ciphertext the best success is p_f=1/2. Given a ciphertext, run T; if T outputs 1 more often on encryptions of m₁ than on encryptions of m₀, use T as the guess, and if the sign is reversed, use 1−T. The success probability is 1/2+δ/2. That beats the a-priori optimum by a non-negligible margin. So semantic security fails. Contrapositive: semantic security implies indistinguishability. That direction is clean.

The other direction is the one that matters and the one that is not obvious: indistinguishability implies semantic security. I have to take a scheme that hides which of any two messages was sent and show it hides *every function* of the message. Suppose it does not — suppose some poly-size circuit C, given E and a ciphertext a ∈ E(m), computes some f_E(m) with probability exceeding p_E + ε, where ε = 1/poly(k) and p_E is the a-priori best after E is fixed. I must manufacture two messages and a distinguisher from C. The advantage C has is *averaged over the message distribution*; I need to localize it to a single pair of messages.

Let me think about what C's advantage means message by message. For a message m and a value v, write r^m_v for the probability that C outputs v on a fresh encryption of m. The success probability is Σ_m Pr[m] · r^m_{f_E(m)}. If C produced essentially the same output distribution on every message, then for any fixed reference distribution s_v its success would be at most Σ_v Pr[f_E(m)=v]s_v, which is no more than p_E because the s_v sum to 1. So the extra ε cannot come from a message-independent output pattern. It must come from messages on which C's output distribution actually moves.

Fix a reference message μ. Call a message ζ "good" if C's output distribution on ζ differs noticeably from its distribution on μ — meaning there is some value v with |r^ζ_v − r^μ_v| greater than ε/10. If ζ is not good, then on ζ's correct value I have r^ζ_{f_E(ζ)} ≤ r^μ_{f_E(ζ)} + ε/10. Let G be the good set and α its message probability. The total success of C is at most the good messages' mass, plus the non-good contribution:

Σ_{ζ∈G} Pr[ζ] · 1 + Σ_{ζ∉G} Pr[ζ](r^μ_{f_E(ζ)} + ε/10).

The middle sum is bounded by Σ_v Pr[f_E(m)=v] r^μ_v, and that is at most p_E. Therefore the whole success is at most α + p_E + ε/10. But I assumed success is greater than p_E + ε. So α must be greater than 9ε/10, in particular greater than ε/10. Good messages have noticeable mass. That is the mass lemma in the only form I need.

Given a good ζ, I still have to find the witnessing value v. I cannot read r^ζ_v and r^μ_v off in closed form, but I can estimate them by sampling, because E is public: draw many fresh encryptions of ζ, run C on each, and tabulate how often C outputs each value; do the same for μ. A value whose true gap exceeds ε/10 has probability at least ε/10 on one of the two sides, so it appears in a polynomial-size sample with high probability. With O(k^c/ε²) samples, the weak law of large numbers makes the empirical frequencies accurate to a small constant fraction of ε; scanning the two tabulations and accepting a value whose empirical gap exceeds 3ε/40 yields, with high probability, a true gap still bigger than ε/20. That is the sampling lemma.

Now I assemble the distinguisher. The message-finder picks a random reference μ and a random candidate ζ from the message distribution. With probability at least ε/10, ζ is good by the mass lemma; when it is, the sampling lemma hands me a value v with |r^ζ_v − r^μ_v| greater than ε/20. The finder outputs the pair (m₀, m₁) = (ζ, μ) together with that v. The line-tapper is simply: on a ciphertext x, output 1 if C(E, x) = v, else 0. Its probability of outputting 1 is r^{m}_v on encryptions of m, so it outputs 1 at rate r^ζ_v on ζ and r^μ_v on μ — a gap exceeding ε/20. That is a poly-size finder and tapper distinguishing two messages with non-negligible advantage. Indistinguishability is broken. Contrapositive: indistinguishability implies semantic security.

Both directions hold, so the two notions are *equivalent*. I will define security by semantic security — it is the meaningful statement, "no computable partial information" — but I will *prove* schemes secure through indistinguishability, the single clean game. That equivalence is the lever.

Now I can go build. I know the scheme must be randomized, and the natural place randomness enters is the smallest unit: a single bit. The card game even told me a single bit is the whole battle — hide one bit perfectly and I am most of the way there. So let me try to encrypt one bit, with many ciphertexts per bit, half of them encoding 0 and half encoding 1.

What object lets the legitimate receiver read off the bit but stops the adversary cold? A trapdoor *function* is the wrong shape — I saw the embedded-bit attacks fail. Let me look at those failures again to learn what shape I need. If I hide my bit as the i-th bit of a random r and send E(r) for a one-way E, the trouble is that hardness of recovering r says nothing about that one bit; a one-way function can leak a specific bit. The sharp example: g^x mod p is one-way (inverting it is discrete log), but the last bit of x is free — x is even exactly when g^x is a quadratic residue mod the prime p, and residuosity mod a prime is decided in polynomial time by Euler's criterion, q^{(p−1)/2} mod p. So the bit I tried to bury is sitting on the surface. And Johnson's cleverer version — bit at a random position, position encoded in the first seven bits — fails the same way: if E leaks those seven bits and any one of the others, the adversary recovers my bit with probability 1/2 + 1/2·(1/93). The lesson is blunt: "x is hard to compute from E(x)" gives me nothing about any *one designated bit* of x. I keep trying to inherit one-bit security from whole-message hardness and it is simply not there to inherit.

So I must stop embedding a bit inside something and instead use a primitive whose hardness *is* the hardness of one bit. I want a predicate B with three properties. Anyone can sample an x with B(x) = 0 or an x with B(x) = 1 — that lets me encrypt either bit. The holder of a trapdoor can, given any z, compute B(z) — that lets the receiver decrypt. And without the trapdoor, no poly-size circuit can compute B(z) noticeably better than guessing — *unapproximable*, meaning it is right on at most 1/2 + 1/poly of inputs. A predicate like that is exactly a hardness-of-one-bit object: the bit B(x) is what is hidden, by construction, not as a hoped-for byproduct of inverting a function.

I also need one more, mundane but essential, property: I must be able to publish an index i naming the predicate B_i *together with* its secret trapdoor σ(i), generating the pair efficiently. If only a tiny fraction of (index, secret) pairs were efficiently generatable, an adversary could take my published i and just resample pairs until he regenerated my i and recovered σ(i). So the sampling of (i, σ(i)) must be efficient and essentially uniform. Call this constructibility. A predicate B with these — samplable both bit-values, trapdoor-evaluable, unapproximable, constructible — I will call an **unapproximable trapdoor predicate.**

Given one, encryption writes itself. The public key is the index i. To encrypt a bit b, sample a random x with B_i(x) = b and send x; there are many such x, so the encoding is randomized — exactly the determinism-killer I needed. To encrypt a message, encrypt each bit independently and send the tuple. To decrypt, the receiver applies the trapdoor to each component and reads B_i off, bit by bit.

But now I owe two things. First, does such a predicate exist concretely? Second — and this is the subtle one — even if each bit is individually hidden, is the *whole message* hidden? Independent per-bit security is not obviously message security; the adversary sees the whole tuple at once.

Take the whole-message question first, because if it fails the construction is pointless. I will prove indistinguishability of the scheme and let the equivalence carry it to semantic security. Suppose not: a finder produces two l-bit messages m₀, m₁ and a tapper T distinguishes their encryptions with advantage exceeding ε. I want to extract from T a way to approximate the underlying predicate B, contradicting unapproximability.

Picture the messages as vertices of the l-dimensional hypercube {0,1}^l. Label each vertex v with the frequency that T outputs 1 on a random encryption of v. The two endpoints m₀, m₁ have labels differing by more than ε. Now walk from m₀ to m₁ flipping one bit at a time — a path of at most l steps, each step changing a single coordinate. The label changes by more than ε over the whole walk, so *some single step* must change the label by more than ε/l. There are two adjacent words s and t with frequencies differing by more than ε/l, and s and t differ in exactly one bit-position, call it d.

Stare at that step. s and t agree everywhere except position d, where one has bit 0 and the other bit 1. To use this against the underlying predicate B, I must preserve the same distribution that defines the labels, so I do not freeze the other coordinates to arbitrary constants. I sample each other coordinate from the public sampler conditioned on its agreed bit, splice my challenge y into position d, and feed the resulting tuple to T. If B(y)=s_d, the tuple has the encryption distribution of s; if B(y)=t_d, it has the encryption distribution of t. The sign of the frequency gap tells me whether to output T's bit or its complement, so on a balanced predicate this guesses B(y) with advantage half the gap. If the two predicate values were badly unbalanced, the majority value would already approximate B; unapproximability rules that out except for a small constant loss. With the usual sampling error for finding the adjacent step, the reduction gets an ε/(5l)-approximator of B. That contradicts unapproximability. This is the hybrid argument: per-bit hardness lifts to whole-message hardness, the price being a 1/l loss in the advantage. So the scheme is indistinguishable, hence semantically secure.

That settles the architecture. Now the existence question: I need a concrete unapproximable trapdoor predicate. And the discrete-log example from a moment ago, the one that *broke* me, is suddenly suggestive. The thing that killed the embedded bit was that residuosity is *easy* mod a prime. So let me move to a *composite* modulus, where it is not.

Let n = p·q, a product of two large distinct primes. Consider the predicate Q_n(x) = 1 if x is a quadratic residue mod n, 0 if not. With the factorization I can compute it: x is a residue mod n iff it is a residue mod p and mod q, each tested by Euler's criterion and combined by the Chinese Remainder Theorem, all in O(|n|³). So it is trapdoor-evaluable. Without the factorization, what do I know? There is the Jacobi symbol (x/n), computable in polynomial time *without* factoring. If (x/n) = −1, then x is a nonresidue for free — no trapdoor needed — so those elements are useless for hiding. The interesting elements are those with (x/n) = +1: among them exactly half are residues and half are nonresidues, and *deciding which* is the Quadratic Residuosity Problem, with no known efficient algorithm when n's factorization is hidden. This is an old hard problem — one of the four Gauss singled out in the Disquisitiones. So I will work entirely inside the Jacobi-+1 elements, where the predicate Q_n is exactly the hard bit.

I will *assume* this is hard — the Quadratic Residuosity Assumption: no poly-size circuit decides Q_n on Jacobi-+1 inputs for a noticeable fraction of hard composites n. That is an assumption about *some* inputs. But my security reduction needs *unapproximability* — hardness on a 1/2 + 1/poly fraction, on the average, for random x. I have to bridge "hard somewhere" to "hard almost everywhere," or the predicate is not the unapproximable object I claimed.

This is the prettiest part. Quadratic residuosity is *randomly self-reducible*. Suppose I had an oracle O that decides Q_n correctly on slightly more than half the Jacobi-+1 inputs — right on a 1/2 + ε fraction, but I do not know which ones. I claim I can then decide Q_n on *every* Jacobi-+1 input with probability as close to 1 as I want. Here is the mechanism. To test an unknown q, generate many random quadratic residues x_i (just square random elements) and form the products q·x_i mod n. The key fact: if q is a residue, then q·x_i ranges over random *residues*; if q is a Jacobi-+1 nonresidue, then q·x_i ranges over random Jacobi-+1 *nonresidues* — multiplication by a random residue is a bijection inside the residue class or inside the nonresidue coset. (Writing every element via the CRT as a^i b^j with generators a, b of the two prime parts, q is a residue iff both exponents are even, and y = q·x with x a residue is a bijection between residues and the q-coset.) So I have turned the unknown q into a fresh random sample of *its own residue class*. Now calibrate: estimate α, the rate at which O says "residue" on genuine random residues, by feeding it squares. Feed O the products q·x_i and compare that "residue" rate to α. If q is a residue, the rate is α; if q is a nonresidue, the rate is β, and the oracle's overall correctness advantage gives 1/2·α + 1/2·(1−β) > 1/2 + ε, hence α−β > 2ε. A polynomial-size sample separates the two cases by the weak law. Whichever side it matches tells me Q_n(q), with confidence I can drive arbitrarily high. So residuosity is *everywhere hard or everywhere easy*: an oracle with a sliver of average advantage yields a near-perfect decider. Contrapositive — under the Quadratic Residuosity Assumption no poly-size circuit even *approximates* Q_n. The predicate is unapproximable. That is the bridge: random self-reducibility converts the worst-case-flavored assumption into the average-case unapproximability my reduction consumes.

Two loose ends for the trapdoor and constructibility. To encrypt a 1 I must sample a random nonresidue with Jacobi +1, but without the factorization I cannot even *recognize* a nonresidue — squaring only gives me residues. The fix: publish, as part of the key, one fixed nonresidue y with Jacobi +1 — a pseudosquare. Then y·(random residue) is a uniformly random nonresidue with Jacobi +1, so the encryptor can sample both bit-values. Does publishing y leak anything? Selecting a nonresidue without the factorization is itself not known to be easy, so I should worry that handing out y compromises the modulus. It does not if y is randomly selected from the Jacobi-+1 nonresidues: a procedure that decides residuosity using such an auxiliary y can be converted into one that decides residuosity without it, by drawing random Jacobi-+1 candidates, using random-square cosets to detect when the auxiliary parameter makes the procedure separate residues from nonresidues, and then invoking the same self-reducibility amplification. So the public pseudosquare is safe. And constructibility holds: flipping 4k coins yields candidate k-bit primes p, q — accepted with good probability by the Prime Number Theorem and a polynomial-time primality test — and a candidate pseudosquare y; the index n#y and its secret (p, q) are generated together, essentially uniformly, in random polynomial time.

So I have everything. Q_n on the Jacobi-+1 elements, with the published pseudosquare y, is an unapproximable trapdoor predicate under the Quadratic Residuosity Assumption: trapdoor-evaluable with (p,q), both bit-values samplable, unapproximable by random self-reducibility, constructible by coin-flipping. Drop it into the bit-by-bit architecture and the scheme is concrete.

Let me write it out. Key generation: pick two k-bit primes p, q; set n = p·q; pick y a Jacobi-+1 nonresidue mod n. Publish (n, y); keep (p, q) secret. To encrypt a bit b, pick a random x in Z_n^*: if b = 0 send x² mod n (a random residue), if b = 1 send y·x² mod n (a random nonresidue with Jacobi +1). To encrypt a message, encrypt each bit independently and send the tuple — one plaintext bit expands to k ciphertext bits. To decrypt each component e, the receiver uses (p, q) to test residuosity: residue means 0, nonresidue means 1.

And the whole chain holds together. Determinism leaks parity, equality, and re-encrypt-and-compare, so encryption must be randomized; the smallest randomized unit is a bit, and the right primitive for one bit is an unapproximable trapdoor predicate, not a trapdoor function — embedding a bit in a function leaks it. Quadratic residuosity mod a composite, restricted to Jacobi +1, is such a predicate: trapdoor by the factorization, unapproximable by random self-reducibility under the residuosity assumption, samplable on both bit-values via a published pseudosquare that is provably safe to reveal. Encrypting bit by bit and stitching the bits with the hypercube hybrid gives indistinguishability of whole messages; and indistinguishability is *equivalent* to semantic security — no efficiently computable function of the plaintext is helped by the ciphertext. So the scheme hides all partial information, for every message distribution, exactly as the card game demanded, and any break of it would decide quadratic residuosity, an algorithm no one has found since Gauss.

```python
from math import gcd

# Number-theoretic primitives (no factorization needed for jacobi).
def jacobi(a, n):
    a %= n; result = 1
    while a:
        while a % 2 == 0:
            a //= 2
            if n % 8 in (3, 5): result = -result   # 2 is a QR mod n iff n = +-1 mod 8
        a, n = n, a
        if a % 4 == 3 and n % 4 == 3: result = -result   # quadratic reciprocity
        a %= n
    return result if n == 1 else 0

def is_qr_mod_prime(a, p):           # Euler's criterion: a^((p-1)/2) = 1 mod p
    return pow(a % p, (p - 1) // 2, p) == 1

# KeyGen: public (n, y) with y a Jacobi-+1 nonresidue (pseudosquare); secret (p, q).
def find_pseudosquare(n, p, q):      # nonresidue mod n with Jacobi +1 = nonresidue mod BOTH primes
    import random
    while True:
        y = random.randrange(2, n)
        if gcd(y, n) == 1 and not is_qr_mod_prime(y, p) and not is_qr_mod_prime(y, q):
            return y                 # safe to publish: revealing a random QNR keeps QR hard

# Encrypt one bit: b=0 -> random QR (x^2); b=1 -> random Jacobi-+1 QNR (y*x^2). Randomized.
def encrypt_bit(b, n, y):
    import random
    x = random.randrange(1, n)
    while gcd(x, n) != 1: x = random.randrange(1, n)
    return (pow(y, b, n) * pow(x, 2, n)) % n

# Decrypt one bit with the trapdoor (p, q): QR mod n iff QR mod p and mod q. QR->0, QNR->1.
def decrypt_bit(e, p, q):
    return 0 if (is_qr_mod_prime(e, p) and is_qr_mod_prime(e, q)) else 1

# Whole message = independent per-bit encryption; the hypercube hybrid lifts per-bit
# indistinguishability to the whole tuple, and indistinguishability = semantic security.
def encrypt(bits, n, y):  return [encrypt_bit(b, n, y) for b in bits]
def decrypt(ct, p, q):    return [decrypt_bit(e, p, q) for e in ct]
```
