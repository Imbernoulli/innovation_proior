I need to stop treating language understanding as a collection of separate supervised products. The usual recipe works only after I choose one task, collect labels for it, and specialize the model or at least specialize a head. Even the stronger transfer recipe still ends with supervised fine-tuning. That means the system is not really general; it is a reusable initialization followed by many small task systems.

The one objective I can train at scale without labels is language modeling. It estimates `p(x)` by the chain rule, `product_i p(s_i | s_1, ..., s_{i-1})`. A task looks different at first because it asks for `p(output | input)`, and a general system should really model `p(output | input, task)`. But the task name, the input, and the output can all be written in language. If the prefix says "translate to French" and then gives English text, the desired French text is just a continuation. If the prefix gives a passage, a question, and an answer marker, the answer is again just a continuation.

Let me make sure this is more than a slogan and actually look at the loss. Take a single sequence `(t_1, ..., t_k, o_1, ..., o_m)` where the first `k` tokens are a task description plus an input and the last `m` are the desired output. The language-modeling loss over the whole sequence is `-sum_i log p(s_i | s_{<i})`. The terms split into two groups: positions `1..k`, which score the prefix, and positions `k+1..k+m`, which score the output conditioned on the prefix. That second group is exactly `-sum_j log p(o_j | t_1, ..., t_k, o_1, ..., o_{j-1})`, which is `-log p(output | input, task)` written out by the chain rule. So the supervised conditional loss for the task is literally a sub-sum of the unsupervised loss on the same string — not an analogy, the same numbers. Minimizing the whole-sequence loss therefore also pushes down the output-span loss, with the only difference being that the prefix terms are also being optimized rather than ignored.

I should be careful not to read more into this than the arithmetic gives. It says the objective contains the task loss as a summand; it does not say a finite model trained on a finite corpus will reach the optimum, and it gives nothing at all if the training text never contains natural demonstrations of the task — in that case the conditional `p(output | input, task)` is being estimated from no relevant data and the decomposition is vacuous. So the objective is not the obstacle. The obstacles are coverage (does the text contain enough demonstrations), capacity (can the model fit them), and optimization.

So the data becomes the first real wall. A single-domain corpus can make a strong language model for that domain, but it will not contain enough forms of translation, summarization, question answering, cloze completion, explanations, tables, and dialogue. The web has that variety, but raw web crawls are full of boilerplate, gibberish, duplicates, and pages no person would choose to read. I need a cheap human filter. A link that has been posted to Reddit and received at least three karma has already passed a weak crowd filter: someone found it interesting, useful, or amusing, and others agreed enough to upvote it. That gives me a way to scrape broad web text without starting from all of Common Crawl.

The corpus should be the text extracted from those outbound links, not the HTML around them. I can use content extractors, deduplicate, and do heuristic cleaning. I also need a cutoff so later evaluation datasets cannot leak backward through fresh links. Wikipedia must be removed because it is a common source for language-modeling and question-answering benchmarks; even if removing it costs useful text, it makes the zero-shot claim cleaner. The result I want is not a task dataset but a broad field of naturally occurring demonstrations.

Now I need the model to read any benchmark without task-specific preprocessing. Lowercasing, word tokenizers, and unknown tokens are all the wrong interface because they make the string space depend on preprocessing choices. Bytes solve coverage: UTF-8 can represent any Unicode string with a base alphabet of 256. But pure byte-level language modeling spends too much sequence length and capacity on low-level spelling and encoding patterns. I want byte coverage with larger useful units.

Byte-pair encoding turns frequent adjacent symbols into longer symbols while leaving rare material decomposable, so it sits between the two extremes. The question is what the base alphabet should be. Standard BPE is usually applied to Unicode code points, and I should check whether that is even affordable as a starting alphabet. If I demand full Unicode coverage before any merges, the base vocabulary is the set of distinct code points I might see. Counting the assigned, non-surrogate code points in Unicode gives 287,346; even restricting to what actually appears across a large multilingual web corpus leaves well over a hundred thousand distinct characters. Either way the base alphabet alone dwarfs the whole vocabulary budget I want for the model (tens of thousands of tokens), before a single useful merge is learned. So code points are the wrong base. Bytes are: there are exactly 256 of them, every string decomposes into bytes losslessly, and merges build everything above that. The fix is to run BPE over byte strings rather than code-point strings, mapped first to printable characters so the merge machinery never has to handle raw whitespace or control bytes.

There is a second trap, and I can see it by tracing what greedy BPE would do on raw byte strings. Consider "dog. dog! dog?". Nothing stops a merge from joining the word with the punctuation that follows it, so the learner would happily mint separate tokens for "dog.", "dog!", and "dog?". Those are not lexical discoveries; they are punctuation attachment, and they waste a vocabulary slot per punctuation variant of every common word. The fix is to forbid merges from crossing most character-category boundaries by pre-splitting the text with a regex: letters with letters, numbers with numbers, punctuation with punctuation, whitespace as its own pattern. Let me actually run a candidate regex on that string and on a few others to confirm it splits where I want. On "dog. dog! dog?" it yields `['dog', '.', ' dog', '!', ' dog', '?']` — the word is cleanly separated from the punctuation each time, so no "dog." token can ever form. On "I have 42 cats" it yields `['I', ' have', ' 42', ' cats']`, keeping digits in their own run away from letters. The one deliberate exception I want is a leading space glued to the following word, because English word boundaries are highly informative and it improves compression, and the trace shows exactly that: ` have`, ` cats`, ` dog` each carry their leading space rather than splitting it off. (On "don't stop" the apostrophe-contraction rules fire too: `['don', "'t", ' stop']`.) So the regex does what I claimed without my having to hand-wave it.

I still owe one detail from the "map bytes to printable characters" step. The BPE and regex machinery operates on strings, so I cannot feed it raw bytes whose characters are spaces, newlines, or control codes — those would interact with the whitespace-splitting regex and corrupt the boundaries I just set up. I need a bijection from all 256 byte values to printable, non-whitespace characters. Building one (reuse the printable Latin-1 ranges where they are already safe, and shift the remaining bytes up into a free block) and then checking it: it covers all 256 byte values, every value maps to a distinct character, and none of the 256 target characters is whitespace or a control category. The reserved bytes land somewhere visible too — byte 32 (space) becomes `Ġ` and byte 10 (newline) becomes `Ċ`, both ordinary printable letters. So the mapping is reversible and inert to the regex, which is exactly the property the rest of the pipeline assumed.

With that pinned down the vocabulary size is determined, not chosen by feel. The base alphabet is the 256 byte characters. To that I add a fixed number of learned merges — I will allocate 50,000 — plus one special end-of-text token to mark document boundaries. That is `256 + 50000 + 1 = 50257` tokens, with no unknown token and no lossy normalization. The "50,257" is not a round magic number; it is exactly what those three counts sum to.

The architecture can start from the decoder-only Transformer used for generative pretraining. It already has causal masked self-attention, learned token and position embeddings, a feed-forward sublayer with a four-times wider hidden dimension, and a tied output projection through the token embedding matrix. For language modeling, I do not need an encoder or cross-attention. I need a larger stack and a longer context.

Depth is where the architecture needs the most care. In the older Transformer block, layer normalization sits after the residual addition: `x = norm(x + attn(x))`. That means the residual stream is renormalized at the output of every block, so across many layers there is no clean additive path from input to output — each layer's signal is immediately rescaled away. Work on very deep residual networks points the other way: keep the residual stream an undisturbed additive path and normalize only the input each sub-block reads. That gives `x = x + attn(norm(x))` and then `x = x + mlp(norm(x))`, with `x` flowing through the stack by addition alone.

Reordering like this has a consequence I have to fix: if every block only normalizes its own input, then after the last residual addition nothing normalizes the final stream before it hits the output projection. The accumulated activations could be at any scale. So I add one final layer normalization after the last block and before the logits. It is not a decorative extra layer; it is the bookend that the pre-norm reordering forces, restoring the single normalization that used to live at each block's output.

Depth changes initialization too, and here I want to be quantitative rather than wave at "the variance grows." A pre-norm residual stack writes the stream as `x_0 + sum_{l=1}^N r_l`, a sum of `N` residual-branch outputs. If each branch output `r_l` is initialized to have roughly unit, independent per-coordinate variance, the variance of the sum is the sum of the variances, so per-coordinate variance grows like `N` and the standard deviation like `sqrt(N)`. Let me check this isn't just a heuristic. Simulating `N = 48` unit-scale branch additions into a stream and measuring the per-coordinate std gives 6.927, and `sqrt(48) = 6.928` — they agree to three digits, so the `sqrt(N)` growth is real, not approximate hand-waving. That is a problem at depth: a 48-layer model would start with a residual stream nearly seven times wider than any single branch, which the final norm would then have to undo and which destabilizes early training. The cure suggested by the same arithmetic is to shrink each branch's initial scale by `1/sqrt(N)`, so the variances sum back to one. Re-running the simulation with each branch scaled by `1/sqrt(N)` gives a per-coordinate std of exactly 1.000 — the accumulated scale is held flat regardless of depth. So I scale the residual-layer weights at initialization by `1/sqrt(N)`. The one subtlety is what `N` counts: each block contributes two residual branches (attention and MLP), so `N` is the number of residual layers in the stack, not the number of Transformer blocks.

The actual model family should test whether capacity is the limiting variable. I keep the smallest model at the previous scale, 12 layers and width 768. Then I make log-spaced larger models: 24 layers at width 1024, 36 at width 1280, and 48 at width 1600. I increase the context from 512 to 1024 tokens because web documents and long-dependency benchmarks need more prefix. I tune only the learning rate per size on held-out corpus text, not on the downstream tasks, because downstream tuning would break the zero-shot claim.

I also need to keep evaluation honest. For language-modeling benchmarks I can compute the likelihood of the benchmark text directly because the tokenizer is reversible and does not require dataset-specific tokenization. For cloze and multiple-choice tasks I can score the candidate continuations under the language model. For generation-style tasks I can prompt with natural text markers, such as an answer marker or a short translation pattern, and decode without changing weights. The prompt is allowed because it is just context; learned task heads are not.

The risk is memorization. A broad web corpus will overlap with public benchmarks in small and annoying ways. Removing Wikipedia helps but does not prove enough. I need n-gram overlap checks between training text and benchmark test text, and I need to interpret gains under that shadow. If overlap changes the metrics only slightly, then the result is still a zero-shot transfer result rather than a memorized-test result. If overlap is large, the benchmark has to be reported differently.

The final shape is therefore simple but specific. Train a large causal Transformer on a Reddit-filtered, deduplicated, non-Wikipedia web corpus using a byte-level BPE tokenizer that preserves arbitrary strings and avoids wasteful cross-category merges. Use pre-layer-normalized residual blocks, a final layer normalization, a longer context, and residual-depth-aware initialization. Then evaluate by prompting or scoring continuations, with no supervised updates. The bet is not that a new task module appears. The bet is that many tasks are already written inside natural text, and a sufficiently strong next-token predictor must model those task-conditioned continuations to model the text well.
