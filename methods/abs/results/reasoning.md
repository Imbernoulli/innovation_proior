Here is the box I am in. I am fine-tuning a language-model policy `π_θ` with RL, and I need a leash to a frozen reference `π_ref`; otherwise the reward can pull the policy toward degenerate text. The principled leash is `KL(π_θ ‖ π_ref)`. But the exact per-token KL is `Σ_v π_θ(v) log(π_θ(v)/π_ref(v))`, a full-vocabulary sum at every generated position, and the actor-loss path does not have that sum materialized. It has two tensors at the sampled response tokens: `log π_θ(x)` and `log π_ref(x)`. Whatever I build has to return one differentiable number per token, then the trainer masks it, averages it, scales it by the KL coefficient, and adds it to the actor loss.

Let me lock the signs before I touch any estimator. For a sampled token `x ∼ π_θ`, write `l = log π_θ(x) − log π_ref(x)`. Then `r = exp(l) = π_θ/π_ref` and `s = exp(−l) = π_ref/π_θ`. The target is `KL(π_θ ‖ π_ref) = E_{x∼π_θ}[l] = E_{π_θ}[−log s]`. The simplest per-sample value is therefore just `k1 = l`: average it over on-policy samples and it is unbiased.

But a per-token loss is not only a report. If I add `k1` to the loss and differentiate the sampled value, `∇_θ k1 = ∇_θ log π_θ(x)` because the reference is frozen. The expectation of that score is zero: `E_{π_θ}[∇_θ log π_θ(x)] = ∇_θ Σ_x π_θ(x) = 0`. So the value estimate is honest, but the actor receives zero-mean score noise instead of a restoring KL gradient. There is also the plain variance problem: `l` is negative on tokens the current policy assigns lower probability than the reference, while the KL mean is non-negative. A signed per-token "penalty" is frequently a reward, and the cancellation happens only after averaging. Wall.

The control-variate route is tempting because `s` gives me one exact mean-zero object. Under `x ∼ π_θ`, `E[s] = Σ_x π_θ(x) π_ref(x)/π_θ(x) = 1`, so `s − 1` can be added to `l = −log s` without changing the mean. I can write `−log s + λ(s − 1)` for any `λ`. The variance-minimizing `λ` would depend on the distributions, so I want a fixed choice that buys something structural. Since `log s ≤ s − 1` for every `s > 0`, choosing `λ = 1` gives `k3 = (s − 1) − log s = (s − 1) + l`, always non-negative and still unbiased for `KL(π_θ ‖ π_ref)`. In code this branch needs `s = exp(−l)`, so the exponent input has to be clamped before the exponential.

The loss gradient is exactly where the trap can hide. With `s = exp(−l)`, `∇_θ s = −s ∇_θ log π_θ(x)`, and `∇_θ(−log s) = ∇_θ l = ∇_θ log π_θ(x)`. Therefore `∇_θ k3 = (1 − s) ∇_θ log π_θ(x)`. Taking expectation under `π_θ`, the first term vanishes by the score identity and the second term leaves `−Σ_x π_ref(x) ∇_θ log π_θ(x)`, which is exactly `∇_θ KL(π_ref ‖ π_θ)`, not `∇_θ KL(π_θ ‖ π_ref)`. The control variate is harmless for the value, but once I differentiate the sample value it changes the divergence that the actor is pushed to minimize. That makes `k3` excellent as a low-variance KL estimate and questionable as a direct actor-side loss for the forward KL I wrote down.

The other smooth option is to fold the sign by squaring: `k2 = ½ l²`. It is non-negative, and near the reference it has the same local curvature as KL. In terms of `s`, the scalar function is `f(s)=½(log s)^2`; `f'(s)=log(s)/s`, `f''(s)=(1−log s)/s²`, so `f''(1)=1`, the same second-order coefficient as `−log s`. That explains why the squared estimator can be close when the policies are close. More importantly for the loss path, differentiating the sampled value gives `∇_θ k2 = l ∇_θ log π_θ(x)`. Since `∇_θ KL(π_θ ‖ π_ref) = Σ_x ∇_θ π_θ(x) l(x) + Σ_x π_θ(x) ∇_θ log π_θ(x)` and the second sum is the zero score, the expectation `E_{π_θ}[l ∇_θ log π_θ(x)]` is the forward-KL gradient. The signed estimator has the right value but the wrong expected loss gradient; the squared estimator has a biased value but the right expected loss gradient.

So why not stop at `k2`? Because long reasoning rollouts do not give me tame log-ratios. Most tokens may sit near zero, but a few rare numerals, formatting choices, or branch-changing tokens can have large `|l|`. The squared penalty gives such a token value `½ l²`, and its gradient weight is `l`. One sampled token with `l = 8` contributes `32` to the scalar penalty and an actor-gradient weight of `8`. That is exactly the squared-loss failure mode: the farther a point is in the tail, the more authority it gets over the update. I still want a non-negative penalty and a restoring gradient, but I do not want the tail to get amplified just because the sample was extreme.

The familiar way to keep a residual from gaining quadratic authority is to stop squaring it and look at its magnitude. I should not assume that this inherits the nice KL geometry, so I check it directly. The candidate value is `|l| = |log π_θ(x) − log π_ref(x)|`. It is non-negative pointwise. Its expectation is biased upward for the target KL: by Jensen's inequality, `E[|l|] ≥ |E[l]| = KL(π_θ ‖ π_ref)` because the KL is non-negative. The bias is real, but the tail growth is only linear in `|l|`, not quadratic.

Does this expectation come from a clean `f`-divergence? Using `s = π_ref/π_θ`, `|l| = |log s|`. On `(0,1]`, `|log s| = −log s`, which is convex. On `[1,∞)`, `|log s| = log s`, which is concave. An `f`-divergence needs a convex scalar function over the whole positive ratio domain, and this one is not globally convex on `(0,∞)`. I cannot borrow the second-order Fisher story that made `k2` attractive. The justification has to be robustness in the actor-loss path, not a divergence pedigree.

The loss path decides whether this is merely a nicer value or a useful actor penalty. Differentiating the sampled value gives `∇_θ |l| = sign(l) ∇_θ log π_θ(x)`, with the zero subgradient at `l=0`. This is the shape I was missing. Where the current policy puts more probability on the sampled token than the reference (`l > 0`), minimizing the loss pushes that log-probability down with unit scalar weight. Where it puts less probability than the reference (`l < 0`), minimizing the loss pushes it up with unit scalar weight. The sign supplies the restoring direction that `k1` lacks, while the scalar multiplier is bounded in a way `k2` is not. A token with `|l| = 8` no longer gets eight times the multiplier of a token with `|l| = 1`; both cast one sign vote.

That makes the tradeoff clear. `k1` is value-unbiased but its differentiated loss averages to the zero score. `k3` is value-unbiased and low-variance, but its differentiated loss follows the reverse-KL gradient `(1−s)∇log π_θ`. `k2` has the correct forward-KL gradient `l∇log π_θ`, but the weight is unbounded and the scalar penalty is quadratic. The absolute log-ratio accepts upward value bias and gives up the `f`-divergence story, but it returns a non-negative per-token penalty with the bounded actor-gradient weight `sign(l)`. For the actor-side KL loss on heavy-tailed token log-ratios, that bounded restoring force is the property I need.

Numerically, this is also the safest branch. There is no exponential and therefore no exponent input to clamp. The implementation should be exactly the subtraction plus `.abs()` that defines the value, and it must stay in the autograd graph because the `sign(l)` subgradient is the whole regularizing mechanism:

```python
import torch


def compute_kl_penalty(
    logprob: torch.Tensor,
    ref_logprob: torch.Tensor,
) -> torch.Tensor:
    # l = log pi_theta - log pi_ref at sampled tokens.
    # The result is differentiable; do not wrap this branch in torch.no_grad().
    return (logprob - ref_logprob).abs()
```

The chain lands here: I start with only two sampled log-probabilities and a forward-KL actor loss. The signed estimator has the right mean but a zero expected sampled-loss gradient. The control-variate estimator has the right mean and low variance but changes the actor gradient to reverse KL. The squared estimator has the right expected forward-KL gradient but lets large-log-ratio tokens dominate. Taking the absolute value of `log π_θ − log π_ref` is the robust compromise: biased upward as a value estimate, not an `f`-divergence, but cheap, non-negative, autograd-friendly, and bounded in per-token gradient weight.
