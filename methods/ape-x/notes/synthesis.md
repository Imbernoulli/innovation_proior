# Ape-X synthesis (Distributed Prioritized Experience Replay, 1803.00933, verified title)

## Pain point / research question
Deep RL scaling: more compute + data → better results in supervised learning. Standard distributed
training parallelizes GRADIENT computation (Dean 2012 DistBelief). For RL, can we instead scale by
generating and SELECTING more experience data? Single-machine deep RL is well studied but harnessing
many machines for *data generation* is underexplored. Want: SOTA on Atari ALE in a fraction of
wall-clock time, no per-game hyperparameter tuning, also works for continuous control.

## Load-bearing ancestors (verify in paper §2, §3)
- **Distributed SGD** (Dean 2012 async; Krizhevsky 2014 sync): parallelize gradient computation.
  Low-latency communication matters because gradients go stale fast.
- **Gorila** (Nair 2015, "Massively Parallel Methods for Deep RL"): FIRST massively distributed deep RL.
  Bundled mode = one-to-one actor + LOCAL replay memory + learner. Learners compute DQN gradients
  locally, send GRADIENTS asynchronously to a central PARAMETER SERVER; Q-nets periodically synced.
  Safeguards: discard gradients older than a staleness threshold; discard gradients whose abs loss
  > mean + k·std. KEY CONTRAST for Ape-X: Gorila distributes gradient computation with per-bundle
  *local* replay sampled UNIFORMLY; Ape-X distributes only data generation/selection, with a SINGLE
  CENTRALIZED SHARED prioritized replay and a SINGLE learner. Sharing experience instead of gradients
  → low latency far less critical (experience goes stale slower than gradients if learner is robust
  to off-policy data), so can batch all comms, even cross-datacenter.
- **A3C** (Mnih 2016): single-machine multi-threaded, async parameter updates + parallel data gen,
  on-policy n-step actor-critic. GA3C / PAAC adapt to GPUs. (on-policy → can't reuse data)
- **Experience replay** (Lin 1992); **DQN** (Mnih 2015): sliding-window 1M buffer, uniform minibatch
  of 32, target network, off-policy → can reuse data generated by previous policies, reduces overfit.
- **Prioritized experience replay** (Schaul 2016): sample ∝ p_k^α / Σ p_j^α, p_k = |δ_k| (proportional)
  or 1/rank. IS weight w = (N·P)^{-β} normalized by max, anneal β→1. New transitions enter at MAX
  priority, updated only once sampled. Rainbow ablation (Hessel 2018) found prioritization the single
  most important ingredient. SUM-TREE for O(log N) sample + update; MIN-TREE for max weight.
- **Double Q-learning** (van Hasselt 2010, 2016): decouple action selection (online) from evaluation
  (target) to reduce overestimation. Multi-step bootstrap targets (Sutton 1988).
- **Dueling network** (Wang 2016): Q = V + (A − mean_a A). Used as function approximator.
- **DDPG** (Lillicrap 2015) / DPG (Silver 2014): off-policy actor-critic for continuous control;
  deterministic actor μ_φ(s), critic q_ψ(s,a), actor updated by ∇_φ q(s, μ_φ(s)). Used for Ape-X DPG.
- **Distributed importance sampling** (Hinton 2007; Alain 2015; Loshchilov 2015): non-uniform sampling
  by gradient L2 norm or by rank-of-loss + IS correction → variance reduction, faster convergence.

## The method (Ape-X), verify §3
Decouple ACTING from LEARNING (like Gorila) but:
1. Centralized SHARED prioritized replay (not per-bundle local uniform). High-priority data found by
   ANY actor benefits whole system.
2. SINGLE learner on GPU samples prioritized batches; hundreds of actors on CPU generate data.
3. Actors COMPUTE INITIAL PRIORITIES ONLINE for new transitions (using their local policy-eval Q-values
   they're already computing — free). WHY: with many actors, the Schaul trick of inserting new data at
   max priority and waiting for the learner to update it does NOT scale — would cause myopic focus on
   newest data (which has max priority by construction). So compute |n-step TD error| on the actor.
4. Off-policy → give different actors DIFFERENT exploration: actor i uses ε_i-greedy with
   ε_i = ε^{1 + (i/(N−1))·α}, ε=0.4, α=7, constant per actor throughout training. Low-ε actors explore
   deep; high-ε prevent over-specialization. Diversity of behavior + prioritization to pick best.
5. Batch all communication with centralized replay (latency OK because experience stale slowly).
6. Actors periodically copy parameters from learner (every 400 frames ≈ 2.8s).
7. Learner: every 100 learning steps removes excess data above capacity in FIFO order (Atari soft cap
   2M). Continuous-control variant evicts via *proportional* sampling with α_evict = −0.4.

### Ape-X DQN loss (verify exact form §3.1, code)
l_t(θ) = ½ (G_t − q(S_t, A_t, θ))²,
G_t = R_{t+1} + γ R_{t+2} + … + γ^{n−1} R_{t+n} + γ^n q(S_{t+n}, argmax_a q(S_{t+n}, a, θ), θ⁻)
double-Q bootstrap, n-step, θ⁻ = target net. Multi-step returns truncated at episode end.
Code uses Huber on the TD error (loss = 0.5δ² if |δ|<1 else |δ|−0.5), weighted by IS weights.
Priority p = |G_t − q(S_t,A_t)| + 1e-6. Actor computes it with its stored q(S_t,*), q(S_{t+n},*).

### Ape-X DPG (verify §3.2, appendix)
Separate policy net π(s,φ) and Q-net q(s,a,ψ). Q loss: l(ψ)=½(G_t − q(S_t,A_t,ψ))²,
G_t = R_{t+1}+…+γ^{n−1}R_{t+n}+γ^n q(S_{t+n}, π(S_{t+n}, φ⁻), ψ⁻). Policy ascent on ∇_φ q(S_t,π(S_t,φ),ψ).
Gaussian exploration noise σ=0.3 (not OU). Critic 400→tanh→300, actor 300→tanh→200, Adam lr 1e-4,
actor grad clipped [−1,1] elementwise, target copy every 100 batches, α_sample=0.6, α_evict=−0.4.

  from prioritized-replay's anneal; treat as a deliberate simplification at scale.)

## Design-decision → why
- Distribute DATA not gradients: gradients go stale fast (need low latency); experience stays useful
  longer if learner robust to off-policy → can batch comms, tolerate latency, run cross-datacenter.
  Also complementary to gradient-distribution, can combine later.
- SHARED centralized replay (vs Gorila per-bundle local): priorities are global, so a rare high-value
  discovery by any one actor is immediately exploitable by the learner for the whole system.
- Actor-side initial priorities: at scale, the "insert at max priority, fix on first sample" trick
  starves old data — too many fresh max-priority transitions flood the top; learner can't keep up →
  myopic focus on newest. Actors already evaluate their policy (compute Q), so |TD error| is free.
- Different ε per actor (held constant): one schedule can't both explore deeply and avoid
  over-specialization; a *spread* does both at once, and off-policy learning + prioritization lets the
  learner cherry-pick whichever behaviors paid off. ε_i = ε^{1+...α} concentrates most actors near
  small ε with a few large — geometric spacing.
- n-step return, no off-policy correction: bias acknowledged but empirically fine; speeds value
  propagation. Multi-step + double-Q + dueling = the strong Rainbow-flavored base.
- cut lr by 4 (0.00025→6.25e-5): prioritization picks high-|δ| transitions more often → larger typical
  gradient → must lower lr to compensate (same reasoning as prioritized-replay). Confirmed: paper uses
  0.00025/4. Larger batch (512) helped when many actors (tried 32..1024, benefit up to 512).
- single GPU learner: data generation is the bottleneck being scaled; one learner suffices because
  the number of learning updates is fixed regardless of actor count (key scalability fact).
- soft replay cap + lazy bulk FIFO eviction: never block actors (always allow add), evict in bulk
  occasionally → throughput.

## Self-derivation math checks
- ε_i: i=0 → ε^1 = 0.4 (greediest actor uses ε=0.4? No — wait). i=0 → exponent 1+0 = 1 → ε_0=0.4.
  i=N−1 → exponent 1+α=8 → ε^8 = 0.4^8 ≈ 6.5e-4 (most greedy). So actor 0 is MOST exploratory (0.4),
  actor N−1 most greedy. Confirmed consistent with appendix evaluator using ε=0.00164 ≈ greediest.
  0.4^7 ≈ 0.0016 — matches the evaluator footnote roughly (they use ε^{1+α} at N−1, ≈0.4^8). OK.
- n-step double-Q: G = Σ_{k=0}^{n-1} γ^k R_{t+1+k} + γ^n q(S_{t+n}, argmax q(·,θ), θ⁻). code uses
  rewards (already multi-step-discounted on actor) + γ^n · next_q_a · (1−done). Consistent.
- IS weight: w_i = (N·P(i))^{−β} / max_j (N·P(j))^{−β}; max weight from min P (min-tree). Confirmed in code.
