OK, let me start from what's actually scarce. I have a value-based deep-RL agent — Q-learning with a deep network, a replay buffer, a target net — and on Atari it works, but it takes ten days on a single GPU to reach the state of the art on 57 games. Meanwhile in supervised learning the story of the last few years is brutally simple: more compute, bigger models, more data, better results, and people scale all three by throwing machines at the problem. The obvious thing to ask is whether deep RL has the same knob. And the examples are right there — Gorila, A3C, GPU A2C, distributed PPO, AlphaGo all got their wins substantially from using more machines, not from a cleverer loss. So the prevailing single-machine research framing feels like it's leaving the biggest lever untouched. Let me try to actually pull that lever.

How does the rest of the field scale a neural network across machines? Distributed SGD. You replicate the model on many workers, each computes a gradient on its shard of data, and you push those gradients to a parameter server that applies them — asynchronously (Dean et al. 2012, DistBelief) or synchronously (Krizhevsky 2014). The thing being parallelized is *gradient computation*. And the thing that always bites you is staleness: a worker computes ∇L(θ_old), but by the time it lands at the server the parameters have moved, so you're applying a gradient evaluated at the wrong point. The more workers and the higher the latency, the staler the gradients, the worse the estimate. So distributed SGD lives or dies on low-latency, tightly-synchronized communication.

Gorila (Nair et al. 2015) took exactly this template and dropped deep RL into it. In its bundled mode you replicate a full unit many times: an actor in its own copy of the environment, *its own local replay memory*, and a learner. Each learner samples from its local replay, computes the DQN gradient, and ships that gradient to a central parameter server; the Q-networks in actors and learners get periodically re-synced from the server. And because it's gradient-passing it inherits all of distributed SGD's fragility — Gorila has to discard gradients that are too old, and even discard gradients whose loss is anomalously large, just to keep asynchronous gradient aggregation from blowing up.

Let me stare at that bundled design, because something about it feels wasteful and I want to name what. Two things bother me. First, each bundle has its *own* replay, sampled uniformly, and the bundles only ever talk to each other through the parameter server. So if actor 7 stumbles into a rare, hugely informative transition — say it finally completes the precise sequence that unlocks a reward — that transition sits in bundle 7's local memory and bundle 7's local memory only. No other learner ever sees it. The single most valuable piece of data in the whole system is quarantined. Second, the quantity being passed around is gradients, which is precisely the quantity that goes stale fastest and forces the low-latency, tight-sync regime.

So what if I flip both? Suppose the thing I distribute isn't gradient computation at all, but *data generation* — and what gets shared isn't gradients but *experience*. Before I commit to that, I should pin down whether experience even can be shared cheaply, because the whole bet rests on it. A transition (S, A, R, S′) computed by an actor a few seconds ago: is it stale? Only to the extent that the *behavior policy* that generated it has drifted from the current policy — but I'm doing off-policy Q-learning, which is built to learn from data generated by other policies. Let me sanity-check the magnitude against something I already trust: DQN's own replay buffer holds a million transitions, and at ~one update per four frames it generates and consumes them slowly enough that the oldest transitions in that buffer were laid down by a policy on the order of 10⁵–10⁶ updates ago — and DQN learns fine from them. So a transition that is merely a few seconds and a handful of parameter-refreshes old is, by comparison, almost fresh. That's the asymmetry: a gradient is evaluated at a specific θ and is wrong the instant θ moves, whereas a transition stays a valid sample of the environment dynamics no matter how θ moves, decaying only through behavior-policy drift that off-policy learning already absorbs. Gradients rot in milliseconds; experience keeps for minutes. If I'm shipping experience instead of gradients, low latency stops mattering — I can batch all my communication into big chunks, eat the latency, and not care, even across datacenters.

That reshapes the architecture. Keep Gorila's decoupling of acting from learning, but drop the part where every bundle has a private uniform replay shipping gradients to a server. If experience is the shared currency, the natural shape is **many actors all feeding one shared replay** and **one learner** reading from that replay and doing all the gradient work — hundreds of cheap CPU actors generating experience as fast as they can, a single GPU learner consuming it. And this directly answers the quarantine complaint: with one replay there is no bundle 7's private memory for a rare transition to be trapped in.

Now I have to decide how the learner picks what to train on, because it's drinking from a firehose — hundreds of actors at ~50K frames per second between them. It cannot and should not train on all of it; the learner's update rate is fixed by its single GPU, independent of how many actors I add. So out of this enormous incoming stream the learner must *select* the most useful transitions. The tool I'd reach for is prioritized replay (Schaul et al. 2016): sample transition k with probability P(k) = p_k^α / Σ_j p_j^α, priority p_k = |δ_k| the absolute TD error, so the learner spends its limited updates on the transitions it's most wrong about. Correct the sampling bias with an importance weight w_k = (N·P(k))^{−β}, normalized by the largest possible weight in the current replay so weights only ever scale steps down. Sum-tree for O(log N) sampling and update, min-tree for the smallest sampling probability and therefore the max weight.

Let me actually run those formulas on a toy buffer to make sure they do what I'm claiming and that the bias correction has the sign I think it does. Take four transitions with priorities (5, 1, 1, 0.5) — one transition the learner is badly wrong about, three it mostly understands — and use α = 0.6, β = 0.4. Then p^α = (2.627, 1, 1, 0.660), and dividing by the sum 5.286 gives P = (0.497, 0.189, 0.189, 0.125). So the high-error transition is sampled with probability 0.497 against 0.125 for the most-confident one — a ratio of 3.98×, close to (5/0.5)^0.6 = 10^0.6 ≈ 3.98, which is the right ballpark for α compressing the raw 10× priority ratio. Good, oversampling-by-surprise works as intended. Now the weights: (N·P)^{−β} with N = 4 gives raw (0.760, 1.118, 1.118, 1.320), and the most-sampled (highest-P) transition gets the *smallest* weight, 0.760 — exactly the down-weighting that has to happen to undo the oversampling. Normalizing by the max, 1.320 (which lands on the least-probable transition, as the min-tree promises), gives w = (0.575, 0.847, 0.847, 1.0), all ≤ 1, so every weight only ever scales a gradient step down, never up. That matches the intent, and it confirms the min-tree is exactly the structure I need: the maximum weight is governed by the *minimum* sampling probability, so a min-tree over p^α gives it in O(log N).

And it's not a side feature — the Rainbow ablation found prioritization the single most important ingredient in a value-based agent. In a distributed setting it does something extra: because the replay is *shared*, priorities are *global*. That rare transition actor 7 found doesn't sit quarantined in a local buffer anymore — it lands in the one shared memory, and if its priority is high the toy computation above says the learner draws it ~4× more often than a routine transition. A valuable discovery by *any* actor is something the *whole* system then trains on preferentially. That is the lever that turns "one shared buffer" from a mere convenience into the thing that fixes the quarantine.

Good — but now a scaling problem with prioritized replay surfaces that didn't exist on a single machine, and I want to put numbers on it before deciding it's real. In single-machine prioritized DQN, a brand-new transition has no TD error computed yet, so the rule is: insert it at the *maximum* priority seen so far, and only refine that priority the first time the learner happens to sample it. That guarantees every new transition is seen at least once. On one machine the production rate and the consumption rate are matched — roughly one new transition per learner update — so the pool of un-refined max-priority transitions stays small. Now do the rate arithmetic for the distributed case. Actors produce at ~50K transitions/s between them; a single GPU learner does maybe ~20 batches/s of 512, i.e. ~10⁴ samples/s, and that's the *consumption* ceiling no matter how many actors I add. So new transitions arrive at ~5× the rate the learner can sample anything at all — and every one of those arrivals is stamped at the maximum priority. The set of un-refined, maximum-priority transitions therefore *grows* without bound, and since they all share the top priority the learner's prioritized sampler draws almost exclusively from them, i.e. from the most recent few seconds of data. The "insert at max, refine on first sample" rule doesn't just degrade — at this production/consumption ratio it collapses into "always train on the newest data," which is the opposite of prioritizing by *informativeness*. So the single-machine insertion rule is not something I can carry over; it actively breaks at scale.

So I need real priorities on new transitions *before* they enter the shared buffer — I can't wait for the learner. Where would those come from? The actors are already evaluating their local copy of the network at every step to choose actions, so they compute the Q-values anyway, and the TD error is built from exactly those Q-values. So the actor could compute the initial priority itself, online, from the Q-values it already has in hand, at essentially no extra cost — and then data enters the replay with an informative priority instead of a placeholder maximum, which is precisely what kills the flood-the-top collapse: nothing enters at the maximum anymore, so the un-refined max-priority pool that grew without bound above simply never forms.

But I should check one thing before trusting this, because the actor and the learner don't compute the TD error the same way. The learner's target is double-Q: pick the bootstrap action with the online net, *evaluate* it with the target net θ⁻. The actor has only one network copy, so the best it can do is q(S_{t+n}, argmax_a q(S_{t+n},a), θ_actor) — argmax and evaluation under the same parameters. Are these even in the same ballpark? Take a concrete state with actor Q-values at the bootstrap state of (10, 7, 3). The actor's bootstrap value is just the max, 10. The learner's double-Q would argmax under the online net → index 0, then evaluate index 0 under the target net. Whenever the online and target nets have not drifted apart — which is *exactly* the actor's situation, since it holds a single coherent set of parameters — the evaluated value is also 10. So with rewards (1, 0, 2), n = 3, γ = 0.99 and an action whose q(S_t, A_t) = 5, the actor's return is 1 + 0.99²·2 + 0.99³·10 = 2.960 + 9.703 = 12.663, giving an initial priority |12.663 − 5| = 7.663. The learner, on the same parameters, would land on the same 7.663. They coincide in the no-drift case and differ only by (a) the online/target gap and (b) the actor's parameters being a few refreshes stale — and (b) is exactly the off-policy slack I already decided to tolerate. So the actor's initial priority is a faithful preview of what the learner will compute, not a different quantity. Good enough to insert on.

Now, why bother with hundreds of actors at all if the learner's update count is fixed? If I'm not doing more gradient steps, what does more data even buy me? Let me think about a failure mode that's everywhere in deep RL: the policy settles into a local optimum in parameter space — a behavior that's locally good but globally mediocre — usually because it never explored the action that would have revealed something better. A single ε-greedy actor explores at one fixed rate, and that rate is a compromise: low ε exploits and digs deep into the environment but rarely tries anything new; high ε tries lots of new things but never commits long enough to see a long-horizon payoff. You can't get both from one ε. But I don't have one actor — I have hundreds, and I'm off-policy, so I'm free to let each actor run a *different* behavior policy. Give actor i its own exploration rate. Let a few actors run very low ε so they exploit and probe deep into the environment, and let others run high ε so the population keeps trying fresh things and nobody over-specializes. The whole population sweeps a spectrum of exploration simultaneously, and prioritized shared replay does the rest: when *any* actor's exploration pays off and produces a surprising, high-error transition, it enters the shared buffer at high priority and the learner pounces on it. Diverse generation plus prioritized selection — generate widely, then identify and learn from the most useful events. That's why more actors help even with a fixed learning rate.

What spread of ε do I want across N actors? Most actors clustered at fairly low ε (exploiting, going deep) and a thin tail at high ε (keeping diversity alive), not a uniform spread. A geometric schedule does that: actor i ∈ {0,…,N−1} runs ε_i = ε^{1 + (i/(N−1))·α} with ε = 0.4 and α = 7, held constant for that actor throughout training. Let me check it actually has that shape rather than just assuming it. The endpoints first: at i = 0 the exponent is 1, so ε_0 = 0.4 — the most exploratory actor; at i = N−1 the exponent is 1+α = 8, so ε_{N−1} = 0.4⁸ ≈ 6.6×10⁻⁴ — nearly greedy. The exponent moves linearly from 1 to 8 across the actors, but ε is exponential in it, so the *values* are bunched. Counting for N = 256: the median actor (i = 128) sits at 0.4^4.5 ≈ 0.016, and tallying the whole array, about 78% of actors run ε < 0.1 and 68% run ε < 0.05, while only ~11% run ε > 0.2. So the population really is mostly near-greedy exploiters with a thin exploratory tail — the asymmetric shape I wanted, not a uniform sweep. (And it lines up with how I'd evaluate the trained agent: a near-greedy evaluator at ε ≈ 0.4^7 ≈ 0.0016 sits right alongside the greediest training actors at 0.4^8 ≈ 6.6×10⁻⁴, so the policy I deploy is one the population was actually generating data under.)

Now let me pin down the learner's actual update, because I want it to be a strong value-based rule, not vanilla DQN. The bootstrap target should not take a plain max — that overestimates, because the same noisy network both picks and evaluates the bootstrap action. Decouple them: pick the action with the online net, evaluate it with the target net θ⁻ (double Q-learning). And reward propagates faster if I bootstrap after n steps instead of one, summing n discounted rewards first (multi-step return). And I'll use a dueling head as the approximator, Q = V + (A − mean_a A), since it estimates state value and per-action advantage separately. Put together, the loss for a sampled transition starting at (S_t, A_t) is

  l_t(θ) = ½ ( G_t − q(S_t, A_t, θ) )²,
  G_t = R_{t+1} + γ R_{t+2} + … + γ^{n−1} R_{t+n} + Γ_t · q( S_{t+n}, argmax_a q(S_{t+n}, a, θ), θ⁻ ),

where Γ_t is γⁿ unless the episode ends inside the n-step window, in which case the return is truncated and Γ_t is 0. Let me check the terminal bookkeeping on a concrete case so I don't get it wrong in code: a window of n = 3 with rewards (1, 0, 2) and no terminal gives G = 1 + 0.99·0 + 0.99²·2 + 0.99³·q = 2.960 + 0.970·q, i.e. the bootstrap carries discount γ³; but if the episode ends after the second step, the return truncates to 1 + 0.99·3 = 3.97 with Γ_t = 0 and *no* bootstrap term. So Γ_t = γᵐ where m is the number of real steps before termination, collapsing to 0 exactly when m < n — that's the rule the code has to implement. There's a known sin I should name too: I'm using an n-step return on off-policy data with *no* importance correction for the intermediate actions, which in principle biases the value estimate, growing with n and with how far the behavior policy drifts from the target. At n = 3 I expect that bias to be small relative to the gain in credit-propagation speed, but I can't actually settle that from first principles here — I'd want to confirm it by sweeping n on a few games and watching whether large-n hurts. I'll start at n = 3 and treat it as something to verify, not something proven.

The priority that goes with this transition is its absolute n-step TD error, p = |G_t − q(S_t, A_t)| + ε_p, with a tiny ε_p so nothing ever drops to exactly zero probability. And crucially, the actor can compute the initial version of this error from the Q-value estimates it already produced while acting — it stored q(S_t, ·) and q(S_{t+n}, ·) as it stepped, and uses its own bootstrap estimate for q(S_{t+n}, ·) rather than waiting for the learner's target network. For numerical robustness on the learner I'll regress with a Huber loss on the TD error rather than pure squared error (½δ² for |δ|<1, |δ|−½ otherwise), and weight each transition's loss by its IS weight w_k.

One hyperparameter has to move relative to single-machine prioritized DQN, and it's the same reasoning as before: prioritization deliberately oversamples high-|δ| transitions, so the typical gradient magnitude per batch goes *up*. If I keep the old learning rate the effective rate is now too hot and training destabilizes. So cut the learning rate. The single-machine value was 0.00025; I'll use 0.00025/4 = 6.25×10⁻⁵. (I'll also use a large batch — with this many actors feeding data, bigger batches help; sweeping {32,…,1024} the benefit holds up to 512, beyond which raising the rate to compensate destabilized some games. So batch 512, centered RMSProp, decay 0.95, ε 1.5e-7, no momentum, gradient norm clipped to 40, target net copied every 2500 learner batches.)

Let me now think about the plumbing, because the whole bet was that sharing experience tolerates latency. Each actor batches its experience locally and ships it to the shared replay in chunks rather than one transition at a time — I buffer up to ~100 transitions and send in batches of B = 50 — so the replay server isn't hammered with tiny requests. The learner prefetches batches asynchronously (up to 16 batches of 512) so the GPU is never idle waiting on the network; while the GPU computes gradients, a background thread is already pulling and decompressing the next batches. Observations are PNG-compressed in transit and in storage to save bandwidth and RAM. Actors refresh their copy of the parameters from the learner only periodically — every 400 frames, a couple of seconds — which is fine precisely because slightly-stale behavior just means slightly-off-policy data, which I tolerate. None of this needs the millisecond-scale synchronization that gradient-passing would have demanded; I've traded latency for throughput everywhere.

The shared replay has finite capacity, and with hundreds of actors it fills and churns fast. I never want to block an actor's add (that would stall data generation, the thing I'm scaling), so adds are always accepted, and only every ~100 learner steps does the buffer bulk-evict everything above its capacity in FIFO order. On Atari I soft-cap it around 2 million transitions.

Let me also confirm the framework isn't secretly DQN-specific, because if "decouple acting from learning, share a prioritized replay, compute priorities on the actor" is a real architecture it should accept any off-policy update. Swap the DQN learner for a continuous-control one: a DDPG-style actor-critic. Now an actor carries an explicit policy network π(s,φ) in addition to the critic q(s,a,ψ). The critic trains by the same multi-step TD regression, except the bootstrap action comes from the (target) policy instead of an argmax:

  G_t = R_{t+1} + … + γ^{n−1} R_{t+n} + Γ_t · q( S_{t+n}, π(S_{t+n}, φ⁻), ψ⁻ ),

and the policy is updated by ascending the critic, ∇_φ q(S_t, π(S_t, φ), ψ) — the gradient that flows back into φ only through the action π(S_t, φ) fed into the critic. Now let me actually account for what had to change versus what didn't, because that's the test of whether this is an architecture or a DQN-shaped accident. What changed: the bootstrap action source (target policy instead of online argmax), an added policy-gradient update, and the exploration mechanism (additive Gaussian action noise instead of per-actor ε). What did *not* have to change: the shared prioritized replay, the priority being the critic's absolute TD error, the actor computing that priority from the Q-values it already has, the multi-step return and its terminal discount, the batched latency-tolerant communication, the parameter-publishing loop. The pieces I touched are all confined to the learning rule and the action distribution; the entire acting/learning/replay scaffolding is untouched. That separation — the scaffolding being indifferent to which off-policy update sits inside it — is what tells me I built an architecture rather than a one-off DQN trick.

Let me put the pieces down as real code, organized exactly as the architecture splits the work: a model, an actor-side n-step storage that also emits the initial priority, a shared prioritized buffer (the sum-tree/min-tree machinery from prioritized replay), the learner's loss, and the two loops.

The dueling network the actors and learner share, with a Q-value call that the actor reuses both for action selection and for initial priorities:

```python
import random
import numpy as np
import torch
import torch.nn as nn


class DuelingDQN(nn.Module):
    def __init__(self, env):
        super().__init__()
        self.input_shape = env.observation_space.shape
        self.num_actions = env.action_space.n
        self.features = nn.Sequential(
            nn.Conv2d(self.input_shape[0], 32, kernel_size=8, stride=4), nn.ReLU(),
            nn.Conv2d(32, 64, kernel_size=4, stride=2), nn.ReLU(),
            nn.Conv2d(64, 64, kernel_size=3, stride=1), nn.ReLU(),
        )
        feat = self.features(torch.zeros(1, *self.input_shape)).view(1, -1).size(1)
        self.advantage = nn.Sequential(nn.Linear(feat, 512), nn.ReLU(),
                                       nn.Linear(512, self.num_actions))
        self.value = nn.Sequential(nn.Linear(feat, 512), nn.ReLU(), nn.Linear(512, 1))

    def forward(self, x):
        x = self.features(x).view(x.size(0), -1)
        a = self.advantage(x)
        v = self.value(x)
        return v + a - a.mean(1, keepdim=True)            # Q = V + (A - mean A)

    def q_values(self, state):
        with torch.no_grad():
            q = self.forward(state.unsqueeze(0))
        return q.cpu().numpy()[0]

    def select_action_from_q(self, q_values, epsilon):
        if random.random() < epsilon:
            return random.randrange(self.num_actions)
        return int(np.argmax(q_values))
```

The actor-side storage: it folds a run of steps into n-step transitions, flushes all shorter terminal tails, and — because it kept Q-values for the start and bootstrap states — computes the initial absolute n-step TD error without re-running the network:

```python
from collections import deque

class BatchStorage:
    """Per-actor n-step accumulator that also produces initial priorities."""
    def __init__(self, n_steps, gamma=0.99):
        self.n_steps, self.gamma = n_steps, gamma
        self.pending = deque()
        self.ready = []

    def __len__(self):
        return len(self.ready)

    def _emit_one(self):
        m = min(self.n_steps, len(self.pending))
        ret = sum((self.gamma ** i) * self.pending[i]["reward"] for i in range(m))
        terminal = any(self.pending[i]["done"] for i in range(m))
        discount = 0.0 if terminal else self.gamma ** m
        first, last = self.pending[0], self.pending[m - 1]
        self.ready.append((first["state"], first["action"], ret, discount,
                           last["next_state"], first["q_values"], last["next_q_values"]))
        self.pending.popleft()

    def add(self, state, action, reward, next_state, done, q_values, next_q_values):
        self.pending.append(dict(state=state, action=action, reward=reward,
                                 next_state=next_state, done=done,
                                 q_values=q_values, next_q_values=next_q_values))
        while self.pending and (len(self.pending) >= self.n_steps or done):
            self._emit_one()

    def compute_priorities(self):
        actions = np.array([x[1] for x in self.ready])
        rewards = np.array([x[2] for x in self.ready], dtype=np.float32)
        discounts = np.array([x[3] for x in self.ready], dtype=np.float32)
        q = np.stack([x[5] for x in self.ready])
        nq = np.stack([x[6] for x in self.ready])
        q_a = q[range(len(q)), actions]                                 # q(S_t, A_t)
        next_q_a = nq.max(1)                                            # bootstrap value, actor-side
        target = rewards + discounts * next_q_a
        return np.abs(target - q_a) + 1e-6                              # |n-step TD error| + eps

    def make_batch(self):
        prios = self.compute_priorities()
        states, actions, rewards, discounts, next_states, _, _ = zip(*self.ready)
        batch = [list(states), list(actions), list(rewards), list(discounts), list(next_states)]
        self.ready.clear()
        return batch, prios
```

The shared prioritized buffer is the prioritized-replay machinery, with one change that matters at scale: `add` takes the actor-computed priority directly, so a transition never has to enter at a placeholder maximum and wait for the learner to refine it:

```python
class CustomPrioritizedReplayBuffer:
    """Single shared store. Sum-tree -> O(log N) sample ~ p^alpha; min-tree -> max IS weight.
    add() accepts an externally (actor-)computed priority."""
    def __init__(self, size, alpha):
        self._storage = []; self._maxsize = size; self._next_idx = 0
        self._alpha = alpha
        cap = 1
        while cap < size: cap *= 2
        self._it_sum = SumSegmentTree(cap)        # holds p_i ** alpha
        self._it_min = MinSegmentTree(cap)        # holds p_i ** alpha
        self._max_priority = 1.0

    def add(self, state, action, reward, discount, next_state, priority):
        idx = self._next_idx
        data = (state, action, reward, discount, next_state)
        if idx >= len(self._storage): self._storage.append(data)
        else: self._storage[idx] = data
        self._next_idx = (self._next_idx + 1) % self._maxsize
        self._it_sum[idx] = priority ** self._alpha          # actor-supplied priority, not a placeholder max
        self._it_min[idx] = priority ** self._alpha
        self._max_priority = max(self._max_priority, priority)

    def _sample_proportional(self, batch_size):
        res = []; p_total = self._it_sum.sum(0, len(self._storage))
        seg = p_total / batch_size
        for i in range(batch_size):                          # stratified: one draw per segment
            res.append(self._it_sum.find_prefixsum_idx(random.random() * seg + i * seg))
        return res

    def sample(self, batch_size, beta):
        idxes = self._sample_proportional(batch_size)
        p_min = self._it_min.min() / self._it_sum.sum()
        max_w = (p_min * len(self._storage)) ** (-beta)      # max weight from least-probable transition
        weights = []
        for idx in idxes:
            p = self._it_sum[idx] / self._it_sum.sum()       # P(idx)
            weights.append(((p * len(self._storage)) ** (-beta)) / max_w)   # (N P)^(-beta), normalized <= 1
        return self._encode_sample(idxes), np.array(weights), idxes

    def update_priorities(self, idxes, priorities):
        for idx, p in zip(idxes, priorities):                # learner writes back refreshed |delta|
            self._it_sum[idx] = p ** self._alpha
            self._it_min[idx] = p ** self._alpha
            self._max_priority = max(self._max_priority, p)
```

The learner's loss — multi-step double-Q target, Huber error, IS-weighted, and it returns the refreshed priorities to write back:

```python
def compute_loss(model, tgt_model, batch, n_steps, gamma=0.99):
    states, actions, rewards, discounts, next_states, weights = batch
    q_a = model(states).gather(1, actions.unsqueeze(1)).squeeze(1)        # q(S_t, A_t)
    next_actions = model(next_states).max(1)[1].unsqueeze(1)             # argmax by ONLINE net (double-Q)
    next_q_a = tgt_model(next_states).gather(1, next_actions).squeeze(1) # evaluate by TARGET net
    target = rewards + discounts * next_q_a                              # discount is gamma^n, or 0 at terminal
    delta = target.detach() - q_a
    abs_delta = delta.abs()
    prios = (abs_delta + 1e-6).data.cpu().numpy()                        # refreshed priority = |delta| + eps
    loss = torch.where(abs_delta < 1, 0.5 * delta ** 2, abs_delta - 0.5)  # Huber on signed TD error
    return (loss * weights).mean(), prios                                # IS-weighted scalar loss
```

The two loops that the architecture splits the computation into — an actor that generates with its own fixed ε_i and ships priced batches, and the learner that samples-trains-rewrites and periodically publishes parameters back:

def actor_loop(env, model, shared_buffer, param_source, actor_id, n_actors, args):
    epsilon = args.eps_base ** (1 + actor_id / (n_actors - 1) * args.eps_alpha)   # ε_i, fixed per actor
    storage = BatchStorage(args.n_steps, args.gamma)
    model.load_state_dict(param_source.latest())
    state = env.reset()
    q_values = model.q_values(torch.FloatTensor(np.array(state)))
    step = 0
    while True:
        action = model.select_action_from_q(q_values, epsilon)
        next_state, reward, done, _ = env.step(action)
        next_q_values = (np.zeros_like(q_values) if done
                         else model.q_values(torch.FloatTensor(np.array(next_state))))
        storage.add(state, action, reward, next_state, done, q_values, next_q_values)
        state = env.reset() if done else next_state
        q_values = model.q_values(torch.FloatTensor(np.array(state))) if done else next_q_values
        step += 1
        if step % args.update_interval == 0:                             # refresh params periodically (~400 frames)
            model.load_state_dict(param_source.latest())
            q_values = model.q_values(torch.FloatTensor(np.array(state)))
        if len(storage) >= args.send_interval:                           # ship a batch (B=50) with priorities
            batch, prios = storage.make_batch()
            for sample, p in zip(zip(*batch), prios):
                shared_buffer.add(*sample, p)                            # priced at insert, no placeholder max


def learner_loop(model, tgt_model, shared_buffer, optimizer, param_sink, args):
    learn_idx = 0
    while True:
        *batch, idxes = shared_buffer.sample(args.batch_size, args.beta)  # prioritized, IS weights
        loss, prios = compute_loss(model, tgt_model, batch, args.n_steps, args.gamma)
        optimizer.zero_grad(); loss.backward()
        torch.nn.utils.clip_grad_norm_(model.parameters(), args.max_norm) # clip grad norm to 40
        optimizer.step()
        shared_buffer.update_priorities(idxes, prios)                     # write refreshed |delta| back
        learn_idx += 1
        if learn_idx % args.target_update_interval == 0:                  # target copy every 2500 batches
            tgt_model.load_state_dict(model.state_dict())
        if learn_idx % args.publish_param_interval == 0:                  # publish params to actors
            param_sink.publish(model.state_dict())
