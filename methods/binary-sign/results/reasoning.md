Let me start from where the money actually goes when I serve one of these large language models. Almost all of the parameters, and almost all of the inference-time arithmetic, sit in the dense linear projections — the Q/K/V and output maps in attention, and the two big matrices in the feed-forward block. Each is a matmul of activations against an FP16 weight matrix, and once the model is several billion parameters the thing that hurts is not the FLOPs in the abstract but the *bytes*: I have to stream those weight matrices out of DRAM into the compute units on every forward pass, and that memory traffic, plus the energy of the floating-point multiply-accumulates, is what sets my latency and my power bill. So the lever is obvious — fewer bits per weight means fewer bytes moved and cheaper arithmetic, roughly linearly. The question is how far down I can push the precision before the model stops being a good language model, and the honest answer from the field is: not very far, not the way people currently do it.

Why not? Because the standard tools are all variations on "train in float, then squeeze." Post-training quantization rounds an already-trained FP16 model onto a coarse grid after the fact — lovely because it needs no retraining, but the model never learned to live on that grid, so as I coarsen the grid the rounding error grows. At eight bits that error is small; the worry is what happens at the extreme of one bit, and I should put a number on it rather than wave at it. Take one output row of a layer at width `n = 4096`, weights at `N(0, 1/n)` and a unit-variance input `x`, and compare the true dot product `y = Σ Wᵢxᵢ` against its best 1-bit replacement — even granting myself the *optimal* per-tensor scale, `β·Σ sign(Wᵢ)xᵢ` with `β = mean|W|`. I'll just simulate it. Over a couple thousand random rows the exact output has RMS magnitude ≈ 1.00, and the RMS error of the 1-bit version is ≈ 0.61, so the relative error per layer is about 0.6 — and that is the *floor*, the error of the least-squares-best rounding, not some sloppy scheme. One layer at 60% relative error, compounded through dozens of stacked projections, is not a degraded language model; it's noise. So rounding a finished float model down to one bit really does detonate, and the reason is structural: the rounding error at one bit is the same order as the signal, every layer. Quantization-aware training is better — it simulates the rounding inside the forward pass so the network learns weights that tolerate it — but the thing being optimized is still a floating-point number, and the discreteness is a fake-quant overlay on top of a fundamentally continuous parameter. What I actually want is different: a weight that *is* one of two values during every single forward pass, from the first training step to the last, trained from scratch. Not a float pretending to be coarse. A genuinely 1-bit weight, where the network gets to adapt the rest of its parameters around the binarization instead of inheriting it cold.

The instant I say "1-bit weight" I hit the first wall, and it's a hard one. If the forward weight is `sign(W)`, then the operation mapping my real parameter to the value I use is the sign function, and the derivative of sign is zero everywhere except at the origin, where it's undefined. Backpropagation multiplies by that derivative, so the gradient that reaches `W` is zero almost surely. The parameter never moves. A network whose every weight is gated by a sign would simply not train — there's nothing to descend.

So before I design anything I need a way to get a gradient through a hard threshold. Let me think about what I'm really asking. The forward pass uses `w_b = sign(W)`; I want a useful gradient `∂L/∂W` even though `∂w_b/∂W = 0`. The cleanest idea, and one with real grounding from the work on training threshold units, is to just *lie* in the backward pass: in the forward direction apply the sign, but in the backward direction pretend the sign was the identity, so the incoming gradient passes straight through to `W` unchanged. That's the straight-through estimator. It feels like cheating, but there's a defensible reason it isn't pure vandalism. Consider the canonical hard-threshold unit, an output `h` that is a step function of a pre-activation `a`, and ask for an estimator of how `L` changes with `a`. For a stochastic binarization where `h` is `±1` with probability tied to `a`, the estimator `(h − sign(a))·L` turns out to be an *unbiased* estimate of the true gradient — it's a special case of the REINFORCE score-function trick, with the sampled binary value playing the role of the action. So "pass the gradient through as if the threshold were identity" is not arbitrary; it's the deterministic shadow of an honest unbiased estimator. Good — that's my license. The sign is in the forward pass; in the backward pass it's a wire.

Now I have a way to train through `sign`, so let me actually design the weight. The naive thing is `w_b = sign(W)`, every forward weight is `±1`. But stare at one output unit's dot product: `y_j = Σ_i sign(W_ji)·x_i`. Every weight has magnitude exactly 1. The real trained weights of a layer are not all magnitude 1 — they have a scale, often well below 1, and forcing them all to unit magnitude inflates the output by a large, systematic factor and throws away the one global piece of magnitude information I could cheaply have kept. So I shouldn't approximate `W` by `sign(W)`; I should approximate it by `α·sign(W)` for some positive scalar `α` that I get to choose, and the scalar is free — one number per tensor, negligible storage, and at inference it factors out of the matmul as a single post-multiply, costing nothing per element. The question is which `α`, and which binary tensor, best stand in for `W`.

Let me make that precise and just solve it. I want the binary `B ∈ {−1,+1}ⁿ` (flatten the matrix to a vector of length `n`) and the scalar `α > 0` that minimize the squared reconstruction error

  J(B, α) = ‖W − αB‖².

Expand it: `J = α²·BᵀB − 2α·WᵀB + WᵀW`. Now look at the three terms. `BᵀB = Σ B_i² = n`, a constant, since every `B_i` is `±1`. `WᵀW` is a constant too — `W` is given. So as a function of `B`, with `α > 0` fixed, the only term that depends on `B` is `−2α·WᵀB`, and since `α` is positive, minimizing `J` over `B` is exactly *maximizing* `WᵀB = Σ_i W_i B_i` subject to `B_i ∈ {−1,+1}`. That maximization decouples coordinate by coordinate: to make `W_i B_i` as large as possible I pick `B_i = +1` when `W_i > 0` and `B_i = −1` when `W_i < 0`; at `W_i = 0` either sign gives the same objective, so I can choose the deterministic convention that sends zero to `−1`. That points at `B* = sign(W)` with a fixed tie rule.

Now the *magnitude*. With `B` fixed at the optimum, set `∂J/∂α = 0`: `∂J/∂α = 2α·BᵀB − 2·WᵀB = 2αn − 2WᵀB`, which vanishes at `α* = WᵀB*/n`. Substitute `B* = sign(W)`: `WᵀB* = Σ_i W_i·sign(W_i) = Σ_i |W_i|`, because `W_i·sign(W_i) = |W_i|`. So

  α* = (1/n)·Σ_i |W_i| = (1/n)·‖W‖₁ = mean(|W|).

That is suspiciously clean, and I want to be sure I haven't fooled myself by assuming the coordinatewise argmax and the stationary `α` can be read off independently — the two are coupled through `B*`, and I derived each holding the other fixed. The honest check is small enough to do by brute force: take a concrete vector, enumerate all `2ⁿ` sign patterns, and for each one solve the (easy, scalar) optimal `α`, then see whether the global minimizer really is `(sign(W), mean|W|)`. Let me take `W = (0.7, −0.2, 0.5, −0.9, 0.1)`, so `n = 5` and 32 patterns. Sweeping all of them, the squared error is minimized at `B = (+1, −1, +1, −1, +1)`, which is exactly `sign(W)`, and the accompanying optimal scale comes out at `α = 0.48`. And `mean|W| = (0.7+0.2+0.5+0.9+0.1)/5 = 2.4/5 = 0.48`. They agree to the digit, and the brute-force winner over *all* patterns is the sign pattern — so the decoupled derivation wasn't a lucky shortcut; it lands on the true joint optimum. So the l2-best 1-bit summary of a weight tensor is `mean(|W|)·sign(W)`, both pieces falling out of one least-squares problem, and the sign function is the optimal binary *direction* rather than a heuristic I reached for. I didn't choose the sign and I didn't choose the absmean; the squared-error objective chose them, and an exhaustive search agrees.

Before I move on, one refinement nags at me. The least-squares fit above treated the threshold as zero, but I have a degree of freedom in *how* I present `W` to the sign. A binary code `{−1,+1}` is symmetric about zero; if the weight tensor has a nonzero mean — say it's slightly positive on average — then the sign threshold at zero is poorly placed relative to the actual distribution, and one of my two symbols does most of the work while the other is starved. I'm wasting half of a one-bit code. The fix is to recenter: subtract the per-tensor mean `ᾱ = (1/n)Σ_i W_i` before taking the sign, so the threshold sits at the center of mass of the distribution and the two symbols split the mass more evenly. This centering step is a capacity choice rather than a second least-squares derivation of the scale; I keep the cheap global magnitude `β = mean(|W|)` from the unshifted fit and use the centered binary direction `B = sign(W − ᾱ)`. Centering buys capacity for one more scalar reduction.

So my forward weight is `W̃ = sign(W − ᾱ)`, used at magnitude rescaled by `β = mean(|W|)`, and the sign is trained through with the straight-through wire. But now I have to confront how the *update* interacts with a binary forward weight, because there's a subtle trap. Suppose I let the optimizer update the binary weights directly. SGD on a deep net makes an enormous number of tiny, noisy steps; the actual learning signal is buried in noise and only emerges as those tiny steps accumulate and average over many iterations. A binary variable cannot accumulate anything — a tiny step almost never flips a sign, so it's simply discarded, and the rare step that does flip a sign is an all-or-nothing jump with no memory of the small pressures that built up to it. The optimization would be driving on a variable that can't integrate its own gradient. The way out is to keep a high-precision *latent* weight `W` that the optimizer updates normally, and binarize it on the fly in the forward pass; the latent weight is where the tiny steps accumulate, and its sign is what the network actually uses. The slogan that justifies this is that only the *expected value* of the weight needs to be high precision — the continuous latent weight holds the running average of the gradient pressure, while the low-bit forward path behaves like a regularized readout of it. At inference, the latent weight is discarded entirely — I store only the bits and the scale. So the picture is: float latent weight in the optimizer, `±1` weight in the forward matmul, straight-through gradient connecting them.

That handles the weights. But a matmul has two operands, and if I leave the activations in FP16 I've only solved half the cost problem — I still stream and multiply full-precision activations. Worse, a true 1-bit-times-FP16 product doesn't even buy the cheap-arithmetic win I wanted, because the expensive floating-point multiply is still there on the activation side. I'd like the activations low-bit too, so the dot product becomes integer accumulation. But here I have to be careful, because activations are a different animal from weights, and I know two uncomfortable things about them. First, activation distributions in trained Transformers are not the flat, well-behaved distributions weights have — they're harder to quantize. Second, and more pointed, LLM activations are known to carry a small number of outlier feature dimensions, on the order of a tenth of a percent of channels, whose magnitudes run tens to a hundred times larger than everything else, and these outliers sit in the *same* channels across tokens. That's a real, documented hazard: any per-tensor scale I pick has to span those outliers, which stretches the dynamic range so far that the ordinary values collapse into a handful of levels.

Given that, going to one bit on activations is reckless — I'd be quantizing the very thing that's hardest to quantize, in the presence of huge outliers, with two levels. So I'll be asymmetric: keep the activations at a *modest* low precision, say 8 bits, where I still have enough levels to absorb the outlier dynamic range, and reserve the extreme 1-bit treatment for the weights, which I just showed are the well-behaved, l2-friendly operand. Eight-bit activations against one-bit weights — call it W1A8 — is the honest split given what each operand can tolerate. For the 8-bit scheme itself the natural choice is the standard symmetric one, absmax: take the absolute maximum of the tensor, `γ = ‖x‖_∞`, and rescale by the half-width convention `Q_b = 2^{b−1}`. For `b = 8`, that is `Q_b = 128`, while the signed int8 storage endpoints are `[-128, 127]`; the positive side must stay below `+128` before it is stored. In the algebra I write the clip as

  x̃ = Clip(x · Q_b/γ, −Q_b + ε, Q_b − ε),

with a small `ε` to keep the clip from touching the overflowing endpoint. In code, the same convention shows up as integer values clamped to `[-128, 127]` with scale `γ/Q_b`. The `Clip` is non-differentiable at its corners just like the sign, so I send the gradient straight through it too. (For activations that are about to hit a ReLU and are therefore one-sided, I'd shift by the minimum first, `η = min(x)`, and map into `[0, Q_b]` instead — same idea, asymmetric range to match a non-negative signal.) Per-tensor scaling is the simple thing during training; at inference I can afford per-token scaling for a tighter fit. I briefly consider a fancier scheme — learn the activation scale and threshold as trainable parameters, an "elastic" quantizer — but a learnable scale is one more thing to optimize through a discretization, it makes the dynamics touchier, and the plain absmax gives me the simple, deterministic scale I need for a large-learning-rate training recipe. So absmax it is.

Now I have to check that this binarized layer doesn't destroy the forward signal, because I'm going to stack tens of these and a deep stack is unforgiving about variance. Take the matmul `y = W̃·x̃` and, as is standard for this kind of stability accounting, treat the entries of `W̃` and `x̃` as independent, identically distributed, and independent of each other. Then for an output that sums over `n` inputs,

  Var(y) = n·Var(w̃·x̃) = n·E[w̃²]·E[x̃²].

Here `w̃` is the *scaled* binary weight, magnitude `β` on every entry, so `E[w̃²] = β²`, and the variance is `Var(y) = n·β²·E[x̃²]`. Now `β = mean(|W|)`, and for a standard zero-mean weight init the mean absolute value scales like `1/√n` (a Gaussian of standard deviation `σ` has mean absolute value `σ√(2/π)`, and inits set `σ ∝ 1/√n` precisely to control this), so `n·β² = n·(mean|W|)²` is a width-independent constant. The stability calculation treats this constant as order one, written as `n·β² ≈ 1`, which leaves `Var(y) ≈ E[x̃²]`. So the binarization itself doesn't blow the variance up with width — good, the absmean scale was doing double duty all along, both as the l2-derived magnitude and as the thing that keeps `n·β²` controlled. The remaining factor is `E[x̃²]`: the output variance is whatever the second moment of the quantized input is. In a full-precision layer with Kaiming/Xavier init that quantity is arranged to sit at order one, which is exactly what keeps a deep network numerically stable. I want the same here. The clean way to *guarantee* the input second moment is to normalize the activation right before I quantize it: insert a LayerNorm so that the tensor entering the quantizer has unit variance by construction. Then `Var(y) ≈ E[LN(x)²] = 1`, the same order as the full-precision layer, and my deep stack of binarized layers inherits the stability of the float one. So a normalization placed *immediately before the activation quantization* isn't decoration — it's the thing that restores the variance the binarization would otherwise leave at the mercy of the incoming activation scale. In a Transformer block this is a sub-layer normalization (SubLN) sitting inside the projection.

Let me now assemble the whole layer in one expression. The activation path is: normalize, then absmax-quantize. The weight path is: center, then sign, with absmean scale. The matmul runs between the 1-bit weights and the 8-bit activations, and then I undo both scales — divide out the activation's `γ/Q_b` and multiply in the weight's `β`. So

  y = W̃ · Quant(LN(x)) · (β·γ / Q_b),   W̃ = sign(W − ᾱ),   β = (1/nm)‖W‖₁,   γ = ‖LN(x)‖∞.

Every multiply in the core matmul is a `±1`-times-int8, i.e. just a conditional add/subtract of an integer; the only real floating-point multiplications left are the two per-output rescalings by `β` and `γ/Q_b`, which is exactly the energy win I was chasing — the bulk of the matmul has become integer addition.

One scaling-up wrinkle I should head off. Model parallelism splits each weight matrix across devices along one dimension, and the prerequisite is that the per-shard computation be independent along the split. But `ᾱ`, `β`, `γ`, and the LayerNorm statistics are all reductions over the *whole* tensor, which couples the shards and would force an all-reduce per parameter per layer — cheap individually but, multiplied by depth, a serialization that throttles the forward pass. The fix is to stop insisting on whole-tensor statistics: partition the weights and activations into groups along the split dimension and estimate each group's `ᾱ`, `β`, `γ`, and normalization locally. Grouped statistics are computed entirely within a shard, so no extra communication is needed, and the slight inexactness versus a global statistic is harmless — these are scale estimates, not quantities that need to be globally consistent. Group quantization plus group normalization keeps the layer parallelizable at scale.

Now the optimization, because there's a failure mode specific to 1-bit training that I have to design around. The gradient that the latent weight receives is computed through the *binarized* forward weight (that's the whole point of the straight-through path), so it's an estimate of the loss sensitivity at the current sign pattern. A small update to the latent weight usually does not change any sign, which means the binary weight — and therefore the actual function the network computes — is unchanged, even though I just spent a gradient step. The update is, in effect, biased: I keep nudging latent weights without flipping the bits that matter, and this is worst at the very beginning of training, when I most need the sign pattern to reorganize quickly. The clean diagnosis is that the latent weight needs to traverse enough distance to actually flip signs, and small steps don't do that. The simplest lever that does is the learning rate: make it large, so latent-weight updates are big enough to cross the zero threshold and flip bits at a useful rate, especially early. I'd normally worry that a large learning rate destabilizes training, but the binarized layer gives me reasons to try it: the sign + scale + pre-quant normalization clamp the forward signal regardless of how the latent weights move, the low-bit path acts as a regularizer, and SubLN holds the variance. So the right recipe is the opposite of timidity: train this thing with a *large* learning rate, larger than I'd first choose for FP16, and let the bits reorganize fast. The rest of the mixed-precision setup follows the same logic — weights and activations are low-bit in the forward pass, but the gradients and the optimizer states stay high precision, because that's where the tiny steps accumulate and where the latent weight lives.

Let me also sanity-check the broader claim I'm implicitly betting on: that this layer, dropped into an otherwise standard decoder Transformer, still scales like a language model. The full-precision scaling law says loss falls as a power law in parameter count, `L(N) = aN^b + c`. The binarized layer is still a stack of linear projections and nonlinearities with the same parameter count, just with a coarser weight representation and a forward path engineered to preserve variance, so the hypothesis is the same power-law shape, perhaps with an offset. That's what I would validate by fitting the law on small sizes and checking it forecasts the larger ones; the design gives the hypothesis a reason to be plausible, and the efficiency math says the inference-cost axis should move in my favor, since the matmul energy drops from FP16 multiplies to int8 adds.

So, to put the whole layer into the code I'd actually drop in, filling the three empty slots — the weight representation, the activation representation, and the forward that ties them together with straight-through gradients. The standard, cleanest way to write the straight-through estimator in an autograd framework is the identity `z_q = z + (quant(z) − z).detach()`: in the forward pass the `z` and `−z` cancel and it equals `quant(z)`, while in the backward pass the whole detached parenthesis contributes zero gradient, so `∂z_q/∂z = 1` — the sign/round/clip inside `quant` is exactly the wire I wanted, with no custom backward needed.

```python
import torch
import torch.nn as nn
import torch.nn.functional as F


def weight_quant(weight):
    # Direction comes from the sign least-squares solution; centering is the
    # capacity fix used before the sign. The scale is beta = mean(|W|).
    beta = weight.abs().mean()
    e = weight.mean()                                    # per-tensor mean (centering)
    w_bin = torch.where(weight > e, torch.ones_like(weight), -torch.ones_like(weight))
    w_ste = weight + (w_bin - weight).detach()           # forward = +-1, backward = identity
    return w_ste, beta


def activation_quant(x):
    # Absmax 8-bit: Q_b = 2^(8-1); signed int8 storage is [-128, 127].
    Qb = 128.0
    gamma = x.abs().amax(dim=-1, keepdim=True).clamp(min=1e-5)
    x_int = (x * (Qb / gamma)).round().clamp(-Qb, Qb - 1)
    x_ste = x + (x_int - x).detach()                    # forward = int8-valued, backward = identity
    return x_ste, gamma / Qb


class BitLinear(nn.Module):
    """Drop-in replacement for nn.Linear with 1-bit weights and 8-bit activations.
    Same path in train and eval. The high-precision self.weight is the LATENT weight the
    optimizer updates; it is binarized on the fly every forward pass and discarded at inference."""

    def __init__(self, in_features, out_features, bias=True):
        super().__init__()
        self.in_features = in_features
        self.out_features = out_features
        self.weight = nn.Parameter(torch.empty(out_features, in_features))
        self.bias = nn.Parameter(torch.zeros(out_features)) if bias else None
        # SubLN placed immediately before activation quantization: forces E[LN(x)^2]=1 so the
        # binarized matmul has Var(y) ~ 1, matching the full-precision layer's stability.
        self.norm = nn.LayerNorm(in_features, elementwise_affine=False)
        nn.init.normal_(self.weight, mean=0.0, std=0.02)

    def forward(self, x):
        x = self.norm(x)                                # normalize before quantizing (variance control)
        x_int, x_scale = activation_quant(x)            # int8-valued activations, STE
        w_bin, w_scale = weight_quant(self.weight)       # +-1 weights, STE
        out = F.linear(x_int, w_bin) * (w_scale * x_scale)
        if self.bias is not None:
            out = out + self.bias
        return out
```

The causal chain, start to finish: I wanted natively 1-bit weights to cut the dominant matmul cost, but `sign` has zero gradient, so first I needed the straight-through estimator — pass the gradient through the threshold as identity, justified as the deterministic shadow of an unbiased stochastic-neuron estimator. A bare `sign` throws away magnitude, so I asked for the least-squares-best `α·B` approximation to `W`, and the squared-error objective *handed me* both pieces: `B* = sign(W)` (the argmax of `WᵀB`) and `α* = mean(|W|)` (from setting the derivative in `α` to zero) — the absmean scale. Centering before the sign recovers wasted capacity in the one-bit code while keeping that absmean scale. Because a binary weight can't accumulate SGD's tiny noisy steps, I keep a high-precision latent weight that the optimizer updates and binarize it on the fly; only its expected value needs precision. Activations are the harder operand — flat-distribution weights binarize cleanly, but activations carry persistent large outliers — so I split asymmetrically: 1-bit weights, 8-bit absmax activations, with the clip also trained through straight-through. To keep a deep stack stable I computed `Var(y) = n·β²·E[x̃²]`, used the absmean scale to keep `n·β²` at order one, and forced the remaining second moment to one by normalizing immediately before the activation quantizer (SubLN). Whole-tensor statistics would break model parallelism, so I group them to keep each shard independent. Finally, because small latent-weight updates fail to flip bits — a bias worst at the start — I train with a deliberately large learning rate, backed by the regularizing low-bit path and the variance-preserving normalization. The result is a drop-in linear layer whose factorized forward path is `F.linear(int8_activation, binary_weight) * beta * gamma / Q_b`.
