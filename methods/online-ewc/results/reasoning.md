Let me start from what actually breaks when I run a regularisation-based continual learner for a long time. I have a network with a fixed number of parameters theta, and contexts arrive one after another: task 1, then task 2, then task 3, and I want it to keep doing all of them. The reigning answer is elastic weight consolidation: after I finish a context t, I freeze a snapshot of where the weights ended up, theta*_t, and I measure how much each weight mattered to that context with the diagonal of its Fisher information F_t, and then while I learn later contexts I add a quadratic penalty that pulls each weight back toward theta*_t with a stiffness proportional to F_{t,i}. The total loss while learning some new context is L_new(theta) + Σ_t (lambda/2) Σ_i F_{t,i} (theta_i - theta*_{t,i})^2, the sum running over every past context t. Each term is a spring: stiff in the directions a past context cared about, slack everywhere else, so the optimiser can move the weights that don't hurt the old contexts and is resisted when it moves the ones that do. It works. But look at that sum. It has one term per past context, and each term carries its own anchor theta*_t (a full copy of the parameter vector) and its own Fisher F_t (another full copy). After a hundred contexts I am storing a hundred parameter-sized anchors and a hundred parameter-sized Fishers, and every single gradient step I have to sweep all hundred springs to form the penalty. The cost — memory and per-step compute both — grows linearly in the number of contexts I have ever seen. For a genuine lifelong agent that is the thing that kills it: the bookkeeping outgrows the model. So the question I actually have to answer is sharper than "how do I not forget." It is: can I keep the protection EWC gives me but pay a cost that is *constant* in the number of contexts?

The naive thing first, just to feel whether there's anything to the linear-growth problem at all. EWC itself notes that two quadratic penalties sum to a single quadratic penalty — a sum of springs is a spring. So can't I just merge them? For one coordinate, take two terms, (1/2) F_A (theta - theta*_A)^2 + (1/2) F_B (theta - theta*_B)^2. Expanding gives (1/2)(F_A+F_B)theta^2 - (F_A theta*_A + F_B theta*_B)theta + const. To write that as one square, the centre has to be m = (F_A theta*_A + F_B theta*_B)/(F_A+F_B), coordinate by coordinate wherever the summed stiffness is nonzero, so the merged penalty is (1/2)(F_A+F_B)(theta-m)^2 + c, where c = (1/2)(F_A theta*_A^2 + F_B theta*_B^2 - (F_A+F_B)m^2), equivalently (1/2)F_A F_B(theta*_A-theta*_B)^2/(F_A+F_B) when F_A+F_B > 0. The constant c does not affect gradients. So the *stiffness* merges into a running sum of Fishers, Σ_t F_t, which is one parameter-sized object regardless of how many contexts. The anchor m is a Fisher-weighted blend of all the past optima, and to update it incrementally when a new context arrives I'd carry both the running Σ_t F_t and the running Σ_t F_t theta*_t and divide — still constant memory, two parameter-sized accumulators. So mechanically I *can* fold EWC's penalties into a single quadratic with constant storage. But all I've done is algebra on a sum of springs I never questioned: is anchoring a fresh penalty at *every* past optimum even what the Bayesian story I started from prescribes? If the springs are wrong, merging them faithfully just preserves the error in a cheaper container.

So back to the derivation, slowly, because EWC came out of a probabilistic argument and I want to redo that argument without assuming the per-context stack. The weights are random; what I know after the data of contexts 1..k is the posterior p(theta | T_{1:k}). The contexts' data are conditionally independent given theta, so the joint likelihood factors, and the posterior builds up recursively: p(theta | T_{1:k}) ∝ p(theta | T_{1:k-1}) p(T_k | theta). Stare at that for a second. It says the posterior after all k contexts is the posterior after the first k-1 — used as the prior — times the likelihood of just the k-th. It is purely sequential and it doesn't care about ordering. And it never asks me to keep the individual context likelihoods p(T_t | theta) lying around: the only object the recursion ever touches is the *running posterior*. The per-context stack of springs is nowhere in this equation; what's in the equation is one evolving distribution. So EWC's linear growth can't be a consequence of the Bayesian math itself — it has to come from how EWC chose to *approximate* it.

How did the approximation go? The posterior over a deep net is intractable, so Laplace's method: near the mode theta*, approximate -log of a distribution by its second-order Taylor expansion. At a mode the gradient is zero, so the expansion is purely quadratic, -log p(theta | ·) ≈ (1/2)(theta - theta*)^T H (theta - theta*) + const, with H the Hessian of the negative log posterior at theta*. Near a well-fit optimum the data part of H is well-approximated by the Fisher F (times the sample count), and EWC keeps only its diagonal — one scalar stiffness per weight, computed from squared first-order gradients, positive semidefinite, the right local curvature. So far so good and identical for everyone. The branch point is *what* you Laplace-approximate. EWC applies the Gaussian approximation to each context's *likelihood* p(T_t | theta) separately — one Gaussian N(theta*_t, F_t^{-1}) per context — and then the product of those Gaussians is the product of springs, one anchored at each theta*_t. That's where the stack is born. But the recursion above suggests a different target: apply Laplace not to each likelihood but to the *whole running posterior* p(theta | T_{1:k-1}), which is the single object the recursion actually carries forward.

Let me run that consistent version and see what it gives, going context by context, because if it differs from EWC the difference can only show up from the third context on (two-context EWC and two-context anything-Laplace agree by construction). Two contexts, A then B. The log posterior is log p(theta | D_A, D_B) = log p(D_B | theta) + log p(theta | D_A) + const, where the constant is independent of theta. The first term is just the loss for B. The second, p(theta | D_A), is the posterior after A — intractable, so I Laplace it around its mode theta*_A: -log p(theta | D_A) ≈ (1/2) Σ_i (F_{A,i}) (theta_i - theta*_{A,i})^2 + const, with F_A the diagonal Fisher at theta*_A (folding the prior's curvature in if I keep one). So learning B I minimise L_B(theta) + (1/2) Σ_i F_{A,i} (theta_i - theta*_{A,i})^2 — one spring around theta*_A. Identical to EWC for two contexts. No disagreement yet.

Now the third context, C, after A and B. The exact log posterior is log p(theta | D_A, D_B, D_C) = log p(D_C | theta) + log p(theta | D_A, D_B) + const. I need to Laplace-approximate the running posterior p(theta | D_A, D_B). But here's the thing I have to be honest about: I already threw D_A and D_B away, so I do not have the *true* p(theta | D_A, D_B) to expand. What I have is the *approximation* I built while learning B, namely log p(theta | D_A, D_B) ≈ log p(D_B | theta) - (1/2) Σ_i F_{A,i} (theta_i - theta*_{A,i})^2 + const. So I Taylor-expand *that* around its mode — and its mode is the point I already found while learning B, which is theta*_B (it's a local optimum of exactly this approximated objective, so the first-order term vanishes there, good). The curvature of this approximated posterior at theta*_B has two pieces: the negative-log-likelihood part of B contributes its Fisher F_B (curvature of -log p(D_B|theta) ≈ F_B), and the spring around theta*_A contributes its own curvature, which is just its stiffness F_A — a quadratic's second derivative is its coefficient, constant everywhere. So the curvature of the running posterior at theta*_B is F_A + F_B. Therefore the consistent Laplace approximation of p(theta | D_A, D_B), expanded at theta*_B, is a *single* quadratic: (1/2) Σ_i (F_{A,i} + F_{B,i}) (theta_i - theta*_{B,i})^2. And the loss for learning C becomes

  L_C(theta) + (1/2) Σ_i (F_{A,i} + F_{B,i}) (theta_i - theta*_{B,i})^2.

So the consistent recursion gives *one* spring for context C, anchored at the most recent optimum theta*_B, with a stiffness that is the *sum* of the past Fishers — where EWC would carry two springs, one at theta*_A and one at theta*_B. The number of penalty terms didn't grow; it stayed one. That's the candidate for constant cost, and it fell out of just being consistent about which object to approximate.

A one-coordinate numeric check, because it's easy to fool myself with this algebra. Take context A with optimum theta*_A = 1, Fisher F_A = 2, and context B with optimum theta*_B-raw = 3, Fisher F_B = 1. If I still had both datasets, the joint negative-log-posterior would be 0.5·2·(t-1)^2 + 0.5·1·(t-3)^2, a single bowl whose mode is (2·1 + 1·3)/(2+1) = 5/3 = 1.6667 and whose curvature is 2+1 = 3. Now the online path: while learning B I don't see A's data, I see B's loss plus the spring at theta*_A, so the point I actually land on is argmin of 0.5·1·(t-3)^2 + 0.5·2·(t-1)^2 = (1·3 + 2·1)/3 = 1.6667 — the *same* point the true joint bowl peaks at. And the consistent single spring I derived has curvature F_A+F_B = 3 anchored there. So the online summary (curvature 3, anchor 1.6667) reproduces the true posterior(A,B) bowl (curvature 3, mode 1.6667) exactly in this case. The recursion isn't just bookkeeping that happens to be cheap; on this example it lands on the genuine joint object.

Now the sharper claim — that EWC's keeping the *extra* spring at theta*_A is not merely redundant but actively wrong, because theta*_B already encodes A's pull. Test it on the same numbers, and the crux is where EWC anchors B's spring: not at B's raw optimum 3, but at the point actually reached while learning B *under* A's spring, which is 5/3. So while learning C, at some test point t = 4 EWC sweeps F_A(t - theta*_A) + F_B(t - theta*_B) = 2·(4-1) + 1·(4-1.6667) = 6 + 2.333 = 8.333, where the consistent single spring exerts (F_A+F_B)(t - theta*_B) = 3·(4-1.6667) = 7.0. They disagree, and the true joint bowl from the check above sides with the single spring: the resting point of EWC's two springs solves 2(t-1) + (t - 5/3) = 0, so t = 11/9 ≈ 1.222, dragged back toward A from the true joint mode 5/3 ≈ 1.667. That gap *is* the double-count: theta*_B already sits where A's pull balanced B's loss, so a fresh spring at theta*_A imposes A's constraint a second time, and the bias points systematically at the *earliest* context. (Had B's spring been anchored at the raw 3, the forces would coincide — but that is just the complete-the-square identity from the start, and it would mean B had been learned unprotected.) So dropping the old anchors is not only cheaper; keeping them is a measured error, not a wash.

Two further situations compound that bias, and the bare three-context Laplace hides both. The first is *recurrence*. Suppose A appears, then B, then A again. The honest Bayesian thing is to *refresh* A's contribution with the new data, replacing rather than re-adding. But EWC's bookkeeping just appends another spring at theta*_A: after A, B, A it carries springs at theta*_A (F_A), theta*_B (F_B), and theta*_A again (F_A) — A's restoring force is now 2·F_A(t-theta*_A), literally counted twice. At t = 2.5 that's a total force F_A(t-1) + F_B(t-3) + F_A(t-1) = 2·(1.5) + 1·(-0.5) + 2·(1.5) = 5.5, of which A contributes 6.0 (the rest pulling back), where a single refreshed summary would have counted A once. The second is the *long chain*: EWC's anchors never fade, so the very first context keeps a spring of weight 1 forever. Over K later contexts the EWC resting point is the Fisher-weighted average of *all* anchors including A at full weight; the early contexts get re-asserted at every step while later ones are only represented once. The lesson for my single-spring summary: collapsing the stack is the consistent move, and for recurrence and the long chain the *right* fix is not to keep the old anchors but to make the summary able to *down-weight* old contributions — which I don't yet have.

So I have a constant-memory method already: keep one mean theta*_{i-1} and one summed Fisher F-summed = Σ_t F_t, update F-summed ← F-summed + F_i each boundary, penalty (1/2) Σ_r [Σ_{t<i} F_{t,r}] (theta_r - theta*_{i-1,r})^2 around the latest mean. The re-centring has a price worth naming. Fisher is a *local* quantity — F_t describes the loss surface only near theta*_t. When I re-centre the accumulated penalty at the new optimum theta*_{i-1}, I'm pretending the old contexts' curvature, measured back at their own optima, is still the right curvature here. Strictly I'd have to recompute every old F_t at the new point, which needs the old data and defeats the purpose. So re-centring is an approximation on top of the diagonal approximation, and concretely it means the earliest contexts get gradually un-anchored: there's no longer a spring sitting at theta*_A directly, only the memory carried forward through the moving mean and the summed Fisher. I should *expect* the earliest contexts to be remembered a little less faithfully than a method keeping an explicit anchor at each. That's a real, predictable trade for constant cost — the recent past is favoured over the distant past — and from the long-chain check above I can even see the direction quantitatively: with literal anchors A holds weight 1, with a decaying summary A's weight shrinks. Which is better depends on the regime: if every context trained to convergence and all penalties can be satisfied at once, the single re-centred summary is the consistent Bayesian object; if the network is undertrained, an explicit heavy anchor on the first context would hold it better. I'd want to confirm that on Permuted-MNIST, but the direction of the bias is clear.

Is the summed-Fisher version the end? Two things make me keep going. First, the fixed-capacity problem I started with hasn't been solved — it's been *compressed*. With F-summed = Σ_t F_t the stiffnesses only ever add, never shrink. Let me see how fast this rigidifies: if every context has comparable Fisher F_t = 1, after n contexts F-summed = n, growing without bound — 200 contexts give a stiffness of 200 in every direction any context ever cared about, and the network, with a fixed number of parameters, becomes so rigid that new contexts can't be learned at all. It's documented that EWC misbehaves when the model runs out of capacity, and my undamped running sum has exactly that disease. Second, recurrence, which I just diagnosed: when A comes around again I want to *update* A's contribution, not append a redundant term — but I deliberately don't store A's term separately (that's what buys constant memory), so I have no way to revise it. Both problems point at the same missing capability: I need to *remove*, or at least *down-weight*, a context's earlier contribution to the shared summary without ever having stored it on its own.

That phrasing rings a bell from the approximate-inference side. Expectation propagation has exactly this move: it keeps one approximating factor per data factor, and to refine one it forms the "cavity" by *dividing that factor out* of the product, recomputes, and multiplies back. The trouble is EP keeps a factor *per* context — linear memory, the very thing I'm fleeing. But stochastic EP relaxes precisely this: instead of one factor per data factor, it keeps a *single shared* averaged factor that stands in for all of them, and it treats any one factor's contribution as a *fraction* of that shared factor. So when you want to update one factor, you don't divide out a stored copy of it — you down-weight the shared factor by the appropriate fraction (a partial removal), refine on the new data, and fold back in. That's exactly the capability I just said I needed, and it costs nothing extra: a scalar down-weighting of the one summary I'm already keeping.

Let me transcribe that into my Fisher accumulation. Right now I add: F-summed ← F-summed + F_i. The stochastic-EP move says: before folding in the new context, *partially remove* the previous shared contribution by scaling it down by a fraction gamma < 1, then add the new Fisher:

  F* ← gamma · F*_old + F_i.

Unroll it to see whether it bounds the growth and does the recurrence job: F*_i = Σ_{t≤i} gamma^{i-t} F_t, so a context that finished k boundaries ago contributes with weight gamma^k inside the stored summary. Take gamma = 0.9 and comparable Fishers F_t = 1 and sum it for real: at n = 10 contexts the stored F* is Σ_{t=1}^{10} 0.9^{10-t} = 6.51; at n = 50 it's 9.95; at n = 200 it's 10.0000; at n = 1000 still 10.0000. So instead of climbing to 200 it saturates, and the saturation value is the geometric-series limit 1/(1-gamma) = 1/0.1 = 10 — the summary stops growing and settles to an effective ~1/(1-gamma) recent contexts. That's the room I need to keep learning on fixed capacity: the loss actually uses gamma·F* (see below), which saturates to gamma/(1-gamma) = 0.9·10 = 9, one factor of gamma below the stored summary. For recurrence: when a context is revisited, scaling the running summary by gamma *is* the fractional removal of its previous presentation before the fresh data is added — A, B, A gives F* = gamma(gamma·F_A + F_B) + F_A = 0.9·(0.9·2 + 1) + 2 = 4.52 with the running example's F_A = 2, F_B = 1, and no separate stored term for A, where EWC's stack double-counted A as I found above. And it needs no task *identity*, only that a boundary occurred, since every context goes through the same gamma. It's also *graceful*: instead of catastrophically dropping an old context, gamma fades it smoothly, which I'd actually want if an old context never converged and its constraint is half-wrong.

The endpoints pin down the indexing, since gamma appears in two places. At gamma = 1, F*_i = Σ_{t≤i} 1·F_t = Σ_t F_t — the strict undecayed sum, and the next loss uses 1·F*, the full summary; so gamma = 1 recovers exactly the summed-Fisher single penalty I had before. At gamma = 0, the stored update after a context is gamma·F*_old + F_new = F_new only — just that context's Fisher — but the next loss uses gamma·F* = 0, so there is *no* protective penalty at all. That's a sensible degenerate point (no memory). A middle value such as 0.9 keeps roughly the last ten contexts strongly represented and lets older ones fade, which matches the saturation count I computed. So gamma is simultaneously the EP partial-removal fraction and an explicit forgetting knob — the same scalar wears both hats.

Now what exactly is the penalty I apply, in terms of this F*? The shared summary after i-1 contexts is the Gaussian with mean theta*_{i-1} and precision F*_{i-1}, and what I regularise the new context against is the *down-weighted* shared factor — the same gamma scaling, because the factor I'm holding the new context to is the cavity-style shared summary, not the raw sum. So the penalty is

  (1/2) ‖theta - theta*_{i-1}‖^2 over gamma · F*_{i-1} = (1/2) Σ_i gamma · F*_{i-1,i} (theta_i - theta*_{i-1,i})^2.

The gamma sits in the loss as well as in the accumulation. There's a genuinely ambiguous reading I want to pin down: one could make the next update decayed but regularise the current context against the *undamped* F*_{i-1}, treating the stored summary as the prior directly. The cavity-style reading is different — before learning the new context I first partially remove the shared factor, and the object I regularise against is already the down-weighted factor. Under that convention the applied stiffness is gamma · F*_{i-1} and the update after the context is gamma·F*_{i-1} + F_i; the two uses of gamma stay tied. That's the convention under which gamma = 0 correctly gives no penalty (the other convention would leave a penalty even at gamma = 0, which is wrong for a "forget everything" knob), so the tied convention is the consistent one.

One more refinement matters in policy settings, and it's about making each context count *equally*, which I claimed gamma helps with but which the raw Fisher quietly undermines. The penalty protects the network's output behaviour in expectation, and the Fisher is the curvature of that output's divergence under weight changes. For a context whose solution is very *deterministic* — a peaked output distribution — a small change in theta produces a large change in the predicted distribution, so the Fisher has a large overall norm; for a context with a flatter, more uncertain solution the Fisher norm is small. So if I accumulate raw Fishers, F* ← gamma F* + F_i, a deterministic context dumps a large-norm F_i into the sum and dominates it, while a soft context barely registers — purely because of the *scale* of its Fisher, not because it's more important to protect. If I want equal task scale, I normalise each context's Fisher before folding it in, dividing F_i by a task-level norm so what enters the running sum is the *relative* importance of weights within that context. For the minimal hook I need here, the invariant is the raw all-label diagonal-Fisher path: compute F_new, make the stored accumulator land on F* ← gamma F*_old + F_new, and use 0.5 · gamma · F* in the loss.

Now the importance estimator itself — what F_i concretely is and how I compute it from squared gradients, because that's the other empty slot. The diagonal Fisher at the finished context's optimum is F_{i,j} = E_x E_{y ~ p_theta(·|x)} [ (∂ log p_theta(y|x) / ∂theta_j)^2 ] — the expected squared score of the model's *own* predictive distribution. I estimate the outer expectation over x by passing examples from the just-finished context through the trained network, and the inner expectation over y by the model's predicted class probabilities. So per example I run a forward pass to get the output logits, take softmax to get the per-class weights p_theta(y=c|x), and for each class c I form the negative log-likelihood of class c, backprop it to get the gradient, square the gradient elementwise, and add it weighted by that class's softmax probability. Summing the squared gradients of every class weighted by its own predicted probability is exactly the Monte-Carlo estimate of the inner expectation over y — this is the "all labels" estimator, the true Fisher rather than the cheaper empirical Fisher that would use only the observed label. I do this in eval mode (so dropout and the like don't perturb the curvature estimate), over a capped number of single-example passes for speed, and I normalise the running sum by the number of examples I actually used. The reason this is affordable on a big net is property (b) of the Fisher: it's all first-order gradients, no Hessian, one diagonal scalar per weight.

The loop follows. After a context finishes: zero an accumulator the shape of the parameters; eval mode; for each of up to N examples, forward, softmax the logits to class weights, and for each class c backprop -log p(c|x) and add softmax_c · grad^2 into the accumulator; divide by the number of examples actually processed; then fold into the running summary with the decay, F* ← gamma F*_old + F_new. At every training step of the next context, the penalty is 0.5 · gamma · Σ over params of F* · (theta - theta*)^2, where theta* is the snapshot from the previous boundary. And the framework I'm dropping into accumulates whatever I return *additively* into its running importance buffer — so to make its additive accumulator end up holding gamma F*_old + F_new, I return the *increment* relative to what's already stored: I compute the full decayed-plus-new Fisher gamma·existing + F_new myself, then subtract the existing value so that "existing + increment" lands exactly on the decayed sum. Trace it on one parameter with gamma = 0.9, because an off-by-one in what I subtract would silently corrupt every later summary. Context 1, F_new = 1.0, nothing stored yet: I return 1.0, the loop adds it to nothing, stored = 1.0 = F*. Context 2, F_new = 0.5: I want stored to become gamma·1.0 + 0.5 = 1.4, so I return (gamma·1.0 + 0.5) − 1.0 = 0.4; the loop does 1.0 + 0.4 = 1.4. Context 3, F_new = 2.0: I want gamma·1.4 + 2.0 = 3.26, so I return 3.26 − 1.4 = 1.86; the loop does 1.4 + 1.86 = 3.26. The stored accumulator walks 1.0 → 1.4 → 3.26, exactly the decayed sum I wanted, so returning the increment under the add-don't-replace convention is correct.

The two hooks follow directly from what is already on the page: the estimator runs the all-label Fisher computation above and returns the increment (gamma · existing + F_new) − existing, and the per-step hook returns 0.5 · gamma · Σ over params of F* · (theta - theta*)^2 against the latest snapshot. Both are written out in full in the answer.
