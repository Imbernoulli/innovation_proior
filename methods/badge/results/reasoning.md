Let me start from what actually hurts. I have a big pool of unlabeled examples and a deep net, and labels cost money, so each round I get to pick a batch of B examples, pay for their labels, retrain, and repeat — and I want to hit a target accuracy with as few labels as possible. The reason I'm forced into batches at all is worth pinning down, because it shapes everything. Classic active learning queries one point, refits, queries the next. For a network with millions of parameters that's hopeless on both ends: one labeled example barely moves the model, and retraining a deep net to convergence after every single query is computationally absurd. So I have to acquire B at a time. And the moment I do that, a failure mode appears that the one-at-a-time world never had: if I score each example on its own and take the top B, nothing stops the B winners from being near-duplicates of each other. They'd all sit in the same confusing corner of input space, all "informative" by the same measure, and a single label would have told the model what all B were going to tell it. I'd have paid for B labels and learned roughly one label's worth.

There's a second constraint I keep underestimating, and it's specific to this setting. In ordinary supervised learning I tune hyperparameters on validation data for free. Here, if I change a hyperparameter, the algorithm queries a *different* set of points, and I have to buy those labels. A hyperparameter sweep is a label-budget sweep. So whatever I build has to "just work" at fixed settings — every knob I add that needs per-dataset tuning is a knob I can't actually afford to turn. That pushes me hard toward a method with *no* free tradeoff parameter.

So what's on the table. Two camps, and the uncomfortable fact is that each only works in part of the world. Uncertainty sampling: score each point by how unsure the model is — least confidence 1 − max_i p_i, or the margin p_(1) − p_(2) between the top two class probabilities, or the entropy H(p) = −Σ_i p_i log p_i — and grab the most uncertain B. The intuition is sound: confidently-classified points teach nothing, the action is near the decision boundary. And empirically this is strong at small batch and with simple models. But it's exactly the method that piles up duplicates: the score is a function of *one* example, there's no term that even looks at the other chosen points, so at large batch it floods the batch with one cluster of uncertain look-alikes. Plus deep-net softmax outputs are notoriously overconfident, so the raw uncertainty numbers are shaky. The other camp is representative or diversity sampling — cover the unlabeled distribution so that training on the batch approximates training on the whole pool, regardless of labels. The cleanest version of this is Core-Set: bound the risk of the subset-trained model by training error plus generalization error plus a "core-set loss," show that minimizing the core-set loss reduces to the k-Center objective — pick B centers minimizing the largest distance from any pool point to its nearest center — and solve it greedily by furthest-first traversal, all in the network's penultimate-layer feature space z(x). This is great at large batch and when the architecture has strong inductive biases so z is meaningful. But it's pure geometry: it has no idea whether a point is informative, so it'll happily spend budget covering regions the model already gets right, and when z isn't meaningful — hard data, weak net — its "representative" batch can be *worse* than random. And which regime am I in? That depends on the data statistics I don't know in advance. So I can't even reliably pick which camp to join.

People have tried to get both. There's a meta-approach, Active Learning by Learning (Hsu & Lin), that runs a bandit to choose at each round whether to fire the diversity rule or the uncertainty rule — but it can only pick among the rules it's handed, it never builds one criterion that has both properties, and it inherits whatever weakness the chosen base rule has right now. What I actually want is a single quantity per example that simultaneously encodes "the model is uncertain about this" and lets me build a batch that's "spread out," with no coefficient I have to tune to balance them.

Let me think about what "the model is uncertain / this point is informative" even means for a net trained by gradient descent. The whole machine moves by gradients: an example that's informative is one that, once I know its label, induces a large gradient of the loss and therefore a large update to the parameters. So the natural currency of informativeness is gradient magnitude. If I knew the true label y of x, I'd compute the gradient g_x^y = ∂/∂θ ℓ(f(x;θ), y) and call ‖g_x^y‖ the example's pull on the model. That's basically the expected-gradient-length idea (Settles): score by gradient size. EGL even handles the unknown label by averaging over it under the model, EGL(x) = Σ_y p(y|x)‖∇_θ ℓ(x,y)‖. But two things bug me about EGL for my purpose. First, it's still a per-example scalar score — it has the exact duplicate-batch disease, no diversity term at all. Second, it averages a full-parameter gradient over all K labels, which is a lot of computation, and the full-θ gradient is a giant object. Let me not take the whole gradient. The cheapest informative slice is the gradient with respect to just the *last* layer's weights, and I want to see whether that slice carries enough structure to be worth the saving.

Set up the network as f(x;θ) = softmax(W z(x;V)), where z(x;V) ∈ R^d is the penultimate-layer embedding and W ∈ R^{K×d} is the final linear layer, row W_i for class i. Cross-entropy loss against a label y is ℓ_CE = −log p_y = log(Σ_j e^{W_j·z}) − W_y·z. I want ∂ℓ_CE/∂W_i. The first term: ∂/∂W_i log Σ_j e^{W_j·z} = (e^{W_i·z}/Σ_j e^{W_j·z}) z = p_i z. The second term −W_y·z contributes −z to the i = y block and nothing elsewhere, i.e. −I(y=i) z. So the i-th block of the last-layer gradient is

  (g_x^y)_i = (p_i − I(y = i)) z(x;V).

Stare at this. The whole gradient embedding is an outer product: g_x^y = (p − e_y) ⊗ z, the probability-residual vector r = p − e_y tensored with the penultimate embedding z. That decomposition is more than a convenience. The z factor is *exactly* the representation space Core-Set lives in — it carries where the example sits, its diversity/identity. The r factor carries how wrong/uncertain the model is — if the model is confident and right, p ≈ e_y, so r ≈ 0; if it's unsure, p is spread out and r is large. So a single object already contains both of the things the two camps were fighting over: representation in z, uncertainty in r, multiplied together coordinate-blockwise. I didn't have to choose a coefficient to balance them — they're entangled inside one vector. That's promising enough that I want to keep pulling on this thread rather than the full-θ gradient.

But there's the obvious snag: this needs the true label y, which is the whole point — I don't have it. EGL's answer is to average over y. Let me try something cheaper and see if it's defensible: just hallucinate the model's own current prediction, ŷ = argmax_i p_i, plug it in, and use g_x = g_x^{ŷ}. One gradient per example, no averaging. The question I have to answer before trusting this is whether the hallucinated-label norm relates in any controlled way to the norm I'd get from the true label — because the norm is what I'm about to use as the uncertainty signal, and if hallucinating the wrong label can inflate that norm arbitrarily, the signal is junk. Let me compute the norm as a function of y.

  ‖g_x^y‖^2 = Σ_i (p_i − I(y=i))^2 ‖z‖^2.

Expand the inner sum: Σ_i (p_i − I(y=i))^2 = Σ_i p_i^2 − 2 Σ_i p_i I(y=i) + Σ_i I(y=i)^2 = Σ_i p_i^2 − 2 p_y + 1. So

  ‖g_x^y‖^2 = (Σ_i p_i^2 + 1 − 2 p_y) ‖z‖^2.

Let me make sure I didn't slip in that derivation before I lean on it. I take a concrete point — K=3 classes, z = (0.5, −1.2), last layer W = [[0.3,0.8],[−0.5,0.1],[0.2,−0.4]] — which gives p = (0.1523, 0.2364, 0.6113). I form the full K·d gradient vector g_x^y = (p − e_y) ⊗ z directly and dot it with itself, for each of the three possible labels, and compare to (Σ p_i^2 + 1 − 2 p_y)‖z‖^2:

  y=0: direct ‖g‖^2 = 1.940557, formula = 1.940557
  y=1: direct ‖g‖^2 = 1.656106, formula = 1.656106
  y=2: direct ‖g‖^2 = 0.388961, formula = 0.388961

They agree to all digits, so the identity is right. Now the thing I actually care about: the only y-dependent piece of that expression is −2 p_y, so to *minimize* ‖g_x^y‖ over y I want to *maximize* p_y — argmin_y ‖g_x^y‖ is the argmax of p_y. In the numeric example the three norms are 1.9406, 1.6561, 0.3890, smallest at y=2, and argmax_y p_y is also class 2 — they coincide, as the algebra says they must. So the label I hallucinate, ŷ = argmax p, is exactly one that gives the *smallest* possible gradient norm. (If several classes tie for maximum probability, every tied class is a minimizer; an implementation can pick any deterministic tie-break, and NumPy's argmax takes the first.) That's the property I needed: for the unknown true label y, ‖g_x‖ = ‖g_x^{ŷ}‖ ≤ ‖g_x^y‖. The hallucinated-label gradient norm is a *lower bound* on the gradient norm the example will really induce once I see its label — so when I pick a point because its g_x is large, it's guaranteed to produce at least that much last-layer gradient. The bound only ever understates, never inflates; the failure mode I was worried about can't happen.

And I should check the other end — does this norm actually behave like an uncertainty score, large when unsure and small when confident? From the formula, a confident point has p ≈ e_ŷ, so Σ p_i^2 ≈ 1 and 2 p_ŷ ≈ 2, making Σ p_i^2 + 1 − 2 p_ŷ ≈ 0. Putting numbers on it with the same z (‖z‖^2 = 1.69): p = (0.02, 0.01, 0.97) gives ‖g‖^2 = 0.0024; the fully one-hot p = (0,0,1) gives exactly 0; and the maximally unsure p = (1/3,1/3,1/3) gives ‖g‖^2 = 1.127. So the gradient embedding nearly vanishes for confident points and is two-to-three orders of magnitude larger for flat ones. ‖g_x‖ is small for confident examples and large for uncertain ones, exactly as an uncertainty score should be — and it's a *conservative* one, by the lower bound. The cheap, lazy choice of label turns out to be the conservative-estimate choice. I'll commit to last-layer gradient embedding at the hallucinated label, g_x = (p − e_ŷ) ⊗ z.

Now the batch. I have, for every pool point, a vector g_x whose *length* means uncertainty and whose *direction* (mostly inherited from z and from which class block r lights up) means identity/representation. I want a batch of B points that are individually long (uncertain) and collectively spread out (diverse), and — the binding constraint — with no tradeoff knob. What's the natural mathematical object for "a set that is high-quality and diverse at once"? A determinantal point process. Sample a size-B set Y with probability ∝ det(L_Y), where L_Y is the Gram matrix of the chosen {g_x}. Why would that be the right object? Because the determinant of a Gram matrix of vectors equals the squared product of their lengths times the squared volume of the parallelepiped they span. The length part rewards big gradients — uncertainty. The volume part is large only when the vectors point in genuinely different directions — diversity — and collapses to zero the instant two chosen vectors are parallel. So det(L_Y) is *simultaneously* a quality score and a diversity score, fused, with no coefficient weighting one against the other. There's a second thing I get for free: it self-adjusts to batch size. When B is small, the volume constraint is easy to satisfy, so the length factor dominates and the sampler leans toward the longest vectors — uncertainty-sampling-like behavior; when B is large, spanning a B-dimensional volume with non-parallel vectors gets hard, so the diversity factor takes over. That's the regime-adaptive behavior I was failing to engineer by hand — small-batch → uncertainty, large-batch → diversity — falling out of the determinant automatically. So the k-DPP over {g_x} is the criterion I'd most like to draw from.

So why not just sample from a k-DPP over the gradient embeddings and call it done? Cost. Sampling from a k-DPP is genuinely expensive — exact samplers are high-order polynomial in the batch size and the embedding dimension, MCMC samplers have a mixing-time problem, and at the large batch sizes I care about it simply blows up memory. The criterion is right but I can't afford to draw from it. So I need something that *behaves* like the k-DPP — favors long, mutually-spread-out vectors — but is cheap and, ideally, has no knobs.

Here's where an old, unrelated tool snaps into place. k-means++ seeding. Its job in life is to initialize Lloyd's algorithm: pick the first center uniformly, then repeatedly sample the next center from the data with probability proportional to its *squared distance to the nearest already-chosen center* — D^2 weighting. In that standard uniform-first form it has the 8(ln k + 2) expected-potential guarantee. I should not automatically transfer that theorem to the variant I'm about to use, because I want the first point to be the largest gradient instead of a uniform draw — that changes the seeding distribution and the guarantee is stated for the uniform-first version. But the D^2 recurrence itself is still the right primitive to inspect. After I've chosen some points, the next point is drawn with probability ∝ (distance to the nearest chosen point)^2. A point that's far from everything already in the batch is much more likely to be picked — that's diversity, and the squaring sharpens it. And the magnitude is in there too, once I'm careful about where distances are measured from: a long gradient vector tends to be far from the others, and if I seed the very first point by its length rather than uniformly, I plant the batch on the single most uncertain example and let D^2 spread out from there. So k-means++-style seeding in gradient space pulls toward the same kind of set a k-DPP would — high-magnitude and diverse — but as a handful of cheap passes over the data with no matrix determinant, no inverse, no MCMC, and no tradeoff hyperparameter.

Let me make the seeding concrete and actually run it on a tiny pool, because I want to see the recurrence do what I'm claiming, not just assert it. The first center: instead of uniform, take the point with the largest ‖g_x‖. Then for each subsequent pick maintain, for every pool point, D(x) = the distance from g_x to its nearest already-chosen gradient embedding, and sample the next point with probability D(x)^2 / Σ D^2; already-chosen points get distance zero so they can't be re-picked. I lay out five points with hand-set (r, z) so I know which has the biggest norm — point 2 has ‖g‖^2 = 36, the rest are ‖g‖^2 = 1 except a near-zero one. Running the seeding for a batch of three:

  step 0: picks point 2 (the largest-norm point, as designed), its own D set to 0;
  step 1: D^2 over the other four is (6.08, 6.08, —, 6.0, 5.57), the draw lands on point 3;
  step 2: chosen points pinned to 0, D^2 becomes (1.0, 0.98, 0, 0, 0.99), the draw lands on point 4;
  final batch [2, 3, 4], all distinct.

Two things I learn from actually running it rather than narrating it. First, the mechanics are exactly as intended: the most uncertain point seeds the batch, chosen points are excluded, and the output is a valid distinct set. Second — and this corrects a sloppy mental model I had — the D^2 draw is *not* "pick the farthest remaining point." At step 1 the four candidates have nearly equal D^2 and the random draw happened to take one that wasn't the strict maximum. D^2 only *biases* toward far (and, through the seeding, long) points; it's a randomized rule, not furthest-first. That's actually the feature I want — furthest-first traversal is deterministic and is what k-Center/Core-Set already does — so this being a *sampler* is what differentiates it from pure geometry.

Now, does this cheap surrogate actually track the k-DPP, or am I just hoping? I can't run a real k-DPP sampler here at scale, so I won't pretend I measured the two side by side. But I can say precisely what I'd want to check and why I expect it to come out favorably. Both samplers are supposed to produce batches that are *diverse* (large Gram determinant of the selected {g_x}) and *high-magnitude* (large average ‖g_x‖). The right experiment is to draw many batches from each on real pool embeddings and compare those two statistics. My expectation: the k-means++ batches should land close to the k-DPP batches on both axes, because both are pulled by the same two forces (length and spanned volume) — the k-DPP through det(L_Y) directly, the surrogate through D^2 seeding from the longest point. If anything I'd expect the k-means++ batches to be a hair *more* extreme on both axes than the k-DPP, since the k-DPP's stochasticity occasionally admits a mediocre set, whereas the greedy length-seeding always anchors on the most uncertain point. And I'd expect furthest-first traversal — the k-Center sampler Core-Set uses — to come out markedly *less* diverse and lower-magnitude in this gradient space, because it optimizes coverage of z alone and never sees the r factor. That's the comparison I'd run before publishing; here I can only verify the surrogate's mechanics (done above) and its zero-knob, low-cost profile, and flag the head-to-head diversity/magnitude comparison as the empirical check I'd most want to see.

There's still a gap I want to close with something I *can* reason through end to end: *why* does diversifying these gradients beat plain uncertainty sampling at the level of the actual model updates? The lower-bound argument tells me the magnitudes are meaningful, but not why spreading them out helps. Take binary logistic regression, the stripped-down case where I can follow the gradients exactly: Y = {−1,+1}, predictive probability p_w(y|x) = σ(y w·x), and the hallucinated label is +1 when p_w(+1|x) > 1/2 and −1 otherwise. The gradient of the logistic loss at the hallucinated label is ĝ_x = ∂/∂w log(1 + e^{−ŷ w·x}) = (1 − p_w(ŷ|x))·(−ŷ x), and at the true label g̃_x = (1 − p_w(y|x))·(−y x). Now restrict attention to the points right on the margin, S_w = {x : w·x = 0} — the genuinely uncertain ones, where p_w(+1|x) = p_w(−1|x) = 1/2 and my tie-break (the `> 1/2` test) hallucinates −1. On that set the hallucinated and true gradients differ only by a sign: ĝ_x = s_x g̃_x with s_x ∈ {±1}. Does a sign flip matter to the sampler? The sampler only ever sees gradients through Gram matrices (the k-DPP through det(L_Y), my surrogate through pairwise distances), and flipping the sign of some columns of a matrix G leaves G Gᵀ — hence its determinant — unchanged; I checked this on a random 4×5 G, det(GGᵀ) = det((−G)(−G)ᵀ) = 8.4394 to the digit. So on the margin set, sampling diverse sets of the *hallucinated* gradients is the same as sampling diverse sets of the *true-label* gradients. Why does that buy anything? Because two facts about that regime are already known: sampling the *uncertain* (low-margin) points makes the procedure behave like preconditioned stochastic gradient descent on the expected 0-1 loss (Mussmann & Liang), and sampling those gradients *diversely* via a determinant *reduces the variance* of the resulting mini-batch gradient estimate (the determinantal mini-batch line). Stack them: restricted to the low-margin region, my method runs the same descent on the 0-1 loss that uncertainty sampling runs, but with a lower-variance update because the batch is diverse rather than redundant. So diversity isn't a separate goal bolted on — it makes the *uncertainty* updates better. And that's exactly why grabbing one cluster of near-identical uncertain points is wasteful: their gradients are nearly collinear, so the batch's combined update is high-variance and mostly one direction repeated.

Now let me write the thing as an algorithm and then as code I'd actually ship. Each round, with the current model θ_t: for every unlabeled x, get the softmax p and the penultimate embedding z, form the residual r = p − e_ŷ (ŷ = argmax p), and the gradient embedding is g_x = r ⊗ z. The reference implementation uses the opposite residual e_ŷ − p, but that only multiplies every gradient embedding by −1 — and I just confirmed a global sign flip leaves every Gram determinant unchanged, and it obviously leaves norms and pairwise distances unchanged too, so the two conventions select identically. Then run k-means++-style seeding on {g_x} to pick B points: first the largest-‖g‖ point, then D^2-weighted draws. Query those, add to the labeled set, retrain from scratch, repeat.

One implementation worry before I code it: the gradient embedding g_x = r ⊗ z is a K·d-dimensional vector per example. For K classes and a few-hundred-dimensional z that's fine to materialize, and the most transparent code does exactly that — build the [n, K·d] matrix of gradient embeddings and run k-means++ on it. But I can avoid ever forming the outer products, which matters when K·d is large, by exploiting the factorization. The only thing k-means++ ever needs is *distances* between gradient embeddings, and for outer-product vectors those collapse. For g_a = r_a ⊗ z_a and g_b = r_b ⊗ z_b,

  ⟨g_a, g_b⟩ = Σ_{i,j} (r_a)_i (z_a)_j (r_b)_i (z_b)_j = (Σ_i (r_a)_i (r_b)_i)(Σ_j (z_a)_j (z_b)_j) = (r_a·r_b)(z_a·z_b),

so the K·d-dimensional inner product is just the product of a K-dimensional inner product and a d-dimensional one. Let me confirm this collapse isn't wishful index-pushing: with random r_a,r_b ∈ R^3 and z_a,z_b ∈ R^4, the full K·d=12-dimensional ⟨g_a,g_b⟩ = 4.507871 and (r_a·r_b)(z_a·z_b) = 4.507871 — match — and the squared distance ‖g_a−g_b‖^2 = 49.881487 computed in the full space equals ‖r_a‖^2‖z_a‖^2 + ‖r_b‖^2‖z_b‖^2 − 2(r_a·r_b)(z_a·z_b) = 49.881487. So in particular ‖g_a‖^2 = ‖r_a‖^2‖z_a‖^2, and the squared distance I need everywhere is

  ‖g_a − g_b‖^2 = ‖g_a‖^2 + ‖g_b‖^2 − 2⟨g_a,g_b⟩ = ‖r_a‖^2‖z_a‖^2 + ‖r_b‖^2‖z_b‖^2 − 2 (r_a·r_b)(z_a·z_b).

So I keep z and r separately, precompute the per-point squared norms ‖z‖^2 and ‖r‖^2, and compute any distance from one z·z and one r·r dot product — never touching the K·d space. The first center is the argmax of ‖g‖^2 = ‖r‖^2 ‖z‖^2, the most uncertain point, exactly as designed. Numerically the distance-squared can come out slightly negative from floating point, so I clip it at zero before the square root. There is one degenerate case to be honest about: if all remaining embeddings have zero distance to the chosen set, the D^2 distribution is undefined and mathematically any remaining tied point is equivalent. The canonical code does not special-case this; if I claim reference faithfulness, I should not hide a robust fallback inside the artifact.

The version I'd really run is therefore the factored reference version: it never builds the K·d embeddings, and it computes distances from the separate z (embedding) and r (probability-residual) factors.

```python
import numpy as np
from scipy import stats


def _distance(R, Z, center):
    # Distance from every (r,z) to a chosen (r0,z0), using the squared-distance
    # identity and then taking a square root, matching the reference helper.
    (r_vec, r_n2), (z_vec, z_n2) = R, Z
    (r0_vec, r0_n2), (z0_vec, z0_n2) = center
    dist = r_n2 * z_n2 + r0_n2 * z0_n2 - 2.0 * (r_vec @ r0_vec) * (z_vec @ z0_vec)
    return np.sqrt(np.clip(dist, a_min=0, a_max=None))   # clip FP-negative dist^2


def query(self, n):
    idxs_unlabeled = np.arange(self.n_pool)[~self.idxs_lb]
    # penultimate embeddings z and softmax probabilities p for the pool
    embs, probs = self.get_embedding(self.X[idxs_unlabeled],
                                     self.Y.numpy()[idxs_unlabeled],
                                     return_probs=True)
    embs, probs = embs.numpy(), probs.numpy()
    m = len(idxs_unlabeled)

    # r = e_yhat - p; this global sign flip preserves norms and Gram matrices.
    z_n2 = np.sum(embs ** 2, axis=-1)               # ||z||^2 per point
    yhat = np.argmax(probs, axis=-1)
    r = -1.0 * probs
    r[np.arange(m), yhat] += 1.0                    # residual; large when uncertain
    r_n2 = np.sum(r ** 2, axis=-1)                  # ||r||^2 per point

    R = (r, r_n2)
    Z = (embs, z_n2)

    chosen, chosen_list, mu, D2 = set(), [], None, None
    for _ in range(n):
        if len(chosen) == 0:
            # first center: largest ||g||^2 = ||r||^2 * ||z||^2  (most uncertain)
            ind = int(np.argmax(r_n2 * z_n2))
            mu = [((r[ind], r_n2[ind]), (embs[ind], z_n2[ind]))]
            D2 = _distance(R, Z, mu[0]).ravel().astype(float)
            D2[ind] = 0.0
        else:
            newD = _distance(R, Z, mu[-1]).ravel().astype(float)
            D2 = np.minimum(D2, newD)               # nearest-chosen-center distance
            D2[list(chosen)] = 0.0
            dist = (D2 ** 2) / sum(D2 ** 2)         # sample proportional to D^2
            sampler = stats.rv_discrete(name='custm', values=(np.arange(m), dist))
            ind = int(sampler.rvs(size=1)[0])
            while ind in chosen:
                ind = int(sampler.rvs(size=1)[0])
            mu.append(((r[ind], r_n2[ind]), (embs[ind], z_n2[ind])))
        chosen.add(ind)
        chosen_list.append(ind)

    return idxs_unlabeled[chosen_list]
```
