I start from the serving bottleneck rather than from the training recipe. During generation I do not process a whole sequence symmetrically; I append one token, compute its query, and compare it with keys and values that have accumulated from earlier tokens. The query is ephemeral. The keys and values persist, and every decode step streams them again. So there are two knobs with different units: how wide the cache is per position, and how many positions the cache and attention are allowed to cover. I want to attack both.

First I try to reduce the per-position width. In ordinary multi-head attention with $h$ heads, every cached position has $h$ key vectors and $h$ value vectors. But the current step only creates the current queries; it does not cache queries. That asymmetry is the opening: nothing forces the number of stored key/value heads to equal the number of query heads. Suppose I store $g$ key/value heads and keep $h$ query heads, with each stored key/value head serving $h/g$ query heads. The cache width is then $2g$ vectors per position instead of $2h$, a reduction of $h/g$. The boundary case $g=1$ is multi-query attention; it minimizes bandwidth but collapses every query head onto a single shared key/value subspace, which is a blunt change to what the heads can specialize on. So $g$ is a real dial, not a binary, and I should pick an interior value. With $h=32$, taking $g=8$ gives a width factor of $32/8 = 4$ while leaving eight distinct key/value subspaces — and that factor is exactly 4, not 4 compounded twice into 8: dense storage per position per layer is $2h = 64$ head-vectors; with $g=8$ it is $2g = 16$; the ratio is $64/16 = 4$, because $K$ and $V$ shrink by the same $h/g$ and contribute a single shared factor rather than two independent ones. Decode-time bandwidth reads exactly these stored vectors, so it falls by the same 4. The attention math is unchanged: I still want 32 query-compatible heads, so after I read the 8 stored key/value heads I repeat each one $h/g = 4$ times just before the attention kernel. This is grouped-query attention, and it is the per-position-width answer.

That only solves a constant factor. If a sequence has length $n$, dense causal attention still reads and scores against an ever-growing history, so the cost per decode step keeps rising with $n$ and prefill is quadratic. The question is whether old positions truly have to remain directly visible at every layer. Try the cheapest restriction: a fixed local band, where at position $i$ a query may attend only to the latest $W$ positions, using the convention $j \in [\max(0, i-W+1), i]$ so there are at most $W$ visible keys. Then prefill attention is $O(nW)$ for fixed $W$ instead of $O(n^2)$, and a decode step reads at most $W$ positions instead of all prior positions. The cost story is clearly better. The worry is correctness: if one layer only ever sees $W$ positions, a token far in the past looks unreachable, and I would be silently truncating the model's memory.

The restriction is per layer, not per model, so what matters is what the stack does to that horizon across depth. I trace a small instance: window $W=3$, several layers, and follow which original input positions can feed the top state at the last position. A layer-0 state at position $p$ depends on inputs $[p-W+1, p]$. A layer-1 state at $p$ attends to layer-0 states in $[p-W+1,p]$, each of which already pulled from its own window, so I take the union. Carrying that forward for the last position $p=11$:

```
after layer 0: pos 11 reaches inputs 9..11    span 3
after layer 1: pos 11 reaches inputs 7..11    span 5
after layer 2: pos 11 reaches inputs 5..11    span 7
after layer 3: pos 11 reaches inputs 3..11    span 9
after layer 4: pos 11 reaches inputs 1..11    span 11
```

So the reachable horizon does grow with depth, which settles the correctness worry: a local window is not the same as truncating history, because information moves one window deeper at each layer through a transitive chain, not through direct attention. But the trace also corrects a sloppy version of the claim I was about to make. The span is not exactly $kW$ after $k$ layers. Here it is $3, 5, 7, 9, 11$, which is $1 + k(W-1)$ with $W=3$: each layer above the first adds $W-1$, not $W$, because successive windows overlap by one position. So $kW$ is an over-estimate, and I should treat it only as the order-of-magnitude span, not an identity. At $W=4096$ and $32$ layers the rough figure is $32 \cdot 4096 = 131072$, about 131K tokens; the honest value $1 + 32 \cdot 4095 = 131041$ is within $0.02\%$ of it, so "about 131K" is fine to state as long as I do not dress it up as exact direct attention to 131K tokens.

Now the cache length should follow from the same fixed-window rule, and I want to push it down too. If a query at position $i$ can see at most the last $W$ positions including itself, then once position $i$ is written, the oldest key any future query will ever look at is position $i-W+1$. Position $i-W$ has just fallen outside every future local window, so keeping it cannot change any future attention result and its slot is free to reuse. That suggests storing only $W$ positions in a ring buffer: write absolute position $i$ to slot $i \bmod W$. But $i \bmod W$ scrambles physical order once the buffer wraps, so both the eviction-safety claim and time-ordered readout need checking together. Trace it with $W=4$, writing absolute positions $0$ through $6$:

```
slot = i mod 4:  0->s0, 1->s1, 2->s2, 3->s3, 4->s0, 5->s1, 6->s2
physical slots after writing 0..6:  [4, 5, 6, 3]
```

Two things to read off. First, the overwrite is exactly the safe one: writing position 4 lands on slot 0, evicting position 0, and at that point the live window for the query at position 4 is $\{1,2,3,4\}$, so position 0 was already invisible — eviction never drops a key that some current query still needs. The retained set after writing position 6 is $\{3,4,5,6\}$, and its oldest element is $3 = \text{seqlen} - W = 7 - 4$, matching the $i-W+1$ horizon I argued for. Second, the physical order $[4,5,6,3]$ is not chronological, so to splice this cache against a later prompt chunk I must rotate it back. The wrap point is $p = \text{seqlen} \bmod W = 7 \bmod 4 = 3$; concatenating slots $p{:}$ then slots ${:}p$ gives $[3,4,5,6]$, which is chronological. So the read path is: ring-write by $i \bmod W$, and unrotate by splitting at $\text{seqlen} \bmod W$ before interleaving.

Chunked prompt processing then needs care, because the cache update and the attention mask genuinely differ across cases — conflating them would be a silent bug rather than a slowdown. On the first prompt chunk there is no prior cache, so the mask is just block-diagonal causal attention, made local to the window. On a later prompt chunk, the keys presented to attention are the unrotated cached keys from previous chunks interleaved with the current chunk's keys, and the mask has to align from the bottom right so each new query sees the allowed suffix of prior cache plus the causal positions in the current chunk. On one-token decode, the cache is rectangular and padded to the per-layer cache size, and the mask supplies the true valid key lengths after the new token is written. These are three distinct cases; treating them as one generic "sliding mask" would hide the alignment detail and let a query attend to the wrong physical slots.

The cache update also has two indices, and the receptive-field trace above is the reason they must differ. Absolute positions are kept for rotary embeddings, because RoPE depends on true sequence position and the depth-expanded span only means anything if positions stay honest. Cache slots use those absolute positions modulo the cache size, plus a batch offset so different batch elements write into disjoint flattened ranges. When a chunk is longer than the cache size, only the last cache-size tokens of that chunk are written, because the earlier chunk tokens are already unreachable by the local window after the chunk has been consumed — the same eviction logic I verified above, applied within a single chunk.

These pieces compose into one system rather than staying separate tricks: grouped-query attention (32 query heads over 8 stored key/value heads) removes the constant factor in cache width, sliding-window attention turns score work from quadratic into linear in sequence length, $O(nW)$ for fixed $W$, while depth still carries information across roughly $kW \approx 131$K tokens, and the ring buffer turns that same fixed window into a fixed cache size — $8\times$ less cache at 32K tokens than a dense growing cache. Chunked prefill is what keeps all three exact once a prompt has to be split into window-sized chunks, since first prefill, later chunks, and single-token decode each need their own mask to keep the visible keys matching the window precisely. Everything outside these mechanisms is the ordinary pre-norm decoder: RMSNorm, rotary query/key embeddings, SwiGLU.
