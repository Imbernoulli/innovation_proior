I start from the serving bottleneck rather than from the training recipe. During generation I do not process a whole sequence symmetrically; I append one token, compute its query, and compare it with keys and values that have accumulated from earlier tokens. The query is ephemeral. The keys and values persist, and every decode step streams them again. So there are two knobs with different units: how wide the cache is per position, and how many positions the cache and attention are allowed to cover.

First I try to reduce the per-position width. In ordinary multi-head attention with $h$ heads, every cached position has $h$ key vectors and $h$ value vectors. But the current step only creates the current queries; it does not cache queries. That asymmetry means I can keep many query heads while storing fewer key/value heads. If I store $g$ key/value heads and keep $h$ query heads, each stored key/value head serves $h/g$ query heads, and the cache width falls by $h/g$. The extreme $g=1$ is multi-query attention. It is attractive for bandwidth, but collapsing every query head onto a single key/value subspace is a blunt change. I want the intermediate point: grouped-query attention. With $h=32$ and $g=8$, the cache stores 8 key heads and 8 value heads instead of 32 of each, so both cache memory and decode-time KV bandwidth fall by $32/8 = 4$. The attention computation still has 32 query heads because the 8 stored key/value heads are repeated after they are read, just before the attention kernel.

That only solves a constant factor. If a sequence has length $n$, dense causal attention still reads and scores against an ever-growing history. I need to decide whether old positions truly have to remain directly visible at every layer. A fixed local band is the obvious cost reduction: at position $i$, allow only the latest $W$ previous-layer states, using the operational convention $j \in [\max(0, i-W+1), i]$ so there are at most $W$ visible keys. Then prefill attention is $O(nW)$ for fixed $W$ instead of $O(n^2)$, and a decode step reads at most $W$ positions instead of all prior positions.

The objection is immediate: if one layer only sees $W$ positions, a token far in the past seems unreachable. The stack changes that. At layer 1, position $i$ can depend on input positions roughly one window back. At layer 2, it attends to layer-1 states in its own window, and the oldest of those layer-1 states already summarized another window below it. Inductively, after $k$ layers the state at position $i$ can carry information from about $kW$ positions back. This is a transitive receptive field, not direct attention, so I should not claim every layer sees the entire long context. But it is enough to explain why a local window is not the same as truncating the model's effective history. With $W=4096$ and 32 layers, the theoretical span is $32 \cdot 4096 = 131072$, about 131K tokens.

Now the cache length follows from the same fixed-window rule. If a query at position $i$ can see at most the last $W$ positions including itself, then the oldest useful key/value after writing position $i$ is no older than roughly $i-W+1$. Position $i-W$ has just fallen outside every future local window. Keeping it cannot change any future attention result, so it can be overwritten. A ring buffer is the natural storage: write absolute position $i$ to slot $i \bmod W$. Before the buffer wraps, slots are chronological. After it wraps, slot order is physical rather than temporal, so for prefill chunks I recover chronological order by splitting at $p = \text{seqlen} \bmod W$ and concatenating slots $p{:}$ followed by slots ${:}p$. That is the unrotation step.

I need to be careful about chunked prompt processing, because the cache update and the attention mask are not the same in every case. On the first prompt chunk there is no prior cache, so the mask is just block-diagonal causal attention, made local to the window. On a later prompt chunk, the keys presented to attention are the unrotated cached keys from previous chunks interleaved with the current chunk's keys; the mask has to align from the bottom right so each new query sees the allowed suffix of prior cache plus causal positions in the current chunk. On one-token decode, the cache is rectangular and padded to the per-layer cache size; the mask supplies the true valid key lengths after the new token is written. These are three distinct cases, and treating them as one generic "sliding mask" would hide a real implementation detail.

The cache update also has two indices. Absolute positions are kept for rotary embeddings, because RoPE depends on true sequence position. Cache slots use those absolute positions modulo the cache size, plus a batch offset so different batch elements write into disjoint flattened ranges. When a chunk is longer than the cache size, only the last cache-size tokens of that chunk are written; earlier chunk tokens are already unreachable by the local window after the chunk has been consumed.

The pieces now compose. Grouped-query attention cuts cache width by four with 32 query heads and 8 key/value heads. Sliding-window attention cuts long-sequence score work to $O(nW)$ and gives a depth-expanded theoretical span of about 131K tokens at 32 layers and window 4096. The ring buffer makes cache length fixed at $W$, so a 32K sequence uses $32768/4096 = 8$ times less cache length than a dense growing cache, while an 8192-token prompt uses half the cache length. Chunked prefill keeps prompt ingestion bounded by using the same window-sized cache and the correct mask case for first prefill, later prefill, and decode. The architecture around those mechanisms stays the usual pre-norm decoder with RMSNorm, rotary query/key embeddings, and a SwiGLU feed-forward.
