The thing that keeps bothering me about how we train vision models is that the supervision is shaped like a multiple-choice exam written before the model is born. You pick a thousand categories, you pay people to put each photo into exactly one of those thousand buckets, you train a softmax with a thousand outputs, and now the model knows a thousand things forever. Want it to know a thousand-and-first? Collect a new labeled set, bolt on a new head, retrain. The label vocabulary is a ceiling, and worse, the prediction mechanism is welded to that ceiling — a static softmax over fixed classes has no way to emit something it wasn't built to emit. So "zero-shot" in vision means almost nothing.

Meanwhile, over in language, none of this is true anymore. You train on raw web text with an objective that needs no human labels — predict the next token, or fill in a masked one — and the thing just gets better and better as you pour in compute and data. And because everything is text in, text out, you can transfer to a brand-new task by *describing the task in words*. No new head, no new labeled set. The model learned the task as a side effect of pretraining, and you elicit it with a prompt. That's the property I want for vision: supervision that isn't capped by a label list, and a model whose output I can *specify at test time with language*.

So the goal is concrete: train a visual model where (a) the supervision comes from the open vocabulary of natural language, not a closed set of class ids; (b) it's cheap enough per example to swallow hundreds of millions of image–text pairs, because that's the only way to get to the scale where the NLP magic showed up; and (c) at test time I can hand it a few sentences describing the classes I care about and it builds a classifier from those, with no training.

Learning from paired text isn't a new idea, so let me actually go through what's been tried and figure out where each one runs out of road, because the gap is the whole point.

The oldest versions just predict words from the text paired with an image — Mori back in '99 predicting nouns and adjectives for retrieval, Quattoni learning representations off caption-word classifiers. Then Joulin and collaborators modernized it: take the messy metadata of web photos — title, description, hashtags — turn it into a bag of words, and train a CNN to predict that bag as a multi-label problem. And it works surprisingly well; the features rival supervised ImageNet pretraining on transfer. Good. But notice what it still is: a prediction over a fixed word vocabulary. And it has no built-in way to be re-pointed at arbitrary new classes — at the end you have a CNN with a multi-label word head, not something you can interrogate with a sentence.

Then Li and collaborators — Visual N-Grams — pushed on exactly the part I care about. They learn parameters for a big dictionary of visual n-grams and, crucially, they do *zero-shot transfer to standard classification datasets*: take each dataset's class names, convert them to n-grams, score them under the model, predict the top one. That's the first time anyone took a generically pretrained image-text model and pointed it at, say, ImageNet's class names with no ImageNet training. The architecture of the idea is right. The problem is the number: 11.5% on ImageNet. Classic pre-deep-learning vision gets ~50%; the supervised state of the art is ~88%. So 11.5% is a proof that the *interface* can exist, and a proof that, as built, it's nowhere near usable.

Why so weak? Two things jump out when I line up this whole family. One is that it predates transformers and runs at tiny data scale. The other — and this is the one I keep circling — is the *objective*. And there's a more recent, more promising-looking family: VirTex trains an image CNN plus a text transformer to *generate the caption*, autoregressively, the exact token sequence. Very expressive. ConVIRT, on medical images, instead uses a *contrastive* image–text objective. These are exciting as proofs of concept but still rare in practice, and the honest reason is they've only been run for accelerator-days on a hundred-or-two thousand images, while the weakly-supervised hashtag and JFT efforts run for accelerator-*years* on millions to billions. Nobody has run natural-language supervision at the scale where it could actually win. So scale is part of my plan no matter what. But scale makes the *cost per example* the thing that decides whether the plan is even feasible. Which sends me straight at the objective.

Let me just try the obvious thing first and see where it breaks. Start like VirTex: jointly train an image encoder and a text transformer from scratch to predict the caption of each image. I'll size the text model reasonably — call it a 63-million-parameter transformer language model. Right away I notice that's already twice the compute of its ResNet-50 image encoder, which is annoying because the image is supposed to be the point. Fine, run it. And here's the wall: I measure how fast it learns to recognize ImageNet classes, and a far simpler baseline — just predict the bag of words of the same text, the Joulin-style multi-label target — learns about three times *faster*. The expensive, expressive caption generator is the slow one.

Stare at this for a second, because it's telling me something. Why would generating the caption be so much harder than predicting its word-set? Because the same image co-occurs with wildly different text. One photo of a dog might be paired with "my dog at the beach," another near-identical one with "Rex, 3 years old, such a good boy!!! #goldenretriever," another with a camera-settings dump. To *generate* the exact sequence, the model has to model word order, function words, punctuation, style, the photographer's mood — an enormous amount of variation in *how* things are said, almost none of which is about *what's in the image*. The visual content is a thin signal buried in a thick distribution of phrasings, and an autoregressive objective makes me pay, token by token, for nailing all of that phrasing. There's a known echo of this: generative models of images can learn fine representations, but they need over an order of magnitude more compute than contrastive ones for the same quality. Generation is just an expensive way to learn a representation when you only care about the content.

So the objective is fighting me. What do I actually need? I don't need the words. I need the *association* — that this image goes with this text and not those other texts. And there's a relaxation hiding in that sentence. Instead of "produce the caption for this image," ask the much cheaper question: "of the texts I happen to have in this batch, which one belongs to this image?" That throws away every bit of effort spent on phrasing. I'm not asking the model to write; I'm asking it to *match*. There's also direct evidence this is the right move: contrastive objectives have been found to learn better representations than the equivalent predictive objective. So let me take the bag-of-words baseline and swap its predict-the-words objective for a contrastive match-the-text objective, and measure again. Another roughly 4× speedup in how fast it picks up ImageNet. Stacked on top of the 3× from dropping generation, that's around an order of magnitude, which makes web-scale training plausible instead of wishful.

Let me try to account for *why* matching is so much cheaper, because I'd rather understand the speedup than just bank on the measurement. Caption generation, per image, is a softmax over the whole vocabulary at every position of a length-L sequence — order L×V targets — and you only "win" by getting the entire sequence right. Matching, per image, is a *single* classification: among the N texts in this batch, point to the right one. One softmax over N items. No decoder, no per-token decoding loop. And the negatives — the wrong texts — are simply the *other examples already in the batch*. I don't run a single extra forward pass to get them; I already encoded them because they're the next training examples. So the supervision per image collapses from "reproduce these L tokens out of V each" to "pick 1 of N," and the N−1 distractors come for free with the forward pass I was doing anyway. That accounts for the bulk of the gap I measured — it's a structural change in what's being predicted, not a tuning trick — though I'd stop short of claiming I've *derived* the exact factor; the measured ~4× is the thing I'd actually trust.

Now let me actually build the objective from this matching idea and see what it has to be. Take a batch of N image–text pairs. Run the images through the image encoder, the texts through the text encoder. They land in different spaces with different dimensions, so project each into a shared d-dimensional space with a learned linear map, one for each modality. Now I want a similarity I can compare across pairs without magnitude getting in the way, so L2-normalize both sets of vectors; then the dot product of an image vector and a text vector is just their cosine similarity, living in [−1, 1]. Stack them up and the natural object is the N×N matrix of all image-to-text cosine similarities. The N diagonal entries are the true pairs; the N²−N off-diagonal entries are mismatches. I want the diagonal high and everything off it low.

How do I turn "diagonal high, off-diagonal low" into a loss? Read the matrix one row at a time. Row i is image i's similarity to all N texts. The right answer is text i. That's a classification problem: softmax over the row, cross-entropy against the label "i." Do that for every row and average — now every image is being trained to retrieve its own text out of the batch. But the relationship is symmetric: a text should also retrieve its image. So read the matrix the other way too, column by column: column j is text j's similarity to all N images, the right answer is image j, softmax over the column, cross-entropy against "j." Both modalities are first-class here — unlike a fixed-class softmax where there's one privileged "label" axis — so I average the image-side and text-side cross-entropies. The labels, in both directions, should just be the indices 0…N−1 if the matched pair really does sit on the diagonal.

I want to actually run this on a tiny batch rather than trust the indexing in my head. Take N=3, embedding dim 4. Build a deliberately easy case: make each text embedding its image embedding plus a little noise, L2-normalize both, scale by 1/0.07≈14.29, and look at the 3×3 logit matrix:

```
[[14.29   6.96   9.70]
 [ 7.15  14.28  -3.67]
 [10.38  -2.94  14.22]]
```

The diagonal is the largest entry in every row and in every column, so argmax along the rows gives (0,1,2) and argmax down the columns also gives (0,1,2) — labels = arange(N) is the right target in both directions, which is the thing I wanted to confirm before writing it down. The averaged loss here is ≈0.011, near zero, as it should be for an almost-perfect matcher. Now the opposite extreme: collapse all three text embeddings onto a single vector so the model has thrown away all discriminative information. Every row of the similarity matrix becomes constant, the softmax is uniform, and the per-row cross-entropy is −log(1/3) = log 3 = 1.0986 — and that's exactly the number that comes out. Good: the floor is log N and the loss only drops below it to the extent the diagonal genuinely separates from the off-diagonal. The objective is wired correctly and it punishes representational collapse rather than tolerating it.

This object isn't novel out of thin air — it's the multi-class N-pair loss from deep metric learning, one positive against many negatives scored by softmax, the same thing known as InfoNCE for contrastive representation learning, here made symmetric and applied across the image/text modalities the way it was recently used for medical image–text pairs. That it coincides with a well-studied objective is reassuring rather than load-bearing, but it does buy me one interpretive handle: the InfoNCE form is a lower bound on the mutual information between the matched image and text, so minimizing it is pushing the embedding to capture whatever is shared between picture and words — which is the right thing to be maximizing if I want the visual side to absorb the semantic content of the caption and ignore the phrasing.

But something nags at me as soon as I write "softmax over cosine similarities." The cosine similarities live in [−1, 1]. The largest possible gap between the right answer and a wrong one is 2. Let me actually work out how confident a *geometrically perfect* row can be. Put the positive at +1 and all N−1 negatives at −1 — the best the geometry can ever do. The probability the softmax assigns the diagonal is exp(1) / (exp(1) + (N−1)·exp(−1)), which tidies to exp(2)/(exp(2)+N−1). I don't trust my feel for that number, so I'll evaluate it. For a small batch, N=32, it's 0.192 — already under 20% even with a flawless geometry. Push to a realistic contrastive batch: at N=4096 it's 0.0018, and at N=32768 it's 0.000225. So the cross-entropy floor for a *perfect* row at the big batch is −log(0.000225) ≈ 8.4 nats. That is brutal: no matter how well the encoders separate the pairs, the loss can't get near zero, because the softmax over raw cosines structurally cannot concentrate probability on one item out of tens of thousands when the logits only span [−1,1]. The objective keeps treating already-separated pairs as uncertain, and gradients keep flowing on examples that are in fact solved. I need to scale the logits up before the softmax — multiply the similarities by some large factor (equivalently divide by a temperature τ) to widen their range. That factor is an inverse temperature, and it controls how peaked the softmax is.

Let me check that scaling actually fixes the ceiling I just computed, not just nudges it. With the positive at +s and negatives at −s, the perfect-row probability becomes exp(s)/(exp(s)+(N−1)·exp(−s)). At N=32768, s=1 gave 0.000225; bumping s to 1/0.07 ≈ 14.29 sends it to ≈1.0 (the CE floor collapses from 8.4 to ~0). So a fairly modest inverse temperature is enough to let the softmax express certainty at batch size 32k — the fix works, and works with room to spare.

What value? This is exactly the kind of thing that interacts with batch size, learning rate, and the geometry of the embedding space, and it changes over training as the embeddings spread out — so pinning it as a hand-tuned hyperparameter is both fragile and wasteful. Let the optimizer set it: make the inverse-temperature multiplier a learned parameter and train it jointly. The multiplier on the logits must stay positive — a negative scale would invert the whole objective — so I parameterize the log of it, a scalar t, and use exp(t) as the actual multiplier; now any real value of t gives a positive scale and the gradient is clean. logits = (image·text) · exp(t). Initialize t so the scale starts around 1/0.07, borrowing the 0.07 temperature that instance-discrimination work found sane for normalized-feature softmaxes.

Once the true pairs start separating from the impostors, the loss can always be nudged lower by making the softmax sharper, i.e. by cranking exp(t) up. Nothing in the objective pushes back on a runaway temperature — and from the ceiling computation above I can see the danger concretely: the same exp(s) that lifted a perfect-row probability from 0.0002 to 1.0 at modest s will, if s keeps climbing, send logit *differences* into the hundreds, where one mislabeled or noisy pair produces an enormous loss and an enormous gradient. So I expect an unclamped scale to make training unstable, and I cap it: don't let exp(t) exceed 100. I can't prove 100 is the right ceiling from first principles — it's a guardrail I'd tune if training still diverged — but it bounds the worst-case logit spread while leaving plenty of headroom above the ~14 the ceiling analysis says is already enough. With that, the core training objective is compact: embed, normalize, matrix-multiply, scale by exp(t), two cross-entropies against arange(N), average.

The training objective is only useful for transfer if I can turn it around at test time. Look at what pretraining gave me. The model answers one question extremely well: "does this text describe this image?" Classification is just that question asked K times. Suppose I want to sort an image into classes c₁…c_K with zero training examples. For each class, write its name as a piece of text, run it through the *text* encoder, normalize — that gives me a vector per class. Run the image through the *image* encoder, normalize. Score the image against each class vector with the same scaled cosine similarity I trained on, softmax over the K classes, take the argmax. No labels, no fitting — the classifier was assembled out of the class *names*.

Let me look hard at the shape of that prediction rule. It's image-vector dotted with one weight-vector per class, the inputs L2-normalized, the weights L2-normalized, no bias, an inverse-temperature scale, then softmax over classes. Line that up against a textbook multinomial logistic regression: features x, weight rows w_k, logits w_k·x, softmax. They match term for term — the class vectors are playing the role of the weight rows w_k — so this *is* a linear classifier, with the one difference that I never fit the weights by gradient descent on labels. The text encoder produced the weight vectors from language. That makes the text encoder a hypernetwork: feed it a description of a class and it emits the row of classifier weights for that class — the old "write a classifier" idea, here as the test-time face of a model that was only ever trained to match images and texts. And the open vocabulary I wanted is right here: any class I can describe in words I can classify, because under this rule describing a class *is* building its classifier weight.

There's a tighter correspondence worth pinning down, since it would explain *why* the zero-shot transfer should work at all rather than just asserting it will. Compare the two operations side by side. A training row: image i scored by scaled cosine against the N text embeddings in the batch, softmax, the correct text is the target. A zero-shot prediction: the image scored by scaled cosine against the K class-name embeddings, softmax, pick the argmax. These are the *same* computation — scaled-cosine-then-softmax of an image against a set of text vectors — with the batch's texts swapped for the task's class names. So every training step was already solving a tiny classification problem: a freshly-drawn, one-text-per-class task with N classes, the classes defined by whatever captions happened to be in the batch. Zero-shot at test time just supplies a fixed, curated set of "captions" (the class names) instead of a random batch. That the same operation is trained and then deployed is the concrete reason I'd expect transfer, not a hope.

There's a practical sweetener: I can compute those K class vectors once and cache them. Then classifying an image is one matrix-multiply against the cached weights. The cost of "building the classifier from language" is paid once and amortized over the whole dataset.

Now let me poke at the zero-shot mechanism, because the naive version — embed the bare class word — has problems I can predict. Polysemy is the obvious one. If I hand the text encoder just "crane," it has no idea whether I mean the bird or the construction machine; with no context the word is genuinely ambiguous, and there are datasets that contain *both* senses as separate classes. Same with "boxer" — the dog breed or the athlete? A single word doesn't carry the disambiguating context. There is also a distribution mismatch: in the pretraining data the text paired with an image is almost never a lone word — it's a sentence. So a bare label is *off-distribution* for the text encoder; I'm querying it with inputs unlike anything it saw. The fix for both is to wrap the label in a templated sentence. "A photo of a {label}." immediately tells the encoder the text is *about the content of a photo*, which nudges it toward the right sense and back onto its training distribution. I can inject task context the same way: "a photo of a {label}, a type of pet." for a pets dataset, "a satellite photo of a {label}." for overhead imagery, quotes around the characters for OCR. The class name was always the same; the prompt supplies the missing context.

Can I do better than one prompt? Yes — ensemble over several templates ("a photo of a big {label}", "a photo of a small {label}", …). But I want the ensemble *free*. If I averaged in probability space I'd have to run K-times-many text encodings per prediction. Instead, average the *embeddings*: for each class, encode all its prompt variants and average the resulting vectors into a single class vector (renormalized). Now the ensemble collapses back to one weight vector per class and one cached matrix, with the same inference cost as a single prompt. Same amortization trick, applied to ensembling.

The objective is now clear, but it still needs encoders that can spend compute where it matters. For the image side I'll support two backbones. A ResNet-50, modernized — the ResNet-D stem-and-downsample tweaks and antialiased blur-pooling, both known to help — and one real change: replace the final global-average-pool with an attention-pooling layer. Instead of averaging the spatial feature map into one vector, I run a single transformer-style multi-head QKV attention where the query is conditioned on the global-average-pooled feature; the pooled output is a learned, content-weighted aggregation of the spatial positions rather than a flat mean. The second backbone is a Vision Transformer, which on large data is more compute-efficient than a convnet — I follow it closely, adding one extra layer normalization on the combined patch-plus-position embeddings before the transformer for stability.

For the text side, reuse what already scales in NLP: a GPT-2-style Transformer over byte-pair-encoded tokens, a ~49k vocabulary, sequences capped at 76 tokens for efficiency, bracketed with start- and end-of-sequence tokens. I take the activation at the end-of-sequence position in the top layer as the text's summary vector — that token has attended over the whole sequence — then layer-norm and linearly project it into the shared space. One deliberate choice: I use *masked* (causal) self-attention in the text encoder even though I'm not doing language modeling here, so I keep the door open to initialize from a pretrained language model or add a language-modeling auxiliary objective if that becomes useful. And I scale only the text encoder's *width* with the image encoder, not its depth: the expensive part of the problem is visual recognition, while the text tower mostly has to map short descriptions into the shared space, so extra depth there is a weak place to spend compute.

A couple of simplifications relative to the contrastive image–text recipe I'm building on, each with a reason. That recipe used a *non-linear* projection head into the contrastive space, popularized by image-only self-supervised methods. Here the embedding space is not a disposable training head; it is exactly the space I will use for retrieval and zero-shot classification, so a linear projection is the cleaner default unless the non-linearity earns its cost. I also drop its function that samples a single sentence from the text, because most of my web pairs are already a single sentence, so there's nothing to sample. And I cut the heavy image augmentation down to a single random square crop — with hundreds of millions of pairs, overfitting just isn't the threat that would justify aggressive augmentation, and I'm training from scratch with no pretrained initialization on either tower for the same reason: at this data scale, initialization buys me nothing and removing it simplifies everything.

Scaling: when I make the image model bigger, I don't just widen it or just deepen it — I push width, depth, and input resolution up together, in the spirit of compound scaling, which has been shown to beat dumping all the extra compute into one dimension. Optimization is standard and known to scale: Adam with decoupled weight decay (applied to the weights but not to gains or biases), a cosine learning-rate schedule, a large minibatch, mixed precision. The large batch isn't incidental, by the way — each row of the similarity matrix is a classification against batch-minus-one negatives, so a bigger batch means harder, richer contrastive problems; the quality of the signal scales with the number of in-batch negatives, which is one more reason to push the batch as large as the hardware allows.

Let me write the core, the way I'd actually sketch it — numpy-like, because the whole method really is this small.

```python
# image_encoder: ResNet or Vision Transformer
# text_encoder : Transformer over BPE tokens
# W_i [d_i, d_e], W_t [d_t, d_e]: per-modality linear maps into the joint space
# t: learned log inverse-temperature scalar

# 1. encode each modality
I_f = image_encoder(I)        # [n, d_i]   from the batch of images
T_f = text_encoder(T)         # [n, d_t]   from the batch of texts

# 2. project into the shared space and L2-normalize so dot product = cosine
I_e = l2_normalize(I_f @ W_i, axis=1)   # [n, d_e]
T_e = l2_normalize(T_f @ W_t, axis=1)   # [n, d_e]

# 3. scaled pairwise cosine similarities; exp(t) keeps the scale positive,
#    and the inverse temperature is clamped to <= 100
logit_scale = min(np.exp(t), 100.0)
logits = (I_e @ T_e.T) * logit_scale    # [n, n], diagonal = true pairs

# 4. symmetric loss: every image picks its text (axis along texts),
#    every text picks its image (axis along images); labels are the diagonal
labels = np.arange(n)
loss_i = cross_entropy(logits, labels, axis=1)     # each image row picks its text
loss_t = cross_entropy(logits.T, labels, axis=1)   # each text row picks its image
loss   = (loss_i + loss_t) / 2
```

And the test-time classifier, built from language, reusing the very same scaled-cosine scoring:

```python
# zero-shot: turn class names into classifier weights via the text encoder
prompts        = [f"a photo of a {name}." for name in class_names]   # one (or many) per class
class_weights  = l2_normalize(text_encoder(tokenize(prompts)) @ W_t, axis=1)  # [K, d_e]
# (for ensembling: average several prompts' embeddings per class, then renormalize)

img            = l2_normalize(image_encoder(image) @ W_i, axis=1)    # [1, d_e]
class_logits   = (img @ class_weights.T) * min(np.exp(t), 100.0)     # [1, K]
prediction     = class_logits.argmax(axis=1)                         # no labels were fit
```

So the causal chain, start to finish: I wanted language-shaped, uncapped supervision and a classifier I could specify in words, which demanded web scale, which made per-example cost decisive; generating captions was too expensive because it models phrasing I don't care about, so I relaxed "produce the caption" to "match the right text among the batch," which is one batch softmax in each direction with free in-batch negatives; reading the resulting similarity matrix by rows and columns gave a symmetric cross-entropy with diagonal labels — the N-pair / InfoNCE objective across modalities; the raw cosine logits had too little dynamic range for a large-batch softmax, so I scaled them by a learned, log-parameterized, clamped inverse temperature; and then the trained text encoder turned out to be a hypernetwork that emits a linear classifier's weights from any class description, so zero-shot classification is just the same scaled-cosine matching asked once per class name — with templated, embedding-space-ensembled prompts to supply the context a bare word lacks.
