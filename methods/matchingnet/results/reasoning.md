I start with a classifier that learns in exactly the wrong place. Its categories live in fixed output weights, and the only way to add a category is to move weights with gradient descent. One labeled example is not enough evidence for that, and a few aggressive steps on a new class can overwrite the old representation. But the test condition does not let me fine-tune anyway. I need the support example itself to become usable immediately.

Nearest-neighbor methods already have that property. If I store a new example, it is instantly part of the classifier. The weakness is that all the intelligence sits in the distance function, and a fixed distance in pixel space or in a feature space trained for base-class softmax labels is only accidentally aligned with the new task. So I do not want to abandon non-parametric classification; I want to learn the space and the comparison rule while keeping the support set as the memory.

NCA gives me the first move. Hard nearest-neighbor classification is discontinuous in the metric, but a soft stochastic neighbor is smooth: `p_ij = exp(-||A x_i - A x_j||^2) / sum_{l != i} exp(-||A x_i - A x_l||^2)`, with `p_ii = 0`, and the probability of correct classification is the total mass on same-class neighbors. The sign is worth pausing on, because I will reuse this structure: NCA puts a minus on the squared distance, so that as a point gets nearer its mass goes up. If I later replace the negative distance with a positive similarity score, the inequality has to keep pointing the same way — more similar must mean more mass — or the soft vote will reward the wrong neighbors. That solves differentiability, but not the whole problem. A fixed linear map `A` and a leave-one-out objective over the training set are not shaped like an episode with a fresh support set.

So I make the support set the reference set directly. For a query `x_hat` and support `S = {(x_i, y_i)}_{i=1}^k`, I write the prediction as `y_hat = sum_i a(x_hat, x_i) y_i`. If each `y_i` is one-hot and the weights `a_i` are nonnegative and sum to one, then `y_hat` should be a distribution over the episode's labels. I want to be sure that is actually true and not just a hope, because the whole loss design downstream depends on the output being a probability vector. Take a concrete case: two classes, two shots each, so four support items with labels `[0, 0, 1, 1]` and one-hot rows stacked as the matrix `Y`. If the attention weights are `a = [a_0, a_1, a_2, a_3]` with `sum_i a_i = 1`, then `y_hat = a Y` has entry `n` equal to `sum_{i: y_i = e_n} a_i`. For class 0 that is `a_0 + a_1`; for class 1 that is `a_2 + a_3`; and the two entries sum to `a_0 + a_1 + a_2 + a_3 = 1`. I actually ran this with `a = [0.316, 0.238, 0.207, 0.238]`: the row sums to `1.000`, the matrix-times-labels gives `[0.555, 0.445]`, and `0.316 + 0.238 = 0.555` matches the class-0 entry exactly while `0.207 + 0.238 = 0.445` matches class-1. So the multiply by one-hot labels is literally the per-class sum of attention, and the output is a genuine distribution over the episode's labels. That also tells me how multi-shot works for free: the probability of class `n` is not a separate learned output unit, it is the total attention paid to support examples carrying label `n`. One-shot is just the special case of one term per class. No part of the parameterization changes with the shot count.

The weights should be smooth, so I use a softmax over similarities: `a_i = exp(c(f(x_hat), g(x_i))) / sum_j exp(c(f(x_hat), g(x_j)))`. This is the NCA soft-neighbor trick moved from a fixed training set onto the episode's support set, and it is here that I have to honor the sign concern from before. NCA exponentiated a negative distance; I am exponentiating `c` directly, so `c` has to be a similarity — larger when the query and a support item agree — for "more similar gets more mass" to hold. The embeddings `f` and `g` can be deep, so I am no longer limited to a Mahalanobis metric. The readout also has the right memory interpretation: support labels are values bound to support inputs, and the query reads a convex combination of those values.

What should `c` be? Raw dot product would let an embedding with a large norm dominate the vote regardless of direction, which is the magnitude problem I just argued against. Cosine removes norm from the comparison, so the vote depends on direction in feature space. But I should be careful about what the implementation actually computes, because the reference I am following does not normalize symmetrically: it normalizes the support embeddings before the final matrix multiply and leaves the query embeddings as they are. So the implemented score is `f^T (g / ||g||)`, not `(f/||f||)^T (g/||g||)`. Is that the same thing or not? Expand it: `f^T (g/||g||) = (f^T g)/||g|| = ||f|| * (f^T g)/(||f|| ||g||) = ||f|| * cos(f, g)`. So it is cosine scaled by the query norm `||f||`. I checked it on a single pair, `f = [0.8, 0.2, 0.6]`, `g = [0, 1, 1]`: the normalized-support dot product is `0.5657`, and `||f|| * cos(f,g)` is also `0.5657`, while the symmetric cosine alone is `0.5547` — they differ by the factor `||f|| = 1.020`. That factor is not a constant across queries, so it acts as a per-query softmax temperature on the vote. It is reference-faithful, but I should not write "cosine" in the code comments and pretend the query norm is gone; it is a scaled cosine.

Now I need training to match the test condition. If I train a base-class softmax and then hope its hidden layer supports nearest-neighbor transfer, I am back to borrowed features. Instead I sample tasks during training. Draw a label set `L` from a task distribution, draw a support set `S` and a query batch `B` from those labels, and maximize `E_{L} E_{S,B} sum_{(x,y) in B} log P_theta(y | x, S)`. The objective is an argmax of a log probability, so the implementation should minimize its negative. There is a sign subtlety I want to nail down rather than assert: the forward pass already produces a probability vector, then I take its log, so the network output is log-probabilities, not logits. Cross-entropy in a framework expects logits and applies its own log-softmax; feeding it my already-log'd probabilities would log-softmax them a second time. The matching loss therefore has to be NLL, which simply indexes the true class and negates. I traced it on the example above: log-probs `[-0.589, -0.809]` with true class 0 give `nll_loss = 0.589`, and `-log(0.555) = 0.589` — they agree, confirming NLL on these log-probs is exactly `-log P(true)`. So the derivation's argmax-of-log and the implementation's minimize-NLL are the same objective with the sign flipped, as long as the output stays log-probabilities.

At this point the classifier is already usable, but the embeddings are myopic. If each support vector `g'(x_i)` is computed independently, it cannot react to another support example of a different class sitting nearby. If the query vector `f'(x_hat)` is computed independently, it cannot emphasize the features that distinguish the particular candidate classes in this episode. The embedding functions themselves should be conditional on the support set.

For support embeddings, I run a bidirectional LSTM over the support features and add a residual connection: `g(x_i, S) = h_forward_i + h_backward_i + g'(x_i)`. The forward recurrence reads from the start up to `i`, the backward recurrence reads from the end down to `i`, and their sum gives each position context from the whole support sequence. The skip connection matters because the recurrent part only has to learn a contextual correction; the raw feature is preserved. There is a wrinkle I should not gloss over: the support set has no natural order, but an LSTM does impose one. I am accepting that the bidirectional pass with a residual is meant to wash most of the order-dependence out rather than eliminate it — the residual guarantees that if the recurrent contribution is small, the embedding falls back to the order-free raw feature. I would want to confirm empirically that shuffling the support order barely moves accuracy; I cannot prove it from the construction alone.

For query embeddings, I use an attention LSTM that repeatedly reads the support memory. At processing step `k`, it has previous hidden state `h_{k-1}` and previous readout `r_{k-1}`. It computes `h_hat_k, c_k = LSTM(f'(x_hat), [h_{k-1}, r_{k-1}], c_{k-1})`, sets `h_k = h_hat_k + f'(x_hat)`, and forms the readout `r_{k-1} = sum_i softmax_i(h_{k-1}^T g(x_i)) g(x_i)`. The softmax here is over support index `i`, and it is a different softmax from the final class vote: its scores are dot products used to mix the support memory inside the refinement loop, and its output is a context vector, not a class probability. After `K` reads, `f(x_hat, S) = h_K`. I take `K` to be a fixed number of processing steps. The local port uses the number of support examples as the loop count, while the Facebook reference accepts `K` as a constructor argument.

Putting the tensor path together: contextualize the support features with the bidirectional LSTM, one-hot the support labels, contextualize each query against that support memory with the attention LSTM, normalize the support embeddings only, compute the scaled-cosine score matrix, softmax over support examples, multiply by the one-hot support-label matrix, and take a log before NLL. From the small trace I already know each link is sound: the softmax row sums to one, the matmul-by-labels turns it into a per-class distribution, and NLL on the log of that distribution is `-log P(true)`. One last numerical point: the local implementation adds `1e-6` before the log as a floor against `log(0)` when a class receives no mass. That breaks exact normalization — my `[0.555, 0.445]` becomes `1.0000020` after the floor — but the perturbation is negligible and it removes a real `-inf` failure mode.

What I end up with is a model whose learned parameters define the comparison machinery, while the episode's support set supplies the actual class-specific memory. The non-parametric ability to install a new classifier by changing the support set is kept; the soft attention vote makes the metric trainable; the episodic objective gives training the same shape as evaluation; and conditioning both support and query embeddings on the support set keeps the comparison from being blind to the particular candidates in the episode.
