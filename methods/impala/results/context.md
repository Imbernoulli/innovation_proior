# Context

## Research question

How can deep reinforcement learning be scaled to very high data throughput — many
machines feeding GPU learners — so that a single agent can be trained efficiently,
including on many tasks at once, while preserving data efficiency and training
stability?

State-of-the-art actor-critic agents of the time (A3C and its descendants) need on
the order of a billion environment frames and several days of wall-clock to master
a *single* domain; training one agent on tens of domains simultaneously multiplies
that cost. Once networks get deep, a single GPU's speed becomes the limiting factor.

A natural route to throughput is to let many *actors* generate experience in
parallel and feed a small number of *learners* that compute gradients. When acting
is decoupled from learning, the data a learner trains on is **off-policy**: by the
time a learner computes a gradient from a trajectory, the policy that generated
that trajectory is several updates behind the learner's current policy. The setting
therefore couples a distributed systems question (how to organise actors and
learners for throughput and GPU utilisation) with a learning question (how an
actor-critic that learns a state-value function should learn from data produced by
a stale behaviour policy).

## Background

**The policy gradient / actor-critic objective.** For a parameterized stochastic
policy π in a discounted MDP, the goal is to maximise
`V^π(x) = E_π[ Σ_{t≥0} γ^t r_t ]`. The on-policy policy gradient is
`∇V^μ(x_0) = E_μ[ Σ_s γ^s ∇log μ(a_s|x_s) Q^μ(x_s,a_s) ]`, usually implemented by
ascending `∇log μ(a_s|x_s) · q_s` with `q_s` an estimate of `Q^μ` (e.g. an n-step
return), states sampled under the behaviour policy, and a state-value baseline
`V(x_s)` subtracted from `q_s` to reduce variance. An "actor-critic" jointly learns
the policy (actor) and the value function `V` (critic), the critic trained by
regression toward a bootstrapped return target.

**On-policy vs off-policy.** On-policy methods use data from (approximately) the
policy being updated. Off-policy methods use data generated by a *behaviour* policy
μ to learn about a *target* policy π ≠ μ. The canonical off-policy device is
**importance sampling**: an expectation under π is estimated from μ-samples by
reweighting with the ratio π/μ. Over a multi-step return this becomes a *product*
of per-step ratios `Π_t π(a_t|x_t)/μ(a_t|x_t)`, which is unbiased; its variance
grows with the horizon and with how dissimilar π and μ are.

**n-step returns and temporal differences.** The one-step TD residual is
`δ_t = r_t + γ V(x_{t+1}) − V(x_t)`; summing discounted residuals telescopes the
intermediate value terms, recovering n-step returns
`Σ_{t} γ^{t} r_t + γ^n V(x_{s+n})` and interpolating between low-variance
high-bias bootstrapping (small n) and high-variance low-bias Monte-Carlo (large
n). These on-policy targets are exact when the data is generated by the policy
being evaluated.

**Policy-lag.** When acting is decoupled from learning and run asynchronously
across many actors, the behaviour policy that produced a trajectory is *stale* —
several gradient updates behind the learner's current policy at the moment of the
update. The larger the throughput and the number of machines, the larger this lag
becomes. The action distribution under which the data was generated thus differs
from the learner's current policy distribution.

## Baselines

**A3C — asynchronous advantage actor-critic (Mnih et al., 2016).** Many worker
threads each hold a copy of the parameters, step their own environment with their
local policy, compute an on-policy n-step advantage actor-critic gradient against
that local copy, and asynchronously push the *gradients* to a central parameter
server (Hogwild-style asynchronous SGD, after Dean et al., 2012). It is simple,
distributed, and fault tolerant; communication carries gradients, whose size
scales with the model. UNREAL and Gorila (a distributed DQN) share the same
asynchronous-SGD-of-gradients template.

**Batched A2C (Clemente et al., 2017).** A synchronous design: one learner steps a
whole *batch* of environments together, producing a batch of actions per step.
This batches the expensive tensor ops, with each batch step advancing once all
environments in the batch have produced their step. Variants synchronise every
step or every n steps.

**GA3C (Babaeizadeh et al., 2016).** Decouples the acting/forward pass from the
gradient/backward pass using dynamic batching to use the GPU. The actor/learner
asynchrony makes the collected data off-policy while the learning rule is
on-policy; it adds a small constant to action probabilities when estimating the
policy gradient.

**Importance-sampled / multi-step off-policy corrections.** A line of work
(Precup et al., 2000/2001 eligibility traces and off-policy TD; Wawrzyński, 2009;
Geist & Scherrer, 2014; O'Donoghue et al. PGQ, 2016; Harutyunyan et al. Q(λ),
2016) corrects off-policy multi-step returns with importance weights or trace
coefficients. **Retrace(λ) (Munos et al., 2016)** is an off-policy multi-step
correction whose per-step trace coefficients `c_i = λ min(1, π/μ)` are truncated at
1, giving convergence for any μ, π and low variance. Retrace corrects a
**state-action value Q** and learns Q.

## Evaluation settings

The natural yardsticks are large-scale deep-RL benchmarks. **Atari-57 / the
Arcade Learning Environment** (Bellemare et al., 2013): 57 Atari 2600 games from
pixels, the standard single-task and (across all games) multi-task deep-RL suite;
feed-forward conv nets on 84×84 grayscale frames, frame stacking, action repeat,
reward clipping to [−1,1]. **DeepMind Lab** (Beattie et al., 2016): a 3D
first-person environment that is comparatively expensive and high-variance to
simulate; from it, a 30-task cognitive challenge set covering navigation, memory,
language and laser-tag, used to probe multi-task learning and transfer. Agents use
a conv stack followed by an LSTM and policy/value heads. Standard reporting uses
human-normalised scores `(s − r)/(h − r)` against human `h` and random `r` baselines,
averaged (capped at 1 per task) across the suite. Throughput is measured in
environment frames per second and in wall-clock to a score. The systems axis —
how throughput, GPU utilisation and stability behave as the number of actors,
learners and the resulting policy-lag scale — is part of the protocol.

## Code framework

agent (conv stack → recurrent core → policy logits and a scalar value head), an
optimiser (RMSProp), and standard losses. What does *not* yet exist is the rule
that turns trajectories collected under a stale behaviour policy into a value
target and policy-gradient signal for the current learner policy. The scaffold
below lays out the decoupled actor/learner harness with that one slot left empty.

# --- Actor-critic losses (generic) ---
def baseline_loss(value_targets, values):
    return 0.5 * tf.reduce_sum(tf.square(value_targets - values))

def policy_gradient_loss(target_policy_logits, actions, pg_advantages):
    neg_log_pi = tf.nn.sparse_softmax_cross_entropy_with_logits(
        labels=actions, logits=target_policy_logits)
    return tf.reduce_sum(neg_log_pi * tf.stop_gradient(pg_advantages))

def entropy_loss(target_policy_logits):
    policy = tf.nn.softmax(target_policy_logits)
    log_policy = tf.nn.log_softmax(target_policy_logits)
    return -tf.reduce_sum(tf.reduce_sum(-policy * log_policy, axis=-1))


# --- Learner: batched SGD over trajectories from the queue ---
