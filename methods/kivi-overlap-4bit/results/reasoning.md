Let me start from what actually hurts when I serve one of these models. To make a single request cheap I batch a lot of requests together, but the moment I do that the key-value cache eats me alive. Every past token leaves behind a key vector and a value vector in every layer so I never recompute attention over the history, and the cache is `b × (l_prompt + l_gen) × d` per layer — it grows linearly with the batch and linearly with the sequence. For a 540B model at batch 512 and context 2048 that is a few terabytes, several times the size of the weights themselves. And it is not just memory: decoding is memory-bound. To produce one token I have to stream the whole cache out of HBM into SRAM, and while that load is happening the compute cores are idle. So the cache is the bottleneck twice over, and the obvious lever is to spend fewer bytes per cached element — quantize the keys and values from 16-bit float down to a few-bit integer. The catch is that I can't retrain or even calibrate; the model is already trained and I want plug-and-play. And the cache is a *streaming* object: during decode the keys and values arrive one token at a time and get appended, so whatever I do has to work incrementally on a thing whose length keeps growing. That immediately kills the accurate optimization-based post-training quantizers like GPTQ — they solve a per-layer reconstruction problem, and I cannot afford to re-solve anything every single decode step. What survives that constraint is the cheap, local thing: group-wise round-to-nearest. Pick a group of contiguous elements, take its min and max, set a zero-point `z = min` and a scale `s = (max − min)/(2^B − 1)`, round `(x − z)/s` into `[0, 2^B − 1]`, and dequantize as `q·s + z`. No calibration, runs on a single token, dequantizes back to FP16 before the matmul. That is what FlexGen does — 4-bit, group size 64, applied to both weights and the cache.

So let me just take the standard recipe and push the bits down, because the bytes scale with the bit width and 2-bit is one eighth of FP16 before metadata and residuals. FlexGen quantizes the cache the streaming-natural way: group along the hidden dimension *within each token*, i.e. per-token, because a freshly generated token is a complete group that I can append along the token axis without touching anything that came before. Let me sanity-check what per-token group RTN actually buys me as I drop the bits. At 4-bit, per-token for both key and value, on a 13B model the numbers barely move — call it 66.5 versus a 16-bit 66.4 on CoQA, and the same story on TruthfulQA. 4-bit is essentially free. Now drop to 2-bit, still per-token for both, and it falls off a cliff: 53 from 66, a TruthfulQA collapse too. So the naive "same recipe, fewer bits" plan dies exactly where I need it to live. There is no in-depth study of *why* the cache resists low-bit quantization — everyone just reports that 4-bit works and 2-bit doesn't. I have to actually look at what the elements look like before I can fix this.

The magnitude maps give me the next clue. The key cache and the value cache are not the same picture. In the **key** cache, a few *fixed* channels light up with magnitudes far above everything else, and the same channels are large for essentially every token — vertical stripes. In the **value** cache, nothing comparable appears: no channel stands out, no obvious outlier structure, just diffuse magnitudes. That asymmetry lines up with something already well known: transformer activations carry outliers in a small number of *fixed* channels, on the order of a hundred times the rest, and an outlier channel is an outlier for all tokens. That is the exact thing that makes activation quantization hard. The key cache is `X·W_K`, an activation, so it can inherit the structure; the value cache apparently does not, at least not in a channel-locked way.

Now think about what an outlier channel does to per-token grouping. A per-token group is a slice across channels within one token — so it *contains* the outlier channel. The group's max is set by that one huge element, the scale `(max − min)/(2^B − 1)` blows up, and every normal element sharing the group gets quantized with that gigantic step. At 4-bit, the largest integer code is 15 and there are 16 representable levels, so the step is coarse but the normal elements survive; at 2-bit, the largest code is 3 and there are only 4 levels, so one outlier in the group and the normal elements are basically thrown away. That intuition should survive an actual computation. Take three tokens of a four-channel key where channel 0 is the outlier — values around 100 — and the other three channels are O(1): `[100, 1, 2, 0.5]`, `[98, 1.5, 1, 1.2]`, `[102, 0.8, 2.5, 0.9]`. At 2-bit, per-token (each row its own group) the min of the first row is 0.5 and the max is 100, so the scale is `(100−0.5)/3 ≈ 33`; the three normal channels 1.0, 2.0, 0.5 all round to the same code and come back as 0.5, 0.5, 0.5 — utterly flattened. Per-channel (each column its own group), the normal channels never share with the outlier: channel 1's group is `[1, 1.5, 0.8]`, scale `(1.5−0.8)/3 ≈ 0.23`, and it reconstructs as 1.03, 1.5, 0.8 — essentially exact. Measuring the mean absolute error on the three normal channels only, per-token gives 0.500 and per-channel gives 0.011, a 45× gap. So the outlier really does destroy its groupmates under per-token grouping at 2-bit, and grouping along the **channel** dimension instead — each channel's elements (across tokens) in their own group, with their own min and scale — confines the outlier's huge range to its own group and leaves the normal channels a tight scale. Per-channel quantization confines the error to the channel that causes it.

The diagnostic comparison backs up that reasoning. For the key cache, the relative reconstruction error is ~13.7 with per-token grouping and ~4.6 with per-channel grouping. More importantly, the key only matters through the attention scores, `Softmax(t_Q X_K^T)`, and the attention-score error is ~47 with per-token grouping versus ~9.6 with per-channel grouping, about a 5× improvement. So the key wants per-channel, and that follows from first principles: outliers live in fixed channels, so isolate them by channel.

The value cache is where I almost trip. It has no channel outliers, so my outlier-confinement argument says nothing — per-token and per-channel should be roughly interchangeable for the value. And indeed the raw reconstruction errors are comparable, ~4.6 per-token versus ~3.7 per-channel, the per-channel one even slightly smaller. So by reconstruction error alone I would happily quantize the value per-channel too, or either way. But the diagnostics give a brutal surprise: at 2-bit, quantizing the value per-channel collapses the whole model — single-digit accuracy — *regardless* of how the key is quantized. Per-channel key plus per-channel value: 2.9. Per-token key plus per-channel value: 2.8. The value being per-channel poisons everything. Yet reconstruction error said per-channel value was fine. So reconstruction error of the value cache is the *wrong* thing to look at, and I need to figure out why before I can trust any choice here.

Where does the value cache actually enter the computation? Not as a thing I reconstruct — as a thing I *mix*. The attention output row is `[A X_V]_{i*} = Σ_j A_{ij} [X_V]_{j*}`: a weighted sum over tokens `j` of the value *rows* `[X_V]_{j*}`, with the attention scores as weights. The value cache is a mixer, and the object I must protect is the mixed output `A X_V`, not the value entries in isolation. The right diagnostic is therefore the relative error of `A X_V` with quantized values. There the numbers separate: per-token is ~3.6, per-channel is ~49.9, about 15× worse for per-channel even though the raw value error was comparable.

A tiny case I can compute by hand should reproduce the same reversal, since it runs opposite to what reconstruction error predicts. Four value tokens, four channels, no channel outliers but with two tokens carrying larger magnitudes: row 0 `[1, −2, 0.5, 3]`, row 1 `[9, 8, −7, 6]`, row 2 `[−8, 7, 9, −6]`, row 3 `[0.5, 1.5, −1, 2]`. Make attention sparse — almost all the mass on tokens 0 and 3, `A = [0.55, 0.01, 0.01, 0.43]` — so tokens 1 and 2 are present in the cache but the output barely uses them. At 2-bit, the raw mean-absolute reconstruction error is 0.594 per-token and 0.635 per-channel, basically the same, exactly the trap. But propagate through the mixer: the true output `A·X_V` is `[0.775, −0.305, −0.135, 2.510]`; with per-token values it comes back as `[1.173, −0.490, 0.323, 2.467]`, a relative error of 0.24; with per-channel values it comes back as `[3.277, −0.367, −1.613, 1.960]`, a relative error of 1.12 — about 4.6× worse, and the first component is off by a factor of four. So even on four tokens the per-channel scheme corrupts the output far more than its reconstruction error suggests, matching the direction of the full diagnostic. That is the explanation for the collapse, and now I want to see *why* the mixer amplifies per-channel error so much.

Bring in the other known fact: attention is sparse — a query attends strongly to only a few tokens, the rest of the weights are near zero, around 84% sparsity on this model. So the output row is effectively a combination of a *few* important tokens' value rows; the unimportant tokens' rows are multiplied by ~0 and don't matter. Now look at what per-token versus per-channel quantization does to that picture. **Per-token** quant gives each token its own group — its own min and scale — so the quantization error of token `j` lives entirely inside row `j`. The unimportant tokens carry whatever error they carry, but they are weighted by ~0 in the mix and contribute nothing; the few important tokens are each cleanly quantized within their own row. Error stays confined to rows, and the sparse mixer discards the rows it doesn't use. **Per-channel** quant groups across tokens within a channel: a single channel-group spans the important tokens *and* the unimportant ones together, so the group's min/max — hence its scale — is set partly by tokens the output never uses, and the error from those tokens leaks, through the shared channel scale, into the channel values of the *important* tokens. There is no sparsity to suppress it, because the corruption is riding on the channel, which runs through the rows that do get used. So per-token quantization aligns the error-confinement with the sparsity structure of the mixer, and per-channel quantization breaks it. That is why the value must be per-token even though it has no outliers — the reason is the mixing geometry, not the element distribution. The key wants per-channel for outlier confinement; the value wants per-token for mixer-error confinement. Asymmetric, for two completely different reasons.

So the two caches pull in opposite directions, each for its own concrete reason: the key wants per-channel because that confines its fixed-channel outliers, the value wants per-token because that aligns its error with the sparse mixer. The axis choice is not a cosmetic variant; it follows from two different failure modes, one in the key distribution and one in the value mixer. That settles which axis goes with which cache. What is left is whether per-channel key quantization can even be *implemented* on a streaming cache, and that is where it gets hard.

Per-token value quantization is trivial in streaming: a new token's value vector is a complete group along the channel axis the instant it arrives; quantize it and append along the token dimension; the existing quantized rows are untouched. But per-channel key quantization groups *along the token axis* — I want, say, `G = 32` tokens' worth of a channel together in one group. During decode tokens arrive one at a time, and the number of tokens is arbitrary, so I cannot form a channel-group of 32 tokens until 32 tokens have shown up, and even then the running count is almost never a clean multiple of 32. I can't just quantize the key cache wholesale per channel at every step; the group boundaries move as the cache grows.

Let me think about what part of the cache I *can* group cleanly and what part I can't. At any moment the key cache has `l` tokens. The minimal divisibility problem says I only want to quantize a prefix whose length is a multiple of `G`, because after transposing I need whole groups of `G` tokens for each channel. But I also want a recent full-precision window, so I should flush in blocks rather than one group at a time. Let `R` be that block length, with `R` divisible by `G`. Then the actual split is governed by `r = l mod R`: the grouped prefix `X_{K_g} = X_K[:l−r]` is a whole number of `R`-blocks, hence also a whole number of `G`-token channel groups, and the residual `X_{K_r} = X_K[l−r:]` stays in FP16. As decoding proceeds I append each new key to the residual; once the residual reaches `R` tokens, I quantize that whole block per-channel, concatenate it onto the grouped part, and reset the residual to empty. So the key residual is `l mod R` tokens — it sawtooths from 0 up to `R−1` and resets — and the rest is quantized.

Now I need attention to still work across this split. The query has to dot with *all* keys, but the keys live in two formats — quantized grouped and full-precision residual. That's fine, the dot product is linear and separable: compute the logits against the quantized grouped keys in one (mixed-precision, dequant-fused) matmul, `A_g = t_Q Q(X_{K_g})^T`; compute the logits against the FP residual keys in an ordinary matmul, `A_r = t_Q X_{K_r}^T`; and concatenate, `A = Concat([A_g, A_r])`, before the softmax. Tiled matmul handles the two pieces. So the streaming-incompatibility of per-channel quant is solved by carving off the not-yet-groupable tail and keeping it exact.

I'll do the analogous split for the value, even though per-token value quant is already streaming-friendly, because there is a second reason to keep a full-precision tail that has nothing to do with grouping. Keep the most recent `R` value rows in FP16 as a sliding window: maintain a queue of recent rows, and once it exceeds `R`, pop the oldest row, quantize it per-token, and append it to the quantized grouped part. The value residual is therefore the tail of length `min(R, l)`, exactly `R` once the cache is at least `R` tokens long, and the output splits the same separable way: `t_O = A_g Q(X_{V_g}) + A_r X_{V_r}`, where `A_g`/`A_r` are the softmaxed weights over the quantized-old and FP-recent tokens. Note the asymmetry in *how* the two residuals behave even though both use the same `R`: the **key** residual is the leftover `l mod R` that periodically empties to zero when `l` hits a multiple of `R` — its full-precision window averages about `R/2` over time — while the **value** residual is a steady sliding tail of `R` rows after warmup. That falls straight out of the two split rules: the key residual exists because of group divisibility (it must be a partial block, so it resets), the value residual exists because I deliberately hold a window (so it stays full).

Why hold that full-precision window at all, beyond the key's divisibility excuse? Because the most recent tokens are the locally relevant ones, and hard generation tasks are exactly where a small amount of noise on the immediately preceding keys and values can derail the next step. Picture a multi-step arithmetic chain: each generated digit depends sharply on the last few tokens, so the cache positions nearest the query deserve special protection. The residual is doing double duty: it makes per-channel key quant *possible* by isolating the partial group, and it makes low-bit quant safer by keeping the local tail exact. And it costs essentially nothing — there are at most `R` residual tokens against a sequence that is far longer than `R`, so the FP tail is a vanishing fraction of the cache.

Let me pin down the constants, because they should follow from the structure rather than be pulled from the air. The **group size** `G`: smaller groups mean tighter per-group ranges and lower quantization error, but every group costs me a stored `(min, scale)` pair, so too-small groups drown the savings in metadata, and on a long sequence the metadata-per-element is what determines the real compression. I need a group fine enough to keep outlier damage local and coarse enough to amortize metadata, and the inherited fine-grained KV quantization recipes already live at small powers of two, so `G = 32` is the natural default here. The **residual length** `R`: it must be a multiple of `G` so the flushed key block is a whole number of groups, and it should be large enough to protect a meaningful recent-token window without letting FP memory dominate. I'll take `R = 128` as the safe default, four groups at a time; `R = 32` would be the aggressive setting that compresses more. And **asymmetric** quantization, zero-point at the min rather than a symmetric range around zero: KV distributions are not centered at zero — the keys especially are skewed by their outlier channels — so an asymmetric `[min, max]` range uses the whole integer codebook, where a symmetric range would waste half of it.

Now the bit width. Everything above holds for any `B`, and the general method spans the 2-bit and 4-bit regimes. The extreme case is `B = 2`, where the axis choice is do-or-die. But there is a safer operating point at `B = 4` — the K4/V4 setting — where the same asymmetric scheme (key per-channel, value per-token, group 32, the residual windows) uses fifteen as the largest integer code instead of three. Why would I ever want 4-bit when 2-bit is possible? Because some models leave me less slack: a multi-query-attention model like Falcon already has a single KV head, so its cache is already heavily compressed and there is far less redundancy to spend. Same for grouped-query-attention models with few KV heads. So the 4-bit configuration is the conservative member of the family: identical machinery, `B = 4`, used where the cache should absorb less quantization noise. That's the version I'll write down here.

One more thing before code: during the **prefill** phase I pass the *exact* full-precision keys and values forward to the next layer's attention, and only the *quantized* cache is what I retain in memory. The prefill output should not be degraded by quantization noise — I have the real tensors in hand at that moment — so I compute attention with them and only quantize for storage. That keeps the prompt encoding lossless and confines all quantization effects to the decode path, which is the only place the memory pressure actually bites.

Let me write the algorithm as the prefill/decode procedures so I can see all the moving parts, then collapse it to the fake-quant tensor form the harness wants. At prefill, with `l_prompt` tokens: split the value into grouped `X_V[:max(l_prompt − R, 0)]` (quantize per-token, groups along channel) and residual `X_V[-min(R, l_prompt):]` (FP); for the key, set `r = l mod R`, grouped `X_K[:l−r]` quantized per-channel (groups of `G` along the token axis), residual `X_K[l−r:]` FP; cache the four pieces and return the exact `X_K, X_V`. At each decode step: append the new key to the key residual and the new value to the value residual; if the key residual has reached `R` tokens, quantize it per-channel and concat onto the quantized grouped key, then empty it; if the value residual exceeds `R`, pop the oldest row, quantize it per-token, concat onto the quantized grouped value, leaving the last `R` rows FP; compute logits as `Concat([t_Q Q(X_{K_g})^T, t_Q X_{K_r}^T])`, softmax, and output `A_g Q(X_{V_g}) + A_r X_{V_r}`.

For the harness I don't need the packed integer storage or the fused kernels — the tensor-replay harness scores quality, so I implement the **fake-quant** form: at each step take the real key/value tensor of shape `[batch, heads, seq_len, head_dim]`, quantize-then-dequantize the grouped part with the right axis, leave the residual untouched, and hand back a tensor of the same shape that equals what the model would see after dequantization. The grouped/residual split is computed from the current `seq_len`. The one primitive underneath is the group-wise asymmetric RTN over the last dimension: split the last axis into groups of `G`, per group `z = min`, `s = (max − min)/(2^B − 1)` (floored away from zero so a constant group doesn't divide by zero), `q = clamp(round((x − z)/s), 0, 2^B − 1)`, return `q·s + z`. For the value path in this fake-quant harness, I transpose to `[batch, seq_len, heads * head_dim]` so the last axis contains a token's features and then group that axis. For the key path, I transpose the token and channel axes so the last axis becomes the token axis, and a group of `G` is `G` tokens of one channel — then transpose back.

The key path's split applies that same block-modulo rule to a fixed-shape tensor: residual `r = seq_len mod R` stays FP, and the quantized prefix `seq_len − r` is a multiple of `R` hence of `G`, so after transposing, the token axis is cleanly groupable. As a defensive fallback, if some shape's groupable token count isn't a clean multiple of `G` regardless, I trim to the largest multiple and let the trimmed tokens fall into the FP set, so a partial channel-group is never formed. The value path's residual is simply the tail of length `R` once the sequence reaches `R`, and its quantized prefix groups along the flattened per-token feature axis without needing any such edge case.

Finally the harness-side effective-bits accounting, so I can report the data-bit footprint the fake-quant policy is using; the scale/min metadata is handled separately in the packed implementation and is not part of this hook. A quantized token costs `B` bits per element; a residual (FP) token costs 16. With `quant_tokens = seq_len − residual` and `residual` the FP count for that cache, the average is `(quant_tokens·B + residual·16)/seq_len`. For the key, `residual = seq_len mod R`; for the value, `residual = min(R, seq_len)`, which is exactly `R` once the sequence is long enough. Let me actually walk a few sequence lengths with `R = 128`, `B = 4`, both to check the formula and to see the sawtooth I claimed for the key. At `l = 127` nothing is groupable yet — the key residual is `127 mod 128 = 127` and the value residual is `min(128, 127) = 127`, so everything is FP and both report 16 bits, as they should. At `l = 128` the key residual drops to `128 mod 128 = 0` and the key snaps to exactly 4 bits, while the value holds all 128 rows FP and is still 16; that is the sawtooth resetting precisely when `l` crosses a multiple of `R`. At `l = 129` the key residual is back up to 1 (`4.09` bits) and the value residual is 128 (`15.91` bits). At `l = 256` the key is again 4.0 bits and the value is `(128·4 + 128·16)/256 = 10` bits. And at the reference span of 4096: the key residual is `4096 mod 128 = 0` so the key averages exactly 4 bits, and the value keeps 128 FP rows out of 4096, `(3968·4 + 128·16)/4096 = 4.375` bits. So the key sawtooths between 4 and a hair under 16 and is 4 exactly at every multiple of `R`, while the value decays toward `B` as the FP tail becomes a vanishing fraction — both consistent with the formula. Across same-sized key and value elements at 4096, that is `(4 + 4.375)/2 = 4.1875` bits per cached KV element on average.

That's the whole design, and filling the harness's `AdaptiveKVQuantizer` slot with it is a direct translation, not new reasoning: pick the axis per cache — channel for key, token for value — compute the residual under the right policy (block-modulo for the key, a sliding tail for the value), run the group-wise asymmetric RTN primitive on the grouped part along the appropriate last dimension after transposing, leave the residual tokens untouched, and return the average-bits figure from the formula above.
