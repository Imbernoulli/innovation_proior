Let me start from the thing that actually breaks, stated as sharply as I can. I am minimizing the average device loss `ell(theta) = (1/m) sum_k L_k(theta)`, and a global solution `theta^*` is characterized by `grad ell(theta^*) = (1/m) sum_k grad L_k(theta^*) = 0`. Now look at what an individual device wants. A device that minimizes its own `L_k` lands at `theta_k^*` with `grad L_k(theta_k^*) = 0`. Under heterogeneous data these two conditions are incompatible: at the *global* optimum the per-device gradients `grad L_k(theta^*)` are individually non-zero — they only sum to zero — so no device is at rest at `theta^*`, and conversely if every device comes to rest at its own `theta_k^*`, the consensus of those rest points is `(1/m) sum_k theta_k^*`, which is not `theta^*`. The two fixed points disagree. Every local-averaging method I know of is fighting this disagreement rather than removing it: I broadcast a model, let devices descend their own losses, and average — and the average is dragged toward the wrong fixed point, the more so the more local work I allow. So the real question is not "how do I damp the drift" but "can I change what each device is solving so that its rest point coincides with the global one." If I can make the local stationary point equal the global stationary point, then converging the local models *to consensus* automatically converges them to `theta^*`, and the whole inconsistency evaporates.

Let me first see precisely why the standard repairs do not do this, because the failure tells me what shape the fix must have. The proximal repair has each device minimize `L_k(theta) + (mu/2)||theta - theta^{t-1}||^2`, a spring anchored at the broadcast model. Compute its stationary point: `grad L_k(theta) + mu(theta - theta^{t-1}) = 0`. At a true global optimum I would need this to be solved by `theta = theta^*`, i.e. `grad L_k(theta^*) + mu(theta^* - theta^{t-1}) = 0`. But `grad L_k(theta^*)` is some fixed non-zero vector (the device's residual gradient at the consensus), so the equation forces `theta^* - theta^{t-1} = -(1/mu) grad L_k(theta^*)`, which is a different displacement for every device — they cannot all be satisfied by a common `theta^{t-1} = theta^*`. The penalty is the *same quadratic every round*; its minimizer is biased by exactly the term `-(1/mu) grad L_k`, and no amount of iterating fixes a bias that the objective itself bakes in. That is why, for the proximal method, inexact minimization is not a compromise but a *necessity* — solving the subproblem exactly would still leave the device at the wrong point. So a static regularizer is structurally incapable of aligning the fixed points. The regularizer has to *move*.

What would I need to add so that `theta = theta^{t-1}` (consensus) is itself the stationary point of the local subproblem? Suppose the local objective is `L_k(theta) + <a_k, theta> + (mu/2)||theta - theta^{t-1}||^2` for some per-device linear term `a_k` I get to choose. Its gradient is `grad L_k(theta) + a_k + mu(theta - theta^{t-1})`. Evaluate at the consensus point `theta = theta^{t-1}`: the quadratic contributes nothing there, and I am left with `grad L_k(theta^{t-1}) + a_k`. For consensus to be a stationary point of *every* device's subproblem I need `a_k = -grad L_k(theta^{t-1})`. That is the whole idea in one line: subtract off, with a linear term, the device's own gradient at the consensus point, so that *at* the consensus the device feels no pull away from it. The regularizer is "dynamic" precisely because `a_k` tracks the device's latest gradient and changes round to round. Static penalties damp the move; this linear term cancels the residual gradient that *causes* the move.

Let me write the local objective the device actually minimizes in round `t`. The device holds its model from last time it participated, `theta_k^{t-1}`, and its gradient there. The subproblem is

```
theta_k^t = argmin_theta [ L_k(theta) - <grad L_k(theta_k^{t-1}), theta> + (alpha/2) ||theta - theta^{t-1}||^2 ].
```

The linear term uses the gradient at the device's *own previous iterate* `theta_k^{t-1}` (which the device knows), not at the current broadcast `theta^{t-1}` (which would require an extra gradient pass). The first-order condition for the local optimum is

```
grad L_k(theta_k^t) - grad L_k(theta_k^{t-1}) + alpha(theta_k^t - theta^{t-1}) = 0.
```

This is the recursion I need, and it is self-referential in a useful way: it expresses `grad L_k(theta_k^t)`, the new gradient, in terms of the old gradient and the displacement `theta_k^t - theta^{t-1}`:

```
grad L_k(theta_k^t) = grad L_k(theta_k^{t-1}) - alpha(theta_k^t - theta^{t-1}).
```

So I never need to recompute the gradient explicitly to maintain the linear term — solving the subproblem *gives* me the updated gradient for free, as a function of where the device ended up relative to the broadcast model. That is the dynamic part of the regularizer realized at zero extra cost: each round the device's stored gradient state is refreshed by the displacement it just took, and that refreshed gradient becomes next round's linear term. Let me name the stored state. Define `h_k` so that the linear term is `-<grad L_k(theta_k^{t-1}), theta> = +alpha <h_k, theta>` with `h_k = -grad L_k(theta_k^{t-1}) / alpha`. Then the recursion above says `h_k` accumulates the displacements: `h_k^{t} = h_k^{t-1} + (theta_k^t - theta^{t-1})`, starting from `h_k = 0` (the input condition `grad L_k(theta_k^0) = 0`). So `h_k` is just the running sum of the device's net moves away from the broadcast models — a memory of how far this device has consistently pulled.

Now I have to handle the inexactness honestly. No device solves the argmin exactly; it runs a few local SGD steps. But notice the linear and quadratic terms are *closed-form* in `theta` — their gradient is `-grad L_k(theta_k^{t-1}) + alpha(theta - theta^{t-1})`, which I can add to the data gradient at every local step. Concretely, the regularizer gradient is `alpha(theta - theta^{t-1}) + alpha h_k`: an `alpha`-strength weight-decay-like pull back toward the broadcast model plus a fixed `alpha h_k` offset that is the accumulated correction. So the device runs plain SGD on `L_k`, and before each step adds `alpha(theta - theta^{t-1} + h_k)` to the gradient. The quadratic `(alpha/2)||theta - theta^{t-1}||^2` contributes the `alpha(theta - theta^{t-1})` part — exactly an L2 pull to the anchor — and the dynamic linear term contributes the `alpha h_k` part. This is the minimal change to the local loop, and crucially it requires the device to know only `theta^{t-1}` (broadcast) and its own `h_k` (stored locally). Nothing global beyond the model crosses the wire — unlike a control-variate method, I am not shipping a separate correction vector each round; the correction lives in the *device's* persistent state and in the *server's* aggregation, both reconstructed from the models alone.

The server side is where the alignment is sealed, and I have to get it exactly right or the fixed-point argument collapses. The naive thing is to average the active devices' models: `(1/|P_t|) sum_{k in P_t} theta_k^t`. But that average is the consensus of points whose subproblems were corrected by the *per-device* linear terms, and the correction has to be re-injected at the server so the global model sits at the aligned point rather than the biased average. Let me derive what to add. Maintain a server state `h` that mirrors the device states: I want `h` to track `(1/m) sum_k grad L_k(theta_k^t)` so that as the device gradients drive toward summing to zero, `h` drives to zero too. Using the per-device recursion `grad L_k(theta_k^t) = grad L_k(theta_k^{t-1}) - alpha(theta_k^t - theta^{t-1})` and updating only the active devices, the server state evolves as

```
h^t = h^{t-1} - alpha * (1/m) * sum_{k in P_t} (theta_k^t - theta^{t-1}).
```

The divisor is `m`, the *total* number of devices, not `|P_t|`, the number active — because `h` is meant to be the average over *all* devices' gradient states, and only the `|P_t|` active ones changed this round; dividing by `m` keeps `h` an honest all-device average despite partial participation, exactly as a control-variate server update would. Then the global model is the active-device average *corrected* by the server state:

```
theta^t = (1/|P_t|) sum_{k in P_t} theta_k^t - (1/alpha) h^t.
```

Let me check this lands at the right fixed point, because that is the only thing that justifies the whole construction. At convergence, suppose `theta_k^t -> theta^infty` for all `k`. Then the device recursion `grad L_k(theta_k^t) -> grad L_k(theta_k^{t-1})` forces the displacements `theta_k^t - theta^{t-1} -> 0`, so `h^t` stops changing and `theta^t -> theta^infty` as well, meaning `theta^infty` is independent of `k` and equals the consensus. With the displacements going to zero, the stored gradients `grad L_k(theta_k^t)` converge to `grad L_k(theta^infty)`. And the server-state recursion, unrolled, makes `h^t` equal to `(1/m) sum_k grad L_k(theta_k^t)` (it is built to accumulate exactly those), which at the fixed point is `(1/m) sum_k grad L_k(theta^infty) = grad ell(theta^infty)`. For `theta^t` to be a fixed point of the server update I need the correction `-(1/alpha) h^t` to vanish into the average consistently, which holds only when `h^infty = 0`, i.e. `grad ell(theta^infty) = 0`. So *the only consensus the method can rest at is a stationary point of the global loss.* That is the alignment I set out to get: the dynamic regularizer makes consensus a local stationary point, and the server correction makes consensus a global stationary point, so the two fixed points are forced to coincide. The static-penalty method could never close this loop because its `h_k` was frozen at zero.

Now let me make sure the server reconstruction is the same object as the device states, because that consistency is what keeps the bookkeeping exact under sampling. Unroll `-(1/alpha) h^t`: from `h^t = h^{t-1} - alpha(1/m) sum_{k in P_t}(theta_k^t - theta^{t-1})`, dividing by `-alpha` and unrolling from `h^0 = 0` gives `-(1/alpha) h^t = (1/m) sum_over_rounds sum_{active}(theta_k - theta^{t-1}) = (1/m) sum_{all k} H_k`, where `H_k` is device `k`'s accumulated displacement — exactly the `h_k` I defined on the device (`H_k = h_k`, the running sum of net moves). So the server correction is just the mean over all `m` devices of their accumulated displacement states, and the global update is `theta^t = active-average + (1/m) sum_k H_k`. The server can maintain this incrementally: each round add `(1/m) sum_{k in P_t} (theta_k^t - theta^{t-1})` to its running `h_sum` and set the model to the active average plus `h_sum`. This is the canonical form, and it is reassuring that the server state and the device states are literally the same accumulated-displacement quantity viewed at two levels — there is one bookkeeping object, not two.

A few choices I want to pin down before writing it. First, why does the linear term use `grad L_k(theta_k^{t-1})` rather than `grad L_k(theta^{t-1})`? Because the device already has `theta_k^{t-1}` and the recursion refreshes the gradient there for free; reading the gradient at the broadcast model would cost an extra full pass each round and buy nothing the recursion does not already give. Second, `alpha` is the single knob: it is the strength of the quadratic anchor *and* the scale relating the stored gradient to the displacement. Larger `alpha` makes the local subproblem more strongly convex (the Hessian is `grad^2 L_k + alpha I`, so `alpha` past the negative-curvature floor convexifies it, giving a unique local solution and a clean displacement bound) and pulls the device harder toward the broadcast model; smaller `alpha` lets devices move more but weakens the correction. The analysis goes through for a suitably chosen `alpha > 0`, and the convergence rates I can prove from it — linear for strongly convex device losses, `O(1/T)` for non-convex, with an `m/P` partial-participation factor — hold without any bounded-dissimilarity assumption, which is the formal statement that heterogeneity has been removed from the rate: it is the alignment of the fixed points, not a bound on how dissimilar the devices are, that drives the rate. For deep nets on a Dirichlet-split CIFAR-style benchmark I will take `alpha = 0.1` as the default (with a sweep `{1e-3, 1e-2, 1e-1}` and a per-device scaling by data fraction), since the harness here is exactly a Dirichlet non-IID image/character setting. Third, plain SGD locally: like the control-variate methods, the clean recursion that gives the gradient state for free relies on the local step being an ordinary gradient step plus the fixed regularizer offset; a momentum buffer would tangle the relationship between net displacement and accumulated gradient, so I keep the local optimizer plain and put all the structure in the added gradient term and the server correction.

There is one more comparison I want to make explicit, against the closest gradient-augmented method, because it sharpens why this is worth doing. A control-variate method also removes heterogeneity from the rate, but it transmits a control variate *in addition to* the model every round — twice the bits — because the server needs each device's latest gradient estimate, which is not recoverable from the model alone. Here, the device's correction is its accumulated displacement `H_k`, which the *server* can reconstruct from the model deltas it already receives; nothing extra goes over the wire. So I get the same heterogeneity-free alignment at half the communication, and because the method is built on (inexact) minimization of a well-posed local objective rather than a tuned SGD schedule, it is far less sensitive to hyperparameter tuning. That is the trade I want: a single `alpha`, one model per round each way, and a fixed-point argument that says the only place the federation can come to rest is a stationary point of the true global loss.

So the construction assembles itself, and it drops into the harness as a fill of the strategy: keep a per-device accumulated-displacement state `H_k` and a server running sum `h_sum = (1/m) sum_k H_k`; in the local loop, add `alpha(theta - theta^{t-1} + H_k)` to the gradient at every plain-SGD step; after training, accumulate `H_k += (theta_k^t - theta^{t-1})`; on the server, average the active models, add the contributions to `h_sum`, and set the new global model to `active-average + h_sum`. The full module is in the answer.
