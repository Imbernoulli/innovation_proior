Let me start from what actually breaks when I point a frozen LLM at a real multi-tool task. The task is a conversation: I read the instruction, I think, I pick one API out of a pile, I fill in its parameters, the environment actually runs the call and hands me back a real response, and I condition my next move on everything so far. There are two finishing moves always on the table — "Finish with Final Answer", which ends successfully, and "Finish by Giving Up", which I use when the tools just can't do it. The thing I keep slamming into is that this is not a tidy little search. The move at each round is a free-form thought, *plus* which API to call, *plus* what arguments to pass it, and the product of those three is unbounded — there's no finite menu of next states I could enumerate and score. And the world bites back: a call can 404, time out, return garbage, or return something almost-but-not-quite useful, and the instant I commit to a bad call, that bad call is now sitting in my conversation history, and I'm forced to keep conditioning on it.

So what do I have to build on? The grounded-acting template is clear and I want to keep it. The clean way to think about an agent is: at step t I see an observation o_t, I take an action a_t from some action space A under a policy π(a_t | c_t), where the context c_t = (o_1, a_1, ..., o_t) is the whole interaction. The trick that makes language work here is to *augment* the action space, Â = A ∪ L, where L is language: a "thought" is an action in L that doesn't touch the environment and returns no observation — it just rewrites the context so my next real action is better reasoned — while a real action in A executes a tool and returns a real observation. Interleaving thought and action this way is exactly the right per-round shape for tool use: think, call an API, read the real response, think again. I'm keeping that. Pure reasoning without the acting — just a chain of thoughts z_1, ..., z_n and then an answer — is a non-starter the moment tools are involved, because it reasons over an internal black box with no way to check what an API actually returns, so it cheerfully hallucinates facts and even hallucinates APIs that don't exist. The grounding is the whole point.

But here's the thing the grounded-acting template gives me and the thing it doesn't. It gives me a single linear trajectory: think, act, observe, think, act, observe, forward and forward until I hit a finish or run out of steps. What it does *not* give me is any way to take back a decision. And when I run this on the hard instructions, two failures show up over and over, and they're really the same disease seen from two angles. First, error propagation: I make one wrong call — wrong parameters, or an API I imagined into existence — and now that mistake is in my history, so I condition on it and make a *related* mistake, and I get stuck in a faulty loop, hammering the same API the same wrong way or chasing the same nonexistent function. Second, limited exploration: a single linear path commits to exactly one route through that infinite action space, and if the route is unlucky the attempt just dies. And — this is the part that stings — rerunning the same flat procedure from scratch doesn't reliably rescue the hard ones; even a very strong model often can't find a single valid path on a complex multi-tool instruction with one linear pass. So the flat trajectory both *propagates* an early error and *under-explores*. A forward-only chain has no answer to either.

The obvious instinct is: if a forward-only chain can't take anything back, give it a way to back up. So what does backing up even mean here? The cleanest existing idea for using a failure is the verbal-reinforcement loop — when an attempt fails, reflect in language on what went wrong, stash that reflection in memory, and retry the *whole task from the beginning* with the reflection prepended. That's genuinely better than nothing; it converts a binary fail into a textual hint for the next try. But stare at the unit of backup: it's the entire episode. Every retry restarts at the root. If my trajectory was perfectly fine for eight steps and then I fumbled step nine, this scheme throws away all eight good steps and redoes them, just to get another shot at step nine. That's enormously wasteful in a setting where every step is a real, paid LLM call. What I actually want is to keep the good prefix and only revise the bad suffix — to back up *a little*, to the place where things went wrong, not all the way to the start. The backup wants to be local, not global.

The instant I say "keep the prefix, revise the suffix, back up to where it went wrong," I'm not describing a chain anymore. I'm describing a *tree*. A node is a partial action sequence — a state I've reached after some thoughts and calls. The root is the bare task. A leaf is a finish: either a final answer (a good terminal) or a give-up / API-error dead end (a pruned node). Backing up = walking up from a dead leaf to an ancestor and trying a *different* child from there. That's exactly "keep the prefix up to the ancestor, revise the suffix below it." So the structure is forced: search over a tree of partial trajectories, where I can descend a promising branch and, on hitting a dead end, retract to a nearby node and branch off in a new direction.

Now, how do I actually search this tree, and this is where I have to be careful, because the standard recipes will quietly bankrupt me. The textbook deliberate-search approach over a tree of thoughts is: at each node, generate k candidate children; *score* them with a value heuristic — ask the model to rate each state, or to vote on which is best — and use that score both to pick which child to expand first and to prune branches whose value falls below a threshold; then backtrack when a branch looks hopeless. On a small, brute-forceable problem — an arithmetic puzzle, a crossword — that's lovely, because there a value heuristic is cheap and meaningful and the whole space is small. But look at my cost structure. I have no value oracle. The *only* way to judge a partial trajectory, or to compare two candidate actions, is to spend more LLM calls. If I generate n children at a node and rank them pairwise with the LLM to pick the best, that sorting costs on the order of O(n log n) LLM calls — and that comparison is the single most expensive thing in the entire procedure. So if I adopt the value-guided recipe, I'd be pouring the bulk of my budget into *scoring nodes*, in a space that's infinite, when all I actually need is *one* valid path. That's the wrong place to spend money. Wall.

Let me make the constraint explicit because it changes the whole shape of the answer: I do not need the *best* path, and I do not need to *rank* anything. I need exactly one path that reaches a final answer, found within a tight query budget. Every LLM call I spend on comparing children instead of advancing toward a terminal is, for my goal, wasted. So the design pressure is the opposite of the value-guided recipe: spend calls on going forward and on diversifying when I'm stuck, and spend as close to zero as possible on evaluation.

Take the easy decision first. Breadth-first or depth-first? Breadth-first expands every node at the current frontier before going deeper — it would generate and partially run a whole layer of alternative trajectories before reaching *any* terminal. But I only need one path to terminate. Depth-first drives straight down one branch to a terminal as fast as possible, and only branches sideways when that branch fails. Since "found one valid path" is my entire stopping condition, depth-first reaches that condition with far fewer calls; breadth-first would burn excessive API calls expanding siblings I don't need. Depth-first it is.

Now the expensive decision: the ranking. The value-guided recipe ranks the children at each node and expands the best first. I can't afford that. But do I even need it? Let me check what the ranking actually buys me. Its job is to make me expand the *most promising* child first so I descend toward a terminal faster. Suppose I just... skip it, and expand children in the order I generate them — generate one child, immediately recurse into it, and only generate a second child at this node if the first child's whole subtree dead-ends. That's a pre-order traversal of the tree, no sorting anywhere. Is it much worse? Empirically, when I look at what the ranking does on these tasks, the child that ends up ranked highest is, most of the time, the one the model generated *first* — the model's own first instinct at a node is usually its best instinct. So the ranking is, in the common case, paying O(n log n) LLM calls to re-derive an ordering that "generation order" already approximates for free. Drop it. Generate one child, recurse; if that fails, come back and generate the next.

And this skip-the-ranking choice has two properties that tell me it's not a hack, it's the right generalization. First, graceful degradation downward: on a simple instruction where the model never has to retract — it just thinks, calls the right API, finishes — my tree is a single chain with one child per node, and the procedure is *identical* to the plain forward-only grounded-acting agent, at the same cost. I haven't made the easy case more expensive; I've left it exactly where it was. Second, graceful degradation upward: after the whole search finishes, the set of nodes I actually explored is almost the same set a full ranked depth-first search would have explored — because depth-first visits the same nodes regardless of the *order* it visits siblings; ranking only changes the order, not the eventual set, and I terminate as soon as I find a valid path either way. So I keep the hard-instruction-solving power of real DFS while shedding its dominant cost. That's the whole trade: pre-order traversal sits exactly between "no exploration at all" (the flat agent it degrades to) and "full ranked search" (the node set it reproduces), and it pays for neither's worst case.

But now there's a subtlety I have to handle or the backtracking is useless. When a branch dead-ends and I retract to an ancestor to try a different child, what stops the model from just... generating the same action again? It's the same node, the same context, the same most-likely next token — left alone, it'll regenerate the very action that just failed, and I'll loop forever expanding identical children. So when I come back to a node that already has children, I have to actively *push* the model off its previous choice. Before it generates the next child, I prepend a message that shows it the actions it already tried at this state and explicitly tells it: this isn't your first attempt here, all the previous tries failed, generate something *different* from all of these. That diversity nudge is what turns "retract and re-expand" into genuine exploration instead of repetition. One bookkeeping detail: that diversity message is scaffolding for *generating* the new child — it shouldn't live on in the child's own context or contaminate downstream reasoning (or, if these trajectories get reused as training data, pollute the recorded messages). So once I've used it to produce the child, I mark it invalid so it's filtered out of the persistent message history. The nudge does its job at generation time and then disappears.

Now the part that actually controls the search dynamics: when a branch ends, how *far* do I back up? This is the one real knob, and I want to reason it out rather than pick a number. Picture the recursion: I descend into a child, and that descent returns some value telling me "back up this many levels." The simplest behavior — back up exactly one level — means: this branch failed, just try the next sibling right here at the immediate parent. That's the tightest possible retraction, a pure local retry. The opposite extreme is to back up an unbounded number of levels: any failure unwinds me all the way to the root and I start over — which is precisely the whole-episode restart I rejected, the reflect-and-retry loop; pushed to the limit it's a flat chain that resets on every failure, which is no better than the forward-only agent I'm trying to beat. So the back-up distance is a dial between "local retry" at one end and "global restart" at the other, and the more I back up, the more diverse my exploration gets — at the extreme of infinity it becomes "regenerate the whole trajectory from scratch n times," a greedy resample. Neither extreme is what I want. I want something in the middle: when a branch genuinely dead-ends, escape the *local subtree* that produced the dead end — get out of the small neighborhood of bad decisions — but don't throw away the entire prefix.

So let me set the distance deliberately. When does a "real" dead end happen? Two cases. One: the model explicitly calls "Finish by Giving Up" — it has decided, from this state, the tools can't get there. Two: I've actually reached a final-answer terminal and want to keep looking for *another* valid path (because I might want more than one). In both of those cases, backing up only one level — just trying a sibling of the give-up node, one step above where I gave up — leaves me right next to the decisions that led into the dead subtree; I'd likely walk straight back into it. Backing up *two* levels lifts me out of that immediate bad neighborhood, to a genuinely earlier branch point, while still preserving everything above it. So I set the back-up distance to 2 for a give-up and 2 for a found-answer. For an *ordinary* prune — a single failed API call that didn't escalate to a full give-up — I only back up one level: try the next sibling here, because a lone failed call is local, not a sign the whole subtree is poisoned. So the back-up length encodes *how bad* the dead end was: 1 for a routine failed call (local retry), 2 for a give-up or a completed answer (escape the local subtree), and the same machinery would give a full restart if I ever cranked it toward infinity.

Let me make the recursion concrete, because the "back up N levels" has to propagate correctly up the call stack. The search is recursive over nodes; each call returns an integer = how many more levels the caller should keep backing up. Base cases first: if I'm at a node that's at the depth limit, or already pruned, or terminal, I stop descending. If it's terminal (a real final answer), I record it as a solution and return the answer-back-length, 2. If it's pruned, I mark it pruned and check *why*: if its observation code is 4, the node is the give-up case, so I record it as a give-up node and return the give-up-back-length, 2; otherwise it's an ordinary failed call and I return 1. If it's not a base case, I loop up to the branching factor (`tree_beam_size`) times. Each iteration starts again from the same ancestor node, generates exactly one new child chain from it, and recurses into the last node of that chain immediately. If this ancestor already has children, I temporarily append the diversity nudge before generation and mark that message invalid afterward. The step itself is the same one-step parser as the tool-use harness: an assistant content field becomes a Thought node; a function call becomes an Action node and then an Action Input node; status 4 marks the new leaf pruned as a give-up, status 1 rewrites a hallucinated function name to a sentinel, and status 3 marks the leaf terminal and marks the finishing action-input ancestors with the answer-back length. The implementation increments the query counter right after the LLM parse; if that just-spent call hits the hard budget, I return a giant value before descending further. Otherwise I recurse into the new leaf and read its returned back-up value. Three things can come back. If I now have enough recorded answers (≥ `answer_count`), I'm done with the whole search and return a huge number so every frame above unwinds and exits. If the returned value is greater than 1, a descendant wants me to keep backing up past this level, so I decrement it and return `result - 1`. If the returned value is exactly 1, the request has been fully consumed at this frame, so I stay in this loop and generate the next sibling here. If the loop exhausts all branching attempts without a propagating return, I return 1 and let my own parent try its next sibling.

Let me trace the propagation once to be sure it's right. Say deep down a node gives up; it returns 2. Its parent's loop sees result = 2 > 1, so the parent returns 2 − 1 = 1. The grandparent's loop sees result = 1, so it does *not* return — it stays in its loop and generates its next child, i.e. a sibling two levels above the give-up. That's exactly "back up two levels and branch": the give-up frame and its parent frame both unwound, and the grandparent is where the new exploration happens. The decrement-and-propagate is what converts a single integer into "unwind exactly N frames then branch." If instead the give-up had returned 1 (an ordinary failed call), the parent's loop would see 1, not return, and immediately try the give-up node's *sibling* — a one-level local retry. The integer is doing all the control flow, and the values 1 and 2 are precisely the local-retry / escape-the-subtree distinction I reasoned out.

A couple of bookkeeping pieces fall out naturally. Depth: each logical tool-use round is really a small group of nodes — a Thought, then an Action (the API name), then an Action Input (the parameters and the resulting observation) — so the raw tree depth advances by three per logical round; the implementation compares `get_depth()` to the configured depth limit in those raw node units, and callback displays divide by three to recover logical step count. The status codes from executing a call carry the meaning the recursion keys on: a normal success, a "hallucinated API name" code (which I rewrite to a sentinel so the model's own message records that it invented a nonexistent function), a "final answer" code that flips the node to terminal and marks the finishing nodes done, and a "give up" code that triggers the deeper backtrack. I stamp each node with a monotone visit counter as I expand it, and I keep two running lists — the terminal answer nodes and the give-up nodes — because at the end, if no branch produced a real final answer, a recorded give-up is still a legitimate, recordable outcome (it's the honest "the tools can't do this" path).

Let me step back and make sure I haven't fooled myself on cost, because cheapness was the whole motivation. Per node I spend exactly one LLM call to generate its (one) child via the step primitive — no ranking, no value scoring, ever. The diversity nudge doesn't add a call; it's a prefix on the same generation call. So the total LLM calls is essentially the number of nodes I expand, capped hard by max_query_count. On a simple instruction that's a single chain — same cost as the flat agent. On a hard one it's a chain plus however many sibling branches the dead ends forced me to open, each one node-for-node, and I stop the instant I either find enough answers or hit the budget. That's the trade I wanted: I bought decision-retraction and real exploration of an infinite space, and I paid for it only in forward-generation calls, not in the O(n log n) evaluation calls that the ranked search would have charged me.

So let me write it as the actual search policy filling the empty slot in the harness — the recursive depth-first walk over the tree, pre-order (no ranking), with the diversity nudge on re-expansion and the back-up-length-encoded-as-return-value backtracking:

```python
def search(self, root_node):
    """Depth-first, pre-order traversal over the tree of partial action sequences.
    One LLM call per node, no child ranking. Diversity nudge on re-expansion;
    backtrack distance encoded as the recursion's return value."""
    self._dfs(root_node)

def _dfs(self, now_node):
    """Recursive DFS. Returns how many more levels the caller should keep backing up."""
    # How far a dead end unwinds: 1 = local sibling retry (ordinary failed call),
    # 2 = escape the local subtree (give-up or a completed answer). Pushed toward
    # infinity this would become a full restart (flat resample); 2 is the middle dial.
    final_answer_back_length = 2
    prune_back_length = 2

    now_node.expand_num = self.now_expand_num          # monotone visit counter
    self.now_expand_num += 1

    # get_depth() is raw tree-node depth: Thought, Action, Action Input consume
    # three levels for one logical tool-use round.
    if (now_node.get_depth() >= self.single_chain_max_step
            or now_node.pruned or now_node.is_terminal):
        if now_node.is_terminal:                       # a real final answer
            self.status = 1
            self.terminal_node.append(now_node)        # record this solution path
            return final_answer_back_length            # keep looking elsewhere: back up 2
        else:
            now_node.pruned = True
            if now_node.observation_code == 4:         # status 4 == "Finish by Giving Up"
                self.give_up_node.append(now_node)
                return prune_back_length                # escape this dead subtree: back up 2
            return 1                                    # ordinary failed call: local retry

    for _ in range(self.tree_beam_size):
        added_diversity = False
        if len(now_node.children) > 0:
            # We have backed up to this node; append a temporary prompt built from
            # previous children so the next generated child differs from them.
            now_node.messages.append(self._diversity_message(now_node))
            added_diversity = True

        # One canonical step: parse assistant content into a Thought, parse a
        # function_call into Action and Action Input, execute the tool, then handle
        # status 4 (give-up/pruned), status 1 (hallucinated API sentinel), and
        # status 3 (terminal final answer + make_finish(final_answer_back_length)).
        leaf = self._step(now_node)

        if self.query_count >= self.max_query_count:   # budget is checked after the LLM call
            return 100000                              # abort: unwind every frame and exit

        if added_diversity:
            now_node.messages[-1]["valid"] = False     # drop the temporary prompt from history

        result = self._dfs(leaf)                       # descend immediately (pre-order, no ranking)

        if len(self.terminal_node) >= self.answer_count:
            return 10000                               # enough valid paths: unwind and stop
        elif result > 1:                               # a descendant wants to keep backing up
            return result - 1                          # propagate the retraction up one level
        # result == 1 -> a local sibling retry == this loop's next iteration; fall through

    return 1                                            # no more children here: let parent retry
```

Let me close the loop on the causal chain. I started from a grounded acting agent — interleave thoughts and real API calls, condition on real observations — because pure reasoning over an internal black box hallucinates tools and facts. But a single forward-only trajectory has no way to retract a decision, so an early wrong call propagates into a faulty loop and one linear path under-explores an infinite action space; even strong models then fail the hard instructions. Wanting to keep a good prefix and revise only a bad suffix forced the structure from a chain to a tree of partial trajectories, where backing up means retracting to an ancestor and branching anew. The reflect-and-retry idea backs up the right way but at the wrong granularity — it restarts the whole episode — so I needed local backup, not global. Searching the tree with the standard value-guided recipe was unaffordable, because ranking children costs O(n log n) LLM calls and I have no value oracle and only need *one* valid path, not the best; so I dropped ranking entirely and chose depth-first (one path suffices, breadth-first wastes calls) with pre-order traversal (generate one child, recurse immediately), which costs one call per node, degrades exactly to the flat agent on easy instructions, and reproduces full-DFS's explored node set on hard ones. To make re-expansion productive rather than repetitive I push the model off its previous actions with a diversity nudge that's discarded after it's used. And the backtrack distance — encoded as the recursion's return value, decremented and propagated up — became the single dial of the method: 1 for a local sibling retry after an ordinary failed call, 2 to escape the local subtree after a give-up or a completed answer, with the same machinery yielding a full restart in the limit. The whole thing buys decision-retraction and genuine exploration of an infinite tool-use space, paid for only in forward-generation calls under a hard budget, never in evaluation calls.
