I start by trying to simulate the nondeterministic machine directly, and every direct version spends the wrong resource. If I keep the set of all possible current configurations, I am storing an exponentially large frontier. If I search one branch at a time, I still have paths whose natural length bound is the number of configurations, and I need memory to control revisits. If I write a whole accepting history, I have made the certificate exactly as long as the thing I cannot afford to store. The machine itself is not the hard part; the hard part is that deterministic certainty seems to require remembering a global search object.

So I change the object I am thinking about. A computation branch is a path in a configuration graph. A configuration is small, only `O(S(n))` bits, while the graph has up to `2^{O(S(n))}` vertices. That contrast is the opening. I should not build the graph and I should not store paths through it. I should ask whether a reachability fact can be verified while holding only a few vertex names.

The tempting reachability routine is depth-first search, but that is still thinking path-first. It carries a path prefix and needs a visited discipline. I need a way to talk about a long path through a short name. A path from `u` to `v` of bounded length has such a name: somewhere around the middle it passes through a vertex `z`. Naming `z` does not describe the path, but it splits the claim into two smaller claims. That is the useful proof unit. I stop asking for the sequence of moves and instead ask for bounded reachability between two endpoints.

Let `Reach(u, v, i)` mean that `v` can be reached from `u` in at most `2^i` steps. At level zero I only need to check equality or one legal edge. At a larger level, the statement is equivalent to the existence of a middle configuration `z` such that `Reach(u, z, i - 1)` and `Reach(z, v, i - 1)` both hold. If the path is shorter than half the budget, I can choose `z = v`; otherwise I choose the configuration reached halfway along the path. Conversely, two such shorter paths concatenate into a path within the original budget.

Now the deterministic simulator can enumerate possible `z` values one at a time. It stores the current endpoints, the candidate middle configuration, and the level. It checks the left half recursively; if that succeeds, it checks the right half using the same recursive workspace. The two subcomputations do not have to coexist in memory. This is the point where the proof is really about space rather than time: time branches and accumulates, but workspace from a failed or completed recursive check can be overwritten.

For an `N`-vertex graph, each vertex name costs `O(log N)` bits and the useful path length is at most `N - 1`, because cycles can be deleted. So I start with about `log N` levels. Each recursion frame stores only a constant number of vertex names and counters, so the stack costs `O(log N)` frames times `O(log N)` bits per frame, which is `O(log^2 N)`. The algorithm is slow because every level scans all middle vertices, but the theorem is not trying to preserve time.

Putting the Turing machine back into the picture, `N` is at most `2^{O(S(n))}` because a configuration has `O(S(n))` bits. Thus `log N = O(S(n))`. Enumerating candidate configurations and checking whether one follows from another by a legal move can both be done in `O(S(n))` space. The reachability recursion therefore uses `O(S(n))` levels and `O(S(n))` space per level, giving `O(S(n)^2)` deterministic space.

This is the distinctive reframing. Nondeterminism first looks like a branching tree whose successful branch must somehow be found. I instead treat it as an implicit finite graph and replace a long branch by recursively named middle configurations. The simulator never stores the branch, never stores a frontier, and never stores the graph. It buys certainty with enormous time, but it spends only a square in the original space bound.
