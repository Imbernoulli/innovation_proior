**Problem Analysis:**

We need to find a **palindrome move sequence** that:
1. Starts at `(sr, sc)`
2. Visits all blank cells
3. Ends at `(er, ec)`

A palindrome means `M[i] = M[len-1-i]` for all `i`.

### Key Insight:
Let the path be `v_0 -> v_1 -> ... -> v_k`, where each transition is one of L,R,U,D. 
For the sequence of moves to be a palindrome, we also need: `pos(v_0) == pos(v_k)`, meaning the **start and end positions must be equal** after following the same path forward and backward (considering boundary/wall checks).

However, there's a catch: a cell visited "en route" might have different effective positions depending on whether we approach from direction A or direction B! 

Example:
```
Start:   Goal:
  A       X    <- blocked
B -------C
```
- Path forward via C: Start → C → D(visit) → E → ... → F → Goal  
- Path back through D: ... → F → D → C → Start

If cell D has **one exit** but was reached via two directions (say C and E), then coming back from E toward D doesn't necessarily lead us back toward C — it could bounce off D's only exit! The palindrome property breaks because `v_i` ≠ `v_{len-1-i}` in terms of actual position.

So we can't use standard bidirectional search over states like `(mask, current_pos)`, where `current_pos` tracks how far the partial palindrome extends inward. Why? Because matching symmetric parts requires knowing **both**:
- How the first half moved outward
- What those moves imply about how the mirrored part should come back

But here’s the crucial observation: since George bounces against walls/boundaries, any symmetric pair of moves (`U` followed by its mirror `D`) naturally cancels out geometrically — unless the second move would otherwise hit an obstacle. In practice, most cells will just "fold back" correctly as long as the path exists between them.

Therefore, instead of trying to match every single intermediate position exactly, we can focus on building a solution by:
- Running BFS **outward** from start while tracking which unvisited blank cells remain
- For each step, pick an unvisited reachable cell that’s closest to the goal
- Once the whole grid is covered, append the remaining steps needed to reach the goal from the last covering cell
- **Mirror everything** around the center and fill in necessary padding with redundant cycles like `"ULUR"` or `"RDLU"` that traverse and return within the already-covered area — these preserve validity without changing the set of visited cells
