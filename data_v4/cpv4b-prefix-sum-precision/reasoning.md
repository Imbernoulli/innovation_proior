Two numbers settle the shape of this before I pick an algorithm. I have `n` daily revenues `a[0..n-1]`, each in `[-10^9, 10^9]`, a minimum length `L`, and `1 <= L <= n <= 2*10^5`; over every contiguous window `[i, j)` with `j - i >= L` I maximize the average `(a[i] + ... + a[j-1]) / (j - i)` and print it as a *reduced* fraction `p/q` with `q > 0`. First, a window sum reaches `2*10^5 * 10^9 = 2*10^14`, well past 32-bit, so prefix sums and window sums are 64-bit `long long` from the start. Second, the output is a fraction and rounding is forbidden, so the operation I actually have to get right is *comparing two averages* — the instant I compute one as a decimal I have lost. This is a comparison problem, not a search problem, and that framing dominates everything below.

With `S[0] = 0` and `S[k] = a[0] + ... + a[k-1]`, window `[i, j)` has sum `S[j] - S[i]` and average `(S[j] - S[i]) / (j - i)` — exactly the *slope* of the segment connecting prefix points `P_i = (i, S[i])` and `P_j = (j, S[j])`. So I am maximizing, over `0 <= i < j <= n` with `j - i >= L`, the slope from a left prefix point to a right one.

The textbook route bisects on the answer `x`: subtract `x` from every element and ask whether some window of length `>= L` has nonnegative sum — a prefix-min scan, `O(n)` per guess, `O(n log(range/eps))` overall. Clean and short, but `x` is a `double`, and that is exactly the trap. With denominators up to `2*10^5`, two *distinct* achievable averages can differ by as little as `1/(d1 d2) ≈ 1/(2*10^5)^2 = 2.5*10^-11`. The values live near `10^9`, where a `double`'s resolution is about `10^9 * 2^-52 ≈ 2*10^-7` — four orders of magnitude too coarse to separate two such averages. Nor is this just an artifact of being near `10^9`: adding a constant to every element shifts every average equally, so two averages `10^-11` apart near zero stay `10^-11` apart once pushed up to `10^9`, where the `double` can no longer resolve them. Float bisection can return a value that rounds to the wrong fraction, and then I cannot even recover `p/q`. So I never divide: to compare `s1/d1` against `s2/d2` with positive denominators, compare `s1 * d2` against `s2 * d1`.

For each right endpoint `j` I want the left endpoint `i` in `{0, 1, ..., j - L}` giving the steepest segment to `P_j`. As `j` increases by one, the allowed set grows by exactly one new index, `j - L`. The steepest left endpoint to a point that lies up-and-to-the-right always sits on the **lower convex hull** of the candidates — any point above the hull is dominated by the hull vertex beneath it, which gives at least as steep a segment to `P_j`. So I maintain the lower hull of `{P_0, ..., P_{j-L}}` incrementally: push `P_{j-L}`, popping while the last turn is not a proper lower-left turn.

Along the lower hull the edge slopes strictly increase. For a fixed `P_j` above the hull, the slope from vertex `hull[t]` to `P_j` rises while the local edge is shallower than that segment and falls once the edge is steeper — so it is unimodal in `t`, and I binary-search the switch where consecutive slopes stop increasing. A monotone sequence or a one-vertex hull simply lands on an endpoint, which the search handles.

The one thing easy to invert in the pop condition is the sign of the cross product. A positive `cross(A, B, C)` means `C` lies left of ray `A -> B` — a counterclockwise turn, which for the *lower* hull is a vertex I keep; I pop on clockwise-or-collinear turns, `cross(A, B, C) <= 0`. The sample's first three prefix points pin the sign concretely: `P_0 = (0,0), P_1 = (1,1), P_2 = (2,13)` give `cross = 1*13 - 1*2 = 11 > 0`, and `P_1` genuinely lies *below* chord `P_0 P_2` (slope `6.5` predicts `6.5` at `x = 1` versus `P_1.y = 1`), so it belongs on the lower hull and must not be popped. A pop-on-`>= 0` rule would discard it — backwards. Pop on `<= 0`; the `<=` also drops collinear middles, harmless since they give the same slope. Carrying the sample through (`n = 7, L = 3, a = [1,12,-5,6,7,-2,3]`, `S = [0,1,13,8,14,21,19,22]`), at `j = 5` the tangent picks `i = 1`: `(S[5]-S[1])/(5-1) = 20/4 = 5`, reduced to `5/1`, window `[1,5) = 12,-5,6,7` — the stated answer.

Every comparison multiplies a sum-difference by a length: worst case `~2*10^14` times `~2*10^5 = ~4*10^19`, while signed 64-bit tops out at `2^63 - 1 ≈ 9.22*10^18` — so the product overflows by roughly a factor of four and wraps, and a wrapped comparison can flip the winner. A concrete flip: two windows of equal length `d = 200000`, with sums `s_A = 46116860184273` and `s_B = s_A + 1`; both averages are about `2.3*10^8`, comfortably feasible, and B is strictly larger by `1/d`. Exactly, `s_A * d = 9223372036854600000 < 2^63 < 9223372036854800000 = s_B * d`, so B wins. But in signed 64-bit `s_B * d` wraps to `-9223372036854751616`, and the test `s_B*d > s_A*d` becomes `(-9.2*10^18) > (9.2*10^18)` — false — so the code keeps the *worse* window A. Exact arithmetic says B, `long long` arithmetic says A: a real, reproducible flip, precisely the failure these constraints are built to expose.

So every product — the hull `cross`, the tangent comparison `(S[j]-S[A])*(j-B)`, and the best-fraction comparison `num*bestDen` — moves to `__int128`, whose range `~1.7*10^38` swallows `4*10^19` with enormous margin. The *stored* quantities stay 64-bit: `|S| <= 2*10^14`, lengths `<= 2*10^5`, and no `__int128` value is ever narrowed back — each only feeds a `<`/`>` comparison. Only the products need the wider type.

Edge cases. `L = n`: the loop runs once at `j = n` with the sole left index `0`, the whole-array window; `n=3, L=3, a=[-1,-2,-3]` prints `-2/1`. Negative answers are legal because a window has length `>= L >= 1`, so unlike max-subarray there is no empty-window escape; `n=1, L=1, a=[-4]` prints `-4/1`. All-equal `a=[7,7,7,7,7], L=2` prints `7/1`. Reduction keeps the sign in the numerator (`bestDen = j - bi >= L >= 1 > 0`) via `g = gcd(|bestNum|, bestDen)`; `gcd(0, d) = d` reduces `0/d` to `0/1`, and the `g==0` guard is cheap insurance. `scanf` skips whitespace, so the two-line input parses regardless of layout.

Against an exact-`Fraction` brute force over 800 random small cases — uniform values, all-`±10^9`, tiny ranges, every `L` from `1` to `n` — zero mismatches; and against an independent `O(n^2)` reference on 150 cases with `n` in `2000..4000` (products already in the `10^16`-`10^18` range), again zero. The sample returns `5/1`. At `n = 2*10^5` the run is ~20 ms and ~7 MB, inside the 1 s / 256 MB budget. That is `O(n)` time and `O(n)` memory; the full self-contained module is in the answer.
