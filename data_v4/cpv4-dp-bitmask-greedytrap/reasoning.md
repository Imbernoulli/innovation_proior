I have `m` required skills numbered `0..m-1` and `k` contractors, each with a fee `cost[i]` and an `m`-bit skill mask `mask[i]`, and I want the cheapest subset of contractors whose masks OR together to the all-ones set — weighted set cover over a universe of at most 18 elements, printing `-1` when nothing covers everything. The two constraints that decide the whole approach are `m <= 18` and `k <= 100`: `2^18 = 262144` coverage states times 100 contractors is about `2.6*10^7` elementary relaxations, which an exact bitmask DP clears well inside 2 seconds — so I do not have to settle for the approximation that the phrase "set cover" usually forces on me. Fees run to `10^6` and I can hire all `k`, so a total tops out near `10^8`; that fits a 32-bit `int` with room, but I keep the cost accumulators in `long long` anyway and a mask stays an `int` (18 bits).

The tempting shortcut is the classic set-cover greedy: repeatedly hire the contractor with the best fee-per-newly-covered-skill ratio until everything is covered. It is a few lines and fast, but it is a known `ln m` approximation, and this task wants the exact minimum, so the real question is whether it can be wrong on inputs this problem actually allows. It can, and a small instance shows it cleanly. Take `m = 6` with `H1 = (2, {0,1,2})` mask `7`, `H2 = (2, {3,4,5})` mask `56`, and a lure `L = (3, {1,2,3,4,5})` mask `62`. Greedy starts empty; `L` covers 5 new skills for fee 3 (ratio `0.6`), each `H` covers 3 for fee 2 (ratio `0.667`), so greedy grabs `L`. That leaves only skill `0`, forcing `H1` for fee 2, and greedy pays `5`. But `H1 + H2` covers `{0,1,2} ∪ {3,4,5}` for `4`. The lure re-covers four skills (`1,2,3,4`) that the optimal pair already supplies for free; local fee-efficiency is blind to the overlap that the union constraint cares about globally. So greedy is out, and this instance stays as the worked sample.

The exact route treats the only thing about a partial hire that constrains the future — which skills are already covered — as the state. Let `dp[s]`, over coverage sets `s ∈ {0,1}^m`, be the minimum fee to reach a hired set whose pooled skills cover `s`. Base case `dp[0] = 0`: covering the empty set is free. From any reachable `s`, hiring contractor `i` moves to `ns = s | mask[i]` at added cost `cost[i]`, relaxing `dp[ns] = min(dp[ns], dp[s] + cost[i])`. The answer is `dp[FULL]` with `FULL = 2^m - 1`, or `-1` if it is never reached.

Order matters, because this is a relaxation over a graph and each state must be final before I propagate out of it. Every transition goes from `s` to a *superset* `s | mask[i]`: as integers a superset carries all of `s`'s bits plus possibly more, so `ns >= s`, and `ns != s` forces `ns > s`. Iterating `s` from `0` to `FULL` in increasing order therefore applies every incoming relaxation — all coming from strictly smaller states — before the outer loop reaches `ns`. One forward pass suffices, like a DAG shortest path; no Dijkstra, no repeated sweeps. On the trap sample it lands where greedy could not: from `s=0`, `L` sets `dp[62]=3`, `H1` sets `dp[7]=2`, `H2` sets `dp[56]=2`; when the loop reaches `s=7`, `dp[7]` is already final, and hiring `H2` gives `7|56 = 63` at `2+2 = 4`, beating the `0 -> 62 -> 63` path's `5`. So `dp[63]` settles at `4` along `0 -> 7 -> 63`, i.e. `H1 + H2` — the optimum.

One transcription trap is real enough to dodge up front. The relaxation adds `cost[i]` to `dp[s]`; if I initialize the table to `LLONG_MAX` and do not gate on reachability, an unreachable state feeds `LLONG_MAX + cost[i]` — signed overflow, undefined behavior, and the wrapped-negative result can beat a legitimate `dp[ns]` and fabricate a "reachable" state that is not. The smallest witness is `m=1, k=1` with contractor `(5, mask 0)`: skill `0` is required but nobody has it, so the answer is `-1`, yet an ungated loop would touch `dp[1] = LLONG_MAX` and overflow. Two coupled guards kill it: skip any `s` whose `dp[s]` is still the sentinel — an unreachable state has nothing to propagate — and set `INF = LLONG_MAX/4` so even a stray addition keeps slack. With the guard only genuine finite costs ever feed the `+ cost[i]`, and impossibility reports as `dp[FULL] >= INF`; equivalently `== INF`, since relaxations only ever lower a value so an unreached state stays exactly at the sentinel, but the `>=` is robust to the arithmetic.

The remaining corners fall out of the same `min` relaxation with no special cases. Overlap where two cheap pieces beat one bundle — `m=2` with `(2, skill 0)`, `(3, skill 1)`, `(10, both)` — takes `0 -> 1 -> 3` for `2 + 3 = 5` over the single `(10)` bundle that a "grab the one all-covering contractor" heuristic would overpay for; duplicate masks at different fees keep the cheaper automatically. The table for the largest instance `m=18, k=100` is `2^18` longs, about 2 MB, comfortably inside 256 MB.

So I ship the `O(2^m * k)` bitmask DP rather than the greedy I broke: a forward pass over coverage states in increasing order, the sentinel guard, and `dp[FULL] >= INF -> -1`. Run against a brute-force subset enumeration over several hundred random small instances, it agrees everywhere the two overlap, including the greedy-trap family it was built to catch.
