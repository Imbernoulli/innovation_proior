I'm counting, for each query `n a b`, how many box widths `x in [1, n]` leave `floor(n/x)` inside the band `[a, b]`. Two numbers in the contract fix the shape of any solution before I write a line: `q` reaches `2*10^5` and `n` reaches `10^12`. A per-query scan over `x` is `O(n)`, and a single large query is already `~10^12` operations, so scanning survives only as a brute oracle to check against — the submission has to answer each query in `O(1)`. And since `n` and `b` sit at `10^12` and I will form `b+1`, everything is `long long`; an `int` anywhere is a silent wrong answer on the large tests.

The `O(1)` route rests on `floor(n/x)` being non-increasing in `x`: a wider box never raises the full-box count. So each half of the band becomes a one-sided bound on `x`, the valid widths form one contiguous interval, and the answer is its length. The whole difficulty is that the two ends carry *opposite* inclusivity.

Upper end, from `floor(n/x) >= a` with `a >= 1`: `floor(n/x) >= a` iff `n/x >= a` iff `x <= n/a` iff `x <= floor(n/a)`, the last step because `x` is an integer. So this is an inclusive cap `hi = floor(n/a)`. When `a <= 0` the condition is vacuous — every quotient is `>= 0 >= a` — so `hi = n`, and I must not evaluate `n/a` (division by zero at `a = 0`).

Lower end, from `floor(n/x) <= b`: here the strictness is everything. `floor(n/x) <= b` iff `n/x < b+1` iff `x > n/(b+1)` iff `x > floor(n/(b+1))`. So this is an *exclusive* floor `loExcl = floor(n/(b+1))`, valid `x` satisfying `x > loExcl`. The denominator is `b+1`, not `b`, straight from the strict `< b+1`. Writing `floor(n/b)` instead would encode `floor(n/x) <= b-1`, the whole band shifted down by one — the exact slip this problem is built to punish. The single-value band `n=10, a=1, b=1` makes it visible: the answer is `5` (widths `6..10`), and `floor(10/(1+1)) = 5` gives `hi - loExcl = 10 - 5 = 5`, while the tempting `floor(10/1) = 10` gives `0`. So the `b+1` stays.

Valid `x` therefore lie in `(loExcl, hi]` — open left, closed right — and the count of integers there is `hi - loExcl`, no `+1`, since they are exactly `loExcl+1, ..., hi`. Reading `n=10, a=2, b=3` off the formula: `hi = floor(10/2) = 5`, `loExcl = floor(10/4) = 2`, count `3`, the widths `{3,4,5}`. Here `x=3` has quotient exactly `b=3` and is kept while `x=2` has quotient `5` and is dropped, so the open-on-`b` / closed-on-`a` split is doing the work it has to.

Now the corners the band endpoints reach. With `a = 0` I already cap `hi = n`; the awkward member is `[0,0]`, where the `<= 0` side forces `x > floor(n/1) = n` while the `>= 0` side allows all `x <= n`, so `(n, n]` is empty and the count is `0` — e.g. `n=7, [0,0]` gives `hi = 7, loExcl = 7, 0`. The open lower bound is what makes this right; an inclusive lower side would wrongly count the widths with quotient `1`. A band entirely above the maximum quotient collapses through `hi`: `n=6, [7,9]` gives `hi = floor(6/7) = 0`, `loExcl = floor(6/10) = 0`, count `0`, no special case. The full band `[1,n]` gives `hi = floor(n/1) = n`, `loExcl = floor(n/(n+1)) = 0`, count `n` — the `floor(n/(n+1)) = 0` is exactly what lets the interval reach down to `x = 1`. And `n = 1`: band `[1,1]` gives `1`, band `[0,0]` gives `0`.

Two guards fall out of the degenerate orderings the contract allows: `a > b` is an empty band, and `b < 0` means `floor(n/x) >= 0 > b` for every `x` — both return `0`, and the `b < 0` guard also keeps me from forming `n/(b+1)` with a non-positive denominator. A defensive `n <= 0 -> 0`, a cap `hi <= n`, and a floor `loExcl >= 0` never fire under the stated bounds but cost nothing and keep the routine sane on those degenerate orderings; the final `max(0, hi - loExcl)` closes any residual empty interval. Overflow is a non-issue: the largest quantity I form is `b+1 = 10^12 + 1`, far under the `long long` ceiling, and no two large numbers are ever multiplied.

The algebra is delicate enough at the endpoints that I want an independent check, so I match the closed form against an `O(n)` reference scan on random small cases — `n` up to a few hundred, `a, b` both near real quotient values and out past `n`, single-value bands `[k,k]`, `a = 0`, and bands above the max quotient — the exact corners where an off-by-one would hide; they agree throughout. So `solve` reads the two bounds, clamps them with the guards above, and returns `hi - loExcl`.
