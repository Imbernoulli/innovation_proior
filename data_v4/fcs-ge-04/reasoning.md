The numbers in the contract decide the whole shape of this. `n` is only up to `10^5`, but coordinates run to `10^9` in magnitude, and that mismatch is the trap. A polygon at this scale can enclose on the order of `10^18` lattice points, so the answer needs 64 bits just to hold — but worse, any method that touches shoelace cross-products multiplies two `10^9` numbers to get `~10^18` and then sums `10^5` of them, so a running area accumulator climbs toward `10^{23}`. That is four orders of magnitude past what a signed 64-bit integer holds. Whatever algorithm I pick, the area bookkeeping cannot live in a `long long`. I write that down first, because it is the one thing the small examples will never surface.

The brute-force method is obvious and I would trust it completely: enumerate every lattice point of the bounding rectangle and test each for strict containment by ray casting, with an on-edge check to drop boundary points. It is exact and short. But its cost scales with the *enclosed area*, not with `n`: the bounding box spans up to `2*10^9` by `2*10^9`, i.e. `4*10^{18}` cells, which is centuries against a 1-second limit. It only survives on tiny coordinates. That rules it out as the solution but makes it an ideal independent oracle later. I need a cost that depends on `n`, not on the area.

Two integer quantities are computable exactly in `O(n)` straight from the vertices. The doubled area, from the shoelace formula `2A = |sum_i (x_i y_{i+1} - x_{i+1} y_i)|` — every term a product of integers, so `2A` is an exact integer with no floating point. And the number of boundary lattice points: an edge from `(x_i,y_i)` to `(x_{i+1},y_{i+1})` carries exactly `gcd(|dx|,|dy|)` lattice points if I count one endpoint per edge — parametrize the segment and a lattice point falls every `1/gcd` of the way, giving `gcd` points per edge including its start vertex; summed around the closed boundary the start vertices are exactly all the vertices, so `B = sum_i gcd(|dx_i|,|dy_i|)`. `gcd(0,k)=k` handles axis-aligned edges correctly.

The question is whether the interior count `I` is a function of just `2A` and `B`. Pick's theorem says yes: `A = I + B/2 - 1`. Rearranged, `I = A - B/2 + 1`, and scaling by 2 to stay in integers, `I = (2A - B)/2 + 1`. The quantity `2A - B` is provably even for an integer polygon — that is exactly what makes Pick's `I` an integer — so the halving is exact. This collapses the whole problem to two accumulators over a single pass.

The given L-hexagon `(0,0),(4,0),(4,2),(2,2),(2,4),(0,4)` is non-convex, which is exactly where the additive constant and orientation are easiest to get wrong, so it is the case worth running through. Shoelace gives `2A = 24`; the edge deltas `(4,0),(0,2),(-2,0),(0,2),(-2,0),(0,-4)` have gcds `4,2,2,2,2,4` summing to `B = 16`; so `I = (24-16)/2 + 1 = 5`, matching the stated answer. The `-1` holds on a concave shape.

Now the implementation, and this is where the `10^{23}` note from the top bites. The loop, for each `i`, adds `x[i]*y[j] - x[j]*y[i]` to the area accumulator and `gcd(|dx|,|dy|)` to the boundary count. Each individual product `x[i]*y[j]` is at most `10^{18}` and fits in a signed 64-bit integer, so it is tempting to keep the accumulator in `long long`. But the *sum* of up to `10^5` such terms, when they share a sign — a long thin polygon hugging the coordinate extremes does exactly that — reaches `~10^{23}` and silently wraps modulo `2^{64}`. The result is a garbage area and therefore a garbage `I`, and it strikes precisely on the large tests the problem is built around, never on the small samples. So the doubled-area accumulator has to be `__int128` (good to `~1.7*10^{38}`), with each product cast to `__int128` before the multiply so the multiplication itself is 128-bit. The boundary sum stays safe in `long long`: each gcd is at most `~1.4*10^9` across `10^5` edges, so `B <= ~1.4*10^{14}`.

A second overflow hides in the sign handling: to handle clockwise input I take the absolute value of the signed area, but `llabs` is a 64-bit function — applied to the 128-bit accumulator it would truncate and reintroduce the exact overflow I just removed. The negation has to be hand-written, `if (twiceArea < 0) twiceArea = -twiceArea;`. And `cout` cannot print `__int128` at all, so I peel off decimal digits by hand into a string and reverse it.

That leaves the corner cases the evaluation advertises, and they fall out of the two formulas cleanly rather than needing special code. Clockwise input: the signed shoelace comes out negative and the manual negation fixes it — the reversed square `(0,0),(0,4),(4,4),(4,0)` gives `2A = -32 -> 32`, `I = 9`, same as CCW. A triangle whose only lattice points sit on its boundary, `(0,0),(2,0),(0,2)`: `2A = 4`, gcds `2,2,2` so `B = 6`, `I = (4-6)/2 + 1 = 0`. A thin sliver `(0,0),(5,1),(10,0)`: `2A = 10`, gcds `1,1,10` so `B = 12`, `I = 0` — positive area, no interior point. Redundant collinear vertices along an edge contribute zero area and leave `B` unchanged, so `I` is untouched: Pick's theorem does not care how the boundary is subdivided. Negative and large coordinates near `10^9` are absorbed by the `__int128` area. The one input where a naive check would "disagree", three collinear points with zero area, is a degenerate polygon the contract explicitly excludes.

Finally I keep the rejected box scan useful by running it as an oracle. A generator emits random simple polygons — points sorted by polar angle around their centroid, convex hulls, axis-aligned shapes — each filtered through an explicit non-self-intersection test so the ray-cast containment is valid, and on small coordinates the scan and the Pick computation are compared over several hundred cases plus the hand-built corners, with no mismatch. The area-proportional method I threw away for being too slow is exactly the thing that certifies the linear one.
