The scale sets two arithmetic traps before any algorithm is on the table. I have up to `n = 2*10^5` people, each weight up to `2*10^9`, and a cart capacity `L` up to `4*10^9`; a two-seat cart launches iff its two riders' weights sum to at most `L`, each person rides at most once, and I want the maximum number of carts. The *count* of carts is tiny — at most `n/2 = 10^5` — but the arithmetic around it is not. `L = 4*10^9` on its own already exceeds the signed 32-bit ceiling of `~2.147*10^9`, so reading `L` into an `int` corrupts it at parse time. And a pair-sum `w[i] + w[j]` reaches `2*10^9 + 2*10^9 = 4*10^9`, again past 32-bit range. So `L`, the weights, and every sum I compare against `L` must be 64-bit — `long long` throughout. An `int` here is not a crash; it is a silent wrong-answer on exactly the large tests. That is settled first, because it fixes the types the rest of the code is written against.

The task itself is, abstractly, a maximum matching: connect person `i` to person `j` iff `w[i] + w[j] <= L`, and find the largest set of vertex-disjoint edges. General maximum matching is absurd overkill for `n = 2*10^5`, but the graph is not arbitrary — an edge exists purely because of a *threshold on a sum*, and that kind of structure usually collapses to a greedy sweep. Two cheap `O(n log n)` routes tempt me, and I want the one I can prove, not the one that is easiest to type: pair the two lightest remaining people repeatedly, or run two pointers from both ends (pair the lightest free person with the heaviest free one when they fit).

Before typing either, I break the lightness-greedy on a concrete instance, because "seating cheap couples feels efficient" is how wrong solutions get shipped. Take `[1, 1, 2, 3]`, `L = 4`. The greedy sorts and pairs the two lightest: `1 + 1 = 2 <= 4`, one cart, consuming both `1`s; the leftover `2` and `3` give `2 + 3 = 5 > 4`, no cart. Greedy total: **1 cart**. But pair `1 + 3 = 4` and `1 + 2 = 3` and every person rides: **2 carts**. Greedy is wrong, and the reason is structural. By marrying the two `1`s to each other it wasted the light people, whose real value is as partners for the heavy people who cannot otherwise ride at all — the `3` needs a `1` alongside it, and spending both `1`s on a `1+1` cart strands the `3`. So the pairing must run from the *heavy* end, not the light end.

That fixes the rule: sort ascending, put `lo` at the lightest and `hi` at the heaviest. If `w[lo] + w[hi] <= L`, pair them and step both inward; otherwise `hi` is the heaviest remaining and even the lightest cannot lift it, so it can pair with no one — drop it (`hi--`). Two claims need nailing before I trust the count is optimal.

The discard is forced. If `w[lo] + w[hi] > L`, every other free person is `>= w[lo]`, so `hi`'s sum with *anyone* exceeds `L`; `hi` is unpaired in every feasible solution, and removing it loses nothing.

The pairing is safe by an exchange argument. Given `w[lo] + w[hi] <= L`, I claim some optimal solution pairs them together. Take any optimal `S`; if `{lo, hi}` is already in `S`, done. Otherwise look at `hi`. If `hi` is paired with some `p != lo` and `lo` is free, swap `p` out for `lo`: `{lo, hi}` is feasible, count unchanged. If `hi` is paired with `p` and `lo` is paired with `q`, re-pair into `{lo, hi}` and `{p, q}`; the latter is feasible because `w[q] <= w[hi]` (nothing is heavier than `hi`) gives `w[p] + w[q] <= w[p] + w[hi] <= L`, and the count is preserved. If `hi` is free, then `lo` must be paired — were both free we could add the feasible cart `{lo, hi}` and beat `S` — so with `lo` paired to some `q`, form `{lo, hi}` and free `q`, count unchanged. Every case produces an optimal solution pairing `{lo, hi}`; induct on the rest. The crux is `w[q] <= w[hi]`: the heaviest person is the most constrained, so it deserves the lightest free partner, which is exactly why pairing from the heavy end is right.

Clean math transcribes dirty, and my first cut of the sweep carries an off-by-one the constraints invite:

```
sort(w.begin(), w.end());
int lo = 0, hi = n - 1;
long long pairs = 0;
while (lo <= hi) {
    if (w[lo] + w[hi] <= L) { pairs++; lo++; hi--; }
    else { hi--; }
}
```

The guard `lo <= hi` is wrong, and a single person exposes it: `n = 1`, `w = [5]`, `L = 100`, where the answer must be `0`. Here `lo = hi = 0`, `0 <= 0` is true, `w[0] + w[0] = 10 <= 100`, so `pairs` becomes `1` — person 0 gets paired *with itself*, because at `lo == hi` both pointers name the same person and the sum double-counts one weight. Any odd-sized all-fit group would self-pair its middle person the same way. The guard must be `lo < hi` strictly; then `n = 1` gives `0 < 0` false and outputs `0`, and an even pair like `[1, 3]`, `L = 4` still fires once and stops at `1 < 0`.

The comparison `w[lo] + w[hi] <= L` rides entirely on the 64-bit element type. With `vector<long long>` weights and `long long L`, the addition happens in 64-bit and cannot overflow at `4*10^9`. Declared `vector<int>`, the same sum would overflow to a negative value *before* the comparison widened to `long long`, spuriously passing `<= L` — the silent wrong-answer I flagged at the start. The boundary confirms it directly: `[2*10^9, 2*10^9]` returns `1` at `L = 4*10^9` (sum equals `L`) and `0` at `L = 4*10^9 - 1` (sum exceeds `L`), both correct only because the arithmetic stays 64-bit.

The remaining corners fall straight out of the `lo < hi` loop. `n = 0`: nothing is read into the weights, `hi = -1` is used only in the guard (`0 < -1` false), output `0`. `n = 2` fits → `1`; doesn't fit → the `else` discards `hi`, then `0 < 0` false, output `0`. All pairs fit (say all weights `1`, large `L`) → the sweep pairs every inward step for `floor(n/2)`, the absolute maximum since each cart eats two people. No pair fits → the first check fails every step, count stays `0`. Odd `n` leaves exactly one correct leftover. The `pairs` accumulator tops out near `10^5`, so even an `int` would hold it; I keep it `long long` to stop thinking about it.

Against the documented sample — `n = 4`, `L = 4`, `[1, 1, 2, 3]`, answer `2` — the sweep pairs `1 + 3` then `1 + 2` for `2`, matching, and it is the very instance that broke the lightness-greedy, so it doubles as the demonstration that the heavy end matters. Beyond hand traces I ran the sweep against an independent brute force that computes the true maximum matching by exhaustive recursion (take the lowest free person, either leave them out or pair them with each compatible later person, recurse for the max), over tiny tie-heavy cases plus an occasional large-weight regime to hit the 64-bit boundary: zero mismatches across 600+ cases.
