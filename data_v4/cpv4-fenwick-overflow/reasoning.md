I'm summing `a[i]*a[j]` over every out-of-order pair `i<j` with `a[i] > a[j]`. The comparison is strict, so ties never contribute, and a non-decreasing sequence gives `0`. Two constraint numbers settle the shape of the solution before I write anything. With `n` up to `10^5`, the pair count `C(n,2) ~ 5*10^9` makes the naive double loop hopeless under a one-second limit, so I need `O(n log n)`. And with each `a[i]` up to `30000`, the *product weighting* is where this differs from plain inversion counting: one product is up to `30000*29999 ~ 9*10^8`, already close to the signed 32-bit ceiling `~2.1*10^9`, and the whole sum can reach `C(10^5,2)*30000^2 ~ 4.5*10^18`. That fits a signed 64-bit `long long` (ceiling `~9.2*10^18`) but nowhere near `int`, so the accumulator is `long long` — and, the part I'll come back to, so is anything that sums values *inside* the algorithm.

**Sweep vs. merge-sort.** Two routes reach `O(n log n)`. I could adapt merge-sort inversion counting, adding `y*(sum of pending larger left elements)` at each merge step instead of a count; it works but rides on getting the merge boundaries and the "pending left = smaller original index" bookkeeping exactly right. The Fenwick sweep has a cleaner invariant, so I take it: sweep `j` left to right with a tree, indexed by compressed value-rank, holding the sum of *values* of all already-inserted (earlier) elements. "The tree holds exactly the elements left of `j`" is maintained for free by inserting after querying.

**The decomposition.** Every inversion has a unique larger index `j`; the inversions ending at `j` are the earlier positions `i<j` with `a[i] > a[j]`, each contributing `a[i]*a[j] = a[j]*a[i]`. So `contribution(j) = a[j] * (sum of a[i] over i<j with a[i] > a[j])`, and summing over `j` counts every inversion once — no double counting, no omission. Online, I need "sum of already-inserted values whose value is strictly greater than `a[j]`". A Fenwick tree over value-ranks storing value-sums gives `query(r) = sum of inserted values with rank <= r`; the strictly-greater sum is `query(m) - query(r)`, where `r` is `a[j]`'s rank and `m` the number of distinct values. Values top out at `30000` so I could index by raw value, but I coordinate-compress to ranks `1..m` to keep the index space `O(n)` and the code general.

**The overflow trap.** Making `answer` a `long long` feels like it settles the arithmetic, but that is the mislead. The Fenwick bucket for a value accumulates `count * value`; with many copies of `30000` that single bucket alone passes `3*10^9`. If the tree storage and the `query` return are `int`, `query(m) - query(r)` has already wrapped past `2^31` *before* the multiply — and no `(long long)` cast on the product can recover a value corrupted upstream. To see it fail at small scale, an all-`int` version on `[30000,30000,30000,29999,29999,29999]` (nine cross-inversions, true answer `9*899970000 = 8099730000`) prints `-490204592`: the silent-overflow signature, a plausible negative number, never a crash. So every quantity on the value path — the tree storage, the query return, the greater-sum — is `long long`, and I give `v` type `long long` too so the multiply is unambiguously 64-bit rather than relying on a cast landing in the right spot.

**Strict-greater and insert order.** Two loop decisions are load-bearing, and `a = [2,2]` (equal values, answer must be `0`) pins both. Both have rank `1`, `m = 1`. At `j=0`: `greater = query(1) - query(1) = 0`, contribute `0`, then insert `2`. At `j=1`: `greater = query(1) - query(1) = 2 - 2 = 0` — the equal earlier element sits at rank `r`, inside `query(r)`, which I subtract off, so the tie is correctly excluded; total `0`. The wrong variants show what each guard buys: `query(m) - query(r-1)` would give `2` at `j=1` and count the tie as an inversion; and inserting before querying would let an element see itself or a later equal, violating `i<j`. So: query first, then insert; strictly-greater via `query(m) - query(r)`.

**Edges.** `n=0` — `cin >> n` reads `0`, the loops never run, `answer` stays `0` (a truly empty stream fails the read and returns `0` having printed nothing, but the contract always supplies `n`). `n=1`, strictly increasing, and all-equal all yield `0` for the same reason: no earlier element is ever strictly greater. With `m` distinct values the tree has size `m+1` and `query(m)` covers everything, so there is no off-by-one; `m=0` when `n=0` is safe since the loop never queries.

The full program is in the answer: coordinate-compress, one left-to-right Fenwick sweep accumulating `v * (query(bitSum, m) - query(bitSum, r))`, every value-path quantity 64-bit.
