Three machines at most, each firing at every multiple of its period, and I want the K-th *distinct* instant at which any of them fires. The distinctness is the whole problem: when two or three machines fire together it is one moment, and the naive count of firing times double-counts exactly those coincidences. Two numbers set the scale and rule things out before I write anything. The answer is at most `min(p) * K` — the smallest-period machine alone supplies `K` firing times by then — and with both near `10^9` that is `10^18`, inside signed 64-bit (`~9.2 * 10^18`) but nowhere near 32-bit, so every time, bound, and accumulator is `long long`. The other number is the `lcm`: two periods near `10^9` give an `lcm` near `10^18` (fine), but three give near `10^27`, which overflows 64-bit. That is a hazard I have to design around rather than trip over.

`K` reaching `10^9` also kills the direct approach. A k-way heap merge — keep each machine's next firing time, pop the smallest, skip it if it equals the previous pop (that is the de-dup), stop at the K-th distinct value — is obviously correct because it literally builds the sorted distinct sequence, but it is `O(K log n)` and a billion heap operations will not finish in a second. It stays useful only as a reference oracle on small inputs.

So: binary search on the answer. Let `C(x)` be the number of distinct firing times in `[1, x]`. It is non-decreasing, and it increases only *at* firing times (flat between consecutive ones), so the smallest `x` with `C(x) >= K` is itself a firing time and is exactly the K-th. Everything now rests on computing `C(x)`.

The firing times of machine `i` in `[1, x]` are the multiples of `p[i]`, numbering `floor(x / p[i])`. Summing that over the machines counts every pairwise coincidence twice and every triple three times, because a common multiple of two periods is a multiple of each. The shared instants of `i` and `j` are the multiples of `lcm(p_i, p_j)`, so inclusion-exclusion over the union of the three multiple-sets gives

`C(x) = sum_i floor(x/p_i) - sum_{i<j} floor(x/lcm(p_i,p_j)) + floor(x/lcm(p_1,p_2,p_3))`.

For `n <= 3` the clean encoding is to iterate non-empty subsets as bitmasks `1..2^n-1`, each contributing `sign * floor(x / lcm(subset))` with `+` for odd-size subsets and `-` for even, building the subset `lcm` incrementally from `L = 1`.

The correction terms are not decoration. If I count only `sum_i floor(x/p_i)`, the over-count shifts the rank by the number of overlaps, and the binary search then returns a real firing time that is simply the wrong one — the most deceptive kind of failure. On the sample `p = [2,3], K = 5` (distinct times `2,3,4,6,8,...`, answer `8`) the naive count gives `countLE(6) = 6/2 + 6/3 = 5`, so the search stops at `6`, off by one position because the shared instant `6` was counted twice. With the full formula `C(6) = 3 + 2 - floor(6/6) = 4` and `C(8) = 4 + 2 - floor(8/6) = 5`, so the search first reaches `K=5` at `x=8`. The `-floor(x/6)` term is doing precisely the de-dup of the coincidence at `6`.

Now the overflow. The subset `lcm` for three periods near `10^9` is near `10^27`; folding `L = L / gcd * p[i]` overflows and the sign logic downstream becomes garbage. The observation that rescues it: any `lcm` larger than the search ceiling can never divide any `x` I query, so `floor(x / lcm) = 0` and that subset contributes nothing. I never need the true `lcm` value, only whether it exceeds the range. So I saturate: pick `CAP = 4 * 10^18` (above the largest `hi = min(p)*K <= 10^18`, still under the 64-bit limit), and when folding `lcm(L, p_i)` would exceed `CAP`, return `CAP + 1` and treat the subset as contributing `0`. The multiply `q * p_i` (with `q = L / gcd`) is itself the overflow site, so the test has to come *before* it: `if (q > CAP / p_i)` saturate, never compute `q * p_i` first. On the dangerous triple `[10^9, 999999999, 999999998]`: folding the first two stays near `10^18 < CAP`, then folding the third has `q ≈ 10^18 > CAP / 999999998 ≈ 4*10^9`, so it saturates to `CAP+1` and the triple term is `0` — correct, since no `x <= 10^18` is a common multiple of all three.

The binary search itself needs the lower-bound form to land on the right rank: `mid = lo + (hi - lo)/2` (the `lo + hi` form would overflow near `10^18`), the predicate `C(mid) >= K` with `hi = mid` / `lo = mid + 1`, and return `lo`. Using `>` or returning `lo - 1` would give the neighbouring firing time. The ceiling `hi = min(p) * K` is valid because at that `x` the smallest-period machine alone already gives `C(x) >= K`, so the predicate holds at the top and the search cannot run off it.

The edge cases the periods can produce all fall out of the same formula. With `n = 1` only the singleton subset survives and `C(x) = floor(x/p)`, so the answer is `K * p` — exactly the ceiling `min(p)*K`. `K = 1` gives `min(p)`, the smallest period, e.g. `p=[6,4]` yields `4`. Equal periods `[2,2]` stress the correction term: `C(x) = 2 floor(x/2) - floor(x/lcm(2,2)) = 2 floor(x/2) - floor(x/2) = floor(x/2)`, so the duplicate machine cancels exactly and `K=5` gives `10`. A period of `1` makes every integer a firing time, so `K = 10^9` gives `10^9`. A divisor chain like `[2,3,4]` has heavy overlap between `2` and `4` that only the signed subset sum nets correctly.

For validation the merge oracle runs on small periods and small `K`, producing the true distinct sequence directly, and every comparison agrees. It cannot reach the `10^9` regime, so for large periods I cross-check against an unbounded big-integer binary search — no overflow, so it pins the saturation logic specifically — and those agree too.

The full program is in the answer: `gcd_ll`, the saturating `lcm_sat` with its pre-multiply cap test, the subset-mask `countLE` implementing the signed inclusion-exclusion, and the lower-bound loop over `[1, min(p)*K]`.
