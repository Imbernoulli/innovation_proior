With `n, m <= 10^5` the product table has up to `n*m = 10^10` entries, so materializing and sorting it is out before I start. And with `a[i], b[j] <= 4*10^9` a single cost `a[i]*b[j]` reaches `4*10^9 * 4*10^9 = 1.6*10^19`. That second number is the one that decides this problem: it sits above the signed 64-bit ceiling (`~9.22*10^18`), above the upper bound of any answer-value search I would run (`1.6*10^19` itself), and even unsigned 64-bit only reaches `~1.84*10^19`, so there is barely a whisker of headroom. The answer I must print can itself *be* that maximal product, so overflow is on the main path, not a corner I patch later. I carry the candidate answer and every product in `__int128`, and I make that decision first.

So I need an implicit K-th selection. A heap that pops the smallest-unseen frontier `K` times dies on contact: `K` can be `n*m = 10^10`. What survives is binary search on the answer value. All weights are positive, so `f(x) = #{(i,j) : a[i]*b[j] <= x}` is nondecreasing in `x`, and the `K`-th smallest cost is the smallest `x` with `f(x) >= K`. That `x` is necessarily an attained product: just below it `f < K`, so the count can only cross `K` by passing a real cost, which means the value I converge to equals that cost — the search target is well defined and lands on an actual pairing. Bisecting `x` over `[1, 1.6*10^19]` with a count that costs `O(n log m)` gives `O(n log m * log(MAXPROD))`; at `n = m = 10^5` that is `n * log2 m ~ 1.7*10^6` comparisons per outer step across ~64 steps, about `1.1*10^8` `__int128` comparisons, and a run at that size finishes in roughly 0.58 s using under 5 MB — inside the 2 s / 256 MB budget.

For `f(x)` I sum over bases the count `#{j : a[i]*b[j] <= x}`. Sort `b` once ascending; since `a[i] > 0`, the product `a[i]*b[j]` increases in `j`, so the qualifying catalysts form a prefix and I binary-search its length.

The tempting way to find that prefix is to divide: count `b[j] <= x / a[i]`. For positive integers `a[i]*b[j] <= x` iff `b[j] <= floor(x / a[i])`, so *integer* floor division would in fact be correct — but `x` runs to `1.6*10^19` and does not fit the integer type, and the reflex fix for that, `(double)x / a[i]`, is poison at this scale. A `double` carries ~53 mantissa bits, so `9000000000000000001` is not representable and rounds to `9000000000000000000.0`; then a comparison like `9e18 <= (double)(9e18 + 1) / 1` returns false and `f` undercounts by one at exactly the magnitudes the hidden tests aim at — the binary search would converge one product off with no crash to warn me. So I never form a `double` from these values. Instead I keep the test as an exact product: for each base I binary-search the first `j` where `(i128)a[i] * b[j]` exceeds `x`, and add that index. No division, no float — every comparison is an exact 128-bit product against an exact 128-bit threshold.

On the small instance the count and its boundary behave as the crossing argument claims. With `a = [4, 2, 3e9]`, sorted `b = [2, 5, 3e9]`, `K = 8`, at `x = 15e9` bases `4` and `2` admit all three catalysts and base `3e9` admits only `{2, 5}` (since `3e9*3e9 = 9e18 > 15e9`), so `f = 3 + 3 + 2 = 8`; drop to `x = 15e9 - 1` and the `3e9` row falls to `{2}`, giving `f = 7`. So `f` steps across `K` exactly at `15000000000`, and that value is an attained product — the 8th cost.

The outer loop is a plain `lo = 1`, `hi = 1.6e19` bisection that pulls `hi` down to the smallest tested `x` with `f(x) >= K`. I form the bound as `(i128)4000000000ULL * (i128)4000000000ULL`, with the factors widened *before* the multiply so the product is built in 128-bit. Done in `u64` the current constants happen to fit, but that pattern wraps silently the instant a bound grows, and there is no reason to leave the landmine armed. I write the midpoint as `lo + (hi - lo)/2` so it cannot overflow even if the range widens. `K` itself reaches `10^10`, so it lives in a `long long` and I compare the count against `(i128)K`.

One 64-bit assumption can still slip back at output. `cout` has no `operator<<` for `__int128`, and `(long long)lo` on a `9e18` answer would print a wrapped negative. So I render the result to decimal by hand — peel digits with `% 10` and `/= 10`, reverse — behind a `v == 0` guard that never fires here (all weights are `>= 1`) but costs nothing.

The extremes then fall out. `K = 1` returns the minimum product and `K = n*m` the maximum — the maximum being the most overflow-prone value in the problem. Ties are honored automatically: when several pairings share a cost, `f` jumps by their multiplicity, so any `K` inside that run returns the shared value, matching the "each equal cost occupies its own slot" rule. Values up to `4*10^9` exceed `int`, so `a` and `b` are read into `u64`; an `int` read would corrupt them before any arithmetic touches them, and `cin >>` skips arbitrary whitespace so the input's line breaks do not matter.
