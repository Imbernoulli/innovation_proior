Every one of the `T` tasks must go to exactly one of the `W` workers, and the twist that sets this apart from a plain assignment is that a worker's cost is not additive in its tasks. On top of the base cost `c[i][j]`, the `m`-th task handed to worker `i` carries a marginal surcharge `s_i(m) = base[i] * max(0, m - q[i])`, with `q[i]` a free quota. So the price of giving a worker one more task depends on how many it already holds — the per-task decisions are coupled through each worker's running load, and that coupling is the entire difficulty. Before anything else I fix the scale, because it dictates the arithmetic. `W, T <= 100`, base costs up to `10^9`, slopes up to `10^6`. The base part alone can reach `100 * 10^9 = 10^11`; one worker taking all 100 tasks at slope `10^6` pays overtime `10^6 * (1+2+...+100) ≈ 5*10^9`; the grand total stays under `10^14`. That is well past the 32-bit ceiling of `~2.1*10^9`, so every accumulator and every edge cost has to be `long long`. An `int` anywhere in the cost path is a silent wrong answer on the large tests, not a crash — the overflow trap the constraint sizes set up.

The shape "workers to tasks, minimize cost" is the assignment problem, and the reflex is the Hungarian algorithm in `O(n^3)`. But Hungarian prices a matching as a sum of fixed, independent edge weights `c[i][j]`, with each worker taking exactly one task. Both assumptions are wrong here: a worker takes many tasks, and the cost of its `m`-th task is not a number I can write on an edge, because `m` is decided by the global assignment, not by the edge in isolation. The natural rescue is to clone worker `i` into `T` "slots" `i_1..i_T` and put the marginal `s_i(k)` on the edges into slot `i_k`, then run a min-cost matching. It leaks, and the reason is precise: a plain matching does not *order* the slots. Nothing stops it from routing a single task into slot 2 and paying `s_i(2)` while slot 1 sits empty — charging a worker for its second task when it holds only one (or, under a different charging scheme, undercounting). To make it correct I would need "fill slot 1 before slot 2 before slot 3," an ordering constraint no matching can express. A fixed-weight, unordered structure simply cannot price a cost that depends on a worker's eventual load. Hungarian is structurally inadequate, not just slow.

The operation Hungarian cannot do — route several units through one worker and price each unit separately *and in order* — is exactly what a flow network gives for free, provided the per-unit prices are non-decreasing. Build a directed network: a source `S` and sink `K`; one node per task, one per worker; `S -> task_j` with capacity 1, cost 0; `task_j -> worker_i` with capacity 1, cost `c[i][j]`; and from each worker `i` to the sink, `T` **parallel unit-capacity edges**, the `m`-th costing the marginal `s_i(m) = base[i] * max(0, m - q[i])`. Find a min-cost flow of value `T`. A flow of value `T` saturates all source edges, so every task ships its one unit to some worker; the units a worker receives must leave through its parallel sink edges, and their cost is exactly that worker's overtime. Total flow cost = chosen base costs + overtime = the objective.

The correctness hinges on convexity. The marginals `s_i(1) <= s_i(2) <= ... <= s_i(T)` are non-decreasing in `m` because `max(0, m - q[i])` is. A min-cost flow, augmenting along shortest paths, always consumes a worker's cheaper parallel edges before its more expensive ones — at the optimum no worker pays for its `m`-th edge while its `(m-1)`-th sits unused, since rerouting onto the cheaper edge can only lower cost. So "fill the cheap slots first, in order" — the exact constraint that defeated the matching model — falls out of optimality automatically. This is the whole reason the encoding is valid; if the cost were concave (marginals decreasing), the flow would still grab the cheap edges first and would misprice it, and a genuinely different (integer-programming) method would be needed. The problem hands me convexity, and that is the precondition that makes the per-unit-edge trick exact. The per-worker total `base[i]*(m-q[i])(m-q[i]+1)/2` for `m > q[i]` is the quadratic load-balancing pressure, and the flow finds the optimal split across workers on its own.

At these sizes the network has `W + T + 2 <= ~200` nodes and roughly `2*10^4` edges (`T` source edges, `W*T <= 10^4` task-worker edges, `W*T` parallel sink edges), and I need a flow of value `T <= 100`. Successive shortest paths (SSP) fits cleanly: repeatedly find the cheapest augmenting path and push one unit (every path here is bottlenecked at 1 by the unit source edges), so `T` augmentations. All original edge costs are non-negative, but residual edges after the first push are negative, so I run Dijkstra on **Johnson-reduced costs**: keep a potential `h[v]` equal to the previous round's shortest distance and relax on `cost(u,v) + h[u] - h[v] >= 0`. Each augmentation is `O(E log V)`, giving `O(T * E log V) ≈ 100 * 2*10^4 * 8 ≈ 1.6*10^7` operations — trivial against a 2-second limit.

The one place clean math transcribes into a subtle bug is that reduced cost. The relaxation must read `dist[u] + e.cost + h[u] - h[e.to]`; all four terms are load-bearing. The tempting slip is to fold `h[u]` into `dist[u]` and drop it, writing `dist[u] + e.cost - h[e.to]`. That silently breaks the invariant that reduced costs are non-negative and that `h[t]` after the update equals the true shortest-path cost, so on the second and later augmentations the parallel sink edges get mis-ranked and the accumulated overtime comes out too low (the first augmentation is unaffected because `h` starts at all zeros). I keep the full `+ h[u]` term, and I guard against unreachable nodes: never fold an infinite `dist` into `h`, and skip relaxation out of a node whose potential is `LLONG_MAX`, so the arithmetic never overflows on an unreached vertex.

The edge cases the hidden tests lean on all fall out of the model rather than needing special code. `T = 0`: the flow loop wants zero units and returns cost 0 immediately; the empty cost rows parse to nothing. `W = 1`: the single worker absorbs every task through its parallel sink edges, paying the full convex overtime. `q[i] >= T`: `max(0, m - q[i]) = 0` for every `m <= T`, so all overtime edges cost 0 and the network degenerates to a pure linear assignment; `base[i] = 0` degenerates the same way regardless of quota. `q[i] = 0`: overtime bites from the first task, marginals `base, 2*base, ...`. I confirmed the model against an exhaustive `W^T` brute force that recomputes each worker's convex overtime directly — 1500+ random small cases (`W <= 4`, `T <= 7`) plus these explicit edges, zero mismatches — and the documented sample reproduces its answer of 16. At maximum scale (`W = T = 99`, costs near `10^9`) it runs in about 0.01s.
