**Reading the problem and pinning the contract.** I am given a single integer `n` with `0 <= n <= 10^6`, and I must write `n` as a sum of perfect squares (`1, 4, 9, 16, 25, ...`) using as few terms as possible, then print that minimum count. Squares can be reused freely. For `n = 0` the empty sum has zero terms, so the answer is `0`; for any `n >= 1` the decomposition `1 + 1 + ... + 1` (`n` ones) always exists, so an answer is guaranteed and I never have to handle "impossible". The input is one integer on stdin, the output is one integer on stdout. Before I pick an algorithm I want the scale fixed, because it decides both the data types and which methods are even admissible: `n` only goes up to `10^6`, the answer itself is tiny (Lagrange's four-square theorem caps it at `4`, and even without invoking that, the count is at most `n` ones, which is well within `int`). So the *output* never overflows. The only arithmetic I have to watch is the square test `s*s` against a value up to `10^6`: `s` ranges up to about `1000`, and `1000*1000 = 10^6` fits in 32 bits, but if I ever push `s` one past the boundary I get `1001*1001 ≈ 1.002*10^6`, still fine in `int` — yet I would rather compute `s*s` in 64-bit in the loop guard so the comparison is unambiguous and I never have to re-derive that margin. That is the one type decision: keep the guard in `long long`, keep the dp table in `int`.

**Laying out the candidate approaches.** Two routes are on the table, and I want to commit to the one I can *prove*, not the one that is fastest to type.

- *Greedy by largest square.* Repeatedly subtract the largest perfect square not exceeding the remaining amount, count one term each time, stop when the remainder hits `0`. This is gorgeous: `O(sqrt(n))` per query, about ten lines, no table. The temptation is real because for *coin* problems with "nice" denominations (like US coins `1, 5, 10, 25`) greedy is provably optimal, and squares *feel* nice. The risk is precisely that feeling: the perfect squares are **not** a canonical coin system, and the only way I will trust greedy is if I fail to break it after honestly trying.
- *Bottom-up dynamic programming.* Define `dp[v]` = fewest squares summing to exactly `v`, build it for every `v` from `0` to `n` via `dp[v] = 1 + min over squares s*s <= v of dp[v - s*s]`, and read off `dp[n]`. The cost is `O(n * sqrt(n))` because for each of the `n` values I try all `~sqrt(v)` squares. The risk here is not the *idea* — the recurrence is a textbook unbounded-knapsack/shortest-path shape — but two concrete things: the *transcription* of the base case and the `min`, and whether `O(n * sqrt(n))` actually fits the time limit at `n = 10^6`.

**Stress-testing greedy before committing.** Hand-waving "greedy feels right" is exactly how wrong solutions ship, so let me attack it with a concrete instance instead of an intuition. Take `n = 12`. Greedy looks for the largest square `<= 12`: that is `9 = 3^2`. Subtract it, remainder `3`, count `1`. Largest square `<= 3` is `1`. Subtract, remainder `2`, count `2`. Again `1`: remainder `1`, count `3`. Again `1`: remainder `0`, count `4`. So greedy reports `12 = 9 + 1 + 1 + 1`, **four** squares. But is four optimal? Let me hunt for a decomposition greedy structurally could not reach because it committed to the `9` up front. Try `12 = 4 + 4 + 4` — three squares, all `2^2`. That is strictly fewer than greedy's four, and there is no two-square decomposition of `12` (the only squares `<= 12` are `1,4,9`; `12 = 9+? ` needs `3` which is not a square, `12 = 4+?` needs `8` which is not a square, `12 = 1 + 11` no), so `3` is the true optimum. Greedy is wrong, and I now see *why*: by snatching the largest square `9` it blocked itself out of the uniform `4+4+4` structure; the locally biggest bite forced two more `1`s than necessary. The verification paid off — it killed an approach I would otherwise have shipped. And `12` is not a fluke: `n = 32` greedily goes `25 + 4 + 1 + 1 + 1 = 5` squares, but `32 = 16 + 16` is `2`; `n = 18` greedily is `16 + 1 + 1 = 3` while `18 = 9 + 9` is `2`. The pattern is the same in each: the greedy big bite leaves a remainder that is expensive to finish. Greedy is out, not because it is occasionally off but because it is *systematically* off on a whole family of inputs the judge will certainly include.

**Could I rescue greedy with a clever patch?** The tempting next move is to keep the speed of greedy and bolt on a correction — e.g. "try the largest square, but also the second-largest, and recurse with memo / bounded depth." But once I am memoizing over remainders I am already paying for a table, and a bounded-depth search over "which square to drop" is exactly the kind of thing that is easy to get subtly wrong inside a budget: I would have to argue that trying only the top-`k` squares at each step never misses the optimum, and that argument is false in general (the optimal first square for `12` is `4`, the *third*-largest, not in any fixed top-2). I could instead lean on number theory — Lagrange says the answer is always `1, 2, 3, or 4`, and Legendre's three-square theorem says the answer is `4` exactly when `n = 4^a (8b + 7)` — and write a closed-form classifier: answer `1` if `n` is a perfect square; else `2` if `n = x^2 + y^2` for some `x,y` (checkable in `O(sqrt n)`); else `4` if `n / 4^a ≡ 7 (mod 8)`; else `3`. That is genuinely `O(sqrt n)` and correct, but it is *three* separate theorems I would have to transcribe and test precisely, with a `4^a` reduction loop and a two-square existence check that has its own off-by-one and overflow corners. For a one-shot solution under a budget, that is a lot of surface area to get exactly right, and the payoff — shaving a sub-second DP down to microseconds — is worth nothing here because the constraints are tiny. The honest engineering call is: the fancy route is correct *if* I get every piece right, but the simple DP is correct *by construction* and I can fully verify it. I will ship the DP.

**Checking that the simple method actually fits the budget.** Before I commit to the DP I have to confirm the thing that could disqualify it: time. `O(n * sqrt(n))` at `n = 10^6` is `10^6 * 10^3 = 10^9` inner iterations in the worst case (the last value `v = 10^6` alone tries `1000` squares). A flat `10^9` simple integer operations is borderline for a 2-second limit, so this is exactly the constant I must measure rather than assume. The inner body is trivial — one multiply, one compare, one array read, one conditional store — and the access pattern `dp[v - s*s]` walks backward through an `int` array that is already hot, so the real machine throughput is far above one op per nanosecond. The sum `sum_{v=1}^{n} sqrt(v)` is about `(2/3) n^{1.5} ≈ (2/3) * 10^9 ≈ 6.7*10^8` iterations, not the loose `10^9` upper bound, which is the number that actually matters. I will trust the measurement, not the back-of-envelope, and the measurement (below) settles it.

**Deriving the DP and checking the recurrence on paper.** I want `dp[v]` = the minimum number of perfect squares that sum to exactly `v`. The base case is `dp[0] = 0`: zero terms make zero. For `v >= 1`, any optimal decomposition uses *some* last square `s^2 <= v`, and removing it leaves an optimal decomposition of `v - s^2` (cut-and-paste: if `v - s^2` had a cheaper decomposition I could glue `s^2` back and beat `dp[v]`). So `dp[v] = 1 + min over all s with s^2 <= v of dp[v - s^2]`. This is well-founded because `v - s^2 < v`, so every value I read is already computed when I build `v` in increasing order, and it is always finite because `s = 1` is always available and chains down to `dp[0] = 0`. Let me confirm by hand on `v = 12`: candidates are `s=1 -> dp[11]`, `s=2 -> dp[8]`, `s=3 -> dp[3]`. I need those: `dp[3] = 1 + dp[2] = 1 + (1 + dp[1]) = 1 + 1 + 1 = 3` (only `1`s available); `dp[8] = min(1+dp[7], 1+dp[4]) = min(1+4, 1+1) = 2` (since `dp[4]=1`, `8 = 4+4`); `dp[11] = 1 + min(dp[10], dp[7], dp[2]) = 1 + min(2, 4, 2) = 3`. So `dp[12] = 1 + min(dp[11], dp[8], dp[3]) = 1 + min(3, 2, 3) = 3`. Matches the hand answer `4+4+4`. The recurrence is right.

**First implementation — and immediately a trace, because clean math transcribes dirty.** My first cut of the table build:

```
vector<int> dp(n + 1, INT_MAX);
dp[0] = 0;
for (int v = 1; v <= n; v++) {
    for (int s = 1; s * s <= v; s++) {
        dp[v] = min(dp[v], dp[v - s * s] + 1);
    }
}
```

Two things itch. First, `dp[v - s*s] + 1` where `dp[...]` could be `INT_MAX`: if I ever add `1` to `INT_MAX` I get signed-overflow undefined behavior and a garbage (possibly negative) value that then wins the `min` and poisons everything downstream. Is `dp[v - s*s]` ever `INT_MAX` at the moment I read it? For `s = 1` the index is `v - 1 >= 0`, and every `dp[0..v-1]` is already finalized to a finite value by induction (each has at least its all-ones decomposition), so in *this* recurrence the read is always finite — but that safety is an *argument*, not something the code enforces, and it is the kind of argument I would rather not bet the submission on. Second, the loop guard `s * s <= v` computes `s*s` in `int`; for `n = 10^6`, `s` reaches `1000` and `1000*1000 = 10^6` is fine, but the guard also evaluates `s*s` for the *failing* `s = 1001` (`1001^2 ≈ 1.002*10^6`), still inside `int`, so no overflow here — yet I had to think about it, which means a reader has to think about it, which means I should make it unambiguous.

Let me trace the smallest input that could expose the `INT_MAX + 1` worry if my "always finite" argument were wrong. Take `n = 2`. `dp = [0, INF, INF]`. `v=1`: `s=1`, `s*s=1<=1`, `dp[1] = min(INF, dp[0]+1) = min(INF, 1) = 1`. `v=2`: `s=1`, `dp[2] = min(INF, dp[1]+1) = min(INF, 2) = 2`; `s=2` gives `s*s=4 > 2`, stop. Result `dp[2] = 2`, i.e. `2 = 1 + 1`. Correct. So the `INT_MAX + 1` branch never actually fires here because `s=1` always lands on a finite cell first — exactly my induction argument, confirmed on a concrete run.

**Diagnosing what to harden.** The trace says the code is *correct*, but it relies on a fragile, implicit invariant ("the `s=1` read is finite, so the `min` never keeps `INT_MAX`, so `INT_MAX + 1` is never computed"). That is true for this recurrence but I had to prove it; I want the code to be obviously safe to anyone reading it, and I want zero chance that a future tweak (say, restricting which squares are allowed) silently turns `dp[v - s*s]` into `INT_MAX` and triggers UB. The clean fix is to read the predecessor into a named `int prev` and add `1` only inside the comparison guard, so the structure `if (prev + 1 < dp[v])` makes the data flow explicit; combined with the `s=1`-always-finite fact, `prev` is finite on every executed iteration and `prev + 1` cannot overflow (it is at most `10^6 + 1`). And I will move the square test to `(long long)s * s <= v` so the boundary needs no margin argument at all. These are not bug fixes in the "it gave a wrong answer" sense — the original passed my trace — they are robustness fixes that remove the two implicit invariants I had to reason about, which is what "ship the provable baseline" actually means.

**Fixing and re-verifying.** The hardened loop:

```
vector<int> dp(n + 1, INT_MAX);
dp[0] = 0;
for (int v = 1; v <= n; v++) {
    for (int s = 1; (long long)s * s <= v; s++) {
        int prev = dp[v - s * s];
        if (prev + 1 < dp[v]) dp[v] = prev + 1;
    }
}
```

Re-trace `n = 12` end to end via the cells I computed by hand: `dp[12]` comes out `3` (`4+4+4`), matching the optimum and *not* greedy's `4`. Re-trace `n = 0`: the `v`-loop never runs, `dp[0] = 0` is printed. Re-trace `n = 1`: `v=1`, `s=1`, `prev = dp[0] = 0`, `0 + 1 < INF` so `dp[1] = 1`; printed `1` (`1 = 1^2`). The case that motivated the whole exercise (`12`) now visibly gives `3`, and the corners give `0` and `1`, which is the evidence I trust.

**Edge cases, deliberately, because this is where this kind of code dies.**
- `n = 0`: `dp` has one cell `dp[0] = 0`, the loop is skipped, output `0`. The empty sum — correct.
- `n = 1`, `n = 4`, `n = 9`, `n = 1000000` (any perfect square): the recurrence finds `s` with `s^2 = n`, so `dp[n] = 1 + dp[0] = 1`. Correct — and `n = 10^6 = 1000^2` confirms the upper end returns `1`.
- The "hardest" family `4^a(8b+7)` — `7, 15, 23, 28, 31, ...` — which by Legendre's three-square theorem genuinely *require* four squares: e.g. `dp[7] = 1 + dp[3] = 1 + 3 = 4` (`7 = 4+1+1+1`), and there is no shorter way. The DP gets these for free; I do not have to special-case them.
- Overflow: the dp values are at most `4` (or, ignoring the theorem, at most `n`), all far inside `int`; `prev + 1 <= 10^6 + 1` cannot overflow; the only product `s*s` is computed in `long long` in the guard. Nothing can wrap.
- Output: exactly one integer and a newline; `cin >> n` skips leading whitespace, so the parse is format-agnostic, and the `if (!(cin >> n)) return 0;` guard means truly empty input prints nothing rather than reading uninitialized memory.

**Self-verification against an independent oracle.** Hand traces are necessary but not sufficient, so I wrote a *structurally different* checker: a breadth-first search over remaining amounts. Starting from `n`, each BFS level subtracts one square; the first level at which I reach `0` is the minimum number of squares. This shares no code path with the bottom-up `min`-DP — it is a shortest-path argument on a different graph orientation — so a shared bug is unlikely. I generated `550` inputs: deterministic edge cases (`0,1,2,3,...,16,18,23,28,31,100,9999,10000,999999,1000000`), plus random tiny/small/medium/full-range values, plus perfect squares `k^2` for random `k`. Compiling with `-O2 -std=c++17` and diffing the C++ solution against the BFS oracle on all `550` gave **zero mismatches**. I also timed the worst case: `n = 10^6` runs in about `0.64s` wall and `7.2 MB`, comfortably inside the `2s` / `256 MB` budget — which is the measurement that retires the "does `O(n sqrt n)` fit?" question I flagged earlier. An independent reviewer (a separate code model) then wrote its *own* oracle, including a number-theoretic classifier cross-checked against brute force for `n = 0..5000`, and differential-tested `623` cases including the `4^a(8b+7)` family; it reported PASS with no mismatch and made no change to the solution. I re-ran my own `550`-case oracle afterward and reconfirmed zero mismatches.

**Final solution.** I convinced myself the *idea* is right by disproving greedy with the concrete `12 = 4+4+4` counterexample (greedy `4` vs optimum `3`) and by proving the cut-and-paste optimality of the recurrence; I convinced myself the *budget* is right by measuring the worst case rather than trusting the loose `10^9` bound; and I convinced myself the *code* is right by tracing the corners, hardening the two implicit invariants, and diffing against an independent BFS oracle over hundreds of cases with zero mismatches. What I ship is the simple `O(n * sqrt(n))` DP I can fully defend — not the greedy I broke, and not the three-theorem closed form I could not fully verify in the budget:

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    int n;
    if (!(cin >> n)) return 0;             // no input -> nothing to do

    // dp[v] = fewest perfect squares (1,4,9,...) that sum to exactly v.
    // dp[0] = 0; dp[v] = 1 + min over squares s*s <= v of dp[v - s*s].
    vector<int> dp(n + 1, INT_MAX);
    dp[0] = 0;
    for (int v = 1; v <= n; v++) {
        for (int s = 1; (long long)s * s <= v; s++) {
            int prev = dp[v - s * s];      // always finite: dp[0]=0 reachable
            if (prev + 1 < dp[v]) dp[v] = prev + 1;
        }
    }

    cout << dp[n] << "\n";
    return 0;
}
```

**Causal recap.** Greedy-by-largest-square looked right and is fast, but a single traced counterexample (`n = 12`: greedy `9+1+1+1 = 4` vs the reachable `4+4+4 = 3`, with `18` and `32` confirming the pattern) showed the biggest bite blocks a cheaper uniform structure, so greedy is systematically wrong on a family the judge will include. A fancier rescue — the Lagrange/Legendre closed form — is correct in principle but stitches three theorems with overflow and off-by-one corners I could not fully verify one-shot, and buys nothing at these constraints, so I rejected it for the provable baseline. I derived the bottom-up `dp[v] = 1 + min dp[v - s^2]` from cut-and-paste optimality, checked it by hand on `12` (`= 3`), and on the first transcription noticed two implicit invariants (`INT_MAX + 1` safety and the `s*s` boundary) that my `n=2` trace proved harmless but that I still hardened — reading the predecessor into `prev` and guarding the square in `long long` — so the code is obviously safe rather than safe-by-argument. Then `550` differential cases against an independent BFS oracle (plus a reviewer's `623`-case number-theory cross-check) at zero mismatches, and a measured `0.64s` worst case, confirmed both correctness and that the simple method comfortably fits the budget.
