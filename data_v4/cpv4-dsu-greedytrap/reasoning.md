The objective is the whole problem: for each request `(s, t)` I want the *maximum over all `s`‚Äì`t` paths* of the *minimum* capacity along the path ‚Äî a maximin bottleneck, not a shortest path. The constraints are lopsidedly on my side: `m` up to `2¬∑10^5` bridges, but only `n, q <= 2000` islands and requests. So I can spend something near-linear in `m` on a one-time preprocess and then as much as `O(n)` per request. Capacities reach `10^9` and the answer is always a single capacity ‚Äî never a sum ‚Äî so 32 bits would technically hold it, but I keep capacities and the running bottleneck in `long long` so I can reserve `LLONG_MAX` / `LLONG_MIN` as "+infinity bottleneck" and "unvisited" sentinels that no real capacity (`>= 1`) can collide with. Island indices stay `int`.

**Greedy is the trap, and both flavours break.** The tempting cheap answer is local, and the problem is built to punish that; both local rules break on small instances. *Direct-edge greedy* takes the strongest bridge straight between `s` and `t`: on `0‚Äì1:6, 1‚Äì2:5, 2‚Äì3:4, 0‚Äì3:2`, request `0 3` reports the direct `2`, but the detour `0‚Üí1‚Üí2‚Üí3` has throughput `min(6,5,4)=4`. The widest path is not the shortest. *Highest-neighbour walk* always steps to the widest incident bridge: on `0‚Äì1:10, 1‚Äì3:1, 0‚Äì2:3, 2‚Äì3:8`, request `0 3` takes `0‚Üí1` (cap 10) and then dies on the only exit `1‚Äì3` of capacity `1`, returning `1`, while `0‚Üí2‚Üí3` gives `min(3,8)=3`. A wide first hop dead-ends into a narrow exit. Both fail for the same structural reason ‚Äî maximin is a global property, and no local rule reconstructs it.

**The exact global characterization.** Sort the bridges by capacity high-to-low and add them one at a time into a disjoint-set union. Claim: the widest-path bottleneck of `(s, t)` is the capacity `c*` of the edge whose insertion first places `s` and `t` in the same component. *Achievable:* at that merge every edge added so far has capacity `>= c*` (we descend), and `s, t` are connected using only those, so an `s`‚Äì`t` path of throughput `>= c*` exists. *Optimal:* just before `c*`, `s` and `t` sat in different components, so every capacity-`> c*` edge lies inside components that don't bridge them; any `s`‚Äì`t` path must cross an edge of capacity `<= c*`. Hence exactly `c*`. That `c*` is the minimum-capacity edge on the unique `s`‚Äì`t` path of the **maximum spanning tree** the DSU builds by keeping each merging edge. With `n, q <= 2000` I build the tree once and answer each request by walking its tree path carrying the running minimum ‚Äî `O(n¬∑q) <= 4¬∑10^6`, trivially inside 2 s.

**The sort direction is the one transcription trap.** A plain `sort` on `(c, u, v)` ascends, which is exactly a *minimum* spanning tree ‚Äî its `s`‚Äì`t` path is dominated by *low* capacities and its path-minimum is a lower bound on connectivity, the opposite of maximin. On the sample, ascending Kruskal keeps `0‚Äì3(2), 3‚Äì2(4), 2‚Äì1(5)` and answers `0 3` with `2` ‚Äî the very direct-edge greedy value I already disproved. So the comparator must descend:

```cpp
sort(edges.begin(), edges.end(),
     [](const array<long long,3>& a, const array<long long,3>& b){ return a[0] > b[0]; });
```

Descending, the DSU keeps `0‚Äì1(6), 1‚Äì2(5), 2‚Äì3(4)` and drops `0‚Äì3(2)` as a cycle; `0 3` walks `min(6,5,4)=4` and `1 3` walks `min(5,4)=4`, both matching the sample, and the discarded `0‚Äì3` is precisely the greedy trap the method now sidesteps.

**The per-query walk has its own trap in the seed value.** The running bottleneck at `s`, having crossed no bridge, is *unconstrained* ‚Äî `+infinity`, not `0`; seeding `best[s] = 0` would collapse every `min(best[x], c)` to `0` and report a bottleneck of zero. So `best[s] = LLONG_MAX`, and the first real edge sets the running minimum to its own capacity. I mark unreached nodes with `LLONG_MIN`; since capacities are `>= 1` and the start carries `LLONG_MAX`, that sentinel never equals a real bottleneck. On the branching tree `0‚Äì1(10), 0‚Äì2(3), 2‚Äì3(8)`, query `0 3`: `best[0]=MAX`, `best[1]=10`, `best[2]=min(MAX,3)=3`, `best[3]=min(3,8)=3` ‚Äî the value the highest-neighbour walk failed to find.

**Boundaries the constraints invite.** `n = 2` with one bridge: the tree is that single edge and the query returns its capacity. Parallel edges between a pair: the descending sort places the stronger first, the DSU merges on it and drops the weaker as a cycle, so the max-spanning-tree argument keeps the better parallel edge with no special-casing. Capacities near `10^9`: the answer is one capacity, no addition ever occurs, and `LLONG_MAX` is only ever consumed by `min`, never added to, so nothing overflows. Connectivity is guaranteed, so every `t` is reached and `best[t]` is always a real value.

**Verification and landing.** I checked the whole thing against an independent maximin Floyd‚ÄìWarshall ‚Äî set `w[i][j]` to the strongest direct edge, then relax `w[i][j] = max(w[i][j], min(w[i][k], w[k][j]))` over all `k`, an all-pairs method with no DSU and no spanning tree ‚Äî over hundreds of random small multigraphs with parallel edges and varied capacity ranges. Zero mismatches; the sample returns `4, 4` and the two greedy-trap instances return `4` and `3`. What ships is one self-contained C++17 file ‚Äî sort descending, build the max spanning tree with a union-by-rank DSU, answer each request with a path-minimum BFS over the tree carrying the bottleneck in `long long` (full module in the answer).
