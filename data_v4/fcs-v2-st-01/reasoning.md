Two substrings named by `(l1, len1)` and `(l2, len2)`, compared lexicographically, `-1/0/1` per query — and both `n` and `q` reach `2*10^5`. The output is tiny; the whole design lives in the per-query cost. What makes this dangerous is the tie-break structure: plain dictionary order, so the first differing character decides, and if one substring runs out while everything matched, the shorter one is smaller. That "runs out" corner is where a comparator that only looks at characters gets subtle. On the arithmetic side there is nothing to fear — every quantity is an index bounded by `2*10^5`, so `int` is enough and I never touch 64-bit. The starts are 1-indexed; internally I work with `i = l1-1`, `j = l2-1`.

The direct comparator just walks both substrings: compare `s[i+t]` against `s[j+t]` for `t = 0, 1, ...` up to `min(len1, len2)`, return on the first difference, else break the tie by length. Correct, three lines, and hopeless on the natural adversarial input. Take `s = "aaaa...a"`, `n = 2*10^5`, and let every query ask `1 n 1 n`: each one scans all `n` characters before concluding "equal", `q` times, for `n*q = 4*10^10` character comparisons. Repetitive strings are exactly what a hidden test set will throw here, so this is not a corner I can wish away. The per-query cost must not scale with how long the two substrings agree.

That agreement length is the whole problem, and it has a name. Comparing the two substrings is almost the same as comparing the two full **suffixes** they start at, because lexicographic order is fixed by the first position where the suffixes differ — the longest common prefix (LCP) of `suffix i` and `suffix j`. The substring lengths only re-enter once I know whether that first difference falls inside the compared region. So let `common = LCP(suffix i, suffix j)` and `cmpLen = min(len1, len2)`:

- if `common < cmpLen`, the suffixes already disagree inside the compared region, and the character at offset `common` decides: `s[i+common]` vs `s[j+common]`;
- if `common >= cmpLen`, the substrings agree on every compared character, so it is a pure length tie-break — shorter is smaller, equal lengths give `0`.

The naive comparator's pathology, long runs of agreement, collapses into that single number `common`. So if I can get `LCP(suffix i, suffix j)` for *arbitrary* `i, j` in `O(1)`, every query is `O(1)` and the whole thing is `O(q)` after preprocessing. Everything now hangs on fast arbitrary-pair suffix LCP.

Two tools give that. A rolling hash with binary search finds the LCP in `O(log n)` per query but is randomized — a collision silently returns a wrong answer, and adversarial repetitive input is precisely where I would not want to gamble on hashing. The deterministic route is the suffix array stack: sort all suffixes into `sa` (with inverse `rnk`), compute the LCP of sorted-*adjacent* suffixes with Kasai into `lcp[r]`, and get the LCP of two *arbitrary* suffixes with ranks `ri < rj` as `min(lcp[ri+1 .. rj])` — walking the sorted order from one suffix to the other, the common prefix can only shrink, so the bottleneck is the smallest adjacent LCP along the way. A range-minimum over the static `lcp` array is `O(1)` with a sparse table. Deterministic, `O(1)` per query, and no collision risk; I take it.

For construction, `n <= 2*10^5` does not justify the linear SA-IS machinery — it is fiddly and easy to get wrong for no asymptotic win that matters at this scale. Prefix doubling with radix sort is `O(n log n)` and comfortably fast enough: rank suffixes by their first character, then at round `k` order them by the pair `(rnk[i], rnk[i+k])`, sort those pairs with two stable counting sorts (second key, then first), and re-rank; after `O(log n)` rounds every suffix has a distinct rank. Kasai gives the LCP array in `O(n)` off `sa` and `rnk` using the invariant that the common-prefix length drops by at most one as the start position advances. The sparse table over `lcp` is the standard `min` doubling, and a query covers `[lo, hi]` with two overlapping power-of-two blocks; the same-suffix case (`i == j`) has no adjacent-LCP range, so I short-circuit it to the suffix length `n - i` before touching the table.

The one genuine trap in the construction is the counting-sort bucket size, and it is specific to this "small string, doubling loop" setup. My instinct is to seed round-one ranks straight from the character:

```
rnk[i] = (unsigned char)s[i];   // ~97..122 for 'a'..'z'
```

and to size the counting-sort buckets to the rank range, which after the first re-ranking is `[0, n-1]`:

```
int maxKey = n + 1;
vector<int> cnt(maxKey + 1, 0);
cnt[secondKey(i) + 1]++;         // secondKey is a rank
```

Those two decisions are individually reasonable and jointly wrong. Re-ranking compresses ranks into `[0, classes-1] ⊆ [0, n-1]` for every round *after* the first — but on round one the ranks are raw character codes around `97..122`, so `secondKey(i)+1` indexes far past a `cnt` sized for `n`. On a short string this writes off the end of the heap: running `banana` (`n = 6`, `cnt` has 8 slots, index ~123) corrupts the allocator and AddressSanitizer flags the write at `cnt[secondKey(i)+1]++` on the first round. The fix is to make the initialization establish the invariant the loop assumes — coordinate-compress the distinct characters into `[0, classes-1]` up front (`lower_bound` into the sorted unique characters), so every rank is in `[0, n-1]` from round one and `maxKey = n+1` bounds every key in every round. After that `banana` runs clean, and its comparisons come out right — `"ana"` vs `"ana"` is `0`, matching the contract example.

The edge cases that break suffix code are the degenerate sizes and the tie-break boundary. `n = 1` with query `1 1 1 1` hits the `i == j` short-circuit (`n - i = 1`), `cmpLen = 1`, so `eq == cmpLen` and equal lengths give `0` — and the sparse table, though sized for `n = 1`, is never indexed because the short-circuit fires first. All-identical strings, the naive comparator's worst case, now answer every equal-length query in `O(1)` via a single large `common`. Periodic strings like `abcabcabc` exercise the `common >= cmpLen` branch when one substring is a prefix of the other (`"abc"` vs `"abcabc"` → `-1`); this is the branch that must *not* fall through to a character comparison, because there is no differing character to read. Against a trivial slicing oracle over a thousand random small cases (short strings, small alphabets so substrings collide often) the two agree everywhere, and at full scale — `n = q = 2*10^5`, a 3-letter alphabet to force the most doubling rounds and longest LCPs — the `O(n log n)` preprocessing plus `O(q)` querying finishes comfortably inside the 2-second limit and well within the 256 MB.

Output is `q` lines of `-1/0/1`; I accumulate them into one `string` and flush once, so `2*10^5` lines cost one write. The final module is in the answer: suffix array by prefix doubling with radix sort, Kasai LCP, sparse-table RMQ, and the case-split query that turns each comparison into an `O(1)` LCP lookup capped by a length tie-break.
