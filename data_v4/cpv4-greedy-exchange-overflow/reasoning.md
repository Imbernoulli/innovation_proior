I have `n` jobs that run one at a time on a single bench, back to back with no gaps and no preemption. Once I fix a permutation, job `i` finishes at the running sum of processing times up to and including it, `C[i]`, and costs `w[i] * C[i]`. Every job runs — I am ordering the whole set, not selecting a subset — so the only decision is the permutation and I want to minimize `sum_i w[i] * C[i]`. Read `n`, then the `n` processing times, then the `n` weights; print one integer.

What makes *this* instance dangerous before I even pick an algorithm is the scale, because three different quantities cross the 32-bit line at three different points. `n <= 2*10^4` and `p[i], w[i] <= 10^5`. The running clock reaches `n * max(p) = 2*10^4 * 10^5 = 2*10^9`, already past the signed-32 limit `~2.147*10^9`. The final answer, everything maxed, is roughly `w*p*n(n+1)/2 = 10^5 * 10^5 * (2*10^4)(2*10^4+1)/2 ≈ 2*10^18`, which fits a signed 64-bit integer (`~9.2*10^18`) with about 4.6x margin but is nowhere near 32-bit. And, less obviously, whatever key I sort on will end up multiplying a `p` by a `w` and hit `10^5 * 10^5 = 10^10`. So the clock, the accumulator, and every product live in `long long`. This objective is built to punish a stray `int`, and the comparator is the sneakiest place it hides.

Two routes to the ordering. Direct permutation or bitmask search — `O(n!)` or `O(2^n n)` — is correct but unusable at `n = 2*10^4`; I keep it only as a tiny-input oracle. The real candidate is a sequencing rule: sort by one per-job key and run in that order. For weighted completion time the natural key is the ratio `p[i] / w[i]` — short jobs and heavy jobs both want to go early, and the ratio balances the two, `O(n log n)`. Two things to settle: does the local swap argument actually give a global optimum, and how do I compare `p/w` without floating point.

"Sort by ratio" is folklore, so let me derive the swap rather than ship it on faith. Take any order and two jobs `A`, `B` adjacent in it, `A` immediately before `B`. Let `T` be the clock right before `A` starts. Every job other than `A` and `B` keeps its completion time regardless of how I order this pair, because the block `{A,B}` occupies the same window `[T, T + p_A + p_B]` either way and everything after starts at the same instant `T + p_A + p_B`. So only the two terms for `A` and `B` change. With `A` first: `w_A(T+p_A) + w_B(T+p_A+p_B)`. With `B` first: `w_B(T+p_B) + w_A(T+p_A+p_B)`. Subtract A-first minus B-first: the `w_A T`, `w_B T`, and shared `+p_A+p_B` cross-terms cancel, leaving `w_B p_A - w_A p_B`. So A-first is no worse than B-first exactly when `p_A w_B <= p_B w_A`, i.e. `p_A/w_A <= p_B/w_B` — putting the smaller ratio first never increases cost. Any order that is not ratio-sorted has an adjacent inversion I can fix without increasing cost, and repeatedly fixing inversions reaches the ratio-sorted order, so ratio-sorted is optimal.

The inequality that fell out is already in cross-multiplied integer form: order `A` before `B` when `p_A * w_B < p_B * w_A`, ties broken arbitrarily since the swap difference is then exactly zero. I never divide — which sidesteps floating error on near-equal ratios — but `p_A * w_B` reaches `10^10`, so the comparator operands and products are `long long`.

On the given sample `p = [3,1,2]`, `w = [1,2,3]`: ratios `3`, `0.5`, `0.667`, so the order is job1, job2, job0. Job1 finishes at `1`, cost `2`; job2 at `3`, cost `9`; job0 at `6`, cost `6`; total `17`, matching.

The comparator is where the overflow bites without any warning, so I follow what a 32-bit build would actually compute. Suppose `p`, `w` were `int` and the comparator computed `int` products. Take two jobs with `p = [60000, 40000]`, `w = [50000, 90000]`; the true ratios are `1.2` and `0.444`, so the second must go first. Deciding "does job0 precede job1" needs `p0*w1 = 60000*90000 = 5,400,000,000` versus `p1*w0 = 40000*50000 = 2,000,000,000`, and `5.4e9 > 2.0e9` says job0 does not precede job1 — correct. But in 32-bit, `60000*90000` overflows: modulo `2^32` it becomes `5,400,000,000 - 4,294,967,296 = 1,105,032,704`, positive and small, so `1,105,032,704 < 2,000,000,000` is now true and the buggy comparator puts job0 *first* — the exact opposite. The sort order is corrupted, and since the whole method rides on that one comparison the total comes out grossly wrong, not a near miss. With `p`, `w` as `long long` the product stays exact well inside 64-bit; every value touching a product or a running sum is `long long`, and the only `int`s left are the loop counter and the index, both bounded by `n`.

One correctness obligation remains: `std::sort` needs a strict weak ordering, or the sort is undefined behavior. Equal ratios give equal products, so `comp(i,j)` and `comp(j,i)` are both false — the pair compares equivalent, which is legal, and by the zero swap-difference above their relative order is free anyway. So a plain `<` is valid; I add an `i < j` tie-break only to pin the output deterministically across library versions, and it cannot change the cost.

Edge cases. `n = 0`: the read loops and the accumulation loop all run zero times, `answer` stays `0`; I guard the head with `if (!(cin >> n)) return 0;` so an empty stream also degenerates to no output cleanly. `n = 1`: one order, `answer = w[0]*p[0]`. All equal ratios (`p = w = 10^5`, `n = 2*10^4`): order-independent, and the cost is `10^10 * n(n+1)/2 = 2000100000000000000`, which exercises the accumulator at its maximum magnitude and confirms it holds without overflow.

Idea-correct and code-correct are separate claims, so I cross-check against an independent brute that, for tiny `n`, tries *every* permutation and takes the minimum — no greedy reasoning, just the definition. Across 600 random small instances (`n` up to 7, values up to 50, ties and `n = 0` included), Smith's rule matches the permutation brute with zero mismatches, pinning down both the exchange argument and the accumulation. The max instance runs in well under 10 ms inside the 1 s / 256 MB limits.

What I ship is one self-contained file: read into `long long` vectors, sort an index array by the division-free cross-multiplied comparator with an `i < j` tie-break, then sweep once keeping a `long long` clock and accumulating `w[i] * clock`. The full module is in the answer.
