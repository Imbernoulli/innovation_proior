A row of `n` crate stacks with sizes `w[0..n-1]`, collapsed by repeatedly shoving two *adjacent* stacks together; shoving sizes `x` and `y` costs `x + y` and leaves one stack of `x + y`. Keep going until one stack remains; minimize the total effort. Input is `n` then the `n` sizes on stdin, output one integer, and `n <= 1` means no shoves so the answer is `0`. The two constraints that shape everything are `n <= 500` and `w[i] <= 10^6`, and their tension is what decides the design. The grand total of all crates is at most `500 * 10^6 = 5 * 10^8`, which sits comfortably inside a 32-bit int (ceiling `~2.147 * 10^9`), and every individual stack is at most that total. But the *answer* adds up a merge cost once per shove, and each merge cost is itself a stack size, so it accumulates the grand total roughly once per level of the merge tree — it can run several times the grand total while no single quantity in sight ever leaves 32-bit range. That is the number the input constraints hide, so the data type of the accumulator, not the size of the inputs, is what decides correctness here. I'll pin the exact bound once I have the recurrence.

Two routes to the value itself. *Greedy:* always shove the cheapest adjacent pair, repeat — short and intuitive, "do the cheap work first," but a line merge has long-range structure so a locally cheapest shove need not be globally cheapest. *Interval DP:* every stack that ever exists is the sum of a *contiguous* run `w[l..r]` (adjacency is never broken), so the natural subproblem is `dp[l][r]` = least effort to fuse the range `[l..r]` into one stack; the last shove of that range happens at some split `k`, fusing the one stack over `[l..k]` with the one over `[k+1..r]` at a cost equal to all the crates in `[l..r]`, and that per-range cost does not depend on `k`. I want the route I can prove, so I try to break greedy first.

My first instinct for a counterexample is a big element among small ones, `w = [1, 1, 100, 1, 1]`. Greedy shoves the two `1+1` pairs (cost `2` each) reaching `[2, 100, 2]`, then must fuse the `100` with both neighbours in the last two shoves for `102 + 104`, total `210`. But here every order pays the `100` in both of its two shoves — it has exactly two neighbours and both must merge into it eventually — so `210` is actually optimal, and greedy survives. That sharpens what I'm hunting: greedy breaks not when a big element is *unavoidable* but when a cheap early shove *forces* an expensive structure above it. So I balance a case: `w = [6, 4, 4, 6]`. Greedy grabs the cheapest pair, the middle `4 + 4 = 8`, reaching `[6, 8, 6]`, then pays `14 + 20`, total `8 + 14 + 20 = 42`. The balanced order fuses the outer pairs instead — `[6,4] -> 10`, `[4,6] -> 10`, then `[10,10] -> 20` — for `40 < 42`. Greedy is wrong, and I can see exactly why: the cheap middle shove inflated the `8` that both later shoves then had to carry, whereas balancing kept every intermediate stack smaller. Interval DP it is.

Define `dp[l][r]` = minimum effort to fuse `l..r`, with `dp[i][i] = 0`, and

`dp[l][r] = min over l <= k < r of ( dp[l][k] + dp[k+1][r] + seg(l,r) )`,

where `seg(l,r) = w[l] + ... + w[r]` is the size of the final combined stack and thus the cost of the last shove of the range. Since `seg(l,r)` is constant across `k`, the inner loop only minimizes `dp[l][k] + dp[k+1][r]`, and I get `seg` in O(1) from a prefix array `prefix[i] = w[0]+...+w[i-1]`, so `seg(l,r) = prefix[r+1] - prefix[l]`. On the sample `w = [3, 1, 4, 2]` (claimed `20`), prefix `[0,3,4,8,10]`. Length 2: `dp[0][1] = 4`, `dp[1][2] = 5`, `dp[2][3] = 6`. Length 3: `dp[0][2] = min(0+5+8, 4+0+8) = 12`, `dp[1][3] = min(0+6+7, 5+0+7) = 12`. Length 4: `dp[0][3] = min(0+12+10, 4+6+10, 12+0+10) = 20`, at split `k=1` (`[3,1] | [4,2]`), which matches the "shove the 3 and 1, shove the 4 and 2, then shove the halves" order in the statement.

Transcribing to a filled table, the one thing I must not get wrong is read order: `dp[l][k]` spans length `k-l+1 <= len-1` and `dp[k+1][r]` spans `r-k <= len-1`, both strictly shorter than the current `len`, so iterating `len` from small to large guarantees both operands are already computed — no stale reads. The other trap is the min-scan seed: I must start `best` at a value larger than any real candidate, not at `0`. With `best = 0`, on `w = [5, 7]` the only candidate `dp[0][0]+dp[1][1]+12 = 12` loses to `min(0, 12) = 0` and I'd report a zero-cost fusion; seeding `best = INF` gives the correct `12`.

Now the pitfall the problem is actually built around, which no small test can expose. If I write `dp` and `prefix` as `int`, `[3,1,4,2]` passes (its largest value is `22`), so I have to reason about the *largest legal* case: `n = 500`, every `w[i] = 10^6`. The grand total `seg(0, n-1) = 5 * 10^8` still fits in an int. But `dp[0][n-1]`? With all weights equal, the line merge is forced to balance like a binary tree of `n` leaves and the total cost is `c` times the sum of subtree leaf-counts over internal nodes, about `c * n * log2(n) = 10^6 * 500 * 8.97 ~ 4.5 * 10^9` — already roughly double the int ceiling `2.147 * 10^9`, while every `seg` and every `w[i]` stayed in range. Running the 64-bit DP on that exact input returns `4488000000`, agreeing with the `4.5 * 10^9` estimate to two figures, and it exceeds `2^31 - 1`. Running the same DP with `int` types prints `536870911`, which is `INT_MAX / 4` — the `INF` sentinel itself, meaning intermediate int sums overflowed into the comparisons and corrupted the minima; the failure is not "wrong by a wrap amount" but total garbage. So `prefix`, `seg`, `dp`, and every accumulator are `long long`. I keep `prefix` in 64 bits too so `seg` is never formed by a 32-bit addition, even though the total alone would fit.

One observation is not a proof, so I bound the worst case in general. Total cost <= (grand total) * (number of merges) = `5 * 10^8 * 499 ~ 2.5 * 10^11`, a gross overestimate of the true `4.488 * 10^9` but a genuine worst case, and it sits seven orders of magnitude below the `long long` ceiling `~9.2 * 10^18`. No overflow is possible. The sentinel `INF = LLONG_MAX / 4 ~ 2.3 * 10^18` is larger than any real `dp` value yet leaves headroom against addition, and since `dp[l][k]` and `dp[k+1][r]` are always real values for `l <= k < r`, `INF` is never actually summed anyway.

The corners fall out of the recurrence: `n = 0` and `n = 1` hit the early `cout << 0` (an empty or singleton row needs no shove); `n = 2` gives one shove `dp[0][1] = w[0]+w[1]`; zero-weight stacks are allowed and simply give `seg = 0`, so an all-zero row returns `0` at every cell. The `cin >>` parser skips arbitrary whitespace, so the token stream is format-agnostic.

For the decisive check I lean on an independent oracle rather than my own factorization of the process: a memoized brute force that represents the row as a tuple of current sizes and recurses over *every* choice of which adjacent pair to shove next. If my "the last shove splits at some `k`" reasoning were wrong, it would disagree. Across 650 random small rows (sizes `0..8`, weights up to `10^5`, including the `n = 0,1,2` corners) there are zero mismatches. On cost, the DP is `O(n^3)`: `n = 500` is about `1.25 * 10^8` cheap inner steps, roughly `20` ms and `6` MB on the all-`10^6` worst case — well inside `2` s / `256` MB, so the Knuth-Yao `O(n^2)` quadrangle speedup is unnecessary and I'd rather ship the cubic version I can prove correct.

The final program reads `n` and the sizes, early-returns `0` for `n <= 1`, builds the `long long` prefix array, fills `dp` by increasing length with the recurrence above seeded at `INF`, and prints `dp[0][n-1]`; the full self-contained module is in the answer.
