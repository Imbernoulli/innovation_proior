The word that governs this whole problem is *subsequence*, not *substring*. I have to return the length of the longest palindromic subsequence of a lowercase string `s` — a palindrome I build by deleting characters from anywhere, the survivors not needing to be adjacent. That one word is the entire trap. In `character`, the longest palindromic subsequence is `carac`, length `5`, pulled from scattered positions, while no palindromic *substring* comes close. Confuse the two and I write something that looks right, passes the sample, and dies on the hidden tests. The contract is otherwise mild: one token on stdin, `0 <= |s| <= 2000`, output one integer; empty string is a palindrome so the answer is at least `0`, and any non-empty `s` is at least `1`.

With `n <= 2000`, anything `O(n^2)` — four million cell updates — clears the one-second limit with room to spare, so the algorithmic question is which `O(n^2)` I can actually defend, and the only real engineering question is memory.

Two routes are on the table. Expand-around-centers is the tempting one: `2n-1` centers, push two pointers outward while the sides match, track the longest run — twenty lines I've typed from memory a dozen times. The problem *sounds* like longest palindromic substring, and the trick is right there. The plainer route is an interval DP: a value per interval `s[i..j]`, grown from short intervals to long by looking at the endpoints. Before reaching for the trick on reflex, I want to try to break it, because if it survives it is the shorter solution.

It does not survive. Take `s = bbbab`, indices `0..4`: `b b b a b`. The true answer is `4` — delete the `a` at index 3, keep `b b b b` at indices `0,1,2,4`. Center expansion only ever grows a *contiguous* run; from the center `b` at index 1 it matches index 0 against index 2 and then stops, because index 3 is `a`. The longest palindromic *substring* here is `bbb`, length `3`, and a contiguous expander physically cannot step over the obstructing `a`. So it returns `3` where the answer is `4`. This is not bad luck, it computes a different quantity: longest contiguous palindrome. Patching it to "expand but skip mismatches" turns each mismatch into a branching choice — exponential, and no longer the clean trick. Expand-around-centers is out.

So the interval DP. Let `dp[i][j]` be the length of the longest palindromic subsequence of `s[i..j]`, `i <= j`, and reason from the outer pair, since a palindrome is governed by its endpoints:

- If `s[i] == s[j]`, use both endpoints as the outer pair wrapped around the best palindrome of the strict interior: `dp[i][j] = dp[i+1][j-1] + 2`. Using both equal endpoints is never worse, since wrapping a matched pair around an interior palindrome stays a palindrome and adds 2.
- If `s[i] != s[j]`, the endpoints cannot both be an outer pair, so the best palindrome drops at least one: `dp[i][j] = max(dp[i+1][j], dp[i][j-1])`.

Base cases: `dp[i][i] = 1`, and an empty interval (`i > j`) is `0` — which is exactly what makes the equal branch correct at length 2, where the interior is empty and `dp[i][i+1] = 0 + 2 = 2` for equal characters. The answer is `dp[0][n-1]`, and `0` for the empty string. The top level of `bbbab` shows the subsequence power directly: `s[0]==s[4]` (`b==b`) gives `dp[0][4] = dp[1][3] + 2`, and `dp[1][3]` is the `bb` at indices 1,2, so `2 + 2 = 4` — the equal endpoints wrap around the interior and step straight over the `a` that center expansion choked on.

Memory next. A full `2000 x 2000` `int` table is `16` MB, comfortably under 256 MB, so the naive version would pass. But the recurrence never reaches back more than two interval-lengths: filling length `L` reads length `L-1` (the unequal branch's `dp[i+1][j]` and `dp[i][j-1]`) and length `L-2` (the equal branch's interior `dp[i+1][j-1]`). Three rolling rows indexed by the left endpoint `i` suffice — `cur` for length `L`, `prev` for `L-1`, `prev2` for `L-2` — giving `O(n)` memory. Writing the index translation out so I do not fumble it: `cur[i]` is `dp[i][i+L-1]`; the equal branch's interior `dp[i+1][i+L-2]` is a length-`(L-2)` interval starting at `i+1`, i.e. `prev2[i+1]`; the unequal branch's `dp[i+1][j]` and `dp[i][j-1]` are both length-`(L-1)`, i.e. `prev[i+1]` and `prev[i]`.

The rolling shift is where this kind of code actually breaks, and it did. At the end of each `L` I have to demote the rows, and my first version wrote `prev = cur` before `prev2 = prev` — which overwrites `prev` and leaves both `prev2` and `prev` holding the current row's values, collapsing two distinct lengths onto one. What makes this insidious is that palindrome-dense strings produce *lucky agreements*: adjacent-length dp values frequently coincide, so `aba`, `cbbc`, even `cbbbc` all still returned the right answer under the broken shift, and hand-tracing kept ratifying a bug. I stopped hunting for a witness by eye and leaned on the differential harness, which flagged a mismatch on a random string on its first real run. The fix is to demote oldest-out-first: `prev2 = prev` (the old length-`(L-1)` row becomes length-`(L-2)`) *then* `prev = cur`.

I did not want to trust hand-traces that had already fooled me, so the oracle rests on a different principle than the interval DP: the longest palindromic subsequence of `s` equals `LCS(s, reverse(s))`, computed with the standard `O(n^2)` LCS table — no shared structure with the endpoint recurrence, so agreement between them is real evidence. For `|s| <= 18` I added a third check, enumerating every subsequence by bitmask and asserting the longest palindrome among them matches the LCS value, triangulating three independent methods on small inputs. Across buckets — empty and length-1 edges, palindrome-dense 2-letter strings, small strings over alphabets of size 1..6, constructed even/odd palindromes, medium random strings, and one character repeated to the limit — I ran 1400 cases with zero mismatches. A second pass with its own bitmask enumeration oracle differential-tested several hundred more cases plus the large corners and agreed on every one.

The corners resolve cleanly. `|s| = 0` (no token): `cin >> s` fails, print `0`. `|s| = 1`: fill the length-1 row, early-return `1` before entering the `L`-loop. `|s| = 2` equal (`aa`): empty interior, `0 + 2 = 2`; unequal (`ab`): `max(1,1) = 1`. All distinct (`abcde`): every off-diagonal cell takes the unequal branch and stays at `1`. One character repeated `n` times: every interval is a palindrome, `dp[0][n-1] = n`, verified at `n = 2000`. Even/odd palindromes climb by 2 per layer to the full length. The answer is bounded by `2000`, so `int` is safe — no 64-bit arithmetic. Worst case `n = 2000` runs in about 10 ms with three `int` rows of a few KB.

That is what I ship: the plain, provable interval DP rolled down to three rows, not the center trick I broke on `bbbab`. The full module is in the answer.
