**The trap this problem is built around.** This is a maximum-drawdown / worst-decline computation: over `n` days a reservoir level walks by prefix sums `P[i] = d[0] + ... + d[i]` from a start level `P[-1] = 0`, and I report the largest drop `P[i] - P[j]` over `-1 <= i <= j <= n-1`. Two features of the contract dictate everything before I pick an algorithm. First, `i` ranges down to `-1`: the *starting* level `0` is an eligible peak, which flips the intuition from the maximum-subarray family — an all-negative season does not answer `0`, it answers the full fall from the start. Second, the scale: `n <= 2*10^5` and `|d[i]| <= 10^9`, so a level reaches `2*10^5 * 10^9 = 2*10^14` and a decline can be about that large — two orders past the 32-bit ceiling of `~2.1*10^9`. Every accumulator, the running peak, and the answer must be `long long`; an `int` here passes every small sample and silently wraps on the hidden large tests.

**Why the start level is load-bearing.** Take a single negative day, `d = [-7]`: the levels are `P[-1] = 0` and `P[0] = -7`, and the only nontrivial pair `(i = -1, j = 0)` gives `0 - (-7) = 7`. Code that ranges the peak only over the actual days — never over index `-1` — concludes "the level never rose, answer `0`" and is wrong. The peak has to carry the start level from the outset.

**The sweep.** All-pairs `P[i] - P[j]` is `O(n^2)` (`~4*10^10` operations at the limit) — a reference oracle only. For a fixed later `j`, `P[i] - P[j]` is largest at the highest `P[i]` with `i <= j`, so one left-to-right sweep carrying `peak = max(P[-1 .. current])` and the current level `prefix` gives, at each day, the best decline `peak - prefix`. For any `(i, j)` with `i <= j`, `P[i] - P[j] <= peak_{<=j} - P[j]` since `peak_{<=j} >= P[i]`, with equality at the maximizing `i`, so taking the max over `j` equals the all-pairs maximum exactly. `O(n)` time, `O(1)` space.

The start level enters by initializing `peak = 0` (that is `P[-1]`) and `prefix = 0` *before* the loop, so every day's decline already sees index `-1` as a candidate peak; the `i = j` drop of `0` is covered by starting `answer = 0`. I measure `peak - prefix` *before* folding today's level into `peak`, which keeps `peak` the best level strictly earlier than the day being measured. Folding first would only add today's own `P[i] - P[i] = 0`, so either order is correct — I keep measure-then-update because it makes the eligibility invariant one clean sentence.

**The base-case pitfall, concretely.** The seductive wrong move seeds the peak from the first day's level rather than before the loop:

```
if (i == 0) peak = prefix;          // "the first level is the first peak"  <-- drops P[-1]
else        peak = max(peak, prefix);
```

This throws away the start level `0`. On `[-7]` it sets `peak = -7` and reports `0` instead of `7`; on all-negative `[-2, -3, -1, -4]` — the level falling monotonically from `0` to `-10`, true answer `10` — it seeds `peak = -2` and reports `8`, undercounting by the start level's `0 - (-2) = 2` on every such input. Seeding `peak = 0` before the loop restores index `-1`, and both come out right: `[-7] -> 7`, `[-2, -3, -1, -4] -> 10`.

**Corners and overflow.** `n = 0`: the loop never runs, `answer = 0` (and `if (!(cin >> n)) return 0;` guards truly empty input). Single positive `[4]` -> `0`; single negative `[-7]` -> `7`; all non-negative or all zeros -> `0` via the floor; all negative -> the peak stays pinned at the start `0`, so the decline is the total fall. Running `n = 2*10^5` all `-10^9`, the level dives to `-2*10^14` and the program prints `200000000000000` — no truncation, where an `int` would have wrapped. I also ran the sweep against an independent `O(n^2)` all-pairs brute force (which builds the level list including the start `0` and maximizes over every `i <= j`) on a few hundred random small cases biased toward negatives, zeros, and all-negative/empty inputs: zero mismatches, a genuinely different method agreeing.

That fixes the whole program: seed `prefix = peak = answer = 0`, then per day accumulate `prefix += d`, take `answer = max(answer, peak - prefix)`, and update `peak = max(peak, prefix)` — the complete `long long` C++ file is the frozen answer.
