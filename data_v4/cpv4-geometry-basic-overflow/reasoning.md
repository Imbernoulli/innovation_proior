**The size of `n` decides the algorithm; the size of the coordinates decides everything else.** With `n <= 500` there are at most `500^3 / 6 ≈ 2.1 * 10^7` triples, so an `O(n^3)` scan over every triangle is comfortably inside a 2-second limit and is the obviously-correct choice — I do not reach for a convex-hull search, which would only add collinear-hull edge cases for a speedup I do not need at this `n`. The real constraint is `|x|, |y| <= 10^9`. A coordinate difference reaches `2 * 10^9`; a single product `(Bx-Ax)*(Cy-Ay)` reaches `4 * 10^18`; the cross product is a difference of two of those, so it can reach about `8 * 10^18` in magnitude. That fits a `long long` (max `≈ 9.22 * 10^18`) but is roughly four billion times too large for a 32-bit `int` (max `≈ 2.1 * 10^9`). An `int` overflow here does not crash — it silently returns a wrong number — so the discipline is fixed from the start: every coordinate, every difference, every product, and the running best must be 64-bit.

**Reporting twice the area keeps the answer an exact integer.** Twice the signed area of triangle `ABC` is the 2-D cross product `cross = (Bx-Ax)*(Cy-Ay) - (Cx-Ax)*(By-Ay)`, and twice the *area* is `|cross|` — an integer for integer inputs, no division, no floating point. Reporting the area itself would give a half-integer near `4 * 10^18` and tempt a `double`, whose 53-bit mantissa is exact only to about `9 * 10^15` and cannot represent it. So the contract works in my favour: I only have to hold `|cross|` in the right integer type. And since only the magnitude matters, the order of the three points inside a triple is irrelevant, so iterating `i < j < k` once per triple is enough.

**Why the type, not the formula, is the trap.** The seductive mistake is that the coordinate *differences* `2 * 10^9` do fit a 32-bit `int` (`2000000000 < 2147483647`, and `-2 * 10^9` clears the negative bound), so the expression looks safe — but the *product* of two such differences is not. Take `A=(0,0)`, `B=(10^9,0)`, `C=(0,10^9)`, twice-area exactly `10^18`: the product `10^9 * 10^9` computed in 32-bit wraps modulo `2^32` to `-1486618624` — wrong magnitude, and because the wrap crosses the sign bit, wrong sign — while `long long` returns `1000000000000000000`. The formula is exact; only the type is at fault, so coordinates, differences, product, `cross`, and `best` are all `long long`, after which `8 * 10^18` still has headroom.

**One more type trap on the way out: `abs`.** Plain `abs` / `std::abs(int)` takes and returns `int`; called on a `long long cross` it can bind to the `int` overload and truncate a correct 64-bit value back to 32 bits — re-introducing the overflow *downstream* of arithmetic I just made safe. That is nastier than the first bug because it only bites when `cross` exceeds `int` range, i.e. exactly on the large hidden tests and never on small samples. I use `llabs`, unambiguously `long long -> long long`, so `llabs(8 * 10^18)` stays `8 * 10^18`.

**The edge cases fall out of the loop plus `best = 0`.** For `n < 3` no triple `i < j < k` ever runs and `best` stays `0`. All-collinear or coincident inputs give `cross = 0` on every triple — a collapsed edge vector or a zero determinant — so `best` stays `0`; the initialization already encodes "no positive-area triangle exists," and duplicates need no special case since the problem allows them and the cross product zeroes them out. The one case that actually exercises the type is the extreme: the four corners of a `±10^9` square, where the largest triangle is half the square with twice-area `4 * 10^18` — the concrete value an `int` can never hold. Parsing is `cin >>`, which skips arbitrary whitespace, so line-broken or space-separated coordinates both read.

**Cross-checking against an independent brute.** To test the code and not just my arithmetic, I run the `long long` solution against a Python brute that computes twice the area by the shoelace expansion `ax*(by-cy) + bx*(cy-ay) + cx*(ay-by)` in arbitrary-precision integers — a different arrangement of the same cross product, so agreement is real evidence rather than an echo of my own formula, and it can never overflow by construction. A generator sweeps four regimes: tiny `[-5,5]` dense with collinear and duplicate triples, small `[-50,50]`, full magnitude `[±10^9]` (the overflow regime), and corners pinned to `±10^9`. They agree on every case. One earlier run *looked* like it had mismatches, but tracing them showed my harness was reusing one shared temp file across iterations so outputs interleaved; isolating each case to its own file made them vanish — the arithmetic was never wrong, the test rig was.

**What I ship** is the single `O(n^3)` file: read the points into `long long` vectors, scan `i < j < k`, form `cross = abx*acy - acx*aby` entirely in 64-bit, take `llabs`, and keep the maximum. The full module is in the answer.
