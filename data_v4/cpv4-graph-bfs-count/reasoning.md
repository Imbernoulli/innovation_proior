The word that decides this whole problem is *distinct*, and the contract pins its meaning to the **sequence of hops** taken: two stations joined by two parallel connections give two different itineraries. So parallel edges stay in, each contributing separately, and the count is fundamentally a sum over edges rather than over predecessor vertices — no vertex-path deduplication anywhere. Three corners are stated and I fix them before touching an algorithm: unreachable `t` prints `0`; `s == t` prints `1` (the empty zero-hop itinerary is the unique answer); self-loops and parallel edges may appear, with a self-loop never shortening a route. Scale is `n` up to `2*10^5`, `m` up to `4*10^5` — sparse, so an adjacency-list BFS is `O(n+m)`, comfortably inside 1 second. The count itself is the hazard: a layered graph of width `W` and depth `k` has `W^(k-1)` shortest routes, past 64-bit almost immediately, so every running count lives modulo `1e9+7` and the reduction must happen *during* accumulation, not at the end, or the intermediate sums overflow.

The distance layer is uncontroversial — plain BFS from `s` gives `dist[v]`, since BFS finalizes vertices in nondecreasing distance and the first arrival at a vertex fixes its minimum hop count. The count layer, `ways[v]`, rides on top. A hop `u -> v` extends a shortest route to `v` exactly when `u` is one layer closer, `dist[u] + 1 == dist[v]`, and then every shortest route to `v` is a shortest route to some such `u` extended by that hop:

  `ways[v] = sum over hops (u,v) with dist[u] + 1 == dist[v] of ways[u]`,  with `ways[s] = 1`.

Parallel hops simply appear multiple times in the sum, which is exactly right under the hop-sequence definition. Everything hinges on getting one predicate exact: "previous layer" is `dist[u] + 1 == dist[v]`, and I have to positively *exclude* "same layer" (`dist[u] == dist[v]`) and "behind" (`dist[u] >= dist[v]`). The same-layer edge is the heart of the difficulty — the sample plants `2-3` and `4-5`, both joining stations at equal distance from `s`, and they must contribute nothing.

Folding the count into the BFS is tempting to write loosely: on first discovery of `w`, seed `ways[w] = ways[u]`; otherwise, in an `else`, add `ways[u]` into `ways[w]`. That blanket `else` is wrong here, for two concrete reasons that are exactly the traps this problem is built from. Because every undirected edge is stored twice, dequeuing the deeper endpoint sees the shallower one again: when `2` is processed it looks back at the source `1` and would add `ways[2]` into `ways[1]`, counting a route backwards. And the decoy edges `2-3`, `4-5` join same-layer stations, so `2` would add into `ways[3]` across an edge that lies on no shortest route. Both are fixed by gating the addition on the exact previous-layer test:

```
if (dist[w] == INF) {            // first arrival
    dist[w] = dist[u] + 1;
    ways[w] = ways[u];
    q.push(w);
} else if (dist[w] == dist[u] + 1) {
    ways[w] = (ways[w] + ways[u]) % MOD;   // genuine previous-layer contributor
}
// dist[w] == dist[u] (same layer) or dist[w] < dist[u] (behind): nothing
```

Is this addition ordering-safe — is `ways[u]` final when I read it? BFS dequeues in nondecreasing distance, and every contribution into `ways[u]` comes from a vertex on layer `dist[u]-1`, all dequeued strictly before any layer-`dist[u]` vertex. So by the time `u` is dequeued, its count has received all its contributors and is final; the forward push into a deeper `w` therefore reads a finished value. The queue serializing by layer is precisely what makes the single pass correct.

Tracing the sample fixes the distances and shows the decoys dying. From `s = 1`: layer 0 `{1}`, layer 1 `{2,3}`, layer 2 `{4,5}`, layer 3 `{6}`, so `dist[6] = 3 = dist[t]`. Now the counts. `ways[1] = 1`. Dequeue `1`: discover `2:(1,1)`, `3:(1,1)`. Dequeue `2`, neighbours `1,4,3`: `1` is behind (`dist 0 != 2`), skip; `4` first-seen -> `(2,1)`; `3` is same-layer (`dist 1 != 2`), skip — the decoy `2-3` contributes nothing. Dequeue `3`, neighbours `1,4,5,2`: `4` is previous-layer (`dist[4]=2 == dist[3]+1`) -> `ways[4] = 1 + 1 = 2`; `5` first-seen -> `(2,1)`; `1` and `2` skip. Dequeue `4`, neighbours `2,3,6,5`: `6` first-seen -> `(3,2)`; `5` has `dist 2 != 3`, skip (decoy `4-5`). Dequeue `5`, neighbour `6` is previous-layer -> `ways[6] = 2 + 1 = 3`. Dequeue `6`: both neighbours behind. Final `ways[6] = 3`, matching `1-2-4-6`, `1-3-4-6`, `1-3-5-6`, with both decoy edges excluded exactly as required.

The natural alternative is two passes — BFS for `dist[]`, then a loop over edges adding `ways[u]` into `ways[v]` wherever `dist[u]+1 == dist[v]`. The predicate is identical, so it looks equivalent, but the recurrence only holds if `ways[u]` is *final* when read, and finality is guaranteed by *layer* order, not by whatever order the loop visits vertices. Iterate `v` from `1` to `n` in index order and it breaks the moment a contributor of `v` carries a larger index. On `1-7, 1-8, 7-2, 8-2, 2-3` with `s=1, t=3` (true answer `2`): processing `v=2` pushes `ways[2]` forward to `3` before `7` and `8` — its layer-1 contributors, both higher-indexed — have filled `ways[2]`, so `ways[3]` gets `+= 0` and the run prints `0`. The off-by-one in thinking is "the predicate is right, so the loop is right", but the predicate governs *which* edges contribute, not *when* the values are ready. The cure is to iterate in nondecreasing-`dist` order, which is exactly what the one-pass BFS hands me for free — so I keep the one pass and drop the index-order two-pass.

The corners fall out of the same predicate. `s == t`: `ways[s] = 1` is set pre-loop, and a neighbour `u` tests `dist[s] == dist[u] + 1`, i.e. `0 == 2`, false, so it is never overwritten -> output `1`. Unreachable `t`: `dist[t]` stays `INF`, output `0` via an explicit `dist[t] == INF` guard (the intent made unmistakable rather than relying on `ways[t]` being `0`). Single isolated station (`s = t = 1`, no edges): loop does nothing -> `1`. Self-loops: `dist[u] == dist[u] + 1` is always false, so they are harmless, but I still drop them at read time to keep adjacency clean. Parallel edges: kept — a doubled `1-2` with `s=1,t=2` seeds `ways[2] = 1` then adds `1` for the second hop -> `2`, the two distinct one-hop itineraries. Overflow: every addition is reduced `% MOD`, and the pre-reduction sum `ways[w] + ways[u] < 2*MOD < 2.2e9` fits in `long long`, so the reduction genuinely happens during accumulation and never after a silent 64-bit wrap — the distinction that would bite on the large layered tests.
