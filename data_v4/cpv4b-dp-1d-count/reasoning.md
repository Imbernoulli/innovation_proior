The object here is the count of distinct subsequences of a length-`n` tone log, taken by *content*: two index selections that render the same tone-id string collapse to one reel, and I want the number of non-empty distinct renderings mod `p = 1000000007`. Two facts about the constraints steer the solution before I write a line. Ids run up to `10^9`, so any "where did this tone last appear" bookkeeping cannot be an array indexed by tone id — it has to be a hash map. And the count itself can reach `2^n - 1` with `n` up to `2*10^5`, which is why everything runs mod `p`; because the recurrence I am about to build *subtracts*, I have to keep every intermediate residue non-negative, and C++'s truncating `%` will not do that for me on its own. Empty input is `n = 0`, answer `0`.

Brute force is only an oracle. Enumerate all `2^n` index masks, render each to its tone-id sequence, drop the empty one, and count the distinct renderings in a set. That is the definition transcribed verbatim, so it is unbeatable for correctness — and hopeless past `n ~ 25`. I keep it purely to validate the real algorithm on tiny inputs.

The real algorithm is a left-to-right counting DP. Let `dp` be the number of distinct subsequences of the prefix seen so far, *including the empty one*. Including the empty subsequence is what makes the doubling clean; I subtract it back out exactly once at the end. Start `dp = 1`, the empty prefix having only the empty subsequence. Appending a tone `x` sends each distinct subsequence `s` of the old prefix to two candidates, `s` and `s` with `x` appended, so the first guess is `dp = 2*dp`. That is exact only when `x` is new — then no two of those candidates coincide. If `x` occurred before, at its latest earlier position `j`, then every subsequence "ending in this `x`" — that is, `s'` followed by `x`, for `s'` ranging over the distinct subsequences of `t[0..j-1]` — was already produced back when I appended that earlier `x`. The number of those already-produced, now-duplicate subsequences is exactly the `dp` value that held *immediately before that earlier `x` was appended*. So the recurrence is:

- `x` new: `dp = 2*dp`.
- `x` repeat: `dp = 2*dp - last[x]`, where `last[x]` is the `dp` from just before the previous occurrence of `x`.

and after each step I set `last[x] = old`, the `dp` from just before *this* append, so the next reuse subtracts the right prefix count. The final answer is `dp - 1`.

Tracing the given example `t = [1,2,1]` grounds the correction. `dp` starts at `1`; appending `1` gives `2` and records `last[1]=1`; appending `2` gives `4` and records `last[2]=2`; the second `1` is a repeat, so `2*4 - last[1] = 8 - 1 = 7`. The distinct subsequences including the empty one are the six reels `1, 2, (1,2), (2,1), (1,1), (1,2,1)` plus `{}`, so `7` is right and `7 - 1 = 6`.

Two transcription hazards fire only at scale, invisible on any hand input. The first is the modular subtraction. After `dp = (2*old) % MOD`, `dp` lives in `[0, p)`, and the subtracted `last[x]` is itself a stored `old`, also in `[0, p)`, so `dp - last[x]` can be as low as `-(p-1)`. On tiny hand inputs `dp` stays a small positive integer and the difference is never negative — which means no small trace can expose this; it fires only once the running count has wrapped past `p`, i.e. only on large inputs, as a silent wrong answer. Since C++ `%` truncates toward zero, a negative dividend yields a negative residue that then propagates into later doublings and the printed answer. The fix is the standard `+MOD` before reducing: `dp = (dp - last[x] + MOD) % MOD`, and likewise `(dp - 1 + MOD) % MOD` for the final subtraction of the empty subsequence.

The second hazard is *what* goes into `last[x]` and *when*. The subtracted term must be the `dp` from before the previous `x`, which means storing `old` (not the post-doubling `dp`) and overwriting it on *every* occurrence, not just the first. Both mistakes are easy, and both are visible on `[1,1,1]`, whose true answer is `3` (`1, (1,1), (1,1,1)`, so `dp = 4` counting the empty one). Storing the post-doubling `dp` makes the subtracted term too large — it over-subtracts and the count collapses to `1`. Storing `old` but freezing it at the first occurrence makes the third `1` subtract a stale `1` where it should subtract the dp-before-the-second-`1`, which is `2` — it under-subtracts by one and double-counts a reel, inflating the answer to `4`. Storing `old` and overwriting unconditionally lands on `3`. So the unconditional `last[x] = old` is load-bearing, and that is what the loop does.

That leaves the corner inputs. With a `0` token present, `cin >> n` reads `n = 0`, the loop never runs, `dp` stays `1`, and the answer is `(1 - 1 + MOD) % MOD = 0`. The one wrinkle is genuinely empty stdin: `if (!(cin >> n)) return 0;` prints nothing rather than a literal `0` — acceptable here because any test that wants the value `0` supplies the token `0`, and for every input containing a number the answer is computed. A single tone gives `dp = 2`, answer `1`. An all-identical log climbs `dp = 1,2,3,...,n+1` (each repeat step is `2*old - old`), so the answer is `n`, matching the `n` distinct runs `7, 77, ...`. An all-distinct log fires no subtraction, `dp = 2^n`, answer `2^n - 1`. Big ids force the `unordered_map<long long,long long>`; I reserve `2n` buckets to hold the expected `O(n)`, and at `n = 2*10^5` the sweep is ~24 ms, comfortably inside 1 s.

Checked against the enumeration oracle over 400 random small cases — heavy on tiny alphabets, where double-counting bites hardest — the DP agrees everywhere with zero mismatches. That is the whole `O(n)` submission.
