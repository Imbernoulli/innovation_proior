The scale sets the algorithm class: `n, m <= 2*10^5`, and every value is a star index bounded by `n`, so nothing overflows an `int` — no arithmetic-width worry on the data. The entire difficulty is **time**, and underneath it, the **boundary** of a range union. Operation `(l, r)` declares stars `l, l+1, ..., r` mutually connected; bridges are permanent, operations overlap or repeat or collapse to a single point, and at the end I count clusters (an untouched star is its own singleton). Connectivity means disjoint-set union is the vehicle. The sharp question is: what does `[l, r]` actually link, and how do I do it without quadratic blowup?

Concretely, range `[l, r]` makes stars `l..r` one cluster, and the minimal set of adjacency links that achieves this is the chain `l-(l+1), (l+1)-(l+2), ..., (r-1)-r`. Those links are indexed by their left endpoint `i`, running `i = l, l+1, ..., r-1` — that is `r - l` links. So a single-star range `l == r` introduces `r - l = 0` links: it connects nothing new, star `r` is touched but its cluster is unchanged. `[2,4]` introduces `2-3` and `3-4`: three stars, one cluster. Two ranges that touch at a shared endpoint, `[1,3]` then `[3,5]`, share star `3` and fuse to `{1,2,3,4,5}`; two ranges with a one-index gap, `[1,3]` then `[4,6]`, share no star and there is no link `3-4`, so they stay `{1,2,3}` and `{4,5,6}`. Touch-merges, gap-does-not: the inclusive/exclusive distinction is the whole game. The invariant I hold onto is that the largest left endpoint ever passed to a union is `r - 1`, never `r`; a link with left endpoint `r` reaches star `r+1`, outside the range.

**Two routes.** Plain DSU links every adjacent pair in every range: `for i in l..r-1: unite(i, i+1)`. It is transparently correct — it literally builds the chain — but costs `O(r - l)` per operation, so `O(sum of range lengths)`. With `m` ranges each as long as `n`, that is `O(n*m) = 4*10^{10}` union calls, orders of magnitude over a 1-second limit. It survives only as a reference oracle on small inputs. The wasted work is re-touching links that already exist: a second `[1,100]` after a first re-calls `unite` 99 times for nothing. So I want each link `i -> i+1` created **at most once over the whole run**. Keep a second pointer `nxt[i]` = the smallest index `>= i` whose outgoing link has not yet been built; when I build link `i -> i+1` I consume `i` by setting `nxt[i] = i + 1`, and a later walk hitting `i` skips straight past. Each index is consumed at most once, so total union calls `<= n - 1`, and with path compression on both the DSU and the skip pointer the whole thing is near-linear, `O((n + m) * alpha)`. This is the submission — provided the loop boundary is exact.

**The walk.** `par[]` is ordinary union-find over the stars; `nxt[]` is the skip pointer, where `nxt[i] == i` means index `i` is still active (its outgoing link unbuilt). To process `(l, r)`: jump to the first active index at or above `l` via `i = fnext(l)`; while `i` is below `r`, build `unite(i, i+1)`, consume `i` with `nxt[i] = i + 1`, and advance `i = fnext(i + 1)`; stop once `i` reaches `r`.

Everything rides on the loop guard. The legal left endpoints are `l .. r-1`, so the cursor must be allowed to reach `r - 1` and then stop — `while (i < r)`, strictly less. The tempting `i <= r` is exactly the off-by-one this problem is built around: at `i = r` it calls `unite(r, r+1)`, the illegal link that drags in star `r+1`. To see the damage, take `n = 3`, op `(1, 2)`; the answer should be `{1,2}` and `{3}`, i.e. `2`. With `i <= r` the walk does `unite(1,2)`, then at `i = 2 = r` still passes the guard and does `unite(2,3)`, collapsing all three stars to one cluster and printing `1`. The strict guard `i < r` stops after `unite(1,2)`, leaving `{1,2}` and `{3}` — correct. The same guard makes the single-point range free: for `(3,3)`, `fnext(3) = 3` and `3 < 3` is false immediately, so zero links are built and no phantom `unite(3,4)` appears.

The subtler hazard is the skip pointer itself: when I advance `i = fnext(i + 1)`, it must not overshoot `r` and miss a link, nor land somewhere that builds a link it shouldn't. The case that exercises it is a later `l` landing *inside* an already-consumed run — `n = 6`, ops `(1, 4)` then `(2, 6)`. After `(1,4)`: links `1-2, 2-3, 3-4` built, `nxt = [_,2,3,4,4,5,6]`, cluster `{1,2,3,4}`. Now `(2, 6)`: `fnext(2)` follows `nxt[2] = 3 -> nxt[3] = 4 -> nxt[4] = 4`, landing on `4` — it jumps over the already-built `2-3, 3-4` links straight to the first active index, and lands not past it but exactly on it. Then `4 < 6`: `unite(4,5)`, `nxt[4] = 5`, `i = 5`; `5 < 6`: `unite(5,6)`, `nxt[5] = 6`, `i = 6`; `6 < 6` stops. Cluster `{1..6}`, count `1` — correct, and no redundant relinking. So `fnext` lands on the first active left endpoint and the `< r` guard still bounds it.

Counting is then the number of DSU roots among `1..n`, `#{ v : find(v) == v }`; untouched stars are their own roots and count as singletons. The given sample checks out end to end: `n = 8`, ops `(2,4), (4,6), (7,7)` — `(4,6)` shares star `4` with `{2,3,4}` and fuses into `{2,3,4,5,6}`, `(7,7)` builds nothing, leaving `{1}, {2,3,4,5,6}, {7}, {8}`, four clusters.

**Edge cases.** `m = 0`: the operation loop never runs, every star is its own root, count `= n`; the `if (!(cin >> n >> m)) return 0;` guard reads a present `n` with `m = 0` fine. `n = 1`: the only possible range `(1,1)` has zero links, count is always `1`. Fully overlapping ranges are the reason the skip pointer exists — `(1,n)` applied `m` times builds all `n-1` links on the first pass, and every later pass does `fnext(1)` which resolves straight to `n` (all lower indices consumed), so `n < n` fails and it costs `O(alpha)`, not `O(n)`; total `O(n + m*alpha)` instead of `O(n*m)`. On array bounds, the only transient out-of-range access is `nxt[i+1]` / `par[i+1]`, so I size both `n + 2` and initialise index `n+1` as a self-root; and since the `< r` guard with `r <= n` stops the cursor at `n - 1`, `i + 1 <= n` always, so the extra slot is never actually needed for a real link. `cin >>` skips whitespace, so the input's line structure is irrelevant to parsing, and I print exactly one integer and a newline.

Each adjacency link is built at most once (consumed by `nxt[i] = i + 1` and skipped thereafter), so `unite` runs `<= n - 1` times; each `fnext` and `find` is amortised `O(alpha)`, reading is `O(m)`. Total `O((n + m) * alpha(n))` time, `O(n)` memory — a couple of hundredths of a second on the `n = m = 2*10^5` adversarial input. The full program is in the answer: a DSU over stars `1..n` carrying the extra `nxt[]` skip pointer, the `while (i < r)` walk per operation, and a final pass counting roots.
