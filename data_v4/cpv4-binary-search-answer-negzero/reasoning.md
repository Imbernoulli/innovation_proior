The readings `a[0..n-1]` are integers in `[-10^9, 10^9]` — negative and zero are legal, since the sensor sits on an arbitrary baseline — and I must pick a contiguous block of length **at least** `k`, score it by the minimum reading inside, and maximize that block-minimum. If `n < k` no such block exists and I print `INFEASIBLE`. "Maximize the minimum" is the textbook cue for a binary search on the answer, so the algorithm is not really in question. What makes *this* instance dangerous lives entirely in the sign domain: the optimal floor can be negative, can be zero, the whole array can be all-negative, and `n` can be `0`. Any reflex that assumes "the answer is at least 0" or "start the search at 0" is wrong here, and wrong in the worst way — silent on every all-positive test, detonating only on the negative ones.

Scale first: `n, k <= 2*10^5`, `|a[i]| <= 10^9`. The answer is some single reading's value, so it fits in 32 bits, but I carry everything in `long long` anyway — the midpoint arithmetic and the `n` vs `k` comparison are exactly the places a stray `int` or signed/unsigned slip would bite. So `n` is an `int` (it sizes the vector), `k` is a `long long`, the values are `long long`, and `(long long)n < k` together with any `run >= k` test are unambiguously signed 64-bit.

Two routes to the objective. Binary search on a floor `x`: does some run of `>= k` consecutive positions all satisfy `a[i] >= x`? A single linear scan with a run counter, reset on every reading below `x`, answers it. Raising `x` can only break runs, never create them, so feasibility is non-increasing in `x` and the largest feasible `x` is the answer — `O(n log(range))`, range about `2*10^9`, so ~31 scans. The alternative is a monotonic-deque sliding-window minimum at `O(n)`, but the deque bookkeeping is fiddly and the asymptotic win is irrelevant inside a one-second limit, so I take the binary search for its trivially-checkable predicate.

One equivalence I need before trusting that predicate: "some block of length `>= k` has minimum `>= x`" is the same statement as "some run of `>= k` consecutive `>= x` exists," because any all-`>= x` block of length `> k` contains an all-`>= x` block of length exactly `k`. So the predicate *is* the objective, not an approximation of it.

Now the bounds, where the signs bite. The answer is the minimum of some block, so it equals some `a[i]` and lies in `[min(a), max(a)]`. `x = min(a)` is always feasible: past the infeasible guard the whole-array window has length `n >= k` and every element is `>= min(a)`. `x = max(a)+1` is never feasible: no element reaches it. So `lo = min(a)`, `hi = max(a)`, and I search for the largest feasible `x` between them. What I must *not* do is anchor `lo` at `0`: on an all-negative array that searches `[0, max(a)]` with `max(a) < 0`, an inverted interval that returns garbage. The data-driven bounds sidestep it.

Checking against the sample `a = [2, -1, 3, 3, 5, -4, 3]`, `k = 3`: `lo = -4`, `hi = 5`. feasible(3) holds via indices 2..4 = `[3, 3, 5]`; feasible(4) fails because the only reading `>= 4` is the isolated `5` at index 4, run length 1. The search lands on `3`, matching the stated answer.

Two reflex implementations go wrong here in ways this problem's signs specifically invite. The first is the anchoring I already flagged: `lo = 0; hi = 0; for v: hi = max(hi, v);`. Trace it on `n = 2, k = 1, a = [-5, -3]`, whose true answer is `-3` (the best single element). `hi` stays `0` — both `max(0, -5)` and `max(0, -3)` give `0` — so the loop `while (lo < hi)` never runs and it prints `0`, a floor no element even reaches. That all-negative-returns-`0` output is the exact fingerprint of the sign bug, and it is why the bounds are `lo = min(a)`, `hi = max(a)`.

The second defect is subtler and only surfaces once the bounds are fixed. Take the standard "move `lo` up" search with a floor midpoint `mid = (lo + hi) / 2` and branches `feasible(mid) ? lo = mid : hi = mid - 1`. Termination needs `lo < mid <= hi` whenever `lo < hi`, so the `lo = mid` branch strictly advances `lo`. On the negative interval `[-5, -3]` it happens to work: `(-5 + -3)/2` truncates toward zero to `-4`, then `(-4 + -3)/2` truncates to `-3` — but that is C++ truncation-toward-zero doing me a favor, not correctness. On the positive adjacent interval `[2, 3]`, `(2 + 3)/2 = 2 = lo`, `feasible(2)` is true (`lo` is always feasible), `lo` stays `2`, and the loop spins forever. So the floor midpoint is a latent hang whose sign-asymmetry is pure truncation accident. The sign-uniform fix is the upper midpoint `mid = lo + (hi - lo + 1) / 2`: since `hi - lo >= 1`, this gives `lo + 1 <= mid <= hi`, strictly advancing `lo` and strictly shrinking `hi` regardless of sign, and it is computed from the non-negative difference `hi - lo`, so truncation direction is irrelevant. On `[2, 3]` it yields `mid = 3` and terminates in one step; on `[-5, -3]` it steps `-4` then `-3` by construction rather than by luck.

The `n < k` guard goes first, before computing `min`/`max`. Since `k >= 1`, `n = 0` always trips it, so the empty array — where folding `min`/`max` over nothing would be undefined — is covered by the same guard; there is no separate empty-array branch to get wrong.

The corners then fall out. `k = 1`: feasible(x) is "some reading `>= x`," so the answer is `max(a)`, the element itself. `k = n`: the only block is the whole array, answer `min(a)`. All-negative `[-5, -1, -1, -9], k = 2`: `lo = -9`, `hi = -1`; feasible(-1) via the adjacent `-1, -1`, feasible(0) fails, answer `-1` — negative, not `0`, precisely what the anchoring bug got wrong. All-zero: `lo = hi = 0`, the loop skips and prints `0`. Single element `[-8], k = 1`: `1 < 1` is false, `lo = hi = -8`, prints `-8`.

As a final guard I cross-check the predicate and bounds against an independent brute force that enumerates every block of length `>= k`, takes each block's minimum directly, and maxes them — it shares no run or monotonicity logic with the search, so agreement is evidence rather than a tautology. I stress it on tiny cases in exactly the dangerous regimes where the anchoring and midpoint bugs would surface: all-negative, negatives-with-zeros, all-zeros, mixed signs, `n = 0`, `n < k`, `k = 1`, `k = n`.

The full module is in the answer: the `O(n log range)` binary search on the answer with data-driven `min`/`max` bounds, the sign-uniform upper midpoint, and the `n < k` guard in front.
