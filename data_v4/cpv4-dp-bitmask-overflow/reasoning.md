A relay with `n` legs and `n` runners, an `n x n` synergy table `s`, and I need the permutation of runners to legs that maximizes the summed synergy — the assignment problem on a complete bipartite graph. Two numbers decide everything before I pick an algorithm: `n <= 18` and `0 <= s[i][j] <= 10^9`. The first says `2^n = 262144` subsets, small. The second, against a sum of exactly `n` entries, says the optimum can reach `18 * 10^9 = 1.8 * 10^10` — almost ten times `INT_MAX ≈ 2.147 * 10^9`. So the answer, never mind the partial sums the DP will compare, does not fit in a 32-bit `int`. Every accumulator, DP cell, and comparison value here has to be 64-bit `long long`. That is the one landmine this table's value scale is built to detonate, and I fix the width before writing a line of logic.

Brute force over the `n!` assignments is trivially correct and makes a fine oracle for tiny `n`, but `18!` ≈ `6.4 * 10^15` is unrunnable — no time limit survives it. The structure to exploit instead: fill legs in index order (leg 0, then leg 1, ...). Once I've placed some runners, the future depends only on *which* runners remain free — not the order they were placed, and not the legs they took, because those legs are exactly `0 .. (placed-1)` by construction. So a subproblem is a *set* of used runners, a bitmask over `n` bits, and `popcount(mask)` equals both the number of legs already filled and the index of the next leg to fill. That collapses the `n!` orderings into `2^n` states. Time `O(2^n * n) ≈ 4.7 * 10^6`, space `2^n = 262144` cells — trivially inside 2 s / 256 MB.

Let `dp[mask]` = the best total synergy using exactly the runners in `mask`, assigned to legs `0 .. popcount(mask)-1`. Base case `dp[0] = 0`: no runners placed, no legs filled. With `leg = popcount(mask)` the next leg, I choose a free runner `i` (a bit not in `mask`), move to `mask | (1<<i)`, and add `s[i][leg]`:

- `dp[mask | (1<<i)] = max(dp[mask | (1<<i)], dp[mask] + s[i][leg])`.

The answer is `dp[(1<<n)-1]`. This is exact, not a heuristic. Every permutation is exactly one path `0 -> ... -> full` in this DAG: at step `k` the path adds the runner the permutation sends to leg `k`. Conversely every `n`-transition path from `0` to the full mask adds `n` distinct, previously-unused runners — a valid permutation. The DP maxes over all such paths, hence over all permutations, and the state suffices because, with legs filled in order, the remaining problem depends only on the free set and the mask-determined next leg. Iterating masks in increasing order is a valid topological order, since `mask | (1<<i) > mask` whenever bit `i` is newly set, so every predecessor is finalized before its successors.

The overflow is not hypothetical noise about the `n = 18` corner; it bites on the given sample. Trace that optimum (the `10^9` diagonal, `n = 3`): `dp[000]=0`; runner 0 on leg 0 -> `dp[001]=10^9`; runner 1 on leg 1 -> `dp[011]=2*10^9`, still just under `INT_MAX`; runner 2 on leg 2 -> `3*10^9`. That last addition is where an `int` table dies — `3*10^9 > 2147483647` wraps modulo `2^32` to `-1294967296`, and the `max` over the several wrapped paths reaching the full mask prints a nonsensical `-536870912` instead of `3000000000`. There is no careful way to stay inside `int` — the quantity simply doesn't fit. So `s`, `dp`, and `cand` are all `long long`, where the worst case `1.8 * 10^10` sits eight orders of magnitude below `LLONG_MAX ≈ 9.2 * 10^18`.

Two smaller decisions in the transition. For unreachable cells I mark `LLONG_MIN / 4` rather than `LLONG_MIN` and skip them with an `if (dp[mask] == NEG) continue;` guard: the `/4` leaves enough headroom that no addition or comparison against the sentinel can underflow, and the guard keeps a sentinel cell from ever winning a `max`. In this complete table every mask of popcount `< n` is actually reachable, so the guard never fires — but it makes the empty/sentinel corner safe rather than brittle. And the column charged to each runner is `leg = popcount(mask)`, the *next* leg — not the runner index and not a fixed column; getting that index wrong would silently return a non-optimal value while looking plausible.

The edge cases fall out. `n = 1`: the one assignment gives `dp[1] = s[0][0]`. All-zero table: every path sums to `0`, and since synergies are `>= 0` and the full mask is reachable, `dp[full] = 0` cleanly. Ties / non-unique argmax: the DP returns the optimal *value*, which is all the problem asks. The full mask always has popcount `n` and is reachable in a complete table, so `dp[(1<<n)-1]` is never the sentinel and no infeasible branch is needed. `cin >>` is whitespace-agnostic, so the table may span lines or spaces freely, and `if (!(cin >> n)) return 0;` guards truly empty input.

So the submission is this subset DP with every value 64-bit. The full self-contained C++17 module is the answer.
