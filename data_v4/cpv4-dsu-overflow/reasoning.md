The name of the game here is magnitude. The algorithm is textbook incremental connectivity — cables are added one at a time, never removed, and after each one I answer a connectivity-derived question, which is what disjoint-set-union is for — but every quantity I track grows quadratically in cluster size and is then summed over up to `2*10^5` steps, so before I write any DSU I need to know exactly how big the numbers get and pin every type to match. With `n = 2*10^5`, if all servers end up in one cluster the connected-pair count is `C(2*10^5, 2) = 200000*199999/2 = 19,999,900,000 ≈ 2*10^10`, already an order of magnitude past the signed 32-bit ceiling of `2,147,483,647`. The grand total sums up to `2*10^5` such per-cable numbers, so it can reach roughly `2*10^10 * 2*10^5 = 4*10^15`. Both the running count and the grand total must be `long long`; an `int` anywhere on that path is a silent wrong answer on the large hidden tests.

**The algorithm.** Rebuilding components from scratch after each cable — flood-fill the whole edge set, sum `C(size,2)` — is `O(m(n+m))`, about `4*10^10` operations at the top end, hopeless in a second (it survives only as a brute-force oracle, which I use at the end). Since edges are add-only and I keep being asked a connectivity quantity, DSU with union by size and path halving is the fit: near-`O(α(n))` per find, and it hands me the size of each cluster I merge.

**The incremental update.** I do not want to recompute `sum_k C(c_k,2)` each step. I track one running value `P`. Merging two distinct clusters of sizes `sA` and `sB` connects every server of one to every server of the other and touches nothing else, so `P += sA*sB` is the only change; algebraically `C(sA+sB,2) - C(sA,2) - C(sB,2) = sA*sB`, so `P` stays equal to `sum_k C(c_k,2)` exactly, and a redundant cable merges nothing and leaves `P` unchanged. On the given sample — cables `(1,2),(3,4),(2,3),(3,4),(1,5)` — this yields `1,2,6,6,10`: the third cable merges two size-2 clusters for `+4`, the fourth is redundant (`3` and `4` already share a cluster) and adds `0`, and the grand total is `1+2+6+6+10 = 25`, matching. That redundant fourth cable adding zero is the case I care about, because a missing guard would over-count exactly there.

**Where the overflow actually bites.** The trap is not where the eye first goes. I store the running count as `long long pairs`, but the increment is `pairs += sz[ru] * sz[rv]`, and if the size array is `int` then `sz[ru] * sz[rv]` is evaluated in 32-bit *before* the result is widened into the 64-bit accumulator. The `long long` on the left does not save the multiply on the right. Two clusters of size `50000` give a product `50000*50000 = 2,500,000,000`, which exceeds `2^31` and wraps to the negative `int` `2,500,000,000 - 2^32 = -1,794,967,296`; added to `pairs`, the count *drops* by about 1.79 billion — and a connected-pair count can never decrease as cables are added, so the answer is self-evidently corrupt. None of this shows on the sample or any small input, which is precisely the danger of the trap. I store `sz` as a `long long` array so every arithmetic touching a size is 64-bit by default; casting one operand (`(long long)sz[ru] * sz[rv]`) would also work, but widening the storage means I never have to audit *which* size expression might overflow. With `long long sz`, that merge computes `2,500,000,000` exactly and `pairs` climbs from `2*C(50000,2) = 2,499,950,000` to `C(100000,2) = 4,999,950,000` — monotone, matching the closed form.

**The guard, and self-loops.** A single `if (find_(u) != find_(v))` absorbs both redundancy cases: an edge inside an existing cluster (same root, skip, no `sz*sz` added), and a self-loop `u == v`, where `find_(u) == find_(v)` holds trivially so no special case is needed. `n = 1` falls straight out: the only legal cable is `(1,1)`, always redundant, every line `0`.

**Ordering and the corners.** The one ordering constraint that can silently corrupt the answer: read `sz[ru]` and `sz[rv]` for the product *before* the union writes `sz[ru] += sz[rv]`, otherwise I multiply post-merge sizes. In the code the `pairs += sz[ru]*sz[rv]` line sits above the union, so the sizes are pre-merge. For `m = 0` the loop never runs and I still print the grand total `0` unconditionally — one line, which is the `m+1`-line contract at `m=0`. At the top end `n = m = 2*10^5`, the peak per-cable count `~2*10^10`, the single product `~10^10`, and the grand total `~4*10^15` all sit comfortably inside signed 64-bit (`~9.2*10^18`). Output is `~2*10^5` lines of up to ~14 digits; printing per line through synced streams is slow, so I build one string and flush it once.

**Cross-check.** I run the DSU solution against a from-scratch flood-fill brute — rebuild adjacency from all edges so far, BFS each component, sum `C(size,2)`, a genuinely different method — over hundreds of random small cases with self-loops and redundant edges, plus the constructed corners (`m=0`, `n=1`, self-loop-only inputs) and a large `n=m=2*10^5` run to exercise the peak count and the multi-quadrillion total. They agree on every per-cable line and on the grand total. The full self-contained program is in the answer.
