The shortest latencies here are a plain positive-weight Dijkstra; the entire problem is the counting layer bolted on top, and that layer is where the traps live. Two routes are distinct when their *sequence of segments* differs, so two parallel `u -> v` segments of equal weight are two routes, not one — the count is a sum over segments, not over neighbors. The answer is taken mod `1e9+7`; unreachable stations print `-1`, and the source prints `1` for the empty route.

Before any algorithm I fix the scale, because the scale fixes the types. `n <= 2*10^5`, `m <= 5*10^5`, `w <= 10^9`. A shortest distance is a simple path of at most `n-1` edges (positive weights forbid revisits on a minimal route), so `dist` can reach `(n-1)*w_max ~ 2*10^14` — five orders of magnitude past 32-bit, so `dist[]` and the relaxation accumulator `dist[u]+w` must be 64-bit or the large tests are a silent wrong answer. The counts live mod `1e9+7`, small in value, but I keep them in `long long` so the intermediate `cnt[v] + cnt[u]` (up to ~`2e9`) does not overflow before I reduce.

Positive weights also settle whether the count is even finite: a zero-weight cycle on a shortest route could be looped arbitrarily often, making the count infinite, but `w >= 1` rules that out. More usefully, along any minimum-latency route the distance strictly increases at every step, so a shortest route never revisits a station, and every *tight* segment `x -> u` with `dist[x] + w == dist[u]` satisfies `dist[x] < dist[u]` strictly. That strict gap is the lever the whole single-pass count rests on.

There are two ways to count. The obviously-correct one is two-pass: run a plain Dijkstra for all `dist`, keep the tight segments as a DAG, and DP in nondecreasing-`dist` (hence topological) order, `cnt[v] = ` sum over tight predecessors, each parallel tight segment counted separately. The leaner one folds `cnt[]` into the Dijkstra itself: on a strictly shorter route to `v` reset `cnt[v] = cnt[u]`, on an equal-latency route accumulate `cnt[v] += cnt[u]`. I want to ship the single pass and keep the two-pass DP as a brute oracle to test against — but only if I can pin down why the single pass counts each route exactly once.

The invariant I need: *when `u` is popped at its final distance, `cnt[u]` already equals the true number of minimum-latency routes to `u`.* Every such route ends with a tight `x -> u`, and `dist[x] < dist[u]` strictly, so Dijkstra popped every such `x` earlier; by induction each `x` had its final `cnt[x]` at its expansion, and expanding it folded `cnt[x]` into `cnt[u]` exactly once per parallel copy — either as the reset that created `dist[u]` or as a later equal-latency `+=`. So `cnt[u]` is complete by the time `u` is popped. Base case: `cnt[s] = 1`, no predecessors. The invariant holds only if I fold a predecessor's count in exactly when that predecessor is final — once, at its expansion. That "once" is the fragile part, and it is exactly what my first loop gets wrong.

A first cut of the loop:

```
while (!pq.empty()) {
    auto [d, u] = pq.top(); pq.pop();
    for (auto [v, w] : adj[u]) {
        long long nd = d + w;
        if (nd < dist[v]) { dist[v] = nd; cnt[v] = cnt[u]; pq.push({nd, v}); }
        else if (nd == dist[v]) { cnt[v] = (cnt[v] + cnt[u]) % MOD; pq.push({nd, v}); }
    }
}
```

Two things here I want to test rather than trust: there is no stale-entry guard (`if (d > dist[u]) continue;`), and the equal-latency branch pushes `v` again. To break them I need a graph where a station is reached, then a strictly shorter route resets it, with stale copies of it still in the queue. Take stations `0..3`, source `3`, segments `3->1 (2)`, `3->1 (1)`, `3->2 (2)`, `3->1 (1)`, `1->1 (1)`, `3->0 (1)`, `3->2 (4)`, `3->2 (4)`, `0->2 (3)`, `1->2 (1)`. By hand: `dist[3]=0`, `dist[0]=1` (`cnt[0]=1`), `dist[1]=1` via the two `3->1 (1)` (so `cnt[1]=2`), and `dist[2]=2` reached by `3->2 (2)` (contributing `cnt[3]=1`) and `1->2 (1)` (contributing `cnt[1]=2`), so `cnt[2]=3`. The true counts are `cnt = [1, 2, 3, 1]`.

Now the buggy run. Pop `(0,3)`, relax: `3->1 (2)` sets `dist[1]=2, cnt[1]=1`, push `(2,1)`; `3->1 (1)` improves to `dist[1]=1, cnt[1]=1`, push `(1,1)`; `3->2 (2)` sets `dist[2]=2, cnt[2]=1`; the second `3->1 (1)` hits `nd==dist[1]`, so `cnt[1] += cnt[3] => 2` **and pushes `(1,1)` again**; `3->0 (1)` sets `dist[0]=1`; the two `3->2 (4)` exceed `dist[2]=2`, ignored. The queue now holds *two* copies of `(1,1)`. Pop the first `(1,1)`, expand station 1: `1->2 (1)` gives `nd=2==dist[2]`, `cnt[2] += cnt[1] => 3`. Correct so far — but with no stale guard, popping the *second* `(1,1)` expands station 1 again: `1->2 (1)` fires once more, `cnt[2] += cnt[1] => 5` — against a truth of `3`, because station 1 was expanded twice and re-poured its count each time.

Two defects, one root. Pushing `v` on the equal-latency branch manufactures duplicate queue entries for an already-settled station, and with no stale guard those duplicates get fully expanded — violating the "expand once" the invariant demands. The fix restores one expansion per station: (a) never push on the equal branch — an equal-latency arrival does not change `dist[v]`, so nothing new settles, only the count updates; (b) add `if (d > dist[u]) continue;` so any leftover entry whose key exceeds the now-final `dist[u]` is discarded. With both, re-running the same graph, the stale `(2,1)` from the very first `3->1 (2)` is popped and skipped, station 1 is expanded exactly once, and `cnt` lands on `[1, 2, 3, 1]` — the exact case that produced `5` now produces `3`.

There is a second, independent boundary to get right, inside a *single* expansion. On the documented sample (source `0`, edges `0->1 (2)`, `0->2 (2)`, `1->4 (3)`, two parallel `2->4 (3)`, `0->3 (4)`, `3->4 (1)`), station 4 has `dist=5` reached from 1, 2, and 3. When station 2 is expanded, its two parallel `2->4 (3)` each hit the `nd==dist[4]` branch and each fold `cnt[2]` in, so `cnt[4]` picks up `2*cnt[2]` — parallel tight segments are counted for free because the relaxation loop visits each segment of `adj[u]` independently. With `1->4` contributing `1`, both `2->4` contributing `1` each, and `3->4` contributing `1`, `cnt[4]=4`, matching the oracle.

The comparison operator is where that same expansion can go one notch wrong. Writing the reset branch as `nd <= dist[v]` instead of `nd < dist[v]` would, on station 4 of that sample, *reset* `cnt[4]` on every equal-latency arrival: the first `2->4` would overwrite the `1` already contributed by `0->1->4`, and the final `3->4` reset would leave `cnt[4]=1` instead of `4` — a gross undercount, plus needless re-pushes of settled stations. The boundary must be *strictly shorter resets, equal accumulates*: `<` / `==`, never `<=`. It is the off-by-one twin of the double-count — one notch off and you reset when you should add.

Edge cases. The source keeps `cnt[s]=1, dist[s]=0`; a self-loop `s->s (w>=1)` relaxes to `nd = w > 0`, matching neither `< 0` nor `== 0`, so it is ignored and the empty route stays unique (`printf "2 2 0\n0 0 1\n0 1 3\n"` gives `1`, `1`). A self-loop on any other node likewise relaxes to a strictly larger distance and never inflates a count. `n=1, m=0` prints `1`. An unreachable station keeps `dist=INF` and prints `-1` (`3 1 0 / 0 1 5` gives `1, 1, -1`). Two parallel source edges give `cnt=2`. A chain of 30 doubling diamonds yields `2^30 > 1e9+7`; reducing `cnt[v]` on every `+=` keeps each value below `2*(1e9+7) < 2^31` and prints `2^30 mod (1e9+7)`, and it is what stops a ~64-diamond chain from overflowing `long long`. `INF = LLONG_MAX` is only ever compared, never has `w` added to it (relaxation fires only from popped, finite-distance nodes), so there is no `INF + w` overflow.

The single-pass code agrees with the two-pass DAG-DP oracle on the sample and on several hundred random small graphs with heavy parallel edges and equal-weight ties (no mismatches), so I trust the count semantics. Standard binary-heap Dijkstra is `O((n+m) log n)` time and `O(n+m)` memory, well inside the 2-second budget; the counting adds `O(1)` per relaxation.
