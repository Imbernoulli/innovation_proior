The task is to write `N` as a sum of perfect squares `k*k` (`k >= 1`, repeats allowed) with as few terms as possible, reading `N` from stdin and printing the count. The bound sets the whole strategy: `1 <= N <= 10^9`. That kills anything linear in `N` — an `O(N)` DP array is too big and an `O(N*sqrt N)` fill is `~3*10^13` operations, hopeless in a second. And `10^9` sits just under `2^31`, so `N` fits in 32 bits, but `sqrt(N) ~ 31623` and any product of two such values lands near `10^9`; multiplying them in `int` overflows silently. So `long long` for `N` and every square from the start.

Three approaches present themselves. Greedy — subtract the largest square `<= ` the remainder and count — is `O(sqrt N)` and four lines, but it makes a local choice against a global objective, and coin-style greedy is correct for some systems and wrong for others (`{1,3,4}` making `6`). The exact DP `dp[v] = 1 + min_{s<=v} dp[v-s]` is trivially correct but `O(N*sqrt N)` time and `O(N)` memory — usable only as a small-`N` oracle. And number theory: Lagrange's four-square theorem says every positive integer is a sum of at most four squares, so the answer is always `1, 2, 3,` or `4`; if I can cheaply decide which, `N`'s magnitude stops mattering. That last route is the only one that survives `10^9`, so it is the one I have to get right — but first I want to know whether the cheap greedy is secretly fine, because if it were I would just take it.

It is not. On `N = 12`, greedy grabs `9`, then `1+1+1`: four terms; but `12 = 4+4+4` is three. On `N = 32`, greedy grabs `25`, then `4`, then `1+1+1`: five terms; but `32 = 16+16` is two — a gap of three. The mechanism is clear: snatching the largest square strands a remainder (`3`, `7`) that squares represent badly, whereas declining it and taking equal mid-size squares wins globally. Greedy is out; I keep `12` and `32` as regression cases.

So I build the `{1,2,3,4}` classifier, deciding in increasing order of cost. Answer `1` iff `N` is itself a perfect square. Answer `2` iff `N = a^2 + b^2` with `a >= 1`: loop `a` from `1` while `a*a <= N` and test whether `N - a*a` is a square — `O(sqrt N) ~ 31623` iterations, cheap. (There is a prime-factorization criterion for sums of two squares, but factoring `10^9` is itself `O(sqrt N)` and more error-prone, so the direct loop wins.) Answer `4` via Legendre's three-square theorem: a non-negative integer is *not* a sum of three squares iff it has the form `4^k(8m+7)`; strip all factors of `4`, then check `≡ 7 (mod 8)`. Answer `3` otherwise — the fall-through, justified because Lagrange guarantees four always suffice and the three cheaper cases are already excluded. Order matters: test `1`, `2`, the `4`-form, then conclude `3`; the `4^k(8m+7)` numbers are never squares or sums of two squares, so testing them last is safe, and testing `1`/`2` first is what makes the fall-through clean.

The four contract samples exercise the order end to end: `12` falls through to `3` (not a square, no two-square split, `12/4 = 3 ≡ 3` not `7 (mod 8)`) matching `4+4+4`; `7 ≡ 7 (mod 8)` gives `4` matching `4+1+1+1`; `13` catches at `a=2` since `13-4=9=3^2`, giving `2`; `25=5^2` gives `1`.

Two implementation pitfalls that the constraint sizes genuinely invite. First, the integer square root. `(long long)sqrtl((long double)n)` floors a value that floating point may have nudged below the true root, so near a large perfect square it can come out one too small: `999950884 = 31622^2`, but `sqrtl` may return `31621.9998`, truncating to `31621`, whose square is `999886641 != N` — and the program would then report `2` for a number that is one square. I correct with two nudges: `while (r*r > n) r--;` pulls down an overshoot, `while ((r+1)*(r+1) <= n) r++;` pushes up an undershoot. On `999950884` the up-nudge restores `r = 31622` and the test passes; on a nearby non-square like `999950885` it stops at `r = 31622` with `r*r != n`, correctly false. Both rounding directions are covered.

Second, stripping the factor of `4` in the Legendre test must be a loop, not a single division. Take `N = 112 = 4^2 * 7`, which is `4^k(8m+7)` with `k=2`, so the answer is `4`. A single `if (m%4==0) m/=4;` leaves `m = 28`, and `28 % 8 = 4 != 7`, wrongly falling through to `3`. With `while (m%4==0) m/=4;`, `m` reduces `112 -> 28 -> 7`, and `7 % 8 == 7` gives `4`. The `while` is load-bearing, and the brute force confirms `112 -> 4`.

Two smaller points. The two-square loop starts at `a = 1`, not `0`, so its two terms are genuinely positive (and it cannot silently re-enter the answer-1 case). And every product in the program — `r*r`, `(r+1)*(r+1)`, `a*a` — peaks near `31623^2 ~ 10^9`, well inside 64-bit range with no 32-bit multiply anywhere. The only loop is the answer-2 search, `O(sqrt N)`; a worst case (an answer-3 value near `10^9` that runs the loop to the end) measures `0.00 s`.

Edge cases fall out: `N=1 -> 1`, `N=2 -> 2` (`1+1`), `N=3 -> 3` (`1+1+1`), `N=7 -> 4`, the largest square `<= 10^9` (`31622^2`) `-> 1`, and `N=10^9 -> 2` (it is `2^9 * 5^9`, a sum of two squares, found quickly by the loop).

Against the obviously-correct `O(N*sqrt N)` DP — exhaustively over small `N` and on random cases weighted toward perfect squares, `4^k(8m+7)` forms, and generic values — the classifier matches with zero mismatches; the `4^k(8m+7)` agreement is the one I care about, since it is where the `while`-reduction and the three-vs-four split could have been wrong. So the final program is the four tests in order, `O(sqrt N)`, indifferent to `N = 10^9`.
