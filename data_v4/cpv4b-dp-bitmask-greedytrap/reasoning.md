The number that decides this problem is `m <= 18`. Firing burst `j` costs `c[j]` and delivers the channel set `mask[j]`; I may fire any subset, redundant re-delivery is harmless, and I want the cheapest subset whose masks cover all `m` channels, or `-1`. That is weighted set cover, which is NP-hard in general — but with `m <= 18`, a set of delivered channels is a subset of an 18-element universe, so there are only `2^18 = 262144` distinct coverage states, each fitting in a 32-bit `int`. With `k <= 200`, a DP that touches every state and tries every burst is `2^18 * 200 ≈ 5.2*10^7` relaxations, comfortably under a second. So the small universe turns an intractable problem into an exact one, and I want the exact optimum, not an approximation.

One arithmetic note before anything else: costs run to `10^6` and a cover can stack many bursts, so a total can reach `k * 10^6 = 2*10^8` — inside 32-bit, but the unreachable-state sentinel plus a cost is the classic silent-overflow trap, so cost accumulators are `long long` from the start.

**Why I distrust the greedy.** The textbook set-cover heuristic — repeatedly fire the burst with the best ratio of `cost / newly-covered channels` — is fast and tempting. But it is famous precisely as an *approximation* with a logarithmic-factor guarantee, i.e. provably suboptimal on some inputs. Coverage is a global property and the ratio a local score, exactly the setup where greedy strands itself. The trap has a definite shape: a burst that covers many channels cheaply per channel (great ratio, greedy grabs it) but leaves an awkward leftover expensive to finish, while a different pair partitions the universe cleanly for less.

Four channels `{0,1,2,3}`, three bursts: burst A covers `{0,1,2}` at cost `5` (ratio `5/3 ≈ 1.67`), burst B covers `{0,1}` at cost `4` (ratio `2.0`), burst C covers `{2,3}` at cost `4` (ratio `2.0`). Greedy fires A first for its best ratio; now `{0,1,2}` is covered and only channel `3` remains, which only C can finish, adding `4` for a total of `9`. But B `{0,1}` plus C `{2,3}` covers everything for `4 + 4 = 8`. Greedy loses, and I can see why: A's spend on `{0,1}` was wasted because B covers that pair more cheaply in context, while A only half-covered the `{2,3}` pair that C completes anyway. The leftover cleanup is where greedy bleeds. That kills the heuristic; I commit to the exact DP.

**The DP.** `dp[S]` is the minimum cost of a set of fired bursts whose masks union to a set containing `S`, with `dp[0] = 0` and everything else `+inf`. From any reachable `S`, firing burst `j` reaches `S | mask[j]` at extra cost `c[j]`, and the answer is `dp[full]` for `full = (1<<m) - 1`, or `-1` if it stayed infinite.

The structural fact that makes a single ascending pass correct: `S | mask[j] >= S` as an integer, because OR only ever *sets* bits, never clears them. So every useful transition goes from `S` to a strictly larger state (it equals `S` exactly when `mask[j]` is already a subset of `S`, a no-op I skip). Iterating `S` from `0` upward therefore means that when I process `S`, its `dp[S]` is final — it could only have been improved from strictly smaller states, all already handled. No queue, no topological sort; plain ascending iteration is the correct order, and it is not a stylistic choice. The reflex to reach for the descending `for (S = full; S >= 0; S--)` loop that subset-sum DPs use would break this DP: transitions flow low-to-high, so a descending pass reaches the small source states like `S=0` last, having already passed the states they feed and never relaxing out of them — `dp[full]` stays `INF` and it prints `-1` on a solvable instance. The direction is load-bearing.

**Sentinel and no-ops.** Two transcription hazards in the inner loop. First, a burst whose mask is already a subset of `S` gives `ns == S` and only wastes a comparison; a zero-channel burst (`t = 0`, which the contract allows) is the degenerate case, a self-loop that does nothing. I skip `ns == S` outright — cleaner and it removes redundant work. Second, the unreachable sentinel: I guard with `if (dp[S] == INF) continue`, so I never compute `INF + cost` on the current code path, but `INF = LLONG_MAX` is fragile — any later refactor to a backward transition, or dropping the guard, makes `LLONG_MAX + cost` overflow to a negative that silently wins the `min`. Setting `INF = LLONG_MAX / 4 ≈ 2.3*10^18` defuses that: even an accidental `INF + cost` stays comfortably positive and never beats a real cost. I'd rather remove the landmine than trust the guard forever.

**Edge cases.** `m = 0`: `full = 0`, `dp[0] = 0` already, so I print `0` — every requirement vacuously met by firing nothing, matching the contract. The state vector has size `1 << 0 = 1`, no out-of-bounds. `k = 0` with `m > 0`: `dp[full]` is unreachable, stays `INF`, prints `-1` — nothing covers anything with no bursts. Zero-channel bursts are absorbed by the `ns == S` skip. Heavy redundancy (two bursts with the same mask at different costs): from `S=0` the cheaper one sets the state first and nothing beats it, so the DP naturally takes the cheaper redundant burst. Memory: `2^18` `long long`s is about 2 MB, trivial against 256 MB. And the largest total the DP can produce is bounded well inside 64-bit, so `long long` accumulators never overflow.

The worst case for the state count is `m=18` with all single-channel bursts, which makes nearly the whole `2^18` space reachable; even there the `5.24*10^7` inner iterations of a couple of integer ops each stay well under the `1 s` limit. So the exact `O(2^m k)` forward-relaxation DP — ascending states, `INF = LLONG_MAX/4`, the `ns == S` skip, `dp[full]` or `-1` — is what I ship; the full module is in the answer.
