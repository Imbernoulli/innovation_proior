The numbers fix the types before the algorithm does. There are up to `n = 2*10^5` beds, each wanting up to `a[i] = 10^9` liters, so the total `S = sum(a)` can reach `2*10^14` — three orders of magnitude past the 32-bit ceiling of `~2.1*10^9`. Every prefix sum, every accumulator, the bounds, and the answer itself have to be 64-bit `long long`; an `int` total is a silent wrong answer on exactly the largest tests. And `k` runs up to `10^9` while `n` stays at `2*10^5`, so `k` can dwarf the bed count — I carry it as `long long` too, so a comparison like `blocks <= k` never trips on a stray promotion. The task: cut the row into `k` contiguous blocks, one per robot, each robot's busy time being its block's water sum; robots run in parallel, so the greenhouse finishes at the *largest* block sum, and I want that maximum as small as possible.

Two routes are on the table. The balance-by-average greedy aims each block at the fair share `S/k`: walk left to right and cut a block once its running sum reaches about `S/k`. It is `O(n)` and a dozen lines — but "make every block near the average" and "make the maximum block small" are different objectives, and contiguity removes any freedom to shuffle beds around. That is exactly the setup where a greedy optimizing the wrong local quantity gives a worse global maximum, so I want to break it before trusting it. The alternative is binary search on the answer: fix a candidate finish time `T` and ask whether the row can be cut into at most `k` contiguous blocks each summing `<= T`. If a `T` works, every larger `T` works too, so feasibility is monotone and the least feasible `T` falls out of a bisection.

So I attack the average greedy with a concrete instance: `a = [7, 1, 2, 9, 2, 6]`, `k = 3`. Here `S = 27` and the share is `9`, and the greedy drops its two cut points near prefix sums `9` and `18`. The prefix sums are `7, 8, 10, 19, 21, 27`; nearest to `9` is `8` (after index 1, distance `1`), nearest to `18` is `19` (after index 3). That gives `[7,1] | [2,9] | [2,6]` with sums `8, 11, 8`, maximum **11**. But `[7,1,2] | [9] | [2,6]` has sums `10, 9, 8`, maximum **10** — strictly better, and structurally unreachable by the average rule, because isolating the lone `9` into its own block pushes that block *below* the average, which "stay near `S/k`" actively refuses. Minimizing the max sometimes wants deliberately unbalanced blocks.

There is a second, cleaner way it dies: the average itself can be infeasible. On the sample `a = [2, 3, 5, 8, 1, 1, 4]`, `k = 3`, the share is `8`; filling each block up to `8` gives `[2,3] | [5] | [8] | [1,1,4]` — **four** blocks — because the bed `8` plus anything already exceeds `8`. The average is below an unavoidable single block, so the greedy cannot even hit `k = 3`, while the true answer is `10`. The floor on the answer is never `S/k` alone; it is `max(a[i])`, and only the smallest *feasible* `T` is the answer. The average greedy is out.

For a fixed `T`, the minimum number of contiguous blocks with each sum `<= T` comes from the obvious greedy: start a block, keep adding beds while the running sum stays `<= T`, and the instant the next bed would push it over, close the block and start a new one there. Extending greedily is optimal for the count — closing a block early can never let a later block hold more (a standard exchange argument). One guard: if any single bed has `a[i] > T`, no block can hold it and `T` is infeasible outright. The decision is then just "block count `<= k`?". For the bisection, `feasible(T)` is false for small `T` and true for large `T`, so I search `[lo, hi]` with `lo = max(a[i])` (some block must contain the biggest bed) and `hi = S` (one block holding everything is always feasible, since `k >= 1`).

Transcribing that, the natural first cut of the feasibility lambda starts the counter at zero and only bumps it on an overflow:

```
auto feasible = [&](long long T) -> bool {
    long long cur = 0, blocks = 0;
    for (int i = 0; i < n; i++) {
        if (a[i] > T) return false;
        if (cur + a[i] > T) { blocks++; cur = a[i]; }
        else cur += a[i];
    }
    return blocks <= k;
};
```

That undercounts. The block count is "one initial block plus one per overflow," so starting at `0` reports one fewer block than any partition actually uses — and the undercount flips a real verdict. Take `a = [3, 3, 3]`, `k = 2`, `T = 3`, which truly needs `[3]|[3]|[3]` = 3 blocks and is infeasible: the lambda sets `cur = 3` at index 0, overflows to `blocks = 1` at index 1 and `blocks = 2` at index 2, and returns `2 <= 2` → **true**. Reporting an infeasible `T` as feasible drives the bisection below the true optimum — a real wrong answer. The fix is to start `blocks = 1` and bail the instant it exceeds `k`; the early bail also keeps the counter from running far past `k` when `k` is near `10^9`. With `blocks = 1`, the same `[3,3,3]` case reaches `3` at the last overflow and rejects `T = 3` correctly.

The bisection is the standard lower-bound form: while `lo < hi`, test `mid = lo + (hi - lo) / 2` (the subtraction form, never `lo + hi`, which would overflow near `2*10^14`); if feasible set `hi = mid`, else `lo = mid + 1`; the loop ends with `lo` the least feasible finish time. One corner needs an explicit hand: `n = 0` gives `lo = hi = 0`, the loop never runs, and printing `lo` would print `0` — correct, but only by accident, so I short-circuit `n = 0` to `0` before the loop to make the intent unambiguous. The rest fall out of the bounds: `k = 1` forces one block, so the least feasible `T` is `hi = S`, the whole sum; `k >= n` lets every bed stand alone, and the `lo = max(a[i])` floor plus the `a[i] > T` guard drive the answer straight to the largest bed; all-zero beds give `lo = hi = 0` → `0`; and a lone bed far above the average sets the floor exactly as the `[7,1,2,9,2,6]` trap warned.

That closes it: `lo = max(a[i])`, `hi = S`, the `n = 0` short-circuit, and the lower-bound bisection over the fixed feasibility greedy, all in 64-bit arithmetic — `O(n log S)` time, `O(n)` space. The full listing is the answer.
