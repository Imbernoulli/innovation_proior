The distinctive part of this borehole-spacing task is all in the corners, not the algorithm. Coordinates run over `-10^9 <= x[i] <= 10^9`; they can be negative, sit exactly on the origin stake (`0`), and repeat, and I must pick exactly `k` of `n` (`2 <= k <= n <= 2*10^5`) to maximize the minimum pairwise distance. Three consequences shape everything. First, the answer can legitimately be `0` — if `k` exceeds the number of distinct coordinates, or `k = n` over an array with a repeated value, some chosen pair is forced to coincide. Second, the answer is never negative even when every borehole lies west of the stake, because only differences of coordinates reach the output, never a raw coordinate. Third, a span `x[max] - x[min]` can reach `10^9 - (-10^9) = 2*10^9`; that sits just under the signed-32-bit ceiling `2147483647`, so a single difference fits an `int` but with almost no headroom, and a naive midpoint `lo + hi ~ 4*10^9` would blow past it. So I hold every coordinate and every distance in `long long` from the start.

Sort the coordinates once, and the isolation of any chosen set is just the smallest gap between adjacent chosen coordinates: a non-adjacent chosen pair has a chosen coordinate between them, so its distance is a sum of adjacent gaps and can never be the minimum. "Minimize over all pairs" is therefore "minimize over adjacent chosen pairs after sorting," and I only ever reason about gaps in the sorted array.

That makes the answer binary-searchable. The predicate "can I choose `k` coordinates pairwise at least `d` apart?" is monotone: a placement valid at `d` stays valid at any smaller `d`, so the achievable distances form an interval `[0, D]` and I want its right endpoint `D`. The feasibility test is a linear greedy on the sorted array — keep the leftmost borehole, then scan right taking the next coordinate that is at least `last + d`; feasible iff at least `k` get placed. Leftmost-first is optimal because any valid placement can be shifted left sensor by sensor onto the greedy without losing anyone. Brute force over all `C(n,k)` size-`k` subsets is my correctness oracle for tiny `n`; the binary search is `O(n log span)` for the real sizes. On the sample `k=3`, `[-7,-3,0,0,4,9]`: `d=7` places `-7, 0, 9` (three, feasible); `d=8` places `-7, 4`, then nothing (two, infeasible); so the largest feasible `d` is `7`, matching the expected answer.

The search interval is where this problem's trap lives. My instinct is `lo = 1, hi = span`, but the answer can be `0`, and `lo = 1` never evaluates `feasible(0)`. Take `x=[-9,-5,-5,-1]`, `k=4` (forced; gaps `4,0,4`; true answer `0`). With `lo=1, hi=8` the loop does run, but every probe is `d >= 1`: `feasible(1)` places `-9, -5, -1` — only three, since the duplicated `-5` collapses — so it is infeasible, the search drives `hi` down, and it exits with `ans` never updated. The output is correct only because I happened to initialize `ans = 0`; had I reached for a "no feasible distance yet" sentinel like `-1`, the same code would print `-1`. The fix is to make `0` a real member of the search: `lo = 0`. Then `feasible(0)` is genuinely tested, and on a sorted array `x[i] - last >= 0` always holds, so the greedy places all `n` and `feasible(0)` is always true — exactly right, since any `k` boreholes have some (possibly zero) closest gap. The answer is now certified by construction, not by the initializer.

The greedy's base case `cnt = 1, last = x[0]` has to count duplicates correctly, and the comparison direction is load-bearing. With `>= d` and `d = 0`, equal coordinates satisfy `x[i] - last >= 0` and count toward `k`, so `feasible(0)` reaches `k`; with `> d` the equal coordinates would be rejected, `feasible(0)` could fall short of `k`, and a genuinely-zero answer would come back "infeasible." On `[2,2,2]`, `k=2`: `feasible(1)` places one and stops (`0 >= 1` fails), `feasible(0)` places all three. The `>=`, not `>`, is what makes the duplicate corner work.

Two sign points close it out. `hi = x[n-1] - x[0]` is a difference of sorted endpoints, so `hi >= 0` even when every coordinate is negative (`[-9,-5,-1]` gives `hi = 8`) — I never seed the bound with a raw coordinate. And forming the midpoint as `mid = lo + (hi - lo) / 2` with `0 <= lo <= hi` keeps the numerator non-negative, so the `lo + hi ~ 4*10^9` I flagged at the outset never materializes.

The remaining corners fall out: all-identical coordinates give `hi = 0`, only `feasible(0)` runs, answer `0`; all-negative `[-9,-5,-1]`, `k=2` searches up to `hi=8` and places the two extremes for a positive `8`; `k=n` forces every borehole and lands on the minimum adjacent gap, possibly `0`; `k=2` places the two extremes for exactly `span`.

Running the binary search against the subset-enumeration brute force on small cases weighted toward negative, zero, and duplicated coordinates turns up no mismatches. At the largest size, `O(n log span)` is roughly `2*10^5 * 31 ~ 6*10^6` greedy steps, comfortably inside the one-second limit.

The full solution is the single-file `O(n log span)` binary search in the answer: `lo = 0`, the `>=` leftmost greedy, and `long long` throughout.
