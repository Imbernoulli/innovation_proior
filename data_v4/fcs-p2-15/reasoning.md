I have `n` keys in fixed sorted order with access frequencies `f[1..n]`, and I want the cheapest binary search tree under `cost = sum_i depth(i)*f[i]` with the root at depth 1. The two constraint numbers shape everything: `n <= 500` and `0 <= f[i] <= 10^9`. The second is a landmine. The total weight `sum f` reaches `500 * 10^9 = 5*10^11`, and a key in a degenerate chain sits at depth up to `n = 500`, so the cost can reach `500 * 5*10^11 = 2.5*10^14` — well past the 32-bit range of about `2.1*10^9`. Every accumulator has to be 64-bit or the large tests silently wrong-answer. The first number cuts the other way: `n = 500` means an `O(n^3)` interval scan is only about `n^3/6 ≈ 2*10^7` operations, so there is no speed pressure at all and I can pick the algorithm I can prove rather than the fastest one.

The structural hinge is that, because the keys are fixed in sorted order, the in-order traversal of any legal tree is `1..n`, so a tree is determined recursively by *which key roots each contiguous range*. Making key `r` the root of `[i..j]` forces keys `i..r-1` into the left subtree and `r+1..j` into the right — no freedom to scatter keys, the root choice alone splits the range into a left and a right contiguous block. This is what makes the greedy tempting and also what makes it wrong.

Two routes are on the table. The greedy "most frequent at the root" — put the highest-frequency key of a range at its root, recurse on the two halves — is three lines and nearly linear, and the intuition that the most-searched key should be shallowest is seductive. The interval DP — `dp[i][j]` = optimal cost on keys `i..j`, try every root, combine — is provably exhaustive but I have to get the depth bookkeeping right. Before trusting greedy I try to break it.

Take `f = [2, 3, 4]`, distinct so there is no tie to argue about. Greedy roots `[1..3]` at key 3 (frequency 4); keys 1 and 2 are both smaller, so they chain into the left subtree, and greedy then roots that pair at key 2 with key 1 below it: key 3 at depth 1, key 2 at depth 2, key 1 at depth 3, cost `4*1 + 3*2 + 2*3 = 16`. Root at the *middle* key 2 instead and keys 1 and 3 become its two children at depth 2, cost `3*1 + 2*2 + 4*2 = 15`. Greedy is wrong, and the mechanism is clear: snatching the biggest frequency to the root forced the other two keys into a chain below it, while the balanced choice keeps both at depth 2, and the weight that drops out of the deep chain (`2 + 4`) outweighs the one extra unit of depth charged to the big key. Greedy optimizes one key's depth in isolation, but the cost sums over all keys and the BST constraint couples them through the partition, so no local rule can be optimal. Greedy is out.

For the DP, `dp[i][j]` is the minimum cost over keys `i..j` with that subtree's own root at depth 1, and the empty range is `dp[i][i-1] = 0`. Choose root `r`: the left subtree is optimal on `[i..r-1]`, the right on `[r+1..j]`. The subtle part is that when those two subtrees hang under `r`, every node in them drops one level, so each subtree contributes its standalone `dp` *plus its total weight* (one added unit of depth per node), and `r` itself contributes `f[r]` at depth 1. But `f[r] + W(i,r-1) + W(r+1,j)` is just `W(i,j)`, the whole range's weight, so the recurrence collapses to

```
dp[i][j] = W(i, j) + min over r in [i..j] of ( dp[i][r-1] + dp[r+1][j] )
```

with `W(i,j) = prefix[j] - prefix[i-1]` from precomputed prefix sums. The reading is that the range weight is added once per level of recursion, which is exactly how each key ends up paying `depth * f`. I fill `dp` by increasing range length, since every candidate sub-range is strictly shorter. (Knuth–Yao would bring this to `O(n^2)` off the monotonicity of the optimal root, but at `n = 500` the cubic is `~2*10^7` and I would rather ship the version I can trace than the fiddlier one.)

Running the collapsed recurrence on the sample confirms the `+W` mechanics reproduce the hand-computed number. `f = [2,3,4]`, `prefix = [0,2,5,9]`. The length-3 cell scans roots — `r=1` gives `dp[1][0]+dp[2][3] = 10`, `r=2` gives `dp[1][1]+dp[3][3] = 2+4 = 6`, `r=3` gives `dp[1][2]+0 = 7` — so `dp[1][3] = 9 + 6 = 15`, optimal root at the middle key `r=2`, the choice greedy refused, matching the expected 15.

The one place this recurrence bites is the weight term, so I state the pitfall before falling into it: if I treat `dp` as a plain sum of subtree costs and write `cand = dp[i][r-1] + dp[r+1][j]` without `+ W(i,j)`, I charge nothing for descending a level. On the singleton `f = [5]` that gives `dp[1][1] = dp[1][0] + dp[2][1] = 0`, not `5` — the single key sits at depth 1 and its whole cost *is* `W(1,1)`. The fix is to add the range weight exactly once, `dp[i][j] = best + w`, where `best` is the pure `min(left + right)` over roots; adding it both inside each candidate and again on assignment would double it (`10` for `[5]`). With `best + w`: `dp[1][1] = 0 + 5 = 5`, correct.

The corners. `n = 0`: empty input returns 0 via `if (!(cin >> n)) return 0;`, and a read `n = 0` runs no length loop and prints `dp[1][0] = 0` from the zero-initialized table. `n = 1`: prints `W(1,1) = f[1]`. All-zero frequencies: every `W` is 0, answer 0. All-equal frequencies: the DP finds the balanced optimum rather than the greedy chain (`[4,4,4]` gives `4 + 8 + 8 = 20`, not `24`). A single huge frequency: the DP places it at or near the root by minimization with no special-casing. Overflow is covered by `long long`; the `LLONG_MAX` sentinel is only ever compared against and overwritten on the first candidate (every non-empty range has a root), never added to, so it cannot overflow.

To cover the cases I did not trace by hand I differential-tested against an independent oracle that enumerates every BST shape explicitly and walks each tree's `sum depth*f` directly — a code path with no shared formula with the collapsed-weight DP — over the structured edges (`n = 0/1`, all-zero, all-equal, monotone, a giant frequency at an end and in the middle) plus random instances up to `n ≈ 11`, with no disagreement. The `n = 500` all-`10^9` worst case runs in a few hundredths of a second, far inside the 2-second limit.

The landing is the simple provable cubic interval DP with `long long` throughout and prefix sums for the range weight; the full module is in the answer.
