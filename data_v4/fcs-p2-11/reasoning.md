This is word break: decide whether `s` is a back-to-back concatenation of reusable dictionary words, output `YES` or `NO`. The scale tells me what is affordable. `|s| <= 5000` is short, `n <= 10^5` words are allowed but their total character mass is capped at `2*10^5`, so the longest single word is at most `5000` and the whole dictionary is bounded in total size. The answer is one boolean, so nothing overflows any integer type and the only real questions are the boundary logic and time. One parsing corner needs care: I read `n`, then `n` word tokens, then `s` as one more token — but if `s` is empty it contributes no token at all, so `cin >> s` simply leaves `s` empty. A missing final token is the empty-string case (`YES`, the empty concatenation), not malformed input. The substantive question to settle before writing any code is whether a single left-to-right greedy pass that commits to one match per position is correct, or whether every boundary has to stay open.

Three approaches are on the table. Greedy longest-match and greedy shortest-match each stand at the current position, take one match, and jump past it; both are `O(|s| * maxLen)` and a dozen lines, but each makes an irrevocable local commitment against a global feasibility question. The alternative is a prefix-reachability DP that keeps every boundary open. The verb "split `s` into words" pulls hard toward greedy — which is exactly the locally-greedy rule that adjacency-feasibility problems punish — so I try to break it before trusting it. Greedy longest-match fails precisely when the longest match at some reachable position leads only to dead ends while a shorter match at that same position leads to a full segmentation. The trap is easy to hide by accident: if the dictionary contains a universal short word like a lone `a`, all-`a` strings tile trivially and greedy never fails, so I need a dictionary with no universal rescue and a long word that eats a boundary character a shorter word would have preserved. Set a short word and a longer word that share a start, so the long word consumes the character the only valid second word needs. Dictionary `{do, dog, gs}`, `s = "dogs"`: longest-match grabs `dog`, lands at position 3 with `"s"` left, which is not a word, and reports `NO`. But `do | gs` covers `"dogs"` exactly, so the truth is `YES` — grabbing `dog` stranded the `g` that the only valid second word `gs` required. Greedy longest-match is out.

Shortest-match dies the same way from the other end. Dictionary `{aa, aab}`, `s = "aab"`: shortest-match takes `aa`, strands `"b"`, reports `NO`, but the single word `aab` covers everything, so the truth is `YES`. Both greedy rules fail for one reason: a single irrevocable local commitment to one match destroys boundary options the global feasibility question depends on. I want the method that keeps every boundary open.

The only state that matters at a cut point is "is the prefix up to here segmentable?" — nothing about *how*, because the words are independent and reusable. So `dp[i]` = true iff `s[0..i-1]` is a concatenation of dictionary words, with `dp[0] = true` for the empty prefix, and for `i >= 1`,

`dp[i] = OR over j in [0, i-1] of ( dp[j] AND s[j..i-1] is in the dictionary )`,

with the answer `dp[m]`, `m = |s|`. Two refinements make it fast. The inner `j` only needs to range over the last `maxLen` characters (`maxLen` = longest dictionary word), because `s[j..i-1]` can only be a word if its length `i - j <= maxLen`; so `j >= max(0, i - maxLen)`, turning the inner loop from `O(i)` into `O(maxLen)`. Dictionary membership is a hashed-set lookup.

My first cut of the core scans `j` from `i-1` down to `0` with no `maxLen` cut and no `break`:

```
for (int i = 1; i <= m; ++i)
    for (int j = i - 1; j >= 0; --j)
        if (dp[j] && words.count(s.substr(j, i - j))) dp[i] = 1;
```

That is logically correct but the inner loop is `O(i)`, so with `m = 5000` I build about `m^2/2 ≈ 1.25*10^7` substrings whose lengths sum to roughly `m^3/6 ≈ 2*10^10` hashed characters — far too slow. The fix is exact, not a heuristic: bound `j >= max(0, i - maxLen)`, since a longer final chunk cannot match, and `break` on the first hit, since the OR is settled once any disjunct is true. On the `{do, dog, gs}` / `"dogs"` case the fixed loop reaches `dp[2]` via `do`, then `dp[4]` via `gs` — recovering exactly the `do | gs` split greedy stranded.

The remaining worry is worst-case time, because the `substr` call allocates and hashes its whole length: cost is the number of `(i, j)` pairs actually examined times the substring length. The `break` helps when matches are common; an adversary keeps matches rare so the inner loop runs its full `maxLen` on long substrings. The pathological shape is "no short word": a dictionary of only long all-`a` words, say `a^2500 ... a^5000`, against `s = a^5000`, which runs around `0.5 s`. But that family needs roughly `2500` words averaging length `~3750`, about `9*10^6` characters — far past the `2*10^5` total-length cap. Under the actual cap, the worst all-`a` adversary I can build measures about `0.05 s`, comfortably inside the `2 s` limit. The constraint is doing the real work: it forbids exactly the inputs that would strain the simple substring DP.

The corners where this kind of code dies, checked against the recurrence: empty `s` (`m = 0`, the outer loop never runs) gives `dp[0] = 1 -> YES`; empty dictionary with non-empty `s` misses every lookup so `dp[i]` stays `0 -> NO` (and `YES` when `s` is also empty); `maxLen = 0` happens only when `n = 0`, and then `lo = max(0, i) = i` so the inner loop never runs — consistent with the empty-dictionary case, no division or out-of-range. One unmatchable letter `{a, b}` / `"c"` gives `NO`; a single exact match `{abc}` / `"abc"` gives `dp[3]` via `dp[0]` and `abc`, so `YES`. Duplicate words collapse harmlessly in the `unordered_set`, and `cin >>` consumes arbitrary whitespace so parsing is format-agnostic.

Hand traces convince me of individual cases; to trust the whole space I differential-test the DP against an independent memoized reachability oracle (a recursive `can(start)` search written separately, so a shared logic bug is unlikely). The cases mix edge inputs, small-alphabet random instances where accidental segmentations are common and the DP is genuinely exercised, and the greedy-trap families (`dogs`, `aab`) — roughly 750 cases, and the DP and the oracle agree on every one. So I ship the prefix-reachability DP, `O(|s| * maxLen)` time and `O(|s|)` space, not the greedy I disproved; the full self-contained program is in the answer.
