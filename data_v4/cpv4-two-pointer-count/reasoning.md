The first thing this problem decides for me is not the algorithm but the data types, and the deciding quantity is the *answer itself*, not the window sum. I have `n` positive pulses `a[0..n-1]` (`a[i] >= 1`, up to `10^9`), and I must count the contiguous runs `[i..j]` whose sum lands in `[L, R]`, distinct by position — two windows with equal total but different endpoints both count. Input is `n L R` then the `n` values; I print one integer. Now the scale: when `L = 1` and `R` is huge, *every* subarray qualifies, so the count reaches `n(n+1)/2 ≈ 2*10^10` at `n = 2*10^5` — already past the 32-bit ceiling of `~2.1*10^9`. The window sums overflow too (one window can hit `2*10^5 * 10^9 = 2*10^14`, and `R` is allowed up to `10^18`), but that danger is loud. The count overflow is the quiet one: an `int` accumulator would sail through every small test and only die on the all-ones giants. So the values, the running sum, and the count are all `long long`.

Two pointers is available here only because `a[i] >= 1`. Define prefix sums `P[0] = 0`, `P[k] = a[0]+...+a[k-1]`; positivity makes `P` strictly increasing, so window `[i..j]` has sum `P[j+1] - P[i]`, and for a fixed right endpoint that sum strictly *decreases* as the start `i` moves right. That monotonicity is the whole engine: it lets one shrinking pointer track the "too big / fits" boundary without ever backing up. With a single zero or negative it would break, so I want the solution to depend on positivity by design, not on the tests happening to be positive.

The clean reduction is to `atMost(X)` = the number of subarrays with sum `<= X`. Then the range count is `atMost(R) - atMost(L-1)`: subarrays with sum `<= R` minus those with sum `<= L-1` leaves exactly those in `[L, R]`, and each `atMost` is one `O(n)` sweep. (The brute `O(n^2)` over all `(i,j)` is `~4*10^10` operations at full size — too slow to submit, but it is my correctness oracle later.) The idea is not the risk; the transcription is: the shrink condition, the per-step count, and the `L-1` boundary are each an off-by-one waiting to happen.

For `atMost(X)` I sweep `right` from `0` to `n-1`, maintaining `left` and the current window sum. Each step I add `a[right]`, then shrink `while (sum > X)` by subtracting `a[left]` and advancing `left`. The shrink has to be a `while`, not an `if`, and this is the trap the problem genuinely invites: one large `a[right]` can overshoot `X` by more than a single left element's worth, so a lone subtraction leaves the window still oversized. Concretely on `a = [1,1,5]`, `X = 2`, the subarrays with sum `<= 2` are `[1]@0`, `[1]@1`, `[1,1]` — three of them. With an `if`, at `right=2` I add `5` (sum `7`), subtract `a[0]=1` once (sum `6`, still `> 2`), stop, and then count `right - left + 1 = 2` windows — but those are `[1,5]` (sum 6) and `[5]` (sum 5), both violating the constraint, giving a wrong `5`. The `while` instead drains `a[0], a[1], a[2]` until `sum = 0` with `left = 3 > right = 2`, contributing `0`, and returns the correct `3`. After the loop, `[left..right]` is the longest window ending at `right` with sum `<= X`, and by monotonicity every start in `[left, right]` is valid — exactly `right - left + 1` of them. Note the loop can push `left` to `right + 1` (empty window, contribution `0`), so the code must tolerate `left > right` without going negative.

The difference boundary must be `L-1`, never `L`, and the discrete case is unforgiving. Take `a = [3]`, `L = R = 3`: the one window has sum `3`, in range, answer `1`. Here `atMost(3) = 1` and `atMost(2) = 0` (sum `3 > 2` shrinks to empty), so `atMost(R) - atMost(L-1) = 1`. Had I written `atMost(L) = atMost(3) = 1`, the difference would be `0` — subtracting `atMost(L)` erases every window whose sum equals `L` exactly. And `L-1` has to be computed in 64-bit since `L` reaches `10^18`.

One more thing about that boundary: `atMost` with a negative argument is a latent out-of-bounds read. The contract gives `L >= 1`, so `L-1 >= 0` and `atMost` never sees `X < 0` today — but if it did, `sum` is a total of positives and can never drop below `0`, so `while (sum > X)` with `X < 0` would never stop, subtracting past `left = right + 1` and reading `a[left]` off the end. A guard `if (X < 0) return 0;` makes `atMost` total (no subarray sum can be `<= -1`) and is a no-op for `X >= 0`.

The corners fall out cleanly. `n = 0`: both sweeps loop zero times, answer `0`. Single element below/above/in range: `a=[7]`, `L=8`, `R=10` gives `atMost(10) - atMost(7) = 1 - 1 = 0`, the `L-1 = 7` boundary correctly excluding the sum-`7` window. All-ones at `n = 2*10^5`, `L = 1`, `R = n` gives `n(n+1)/2 = 20000100000` — fits `long long`, overflows `int`. Max stress `a[i] = 10^9` keeps every running sum `<= 2*10^14 < 9.2*10^18`, so no overflow anywhere. I ran the two-pointer against an `O(n^2)` brute on randomized cases — small `n <= 8` with all-ones runs and exact-value (`L = R`) queries, plus larger `n` with values near `10^9`, all with `1 <= L <= R` as the contract requires — and got zero mismatches.

So I ship one self-contained stdin/stdout file: read `n, L, R` and the array, print `atMost(R) - atMost(L-1)`, where `atMost` guards `X < 0`, `while`-shrinks the window, and adds `right - left + 1` per right endpoint — every accumulator a `long long`.
