The graph is undirected and unweighted and I want the shortest hop-distance from a single source `s` to every node, then the maximum over distance-layers of the summed brightness. The moment I read "shortest hop-distance" the algorithm is settled — that is unweighted single-source shortest path, which a breadth-first search computes directly. What makes this more than a five-line exercise are three traps the constraints stack around that BFS, and I want all three pinned before I touch the accumulation loop.

First, integer width. `n, m <= 2*10^5` and `|w[i]| <= 10^9`. A single layer can hold nearly every node — a star with `s` at the center puts all `n-1` leaves at distance `1` — so one layer-sum can reach `2*10^5 * 10^9 = 2*10^14`, roughly a hundred times past the 32-bit ceiling near `2.1*10^9`. Every brightness, every per-layer accumulator, and the running maximum must be 64-bit. An `int` here is not a crash; it is a silent wrong answer on the large tests, which is worse.

Second, the answer is not floored at `0`. Brightnesses may be entirely negative, and layer `0` always exists because it holds `s`, so the brightest layer can itself be negative. The running maximum must therefore be seeded below everything attainable — `LLONG_MIN`, or any sentinel under `-2*10^14` — never `0`. Seed it at `0` and every all-negative graph silently returns `0`, inventing a "broadcast nothing, score 0" option that the problem does not offer: `s` is always lit, so there is always a layer to report.

Third, unreachable nodes. If I mark unreachable with `dist[v] = -1`, then the natural accumulation "for every node `v`, add `w[v]` into `layerSum[dist[v]]`" indexes the bucket array at `-1` for any isolated node — out of bounds, undefined behaviour, and wrong even if it doesn't fault, since unreachable nodes belong to no layer and must contribute to none. The accumulation must guard on `dist[v] != -1`.

So the shape is fixed before I write it: BFS from `s` giving `dist[v]` (`-1` if unreachable), a bucket `layerSum[d]` fed only by reachable nodes, and `best = max over d of layerSum[d]` seeded at `LLONG_MIN`.

The BFS itself is standard: `dist[s] = 0`, push `s`, and on popping `u` relax each neighbour `v` with `dist[v] == -1` to `dist[u] + 1`. Because edges are unit-weight and BFS expands in nondecreasing distance order, the first time a node is labeled is along a shortest path — the property I lean on for correctness. The labels come out contiguous: if some node sits at distance `d`, it was reached from a node at distance `d-1`, which therefore exists, so `0, 1, ..., maxDist` has no gaps and every index is a real, non-empty layer.

That contiguity is what lets me refuse to detect "layer non-empty" from `layerSum[d] != 0`. A layer can legitimately sum to `0` — a lone zero-brightness node, or `+4` and `-4` sitting at the same distance. Emptiness is a property of the distance grouping, not of the sum, so I iterate the whole `0..maxDist` range and fold every bucket in, zeros included. Gating on `!= 0` would drop a single-zero-node layer, and on `n=1, w=[0]` it would fall through to print the sentinel.

Tracing the given sample: from `s=1`, `dist = [_,0,1,1,2,3,4]`, so the layers sum to `3, -5, 7, 2, -1` and the maximum is `7` as stated. But the sample cannot catch the base-case trap — layer `2` is `+7`, so even a `best=0` seed returns `7` by luck. The trap only surfaces on a graph whose every layer is negative, which the sample is not, so the negative case has to be constructed by hand.

Take the case that hits both remaining traps at once: `n=4, m=2, s=1`, `w = [_,-2,-9,-1,-4]`, edges `1-2, 1-3`, with node `4` isolated. BFS gives `dist = [_,0,1,1,-1]`, `maxDist = 1`. The guarded accumulation skips `v=4` — without the guard, `layerSum[-1] += -4` would write one slot before the array — leaving `layerSum = [-2, -10]`. The `LLONG_MIN`-seeded maximum then returns `-2`, layer `0` where `s` sits alone, whereas a `0` seed would print `0`. The index guard and the sentinel are independent defects that happen to live at the same negatives/zeros/unreachable corner, so this one graph exercises both.

The zero-node confirmation: `n=1, m=0, w=[0]` gives `layerSum = [0]`, and `max(LLONG_MIN, 0) = 0` — correct, and precisely the case a `!= 0` emptiness gate would break by printing the sentinel.

One last thing to settle about the sentinel itself: `LLONG_MIN` is only ever read inside `max`, never accumulated into, and layer `0`'s real value always overwrites it before output — so it can neither underflow nor reach stdout, which is the standard failure mode of an under-thought sentinel.

For coverage past the hand-traced cases I differential-test against a brute that computes distances by repeated unit-edge relaxation — a mechanism distinct from a queue-driven BFS, so a shared bug is unlikely — grouping reachable nodes and maximizing layer sums over many small random graphs skewed toward negatives, zeros, all-negative regimes, and sparse graphs with unreachable nodes and random sources. The two agree throughout, and a clear majority of the answers come out negative, so the sign corner is genuinely stressed and a `best=0` variant would fail a large fraction of them.

That fixes the whole program: an `O(n + m)` BFS filling `dist`, a per-distance bucket fed only by reachable nodes, `long long` everywhere, and the maximum over `0..maxDist` seeded at `LLONG_MIN`. The full module is in the answer.
