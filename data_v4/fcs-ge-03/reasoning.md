Two things about this closest-pair instance decide the design before I pick an algorithm. First, the answer is the *squared* distance, an integer — so every comparison can stay in exact integer arithmetic and the judge needs no floating tolerance. Second, the scale: coordinates reach `+-10^9`, so a coordinate difference reaches `2*10^9` and a squared distance reaches `(2*10^9)^2 + (2*10^9)^2 = 8*10^18`. That sits just under the signed 64-bit ceiling of `~9.2*10^18` — it fits, but with under one order of magnitude of headroom, and it overflows 32-bit by nine orders. So coordinates, differences, and squared distances are all `long long`; an `int` anywhere in the distance computation is a silent wrong answer on the large-coordinate tests. And if `n < 2` there is no pair: print `-1`, an unambiguous sentinel since real squared distances are `>= 0`.

With `n` up to `2*10^5`, the brute-force `O(n^2)` double loop is `~2*10^10` distance checks — twenty-plus seconds, an order of magnitude over the two-second limit. It survives only as a trusted oracle on small `n`. I need `O(n log n)`, and two methods are standard:

- *Divide and conquer.* Sort by `x`, split at the median, recurse on each half to get its best `delta`, then merge across a vertical strip of width `2*delta` sorted by `y`, where each point compares against a constant number of `y`-neighbours. `O(n log n)`, but the strip merge is where off-by-ones live — threading a `y`-order through the recursion, getting the strip filter and the comparison window right.
- *Sweepline with an ordered active set.* Sweep left to right by `x`, hold the recently-seen points in a `std::set` keyed by `y`, and for each new point query only the `y`-band within the current best distance. Also `O(n log n)`, and the bookkeeping is two clean moves — evict-by-`x`, range-query-by-`y` — with no fragile merge.

That decides it: the sweepline is the simpler `O(n log n)` to get exactly right here.

The correctness-and-speed argument for the sweepline: suppose my best squared distance so far is `best`, and `d = sqrt(best)`. Processing in increasing `x`, when I reach `p_i` any earlier `p_j` that could improve on `best` has `|x_i - x_j| < d` and `|y_i - y_j| < d`, so it lives in a box of width `d` behind the sweep and height `2d` centred on `y_i`. The counting fact that makes this fast: at most a constant number of points can pack into a `d x 2d` region while keeping all pairwise distances `>= d`, and every point currently in the active set is already `>= d` apart from the others (otherwise `best` would be smaller) — so the `y`-band query returns `O(1)` points amortized. The engine per point is: evict points whose `x` is more than `d` behind, query the `y`-band `[y_i - d, y_i + d]`, compare against the `O(1)` survivors, update `best`, insert `p_i`. `n` set operations, `O(n log n)` total.

I key the set on `(y, x)` so a `lower_bound`/`upper_bound` on `y` extracts the band directly. Eviction uses a `left` pointer into the `x`-sorted array: `p[left..i-1]` are the inserted points, and I advance `left` (erasing) while the front point's `x`-gap satisfies `dx*dx >= best`. Because the array is `x`-sorted the front point has the largest gap, so the first survivor lets me `break`. I keep this test in squared form — `dx*dx >= best` — so eviction stays in exact integers and needs no `sqrt` at all.

The one place I genuinely need a real `d` is the `y`-band: `lower_bound`/`upper_bound` compare against integer `y` values, so `d` has to be a concrete integer, and the band `[y_i - d, y_i + d]` must not drop any point that could still improve `best` (a point with `dy^2 < best`). The delicate spot is computing that integer at scale. `best` can be `8*10^18`, well past `2^53` where a `double` can no longer represent every integer, so `(long long)sqrt((double)best)` is not trustworthy: the rounded `double` can land below the true integer square root, and then `d` is one (or more) too small, the band comes out too narrow, and it can exclude the genuinely closest earlier point — a silent too-large answer. So I pin `d` to the exact integer square root of `best` rather than trust the float. Starting from `d = ceil(sqrt((double)best))`, `while (d*d < best) d++;` guarantees `d*d >= best` and `while (d > 0 && (d-1)*(d-1) >= best) d--;` makes `d` the smallest such integer — exactly `ceil(sqrt(best))`, with no floating slack. Biasing toward the up-correction is deliberate: a band one too wide costs a few extra comparisons, a band one too narrow loses the answer.

One wrinkle before the band path can run: at the start `best = LLONG_MAX`, `sqrt(LLONG_MAX)` is meaningless, and the band would be the whole set anyway. So I special-case `best == LLONG_MAX` — scan whatever is in the window, which is at most one point at that stage — until `best` becomes finite and the banded path takes over. No `sqrt(LLONG_MAX)` is ever computed.

The `x`-eviction relies on the array being `x`-sorted, so a vertical line — every point sharing one `x` — is its adversary: `dx = 0` for every pair, the prune test `0 >= best` never fires, and the active set holds all `n` points at once. It still runs fast, because the speed guarantee comes from the `y`-band, not the eviction: once `best` shrinks to the minimum `y`-gap, the band around each new point contains only `O(1)` points, so each query is cheap even against a huge set. The eviction is an optimization for the common case; the `y`-band is what bounds the work, and it degrades gracefully on its own worst input.

The corners: `n = 0` / `n = 1` short-circuit to `-1`. Coincident points give `dx = dy = 0`, squared distance `0`; the subtle part is that `std::set` deduplicates identical `(y, x)` keys, so inserting a second identical point is a no-op — but that costs nothing, because the duplicate's distance `0` is found the moment the second point is processed (the first is still in the set and the band sees it), and once `best = 0` the `d`-correction collapses the band to `[y_i, y_i]` and nothing can beat it. Collinear-horizontal (shared `y`) is carried by `x`-eviction with a single-value band. The extreme `(-10^9,-10^9)`–`(10^9,10^9)` returns `8*10^18` exactly; the `LLONG_MIN`/`LLONG_MAX` values appear only as set-key extremes in the band bounds, never inside arithmetic.

I check the sweep against an independent `O(n^2)` brute force over a randomized generator spanning the four regimes that matter here — tiny range with many ties, medium range, extremely dense for duplicates, and full `+-10^9` for overflow stress — plus the explicit edge cases above, and against the `2*10^5` adversaries (random, horizontal line, vertical line, grid). The sample `(0,0),(7,6),(3,4),(1,1),(8,7),(10,2)` yields `2`, the closest pair `(7,6)`–`(8,7)`.

So the sweepline ships: an ordered-set window of width equal to the current best, eviction in exact integer `dx*dx >= best`, the `y`-band half-width `d` pinned to an exact integer `ceil(sqrt(best))`, and `long long` throughout. The full program is the answer.
