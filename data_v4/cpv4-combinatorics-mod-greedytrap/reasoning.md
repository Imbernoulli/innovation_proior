The scale is what forces the whole design here. I have `q <= 2*10^5` scenarios, each asking for the number of safe reveal orders of `a` A-ballots and `b` B-ballots with `a, b <= 10^6` and `a + b <= 2*10^6`. An order is *safe at margin `m`* if, reading left to right, the running `lead = (#A) - (#B)` never drops below `-m` (B never leads by more than `m`). A per-query `O(a + b)` scan would be `2*10^5 * 2*10^6 = 4*10^11` operations — hopeless — so after one shared precompute each query has to be essentially `O(1)`, which points straight at a closed-form binomial count rather than any per-order walk. The modulus is a prime `p`, and the constraint hands me `p > 2*10^6 >= a + b`: that means every factorial `0!, ..., (a+b)!` is nonzero mod `p`, so Fermat inverses exist and `C(n, k) mod p` is `O(1)` after an `O(maxN)` factorial table. One thing to fix before touching data types: modular products reach `(p-1)^2 ~ 4*10^18`, and chained `a*b%p*c` multiplies clear the `~9.2*10^18` signed-64 ceiling, so every modular multiply routes through `__int128`.

The clean way to think about "B never leads by more than `m`" is a lattice path. Map each A-ballot to a `+1` step and each B-ballot to a `-1` step. A reveal order is a path from height `0` with `a` up-steps and `b` down-steps, ending at height `a - b`, and "after every reveal `lead >= -m`" is exactly "the path never dips below height `-m`". So I am counting lattice paths that stay `>= -m` at every prefix; a path is *unsafe* precisely when it touches the barrier line `y = -m - 1`. That reframing unlocks two candidate methods.

The first is a *local-multiply greedy*: insert the `b` B-ballots one at a time, each into the earliest position the margin still allows, multiply the number of legal slots each B has, then divide by `b!` to undo the ordering of identical B's. It is `O(b)` per query and feels like the staircase product that counts one-sided threshold matchings. The structural risk is whether the per-B slot sets are really independent — placing one B early might remove a slot a later B was counting on. The second is *reflection* (Andre / cycle lemma): count all `C(a + b, a)` interleavings and subtract the unsafe ones, which the reflection principle collapses to a single binomial. That is `O(1)` per query after the factorial table; its only risk is transcribing the reflected index and feasibility guard without a sign or off-by-one slip.

Before committing I try to break the greedy on the smallest scenario where the constraint bites: `a = 3, b = 2, m = 0`, barrier `y = -1`, so a safe path keeps `lead >= 0` everywhere. The greedy reasons that the `j`-th B (left-to-right) needs `j` A's before it, so it can occupy 0-indexed positions `>= 2j - 1`, giving `n - (2j - 1) - m` legal slots with `n = 5`. `B#1`: `5 - 1 - 0 = 4` slots (positions 1..4); `B#2`: `5 - 3 - 0 = 2` slots (positions 3..4). Product `4 * 2 = 8`, divided by `2!`, is `4`. Enumerating the truth by hand, the safe strings are `AAABB, AABAB, AABBA, ABAAB, ABABA` — **5**, the Catalan-style ballot number. They disagree, and the *why* is diagnostic: the 6 ordered placements `(p1, p2)` with `p1 in {1,2,3,4}`, `p2 in {3,4}`, `p1 != p2` include `(3,4)` and `(4,3)`, which are the same set `{3,4}` = `AAABB` — they collide. The slot ranges `{1,2,3,4}` and `{3,4}` overlap, so the `b!` division assumes a symmetry that isn't there. It is not off by a fixable constant; the independence premise is false. Greedy is out.

For the reflection, I want the number of paths with `a` up, `b` down, staying `>= -m` at every prefix. Total ignoring the barrier: `C(a + b, a)`. To subtract the unsafe paths, take an unsafe path, find its first contact with `y = -m - 1`, and reflect the portion before that contact across the line. Reflecting flips the start point from `0` to `2(-m - 1) = -2(m+1)`. A path from `-2(m+1)` to `a - b` in `n = a + b` steps has up-steps `u` with `2u - n = (a - b) + 2(m+1)`, i.e. `u = a + m + 1`, so the unsafe count is `C(n, a + m + 1) = C(n, b - m - 1)`. Hence

`safe(a, b, m) = C(a + b, a) - C(a + b, b - m - 1)`,

with the convention `C(n, k) = 0` for `k < 0` or `k > n`, so the subtracted term vanishes when `b - m - 1 < 0`, i.e. `m >= b`, the unconstrained regime. One more guard: if the path cannot even end at or above the barrier — `a - b < -m` — no safe path exists, and I short-circuit to `0` rather than push the formula into a regime I never re-derived. On the sample `a = 3, b = 2, m = 0` the formula gives `C(5,3) - C(5,1) = 10 - 5 = 5` — the truth the greedy missed — and `a = 4, b = 4, m = 0` gives `C(8,4) - C(8,3) = 70 - 56 = 14`, the fourth Catalan number.

Now the code, where the reflected index legitimately goes out of range, so my first unguarded binomial is a landmine:

```
ll C(ll n, ll k) {                                     // no range guard yet
    return (__int128)fct[n] * inv_fct[k] % MOD * inv_fct[n-k] % MOD;
}
```

Trace `a = 5, b = 0, m = 0` (only A's, safe answer obviously `1`): `total = C(5,5) = 1`, but `bad index = b - m - 1 = -1`, so `C(5, -1)` reads `inv_fct[-1]` — a negative `vector` index, undefined behaviour that returns junk and gives `safe = 1 - junk`. The fix is exactly the `C(n,k) = 0` convention the derivation already assumed; I just have to encode it:

```
ll C(ll n, ll k) {
    if (k < 0 || k > n) return 0;
    return (__int128)fct[n] * inv_fct[k] % MOD * inv_fct[n-k] % MOD;
}
```

Re-tracing, `bad = C(5, -1) = 0`, `safe = 1`; and `a = 2, b = 5, m = 10` (B dominates but the barrier `y = -11` is unreachable in 7 steps) gives `bad index = -6`, `bad = 0`, `safe = C(7,2) = 21` — all orders safe, correct.

A second failure only surfaced when I wired up an independent brute force (enumerate every interleaving, check the running balance) against a small-case generator and ran a few hundred comparisons: a wall of mismatches. The first was `p = 3`, query `a=4, b=1, m=3`: brute says `2`, mine says `0`. Here `safe = C(5,4) mod 3`, and `C(5,4) = 5`, so the answer is `5 mod 3 = 2` — yet I printed `0`. Printing `fct` mod 3 shows why: `fct = [1,1,2,0,0,0]`, because at `i = 3`, `fct[3] = fct[2]*3 mod 3 = 0`, and every later factorial and every Fermat inverse is then `0`. The Fermat-factorial method is *fundamentally invalid* when `p <= n`, since `p | n!` and `0` has no inverse. This is a modeling boundary, not a coding slip. Lucas' theorem would generalize it, but that costs `O(log_p n)` per binomial and buys generality the problem never asks for: the contract guarantees `p > 2*10^6 >= a + b`, so no factorial in range is ever divisible by `p` and the simple `O(1)` method is exactly correct. The honest resolution is to hold the stress-test to that same contract — the smallest prime I feed it is `17`, above any tiny `a + b <= 16` — rather than bolt on Lucas and slow the `2*10^5`-query batch. With that, the harness goes green: 900 random small cases across `m = 0`, large `m`, `a < b`, `a = b`, and empty scenarios, 0 mismatches, plus a full-scale check against exact Python binomials. Because the brute enumerates and re-checks every order by a completely different route than the reflection identity, that agreement is real evidence the reflected index and both guards are right.

The boundaries all fall out of the same guards. `a = b = 0` gives `maxn = 0`; `build(0)` produces `fct = [1]`, `inv_fct[0] = 1`, and `safe = C(0,0) = 1` — the one empty order. `b = 0` gives `bad index = -m - 1 < 0`, `safe = C(a,a) = 1`. The reflected index crossing `m >= b` is continuous: `m = b - 1` gives `bad = C(n,0) = 1`, a real subtraction; `m = b` gives `bad = 0` via the guard. Two `long` tables of size `maxn + 1 <= 2*10^6 + 1` are ~32 MB, inside 256 MB, and the subtraction `total - bad` can land in `[-(p-1), 0)`, so I add `MOD` back when negative; a full run finishes in about 0.1 s.

That is the shippable solution: the reflection `safe = C(a+b, a) - C(a+b, b-m-1)` with the `C(n,k)=0` guard, the `a - b < -m` short-circuit, and `__int128` products throughout, all `O(maxN + q)`. The full program is in the answer.
