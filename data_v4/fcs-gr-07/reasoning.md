**The scale of `t` is the whole problem.** A query gives me a start `s` and a step count `t` up to `10^18`, and asks for `f^t(s)` — the node reached after following the single out-edge `t` times. I cannot simulate `10^18` steps, and with `n, q` up to `2*10^5` I cannot afford a per-query structure that is heavy in memory. The first concrete consequence is the data type: `10^18` overflows 32-bit by a wide margin and sits near the top of signed 64-bit (`~9.2*10^18`), so `t` and every quantity derived from it — `t` minus a tail depth, a remainder — lives in `long long`. Node labels are `< 2*10^5` and stay `int`. Keeping that split clean is not pedantry: mixing an `int` step count into a subtraction is exactly how a silent wrong answer appears only on the `t = 10^18` tests, which are precisely the ones I cannot eyeball.

**Out-degree one forces a rho shape.** Following edges from any node is a deterministic walk on a finite set, so it must revisit a node and from the first revisit on loops forever with a fixed period. Every component is therefore a **rho (ρ)**: a forest of tails (trees whose edges all point inward) hanging off a single directed cycle. From any start I walk `d >= 0` steps down a tail to the cycle, then circulate with period `L`. This is forced, not heuristic, and it is the entire lever for huge `t`: the walk is ultimately periodic, so `f^t(s)` depends on `t` only through "how far down the tail" and then "where on the cycle mod `L`".

**Binary lifting is correct but the wrong default, and the numbers say why.** The textbook tool is binary lifting: precompute `up[k][v] = f^{2^k}(v)` for `k = 0..59` (since `2^60 > 10^18`) and follow the binary digits of `t`, `O(log t)` jumps per query. The table is `n * 60 = 1.2*10^7` entries — 48 MB at 4 bytes each, ~96 MB if I slip and make them 64-bit, uncomfortably close to the 256 MB cap once I/O buffers are counted. The query loop scatters `q * 60 = 1.2*10^7` reads across that 48 MB, which is cache-hostile. It works, but it spends a `log t` factor in both memory and time to brute-force a structure I already know is almost entirely periodic — and circulating a cycle is `O(1)` modular arithmetic, not `O(log t)`. Paying 60× the memory to ignore the periodicity out-degree-one hands me is the tell. I design directly against the rho structure.

**Rho decomposition splits every query into two regimes.** Let `depth[s]` be the steps from `s` to its cycle entry `entry[s]`, and `L` the cycle length.

- `t >= depth[s]`: I spend `depth[s]` steps to reach `entry[s]`, and the remaining `t - depth[s]` steps circulate. The answer is the cycle node at `(pos(entry[s]) + (t - depth[s])) mod L`. One modular step, `O(1)`; `t - depth[s]` stays in `long long` and reduces mod `L` immediately, so nothing overflows. Every deep query collapses to one `mod`.
- `t < depth[s]`: the answer is still in the tail, the node exactly `t` steps down from `s`. A naive walk is `O(depth)`, and `depth` can be `Θ(n)` (a single chain of `2*10^5` into a self-loop); `q` such queries would be `Θ(nq) = 4*10^{10}`, far too slow. So the tail also needs an `O(1)` jump — but not a 60-level lifting table; I want memory linear in `n`.

**A tail jump is a k-th-ancestor query, and a DFS stack answers it for free.** Reverse the edges: `u -> f(u)` reversed makes `f(u)` the parent of `u`, cycle nodes are the roots, and the tails form a forest. "Go `t` steps forward down the tail" is "go to the `t`-th ancestor". The clean linear trick avoids any precomputed table: during a depth-first traversal, the live recursion stack *is* the root-to-current path. If the stack is `stk[0..D]` while visiting a node at tail-depth `D` (`stk[D]` the node, `stk[0]` its cycle root), then the node `t` steps up is just `stk[D - t]` — one array index. The catch is that this holds only while the DFS stands on the node, so I answer tail queries **offline**: bucket each query by its start node, run one DFS over the forest, and resolve each node's queries against the live stack as I arrive. Everything becomes `O(n + q)` time and memory — a handful of length-`n` `int` arrays, no `60 * n` table, no `log t` anywhere.

**Finding the cycles.** Standard three-state functional-graph traversal: `state[v]` is `0` unvisited, `1` on the current walking path, `2` finished. From an unvisited `s` I walk forward marking `1` and pushing onto a path stack until I step onto a seen node `v`. If `state[v] == 1`, `v` is on my current path and I have just closed a new cycle running from `v` to the top of the path; I pop the tail nodes above `v`, then walk `f` from `v` back to `v` collecting the cycle, assigning each node a cycle id, a position, and recording its length. If `state[v] == 2`, I walked into finished territory and there is no new cycle. Every node is pushed and popped once, so this is linear.

**The DFS must be iterative, and one ordering is load-bearing.** A chain of `2*10^5` would overflow the native call stack, so I keep an explicit stack `stk` and a per-node child iterator `itChild[v]` walking the reverse-adjacency. Cycle nodes seed the DFS as roots at depth `0`. The one subtlety is the termination test: I pop `v`, *then* check `if (v == root) break`. If I tested before popping I would leave the root on the stack and re-enter it with a stale iterator; pop-then-test leaves the stack empty and stops cleanly.

**The real trap: what the cycle entry is.** The cyclic branch needs `entry[s]` to index into the right cycle. It is tempting to store a per-node `entry` array — and easy to leave it uninitialized as `0`. But every node reached in a given DFS subtree provably enters the cycle at that tree's single root, so the entry *is* `stk[0]`, the `root` of the outer loop; there is no separate lookup to fill. This is not cosmetic. On `f = [1, 2, 0, 2]` (3-cycle `0->1->2->0`, plus `3 -> 2`), the query `(3, 1)` should give `f^1(3) = 2`; node `3` has `depth 1`, `entry = 2`. An `entry` defaulting to `0` computes `(cycPos[0] + 0) mod 3 = 0` and answers `0` instead of `2`. Passing `entry = root` makes it `(cycPos[2] + 0) mod 3 = 2`. Correct. I route the boundary `t == depth` through the tail branch (`t <= D` gives `stk[D - t] = stk[0] = entry`, needing no cycle lookup) so the cyclic branch always sees `into = t - D >= 1`; both formulations agree there, and keeping the boundary in the stack branch is the more robust of the two.

**Corners that actually differ.** `t = 0` resolves through the tail branch as `stk[D - 0] = stk[D] = s` itself, matching `f^0(s) = s` — the one place the I/O contract is unusual. A self-loop (`n = 1`, `f = [0]`) makes `L = 1`, so `(pos + into) mod 1 = 0` sends everything to node `0`. A query on a cycle node is `depth 0`, a pure modular jump. The overflow surface is exactly `into = t - D` with `D < 2*10^5`, which stays positive in `long long` and reduces mod `L` at once. The deep chain reads `stk[D - t]` in `O(1)` with no per-query walk and no recursion, so neither the call stack nor an `O(nq)` loop can blow up.

**Checking against an oracle.** I run the implementation against a brute-force oracle — single-step simulation with a period shortcut for huge `t` — over randomized small instances (self-loops, full cycles, long chains, dense random functional graphs, `t` from tiny up to `10^18`) plus the hand-built cases above; it agrees everywhere. At full scale (`n = q = 2*10^5`, worst-case deep chain, all `t = 10^18`) it runs well under 0.1 s using ~16 MB, against the ~48 MB a binary-lifting table would cost — the rho decomposition pays off exactly where the memory argument said it would.
