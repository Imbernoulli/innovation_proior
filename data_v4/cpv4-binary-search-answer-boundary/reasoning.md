`n` flower beds sit in a row; bed `i` blooms on day `b[i]` and never un-blooms. I want the earliest day `t` on which the row yields at least `m` bouquets, where a bouquet is `k` consecutive simultaneously-bloomed beds and a maximal bloomed run of length `L` gives `floor(L/k)` of them. Input is `n m k` then the `n` bloom days; output is the day or `-1`. What lifts this above a plain scan is packed into the constraint sizes, and two of them are arithmetic traps. `m` reaches `10^9` while `k <= n <= 2*10^5`, so `m*k` reaches `10^9 * 2*10^5 = 2*10^14` — I need that product for the impossibility test `m*k > n`, and it overshoots 32-bit badly, so I form it in `__int128`. And the bloom days go to `10^9`, so a binary search over `[min b, max b]` must never compute `lo + hi` (up to `2*10^9`, past signed 32-bit) — the midpoint has to be `lo + (hi - lo)/2`. Everything stays in `long long`. But the two hazards this problem is really built around are boundaries, not overflow: what "bloomed by day `t`" means, inclusive or exclusive, and which half a binary-search step keeps.

Feasibility is monotone in the day: no bed ever un-blooms, so any set of bouquets assemblable on day `t` is still assemblable on day `t+1`. Feasibility runs `false...false true...true`, and the answer is the single lower-bound transition — that is exactly what binary-search-on-the-answer is for. The naive alternative, scanning every day from `min b` upward until one works, is `O(range * n)` with `range` up to `10^9`, hopeless; but it is obviously correct, so it stays as my independent oracle. Binary-searching the day with an `O(n)` feasibility scan inside is `O(n log range)`, about 30 scans of `2*10^5` beds, comfortably under a second.

The feasibility scan: fix day `t`, sweep left to right with `run` = length of the current bloomed streak and a `bouquets` accumulator. For a bloomed bed increment `run`, and each time `run` hits `k` count a bouquet and reset `run = 0` (those `k` beds are consumed and the next bouquet starts fresh); a non-bloomed bed resets `run = 0`. Inside a maximal run of length `L`, `run` reaches `k` exactly `floor(L/k)` times, so the scan sums `floor(L/k)` over runs — precisely the quantity defined. I stop the moment `bouquets >= m`; that early exit matters because `m` is up to `10^9`, though the accumulator is bounded by `floor(n/k) <= n` regardless.

The predicate "bed `i` is bloomed on day `t`" is `b[i] <= t`: the bed blooms *on* day `b[i]`, so it is usable when `t = b[i]`. The trap is `b[i] < t`, which delays every bed's usability by a day and inflates the answer. That failure is concrete on the sample `n=7, m=2, k=2, b=[3,1,4,1,5,9,2]`, whose answer is `4`. On day 4 with `<=`, beds `0,1,2,3` (days `3,1,4,1`) form a run of length 4 giving `floor(4/2) = 2` bouquets — feasible. With `<`, bed 2 (bloom day 4) is dropped because `4 < 4` is false, the run breaks after two beds, day 4 yields only one bouquet, and the earliest feasible day slides to `5`. So `<=` is not a "loosen it to be safe" hack; it is the literal semantics, and slipping to `<` is exactly the off-by-one this row of beds punishes.

The search range: between consecutive bloom days the bloomed set is constant, so `feasible` only changes on days some bed newly blooms; the earliest feasible day is therefore one of the `b[i]`, and `[lo, hi] = [min b, max b]` brackets it. Below `min b` nothing is bloomed (0 bouquets, and `m >= 1`); on `max b` the whole row is one run of length `n`. An answer exists iff that full run suffices, `floor(n/k) >= m`, which holds iff `n >= m*k`. So I test `m*k > n` up front and print `-1`; otherwise `feasible(hi)` is guaranteed true and the search is well-posed with no sentinel.

The lower-bound loop is `while (L < R) { mid = L + (R-L)/2; if (feasible(mid)) R = mid; else L = mid+1; }`, then print `L`. Keep the candidate when feasible (`mid` itself may be the earliest), discard it when not (the earliest is then strictly greater). It converges because each step strictly shrinks `R - L`: on the feasible branch `mid <= R-1` whenever `L < R`, so `R` drops; on the infeasible branch `L` rises. The other boundary trap lives right here — `R = mid-1` or `L = mid` would over- or under-shoot. The decisive step on the sample is `mid=3`: bloomed beds are `0,1,3,6` (days `3,1,1,2`) with only the `0,1` pair adjacent, one bouquet, infeasible, so `L = mid+1 = 4`; with `feasible(4)` true (the run of 4) the loop settles on `4` rather than sliding past it. That is where `L = mid+1`, not `L = mid`, earns its exact form.

Edge cases where boundary code dies:
- *Impossible, `m*k > n`.* Take `n=7, m=5, k=2`: `m*k=10 > 7`, print `-1`. Without the up-front guard, `feasible(hi)` would be false and the loop would return `hi` — a wrong, non-`-1` answer, so the guard is load-bearing, and it must use `__int128` because `m*k` reaches `2*10^14`.
- *`k=1`.* Every bloomed bed is its own bouquet, so the answer is the `m`-th smallest bloom day; `run==k` fires and resets on each bloomed bed, which is correct.
- *Broken run.* `b=[1,9,1], k=3, m=1` needs all three adjacent beds bloomed at once, so the middle bed forces the answer to day 9 — the answer genuinely depends on adjacency, not the `m`-th smallest day, which is why a feasibility scan and not a sort is required.
- *Answer at the left edge.* `b=[1,5,1,1], k=2, m=1` is already feasible on day 1 (beds 2,3 adjacent), and the loop settles on `lo` without ever probing below it — the left-edge termination I most worried about. Single-bed and all-same-day inputs are the trivial `lo=hi` case where the loop body never runs.

Finally I cross-check against the brute day-by-day oracle, which sorts the distinct bloom days and returns the first feasible one — a completely different control structure, so agreement is real evidence and not a tautology. On randomized small cases with deliberately tight day ranges (to force many tied bloom days and exercise the `<=` boundary on every run) and `m` chosen to straddle feasibility (so the `-1` branch fires often), the two agree; together with the hand traces above, that is what I trust.

So the file is one self-contained `O(n log range)` binary search on the day: the `__int128` impossibility guard, `[min b, max b]` as the search interval, the `b[i] <= t` inclusive feasibility scan, and the `while (L<R)` lower-bound loop with `mid = L+(R-L)/2` and `R = mid` / `L = mid+1`.
