This is weighted interval scheduling dressed as studio bookings: `n` up to `2*10^5` requests, each holding a half-open interval `[s_i, f_i)` and paying `p_i > 0`, and I accept a pairwise non-overlapping subset that maximizes total profit — the empty schedule allowed, so the answer never drops below `0`. What makes it treacherous is the profits: with equal weights a one-line greedy settles it, so before picking an algorithm I want to know which local rule the unequal weights are set up to punish. Half-open is the load-bearing detail — `[s_i, f_i)` and `[s_j, f_j)` are compatible exactly when `f_i <= s_j` or `f_j <= s_i`, so two bookings that merely touch at an endpoint (`f_i == s_j`) both fit, one ending the instant the next begins. That `<=` boundary has to hold everywhere I compare a finish against a start.

Scale fixes the data types before anything else. `n <= 2*10^5`, coordinates and profits up to `10^9`. A profit sum can reach `2*10^5 * 10^9 = 2*10^14`, three-plus orders of magnitude past the 32-bit ceiling of about `2.1*10^9`, so every accumulator that holds a profit must be 64-bit. An `int` there is a silent wrong-answer on the large tests, not a crash — the kind of bug that passes every small local case. Coordinates fit in 32 bits, but I read everything as `long long` to keep comparison and addition widths uniform.

Two routes are on the table. A **sweep-and-grab greedy** sorts by a single key and walks through accepting whatever is compatible with what is already held — `O(n log n)` and a handful of lines. A **sort-plus-DP** sorts by finishing time and, for each request, decides accept-vs-reject by pairing its profit with the best schedule that finishes before it starts, located by binary search — also `O(n log n)`. The greedy is far cheaper to write, but profit makes this a weighted problem and a single sort key is a purely local decision; that is exactly the configuration where greedy fails, and this problem is built to punish it. So I try to break both keys before trusting either.

Earliest-finishing-time is the famous optimal rule — for *counting* non-overlapping intervals, the unweighted problem. The moment profits differ it can lose: take `A = [0,1)` profit `1` and `B = [0,3)` profit `5`. Sorting by finish puts `A` (finishes at 1) before `B` (finishes at 3); greedy accepts `A` for `1`, then `B` overlaps `A` on `[0,1)` and is rejected — total `1`, against the reachable `5` from `B` alone. It grabbed the request that frees the resource soonest, but freeing the resource soonest is worthless when the thing it blocked pays five times as much. Largest-profit-first dies to the mirror image: `[0,4)` profit `10`, `[0,2)` profit `7`, `[2,4)` profit `7`. Profit-first takes the fat `10` and both lean intervals overlap it, so total `10` — but `[0,2)` and `[2,4)` touch at `2` and jointly pay `14`. One fat interval blocks two lean ones that beat it. Both keys fail for one root cause: a local accept commits the shared resource over a span whose global opportunity cost the greedy never measures. That kills the whole greedy family.

So I need to weigh accept against reject *with* the value of what each choice forecloses, and the clean way is to order the requests so that "everything before" collapses to a single number. Sort all requests by finishing time `f` ascending and relabel them `0..n-1`. Let `best[i]` be the maximum total profit using only the first `i` requests — those with the `i` smallest finishing times — with `best[0] = 0`. Extending to `best[i+1]`, request `i` is either rejected, contributing `best[i]`, or accepted, collecting `p_i`. When accepted, no earlier accepted request may overlap `[s_i, f_i)`; since I sorted by finishing time, every earlier request finishes at or before `f_i`, and the compatible ones are exactly those finishing `<= s_i`. Because finishing times are non-decreasing, those compatible predecessors form a *prefix* — there is a count `j` such that requests `0..j-1` all finish `<= s_i` and `j..i-1` do not — so the best I can pair with `i` is `best[j]`. Hence

`best[i+1] = max(best[i], best[j] + p_i)`,

with `j` the number of earlier requests whose finish is `<= s_i`, found by binary search on the sorted finishing-time array; the answer is `best[n]`. Note `<=`, not `<`, matching the half-open convention.

To exercise the recurrence end to end I build a six-request instance: `(0,5,30)`, `(0,2,20)`, `(2,4,6)`, `(4,7,20)`, `(5,9,25)`, `(7,9,8)`. Sorted by `f` they relabel to `(0,2,20)`, `(2,4,6)`, `(0,5,30)`, `(4,7,20)`, `(5,9,25)`, `(7,9,8)`, giving `F = [2,4,5,7,9,9]`. Walking `best` from `best[0]=0`: `(0,2,20)` has no compatible predecessor so `best[1]=20`; `(2,4,6)` reaches back to `best[1]` so `best[2]=26`; `(0,5,30)` has none compatible so `best[3]=max(26,30)=30`; `(4,7,20)` reaches `best[2]` so `best[4]=46`; `(5,9,25)` reaches `best[3]` so `best[5]=55`; `(7,9,8)` reaches `best[4]=46+8=54 < 55` so `best[6]=55`. The optimum `55` is the two requests `(0,5,30)` and `(5,9,25)`. Earliest-finish greedy on the same input grabs `(0,2),(2,4),(4,7),(7,9)` for `20+6+20+8=54` — one short, so the instance is itself a greedy trap.

The one place this transcribes into a bug is the binary-search predicate. A predecessor at position `k` is compatible with request `i` iff `F[k] <= S[i]`, and I want the count of the leading block where that holds — a `lower_bound`-style partition returning the first index where the predicate *fails*. The tempting default is the strict `F[k] < S[i]`, and that is wrong here for exactly the touching-endpoint reason: on the pair `(0,2,20)`, `(2,4,6)` the strict predicate treats the predecessor finishing at `2` as incompatible with the request starting at `2`, so the search lands at count `0` instead of `1`, `(2,4,6)` reaches back to `best[0]` instead of `best[1]`, and the code returns `20` instead of `26`. The predicate must be `<=`:

```
int lo = 0, hi = i;                 // search F[0..i-1]
while (lo < hi) {
    int mid = (lo + hi) / 2;
    if (F[mid] <= S[i]) lo = mid + 1;
    else hi = mid;
}
// lo = number of first-i requests with F <= S[i], a valid index into best
```

`lo` after the loop is the count of the compatible prefix, hence exactly the index to read `best[lo]`; since the search range is `F[0..i-1]`, `lo` lies in `[0,i]` and `best` (size `n+1`) always has that slot. The strictly-overlapping pair `(0,1,1)`, `(0,3,5)` confirms the other direction: the later request has `S=0`, `F[0]=1 <= 0` is false, so its overlapping predecessor is correctly *not* counted and the answer is `5`.

The corners fall out of the recurrence plus `best[0]=0`. With `n = 0` the loop never runs and I print `best[0]=0`; the read guard `if (!(cin >> n)) return 0;` also handles empty input. A single request reaches back to `best[0]` and collects its own profit. When all requests mutually overlap — every `S=0`, and no predecessor can finish `<= 0` since `s < f` — every search yields `lo=0`, so `best[i+1] = max(best[i], best[0]+p_i) = max(best[i], p_i)`, the running maximum single profit, which is the correct "pick one of a fully-overlapping clique." A touching chain `(0,1,1),(1,2,1),(2,3,1)` accepts each on top of the last through the `<=` predicate for the full sum `3`. Sorting indices by `f` alone is enough — correctness needs only that `F` is non-decreasing, not any tie-break — and `best` only ever accumulates real profits from a zero start, so the `long long` accumulator has no sentinel to corrupt and the `~2*10^14` maximum sits comfortably inside 64 bits.

That is the solution I ship: sort by finishing time, DP with the `<=` binary search, `long long` throughout — the full module is in the answer. I checked it against an exhaustive brute force over 601 random small instances (varied span and profit) with zero mismatches, and the `n = 2*10^5` worst case runs well under the time limit, one sort plus `n` binary searches at `O(n log n)`.
