The whole difficulty of this one lives at a single instant. Every car is a half-open interval `[s_i, e_i)` — present at `s_i`, gone by `e_i` — so a car that leaves at time `t` and a car that arrives at `t` reuse the same spot rather than compete for it. With `n` up to `2*10^5` and coordinates up to `10^9`, the algorithm is going to be an `O(n log n)` event sweep whatever I do; the only place it can go wrong is the coordinate where a departure and an arrival coincide, and the hidden tests are described as leaning hard on exactly that — intervals that merely touch (`e_i == s_j`), many intervals sharing one endpoint. The answer itself is at most `n`, so an `int` would hold the count, but coordinates near `10^9` make `long long` the safe habit at zero cost. One contract clause I can't let slip: a record with `s_i >= e_i` occupies no instant and must be ignored.

A difference array over coordinates would be simplest if coordinates were small, but at `10^9` it needs compression, and compression doesn't remove the boundary question — it just relabels it as "which compressed bucket does a departure fall into, relative to an arrival at the same value." The event sweep concentrates that decision into one inspectable place: the sort's tie-break. The danger is localized where I can stare at it.

Coverage `f(t) = #{i : s_i <= t < e_i}` is a step function: `+1` as `t` crosses an arrival (`s_i <= t` turns true at `t = s_i`), `-1` as `t` crosses a departure (`t < e_i` turns false at `t = e_i`), and I want its maximum. At a coordinate `t` where departures `e_j = t` and arrivals `s_k = t` collide, half-open semantics say the departing car is already gone (`t < e_j` is false) while the arriving car is present (`s_k <= t` is true) — so the departures must not be counted alongside the arrivals. In sweep terms: at a shared coordinate, apply every end (`-1`) before any start (`+1`). The opposite order momentarily lumps a leaving car with an arriving one, which is exactly the *closed*-interval answer `[s, e]`, off by one against the spec.

I lean on lexicographic `pair` comparison to enforce that for free. Encode each event as `(coordinate, type)` with `type 0 = end`, `type 1 = start`; sorting ascending then puts all ends before all starts at any shared coordinate. The encoding *is* the correctness argument, so the increment branch has to match it — `+1` on `type 1`, `-1` on `type 0`:

```
ev.push_back({s, 1});   // start -> +1
ev.push_back({e, 0});   // end   -> -1
sort(ev.begin(), ev.end());
for (auto &p : ev) { if (p.second == 1) cur += 1; else cur -= 1; best = max(best, cur); }
```

The trap this problem is built to punish is getting that encoding backwards (`start = 0, end = 1`), which sorts starts first at a tie and silently computes the closed-interval answer. The two-car case that separates the readings is the touching pair `[1,4)` and `[4,7)`: they share no instant, so the answer is `1`. With ends-first, coordinate `4` fires `(4,0)` before `(4,1)` — `cur` drops to `0` and climbs back to `1`, output `1`; starts-first would have peaked at `2`. I have to check the fix doesn't over-correct on a real overlap: `[1,5), [4,7)` sorts `(1,1)(4,1)(5,0)(7,0)`, giving `cur` `1,2,1,0` → `2`, still right. And on the documented sample `[1,4) [2,5) [5,7) [3,4) [4,6)`, two ends pile up at coordinate `4` and sort `(4,0)(4,0)` before the start `(4,1)`, so `cur` dips from `3` to `1` before rising — the peak is the real `3` at `t = 3` (cars `[1,4), [2,5), [3,4)`), never a spurious `4`.

The `s >= e` clause is not cosmetic. If I push `+1` at `s` and `-1` at `e` for a degenerate record, its end-event can fire with no matching live start ahead of it and drive `cur` negative — records `[5,2)` and `[0,3)` sort to `(0,1)(2,0)(3,0)(5,1)`, so `cur` runs `1,0,-1,0`. Here `best` stays `1` by luck, but a count that desyncs from "cars actually present" is how a wrong maximum could sneak onto a hidden test. Skipping any record with `s >= e` before emitting events keeps `cur` equal to the true occupancy throughout. The trivial seams then fall out: `n = 0` leaves the list empty and `best = 0`, a single car peaks at `1`, disjoint intervals rise and fall without coinciding, nested `[0,10) [2,8) [4,6)` peaks at `3` in the middle.

Reasoning on paper isn't the same as running the code, and the tie-break is subtle enough that I want an independent oracle before trusting it. So I check the sweep against a brute force with no events, no sort, no tie-break logic to share my bug: for each distinct start coordinate `t` (coverage can only rise at a start, so the maximum is attained at one), count intervals with `s <= t < e` by direct scan. Driven on inputs biased toward tiny coordinate ranges so endpoints collide constantly, with degenerate `s >= e` records mixed in — the exact regime where the inclusive/exclusive bug lives — the two agree everywhere, which is the confirmation the paper-traces couldn't give on their own. At `n = 2*10^5` the sort dominates and finishes well inside the 1 s limit; since `cur` moves by `±1` over at most `n` starts it never exceeds `2*10^5`, so `int` would hold it and `long long` is free margin.

That leaves the `O(n log n)` sweep of the opening: skip `s >= e`, push `{s,1}` and `{e,0}`, sort, sweep `+1` on `type 1` and `-1` on `type 0` tracking the maximum. Its entire correctness is the ends-before-starts tie-break; the full program is in the answer.
