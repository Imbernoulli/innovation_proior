Two features of this instance fix the whole shape of the solution before I pick an algorithm. First, the values may be negative or zero, so the answer floors at `0` via the empty window and I cannot assume a wider window is better — that is the whole difficulty, and it is what separates this from the plain "largest rectangle under a skyline." Second, the scale: `n` up to `2*10^5` with `|a[i]|` up to `10^9` means a positive window of `10^9`s scores `2*10^5 * 10^9 = 2*10^14`, roughly twenty thousand times past the `~2.1*10^9` reach of 32-bit. Everything that holds a score has to be 64-bit; an `int` product is a silent wrong answer on the large positive tests, not a crash I would catch. So `long long` for the values and for the running best.

The transparent baseline fixes each left endpoint and extends right while tracking the running minimum: `O(n^2)`, obviously correct, and the natural oracle to check against — but `~4*10^10` operations at the top size, hopeless under a one-second limit. The linear route is the monotonic stack: every window's minimum sits at some index, so if for each `i` I take the widest window in which `a[i]` is the minimum and score `a[i]` times that width, the maximum over `i` equals the maximum over all windows. The widths come from nearest-smaller-element on each side, one stack pass each.

The reason this identity is not just the histogram trick in disguise: in the all-non-negative case, "for each index take the widest window" works because widening only multiplies a non-negative minimum by a larger length. That reflex breaks the moment negatives enter — a window whose minimum is `-4` scores `-4 * width`, which grows *more* negative as it widens, so the widest window is the worst candidate. Even staying positive, widening can drag in a smaller minimum. So the claim is not "widest wins"; it is that scoring each index at *its own* maximal window and taking the global max recovers the true optimum. Because negatives make the widening intuition unreliable, this identity is the one place I will lean on the brute oracle rather than a monotonicity argument.

For index `i`, let `left[i]` be the nearest position on the left with value strictly less than `a[i]` (`-1` if none) and `right[i]` the nearest on the right with value `<= a[i]` (`n` if none); the maximal window in which `a[i]` is a minimum is the open interval `(left[i], right[i])`, of width `right[i] - left[i] - 1`. The one real subtlety is runs of equal values. If both boundaries stopped at the nearest smaller-*or-equal* element — both passes popping only on a strict `>` — then inside a run of equal minimums every element's window would be truncated at its equal neighbours and no index would ever see the full run: on `[3,3,3]` each `3` gets width 1, reporting max score `3` when the true answer is `min 3 * width 3 = 9`. To capture the full run exactly once I make the passes asymmetric: pop on `>=` in the left pass, so `left[i]` is the nearest *strictly* smaller element and the run extends left freely, and pop on `>` in the right pass, so `right[i]` stops at the nearest equal-or-smaller. Then the rightmost element of each maximal equal-run owns the whole run and the others stop short — every window counted, none doubled. On `[3,3,3]`: `left = [-1,-1,-1]`; the right pass pops on `>` only, giving `right = [1,2,3]`; widths `1,2,3`, scores `3,6,9`, max `9`, claimed by the rightmost `3`.

Running the convention across the given sample `[2,1,5,6,2,3]` gives per-index scores `2,6,10,6,8,3`, max `10` at index 2 (the window `[5,6]`), matching the expected answer — while index 1, the global minimum `1`, spans the entire array for only `6`: the concrete demonstration that a wider window loses.

The base case is where the negzero contract bites even with the widths right. If I seed `best` from a real window — say `best = a[0] * width_0` before the loop — I have asserted that some non-empty window *must* be chosen, but the contract always offers the empty window at `0`. On all-negative `[-3,-1,-4]` every non-empty score is negative (the widths give `-6, -1, -12`, so the best non-empty is the lone `-1`), and a seeded `best` returns `-1` while the answer must be `0`. Initializing `best = 0` and only ever raising it makes the empty window the floor and closes every degenerate corner at once: all-negative and single-negative return `0`, all-zero returns `0`, and `n = 0` (the loops never run) returns `0`. If the stream carries no token at all, `!(cin >> n)` prints `0` and exits.

`best`, the width, and the product are all `long long`, with the width cast to 64-bit before the multiply so it is never an `int * int` — the `2*10^14` top score fits with room. The whole thing is `O(n)` (two stack passes, one scoring pass) and `O(n)` space, trivially inside the 1s / 256MB budget at `n = 2*10^5`.

Run against the `O(n^2)` brute on random small arrays spanning all-negative, negatives-with-zeros, all-zero, repeated-equal, and full-range inputs, the linear solution agrees everywhere — closing the one gap where the widening intuition could not vouch for the maximal-window identity under negatives.
