I keep an undirected graph on `n` vertices, starting empty, and stream `q` operations in time order: add an edge, remove an edge, or ask whether `u` and `v` are connected at that instant. The scale is `n, q <= 2*10^5` in 2 seconds. The one fact that dictates everything is that edges can be *removed*: insert-only connectivity is the textbook union-find exercise, so if `remove` were absent I would be done in a line. Deletion is the entire problem, and I refuse to reach for a plain DSU and hope.

A disjoint-set union answers insert-only connectivity beautifully ŌĆö each add is a `unite`, each query two `find`s, near-constant amortized ŌĆö but it has no `delete`. Path compression rewires an unknown number of parent pointers straight to the root, so a merge leaves no record of the pre-merge shape to restore. And removing one edge from a component may split it in two or leave it whole, depending on whether other edges still bridge the sides; a compressed DSU can neither tell which nor reconstruct the split cheaply. Components are trivial to grow and effectively impossible to shrink. That asymmetry is the wall.

The obvious correct fallback is to keep the live edge set in a hash structure and, per query, BFS/DFS from `u` over the present edges. Unarguably correct, but it collapses on a path. Wire `n = 2*10^5` vertices into one line `1-2-...-n` (`n-1` adds), then fire ~`10^5` queries `3 1 n`; each walks the whole path to reach the far end, `O(n)`, so ~`2*10^{10}` traversals total ŌĆö tens of seconds. Recomputation survives only as the oracle I check the fast solution against; I need something sublinear per query.

The lever is that all `q` operations are known up front ŌĆö nothing forces me online. So I can look at an edge's whole life: an edge added at `t_add` and removed at `t_remove` is present for exactly the queries in `[t_add, t_remove - 1]`; if never removed, `[t_add, q-1]`. Every edge occupies one *contiguous time interval*. The problem turns from "a graph mutating unpredictably" into "a set of edges each labeled with a time interval; for each query-time, what graph do the covering intervals induce?"

That shape is what a segment tree over time exploits. Split `[0, q-1]`, recurse on halves. An edge whose interval fully spans the current segment is on for the whole segment and can be activated once at the top of that recursion instead of at every leaf inside it; an interval that only partially overlaps gets pushed into the halves it meets. This is the standard segment-tree interval decomposition: each edge interval lands at `O(log q)` canonical nodes, total storage `O(q log q)`. Then I DFS the tree ŌĆö entering a node, `unite` all its edges; at a leaf `t`, the DSU reflects exactly the edges alive at time `t`, so I answer that query with two `find`s; leaving the node, I **undo** exactly the unites done on entry so siblings see the correct state.

Which surfaces the crux: the DSU must support undo, and path compression makes that impossible ŌĆö a single `find` can flatten a long chain, mutating many pointers with no compact way to reverse them. So I drop compression. Union by size alone keeps tree height `O(log n)`: a node's depth grows only when the tree it lives in at least doubles, so `find` is an `O(log n)` walk. In exchange, each real `unite` changes exactly two things ŌĆö one parent pointer `par[b] = a` and one size `sz[a] += sz[b]` ŌĆö so I push one record and pop it to reverse both writes. Rollback is possible *precisely because* I gave up path compression: `O(log n)` finds bought in return for `O(1)` reversible unions.

Costing it out: `O(q log q)` stored (edge, node) pairs, each visited once as a `unite` and once as a rollback at `O(log n)`; each query `O(log n)`. Total `O((n + q log q) log n)`. At `n = q = 2*10^5`, `q log q ~ 3.5*10^6`, times `log n ~ 18` is ~`6*10^7` ŌĆö comfortably inside 2 s. So: segment tree on time + rollback DSU, union by size, no compression.

One detail in the DSU: when `unite(a, b)` finds the roots already equal it makes no structural change, but the DFS still issues one rollback per unite it called, so a no-op union must still push *something* or the stack desynchronizes from the call count. I push a sentinel `-1` for a no-op and the attached child root `b` for a real union; on rollback, `-1` does nothing, otherwise `a = par[b]` recovers the parent and I reverse `sz[a] -= sz[b]; par[b] = b`. Recovering `a` from `par[b]` is valid only because rollbacks run in strict LIFO order ŌĆö no later, not-yet-undone union could have re-parented `a` in between.

The undo bookkeeping has one ordering trap that this structure genuinely invites. I could roll back `seg[node].size()` times ŌĆö every `unite` pushes exactly one record, including the no-op sentinel, so the count is right ŌĆö but tying the undo count to the container is fragile. Snapshotting the stack size and rolling back until it returns is sturdier, and it forces the question of *when* to take the snapshot. It has to be *before* this node's unite loop, not after. Take it after and the snapshot already includes this node's merges, so on the way out the rollback stops with them still in place; a descendant covering a sibling time range where those edges are absent still finds their endpoints merged ŌĆö a ghost edge that leaks spurious `YES`s on churnier instances. Bracketing all the mutations means the save point precedes the first one. On the diamond (`n=4`; add `1-2,1-3,2-4,3-4`, remove `2-4`, query `1-4` at `t=5`) the live edges are `1-2,1-3,3-4`, so `1-3-4` joins `1` and `4` and the answer is `YES` ŌĆö which the snapshot-before form returns and the snapshot-after form corrupts once a neighboring range reuses vertex `1`.

The corners where fully-dynamic connectivity code tends to die:
- `q = 0`: nothing after `n`; I guard the DFS with `if (q > 0)` (a segment tree on an empty range is undefined) and size the vectors with `max(1, q)`.
- Query before any edge exists (e.g. `3 1 3` at `t=0`): no interval covers it, the leaf DSU is all-singletons, so `NO` unless `u == v`.
- Self-query `u == v`: both finds land on the same vertex -> `YES`, which is the contract.
- Edges never removed: they remain in `active` at the end and get inserted on `[l, q-1]`.
- Re-adding an edge after removing it: the `map` key `(u,v)` is reused with a fresh start time, yielding two disjoint intervals.
- Undirected-key normalization: `if (u > v) swap(u, v)` so a `remove 2 1` finds an edge added as `1 2`.
- The long path: `find` stays `O(log n)` under union by size even without compression, so the path query is `O(log n)`, not `O(n)`.

I compiled with `g++ -O2 -std=c++17` and differential-tested against an independent BFS oracle: over 1200 random operation streams (`n` up to 25, `q` up to 80, adds/removes/queries mixed under the edge-set invariants) plus the hand-built battery (`q=0`, query-before-add, self-query, add-remove-re-add, edge-alive-to-end, the diamond) ŌĆö zero mismatches, and the sample reproduces `YES YES NO NO`. At maximum scale with heavy churn it runs ~0.13 s and on the path worst case ~0.29 s, both well inside 2 s and consistent with the `O((n + q log q) log n)` analysis.
