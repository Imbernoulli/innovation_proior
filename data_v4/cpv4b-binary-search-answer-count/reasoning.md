The scale is what I read first, because here it decides the data type before it decides the algorithm. There are `m <= 10` lantern periods `p_i <= 10^6`, an index `k <= 10^12`, and I must report the time of the `k`-th minute that is a multiple of at least one period, counting distinct lit minutes in increasing order. Input is `m` and `k`, then the `m` periods; output one integer. With `k` that large and periods that small, the answer can be enormous: the smallest period `pmin` alone lights `pmin, 2*pmin, 3*pmin, ...`, so `k` distinct lit minutes have already occurred by time `k*pmin <= 10^12 * 10^6 = 10^18`. That `10^18` is the upper bound on the `k`-th lit minute, and it is the first thing that bites. It overflows nothing in `long long` (`LLONG_MAX ≈ 9.2*10^18`), but it sits far above the `10^9`-ish ceiling a "values up to a million" reflex would reach for — a `hi = 10^9` would leave the boundary case `p=[10^6], k=10^12` unreachable and return `10^9` for an answer that is actually `10^18`. So the search ceiling must clear `10^18`; I take `hi = 2*10^18`, still under `LLONG_MAX` so even the count arithmetic evaluated at `x = hi` stays in range.

The count `f(x)` = number of distinct lit minutes in `[1, x]` is non-decreasing in `x`, so the `k`-th lit minute is the smallest `x` with `f(x) >= k` — a binary search on the answer over `[1, 2*10^18]`. Everything reduces to computing `f(x)`: how many integers in `[1,x]` are divisible by at least one period. The word *distinct* is the whole difficulty.

There are two ways to count. The one-liner `f(x) = sum_i floor(x/p_i)` is `O(m)` but double-counts any minute divisible by several periods. Inclusion–exclusion over subsets fixes that: a minute divisible by every period in a subset `S` is a multiple of `lcm(S)`, so

```
f(x) = sum over non-empty S of  (-1)^(|S|+1) * floor(x / lcm(S))
```

odd-sized subsets add, even-sized subtract, at most `2^m - 1 = 1023` terms per query. Its two risks are the sign convention and the LCM overflowing.

The double-count is the whole trap, so I settle the shortcut on the smallest overlapping case. On `p=[2,3]`, `x=6` the lit minutes are `2,3,4,6` (four of them), but `floor(6/2)+floor(6/3) = 3+2 = 5`: minute `6`, a multiple of both, lands in both terms. And that inflation corrupts the answer, not just the count — for `k=10` the buggy count reaches `10` already at `x=12` (`6+4`), so the search returns `12`, which is only the 8th lit minute in `2,3,4,6,8,9,10,12,...`; the correct answer is `15`. The shortcut is dead. Inclusion–exclusion on the same case gives `{2}:+3, {3}:+2, {2,3}:lcm 6 so -1`, total `4`, the `-1` removing exactly the double-counted `6`; the sign convention is right. End-to-end with the correct `f`, `f(14)=9 < 10` and `f(15)=10`, so the smallest qualifying `x` is `15`, matching the stated sample.

Building the LCM by `lcmv = lcmv / gcd(lcmv,p_i) * p_i` and adding `floor(x/lcmv)` by popcount parity is the whole count — except the LCM overflows, and the constraints invite it directly. With pairwise-coprime periods near `10^6`, four of them already give `lcm ≈ 10^24`, far past `long long`; the product wraps to garbage, `x/lcmv` becomes a wild term, and the count is silently wrong. The escape is that I never need an LCM larger than `x`: if `lcm(S) > x` then `floor(x/lcm(S)) = 0` and the subset contributes nothing regardless of sign. So I cap the build. Before each multiply `lcmv *= step` with `step = p_i / gcd(lcmv, p_i)`, I test `lcmv > x / step`; if so the new LCM would exceed `x`, so I abandon the subset and skip its zero term. Every surviving `lcmv <= x <= 2*10^18`, and the one multiply I actually perform satisfies `lcmv*step <= x`, so nothing overflows. The guard does double duty — it prevents the overflow and short-circuits the useless subsets.

The one subtle point is that the guard must be strict `>`. On `p=[2,3]`, `x=6`, subset `{2,3}` has `lcm = 6 = x`; its check is `lcmv = 2 <= 6/3 = 2`, so it is kept and contributes the real `-1` that cancels the double-count. A `>=` test would drop it and silently restore the count of `5`. The direction is provable with integer division: `lcmv > x/step` means `lcmv >= floor(x/step)+1`, so `lcmv*step > x`; and `lcmv <= x/step` gives `lcmv*step <= step*floor(x/step) <= x`. So strict `>` drops exactly the subsets with `lcm > x`, and keeps the boundary subset where `lcm == x`.

Two structural cases confirm no special-casing is needed beyond that. Equal periods describe the same set of minutes, and inclusion–exclusion deduplicates them on its own: for `p=[6,6]`, `lcm(6,6)=6` collapses `floor(x/6)+floor(x/6)-floor(x/6)` to a single `floor(x/6)`, so I leave the array as read rather than `sort+unique` it — a manual dedup would only add a place to plant an off-by-one. Divisibility chains behave the same way: `p=[2,4,8]` should count exactly the even minutes, `f(x)=floor(x/2)`, and it does — the two `floor(x/4)` terms cancel and the four `floor(x/8)` terms sum `+1-1-1+1=0`, leaving `floor(x/2)`.

The remaining cases hold. `m=1` is just multiples of `p[0]`. The boundary `p=[10^6], k=10^12` lands at exactly `10^18`, reachable under `hi=2*10^18` and printable in signed 64 bits. Pairwise-coprime `m=10` visits all `1023` subsets but short-circuits every size-`>=4` one, so no overflow. The search invariant is clean: the predicate is false at `x=0` and true at `hi` (since `f(hi) >= floor(hi/pmin) >= k`), and `mid = lo + (hi-lo)/2` avoids `lo+hi` overflowing. Cost is `1024` subsets times `~10` work times `~61` search iterations, about `6*10^5` operations — trivially under the 1-second limit, `O(m)` memory.

For an independent cross-check I compare against a brute force that walks `t = 1, 2, 3, ...`, marks `t` lit when any `p_i` divides it, and stops at the `k`-th — no inclusion–exclusion and no binary search, so it cannot share a bug. On small random cases seeded toward heavy overlap and divisibility chains with duplicate periods — exactly where a surviving double-count would show — the two agree.
