The numbers in this one are chosen to bite. There are up to `n = 1000` parcels, each payout `v[i]` up to `10^9`, so if every parcel fits the total reaches `1000 * 10^9 = 10^{12}`. A signed 32-bit integer stops near `2.147 * 10^9` — three orders of magnitude short — so the true answer overflows `int` on the large tests, silently, as a wrong number rather than a crash. Volumes and `W` are at most `10^5`, safe in 32-bit, but the entire value path — the payouts, the accumulators, the dp table — has to be 64-bit. I fix that as an invariant before choosing an algorithm and read everything into `long long` so I never mix widths by accident. The task underneath is standard: pick a subset of parcels with total volume at most `W`, each usable once, maximizing total payout; read `n` and `W`, then `n` pairs `w[i] v[i]`, print one integer.

Two routes. Greedy by value density — sort by `v[i]/w[i]`, load the densest that still fit — is `O(n log n)` and a handful of lines, but it is optimal only for the *fractional* knapsack, where I can load part of a parcel to fill the van exactly. Whole parcels are indivisible, and I suspect that breaks it, so I try to break it concretely: `W = 4`, parcels `A = (1, 2)` and `B = (4, 7)`. Densities are `2.0` and `1.75`, so greedy takes A (volume 1, payout 2), then B needs volume 4 with only 3 left and is locked out — greedy scores `2`. But B alone fits exactly and scores `7`. The high-density crumb wasted the slack the one big parcel needed. Greedy is out for 0/1.

So the capacity DP. Let `dp[c]` be the best payout achievable within volume budget `c`, initialized to `0` for all `c` (the empty load). Fold parcels in one at a time; bringing in `(w, v)`, for budget `c >= w` I either skip it (`dp[c]` unchanged) or take it (`dp[c - w] + v`). So `dp[c] = max(dp[c], dp[c - w] + v)`, and the answer is `dp[W]`. That is `O(n*W) = 1000 * 10^5 = 10^8` tight integer operations, comfortably under a second in C++, with `O(W)` memory. `dp` is monotone non-decreasing in `c`, so `dp[W]` is the best over all volumes up to `W`, which is exactly "total volume does not exceed `W`".

Two things have to be exactly right, and they are the only two places this can go wrong.

First, the update direction. The `dp[c - w]` on the right must be the table's value *before* this parcel was folded in, because each parcel is used at most once. If I sweep `c` upward with the table in place, then by the time I reach `c` the entry `dp[c - w]` (a smaller index) has already been raised for this same parcel — so I add the parcel to itself, which is the *unbounded* knapsack. Concretely, `W = 6`, one parcel `(2, 5)`, ascending: `dp[2] = dp[0] + 5 = 5`, then `dp[4] = dp[2] + 5 = 10`, then `dp[6] = dp[4] + 5 = 15` — three copies of a parcel that exists once. Sweeping `c` from `W` down to `w` fixes it: then `dp[c - w]` is still the pre-parcel value when read, because smaller indices are untouched this round. Descending, the same parcel gives `dp[6] = dp[4] + 5 = 5` and stays `5` at every budget — loaded exactly once.

Second, the width — the whole point of the problem, and it will not show up on those tiny traces. `W = 7`, two parcels `(3, 2000000000)` and `(4, 2000000000)`; both fit (`3 + 4 = 7`), so the answer is `4000000000`. Fold the first: `dp[3..7] = 2e9`. Fold the second at `c = 7`: `cand = dp[3] + 2e9 = 2e9 + 2e9`. In 32-bit that sum is `4000000000 > 2^31 - 1 = 2147483647`, so it wraps to `4000000000 - 2^32 = -294967296`, which is *less* than `dp[7] = 2e9`; `max` discards the correct value and the code prints `2000000000`, silently too small — the worst kind of wrong, because it looks plausible. Making `dp`, `w`, `v`, and `cand` all `long long` computes the sum in 64-bit (peak `~10^{12}`, tiny against `long long`'s `~9.2 * 10^{18}`) and the wrap is gone. Any 32-bit cell anywhere in the value path corrupts the big totals; this is precisely the trap the constraints were built around.

Edge cases, where knapsack code usually dies:
- `w[i] > W`: the inner loop's start `W` is already below `w`, so it never runs and the parcel is ignored — correct, since it cannot be in any feasible subset. I still add `if (w > W) continue;` to make that explicit.
- Zero-volume parcels (`w = 0`): the read index equals the write index, but the descending sweep still writes each cell once from its own pre-parcel value, so a `(0, v>0)` parcel is loaded exactly once per budget, not repeatedly. `W = 1`, `(0, 5)`: `dp[1] = 0 + 5 = 5`, then `dp[0] = 0 + 5 = 5`. Correct.
- `n = 0`: the parcel loop never runs, `dp[W] = 0` — the empty load.
- `W = 0`: `dp` is a single cell; only zero-volume positive parcels can raise it to `dp[0] = v`. Correct.
- All parcels oversized: every one skipped, output `0`.
- Ties in value or density: the DP keeps the larger payout, so the chosen subset may differ but the reported maximum is unaffected. All payouts are `>= 0`, so the base case `0` needs no negative sentinel.

The contract sample's own answer, `3300000000`, already sits above `2^31 - 1`, so the very first example doubles as an overflow tripwire — an `int` solution prints the wrong thing on it before any hidden test is reached.
