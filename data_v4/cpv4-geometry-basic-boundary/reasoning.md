I have `n` axis-aligned closed rectangles, up to `2*10^5` of them, each given as two opposite corners with integer coordinates in `[-10^9, 10^9]` in arbitrary order, and I must count the distinct integer lattice points their union covers. Two facts decide the shape of the whole problem before I pick an algorithm. First, this is a *point count*, not an area: a rectangle whose columns run `x1..x2` covers `x2 - x1 + 1` distinct columns, and that inclusive `+1` lives in both dimensions, so the naive area `(x2-x1)*(y2-y1)` is short by a full border on two sides at once — the `(0,0)-(2,2)` square is a `3x3` block of `9` points, but `(2-0)*(2-0) = 4`. Defeating that `+1` at every single rectangle, at every shared edge, and at every degenerate rectangle is the entire problem. Second, the count can be enormous: coordinates spanning the full `2*10^9` extent give on the order of `4*10^18` points, nine orders of magnitude past 32-bit and near the ceiling of 64-bit, so every coordinate, width, and accumulator has to be `long long`; an `int` anywhere is an instant wrong answer. And since the corners arrive in arbitrary order I normalize `x1 = min(a,c), x2 = max(a,c)` (and likewise `y`) up front — otherwise a rectangle handed to me as `(5,5)-(2,3)` gets a negative width and silently vanishes from the count.

A literal grid or hash set that marks every covered `(x,y)` is correct by definition — it enumerates the definition — but it is `O(sum of areas)` and dies the instant coordinates are large. I keep it only as a brute-force oracle for testing. The shippable route is a sweep with coordinate compression: sort the distinct vertical cut lines, walk the plane left to right in vertical slabs, and in each slab multiply the slab width by the union length of the active horizontal intervals. This survives `10^9` coordinates; all the danger is in the inclusive/exclusive bookkeeping that turns continuous lengths into integer counts.

The clean way to make the `+1` automatic rather than something I sprinkle by hand and forget: map each closed integer range `[x1, x2]` to the half-open real interval `[x1, x2+1)`. Its length `(x2+1) - x1 = x2 - x1 + 1` is exactly the column count, and the unit strip `[x, x+1)` stands for the single integer column `x`. So I push `x1` and `x2+1` as compression coordinates, sweep slabs `[xs[k], xs[k+1])` whose width `xs[k+1] - xs[k]` counts the integer columns in the band, and inside each slab collect the y-intervals `[y1, y2+1)` of every rectangle whose x-range covers the *whole* slab, take their union length (the integer rows covered there), and multiply. Because both axes expand by `+1`, the inclusive boundary is handled uniformly in x and y, and any shared edge or corner between two rectangles falls out of the interval union counted exactly once.

The mapping has to reproduce the stated sample of `17`. For `(0,0)-(2,2)` and `(1,1)-(4,3)`: x-endpoints are `{0,3}` and `{1,5}`, sorted-unique `{0,1,3,5}`, giving slabs `[0,1)`, `[1,3)`, `[3,5)`. Slab `[0,1)` (width 1) is covered only by rect1 (`x1<=0` and `x2+1>=1`), y-union `[0,3)` length 3, contributes 3. Slab `[1,3)` (width 2) by both, y `[0,3)` and `[1,4)` union to `[0,4)` length 4, contributes 8. Slab `[3,5)` (width 2) by rect2 only, y `[1,4)` length 3, contributes 6. Total `17`, matching the stated answer.

The two boundary tests the sweep needs — does a rectangle cover a slab, and how do the y-intervals merge — are exactly where an off-by-one hides, so I write them and trace the tightest case:

```
for (int i = 0; i < n; i++) {
    if (X1[i] <= xl && X2[i] >= xr) {            // (A) does rect i cover this slab?
        ivs.push_back({Y1[i], Y2[i] + 1});
    }
}
...
// union of half-open y-intervals
long long curL = ivs[0].first, curR = ivs[0].second, rows = 0;
for (size_t j = 1; j < ivs.size(); j++) {
    if (ivs[j].first >= curR) {                  // (B) start a new piece?
        rows += curR - curL;
        curL = ivs[j].first; curR = ivs[j].second;
    } else {
        curR = max(curR, ivs[j].second);
    }
}
rows += curR - curL;
```

Line (A) is the trap this problem is built to spring. Trace the lone rectangle `(0,0)-(2,2)`, whose answer must be `9`: its x-endpoints are `0` and `x2+1 = 3`, so `xs = {0,3}`, one slab `xl=0, xr=3` of width `3`. Test (A) asks `X1 <= xl` (`0<=0`, true) **and** `X2 >= xr` (`2 >= 3`, false), so the rectangle is judged not to cover its own — its only — slab, `ivs` is empty, and the program prints `0`. The slip is the same inclusive `+1` the whole problem turns on: I expanded the x-range to the half-open `[X1, X2+1)`, so the rectangle's right end for coverage purposes is `X2+1`, not `X2`. Writing `X2 >= xr` compared the *closed* right edge against the slab's *half-open* right edge and dropped the `+1` on exactly the boundary column. The correct test is `X2[i] + 1 >= xr`; re-tracing, `2+1 = 3 >= 3` holds, `ivs = {[0,3)}`, union length `3`, contribution `9`.

Test (B) is the same inclusive/exclusive question one level down. Two half-open y-intervals that touch, `[0,2)` and `[2,5)`, are adjacent and their union is the single `[0,5)` of length `5`; the correct "real gap" predicate is strict `ivs[j].first > curR`. The `>=` I first wrote declares a false gap at the touch point `start == curR` and splits a contiguous run into two pieces. It happens to conserve total length — `2 + 3 = 5` either way, so the *answer* survives — but it misrepresents one maximal interval as two, which is the wrong invariant to carry. I change (B) to `>`. A genuine gap still splits correctly: for `(0,0)-(2,1)` and `(0,3)-(2,4)`, row `2` is uncovered, y-intervals `[0,2)` and `[3,5)`, `3 > 2` closes the first piece, giving rows `2 + 2 = 4` and, over the width-3 slab, `12` points — the columns `{0,1,2}` times rows `{0,1,3,4}`.

The judged edge cases mostly fall out of the two `+1`s: a single point `(3,3)-(3,3)` maps to slab `[3,4)` times row `[3,4)`, one point; a thin line `(0,0)-(5,0)` is a width-6 slab times a single row, six points (the degenerate height still gets its inclusive `+1`); reversed corners like `(5,5)-(2,3)` are equalized by the `min/max` normalization before any of this runs. Two cases carry real subtlety. The empty input `n = 0` leaves `xs` empty, and the slab-loop guard `k + 1 < xs.size()` is `1 < 0` in unsigned arithmetic — false, body never runs, prints `0`; the guard is evaluated before `k` ever moves and `k` only increments, so the unsigned comparison can't wrap into a spurious pass. And corner-touching `(0,0)-(2,2)` and `(2,2)-(4,4)`, sharing only the single point `(2,2)`, expected `9 + 9 - 1 = 17`: rect2 has x-endpoints `2` and `5` (not `3` — the compression coordinate is `x2+1`), so sorted-unique `{0,2,3,5}` gives slabs `[0,2)`, `[2,3)`, `[3,5)`. `[0,2)` (width 2): rect1 only, y `[0,3)`, gives 6. `[2,3)` (width 1): rect1 (`0<=2 && 3>=3`) y `[0,3)`, rect2 (`2<=2 && 5>=3`) y `[2,5)`, union `[0,5)` length 5, gives 5. `[3,5)` (width 2): rect2 only, y `[2,5)` length 3, gives 6. Total `17`, the shared corner counted once — which is precisely why the compression coordinate must be `x2+1` and not `x2`.

Correct is not yet fast enough, though. The sweep as written rescans all `n` rectangles inside every slab, and with up to `2n` distinct cut lines that is `O(n^2 log n)` — at `n = 2*10^5` roughly `4*10^5` slabs times `2*10^5` rectangles, on the order of `10^11` operations against a 2-second limit; timing the implementation confirms it already takes over 3 seconds at `n = 10^4`. The fix is to stop re-deriving each slab's active set from scratch. A rectangle's coverage of slabs is one contiguous run in sweep order, so I turn each rectangle into two events — activate its y-interval `[y1, y2+1)` at `x = x1`, deactivate it at `x = x2 + 1`, the same half-open right end that test (A) taught me — and maintain the union length incrementally in a segment tree built over the elementary y-segments between consecutive compressed values of `{y1, y2+1}`. Each node stores how many active intervals fully cover its segment and the covered length beneath it: a positive count means the whole segment `ys[hi] - ys[lo]` is covered no matter what the children say; otherwise the covered length is the children's sum, `0` at a leaf. A deactivation retraces exactly the decomposition of its activation, so counts never go negative and no lazy propagation is needed. Coverage is constant between consecutive event abscissas, so each band contributes `(x' - x) * cov[root]`. Both traps resurface in the new clothes: deactivating at `X2` instead of `X2 + 1` disowns each rectangle's rightmost column — the lone `(0,0)-(2,2)` square drops to `6` instead of `9` — and the y-interval must update the half-open leaf range `[idx(y1), idx(y2+1))`, exclusive on the right, or the tree annexes the elementary segment just above the rectangle: two isolated points `(0,0)` and `(0,2)` come out as `3` instead of `2`. The interval-merge scan of test (B) disappears entirely — touching intervals fuse automatically when coverage is accounted per elementary segment. Sorting plus `2n` updates of `O(log n)` each is `O(n log n)`, and the event-driven version finishes the full `n = 2*10^5` in well under half a second.

Past the hand traces, I run the swept solution against a brute that marks every covered lattice point in a set (correct by construction for small coordinates) over hundreds of random instances mixing dense overlaps, reversed corners, degenerate points and lines, corner-touches, and empty inputs. A dropped `+1` on a right end truncates single rectangles and the random cases are dense with those, so a surviving slip would show immediately; the swept solution matches the oracle on every instance.
