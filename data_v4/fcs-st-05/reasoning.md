I need `k`, the starting index of the lexicographically least cyclic rotation of `s`, where rotation `k` is `s[k:] + s[:k]`, and on ties the *smallest* such index. Two details fix the shape of the solution before any algorithm. First, scale: `n` reaches `10^6`, so a `Θ(n^2)` compare-all-rotations method is `~10^12` character comparisons — far past the 1-second budget — while a handful of `O(n)` arrays over a `10^6` string sit in tens of MB, well under 256 MB. So the constraints don't merely prefer the linear method; they make the quadratic one time out on the large tests. Second, the output is an *index* in `[0, n-1]`, not the rotated string, and ties break toward the *smallest* index — a rule that has to be baked into the update itself, not patched on afterward.

Compare-all is still worth keeping around as the brute-force oracle: build each of the `n` rotations, take the lexicographic minimum, tie-break by index. It is correct by definition, which is exactly what I want to test against — but each comparison scans up to `n` characters, so `Θ(n^2)` time and `O(n^2)` just to materialize the rotations. Fine as an oracle up to a few thousand characters, fatal at `10^6`.

Before abandoning it I want to see where the quadratic time goes, because that is where the linear idea comes from. Take `s = "aaaa...ab"`: `n-1` copies of `'a'` then a `'b'`. Rotation 0 is `"aa...aab"`, rotation 1 is `"aa...aba"`; they agree on a long run of `'a'`s and differ only near the end, so a comparison scans almost the whole string before deciding — and nearly every adjacent pair does the same. The structural killer is that the rotations share enormous prefixes and the naive method re-reads each shared prefix from scratch every comparison, so every character of `s` is looked at `Θ(n)` times. That re-reading is exactly what a failure function exists to remove.

So I scan `ss = s + s` left to right with a pointer `j`, holding a committed best start `k` and, KMP-style, the length `i+1` of the run currently matched between the candidate rotation at `k` and the text at `j` — that is, `ss[j]` is compared against `ss[k + i + 1]`, the next character the candidate expects, with `i = f[j - k - 1]`. Three outcomes for `c = ss[j]`:

1. **Match** (`c == ss[k+i+1]`): the candidate extends by one; record the longer run in `f`, no decision on `k` yet.
2. **`c` strictly smaller**: the rotation beginning at `j - i - 1` agreed with `k`'s on the first `i+1` characters and is now strictly smaller, so it wins — set `k = j - i - 1`.
3. **`c` larger**: the candidate at `k` wins, and so does every rotation that began inside the matched block, since they all carry the same now-losing prefix; so instead of retreating one character I follow the failure links, `i = f[i]`, jumping past the entire block at once.

That third case is the whole point. Greedy one-step retreat would put me back at `Θ(n^2)`; the failure-link jump discards a whole run of already-compared characters in one move, so every character is charged `O(1)` amortized and the scan is `O(n)`. This is Booth's algorithm, and the block-skip is the precise cure for the re-read-the-shared-prefix waste I diagnosed above.

The tie rule lives in a single character of the update: I move `k` only on a *strict* `<`, never on equality. For an all-equal string like `"aaaa"` no strict improvement ever fires, `k` never leaves its initial `0`, and I return the smallest index as required; a `<=` would drift `k` forward on every tie and hand back a larger index. So the strict comparison is not stylistic — it *is* the tie-break.

I build `ss = s + s` explicitly (clearer than modular indexing, and the memory is fine at `10^6`) and keep `f` of length `2n` initialized to `-1` for "no match yet". The exposure is entirely in the index arithmetic — `f[j - k - 1]`, `ss[k + i + 1]`, `f[j - k]`, and the two sites where `k` jumps (`k = j - i - 1` inside the mismatch loop, and `k = j` when `i == -1`, which coincide there because `j - i - 1 == j`). This is the kind of tightly-interacting KMP index code where a hand-trace is worse than useless: eyeballing `"abaab"` (answer `2`) I misread one `ss[...]` lookup and talked myself into `k = 5`, not even a legal index for a length-5 string, while the compiled program returns `2`. So I don't certify these indices by reading them — I certify by differential testing against the obvious oracle, which is the right discipline for KMP-family code anyway.

I write the brute oracle and a generator biased toward the hard regimes: tiny alphabets, periodic blocks, single characters, the inputs with dense ties and long shared prefixes that stress the block-skip. Exhaustive binary over all 8190 strings of length `1..12`: no mismatches. Exhaustive ternary over all 9840 strings of length `1..8`: no mismatches. About 2800 random cases up to length 2000 over alphabets of size `1..26`, including periodic constructions with random perturbations: no mismatches. Explicit edges — `"a"`, `"ab"`, `"ba"`, `"abab"`, `"aabaab"`, `"mississippi"`, `"zzzzzzzzza"`, `"cabcab"`, the empty input, and the all-equal and `"aaa...ab"` families — all match, including the all-equal answer `0` that confirms the strict-tie rule.

Two checks that actually pin something down. Empty input: `cin >> s` fails, so I print `0` and return before calling anything, and `leastRotation` also guards `n == 0` independently so `ss` is never sized zero. And the `10^6` adversaries where the oracle would time out — `"a"*10^6 -> 0`, `"a"*(n-1)+"b" -> 0`, `"b"+"a"*(n-1) -> 1`, period-2 `"ab"` repeated to `10^6` -> `0` — each finish in about 10 ms and 14 MB, two orders of magnitude inside the budget. Every index is bounded by `2n <= 2*10^6`, so `int` holds it with room to spare — no `long long` — and `k` is only ever assigned a start lying in the first copy of `s`, so the returned value is already in `[0, n-1]` and I report it as is.

That is what I ship: one self-contained C++ file running Booth's `O(n)` algorithm, the tie rule carried in the strict `<`, wrapped in the `main` that reads the single token and prints the index, with empty input short-circuited to `0`.
