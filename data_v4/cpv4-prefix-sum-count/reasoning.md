**Scale first — it fixes the types before any algorithm.** The count runs over all `O(n^2)` windows, up to `n*(n+1)/2 ≈ 2*10^10` of them at `n = 2*10^5`. Two consequences are immediate. Enumerate-and-test is `O(n^2)`, about `2*10^10` operations, hopeless against a 1-second limit; it survives only as a brute-force oracle, never as the submission. And the answer overflows 32-bit signed (`~2.1*10^9`) by an order of magnitude, so the accumulator must be `long long` — `m = 1` makes every window balanced and the count exactly `2*10^10`, a guaranteed hidden test rather than a hypothetical corner. Two more numbers to hold onto: `m` up to `10^6`, and `|a[i]|` up to `10^9`, so a raw running sum reaches `2*10^5 * 10^9 = 2*10^14` (64-bit if I ever carry it unreduced) and negative readings mean negative running sums — which will collide with C++'s sign-keeping `%` once I get to the modulus.

**The reduction to prefix residues.** Define `S[0] = 0`, `S[k] = a[0] + ... + a[k-1]`. Window `[l, r]` has sum `S[r+1] - S[l]`, divisible by `m` iff `S[r+1] ≡ S[l] (mod m)`. Setting `i = l`, `j = r+1`, every balanced window is exactly one unordered prefix pair `0 <= i < j <= n` with equal residues, and conversely. So the answer is the number of same-residue pairs among the `n+1` prefixes `S[0..n]`: bucket residues, and a class of size `c` contributes `C(c,2) = c*(c-1)/2` pairs. `O(n + m)` time, `O(m)` space. And `S[0] = 0` is a real participant — it is the left endpoint of every window that starts at index `0`, so it has to sit in the buckets like any other prefix.

Check the bijection on the sample `m = 5`, `a = [3,1,4,1,5,-4,6]` (claimed `5`). Prefixes `S = [0,3,4,8,9,14,10,16]`, residues `[0,3,4,3,4,4,0,1]`. Classes: residue `0` at `{0,6}` → `1` pair, `3` at `{1,3}` → `1`, `4` at `{2,4,5}` → `3`, `1` singleton → `0`, total `5`. The pairs name the windows exactly: `{2,4,5}` → `[2,3],[2,4],[4,4]`, `{0,6}` → `[0,5]`, `{1,3}` → `[1,2]` — the five the statement lists. The identity holds and the per-class count is `c*(c-1)/2`.

**Three places the tally goes subtly wrong — each a trap these constraints invite.**

*The per-class count is `c*(c-1)/2`, not `c*c/2`.* `c*c` counts ordered pairs including the diagonal `i = i`, so `c*c = c*(c-1) + c`, and halving it is neither the ordered nor the unordered count. On `m = 2, a = [2,4]` (all even, truth `3`: `[0,0],[1,1],[0,1]`) the single class `0` has `c = 3`; `c*c/2 = 4` overcounts, `c*(c-1)/2 = 3` is right. `c*(c-1)` is always even so the division is exact, and it reaches `~4*10^10` — 64-bit again.

*Seed `cnt[0] = 1` for the empty prefix.* Folding the tally into one streaming pass — for each new residue, add `cnt[pref]` (the earlier equal prefixes) then increment its bucket — is cleaner, but only correct if `S[0]` is already in the buckets. On `m = 5, a = [5]` (truth `1`, window `[0,0]` sums to `5`) without the seed: `pref = 0`, `answer += cnt[0] = 0`, done → `0`. That window is the pair `(0,1)` whose left endpoint is `S[0]`; drop `S[0]` and you silently drop *every* window starting at index `0` — a systematic undercount, not a random one. Seeding `cnt[0] = 1` repairs it, and the streaming add-before-increment over a class encountered in order sums to `0 + 1 + ... + (c-1) = c*(c-1)/2`, identical to the offline formula, so the two forms agree by construction.

*Normalize negative residues into `[0, m-1]`.* C++'s `%` keeps the dividend's sign, so a negative running sum yields a negative remainder and `cnt[negative]` is an out-of-bounds read. On `m = 3, a = [-2, 2]` (truth `1`, window `[0,1]` sums to `0`), a plain `(pref + a[i]) % m` gives `pref = -2` at the first step, indexing `cnt[-2]`. The fix is `((pref + a[i]) % m + m) % m`, and the outer mod is not decorative: the inner `% m` lands in `(-m, m)`, `+ m` lifts it to `(0, 2m)`, and when the value was a negative multiple (inner result `0`) it becomes `m`, which only the outer `% m` folds back to `0`. With it: `pref = ((-2)%3 + 3)%3 = 1`, then `((1+2)%3 + 3)%3 = 0`, `answer += cnt[0] = 1` → `1`, no negative index.

**Edge cases.** `n = 0`: the loop never runs, `answer = 0`, no windows — correct (`if (!(cin >> n >> m)) return 0;` guards truly empty input). `n = 1`: one window, balanced iff `a[0] % m == 0` — `[6],m=3` → `1`, `[7],m=3` → `0`. `m = 1`: all prefixes fall in class `0`, streaming sum `n*(n+1)/2` = every window, `~2*10^10` at max `n` — the test that punishes a 32-bit accumulator. Zeros extend a class with no special case (an all-zero window sums to `0`, balanced). `cnt` is `m <= 10^6` longs = 8 MB, inside 256 MB; `pref` is always reduced into `[0, m-1]` so it never grows.

**Landing.** The submission is the streaming `O(n+m)` pass: seed `cnt[0] = 1`, normalize each prefix residue into `[0, m-1]` with the double mod, add `cnt[pref]` before incrementing, and accumulate into a `long long`. The full program is in the answer.
