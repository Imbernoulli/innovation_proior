The thing that decides everything here is the phrase "arbitrary denominations." I have `n <= 100` coin values, each up to `10^6`, a target `S <= 10^6`, an unlimited supply of each coin, and I must hit `S` *exactly* with as few coins as possible or print `-1`. The scale pins the data types first: when an answer exists it uses at most `S` coins (every coin is `>= 1`, so `S` coins of value `1` is the worst case), so it never exceeds `10^6` and fits a 32-bit int — but I will carry sums in `long long` anyway, because `s - c` and the unreachable sentinel are exactly where signedness and overflow bite. The word "arbitrary" is the real content: it rules out the shortcut everyone reaches for.

That shortcut is greedy-by-largest — sort descending, keep subtracting the biggest coin that still fits, count the subtractions. It is what you do at a cash register, and it *is* optimal for canonical currency like `{1, 5, 10, 25}`. But nobody promised me a canonical system, and greedy makes an irrevocable local choice while "fewest coins to reach `S`" is a global optimum. So I try to break it. Take `{1, 3, 4}`, `S = 6`: greedy grabs `4`, is left with `2`, and can only patch that with `1 + 1`, landing on `4 + 1 + 1 = 3` coins — but `3 + 3 = 2` is strictly better. The failure is visible in the mechanism: snatching the largest coin stranded a remainder (`2`) the set fills badly, whereas stepping down to a non-maximal coin lined up a clean tiling. A second family confirms it is not a fluke: `{1, 5, 6, 9}`, `S = 11`, greedy takes `9 + 1 + 1 = 3` while `5 + 6 = 2` tiles it exactly. This is structural, not a patchable corner — salvaging greedy would mean proving the given set is canonical (an easy-to-botch test) or bolting on a fallback, at which point I have written more code than the DP and still need the DP to back it. Greedy is out.

So `dp[s]` = the minimum number of coins summing to exactly `s`, with `dp[s] = INF` when `s` is unreachable. Base case `dp[0] = 0`. For `s >= 1`, any optimal multiset for `s` contains at least one coin `c <= s`; delete it and what remains is a multiset for `s - c` with one fewer coin, and it must itself be optimal for `s - c` (else swap in a cheaper filling and beat the supposed optimum for `s` — the standard exchange argument). Hence

```
dp[s] = 1 + min over denominations c with c <= s of dp[s - c],
```

and `dp[s] = INF` when every predecessor is `INF`. Every transition reads a strictly smaller index (`c >= 1`), so filling `dp` in increasing `s` makes each `dp[s - c]` final before I use it. The unlimited supply falls out of this ordering for free: relaxing *forward* over increasing `s`, the entry `dp[s - c]` may already have used coin `c`, so the same denomination reuses without any bookkeeping — this is precisely the unbounded-knapsack order (the 0/1 case would iterate `s` downward instead). The answer is `dp[S]`, printed as `-1` if still `INF`. I organize the double loop with `s` on the outer index rather than the coin, so the loop bounds and the `c <= s` guard read directly off the recurrence and there is one fewer place to plant an off-by-one.

The recurrence is provably right; the one place it dies in transcription is the sentinel. The naive relaxation `dp[s] = min(dp[s], dp[s - c] + 1)` with `INF = LLONG_MAX` adds `1` to a value at the very top of the `long long` range, and `LLONG_MAX + 1` overflows to `LLONG_MIN` — a huge *negative*, which `min` then latches onto as the "best" answer. The smallest input that trips it is `{4}`, `S = 6`: `dp[1]` is unreachable, but at `s = 5` the guard `4 <= 5` passes and I read `dp[1] + 1 = LLONG_MAX + 1`, which wraps negative; `dp[5]` becomes garbage and `dp[6]` inherits it, printing a nonsensical negative instead of `-1`. The fix is twofold: make the sentinel a finite `4e18` — unmistakably infinite next to any real answer (`<= 10^6`), yet far enough below `LLONG_MAX (~9.2e18)` that `+1` cannot overflow — *and* guard each relaxation on `dp[s - c] != INF` so an unreachable sub-sum never propagates at all, then map a final `INF` to `-1`. With the guard, `{4}`, `S = 6` falls straight through to `-1`.

The corners fall to the same two mechanisms. `S = 0`: the `for s` loop never runs, `dp[0] = 0`, print `0` — zero coins make the empty sum, regardless of denominations. All coins larger than `S` (e.g. `{7, 9}`, `S = 5`): every `v <= s` guard fails, every `dp[s >= 1]` stays `INF` → `-1`. A single coin dividing `S` (`{5}`, `S = 5`): `dp[5]` relaxes from `dp[0]` → `1`. Duplicate denominations (`[3, 3, 3]`): the inner loop just relaxes against the same value repeatedly, `min` is idempotent, no de-duplication needed. Impossible target with a common factor (`{2, 4}`, `S = 3`): every odd `dp` stays `INF` → `-1`. The largest instance, `S = 10^6`, `n = 100`, is `~10^8` `long long` relaxations behind an early `v <= s` guard — roughly `0.1` s, well inside `2` s — and the `dp` array is `~8` MB, well inside `256` MB. The DP I can prove is also the one that comfortably fits.

To catch cases I would not think to hand-trace, I differential-test against a structurally different oracle: a BFS by coin count, where the first layer to reach `S` gives the minimum count — a shortest-path formulation rather than a re-run of the same value relaxation. Across dense small-coin greedy-killer sets, no-`1` sets, impossible targets, `S = 0`, coins exceeding `S`, duplicates, single denominations, and large targets, the two agree with zero mismatches, including the `{1, 3, 4}`, `S = 6` case that started the detour (DP returns `2`).

That fixes the whole program: the `O(S * n)` forward DP with the finite sentinel, the `dp[s - c] != INF` guard, and the `INF → -1` print. The full file is in the answer.
