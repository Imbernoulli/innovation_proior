`n` tops out at 14 while the job times run up to `10^9` — that pairing is the whole character of this instance. Fourteen jobs is small enough that `2^n = 16384` subsets fit comfortably in memory, so an *exact* method that reasons over subsets of jobs is on the table instead of an approximation. But `14 * 10^9 = 1.4 * 10^10` overruns the 32-bit range of about `2.1 * 10^9`, so every press load, the running sum, and the search bounds have to be `long long`; only the job count and a press count stay `int`. An `int` accumulator here is a silent wrong answer on the large tests, so that is settled before anything else. The objective itself: a press's finish time is the sum of the jobs on it, the shop closes with the slowest press, so the closing time is the *maximum* press load, and I want the assignment that minimizes that maximum.

The reflex is the longest-processing-time greedy: sort jobs descending, drop each onto the currently least-loaded press. It is `O(n log n)` and a celebrated 4/3-approximation — exactly the kind of "close enough to feel right" that gets shipped as if it were exact. Before trusting it I try to break it. Take `m = 3`, `t = [5,5,4,4,3,3,3]` (already sorted). LPT puts the two `5`s on presses 0 and 1, then both `4`s onto press 2 (least loaded at 0, then 4), reaching `[5,5,8]`; the three `3`s then land to give `[8,5,8]`, `[8,8,8]`, and finally `[11,8,8]` — closing time 11. But `{5,4}, {5,4}, {3,3,3}` gives loads `9, 9, 9`. LPT scattered the three `3`s onto presses that already held a `5+4` and never saw that the `3`s wanted a press to themselves; a purely local "least-loaded" placement cannot foresee that. So greedy is out and I need an exact method. (A cousin shortcut, outputting `max(maxJob, ceil(sum/m))`, is only a lower *bound* — it ignores that jobs are atomic, so `[7,1,1,1] m=2` truly needs 7, not `ceil(10/2)=5`.)

The exact handle is binary search on the closing time. The predicate `feasible(T)` = "can every job finish by time `T`?" is monotone: if all jobs pack into `m` presses each loaded `<= T`, they certainly pack under any larger `T`. So I search the smallest feasible `T` in `[max t[i], sum t[i]]` — the low end because some press must run the largest job, the high end because one press doing everything is always valid. The loop is standard, with `mid = lo + (hi - lo) / 2` so the midpoint stays safe even though `long long` already has ample room.

Everything then rides on `feasible(T)`: can the jobs split into at most `m` groups each summing `<= T`? The tempting cheap version tracks a single *open* press — state `(presses used, load on the current press)`, and whenever the next job does not fit, close the press and start a fresh one. This is wrong, and the reason is exactly what makes packing hard: once it closes a press it can never come back to top it up. On `[3,3,2,2,2] m=2` at `T = 6` the true packing is `{3,3}` and `{2,2,2}`, both load 6 — feasible. But an open-bin walk that keeps the current load small can strand itself with an open press too small for what remains and be forced to open a third, reporting `feasible(6) = false`; the search then returns 7 instead of the correct 6. A single open bin plus a count is simply not a sound summary of a multi-bin packing.

The fix is a DP whose transition forms an entire press at once. Let `dp[mask]` be the fewest presses (each loaded `<= T`) needed to cover *exactly* the jobs in `mask`. From a covered `mask`, the next press may be **any** non-empty submask `sub` of the remaining jobs `full ^ mask` whose total `sumMask[sub] <= T`, advancing to `mask | sub` at one more press. Because the next press is any valid subset of what is left — not whichever bin happened to be open — this ranges over every possible partition, so `dp[full]` is the genuine minimum group count and `feasible(T)` holds iff `dp[full] <= m`. I precompute `sumMask[sub]` for all subsets in `O(2^n)` by peeling the lowest set bit, and I prune any `mask` whose `dp` already reached `m`. Enumerating submasks of each complement is the classic `3^n` walk.

Back on the sample `[5,5,4,4,3,3,3] m=3`: `feasible(9)` holds via `{5,4}, {5,4}, {3,3,3}`, and `feasible(8)` fails — the two `5`s each need their own press, `5+4 > 8` forbids pairing either with a `4`, and three groups cannot cover all seven jobs under 8 — so the search lands on 9, matching the reachable partition and beating LPT's 11.

Now the corners. `n = 0`: no jobs, print 0 directly, before `full = (1<<n)-1` collapses to 0 and degenerates the DP loops. `m = 1`: only the full set is a valid group, so feasibility holds precisely at `T >= sum` and the search returns the total. `m >= n`: each job can take its own press — singletons are valid submasks, so `dp[full] = n <= m` at `T = max t[i]` and the answer is `max t[i]`; crucially the press count is only a bound compared against `m`, never an array index, so `m > n` never reads out of bounds. `[7,1,1,1] m=2` returns 7, not the average 5, because the dominant job is handled by the feasibility test rather than by any closed-form guess.

Cost: `sumMask` is `O(2^n)`; each `feasible` call is `O(3^n)` worst case, times the roughly 35 binary-search iterations, about `35 * 3^14 ~ 1.7 * 10^8` integer operations — well under the 1-second limit (about `0.16 s` on a random `n = 14, m = 7` case at `10^9` scale). Memory is two arrays of length `2^n`, a few megabytes. Cross-checking the whole thing against a direct assignment brute force over about 1200 random small cases produced no mismatch.
