The scale is what I read first, because it decides the method. With `n <= 200` and each `a[i] <= 1000`, the total `S = sum(a)` is at most `200000`, so a half-total is at most `100000` — small enough that a table indexed over `[0, half]` is nothing to worry about, and small enough that any `O(n * S)` method is affordable. Everything fits in `int`, but I accumulate the running total into a `long long` so the sum over 200 elements needs no second thought. The real question is not the arithmetic; it is which algorithm actually *certifies* a YES/NO answer.

The reduction is the lever. The two subsets together contain every element, so if their sums are equal each must be exactly `S / 2`. That gives two things at once. If `S` is odd, no equal split can exist — instant `NO`. If `S` is even, the answer is YES precisely when some sub-multiset sums to `S / 2`, because its complement then automatically sums to `S / 2` too. So "partition into two equal halves" collapses to "subset-sum to the half-total."

The cheap route to reject first: greedy balancing — sort descending and drop each value into whichever bin is currently lighter (the longest-processing-time rule), then check whether the bins ended equal. It is a local, irrevocable rule answering a global existence question, and that mismatch is exactly where it breaks. A concrete instance settles it: `a = [4, 9, 10, 12, 15]`, total `50`, target `25`, with a genuine equal split `{10, 15} | {4, 9, 12}` sitting right there. Run greedy on the descending order `[15, 12, 10, 9, 4]`: `15 -> (15, 0)`, `12 -> (15, 12)`, `10 -> (15, 22)`, `9 -> (24, 22)`, `4 -> (24, 26)`. It ends `(24, 26)`, reports `NO`, and is wrong. Committing `15` and `12` to opposite bins early made `25/25` unreachable for the later purely-local choices. Keeping the running bins level is simply not the predicate "an exact equal split exists," so greedy is out.

That leaves the boolean subset-sum DP, which I can prove. Carry `reach[s]` over `s in [0, half]` meaning "some sub-multiset of the elements seen so far sums to exactly `s`." Initialize `reach[0] = true` (the empty subset) and everything else false; the answer at the end is `reach[half]`. I cap the array at `half` because sums past the half-total are irrelevant to the question, which also bounds memory at `half + 1` bytes. Folding in a value `v`, a sum `s` becomes reachable if it already was (skip `v`) or if `s - v` was reachable (take `v`): `reach[s] |= reach[s - v]`.

The one thing that can silently go wrong is element reuse, and it lives entirely in the iteration order. Each element may be used at most once, so both sides of `reach[s] |= reach[s - v]` must reflect the state *before* `v` was folded in. If I sweep `s` upward, then by the time I read `reach[s - v]` I may already have set it to true using `v` itself earlier in the same pass — turning 0/1 subset-sum into unbounded knapsack. `a = [3, 1]`, target `2`, makes the failure concrete: the true answer is `NO` (subset sums are `0, 1, 3, 4`). An upward sweep on the `1` first sets `reach[1] = true`, then reads that freshly-set cell to set `reach[2] = true` — a spurious `YES` that counts the lone `1` twice. The fix is to sweep `s` **downward**, from `half` down to `v`:

```
for (long long s = half; s >= v; s--)
    if (reach[s - v]) reach[s] = 1;
```

Now `reach[s - v]` is a strictly smaller index untouched in this element's pass, so it still holds the pre-`v` value and each element is used exactly once. On `[3, 1]` this gives the correct `NO`; on `[4, 9, 10, 12, 15]` the reachable set still hits `25` (via `13 + 12`), giving `YES` — matching the split greedy missed. Cost is `O(n * half) <= 2 * 10^7` boolean updates.

The corners all fall to the parity guard plus this DP. Odd total (`[1, 2, 4]`, `S = 7`) exits `NO` before the DP. A single element `[4]` gives `half = 2`, and folding `4` runs an empty range so `reach[2]` stays false — `NO`, correctly (the other side would be empty). Two equal `[4, 4]` -> `YES`; two unequal `[4, 6]` -> `NO`. All-equal with even count `[1,1,1,1]` -> `YES`, odd count `[1,1,1,1,1]` odd total -> `NO`. Value extremes `[1000, 1000, 1, 1]` reach `1001` -> `YES`; the size extreme `n = 200` of `1000`s gives `half = 100000` and a `reach` array around `100001` bytes — trivial. Input parsing is `cin >>`, which absorbs arbitrary whitespace, so the spaces-vs-newlines layout the contract warns about is a non-issue; output is one line with a trailing newline.

Hand-traces are not enough to trust a DP whose only subtle point is invisible in the output, so I differential-test against an independent oracle that decides the same question two other ways — exhaustive `2^n` enumeration for tiny `n`, cross-checked with meet-in-the-middle — neither of which is the downward-sweep DP, so a shared blind spot is unlikely. Over 1300+ random and edge-weighted instances (tiny multisets, tiny values, planted-YES, forced-odd totals, all-equal, value extremes, single/double elements, powers of two, and the full `n = 200`), zero mismatches, including the motivating `[4, 9, 10, 12, 15]`. The worst case — `n = 200`, all `1000`, `half = 100000` — runs in about 14 ms, far inside the 1-second limit.
