An `n x n` cost matrix, `n <= 20`, and I have to send each courier to a distinct zone minimizing the summed cost — the assignment problem. Two numbers decide everything before I pick an algorithm. `n <= 20` rules out brute force over permutations (`20!` is about `2.4*10^18`) but is small enough for a `2^n` bitmask DP. And `c[i][j]` up to `10^9` across `n = 20` edges means a full assignment can cost `20 * 10^9 = 2*10^10` — about ten times `INT_MAX = 2147483647`. So whatever the algorithm, every running sum has to live in 64-bit; a 32-bit accumulator here is a silent wrong answer on the large tests, not a crash. I design around that from the start.

Before reaching for the DP, is there anything cheaper that works? Greedy — send each courier to its cheapest free zone — is `O(n^2)` and tempting, but the cheapest single edges can collide on one zone. On `c = [[1,2],[1,9]]` both couriers' cheapest edge is zone 0; greedy hands courier 0 zone 0 (cost 1) and strands courier 1 on zone 1 (cost 9), total `10`, while the swap courier0->zone1, courier1->zone0 costs `3`. Off by a factor of three because a local cheap-edge grab ignored the global one-to-one constraint. Greedy is out; the `O(n!)` permutation search stays only as a correctness oracle for small `n`.

For the DP, place couriers in the fixed order `0, 1, 2, ...`. The number placed so far equals the number of used zones equals `popcount(mask)`, so the mask alone tells me which courier is next — the state is just the set of used zones. Let `dp[mask]` be the minimum cost to have assigned couriers `0 .. popcount(mask)-1` onto exactly the zones in `mask`. Placing courier `k = popcount(mask)` into any unused zone `z` gives `dp[mask | (1<<z)] = min(dp[mask | (1<<z)], dp[mask] + c[k][z])`. Base `dp[0] = 0`; the answer is `dp[(1<<n) - 1]`, since a full mask means every courier placed.

Tracing the sample `[[4,2,8],[4,3,7],[3,1,6]]`: from `dp[0]=0`, courier 0 fills `dp[001]=4, dp[010]=2, dp[100]=8`; courier 1 relaxes those to `dp[011]=6, dp[101]=11, dp[110]=9`; courier 2 takes the single remaining free zone from each, and all three routes into `dp[111]` land on `12` (e.g. `dp[011]=6` plus `c[2][2]=6`) — the example's stated total, so the fixed placement order carries the recurrence through.

Coding the loop is direct: over each `mask`, relax `dp[mask]` into `dp[mask | (1<<z)]` for every unused zone `z`, with base `dp[0]=0` and answer `dp[(1<<n)-1]`. It needs one guard. When `popcount(mask) == n` — necessarily the full mask — `courier` would be `n` and `c[courier]` is out of range. The inner loop happens to skip every zone there because all bits are set, but an explicit `if (courier == n) continue;` keeps that index from ever being formed. It also makes `n = 0` fall out cleanly: `full = 1`, the only mask is `0`, `courier = 0 = n`, the body is skipped, and the output is `dp[0] = 0`.

The arithmetic is where this problem is actually set. The small traces I would reach for — `n = 1`, `n = 2` — pass on any integer type, so they say nothing about the trap. Take `n = 3` with every entry `8*10^8`, where every assignment costs `2.4*10^9`. Along any path the accumulator runs `0 -> 8*10^8 -> 1.6*10^9 -> 2.4*10^9`. The first two partial sums sit under `INT_MAX`; the third crosses it, and in 32-bit two's complement `2400000000` wraps to `2400000000 - 2^32 = -1894967296` — a negative total, with no crash and no warning. That pairing of `n = 20` with entries near `10^9` is engineered precisely so a partial sum first crosses `2.147*10^9` on a big test. So `c`, `dp`, and the intermediate `cand = dp[mask] + c[courier][z]` are all `long long`; with both operands 64-bit the sum is formed and stored in 64-bit, and the `2*10^10` worst case is nowhere near the `9.2*10^18` ceiling.

Holding `dp` in `long long` forces the sentinel too. `INF` marks unreached masks, and it has to sit above every reachable total (`> 2*10^10`) — so the reflex `INF = 1e9` is worse than useless: real partial costs exceed it, and `min` against a too-small INF would treat genuine states as unreachable and clobber the answer. I use `INF = 4e18`: above any reachable total, and since the loop guards `if (dp[mask] == INF) continue;` before ever reading `dp[mask]`, INF is never an operand of `+` and cannot itself overflow.

The remaining corners fall out of the same code: `n = 1 -> c[0][0]` from the base case; an all-equal matrix gives `n * value`, since `min` resolves the ties; and `n = 20` with entries near `10^9` totals about `2*10^10`, exact in `long long`, over `2^20 * 20 ≈ 2.1*10^7` transitions and `8` MB of `dp` — comfortably inside the 2 s / 256 MB limits. Reading with `cin >>` consumes arbitrary whitespace, so the row-by-row matrix parses regardless of spacing, and the output is one integer and a newline. The full self-contained program is the answer.
