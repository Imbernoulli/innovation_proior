The thing that fixes the data types here, before any algorithm, is the arithmetic scale, and it is unusually aggressive for a counting problem. With `n` up to `2*10^5` the number of pairs `n*(n-1)/2` reaches about `2*10^10`, well past the 32-bit signed ceiling of `~2.1*10^9`, so the count itself has to be `long long`. Each `a[i]` runs to `10^9` in magnitude and `T` to `2*10^9`, so a pair sum `a[i]+a[j]` sits in `[-2*10^9, 2*10^9]` — already outside 32-bit — and the test `a[i]+a[j] >= T` must be evaluated in 64-bit or it silently wraps and flips admissibility. Values, threshold, and accumulator are all `long long`. That is decision one, and the negatives and zeros this problem is built around are exactly what make the sign and the boundary easy to get wrong.

Brute force over all pairs is `O(n^2)`, about `2*10^10` operations at the top end — dead against the 1 s limit. Sorting is what unlocks a linear count: once the array is monotone, "does some `a[j]` reach `T - a[i]`" turns from a value scan into a position question.

Two sorted approaches present themselves. A binary search: for a fixed left index, the partners with `a[j] >= T - a[i]` form a contiguous suffix locatable by `lower_bound`, sum the suffix lengths, `O(n log n)` — but I have to dodge both the suffix-start off-by-one and double-counting, and forming `T - a[i]` is itself an overflow site (`2e9 - (-1e9) = 3e9` overshoots even 32-bit). A converging two-pointer sweep avoids the inner search entirely: sort ascending, `lo = 0`, `hi = n-1`, and when `a[lo]+a[hi] >= T`, every index `k` in `[lo, hi-1]` also clears it (because `a[k] >= a[lo]` in ascending order), contributing `hi - lo` pairs at once, then `hi--`; otherwise `a[lo]` cannot reach `T` even with the largest remaining partner, so `lo++`. I take the two-pointer sweep — no per-element search, a single arithmetic increment, and each unordered pair is counted exactly once because `hi` only ever pairs with strictly-smaller positions.

The increment is the one place to be careful, and it is a genuine trap here. When `a[lo]+a[hi] >= T` I pair `hi` with the *closed* range `[lo, hi-1]` — that range includes `lo` itself, so it is `hi - lo` partners, not `hi - lo - 1`. The open-interval count `hi - lo - 1` counts the strict interior and drops the pair `{lo, hi}`. On `n=2, T=1, a=[-2,8]` this is unambiguous: sorted `[-2,8]`, `-2+8 = 6 >= 1`, the correct increment `hi - lo = 1` reports the single pair, while `hi - lo - 1 = 0` would report no pairs where there is plainly one. So the increment is `hi - lo`. After adding it I retire `hi` (`hi--`) but leave `lo`, which still has to meet the smaller remaining `hi` values. In the other branch, `a[lo]+a[hi] < T`, the largest remaining partner already fails to lift `a[lo]` to `T`, so `a[lo]` can pair with nothing left and I drop it (`lo++`). Each step removes one element, so the sweep is `O(n)`.

Whether the comparison is `>=` or `>` is a real decision, not cosmetics, and the sample pins it down. For `n=6, T=3, a=[4,-1,0,5,-3,2]`, sorted `[-3,-1,0,2,4,5]`, the sweep hits `lo=1, hi=4` with `-1+4 = 3` — a sum *equal* to `T`. The problem calls that admissible, so I need `>=`; a strict `>` would drop `{-1,4}` and undercount. With `>=` the walk lands on 7, matching the given answer.

The base case needs an explicit guard, and here the negatives resurface as index arithmetic. At `n=1`, `hi = n-1 = 0`, the loop condition `0 < 0` is false, the sweep never runs, count 0 — correct, a lone thruster forms no pair. At `n=0`, `hi = n-1 = -1`, and `0 < -1` is also false, so it prints 0 too — but that leans on an invalid, negative `hi` making the comparison false. I do not want correctness resting on `int` producing `-1`: a later peek at `a[hi]` before the loop would read `a[-1]`. So I add `if (n < 2) { print 0; return; }`, and the sweep only ever runs with two or more valid elements.

The negatives put the sum right at the 32-bit edge, so a worked case is worth having. Take `n=2, T=-2000000000, a=[-1e9, -1e9]`: the sum is `-2*10^9`, equal to `T`, so the pair is admissible and the answer is 1. That sum sits just above `INT_MIN`, but push one value to `1e9` against a positive `T` and a `1e9 + 1e9 = 2e9` sum sails past `INT_MAX` — a 32-bit evaluation wraps and flips admissibility. In 64-bit it stays exact: `-2000000000LL >= -2000000000LL` holds, add 1, output 1.

The sign-driven corners behave as the sweep predicts. All zeros at `T = 0`: every sum is `0 >= 0`, so all `n*(n-1)/2` pairs count. All-negative against a threshold no reverse-thrust pair can reach: the `>=` branch never fires and the count stays 0. All-negative with a very low `T`: every pair clears, the `>=` branch fires at every step, and the sum runs up to `n*(n-1)/2` — the worst case for the accumulator, `~2*10^10`, which is exactly why it is `long long`. A `T` above every reachable sum: `lo` walks up to meet `hi` with no `>=` branch, count 0.

For assurance beyond the traces I check the sweep against an obvious `O(n^2)` brute force that enumerates every pair and tests the threshold directly, over many small random cases spanning all-negative, all-zero, mixed-sign arrays, thresholds below, inside, and above the reachable range, and `n` in `{0, 1, 2..10}` — so the empty, single-thruster, and all-negative-boundary cases all fall inside the distribution rather than being asserted separately. They agree everywhere. At `n = 2*10^5` with `|a[i]|` near `10^9` the sweep is dominated by the sort and finishes well under the 1 s limit.

That is everything the code has to encode: read as `long long`, guard `n < 2`, sort ascending, sweep the two pointers adding `hi - lo` on the `>=` branch. The full module is in the answer.
