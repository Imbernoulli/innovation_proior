The cost in this problem does not sit where a graph-search reflex wants to put it. I pay one *transfer* each time two consecutive connections carry different line labels; riding on within a single label is free, and the first board is free. So the objective is line-changes along a walk, not edges and not stations — and that mismatch is the whole trap, because the route with the fewest connections can cost *more* transfers than a longer route that stays on one line. Scale fixes the data structures: `n, m <= 2*10^5`, labels up to `10^9`. Labels are large and sparse, so I compress them to a contiguous range before anything else — no `10^9`-sized arrays. The answer is bounded by `m` (an optimal walk never repeats a state, so it makes at most `m` line changes), so it fits in a 32-bit `int`; only the label magnitude needs care, and compression handles that.

One contract subtlety has to be nailed before coding, because a first implementation will get it wrong. "Free to stay on a line" does *not* mean "any two stations on line `c` are mutually free." It means: having just arrived on a line-`c` connection, I may continue along another line-`c` connection incident to the *same station* for free. Free movement on line `c` is therefore confined to a *connected component of the subgraph formed by line-`c` connections*. Two connections that share the label `c` but sit in different parts of the network are not mutually free — reaching one from the other costs a transfer onto and off of some bridging line.

Is a plain approach enough? The tempting move is a BFS over stations for the fewest-connections route, or a greedy that extends the current line and switches only when forced. Both are `O(n+m)` and a handful of lines, but they measure connections, or make local switch decisions, while I pay for global line-changes, so a counterexample settles it. Stations `0..4`, target `4`, connections `(0,2,line 1)`, `(2,4,line 2)`, `(0,1,line 5)`, `(1,3,line 5)`, `(3,4,line 5)`. Fewest connections is `0 -> 2 -> 4`: two connections, but line 1 then line 2 — one transfer. Yet `0 -> 1 -> 3 -> 4` rides line 5 the whole way: board once (free), continue twice for free — zero transfers, on the *longer* route. So the optimum is `0`, strictly below what connection-counting reports. Fewest connections is not fewest transfers; greedy and plain BFS are out.

Since "free to stay, pay 1 to switch" is a graph with weights in `{0,1}`, this is the 0-1 BFS setting (a deque: weight-0 relaxations to the front, weight-1 to the back). The state must know which line I am on to decide whether the next move is free, so the natural state is `(station, last-line)`: from `(u, last)`, riding `(u,v,c)` costs `0` if `c == last` or if `last` is "none" (first board), else `1`, landing in `(v, c)`. The answer is the minimum `dist[(n-1, *)]` over lines, or `0` when `n-1 == 0`.

I code that state directly — an adjacency list of `(neighbor, line)`, a `dist` map keyed by encoded `(station, line)`, a deque, and a stale-pop guard so a state improved after queueing is not re-expanded. It agrees with hand-traces on the sample and on `(0,1,10),(1,2,10),(2,3,20),(3,4,30)` (target 4: line 10, free to 10, switch to 20, switch to 30 = 2 transfers). Against an independent Dijkstra over `(station, last-line)`, a few hundred random small cases give zero mismatches. Correct — but correctness is not the constraint that bites here.

The bound is. Consider one hub station of degree `D`, each spoke on its own distinct line, plus a single edge into the hub. The hub is entered on `D` different lines, so it accumulates `D` distinct states `(hub, line)`, and popping each one rescans all `D` incident connections to relax onward — `O(D^2)`. Timed at `D = 20000` it takes about 1.6 s; at the real bound `D` up to `2*10^5` that is minutes, a dead TLE. The `(station, last-line)` state is correct but its per-line re-expansion of a station is quadratic. I need to stop re-examining a station's edges once per in-line.

The fix is to compute "the set of stations reachable for free once on line `c`" once and share it. That set is a *line-component*: within line `c`, the stations partition into connected components, and inside one component I roam free. So I build a layered graph: stations `0..n-1` are ordinary nodes; each `(line, connected component)` mints one super-node; a station connects to a super-node with weight `1` (boarding / switching onto that line-component) and the super-node connects back to each of its stations with weight `0` (riding free within it). Now `dist[station]` counts *boardings*, and since the first board is free, `transfers = boardings - 1`; the answer for `n-1` is `dist[n-1] - 1`. Each super-node is relaxed once, so the work is `O(n+m)` regardless of hub degree.

There is a wrong way to build this, and it is the contract subtlety from the start biting back. If I mint one super-node per line *label* and attach every station touching line `c` to it, the BFS can ride from a station in one line-`c` segment "through" the shared node into a physically disconnected line-`c` segment — a free teleport that does not exist, fabricating reachability. The oracle exposes it at once: a random case where station `0` cannot reach the target at all, yet the one-node-per-label code returns `0` while the brute, counting transfers along real walks, returns `-1` and is right. So the super-node must be per `(line, component)`, not per label. For each line I take only its connections, reset its touched stations to singletons in a DSU, union along that line's edges, and mint one super-node per DSU root; each touched station attaches to its own component's node, so two disjoint segments of a label get two different nodes and cannot teleport. The reset is scoped to the touched stations so `find` never escapes them, and the total reset+union over all lines is `O(m * alpha)`. With the fix the oracle passes, including a generator that reuses a tiny label pool specifically to force disconnected same-label segments — zero mismatches.

On the sample the offset does its job: line 5 is one component over `{0,1,3,4}` -> node `P`, so boarding `P` from `dist[0]=0` costs `1` and riding it free gives `dist[4]=1` — one boarding, answer `dist[4]-1 = 0`. The two-line route reaches `4` at `dist=2` (two boardings, one transfer), and BFS keeps the smaller value.

The corners cluster where this kind of code dies, so I pin each. `n = 1` (start == destination): special-case `n-1 == 0` to `0` before building anything, so the `-1` offset never fires on a never-boarded destination. Unreachable: if `dist[n-1]` stays `INF` I print `-1`, always testing `>= INF` before subtracting so I never compute `INF - 1`. A single direct connection `n=2, (0,1,9)`: one free board, `dist[1]=1`, answer `0`. `m=0` with `n>=2`: no super-nodes, `dist[n-1]` stays `INF` -> `-1`. Self-loops union a station with itself and never move you — harmless. Duplicate / parallel connections add only duplicate `(station, node)` edges, a constant factor, and the union-find dedups them into one component. Distances are bounded by `m`, well inside `int`.

At `n = m = 2*10^5` — the hub that detonated the first model, a dense single line, and a maximal path-plus-random case — the layered 0-1 BFS runs comfortably under the 2-second limit in about 43 MB and agrees with the independent brute across thousands of cases.
