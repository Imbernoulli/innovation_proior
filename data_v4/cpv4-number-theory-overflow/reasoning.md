**Fixing the scale before the algorithm.** I have `n` oscillators with integer frequencies `a[0..n-1]`, a modulus `m`, and a target residue `t` with `0 <= t < m`; I count the unordered pairs `{i, j}`, `i != j`, with `(a[i] + a[j]) mod m == t`, reading `n m t` then the `n` frequencies from stdin and printing one integer. The distinctive thing here is not the counting rule â€” it is the magnitudes. With `n` up to `2*10^5`, the number of pairs alone reaches `C(n, 2) = n*(n-1)/2 â‰ˆ 2*10^10`, roughly ten times `INT_MAX â‰ˆ 2.147*10^9`. The answer simply does not fit in a 32-bit `int`. And frequencies up to `10^9` mean a raw sum `a[i] + a[j]` reaches `2*10^9`, already brushing the signed 32-bit ceiling. So the type discipline is the real content: every quantity that *counts pairs* or *sums two frequencies* has to be 64-bit, or the large tests come back as silent wrong answers, not crashes.

**Approach.** Brute force over all `i < j`, testing `(a[i] + a[j]) % m == t`, is obviously correct but `O(n^2) â‰ˆ 4*10^10` at the top of the range â€” two orders past a 1-second limit. It survives only as an oracle to check the fast version against. The fast version: the predicate depends only on `a[i] mod m` and `a[j] mod m`, so I reduce every frequency to its residue, count how many fall in each residue class, and count *residue pairs* that complete to `t`. There are at most `m <= 10^6` residues, so a single pass over them is `O(n + m)`.

**Deriving the bucket count.** Let `cnt[r]` be the number of indices `i` with `a[i] mod m == r`, for `r in 0..m-1`. A pair `{i, j}` is resonant iff its residues `(r, s)` satisfy `(r + s) mod m == t`. For a fixed `r`, the completing residue is forced: `s = (t - r) mod m`. But I have to count *index pairs*, not residue pairs, and there are two genuinely different cases:

- **Distinct residues, `r != s`.** Every frequency in bucket `r` pairs with every frequency in bucket `s`, giving `cnt[r] * cnt[s]` distinct resonant index pairs. The unordered residue pair `{r, s}` is however visited twice by a full `0..m-1` loop â€” once at `r`, once at `s` â€” so its contribution must be added exactly once.
- **Self-paired residue, `r == s`** (i.e. `2r â‰¡ t (mod m)`). Both elements come from the same bucket, so I choose 2 distinct indices out of `cnt[r]`: `C(cnt[r], 2) = cnt[r] * (cnt[r] - 1) / 2`. Not `cnt[r] * cnt[r]`, since an index cannot pair with itself and order does not matter.

So: as `r` runs `0..m-1`, compute `s = (t - r) mod m`; add `cnt[r] * cnt[s]` only when `r < s`, add `C(cnt[r], 2)` when `r == s`, and skip `r > s` (handled when the loop variable was the smaller residue). Each unordered residue pair is visited once.

On the sample `m = 5`, `t = 3`, `a = [7, 1, 4, 6, 9, 2]`: residues mod 5 give `cnt[1] = 2`, `cnt[2] = 2`, `cnt[4] = 2`, the rest 0. Residue pairs summing to `3 (mod 5)`: `1 + 2` (distinct) and `4 + 4` (self, since `8 â‰¡ 3`). So `cnt[1]*cnt[2] + C(cnt[4], 2) = 4 + 1 = 5`, matching the expected `5`.

**Why the flat loop fails.** The naive `answer += cnt[r] * cnt[s]` over all `r` breaks two ways. On `m = 6`, `t = 0`, `a = [6, 12, 18]` â€” all residue 0, truth `C(3, 2) = 3` â€” it adds `cnt[0] * cnt[0] = 9` at `r = 0`, counting each index with itself and each unordered pair in both orders: the diagonal failure. Off the diagonal, a distinct residue pair completing to `t` gets added twice, once at `r` and once at `s`. The `r < s` / `r == s` / skip split kills both â€” `C(cnt[r], 2)` on the diagonal, exactly one visit per distinct pair off it.

**The overflow, the headline trap.** The counting logic is right; now the magnitudes. Worst case: `n = 2*10^5`, `m = 2`, `t = 0`, every frequency even, so `cnt[0] = 200000` and every pair is resonant. The answer is the self term `cnt[0] * (cnt[0] - 1) / 2 = 200000 * 199999 / 2 = 19,999,900,000 â‰ˆ 2*10^10`. The subtle part is *where* an `int` bites. In C++ the product `cnt[r] * cnt[s]` (or `cnt[r] * (cnt[r] - 1)`) is evaluated in the operands' type *before* being added to `answer` â€” so making only `answer` a `long long` while leaving `cnt` as `int` still overflows inside the product. With `int cnt`, `200000 * 199999 = 39,999,800,000` wraps mod `2^32` to `1,345,094,336`, and the `/ 2` then yields `672547168` â€” which is exactly what the `int` version prints on this input instead of `19999900000`, no warning. The fix has to cover every arithmetic site that can grow past 32 bits: the counts, the products (`~4*10^10`), and the accumulator (`~2*10^10`). Making `cnt` itself `vector<long long>` does it in one stroke â€” every product and the accumulation are then 64-bit, and `~4*10^10` sits far inside the `~9.2*10^18` signed 64-bit range.

**Negative modulo on the completing residue.** I compute `s = (t - r) mod m`. With `t, r` both in `0..m-1`, `t - r` ranges over `-(m-1)..(m-1)`, so a bare `(t - r) % m` can be negative in C++ and index `cnt[-2]` is undefined behavior. Guard it as `s = ((t - r) % m + m) % m`: for `t = 1, r = 3, m = 5` this gives `-2 â†’ +5 â†’ 3`, and indeed `3 + 3 = 6 â‰¡ 1`, the correct completing residue. I apply the same form when reducing each frequency â€” harmless since `a[i] >= 0`, but it keeps the reduction sign-agnostic.

**Edge cases.** `n = 0`: the read loop never runs, all buckets 0, output `0` (and if stdin is empty the guarded `cin >> n >> m >> t` fails and I return, printing nothing). `n = 1`: one bucket of size 1, every term is a zero factor or the self term `1*0/2 = 0`, output `0`. `m = 1`: the only residue is 0 and `t = 0` is forced, so `cnt[0] = n` and the answer is the self term `C(n, 2)` â€” every pair is resonant mod 1, and at `n = 2*10^5` this is another `~2*10^10` big-answer case. `C(k, 2) = k(k-1)/2` is always exact since `k(k-1)` is even, so the integer division never rounds regardless of parity. Frequencies up to `10^9` reduce fine under `%`; the raw `2*10^9` sum that would threaten a brute-force `int` never appears, because the fast solution works purely in residues and counts.

**Validation.** I ran the residue-bucket solution against the `O(n^2)` brute force on random small cases with `m` kept in the single digits and a dozen or so frequencies, so self-paired residues and multiply-populated buckets occur often â€” precisely the regime that stresses the self-pair and double-count logic â€” and it matched everywhere. The small random cases are too small to reach 64-bit territory, so the separately constructed all-even `n = 2*10^5` case does that job, printing `19999900000`.

The submission is one self-contained pass: bucket the frequencies by residue into a `vector<long long>`, then the `r < s` / `r == s` / skip loop â€” with 64-bit types at every counting site, the one decision this problem is really about.
