One string `s`, up to `2*10^5` lowercase letters, and I sum `occ(k)` over every prefix length `k = 1..n`, where `occ(k)` counts the start positions at which `s[0..k-1]` appears as a contiguous substring of `s` (overlaps allowed, distinct starts are distinct). The score is a single integer; read `s` from stdin, print the sum. The thing that decides the shape of the code first is not the algorithm but the range of the answer. The most self-similar string is `"aaaa...a"`: there a prefix of length `k` starts at every position `0..n-k`, so it occurs `n-k+1` times and `score = sum_{k=1}^{n} (n-k+1) = n(n+1)/2`. At `n = 2*10^5` that is `200000*200001/2 = 20000100000`, nearly ten times past the 32-bit signed cap `2147483647`. So the final accumulator has to be 64-bit — an `int` there is a silent wrong answer, not a crash — and I settle that before writing a line.

Two ways to compute the score. Counting each of the `n` prefixes by a separate scan of `s` is `O(n^2)`, about `4*10^10` operations at the limit: dead under one second, useful only as a brute oracle on tiny inputs. The route that fits is a single KMP prefix-function pass, and it fits precisely because the patterns are the string's own prefixes. That self-reference is what makes the single pass possible: the prefix function `pi[i]` (length of the longest proper border of `s[0..i]`) already names, for each position `i`, the longest prefix ending there, and the shorter prefixes ending at `i` are exactly its border chain `i -> pi[i]-1 -> ...`. The only real work left is the aggregation order and the off-by-one in mapping `pi` values to prefix lengths.

Which prefixes end at position `i`? The full prefix `s[0..i]` of length `i+1` ends there trivially. Beyond that, the prefixes ending at `i` are those of length `pi[i]`, then `pi[pi[i]-1]`, and so on down the border chain to `0`. So to get `occ(L)` for every length `L`:

1. Tally the deepest border at each position: for every `i`, the prefix of length `pi[i]` ends at `i`, so `occ[pi[i]]++`.
2. Push those tallies down the chains, longest length first: for `L` from `n` down to `1`, `occ[pi[L-1]] += occ[L]`. A prefix of length `L` ending somewhere also has its own border of length `pi[L-1]` ending at the same spot.
3. Add each prefix's own appearance at the very start of `s`, which is not a border of anything: `occ[L]++` for `L = 1..n`.

Then `occ(L) = occ[L]` and `score = sum_{L=1}^{n} occ[L]`; bucket `occ[0]` is the empty-prefix chain sink and is never summed. The two easy-to-miss points are that step 1 indexes by `pi[i]` — the border *length* — and not `pi[i]+1`, and that the step-3 `+1` is separate mass.

I lock the off-by-one on `s = "ababa"`, where the statement says the answer is `9`. Characters `a b a b a`, positions `0..4`. The prefix function: `pi[0]=0`; `pi[1]`: `s[1]='b'` vs `s[0]='a'` gives `0`; `pi[2]`: `j=0`, `s[2]='a'==s[0]` gives `1`; `pi[3]`: `j=1`, `s[3]='b'==s[1]` gives `2`; `pi[4]`: `j=2`, `s[4]='a'==s[2]` gives `3`. So `pi=[0,0,1,2,3]`. Step 1 over those values: `occ[0]+=2, occ[1]+=1, occ[2]+=1, occ[3]+=1`, i.e. `occ=[2,1,1,1,0,0]`. Step 2, `L=5..1`: `L=5` `occ[pi[4]=3]+=occ[5]=0`; `L=4` `occ[2]+=occ[4]=0`; `L=3` `occ[1]+=occ[3]=1 -> occ[1]=2`; `L=2` `occ[0]+=occ[2]=1 -> occ[0]=3`; `L=1` `occ[0]+=occ[1]=2 -> occ[0]=5`; now `occ=[5,2,1,1,0,0]`. Step 3 adds `1` to `L=1..5`: `occ=[5,3,2,2,1,1]`. That gives `occ(1..5) = 3,2,2,1,1`, summing to `9`, and it matches the by-hand occurrence list in the statement. The aggregation order and both off-by-ones are right. A brute oracle (per-prefix substring count) agrees on this and on ~500 random small strings, zero mismatches, so the small-case logic is settled.

Now the width trap, which the small comparisons structurally cannot reach: at `n <= 14` the largest possible score is `14*15/2 = 105`, nowhere near `2^31`, and a random small generator never visits the periodic extreme. So I have to reason about the large case directly rather than test into it. Take `s = "a"` repeated `200000`. After the three passes `occ[L] = n-L+1`, and the accumulator sums `n + (n-1) + ... + 1`. In 32-bit that running total crosses `2147483647` partway through and every further add wraps mod `2^32`, so the final value is `20000100000 mod 2^32` read as signed: `20000100000 - 4*2^32 = 2820230816`, still above `2^31`, hence `-1474736480`. A negative score is impossible for a sum of nonnegative counts, so that negative output is itself the overflow tell: an `int` build prints exactly `-1474736480`, while the `long long` build prints `20000100000`, matching the closed form `n(n+1)/2`.

The one place I was tempted to overreact is the count array itself. Each `occ[L]` for `L >= 1` is a single prefix's occurrence count, so it is at most `n`. The sink `occ[0]` absorbs whatever falls off the bottoms of the border chains; for `"aaaa...a"` that lands at exactly `n`, and I can't construct a string that pushes it past a small multiple of `n` — the buckets never approach `2^31`. It is only the *sum* over buckets that reaches `2*10^10`. So strictly I could leave `occ[]` as `int` and widen only `score`. But the uniform, cost-free fix is to make the whole count array `long long`; `score += occ[len]` then accumulates in 64-bit by construction, with no per-bucket width to track. That is what I use.

One detail in the step-2 loop: I write the bound `i = n .. 1`, one iteration higher than the `n-1 .. 1` form more usually seen. The `i = n` step does `occ[pi[n-1]] += occ[n]`, and `occ[n]` is `0` after step 1 (no `pi[i]` can equal `n`, since `pi[i] <= i < n`), so that top iteration adds nothing — the extra bound is a harmless no-op, and the `i = 1` step folds the length-1 count into `occ[0]`, which I never read. The loop is correct as written.

The corners fall out of the same recurrence. Empty input: `cin >> s` fails, I print `0` — no prefixes, score `0`. `n = 1`, `s = "z"`: the `pi` loop never runs, step 1 sets `occ[0]=1`, step 3 sets `occ[1]=1`, score `1`. Unique-alphabet string like `"abcd"`: every `pi[i]=0`, so all step-1 mass lands in `occ[0]` and the propagation only ever touches `occ[0]`; step 3 makes each `occ[L]=1`, score `= n`, confirming the border machinery invents no phantom occurrences. And `"aaaa...a"` at `n = 2*10^5` gives score `20000100000`, well inside `long long`, with buckets around `n`.

Prefix function is `O(n)` amortized and the three aggregation passes plus the final sum are each `O(n)`, so total `O(n)` time and memory; at `n = 2*10^5` it runs in a few milliseconds, far under the 1-second limit, with no I/O tuning needed. The finished program is the standard prefix-function computation followed by exactly the three long-long passes derived above and the final sum — the full module is in the answer.
