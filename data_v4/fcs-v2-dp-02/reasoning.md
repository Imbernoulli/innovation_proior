Three numbers decide the algorithm here: `B` up to 20, so `2^B` is at most about `1.05*10^6`; `n` and `q` each up to `10^6`. Every complexity that would blow the 2-second budget is a *product* of two of those — `n*q` for a naive scan, `q*2^B` if I answered each query by touching every mask, `3^B` for the obvious lattice aggregation — so the whole game is keeping the work to a single one of them (or their sum). Counts never exceed `n = 10^6`, comfortably inside an `int`, but I will accumulate in `long long` so a sum of counts is never a question I have to revisit. For each query `m` I need the number of items `x` with `x & m == m`: the *supersets* of `m`, those that carry every bit of `m` and possibly more.

Two routes. The per-query scan tests `x & m == m` over all `n` items for each query: `O(n)` per query, `O(n q)` overall, which at the limits is `10^{12}` AND-tests. Even at an optimistic `10^9` ops per second that is on the order of `10^3` seconds against a 2-second limit — off by three orders of magnitude, not close. It survives only as a brute oracle on tiny `B`. So I aggregate once. The queries only ever read a function of the item multiset — for a given mask, how many items are supersets of it — and that function does not depend on which query asked. So I precompute `S[m]` for **every** mask `m` in `[0, 2^B)` and answer each query with a single array read.

The real question is how to build `S[]`. By definition `S[m] = sum of cnt[t]` over all `t` with `t ⊇ m`, where `cnt[t]` is the number of items equal to mask `t`. The textbook-obvious build enumerates, for each mask `m`, its supersets and sums their `cnt`. The total cost is `sum over m of 2^{zeros(m)}`, and a mask with `k` zero bits has exactly `2^k` supersets, so that sum is the familiar `3^B`. Concretely at `B=2`: `S[00]` sums 4 masks, `S[01]` and `S[10]` sum 2 each, `S[11]` sums 1 — `4+2+2+1 = 9 = 3^2`. At `B=20` it is `3^{20} ≈ 3.49*10^9`, borderline-to-impossible in 2 seconds, and every extra bit multiplies it by 3 while `2^B` only doubles. Against `B*2^B ≈ 2.1*10^7` the gap is about 170x, so the whole problem is finding that `O(B*2^B)` build.

The waste in the `3^B` version is recomputation: when I aggregate the supersets of `m`, I redo work that the aggregation for masks one bit denser than `m` already did. The fix is the fast zeta transform over the subset lattice, run in the *superset* direction. Process the bits one at a time, `b = 0..B-1`, maintaining the invariant that after processing bit `b`, `f[m]` is the sum of `cnt[t]` over all `t` that may carry extra 1s among bits `0..b` but match `m` exactly on bits `> b`. The transition follows from that invariant. For a mask `m` with bit `b` clear, a superset may leave bit `b` at 0 or set it: the "leave 0" case is already in `f[m]`, and the "set to 1" case is exactly `f[m | (1<<b)]` from the previous round, so `f[m] += f[m | (1<<b)]`. For a mask with bit `b` already set, every superset keeps that bit, so nothing changes. After all `B` bits, `f[m]` is the full superset sum `S[m]`. That is `B` passes over `2^B` masks: `O(B*2^B)`.

Checking the transform on `B=2`, `cnt = [c0,c1,c2,c3]` indexed by mask value. Bit 0: masks `0,2` are clear, so `f[0]+=f[1]`, `f[2]+=f[3]`, giving `[c0+c1, c1, c2+c3, c3]`. Bit 1: masks `0,1` are clear, so `f[0]+=f[2]`, `f[1]+=f[3]`, giving `f[0]=c0+c1+c2+c3`, `f[1]=c1+c3`, `f[2]=c2+c3`, `f[3]=c3` — exactly `S[00],S[01],S[10],S[11]`.

The in-place update is the subtle part, so the direction matters. Writing the fold as `f[mask] += f[mask ^ (1<<b)]` for every mask is wrong twice over: it also updates bit-*set* masks, which must stay fixed because a superset cannot drop a set bit, and by flipping with `^` it reads a neighbour that this very pass already mutated — a read-after-write. Both defects vanish if I guard the update to bit-clear masks and read the partner `f[mask | (1<<b)]`. That partner has bit `b` set, so it is never a write target during this pass; the in-place fold then computes exactly what a fresh copy would.

```
for (int b = 0; b < B; b++) {
    int bit = 1 << b;
    for (int mask = 0; mask < SZ; mask++)
        if ((mask & bit) == 0)
            f[mask] += f[mask | bit];
}
```

The corners, where mask code usually breaks. `B=0`: `SZ=1`, the only mask is `0`, the bit loop runs zero times, `f[0]=n`, and any query (necessarily `0`) returns `n`. `n=0`: `f` is all zeros, every query returns `0`. Query `m=0`: mask `0` has every bit clear, so it accumulates every state and `S[0]=n`. Query `m=2^B-1`: the full mask is never a bit-clear mask, so `f[full]` is never written and stays `cnt[full]`, the count of items equal to the full mask, whose only superset is itself. Duplicate item masks accumulate in `cnt[t]` and the transform sums them by construction. A superset sum is at most `n=10^6`, so `long long` is pure defensiveness. For I/O, `2^B` longs is about 8 MB; with `n+q` up to `2*10^6` integers to read I set `sync_with_stdio(false)` and buffer the `q` outputs into one string before a single write, which keeps the run well under a second.

For evidence beyond the derivation, an independent `O(n q)` brute — scan all items per query, test `x & m == m` — on small random instances agrees line-for-line with the SOS build, as do the crafted corners (`B=0`, `n=0`, query `0`, query full, heavy duplicates), and the documented sample (`B=3`, items `7 6 3 5 4 1`, queries `3 1 0 7`) gives `2 4 6 1`.
