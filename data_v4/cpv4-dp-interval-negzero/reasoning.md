The thing that sinks a naive submission here is the sign. A fusion of two adjacent clusters with charges `L` and `R` welds them into one cluster of charge `L + R` and pays reward `L * R`, and I may stop at any point — so "do nothing" always scores `0` and the answer is never below `0`. The tempting shortcut, "everything is negative, so just print `0`," is exactly the trap: a negative times a negative is positive, so fusing two negative clusters *pays*, and an all-negative row usually has a large positive answer. Zeros are the other half of it — a cluster of charge `0` welds for free, neither helping nor hurting — so I have to get products right across all three signs without special-casing any of them. Input is `n` then the `n` charges on stdin; I print one integer.

Scale fixes the data types and the budget. `n <= 400`, `|c[i]| <= 10^4`. A cluster's charge is a sum of original charges, so `|charge| <= 400 * 10^4 = 4*10^6`; one reward is a product of two such charges, up to `~1.6*10^13`, and I sum up to `n` of them, `~6.4*10^15`. That is far past 32-bit range (`~2.1*10^9`), so every accumulator and every reward must be `long long` — an `int` is a silent wrong answer on the larger tests. And `n <= 400` is exactly the size where an `O(n^3)` algorithm fits (`6.4*10^7` inner steps, comfortable in 1 s) while `O(n^4)` at `2.6*10^10` would not.

The structure of a schedule is what makes it tractable. A fusion only ever combines *adjacent* clusters, and adjacency of contiguous runs is preserved by fusion, so at every moment the clusters are a partition of `0..n-1` into contiguous runs. When I stop, the final state is therefore a partition into contiguous blocks: each block is a maximal run I chose to fully fuse into one cluster, and any crystal I left alone is a singleton that paid nothing. Rewards are additive across blocks, because no weld inside one block ever involves a cluster from another. That decomposition is the whole game: (1) for one contiguous block, the best reward to fuse it into a single cluster, and (2) which blocks to actually fuse.

Greedy is tempting — repeatedly fuse the most-rewarding adjacent pair, or all same-sign runs — and cheap, but it is the wrong tool here: a cluster's charge, hence every future reward, shifts as I fuse, so a locally-best weld can poison later ones. And small examples won't expose that — on `[3,1,5,8]`, `[10,1,10]`, `[-5,4,-5]` a greedy one-weld-ahead rule happens to match the optimum, so passing a handful of cases would prove nothing either way. The DP below enumerates every weld order by construction, so I commit to it and drop greedy.

Inner interval DP. Let `mergeAll[i][j]` be the maximum total reward to fuse crystals `i..j` into a single cluster. The *last* weld must join two adjacent clusters that together cover `[i..j]`; being contiguous, they are `[i..k]` and `[k+1..j]` for some `k` with `i <= k < j`. Beforehand each half was independently fused (best rewards `mergeAll[i][k]` and `mergeAll[k+1][j]`, with charges `sum(i..k)` and `sum(k+1..j)`), and the final weld pays their product. So

  `mergeAll[i][j] = max over k in [i, j-1] of ( mergeAll[i][k] + mergeAll[k+1][j] + sum(i..k) * sum(k+1..j) )`,

with base case `mergeAll[i][i] = 0` — a single crystal is already one cluster, no weld, no reward. Contiguous sums come from prefix sums in `O(1)`. Checking against the sample `[3, 1, 5, 8]` (expected `95` for the whole range): the length-2 cells are `3, 5, 40`; `mergeAll[0][2] = 23` and `mergeAll[1][3] = 53`; then `mergeAll[0][3]` gives `95` at every split (`k=0`: `53 + 3*14`; `k=1`: `3 + 40 + 4*13`; `k=2`: `23 + 9*8`), matching.

Partition layer. I need not fuse the whole row — I choose a partition and fully fuse each block, singletons paying `0`. Let `best[p]` be the max total reward over the first `p` crystals `c[0..p-1]`. The last crystal `p-1` sits in some block `[q-1 .. p-1]`, contributing `mergeAll[q-1][p-1]` on top of `best[q-1]`:

  `best[p] = max over q in [1, p] of ( best[q-1] + mergeAll[q-1][p-1] )`, base `best[0] = 0`, answer `best[n]`.

The case `q = p` uses `mergeAll[p-1][p-1] = 0` — the "leave this crystal alone" option. That singleton transition does double duty: block rewards can be negative (`mergeAll` of `[3, -3]` is `-9`), so the DP must be free to leave crystals unmerged, and taking every crystal as its own singleton sums to `0`, which makes `best[n] >= 0` automatically with no separate `max(..., 0)`.

The one place this is genuinely easy to get wrong is the sentinel, and it splits into two *opposite* choices that must not be confused. The table diagonal must be `0`: a length-1 range is a real sub-result with reward `0`, and the length-`>=2` recurrence *reads* those diagonal entries as its two halves. If I instead left the whole table at a `-inf` sentinel and only filled `len >= 2`, then on `[2, 3]` the cell `mergeAll[0][1]` would compute `sentinel + sentinel + 2*3` — a garbage huge-negative — and the partition layer, seeing a hugely negative block, would prefer to do nothing and print `0` instead of `6`. But the *local* per-cell accumulator (the running max over `k`) must start at `-inf`, not `0`: if it started at `0`, a genuinely losing block like `[3, -3]` would be clamped to `0` instead of its true `-9`, and the partition layer would lose the information it needs to choose splitting. So: table diagonal `0` (a singleton pays nothing), local accumulator `-inf` (a negative block is reported honestly). I initialize the whole table to `0` and overwrite only the off-diagonal cells, each from a local `best` seeded at `NEG = LLONG_MIN/4`; the diagonal then stays `0` correctly and off-diagonal cells are never clamped.

With that fixed, `[2, 3]` gives `mergeAll[0][1] = 0 + 0 + 6 = 6` and `[3, -3]` gives `0 + 0 + 3*(-3) = -9`, the latter making the partition layer prefer two singletons for a `0` answer. And the headline trap, all-negative `[-2, -3, -1]`: length-2 cells `6` and `3`, then `mergeAll[0][2] = 11` (both splits: `3 + (-2)*(-4)` and `6 + (-5)*(-1)`), and the partition takes the whole block `11` over `0` — a positive answer, not a default `0`. I confirmed this and the other values against an independent exhaustive brute force.

Edge cases, where this kind of code dies:
- `n = 0`: no charges to read; I short-circuit and print `0`, avoiding a `0`-by-`0` matrix.
- `n = 1`, e.g. `c = [-7]`: the only block is a singleton, `best[1] = 0`. A lone crystal never pays.
- All zeros `[0,0,0]`: every charge sum and product is `0`, answer `0`.
- All negatives `[-3,-3,-3,-3,-3,-3]`: every adjacent product is positive, fusing everything wins (`135` by brute), confirming the sign handling scales past the length-3 case above.
- Mixed sign where stopping is best `[-3, 4, -3]`: products `-12, -12`, every full fusion is a net loss, so the DP leaves all three singletons for `0`. Two negatives `[-4, -5]`: `mergeAll[0][1] = 20`. Both checked against brute.
- Sentinel safety: the local accumulator's initial `NEG` is only ever the left operand of a `max` against a real candidate — the two halves it adds are always real (diagonal `0` or an already-computed cell), so no two sentinels are ever summed and nothing underflows.
- Performance: the inner table is `O(n^3) = 6.4*10^7` for `n = 400`, and `mergeAll` is `400*400*8 = 1.28 MB` — both well inside 1 s / 256 MB. `cin >>` skips arbitrary whitespace, so the parser is format-agnostic; output is one integer and a newline.
