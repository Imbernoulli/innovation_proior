This is the super-egg-drop problem, and its whole difficulty is that the textbook reflex is a trap. I have `k` eggs and a building of `m` floors with a critical floor `c` where a drop at `>= c` breaks and a drop below survives; broken eggs are gone, survivors reusable. I want the strategy whose *worst case* uses the fewest trials, and I output that count. The scale is `1 <= k <= 100`, `1 <= m <= 10000`, two integers on stdin, one out. The answer is a trial count bounded by `m = 10000` (the one-egg scan, below), so it fits a 32-bit int; but the "how many floors can I cover" quantities double per trial, so I hold those in `long long` and clamp them rather than let one overflow silently.

The reflex is binary search: drop from the middle, the break/survive answer halves the range, `ceil(log2(m+1))` trials. That is optimal only with *unlimited* eggs, and the test that matters is whether it survives a small `k`. Take `k = 1, m = 100`: binary search drops from floor 50, and if the egg breaks I have zero eggs left with `c` still unknown across floors `1..49` — the strategy simply cannot finish. It is not merely sub-optimal at one egg, it is *infeasible*. The only guaranteed one-egg strategy is the linear scan `1, 2, 3, …`: a break at floor `j` means `c = j`, no break means `c = m+1`, worst case `m = 100`. So the clever bound (`ceil(log2(101)) = 7`) is off by a factor of fourteen and unreachable. And with two eggs, `k = 2, m = 100`, the real answer is 14 — sacrifice the first egg on a coarsening ladder, then scan the ~14-wide gap where it broke with the second — again not 7. The number of times I am allowed to break an egg caps how much information each trial can buy, and binary search ignores that cap entirely. It is out; I want the minimax value itself.

The natural definition is over floors. Let `T(e, f)` be the fewest worst-case trials to pin `c` among `f` candidate floors with `e` eggs. One drop splits the range at some position `x`: if the egg breaks, `c` is among the `x - 1` floors below with `e - 1` eggs; if it survives, among the `f - x` above with `e` eggs; the adversary takes the worse branch. So `T(e, f) = 1 + min_{x=1..f} max( T(e-1, x-1), T(e, f-x) )`, with `T(e, 0) = 0` and `T(1, f) = f`. Correct by construction — it is literally the game-tree value — but the brute inner minimisation makes it `O(k * m^2) = 100 * 10^8 = 10^{12}`, far past a one-second limit. A monotonicity argument on the optimal split would bring it to `O(k * m)` but is fiddly and easy to get subtly wrong. I would rather have fast *and* obviously correct.

So I flip the question. Instead of "fewest trials for `f` floors," ask "most floors coverable with `e` eggs and `t` trials," call it `cover(e, t)`. One drop resolves `cover(e-1, t-1)` floors below it if the egg breaks, `cover(e, t-1)` above it if it survives, plus the drop floor itself: `cover(e, t) = cover(e-1, t-1) + cover(e, t-1) + 1`, with `cover(e, 0) = cover(0, t) = 0`. This is monotonically increasing in `t`, so the answer to the original question is just the smallest `t` with `cover(k, t) >= m` — no inner minimisation at all, because the `+1` recurrence already encodes the best balanced drop. That is the clean, provable, fast formulation; total work is `O(k * t_answer)`. I will ship this and keep the primal floors DP only as an independent oracle later, since the two are genuinely different algorithms.

Against the given examples the dual checks out. `cover(2, t)` runs `1, 3, 6, 10, … = t(t+1)/2`, the triangular numbers, so `k = 2, m = 6` needs `t = 3` and `k = 2, m = 100` needs the smallest `t` with `t(t+1)/2 >= 100`, i.e. `t = 14` (`14·15/2 = 105 >= 100 > 91 = 13·14/2`) — the classic "100 floors, 2 eggs" answer. And `cover(3, 4) = 4 + 6 + 4 = 14 >= 14` while `cover(3, 3) = 3 + 3 + 1 = 7`, so `k = 3, m = 14 -> 4`.

The one real hazard is the in-place transcription. I keep a 1-D array `cover[e]` and advance one trial at a time, updating every egg count in place. Both terms on the right, `cover[e-1]` and `cover[e]`, must be *previous-trial* values. `cover[e]` is read-then-written at the same index, so it is naturally the old value. The hazard is `cover[e-1]`: if I sweep `e` upward, I overwrite `cover[e-1]` this trial before `cover[e]` reads it. Tracing `kk = 2` from `[0, 0, 0]` on the first trial, sweeping up: `e = 1` sets `cover[1] = 1`, then `e = 2` reads that fresh `cover[1] = 1` and produces `cover[2] = 2` — but `cover(2, 1)` should be `1`. The upward sweep fed a same-trial value into the breaks term. Sweeping `e` from high to low fixes it exactly: computing `cover[e]` before touching `cover[e-1]` leaves the latter at its `t-1` value. Re-traced downward, the first trial gives `cover(1,1) = cover(2,1) = 1` and the second gives `cover(2,2) = 3`, matching the triangular value.

Two bounds keep every stored quantity small. With `t` trials, even unlimited eggs cover at most `2^t - 1` floors; since `2^14 - 1 = 16383 >= 10000`, no instance ever needs more than 14 trials once `k >= 14`, and with at most 14 trials I can break an egg at most 14 times, so a 15th egg can never be dropped. Capping the working egg count at `kk = min(k, 14)` is therefore exact, not an approximation, and holds the array to length 15. And since I only ever compare `cover[kk]` against `m`, I clamp each updated value at `m`; every stored value then stays `<= 10000`, `long long` is never stressed, and the clamp never touches the answer because the loop stops the instant `cover[kk]` reaches `m`.

The corners fall out of the same loop. `m = 1` exits at `t = 1` for every `k` (one drop settles floor 1 versus `m+1`). `k = 1` increments `cover[1]` by exactly one per trial and runs to `t = m`, the linear scan — and that is the longest the loop can run, `t = 10000`, still only `O(k · t) ~ 10^5` work. `k >= 14` caps at `kk = 14` and collapses to the unlimited-egg `ceil(log2(m+1))`; `k = 13` and `k = 14` both give 14 at `m = 10000`, confirming the 14th egg is the last that can matter. Termination is guaranteed since `cover[kk]` gains at least 1 each trial.

To trust it before shipping I differential-tested the dual `cover` against the primal floors DP `T(e,f) = 1 + min_x max(T(e-1,x-1), T(e,f-x))` implemented with a plain inner scan (transparently the game-tree value, and a genuinely different algorithm) over a few hundred random cases plus a curated edge set on the triangular and power-of-two boundaries — zero mismatches — keeping `m` small there so the quadratic oracle stays cheap. I separately checked the extreme `m = 10000` corners for `k in {1,2,3,13,14,50,100}`, which that oracle cannot reach quickly, against the closed form `cover(k, t) = sum_{i=1..k} C(t, i)`, also clean. Two independent algorithms agreeing across every case, including all the textbook values, is enough to ship the dual DP rather than the binary search I disproved. The full program is in the answer.
