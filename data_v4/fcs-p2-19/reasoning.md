The shape here is narrow: pick a subsequence of `a[0..n-1]` (original order) whose values strictly climb to one peak and then strictly fall. Two corners of the spec do all the work, and I nail them before touching an algorithm. First, both comparisons are *strict*, so equal values can never sit adjacent in the chosen chain — `[3, 3]` is not a climb, and `[5, 5, 1]` cannot spend both fives. Second, the ascent must be real: at least one step up before the peak (`p >= 1`), while the descent may be empty, so a strictly increasing run of length `>= 2` is already valid, but a plateau or a strictly falling array is not — those print `0`, as do `n <= 1`. Input is `n` then the values on stdin; output is one integer.

Scale decides the budget. `n <= 5000` and `|a[i]| <= 10^9`. I am computing a *length*, never a sum, so the answer is at most `5000` and an `int` holds everything; I read the values into `long long` only to keep the `10^9`-magnitude comparisons unambiguous. The load-bearing number is `n <= 5000`: an `O(n^2)` method runs about `2.5 * 10^7` comparisons, trivial under a 2-second limit. So I am free to pick the method I can *prove* rather than the asymptotically slick `O(n log n)` one.

The problem literally says "up then down," which tempts a one-pass greedy: sweep once, climb while the next element rises, flip to descending when it falls, track the longest mountain. It is `O(n)` and a dozen lines — and it is wrong, because it only ever compares *contiguous* neighbours, while a subsequence may delete arbitrarily many elements on either slope. The clean way to see the gap is to break it. Take `a = [1, 11, 2, 10, 4, 5, 2, 1]`. The true answer is `6`: `1 (i0) < 2 (i2) < 10 (i3) > 4 (i4) > 2 (i6) > 1 (i7)`, which requires skipping the `11` at index 1 and stitching `i0, i2, i3` across that gap. No length-7 chain exists — `11` can only be a peak, and `1 < 11` then `11 > 10 > 4 > 2 > 1` is again `2 + 5 - 1 = 6`. The contiguous walk cannot express "skip the 11": from index 0 it climbs `1 -> 11`, sees `11 -> 2` fall, then `2 -> 10` rise breaks the descent, so it closes a length-3 mountain and later scrapes `4, 5, 2, 1` of length 4 — it reports about `4`, never `6`. A neighbour-based walk solves the longest contiguous *substring* mountain; the problem asks for the longest *subsequence* mountain. Single-pass is out.

So I go to the method I can prove. Every bitonic subsequence has exactly one peak; left of it is a strictly increasing subsequence ending at the peak, right of it a strictly decreasing subsequence starting at the peak, and given the peak's index those two halves are independent. Let `inc[i]` be the length of the longest strictly increasing subsequence *ending* at `i`, and `dec[i]` the longest strictly decreasing subsequence *starting* at `i`. Then the best bitonic subsequence peaking at `i` has length `inc[i] + dec[i] - 1`, the `-1` because the peak is the last element of the ascent and the first of the descent — counted once. Both arrays are `O(n^2)` LIS-style DPs: `inc[i] = 1 + max{ inc[j] : j < i, a[j] < a[i] }` and `dec[i] = 1 + max{ dec[j] : j > i, a[j] < a[i] }`, each defaulting to `1`. Both use the *same* strict predicate `a[j] < a[i]` — for `inc` the smaller element is the earlier one, for `dec` the later one — which is the strictness enforcing itself and keeps me from writing the two loops backwards.

The one genuine trap is the "must actually increase" corner. If every index may be a peak, an index with `inc[i] = 1` contributes `1 + dec[i] - 1 = dec[i]` — a "peak" with no real step up. A strictly decreasing array is the sharp case: there `inc[i] = 1` everywhere and `dec[0] = n`, so an ungated combine returns `1 + n - 1 = n`, reporting the whole descent as bitonic when the answer must be `0`. The fix is to gate the combine on `inc[i] >= 2`: a peak must have at least one strictly smaller predecessor, which is exactly the `p >= 1` rule. With the gate, strictly decreasing arrays and equal-value plateaus have no qualifying index and return `0` — that single guard carries the entire "must go up first" clause of the contract.

On the sample the arrays come out `inc = [1, 2, 2, 3, 3, 4, 2, 1]` and `dec = [1, 5, 2, 4, 3, 3, 2, 1]`, and three different gated peaks tie at `6` — index 3 as `3 + 4 - 1`, index 1 as `2 + 5 - 1`, index 5 as `4 + 3 - 1` — so the optimal peak need not be unique.

The corners fall out of the recurrence plus the gate. `n = 0` and `n = 1`: no index reaches `inc[i] >= 2`, answer `0`. All-equal `[3, 3, 3, 3, 3]`: the strict predicate never fires, every `inc` and `dec` is `1`, answer `0`. Strictly increasing `[1, 2, 3, 4]`: `inc = [1, 2, 3, 4]`, `dec` all `1`, so the top peak gives `4 + 1 - 1 = 4` — the empty-descent case, where `dec[i] = 1` and the `-1` cancel cleanly. Duplicates mid-array `[2, 2, 3]`: `inc = [1, 1, 2]`, answer `2` for `2 < 3`, the duplicate harmlessly skipped.

I ship the both-ends DP — the version I could both prove and break-test against the single-pass walk I disproved. It reads `n`, special-cases the empty input, fills `inc` and `dec`, and takes the gated maximum of `inc[i] + dec[i] - 1`; the full program is in the answer.
