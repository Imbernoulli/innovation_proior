The array holds up to `3*10^4` integers as large as `10^9` in magnitude, and I want the sum, over all `n*(n+1)/2 ~ 4.5*10^8` contiguous subarrays, of each subarray's minimum. Two numbers settle everything before I pick an algorithm. First, the answer's magnitude: each of the `~4.5*10^8` subarrays contributes a minimum up to `10^9`, so the worst-case total is about `4.5*10^17` — four orders past the 32-bit ceiling of `2.1*10^9`, but a factor of twenty inside signed 64-bit's `9.2*10^18`. So `long long` is mandatory and also sufficient; no `__int128`. I pin that first because the natural way to phrase the answer multiplies three quantities and sums them, and if any of the three or the accumulator is an `int`, the program is a silent wrong-answer on exactly the large tests the judge will run. Second, values may be negative, so the answer itself can be negative — there is no `max(..., 0)` anywhere in this problem, unlike the maximum-subarray cousins whose templates I must not reach for by reflex.

Enumerating subarrays directly — fix `l`, sweep `r` with a running minimum — is `O(n^2) ~ 9*10^8` at the largest `n`, too slow for one second; I keep it only as a brute-force oracle. The submission has to be per-element contribution counting: instead of asking each subarray for its minimum, ask each element how many subarrays it is the minimum of. If `a[i]` is the minimum of `cnt[i]` subarrays then the answer is `sum_i a[i]*cnt[i]`, and a monotonic stack delivers every `cnt[i]` in `O(n)`.

`a[i]` is the minimum of `a[l..r]` exactly when `l <= i <= r` and nothing in the window is smaller than it. The `(l, r)` choices factorize: extend `l` leftward until just before the nearest strictly-smaller element, extend `r` rightward the same way, and the two choices are independent. So `cnt[i] = left[i]*right[i]`, with `left[i] = i - (nearest-smaller-to-the-left index, or -1)` and `right[i] = (nearest-smaller-to-the-right index, or n) - i`. Those nearest-smaller indices are the classic monotonic-stack output: keep a stack of indices whose values increase from bottom to top, and for each new `i` pop the ones that are no longer smaller. On the distinct sample `[3, 1, 2, 4]` this reproduces `17`: the global minimum `1` at index 1 reaches two positions left and three right for `1*2*3 = 6`, and `3`, `2`, `4` contribute `3`, `4`, `4` from their short runs, summing to `17`.

Equal values are where the factorization can double-count, so the strict-vs-non-strict convention needs pinning on an actual tie. Take the symmetric first cut: pop while the top is `>= a[i]` on *both* passes. On `a = [2, 2]`, whose true sum is `[2] + [2] + [2,2] = 6`, the left pass pops the equal left neighbour and gives `left = [1, 2]`; the right pass pops the equal right neighbour and gives `right = [2, 1]`; so `cnt = [2, 2]` and the program prints `2*2 + 2*2 = 8`. The overcount is exactly one extra copy of the minimum: with equals blocking neither direction, index 0 thinks it can extend right across index 1 and index 1 thinks it can extend left across index 0, so both claim the subarray `[2, 2]`.

The cure is to make exactly one side treat an equal element as a boundary. I keep the left pass strict — pop while `>= a[i]`, so an equal left neighbour is popped and does not block — but make the right pass pop only while `> a[i]`, so an equal element to the right survives as the boundary and stops rightward extension. Now on `[2, 2]`: `left = [1, 2]` as before, but the right pass no longer pops index 1 for index 0, giving `right = [1, 1]`, `cnt = [1, 2]`, and `2*1 + 2*2 = 6`. Distinct arrays are unaffected — with no ties, `>=` and `>` find the same boundary — so `[3, 1, 2, 4]` still gives `17`. And on the both-sides-equal case `[2, 1, 2]`, true sum `8`: the middle `1` reaches both `2`s and owns all four spanning subarrays (`1*2*2 = 4`), each outer `2` owns only its singleton, total `8`.

With the counting right, the arithmetic is the second and nastier trap — the one this problem is built around. The tempting wrong accumulation keeps the count and the running sum as `int`. On the adversarial case the judge surely ships, `n = 30000` with every `a[i] = 10^9`, the true answer is `(n*(n+1)/2)*10^9 = 450015000 * 10^9 = 4.50015*10^17`. But a single `left[i]*right[i]` already reaches `~n^2/4 ~ 2.25*10^8`, near the `int` ceiling, and multiplying by `a[i] = 10^9` overflows `int` on the spot; the all-`int` program wraps to `446230528` while the `long long` version prints the correct `450015000000000000`. The defect is purely width, not logic: `left` and `right`, the product, and the accumulator must all be 64-bit. Then each term is at most `~2.25*10^17` and their sum at most the `4.5*10^17` bound I fixed at the start — safely inside `long long`, which confirms `__int128` is unnecessary.

The corners fall out. `n = 0` (guarded by the `if (!(cin >> n))` read, though `n = 0` also just empties both loops) gives `0`, no subarrays. `n = 1`, `a = [-5]` gives `left = right = 1` and answer `-5` — negative, and correctly so, which is why the no-clamp discipline matters. All-negative arrays return a negative sum for the same reason. Strictly increasing `[1, 2, 3]` and strictly decreasing `[3, 2, 1]` both give `10`, matching the brute force. Reading is whitespace-agnostic via `cin >>`, and `sync_with_stdio(false)` keeps the `3*10^4`-token read comfortably under the limit.

Cross-checking the finished program against the `O(n^2)` brute force over many random small cases — negatives, zeros, tie-heavy arrays drawn from `{1, 2, 3}`, and large magnitudes near `+-10^9` — turns up no mismatch, so both the asymmetric tie rule and the 64-bit arithmetic hold. The full program is the module in the answer: two monotonic-stack passes (left strict, right non-strict), then a 64-bit accumulation of `a[i]*left[i]*right[i]`.
