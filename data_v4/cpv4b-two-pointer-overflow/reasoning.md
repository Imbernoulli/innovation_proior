Two separate accumulators here both blow past 32 bits, and that is the whole trap of this problem. I have `n` containers in a line, `n <= 2*10^5`, weights `w[i] <= 10^9`, crane capacity `S <= 10^18`, and I must count contiguous runs `[i..j]` whose total weight is `<= S` — read `n S` and the weights from stdin, print the single count. A single window sum can reach `n * max(w) = 2*10^5 * 10^9 = 2*10^14`. The answer itself is a count of runs, of which there are at most `n*(n+1)/2 ~ 2*10^10`. Both sit far above the signed 32-bit ceiling `2^31 - 1 ~ 2.147*10^9` — the window sum by five orders of magnitude, the count by a factor of ten. So I am carrying two independent 64-bit quantities, not one, and the count is the one I could forget precisely because it never appears in the window arithmetic. `S` also needs `long long`, reaching `10^18`.

Because all weights are non-negative, prefix sums are non-decreasing, and that monotonicity is the only structural lever. It supports a per-right binary search on prefix sums (`O(n log n)`, but it materializes an array of `2*10^14`-sized values and does a width-sensitive subtraction per query — more places to drop a 32-bit type by accident) or a two-pointer sliding window (`O(n)` time, `O(1)` space). I take the window: one running sum, one contribution formula, less surface to get the integer widths wrong.

Fix `right` and maintain a window `[left..right]` with a running `sum`. Since weights are non-negative, `sum(i..right)` is non-increasing in the start `i` — moving the start rightward only sheds weight — so the valid starts form a suffix `{left, ..., right}`. After I advance `right` (adding `w[right]`) and shrink from the left while `sum > S`, `left` is the smallest valid start, and every start in `[left..right]` yields a liftable run: contribution `right - left + 1`. The threshold `left` is itself non-decreasing in `right` (adding weight on the right can only push the smallest valid start further right), so `left` only ever advances and the inner loop costs `O(n)` across the whole scan.

The contribution formula is the part I can get subtly wrong, so I walk the sample `w = [3,1,4,1,5,2]`, `S = 7`:

- `right=0`: sum `3 <= 7`, `left=0`, contributes `1`. (`[0,0]`)
- `right=1`: sum `4 <= 7`, `left=0`, contributes `2`. (`[0,1]`, `[1,1]`)
- `right=2`: sum `8 > 7`; drop `w[0]=3` -> `5`, `left=1`; contributes `2`. (`[1,2]`, `[2,2]`)
- `right=3`: sum `6 <= 7`, `left=1`, contributes `3`. (`[1,3]`, `[2,3]`, `[3,3]`)
- `right=4`: sum `11 > 7`; drop `w[1]=1` -> `10`, still `>7`, drop `w[2]=4` -> `6`, `left=3`; contributes `2`. (`[3,4]`, `[4,4]`)
- `right=5`: sum `8 > 7`; drop `w[3]=1` -> `7`, `left=4`; contributes `2`. (`[4,5]`, `[5,5]`)

Total `1+2+2+3+2+2 = 12`, matching the enumerated runs one-for-one. And `left` took the values `0,0,1,1,3,4` — non-decreasing, advancing only four times over six right-ends, which is the amortization.

One corner the sample never reaches is a window that empties because no start is valid. Take `w = [5,1]`, `S = 3`. At `right=0`, `sum = 5 > 3`; the shrink drops `w[0]` and sets `left = 1`, and now `left > right`, so the `left <= right` guard halts it — contribution `0 - 1 + 1 = 0`. The heavy single is correctly not counted, and the guard is exactly what keeps this at `0` rather than a negative. At `right=1`, `sum = 1`, contribution `1`. Total `1`, the lone liftable run `[1,1]`.

Now the width discipline. There are two distinct overflows here, and they surface on different tests, so each needs its own construction to expose it.

If `count` were `int`, a large all-fits case exposes it: `n = 2*10^5`, every `w[i] = 10^9`, `S = 2*10^14`. The whole line fits, so every run is valid and the answer is `200000 * 200001 / 2 = 20000100000`. A 32-bit signed `count` holds values mod `2^32 = 4294967296`; reducing, `20000100000 - 4*4294967296 = 2820230816`, which exceeds `2^31`, so it prints as `2820230816 - 4294967296 = -1474736480`. The negative sign is the giveaway — there is no negative number of runs — but nothing below `2^31` triggers the wrap, so only a large case reveals it. The addend `right - left + 1` is at most `n`, a fine `int`; it is purely the running total that must be 64-bit.

If `sum` were `int`, that all-fits case hides the bug (the window never has to shrink), so I need a case where an over-capacity window must be rejected: two heavy containers `w = [2*10^9, 2*10^9]`, `S = 2*10^9`. Each single fits, the pair (`4*10^9`) does not, so the answer is `2`. But `2*10^9` fits in `int` while `4*10^9` does not: at `right=1`, `sum` becomes `4000000000 mod 2^32 = 4000000000`, past `2^31`, representing `-294967296`. Then the guard `sum > S` is `-294967296 > 2*10^9` — false — so the window never shrinks and the illegal pair is counted: answer `3`, not `2`. The overflow makes a too-heavy window look light and defeats the shrink test. Both accumulators go to `long long`; then `4*10^9 > S` holds, the window shrinks, and the count never wraps.

The remaining edges: `n = 0` gives an empty scan and count `0` (the `if (!(cin >> n >> S))` also guards a truncated stream). `S = 0` with all-zero weights leaves every prefix `<= 0`, the window never shrinks, count `n*(n+1)/2`. Capacity above the whole-line total behaves the same way — `left` never moves. A single container heavier than `S`, and any momentary window collapse, are the `[5,1]` case above: the `left <= right` guard bottoms the contribution out at `0`. With `sum` and `count` both 64-bit (max `~2*10^14` and `~2*10^10`, comfortably below `9.2*10^18`) and `S` in `long long`, the comparison `sum > S` is 64-bit on both sides and nothing overflows.

That settles the design: a `long long`-throughout two-pointer scan, `left` guarded by `left <= right`, `n = 0` handled by an empty loop. The full C++ program is in the answer.
