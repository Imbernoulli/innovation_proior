Everything distinctive about this task lives in the signs. I stand at the origin with `n` beacons `P[0..n-1]`, each a lattice point `(x_i, y_i)` whose coordinates may be negative or zero, and for an ordered pair `i < j` the signed sweep is the cross product `sweep(i, j) = x_i*y_j - x_j*y_i` — twice the signed area of triangle `O, P[i], P[j]`. I report the maximum sweep over all `i < j`. Two corners are what make this more than a nested loop: the maximum can itself be *negative* (every pair might turn clockwise), and when `n < 2` there is no ordered pair at all and I must print the word `NONE`. So the output is either `NONE` or a single integer that is free to be negative or zero — and a solution that quietly assumes the answer is nonnegative will pass the friendly samples and fail exactly the tests this problem was built around.

Scale fixes the type before anything else. `n <= 2000` and `|x_i|, |y_i| <= 10^6`, so a single product `x_i*y_j` reaches `10^12` and the difference of two such products reaches `2*10^12` in magnitude — an order past the 32-bit range of ~`2.1*10^9`. Every coordinate and every accumulator that touches a product must be 64-bit; with `int` the large-coordinate tests silently wrap to a wrong answer rather than crash. Coordinates go into `long long`, the running maximum stays `long long`.

For the algorithm, two routes are on the table. Sorting by polar angle and pairing the angular extremes is `O(n log n)`, but it fights the problem twice: the objective is over pairs `i < j` *in input order*, so any reordering by angle changes which pairs are even legal, and `atan2` ties on collinear and zero-coordinate beacons are precisely where float comparisons go wrong. Direct enumeration sidesteps both. With `n <= 2000` there are at most `n(n-1)/2 ~ 2*10^6` ordered pairs; evaluating `sweep(i, j)` for every `i < j` and tracking the maximum is `O(n^2)`, pure integer arithmetic, respects `i < j` by construction, and at ~2 million iterations of a couple of multiplications runs well under the 1-second limit. The angular route would be solving an asymptotic problem I do not have while importing floating-point fragility I would have to defend. Enumeration it is; the only remaining risk is transcription — the initializer, the loop bounds, the type.

The sample fixes the orientation of my primitive: pair `(0,1)` on `(2,0), (0,2), (-2,0), (0,-2)` gives `2*2 - 0*0 = 4`, matching the stated answer, so `sweep` is the cross product and not its negation. But the same instance shows why the sample is toothless against the real trap: pair `(0,3)` gives `2*(-2) - 0*0 = -4`, a negative sweep sitting right next to the positive `4`. A max initialized to `0` still reports `4` here — the bug only bites when *every* pair is negative, and this input has a positive one.

That is the pitfall to dodge, because the natural initializer for a running maximum is `best = 0`, and it is wrong here. `0` is not a neutral starting point; it injects a phantom candidate worth `0` that corresponds to no real pair, and it quietly asserts "the answer is at least 0" — a claim about the data, not about `max`. The identity element for maximum over a set is `-infinity`, and `0` is only correct when the problem floors the answer at zero (an empty selection allowed), which this one does not: you must pick a real `(i, j)`. To see it bite, take three beacons fanning clockwise around the origin, `P0 = (0,3)`, `P1 = (3,0)`, `P2 = (4,-1)`. The pairs are `(0,1) = 0*0 - 3*3 = -9`, `(0,2) = 0*(-1) - 4*3 = -12`, `(1,2) = 3*(-1) - 4*0 = -3`, so the true answer is the least-negative pair, `-3`. Run it with `best = 0`: no negative sweep ever exceeds `0`, and the code prints `0` instead of `-3`. Seeding `best = LLONG_MIN` (~`-9.2*10^18`, far below the minimum achievable sweep of `-2*10^12`) fixes it — the first real pair sets `best = -9`, then `(1,2)` raises it to `-3`, correct. The positive sample is unaffected: the first pair sets `best = 4` and nothing beats it. The sentinel itself is safe: `best` is only ever compared (`cr > best`) and assigned (`best = cr`), never added to; all arithmetic happens on real coordinates inside `cr`, bounded by `2*10^12`, so `LLONG_MIN` never participates in a computation and cannot underflow.

The `n < 2` corner is a second place to be careful, because it decides output format, not just a value. I read `n`, read `n` coordinate pairs, *then* check `n < 2` and print `NONE`. Reading the coordinates first is harmless (for `n = 0` the read loop runs zero times) and keeps the guard in one place. The subtlety is that the guard must *print* `NONE`, not merely bail: a present `n = 0` — a header of `0` followed by no points — still owes the literal `NONE`, so a bare early `return` would emit nothing and be judged wrong. That is distinct from an empty input stream with no header token at all, which I handle with `if (!(cin >> n)) return 0;` — there is genuinely nothing to answer, so returning silently is correct. Two early exits kept separate: silent on no header, `NONE` on a real `n < 2`.

The remaining cases fall out with no special handling. Collinear or zero-coordinate beacons give a legitimate `0` sweep — e.g. `(1,1),(2,2)` yields `1*2 - 2*1 = 0` — which is a real candidate distinct from `NONE`; there is no division anywhere, so duplicated points (also sweep `0`) and `(0,0)` beacons just flow through the integer products. And `cin >>` skips arbitrary whitespace, so whether the input is one pair per line or all on one line does not matter.

Header guard, point read, `NONE` on `n < 2`, then the `LLONG_MIN`-seeded `O(n^2)` double loop over `i < j` in `long long` — the full submission is in the answer.
