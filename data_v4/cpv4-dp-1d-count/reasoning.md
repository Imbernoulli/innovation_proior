Two numbers set the whole shape of the arithmetic before I pick an algorithm: `K` and `p` each run up to `10^9`, while `n` runs to `2*10^5`. The count is reported mod `p`, so it stays below `10^9`, but the *intermediate products* are the trap — a residue below `p` times a color factor of `K` or `K-1`, both below `10^9`, and `10^9 * 10^9 = 10^18`. That fits a signed 64-bit `long long` (cap `~9.2*10^18`) but overflows 32-bit by nine orders of magnitude, so every accumulator and factor is `long long` and I reduce mod `p` after each multiply. Two contract corners I pin now: `n = 0` is legal and answers `1` (the empty strip is one design, no bricks placed), and `p = 1` is legal, where everything is `0 mod 1`.

There are two ways to count. Listing every length-`1`/`2` composition and coloring a layout of `t` bricks as `K * (K-1)^{t-1}` is obviously correct, but the number of layouts is `Fib(n+1)` — exponential, fit only as an independent oracle for tiny `n`. The real solution has to be linear: carry a running count `g[i]` = valid colored designs of a length-`i` strip and grow the strip one brick at a time, `O(n)`, which ships at `n = 2*10^5`. The entire difficulty is attaching the color factor to the right predecessor, which is exactly where this flavor of counting DP goes wrong.

The color-blind skeleton is Fibonacci: a length-`i` strip ends in a length-`1` brick on a length-`i-1` strip, or a length-`2` brick on a length-`i-2` strip. Color the new final brick: it must differ from the brick immediately to its left — *unless there is no brick to its left*, i.e. unless the predecessor strip is empty. So the multiplier is `K` when the predecessor has length `0` (this is the very first brick on the wall) and `K-1` otherwise. What makes a single scalar `g[i]` suffice is that this factor depends only on *whether* the predecessor is empty, never on its colors: all `K` colors are symmetric, so whatever the previous last brick's color, exactly `K-1` colors differ from it. That symmetry is why I never track the last color.

So with `g[0] = 1`:
- a length-`1` brick onto strip `i-1`: factor `K` if `i-1 == 0` else `K-1`, times `g[i-1]`;
- a length-`2` brick onto strip `i-2` (only if `i >= 2`): factor `K` if `i-2 == 0` else `K-1`, times `g[i-2]`.

`g[i]` is the sum; the answer is `g[n] mod p`. On the stated sample `n = 3, K = 3`: `g[0]=1`; `g[1] = 3*g[0] = 3` (predecessor empty, factor `K`); `g[2] = 2*g[1] + 3*g[0] = 6 + 3 = 9` (the length-`1` branch touches a non-empty strip, factor `2`; the length-`2` branch sits on the empty strip, factor `3`); `g[3] = 2*g[2] + 2*g[1] = 18 + 6 = 24` (both predecessors non-empty). This matches `24` term-by-term against the direct coloring `3*2*2 + 3*2 + 3*2 = 12+6+6`.

The place this DP dies is the first-brick factor, and it fails in two distinct ways. The obvious one is an off-by-one: slap a uniform `K-1` on every brick — treating them all as touching a predecessor — and `n = 1` gives `g[1] = (K-1)*g[0] = 2` instead of `3`. The leading brick has no left neighbour, so it keeps all `K` colors; a uniform `K-1` strips one color from every design's first brick. The second is sneakier, and it is why I cannot just special-case `i == 1`: the length-`2` branch has its *own* first-brick case. At `i = 2` the length-`2` brick sits on the empty strip and is itself a first brick deserving factor `K`. If I only fixed the `i == 1` single brick, then `n = 2, K = 3` would compute `g[2] = (K-1)*g[1] + (K-1)*g[0] = 2*3 + 2*1 = 8` instead of `9` — the lone length-`2` brick gets `2` colorings when it deserves `3`. So the correct rule is not "the first brick is at `i == 1`" but "the factor is `K` iff *this branch's own* predecessor has length `0`", tested independently on each branch: `c1 = (i-1 == 0) ? K : K-1` for the length-`1` brick, `c2 = (i-2 == 0) ? K : K-1` for the length-`2` brick.

Off-by-one is one counting sin; double-counting is the other, and I need no design produced by two `(i, branch)` decompositions. It cannot be: every design has a *unique* last brick — the one covering position `n` — of a single length, so each design is generated by exactly one branch of exactly one `g[n]`. "Design = (everything before the last brick) + (last brick)" is a bijection. A mixed-layout case confirms it concretely at a different `K`: `n = 3, K = 2` enumerates as `1+1+1 -> 2`, `1+2 -> 2`, `2+1 -> 2`, total `6`; the DP gives `g[1]=2`, `g[2] = 1*2 + 2*1 = 4`, `g[3] = 1*4 + 1*2 = 6`.

The corners fall out of the same recurrence with no special cases. `n = 0`: the loop never runs, output `g[0] = 1 % p`. `n = 1`: `g[1] = K`, output `K mod p`. `K = 1`: `Km1 = 0`, so `g[1] = 1` (one color, one single brick) and `g[2] = 0*g[1] + 1*g[0] = 1` (the single length-`2` design), but `g[3] = 0` and it stays `0` for every longer strip — which matches reality, since any length-`>= 3` layout has at least two bricks and with one color two touching bricks must clash. `p = 1`: `g[0] = 1 % 1 = 0` and every product is `0 mod 1`, so all answers are `0`. Overflow stays bounded because each product is `(< p) * (< p) <= 10^18`, reduced immediately, with `Km` and `Km1` reduced once up front so I never multiply two unreduced values. Memory is one `vector<long long>` of length `n+1`, about `1.6 MB`, well inside `256 MB`; I keep the full array for clarity rather than collapsing to two rolling values.

For a counting problem I check the linear DP against the enumerate-and-color oracle — a genuinely different method — over random small inputs (`n` into the low teens, small `K`, a spread of prime and composite moduli); they agree everywhere. So I ship the `O(n)` counting DP with the predecessor-empty color factor, one self-contained file that reads `n K p` and prints `g[n] mod p`.
