The scale decides everything here, so I read the constraint first: `1 <= |s| <= 2000`. That is small enough that a quadratic method is comfortable — `n^2 = 4*10^6` — and it is exactly the size that signals an interval DP rather than anything cleverer. The answer is bounded by `n - 1 <= 1999` (worst case you mirror every character but one), so an `int` holds every intermediate value with room to spare. Only insertions are allowed, never deletes or replaces; a length-0 or length-1 string is already a palindrome at cost `0`; input is one token on stdin and the output is one integer.

The cheap-looking move is a two-pointer matcher: pointers at both ends, step inward on a match, and on a mismatch `s[i] != s[j]` insert once to mirror one end onto the other and advance that pointer. It is `O(n)` and a handful of lines, and it feels like it can never waste an insertion. But it has to *guess* which end to keep whenever the ends differ, and that guess is committed forever. The minimum insertions equal `n - LPS(s)`, the length of the string minus its longest palindromic subsequence: the characters you keep must, read in order, already form a palindrome, and every other character costs exactly one insertion to mirror. At a mismatch the greedy is choosing which of `s[i]`, `s[j]` to keep in that surviving palindrome — precisely the decision `dp[i][j] = 1 + min(dp[i+1][j], dp[i][j-1])`, a `min` over two interior costs that depend on the entire span `s[i+1..j-1]` the greedy has not looked at. Whenever those two costs differ, a fixed left-or-right rule is wrong on some input.

A concrete witness makes it stick: `s = "abacab"` has answer `1` (keep the subsequence `"bacab"`, mirror the leading `a`). The "always mirror the left end" rule happens to find that `1`; the mirror-the-right-end rule instead peels into `"abaca"`, hits a fresh mismatch its earlier choice created, and spends `3`. Same string, opposite fixed rules, a factor-of-three gap — and no local rule can know in advance which side the interior will reward. The small strings I might reach for first, `"abcda"` or `"abab"`, all sit in the tie regime where the two interior costs are equal and every rule looks correct; that coincidence is exactly what makes the greedy dangerous to trust. With an `O(n^2)` DP free at this scale, I take the DP.

So `dp[i][j]` is the minimum insertions to palindromize `s[i..j]`, read off the two ends. If `s[i] == s[j]` the ends are a free outer layer and `dp[i][j] = dp[i+1][j-1]`; when `j == i+1` the interior is the empty range, so a matched pair costs `0`. If `s[i] != s[j]` I pay one insertion to mirror one end and recurse into the cheaper side: `dp[i][j] = 1 + min(dp[i+1][j], dp[i][j-1])`. Base `dp[i][i] = 0`, answer `dp[0][n-1]`. Both branches want exercising, so I run the tie-regime strings from before: `"abcda"`'s outer `a`s match and drop it to the middle `"bcd"`, worth `1 + min(dp["cd"], dp["bc"]) = 2` (`"abcdcba"`); `"abab"` has mismatched ends and reduces through matched inner pairs to `1 + min(dp["bab"], dp["aba"]) = 1 + min(0,0) = 1`. The greedy agrees on both — as it must where the interior costs tie — so what certifies the recurrence rather than merely echoing the greedy is the `abacab` witness above, the one input where that `min` splits `1` from `3`.

The transition for row `i` reads only row `i+1` (`prev[j]`, `prev[j-1]`) and the current row to its left (`cur[j-1]`), so two rolling `int` arrays of length `n` suffice — `O(n)` memory. Running `i` from `n-1` down to `0`, `prev` holds row `i+1` and `cur` the row being filled, swapping after each `i`. The one place this bites is the matched branch reading `prev[j-1]` for `dp[i+1][j-1]`. When `j == i+1` that index is `prev[i] = dp[i+1][i]`, the empty range, which must be `0` — but in the rolling scheme `prev[i]` is never written during row `i+1` (the inner loop there starts at `j = i+2`), so it carries a stale value from two rows earlier. On `"aa"` it happens to read `0` and the answer comes out right, but only by accident of what was sitting in that cell. I make the empty range explicit rather than trust the array:

```
cur[j] = (j - 1 >= i + 1) ? prev[j - 1] : 0; // length-2 match -> 0
```

Now a length-2 match is `0` by construction and every longer match reads a genuine `dp[i+1][j-1]` cell that row `i+1` actually wrote — correct because of the index guard, not because of luck in the buffer.

The corners map to distinct paths: `n = 1` skips the inner loop and prints `0`; empty stdin is caught by the `if (!(cin >> s))` guard, which prints `0` and returns; a length-2 mismatch like `"za"` gives `1 + min(0,0) = 1`. At `n = 2000` the `4*10^6` `int` operations run in a few milliseconds against the 2 s limit, on two rows of a few kilobytes.

An indexing-heavy DP earns an independent check, so I differential-tested against an oracle that computes the answer a different way — `n - LCS(s, reverse(s))`, itself pinned on short strings by an exhaustive insertion-BFS. Across the edge set, every string over `{a,b,c}` up to length 8, and several hundred random cases (two-, three-, four-letter and full-alphabet strings, near-palindromes, repeated runs), the solution matched the oracle everywhere, with the empty-input and `n = 2000` corners confirmed separately. The stale-`prev[i]` bug never reached this sweep — the hand-trace caught it first, so what the sweep certifies is already the fixed version. The full module is in the answer.
