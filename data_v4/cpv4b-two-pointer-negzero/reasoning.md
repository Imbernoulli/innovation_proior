What makes this pair-counting problem more than a textbook exercise is the sign zoo and the degenerate band. Anomalies run negative, zero, and positive, so a pair sum can be anything in `[-2*10^9, 2*10^9]`; the band `[lo, hi]` can be a single value or, when `lo > hi`, empty. And the scale bites twice on integer width. With `n` up to `2*10^5`, the pair count reaches `n(n-1)/2 ≈ 2*10^10`, already past the 32-bit cap of `~2.1*10^9`, so the answer itself must be 64-bit. Each `|a[i]|` up to `10^9` pushes a pair sum to `2*10^9`, and `lo, hi` reach `2*10^9` with `lo - 1` dropping to `-2*10^9 - 1`. Every anomaly, every accumulator, and every band endpoint has to be `long long`; an `int` anywhere on the value path is a silent wrong answer on the large tests. That is settled before I pick an algorithm.

Brute force checks all `O(n^2)` pairs — `~2*10^10` at the cap, hopeless under one second, but a fine oracle. Two fast routes survive. Sort and binary-search each element: for index `i`, partners lie in `[lo - a[i], hi - a[i]]`, a slice sized by two `lower_bound`/`upper_bound` calls — but then I owe a self-pair subtraction (whenever `lo <= 2*a[i] <= hi`) and a halving to undo the unordered double count, two sign-sensitive corrections. Or sort and sweep with two pointers: define `countLE(K)` as the number of unordered pairs with sum `<= K`, and read off pairs in the band as `countLE(hi) - countLE(lo - 1)`. The sweep only ever pairs `l < r`, so it never sees a self-pair and never double counts — no corrections at all. I take the two-pointer route for exactly that reason, but the reduction hides a precondition I want to keep in view: subtracting `countLE(lo - 1)` from `countLE(hi)` only counts the band when `{s <= lo-1} ⊆ {s <= hi}`, i.e. when `lo <= hi`.

Now `countLE(K)` on `a` sorted ascending. Run `l = 0`, `r = n-1`. If `a[l] + a[r] <= K`, then since the array is sorted, `a[l] + a[m] <= a[l] + a[r] <= K` for every `m` in `(l, r]` — so every pair whose smaller element is `l` qualifies. Those pairs are `(l, l+1), (l, l+2), ..., (l, r)`: that is `r - l` of them, the boundary pair `(l, r)` **included**. This is the one spot the count wants to be off by one — writing `r - l - 1` silently drops `(l, r)` itself (on `a = [1, 2]`, `K = 5`, the true count `1` collapses to `0`). So I add `r - l` and advance `l`, having fully credited it. Otherwise `a[l] + a[r] > K`: the largest free element `a[r]` is too big even for the smallest free `a[l]`, so it pairs with nobody and I drop it, `r--`. The loop stops at `l >= r`, and each unordered pair is counted exactly once — at the moment its smaller index becomes `l`, which only advances after crediting itself.

On the sample sorted to `[-5, -3, -1, 0, 2, 7]` with band `[-3, 4]`, the sweep gives `countLE(4) = 12` and `countLE(-4) = 4`, and `12 - 4 = 8` — the answer the contract states.

The precondition is where this problem is designed to catch me. If I form the answer as `countLE(hi) - countLE(lo - 1)` unconditionally and hand it an empty band, the identity's superset assumption fails and the subtraction can go negative — which for a *count* is structurally impossible, an instant tell. Concretely, an all-negative log with an empty band: `n = 3`, `lo = -7`, `hi = -9`, `a = [-6, -5, -3]`. The true answer is plainly `0` (no sum satisfies `-7 <= s <= -9`, that interval is empty). But the unguarded formula computes `countLE(-9) - countLE(-8) = 2 - 3 = -1`. Here `lo - 1 = -8 > hi = -9`, so the subtracted set is strictly *larger* than the minuend and the difference dips below zero. The fix is to enforce the precondition I named at the outset: if `lo > hi`, the band is empty and the answer is `0`; only otherwise do I subtract. Re-run the breaking case: `lo > hi`, so `ans = 0`, correct. And a *non-empty* all-negative band to check I did not over-correct — `lo = -9`, `hi = -8` on the same array gives `countLE(-8) - countLE(-10) = 3 - 1 = 2`, matching the two pairs `(-6,-3) = -9` and `(-5,-3) = -8` by hand.

The remaining corners fall out of the sweep plus that one guard. The degenerate one-value band `lo == hi` is really "pairs summing to exactly `lo`": `countLE(lo) - countLE(lo - 1)` is pairs with sum `<= lo` minus pairs with sum `< lo`, i.e. sum `== lo` — on `[-1, 0, 1, 2]` with band `[0, 0]` it returns `1`, the single pair `(-1, 1)`. Heavy duplicates still form distinct day-pairs: `[0, 0, 0, 0]` with band `[0, 0]` gives `countLE(0) - countLE(-1) = 6 - 0 = 6 = C(4,2)`. The empty log (`n = 0`) and single day (`n = 1`) never enter the sweep loop (`l < r` is false at once), so both return `0`. The overflow reservation from the outset now pays off: values, endpoints, `cnt`, and `ans` all being `long long`, the `~2*10^10` count and the `2*10^9` pair sum stay inside 64-bit.

Sort, the `long long` `countLE` sweep with increment `r - l`, and the `lo > hi` guard ahead of the subtraction — the full module is in the answer.
