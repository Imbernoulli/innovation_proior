The distinctive thing here is the alphabet. This is "longest repeated substring," but the symbols are integers across `[-10^9, 10^9]` — negatives and zeros included — and with `n <= 2*10^5` the natural solution is a rolling hash, which is exactly the tool a negative or zero symbol quietly corrupts. So before any algorithm I fix the arithmetic. The answer is a length `<= n`, comfortably an `int`; but the fingerprints I compute over windows want a 61-bit modulus, so their products need `__int128` underneath. The symbol map is where the real care goes.

Structure first. "Some block of length `L` repeats" is monotone in `L`: if equal length-`L` windows sit at distinct starts `i != j`, dropping their last element leaves equal length-`(L-1)` windows at the same distinct starts, so `hasDup(L) => hasDup(L-1)`. The answer is therefore the largest `L` with `hasDup(L)` true, reachable by binary search over `[1, n]` in `O(log n)` yes/no queries. Each query asks "do two of the `n-L+1` length-`L` windows coincide?", which a polynomial rolling hash answers in one `O(n)` scan: prefix hashes give any window's fingerprint as `pre[r] - pre[l]*B^{L}` mod `p`, and a hash table catches a collision. Overall `O(n log n)`. (A suffix-array/LCP solution is exact and hash-free, but coordinate-compressing a signed alphabet is its own bug surface and a lot more code; I keep it only as a fallback if hashing won't come out clean.)

Now the trap this problem is built around. A textbook polynomial hash assumes each symbol is a positive digit below the base. If I feed the value straight in — `sym(v) = (u64)v` — a negative `v` casts to `2^64 + v`, and modulo the Mersenne prime `p = 2^61 - 1` that fold is destructive: since `2^61 ≡ 1`, we have `2^64 = 8*2^61 ≡ 8 (mod p)`. So any negative `v` and positive `u` with `u - v ≡ 8` collapse to one residue — take `v = -3`, `u = 5`, then `u - v = 8` and `(u64)(-3) mod p = 5 = (u64)5 mod p`. A window containing `-3` would be declared equal to one containing `5`, inflating `hasDup` and returning a length that is too large. The negativity is not cosmetic; it folds the value space onto itself.

The cure is the standard offset: push every symbol into a strictly positive, injective range before hashing. With `|a[i]| <= 10^9`, map `v -> v + 10^9 + 1`, landing every symbol in `[1, 2*10^9+1]`. That width `~2*10^9` sits far below `p ≈ 2.3*10^18`, so distinct values keep distinct residues; and the minimum is `1`, never `0`, so a symbol of value `0` becomes an ordinary digit (`10^9+1`) rather than vanishing from a prefix hash. Now `-3` maps to `999999998` and `5` to `1000000006` — distinct, collision gone.

Two guards make the result exact rather than probabilistic. I run two independent 61-bit bases combined into one key, so a spurious collision needs both hashes to clash (`~1/p^2` per pair); and on any key match I confirm with a direct `windowEqual` element comparison before returning true. A real 122-bit collision then cannot produce a wrong answer — it costs one extra `O(L)` compare, and only on a genuine match, where I return immediately.

One base case remains, and the constraints invite the off-by-one. The predicate is only meaningful for `L >= 1`. If binary search starts at `lo = 0`, `mid` can reach `hasDup(0)`: every empty window has fingerprint `pre[i] - pre[i]*B^0 = 0`, they all collide, and `windowEqual(.,.,0)` is vacuously true, so `hasDup(0)` returns a meaningless `true`. I start the search at `lo = 1`, and guard the predicate (`L <= 0 => true`, never queried; `L > n => false`) so it is total and correct at its edges.

The corners then fall out, and they drive the one special-case in the code. `n = 0` and the no-token input print `0` and return before any array or base setup — no zero-length vectors, no `rng() % (MOD-300)` on an empty problem. `n = 1` gives a single window at every length, `hasDup(1) = false`, answer `0`. An all-negative array with no repeat: every offset symbol distinct, answer `0`; with a repeated value, answer `1`. All zeros map to the digit `10^9+1`, and the longest repeated block overlaps by one, length `n-1` — for `[0,0,0,0,0]` the search settles at `4` (`hasDup(4)` true via windows at starts `0` and `1`, `hasDup(5)` false with only one window). Large magnitudes near `+-10^9` stay inside `[1, 2*10^9+1]` and never overflow, since `mulmod` routes products through `__int128`.

On the stress inputs — `n = 2*10^5` random, all-zero, and `{-1,0,1}` — the `O(log n)` linear scans finish well under the 2-second limit at ~30 MB. Against a brute-force `O(n^3)` oracle over random small cases (empty, singletons, all-negative, zero-heavy, tiny-alphabet, large-magnitude) it agrees everywhere. The full program is in the answer.
