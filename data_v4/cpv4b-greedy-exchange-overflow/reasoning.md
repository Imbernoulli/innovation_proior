The two numbers that decide this problem are the ranges, not the algorithm. There are `n <= 2*10^5` acts, each fee `p[i] <= 10^9`, each deadline `d[i] <= 10^9`, and nothing forbids all `n` acts being staged when the deadlines are large and distinct. So the collected total can reach `2*10^5 * 10^9 = 2*10^14` — roughly ninety thousand times past the 32-bit signed ceiling of `~2.147*10^9`. The running sum has to be 64-bit. And a deadline of `10^9` means I cannot index anything by a raw deadline value: an array of that length is about 4 GB against a 256 MB limit. Both traps are handed to me by the constraints before I have written a line, so I fix the plan around them: fees and the accumulator live in `long long`, and whatever structure indexes slots must be sized by `n`, never by a deadline.

**The task is job sequencing with deadlines.** Act `i` pays `p[i]` if I put it in some slot numbered `1..d[i]`; one act per slot, one slot per act, and I may drop any act. Maximize the total fee. The candidate I want is the greedy: sort acts by fee descending, and place each act in the **latest still-free slot that is `<= d[i]`**, skipping it if no such slot is free. Grabbing the biggest fees first is the obvious half; placing each act as *late* as legally possible is the subtle half — it keeps the small-numbered slots open for tight-deadline acts that have nowhere else to go. The alternative would be weighted bipartite matching or a slot DP, always correct but far heavier, and its slot side is enormous once deadlines run to `10^9`. I only want the heavy machinery if I cannot prove the greedy, so I try to prove it.

**Exchange argument.** Let `O` be an optimal schedule that agrees with the greedy `G` on as many of the highest-fee acts as possible, and suppose they differ; take the highest-fee act `i` on which they disagree. Every act with fee `>= p[i]` is placed identically by both, by maximality. If `G` stages `i` and `O` does not, then the slots `O` has occupied `<= d[i]` are exactly the ones `G` had occupied when it seated `i`, and `G` only seats `i` when a slot `<= d[i]` is free — so `O` has a free slot there too and can add `i`, or must drop a `>= p[i]` act to make room, which cannot raise the total. If both stage `i` but in different slots, move `i` in `O` to greedy's slot: greedy picked the *latest* legal slot, the least constraining one for everyone else, so the reslot is always feasible. Either way I get an optimal schedule agreeing with `G` on more acts — contradiction. So the greedy total is optimal, and the "latest legal slot" rule is precisely what makes the reslot step go through.

I do not want to ship on prose alone, so I also check the greedy against an independent oracle — a bitmask brute force that, for each act, either skips it or drops it into any free slot `<= d[i]`, memoized on `(index, busy-slot mask)`, with no greedy baked in. Across random small instances with ties, zero fees, and colliding deadlines it never disagrees with the greedy.

**Finding "latest free slot `<= d`" quickly.** Scanning down from `d` to `1` for a free slot is `O(d)` per act, hopeless when `d` is `10^9`. The right structure is a disjoint-set-union where `parent[s]` points to the largest slot index `<= s` that is still free: `find(d)` returns the latest free slot `<= d`, or the sentinel `0` when none is free, and seating an act in slot `s` sets `parent[s] = s - 1` so later queries skip past it. With path compression this is near-`O(alpha)` amortized.

The sample exercises the one non-obvious move. Acts `(100,2),(60,1),(70,2),(40,1),(90,3)` sort by fee to `100(d2), 90(d3), 70(d2), 60(d1), 40(d1)`; starting `parent = [0,1,2,3]`, the `100` takes `find(2)=2` and the `90` takes `find(3)=3`, but the `70` then calls `find(2)`, which chases `parent[2]=1` down to the still-free slot `1` — the DSU stepping past the occupied slot to seat the act below its deadline. Both deadline-1 acts, `60` and `40`, hit `find(1)=0` and are turned away, and the total is `260`. The collision on slot 1 is precisely the case the structure exists to resolve.

**Sizing the DSU without the 4 GB array.** If I let `parent` span `1..max(d)`, one act with `d = 10^9` allocates it at ~4 GB — the memory trap I flagged at the start, and one the sample never exposes because there `max(d) = 3`. The escape is that slot *numbers* above `n` are useless: at most `n` acts are ever seated, so no more than `n` distinct slots are used, and any act with `d >= n` can take any of slots `1..n` just as well. Formally, if a schedule seats `k <= n` acts using some slots, all relevant deadlines are `>= n >= k`, so I can relabel the used slots to `1..k` without violating any deadline — the achievable total is identical. So I cap each deadline to `min(d[i], n)` up front and size `parent` as `n + 1`, giving `O(n)` memory (~800 KB at `n = 2*10^5`). Running the brute force on raw versus capped deadlines agrees on every random instance, so the cap is loss-free.

**The overflow, made concrete.** The exchange argument and the DSU get the *count* of seated acts and their identities right; the one thing left to get wrong is the summation type, and it is invisible on every small test because those totals sit in the hundreds. To surface it I have to build the maximal case: `n = 200000` acts all `(10^9, 10^9)`. After the cap every deadline is `200000`, the greedy seats all `200000` acts into slots `200000..1`, and the true total is `200000 * 10^9 = 2*10^14`. An `int` accumulator on this input prints `552894464`, which is exactly `2*10^14 mod 2^32` — a silent wrap, not a scheduling error, since the seated count is correct. A `long long` accumulator prints `200000000000000` and runs in about 0.02 s using a few MB. This is the headline pitfall of the problem: the `int` bug is a wrong answer on precisely the maximal hidden test and nothing smaller reveals it.

**Edge cases.** `n = 0`: the read loop and greedy loop never run, total `0` (and empty stdin fails `cin >> n`, so I return with no output). `n = 1`, `(5,1)`: `find(1)=1`, total `5`. Two acts sharing `d = 1`, `(9,1)` and `(5,1)`: the `9` takes slot 1, the `5` gets the sentinel and is skipped — only one of two colliding deadline-1 acts fits. A `p[i] = 0` act sorts last and can only ever take a slot no paying act wanted, so it never changes the total. The sentinel slot `0` is guarded by `if (s > 0)`, so no phantom fee is added and `parent[0]` is never written.
