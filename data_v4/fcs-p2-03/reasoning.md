I have to scan every contiguous non-empty subarray of `a[0..n-1]` and report the largest product. The non-empty clause is load-bearing: a single element is always a legal window, so on an all-zero or strictly-shrinking-magnitude input the answer is just the largest single value, and I must never seed the search with a neutral product of `1` — the empty window is disallowed and a phantom `1` would beat a genuine all-negative answer.

The constraints settle the arithmetic before I pick an algorithm. With `n <= 18` and `|a[i]| <= 9`, the extreme product is eighteen nines: `9^18 = 150094635296999121`, about `1.5 * 10^17`. A signed 64-bit integer tops out near `9.2 * 10^18`, so every reachable product, positive or negative, fits in `long long` with more than an order of magnitude of headroom. A 32-bit `int` overflows the moment a window's product passes `~2.1 * 10^9`, which happens fast at these magnitudes and is a silent wrong answer on exactly the maximal `n=18` tests. So 64-bit accumulators throughout, no big integers, no floating point.

The additive maximum-subarray problem has the one-line recurrence `cur = max(a[i], cur + a[i])`, and the tempting move is to swap `+` for `*`: `cur = max(a[i], cur * a[i])`, tracking a single running best. The trouble is specific to multiplication. With addition a larger running value is always at least as good to extend; multiplying by a negative reverses order, so the running product you most want to carry through a negative factor is the *smallest* (most negative) one, and keeping only the maximum throws it away. I run it on `[-1,-2,-3,-4,-5]`, whose best window is the even-length `(-2)(-3)(-4)(-5) = 120`. Running the recurrence by hand: `cur=-1`; at `-2`, `cur=max(-2,2)=2`; at `-3`, `cur=max(-3,-6)=-3`; at `-4`, `cur=max(-4,12)=12`; at `-5`, `cur=max(-5,-60)=-5`; best `12`. It returns `12` against a reachable `120`. A second array pins the mechanism: `[-2,3,-4]`, whole-array product `24`; the scan keeps `cur=3` at index 1 and never sees that the *minimum* product ending there, `-6`, becomes `(-6)(-4)=24`. It returns `3`. Both failures are the same defect — the discarded most-negative running product is exactly what the next negative factor turns into the new maximum. Kadane-on-product is out.

So carry both extremes. For windows ending at index `i`, track the maximum product `curMax` and the minimum product `curMin`. A window ending at `i` is either the singleton `a[i]` or an extension of a window ending at `i-1`, so at element `x` the candidates are `c1 = x`, `c2 = curMax * x`, `c3 = curMin * x`, with `curMax = max(c1,c2,c3)`, `curMin = min(c1,c2,c3)`, and the global answer the running max of `curMax`. The `c3` term is the whole point: it is the negative-times-negative flip Kadane discarded. Seed from the first element, `curMax = curMin = best = a[0]` — legal because `n >= 1` guarantees `a[0]` exists, and seeding from the real element rather than a neutral `1` is what makes a lone negative return itself.

On the array that broke Kadane, `[-2,3,-4]`, the DP now takes `c3 = (-6)(-4) = 24`: the minimum carried through index 1, dead weight at that step, is exactly what the trailing negative converts into the answer — which is why `curMin` has to ride along even when it is losing locally.

One implementation pitfall is real here, and the two negatives in the recurrence invite it. Both updates must read the *previous* `(curMax, curMin)` pair. If I write them in place — assign `curMax`, then reference `curMax` in the `curMin` line — the just-computed maximum feeds back into the minimum and fabricates products belonging to no real window. On `[-3,-4,-5]` (true answer `20`, the window `[-4,-5]`) that mistake corrupts `curMin` to `-48` at index 1, then reads `(-48)(-5) = 240` at index 2 and reports `240`. Snapshotting the three products into `c1,c2,c3` before assigning either variable dodges it, which is why the loop computes the candidates first and only then overwrites.

Edge cases the recurrence has to survive: `n = 1` with `a = [-5]` runs the loop zero times and returns the seed `-5`, since the singleton is the only window; a `0` anywhere zeros `c2` and `c3` while `c1 = x` lets the window restart, so the scan resets cleanly past it; an all-negative even-length array like `[-2,-3,-4,-5]` recovers its positive whole-array product through the carried `curMin`. Overflow is already closed off by the `long long` bound. Output is one integer and a newline, and `cin >>` eats arbitrary whitespace, so the input's mix of spaces and line breaks does not matter.

For confidence in the code and not just the idea, I differential-test against an independent `O(n^2)` brute oracle in Python — exact big integers, double loop over every `(i,j)`, sharing no logic with the `O(n)` DP — over a generator spanning seven distributions (mixed signs, sign-only small values, all-positive, all-negative, zero-sprinkled, tiny `n` for the `n=1,2` corners, and maximal `n=18` at `±9`). Zero mismatches over 13 hand-built edge cases and 600 random cases, and zero again on an exhaustive sweep of every array with `n <= 4` and values in `[-3,3]` (2800 arrays), including the two arrays that broke Kadane. The shipped solution is the `O(n)`, `O(1)`-space min/max DP; the full module is in the answer.
