The distinctive thing about this problem is not the shortest path — it is the inequality at the boundary. Each station `v` carries a curfew `c[v]`, and the courier may be present at `v` only while `t < c[v]`; the statement forbids presence at any `t >= c[v]`, so an arrival landing *exactly* at `t = c[v]` is illegal, not allowed. The weight and curfew ranges are plainly tuned so exact landings are common, which means the whole answer hinges on coding `<` and never `<=`. Two consequences I pin down before touching an algorithm. The start counts as being present: the courier stands at station `1` at time `0`, which is legal only if `0 < c[1]`, so `c[1] = 0` forces `-1` immediately. And with `n, m` up to `2*10^5` and weights up to `10^9`, a path chaining `n - 1` edges accumulates arrival times up to `~2*10^14` — an order of magnitude past the 32-bit ceiling. Distances, the running sum `d + w`, and the curfews all have to be 64-bit `long long`; an `int` here is a silent wrong answer on the large tests, not a crash, which is the worst failure mode.

So the objective is a positive-weight shortest path with a per-node validity window. Does Dijkstra survive the window? The label I want per station is its earliest legal arrival, and that label is monotone in exactly the way Dijkstra exploits: arriving earlier at `u` can only produce smaller successor times, and a smaller arrival satisfies every downstream curfew that a larger one would. Waiting never helps, so the time dimension collapses to a single label per node instead of a time-expanded `(station, time)` graph. Weights are at least `1`, so the settle-once argument holds and the first pop of a node is its true earliest legal arrival. The curfew then acts purely as a filter on relaxation: refusing to create a label when `nd >= c[v]` is the same as deleting that edge for that particular arrival time, which never disturbs the heap ordering. Checking curfews only at the end would be wrong — the globally cheapest route can land on a curfew while a slightly longer legal route exists — so the constraint has to live inside the relaxation, not after it.

Concretely: keep `dist[v]` as the earliest legal arrival, pop `u` with finalized `d` (skipping any stale pop where `d != dist[u]`), and for each edge `u -> v` accept the candidate `nd = d + w` only when it is both legal (`nd < c[v]`) and improving (`nd < dist[v]`). The sample makes the boundary visible: `c = [_, 100, 9, 12, 8]`, edges `1->4 (8)`, `1->2 (3)`, `2->4 (4)`, `3->4 (1)`. The direct hop `1->4` arrives at `8`, and `8 < c[4] = 8` is false, so it is rejected — the cheapest edge is illegal by exactly one unit. The detour `1->2->4` arrives at `3 + 4 = 7 < 8` and wins. That is the case the boundary tests turn on.

Two boundary traps invite an off-by-one, and both come from writing the natural-but-wrong comparison. The first is the gate itself: the legal region is strict, so it must be `nd < c[v]`, not `nd <= c[v]`. The forbidden instant `t = c[v]` is where every distinguishing test lives — a single edge `1->2 (5)` with `c[2] = 5` has true answer `-1`, because arriving at `5` is not before the curfew, yet `<=` would wrongly report `5`. The second trap is the start: it is itself an arrival, at time `0`, and must pass the same strict test. Seeding `dist[1] = 0` unconditionally is the tempting mistake — on `n = 1, c[1] = 0` it would print `0` when the courier cannot legally stand at the origin at all, and with `c[1] = 0` plus outgoing edges it would let an illegal courier depart and relax onward. So I seed `dist[1] = 0` and push it only when `0 < c[1]`; otherwise the frontier starts empty, `dist[n]` stays infinite, and `-1` falls out naturally.

Since the whole verdict rides on one inequality, a hand-trace is not enough; I want an independent method that shares no structure with Dijkstra. Waiting never helps, so an optimal legal route is a simple path — revisiting a node only delays arrivals and can never unlock a curfew — which means I can enumerate every simple path from `1` by DFS, accumulate arrivals, drop a path the instant an arrival violates `t < c[v]` (including the start check `0 < c[1]`), and take the minimum arrival at `n`. No heap, no dominance pruning, no settle-once. On hundreds of random small graphs, weights and curfews kept small so arrivals land on curfews constantly, the two agree with zero mismatches — and a `<=` variant of the relaxation, run against the same enumeration, breaks on exactly the graphs where an optimal arrival hits a curfew. The boundary is genuinely being exercised, and the strict inequality is doing real work.

The remaining cases fall out. `n = 1` gives `0` when `c[1] > 0` and `-1` otherwise, both covered by the seeded start check. An unreachable target, `m = 0` with `n > 1`, and every route into `n` landing on or after a curfew all leave `dist[n]` at infinity and print `-1`. Parallel edges just generate extra candidates, and the minimum legal one survives. For overflow safety I set `INF = LLONG_MAX / 4` and only ever compute `d + w` for finite popped `d`, so `INF` is never summed into and the unreachable test `dist[n] >= INF` stays clean; all comparisons against `c[v] <= 10^9` are well within range.
