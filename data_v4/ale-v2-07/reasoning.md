A grid up to `40x40`, so at most `1600` cells, some blocked and the rest free; on the free cells sit `K` nets, each a pair of distinct terminals I must join with a simple 4-adjacent free path, and no two routed paths may share a single cell. I maximize the count of nets routed. One feature of the scoring dominates every design choice: if any single path is malformed — off-grid, on an obstacle, a non-adjacent step, a repeated cell, wrong endpoints — or any two routed paths touch a common cell, the whole score collapses to `0`. Feasibility is not a tie-breaker here; it is a precondition, and one stray shared cell zeroes a run that might otherwise have scored a dozen nets. So whatever I print has to be provably vertex-disjoint before I care how many nets it packs. Output is `R` followed by `R` lines `i L r0 c0 ... r(L-1) c(L-1)`.

The scale dictates what is affordable. `N = H*W <= 1600`, and `K` is scaled to roughly `free_cells / typical_path_length`, which lands around `25..40` nets. A single shortest-path search over this grid is a few thousand cell-relaxations — microseconds. With a ~1.8 s budget I can run *thousands* of path searches: route every net, rip them all up, reroute them all, many times over. That affordance is what makes an iterative negotiated-routing scheme realistic; if `N` were a million cells I would have to be far stingier, but at `N <= 1600` I can be lavish with re-routing.

The simplest thing that is always feasible: route nets one at a time, and when I route a net, permanently mark its cells as claimed so no later net can touch them; if a net's terminals are already claimed or no path avoids claimed cells, skip it. By construction the committed set is vertex-disjoint, so the output is always feasible. This is sequential routing with hard blocking, my floor. I will call the act of doing this — given an *order* of nets, route each through the not-yet-claimed cells — a "commit", because it is the operation that turns any prioritization into a guaranteed-feasible disjoint solution. Every candidate solution I ever emit comes out of a commit, so I never have to worry about an overlapping output.

The commit is order-sensitive in a way that costs real nets. Suppose nets A and B both want to pass through a narrow corridor. If I route A first and it barrels straight through, B is boxed out and scores nothing — even though A could have taken a one-cell detour and let *both* through. The greedy commit cannot see this: once A is placed it is frozen, and B simply fails. The damage is *irreversibility*. A cheap first improvement is to route short nets first — they use few cells, are easy to place, and rarely block anyone; long nets, placed late, snake through whatever room is left. Sorting by Manhattan distance ascending already beats arbitrary input order by a wide margin. But it is still one-shot: it never reconsiders a placement, so it strands nets that a small reroute of an earlier net would have saved.

That "early net blocks later net, but could have detoured" pathology is exactly the problem FPGA and VLSI routers were built to solve, and the established strong method is PathFinder's negotiated-congestion rip-up-and-reroute. The idea is to stop forcing nets to be disjoint *during* search and instead let them all route on a *shared* grid where cells have a soft cost; cells that many nets want become expensive; nets are repeatedly ripped up and rerouted on the updated cost field; over rounds they negotiate onto private detours, and only then do I extract a disjoint subset. This is the family I want — rip-up-and-reroute with A\* per net and congestion costs. The obvious local search (route disjointly, then try swapping a single net's path) is too weak because the binding interactions are *global*: freeing one net often requires several others to shift in concert, and a single-net move cannot propose that. Negotiated congestion proposes it implicitly, because raising the price of a contested corridor pushes *everyone* off it at once.

For the soft routing I keep, per cell `v`: `occ[v]`, the number of net-paths currently using `v` (present congestion), and `hist[v]`, an accumulated penalty for cells chronically contested across rounds. The cost to enter `v` is `1 + presentFactor*occ[v] + hist[v]`. The base `1` makes A\* prefer short paths when nothing is contested. The `presentFactor*occ[v]` term is the immediate negotiation pressure: if three nets sit on `v`, a fourth pays a stiff premium to join and will detour if any detour is cheaper. The `hist[v]` term is memory: cells that stay shared after a full reroute sweep get their history bumped, so even if present occupancy momentarily drops, a chronically-fought-over cell stays expensive and nets learn to avoid it permanently. This present-plus-history split is the heart of PathFinder — present cost resolves the current round, history cost prevents oscillation where two nets keep swapping onto the same cell forever. I ramp `presentFactor` up over rounds (start `0.5`, add `0.7` each round) so early rounds let nets explore cheaply and later rounds force them apart hard.

Each reroute is a min-cost path on this field, so A\* with the Manhattan-distance heuristic, admissible because every cell costs at least `1` and Manhattan distance is the minimum step count. I run each search with a generation-token trick instead of clearing `dist`/`visited` each call — a per-call `token` counter, and a cell is "seen this call" iff `visToken[v] == token`. That turns each search's setup from `O(N)` clearing into `O(1)`, which matters because I run thousands of searches. The search takes a `passable(v)` predicate so the same routine serves two masters: in the soft phase `passable` only forbids obstacles (overlap is allowed, paid for in cost), while in the commit it also forbids already-claimed cells (hard disjointness). One A\* routine, two cost regimes.

The main loop: each round, reroute every net once (rip it up so it does not pay congestion against its own old path, refresh the field, A\* it back), in a shuffled order so no net is permanently advantaged; then bump `hist` on every still-shared cell; then raise `presentFactor`. After each round I derive a commit from the soft layout — because the soft layout itself is not a legal answer (it has overlaps), it is only a *suggestion* of which corridors each net likes. To commit, I order nets by how clean their soft path is — fewest still-contested cells first, because a net whose soft route is already nearly private is the one most likely to keep it when I force disjointness — and run the hard-blocking commit in that order, lightly biasing the committer's A\* away from historically hot cells (`bias = 0.15*hist`). I keep the best commit ever seen. This is the bridge: the negotiation finds good *routes and priorities*, and the commit converts them into a guaranteed-disjoint solution — the soft layout, with its overlaps, is never emitted.

Negotiated congestion gets the routes mostly disentangled, but at the end I want to squeeze in the last few nets, and that is a job for large-neighbourhood search directly on the committed solution. Repeatedly: take the current best commit, rip up one to a few random *routed* nets, rebuild the claimed grid from the survivors, then reroute (through the freed space) the ripped nets together with all currently-unrouted nets, in random order, hard-blocking as I go. If the resulting count is at least the best, keep it. This is destroy-and-repair: removing a routed net sometimes opens a corridor that lets *two* previously-stranded nets in, a net trade the single-shot commit never finds. Accepting ties (`cnt >= bestRouted`, not `>`) lets the solution drift sideways across equal-scoring layouts, which diversifies the corridors and helps the next rip-up find a strictly-better neighbour.

Best layouts are carried across the PathFinder and LNS phases, and a single mixed-up or in-place-mutated path would ship an overlap and silently zero the run — the scorer just prints `0`. So before output I add a safety re-commit: I walk the chosen paths in index order against a fresh `claimed` grid and re-validate each — endpoints match the net's terminals, every cell free and not already claimed by an earlier kept path, every step 4-adjacent, no repeated cell — and drop any path that fails rather than risk emitting it. Worst case it drops a net; it can never emit an illegal one, so the output is provably feasible regardless of what the optimization did upstream.

The instances are over-subscribed by design: `K` is scaled to *capacity* = `free_cells / typical_path_length`, which matters because each net consumes not one cell but a whole path of length `~(H+W)/2`. On the provided seeds `K` lands around `26..28` and the routing rate is a meaningful `~35%` — enough slack that detours genuinely matter and the negotiation has room to express itself, rather than a saturated packing dominated by "how many of the very shortest nets happen to fit".

I check the solver against two baselines on seeds 1..20, because beating a trivial one is not enough. Every output is feasible (the scorer parses all 20 and zeroes none). The solver means `9.25` nets; the trivial sequential-A\* baseline (input order, hard blocking) means `4.45`; and — the one I actually worried about, since my gain might just be the length-sort — a distance-sorted greedy (the good ordering, still one-shot) means `5.55`. The full negotiated-congestion + LNS solver beats even that smarter greedy by `~67%`, so the rip-up-and-reroute and destroy-and-repair are each pulling real weight beyond a good static priority. Timing on the largest-`K` seed (`K=41`) is `1.80 s`, ~3.6 MB resident.

The boundaries drive two guards in the code. A net whose terminals are walled off — a `3x3` moat, or opposite corners split by an all-`#` middle row — returns an empty path from both the soft A\* and the commit, so it is simply left unrouted and the output stays feasible; that is why "skip a net that can't be placed" has to be a first-class case, not an error. And `K=0` or empty stdin has to short-circuit to `0` before any allocation, which is what the `if(!(cin>>...))` guard at the top is for. The full self-contained C++17 module is in the answer.
