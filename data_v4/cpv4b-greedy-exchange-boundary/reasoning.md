The whole task is: place the fewest integer checkpoints so every one of `n` closed segments `[l_i, r_i]` has a checkpoint sitting on it, where "on it" includes both endpoints. The word that decides everything is *closed*. `[1,5]` and `[5,9]` share only the coordinate `5`, and a single checkpoint there inspects both — so the correctness of the entire program collapses onto one question: does a checkpoint that lands exactly on a segment's endpoint count that segment as inspected? That is a `<` vs `<=` decision, and nothing else in this problem is hard.

Scale fixes the types before I write anything. `n <= 2*10^5`, coordinates in `[-10^9, 10^9]`. The answer is at most `n`, comfortably an `int`, and there is no summation anywhere, so overflow is not a threat once I hold coordinates in 64 bits. The one place types genuinely bite is the sentinel: I will carry `last` = the coordinate of the most recent checkpoint, and its "no checkpoint yet" value must sit strictly below every real coordinate, including `-10^9`. `long long last = LLONG_MIN` is the unambiguous choice; an `int` sentinel near `-2e9` would be fragile, so I keep every coordinate `long long`.

This is the textbook "stab all intervals with the fewest points." Sorting endpoints and reasoning about a maximum set of pairwise point-disjoint segments would also work, but that route drags the same closed-touching subtlety into its disjointness predicate and buys nothing over the greedy — I keep it only as an independent check. So: greedy by earliest right endpoint. Two things need proof rather than assumption: why anchoring the checkpoint at the right endpoint is optimal, and the exact membership test when intervals are closed and may touch.

The exchange argument. Let `s` be the segment with the smallest right endpoint `r_min`. Any solution must inspect `s`, so it has a checkpoint at some `x` with `l_s <= x <= r_min`. Slide that checkpoint rightward to `r_min`. For any other segment `t` the old `x` inspected, `l_t <= x <= r_t`, and because `r_min` is the smallest right endpoint, `r_t >= r_min >= x`; combining, `l_t <= x <= r_min <= r_t`, so the checkpoint at `r_min` still inspects `t`. The slide can only gain coverage, never lose it — so there is an optimal solution with a checkpoint at `r_min`. Remove `s` and everything that checkpoint now inspects, recurse on the rest. That recursion is exactly a sweep: sort by `r`, and for each segment in order, if it is not inspected by the checkpoint at `last`, open a new checkpoint at this segment's `r` and set `last = r`.

Now the boundary. A segment `[l, r]` is inspected by the checkpoint at `last` iff `l <= last <= r`. Because I process in increasing-`r` order, every segment seen after `last` was set has `r >= last`, so the right half `last <= r` holds automatically once any checkpoint exists. The membership test therefore reduces to its left half: inspected iff `l <= last`, needs a new checkpoint iff `l > last`, i.e. `last < l`. The touching case is precisely `l == last`: the checkpoint sits on the segment's left endpoint, so the segment *is* inspected and must be skipped. The condition to open a new checkpoint is the **strict** `last < l`. Writing `last <= l` instead would re-open a redundant checkpoint for every touching segment.

The touching pair `[1,5], [5,9]` makes the operator concrete; its true answer is `1` (a checkpoint at coordinate `5` sits on both). Sorted by `r`: `[1,5]` then `[5,9]`, starting `last = LLONG_MIN`. With the buggy `<=`: first segment `LLONG_MIN <= 1`, open at `5`, `last = 5`; second segment `5 <= 5` is true, open again at `9` — output `2`, wrong. With strict `<`: first `LLONG_MIN < 1`, open at `5`, `last = 5`; second `5 < 5` is false, skip — output `1`, correct, because the checkpoint at `5` already inspects `[5,9]` (`5 <= 5 <= 9`). The `<=` form mis-classifies exactly this equality.

The duality gives an independent read on the same convention. Minimum stabbing points equals the maximum number of pairwise point-disjoint segments, where two closed intervals share a point iff `max(a,c) <= min(b,d)`. On the four-segment sample `[1,5],[5,8],[9,12],[10,14]`, `[1,5]` and `[5,8]` share `5` and `[9,12]`,`[10,14]` share `10..12`, so the two touching pairs contribute one disjoint member each — a maximum disjoint set of size `2`, matching the `2` checkpoints. The point is that "touching counts as overlapping" on the stabbing side must line up with "touching counts as not-disjoint" on the packing side; the strict `<` is what keeps the two sides consistent.

The sort key is load-bearing in its own right, and getting it wrong is subtle because it doesn't fail loudly. Take `[1,10],[2,3],[4,5]`, answer `2` (stab at `3` covering `[1,10]` and `[2,3]`, at `5` covering `[4,5]`). If I had sorted by *left* endpoint, `[1,10]` comes first and I would place a checkpoint at its right endpoint `10`; then `[2,3]` gives `10 < 2` false and I'd skip it — but the checkpoint at `10` does not inspect `[2,3]` (`2 <= 10 <= 3` is false). Sorting by left endpoint destroys the invariant `r >= last` that let me drop the `last <= r` half of the membership test, so a checkpoint parked at a large right endpoint silently mis-reports shorter later segments as covered, and the strict `<` fix cannot save it. Storing pairs as `{r, l}` makes the default `pair` ordering sort by `r` first, which restores the invariant.

Two corners exercise machinery the main argument didn't. `n = 0`: `cin >> n` reads `0`, the loop never runs, output `0`. Negative coordinates like `[-9,-9],[-3,-1]`: only the `LLONG_MIN` sentinel guarantees the first segment opens (an `int` sentinel near `-2e9` would be the fragile alternative), disjoint pair, output `2`. The end-to-end touching chain `[1,5],[5,9],[9,13]` is the boundary case at length: open at `5`, `5 < 5` false skip, `5 < 9` open at `13`, output `2` — no integer lies in all three so `2` is optimal, and this is exactly what a `<=` bug inflates to `3`. Fully disjoint segments hit `open` every step and return `n`; nested and duplicate segments collapse under the skip test to `1`.

At `n = 2*10^5` this is one `O(n log n)` sort and an `O(n)` sweep in `O(n)` memory — about `0.03 s`, far under the 1 s limit. For correctness I ran the sweep against an independent brute that computes the maximum pairwise point-disjoint set by exhaustive subset enumeration (a genuinely different argument) over 400 random small instances with tight coordinate ranges so touching and nesting occur constantly: zero mismatches. Compiling the `<=` variant, it diverged from the brute on most of those cases and returned `2` on `[1,5],[5,9]` — direct confirmation that the boundary operator, not anything cosmetic, is the load-bearing line.

The full module is in the answer: sort by right endpoint, sweep with `last` starting at `LLONG_MIN`, and open a checkpoint exactly when `last < l`.
