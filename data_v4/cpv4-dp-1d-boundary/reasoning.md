The distinctive thing about this rod-cutting problem is that the constraint lives on *billet length* — every piece must span between `L` and `R` segments — while the natural DP transition lives on a *cut-boundary index*. The entire task is getting the translation between those two frames exactly right. With `R <= 50` the transition window is tiny, so an `O(n*R)` partition DP is trivially fast; the only real dangers are an off-by-one in that window and the arithmetic scale.

Scale first, because it fixes the types. `n <= 2*10^5`, `|v[i]| <= 10^9`, `K <= 10^9`, `1 <= L <= R <= 50`. A single billet's `|sum|` is at most `R * 10^9 = 5*10^10`, plus `K`, so one billet costs up to `~5.1*10^10`; with up to `n` billets the total is bounded by `2*10^5 * 5.1*10^10 ~ 1.0*10^16`. That fits signed 64-bit (`~9.2*10^18`) with room to spare but overflows 32-bit by seven orders of magnitude. So prefix sums, the dp array, and every per-billet cost must be `long long` — an `int` anywhere on the cost path is a silent wrong answer on the large tests.

I represent a billet by the half-open range `[j, i)` covering segments `j..i-1`, length `i - j`. Half-open keeps the imbalance clean — `sum on [j, i) = S[i] - S[j]` with `S[i] = v[0] + ... + v[i-1]` — and lets "the prefix `[0, i)` is fully tiled" be a statement about a single index `i` running `0..n`. But it also plants the trap: the DP indexes cut boundaries, the constraint is on length, and mixing those two frames is exactly where an off-by-one will hide.

Before committing to the DP I ask whether any greedy is optimal. On the sample `v = [3,3,-5,-5,3,3]`, `K=2`, `L=2`, `R=3` (`S = [0,3,6,1,-4,-1,2]`), "always cut length `R=3`" gives `[0,3)+[3,6)`, sums `1` and `1`, cost `6`; "always cut length `L=2`" gives `[0,2)+[2,4)+[4,6)`, sums `6,-5,5`, cost `27`. A factor of four between the two simplest rules already says cut *placement* dominates, not cut count — with negative `v` the cost-minimizing split keeps each billet's running sum near zero, and that balance point does not land on any fixed multiple of `L` or `R`. The sharper objection is feasibility: the moment `L = R`, or `n` is not a clean multiple of the chosen length, a fixed rule either gets lucky or leaves a remainder it cannot legally cut, and it has no notion of reporting `-1`. Only a DP that scans the whole legal window both finds the balance-minimizing split and detects infeasibility. Greedy is out.

So let `dp[i]` be the minimum cost to tile the prefix `[0, i)`, with `dp[i] = +infinity` when that prefix cannot be tiled; base case `dp[0] = 0`. For `i >= 1` the last billet is some `[j, i)` of length `i - j`, which must satisfy `L <= i - j <= R`. Solving the double inequality for the predecessor `j`: `i - j <= R` gives `j >= i - R`, and `i - j >= L` gives `j <= i - L`; with `j >= 0` a real boundary, `j` ranges over the inclusive integer window `[max(0, i - R), i - L]`. The transition is `dp[i] = min over those j of ( dp[j] + K + |S[i] - S[j]| )`. The two endpoints check out against the extreme legal lengths: the longest billet, length `R`, starts at the *low* end `j = i - R` (clamped to `0`); the shortest, length `L`, starts at the *high* end `j = i - L`. When the window is empty (`i < L`) `dp[i]` stays infinite. The answer is `dp[n]`, or `-1` if it is still infinite. Running this on the sample reaches `dp[6] = 6` through the cut `[0,3)+[3,6)`, matching the given answer.

Transcribing that window is where the off-by-one lives. The half-open habit is to write the high end as `jhi = i - L + 1`, on the reflex that windows of "length up to something" end at `+1` — but here `j` is a cut boundary, not a half-open bound, and that stray `+1` admits `j = i - L + 1`, a last billet of length `i - j = L - 1`, one segment shorter than the minimum. The smallest input that exposes it: `n = 1`, `K = 37`, `L = 2`, `R = 2`, `v = [9]`, true answer `-1` (a single segment cannot host a length-2 billet). With `jhi = i - L + 1`, `i = 1` gives `jlo = max(0,-1) = 0` and `jhi = 0`, so window `[0,0]` is non-empty and `j = 0` prints `37 + |9| = 46`. The shortest legal billet starts at `j = i - L`, so the correct inclusive high end is `i - L`: then `jhi = -1`, window `[0,-1]` is empty, `dp[1]` stays infinite, output `-1`. And the narrowing does not over-shoot into forbidding a *legal* billet — `n = 2`, `L = R = 2`, `K = 0`, `v = [3,4]` still gives window `[0,0]` and `dp[2] = |S[2]-S[0]| = 7`, the one legal length-2 billet intact. The `+1` slip is not a corner curiosity; it changes the answer on any input where some prefix length equals a multiple-of-`L`-minus-one.

The other boundary is the unreachable predecessor. The transition reads `dp[j]` for `j` in the window, but some of those `j` are themselves untileable (`dp[j] = INF`), and reading them would compute `INF + |S[i] - S[j]|` — a finite-looking huge number that replaces the correct `INF` and makes an impossible prefix look reachable. `n = 3`, `L = R = 2`, `v = [1,1,1]` is untileable (`2+2 > 3`, and one length-2 billet leaves a length-1 remainder) and must print `-1`: `dp[2] = 2` is reachable but `dp[1] = INF`, and at `i = 3` the only predecessor in window `[1,1]` is `j = 1`. Without a guard `dp[3]` would become `INF + 2`, a giant cost printed in place of `-1`; with `if (dp[j] >= INF) continue;`, `j = 1` is skipped and `dp[3]` stays infinite. I test `>= INF` rather than `== INF` so any state accidentally built slightly above the sentinel still reads as unreachable, and I size `INF = 4e18` so that `INF` plus one billet cost (`~5.1e10`) stays under `LLONG_MAX` — the sentinel cannot overflow and wrap negative, which would defeat the comparison.

The corners fall out of the recurrence without special-casing. `n = 0`: neither loop runs, `dp[0] = 0`, output `0`. `n < L`: every window is empty, `dp[n]` stays infinite, output `-1`. `L = R`: the window collapses to a single `j = i - L`, so only multiples of `L` are reachable and the DP still works. Single segment with `L = 1`: window `[0,0]`, cost `K + |v[0]|`. At maximum scale — `n = 2*10^5`, `R = 50`, `|v| = 10^9`, `K = 10^9` — the DP does at most `n*R = 10^7` updates, runs in about `0.02 s`, and its largest reachable value sits near `10^13`, an order of magnitude below the proven `~1.0*10^16` bound and far under `LLONG_MAX`. Output is a single integer; `cin >>` skips arbitrary whitespace, so the empty or absent second line at `n = 0` is harmless.

For validation I compare against an independent brute force that enumerates every composition of the length into parts in `[L, R]` and minimizes `sum(K + |segment sum|)`. It shares no code path with the prefix DP (recursion over compositions vs. a linear scan), and the two agree on several hundred random small cases — including many infeasible `-1` instances — with zero mismatches. In particular, the `jhi = i - L + 1` version disagrees on a large fraction of these while `i - L` matches every one, confirming the window fix rather than papering over it.

What I ship is one self-contained file: read `n K L R` and the `n` values, build prefix sums `S`, run the `O(n*R)` DP over the inclusive window `[max(0, i-R), i-L]` with the `>= INF` guard and `INF = 4e18`, and print `dp[n]` or `-1`.
