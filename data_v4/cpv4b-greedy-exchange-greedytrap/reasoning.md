A single bench runs `n` jobs back-to-back — no gaps, no preemption. Job `i` takes `t[i]` minutes and its customer pays `w[i]` for every minute until *their own* job finishes, so in a given order job `i` completes at `C[i]` = the running sum of durations through it and pays `w[i] * C[i]`. I want the order minimizing `sum_i w[i] * C[i]` — the single-machine total weighted completion time problem, `1 || sum w_j C_j`. Input is `n` then `n` pairs `(t[i], w[i])` on stdin; output is the one minimum.

What decides the data types here is the sheer magnitude of that objective. With `n <= 10^5` and `t[i], w[i] <= 10^4`, the worst case is every job `t = w = 10^4`: completion times run `10^4, 2*10^4, ...`, so the sum is `10^4 * 10^4 * (1 + 2 + ... + 10^5) ~ 5*10^17`. A single completion time already reaches `~10^9`, brushing signed 32-bit (`~2.1*10^9`); the objective overflows it by eight orders of magnitude. So every accumulator is `long long`, and I store `t`, `w` as `long long` too so any product formed during comparison stays 64-bit.

With `n!` orders I need a single sort key that is *provably* optimal, not one that merely reads well. Three are on the table: shortest-processing-time-first (`t` ascending), most-impatient-first (`w` descending), and some ratio rule on `t/w`. The two single-attribute keys are exactly how wrong solutions get shipped — each throws away one of the two attributes the weighted objective depends on. A concrete three-job instance will show whether either survives, and it lets me work the ratio key out instead of recalling it.

Take three jobs `A=(t=3,w=4)`, `B=(t=1,w=1)`, `C=(t=2,w=3)`. Shortest-first orders `B,C,A`: completion times `1,3,6`, cost `1*1 + 3*3 + 4*6 = 34`. Impatient-first orders `A,C,B`: completion times `3,5,6`, cost `4*3 + 3*5 + 1*6 = 33`. Brute-forcing all six orders, the optimum is `32`, at `C,A,B` (`C@2, A@5, B@6`: `3*2 + 4*5 + 1*6`). Both cheap greedies are strictly beaten, and the reason is visible: shortest-first spends the first slot on `B`, whose weight is tiny, while impatient-first puts the long heavy `A` first and pushes everyone behind its three minutes. The winning order balances duration against weight.

To pin down that balance, take any order with two *adjacent* jobs `i` then `j`, and let `P` be the total duration of everything strictly before the pair. Running `i` then `j`, the pair contributes `w_i(P + t_i) + w_j(P + t_i + t_j)`; running `j` then `i`, it contributes `w_j(P + t_j) + w_i(P + t_j + t_i)`. Everything outside the pair is untouched — jobs before see the same `P`, jobs after see the same total `P + t_i + t_j` either way — so the whole difference is

`(run i,j) - (run j,i) = w_j t_i - w_i t_j`.

Putting `i` before `j` is no worse exactly when `w_j t_i - w_i t_j <= 0`, i.e. `t_i / w_i <= t_j / w_j`. So sorting by `t/w` ascending removes every cost-reducing adjacent inversion; by repeatedly fixing inversions I turn any order into the sorted one without ever raising the cost, hence the sorted order is optimal. This is Smith's rule. It also explains the two greedies: when weights all tie the key collapses to `t` (shortest-first), when durations all tie it collapses to `1/w` (impatient-first) — each is the special case of Smith's rule that leaks the other attribute, which is why each looked plausible.

The transcription has one genuine trap. The key is the *ratio* `t/w`, not the *product* `t*w` — and those are different functions: sorting by `t*w` on the counterexample gives `B(1), C(6), A(12)`, i.e. `B,C,A`, reproducing shortest-first's suboptimal `34` while looking like a reasonable "smallest jobs first" key. So I compare with the ratio's cross-multiplied form, `t[i]/w[i] < t[j]/w[j]` <=> `t[i]*w[j] < t[j]*w[i]`, which stays in exact integers — the crossed indices matter (`w[j]` pairs with `t[i]`), and the products `t*w <= 10^8` are far inside 64-bit, no division, no rounding. The crossed indices are the part that is easy to get backwards — `w[j]` pairs with `t[i]`, not `w[i]` — so I run the comparator on the counterexample: `A` vs `C` gives `3*3=9 < 2*4=8`, false, so `C` precedes `A`; the other two comparisons put `A` before `B` and `C` before `B`, giving `C,A,B` — the brute-force optimum, and precisely the reverse of what the `t*w` key produced.

`std::sort` needs a strict weak ordering, so I check the equal-ratio case: when `t[i]*w[j] == t[j]*w[i]` the comparator returns `false` both ways, making equal-ratio jobs incomparable, which is exactly right — and the exchange difference is `0` there, so their relative order costs the same regardless. I break exact-ratio ties by original index for deterministic output; it changes no cost.

Edge cases the hidden tests will probe:
- `n = 0`: the read loop, sort, and accumulation all run zero times, `total = 0` — an empty bench costs nothing. (`if (!(cin >> n)) return 0;` also covers entirely empty input.)
- `n = 1`, `t=[7], w=[5]`: sort is a no-op, `total = 5*7 = 35` — the lone job finishes at its own duration.
- All equal durations, or all equal weights: the rule degenerates to impatient-first / shortest-first respectively, each optimal in its degenerate regime (consistent with the exchange difference reducing to a single-attribute comparison). All equal ratios: every order costs the same.
- Large `n = 10^5`, `t = w = 10^4`: output `500005000000000000`, matching the closed form `10^4 * 10^4 * (10^5*(10^5+1)/2)`, well within the 1 s / 256 MB budget. Each intermediate `w*cur <= 10^4 * 10^9 = 10^13` and the final `~5*10^17` both sit safely in 64-bit.

So the solution is the `O(n log n)` sort under Smith's rule, cross-multiplied to stay in integers, with 64-bit accumulation throughout; the full program is in the answer.
