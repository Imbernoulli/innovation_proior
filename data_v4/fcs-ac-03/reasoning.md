What makes this problem awkward is that one growing set has to answer three questions that look unrelated: the maximum XOR over any subset, whether a given `x` is obtainable as a subset-XOR, and the `k`-th smallest *distinct* obtainable value. Up to `3*10^5` operations arrive online — each answered before the next is read — so I cannot batch, and values run to `2^60`. That 60-bit ceiling settles the arithmetic before anything else: every number and every accumulator is `unsigned long long`, and every shift is `1ULL << b`. A plain `1 << b` is undefined for `b >= 31`, which here would be a silent wrong answer, not a crash.

The definitional method is to keep the inserted numbers and, per query, materialize the set of all subset-XOR values: start from `{0}` and, for each inserted `x`, replace `S` with `S ∪ {v ^ x : v in S}`. Then max is `max(S)`, membership is `x in S`, k-th is `sorted(S)[k-1]`. This is trivially correct — it is literally the definition — and it is exactly the brute force I differential-test against. But `|S|` is up to `2^m` for `m` inserts; even `m = 40` gives `~10^12` entries. It dies past roughly `m = 20`, nowhere near `3*10^5`.

For the max query alone there is a tempting shortcut: sort the numbers descending and XOR each in if it raises the running value. I distrust it. Ordering by *value* is not ordering by *leading bit*, and once two numbers share a top bit, value order can commit to the wrong representative of that bit. More decisively, even a version that happened to get the max right would say nothing about membership and nothing about the k-th-smallest order statistic — two of my three query families have no greedy-by-value story at all. A sort is not the structure I need.

The reframing that serves all three at once: XOR is addition in `GF(2)`, and a 60-bit integer is a vector in `GF(2)^60`. "XOR of a subset of the inserted numbers" is exactly a `GF(2)`-linear combination of them, so the obtainable values are precisely the **linear span** of the inserted numbers. That turns the whole task into linear algebra. The span has a basis of size `r` (the rank), every span element is a unique subset-sum of basis vectors, so there are exactly `2^r` distinct obtainable values — which both answers "how many distinct values" and bounds the valid range for the order query. I maintain the basis incrementally by online Gaussian elimination over `GF(2)`: an array `basis[b]` where, when non-empty, `basis[b]` is a span vector whose highest set bit is exactly `b`. At most one vector per bit, so at most 60 vectors total no matter how many numbers arrive.

Three of the four routines fall out immediately. *Insert*: walk bits high to low; at the first set bit `b` with `basis[b]` empty, store `x` there as a new pivot and stop; otherwise `x ^= basis[b]` and continue. If `x` reduces to `0` it was dependent and nothing changes — `O(60)`. *Max XOR*: greedy on *bits*, not values — `r = 0`, then for `b` high to low do `r ^= basis[b]` whenever that raises `r`; each pivot owns a distinct top bit, so this turns on the highest achievable bits, floored at `0`. *Membership*: reduce `x` against the basis without storing; `x` is in the span iff it reduces to `0`, and `0` always does, so `0` is always `YES`.

The k-th-smallest query is the one that forces something stronger. If — and only if — the basis is in **reduced** row-echelon form, where each pivot bit appears in exactly one basis vector, then the pivots ordered from lowest pivot bit to highest behave as place values: XORing in the `j`-th pivot exactly when bit `j` of `k` is set enumerates the span in strictly increasing order as `k` runs `0..2^r-1`. In a *non-reduced* basis a low pivot's bit can still be set inside a higher vector, which breaks that monotonicity. So the plan is a `reduced` flag: max and membership never need it, so I reduce lazily — only when an order query arrives — and clear the flag whenever a new pivot is inserted. A run of order queries with no intervening pivot then pays for reduction once. Dependent inserts leave rank and basis untouched, so they leave the flag alone.

Reduction is the step that is easy to get subtly wrong, and getting it wrong is invisible until an order query. The mistake is to clear pivot bit `b` out of only the *adjacent* higher vector rather than every higher vector. Concretely, insert `7` (`111`), `2` (`010`), `1` (`001`): that lands `basis[2]=111`, `basis[1]=010`, `basis[0]=001`. Selecting the top pivot alone should yield the smallest value with bit 2 on, i.e. `100 = 4` — but with `basis[2]` still `111` the routine returns `7`. That top-pivot-alone case is 0-indexed position `4` in the span, so the query `4 5` comes out `7` instead of `4`. The invariant I actually need is "pivot bit `b` set in exactly one vector," which demands the full double loop: for `b` ascending, for *every* `c > b`, clear bit `b` from `basis[c]`. Processing `b=0` then `b=1` turns `basis[2]` from `111` into `100`, and now the top pivot alone gives `4`.

For type-4 I compute the distinct count as `1ULL << rank`. Since values are below `2^60` the rank is at most 60, so this is at most `2^60` and never overflows `unsigned long long`; `k` is 1-indexed so it is valid iff `1 <= k <= 2^rank`, otherwise `-1`. The corners then all reduce to the empty and near-empty basis: with no inserts, max is `0`, only `0` is `YES`, and type-4 has `2^0 = 1` value so `k=1 -> 0` and `k>=2 -> -1`; a dependent or duplicate insert leaves the distinct count at `2^rank`; a value using bit 59 is fine because the loops run `b` from 59 down with `1ULL` shifts; and `if (!(cin >> q)) return 0;` makes empty input and `q=0` both silent.

I differential-tested the compiled program against the subset-enumeration brute force across many random small instances — small insert counts so `2^m` enumeration is cheap, narrow bit widths so memberships hit both `YES` and `NO` and values collide, all four op types, and `k` straddling the `2^rank` boundary — plus the documented sample and the edge cases above, with zero mismatches. Each operation is `O(60)`, the basis saturates at 60 vectors (further inserts are dependent), and reductions are `O(60^2)` bounded by the number of pivot-changing boundaries, so even an adversarial insert/type-4 alternation at `q = 3*10^5` runs comfortably inside the 1-second limit.
