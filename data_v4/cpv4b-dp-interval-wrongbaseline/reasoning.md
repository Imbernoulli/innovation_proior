A splice may only fuse two reels that are currently **neighbours** on the circular carousel — first and last reel included — and that single restriction is the whole problem; it is what separates this from the textbook optimal-merge problem. Splicing lengths `x` and `y` costs `x + y` and leaves one reel of that length; I splice until one reel remains and want the minimum total cost, with `0` when `n <= 1`. Scale fixes the data type before anything else: `n <= 1000` and `w[i] <= 10^6`, so any reel's length is at most `S = sum(w) <= 10^9`, and with exactly `n - 1` splices each costing at most `S` the total stays below `(n-1) * S < 10^12` — past 32-bit range (`~2.1*10^9`), so every accumulator is `long long`; an `int` here is a silent wrong answer on the large tests, not a crash.

Two routes are on the table. The min-cost-merge greedy — dump all lengths in a min-heap, repeatedly pop the two smallest, push their sum and charge it — is `O(n log n)` and famously *exactly* optimal for the unconstrained merge-piles problem. But that optimality theorem assumes I may merge any two piles, and here I may only merge neighbours; the freedom Huffman leans on may not be freedom I actually have. The alternative is interval DP: a run of already-fused reels is a contiguous arc, and I compute the cheapest way to fuse each arc into one reel.

Before trusting the greedy I attack it with a carousel where the two smallest reels sit apart. Take `w = [1, 2, 1, 3]` on the 4-cycle: the two `1`s are at slots `0` and `2`, which are opposite, not adjacent. Huffman runs heap `{1,1,2,3}` -> pop the two `1`s (`+2`) -> `{2,2,3}` -> pop `2,2` (`+4`) -> `{3,4}` -> pop `3,4` (`+7`), total `13`. But its very first move fused the two non-adjacent `1`s, which the machine cannot do, so `13` is unreachable. The true minimum by hand: fuse slot `0`(1) with slot `1`(2) -> `3` (`+3`), carousel now `[3, 1, 3]`; fuse `3` with the neighbouring `1` -> `4` (`+4`), carousel `[4, 3]`; fuse `4` with `3` -> `7` (`+7`); total `14`, every splice legal, and no other adjacent order beats it. So the constrained optimum `14` is strictly worse than Huffman's infeasible `13`. That is the trap in this variant: Huffman *under*-reports — it returns a number smaller than any legal answer, so it never looks obviously broken, it just quietly cheats by merging non-neighbours. Greedy is out; I commit to interval DP.

Read the whole splice sequence as a binary tree: leaves are the original reels in carousel order, internal nodes are splices, and since every splice joins adjacent reels, each subtree fuses a contiguous arc — so the subproblems are arcs. For a line of reels `i..j`, let `dp[i][j]` be the min cost to fuse it into one. Its last splice joins `i..k` with `k+1..j` and costs the whole arc's length `W(i,j) = w[i] + ... + w[j]` — the same regardless of how the two sides were built — giving

`dp[i][j] = min over k in [i, j-1] of ( dp[i][k] + dp[k+1][j] ) + W(i, j)`, with `dp[i][i] = 0`.

On `[4, 2, 6]` this reads `k=0`: `0 + (2+6) + 12 = 20`; `k=1`: `(4+2) + 0 + 12 = 18`, so `dp = 18`, matching direct simulation (merge `4+2` then `+6`).

The circle needs one more idea. The final splice of the whole process straddles exactly one carousel gap, so once I fix where the ring is opened into a line the rest is the line DP — but different openings give different costs, so I must try all `n`. To kill the modular arithmetic I double the array: `a[t] = w[t mod n]` for `t in [0, 2n)`, so any arc of `n` reels starting at slot `i` is the plain segment `a[i .. i+n-1]`. The answer is `min over i in [0, n) of dp_line(i, i+n-1)`.

I index the table by left endpoint `l` (`0..2n-1`) and length `len` (`1..n`) rather than by both endpoints — I never need arcs longer than `n`, and this keeps the table at `2n * (n+1)` longs, about `16` MB at `n = 1000`, well inside `256` MB. Arc sums come `O(1)` from a prefix-sum array over the doubled `a`. Time is `O(n^3)`: for each of `2n` left endpoints and each length up to `n`, I scan up to `n` split points. `n^3 = 10^9` looks alarming, but each inner step is one add and one compare, and the `l + len <= 2n` guard skips roughly half the pairs; it comes in near `0.9` s at `n = 1000`.

Transcribing the split into table indices is the part that bites, and my first pass gets it wrong. The arc `[l, r]` split after position `k` has left sub-arc `[l, k]` of length `k - l + 1` and right sub-arc `[k+1, r]` of length `r - k`. My first cut wrote `dp[l][k - l]` for the left — length `k - l`, off by one — so for `k = l` it indexes length `0`, a cell that does not exist in a `len in 1..n` table. A trace of `[4, 2]` (answer obviously `6`) catches it: `dp[0][2]`, `k=0`, wants `dp[0][?] + dp[1][1] + 6`, and `dp[0][0]` is the phantom length-0 slot. Fixing the left index to `k - l + 1`: `dp[0][1] + dp[1][1] + 6 = 0 + 0 + 6 = 6`, correct. I name `leftLen` and `rightLen` explicitly in the code so the arithmetic cannot drift back.

The closing `min over i` is not cosmetic: fixing a single opening (`dp[0][n]`) overshoots on an asymmetric carousel, since the cheapest opening can start anywhere. And it would not surface on `[1, 2, 1, 3]`, where all four openings tie at `14` by symmetry — which is exactly why the min has to be argued from the structure rather than trusted to that one example.

For the sentinel: `INF = LLONG_MAX/4 ~ 2.3*10^18` sits far above the `~10^12` cost ceiling, and I only ever add two finite `dp` values plus a sum. An unreachable arc stays `INF` only through `INF + finite`, never `INF + INF`, so nothing overflows and no real cost can collide with `INF`.

Corners: `n = 0` and `n = 1` print `0`, special-cased before the doubled array is built, so the DP never runs on a degenerate ring; `n = 2` is the single neighbour splice `w[0] + w[1]`, counted once (the 2-cycle's two adjacencies are the same splice); uniform reels merge balanced. A brute force enumerating every legal adjacent-merge order (wrap included) agrees on many hundreds of random asymmetric carousels, where a fixed single opening would have diverged. `cin >>` skips arbitrary whitespace, so input layout is irrelevant, and I print one integer.

I ship the `O(n^3)` interval DP over the doubled array — the full module is in the answer — not the greedy the adjacency constraint broke.
