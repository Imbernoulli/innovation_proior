The score sums `w[i] * (overlapping occurrences of pattern i in T)` over a dictionary of up to `10^5` patterns against one text, with `sum|p_i| + |T| <= 10^6`. Three features of the numbers decide the shape of the solution before I pick an algorithm. First, occurrences are counted at *every* start position including overlaps, so `aa` occurs three times in `aaaa` — the counting has to see nested and overlapping matches, not disjoint ones. Second, weights reach `10^9` in magnitude and a string may be listed many times: `10^5` copies of the pattern `a` at weight `10^9`, scored against `~10^6` copies of `a`, gives a total on the order of `10^20`, which blows past signed 64-bit. So the final accumulator, and the per-terminal weight sum (duplicates pile onto one node), must be `__int128_t` printed by hand; a `long long` total is a silent wrong answer on exactly the duplicate-heavy test the constraints invite. Third, with `m` up to `10^5`, any per-pattern rescan of the text is the thing to beat. I read `m`, then `m` lines of `pattern weight`, then the text (which may be an absent token, in which case the score is `0`).

The objective splits cleanly per pattern, so the naive plan is to match each dictionary entry against `T` independently — KMP, or even `string::find` in a loop — and add `w[i]*count`. That is `O(m*|T| + sum|p_i|)`. The `sum|p_i|` term is bounded by `10^6` and harmless; the killer is `m*|T|`. Concretely: with `m = 10^5` short patterns (total length say `10^5`, leaving `|T|` up to `9*10^5`), `m*|T|` is about `10^5 * 9*10^5 = 9*10^{10}` character comparisons — minutes, not one second. The fault is rescanning the whole text once per pattern, redoing the same character reads `m` times. The fix has to read the text a single time while accounting for all `m` patterns at once.

So I want a machine that, as it consumes `T` left to right, is always in a state that knows which dictionary patterns end at the current position. A plain trie of the patterns gives *prefix* matching but not *substring* matching: a trie walk that mismatches has to restart at the root, throwing away the text suffix already consumed, which might itself begin another pattern. The fix is a **failure function**: from a trie node `v` spelling string `s`, `fail[v]` points to the node spelling the longest proper suffix of `s` that is also a trie node. That is the Aho-Corasick automaton — when the next character has no real trie edge I follow fail links until an edge exists, never re-reading a text character. Building the trie is `O(sum|p_i|)`, building fail links is a BFS, and the scan is `O(|T|)`, for `O(sum|p_i| + |T|)` overall — the target.

The genuinely non-obvious part is recovering *per-pattern* overlapping counts from that single pass. Driving the text through the automaton, at each position I land in exactly one state: the node spelling the longest dictionary prefix that is a suffix of the text read so far. Incrementing a counter only at that landing node undercounts — when a position ends `aaa` it also ends `aa` and `a`, but those are the *fail-ancestors* of the `aaa` node, not visited. The naive repair, walking the whole fail chain up to the root at every text position and bumping each node, is `O(depth)` per character; on `aaaa...a` against `a, aa, aaa, ...` the chain grows linearly and the scan degrades back to quadratic. That is the trap this problem is built to catch.

The way out: I do not need per-position counts, only each pattern's *total* occurrence count at the end. Occurrences of the string at node `u` correspond exactly to text positions whose landing state's fail chain passes through `u` — that is, to landing states `v` for which `u` is an ancestor of `v` in the fail tree (the fail links form a tree rooted at the root). So let `cnt[v]` be the number of text positions whose landing state is exactly `v` — one cheap increment per character, no chain walk during the scan. Then the occurrence count of `u`'s string is the sum of `cnt[v]` over the whole **subtree of `u` in the fail tree**. Pushing each `cnt[v]` up to its fail-parent once, children before parents, turns `cnt[u]` into `u`'s occurrence count, and the answer is `sum_u cnt[u] * wsum[u]` over terminal nodes. One global linear pass replaces the per-character chain walk — that is the move that keeps the whole thing linear.

Two implementation choices keep that honest. During the fail-link BFS I overwrite each missing trie edge `nxt[v][c]` with `nxt[fail[v]][c]`, and set the root's missing edges to loop back to the root. After this `nxt[v][c]` is total and lands directly on the correct next state, so the scan is one flat array lookup per character with no fail-chain walk at scan time — a genuine `O(|T|)`, not an amortized argument. And I need no explicit fail tree for the subtree sum: the BFS visits nodes in nondecreasing fail-tree depth (a node's fail link always points to a shallower node, dequeued earlier), so recording the BFS order and iterating it **in reverse** processes every node before its fail-parent. One reverse loop over `bfsOrder`, no recursion, `O(nodes)`.

Duplicate patterns and prefix/substring relationships need no special case. I keep `wsum[u]` = the sum of weights of all patterns whose terminal node is `u`; two identical patterns share a terminal node and their weights add, and a pattern that is a prefix of another simply owns its own node, counted independently by the subtree sum. Negative and zero weights just flow through the final multiply-and-add.

The one subtlety I want to pin is the order inside that reverse loop:

```
for (int i = bfsOrder.size()-1; i >= 0; i--) {
    int v = bfsOrder[i];
    answer   += cnt[v] * wsum[v];   // score v with its (now-complete) subtree count
    cnt[fail[v]] += cnt[v];         // push v's count up to its fail-parent
}
```

The push always targets `fail[v]` — a *different* node, and one that appears earlier in `bfsOrder` (shallower fail depth), hence *later* in this reverse loop. So within an iteration the score reads `cnt[v]` and the push writes `cnt[fail[v]]`; they touch different cells and the two lines commute. What actually has to hold is that every push *into* `v`, coming from `v`'s fail-children (which appear later in `bfsOrder`, thus earlier in the reverse pass), has already happened before I read `cnt[v]`. Reverse-BFS order guarantees exactly that; the load-bearing fact is the order of the loop, not the order of the two statements.

I confirm the mechanism on the smallest input that mixes a pattern with its own fail-ancestor: dictionary `{a: 1, aa: 10}`, text `aaa`. By hand, `a` occurs 3 times (`3*1 = 3`), `aa` occurs 2 times (`2*10 = 20`), total `23`. Trie: root `0`, node `1 = "a"` (`wsum=1`), node `2 = "aa"` (`wsum=10`); `fail[1]=0`, `fail[2]=1`; `bfsOrder = [1, 2]`. Driving `aaa`: `0 -a-> 1` (`cnt[1]=1`); `1 -a-> 2` (`cnt[2]=1`); node `2` has no real `a`-child, so its completed edge is `nxt[fail[2]][a] = nxt[1][a] = 2`, giving `2 -a-> 2` (`cnt[2]=2`). After the scan `cnt = [0,1,2]`. Reverse loop: `v=2` scores `2*10=20` and pushes `cnt[1] += 2 -> 3`; `v=1` scores `cnt[1]*1 = 3*1 = 3` and pushes to root. Total `23`. The `aa`-count of 2 and the `a`-count of 3 both come out right precisely because the fail-child's count was accumulated into `cnt[1]` before node `1` was scored.

The empty-text and `m=0` corners fall out of the same structure but are worth pinning because they are where I/O handling usually breaks: `cin >> m` reads `0`, the pattern loop runs zero times, `cin >> text` finds no token and leaves `text` empty, the scan runs zero times, `bfsOrder` is empty, and the answer is `0`. The same holds for `m > 0` with an empty text: automaton built, scan runs zero times, all `cnt = 0`, answer `0`. And crucially the scan increments `cnt[state]` exactly once per character at the landing state only — if I also walked the chain during the scan, every occurrence would be counted twice once the subtree sum runs, and the scan would go quadratic. All fail-chain contribution is introduced by the single reverse pass, nowhere else.

The remaining hidden-test categories map onto the structure directly: a pattern longer than the text never reaches its terminal node, so its `cnt` stays `0`; a pattern equal to the whole text is reached once; duplicates with cancelling weights add at the shared terminal node (`{aa:3, aa:-1}` on `aaaa` gives `wsum=2`, `2*3=6`); deeply nested single-letter patterns come out right via the subtree sum (`{a:1,aa:2,aaa:3}` on `aaaaa` gives counts `5,4,3`, score `5+8+9=22`); a negative total is allowed (`{b:-5}` on `bbb` is `-15`); and the overflow case (`10^5` copies of `a` at `10^9` on `900000` `a`s, score `9*10^{19}`) is why `wsum` and `answer` are `__int128_t` and printed manually.

I also want the memory and time to be inside budget, since the `array<int,26>` transition row is the dominant cost. At `~10^6` nodes that row is ~`104` bytes/node, ~`100` MB, inside the `256` MB limit. A node-maximizing instance (`~9*10^5` total pattern length of near-distinct strings, so almost every character is its own node) builds ~`9*10^5` nodes and runs in a couple tenths of a second at ~`120` MB; a balanced instance with `sum|p_i| ~ 5*10^5` and `|T| = 5*10^5` runs in ~`0.13` s at ~`62` MB. Both clear `1` s / `256` MB.

For correctness I differential-test against an independent oracle that, per pattern, counts occurrences with a naive find-loop and sums `weight*count`. I run the automaton against it on the eleven hand-built adversarial cases (empty text, `m=0`, longer-than-text and whole-text patterns, cancelling duplicates, nested single-letter patterns, zero and negative weights, plus the worked example) and the `10^5`-copy overflow case, then on 450 random cases over alphabets of size 1 to 4 (to force dense overlaps), 150 over the full alphabet with up to 12 patterns, and 30 deeply-nested single-letter cases `a, aa, ..., a^n` on `a^{n+5}`. Zero mismatches — including exactly the cases that would expose the fail-chain undercount and the empty-text read.

That is what I ship: one self-contained C++17 file implementing the Aho-Corasick automaton with the goto-completed transition function, per-terminal `__int128_t` weight sums, a single reverse-BFS fail-tree subtree sum, and manual 128-bit decimal printing. The full program is in the answer.
