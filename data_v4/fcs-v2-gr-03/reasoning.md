I have an undirected graph — up to `2*10^5` vertices and edges, possibly disconnected, with parallel edges and self-loops in scope — and up to `2*10^5` queries, each asking how many vertices `w != u, v` lie on *every* `u`-`v` path, i.e. whose deletion disconnects the pair. Print `-1` when `u, v` are in different components, `0` when `u == v`. One query's answer can reach `n - 2` (a long path makes every interior vertex essential) but that fits an `int` and never sums across queries — the pressure is entirely time, with everything at `2*10^5`.

The definition is operational, so a brute force writes itself: for `(u, v)`, delete each candidate `w` in turn and BFS from `u` to test whether `v` is still reachable. That is literally the definition and makes a perfect oracle, but one reachability test is `O(n + m)`, there are up to `n` candidates, and up to `q` queries, so the total is `O(q n (n + m))` — around `10^16` at the limits, and even one worst-case query is `~10^{10}`. Hopeless as the real solution. I need to do the heavy work once and answer each query cheaply.

The brute force recomputes connectivity for every `(w, query)` pair, but "vertices whose removal can disconnect something" is a fixed global property: those are exactly the **articulation points**. A non-cut vertex can never disconnect any pair, so the essential vertices for `(u, v)` are always a subset of the global cut vertices — findable once in `O(n + m)` by a single DFS with low-links. But I still need which cut vertices sit *between* `u` and `v`, and "between" smells like a per-query traversal unless I can turn it into a tree question.

The restructuring: articulation points carve the graph into **biconnected components (BCCs)**, maximal subgraphs with no internal cut vertex. Inside one BCC any two vertices have two vertex-disjoint paths, so no interior vertex is essential for a pair both inside it; the only ever-essential vertices are the cut vertices where BCCs join. Contracting this gives the **block-cut tree (BCT)**: one *block node* per BCC, one *cut node* per articulation vertex, each cut node joined to every block containing it. A non-cut vertex lives in exactly one BCC and maps to that block node; a cut vertex gets its own cut node. The result is a forest — one tree per input component — and the cut nodes on the BCT path between `rep(u)` and `rep(v)` are exactly the articulation points separating `u` from `v`: any cut vertex off that path has a route around it.

On the sample this holds: two triangles `{1,2,3}`, `{3,4,5}` glued at `3`; BCCs `{1,2,3}` and `{3,4,5}`, sole cut vertex `3`. The BCT is `B1 — c3 — B2`; query `1 4` walks `B1 - c3 - B2`, one cut node, so the answer is `1`.

Counting nodes-with-a-property on a tree path is a standard LCA computation. Root each tree, let `cutPrefix[x]` be the number of cut nodes on the root-to-`x` path inclusive; then the count on path `a..b` is

```
cutPrefix[a] + cutPrefix[b] - cutPrefix[lca(a,b)] - cutPrefix[parent(lca(a,b))]
```

— subtract the LCA's prefix and its parent's so the LCA node itself is counted exactly once. Binary-lifting LCA gives `O(log n)` per query after `O(bctN log bctN)` preprocessing, with `bctN` (BCT node count) `= O(n + m)`.

Two corrections the query needs. First, it forbids `w = u` and `w = v`, but if `u` is itself a cut vertex then `rep(u)` is its own cut node sitting at the end of the BCT path and gets counted — so I subtract 1 for each of `u`, `v` that is a cut vertex. This is not cosmetic: on the path `1-2-3-4-5`, query `1 3` should answer `1` (only interior vertex `2`), yet the BCT path from `rep(1)` to `rep(3)` includes `3`'s cut node, which the subtraction removes. Second, if `u, v` are in different input components there is no path and I print `-1` (compare a component id assigned during the DFS); `u == v` prints `0`. Both are cheap guards before any LCA work.

Now the decomposition, and the place the constraints bite. The natural BCC algorithm is recursive Tarjan with an edge stack, but the evaluation explicitly includes a depth-`2*10^5` path, which overflows a recursive DFS's call stack. So I write the DFS **iteratively** with explicit frames `{u, parentEdge, ei}` — `ei` the current adjacency position, `parentEdge` the half-edge I arrived on. I store the graph as half-edges (`e ^ 1` is the twin) and skip the parent by *half-edge id*, not by vertex: with parallel edges, "the parent vertex" and "the parent edge" differ, and skipping the vertex would wrongly discard a genuine second edge back to the parent.

The BCC enumeration uses an edge stack: every tree and back edge is pushed as traversed. When, returning from child `u` to parent `p`, I find `low[u] >= disc[p]`, a component closes — `p` is a cut vertex (root special-cased by child count) — and I pop edges down to the tree edge `p->u`; the endpoints of the popped edges are the block's vertices. An edge stack rather than a vertex stack because a cut vertex belongs to several BCCs at once, and the edge formulation places it into each block naturally, as an endpoint of edges in each.

Here the parallel-edge case actually bites. My first version identified the boundary tree edge by its *endpoints*:

```cpp
if ((a == p && b == u) || (a == u && b == p)) break;   // stop at tree edge p-u
```

It compiled, the sample passed, small hand cases looked fine — then the differential oracle on random small graphs (parallel edges and self-loops in the mix) failed: a graph with parallel edges between `1` and `2` where a query that should answer `2` came back `1`. The cause is exactly the parallel-edge case I meant to handle: the endpoints `{p, u}` do not uniquely name the tree edge, so a parallel or back edge between `p` and `u` gets popped *before* the true boundary edge, closing the BCC early — splitting one component in two and mislabeling `p` or `u` as a cut vertex, which propagates into a wrong BCT and undercounts. The fix is to stop on the exact tree half-edge by id: it is already in hand as `parentEdge` of `u`'s frame, the half-edge pushed when I first descended into `u`, so I capture `childParentEdge` and break when I pop *that* id. Now parallel edges between `p` and `u` are popped *into* the block, where they belong, and only the genuine tree edge ends the pop; the case that returned `1` returns `2`, and the random batch passes.

I then push the differential oracle harder: a couple thousand random small cases (sparse forests, dense blobs, disconnected graphs, parallel edges, self-loops), then explicit corners — `n = 1` (singleton block, `1 1 -> 0`), two isolated vertices (`1 2 -> -1`), the path `1-2-3-4-5` (`1 5 -> 3`, `1 3 -> 1` exercising endpoint-exclusion, `4 5 -> 0`), a 4-cycle (one BCC, every query `0`, the blob case a bridge-based method would get wrong), parallel edges `1=2` with pendant `2-3` (`1 3 -> 1`, `1 2 -> 0` — the case the first version failed), and a star (center separates any two leaves). Zero mismatches.

Full-scale check: `n = m = q = 2*10^5` with the depth-`2*10^5` path (the recursion adversary) plus random chords and random queries — the iterative DFS runs without overflow, about `0.19 s` and `~68 MB`, inside the 2 s / 256 MB budget. A pure deep-path input with hand-computed answers (`1..n -> n-2`, far-apart -> exact interior count) matches, confirming the binary-lifting LCA handles maximal depth.

The complexity is what the limits demand: `O(n + m)` decomposition, `O((n+m) log(n+m))` LCA preprocessing, `O(log(n+m))` per query — against the brute force's `O(q n (n+m))`. The block-cut tree is the structural move that collapses a per-query graph traversal into a single tree-path count, and that is the C++17 program I ship.
