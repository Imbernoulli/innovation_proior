One telescope, one night, and `n <= 2*10^5` candidate observations, each a half-open interval `[s_i, e_i)` carrying a value `v_i >= 0`; I want a pairwise-non-overlapping subset of maximum total value, the empty set allowed so the answer floors at `0`. Two numbers set the tone before any algorithm. First, the sums: `2*10^5` observations worth up to `10^9` each can total `2*10^14`, twenty thousand times past the 32-bit ceiling of `~2.1*10^9`, so every value accumulator and every coordinate has to be `long long` — an `int` here is a silent wrong answer that only the large tests would expose. Second, the overlap convention, which the whole problem hinges on: the intervals are half-open, so `[s_i,e_i)` and `[s_j,e_j)` conflict only when they share an interior point (`s_i < e_j` and `s_j < e_i`); two that merely touch, `e_i == s_j`, may both run. Restated the way I will actually use it: an observation with start `s` is compatible with an earlier one ending at `e` iff `e <= s`. That single `<=`-versus-`<` is the whole problem's off-by-one, so I keep it in front of me.

The tempting cheap answers are greedy, and there are two. Value-greedy takes the highest-value observation that still fits and blocks whatever it overlaps; earliest-finishing scans by finish time and takes each start `>=` the last finish — the textbook rule that maximizes the *count* of chosen intervals. Both are `O(n log n)` and a handful of lines. But the non-overlap constraint is global while a greedy commits locally, and once intervals carry weights the count-maximizer has no reason to chase value. Before trusting either I try to break them, on five observations across a 90-minute night:

```
A [0,30)  v=20
B [0,90)  v=50
C [30,60) v=25
D [60,90) v=25
E [40,50) v=5
```

Value-greedy grabs B `[0,90)` worth 50 and hogs the entire night, blocking everyone else, for a total of `50`. Earliest-finishing sorts by end — A(30), E(50), C(60), B(90), D(90) — takes A, then the scrap E `[40,50)`, then D, for `20+5+25 = 50`. But A `[0,30)`, C `[30,60)`, D `[60,90)` tile the night, touching cleanly at 30 and 60 under the half-open rule, for `20+25+25 = 70`. Both greedies lose, and I can see why: one fat interval hogging the resource, or a tiny early-finishing scrap wrecking a clean tiling, each costs more than it pays. A random audit later confirms this is the common case, not a contrived corner — value-greedy misses the optimum on roughly a fifth of small instances and earliest-finishing on over half. Greedy is out.

So I go to the interval DP I can prove. Sort by finishing time, `e_0 <= e_1 <= ... <= e_{n-1}`, and let `dp[i]` be the best total value over the first `i` sorted observations. For observation `i-1` with start `s` there are two moves: skip it and keep `dp[i-1]`, or take it for `v_{i-1}` plus whatever earlier observations still fit. An earlier `j` fits iff `e_j <= s`, and because the array is end-sorted those compatible predecessors are exactly a prefix `0..p-1`, where `p = #{ j : e_j <= s }`. Hence `dp[i] = max(dp[i-1], v_{i-1} + dp[p])`, with `dp[0]=0` and answer `dp[n]`. The end-sort is load-bearing twice over: it is what makes the compatible predecessors a contiguous prefix, and it is what makes `ends[0..i-2]` an ascending array I can binary-search for `p`. Sort by start instead — an easy slip, since intervals are usually named start-first — and `ends` stops being monotone, the binary search returns garbage, and the recurrence silently builds on incompatible predecessors (`X[0,6)5, Y[1,3)4, Z[3,5)4` would return `9` instead of the true `8`). So the sort key is end, tie-broken by start.

The one transcription trap the half-open rule genuinely sets is the binary-search boundary. I need `p` to count the earlier ends `<= s`, which is the position of the first end strictly greater than `s` — that is `upper_bound`, not `lower_bound`. `lower_bound(ends, s)` returns the first end `>= s`, so it swallows an end equal to `s` and reports that touching predecessor as *incompatible*, exactly backwards. The smallest witness is `[0,5) v=10` and `[5,10) v=10`, which touch at 5 and should sum to 20: for the second observation `lower_bound` lands `p=0` and the DP returns 10, while `upper_bound` lands `p=1` and returns 20. `upper_bound` it is. (And it does not over-shoot: a genuine overlap like `[0,5)` and `[4,10)` gives `s=4`, `upper_bound` finds no end `> 4` among `[5]`, so `p=0` and only one of the two survives — correct.)

On the sample the recurrence climbs through end-sorted order A(30), E(50), C(60), B(90), D(90): `dp = 20` (A), `25` (E on A), `45` (C on A), `50` (B alone), and at D with `s=60` the `upper_bound` lands `p=3` — A, E, C all finish by 60 — reading `dp[3]=45` for `25+45=70`. That last step is the A+C+D tiling both greedies missed, D building on C, C on A, and it is where the compatible-prefix lookup earns its keep.

The I/O contract needs one check the recurrence does not cover: for `n=0`, `cin >> n` reads 0, the loops skip, and `dp[0]=0` prints `0`; a genuinely empty stream fails the read and returns 0 having printed nothing, which the contract permits since it guarantees `n` is present. The remaining corners fall out of the recurrence for free — a touching chain all schedules, a pile of mutually overlapping intervals admits exactly one, zero-value observations drop out of the `max` — and the binary search only ever reads the filled prefix `ends[0..i-2]`.

Against an exponential brute force that checks every subset for pairwise half-open overlap, the DP agrees on 1100+ random small instances — some with tiny coordinate ranges to force exact-endpoint touching and collisions, some with wider `n` and values — with zero mismatches; the same harness is where the greedy miss-rates came from. The `O(n log n)` interval DP — sort by end, `upper_bound` for the compatible prefix, `long long` accumulators — is what I ship; the full program is in the answer.
