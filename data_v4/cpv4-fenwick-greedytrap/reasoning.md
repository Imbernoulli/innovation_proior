The scoring here is a **sum** over a strictly increasing subsequence, and two things about that combination are settled before I choose an algorithm. First, "increasing subsequence" is LIS-shaped phrasing, but the objective is the sum of the chosen prices, not the count — a different optimization on the same feasible set, and conflating the two is the trap the wording sets. Second, the scale fixes the data types: with `n <= 2*10^5` and `|a[i]| <= 10^9`, a strictly increasing chain of length `2*10^5` with values near `10^9` sums to about `2*10^14`, far past the 32-bit ceiling of `~2.1*10^9`; even the innocuous `1..n` case already reaches `~2*10^10`. Every score, accumulator, and Fenwick cell has to be 64-bit `long long`, or the large tests are silent wrong answers.

Two greedy heuristics tempt me: (a) take the longest increasing subsequence — surely more terms means a bigger sum; (b) take the single largest value — one fat element beats a thin chain. One instance kills both: `a = [1, 100, 2, 3, 4, 5, 6]`. The longest strictly increasing chain is `1,2,3,4,5,6`, sum `21`; the largest single value is `100`; but the optimum is `1, 100`, sum `101`. Maximizing the count optimizes the wrong functional, and grabbing one big element forfeits the free additive value of a small compatible predecessor. Greedy is out.

So I want the best score of an increasing subsequence *ending* at `i`. The only thing that lets me append `i` to a prefix is a previously chosen element `j` with `j < i` and `a[j] < a[i]`; among those I take the largest `f[j]`, and I may also start a fresh chain at `i` (no predecessor), i.e. add `0`:

`f[i] = a[i] + max(0, max{ f[j] : j < i and a[j] < a[i] })`,   answer `= max_i f[i]`.

The `max(0, ...)` is what lets a chain restart, so an all-negative array returns its least-negative single element rather than chaining negatives into a worse sum. On the sample `a = [3,1,4,1,5,9,2,6]` the `f` values come out `3, 1, 7, 1, 12, 21, 3, 18` — `f5 = 9 + f4 = 9 + 12 = 21` realizes the chain `3 < 4 < 5 < 9` — and their max is `21`, the expected answer.

The inner `max{ f[j] : j < i, a[j] < a[i] }` is the bottleneck; scanning `j` is `O(n^2)`. Processing left to right and storing each computed `f[i]` at the *value* `a[i]` turns that inner max into a prefix maximum on the value axis over already-inserted elements — exactly what a Fenwick tree delivers in `O(log n)` with point-max update and prefix-max query, for `O(n log n)` total. Values span `[-10^9, 10^9]`, so I coordinate-compress to 1-based ranks `1..m`. A max-Fenwick is valid here only because the updates never lower a cell — I always fold in with `max` and only ever read prefix maxima, which is all the recurrence asks for.

The one place strictness lives is the query bound. "Strictly less than `a[i]`" means ranks `[1 .. rank(a[i]) - 1]`, *not* inclusive of `rank(a[i])`: duplicates share a rank, and an equal predecessor must never extend a strictly increasing chain. That off-by-one is exactly the kind the constraints invite, so I pin it on the minimal witness `a = [2, 2]`, whose answer is `2`. An inclusive query lets the second `2` see the first `2`'s stored `f` and chain to `4`; querying `rank - 1` reads the empty prefix and gives `2`. So the query is `queryPrefixMax(r - 1)`, and at `r = 1` it reads the empty prefix — the correct "no smaller predecessor" signal.

The additive form also needs `max(bestPrev, 0)` rather than a raw `a[i] + bestPrev`. On `a = [-5, -3]`, both single elements and the pair `{-5,-3}` are feasible, and the best is `-3`. A raw add at the second element would compute `-3 + (-5) = -8`, forcing a negative predecessor in; `max(bestPrev, 0)` lets the chain restart and yields `-3`. So the guard is load-bearing for negatives, not merely for the empty/no-predecessor case.

Empty Fenwick cells start at `NEG = LLONG_MIN/4`, small enough that folding several of them through `max` cannot underflow. `NEG` is only ever compared, never summed — I always add `a[i]` to `max(bestPrev, 0)`, so a leaked `NEG` collapses to `0` and never reaches a score.

The corners fall out: `n = 0` prints `0` by convention, short-circuited before building a size-0 Fenwick; `n = 1` returns the single element; a strictly decreasing array leaves every `f[i] = a[i]` and answers the largest element; a duplicate-heavy array cannot chain equals; an all-negative array returns its least-negative element.

An independent exponential brute force — enumerate every non-empty subset, keep the strictly increasing ones, take the max sum — agrees with the Fenwick DP across hundreds of random small cases spanning all-negative, all-positive, wide-range, duplicate-heavy, and spike-among-small-run shapes, and the sample prints `21`. At `n = 2*10^5` the program stays well inside the `1 s` / `256 MB` budget; the cost is `O(n log n)` time and `O(n)` space. The full self-contained program is in the answer.
