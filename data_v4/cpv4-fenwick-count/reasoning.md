Counting unordered index pairs whose absolute difference lands in a band `[L, R]`. Two things about this problem set the shape of everything downstream. First, the band is *two-sided* around each value — a partner of `v` can sit below it (in `[v-R, v-L]`) or above it (in `[v+L, v+R]`) — and when `L = 0` those two sides touch at `v` itself, which is exactly where a naive count doubles. Second, the scale forces my data types before I write a line: `n <= 2*10^5` means the answer can reach `C(n,2) = n(n-1)/2 ~ 2*10^10`, past the 32-bit ceiling of `~2.1*10^9`, so the accumulator is `long long`; and with `|a[i]| <= 10^9` and `R <= 2*10^9`, a band endpoint like `v + R` reaches `~3*10^9`, so values and endpoints are 64-bit too. An `int` anywhere on the count or the endpoints is a silent wrong answer on the large tests.

There are two ways to beat the `O(n^2)` brute force. Sort-plus-two-pointers is `O(n log n)` but the two-sided band means each value's partners split into two windows, and a window counts *ordered* contributions, so I would divide by two and hand-handle the `L = 0` self-overlap — correct but fiddly to keep unordered-exact. A Fenwick sweep gives me a cleaner correctness argument: process `j = 0..n-1`, and before inserting `a[j]` query how many *already-inserted* elements (indices `i < j`) fall in the band around it. Because I only ever query earlier elements, each unordered pair `{i, j}` is counted exactly once, from its later index `j` — the count-once property is structural, not something I have to divide out. I take the Fenwick route.

Fix `j` with value `v = a[j]`. Splitting `L <= |a[i] - v| <= R`: the `a[i] <= v` side gives `a[i] in [v-R, v-L]` (lower band), the `a[i] >= v` side gives `a[i] in [v+L, v+R]` (upper band). For `L >= 1` these are disjoint — the lower top is `v-L <= v-1`, the upper bottom is `v+L >= v+1` — but at `L = 0` both bands include `v`, and that boundary is where I expect trouble. Values span `[-10^9, 10^9]`, too wide to index directly, so I compress; the useful observation is that I only ever *insert* actual array values, so the compressed key set is exactly the distinct `a[i]`, and query endpoints (`v-R`, `v+R`, ...) never need to be inserted — they are just binary-searched into the sorted key list. So `ks` = sorted-unique `a`, a 1-indexed Fenwick `bit[1..m]` accumulating counts at compressed positions, `pref(p)` returning the count at positions `1..p`, and `rangeCount(lo, hi)` = keys with value in `[lo, hi]` = `pref(hiIdx) - pref(loIdx)` where `hiIdx` counts keys `<= hi` and `loIdx` counts keys `< lo`. The inclusive endpoints after compression are the other place an off-by-one hides.

The `L = 0` boundary I flagged wants the smallest input that exercises it. Add both bands unconditionally on `a = [5,5,5,5]`, `L = 0`, `R = 3`. Every pair has difference `0 in [0,3]`, so the true answer is `C(4,2) = 6`. Compressed `ks = [5]`. Sweeping: `j=0` queries an empty tree (0), inserts. `j=1` sees one `5`; the lower band `[v-R, v-L] = [2,5]` counts it (1) and the upper band `[v+L, v+R] = [5,8]` *also* counts it (1), so this `j` adds `2`. `j=2` adds `4`, `j=3` adds `6` — total `12`, exactly double the true `6`. The defect is precisely the boundary I flagged: at `L = 0` the lower band top is `v-0 = v` and the upper band bottom is `v+0 = v`, so their union overlaps at the single value `v`, and every earlier element equal to `v` gets counted once from each band. The fix is to collapse `L = 0` into a single merged band `[v-R, v+R]` — one query, no internal overlap. Re-tracing `[5,5,5,5]` with the merged band `[2,8]` gives `1+2+3 = 6`. For `L >= 1` the two disjoint bands stay.

That leaves the inclusive-range indices, which I stress on a band whose edge lands exactly on a key: `a = [2,5]`, `L = R = 3`, so I want pairs with `|diff| = 3`; `|2-5| = 3`, true answer `1`. The trap is writing the upper index as `lower_bound(ks, hi) - begin` ("first key `>= hi`") when I mean "keys `<= hi`". Trace `j=1`, `v=5`, tree `{2}`, lower band `[2,2]`: with `lower_bound` for the upper end, `hiIdx = lower_bound(ks,2) = 0`, `loIdx = lower_bound(ks,2) = 0`, so `hiIdx <= loIdx` and the count is `0` — the code returns `0`, wrong. The `lower_bound` upper end *excludes* the key equal to `hi`, dropping exactly the band-edge partners (`|diff| = R` and `|diff| = L`). The correct upper index is `upper_bound(ks, hi) - begin` (first key strictly `> hi`): now `hiIdx = 1`, `loIdx = 0`, `pref(1) - pref(0) = 1`, right. So the rule is upper end `upper_bound` (keys `<= hi`), lower end `lower_bound` (keys `< lo`), with a `hiIdx <= loIdx -> 0` guard.

To confirm both fixes together I ran the corrected program against an independent `O(n^2)` brute force over 1500 random small cases — narrow value ranges so equal values and band edges occur constantly, and `L` biased toward `0` to hammer the merged-band path — with zero mismatches. The corners check out for their own reasons: `n = 0` and `n = 1` sweep an empty (or immediately-queried-empty) tree and give `0`; `L = R = 0` becomes the merged band `[v,v]`, counting earlier equal-value elements once each; a band no pair can hit (e.g. `L = R = 5` on `[1,2,3]`) queries empty ranges and gives `0`; and full-range values with `R = 2*10^9` stay inside the 64-bit endpoints and accumulator. The query-before-insert order is load-bearing: inserting first would let an element match itself when `L = 0`, so I always query then insert `a[j]`.

The cost is `O(n log n)` — one sort to compress, then `n` iterations of `O(1)` binary searches and `O(log m)` Fenwick ops — well inside 1 s and a few MB at `n = 2*10^5`.
