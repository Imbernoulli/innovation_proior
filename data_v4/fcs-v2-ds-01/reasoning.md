The statistic is `P(l, r) = sum over distinct v of (count of v in a[l..r])^2`, and the first thing I notice is that it is not a decomposable aggregate. A sum or a max over `[l, r]` can be assembled from prefix data or a segment tree because each element contributes independently; `P` is the opposite, because the contribution of a value `v` is `(count of v)^2`, which couples every occurrence of `v` in the window. Knowing `P(l, m)` and `P(m+1, r)` tells me almost nothing about `P(l, r)`: a value straddling the split has its two partial counts squared separately on the two sides but must be squared *together* in the union. So prefix sums and segment-tree-over-the-array are off the table — I need the whole frequency table of each window. That, plus the fact that all `q` queries are given offline and `n, q` are both up to `2*10^5`, is the signature of a Mo's-algorithm problem, and the whole difficulty is going to sit in a constant factor rather than in the asymptotics.

One scale fact I pin before writing any code, because it is a silent-wrong-answer trap the constraints invite: a full-range query on a constant array gives count `n` on a single value, so `P = n^2 = 4*10^10`. That is past the 32-bit ceiling of `~2.1*10^9`, so the running statistic and every answer must be 64-bit `long long`; an `int` accumulator would overflow exactly on the all-equal large tests, and nowhere else — the kind of bug that passes every small test and fails the one that matters.

The honest brute force is: for each query sweep `a[l..r]`, tally counts in an array indexed by value, sum the squares, zero the counts back out. Obviously correct — it is my reference oracle — but `O(r-l+1)` per query, `O(nq)` overall. At the limits that is `4*10^10` element touches; a test with `q = 2*10^5` full-range queries forces `2*10^5` sweeps of length `2*10^5`, tens of seconds against a 2-second limit. This is the wrong complexity class, not a slow constant, so I have to share work between overlapping windows instead of recomputing each from scratch.

What makes sharing possible is that `P` admits an `O(1)` incremental update. If I keep `cnt[v]` and `cur = sum_v cnt[v]^2` for a window and one element of value `v` (current count `c`) enters, `cur` changes by `(c+1)^2 - c^2 = 2c+1`; when it leaves, `cur` drops by `c^2 - (c-1)^2 = 2c-1`. Both are `O(1)` given `cnt[v]`. So if I process the windows in an order where consecutive windows differ by only a few endpoint positions, the total cost is (total endpoint movement) units at `O(1)` each. The whole game reduces to ordering the queries so the endpoints `(l, r)` travel as little as possible in aggregate.

That is Mo's algorithm. Treat each query as a point `(l, r)` in the plane; moving from one query to the next costs the Manhattan distance between the points, so I want a short tour of the point set. The classic ordering picks a block size `B ~ sqrt(n)` and sorts by `(l / B, then r)`: within an `l`-block `r` sweeps monotonically, so `r` moves `O(n)` per block over `O(n/B)` blocks, `O(n^2/B)` total; `l` stays within its block, `O(qB)`. Balancing at `B = n/sqrt(q)` gives `O((n+q) sqrt n) ~ 9*10^7` moves, which fits.

But the constant is what this problem is really testing, and block ordering has a known weak spot: it is a row-major scan of the 2-D points, discontinuous at block boundaries. When `r` finishes sweeping up to `n` in one block, the next block restarts it near the bottom — a "carriage return" jump. The even/odd-block `r`-direction trick mitigates that particular snap, but two queries physically close in the `(l, r)` plane can still land in adjacent blocks and be visited far apart in the tour. Row-by-row scanning has bad locality at the row ends, and at `n = q = 2*10^5` that inflated constant is the difference between passing comfortably and timing out.

The fix is to linearize the point set along a curve with no such cliffs: a Hilbert space-filling curve. It visits every cell of a `2^k x 2^k` grid in an order where consecutive cells are always grid-neighbors (Manhattan distance 1), and it has the best locality of the standard space-filling curves — nearby 2-D points map to nearby 1-D indices, with no long-range boundary jumps. So I embed each `(l, r)` on a `2^k x 2^k` grid with `2^k >= n`, compute its Hilbert distance, and sort the queries by it. The same `O((n+q) sqrt n)` bound holds — it is a property of any locality-respecting tour of the grid — but the constant is smaller, and that constant-factor win is what the `2*10^5` limits reward.

The one delicate piece is the mapping itself. I need `hilbertOrder(x, y, order)` sending `(x, y)` with `0 <= x, y < 2^order` to its distance along the curve. The canonical iterative `xy2d` walks the bits from most to least significant: at level `s` it reads `rx = bit_s(x)`, `ry = bit_s(y)`, adds the quadrant contribution `((3*rx) ^ ry) << 2s` (the `3*rx XOR ry` is the Gray-code quadrant index that keeps the curve continuous across quadrant boundaries), and then rotates/reflects the coordinates so the recursion into the sub-quadrant uses the correctly oriented copy of the base curve. The reflection is within the *entire* grid — `x -> side-1-x` with `side-1 = (1<<order)-1` — followed by a swap of `x` and `y` when `ry == 0`.

Getting that rotation right is the whole risk, and it fails in an insidious way. If I bungle the reflection or drop the swap, the map often stays a perfectly good bijection — every cell still gets a distinct index — but the resulting order is no longer *continuous*: consecutive cells jump across the grid, and the locality that justified the Hilbert curve evaporates. Crucially, Mo's algorithm prints the correct answers under *any* ordering of the queries, so a broken curve degrades only speed, never correctness. That means a differential test against brute — which only compares answers — can never catch a curve bug. I have to verify the curve on its own terms.

So I check it structurally, not by answers: for every `order` from 1 to 8, enumerate all `side^2` cells, compute `d` for each, and assert (a) every `d` lands in `[0, side^2)`, (b) all `d` are distinct, and (c) sorting cells by `d`, every consecutive pair is a grid-neighbor at Manhattan distance exactly 1. It is (c) that does the discriminating work — a mis-rotated curve sails through the bijection checks (a) and (b) but produces long jumps in (c). The full-grid-mask version passes all three through order 8 (256x256, 65536 cells).

The driver maintains `curL, curR` for the current window, starting empty at `curL = 0, curR = -1` with `cur = 0`, and moves the endpoints for each query with four while-loops whose order matters: grow before shrink — add on the right (`add(++curR)` while `curR < R`), add on the left (`add(--curL)` while `curL > L`), then remove on the right, then remove on the left. Growing first keeps the window a superset of both the old and new windows at each individual add, so no `remove` ever runs on a zero count and `cur` never goes spuriously negative mid-transition. A single-point query `l == r` adds one element and the other three loops are no-ops, giving `cur = 1`.

With the curve trusted, I differential-test the whole program against a Python brute that `Counter`s each subarray and sums squares, over 600 random seeds spanning the stress regimes — heavy collisions with large counts (which exercise the `2c+1`/`2c-1` deltas at large `c`), mostly-distinct arrays, all-equal arrays, and random `[l, r]` including single points — with zero mismatches. The explicit edges also pass: the sample `a = [1 2 1 3 1 2]` gives `14, 5, 3, 1`; `n = 1` gives `1`; all-equal length-5 full range gives `25`; all-distinct full range gives `5`; four single-point queries give `1,1,1,1`; `n = 2` with `[9, 9]` gives `1, 4, 1`.

The remaining details are the ones the constraints force. `cur`, the answers, and the count `c` inside each delta are `long long` — the 64-bit width the opening's overflow argument demanded, since `2*c+1` at `c ~ 2*10^5` and the `4*10^10` answers must not truncate. `cnt` is sized `maxv + 2` so every `cnt[v]` access is in bounds (values are `>= 1`, index 0 unused). I build the whole output into one string and `fputs` it once rather than flushing per line, which matters at `q = 2*10^5`. Measured, the program runs in about 0.18 s and ~12 MB at `n = q = 2*10^5`, well inside 2 s / 256 MB.
