**Problem.** Given one integer `n` (`0 <= n <= 10^6`), count the number of **unordered** pairs `{x, y}` with `1 <= x < y <= n` and `lcm(x, y) <= n`. Each pair is counted once. Read `n` from stdin, print the count. For `n = 6` the answer is `9`.

**Key idea — count by gcd.** Factor out `g = gcd(x, y)` and write `x = g*a`, `y = g*b` with `gcd(a, b) = 1`. Then `lcm(x, y) = g*a*b`, and `x < y` is exactly `a < b`. Because `lcm` is a multiple of both `x` and `y`, the single condition `lcm = g*a*b <= n` already forces `x, y <= n`. For a fixed coprime `a < b`, the admissible `g` are `1 .. floor(n/(a*b))`. Hence

```
count = sum over a < b, gcd(a, b) = 1, a*b <= n  of  floor(n / (a*b)).
```

Enumerate `a` while `a*(a+1) <= n` (so a valid `b > a` with `a*b <= n` can exist), and `b` from `a+1` while `a*b <= n`. This is `O(n log n)`-ish and clears `n = 10^6` in about `0.11 s`.

**Pitfalls.**
1. *The double-count from "count ordered, divide by two."* The gcd parameterization is naturally ordered. The tempting shortcut — sum `floor(n/(a*b))` over *all* coprime `(a, b)` and halve — is wrong twice over: it keeps the diagonal `a = b` (only `a = b = 1`, contributing `n` phantom `{g, g}` "pairs" that don't exist), and any halving must happen *once at the very end*, not inside the loop. The robust cure is to never generate both orientations: enumerate `a < b` strictly (`b` starts at `a + 1`), so there is nothing to halve and no diagonal. A trace of `n = 6` catches the broken version printing `5` instead of `9` (it had passed `n = 2` only by coincidence).
2. *The second double-count from skipping coprimality.* If a non-coprime `(a, b)` is let through, the pair it generates is also generated by its reduced shape, inflating the count (e.g. `{2,4}` would be counted via both `(2,4,g=1)` and `(1,2,g=2)`). The `gcd(a, b) != 1` skip is what makes the reduced shape the unique representative.
3. *64-bit accumulator.* The answer reaches `~3.6*10^7` for `n = 10^6`; use `long long`. Also write `lcm = x / g * y` (not `x*y/g`) in any brute checker to avoid intermediate overflow.

**Edge cases.** `n = 0` and `n = 1`: no pair `x < y` exists, loop guard `a*(a+1) <= n` fails immediately, answer `0`. `n = 2`: only `{1,2}`, answer `1`. All handled by the loop bounds with no special-casing; `a*b <= n <= 10^6` keeps every product inside 64-bit.

**Complexity.** Outer `a` up to `~sqrt(n)`, inner total `~n log n`; `O(1)` extra space.

**Code.**

```cpp
#include <bits/stdc++.h>
using namespace std;

static long long gcdll(long long a, long long b) {
    while (b) { long long t = a % b; a = b; b = t; }
    return a;
}

int main() {
    long long n;
    if (!(cin >> n)) return 0;

    // Count unordered pairs {x, y}, 1 <= x < y <= n, with lcm(x, y) <= n.
    // Write x = g*a, y = g*b, g = gcd(x, y), gcd(a, b) = 1. Then x < y  <=>  a < b,
    // and lcm = g*a*b. The constraint lcm = g*a*b <= n already forces both x, y <= n
    // (since lcm >= y > x). So the count is:
    //     sum over a < b with gcd(a, b) = 1 and a*b <= n  of  floor(n / (a*b)).
    // The a < b ordering is what makes each unordered pair counted exactly once.
    long long ans = 0;
    for (long long a = 1; a * (a + 1) <= n; a++) {       // need a < b  =>  a*b >= a*(a+1)
        for (long long b = a + 1; a * b <= n; b++) {     // strictly a < b: no a == b term
            if (gcdll(a, b) != 1) continue;              // a, b must be coprime
            ans += n / (a * b);                          // g ranges 1..floor(n/(a*b))
        }
    }

    cout << ans << "\n";
    return 0;
}
```
