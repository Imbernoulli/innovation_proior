The cost rule is what lifts this above an ordinary maze: stepping to a strictly higher cell costs `1`, every other step costs `0`, and that asymmetry is *directional*. The edge `(0,0) -> (1,0)` for heights `1 -> 3` is a boost costing 1, but its reverse `3 -> 1` is a free glide costing 0 — the same pair of cells carries different weights depending on which way I cross it. So I cannot model this as an undirected weighted graph; it is a directed graph with edge weights in `{0,1}`, and that single fact decides which algorithm is correct. The scale pins the algorithm class: `n, m <= 1000` gives up to `10^6` cells against a 1-second limit, so whatever I run has to be near-linear — `O(nm)`, no cell touched a non-constant number of times. Heights reach `10^9`, but I only ever *compare* two heights with `>`, never add them, so a plain `int` holds each height with no overflow; the output is a count of boosts, at most one per cell and well inside `int` (I keep the distance array wide anyway, for uniformity).

The reflex for "shortest path on a grid" is plain BFS, but it is the wrong tool here. Plain BFS assumes every edge weighs 1, so it returns the minimum *number of moves* — a different objective from minimum *energy* the moment a free-glide route is longer in moves than a boost-heavy one. The stated `3 x 3` sample is exactly that situation: the Manhattan distance from `(0,0)` to `(2,2)` is 4 and the grid has no walls, so plain BFS returns 4, while the true minimum energy is 1 — one boost `(0,0) -> (1,0)` for `1 -> 3`, then three free glides `3 -> 3 -> 3 -> 1` into the corner. Move-count 4 against energy 1; no tweak to the BFS repairs this, because charging free glides as moves is inherent to counting moves at all. Plain BFS is out.

What I actually have is a shortest-path problem with weights in `{0,1}`, and the specialized tool for that is 0-1 BFS — a deque standing in for Dijkstra's heap. The invariant: the deque holds frontier vertices whose tentative distances are non-decreasing and span at most two consecutive values `d` and `d+1`, so popping the front always yields a global minimum. Relaxing an edge out of a front vertex at distance `d`, a weight-0 (glide) edge reaches its neighbour still at distance `d`, so that neighbour belongs among the current `d`-vertices at the *front*; a weight-1 (boost) edge reaches `d+1`, so it belongs at the *back*. Push-front for 0, push-back for 1 keeps the deque sorted at `O(1)` per push, `O(nm)` overall — each of the `<= 10^6` cells finalized once, each of its `<= 4` edges relaxed a constant number of times.

The edge weight is `w = (h[nr][nc] > h[r][c]) ? 1 : 0`, and the direction matters: inverting it to `h[r][c] > h[nr][nc]` would charge for gliding down and free-ride the boosts — a silent wrong answer, not a crash. The `1 -> 3` step to a higher cell must cost 1, and `h[nr][nc]=3 > h[r][c]=1` gives `w=1`, so the orientation is right.

The one real subtlety in 0-1 BFS is stale deque entries: a vertex can be pushed more than once, because its distance can improve while an earlier copy of it still sits in the deque. The clean way to stay correct without extra bookkeeping is to store nothing in the deque but coordinates, read `d = dist[r][c]` fresh on each pop, and guard every relaxation with `if (d + w < dist[nr][nc])`. A stale pop is then harmless: `dist[r][c]` already holds its final small value, so each relaxation it attempts either genuinely improves a neighbour or is rejected by the `<` guard. A `done`-on-first-pop flag would also work, but it adds a second piece of state to reconcile against the duplicate entries; the single distance guard is enough, so I keep no visited array.

To check the deque discipline I lean on a method that shares no code with it: an independent Bellman-Ford-style relaxation — sweep all four directed edges of every cell repeatedly until no distance changes, obviously correct, no deque order to get wrong. It agrees with the 0-1 BFS on random small grids and on the pointed shapes: flat grids, strictly increasing corridors, and columns of high cells that force exactly one boost to cross.

The edge cases fall out of the model. `1 x 1`: start is the goal, no move, answer 0. Flat grid: every step is a glide, so the answer is 0. Strictly increasing `1 x m` corridor: a boost at each of the `m-1` steps, answer `m-1`. An `n x 1` column runs the same four-direction loop with the two horizontal moves dropping off-grid. The grid is guaranteed connected, so `dist[n-1][m-1]` is finite when I print it.

So the `O(nm)` 0-1 BFS is what I ship: seed the deque at `(0,0)`, front-pop, push glides to the front and boosts to the back under the distance guard, and print `dist[n-1][m-1]`. The full single-file C++ program is the answer.
