The distinctive shape here is a take-away game where a move does not remove stones but *replaces* a pile of size `x > 1` with one of its proper divisors `y` (`y | x`, `1 <= y < x`), and a pile of `1` is dead. With `n` up to `10^5` piles and each `a[i]` up to `10^9`, two limits fix everything downstream: I cannot touch the joint state space, and per-pile work has to sit near `O(sqrt(a))` — a `sqrt(10^9) ~ 31623` factor over `10^5` piles is about `3.2 * 10^9` trivial divisions worst case, borderline but survivable. Output is a single word, `First` or `Second`, and the input may even hand me `n = 0`.

Memoized minimax over positions is the textbook decision procedure — a position is a loss for the mover iff every move hands the opponent a win — but here the state is the *multiset of all pile sizes*, and that product explodes. Three piles of `12` already reach on the order of `12^3` joint positions (each `12` walks down `{1,2,3,4,6}` and further), and at `n = 10^5` the joint space is a product over `10^5` factors. No memoization rescues that; it is exponential in the pile count. It survives only as a brute oracle on tiny inputs.

The blow-up itself points at the fix: a move touches exactly one pile and never references the others, so the whole position is a disjunctive sum of `n` one-pile games. Sprague–Grundy then applies — every position has a Grundy value, the value of a sum is the bitwise XOR of the parts, and the mover loses iff that XOR is `0`. So if I can get `G(x)` for a single pile, the answer is just: XOR all `G(a[i])`, print `First` iff nonzero. The exponential joint search collapses to `n` independent per-pile computations and a fold, and the whole difficulty migrates to one question — what is `G(x)`?

By definition `G(x) = mex { G(y) : y a proper divisor of x }`, with `G(1) = mex(empty) = 0`. Computing the first several by hand, in the hope of a closed form that spares me any per-pile recursion:

- `G(1) = mex{} = 0`; `G(2) = mex{G(1)} = mex{0} = 1`; primes `3,5,7` likewise give `1`.
- `G(4) = mex{G(1),G(2)} = mex{0,1} = 2`; `G(8) = mex{0,1,2} = 3`; `G(16) = mex{0,1,2,3} = 4`.
- `G(6) = mex{G(1),G(2),G(3)} = mex{0,1,1} = 2`; `G(9) = mex{0,1} = 2`; `G(12) = mex{0,1,1,2,2} = 3`.

So `G(1..16) = 0,1,1,2,1,2,1,3,2,2,1,3,1,2,2,4`. Primes give `1`; `G(4)=2, G(8)=3, G(16)=4` track the exponent of two; `G(12)=3` with `12 = 2^2·3` (three prime factors), `G(6)=2` from `2·3`, `G(9)=2` from `3^2`. The value is `Omega(x)`, the number of prime factors of `x` counted with multiplicity.

Sixteen values are a conjecture, and a wrong Grundy formula is a silent wrong answer, so I prove `G(x) = Omega(x)` by strong induction. Base: `Omega(1) = 0 = G(1)`. Assume `G(d) = Omega(d)` for all `d < x`, and write `x` as a multiset of `m = Omega(x)` primes. Any proper divisor `y` drops at least one of those primes, so `Omega(y) in {0,...,m-1}`; and every target `t` in that range is realized — the product of any `t` of the `m` primes is a proper divisor with `Omega = t` (`t = 0` gives `y = 1`). No proper divisor reaches `Omega = m`, since that would force `y = x`. By the hypothesis the option Grundy values are therefore exactly `{0,1,...,m-1}`, a complete prefix missing only `m`, so `mex = m = Omega(x)`. The whole solver is now: XOR `Omega(a[i])`; First wins iff nonzero — no game tree at all.

Computing `Omega(x)` for `x <= 10^9` needs trial division only up to `sqrt(x) < 31623`: strip every prime `p` with `p*p <= x`, and any cofactor `> 1` left over must itself be a single prime (two factors both above `sqrt(x)` would exceed `x`), so it adds exactly one. I sieve the primes up to `31623` once with a linear sieve and divide only by those (~3400 primes, not every integer). Pollard–Rho would be overkill at this scale — it earns its keep past `10^{12}` — and an `Omega`-sieve to `10^9` will not fit in memory; sieve-small-primes plus trial division is the right size.

One genuine trap sits in the trial-division guard. Written as `p * p > x` with `p` an `int`, the product invites overflow: the largest sieved prime is ~`31607`, and `31607^2 ~ 9.99·10^8` sits just under `2^31`, so it is safe at *this* `LIM` — but a guard that is correct only while `LIM < ~46340` is a landmine, so I compute it as `(long long)p * p > x`, and use the same 64-bit product in the sieve's `p*i` bound. The cofactor rule is sound precisely because `a[i] <= 10^9`: the unfactored remainder is always a single prime, never a prime square (a large prime squared would exceed the bound), so `if (x > 1) omega++` never undercounts.

Edge cases fall straight out of `Omega(1) = 0` and the final XOR test. `n = 0` or all-ones piles give XOR `0` → `Second` (First cannot move); both the `if (!(cin >> n)) return 0` path and a zero-length read loop land there. A single prime gives `Omega = 1` → `First`. Equal piles like `[8,8]` or `[4,9]` give equal Grundy values XOR-ing to `0` → `Second`, which is exactly the mirroring strategy — whatever First does to one pile, Second copies on the other. And `Omega <= 30` for any `x <= 10^9`, so the XOR stays a small int.

To trust the game theory rather than a pattern over sixteen numbers, I differential-test the solution against the exponential minimax brute — which assumes no Sprague–Grundy — over several hundred random small cases plus hand-picked corners (`n=0`, all ones, primes, prime powers, XOR-zero pairs); that confirms XOR-of-Grundy with `G = Omega` gives the right verdict wherever the brute can actually run. For the large values the brute cannot touch, I cross-check the `Omega`-XOR against an independent factorization, which pins down the factor-counting at scale.
